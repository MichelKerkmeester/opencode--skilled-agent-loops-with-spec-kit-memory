# v1.0.0.0 - initial release

Initial release of `create-skill`, the skill-authoring workflow packet of the `sk-doc` parent hub's ten-packet workflow-only architecture. It scaffolds, authors, and validates OpenCode skill packages under `.opencode/skills/` through two workflow modes: `create-skill` for a standalone skill with its own advisor identity, and `create-skill-parent` for a parent hub that dispatches to nested workflow or surface packets.

---

## What's Included

- `SKILL.md` - the executable workflow contract (primary): route selection between the two modes, the ordered standalone-skill steps (plan resource placement, scaffold with `init_skill.py`, author `SKILL.md`/`README.md`/`references/`/`assets/`, validate), the parallel parent-hub steps (hub root files, nested-packet shape, `mode-registry.json` and `hub-router.json` rules), plus rules and success criteria.
- `references/` - creation guidance beyond the core workflow: `skill/creation_workflow.md` and `skill/examples_and_maintenance.md` for the standalone path; `shared/overview.md`, `shared/common_pitfalls.md` and `shared/validation_and_packaging.md` for shared authoring concerns; `parent_skill/parent_skills_nested_packets.md` and `parent_skill/parent_hub_router_schema.md` for the parent-hub path.
- `assets/` - copy-from templates: `skill/skill_md_template.md`, `skill/skill_readme_template.md`, `skill/skill_reference_template.md`, `skill/skill_asset_template.md` and `skill/skill_smart_router.md` for standalone skills; `parent_skill/parent_skill_hub_template.md`, `parent_skill/parent_skill_registry_template.json`, `parent_skill/parent_skill_hub_router_template.json`, `parent_skill/parent_skill_description_template.json` and `parent_skill/parent_skill_graph_metadata_template.json` for parent hubs.
- `scripts/` - `init_skill.py` to scaffold a new standalone skill folder and `package_skill.py` as the validation and packaging gate (`--check` for structure and frontmatter checks, plain run to zip for distribution).

---

## Notes

This nested workflow packet carries no `graph-metadata.json` of its own; the single advisor identity and the `mode-registry.json` entries for both modes live at the `sk-doc` hub root.
