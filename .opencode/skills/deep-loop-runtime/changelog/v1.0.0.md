This is the first shipped release of `deep-loop-runtime`. The earlier `v0.1.0` state was a phase-001 scaffold. Version `1.0.0` marks the runtime as populated and verified: shared deep-loop libraries, coverage-graph ownership, direct script entry points, SQLite storage, and runtime tests now live together in the peer skill instead of inside `system-spec-kit/mcp_server/`.

> Spec folder: `.opencode/specs/skilled-agent-orchestration/118-deep-loop-full-isolation-no-mcp/008-verification-changelog-closeout/` (Level 2)

---

## Why This Release Matters

Before 118, every change to the deep-review or deep-research loop infrastructure required touching files inside `system-spec-kit/mcp_server/`. Two consumer skills (`deep-review`, `deep-research`) depended on internals of a third skill (`system-spec-kit`), and the MCP tool surface (`mcp__mk_spec_memory__deep_loop_graph_*`) added a marshalling layer between workflow YAMLs and the SQLite-backed coverage graph. The arrangement was accumulated drift from early MCP server scaffolding; no ADR justified it. Arc 118 (FULL_ISOLATE_NO_MCP, user-directive override of the 117 AI Council SPLIT ruling) consolidated all deep-loop runtime infrastructure into this peer skill, removed the MCP tool surface entirely, and replaced it with direct `.cjs` script invocation — the same pattern `reduce-state.cjs` already used. Workflow YAML calls became one-step (`bash: node .../scripts/<name>.cjs`) instead of two-step (MCP-dispatch then JSON-parse).

This release ships the consolidation. New contributors fixing a deep-review bug now work entirely within `deep-loop-runtime/` and `deep-review/` — no MCP-server traversal required.

## New Features

#### Runtime libraries

- **Deep-loop support code** — Ten shared TypeScript modules now live under `.opencode/skills/deep-loop-runtime/lib/deep-loop/`: executor config, executor audit, prompt-pack rendering, post-dispatch validation, atomic state, JSONL repair, loop locking, permissions, Bayesian scoring, and fallback routing. Each carries a `MODULE:` header per the OpenCode TypeScript style guide.

#### Coverage graph ownership

- **Graph schema and queries** — Three coverage-graph modules live under `.opencode/skills/deep-loop-runtime/lib/coverage-graph/`: `coverage-graph-db.ts` (SQLite schema + connection lifecycle + node-kind allow-list), `coverage-graph-query.ts` (query builders), and `coverage-graph-signals.ts` (convergence signal extraction). The runtime-owned SQLite database is at `.opencode/skills/deep-loop-runtime/storage/deep-loop-graph.sqlite`.

#### Direct script entry points

- **No MCP graph tools** — Four CommonJS entry points replace the removed MCP tool surface:
  - `scripts/convergence.cjs` (replaces `mcp__mk_spec_memory__deep_loop_graph_convergence`)
  - `scripts/upsert.cjs` (replaces `_upsert`)
  - `scripts/query.cjs` (replaces `_query`)
  - `scripts/status.cjs` (replaces `_status`)
- Each script parses CLI args (`--spec-folder`, `--loop-type`, `--session-id`, etc.), opens the SQLite DB in a try block, calls the relevant lib function, writes JSON to stdout, and closes the DB in a finally block.
- Exit codes are standardized: 0=ok, 1=script error, 2=DB error, 3=input validation error.
- Workflow YAMLs and collateral (e.g., `/doctor` health checks, system-code-graph playbook scenario 009) call these scripts directly with `bash: 'node .../scripts/<name>.cjs ...'`.

#### Runtime tests

- **Split by responsibility** — Runtime-focused unit, integration, and lifecycle tests moved into `.opencode/skills/deep-loop-runtime/tests/`:
  - `tests/unit/` (13 files for executor/prompt/state/scoring/routing libs)
  - `tests/integration/` (4 script-spawn tests + 3 Phase B review-depth fixtures)
  - `tests/lifecycle/db-open-close.vitest.ts`
  - `tests/_helpers/spawn-cjs.ts` (shared spawn helper)
- The `system-spec-kit/mcp_server/vitest.config.ts` includes `'../deep-loop-runtime/tests/**/*.{vitest,test}.ts'` for cross-package test discovery, so a single `pnpm vitest run` exercises both surfaces.
- MCP-specific tests for the 4 deleted deep-loop tools were removed; the only retained test in the old location is `mcp_server/tests/deep-loop/review-depth-reducer.vitest.ts` because it tests `deep-review/scripts/reduce-state.cjs` (lives in `deep-review/`, not in the migrated runtime).

#### sk-doc canonical companions

- **feature_catalog/** — 18 files (top-level + 17 per-feature entries across 7 categories: Executor, Prompt Rendering, Validation, State Safety, Scoring, Coverage Graph, Script Entry Points)
- **manual_testing_playbook/** — 18 files (top-level orientation + 17 per-feature operator scenarios)
- **references/** — 4 reference docs: `script_interface_contract.md`, `coverage_graph_schema.md`, `state_format.md`, `integration_points.md`
- **graph-metadata.json** — peer-skill discoverability metadata (trigger phrases, key topics, entities, dependencies, related_to)

## Upgrade

Consumers should call the four `.cjs` scripts directly rather than attempting to use the deleted `mcp__mk_spec_memory__deep_loop_graph_*` MCP tools. The migration source was `.opencode/skills/system-spec-kit/mcp_server/`; after arc 118, the runtime owner is `.opencode/skills/deep-loop-runtime/`. Workflow YAMLs (`spec_kit_deep-{review,research}_{auto,confirm}.yaml`) were updated to invoke the scripts via `bash:` blocks; `/doctor` diagnostics and the `system-code-graph` playbook scenario 009 were updated correspondingly. The `system-spec-kit/mcp_server/vitest.config.ts` glob was extended to discover tests at `../deep-loop-runtime/tests/`.

For workflow YAML migration:

```yaml
# Before (deleted in arc 118):
mcp_tool:
  name: mcp__mk_spec_memory__deep_loop_graph_convergence
  parameters: { specFolder: "{spec_folder}", loopType: "review", sessionId: "{session_id}" }

# After (canonical):
bash: 'node .opencode/skills/deep-loop-runtime/scripts/convergence.cjs --spec-folder "{spec_folder}" --loop-type "review" --session-id "{session_id}"'
outputs:
  - graph_decision
  - graph_signals_json
  - graph_blockers_json
  - graph_blockers_csv
  - graph_stop_blocked
```
