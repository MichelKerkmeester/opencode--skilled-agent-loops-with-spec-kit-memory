### Already covered by audit

Audit already requires severity-ordered P0-P3 findings, evidence, category, impact, recommended fix, owner, and a five-dimension `/20` score. [SOURCE: .opencode/skills/sk-design/design-audit/references/anti_patterns_production.md:29]

### Out of scope or should be constrained

- Do not import seven visual-critique dimensions as new top-level audit score dimensions. Audit already owns a five-dimension score, and changing that would create incompatible report semantics. [SOURCE: .opencode/skills/sk-design/design-audit/references/audit_contract.md:29]
- Do not require `mood.md`, `voice.md`, and `tokens.md` for every audit. The adoptable part is the evidence guard, not a universal project-file convention. The corpus itself says missing files should be noted and skipped rather than invented. [SOURCE: .opencode/specs/design/008-sk-design-parent/external/designer-skills-main/visual-critique/skills/critique-brand-consistency/SKILL.md:14]
- Do not make audit apply token fixes or perform exhaustive token remediation. Audit maps findings to owners; foundations and `sk-code` own the actual changes. [SOURCE: .opencode/skills/sk-design/design-audit/SKILL.md:315] [SOURCE: .opencode/skills/sk-design/design-audit/SKILL.md:381]
- Treat numeric hierarchy thresholds from the corpus as calibration prompts, not hard release gates, unless the issue is accessibility. The visual hierarchy skill suggests 1.5x differentials, while typography suggests >=1.25x scale contrast; audit should file the observed hierarchy failure and impact rather than enforce both numbers mechanically. [SOURCE: .opencode/specs/design/008-sk-design-parent/external/designer-skills-main/visual-critique/skills/critique-visual-hierarchy/SKILL.md:22] [SOURCE: .opencode/specs/design/008-sk-design-parent/external/designer-skills-main/visual-critique/skills/critique-typography/SKILL.md:15]

## Questions Answered

- Which visual-critique rubric language is genuinely net-new or sharper than audit's existing critique references? The strongest additions are hierarchy sub-probes, composition scan, affordance/discoverability checks, information-density scanability, optional brand-evidence guards, measured color evidence, and a compact typography audit probe.
- Which audit target files should receive the future edits? Most belong in `.opencode/skills/sk-design/design-audit/references/critique_hardening.md`; color evidence belongs in `.opencode/skills/sk-design/design-audit/references/audit_contract.md`. No change is justified to the top-level `design-audit/SKILL.md` beyond possibly linking the new sub-probes after the reference edit.
- Which material is already covered? P0-P3 reporting, five-dimension scoring, token drift, production states, touch-target thresholds, and AI-tell detection are already covered by audit and should not be re-imported.
- Which material is out of scope? New score dimensions, mandatory brand-doc conventions, audit-side implementation fixes, and mechanical hierarchy ratio enforcement are out of scope or need constraints.

## Questions Remaining

- In the eventual build phase, should typography and color sub-probes be placed directly in `critique_hardening.md`, or should audit keep them as short pointers to `foundations` and `accessibility_performance` to avoid duplicating deeper system guidance?
- Should the future audit report template include an optional "Visual critique probes assessed" row, or is it enough for findings to cite the relevant sub-probe? My read: keep it out of the template unless users ask for a visual-only audit mode.
- Does the design-systems corpus add a better token-audit evidence schema than the visual-critique brand/token compliance language?
