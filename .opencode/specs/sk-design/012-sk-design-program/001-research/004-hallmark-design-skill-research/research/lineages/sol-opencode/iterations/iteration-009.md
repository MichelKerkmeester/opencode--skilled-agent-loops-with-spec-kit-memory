## Findings

### 1. `assets.md` contributes a provenance receipt, not a source catalog

Shipped Interface already prefers real or generated section-specific imagery, requires correct aspect ratios and alt text, and rejects fake screenshots. Hallmark adds useful operational concerns around remote-host risk and generated-asset provenance, but its named providers, prices, counts, license labels, and model claims are volatile and cannot be copied as authority. Adapt the existing Interface preflight and shared handoff with an asset receipt containing source/provider, acquisition URL or generation model, license/terms URL and retrieval date, owner/author when known, local-versus-remote status, prompt/reference-input sensitivity, post-processing, alt text, dimensions, and unresolved rights. A watermark or user-supplied reference image is provenance evidence, not proof of reuse rights. ### 4. Hallmark's proposed `data-viz.md` is already shipped; two anti-patterns merit surgical precision

Foundations already chooses charts by question, prefers simple encodings, constrains pie/donut use, starts bar axes at zero, keeps gridlines quiet, separates data color from brand color, uses shared scales for small multiples, and requires responsive and accessible alternatives. A new capability or file would duplicate a stronger owner. Add only explicit guidance against perspective/3D encodings and dual-axis charts without a demonstrated shared scale; route observed violations through Audit with measured user harm rather than a universal aesthetic ban. ### 5. Tactile rebellion is an existing Interface adaptation, not a new mode or style

Shipped Interface already permits low-opacity texture and physical grain in brief-specific aesthetic references, while rejecting crude hand-drawn fallback art. The useful addition is a controlled-imperfection decision check in the existing direction/preflight path: name the brief evidence, constrain the irregularity to one or two roles, preserve legibility and hit targets, verify responsive stability, and remove it when it becomes repeated noise. Do not expose `tactile` as a theme, dial, preset, or guaranteed trend. ### 6. The Nanobanana hook is already a provider-neutral image branch plus an external transport

Interface already gates native image generation to net-new, ambiguous, or image-led mid-fidelity work, confirms palette and image role, creates one to three brief-specific directions, critiques them, and waits for approval before code. Open Design already exposes gated `od media generate --surface image` and writes media into the active project. Therefore no Nanobanana-specific core hook, cache contract, or `Plate` theme is justified. Strengthen the existing branch with the asset receipt from finding 1 and route actual generation through a selected external transport. Require explicit confirmation, provider/model availability, cost disclosure, secret isolation, prompt/reference-image sensitivity review, output-rights review, and post-generation accessibility/performance checks. Prompt hashes must not cache sensitive briefs or reference images by default. ### 7. A live-preview MCP would duplicate shipped transports and blur judgment authority

Open Design generation already produces a local `previewUrl`, supports artifact inspection, and routes revisions through the run conversation. MD-GENERATOR already emits proof and preview artifacts, while `mcp-chrome-devtools` owns last-mile browser inspection. The real-UI loop explicitly rejects a heavyweight visual-regression engine and keeps judgment over the real render as the gate. Skip a new MCP server. A future orchestration improvement may normalize a read-only preview receipt across existing transports, but it must not auto-accept a design, silently mutate files, or treat a renderable page as a quality pass. ### 8. Sound needs a tighter existing policy; web haptics should be deferred

Motion already prohibits sound without system-setting respect and mute controls; Audit already checks captions, reduced motion, and no sound autoplay. Add an applicability card to Motion: sound must communicate a user-relevant state or belong to an explicitly audio-centric experience, default muted where appropriate, expose independent volume/mute, never become the sole feedback channel, respect interruption and repeated-frequency cost, and include captions/transcripts where media carries meaning. Haptics have no general web platform or product contract here; defer until a named target runtime, supported API, user value, accessibility alternative, permission/privacy behavior, and device test matrix exist. Open Design's ability to generate audio is transport capability, not evidence that a product should play sound. ## Phased Dependencies

### Phase A: Existing-owner hardening

1. Add the asset receipt, stylesheet-preservation field, chart cautions, controlled-imperfection check, and sound applicability card in their existing owners.
2. Add parity fixtures for handoff/preflight changes and preserve Audit's measured severity authority.
3. Keep wording independent; if substantial Hallmark prose, tables, or examples are copied, include its MIT notice.

### Phase B: Generated-media transport integration

1. Keep Interface as judgment and Open Design or another selected tool as transport.
2. Require explicit mutation approval, provider readiness, secret isolation, cost and sensitive-input disclosure, and the Phase A asset receipt.
3. Verify output rights, accessibility, performance, and post-processing before implementation handoff; never treat a watermark as rights proof.

### Phase C: Preview normalization only on demonstrated need

1. Measure whether users actually lose time moving among `previewUrl`, browser inspection, and MD-GENERATOR proof artifacts.
2. If a gap exists, define a read-only transport receipt; do not add another server, auto-critique loop, or acceptance authority.

### Phase D: Target-specific sensory capability

1. Reopen haptics only for a named runtime and product task with supported APIs and user research.
2. Require permission/privacy, device coverage, user controls, and a non-haptic accessible equivalent.

## Ruled Out

- A provider catalog or copied price/license table inside sk-design; those facts are volatile and source-specific.
- A Nanobanana-specific core hook, prompt-hash cache, or generated-image provider lock.
- A reusable image-led `Plate` theme.
- A second `data-viz.md` or standalone charts command.
- A `tactile` theme, dial, preset, or trend mandate.
- A new live-preview MCP server or automated render-to-acceptance loop.
- General web haptics guidance without a supported target runtime and current user-value path.
- Treating generated-media watermarks, URLs, or user-supplied references as proof of copyright or exact-reuse rights.

## Dead Ends

- Filename parity again overstated gaps: Hallmark's proposed `data-viz.md` maps to a complete shipped Foundations reference.
- Tool capability is not product justification: Open Design can generate image, video, or audio, but transport availability does not establish that a surface needs those media.
- A rendered preview is not a quality verdict. Existing transport output remains subordinate to Interface/Audit judgment and proof gates.
- Hallmark's provider counts, costs, model versions, and license shorthand are snapshots, not durable implementation authority.
- `preview-examples.md` demonstrates Hallmark's own presets; abstracting those examples into a sk-design catalog would revive an exhausted direction.
