# Webflow Staging vs Production Workflow

Promotion workflow from the Webflow staging subdomain (`.webflow.io`) to a production custom domain — covering CSS export, asset versioning, custom code panel sync, and CDN cache invalidation.

---

## 1. OVERVIEW

### Purpose

Webflow ships every site to a free `.webflow.io` staging subdomain by default. Production publishes to a custom domain only after an explicit "Publish to Selected Domains" action. The promotion workflow needs discipline because: (a) custom code panels are per-site (NOT per-environment), (b) CSS is generated by Webflow Designer and ALSO can be customized via Custom CSS panel, (c) externally-hosted JS bundles (e.g. R2 CDN) need version-pinning to prevent staging-vs-production drift.

### When to Use

- Promoting Webflow Designer changes from staging to production
- Updating externally-hosted JS bundles (R2 CDN) without breaking either environment
- Publishing CSS Custom Code changes (the Custom CSS panel)
- Coordinating Designer changes with R2-hosted custom JS releases
- Setting up a new Webflow project's staging-vs-production promotion discipline

### Core Principle

Treat staging as the canonical pre-production environment — every change tested there before "Publish to Selected Domains" promotes to production. Externally-hosted assets (JS bundles, fonts) must be version-pinned to survive the promotion.

---

## 2. WEBFLOW'S BUILT-IN STAGING DOMAIN

### What's automatic

- Every Webflow site has a free `[site-name].webflow.io` staging subdomain
- "Publish to" panel (top-right of Designer) shows checkboxes for staging + each connected custom domain
- Publishing to staging only → site updates at `.webflow.io` URL; custom domain unchanged
- Publishing to custom domain(s) → site updates at production URL

### Pre-publish checks

Before clicking "Publish to Selected Domains":

```markdown
□ Test latest changes on staging URL first
□ Custom code panel (Site Settings → Custom Code) reviewed for new vs removed scripts
□ External CDN assets pinned to specific version (?v=X.Y.Z) so promotion doesn't pick up untested code
□ Designer-generated CSS changes (color tokens, fluid typography) verified at multiple breakpoints
□ Form submissions tested on staging (forms post to Webflow's collection regardless of environment)
□ Schema markup / structured data validated (use Schema.org Validator on staging URL)
```

### Common staging gotcha

Webflow's `.webflow.io` subdomain has `noindex` headers — useful for preventing search-engine indexing of staging content, BUT means SEO tooling that requires crawlable URLs (Google Lighthouse for SEO scoring, schema validators that fetch via HTTP) may produce false-negative results on staging. Always re-run SEO checks on production URL after promotion.

## 3. CSS WORKFLOW (DESIGNER-GENERATED + CUSTOM CSS PANEL)

### Designer-generated CSS

The bulk of CSS comes from the Webflow Designer (class panel, style panel). This CSS is:
- Auto-generated when you publish
- Lives at `https://cdn.prod.website-files.com/[site-id]/css/[site-slug].webflow.[hash].css`
- Cache-busted by hash on every publish — no manual versioning needed
- NOT editable outside Designer

### Custom CSS panel content

Site Settings → Custom Code → Head Code can include `<style>` blocks for rules Designer can't express (advanced selectors, CSS custom properties, `@supports` queries, third-party widget overrides).

**Promotion workflow:**

1. Add `<style>` block to staging Custom Code panel
2. Publish to staging only
3. Test rules at every breakpoint (375px, 991px, 1920px) using DevTools mobile emulation
4. Once verified, click "Publish to Selected Domains" to promote
5. Re-test on production URL — CDN cache may take 1-2 minutes to invalidate

### Per-environment Custom CSS divergence

Webflow does NOT support per-environment custom CSS (one panel per site). If you need staging-only or production-only rules, the workaround:

```html
<!-- In Custom Code panel -->
<script>
  if (window.location.hostname.includes('webflow.io')) {
    document.documentElement.classList.add('env-staging');
  } else {
    document.documentElement.classList.add('env-production');
  }
</script>

<style>
  .env-staging .debug-panel {
    display: block !important;
  }
  .env-production .debug-panel {
    display: none !important;
  }
</style>
```

The `.env-staging` / `.env-production` body classes let CSS rules target one environment without forking the panel.

### CSS export workflow (for source-control)

When you need staging-and-production CSS in source control (typical for A. Nobel-style projects with `src/1_css/` mirror):

1. Make changes in Webflow Designer
2. Publish to staging
3. Use Webflow's "Export Code" feature (Site Settings → Export Code, requires Workspace plan) OR manually fetch the live CSS via DevTools (Network tab → filter `*.css` → save)
4. Save exported CSS to `src/1_css/z__nobel_general.css` (or equivalent) for version control
5. Custom CSS additions go in source-controlled per-component files (`src/1_css/[component]/[variant].css`)
6. Promotion to production via Designer "Publish to Selected Domains" — exported source CSS is for archive/reference only, NOT actively served

## 4. EXTERNALLY-HOSTED JS BUNDLE WORKFLOW

When custom JS lives on an external CDN (R2, jsDelivr, unpkg) and is loaded via the Custom Code panel, version pinning is critical for staging-vs-production discipline.

### Version-pin pattern (in Site Settings → Custom Code → Footer Code)

```html
<!-- Performance -->
<script src="https://[your-cdn].r2.dev/attribute_cleanup.min.js?v=1.4.0" defer></script>

<!-- Navigation -->
<script src="https://[your-cdn].r2.dev/nav_dropdown.min.js?v=1.3.1" defer></script>
<script src="https://[your-cdn].r2.dev/nav_mobile_menu.min.js?v=1.3.1" defer></script>
```

The `?v=1.3.1` query string serves three purposes:
1. **Cache-busting** — browsers and Cloudflare R2 treat each version as a distinct asset, never serving stale code
2. **Promotion gating** — staging and production reference the SAME version, so what you tested on staging IS what ships to production
3. **Rollback hook** — flip `?v=1.3.1` → `?v=1.3.0` to instantly revert a bad release

### Promotion workflow for JS bundle releases

1. Make changes to source JS in `src/javascript/[component]/[file].js`
2. Run minification:
   ```bash
   npx terser src/javascript/[component]/[file].js \
     --compress --mangle \
     -o z_minified/[file].min.js
   ```
3. Verify minification:
   ```bash
   node .opencode/skills/sk-code/code-webflow/assets/scripts/verify-minification.mjs
   node .opencode/skills/sk-code/code-webflow/assets/scripts/test-minified-runtime.mjs
   ```
4. Bump version in source: change `?v=1.4.0` → `?v=1.4.1` in BOTH the Custom Code panel AND `src/0_html/global.html` (mirror file)
5. Upload `[file].min.js` to R2 dashboard (or via `wrangler` CLI)
6. Publish staging → test the new version
7. If staging passes, publish production
8. If staging fails, the OLD `?v=1.4.0` version is still live — just revert the `?v` bump and re-publish staging

### Don't do this

- Never serve external JS without `?v=` version pin — staging/production drift becomes inevitable
- Never overwrite an existing `[file].min.js?v=1.4.0` in R2 with new code — bump the version instead. Overwrites cause cache-poisoning where some users get old code, some get new
- Never use `latest` or floating tags (`@latest`, branch names) for production JS — pin to a specific semver

## 5. SCHEMA MARKUP / STRUCTURED DATA SYNC

Schema.org JSON-LD lives in the Custom Code panel (Head Code). Promotion considerations:

- Schema validators (Schema.org Validator, Google Rich Results Test) fetch via HTTP — staging's `noindex` headers don't block fetching
- Always validate on staging URL FIRST, before promoting to production
- Production schema gets indexed by Google within hours of publish — broken schema can cause SEO regression

### Pre-publish schema checklist

```markdown
□ JSON valid (no trailing commas, proper escaping)
□ @context, @type, name, url all present
□ URL fields point to PRODUCTION domain (not .webflow.io) even when on staging
□ Validator returns no errors at staging URL
□ All linked resources (logo, image fields) are absolute production URLs
□ openingHoursSpecification dayOfWeek values use canonical Schema.org strings
□ Test in Google Rich Results Test for any rich-result-eligible schemas (LocalBusiness, FAQ, Article)
```

## 6. CDN CACHE INVALIDATION

### Webflow's own CDN

Publishes invalidate automatically. CSS is hash-busted, so no manual cache-busting needed.

### External CDN (R2, jsDelivr)

When you bump `?v=1.4.0` → `?v=1.4.1`:
- Browsers immediately treat as new asset → fresh fetch, no cache hit
- R2 / jsDelivr serve `[file].min.js` (the underlying file) — query string doesn't change which file is served, just bypasses cache
- 5-15 minute propagation across the CDN edge network — production users may see old version briefly

### Forcing invalidation

If you must serve fresh code at the SAME version (rare — usually a sign you should bump version instead):

- R2: Cloudflare dashboard → R2 → bucket → "Purge Cache" (covers your custom domain only; jsDelivr/unpkg edge cache is separate)
- jsDelivr: append `?nocache=1` to bypass — but this is a one-shot, not a permanent fix; bump version for a real fix

## 7. QUICK REFERENCE — PROMOTION CHECKLIST

```markdown
□ STAGING TEST
  □ Latest Designer changes published to staging only
  □ External JS bundles version-pinned (?v=X.Y.Z)
  □ Custom Code panel reviewed (no debug `console.log`, no commented-out code)
  □ Tested on Mobile (375px), Tablet (991px), Desktop (1920px)
  □ Forms submit successfully
  □ Schema validates with no errors
  □ Console clean (no errors, no 404 network requests)
  □ Animation perf 60fps on throttled CPU (DevTools 4x slowdown)

□ PROMOTE
  □ Click "Publish to Selected Domains" in Designer
  □ Wait 1-2 minutes for CDN propagation
  □ Re-test production URL with same checklist as staging
  □ Verify no SEO regression (Lighthouse SEO score, schema validator)

□ POST-PROMOTE
  □ Tag the JS source release in git (if applicable)
  □ Update release notes / changelog
  □ Monitor for production errors via Sentry / your error tracking
```
