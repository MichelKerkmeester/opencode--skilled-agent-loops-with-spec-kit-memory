`deep-alignment` joins `system-deep-loop` as a new mode-packet: a structured conformance-review loop that checks artifacts against a named authority's own creation standards (starting with sk-doc, sk-git, sk-design and sk-code) instead of general code correctness. This release ships the mode's skeleton only -- the contract, the routing wiring and the packet directories -- so the scoping engine, the per-authority adapters and the loop runtime can be built against a fixed shape with no open design questions left over.

> Spec folder: `.opencode/specs/system-deep-loop/059-deep-alignment-mode/003-scaffold-mode-packet` (Level 1)

---

## What Changed

#### Mode Contract

- **New `deep-alignment` mode-packet** -- `system-deep-loop` previously had no first-class way to check whether an artifact follows a named authority's own standards, only general-purpose review. `SKILL.md` now states the mode's boundary against `deep-review` (general correctness) and `parent-skill-check.cjs` (hub structure), and carries the state machine (`INIT -> SCOPE -> DISCOVER -> ITERATE -> CONVERGE -> REPORT -> optional REMEDIATE`) and the four alignment invariants: verify-first, known-deviation suppression, read-only default and gated remediation.

#### Routing

- **Registry and router wiring** -- `mode-registry.json` gains an `alignment` entry (`packetKind: workflow`, reusing the shared runtime convergence engine) and `hub-router.json` gains a matching `routerSignals.alignment` block plus a `tieBreak` slot, so alignment-intent requests have somewhere to route once the loop engine ships.

## Upgrade

No migration required. The mode is not runnable yet: the scoping engine, the per-authority adapters and the loop scripts ship in later releases, and the `/deep:alignment` command does not exist until then.
