# /create:benchmark Presentation Contract

This file is the single source of truth for user-facing presentation in `/create:benchmark`: startup questions, setup/status dashboards, and result display. The command router owns asset routing only. The workflow YAML owns execution behavior.

## 1. Phase 0 Verification Display

Run this automatic self-check before setup. This is not a user question.

```text
SELF-CHECK: Are you operating as @markdown?

Indicators:
- Invoked with @markdown prefix
- Template-first benchmark report and SOURCE.md generation behavior is available
- sk-doc quality validation behavior is available

If all indicators are present:
- create_benchmark_verified = true
- phase_0_status = PASSED

If any indicator is missing or uncertain:
- Stop before loading workflow YAML
- Display the hard-block message below
```

Hard-block message:

```text
This command requires @markdown for template-first benchmark authoring and sk-doc validation.
Restart with: @markdown /create:benchmark <skill-name> <spec-packet> [create|update]

STATUS=FAIL ERROR="Markdown agent required"
```

## 2. Auto Setup Resolution

For `:auto`, resolve setup using the system auto-mode contract, then load `create_benchmark_auto.yaml` only when every required field is available.

Pre-bound setup schema:

```yaml
PRE-BOUND SETUP ANSWERS:
  skill_name: system-spec-kit
  source_packet_path: .opencode/specs/example-benchmark-packet
  operation: create
  source_strategy: promote-benchmark
  benchmark_date: 2026-05-17
  skill_path: .opencode/skills/
  benchmarks_root: .opencode/skills/system-spec-kit/mcp_server/benchmarks
  benchmark_folder: .opencode/skills/system-spec-kit/mcp_server/benchmarks/benchmark-2026-05-17
  execution_mode: AUTONOMOUS
  memory_choice: skip
```

Default resolution table:

| Field | Required | Resolves Via | Default | Tier-2 Candidate |
| --- | --- | --- | --- | --- |
| `skill_name` | Yes | First positional token or marker | none | No |
| `source_packet_path` | Yes | Second positional token or marker | none | No |
| `operation` | Yes | positional token, marker, or target existence | none | Yes |
| `source_strategy` | Yes | marker or targeted promotion type choice | `promote-benchmark` | Yes |
| `benchmark_date` | Yes | `--date`, marker, or benchmark evidence date | none | Yes |
| `skill_path` | Yes | `--path`, marker, or default | `.opencode/skills/` | No |
| `benchmarks_root` | Yes | marker or derived from skill path/name | derived | No |
| `benchmark_folder` | Yes | marker or derived from root/date | derived | No |
| `execution_mode` | Yes | `:auto`, `:confirm`, or marker | `AUTONOMOUS` under `:auto` | No |
| `memory_choice` | No | marker, prior-work detection, or default | `skip` when no prior continuity exists | No |

## 3. Consolidated Startup Prompt

Ask one prompt containing only applicable questions. Do not split these questions across turns.

```text
Before proceeding, please answer:

Q0. Skill Name (if missing)
Required format: hyphen-case folder name for a skill that has mcp_server/.

Q1. Source Spec Packet (if missing)
Provide the .opencode/specs/... packet path that owns the benchmark decision and evidence.

Q2. Operation (if missing)
A) create
B) update

Q3. Promotion Type
A) promote-benchmark: create a skill-local folder from accepted benchmark evidence
B) rerun-update: update an existing folder after a confirming re-run
C) retirement-update: mark an existing benchmark retired while preserving the folder

Q4. Benchmark Execution Date (if missing)
Required format: YYYY-MM-DD. Use the date the benchmark ran, not today's date.

Q5. Prior Work Context (when the existing spec has prior continuity)
A) Load handover plus implementation-summary continuity
B) Load canonical spec docs up to a recent cap
C) Skip

Q6. Execution Mode (if no :auto/:confirm suffix)
A) Interactive (:confirm)
B) Autonomous (:auto)
```

Hard stops:

- Do not infer missing operation from context.
- Do not split setup across multiple prompts.
- Do not continue with missing required fields.
