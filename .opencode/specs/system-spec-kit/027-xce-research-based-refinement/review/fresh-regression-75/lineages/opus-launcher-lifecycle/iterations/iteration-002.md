## Analysis: launcher stale-reclaim vs reap+respawn (true adoption)

**Scope traced:** `mk-spec-memory-launcher.cjs`, `mk-skill-advisor-launcher.cjs`, `mk-code-index-launcher.cjs`, and `lib/launcher-ipc-bridge.cjs` — specifically the owner-lease acquisition, daemon-lease classification, adopt-vs-reap decision, and spawn serialization.

### spec-memory — robustly guarded (no double-spawn)
The `staleReclaimable` branch (`mk-spec-memory-launcher.cjs:1570-1652`) is correct: it reads the recorded `childPid`, checks liveness, and for a live orphan it runs a **deep JSON-RPC probe** (`probeLeaseHolderWithRetries`, not just `bridgeReadiness` socket-existence) before adopting — explicitly defending against a *wedged* daemon (1584-1591). Only a dead/wedged/unbridgeable daemon falls through to reap+respawn, which is serialized by the exclusive respawn lock (1618), re-validates `childPid` under the lock (1631-1639), and refuses to spawn unless the reap *confirms* the child is dead (`reapLeaseChildBeforeRespawn`, 709-752). Two launchers racing a crashed owner are gated by the owner-lease `O_EXCL`/last-writer-reread (452-489) and again by the respawn lock. The bridge decision in `maybeBridgeLeaseHolder` requires N consecutive deep-probe failures before declaring dead (`launcher-ipc-bridge.cjs:411-424`). Sound.

### skill-advisor — guarded via a different but sound model
`checkStrictSingleWriter` (`mk-skill-advisor-launcher.cjs:731-760`) bridges a live, socket-usable orphan through `maybeBridgeLeaseHolder` (which deep-probes and returns `respawn` on a dead/wedged daemon), and otherwise defers the orphan reap to `scheduleBootstrapReap` → executed *inside* the bootstrap-lock critical section (1241-1251). Its `acquireBootstrapLock` never returns `false` (only `return true`/throw, 1114-1142), so reap+spawn always share that one serialization point. Sound.

### code-index — **gap: alive-but-stale orphan can double-spawn**
Code-index does **not** track a daemon `childPid` in its lease (`writeLeaseFile` writes only `pid: process.pid` = the launcher, `mk-code-index-launcher.cjs:705-710`), and after spawn it refreshes the *owner* lease's `ownerPid` to the child (882-890). Its `staleReclaimable` branch is a **bare log with no reap/adopt** (979-981), falling straight through to `acquireBootstrapLock` → `writeLeaseFile` → `launchServer()` (1054-1061).

Reachable scenario: launcher L spawns non-detached daemon C; L is SIGKILLed → C is reparented to init and keeps running, still holding `code-graph.sqlite`. New launcher N:
- Owner lease `ownerPid=C` (alive, `ppid==1`) → `classifyOwnerLease` returns `ppid-1-orphan` (427-435) → **reclaimable**, so `acquireOwnerLeaseFile` overwrites it **without reaping C** (450-486).
- Daemon lease `pid=L` (dead) → `staleReclaimable:true` → log-only (979-981).
- N spawns a **second** daemon C2 against the same DB. The orphan C is never found or reaped — and cannot be, since `childPid` is never recorded.

Both sibling launchers explicitly prevent exactly this; code-index has no equivalent guard and no comment justifying the no-op (every other tricky decision in these files is heavily commented). The ultimate backstop is whatever DB-lock the code-index daemon itself holds (outside this slice), so I rate this P1, not P0 — but it is a real, asymmetric correctness gap in the alive-but-stale branch the seat was asked to verify.
