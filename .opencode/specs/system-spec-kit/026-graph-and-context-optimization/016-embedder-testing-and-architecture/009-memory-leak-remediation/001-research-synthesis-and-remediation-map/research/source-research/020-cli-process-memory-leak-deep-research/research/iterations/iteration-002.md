# Iteration 002: State-machine, advisory-lock, reducer, and post-dispatch failure-recovery audit

## Focus
Trace the `/spec_kit:deep-research` state machine end-to-end to find paths where a failed, partial, or mis-typed dispatch can leave research state inconsistent or launch the next iteration with wrong values. In-scope surfaces: `step_acquire_lock` and its `on_*` release hooks; `pre_dispatch_audit → writeFirstRecordExecutor` interaction with `previousStateLogSize`; the heredoc executor-shape contract; `post_dispatch_validate` recovery branches; `reduce-state.cjs` failure modes on missing/corrupt inputs; `step_loop_decision → step_read_state` re-entry; `step_marker_scan` against the actual rendered prompt template; cleanup ownership for `{state_paths.prompt_dir}` scratch files. Out-of-scope (iteration 001 already covered): the spawn entrypoint inventory itself.

## Findings

### A. Heredoc executor-shape contract violates the audited-wrapper schema

1. The three audited heredoc branches (`if_cli_codex`, `if_cli_gemini`, `if_cli_claude_code`) construct a local `executor` object literal using the field name `type:` ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:592-599], [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:638-645], [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:683-690]) and pass that object to `runAuditedExecutorCommand({ executor, ... })`. The function's TypeScript signature requires `executor: ExecutorConfig`, which is `z.infer<typeof executorConfigSchema>` and uses the field name `kind:` ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/executor-config.ts:21-28]). The heredoc lives inside a YAML string so TypeScript never type-checks it. At runtime, `executor.kind` is `undefined` and `executor.type` is the populated value.
2. The downstream consequence: `buildExecutorAuditRecord(executor)` reads `kind: executor.kind` and returns `{ kind: undefined, model, reasoningEffort, serviceTier }` ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/executor-audit.ts:10-17]). `JSON.stringify` drops `undefined` so the persisted dispatch-failure event carries `executor: { model, reasoningEffort, serviceTier }` with NO `kind`. `post_dispatch_validate`'s `executor_missing` check is `!isObjectRecord(parsedRecord.executor)` ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:405-411]) — it asserts the object exists but never asserts `kind` is populated. So failures on cli-codex / cli-gemini / cli-claude-code dispatches pass the executor_missing check while silently losing their executor-kind provenance, and any reducer or dashboard that filters JSONL events by `executor.kind === 'cli-codex'` will undercount those failures.
3. `executor.kind === 'native'` early-exits at three sites (`writeFirstRecordExecutor` line 98, `emitDispatchFailure` line 146, `appendExecutorAuditToLastRecord` line 230) ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/executor-audit.ts:97-152]). When called from the heredoc with `executor.kind === undefined`, none of these short-circuit. That is the correct outcome (non-native dispatches MUST audit), but the path documents that the runtime accidentally treats `undefined` as non-native, which is fragile if `executor.kind` is ever consulted for routing rather than provenance.
4. The YAML `record_executor_audit` step uses the same `type:` field name as the heredoc ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:728-732]). So the YAML-runtime-side substitution and the heredoc-side construction are internally consistent — they both diverge from the `kind:` schema. The contract drift is between the executor-config TS module and every YAML/heredoc consumer in the file.

### B. `step_marker_scan` self-triggers on the canonical prompt template

5. The `render_prompt_pack` step renders `.opencode/skills/deep-research/assets/prompt_pack_iteration.md.tmpl` into `{state_paths.prompt_dir}/iteration-{current_iteration}.md` via `renderPromptPack` ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:552-569], [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/prompt-pack.ts:36-54]). The template's literal first line is `DEEP-RESEARCH` — no token substitution applies to it ([SOURCE: .opencode/skills/deep-research/assets/prompt_pack_iteration.md.tmpl:1]).
6. `step_marker_scan` reads `{state_paths.prompt_dir}/iteration-{previous_iteration}.md` on iteration ≥2 and aborts with exit code 2 when `head -1` matches the anchored regex `^(DEEP-REVIEW|DEEP-RESEARCH|CODE-REVIEW)$` ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:523-538]). The template-rendered first line `DEEP-RESEARCH` matches this regex exactly. Under the official YAML asset, every multi-iteration deep-research run that uses `render_prompt_pack` would HALT at the start of iteration 2.
7. Empirical check: the live packet (`020-cli-process-memory-leak-deep-research/research/prompts/iteration-001.md`) begins with `# Iteration 001 Prompt`, NOT `DEEP-RESEARCH` ([SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/009-memory-leak-remediation/001-research-synthesis-and-remediation-map/research/source-research/020-cli-process-memory-leak-deep-research/research/prompts/iteration-001.md:1]). So this packet is being driven by an operator-authored orchestrator that bypasses `render_prompt_pack` and writes a sentinel-free prompt header. The latent bug is only suppressed because the operator orchestrator dodges the canonical template; any future caller that re-enables `render_prompt_pack` re-introduces the false-positive HALT. [INFERENCE: the `step_marker_scan` notes describe the regex as "False-positive prevention" predicated on the template NOT starting with the marker, contradicting the template that ships in `assets/`.]

### C. `pre_dispatch_audit` invalidates the `previousStateLogSize` invariant

8. `validateIterationOutputs` performs its FIRST check as `statSync(input.stateLogPath).size <= input.previousStateLogSize → jsonl_not_appended` ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:308-314]). The intent is "did the dispatched executor write anything?".
9. But `pre_dispatch_audit` calls `writeFirstRecordExecutor` BEFORE the executor runs ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:570-576]). When no prior `iteration` / `iteration_start` record exists for the current iteration number, `writeFirstRecordExecutor` APPENDS a fresh `{"type":"iteration_start","iteration":N,"executor":{...},"timestamp":...}` row ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/executor-audit.ts:97-129]). That append grows the state log BEFORE dispatch.
10. Result: even if the executor produces zero output and never appends a single line, `statSync().size > previousStateLogSize` because the `iteration_start` row alone advanced it. The validator then reaches the last-line parse and the parsed record has `type === 'iteration_start'`, which fails the canonical-type guard (`jsonl_wrong_type` with detail `last record uses type='iteration_start' (expected 'iteration')`) ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:348-356]). Net effect: instead of the more precise `jsonl_not_appended` (executor wrote nothing), the operator sees `jsonl_wrong_type` — misleading triage signal.
11. The validator's `findLatestIterationEvent` and `findLatestIterationRecordIndex` both accept `iteration_start` as a "latest iteration record" ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/executor-audit.ts:57-95]), so a retry call to `writeFirstRecordExecutor(N)` will UPDATE the existing `iteration_start` (line 105-121) instead of stacking duplicates. So consecutive retries do not balloon the log. But the validator's `previousStateLogSize` snapshot, if taken AFTER `pre_dispatch_audit`, would need to encompass that append. The YAML does not specify when the snapshot is captured; if the YAML runtime takes the snapshot before `pre_dispatch_audit`, the off-by-one survives.

### D. Reducer fails hard on missing iteration files

12. `reduceResearchState` walks `iterationDir` and calls `parseIterationFile(path)` on every match of `^iteration-\d+\.md$` ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:871-876]). `parseIterationFile` calls `readUtf8(iterationPath)` which is `fs.readFileSync(filePath, 'utf8')` ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:20-22], [SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:174-198]). If the file disappears between `readdirSync` and `readFileSync` (TOCTOU during concurrent runs, or operator deleting a stale iteration mid-loop), the reducer throws ENOENT.
13. The CLI entry's catch-all maps any non-`STATE_CORRUPTION` error to `process.exit(3)` ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:998-1005]). The YAML's `step_reduce_state` documents `failure_modes.missing_iteration_file: "no-op + error"` ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:852-855]) — but the implementation is "hard crash + exit 3", not a no-op. The next workflow step (`step_graph_upsert`) never gets a chance to absorb the failure.
14. The reducer-as-source-of-truth divergence: the YAML iteration_count is `Count lines where type === 'iteration'` ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:382]) and the reducer counts `parsedRecords.filter((record) => record.type === 'iteration').length` ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:867], [SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:612]). Both ignore `iteration_start`. So any `iteration_start` row that never got upgraded to a full `iteration` row (e.g., dispatch crashed AFTER pre_dispatch_audit but BEFORE executor wrote its record) is invisible to iteration accounting. The audit trail keeps the iteration_start as forensic evidence, but the loop never knows that an attempt happened — the next iteration reuses the same iteration number and overwrites the iteration_start record (writeFirstRecordExecutor's update path), losing the original attempt's timestamp.

### E. Reducer parses delta-file payloads but cannot verify them

15. `loadDeltaPayloads(deltaDir)` is only invoked when `emitResourceMapOutput === true` ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:112-130], [SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:908-928]) — i.e., only during the synthesis phase, not during per-iteration reductions. So the per-iteration `reduce-state.cjs <spec-folder>` invocation (line 843 of YAML) never touches the `deltas/` directory. The delta files exist only for two consumers: (a) `post_dispatch_validate.deltaFilePath` existence/non-empty/has-`type:iteration` checks, (b) the synthesis-only resource-map extractor. Consequence: a delta file with garbage payload, contradictory `findingsCount`, or wrong `iteration` number passes per-iteration reduction silently. Corruption detection happens once at synthesis time — far from the iteration that produced it.
16. The validator's delta-file check uses `findLastIterationRecord` (filters `type === 'iteration'`) on the per-iter delta file ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:133-148], [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:413-430]). It does NOT cross-check that the delta-file `iteration` field matches the current iteration N, that `findingsCount`/`newInfoRatio` agree with the state-log `iteration` row, or that the delta file's executor matches the audit record's executor. A misaligned executor in the delta survives validation.

### F. Advisory lock has no liveness probe and one-shot release events

17. `step_acquire_lock` opens `{artifact_dir}/.deep-research.lock` with "exclusive non-blocking advisory locking semantics; fail closed on contention" ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:179-195]). The note is explicit: "macOS/BSD locking here is advisory. Stale-lock override is confirm-only or explicit recovery-only." In `auto` mode (this packet's `executionMode`), there is no operator to confirm — so a stale lock blocks every future `auto` run on this packet until a human intervenes.
18. There is NO PID file alongside `.deep-research.lock` and no liveness probe. The `on_halt` / `on_cancel` / `on_workflow_exit` hooks describe ACTIONS ("Remove or close-with-release {state_paths.lock_file}") but the YAML carries no executable handler — implementation is delegated to the YAML runtime. If the runtime itself is SIGKILLed (or the OS reboots) between `step_acquire_lock` and a release hook, the lock survives and no probe can decide whether the prior owner is gone. Compare against `references/launcher-lease.md` (cited in iteration 001 finding 15), which DOES use `process.kill(pid, 0)` to probe ownership before reclaim.
19. The only persisted `lock_released` JSONL event is emitted in `step_release_lock` inside `phase_save` ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:1055-1058]). The three release-hook paths (`on_halt`, `on_cancel`, `on_workflow_exit`) do NOT emit a JSONL event ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:186-195]). So an audit walking the state log cannot distinguish "lock cleanly released after halt" from "lock leaked after a halt path that crashed before releasing".

### G. Stuck-recovery and dispatch-failure cleanup are absent

20. `step_handle_convergence.if_stuck_recovery` writes a `stuckRecovery` event, resets `stuck_count` to 0, and biases `next_focus`, but performs ZERO cleanup of side-effect files ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:505-509]). Files left behind across stuck-recovery: `{state_paths.prompt_dir}/iteration-{N}.md` (rendered prompt), `{state_paths.prompt_dir}/agent-config-iter-{N}.json` (devin), `{state_paths.prompt_dir}/devin-stdout-iter-{N}.log`, `{state_paths.prompt_dir}/devin-stderr-iter-{N}.log` ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:759-816]). These accumulate one-set-per-iteration with no rotation policy.
21. `post_dispatch_validate.on_failure` documents `action: "emit schema_mismatch canonical conflict event"` and `escalation: "After 3 consecutive failures, trigger stuck_recovery event with diagnostic context"` ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:825-827]). Neither the schema_mismatch event nor the diagnostic-context capture is wired in the workflow body — they live only in prose. There is no `step_post_dispatch_failure` to consume the result of the validator. `step_evaluate_results.on_missing_outputs` writes a synthetic `{"type":"iteration","status":"error","findingsCount":0,"newInfoRatio":0.0,...}` row ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:892-897]) — but it triggers on missing outputs, not on `dispatch_failure_logged`. If the validator returns `dispatch_failure_logged`, no workflow step is explicitly named to fall through to the synthetic record. The implicit "continue" path means the iteration may be miscounted: a `dispatch_failure` event exists (advancing the state log) but no `type:iteration` row was written, so the reducer's `iterationsCompleted` does not advance — and the next loop will reuse the same `current_iteration`, re-rendering the same prompt and re-attempting the same executor, possibly forever (the 3-consecutive-failure rule is only an "escalation" note, no enforcing step exists).

### H. Resume-event continuedFromRun arithmetic and lineage drift

22. `on_resume` emits `"continuedFromRun":{iteration_count}` using the iteration_count VALUE BEFORE classification ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:216]). `iteration_count` is the count of `type === 'iteration'` rows, which excludes `iteration_start` and excludes any in-flight iteration that aborted before writing its row. So `continuedFromRun` reflects "last fully-completed iteration", which is a defensible semantic — but the dashboard label `continuedFromRun: ${continuedFromRun}` ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:764]) presents the value without context, and operators may read it as "the iteration we are about to resume", off by +1.
23. `buildLineageState` resolves `sessionId` from the latest `resumed`/`restarted` event, else from config ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:311-353]). On `on_restart`, the YAML sets `parent_session_id: "{prior_session_id}"` and emits the event with that field ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:218-225]). But the `on_resume` event emits `"parentSessionId":"{session_id}"` — i.e., `parentSessionId === sessionId` for `resume` events ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:216]). This is self-referential. The reducer reads `parentSessionId` faithfully ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:330-336]) and dashboards display `Parent Session: ${parentSessionId}` ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:761]). On any resume event, the dashboard will show the session as its own parent — a likely template-substitution typo (`{session_id}` should be `null` or `{parent_session_id}` for a resume's parent).

### I. Synthesis-time idempotency vs `step_stage_artifact_dir`'s git side-effects

24. `step_stage_artifact_dir` runs `git add {state_paths.artifact_dir}` as part of synthesis ([SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:1026-1030]). This mutates the operator's git index without a commit. If a deep-research run is interrupted between synthesis and save (e.g., process killed during `step_index_memory`), the operator inherits a dirty index they did not author. `on_fail` is documented as "Log warning. Do not block synthesis" but staging itself is not idempotent across re-runs: each `git add` may re-stage files that the operator unstaged manually during recovery.

### J. Validator's verification-pass writes into the live state log without locking

25. `runOptionalVerificationPass` is gated by `recipeConfig.verification_enabled` ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:256-305]). When degraded, the validator `appendFileSync` a `verification_degraded` event directly to the state log ([SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:432-443]). The post-dispatch validator runs synchronously inside the workflow step; the state-log advisory lock semantics described in `step_acquire_lock` cover the LOCK FILE only, not the state log file itself. No concurrency protection on `appendFileSync` is documented — concurrent writers from `writeFirstRecordExecutor`, `emitDispatchFailure`, the validator's verification event, and the YAML's `append_to_jsonl` substitutions can interleave at the byte level under macOS without atomic guarantees, producing malformed JSONL that the reducer's fail-closed corruption check then rejects with exit code 2 ([SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:132-145], [SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:994-1002]).
