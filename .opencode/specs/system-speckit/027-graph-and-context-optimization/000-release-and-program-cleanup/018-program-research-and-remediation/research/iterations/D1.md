### Angle 13: RSS behavior / watchdog threshold of hf-local server

- **Framing:** The opt-in RSS watchdog threshold (`SPECKIT_HF_MODEL_SERVER_MAX_RSS_MB`) for the hf-local model server is well-calibrated, and a memory-sufficiency preflight should exist before spawning.
- **Repo evidence:** `ENV_REFERENCE.md:419-420` — `SPECKIT_HF_MODEL_SERVER_MAX_RSS_MB` (unset → disabled) is the ceiling; `SPECKIT_HF_MODEL_SERVER_RSS_SELF_EXIT` enables graceful self-exit on breach. `model-server-supervision.cjs` implements the watchdog via `createModelServerControl(deps)`. `embedder_architecture.md:134-143` — RSS expectations table: hf-local model state lives in a separate launcher-supervised Node process, spec-memory holds only the HTTP client. `changelog-013-003-front-proxy-in-place-recycle.md` — the front-proxy makes RSS-recycle transparent to clients. `launcher-watchdog.vitest.ts:35,116,133` — tests confirm rollup of daemon child+grandchild RSS, grace clamping, and default-off behavior. `changelog-007-006-graceful-exit-watchdog.md` — the launcher RSS-ceiling watchdog and graceful-exit supervision shipped.
- **Findings:** The watchdog is **opt-in and uncalibrated**. No default value is set (`SPECKIT_HF_MODEL_SERVER_MAX_RSS_MB` is unset by default), meaning the watchdog never fires in a vanilla install. The daemon's own RSS-recycle target is ~1 GB (documented in front-proxy spec), but the model-server watchdog operates on a separate process tree. There is no memory-sufficiency preflight — the server spawns regardless of available RAM, and if the model loads successfully, RSS grows organically until either the watchdog fires (if configured) or the OS OOM-kills it. The idle eviction monitor (`SPECKIT_HF_MODEL_SERVER_IDLE_TIMEOUT_MIN`, default 0/off) is a separate mechanism that reaps idle servers but doesn't prevent spawn on low-memory hosts. The 250-600 MB model size range (depending on dtype) plus Node runtime overhead means a 2 GB host could be fine or tight depending on concurrent processes.
- **Next experiment:** Measure peak RSS of the hf-model-server process tree under sustained embed load (100 sequential requests, production-length docs) on Apple Silicon. Report the number. Then set a sane default for `SPECKIT_HF_MODEL_SERVER_MAX_RSS_MB` (likely 800-1000 MB for q8, 1200-1500 for fp32). Add a preflight check in `handleModelServerDemand` that reads available system memory and logs a warning if free RAM < 1.5× expected model RSS.
- **Verdict:** The watchdog mechanism is sound (tested, transparent via front-proxy) but **defaults to disabled**. Without a calibrated default, operators must discover the right ceiling empirically. The lack of a memory-sufficiency preflight means low-memory hosts will OOM silently. **Recommend:** measure peak RSS, set a conservative default ceiling, and add a preflight warning.

### Angle 15: Cold-start latency of the Python-free hf-local server

- **Framing:** The cold-start latency of the hf-local model server (~15-30s, ~274MB) can be reduced to <5s via model caching or a warm pool, making it viable as a default without user-facing delay.
- **Repo evidence:** `INSTALL_GUIDE.md:108` — "The server is lazily spawned on first embed; the first run downloads the model to `~/.cache/huggingface/hub` (roughly 250-600 MB depending on dtype/artifacts; expect a 15-120 s cold start)." `ENV_REFERENCE.md:443` — "Expect roughly hundreds of MB of cache growth (about 250-600 MB depending on dtype/artifacts) and a 15-120 s first request on a typical connection." `ENV_REFERENCE.md:421` — `SPECKIT_HF_MODEL_SERVER_LOADING_MAX_MS` defaults to 150s. `changelog-018-front-proxy-recycle-hardening.md:36` — "Proxy cold-start attempts default from 120 to 30, reducing the silent wait from about 176 seconds to about 41 seconds." `launcher-session-proxy.cjs` — `DEFAULT_MAX_COLD_START_ATTEMPTS` is 30 with env override. `hf-model-server.cjs` — binds listener before model load; `/api/health` returns `loading` during cold start.
- **Findings:** Cold-start latency has two distinct phases: (1) **model download** (~15-120s, first-time only, depends on network and model size), and (2) **model load into memory** (~seconds, every spawn). The download happens once and is cached at `~/.cache/huggingface/hub`; subsequent cold starts only pay the load cost. The proxy was hardened from ~176s silent wait to ~41s (30 attempts). The idle eviction monitor (default off) keeps the server warm when enabled. There is **no warm pool or model pre-caching mechanism** — each launcher spawn starts a fresh Node process that loads the model from the HF cache directory. The `SPECKIT_HF_READY_LATCH_TTL_MS` (default 30s) skips health probes for 30s after a successful embed, reducing probe overhead but not cold-start time. No model pre-warming, no persistent model server across daemon restarts, no shared model memory between launcher instances.
- **Next experiment:** Measure the model-load-only cold start (cached model, fresh server process) on Apple Silicon M-series. If >5s, investigate: (a) ONNX model warm-up via a `/api/warmup` endpoint that pre-runs one inference, (b) keeping the model server alive across daemon recycles (currently the front-proxy recycles the daemon but kills the model server child first), (c) a persistent model server systemd/launchd service independent of the MCP launcher.
- **Verdict:** The 15-120s range is dominated by first-time download; cached cold start is likely 3-10s (unmeasured in 026). The proxy hardening (176s → 41s) helps but is a timeout bound, not a latency reduction. **The real lever is keeping the model server alive across daemon recycles** — the front-proxy already handles daemon transparency but explicitly kills the model server child before respawn. Making the model server a peer process (not a child of the launcher) would eliminate cold-start entirely for the steady state.
