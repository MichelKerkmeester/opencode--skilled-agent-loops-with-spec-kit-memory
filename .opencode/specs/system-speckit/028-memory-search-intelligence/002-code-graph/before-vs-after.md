# What Changed in the Code Graph: Determinism, Freshness, Parser Resilience and Structural Ranking

> This is the subsystem-scoped companion to the packet-root `before-vs-after.md`. It covers only the Code Graph, the structural index behind packet 028's `mk-code-index` server that answers impact, dependency, neighborhood and outline questions over the parsed symbol graph. The Code Graph entered 028 fast but brittle. Its impact ranker trusted raw SQLite row order and ignored the confidence and evidence-class metadata already carried on every edge. Its freshness signal was a wall-clock staleness bucket with no monotonic generation. A refactored dependency silently orphaned its importers' edges. A single transient parser crash quarantined a file forever. A markdown or config document was nothing but a write-only content hash. Across a Wave-0 spearhead and eight phase children it became a reproducible, trust-aware and self-healing index that answers questions over documents as well as code. The kept work shipped additive and default-safe against confirmed seams. The schema-migration and net-new-ranking experiments shipped behind default-off flags, were measured against the real graph and were then removed by the flag-resolution reckoning when the evidence did not support them. Nothing here contradicts the root doc. It tells the same story in more depth.

---

## 1. DETERMINISM AND RANK-TIME TRUST

The trust blend that shipped in Wave-0 and the walk-order determinism phase (`001-determinism-walk-order`) that followed it. One seam, the `rankContextEdges` / `finalize()` path in `code-graph-context.ts`, no schema migration.

**Before**

The impact context ranked callers and callees by raw database row order. The edges already carried `confidence` and `evidenceClass` metadata, but the ranker ignored both, so a high-confidence structurally-extracted edge sorted no higher than an ambiguous inferred one. Worse, the equal-score tiebreak was the DB-iteration index. The Code Graph reindex is a destructive DELETE-plus-INSERT swap with no temporal columns, so the physical row order of edges can shift between scans, and two edges with the same score could re-order whenever a rebuild moved their rows. Impact, dependency and outline output was therefore not reproducible across scan rebuilds. The dual reverse-edge channels of the impact walk, CALLS and IMPORTS, were concatenated in DB order rather than fused by rank, so the cross-channel ordering the shared `fuseResultsMulti` primitive provides was unavailable here.

**After**

The Wave-0 trust blend (commit `e21caf5de6`) folds confidence and evidence class into ranking as an RRF-additive term, `rankScore = 1/(60+index+1) + clamp(confidence)*evidenceClassFactor`, with the structural edge weight never mutated. A neutral edge that carries no trust metadata gets a reliability of zero and stays byte-identical to the rowid baseline, proven byte-for-byte against the pre-change order. The walk-order phase then adds `det-context-order-global`. `rankContextEdges` now derives a content-derived stable key from the related node content hash coalescing to the related symbol id, then the file or fully-qualified name, the edge type and the source and target ids, and assigns both the baseline rank and every equal-score tie from that key. The comparator is a hand-written total order over string keys rather than JS subtraction, which a NaN or a negative zero would poison. The one seam covers impact, dependency or callees and outline-export together. There is no schema migration. Two follow-ups stayed gated. The Q4-C1 boost magnitudes (EXTRACTED and STRUCTURED at 0.01, INFERRED at 0.004, AMBIGUOUS at 0.002) are unbenchmarked defaults that wait on a code-graph retrieval benchmark that does not exist campaign-wide. The dual-channel fuser adapter that would promote `fuseResultsMulti` for CALLS and IMPORTS waits on an isolation-compatible way to consume the shared fuser without forking it or adding a forbidden code-graph source dependency.

**Impact**

The impact ranker can now lift trusted edges without disturbing peers that carry no trust signal, and the byte-identical neutral fallback guarantees it adds order only where trust metadata exists. The walk output is reproducible across a scan rebuild that shifts DB row order, so a re-index of unchanged content returns the same ranking. The cross-channel fuse and the magnitude tuning exist as named, gated follow-ups rather than silent gaps.

**Why**

Both shipped pieces are ordering-only and additive. The trust blend never touches structural weight and the det-order seam only changes tie resolution, so each ships with its default behavior unchanged and proven against the rowid baseline. The fuser adapter and the magnitude tuning are ranking-sensitive, so they stay pending behind a named gate, the isolation-compatible consume path for the fuser and a real retrieval benchmark for the tuning, rather than shipping on a structural guess.

## 2. FRESHNESS AND EDGE-LIFECYCLE CORRECTNESS

The generation watermark (`003-generation-watermark`) and the one real edge-staleness bug (`002-edge-staleness-correctness`). Both are correctness work against confirmed seams in the scan loop.

**Before**

Freshness carried no monotonic generation counter. `computeFreshness()` reported only a wall-clock staleness bucket of fresh, recent, stale or unknown derived from the maximum `indexed_at`, and the read-path gate was a binary block on anything other than fresh. There was no comparable number to express results as of scan generation N. Separately, the scan silently lost cross-file edges when a file's dependency was refactored. Symbol ids are path-coupled as `sha256(filePath + '::' + fqName + '::' + kind)`, so a rename, kind-flip or move of an exported symbol mints a new id, while a dependent that imports it is content-hash-stable and gets skipped by the incremental scan. The dependent's edge to the old id is never re-derived, the old target node is dropped on reindex and `pruneDanglingEdges` physically deletes the now-dangling edge. The relationship vanished silently until the dependent itself changed. The lever to fix it already existed, `queryFileImportDependents`, but it was wired only to the read path and the scan loop never consulted it.

**After**

The soft generation watermark (Q6-C2) stores a monotonic counter in the existing `code_graph_metadata` KV table through a typed `getCodeGraphGeneration()` / `bumpCodeGraphGeneration()` pair, bumps it once from the `scanPromotable` finalize block in `handlers/scan.ts` so it fires on both a full scan and a selective reindex, and surfaces it as an additive `generation` field on the freshness envelope. This corrected the refuted `ensure-ready.ts:497` bump site, which only fired on an out-of-scope HEAD branch and never advanced on a real reindex. There is no read-filter change and the node and edge result set stays byte-identical. The edge-staleness phase makes the incremental scan dependency-transitive. Before the per-file skip loop it snapshots the stale set and expands it with reverse-dependents, force-including the importer files through a `forceParse` set that overrides the skip-fresh short-circuit, so a dependency change drags its importers back into the parse batch and their edges re-derive against the new symbol ids. The reverse-dependents are captured before any `replaceNodes` on the changed file, because the join onto live target nodes returns nothing once the edge already dangles, and a path-filtered `queryImportersOf(stalePaths)` replaces the full-table scan for the scan-loop path. The phase also emits an additive `SUPERSEDES` rename-lineage edge keyed on matching content hash, reusing the off-by-default tombstone machinery as its substrate with `SCHEMA_VERSION` unchanged at 5 and the absent-edge read path byte-identical.

**Impact**

Freshness now carries a comparable generation number that advances on every promoted scan. A kind-flip rename no longer strands a stale import edge pointing at a dead symbol id, the relationship re-derives in the same scan, and rename lineage is preserved rather than discarded as a delete-then-create. A body-only edit, which keeps the symbol id stable, triggers no extra dependent re-parse, so the common case is untouched.

**Why**

The soft watermark is additive telemetry with no read-filter change, so it ships always-on and byte-identical. The reverse-dependency repair is a correctness fix, but its fan-in cost is real, so it ships behind `SPECKIT_CODE_GRAPH_REVERSE_DEP_FORCE_PARSE` with default behavior unchanged until a fan-in re-parse benchmark on a hot high-importer file clears the cost-bearing default-on flip. The `SUPERSEDES` edge rides the existing tombstone substrate, gated by `SPECKIT_CODE_GRAPH_TOMBSTONES`, and is strictly additive, so its absence leaves every existing query byte-identical. The hard as-of-generation error gate (Q6-C1) that would build on the watermark stayed documented-only and deferred-speculative, because it needs the bitemporal schema cluster and a named consumer that do not exist.

## 3. PARSER RESILIENCE AND THE DOC-SYMBOL LANE

The transient/fatal parser retry axis (`007-parser-resilience`) and the document-symbol lane (`008-doc-symbol-lane`). Two capability adds that widen what the index survives and what it can answer.

**Before**

A single crashing file was permanently removed from the graph with no bounded-retry escape. The parser skip-list classified files only by tree-sitter WASM crash cohort, B1, B2 or OTHER, which is a crash taxonomy and not a retry policy. Adding a file only bumped its attempt count, there was no `max_retries` ceiling, and removal was an intentional permanent no-op whose `recordSuccess` was a deprecated empty body documented to never auto-unskip. So a file that crashed once, even for a transient out-of-memory or timeout reason, was skipped forever short of manual review. Separately, the document lane was a write-only content hash that answered no questions. When the language was `doc` the indexer short-circuited to an empty result, so markdown headings and config keys were never queryable as graph nodes. The launcher classified lease-lifecycle events for its own control flow but had no metrics sink, so that classification was computed and dropped.

**After**

The parser phase (Q2-C1) splits the skip-list into an orthogonal transient and fatal retry axis. A TRANSIENT file, which covers a WASM memory trap, an out-of-memory, a timeout or a deadline-abort, stays eligible and is re-attempted on later scans until its durable attempt count reaches a default of five, at which point it is promoted to FATAL. A FATAL file, which covers genuinely unparseable content or an exhausted transient, is skipped fail-closed from the first failure, and an unknown or ambiguous error defaults to FATAL so a misclassification can never loop. The retry budget is read from the durable `attempt_count` column rather than an in-memory counter, so a mid-scan crash does not hand a file a fresh budget, and the deliberate no-op `recordSuccess` is lifted to clear a TRANSIENT entry on a clean re-parse so the file self-heals back into the graph. Poison-pill isolation is preserved, the scan proceeds past the failing file in either class, and the new `retry_class` storage is additive with legacy rows defaulting to fatal. The ceiling is configurable through `SPECKIT_PARSER_SKIP_LIST_MAX_RETRIES`. The doc-symbol lane (Q5-C1) turns the empty `doc` early-return into real structure. Markdown ATX and Setext headings become `heading` nodes nested by level through CONTAINS edges, and the keys of json, jsonc, yaml, yml and toml documents become `key` nodes through a shallow key walk. Every id is content-derived and idempotent across rescans, with no LLM and no network, and the `SymbolKind` union gains `heading` and `key` with the context renderer made tolerant of those non-code kinds. Markdown inclusion stays an explicit opt-in, the lane runs json, yaml and toml first because those already index as doc rows, so the heading extractor is not a silent no-op. The same phase adds a launcher lease-churn classifier that names the held-by-other, bridged-secondary, stale-reclaimed and respawned transitions and routes them through a sink-agnostic `emitLeaseMetric` stub that defaults to a no-op.

**Impact**

A transiently failing file now gets bounded retries and a clean self-heal instead of a permanent skip, while a genuine poison file is still quarantined after at most five attempts, and in neither case can one file wedge the scan for the others. Markdown and config documents become queryable graph structure instead of opaque hashes, and the lease classifier is ready to surface churn the moment a metrics sink exists.

**Why**

The parser retry axis is a correctness improvement that is strictly no worse than the prior permanent-skip behavior, it reuses the existing durable budget column and fails closed on ambiguity, so it ships always-on with the cohort labels left intact and orthogonal. The doc lane is additive and sits entirely off the destructive-reindex transaction boundary, behind the existing `doc` guard, so the code parser lanes and the edge-write path are untouched. The lease telemetry ships as a no-op-default stub because there is no metrics sink today, so the classifier is the durable deliverable and the emit changes no behavior while the sink is absent.

## 4. THE GATED FRONTIER AND WHAT THE RECKONING REMOVED

The schema-migration and net-new-ranking experiments: the seeded-PPR impact ranking (`005-seeded-ppr-ranking`), the code-edge bitemporal cluster (`004-code-edge-bitemporal`) and the closed-vocabulary edge governance (`006-edge-governance-vocab`). All three were built behind default-off flags, measured against the real graph and then removed by the flag-resolution reckoning.

**Before**

The impact walk ranked callers through flat reverse CALLS and IMPORTS enumeration in DB order, with no centrality and no mass propagation, and its 400ms latency budget could truncate that flat list in arbitrary order so the most-central blast-radius callers were not guaranteed to surface before the cut. The Code Graph edge table was also ungoverned and atemporal. `code_edges` carried no validity window, so currentness equalled physical edge presence and there was no way to ask what the call graph looked like at a prior generation. The `edge_type` column was TEXT with no CHECK, so only writer discipline kept it to the known vocabulary, and the tombstone prune tiebreak ordered on the rowid arrival order, which is non-deterministic across rescans of the same content.

**After**

Seeded PPR (Q3-C1) was built as a bounded personalized-PageRank primitive behind `SPECKIT_CODE_GRAPH_SEEDED_PPR_RANKING`. It seeds mass on the subject symbols and spreads it over the Memory weighted-walk substrate through a code-graph edge adapter rather than forking a second walk engine, gates expansion to the impact and multi-hop classes through a new SingleHop, MultiHop and Entity taxonomy so single-hop precision is never degraded, projects the working graph undirected so seed mass reaches callers instead of sinking and folds confidence and evidence reliability into the transition weights so an inferred far-hop path ranks below an observed near-hop one. A bounded power-method cap keeps it inside the 400ms budget and a budget cut returns the best-ranked prefix rather than an arbitrary truncation. The lexical-vector seed union was recorded as a NO-GO and cut, because the module deliberately has no vector backend. The code-edge bitemporal cluster (Q1-C1) shipped a schema foundation that bumped `SCHEMA_VERSION` from 6 to 7, added nullable `valid_at` and `invalid_at` generation columns to `code_edges` with idempotent fail-closed UP, DOWN and BACKFILL helpers and a fresh-init path, and kept temporal reads off behind `SPECKIT_CODE_GRAPH_EDGE_BITEMPORAL_READS`, while the live-view chokepoint, the edge-lifecycle layer and the symbol-timeline read stayed gated on a named as-of consumer that 200 iterations never found. The closed-vocabulary governance (CG-closed-vocab-check) shipped the first table-rebuild migration behind `SPECKIT_CODE_GRAPH_EDGE_GOVERNANCE_VOCAB`, adding a CHECK over the ten live relations plus SUPERSEDES with a mandatory pre-rebuild DISTINCT scan, an all-or-nothing abort on any out-of-vocab value, a rollback helper and a `SCHEMA_VERSION` bump from 7 to 8, with the churn cap, the audit-subgraph and the derived-clock tiebreak left as deferred siblings. The flag-resolution reckoning then ran each of these against a fair real-world load. Seeded PPR went negative on the real CALLS graph and was removed with its code. The code-edge bitemporal cluster and the closed-vocabulary governance never found a worthwhile consumer and were caller-less or integrity-only on the live graph, so both were removed with their code too.

**Impact**

The impact path carried a real personalized-PageRank primitive and the edge table carried an explicit validity window and a write-time vocabulary CHECK, each as a tested, reversible, default-off seam, until the reckoning measured them and decided none earned its keep. What the subsystem retains from these phases is the negative result recorded as evidence rather than dormant code, and the determinism win from PPR's score-first ordering is independently preserved by the det-order tiebreak in the determinism phase.

**Why**

Every one of these was ranking-sensitive or schema-bearing, so each shipped default-off and byte-identical with its flag unset and named the gate it waited on, a code-graph retrieval benchmark for PPR and a named as-of or governance consumer for the migrations. The reckoning held them to the same eval-versus-prod discipline the whole packet uses, where a feature earns a keep only by improving the served result the reader sees and the prod retrieval route guarantees a never-cut-below-three minimum rather than a cap at three, trimming the longer tail only at a confidence cliff and under a running token budget. PPR went negative on the real CALLS graph, the bitemporal cluster had no as-of consumer and the closed-vocab CHECK was integrity-only with no recall effect, so all three were deleted rather than left dormant. The as-of-generation hard gate that the bitemporal cluster depended on was never built at all, only documented and deferred.

## CURRENT STATE

The Code Graph shipped its rank-time trust blend in Wave-0 at commit `e21caf5de6` and then built eight phase children under `028-memory-search-intelligence/002-code-graph`. Five carried into the shipped state as always-on or default-off-by-default correctness and capability work: the walk-order determinism tiebreak, the soft generation watermark, the reverse-dependency edge-staleness repair with its additive `SUPERSEDES` lineage edge, the transient and fatal parser retry axis and the doc-symbol lane with its lease-churn classifier. Three were built behind default-off flags, measured against the real graph and then removed by the flag-resolution reckoning: seeded PPR for going negative on the real CALLS graph, and the code-edge bitemporal cluster and the closed-vocabulary edge governance for being caller-less or integrity-only on the live graph. Every shipped member is additive against a confirmed seam, the always-on members are byte-identical where they touch ranking or freshness and the cost-bearing or ranking-adjacent members carry an explicit gate and unblock condition rather than a forced flip. The per-flag deciding evidence for the removed three lives with the packet-root reckoning record. For the full cross-subsystem story see the packet-root `before-vs-after.md`, Section 3.
