# AI Council Report — 013 follow-ups + MCP stability

## Summary

The through-line is that the new self-maintaining index still treats “same file, same content hash” as “nothing to do” in places where reprocessing is semantically required: provider migration, mtime backfill, force reindex, and spec-doc coverage all need a stronger logical identity and refresh contract. The single most important fix is to make reindex/force paths update the existing logical row or explicitly supersede it, while always backfilling `file_mtime_ms`; without that, the daemon keeps drifting and MCP recycle work only makes the drift harder to observe.

## Findings

### 1. Indexer Refresh Gap — Priority P0

- Root cause: re-embed does not rewrite `file_mtime_ms`; it vector-reindexes rows selected from `memory_index` by `id/content_text/title/file_path` and then only flips `embedding_status`, `embedding_generated_at`, `updated_at`, and `failure_reason` (`.opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:246`, `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:497`, `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:549`). Normal memory creation does backfill mtime (`.opencode/skills/system-spec-kit/mcp_server/handlers/save/create-record.ts:344`), but append-only lineage insertion omits `file_mtime_ms` from its post-insert metadata (`.opencode/skills/system-spec-kit/mcp_server/lib/storage/lineage-state.ts:495`). `force:true` reaches `indexSingleFile` (`.opencode/skills/system-spec-kit/mcp_server/handlers/memory-index.ts:615`), but the save-layer same-path predicate still returns `unchanged` when `content_hash`, metadata, and an eligible embedding status match (`.opencode/skills/system-spec-kit/mcp_server/handlers/save/dedup.ts:265`). `handover.md` is in the spec-doc allowlist (`.opencode/skills/system-spec-kit/mcp_server/lib/config/spec-doc-paths.ts:7`) and discovery path (`.opencode/skills/system-spec-kit/mcp_server/handlers/memory-index-discovery.ts:149`), but scoped scans compare the path relative to `.opencode/specs` against the raw `specFolder` argument (`.opencode/skills/system-spec-kit/mcp_server/handlers/memory-index-discovery.ts:97`); the shipped handover itself documents calls using `.opencode/specs/...`, which will not match that root-relative comparison (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-and-causal-runtime/013-memory-index-scan-implementation/handover.md:122`).
- Proposer: Add a reprocess mode for scan/reindex that bypasses `unchanged` when `force:true`, provider/model changed, or `file_mtime_ms IS NULL`; backfill `file_mtime_ms` in embedder reindex completion and append-only insert; normalize `specFolder` to specs-root-relative before discovery.
- Skeptic: Blindly bypassing `unchanged` can recreate the duplicate-row problem if it inserts instead of updating. Backfilling mtimes during vector-only reindex needs a safe path for missing/deleted files.
- Referee (converged): Minimal fix is update-in-place for the active logical row on force/provider refresh, not append. Treat `force:true` as “recompute embedding and metadata for this logical file row”; only append when content changed and lineage semantics require it. Normalize `specFolder` inputs and add a regression proving `handover.md` indexes from `.opencode/specs/...`, `specs/...`, absolute, and root-relative forms.
- Effort: M · Risk: med · Blast radius: `memory-index.ts`, `memory-save.ts`, `save/dedup.ts`, `lineage-state.ts`, `embedders/reindex.ts`, spec-doc discovery.

### 2. Duplicate Rows On Provider Migration — Priority P1

- Root cause: the table constraint is `UNIQUE(spec_folder, file_path, anchor_id)` (`.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-schema.ts:2419`), which does not protect canonical path and allows duplicate `NULL` anchors under SQLite semantics. The active lookup goes through `active_memory_projection` (`.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:1009`), and the projection is only an upserted pointer keyed by `specFolder::canonicalPath::anchor` (`.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:146`). So an old failed row can remain outside the active projection while a new success row is inserted for the provider swap.
- Proposer: Define one durable logical key for memory rows: `spec_folder + canonical_file_path + COALESCE(anchor_id,'_') + scope tuple`. Reindex/provider migration should update or supersede that key in one transaction.
- Skeptic: A new unique index will fail until existing duplicates are cleaned. Scope and parent/chunk rows complicate a naive uniqueness constraint.
- Referee (converged): First ship a cleanup/supersede migration that collapses duplicate logical keys and points `active_memory_projection` at the winner. Then add a generated/stored `logical_key` or expression unique index that coalesces nullable anchor/scope values. Provider changes should update the active row’s embedding metadata and mark stale failed siblings deprecated/superseded, never create a second active logical row.
- Effort: M · Risk: med · Blast radius: schema migration, `vector-index-mutations.ts`, `vector-index-store.ts`, lineage/projection maintenance.

### 3. Checkpoint Fails On Large Index — Priority P1

- Root cause: `checkpoint_create` calls `checkpoints.createCheckpoint` directly (`.opencode/skills/system-spec-kit/mcp_server/handlers/checkpoints.ts:398`). Storage snapshots every manifest table into in-memory `rows` arrays (`.opencode/skills/system-spec-kit/mcp_server/lib/storage/checkpoints.ts:914`), stores all table snapshots in one object (`.opencode/skills/system-spec-kit/mcp_server/lib/storage/checkpoints.ts:1567`), then does one `JSON.stringify(snapshot)` and `gzipSync(Buffer.from(snapshotJson))` (`.opencode/skills/system-spec-kit/mcp_server/lib/storage/checkpoints.ts:1604`). The checkpoint table only has one `memory_snapshot BLOB` slot (`.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-schema.ts:2209`), so the design materializes the full ~33k-row index as one V8 string.
- Proposer: Replace monolithic JSON with chunked NDJSON or table-chunk blobs written page-by-page. Store a manifest in `checkpoints.metadata`, and keep each table chunk bounded.
- Skeptic: Restore currently expects one parsed snapshot envelope; changing the format risks rollback reliability unless legacy snapshots still restore.
- Referee (converged): Add v2 checkpoint format, keep v1 restore. For v2, stream table rows by primary-key windows into compressed chunks, or use SQLite online backup/VACUUM INTO for full-db checkpoints and a manifest for scoped checkpoints. Also expose `includeEmbeddings` in the handler API; storage supports it, but the handler args do not pass it through (`.opencode/skills/system-spec-kit/mcp_server/handlers/checkpoints.ts:40`, `.opencode/skills/system-spec-kit/mcp_server/handlers/checkpoints.ts:398`).
- Effort: L · Risk: med · Blast radius: checkpoint create/restore, schema for checkpoint chunks or external artifact paths.

### 4. Residual Failed Embedding Rows — Priority P2

- Root cause: the retry manager only queues `pending` and `retry`, not `failed` (`.opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:557`). `memory_embedding_reconcile` defaults to dry-run (`.opencode/skills/system-spec-kit/mcp_server/handlers/memory-embedding-reconcile.ts:19`) and apply only mutates vector-present stale rows to success or retry-retention failures back to retry (`.opencode/skills/system-spec-kit/mcp_server/lib/embedders/embedding-reconcile.ts:217`, `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/embedding-reconcile.ts:240`, `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/embedding-reconcile.ts:376`, `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/embedding-reconcile.ts:389`). Provider failures missing vectors are counted separately and left report-only (`.opencode/skills/system-spec-kit/mcp_server/lib/embedders/embedding-reconcile.ts:247`).
- Proposer: Run `memory_embedding_reconcile` dry-run, then apply if the 6 rows fall into vector-present stale or retry-retention buckets.
- Skeptic: If the 6 rows are genuine provider failures with no vector surface, reconcile apply will not fix them by design.
- Referee (converged): Safe remediation is bucket-dependent. Use reconcile apply only for rows it explicitly owns. For genuine failed/no-vector/no-success rows, targeted re-embed/reset is safer: verify the source file still exists, verify no newer success row for the same logical key, verify active shard attachment, then move selected IDs to `retry` or reprocess their files through the corrected force path.
- Effort: S · Risk: low · Blast radius: operational repair script or reconcile handler guard.

### 5. MCP Disconnects Mid-Session — Priority P0

- Root cause: the configured MCP command launches `node .opencode/bin/mk-spec-memory-launcher.cjs` in both OpenCode and Claude config (`opencode.json:18`, `.claude/mcp.json:9`). The launcher spawns `context-server.js` with `stdio: 'inherit'` (`.opencode/bin/mk-spec-memory-launcher.cjs:808`), while the child creates the real `StdioServerTransport` (`.opencode/skills/system-spec-kit/mcp_server/context-server.ts:2067`). Transport-severing paths:
  - RSS self-exit: watchdog breach calls `recycleViaGracefulSelfExit`, kills model/context children, clears lease, and `process.exit(0)`s (`.opencode/bin/mk-spec-memory-launcher.cjs:676`, `.opencode/bin/mk-spec-memory-launcher.cjs:701`, `.opencode/bin/mk-spec-memory-launcher.cjs:709`). Recovery is not transparent; prior art says today’s recovery is manual `/mcp` reconnect and relies on host relaunch (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/007-mcp-daemon-reliability/006-graceful-exit-watchdog/spec.md:84`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/007-mcp-daemon-reliability/006-graceful-exit-watchdog/spec.md:157`).
  - Crash-loop give-up: child exit goes through `superviseChildExit`; after too many deaths it clears lease and exits or mirrors the signal (`.opencode/bin/lib/model-server-supervision.cjs:251`, `.opencode/bin/mk-spec-memory-launcher.cjs:821`). Recovery is not transparent.
  - Child relaunch under inherited stdio: below crash-loop threshold the launcher schedules `launchServer()` again (`.opencode/bin/mk-spec-memory-launcher.cjs:829`), but the new child creates a fresh MCP `Server` that has not performed the client’s original initialize handshake. Prior art explicitly rejected transparent in-place respawn because re-piping stdio cannot restore the per-server MCP initialize session (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/007-mcp-daemon-reliability/006-graceful-exit-watchdog/spec.md:103`).
  - Bridge socket close: secondary launchers pipe stdio to the daemon socket once (`.opencode/bin/lib/launcher-ipc-bridge.cjs:95`); socket error/close calls `closeOnce` and exits, with no reconnect loop (`.opencode/bin/lib/launcher-ipc-bridge.cjs:101`, `.opencode/bin/lib/launcher-ipc-bridge.cjs:106`). Manual reconnect or host relaunch is required.
  - Model-server cascade: ordinary model-server crash-loop give-up logs that the daemon remains running (`.opencode/bin/lib/model-server-supervision.cjs:861`), so that path should not sever MCP. But model-server RSS breach is wired to the launcher self-exit callback (`.opencode/bin/mk-spec-memory-launcher.cjs:484`, `.opencode/bin/mk-spec-memory-launcher.cjs:491`), which does sever MCP. Transparent recovery is UNKNOWN and host-dependent.
- Proposer: Put a stable MCP front proxy in front of the worker. The proxy owns the client stdio session and IPC socket, survives worker restarts, and talks to the worker over an internal RPC channel.
- Skeptic: A true proxy must handle MCP initialize, capabilities, in-flight calls, cancellation, notifications, and tool-list changes. It can become the new hard-to-debug daemon.
- Referee (converged): Do not relaunch a stdio MCP child behind a live initialized client. Short-term: on worker death, close the client transport intentionally and emit a clear reconnect-required error. Medium-term: build a stable front proxy that owns MCP session state and treats `context-server` as a restartable backend. RSS recycle should restart only the backend when possible; launcher self-exit should not be the primary recycle mechanism for connected clients.
- Effort: L · Risk: high · Blast radius: launcher, IPC bridge, context-server transport lifecycle, MCP configs, daemon reliability tests.

## Cross-cutting risks

- Logical identity is split across `file_path`, `canonical_file_path`, nullable `anchor_id`, active projection, and lineage. Until this is unified, fixes can repair one path and duplicate another.
- “Force” currently means different things at scan, save, dedup, and embedder-reindex layers. That semantic mismatch is the P0 index bug.
- The MCP launcher is doing both process supervision and client transport ownership by implication. That coupling makes memory recycling look like availability recovery while actually breaking initialized sessions.
- Large operational tables now exceed assumptions baked into JSON-in-memory tooling: checkpointing, reporting, and health surfaces should avoid whole-database materialization.

## Recommended remediation ORDER (with rationale)

1. Fix index refresh semantics first: force/provider reprocess must update active logical rows and backfill `file_mtime_ms`. This stops ongoing index drift.
2. Normalize `specFolder` discovery and add handover/spec-doc coverage tests. This is small and directly addresses the missing `handover.md`.
3. Add duplicate-key cleanup plus logical-key uniqueness. Do this after the force semantics are clear, or the migration may encode the wrong identity.
4. Remediate the 6 failed embeddings using dry-run reconcile, then targeted retry for rows outside reconcile’s ownership.
5. Replace checkpoint serialization with chunked/streaming v2. It is P1, but lower operational urgency than the P0 index correctness and MCP stability paths.
6. Redesign MCP lifecycle around a stable front proxy or explicit reconnect contract. This is the largest change; start with “fail closed and reconnect clearly” before attempting transparent backend restarts.
