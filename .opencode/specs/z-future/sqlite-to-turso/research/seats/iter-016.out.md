# Iteration 16: W6 pivot: ATTACH experimental gate and shard-architecture fallout

## Focus
Adversarial revalidation of iteration 13's pivotal claim that the Turso JS SDK at v0.7.0-pre.6 cannot enable SQL ATTACH (zero grep hits for `experimental_attach` under bindings/javascript), which would strand the spec-kit vector-shard architecture (ATTACH `active_vec`, cross-schema KNN queries, `VACUUM active_vec INTO` checkpoint snapshots). I exhaustively enumerated the JS-enableable experimental features (Q1), read the VACUUM translation and execution paths to settle whether schema-qualified `VACUUM INTO` works (Q2), searched for programmatic attach surfaces in the napi layer (Q3), and re-derived the real blocker list plus an evaluation of the three fallback shapes against the actual spec-kit shard code (Q4).

## Findings
- NEW — **Iteration 13's pivot premise is REFUTED: the JS native SDK CAN enable SQL ATTACH.** The connect option `experimental: ["attach"]` maps directly to `"attach" => core_opts.with_attach(true)` [SOURCE: bindings/javascript/src/lib.rs:283], which sets the same core `DatabaseOpts` flag the CLI's `--experimental-attach` sets via `with_attach` [SOURCE: core/lib.rs:269, core/lib.rs:2718]. Iteration 13's zero-hit grep was a naming artifact: the JS binding calls the builder `with_attach`, never the accessor string `experimental_attach`.
- NEW — **Working ATTACH-from-JS is proven by in-tree tests**: `connect(path1, { experimental: ["attach"] })` followed by `db1.exec("ATTACH '${path2}' as secondary")` in the promise API test [SOURCE: bindings/javascript/packages/native/promise.test.ts:220-231] and the better-sqlite3-compat `new Database(path1, { experimental: ["attach"] })` test [SOURCE: bindings/javascript/packages/native/compat.test.ts:115-126]. Turso's own SQL-runner harness also passes `experimental: ['triggers', 'attach', 'generated_columns', 'without_rowid']` [SOURCE: bindings/javascript/turso-sql-runner.mjs:86].
- NEW — **Q1 exhaustive enumeration of JS-enableable experimental features** (native napi binding, single match block; unknown strings silently ignored via `_ => core_opts`) [SOURCE: bindings/javascript/src/lib.rs:273-288]: `"views"`->`with_views`, `"strict"`->no-op (always on, back-compat), `"custom_types"`->`with_custom_types`, `"encryption"`->`with_encryption`, `"index_method"`->`with_index_method`, `"autovacuum"`->`with_autovacuum`, `"vacuum"`->`with_vacuum`, `"attach"`->`with_attach`, `"generated_columns"`->`with_generated_columns`, `"multiprocess_wal"`->`with_multiprocess_wal`, `"without_rowid"`->`with_without_rowid`. Core builder methods confirmed at [SOURCE: core/lib.rs:239-279].
- NEW — **TS type/native drift (minor):** the common TS union lists `'triggers'` but the native match block has NO `"triggers"` arm (silently ignored), while native-accepted `'custom_types'` and `'generated_columns'` are missing from the union [SOURCE: bindings/javascript/packages/common/types.ts:1 vs src/lib.rs:275-288]; the napi `.d.ts` is an untyped `Array<string>` so all strings pass through [SOURCE: bindings/javascript/packages/native/index.d.ts:155-156].
- NEW — **Scope limits of attach-from-JS:** (1) WASM/browser package: attach test is commented out — "attach is not supported in browser for now" [SOURCE: bindings/javascript/packages/wasm/promise.test.ts:208-219]. (2) Sync SDK: the sync binding constructs its inner `DatabaseOpts` with hardcoded `experimental: None`, so NO experimental feature (attach included) is enableable through the sync packages [SOURCE: bindings/javascript/sync/src/lib.rs:252-260]. Neither limit affects spec-kit's local Node daemon use case.
- NEW — **Q2: the main-only rejection applies ONLY to bare in-place VACUUM, not VACUUM INTO.** `translate_vacuum` has two branches: `into: Some(...)` emits `Insn::VacuumInto { schema_name, dest_path }` with no experimental gate and no schema check [SOURCE: core/translate/vacuum.rs:28-37]; the `None` branch carries the experimental-vacuum gate (lines 39-44), the multiprocess-WAL incompatibility (45-49), and the `schema_name != "main"` rejection (51-57) [SOURCE: core/translate/vacuum.rs:38-62]. At execution, `op_vacuum_into_inner` resolves the schema generically via `get_database_id_by_name(schema_name)` (any attached DB; `temp` is a SQLite-compatible no-op) [SOURCE: core/vdbe/execute.rs:16041-16054]. Therefore spec-kit's `VACUUM active_vec INTO ...` checkpoint snapshot [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/storage/checkpoints.ts:2284] compiles and executes — provided `active_vec` is attached — and `VACUUM main INTO` [checkpoints.ts:2281] works unconditionally. Constraint: VACUUM INTO refuses to run inside an open transaction [SOURCE: core/vdbe/execute.rs:16069-16076].
- NEW — **Q3: no dedicated programmatic attach API exists in the napi layer, and none is needed.** Greps for `attach_database`/`fn attach` over core/lib.rs and bindings/javascript/src returned nothing; ATTACH flows through the ordinary SQL exec path once the flag is set. The binding's `classify_sql` even classifies `Stmt::Attach`/`Stmt::Detach` as `"read"`, i.e. ATTACH is routed like any normal statement with no binding-level block [SOURCE: bindings/javascript/src/lib.rs:608-612].
- PARTIAL — **Q4 revised real-blocker list for the shard architecture.** ATTACH availability drops OFF the blocker list. What remains: (1) KNOWN (prior iterations) — `vec_memories` is a sqlite-vec virtual table loaded via `sqliteVec.load(database)` [SOURCE: lib/search/vector-index-store.ts:1090; lib/search/db-shard-migration.ts:8-9,242] and the hot queries call `vec_distance_cosine` [SOURCE: lib/search/vector-index-queries.ts:314,403,407] — porting the vec table/function surface to Turso's native vector support is still the dominant migration work; (2) NEW — schema-qualified PRAGMA on the attached shard (`PRAGMA shard.journal_mode = WAL` [SOURCE: db-shard-migration.ts:245]) is unverified in Turso this iteration; (3) NEW — experimental-status risk: attach works but is gated precisely because it is experimental (cross-attached transaction semantics/WAL behavior unverified); (4) KNOWN — the migration utility itself is better-sqlite3-specific and double-ATTACHes legacy+shard [SOURCE: db-shard-migration.ts:243-244].
- NEW — **Cross-schema SQL joins DO exist on the hottest path**, which is decisive against fallback (b): the KNN search runs a single SQL statement joining main-schema `memory_index` + `active_memory_projection` with the attached vec table (`JOIN ${vectorSource.tableName} v ON m.id = v....` where query paths use `active_vec` after attach) [SOURCE: lib/search/vector-index-queries.ts:311-324; vector-index-store.ts:546 "Query paths use active_vec after attach"], plus the multi-concept distance variant [vector-index-queries.ts:400-419]. Other cross-schema touchpoints: embedding-reconcile reads `active_vec` for coverage repair [SOURCE: lib/embedders/embedding-reconcile.ts:115,161-170,299], checkpoints introspect/count `active_vec.sqlite_master` / `active_vec.vec_memories` [SOURCE: checkpoints.ts:653,750,757,1117-1124], and the semantic trigger matcher targets `ACTIVE_VECTOR_SCHEMA = 'active_vec'` [SOURCE: lib/triggers/semantic-trigger-matcher.ts:95].
- NEW — **Shard-split purpose (fallback (c) context):** the split moves heavy embedding payloads (`vec_memories`, `embedding_cache`, all `vec_*` tables) out of the canonical DB into a per-profile shard file (`get_vector_shard_path(profile, baseDir)`) for "canonical slimming" — the shard copy is authoritative and the tables are dropped from main [SOURCE: db-shard-migration.ts:17-18,43-47,126-131; vector-index-store.ts:1096-1098]. Consolidating back (c) re-bloats the canonical DB and forfeits cheap per-profile shard swapping.
- NEW — **Fallback evaluation and recommendation: choose (a).** (a) Keep ATTACH, enable `experimental: ["attach"]` at connect — effort is near-zero (one option in the connect call; add a startup capability probe), preserves all SQL shapes including cross-schema KNN joins and `VACUUM active_vec INTO`; residual risk is experimental-tier stability. (b) Separate connections per shard — must rewrite the hot KNN and multi-concept queries into two-phase app-level joins (shard KNN -> id/distance list -> hydrate from main), plus rework embedding-reconcile, checkpoints introspection, and the trigger matcher; medium-high effort and loses single-connection visibility between main and shard. (c) Consolidate the shard into main — medium effort, but an architectural regression against the explicit canonical-slimming design. Recommend (a), with (b) as the contingency if Turso's experimental attach proves unstable for write-bearing maintenance paths.

## Ruled Out
- "The JS SDK cannot enable SQL ATTACH at all" (iteration 13's pivotal finding) — refuted by `"attach" => core_opts.with_attach(true)` [bindings/javascript/src/lib.rs:283] plus passing in-tree tests; the original grep keyed on the wrong identifier (`experimental_attach` vs `with_attach`).
- "VACUUM active_vec INTO is blocked by the main-only schema rejection" — the rejection at core/translate/vacuum.rs:51-57 is inside the in-place (`into: None`) branch only; VACUUM INTO resolves arbitrary attached schemas at execution.
- "A separate napi attach method is required for JS shard attachment" — the normal exec path carries ATTACH once the flag is set; no binding-level interception exists.

## Dead Ends
- Grep for `attach_database` / `fn attach` across core/lib.rs and bindings/javascript/src: zero hits — no programmatic attach API surface to document.
- Sync-package experimental forwarding: the sync binding hardcodes `experimental: None` (sync/src/lib.rs:258), so chasing a sync-SDK attach path is futile at this tag.
- Searching for SQL-level `JOIN ... active_vec` literals in spec-kit lib found none directly — the join is parameterized (`${vectorSource.tableName}`), only resolvable by reading vector-index-queries.ts around lines 311-324.
