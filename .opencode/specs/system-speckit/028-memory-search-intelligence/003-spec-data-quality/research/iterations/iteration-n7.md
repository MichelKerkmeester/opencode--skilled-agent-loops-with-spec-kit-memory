# Iteration N7 - Rollout and Migration Sequencing (opus, prove-first)

## TITLE

Cohort N7 rollout and migration. Model opus. Angle designs the safe staged rollout across the existing 2022-packet corpus for the two confirmed levers from wave 1 and 2: the JSON-schema metadata gate wired into validate.sh strict, and the header-path embedding prefix behind an embedding coverage guard. It nails the order of operations so no gate blocks the live corpus before its backfill runs, with a per-stage rollback and a per-stage validation checkpoint, all grounded in re-measured corpus counts.

## FINDINGS

The migration surface is real but the wave-2 absolute counts are root-and-doctype dependent. A fresh read-only re-measure on the live `.opencode/specs` root finds 11 graph-metadata.json files that fail JSON.parse, not 14. The 14 in R2 counted both spec roots together. The legacy `specs` root has 0 invalid graph files (`find specs -name graph-metadata.json` yields 0 invalid), so the live repair surface is 11. All 11 are byte-identical in shape. They open `Packet: <slug>` then `Spec Folder:`, `Status:`, `Importance Tier:`, `Summary:`, `Parent:`. This is the indexable-text mirror format written into a graph-metadata.json filename, the same class R2 named.

Every one of the 11 sits in a `research/<NNN-phase>/iterations/` directory and every one has a description.json but no spec.md. That matters for sequencing because the validator already treats a folder with description.json as a Level-1 spec folder. A live strict run on one of them returns exit 2 today, but it fails on `FILE_EXISTS: Missing 4 required file(s) for Level 1`, while `GRAPH_METADATA_PRESENT` reports pass on the same folder because it only checks presence (`orchestrator.ts:397`). So the garbage text-as-JSON file is invisible to the current gate. The new JSON-shape rule catches a defect the current gate is structurally blind to, but the same 11 folders are also mis-classified non-packet iteration containers. The sweep has to fix the classification, not only re-emit JSON.

No current validation rule parses graph-metadata.json as JSON. A grep for `JSON.parse` of the graph file or `graphMetadataSchema.safeParse` across the validation lib is empty. The only graph check on the validation path is the presence warn at `orchestrator.ts:397`. This confirms the candidate-4 gap end to end and means the schema rule is net-new validation, not a tightening of an existing parse.

The strict bypass it would replace is dormant, which de-risks stage 1. A corpus grep for `legacy_grandfathered: true` returns 0 packets. The bypass at `validate.sh:175-183` and the strict-warn suppression at `print_summary` and `generate_json` are live code but no packet currently flips them on. So removing the legacy-grandfathered escape hatch breaks no packet that relies on it today, because none do. The escape hatch is a future-proofing artifact, not a load-bearing migration crutch.

The frontmatter and continuity surface is the genuine staging hazard, and it is larger than a single number. Re-measured across the ten canonical authored doc types under the live root excluding z_archive, 504 of 5453 canonical docs miss frontmatter (first line not `---`). Re-measured for the two continuity-bearing doc types, 73 of 2100 spec.md plus implementation-summary.md docs miss a `continuity:` block. R2's 287 and 913 come from a wider markdown sweep and a different doctype scope. The exact integer is not the rollout-critical fact. The rollout-critical fact is that hundreds of canonical docs fail a frontmatter hard gate on day one and dozens fail a continuity hard gate, so neither can flip to error before its backfill lands.

Today's strict mode already error-promotes the warn rules, which reshapes the staging design. `validateFrontmatterBasics` (`orchestrator.ts:362`) and `FRONTMATTER_MEMORY_BLOCK` (`orchestrator.ts:393`) both emit warn, and under `--strict` with no legacy-grandfathered a warn becomes RESULT FAILED exit 2 (`print_summary` strict branch, `generate_json` passed flag). So the corpus is not gate-free today. A packet missing its continuity block already strict-fails. The staging lever is therefore not warn-versus-error on these existing rules, it is whether the NEW JSON-parse-and-shape rule lands as warn first then error, and whether the existing frontmatter warns are left untouched until the backfill clears them. Staging the new rule as warn keeps it from compounding the existing strict failures during the backfill window.

The header-path prefix forces a full re-index and there is no named coverage guard variable to gate it. The embedding cache is content-hash keyed (`embedding-pipeline.ts:121-123`, R1 and L2 confirm). Changing embedded text changes the hash, so the prefix only takes effect after a full re-index that re-embeds every chunk. There is reindex machinery (`reindex.ts:790` startReindex, `:895` startVectorShardRepairReindex, `:838` getJobStatus) and a trigger-embedding-backfill path, but a grep for `embeddingCoverage`, `coverage_guard`, `coverageThreshold` finds nothing. The coverage guard the brief and the R-cohort assume is a concept, not an existing gate. So the rollout must introduce a coverage readout (count of chunks embedded under the current embedding_context_version over total chunks) and refuse to publish recall numbers while that ratio is below threshold, rather than wire into a guard that already exists.

The 3-result floor is code-confirmed at `confidence-truncation.ts:35` (DEFAULT_MIN_RESULTS = 3). This is why recall measured under partial re-index coverage is doubly untrustworthy. A half-re-indexed corpus mixes prefixed and unprefixed vectors, and the floor then truncates to three results that may straddle both vector generations, so any recall delta is confounded by coverage, not just by the prefix. The coverage guard exists precisely to refuse trusting recall until coverage is complete.

## CONCRETE CHANGE

The staged plan, ordered so no gate blocks the live corpus before its backfill runs.

Stage 0, baseline capture, read-only. Record the real starting numbers before any change: 11 invalid graph files (live root), 0 legacy-grandfathered packets, 504 canonical docs missing frontmatter, 73 spec-plus-impl docs missing the continuity block, and the current embedded-chunk count from the index. This is the delta baseline for every later checkpoint. Rollback: none, read-only. Checkpoint: the four counts plus the chunk count are written to the packet before any mutation.

Stage 1, sweep the 11 text-not-JSON graph files. For each of the 11 iterations directories, either regenerate a valid graph-metadata.json through the graph-metadata refresh path or, because these are iteration containers not packets, remove the stray graph-metadata.json and the stray description.json so the folder stops being detected as a Level-1 spec folder. Decide per folder against whether it is meant to be a packet. Rollback: git revert the sweep commit, the 11 files are tracked. Checkpoint: re-run the JSON.parse scan, expect 0 invalid graph files on the live root, and re-run strict on each of the 11 parents to confirm the FILE_EXISTS failure also clears if the classification fix was chosen.

Stage 2, land the JSON-schema rule as warn-only. Add the metadata-artifact-schema rule (graph must JSON.parse and pass graphMetadataSchema.safeParse, description must pass the shared description schema) wired into the orchestrator and a shell fallback, but emit warn not error, and leave the dormant legacy-grandfathered bypass in place for now. Run it across the whole corpus as a report. Rollback: revert the rule commit, behavior returns to presence-only. Checkpoint: the warn report shows 0 graph or description shape warnings corpus-wide (because stage 1 cleared the only 11), proving the corpus is gate-ready before the rule hardens.

Stage 3, flip the JSON-schema rule to error and remove the legacy-grandfathered bypass. Only after stage 2 reports a clean corpus. The graph and description shape rule becomes a hard strict failure exit 2, and the `legacy_grandfathered` escape hatch is deleted since 0 packets use it. Rollback: revert to the warn severity and restore the bypass branch, both single-commit reversions. Checkpoint: strict on every packet still exits 0 for graph and description shape, and a deliberately corrupted graph file in a scratch packet now exits 2.

Stage 4, frontmatter and continuity backfill, behind a deferred gate. Run a dry-run backfill report that lists the 504 docs missing frontmatter and the 73 missing the continuity block. Apply the backfill in batches through the description and continuity refresh paths. Leave the existing frontmatter warn rules exactly as they are, because they already strict-fail and tightening them further before the backfill would only widen the day-one breakage. Do not introduce any new frontmatter or continuity hard rule until this backfill report reads zero. Rollback: the backfill is additive frontmatter writes, revert the batch commits. Checkpoint: re-measure both gaps, expect the missing-frontmatter count and the missing-continuity count to fall to 0 before any frontmatter rule is considered for promotion.

Stage 5, header-path prefix behind the embedding coverage guard, retroactive by necessity. Introduce an `embedding_context_version` and a coverage readout that counts chunks embedded under the current version over total chunks. Ship the prefix on-write first so new and re-saved packets get it, then run a full re-index. Refuse to publish or trust any recall number while coverage is below threshold, because partial coverage mixes prefixed and unprefixed vectors under the 3-result floor and confounds the delta. Rollback: the version field lets the system fall back to the prior vectors, and the prefix can be disabled at the chunking seam without a data migration since old vectors remain valid under their old version. Checkpoint: coverage readout reaches 100 percent under the new version, then a dual-mode eval read confirms the prod-mode recall number moves, not just the eval-mode number.

The order is the safety property. Stage 1 clears the only shape defects, stage 2 proves the corpus is clean before stage 3 hardens, stage 4 backfills frontmatter before any frontmatter rule could harden, and stage 5 gates recall trust on full coverage. No gate reaches error before its backfill has run.

## READER

Adherence and logic primary for stages 1 through 4, the metadata gate improves the quality of indexed material rather than the live ranking, so retrieval benefit is downstream hygiene. Retrieval primary for stage 5, the header-path prefix targets vector and hybrid prod ranking but only after full coverage, and the coverage guard exists to stop a partial-coverage recall number from misleading either reader.

## ON-WRITE OR RETROACTIVE

Both, staged. The schema rule is on-write at the workflow choke point plus a retroactive corpus sweep, and stages 1 through 3 make the retroactive sweep safe before the on-write gate hardens. The frontmatter and continuity backfill is retroactive batch repair, with the existing on-write warn rules left untouched until the backfill clears. The header-path prefix is on-write for new and re-saved packets plus a mandatory retroactive full re-index, since the content-hash cache key means the prefix is inert on existing vectors until they are re-embedded.

## RISK

The biggest risk is hardening a gate before its backfill runs, which the stage order is built to prevent. Three residual hazards remain. First, the 11 iteration containers are mis-classified non-packets, so a sweep that only re-emits JSON without fixing the classification leaves them strict-failing on FILE_EXISTS, which would mask the schema-rule signal. Second, the frontmatter count is doctype and root dependent, so the dry-run report must be the source of truth for the stage-4 batch, not any single quoted integer, and the two roots must be deduplicated before quoting a repair count. Third, the coverage guard does not exist yet, so stage 5 ships the guard and the prefix together or the recall numbers it produces under partial coverage will be trusted wrongly, exactly the 028 eval-versus-prod trap restated at the migration grain.
