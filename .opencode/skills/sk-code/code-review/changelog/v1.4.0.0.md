This release folds the sk-code-review slice of the packet 146 ponytail-based refinement into the review baseline — a set of reusable review teachings transplanted from an external reference and added on top of the existing contract without moving any of its floors. The theme is restraint and durability: the reviewer now actively looks for code that reinvents something the language or platform already provides for free, asks whether new code was ever asked for (and recommends removal, not just simplification, when it was not), honors a concrete "this is a deliberate shortcut" note as evidence that a too-simple finding is intentional, lets a caller name and persist the review depth they want, and ships a CI canary that fails loudly the moment the load-bearing verdict vocabulary or the cross-document Iron Law wording drifts out of agreement.

> Spec folder: `.opencode/specs/skilled-agent-orchestration/142-sk-code-ponytail-based-refinement` (Level 2)

---

## New Features

All five additions land in the review references plus one new script. None of them change the public review API or relax a baseline minimum; they change what reviewers look for and add a drift guard around the wording the dispatcher depends on.

#### Reinvent-the-wheel review rows

Reviews used to wave through code that re-implements behavior the language or the platform already hands you. Two new §6 Maintainability checks in `references/code_quality_checklist.md` name the smell directly: hand-rolled standard-library behavior where a clear, readable primitive already exists and its behavior and edge cases match, and custom code or a dependency that duplicates a native platform or runtime capability with no current requirement the native feature cannot satisfy. The recommended fix in both cases is to prefer the standard or native API instead of carrying a bespoke copy.

&nbsp;

#### Needed-ness prompt

The §7 KISS pass used to ask "can this be simpler?" but never "should this exist?". New code that traces to no current requirement — a feature, parameter, branch, or config that nothing in the stated scope asked for — now earns a removal recommendation rather than only a simplification suggestion, cross-referenced to `removal_plan.md`. The reviewer asks whether the code was asked for and whether anything would break if the requirement were dropped. The finding defaults to P2 and escalates to P1 only when the unneeded code adds attack surface, contract obligations, or regression and maintenance risk. The companion `references/removal_plan.md` §2 safe-to-remove-now table gains a new `Replacement` field that records what takes the removed code's place — nothing, a standard-library API, a native platform feature, or a shorter equivalent expression.

&nbsp;

#### Intentional-simplification evidence

A deliberate shortcut used to read the same as an oversight, so reviewers flagged both. A new §7 rule treats a concrete `ceiling:` comment — one that names the shortcut, its known ceiling, and the upgrade path or trigger — as evidence that a "too simple" or "missing feature" KISS or YAGNI finding was a choice, and downgrades or suppresses that P2. The exemption is deliberately narrow: it NEVER applies to security, authentication, persistence, sandboxing, public-contract, or correctness findings, which are judged on their own contract no matter what any comment says.

&nbsp;

#### Review-depth alias

Callers who wanted a consistently deep, or consistently light, review had to repeat "comprehensive" or "full review" on every run. The new §9.3 `SK_CODE_REVIEW_DEPTH=lite|full|ultra` alias names and persists routing that already existed, resolved env over config over default in the same idiom as `SK_CODE_REVIEW_MIN_CHANGED_LINES`. The `full` value (the unset default) is the normal ALWAYS plus CONDITIONAL plus ON_DEMAND routing; `ultra` biases intent selection toward the existing ON_DEMAND deep-dive reference set for the session; `lite` maps to the existing M-2 conservative skip. It adds no new tier and relaxes no floor — `lite` never lowers the ALWAYS tier, the baseline security and correctness minimums, or the P0/P1/P2 contract, and it can never skip a review on a sensitive path (auth, config, persistence, dependencies, sandbox, or public-response).

&nbsp;

#### Rule canary (drift guard)

Some rules have to read identically in more than one place: the `Review status:` verdict triplet that downstream PR-state dedup logic keys on, and the cross-document Iron Law that forbids completion claims without verification. When an editor updated one copy and forgot the others, the docs silently disagreed and the guarantee rotted. A new `scripts/check-rule-copies.js` canary asserts the load-bearing substrings still exist — the exact verdict strings in `SKILL.md` and `README.md`, the COMMENTED status in the v1.3.0.0 changelog and the dedup reference, and an Iron Law line carrying both "completion claim" and "verification" in sk-code `SKILL.md`, the root `CLAUDE.md`, and `AGENTS.md`. It validates every Iron Law line in each of those files, not just the first match, so a tampered duplicate cannot hide behind a clean leading line — and it fails loudly when any copy drifts. It is a checker, not a generator: it never rewrites anything. A `scripts/check-rule-copies.test.sh` self-test and a `.github/workflows/rule-canary-sync.yml` workflow — which fails closed if the canary is ever deleted or moved — wire it into CI.

## Upgrade

No migration required. The new review rows and prompts change what a reviewer looks for, not the output contract. The depth alias is opt-in and preserves prior behavior when unset. The canary only runs in CI and as an explicit local check, and every review keeps emitting the same `Review status:` final line.
