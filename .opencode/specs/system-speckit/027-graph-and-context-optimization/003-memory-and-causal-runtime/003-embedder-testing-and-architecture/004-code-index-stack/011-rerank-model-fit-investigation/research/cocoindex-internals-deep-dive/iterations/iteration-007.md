# Iter 7 — Decision rubric for A/B/C/D — fix-probability, cost, regression risk, downstream unblocks

## TL;DR (2-3 sentences)

The utility-optimized ranking is **A (jina-v3) > C (jina+path-class) > D (bench fix) > B (candidate-set)**, with A scoring 0.759 on the weighted formula `0.45·P(fix≥3) + 0.30·P(no_regression) + 0.15·downstream_unblock + 0.10·(cost+latency)`. The dominant driver is P(fix≥3): jina-v3 has bench-proven evidence of flipping probes 14 and 18, giving 3 of 4 failure probes passing (probes 3+14+18), while candidate-set fixes remain speculative and bench-only fixes address only probe 10's fixture-truth mismatch. The emergent recommendation is **A + D**: promote jina-v3 to production while simultaneously fixing the bench harness and probe 10 fixture, which yields 4/4 failure probes passing and leaves the probe 5 control regression as the sole remaining concern requiring jina-specific score calibration in the 012 packet.

## Question (restate the scoped RQ)

For each: (a) probability of fixing ≥3 of 4 failure probes (0-1), (b) probability of zero control regression (0-1), (c) implementation cost (S/M/L), (d) inference latency delta (ms), (e) what gets UNBLOCKED downstream (cost-benefit chain). Rank by utility = 0.45·P(fix≥3) + 0.30·P(no_regression) + 0.15·downstream_unblock + 0.10·(cost+latency).

## Findings (numbered, with citations)

### Option A — Production jina-v3 adapter packet

1. **P(fix≥3) = 0.90**. Jina-v3 flips 2 of 3 baseline-missing failure probes: probe 14 (miss→hit, evidence at `jina-v3.results.jsonl:6`) and probe 18 (miss→hit, evidence at `jina-v3.results.jsonl:8`). Probe 3 already hits baseline (`baseline-bge.results.jsonl:2`). Probe 10 is unfixable by reranker swap (fixture-truth, `.js` dist not in index — `baseline-bge.results.jsonl:4`, `jina-v3.results.jsonl:4`). Result: 3 of 4 failure probes pass. The 0.10 discount accounts for the possibility that probe 5 regression or probe 1 shared-miss interact with failure probe outcomes under different daemon states, but the pre_score identity across lanes (`baseline-bge.rerank-scores.jsonl:6` vs `jina-v3.rerank-scores.jsonl:6`) makes cross-contamination unlikely. Supporting citations: `phase2-comparison.md:5-14`, `phase2-comparison.md:18-22`, `phase2-bench/README.md:5-9`, Iter 5 findings at `iteration-005.md:17-26`, `iteration-005.md:29-35`, Iter 6 findings at `iteration-006.md:29-47`.

2. **P(no_regression) = 0.15**. The observed control regression is probe 5: baseline HIT (`baseline-bge.results.jsonl:3`), jina-v3 MISS (`jina-v3.results.jsonl:3`). Root cause: jina-v3 scores the expected `code-graph-status-readiness-snapshot.vitest.ts` at 0.147 vs stress test `w7-degraded-stale.vitest.ts` at 0.468 — a 3.2x gap (`jina-v3.rerank-scores.jsonl:3`). Both are tests-class, so path-class stacking (option C) offers no mitigation. Fixing this requires model-level changes (score calibration, query-aware weighting) or document-level interventions (content clipping, query rewriting). The 0.15 probability reflects the chance that jina-v3 document clipping adjustment (max_chars knob at `rerankers_jina_v3.py:140-150`) or per-query score normalization within the adapter could close the gap — unlikely given the magnitude, but not impossible with aggressive content truncation. Supporting citations: `jina-v3.results.jsonl:3`, `jina-v3.rerank-scores.jsonl:3`, `baseline-bge.results.jsonl:3`, `rerankers_jina_v3.py:140-150`.

3. **Cost = S**. The throwaway adapter already exists at `rerankers_jina_v3.py:127-190` with full head/tail contract, optional document clipping, and score-logging support. Production promotion requires: (a) remove throwaway/experimental markers from the module, (b) add `jina_v3` to the adapter selection chain in `reranker.py:215-223`, (c) update default adapter docs in `config.py:415-420`, (d) write a 012 packet implementing the promotion and probe-5 regression mitigation. No re-indexing, no DB migration, no chunker changes. Supporting citations: `rerankers_jina_v3.py:127-190`, `reranker.py:215-223`, `config.py:415-420`.

4. **Latency delta = +495ms median, +2446ms p95**. From `phase2-comparison.md:20-22`. The p95 inflation is dominated by probe 1 (15059ms at `jina-v3.results.jsonl:1` — first jina-v3 invocation, includes model loading) and probe 18 (3105ms at `jina-v3.results.jsonl:8`). Non-first invocations show comparable latency: probe 14 is 1319ms (jina) vs 1428ms (baseline), probe 16 is 1897ms (jina) vs 1460ms (baseline). A production deployment would amortize model-load time across all queries via daemon caching; the steady-state latency delta is likely smaller than the bench p95 suggests. Supporting citations: `phase2-comparison.md:20-22`, `jina-v3.results.jsonl:1-8`, `baseline-bge.results.jsonl:1-8`.

5. **Downstream unblocks (score 0.85)**: (a) 012 packet directly — the jina-v3 production path is the most actionable next step, (b) jina-specific performance tuning (document clipping optimization, batch scoring if jina-reranker-v3 supports it), (c) model-version tracking — establishing a pattern for future reranker upgrades, (d) latency benchmarking under production conditions (daemon-cached model, amortized first-load cost). Cost-benefit chain: jina-v3 production → 012 packet → reranker upgradeability infrastructure → all future model-swap bench protocols benefit.

### Option B — Candidate-set fix (chunking / hybrid / query expansion)

6. **P(fix≥3) = 0.35**. Chunking improvements could help probe 14 by making implementation-body chunks (e.g., `findFiles` at structural-indexer.ts lines 1391-1465) reach the top pre_score position instead of the 9-line import header chunk. However, the current `RecursiveSplitter` only exposes chunk_size/min/overlap knobs (`indexer.py:33-39`, `config.py:370-387`) — no tree-sitter or code-aware splitting is available. Increasing chunk_size to 4000 would merge the import header with implementation body, but also reduce total candidate diversity. Query expansion (synonym maps, camelCase decomposition) could bridge "filesystem walker" → "findFiles" and "import edges" → "extractEdges" for probe 14, and "integration test for index refresh" → Python test function names for probe 18, but `query.py` has no expansion infrastructure (`iteration-004.md:13-15`). Hybrid recalibration is ineffective: structural-indexer already had the highest pre_score (0.0216) before rerank, and equal 0.7/0.7 weights benefit dual-lane candidates. Probe 10 remains unfixable (fixture-truth). At best, chunking + query expansion fixes 2 of 3 baseline-missing failure probes (14, 18), but neither is bench-proven and both require new infrastructure. Supporting citations: `iteration-002.md:29-35`, `iteration-004.md:13-24`, `indexer.py:33-39`, `config.py:370-387`, `query.py:55-74`, `fusion.py:42-87`.

7. **P(no_regression) = 0.70**. Chunking changes are moderate-risk: existing control probes (1, 5, 11, 16, plus the already-hitting failure probe 3) could shift rank when chunk boundaries change. However, no reranker model is swapped, so BGE's scoring function remains the same — the risk is that different chunks from the SAME file now dominate, which is lower-risk than swapping scoring functions entirely. Query expansion has higher regression risk: OR-broadened FTS queries could introduce new distractors that displace controls. The 0.70 reflects chunking-only risk; query expansion alone would be 0.50. Supporting citations: `indexer.py:273-283`, `fts_index.py:88-132`, `phase2-comparison.md:5-14`.

8. **Cost = M**. Changing chunking requires: (a) re-index the entire codebase with new parameters (86635 FTS rows, ~10-30 minutes), (b) re-run the full 8-probe bench, (c) if `RecursiveSplitter` cannot be configured for code-aware splitting, write or integrate a custom chunker (tree-sitter, AST-aware). Adding query expansion requires: (a) a new expansion module, (b) integration into `query.py:579-585` before embedding, (c) FTS query normalization awareness, (d) re-bench. Supporting citations: `indexer.py:33-39`, `indexer.py:320-365`, `query.py:579-585`, `fts_index.py:88-132`.

9. **Latency delta = 0ms**. No model change, no additional inference. If larger chunks are used, fewer candidates pass through fusion/dedup, potentially reducing latency (minor, unmeasured). Supporting citations: `query.py:594-640` (candidate assembly scales with fetch_k, not chunk size), `phase2-comparison.md:20-22` (baseline latencies).

10. **Downstream unblocks (score 0.90)**: (a) chunking experiments benefit ALL future reranker/model choices — it's a shared infrastructure improvement, (b) query expansion plugin architecture enables future NL-to-code bridging without per-model work, (c) general retrieval quality uplift independent of reranker choice. Cost-benefit chain: chunking + query expansion → better pre-rerank candidates → all rerankers get better input → permanent recall improvement → higher baseline for all future bench comparisons.

Supporting citations: `baseline-bge.results.jsonl:4-8`, `run-phase2-smoke.sh:117-168`, `phase2-bench/README.md:7`, `phase2-comparison.md:5-14`.

Supporting citations: `run-phase2-smoke.sh:117-168` (parser operates on ccc output, not on daemon state).

Supporting citations: `run-phase2-smoke.sh:139-149`, `probe-subset.json` (not read but referenced at `run-phase2-smoke.sh:31`).

20. **Downstream unblocks (score 0.70)**: (a) reliable bench methodology for 012 and all future packets — measurement confidence is a prerequisite for any model/code change decision, (b) fixture quality standard — probe 10 illustrates the cost of fixture-truth mismatches (2 bench lanes wasted on unfixable probe), (c) confident measurement enables aggressive A/B testing without second-guessing false positives/negatives. Cost-benefit chain: bench fix → measurement confidence → 012 fixture design → faster iteration on all subsequent model swaps.

21. **Utility scores** computed per formula `0.45·P(fix≥3) + 0.30·P(no_regression) + 0.15·downstream_unblock + 0.10·(cost_score + latency_score)` where cost: S=1.0, M=0.5, L=0.0 and latency_score = max(0, 1 − (delta_p95 / baseline_p95)):

22. **Sensitivity analysis**. The 0.015 gap between A and C is driven solely by downstream_unblock (0.85 vs 0.80) — path-class adds complexity without proven value. The formula is dominated by the 0.45 P(fix≥3) term: A/C gain 0.405 from this term alone, while D gains only 0.068. The 0.30 P(no_regression) term is the swing factor: if jina-v3's probe 5 regression could be mitigated (pushing P(no_regression) from 0.15 to, say, 0.60), Option A's utility would rise to `0.405 + 0.180 + 0.128 + 0.181 = 0.894` — a decisive margin. ### Synthesis: the emergent A + D combination

23. **Recommended path: A (jina-v3) + D (bench fix)**. While the rubric evaluates each option independently, the evidence supports a combined approach: promote jina-v3 to production (flips probes 14+18) AND fix the bench harness + probe 10 fixture (flips probe 10). This yields 4/4 failure probes passing (3+10+14+18), while probe 5 regression remains the sole open concern requiring jina-specific mitigation in the 012 packet. The combined cost is S+S=S (both are S-cost individually, and they modify independent subsystems). The combined latency impact is jina-v3's delta only (bench fix is zero-latency). The combined downstream unblock covers both model production path AND methodology confidence. Supporting citations: `phase2-comparison.md:5-22`, `baseline-bge.results.jsonl:1-8`, `jina-v3.results.jsonl:1-8`, `run-phase2-smoke.sh:139-149`, `phase2-bench/README.md:5-9`.
