## Mechanism
MemClaw separates lifecycle into status-only archive, soft-delete grace, and later hard purge.

Archive criteria are SQL predicates on live memories. `memory_archive_expired` marks active, non-deleted rows as `outdated` when `ts_valid_end < NOW()` and limits each batch to 500 (`core-storage-api/src/core_storage_api/services/postgres_service.py:1575`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1590`). `memory_archive_stale` marks active, non-deleted rows as `archived` when they are older than `stale_days`, have `recall_count = 0`, and `weight < max_weight` (`core-storage-api/src/core_storage_api/services/postgres_service.py:1606`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1628`). Archive operations are idempotent because rows leave the `status = 'active'` candidate set after the first update (`core-storage-api/src/core_storage_api/services/postgres_service.py:1596`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1636`).

Soft-delete is not a hard delete. `memory_soft_delete` sets `deleted_at = now()` and `status = "deleted"` (`core-storage-api/src/core_storage_api/services/postgres_service.py:418`). Hot-path updates reject soft-deleted rows and keep `deleted_at IS NULL` guards on update statements so later writes do not resurrect deleted rows (`core-storage-api/src/core_storage_api/services/postgres_service.py:486`, `core-storage-api/src/core_storage_api/services/postgres_service.py:526`).

Purge is the hard-delete phase. `memory_purge_soft_deleted` deletes rows whose `deleted_at` is older than `retention_days`; its docstring says this removes the memory row plus embedding, entity links, and idempotency response cache lines through FKs (`core-storage-api/src/core_storage_api/services/postgres_service.py:1646`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1653`). The DELETE predicate is `deleted_at IS NOT NULL AND deleted_at < NOW() - INTERVAL '1 day' * :retention_days` (`core-storage-api/src/core_storage_api/services/postgres_service.py:1674`). Purge fanout targets only tenants with soft-deleted rows older than the maximum retention window, using DB clock to avoid client-clock drift (`core-api/src/core_api/services/tenants.py:25`, `core-api/src/core_api/services/tenants.py:37`).

Retention is a bounded policy snapshot. `LifecyclePurgeRequest` carries `retention_days`; the comments state fanout reads org settings at cron time and consumers trust that snapshot (`common/events/lifecycle_purge_request.py:1`, `common/events/lifecycle_purge_request.py:4`). The configured range is 1-30 days (`common/events/lifecycle_purge_request.py:21`), settings validation reuses those constants (`core-api/src/core_api/services/organization_settings.py:340`), and unresolved config defaults to 30 days (`core-api/src/core_api/services/organization_settings.py:537`).

Supersession is status/link lifecycle, not deletion. `Memory` has `status` and `supersedes_id` fields (`common/models/memory.py:56`, `common/models/memory.py:79`). `memory_update_status` can set, clear, or CAS-check `supersedes_id` (`core-storage-api/src/core_storage_api/services/postgres_service.py:538`). Default search excludes `outdated` and `conflicted` memories while allowing explicit `status_filter` inspection (`core-storage-api/src/core_storage_api/services/postgres_service.py:1149`).

Lifecycle audit is pre-published, stateful, and sticky-success. `LifecycleAudit` stores `org_id`, `action`, `triggered_by`, `started_at`, `finished_at`, `status`, `stats`, and `error_message` (`common/models/lifecycle_audit.py:24`). The fanout path creates the audit row before publishing the event so publish failure leaves an observable `pending` row (`core-api/src/core_api/routes/lifecycle.py:99`). Handlers move the same audit row through `in_progress`, `success`, or `failure`, and SQL ops are described as safe under redelivery (`common/events/lifecycle_handlers.py:127`). Finalization stamps `finished_at` only for terminal states and blocks updates once status is already `success`, preserving original stats on redelivery (`core-storage-api/src/core_storage_api/services/postgres_service.py:4315`, `core-storage-api/src/core_storage_api/services/postgres_service.py:4350`).

## Negative knowledge
Do not copy MemClaw’s multi-tenant fanout machinery. The cron service only POSTs a fanout endpoint while core-api enumerates orgs and publishes per-org events (`core-operations/src/core_operations/tasks.py:1`, `core-api/src/core_api/routes/lifecycle.py:138`). Spec Kit is local/single-user and should use an in-process sweep, not Pub/Sub or REST fanout.

Do not copy fleet scoping. MemClaw carries optional `fleet_id` through lifecycle payloads and SQL predicates (`common/events/lifecycle_archive_request.py:31`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1584`). Spec Kit has no fleet dimension.

Do not copy per-org retention settings. MemClaw validates `lifecycle.memory_retention_days` per org (`core-api/src/core_api/services/organization_settings.py:329`). Spec Kit should use one local retention policy unless a concrete multi-profile need appears.

Do not copy distributed-scheduler dedup windows for local SQL lifecycle. MemClaw’s 23-hour dedup gate is for pipeline ops and cron double-fires (`common/events/lifecycle_handlers.py:100`, `core-storage-api/src/core_storage_api/services/postgres_service.py:4376`). Local retention/tombstone sweeps should rely on idempotent SQL predicates plus audit, not time-window distributed dedup.

Do not copy Postgres-specific partial-index assumptions directly. MemClaw uses Postgres partial indexes for live rows and purgeable rows (`common/models/memory.py:118`, `common/models/memory.py:131`). SQLite can use partial indexes too, but the exact pgvector/Postgres indexing strategy is fleet-storage machinery.
