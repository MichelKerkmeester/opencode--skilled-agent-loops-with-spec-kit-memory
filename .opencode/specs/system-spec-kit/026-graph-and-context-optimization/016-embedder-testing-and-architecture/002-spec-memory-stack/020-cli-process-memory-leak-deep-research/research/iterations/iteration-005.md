# Iteration 005: Apple Silicon swap, wired memory, sidecar/reranker/embedder/Ollama, and expected-daemon classification

## Focus

Audit how `.opencode/skills/system-spec-kit` and related cli-X / deep-research docs distinguish (a) user-process RSS leaks (Node MCP children, embedder sidecar forks, rerank sidecar Python process, ccc run-daemon, leftover cli-codex/cli-devin/cli-opencode shells) from (b) kernel-side / MPS-wired memory pressure and from (c) expected long-lived services (Ollama on :11434, rerank sidecar on :8765, mk-spec-memory daemon owning `daemon-ipc.sock`, cocoindex `ccc run-daemon`). Find what is documented, what is missing, and where the missing classification has already caused operator sweeps to over-kill or under-kill.

8. **The embedder Node-fork sidecar's idle-evict is silent process termination by design.** `SidecarClient.scheduleIdleEviction()` arms a `setTimeout(this.idleMs, ...)` (default 300 s) that calls `killWorker()` → `child.kill('SIGTERM')` (sidecar-client.ts:294-300, 419-429). The next embed call respawns through `ensureWorker()`, paying cold-load cost plus Apple Silicon MPS warmup. No doc warns operators that a long-running deep-research iteration that idles the embedder for > 300 s will see the sidecar self-evict and a fresh `[sidecar:<pid>]` line in MCP stderr — they may mistake the SIGTERM exit + respawn for a leak or crash. [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/sidecar-client.ts:72,154-156,294-302,419-429]

## Edge Cases

- **Ambiguous input:** The deep-research-config.json's `recordedSwapUsage` field reads like a runtime gate but is dead metadata. Future audits could spuriously assume the system enforces a preflight that does not exist.
- **Contradictory evidence:** Arc 008 phase 005 HOLD verdict cites p95 latency; system-rerank-sidecar/SKILL.md:237 cites host swap pressure. Both can be true (latency on CPU is bad; MPS pressure is bad on CPU-less devices), but the two reasons are not reconciled into a single decision record.
- **Missing dependencies:** Ollama is referenced 30+ times in audit scope but its lifecycle is owned by the operator (`brew services start ollama` or `ollama serve`) and no audit-scope doc declares this contract. Likewise PyTorch's MPS-wired allocation is invisible to all in-process telemetry.
- **Partial success:** `memory_health` exposes `sidecar_workers` for the Node embedder fork only (Finding 9). It does not expose rerank-sidecar PID, Ollama PID, `ccc run-daemon` PID, or mk-spec-memory daemon PID, so the user-facing health surface presents a partial process picture.
