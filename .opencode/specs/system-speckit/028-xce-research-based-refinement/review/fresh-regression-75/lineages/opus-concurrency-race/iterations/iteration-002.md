## Analysis — Lock/state-dir isolation across same-kind replicas (`SPECKIT_*_STATE_DIR`)

**What I verified (file:line evidence):**

1. **Speckit-managed per-replica state/artifact dirs ARE properly isolated.** In `fanout-run.cjs:387-390`, each lineage gets `lineageDir = lineages/<label>` and `stateDir = <lineageDir>/.executor-state`, keyed entirely by `lineage.label`. Labels are validated **unique** at both the base-label level (`executor-config.ts:291-300`) and the post-expansion level (`executor-config.ts:302-311`, catching `foo`+count2→`foo-1` colliding with an explicit `foo-1`), and constrained **dir-safe** by `/^[a-z0-9][a-z0-9-]*$/` (`executor-config.ts:239-241`) — no path traversal, no `/` escape. **Same-kind replicas (count>1 → `label-1`, `label-2`) therefore cannot collide on the speckit state/artifact dir.** This part is sound.

2. **But `SPECKIT_*_STATE_DIR` does NOT isolate the native CLI's real lock — it is functionally inert for that purpose.**
   - `fanout-run.cjs:416-419` sets `SPECKIT_CODEX_STATE_DIR=<unique stateDir>` in the child env.
   - That value is **never mapped to the CLI's real home env** (`CODEX_HOME` / `OPENCODE_HOME` / `CLAUDE_CODE_HOME`). No wrapper in `cli-codex` / `cli-opencode` / `cli-claude-code` performs the mapping (grep: zero hits).
   - When the native CLI is actually dispatched, `buildExecutorDispatchEnv` (`executor-audit.ts:551-558`) filters env to the per-kind prefix allowlist (`CODEX_`/`OPENAI_`/`AZURE_OPENAI_` for codex — `executor-audit.ts:86-90`). `SPECKIT_CODEX_STATE_DIR` begins with `SPECKIT_`, so it is **dropped** and never reaches the native process. `runAuditedExecutorCommand` (`executor-audit.ts:676`) uses that filtered env directly without re-adding it.
   - The native CLI therefore writes its real lock to its **default shared home** (`~/.codex` etc.), which all same-kind replicas share. `getDefaultStatePaths` (`executor-audit.ts:184-188`) confirms `$HOME/.<home>` is **always** scanned, in addition to the per-replica dir.
   - The only thing `SPECKIT_CODEX_STATE_DIR` does is add the (empty) per-replica dir to the **detection** scan set in `getDefaultStatePaths` — and `detectFromLockfile` (`executor-audit.ts:448-468`) only matches speckit-guessed names (`codex.lock`, `cli-codex.lock`, `speckit-cli-dispatch-cli-codex.lock`, `executor-audit.ts:196-199`) that **nothing in the repo ever writes** (grep for any writer: zero hits). The layer is inert.

   Net: the comment "Isolate per-kind state dirs so same-kind replicas do not collide on lockfiles" (`fanout-run.cjs:415`, echoed `fanout-run.cjs:98`) asserts a guarantee the mechanism does not deliver for the native lock. Isolation that actually matters (artifact dirs) comes from `lineage.label`, not from this env var. Sharing the home is plausibly intentional (auth keys must pass through — `executor-audit.ts:78-84`), which makes this a claim/code legibility defect rather than a functional break.

3. **The regression test overstates its coverage.** `fanout-run.vitest.ts:184` ("sets distinct `SPECKIT_CODEX_STATE_DIR` ... to prevent lockfile collisions") uses two **distinct base labels** with `count:1` (`alpha`, `beta`), not the same-kind-**replica** path (`count>1` → `expandLineages` `label-N`). Its only assertions (`:226-230`) check filesystem dir distinctness; despite the comment at `:186`, it never asserts the stub's echoed `STATE_DIR`/env propagated distinctly, and never exercises the count-expansion path that is the actual same-kind-replica scenario.

**Not findings (checked, sound):** label uniqueness + dir-safety (no traversal via label into `lineageDir`); the `extraEnv`-after-filter ordering in `fanout-run.cjs:444` correctly re-adds `SPECKIT_*` to the *subprocess* env (just not to the native-CLI grandchild); the stack-layer recursion guard (`fanout-run.cjs:432`) is correctly the only layer at fan-out level.

No P0/P1: the speckit state/artifact dirs are genuinely collision-free, and the inert detection layer cannot produce a false-positive from any in-repo writer. The two issues are P2 legibility/coverage gaps where a load-bearing comment and a test name claim more than the code establishes.
