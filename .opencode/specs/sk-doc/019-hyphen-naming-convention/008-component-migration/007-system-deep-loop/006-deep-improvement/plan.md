# Implementation Plan: Deep-improvement filesystem names

<!-- SPECKIT_LEVEL: 2 -->
<!-- SPECKIT_TEMPLATE_SOURCE: plan-core | v2.2 -->

<!-- ANCHOR:summary -->
## 1. SUMMARY

| Aspect | Value |
|--------|-------|
| **Surface** | `.opencode/skills/system-deep-loop/deep-improvement/` |
| **Change class** | Multi-lane filesystem rename plus shared host/reference repair |
| **Execution** | Isolated worktree using the pinned BASE, classified map, and generated-output manifest |
| **Verification** | Lane routing, imports, fixture/profile paths, benchmark parity, tests, and exemption checks |

### Overview

Handle the largest child as dependency-closed lane batches. Rename authored resources across assets, references, catalogs, playbooks, scripts, and component-owned benchmark storage; classify Python/package names and generated reports before touching them; then repair the shared loop host and all lane consumers.
<!-- /ANCHOR:summary -->

<!-- ANCHOR:quality-gates -->
## 2. QUALITY GATES

### Definition of Ready

- [ ] The frozen map covers all 22 directory families and 250 underscore-bearing files with one disposition each.
- [ ] Python/package, generated-output, lockfile, tool-name, and lane-data manifests are attached.
- [ ] BASE lane-routing, fixture/profile, benchmark, import, and test-discovery evidence is captured.

### Definition of Done

- [ ] Authored deep-improvement paths and active consumers are kebab-clean and resolvable.
- [ ] All three lanes retain their shared loop-host behavior and benchmark/test parity.
- [ ] Python/package, generated, key/identifier, tool, and frozen exemptions are evidenced explicitly.
<!-- /ANCHOR:quality-gates -->

<!-- ANCHOR:architecture -->
## 3. ARCHITECTURE

- **Lane resource layer**: rename the three lane directory families, asset/reference/catalog/playbook categories, and authored fixture/profile paths through the semantic map.
- **Host/consumer layer**: update shared loop-host, command bridges, registries, tests, imports, and path-valued configs in the same reference closure.
- **Benchmark layer**: distinguish authored fixture/profile/storage names from generated report output; root `benchmark/` remains phase 009 ownership.
- **Python/data boundary**: preserve `.py` scripts, Python import-package directories, lane keys, JSON/YAML/TOML keys, and generated schemas.
<!-- /ANCHOR:architecture -->

<!-- ANCHOR:phases -->
## 4. IMPLEMENTATION PHASES

### Phase 1: Setup

- [ ] Load the deep-improvement map, BASE manifests, lane inventory, and generated-output/Python disposition lists.
- [ ] Trace loop-host, command, registry, import, benchmark, fixture/profile, and dynamic path consumers.

### Phase 2: Core Implementation

- [ ] Rename authored lane/resource directory families and the component-owned benchmark storage labels through dependency-closed batches.
- [ ] Update assets, references, catalogs, playbooks, scripts, tests, imports, registries, and path-valued configuration.
- [ ] Leave Python/package, generated/lockfile, tool-mandated, data-key, identifier, and frozen names unchanged.

### Phase 3: Verification

- [ ] Resolve every old/new lane, fixture/profile, import, host, and benchmark path.
- [ ] Compare lane routing, test discovery, fixture/profile counts, benchmark scenario IDs, and score semantics with BASE.
- [ ] Run Python import checks, Node syntax/tests, and non-zero lane/benchmark checks within the worktree.

## 7. ROLLBACK PLAN

- **Trigger**: Python import failure, generated-output loss, lane routing drift, benchmark regression, stale host path, or collision.
- **Procedure**: Stop the current lane batch; revert only the scoped commit, restore the generated/Python manifests and BASE reports, then rerun import and path closure checks before retrying.
