# Iteration 002 — Track B (mimo)

## Focus
Extraction gaps that force invention (interaction OFF by default, a11y async half uncalled, single-page motion, contrast top-20 cap, gradient non-decomposition)

## Findings
1. **[P0] B1: Interaction capture OFF by default** — Confirmed. `extract.ts:81` sets `noInteraction = true` as default; `--fast` (line 118-119) forces it off. When off, `captureInteractions` is never called (line 322), so `interactionSets` is empty. This nullifies: (a) all component variant state changes (hoverDiff, focusVisibleDiff, focusDiff, activeDiff, disabledStyle) in cluster.ts:1347-1351, (b) focusIndicator.style stays empty `{}` (a11y-extract.ts:101-103), (c) loadingStates/emptyStates/errorStates never populated. §11 State Matrix requires 4+ components × 5 state columns — entirely fabricated without interaction data. §4 Component Stylings requires hover/focus/active/disabled lines with rationale — all null. §9 Focus Indicators has empty style object.
   - Recommendation: Default `noInteraction` to `false` (opt-out instead of opt-in). Remove the `noInteraction = true` from `--fast` or add `--fast-with-interaction` escape hatch. At minimum, always capture a lightweight pass (focus + disabled only, skip hover/active) to guarantee minimum §11/§9 data.
   - Evidence: extract.ts:81 (default true), extract.ts:117-119 (--fast forces off), extract.ts:321-328 (conditional call)
2. **[P0] B2: extractA11yAsync never called — 5 §9 fields always null** — Confirmed. `extract.ts:445` calls `extractA11y(domCollections, interactionSets)` — the sync version. The async variant `extractA11yAsync` (a11y-extract.ts:361-391) which returns `tabOrder`, `langAttribute`, `skipLinkDetected`, `altTextCoverage` is exported but never imported or called from extract.ts. §9 Accessibility Contract requires: ARIA patterns (OK from ariaRoleStats), contrast ratios (OK), focus indicators (empty without B1), touch targets (OK), reduced-motion support (always undefined — see B2a), lang attribute (never populated), skip link (never populated), alt text coverage (never populated). The AI invents all five missing fields.
   - Recommendation: Call `extractA11yAsync` inside the per-page extraction loop (extract.ts:303-336), merge results into `a11yTokens` after the sync call. Requires passing `page` to the async call. Alternatively, move the page-dependent extractions (tabOrder, langAttribute, skipLink, altText) into the existing Playwright page context before `context.close()` at line 334.
   - Evidence: extract.ts:445 (sync call only), a11y-extract.ts:361-391 (async fn unused), a11y-extract.ts:148-249 (async helpers exist but unreachable)
3. **[P0] B2a: cssAnalyses not passed to extractA11y — reducedMotionSupport always undefined** — Confirmed, compounding B2. `extractA11y` signature accepts optional `cssAnalyses` parameter (a11y-extract.ts:260). `extract.ts:445` passes only `(domCollections, interactionSets)` — omits `cssAnalyses`. The `extractReducedMotionSupport` helper (a11y-extract.ts:213-220) checks for `prefers-reduced-motion` media breakpoints but is never reached. §9 'Reduced-Motion Support' subsection and §6.5 'Reduced-Motion Fallback' both require this data. Always undefined → AI fabricates a recommendation instead of reporting observed behavior.
   - Recommendation: Pass `cssAnalyses` as third argument: `extractA11y(domCollections, interactionSets, cssAnalyses)` at extract.ts:445. Single-line fix.
   - Evidence: extract.ts:445 (missing 3rd arg), a11y-extract.ts:260 (optional param), a11y-extract.ts:345-347 (conditional logic never reached)
4. **[P1] B7: Motion extracted from first page CSS only** — Confirmed. `extract.ts:439` passes `cssAnalyses[0]` to `extractMotion` — only the first page's CSS analysis. `extractMotion` (motion-extract.ts:95-173) iterates `cssAnalysis.transitions` and `cssAnalysis.animations` from that single analysis. If page 2+ has unique keyframe animations or distinct timing functions, they are discarded. §6.5 Motion System requires duration scale, easing functions, and keyframe animations — thin data forces the AI to invent a 'motion philosophy' from incomplete observations.
   - Recommendation: Merge all `cssAnalyses` before passing to `extractMotion`: create a combined CSSAnalysis with concatenated transitions/animations arrays from all pages. Or loop `extractMotion` per-page and merge the resulting MotionSystem objects (deduplicating by animation name).
   - Evidence: extract.ts:439 (`cssAnalyses[0]` only), motion-extract.ts:102-111 (iterates single analysis)
5. **[P1] B5: colorRelationships.contrastPairs capped at top-20 colors (§2, not §9)** — Partially confirmed, scope correction needed. `cluster.ts:799` filters to top-20 highest-frequency colors (`frequency >= 5`, then `.slice(0, 20)`), then computes pairwise contrast among those 20. This populates `tokens.colorRelationships.contrastPairs`. However, `extract.ts:469` OVERWRITES `tokens.a11yTokens` with the output of `extractA11y`, which computes contrast pairs from actual DOM element color/backgroundColor pairs WITHOUT a cap (a11y-extract.ts:265-304). So §9 a11y contrast data is NOT capped — it comes from extractA11y. The cap affects `colorRelationships.contrastPairs` which informs §2 Color Palette color relationship data. Real text-on-bg pairs involving lower-frequency colors are missed in the colorRelationships view.
   - Recommendation: Increase the cap or change the filter. Either raise `.slice(0, 20)` to `.slice(0, 50)` or filter by `frequency >= 2` instead of `>= 5`. Also consider using the actual DOM-derived pairs (from extractA11y) for colorRelationships instead of the synthetic pairwise computation.
   - Evidence: cluster.ts:799 (slice cap), a11y-extract.ts:265-304 (uncapped DOM pairs), extract.ts:469 (overwrite)
6. **[P2] B6: Gradients stored as raw strings — no structural decomposition** — Confirmed. `types.ts:124-130` GradientInfo stores `value` as a raw CSS string (e.g. `linear-gradient(135deg, #667eea 0%, #764ba2 100%)`). `cluster.ts:1449-1483` aggregates gradients with `type`, `value`, `elementTag`, `location` — no parsing of angle, stop positions, or stop colors into structured fields. §6 Depth & Elevation 'Decorative Depth' subsection and §2 gradient endpoint colors require parsed data. The AI writer must regex-parse raw gradient strings to extract angles and stops, enabling angle/stop fabrication.
   - Recommendation: Add a gradient parser to cluster.ts that decomposes `value` into `{ angle: number, stops: { color: string, position: number }[] }`. Extend `GradientInfo` type in types.ts with structured fields. Use the same color parser (`parseColor`) already available in cluster.ts.
   - Evidence: types.ts:124-130 (raw value field), cluster.ts:1476-1481 (no decomposition)
7. **[P1] NEW: §0/§1/§7/§8 have zero extraction tokens — pure interpretation** — Four required sections have NO backing token fields: §0 Brand Context (identity, audience, personality), §1 Visual Theme & Atmosphere (design principles, comparative framing), §7 Content & Voice (tone, capitalization, button labels, error copy), §8 Do's and Don'ts (implementation rules). The extraction provides raw CSS values but zero semantic/interpretive data for these sections. The AI must invent ALL prose, adjectives, tone descriptors, and rules from raw hex/px/font tokens. These sections are the highest fabrication risk after §11 because they require subjective interpretation with no data guardrails.
   - Recommendation: For §7: Extract button label text from DOM (`textContent` already collected per element in ElementStyle). Extract heading text for tone analysis. For §0: Extract `<meta description>` and `<title>` from crawled pages. For §1/§8: These are inherently interpretive — add a 'confidence annotation' requirement so the AI marks low-confidence claims. Consider making §8 optional or data-derived (auto-generate Do/Don't from detected patterns).
   - Evidence: design_md_format.md §0 (lines 74-101), §1 (104-147), §7 (647-726), §8 (729-762) — no corresponding token fields in types.ts DesignTokens
8. **[P1] NEW: §11 State Matrix — global states not linked to components** — Even when interaction capture IS enabled, `interaction-capture.ts:394-690` detects loading/empty/error states as GLOBAL page patterns (class-based and ARIA-based), not per-component. `InteractionData.loadingStates/emptyStates/errorStates` are flat arrays with no component-type binding. `cluster.ts:1283-1303` links interaction CAPTURES to components by tag+classes, but loading/empty/error states are not linked. §11 requires a component×state matrix (Button×Loading, Card×Empty, etc.) — the AI must fabricate which global state pattern maps to which component.
   - Recommendation: During `detectLoadingStates`/`detectEmptyStates`/`detectErrorStates`, also capture the parent component context (nearest interactive ancestor's tag/role). Add a `componentType` field to `LoadingStateInfo`/`EmptyStateInfo`/`ErrorStateInfo`. Or post-hoc in cluster.ts: match state detection selectors against identified component elements.
   - Evidence: interaction-capture.ts:394-487 (global detection), types.ts:177-199 (no component binding), cluster.ts:1283-1303 (capture-to-component linking exists for InteractionCapture but not for state detections)
9. **[P2] NEW: §10 Responsive collapsing strategy has no extraction backing** — Breakpoints are extracted from CSS media queries across all pages (OK). But §10 'Collapsing Strategy' requires observing how specific elements change size/layout across breakpoints (e.g. 'Hero headline: 56px → 44px at 1024px'). The extraction collects elements at a single viewport (1440×900, extract.ts:298) per page — no multi-viewport element comparison. The AI must fabricate all pixel-transition claims.
   - Recommendation: Run DOM collection at 2-3 viewport sizes (1440, 768, 375) per page and diff element rect/fontSize across viewports. This is expensive but the only way to produce real collapsing data. Alternatively, parse CSS media query rules for responsive font-size/padding declarations and map them to element selectors.
   - Evidence: extract.ts:298 (single viewport 1440×900), design_md_format.md §10 lines 866-871 (requires exact pixel transitions)
10. **[P2] NEW: §12 Iconography — icon-to-text alignment not extracted** — Icon detection (icon-detect.ts) provides library, sizeScale, strokeWidth, totalCount, colorMode. But §12 requires 'Icon-to-Text Alignment' (gap values, vertical centering method). No extraction of icon-adjacent-element spacing exists. The AI fabricates alignment claims from DOM structure alone.
   - Recommendation: Low priority. Icon-to-text gap can be inferred from DOM element rects (icon element + adjacent text element gap). Add a post-processing step in cluster.ts or icon-detect.ts that finds SVG elements with adjacent text siblings and computes the gap.
   - Evidence: design_md_format.md §12 lines 952-958 (alignment requirement), icon-detect.ts (no adjacency analysis)

## Summary
Three P0 gaps confirmed: (1) interaction capture is off by default, starving §11 State Matrix, §4 component states, and §9 focus data entirely; (2) extractA11yAsync is dead code, leaving 5 §9 fields (tabOrder, langAttribute, skipLink, altText, reducedMotion) permanently null; (3) cssAnalyses is not passed to the sync extractA11y call, so reducedMotionSupport is always undefined despite the helper existing. The core mechanism: extraction produces structured data for ~10 of 17 sections; the AI must invent content for the rest, and the prompt template's cardinal rules ('every value must come from tokens') conflict with the absence of tokens for those sections.
