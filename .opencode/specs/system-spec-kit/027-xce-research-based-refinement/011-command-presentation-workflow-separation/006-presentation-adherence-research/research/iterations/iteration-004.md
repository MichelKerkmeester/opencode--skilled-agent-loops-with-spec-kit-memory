# Iteration 004 (mimo-v2.5-pro)

**Summary:** The /memory:search render-contract failure correlates with two compounding factors: its primary render template is buried under 25 lines of preamble (vs. 8-18 in succeeding commands) and its command .md uses a weaker 'read the presentation asset' instruction instead of the doctor family's stronger 'Load the presentation contract before showing...' phrasing. Additionally, search_presentation.md has the highest template density (7 competing fenced blocks in 243 lines), and there is no explicit marker convention to signal which template block is primary — a gap that will affect the deep/* family (328-463 lines) even more severely when dispatched to mid-tier models.

## [P1][REFINEMENT] Primary render template buried under 25 lines of preamble in the failing command
- Evidence: search_presentation.md:26 — the retrieval result display template (the one /memory:search ignored) starts at line 26 after 25 lines of title, description, and startup-question policy. By contrast, doctor_speckit_presentation.md:9 places its router menu at line 9 with only 8 lines of preamble.
- Detail: The command that failed (/memory:search) has the highest preamble-to-template ratio among the memory family: 25 lines before the first render template vs. 18 for save, 13 for manage, and 17 for learn. Mid-tier models appear to lose adherence to templates that are not the first code-fence block in the asset.
- Fix sketch: Move the retrieval result display template (section 2) to immediately after the title block, before the startup-question policy, so the primary render contract is the first fenced block a model encounters.

## [P1][REFINEMENT] Weaker pre-read instruction in the failing command vs. doctor commands
- Evidence: search.md:18 says 'Before asking startup questions or displaying results, read the presentation asset and use it as the display source of truth.' doctor_speckit.md:16 says 'Load the presentation contract before showing startup questions, setup dashboards, approval prompts, diagnostic dashboards, result summaries, or next-step text.' The doctor phrasing uses 'Load' (implying full ingestion) and enumerates every surface; the search phrasing uses 'read' and is shorter.
- Detail: Every doctor command (speckit, mcp, update) uses the stronger 'Load the presentation contract before showing...' phrasing and all rendered correctly. The memory family uses the weaker 'Before rendering... read the presentation asset' phrasing. The word 'Load' paired with exhaustive surface enumeration may signal higher priority to mid-tier models.
- Fix sketch: Standardize all command .md files to use the doctor-style 'Load the presentation contract before showing <exhaustive-surface-list>.' phrasing.

## [P2][DOC-DRIFT] search_presentation.md has 7 template sections — highest template density dilutes primary render signal
- Evidence: search_presentation.md has 7 distinct fenced template blocks across 243 lines (startup, retrieval, empty-result, analysis-overview, plus 6 analysis sub-displays). save_presentation.md has 5 blocks in 89 lines. manage_presentation.md has 13 blocks in 227 lines. The high block count in search means the model sees multiple competing output formats.
- Detail: When a mid-tier model reads a presentation asset with many fenced template blocks, it may pattern-match to any block rather than the one relevant to the current dispatch path. /memory:search's retrieval template (section 2) competes with the analysis-overview template (section 5) and six analysis sub-templates (section 6).
- Fix sketch: Split search_presentation.md into a primary retrieval-contract file (sections 1-4) and a separate analysis-contract file (sections 5-7), with the command router loading only the relevant file per dispatch path.

## [P2][REFINEMENT] deep/* assets are 328-463 lines — highest risk of template non-adherence for mid-tier models
- Evidence: deep_start-model-benchmark-loop_presentation.md:463, deep_start-agent-improvement-loop_presentation.md:461, deep_start-review-loop_presentation.md:432, deep_start-context-loop_presentation.md:390. The consolidated setup prompt template sits at lines 62-181 (benchmark), 58-173 (agent-improvement), 81-209 (review), 62-180 (context) — each preceded by 60-80+ lines of auto-resolution tables and PRE-BOUND schema blocks.
- Detail: These assets were not tested in the reported live tests, but they share the same pattern that broke /memory:search: long preamble before the primary render template. The deep/* family has the largest assets (328-463 lines) and the deepest template placement (60-230 lines of preamble). They are at high risk of the same adherence failure when dispatched to mid-tier models.
- Fix sketch: For deep/* assets, extract the consolidated setup prompt into a top-level section that appears before the PRE-BOUND schema and auto-resolution reference material.

## [P2][NEW-FEATURE] No explicit 'primary template' marker convention in presentation assets
- Evidence: Across all 24 presentation assets, no file uses a marker like 'PRIMARY RENDER TEMPLATE' or '## 0. Render Contract' to signal which fenced block is the one the model must honor. The model must infer priority from document order and section numbering.
- Detail: Mid-tier models have no explicit signal for which template block is the primary render contract vs. auxiliary reference templates. Adding a machine-parseable marker (e.g., a YAML frontmatter field `primary_template_section: 2` or a bold label before the first relevant fenced block) would give models an unambiguous anchor.
- Fix sketch: Add a `<!-- PRIMARY RENDER TEMPLATE -->` HTML comment or a YAML frontmatter `primary_template_line` field to each presentation asset marking the exact fenced block the model must honor for the default dispatch path.

## [P3][REFINEMENT] save_presentation.md at 89 lines succeeded — size ceiling may be ~150 lines for reliable mid-tier adherence
- Evidence: save_presentation.md:89 lines — rendered correctly. manage_presentation.md:227 lines — rendered correctly. learn_presentation.md:166 lines — rendered correctly. search_presentation.md:243 lines — failed. The doctor assets (195-205 lines) rendered correctly but have stronger pre-read instructions.
- Detail: The memory family's three succeeding assets (89, 227, 166 lines) all rendered correctly with the same weak pre-read instruction as search. The failing asset at 243 lines is the largest in the family. Combined with doctor's success at 195-205 lines with stronger instructions, the effective size ceiling for reliable adherence without stronger loading cues appears to be approximately 150-200 lines for mid-tier models.
- Fix sketch: Target presentation assets under 200 lines; split assets exceeding that threshold into per-dispatch-path sub-files loaded by the command router.
