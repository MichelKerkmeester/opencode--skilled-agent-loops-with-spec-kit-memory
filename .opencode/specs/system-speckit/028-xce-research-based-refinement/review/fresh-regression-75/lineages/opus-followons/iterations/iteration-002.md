# Seat opus-followons — iteration 2 (global #37)

- Model: `claude-opus-4-8` · Executor: claude · Angle: opus-followons
- Verdict: **CONDITIONAL** · Findings: P0=0 P1=0 P2=4

## Analysis

I've traced the full idempotency slice: the library (`idempotency-receipts.ts`), both call sites (`memory-save.ts`, `memory-crud-update.ts`), the schema/table (`vector-index-schema.ts`), and the only prune caller (`context-server.ts`). Here is my analysis on the assigned angle — enablement gating, receipt-key variance, force-retry-conflict, and receipt TTL.

## What's solid
- **Flag gating fails safe.** Both handlers gate on `isMemoryIdempotencyEnabled()` (default-OFF) and wrap the entire lookup in try/catch that resets `idempotencyKey = null` on any error (missing table, null db). A non-migrated DB or null database degrades to normal save, never crashes. (`memory-save.ts:3547,3629-3633`; `memory-crud-update.ts:225,285-288`)
- **`force` is in the key material**, so a `force`-flipped retry derives a *different* receipt key (a miss) rather than colliding. (`memory-save.ts:3562-3566`)
- **Stale-replay safety via active-row recheck**, not just TTL: a replay is honored only when the live row's `content_hash` still matches the receipt's. (`memory-save.ts:3584-3611`; `memory-crud-update.ts:262-267`)

## Findings

**1 — Conflict detection is structurally unreachable (force-retry-conflict angle).** Both call sites pass `payload` identical to `requestFingerprint` (`memory-save.ts:3570-3571` — same `idempotencySemantics` object; `memory-crud-update.ts:239-253` — structurally identical literals). Therefore `payloadHash === requestFingerprintHash` always. Since `receiptKey` is derived *from* `requestFingerprintHash` (`idempotency-receipts.ts:88-97`), any receipt-key match guarantees a payload-hash match — so the `conflict` branch (`idempotency-receipts.ts:136-137`) can never fire through these paths, and `markResponseWithReceiptStoreConflict` (store-race winner is always `'replay'`) is dead too. The comment at `memory-save.ts:3562-3565` justifying `force` in the key ("rejected as a conflict") describes an impossible event — a force-flip yields a *miss*, never a conflict. Fails safe, but the advertised "conflict protection" provides zero protection; operators enabling the flag may believe otherwise.

**2 — Stale-id replay via the planner-save bypass.** The replay active-row check verifies `content_hash` only (`memory-save.ts:3585,3609`) and never compares the live row's *id* against the id embedded in the cached response. Canonical/planner saves (`shouldPlanCanonicalSave`) skip the idempotency block entirely (lookup *and* store — the planner branch returns at `memory-save.ts:3636-3644` before the store at 3756). So if a memory is hard-deleted (FK nulls `receipt.memory_id` but the JSON `response_payload` keeps the old id, `vector-index-schema.ts:668`) and the same content is re-created through the planner path with a new id, the stale `receipt_A` is never refreshed. A later direct save then matches `content_hash`, honors the replay, and returns a response reporting the **deleted id**. Narrow (needs flag-ON + mixed planner/direct saves + delete within TTL) but a genuine correctness gap to close before enablement.

**3 — TTL enforced only at daemon boot.** `pruneExpiredIdempotencyReceipts` has exactly one caller, at init (`context-server.ts:1824`). These are warm, long-lived daemons; between restarts receipts grow unbounded and rows past the 30-day TTL are never swept while running. The active-row recheck contains the *staleness* risk, so impact is bounded to disk growth + a weaker-than-it-looks TTL guarantee — worth noting before flag-ON.

**4 — Prune forfeits its index (minor).** The only index is on `(operation, created_at DESC)` (`vector-index-schema.ts:630-631`) but prune filters `WHERE updated_at < ...` (`idempotency-receipts.ts:267`). Since no code path ever issues an `UPDATE` (`storeIdempotencyReceipt` uses `ON CONFLICT DO NOTHING`), `updated_at` is immutable and always equals `created_at`; pruning on the indexed `created_at` would be behaviorally identical and indexed.

No P0/P1-with-current-impact (feature is default-OFF and every failure path degrades to plain save). Verdict CONDITIONAL on finding 2 being addressed before the flag is turned on.
