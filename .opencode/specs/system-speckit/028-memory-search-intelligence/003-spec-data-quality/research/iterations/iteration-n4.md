# Iteration N4 - Metadata Enrichment Fields (opus, schema-slot design)

## TITLE

Cohort N4 metadata-artifact angle. Model opus. Angle designs the two new per-doc metadata field families the brief recommends, grounded against the real `031/description.json` and the `generatePerFolderDescription` writer: (a) semantic-intent plus content-type tags plus answerable-questions per arXiv 2512.05411, and (b) temporal and freshness governance fields per the RAG-freshness source. Each field gets an exact schema slot, an on-write population path, a retroactive backfill path, the reader it serves, and a go or no-go under the truncation law.

## FINDINGS

The description schema already accepts new fields without breaking, but the write payload silently drops anything not on a hardcoded allowlist. `perFolderDescriptionSchema` ends in `.passthrough()` at `description-schema.ts:69` so an unknown key validates rather than throwing, but `pickIncomingAuthoredOptionalFields` at `folder-discovery.ts:216-228` forwards only `title`, `type`, `trigger_phrases`, `path` from a fresh write, and `generatePerFolderDescription` at `:893-903` constructs the object from a fixed nine-field literal. So a new field that is not added to `DESCRIPTION_KNOWN_AUTHORED_OPTIONAL_KEYS` at `description-schema.ts:25-31` AND to the pick allowlist AND to the `PerFolderDescription` type is born unpopulated and erased on the next canonical save. This is the load-bearing wiring fact for both families.

Retroactive backfill survives re-saves through a different door than on-write. `mergeDescription` calls `copyUnknownPassthrough` at `description-merge.ts:62` which preserves every non-reserved EXISTING key. So a field hand-written into an existing `description.json` is durable across `savePerFolderDescription` so long as it is not in `DESCRIPTION_RESERVED_KEYS` (`description-schema.ts:33-38`) and so long as the regenerate-from-scratch branch at `workflow.ts:1673-1689` does not fire, because `genPFD` returns the fixed literal and drops unknowns. The real `028/description.json` proves passthrough is live: it carries a `level` key the live schema never lists, retained across four `memorySequence` bumps. So retroactive backfill is durable by construction, on-write derivation is the part that must be coded.

The fields reach the embedding vector only through one seam. `descriptionMetadataToIndexableText` at `memory-parser.ts:595-607` is the function that turns `description.json` into the embedded text of the `description_metadata` document, and `DescriptionMetadataContent` at `:528-538` is its typed carrier. A field added to `description.json` but not threaded through `parseDescriptionMetadataContent` at `:540-593` and this builder is invisible to retrieval, it sits in the file as governance-only data the reader never embeds. This splits the two families cleanly: family (a) intent and answerable-questions wants to reach this builder, family (b) temporal governance deliberately should not.

Content-type today is structural not semantic, and lives at the wrong granularity for the brief. `extractDocumentType` at `memory-parser.ts:420-453` already derives a per-FILE type from filename alone, `spec.md` to spec, `plan.md` to plan, and so on. The brief's content-type axis (procedural / conceptual / reference / example) is orthogonal, a single `spec.md` holds conceptual requirements and procedural acceptance steps in different chunks, so a filename map cannot infer it and a per-folder `description.json` slot is too coarse to label a chunk. Keyword derivation is equally mechanical, `extractKeywords` at `folder-discovery.ts:514-535` lowercases and de-stops tokens from the description string with zero semantic intent, so semantic-intent is genuinely net-new signal not a rename of keywords.

The truncation law from 028 governs which of these can move retrieval. The floor is `DEFAULT_MIN_RESULTS = 3` at `confidence-truncation.ts:30-35` and the 028 doctrine at `028/before-vs-after.md:169` is that a feature earns a prod keep only by changing the composition of the truncated top-3, never by adding rows the floor cuts. As X1 established, this is a tax on retrieval candidates only, so write-time and governance fields bypass it by serving a different reader. Family (a) must therefore prove it reorders the top-3, family (b) is explicitly scoped as governance and is a go on the bypass not on a ranking claim, which is also why it does not repeat the 028 bitemporal mistake of being sold as a recall lever.

## CONCRETE CHANGE

Family (a): semantic-intent, content-type, answerable-questions.

- `semantic_intent` string. Slot: a new authored-optional key in `description.json`. One sentence naming what the packet decides or builds, distinct from the descriptive `description`. Register in `DESCRIPTION_KNOWN_AUTHORED_OPTIONAL_KEYS` (`description-schema.ts:25-31`), add to `PerFolderDescription` and to `pickIncomingAuthoredOptionalFields` (`folder-discovery.ts:216-228`), then thread through `DescriptionMetadataContent` (`memory-parser.ts:528-538`) and emit a `Semantic intent: ...` line in `descriptionMetadataToIndexableText` (`:595-607`) so it joins the embedded text.
- `answerable_questions` string array. Slot: same authored-optional path. Three to five questions this packet answers, mined from spec.md requirement headings. Emit as `Answers: q1 | q2 | q3` in the indexable-text builder so it enters the vector as the 2512.05411 metadata side of the fusion.
- `content_type` enum (procedural / conceptual / reference / example). Slot: NOT a `description.json` field, it belongs at chunk granularity. Carry it as a chunk-level tag alongside `anchor_id` and `chunk_label` in the chunking-orchestrator seam R1 mapped at `chunking-orchestrator.ts:315,347`, inferred per anchor section, and prepend it to the deterministic breadcrumb X1's rank-3 change already prepends so it rides the existing re-index rather than a new pass.

Family (b): temporal and freshness governance.

- `last_verified` ISO string. Slot: new authored-optional key. Set by a human or a verify step when a packet's claims are re-confirmed, distinct from the machine-written `lastUpdated` at `workflow.ts:1719` which only marks the last save.
- `shelf_life_class` enum (durable / volatile / pinned). Slot: authored-optional. Declares how fast the doc rots, durable for decision records, volatile for status-bearing docs, pinned for constitution-class content.
- `stale_after` ISO string. Slot: authored-optional, derived as `last_verified` plus a class-driven interval. Governance only.
- Wiring: register all three as authored-optional keys and in the pick allowlist so on-write survives, but deliberately do NOT thread them into `descriptionMetadataToIndexableText`. They are read by the resume ladder, the freshness consumer that already reads `lastUpdated` per the note at `workflow.ts:1714-1718`, and a future staleness sweep, not by the embedder.

- Schema accepts unknowns, payload drops them: `description-schema.ts:69` passthrough, `:25-38` reserved and known-optional key lists, `folder-discovery.ts:216-228` pick allowlist, `:893-903` fixed-literal generator.
- Retroactive durability via passthrough: `description-merge.ts:33-44,62`, reserved-key gate `description-schema.ts:33-38`. Live proof of a non-schema `level` key surviving four sequence bumps: `028-memory-search-intelligence/description.json`.
- The single embed seam: `memory-parser.ts:528-538` carrier, `:540-593` parser, `:595-607` indexable-text builder.
- Content-type is structural and per-file today: `memory-parser.ts:420-453`. Chunk-label seam for a per-chunk tag: `chunking-orchestrator.ts:315,347`. Keyword derivation is mechanical: `folder-discovery.ts:514-535`.
- Real target file lacks every proposed field: `003-spec-data-quality/description.json:1-23` (level, specFolder, description, keywords, lastUpdated, specId, folderSlug, parentChain, memorySequence, memoryNameHistory only).
- `lastUpdated` already machine-bumped on every save, the freshness hook to extend: `workflow.ts:1714-1720`.
- Truncation law: `confidence-truncation.ts:30-35`, doctrine `028/before-vs-after.md:169`. Brief sources: metadata fusion at 82.5 vs 73.3 precision `stage-0-external-findings.md:12`, intent and answerable-questions tag `:42`, temporal block `:44-46`, candidate 10 freshness `:99`.

## READER

Family (a) targets retrieval. `semantic_intent` and `answerable_questions` reach the embedded `description_metadata` text, so they are the 2512.05411 metadata side of a content-plus-metadata fusion and CAN reorder the top-3 of a broad intent query, the only retrieval class that survives the floor. `content_type` serves retrieval secondarily as a re-rank or filter key but only if a downstream consumer reads it, otherwise it is a floor-mirage exactly like the deleted 028 lanes. Family (b) targets logic and governance, the resume ladder and a staleness sweep read it to decide what to re-verify, it never touches a retrieval candidate so it bypasses the floor and is not a ranking claim.

## ON-WRITE OR RETROACTIVE

Family (a) is on-write primary. `semantic_intent` and `answerable_questions` should be derived in `generatePerFolderDescription` from spec.md (the description and requirement headings the function already reads at `folder-discovery.ts:864-872`), so every new packet earns them by default, with a retroactive corpus pass for existing packets since passthrough makes a hand-written value durable. `content_type` is retroactive by necessity, it only takes effect after the re-index behind the coverage gate that X1's breadcrumb change also waits on. Family (b) is on-write for the machine-derivable part, `stale_after` computes from `last_verified` plus the class interval, but `last_verified` and `shelf_life_class` need a human or a verify-step author, so the retroactive backfill should default `shelf_life_class` by document type (decision_record durable, status docs volatile) and leave `last_verified` null until a real verification happens, never fabricated.

## RISK

Family (a)'s retrieval win is a hypothesis until measured on the prod path, the brief's 82.5-vs-73.3 precision is a recall-style metric the floor hides, so it must be confirmed by a dual-mode eval read that the prod-mode number moves, same caveat R1 and X1 raised for the prefix and fusion changes. Auto-deriving `semantic_intent` and `answerable_questions` from spec.md headings risks restating the existing `description` and adding embedding noise rather than orthogonal signal, the fields earn their slot only if they carry intent the description does not, an LLM-judged pass would help but adds a dependency and cost. The biggest correctness hazard is the wiring asymmetry, a field added to the schema and the file but missed in `pickIncomingAuthoredOptionalFields` is silently erased on the next save and a field missed in `descriptionMetadataToIndexableText` is silently absent from the vector, both fail green, so the change needs a round-trip test asserting each new key survives a save and appears or is absent in the indexable text as intended. Family (b)'s only honest framing is governance, if it is ever sold as a retrieval ranker it repeats the 028 bitemporal deletion, and a fabricated `last_verified` is worse than a null one because it makes stale content look fresh.
