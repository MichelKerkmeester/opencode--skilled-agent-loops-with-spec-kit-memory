# Iteration 9: Deterministic Ranking Core and Oracle Parity

## Focus

Pressure-test deterministic ranking kernels, reusable pure-core boundaries, and TypeScript-oracle parity across napi-rs, WASM, and sidecar adapters. Decide whether deterministic RRF, hashing, and ranking constitute a standalone Rust opportunity or only supporting infrastructure for a material feature.

## Actions Taken

1. Read the externalized state, reducer-owned strategy, prior iteration, and iteration contract.
2. Traced the styles database's structured, FTS, vector, and weighted-RRF ranking paths, including candidate bounds and deterministic tie-breaks.
3. Audited the existing retrieval tests as a prospective TypeScript oracle.
4. Compared the repository's pure-core, adapter, numeric, serialization, fallback, and differential-parity rules for napi-rs, WASM, and sidecars.

## Findings

### F1. Weighted RRF is bounded orchestration, not a material standalone Rust kernel

The fusion loop consumes already-ranked channel lists, adds one reciprocal-rank contribution per result, and sorts by fused score with ID as the terminal key. The request contract caps each lane at 200 candidates and defaults to 50; with the current three lanes, fusion processes at most about 600 contributions regardless of whether the corpus contains 1,290, 12,900, or 129,000 bundles. Corpus growth can make eligibility, vector search, and lane construction material, but it does not make this bounded RRF loop material by itself. [SOURCE: .opencode/skills/sk-design/styles/_db/retrieval.mjs:23-24] [SOURCE: .opencode/skills/sk-design/styles/_db/retrieval.mjs:252-283] [SOURCE: .opencode/skills/sk-design/styles/_db/retrieval.mjs:361-408]

Rust ownership of weighted RRF would therefore be a forbidden like-for-like rewrite. The deterministic kernel becomes useful only as a private component inside an independently justified feature, such as a filter-aware ANN query that returns candidate lists or a resident multimodal worker that also performs a measured batch-ranking stage. Even then, TypeScript should continue to own profile selection, channel degradation, fallback, cursors, and public response assembly.

### F2. A reusable pure core is an implementation boundary, not a cross-system product

Repository standards deliberately place measured ranking, hashing, numeric operations, and validation in a transport-neutral core while leaving transport, flags, fallback, and routing in TypeScript. That shape permits one qualified core to sit behind separate napi-rs, WASM, or sidecar adapters without duplicating parity-sensitive algorithms. It does not establish that the core should exist before a measured Rust workload does. [SOURCE: .opencode/skills/sk-code/code-opencode/references/rust/style-guide/interop-model.md:33-52] [SOURCE: .opencode/skills/sk-code/code-opencode/references/rust/style-guide/interop-model.md:112-123]

The prior cross-system result remains intact: styles retrieval and Spec Kit Memory have different ranking policy, persistence, publication, and fallback contracts, while system-code-graph has no vector workload. A shared generic ranking platform would move policy into a lowest-common-denominator ABI and create coordinated release ownership without adding capability. Reuse should be limited to transport-neutral primitives inside a component already warranted by local inference, ANN, or measured build preprocessing; subsystem-specific scoring and fusion profiles remain local.

### F3. The current retrieval test is a useful behavioral seed but not a release-grade parity oracle

The existing weighted-RRF test confirms rank-only behavior, attribution keys, and deterministic ID ordering for two channels and two IDs, using semantic `deepEqual`. It does not pin serialized bytes, channel insertion-order permutations, duplicate IDs within a lane, zero or invalid weights, complete ties, Unicode ID ordering, negative zero, number spelling, omitted fields, error shapes, or all three channels. [SOURCE: .opencode/skills/sk-design/styles/_db/__tests__/retrieval.test.mjs:18-31]

Repository Rust rules require fixtures generated by the pinned TypeScript implementation and byte-for-byte comparisons; semantic equality is explicitly insufficient because JavaScript and Rust may differ in floating-point reduction order, negative zero, number spelling, object key order, and map iteration. [SOURCE: .opencode/skills/sk-code/code-opencode/references/rust/style-guide/interop-errors-and-parity.md:152-188] [SOURCE: .opencode/skills/sk-code/code-opencode/references/rust/style-guide/interop-errors-and-parity.md:271-312]

Before any Rust prototype, freeze the supported domain and generate oracle fixtures from TypeScript. Preserve lane iteration order and sequential floating-point accumulation exactly; do not parallelize parity-visible reductions. Require a complete terminal comparator key and canonical boundary serialization. This oracle investment is reusable supporting infrastructure, but it is not independently a reason to ship Rust.

### F4. Adapter choice follows residency and failure isolation; ranking does not choose it

For a Node-only, in-process, CPU-bound kernel that demonstrably blocks the event loop, napi-rs is the lowest-friction adapter and can use an async task after measurement. For browser portability, WASM is viable, but the styles database has no browser execution requirement and baseline WASM is single-threaded; Promise bridging does not create CPU parallelism. For a resident model or large ANN process, a sidecar can justify IPC through crash containment, independent RSS lifecycle, model residency, and potentially multi-client service. [SOURCE: .opencode/skills/sk-code/code-opencode/references/rust/style-guide/interop-errors-and-parity.md:190-213] [SOURCE: .opencode/skills/sk-code/code-opencode/references/rust/style-guide/interop-model.md:160-178]

None of those advantages applies to at most 600 RRF contributions. A sidecar would add framing and process latency, WASM would add packaging without required portability or parallelism, and napi-rs would add native release complexity without a material compute win. If ANN or inference later clears its own gate, its residency determines the adapter and the bounded ranking primitive may travel inside that already-funded boundary.

### F5. Deterministic hashing and ranking are necessary parity infrastructure for material features, not separate opportunities

The durable SHA-256 identities and comparator chains protect cache, generation, cursor, and output contracts. Rust must reproduce the exact TypeScript preimage, field order, separators, UTF-8 encoding, hash algorithm, hexadecimal form, numeric reduction order, and terminal tie-break; a semantically similar implementation is not enough. [SOURCE: .opencode/skills/sk-code/code-opencode/references/rust/style-guide/interop-errors-and-parity.md:127-150] [SOURCE: .opencode/skills/sk-design/styles/_db/retrieval.mjs:218-249]

This makes deterministic primitives a mandatory acceptance layer for any future Rust ANN, inference, or build accelerator. It does not make them a user-visible feature and does not clear residency or materiality at current, 10x, or 100x corpus scale. The scale gates must stay attached to the unbounded or resident work: eligible-vector search, model execution, image preprocessing, or cold-build verification and parsing. Candidate-bounded fusion and native SHA-256 remain TypeScript/Node baselines unless bundled with such a feature and proven byte-identical.

## Questions Answered

- **Is deterministic RRF, hashing, or ranking a standalone Rust opportunity?** No. RRF is capped independently of corpus size, hashing is already native, and deterministic primitives add no user capability by themselves.
- **Is a reusable pure core valuable?** Yes, but only as an internal boundary for an independently material Rust component. It is not justification for a shared cross-system search platform.
- **Which adapter should host a deterministic kernel?** The owner workload decides: napi-rs for measured in-process Node CPU work, WASM only for a real portability requirement, and a sidecar for resident-model or ANN isolation. Ranking alone warrants none.
- **What parity work is required?** A pinned TypeScript fixture generator, byte-level expected outputs, exhaustive numeric/order/error cases, sequential parity-visible reductions, and adapter-specific differential tests.
- **Which scale gate clears?** None for standalone fusion or hashing at current, 10x, or 100x. Only the surrounding unbounded or resident feature can clear a gate.

## Ruled Out Directions

- Standalone Rust weighted-RRF package: the kernel is capped at about 600 contributions and adds no capability.
- Cross-system generic ranking core: subsystem policies and ownership contracts are intentionally different.
- Sidecar for bounded ranking: process framing and lifecycle cost exceed the work being isolated.
- WASM as the default styles adapter: there is no browser portability requirement and baseline WASM adds no CPU parallelism.
- Treating semantic deep equality as oracle parity: release equivalence requires TypeScript-authored byte fixtures.
