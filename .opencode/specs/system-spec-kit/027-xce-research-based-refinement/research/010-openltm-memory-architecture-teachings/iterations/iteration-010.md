## Mechanism
OpenLTM’s `learn()` idempotency is normalized-text keyed, not a verified hash: `normalizeKey()` lowercases, strips punctuation, collapses whitespace, trims, and truncates to 200 chars, while `memories.dedup_key` is `UNIQUE` (`packages/openltm-core/src/dedup.ts:5`, `packages/openltm-core/src/dedup.ts:6`, `packages/openltm-core/src/schema.sql:45`). `learn()` scrubs secrets before dedupe, computes that key, then looks up duplicates by `WHERE dedup_key=?` only, with no category/project/workspace predicate (`packages/openltm-core/src/db.ts:307`, `packages/openltm-core/src/db.ts:314`, `packages/openltm-core/src/db.ts:317`).

On duplicate, OpenLTM reinforces the existing row: `confirm_count=confirm_count+1`, `last_confirmed_at=datetime('now')`, and `confidence=MIN(1.0, confidence+0.05)`, then returns `{ action: "reinforced", id, confirm_count }` (`packages/openltm-core/src/db.ts:321`, `packages/openltm-core/src/db.ts:324`, `packages/openltm-core/src/db.ts:347`). On first insert, it stores content/title/category/defaults, writes `dedup_key`, and returns `{ action: "created", id, confirm_count: 1 }` (`packages/openltm-core/src/db.ts:350`, `packages/openltm-core/src/db.ts:352`, `packages/openltm-core/src/db.ts:414`). Reinforcement affects lifecycle because decay never deprecates memories with `confirm_count>=5` (`packages/openltm-core/src/db.ts:246`, `packages/openltm-core/src/db.ts:258`).

There are two queue concepts. Embedding generation uses a durable Honker queue named `ltm-embeddings` with `EMBED_MAX_ATTEMPTS = 5`; `learn()` enqueues an embedding job and falls back to inline embedding if enqueue returns `null` (`packages/openltm-core/src/queue/index.ts:12`, `packages/openltm-core/src/queue/index.ts:16`, `packages/openltm-core/src/db.ts:396`, `packages/openltm-core/src/db.ts:401`). The worker retries failures with exponential backoff from 5 seconds and acks on success (`packages/openltm-core/src/queue/worker.ts:25`, `packages/openltm-core/src/queue/worker.ts:55`, `packages/openltm-core/src/queue/worker.ts:60`). Separate write serialization is `WriteQueue`: a promise-tail queue wrapping DB writes in `BEGIN IMMEDIATE` / `COMMIT`, rolling back on error (`packages/openltm-core/src/lib/writeQueue.ts:7`, `packages/openltm-core/src/lib/writeQueue.ts:13`, `packages/openltm-core/src/lib/writeQueue.ts:16`, `packages/openltm-core/src/lib/writeQueue.ts:19`). Tests assert transaction wrapping, rollback, no interleaving, and recovery after rollback (`src/__tests__/writeQueue.test.ts:24`, `src/__tests__/writeQueue.test.ts:31`, `src/__tests__/writeQueue.test.ts:40`, `src/__tests__/writeQueue.test.ts:53`).

Cross-runtime write access is adapter-specific today. OpenCode exposes `ltm_learn` and passes actor `opencode:ltm_learn` into core `learn()` (`packages/adapter-opencode/src/tools.ts:30`, `packages/adapter-opencode/src/tools.ts:44`, `packages/adapter-opencode/src/tools.ts:50`). Pi has a direct legacy `ltm_learn` path with actor `pi:ltm_learn`, while its newer bridge discovers MCP tools through `tools/list` and proxies `tools/call` (`packages/adapter-pi/src/tools.ts:42`, `packages/adapter-pi/src/tools.ts:61`, `packages/adapter-pi/src/tools.ts:67`, `packages/adapter-pi/src/index.ts:207`, `packages/adapter-pi/src/index.ts:211`, `packages/adapter-pi/src/index.ts:220`). The stable cross-plugin contract is roadmap, not fully current: Phase 5 says “Other plugins read/write memories through a versioned interface,” with “versioned MCP API + capability discovery” and exit criteria that another plugin can `recall+learn` without coupling (`docs/internal/ROADMAP.md:145`, `docs/internal/ROADMAP.md:147`, `docs/internal/ROADMAP.md:151`, `docs/internal/ROADMAP.md:157`). PRD frames the same goal as a stable API so other plugins can write with category/source tags (`docs/internal/PRD.md:391`, `docs/internal/PRD.md:392`, `docs/internal/PRD.md:393`).

## Negative knowledge
- Do not copy the 200-char normalized-text dedupe key as-is; it is not a hash and can over-merge across project/category boundaries.
- Do not copy adapter-specific duplicated tool definitions; OpenLTM itself shows drift between OpenCode’s 5 tools and Pi’s 3 direct tools.
- Do not replace system-spec-kit’s FSRS/causal-edge model with `confirm_count` thresholds; use reinforcement only as an input signal.
- Do not expose the full maintenance/admin MCP surface as the cross-plugin write contract; plugins need a small, versioned write/read subset.

## Open questions
- Should system-spec-kit’s idempotency key include `specFolder`, `contextType`, normalized content hash, and provenance actor, or should reinforcement intentionally cross folders?
- Should reinforcement update FSRS scheduling directly, or only create a validation event consumed by existing reducers?
- Should the stable write contract be MCP-only, or also a local CLI/JSON schema for non-MCP hooks?
