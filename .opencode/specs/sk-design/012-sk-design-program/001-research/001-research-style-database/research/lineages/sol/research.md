## Iteration 1 — Current-state baseline

The current corpus has 1,290 live style directories and manifest records, despite the research topic's 1,291 label. Its retrieval manifest is a strict 5.89 MB denormalized JSON projection keyed by UUID-first identity with slug retained as a unique filesystem locator. It includes derived facets, token-axis counts, capabilities, section pointers, provenance, artifact hashes, hydration estimates, content hashes, and a whole-corpus generation hash. A local all-eligible query took 6,246.5 ms over 1,290 records. The migration target should therefore preserve the existing eligibility, generation, provenance, rights, containment, and artifact-hash guarantees while moving freshness work to incremental writes and adding persistent lexical, embedding, and relationship indexes. ## Iteration 4 — Crash-safe incremental sync and generation publication

The style indexer should execute an explicit `DISCOVER -> VERIFY -> PARSE_VALIDATE -> COMMIT -> VECTOR_DRAIN -> PUBLISH` lifecycle. Mtime, size, and ctime narrow candidates, but canonical all-artifact hashes remain freshness authority; parse and validation complete before a short `BEGIN IMMEDIATE` transaction atomically replaces normalized source children, upserts the deterministic retrieval document, lets external-content FTS triggers synchronize, and records pending vector work. Success hints advance only after commit, so any failed style naturally retries. [SOURCE: .opencode/skills/sk-design/styles/_engine/manifest.mjs:584-705] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/storage/incremental-index.ts:283-321] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/storage/ports/contention-policy.ts:202-234] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/vector-index-schema.ts:399-432]

Vector work is a rebuildable queue uniquely keyed by style identity, retrieval hash, and embedding profile. Workers use cached embeddings and bounded retry/backoff, then compare the queued hash/profile with current state before atomically writing payload and success status; stale jobs are superseded, while pending or failed vectors never block lexical retrieval. Missing styles first enter a durable suspect quarantine, become tombstones only after an independently separated known-absent confirmation, and are hard-deleted only after retention plus a newer verified rebuild or explicit maintenance checkpoint. [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/providers/retry-manager.ts:652-699] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/providers/retry-manager.ts:793-879] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/handlers/memory-index.ts:1013-1079] [INFERENCE: retrieval-hash/profile comparison prevents stale row-id worker publication]

Each successful sync publishes an immutable corpus-generation row and current pointer in the same relational transaction, so a query binds one SQLite read snapshot and returns generation-stamped cards for guarded hydration. Full rebuilds write generation-addressed staging DB/vector files, validate schema/FTS/count/hash invariants, checkpoint WAL, and atomically switch only new readers; existing readers finish on the old generation. Keeping that prior generation makes rollback a blocked, durable pointer reversal, while partial staging files are disposable and transaction failures leave success markers untouched. ## Iteration 5 — Generation-consistent retrieval contract

The native retrieval API should separate hard eligibility (`requiredFacets`, exclusions, provenance, exact-reuse rights) from structured ranking signals (`axes`, `needs`), then execute structured, persistent FTS5, and vector top-k lanes against one published SQLite generation. Every lane must join the eligible ID set before ranking. Versioned weighted RRF over ranks—not addition of incomparable BM25/cosine/structured raw scores—provides deterministic fusion and per-channel attribution; initial style-specific weights remain neutral, and memory-specific adaptive intent, graph, cognition, and post-fusion stages stay out of scope. [SOURCE: .opencode/skills/sk-design/styles/_engine/eligibility.mjs:15-80] [SOURCE: .opencode/skills/sk-design/styles/_engine/rank-fts.mjs:46-73] [SOURCE: .opencode/skills/system-spec-kit/shared/algorithms/rrf-fusion.ts:445-576]

Responses retain generation hash, degradation mode, eligibility counts, and bounded cards, while the native surface adds query/fusion fingerprints, channel health, source ranks/contributions, a generation-bound keyset cursor, and an opaque hydration token. The cursor pins generation, request/filter hash, fusion profile, candidateK, and the last `(fusedScore,id)` key. Tokens do not grant rights: hydration still validates generation, mode allowlists, exact-reuse rights, containment, artifact hashes, and byte caps. A bounded pressure test confirmed eligibility-first fusion, deterministic UUID tie-breaking, stable continuation, and lexical-only fallback for a stale vector lane. [SOURCE: .opencode/skills/sk-design/styles/_engine/style-library.mjs:136-155] [SOURCE: .opencode/skills/sk-design/styles/_engine/hydrate.mjs:225-370] [SOURCE: command: bounded Node RRF/filter/cursor pressure test, 2026-07-19]

Degradation is channel-local: hybrid; structured+FTS; structured+vector; then structured-only SQL. A missing or invalid authoritative generation fails closed because stale rights cannot be safely ranked. Existing `runQuery`, `runHydrate`, CLI, card fields, and refusal codes remain behind a `legacy|shadow|persistent` adapter; corpus walking is legal only in explicit migration/rollback mode and disappears from the normal persistent path after cutover. Q4 is answered. Next, resolve Q5's source-of-truth, dual-read, cutover, rollback, and repair stages. ## Iteration 7 — Minimum topology and graph promotion boundary

The named council and coverage “graph databases” are SQLite adjacency projections: composite node/edge keys, endpoint foreign keys and indexes, WAL, and transactional batch writes. Their direct query contracts are mostly one-hop predicates, joins, and degree aggregates; only evidence/provenance chains use bounded application BFS, while coverage depth is computed in memory. Graph-shaped relationships therefore do not by themselves justify a dedicated graph engine. [SOURCE: .opencode/skills/system-deep-loop/runtime/lib/council/council-graph-db.ts:119-176] [SOURCE: .opencode/skills/system-deep-loop/runtime/lib/council/council-graph-query.ts:159-227] [SOURCE: .opencode/skills/system-deep-loop/runtime/lib/coverage-graph/coverage-graph-db.ts:234-298] [SOURCE: .opencode/skills/system-deep-loop/runtime/lib/coverage-graph/coverage-graph-query.ts:565-699]

Style sources do contain explained `designSystem.similar` targets, but the current retrieval projection omits them. Across 1,290 canonical styles, a bounded scan found 5,812 labels: 2,269 uniquely name-resolved, 440 ambiguous, and 3,103 unresolved. The immediate need is a normalized, generation-bound relationship table that preserves raw target labels, rationale, resolution state, confidence, and nullable canonical targets—not transitive traversal or a graph service. Shared facets remain relational joins and semantic neighbors remain a vector lane. [SOURCE: .opencode/skills/sk-design/styles/shade/shade-canonical.json:102-118] [SOURCE: .opencode/skills/sk-design/styles/_engine/manifest.mjs:43-73] [SOURCE: command: bounded Node canonical-style relationship inventory, 2026-07-19]

Q1 is answered: ship one published SQLite generation containing normalized style state, FTS5, and indexed relationship edges, with profile-addressed vectors as a rebuildable projection. A future graph may be added only as a disposable, generation-stamped projection when a concrete variable-depth/path or graph-algorithm workload exists, indexed SQLite misses its SLO, graph ranking proves relevance lift, canonical relationship resolution is production-quality, and lag/rebuild/rollback/graph-down operations are drilled.
