## Findings (file:line)

### A. The 5-lane fusion scorer — fusion point + weights
- **F1 — Lane weights live in the lane-registry.** `lib/scorer/lane-registry.ts:8-19` `LANE_DEFINITIONS`: `explicit_author 0.42`, `lexical 0.28`, `graph_causal 0.13`, `derived_generated 0.12`, `semantic_shadow 0.05` — all `live: true`, each with a separate `defaultShadowWeight`. Weights are env-overridable via `SPECKIT_ADVISOR_LANE_WEIGHTS_JSON` (`lane-registry.ts:41-70`, validated `>=0 && <=1`). `weights-config.ts:14-22` re-exports them as `DEFAULT_SCORER_WEIGHTS` / `SCORER_LANES`. [SOURCE: lib/scorer/lane-registry.ts:8-19, 41-70; lib/scorer/weights-config.ts:14-22]
- **F2 — The fusion point is a non-deterministic weighted SUM, not RRF.** `lib/scorer/fusion.ts:359-372`: per skill, each lane produces `weightedScore = rawScore * weights[lane]` (`fusion.ts:366`), and the fused `score` is `contributions.reduce((total, c) => total + c.weightedScore, 0)` (`fusion.ts:372`). This sums RAW per-lane scores scaled by weight — it does NOT fuse by rank. This is the exact "comparable-score" surface Q2 targets. [SOURCE: lib/scorer/fusion.ts:359-372]
- **F3 — Ranking is by summed score with ad-hoc tiebreaks.** `fusion.ts:425-433`: `ranked = recommendations.sort(...)` orders by `(score + commandBonus + intent)` desc, then `confidence` desc, then `skill.localeCompare` (stable id tiebreak). The score field itself is `Number(score.toFixed(6))` (`fusion.ts:409`) — float rounding, not a fixed-order deterministic sum. [SOURCE: lib/scorer/fusion.ts:409, 425-433]
- **F4 — `liveTotal` normalization.** `fusion.ts:343-345` sums live-lane weights (skipping disabled/shadow); `liveNormalized = score / liveTotal` (`fusion.ts:388`) feeds confidence. So the raw weighted sum is also the normalization base. [SOURCE: lib/scorer/fusion.ts:343-345, 388]
- **F5 — There is NO query-class router.** Intent enters today only via two ad-hoc mechanisms, not a class→profile map: (i) `TASK_INTENT` / breadth / multi-concern REGEXES used for confidence + abstention (`fusion.ts:44-56, 484-513`), and (ii) `primaryIntentBonus()` — a long hand-maintained list of per-phrase, per-skill additive score bonuses/penalties applied at sort time (`fusion.ts:259-332, 428-430`). This is the closest current analog to aionforge's per-class profile weights, but it is per-(phrase,skill) hardcoding, not a 5-class router that reweights LANES. [SOURCE: lib/scorer/fusion.ts:44-56, 259-332, 428-430, 484-513]

### B. Skill-graph traversal (graph_causal lane)
- **F6 — BFS over projection edges with edge-type multipliers + depth decay.** `lib/scorer/lanes/graph-causal.ts:21-104`: seeds from explicit+lexical+derived matches (`fusion.ts:196-202`), BFS `maxDepth=2`, `maxBreadth=4` (`graph-causal.ts:27-28`); `EDGE_MULTIPLIER` enhances 0.55 / siblings 0.35 / depends_on 0.35 / prerequisite_for 0.30 / conflicts_with -0.35 (`graph-causal.ts:13-19`); propagation `strength * edge.weight * |mult| * 1/(depth+1)`, pruned `< 0.05` (`graph-causal.ts:70-71`); output clamped to `[-1,1]` (`graph-causal.ts:101`). Negative (conflict) contributions are preserved through emit but not enqueued (`graph-causal.ts:89-103`). [SOURCE: lib/scorer/lanes/graph-causal.ts:13-104]

### E. External anchors (aionforge)
- **F9 — aionforge query-class router: 5 classes → per-signal profile weights.** `retrieval.md:107-143`: `QueryClass { SingleHopFactual, MultiHop, Temporal, Entity, Quote }`; a heuristic regex classifier (most-specific-first) maps each class to a `RetrievalProfile` with per-signal weights from 4 levels (HEAVY 1.0 / MODERATE 0.6 / LIGHT 0.3 / OFF 0.0) plus behavior flags (`graph_expansion`, `exact_rerank`, etc.). Misclassification "degrades gracefully" to a usable ordering; the class is reported in the explanation. `mode_override` lets a caller force a class. [SOURCE: retrieval.md:107-143]
- **F10 — aionforge deterministic RRF (k=60).** `retrieval.md:77-105`: fused score = weighted sum of `1/(k+rank+1)` across ranking signals, default `k=60`; "BM25 and cosine are not comparable, so fusion never tries... fusion reads only the RANK" (`retrieval.md:39-43`). A **zero weight elides the signal entirely** (`retrieval.md:88-90`). Determinism is a HARD requirement: fixed-order sum by `Signal`, tie-break by node id ascending via `total_cmp` (`retrieval.md:100-105`). [SOURCE: retrieval.md:39-43, 77-105]
- **F11 — aionforge graceful degrade (Q6 anchor).** `retrieval.md:295-301`: if the embedder is unreachable the dense signal "returns an empty ranking; it never fails the recall" — degrade not error; reported via `embedder_available: false`. Direct analog for a graph-unavailable graph_causal lane. [SOURCE: retrieval.md:295-301]
- **F12 — aionforge reliability-weighted Beta posterior (Q3 anchor).** `attestation-and-promotion.md:31-66`: each attester contributes reliability `r∈[0,1]` as evidence (`r` toward correct, `1-r` against) over a prior; posterior bounded by attester quality `(alpha+k)/(alpha+beta+k)`; cold-start contributes uninformative `0.5` so "nothing promotes" by default. Bounded, anti-overfit shape. [SOURCE: attestation-and-promotion.md:31-66]

### Q1 hypothesis (query-class routing vs fixed 0.42/0.28/0.13/0.12/0.05)
A query-class router is plausibly a NET IMPROVEMENT and is additive, not a replacement, mirroring aionforge `retrieval.md:107-143`. The current "intent" path is two ad-hoc surfaces — `TASK_INTENT`/breadth regexes (`fusion.ts:44-56`) and the per-phrase `primaryIntentBonus` table (`fusion.ts:259-332`) — which scale poorly (every new disambiguation = a new hardcoded branch). A small class set (e.g. task-write / read-only-explain / deep-loop / breadth-abstain / memory-save) mapping to per-LANE weight multipliers would generalize that table. Injection point: `effectiveScorerWeights()` (`fusion.ts:69-82`) already merges a `laneWeightsOverride`; a router could compute a class→lane-multiplier and feed it there BEFORE the per-skill loop, keeping explicit_author dominant (Known Context: "additive, not a substitute for explicit-author lane"). INFERENCE — needs a held-out routing-quality check; aionforge warns the costly error is the false positive that hurts single-hop precision (`retrieval.md:137-143`), so the analog risk is a misrouted class demoting the right skill.

### Q2 hypothesis (deterministic RRF vs the current weighted sum)
The current fusion IS a raw-score weighted sum (`fusion.ts:372`) over heterogeneous per-lane raw scores: explicit/lexical produce text-overlap magnitudes, graph_causal produces a clamped `[-1,1]` propagation score (`graph-causal.ts:101`), semantic_shadow a cosine. These are NOT on a comparable scale — the exact problem aionforge cites for BM25-vs-cosine (`retrieval.md:39-43`). RRF (`retrieval.md:77-105`) would fuse by RANK within each lane instead, with `k=60`, zero-weight elision, fixed-order sum, node-id tiebreak — directly addressing comparability AND determinism (today's determinism rests on `toFixed(6)` + `localeCompare`, `fusion.ts:409,432`, which is fragile under float non-associativity). FEASIBILITY CAVEAT (inference): each lane already emits a list of `LaneMatch{skillId,score,evidence}` (`buildLaneMatchIndex`, `fusion.ts:100-111`), so producing a per-lane RANKED list is cheap — but graph_causal's NEGATIVE (conflict) contributions have "no rank-fusion meaning" in aionforge (`retrieval.md:90-92`), so a naive RRF port would drop conflict suppression. Need next iteration to confirm how conflict edges must be preserved (re-rank vs separate signal).
