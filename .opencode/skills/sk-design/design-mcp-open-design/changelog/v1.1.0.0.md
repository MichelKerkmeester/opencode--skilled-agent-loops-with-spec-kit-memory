# v1.1.0.0 - Generation Flow Correction

**Released:** 2026-06-14

Corrects the skill to the live-verified reality of how Open Design generates designs. The v1.0.0 release was built from a reverse-engineering pass and described generation as a one-shot `start_run`. A live generation test this session proved that generation is multi-turn, and that `od artifacts create` only adds a file rather than creating a rendered design. This release brings every doc in line with what was proven live, adds the `od run` verb family and the HTTP API surface, and preserves everything the prior release got right.

---

## What's New

### Corrected: generation is multi-turn, not one-shot

The load-bearing fix. A single `start_run` (MCP) or `od run start` (CLI) fires turn 1 only, which returns a GenUI discovery question-form and ends `awaiting_input` with zero files. The design is built only after the form is answered with `od ui respond` (or a follow-up message), which fires the build run that writes `index.html` and gives the project an `entryFile` and a `previewUrl`. Live proof this session: turn 1 produced a question-form with zero files, and after it was answered a build run wrote a 26 KB `index.html` and the project rendered with a `previewUrl`.

- `SKILL.md` Section 3 Run Direction rewritten to the multi-turn flow, with a callout that adding a file is not creating a design. New ALWAYS rule 8 and NEVER rule 5 lock the correction in.
- `references/tool-surface.md` Section 5 rewritten to the multi-turn loop, and the `start_run` description corrected.
- `references/od-cli-reference.md` Section 5 rewritten with the turn-1, answer-the-form, build sequence and the live proof.
- `feature-catalog/04--runs/headless-runs.md`, `feature-catalog/feature-catalog.md`, and the `README.md` run sections updated.
- `manual-testing-playbook/03--gated-runs/gated-verb-confirm.md` and the root playbook RUN-001 scenario now exercise the multi-turn flow, including answering the discovery form, and fail a single turn-1 "finished design" claim.

### Corrected: `od artifacts create` adds a file, it does not create a design

`od artifacts create --name <path> --input <file>` only adds one file to a project (the active one when `--project` is omitted). It does not spawn a run, render a design, or update the project preview. Every run-direction doc now separates "add a file" from "create a visible design" so the user-facing bug cannot recur.

### Added: the `od run` verb family

`od run start|watch|cancel|list|info` were missing from `od-cli-reference.md` and are now documented in the verb table and the generation flow, with mutation classes.

### Added: the HTTP API surface

`od-cli-reference.md` now documents the four interchangeable surfaces (CLI, MCP, HTTP, Skills) and the HTTP API at `http://127.0.0.1:<port>/api/*`, with the verified endpoints `/api/health`, `/api/projects` (GET and POST), `/api/mcp/install-info`, and `/api/projects/:id/chat` (SSE). `/api/mcp/install-info` is named as the canonical config source. Two cautions are recorded: the HTTP port is ephemeral and rotates on every daemon restart (rediscover it, never hardcode it), and `autoSendFirstMessage:true` on POST `/api/projects` did not reliably fire the run on this build. The `od://app` alias is internal and does not resolve from a plain shell `curl`.

### Confirmed: the MCP wiring config shape

`/api/mcp/install-info` confirmed the install entry: `command[0]` is the "Open Design Helper" Electron binary, `args` is `[<daemon-cli.mjs>, "mcp"]`, and the `env` is `{OD_DATA_DIR, OD_SIDECAR_IPC_PATH, ELECTRON_RUN_AS_NODE:"1"}`. `mcp-wiring.md` now names install-info as the canonical source and marks the shape live-verified. The previously open `command[0]` uncertainty in `od-cli-reference.md` Section 7 is marked resolved.

## Preserved From v1.0.0

These were confirmed correct and are unchanged in substance: the MCP wiring config shape, the gate-every-mutation policy, the daemon-must-be-running requirement, the `node daemon-cli.mjs` invocation (no global `od` on PATH), the socket-discovered ephemeral daemon, and the roughly 18-tool live surface.

---

## Upgrade Notes

No config or wiring change is required. The MCP entry written by v1.0.0 remains valid. The behavioral change is in how an agent should drive generation: answer the discovery form to build a design, and never treat a single turn-1 call or an artifact write as a finished design.

---

## Known Limitations

- The desktop app must be open for any tool call, because the daemon it hosts answers every call.
- The HTTP port is ephemeral and rotates on daemon restart, so it must be rediscovered from `/api/mcp/install-info` or the socket each session.
- Per-verb auth and token requirements when a user (rather than a daemon-spawned agent) invokes generation or media are still partly open (see `od-cli-reference.md` Section 7).
- The headless `od --no-open` path was not re-exercised this session, so its full drive-without-GUI behavior remains inferred.
