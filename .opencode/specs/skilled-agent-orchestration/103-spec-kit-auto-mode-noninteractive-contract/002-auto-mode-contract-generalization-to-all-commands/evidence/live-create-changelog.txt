    48	## §2. PRE-BOUND SETUP ANSWERS Schema
    49	
    50	The `PRE-BOUND SETUP ANSWERS:` block is a structured marker the dispatched AI parses out of the prompt body. It allows callers to pre-bind setup field values without using individual `$ARGUMENTS` flags.
    51	
    52	Grammar (YAML-style key/value, one field per line):
    53	
    54	```
    55	PRE-BOUND SETUP ANSWERS:
    56	  <field_name>: <value>  # optional inline comment
    57	  <field_name>: <value>
    58	  ...
    59	```
    60	
    61	Rules:
    62	
    63	- The block opens with `PRE-BOUND SETUP ANSWERS:` (exact case-sensitive heading, followed by colon). Indentation MAY be inside a markdown fenced code block or inline in the prompt text.
    64	- Each field is `  <key>: <value>` indented two spaces, one per line.
    65	- Values are interpreted by the command's Default Resolution Table (§3). Field types and value enums are command-specific.
    66	- Comments after `#` on a value line are ignored.
    67	- Unknown fields produce a warning (do not abort), so callers may include forward-compatible fields.
    68	- Fields not specified fall back to: command-specific Default Resolution Table → command-specific default value → unresolved (triggers Tier 2 or Tier 3 per §1).
    69	- Malformed lines (e.g. missing `:`, malformed value type) emit a parse error naming the line. Known fields parsed before the error may still be used; remaining unresolved fields continue to Tier 2 or Tier 3.
    70	- A second `PRE-BOUND SETUP ANSWERS:` block in the same prompt body produces a duplicate-block error.
    71	
    72	The marker block applies ONLY under `:auto`. Under `:confirm`, the block is informational and does not skip the consolidated question prompt.

    77	
    78	Every command MUST publish a Default Resolution Table in its §0 (after citing this contract). The table lists each setup field with the following columns:
    79	
    80	| Field | Required | Resolves Via | Default | Tier-2 Candidate |
    81	|-------|----------|--------------|---------|------------------|
    82	
    83	Column meanings:
    84	
    85	- **Field**: the field name (matches the schema in §2 and the question label in the legacy consolidated Q-block).
    86	- **Required**: `Y` (must be resolved before YAML load) or `N` (optional).
    87	- **Resolves Via**: ordered list of resolution sources, e.g. `flag → $ARGUMENTS` / `marker → PRE-BOUND` / `default → "<value>"` / `auto-detect from <other field>` / `requires-ask`.
    88	- **Default**: the documented default value when no flag, marker, or auto-detection applies. May be `none` (no default — Tier 2 or Tier 3 candidate).

    90	
    91	A field is a Tier-2 candidate only when ALL of:
    92	- No flag exists for it.
    93	- No marker is provided.
    94	- No sensible default exists.
    95	- The field is genuinely required (Required=`Y`).
    96	- A user choice is the only resolution path (ambiguity, not absence).
