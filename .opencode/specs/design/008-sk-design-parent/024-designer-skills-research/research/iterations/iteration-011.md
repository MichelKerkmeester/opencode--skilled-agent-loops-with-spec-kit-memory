# Iteration 11: Design Systems Craft Split

## Focus

Deep-read the `design-systems` plugin and split practical system craft from design-system lifecycle material. The in-scope craft lens was narrow: tokens, component specs, naming, theming, icons, and motion where those improve `sk-design` foundations, audit, or motion. Governance, adoption, documentation-library maintenance, and localization programs were treated as out of scope unless they reduced to immediate build/audit checks.

## Questions Answered

- Which design-systems skills are practical system craft? `component-spec`, `design-token`, `icon-system`, `motion-system`, `naming-convention`, and `theming-system` have adoptable craft, but each should enter as a minimal rule in an existing sk-design file.
- Which are already covered? `accessibility-audit` is covered by `design-audit`; most `localization-design` audit/hardening content is already covered by typography and hardening references.
- Which are out of scope? `design-system-governance`, `documentation-template`, and `pattern-library` are lifecycle/documentation/governance capabilities, not sk-design build/visual taste capabilities.
- Does this require a new sk-design mode? No. The correct homes are existing `foundations`, `motion`, `audit`, and shared vocabulary files.

## Questions Remaining

- Should the future build put component-level handoff completeness in `design-foundations/SKILL.md` or in the shared sk-code handoff envelope? This iteration targets `design-foundations/SKILL.md` because that was in the requested read scope.
- Should the icon craft rule live in foundations layout, shared token vocabulary, or parent-level design instructions? Foundations layout is the smallest build-facing target from the files read here.
- Should direction-neutral naming be added as part of the token-layer edit, or left only as an audit hardening finding? My read: add one naming sentence to shared vocabulary, but do not import localization as a standalone topic.

## Next Focus

If another iteration continues this track, validate the five low-effort backlog items against the current sk-design parent hub and shared handoff file before implementation. The main risk is bloat: each adoption should be one compact rule, not a new lifecycle workflow.
