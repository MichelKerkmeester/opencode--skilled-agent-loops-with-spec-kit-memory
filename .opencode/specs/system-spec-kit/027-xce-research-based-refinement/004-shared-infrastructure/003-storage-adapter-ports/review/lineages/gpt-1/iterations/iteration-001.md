## Findings

### P0, Blocker

### P1, Required

- **F001**: `BetterSqliteVectorStore` ignores caller supplied record IDs. The port interface promises `upsert(record)`, `get(id)`, and `delete(id)` around a `VectorRecord.id`, but the better-sqlite adapter delegates record upsert to the string overload without persisting the supplied ID, then `index_memory` generates its own `lastInsertRowid`. A caller using the interface literally cannot rely on `get(record.id)` matching the record just upserted. [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/storage/ports/vector-store.ts:80-87] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/storage/ports/vector-store.ts:175-187] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:320-337]
- **F002**: `VectorStore.clear()` deletes memory metadata, not just vector records. The interface says `clear()` removes all vector records, but the better-sqlite implementation deletes from the active vector table, `active_memory_projection`, and `memory_index`. That is a wider destructive operation than the port contract advertises. [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/storage/ports/vector-store.ts:95-96] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/storage/ports/vector-store.ts:264-271]
