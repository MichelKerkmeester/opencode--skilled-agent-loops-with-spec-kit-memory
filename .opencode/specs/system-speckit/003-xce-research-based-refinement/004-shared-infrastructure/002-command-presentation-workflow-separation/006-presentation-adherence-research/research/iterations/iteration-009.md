# Iteration 009 (deepseek-v4-pro)

**Summary:** The worst-adherence surface is the memory family (search/manage/learn/save) whose routers gate critical parseable render templates behind external 166-243 line presentation files that mid-tier models skip or misparse — confirmed by live test where /memory:search ignored its §2 template. Inlining 5-6 line compressed templates into each router .md eliminates the external-Read failure mode, while the deep family's 328-463 line presentation files similarly need 5-line convergence dashboards inlined to prevent display-contract drift.

## [P1][DOC-DRIFT] Memory family has no YAML workflow assets; render template gated behind external file read
- Evidence: .opencode/commands/memory/search.md:18 `Before asking startup questions or displaying results, read the presentation asset` vs .opencode/commands/doctor/speckit.md:10-23 which inlines its menu directly. Live test: /memory:search ignored search_presentation.md §2 (lines 30-41) and answered in free prose; /doctor rendered its router menu verbatim.
- Detail: All other families (doctor, speckit, create, deep) have _auto.yaml + _confirm.yaml workflow assets providing step-by-step execution guardrails. The memory family has none. Its routers depend entirely on external presentation .md files (725 lines across 4 files) via a single 'read the presentation asset' instruction. A mid-tier model that skips this read has zero fallback render guidance, and even when it reads the file, 11+ competing templates cause selection errors.
- Fix sketch: Inline the critical parseable render template (<=6 lines) directly into each memory router .md file as a fenced code block, keeping the full presentation asset as authoritative reference only.

## [P1][DOC-DRIFT] search_presentation.md packs 11 competing templates into 243 lines causing model selection failure
- Evidence: .opencode/commands/memory/assets/search_presentation.md: contains sections 1-7 with 11 distinct templates (startup, retrieval, empty-fallback, analysis-overview, preflight, postflight, history, causal, link/unlink, causal-stats, ablation+dashboard, error). The retrieval-result template (§2 lines 30-41) is only 12 lines but competes with 231 lines of other templates.
- Detail: A mid-tier model scanning 243 lines must correctly (a) identify that it's in retrieval mode, (b) locate §2 among 7 sections, (c) suppress the other 10 templates. Live test confirmed this fails: the model answered in free prose instead of the parseable `MEMORY:SEARCH ... STATUS=OK` format. The doctor router avoids this by inlining only the currently-needed template.
- Fix sketch: Compress all 11 templates into a single 10-line master format: `MEMORY:SEARCH "<q>" i=<intent> n=<N>\n---\n<leaf>/ <score> #<id> <title>\nSTATUS=OK N=<N> I=<intent>` with variants collapsed to comment hints.

## [P1][REFINEMENT] Deep-family presentation files are 328-463 lines — too large for consistent mid-tier model adherence
- Evidence: .opencode/commands/deep/assets/: deep_start-model-benchmark-loop_presentation.md (463 lines), deep_start-agent-improvement-loop_presentation.md (461 lines), deep_start-review-loop_presentation.md (432 lines), deep_start-context-loop_presentation.md (390 lines), deep_start-research-loop_presentation.md (374 lines), deep_ask-ai-council_presentation.md (328 lines).
- Detail: Each deep presentation file contains complex auto-mode resolution tables, pre-bound YAML schemas (30+ fields), convergence semantics with multi-gate quality checks, fan-out executor configuration, and offline optimization notes. A mid-tier model processing 400+ lines of presentation rules has low probability of selecting and applying the correct display template for its current mode. The convergence dashboard and iteration-progress display are the critical output surfaces most at risk.
- Fix sketch: Inline a 5-line convergence dashboard template into each deep router .md: `DEEP:<LOOP> Iter<N>/<M> Mode<auto|confirm> Converge<ratio> Status<converging|converged|completed>\nFindings: P0<N> P1<N> P2<N> | Artifacts: <dir>\nSTATUS=OK ACTION=<running|completed>`.

## [P2][REFINEMENT] manage_presentation.md has 9 competing templates with no mode-routing anchor
- Evidence: .opencode/commands/memory/assets/manage_presentation.md: sections 2-8 present 9 distinct templates (stats, scan-confirm, scan-result, cleanup, retention, bulk-delete, tier/triggers/validate/delete, health, checkpoint/ingest) across 227 lines with no 'if mode=X use template Y' dispatch table.
- Detail: The manage router (manage.md:24-37) lists 12 recognized modes but has no per-mode render-template selector. A model must read the full 227-line presentation file and mentally map each mode to its template. In contrast, the doctor router (speckit.md:60) delegates rendering to the presentation file only for display, keeping routing logic separate. This split pattern works when the model actually reads the presentation file, but fails when it doesn't.
- Fix sketch: Add a 6-line dispatch table to manage.md mapping each mode to its 2-3 line compressed template: `stats → MEMORY:STATS Total<N> | Tiers:...`, `scan → MEMORY:SCAN Mode<N|F> Indexed<N>`, `health → MEMORY:HEALTH Status<healthy|degraded> Schema<vN>`.

## [P2][REFINEMENT] Compressed inline render template for /memory:search (5 lines, down from 243)
- Evidence: .opencode/commands/memory/assets/search_presentation.md §2 lines 30-41 (current 12-line retrieval template), §3 lines 50-64 (empty fallback), §7 lines 236-242 (error). The proposed compression collapses all modes to 5 lines.
- Detail: The retrieval template itself is already compact at 12 lines. The compression value is not in shortening individual templates but in (a) inlining the master template into the router so no external Read is needed, (b) collapsing 11 mode-specific templates into one format with comment-annotated variants, and (c) making the STATUS line the single parseable anchor for downstream consumers. Proposed: `MEMORY:SEARCH "<q>" i=<I> n=<N>\n---\n<leaf>/ <score> #<id> <title>\nSTATUS=OK N=<N> I=<I>\n# n=0: append TRIGGERED=<n> CONSTITUTIONAL=<n>; Error: STATUS=FAIL E="<m>"`
- Fix sketch: Replace search.md:47 `Render compact, parseable output using the presentation asset` with the 5-line fenced render template block inlined in the Retrieval Mode section (search.md:39-57).

## [P3][REFINEMENT] Compressed inline render template for /memory:manage (6 lines, down from 227)
- Evidence: .opencode/commands/memory/assets/manage_presentation.md: 9 templates across 227 lines. Compressed to: `MEMORY:<MODE> <key=val...>\n---\n<mode-specific lines>\nSTATUS=OK|AWAITING_INPUT|FAIL` where MODE∈{STATS,SCAN,CLEANUP,RETENTION,BULK-DELETE,TIER,TRIGGERS,VALIDATE,DELETE,HEALTH,CHECKPOINT,INGEST}.
- Detail: The manage presentation file has no dispatch anchor — every mode's template is presented as a sibling section. A compressed inline version groups all modes under a single format contract, reducing the model's selection surface from 227 lines of prose to 6 lines of structured format. The mode-specific output lines are collapsed to `key=val...` with the understanding that each mode's MCP tool return value maps naturally to its dashboard fields.
- Fix sketch: Add a 6-line render contract block to manage.md §3 replacing the external-presentation reference: one format line, 2 example mode lines (stats, scan), 1 confirmation-variant line, 1 error line.

## [P2][NEW-FEATURE] Missing render-template dispatch discipline: no family standard for when to inline vs. externalize
- Evidence: Compare: .opencode/commands/doctor/speckit.md:69-73 explicitly lists what lives in presentation (menu, help, dashboards) vs. router (target resolution). .opencode/commands/memory/search.md:18-18 has a single line `read the presentation asset` with no boundary definition. No command family documents a policy for what MUST be inlined vs. what may be external.
- Detail: The current architecture has no standard for render-template locality. The doctor family inlines its startup menu (proven effective) but externalizes diagnostic dashboards. The memory family externalizes everything. A standardized policy would state: 'Critical output templates that produce parseable STATUS lines must be inlined in the router. Optional descriptive text, help blocks, examples, and offline optimization notes may remain external.' This gives every command family a consistent baseline.
- Fix sketch: Adopt a render-template locality policy: any template producing a `STATUS=OK|FAIL|AWAITING_INPUT` line must be inlined (<=10 lines) in the router; descriptive prose, help text, examples, and optimization notes stay in the external presentation asset.
