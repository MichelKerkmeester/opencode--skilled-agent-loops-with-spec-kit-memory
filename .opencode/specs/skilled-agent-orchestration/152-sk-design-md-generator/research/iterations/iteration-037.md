# Iteration 037 — Track A (mimo)

## Focus
Per-section anti-fabrication ruleset (FORBID/CITE-TOKEN/GATE/LABEL per claim type)

## Findings
1. **[P0] §0 Brand Context — personality adjectives with rationale** — The spec requires '3-5 adjectives, each with a parenthetical rationale tied to an observable design decision.' The adjectives themselves (e.g. 'Authoritative', 'Technical', 'Premium') are interpretive labels. The rationales attempt grounding by citing CSS values (e.g. 'weight 300 headlines convey confidence') but the causal link between a CSS value and a personality trait is fabricated inference. There is no extraction pipeline that maps font-weight to 'confidence.'
   - Recommendation: GATE-ON-EVIDENCE: allow personality adjectives ONLY if each rationale cites at least one concrete token value (hex, weight, spacing). LABEL-INFERENCE: prefix the adjective list with '**Inferred brand personality:**' so the reader knows these are interpretive, not extracted. FORBID: adjectives from the banned-word list (writing_style_guide.md §2).
   - Evidence: design_md_format.md §2 (Brand Context), writing_style_guide.md §2 (Banned Words)
2. **[P0] §1 Visual Theme — opening sentence (essence capture)** — The spec requires 'ONE sentence' that 'captures the design system's essence' and passes the 'could this describe 3 other sites?' test. The style guide provides examples like 'Vercel's website is the visual thesis of developer infrastructure made invisible.' These are entirely fabricated metaphors with zero extraction grounding. No CSS token produces 'developer infrastructure made invisible.'
   - Recommendation: FORBID: never generate the opening sentence without at least 2 inline token references (hex, font, px). The sentence MUST embed concrete values (e.g. 'a near-black canvas (#08090a) where content emerges from darkness'). Any metaphor must be LABEL-INFERENCE.
   - Evidence: design_md_format.md §3 (Visual Theme), writing_style_guide.md §3 (Opening Paragraph Rules)
3. **[P0] §1 Visual Theme — comparative framing ('Unlike most systems...')** — The spec mandates 'at least one sentence in the prose must use a comparative structure: Unlike most systems..., Where others..., This is not...' The style guide reinforces with templates like 'Where most systems use X, this system uses Y.' The tool has no corpus of 'most systems' to compare against. Every comparative claim is fabricated from the LLM's training data, not from extraction.
   - Recommendation: FORBID: ban comparative framing unless the comparison is grounded in a cited token contrast (e.g. 'weight 300 at display sizes — most design systems use 600-700'). If no corpus comparison is possible, use LABEL-INFERENCE: '**Inferred convention contrast:** most systems use 600-700 for headlines; this system uses 300.'
   - Evidence: design_md_format.md §3 (Visual Theme, item 4), writing_style_guide.md §15 (Comparative Framing Pattern)
4. **[P0] §1 Visual Theme — named design principles** — The spec requires 'name and briefly define the 2-4 most distinctive principles of the system' with examples like 'Chromatic Depth: Shadows carry brand color.' These are interpretive labels invented by the writer. No extraction pipeline produces 'Chromatic Depth' or 'Compression as Identity.' The names and definitions are pure fabrication.
   - Recommendation: CITE-TOKEN: each named principle must reference at least one concrete CSS value in its definition. LABEL-INFERENCE: prefix with '**Inferred principles:**' so the reader knows these are interpretive labels, not extracted system names.
   - Evidence: design_md_format.md §3 (Visual Theme, item 3), writing_style_guide.md §16 (Named Principle Pattern)
5. **[P0] §1 Visual Theme — key characteristics front-half (design intent)** — The spec requires each key characteristic to have a 'front half that describes WHY (design intent).' This half is entirely fabricated. For example, 'text as compressed infrastructure' — no token produces this phrase. The back half (CSS values) is groundable, but the front half is interpretive prose.
   - Recommendation: LABEL-INFERENCE: the front half of each key characteristic must be marked as inference. CITE-TOKEN: the back half must contain at least one verifiable token value. If the intent cannot be grounded, use a neutral descriptor like 'negative letter-spacing at display sizes' instead of a fabricated metaphor.
   - Evidence: design_md_format.md §3 (Visual Theme, item 5), writing_style_guide.md §5 (Key Characteristics Format)
6. **[P1] §1 Visual Theme — expansion paragraphs P2-P4 personality sentences** — The style guide requires each paragraph to include 'personality' descriptions like 'a rich, saturated violet that reads as confident and premium' or 'whispered authority.' These are fabricated interpretive claims about visual quality. The CSS values within the paragraphs are groundable, but the personality framing is not.
   - Recommendation: GATE-ON-EVIDENCE: personality descriptions are allowed ONLY if they reference a measurable visual property (hue angle, saturation %, lightness value). FORBID: emotional/subjective personality claims ('reads as confident', 'whispered authority') without a measurable anchor. If no measurable anchor exists, omit the personality sentence.
   - Evidence: writing_style_guide.md §4 (Layered Expansion), writing_style_guide.md §6 (Color Descriptions — personality sentence)
7. **[P1] §2 Color Palette — personality sentence per color** — The spec requires every color entry to have a 'Personality sentence: One sentence about the visual quality (not emotion).' Examples: 'A saturated blue-violet that reads as financially authoritative' or 'The yellow undertone is key.' While the hex values are groundable, the personality sentence is a fabricated interpretation of what that hex 'reads as.' No extraction pipeline maps #533afd to 'financially authoritative.'
   - Recommendation: GATE-ON-EVIDENCE: allow personality sentences ONLY if they reference a measurable color property (hue, saturation, lightness, temperature). Example: '#533afd has hue 255°, saturation 96% — a high-saturation blue-violet.' FORBID: emotional personality claims ('reads as authoritative', 'signals confidence'). LABEL-INFERENCE: if a personality sentence is provided without measurable anchoring, prefix with '**Inferred:**'.
   - Evidence: design_md_format.md §4 (Color Palette, entry format), writing_style_guide.md §6 (Color Descriptions)
8. **[P1] §2 Color Palette — color boundary rules** — The spec requires 2-4 rules 'explaining which colors are system-level vs content-level.' These rules require understanding the design system's intent (e.g. 'Product accent colors appear at frequency 1-3, confined to product tile backgrounds. They are NOT part of the system palette'). While frequency data is groundable, the boundary classification (system vs content vs campaign) is an interpretive judgment.
   - Recommendation: GATE-ON-EVIDENCE: boundary rules must cite frequency counts and usage breakdowns. LABEL-INFERENCE: the classification itself (L1/L2/L3/L4) must be marked as inference unless explicitly present in extraction metadata.
   - Evidence: design_md_format.md §4 (Color Boundary Rules)
9. **[P2] §2.5 Dark Mode — strategy name + feel comparison** — The spec requires 'Name the dark mode approach with an evocative label' and 'a comparative observation: how does the dark mode FEEL different from light?' The strategy name (e.g. 'Inverted Gray Ladder') and the feel comparison are interpretive. All color mapping values are groundable.
   - Recommendation: LABEL-INFERENCE: strategy name and feel comparison must be prefixed with '**Inferred:**'. GATE-ON-EVIDENCE: the feel comparison must reference at least one measurable structural difference (e.g. 'gray ladder compresses from 8 steps to 5').
   - Evidence: design_md_format.md §5 (Dark Mode System, strategy name)
10. **[P1] §3 Typography — named strategies with architectural benefit** — The spec requires '3-5 bullet points, each with a bold name, 2-3 supporting values, and an architectural benefit.' The names (e.g. 'Light weight as signature', 'Bilingual parallelism') and architectural benefits (e.g. 'text hierarchy is driven by size and tracking, not weight') are interpretive. The values (font, weight, size) are groundable.
   - Recommendation: CITE-TOKEN: each named strategy must include at least 2 concrete values from tokens.json. LABEL-INFERENCE: the strategy name and architectural benefit must be prefixed with '**Inferred:**'. FORBID: architectural benefits that make causal claims without evidence (e.g. 'enabling a single-weight visual identity' — this is a design claim, not an observed fact).
   - Evidence: design_md_format.md §6 (Typography, Named Strategies)
11. **[P2] §4 Component Stylings — state change rationale** — The spec requires 'hover, focus, active, disabled lines must explain WHY, not just WHAT' with examples like 'darkens to signal actionability without changing the color family.' The WHY is interpretive. The CSS values (colors, transitions) are groundable.
   - Recommendation: LABEL-INFERENCE: state change rationale must be marked as inference. CITE-TOKEN: the rationale must reference the actual CSS value change (e.g. 'background shifts from #171717 to #4434d4 — 18% lightness reduction'). FORBID: causal claims ('to signal actionability') without a measurable anchor.
   - Evidence: design_md_format.md §7 (Component Stylings, item 2)
12. **[P2] §5 Layout Principles — whitespace philosophy** — The spec requires '2-3 bold-labeled bullet points' each with 'a contrast statement that names what the system does differently from convention' and 'exact px values as evidence.' The contrast statement is interpretive (requires knowledge of 'convention'). The px values are groundable.
   - Recommendation: GATE-ON-EVIDENCE: whitespace philosophy must cite at least 2 px values. LABEL-INFERENCE: contrast statements must be prefixed with '**Inferred convention contrast:**'. FORBID: claims about 'convention' without citing what the convention is.
   - Evidence: design_md_format.md §8 (Layout Principles, Whitespace Philosophy)
13. **[P1] §6 Depth & Elevation — depth philosophy + contrast prose** — The spec requires 'state the depth philosophy upfront using one of these categories' and '50-100 words' contrasting with 'conventional approach.' The category selection (e.g. 'Chromatic depth') is interpretive. The contrast prose is fabricated — the tool has no corpus of 'conventional approaches.' The shadow table values are groundable.
   - Recommendation: LABEL-INFERENCE: depth philosophy category must be prefixed with '**Inferred depth approach:**'. CITE-TOKEN: the contrast prose must reference at least 1 concrete RGBA value from the extraction. FORBID: claims about 'the conventional approach' without citing what that approach is. If no contrast can be grounded, write: 'The shadow system uses [specific values]. No conventional comparison is available from extraction data.'
   - Evidence: design_md_format.md §9 (Depth & Elevation), writing_style_guide.md §12 (Shadow Philosophy Pattern)
14. **[P2] §6.5 Motion System — motion philosophy** — The spec requires '1-2 sentences describing the system's approach to motion. Not just uses 300ms but the intent.' The intent is interpretive. Duration/easing values are groundable.
   - Recommendation: LABEL-INFERENCE: motion philosophy must be prefixed with '**Inferred motion intent:**'. CITE-TOKEN: must reference at least 1 concrete duration or easing value.
   - Evidence: design_md_format.md §10 (Motion System, Motion Philosophy)
15. **[P1] §7 Content & Voice — vibe paragraph** — The spec requires '2-3 sentences capturing the brand personality as expressed through copy.' The style guide example: 'Stripe writes like a senior engineer explaining a product to a peer -- precise, specific, never patronizing.' This is entirely fabricated interpretation. No extraction pipeline produces 'senior engineer' or 'never patronizing.'
   - Recommendation: FORBID: the vibe paragraph as a standalone prose section. REPLACE WITH: a structured table of observed voice characteristics (person, tense, jargon density, sentence length) grounded in extracted sampleTexts. If prose is retained, LABEL-INFERENCE: prefix with '**Inferred voice characterization:**'.
   - Evidence: design_md_format.md §11 (Content & Voice, Vibe Paragraph), writing_style_guide.md §19 (Brand Voice Detection Pattern)
16. **[P2] §7 Content & Voice — tone descriptors** — The spec requires '3-5 tone descriptors, each with a real example from the site.' If the examples are real (from sampleTexts), this is groundable. But the descriptors themselves (e.g. 'Direct', 'Technical', 'Confident') are interpretive labels applied to observed text.
   - Recommendation: GATE-ON-EVIDENCE: tone descriptors are allowed ONLY if paired with a real quote from extraction. LABEL-INFERENCE: the descriptor itself is inference, but the quote is groundable.
   - Evidence: design_md_format.md §11 (Content & Voice, Tone)
17. **[P1] §7 Content & Voice — error/empty state copy (when not observed)** — The spec says 'If no error/empty state copy was observed, state that and provide tone-consistent recommendations.' The 'tone-consistent recommendations' are fabricated — the tool invents copy that 'sounds like the brand' without any extraction evidence.
   - Recommendation: GATE-ON-EVIDENCE: if no error/empty state copy was observed, stamp ABSENT. Do NOT generate 'tone-consistent recommendations.' Write: 'No error/empty state copy observed in extraction. Implementers should derive from observed tone characteristics above.'
   - Evidence: design_md_format.md §11 (Content & Voice, Error/Empty State Copy)
18. **[P1] §8 Do's and Don'ts — counter-intuitive claims** — The spec requires Don'ts that 'would surprise a competent developer who knows CSS but not this specific system.' This encourages fabrication of surprising claims. The style guide example: 'Don't use weight 600-700 for sohne-var headlines -- weight 300 is the brand voice.' While weight 300 may be observed, the claim that this is 'counter-intuitive' or that 'weight 300 is the brand voice' is interpretive.
   - Recommendation: CITE-TOKEN: every Do and Don't must include at least 1 concrete CSS value. FORBID: 'brand voice' or 'brand personality' claims in Do/Don't rationale. LABEL-INFERENCE: the 'counter-intuitive' framing is inference — mark it as such or omit it.
   - Evidence: design_md_format.md §12 (Do's and Don'ts), writing_style_guide.md §8 (Do's and Don'ts Quality)
19. **[P0] §9 Accessibility — inferred WCAG target level** — The spec requires 'State the inferred WCAG level based on observed behavior' with examples like 'Inferred target: WCAG 2.1 AA. Evidence: focus indicators present on all interactive elements, color contrast ratios exceed 4.5:1 for body text.' The tool cannot actually test WCAG compliance. It observes CSS values and infers compliance, which is a fabrication. Focus indicators being 'present' does not prove WCAG AA compliance.
   - Recommendation: FORBID: do not claim a WCAG level. REPLACE WITH: '**Observed accessibility signals:**' — list what was actually detected (focus indicators, contrast ratios calculated from hex values, touch target sizes). Present as a table of observed evidence, not a compliance claim. Let the implementer determine the WCAG level.
   - Evidence: design_md_format.md §13 (Accessibility Contract, WCAG Target)
20. **[P2] §9 Accessibility — 'consistent'/'compliant' boolean claims** — Claims like 'focus indicators present on all interactive elements' or 'color contrast ratios exceed 4.5:1' are boolean assertions that may not hold across all extracted elements. The tool samples, it doesn't exhaustively test.
   - Recommendation: GATE-ON-EVIDENCE: boolean claims about consistency require N observations (e.g. 'focus indicators observed on 47/52 interactive elements'). If N is too low, stamp ABSENT. FORBID: universal claims ('all', 'every', 'consistent') without count evidence.
   - Evidence: design_md_format.md §13 (Accessibility Contract)
21. **[P2] §10 Responsive Behavior — collapsing strategy descriptions** — The spec requires '6-10 bullet points using arrow format with exact pixel transitions.' The px values are groundable, but the descriptive labels (e.g. 'Hero headline') require understanding component semantics, which may be inferred.
   - Recommendation: CITE-TOKEN: px values must be from extraction. LABEL-INFERENCE: component role labels (e.g. 'Hero headline') are inference from DOM structure.
   - Evidence: design_md_format.md §14 (Responsive Behavior, Collapsing Strategy)
22. **[P2] §11 State Matrix — unobserved state cells** — The spec requires '4+ component rows x 5 state columns' but many states may not have been observed. The spec says to use '-' for unobserved states, but the style guide encourages generating plausible treatments. Fabricated state treatments (e.g. 'Spinner icon, opacity 0.6' for a loading state never observed) are common.
   - Recommendation: GATE-ON-EVIDENCE: each cell must have observation data or stamp '-'. FORBID: generating plausible state treatments for unobserved states. If >50% of cells are '-', note: 'State matrix is sparse — most states were not observed during extraction.'
   - Evidence: design_md_format.md §15 (State Matrix)
23. **[P2] §12 Iconography — substitution recommendation** — The spec requires recommending 'closest open alternative' for proprietary icon sets. The recommendation (e.g. 'Lucide is closest because matching stroke weight 1.5px') is an interpretive comparison the tool makes without a systematic icon-set corpus.
   - Recommendation: LABEL-INFERENCE: substitution recommendations must be prefixed with '**Inferred substitution:**'. CITE-TOKEN: must reference measurable properties (stroke weight, grid size) that justify the recommendation.
   - Evidence: design_md_format.md §16 (Iconography, Substitution Recommendation)
24. **[P2] §13 Agent Guide — iteration guide interpretive claims** — The iteration guide requires '6-8 numbered steps, each with at least one concrete value.' Steps may include interpretive claims like 'shadow-as-border instead of CSS border' which names a technique that is inference from observed CSS patterns.
   - Recommendation: CITE-TOKEN: each iteration step must include at least 1 verifiable CSS value. LABEL-INFERENCE: technique names (e.g. 'shadow-as-border') are inference.
   - Evidence: design_md_format.md §17 (Agent Prompt Guide, Iteration Guide)

MOSTLY-GROUNDABLE sections with isolated high-risk claims: §0 (personality adjectives), §2 (color personality sentences), §6 (depth philosophy + contrast prose), §9 (WCAG target inference). FULLY GROUNDABLE sections: §2.5 Dark Mode (except strategy name), §4 Components (except state rationale), §5 Layout (except whitespace philosophy), §6.5 Motion (except philosophy), §10 Responsive, §11 State Matrix (if cells are observed), §12 Iconography (except substitution), §13 Agent Guide (if values are copied). The critical pattern: every section has a 'prose wrapper' that the format spec and style guide explicitly require, and that prose wrapper is where 90%+ of fabrication occurs. The token-level data (hex, px, weight, frequency) is groundable; the interpretive framing around it is not. The writer contract must separate these two layers and enforce different rules for each.
