## Findings (file:line)

### Q6 â€” Decay as pure rank-time function (KEY STRUCTURAL FINDING â€” the split is already half-built)
1. **The rank-time decay/freshness path is already pure and write-free.** `fsrs.ts` is the rank-time structural-freshness module: `computeStructuralFreshness(stability, centrality) = clamp(stability) * clamp(centrality)` and `computeGraphCentrality` are pure functions â€” no DB handle, no write [SOURCE: lib/search/fsrs.ts:40-47, 63-82]. This is the "central nodes decay slower" rank-time signal (step 6 of the 13). It already matches aionforge's "decayed importance the re-rank computes" with no persistence [SOURCE: external/aionforge-memory-development/docs/retrieval.md:316-317].
2. **The ONLY write-back is `strengthenOnAccess`, and it is the FSRS reinforcement ("testing effect"), NOT the decay computation.** `strengthenOnAccess` does `UPDATE memory_index SET stability=?, last_review=CURRENT_TIMESTAMP, review_count=review_count+1, access_count=access_count+1, last_accessed=?` [SOURCE: lib/search/pipeline/stage2-fusion.ts:602-610]. It computes a *new* stability via `fsrsScheduler.updateStability(..., GRADE_GOOD, clampedR) * (1 + difficultyBonus)` where `difficultyBonus = max(0, (0.9 - R) * 0.5)` â€” i.e. low-retrievability memories get the biggest stability boost [SOURCE: stage2-fusion.ts:592-600]. This is retrieval-reinforcement (spaced-repetition strengthening), a *different concept* from rank-time decay scoring.
3. **The rank-time effective value `R(t)` is already computed independently, BEFORE the write fires.** `applyTestingEffect` (`stage2-fusion.ts:889-916`) reads stored `stability`+`last_review`, computes `elapsedDays = calculateElapsedDays(lastReview)` and `currentR = calculateRetrievability(stability, max(0, elapsedDays))`, and only *then* calls `strengthenOnAccess(db, row.id, currentR)` [SOURCE: stage2-fusion.ts:897-910]. So the decay-at-read math (`calculateRetrievability` over elapsed time) and the persistence (`strengthenOnAccess`) are **already two distinct calls in sequence** â€” the split aionforge wants is structurally half-present; the coupling is only that both sit inside one `trackAccess`-gated step.
4. **The write-back is already opt-in (`trackAccess`), so decay scoring is ALREADY pure when `trackAccess=false`.** The testing effect "Called only when `trackAccess` is true (explicit opt-in to avoid unintended write side-effects during read-only searches)" and is gated `if (config.trackAccess)` [SOURCE: stage2-fusion.ts:876-877, 1237-1247]. So a read-only recall (default) does NOT write back today â€” the never-written-back behavior already exists for the common path; the divergence from aionforge is only that an *opt-in* write path exists at all, and that it uses an *ambient* `CURRENT_TIMESTAMP`/`Date.now()` clock [SOURCE: stage2-fusion.ts:605, 608] rather than a caller-supplied one [aionforge contract: retrieval.md:307-310 "no ambient clock ... always caller-provided"].
5. **What would break if the write-back were simply removed:** the FSRS spaced-repetition reinforcement â€” `review_count`, `access_count`, `last_review`, and the difficulty-weighted `stability` boost that makes frequently-needed-but-weak memories more retrievable over time. That signal feeds back into finding #3's own `calculateRetrievability` on the *next* recall and into `last_accessed`-derived recency. Removing it would make decay monotonic (memories only ever decay, never re-strengthen on use), losing the "testing effect." So a naive "never write back" is NOT safe here â€” unlike aionforge, internal *uses* the write-back as a learning signal.

### Q6 candidate proposal
- **C6-A (Sâ†’M): Split into (a) pure rank-time effective-score, always on, caller-clock; and (b) reinforcement as a separate, optional, explicitly-emitted event.** (a) Promote the already-pure `calculateRetrievability(stability, elapsedDays)` decay to a standard rank-time signal computed for *every* recall (not only `trackAccess`), reading elapsed time against a **caller-supplied `nowMs`** instead of an ambient clock, so decay scoring is deterministic and write-free by construction [evidence: stage2-fusion.ts:897-908 already computes this; aionforge retrieval.md:307-310, 316-317]. (b) Keep `strengthenOnAccess` but rename/reframe it as an explicit *reinforcement event* decoupled from ranking â€” fired only on a deliberate "this memory was actually used" signal, not as a side-effect of every `trackAccess` recall, preserving the FSRS testing effect without coupling it to rank order. Effort **S** for the clock-injection + always-on rank-time decay (the function already exists and is already separate); **M** if reframing the reinforcement trigger (touches the `trackAccess` contract and any caller relying on recall-as-reinforcement). This is the minimal split that gets aionforge's determinism (finding #3/#4) WITHOUT losing the testing effect (finding #5).

### Q7 â€” Session-diversity cap (CONFIRMED GAP â€” clean, no internal equivalent)
6. **There is NO session-diversity / single-spec-folder dominance cap anywhere in the search pipeline.** A grep across `lib/search/ handlers/` for `session.?divers|diversity|per.?session|dominanc|spill|maxPerSession|maxPerFolder|maxPerSpec` returned **zero** session/folder-cap logic â€” every "diversity" hit is MMR (embedding similarity), not session [SOURCE: grep lib/search/ handlers/ â€” only apply_n/MMR/INTENT_LAMBDA hits]. So a single chatty session or one dominant spec-folder can occupy the entire top-k.
7. **The result-assembly truncation is two bare `slice(0, limit)` steps with no per-source cap.** (a) Stage-2 artifact limiting: `results = results.slice(0, config.artifactRouting.strategy.maxResults)` (only when routing confidence>0) [SOURCE: lib/search/pipeline/stage2-fusion.ts:1388-1396]; (b) Stage-4 final cap: `workingResults = workingResults.slice(0, config.limit)` ("Enforce caller's limit after all filtering") [SOURCE: lib/search/pipeline/stage4-filter.ts:305-309]. Both truncate the score-ordered list head-first; neither considers session or spec-folder provenance. (Stage-4 also has per-*memory-state* tier limits at :170-177, but those cap by lifecycle state, not by session.)
8. **The existing MMR "diversity" is an orthogonal axis (embedding similarity), not session diversity.** `apply_n` (aliased `applyDiversity`) computes `mmr_score = relevance - (n_factor * max_similarity_to_selected)` over embeddings, with intent-driven lambda (`understandâ†’0.5` more-diverse, `fix_bugâ†’0.85` more-relevant) [SOURCE: lib/search/vector-index-queries.ts apply_n; lib/search/intent-classifier.ts INTENT_LAMBDA_MAP; stage3-rerank.ts MMR step]. This demotes semantically-redundant results, NOT results from the same conversation â€” so it does not satisfy aionforge's session cap, which is explicitly "an episode notion" about provenance, not content similarity [SOURCE: external/aionforge-memory-development/docs/retrieval.md:289-293].
9. **aionforge's contract that internal lacks:** "The `session_diversity_cap` (default 3) is the most memories from one session allowed into the primary set before the rest spill. Spilled memories are appended only if the bundle is under-filled, so the cap demotes a single conversation that would otherwise dominate without ever returning fewer results than it could. The cap is an episode notion â€” facts have no session and always go straight to the primary set in fused order." [SOURCE: external/aionforge-memory-development/docs/retrieval.md:289-293].

### Q7 candidate proposal
- **C7-A (S): Insert a session/spec-folder diversity cap immediately before the final truncation.** Add a cap pass between `sortDeterministicRows` and the stage-4 `slice(0, config.limit)` (or as a stage-4 sub-step before :307): walk the score-ordered list, admit at most `N` (default 3) rows per `sessionId`/`spec_folder`, send overflow to a spill list, and append spill only if the primary set is under `config.limit` â€” exactly aionforge's spill-if-underfilled semantics [evidence: stage4-filter.ts:305-309; aionforge retrieval.md:289-293]. Effort **S** â€” a single in-place reorder/partition before the existing slice; no scoring change; the score-ordered input already exists. Open design question (defer): the internal grain is `spec_folder` rather than `session` (the natural "one conversation dominates" unit here is a spec-folder); the cap key should likely be configurable (`spec_folder` primary, `sessionId` secondary). Note aionforge caps only *episodes* and lets *facts* bypass â€” internal has no episode/fact split, so the cap would apply uniformly unless a memory-type exemption is added.

## Questions Answered
- **Q6 (ANSWERED):** Decay scoring is **already a pure rank-time function** (`fsrs.ts` structural freshness + `calculateRetrievability` at `stage2-fusion.ts:905-908`) and is **already write-free on the default read-only path** (the write is `trackAccess`-gated). The single write-back, `strengthenOnAccess` (`stage2-fusion.ts:602-610`), is FSRS *reinforcement* (testing effect), not decay computation â€” and it is already a separate call from the rank-time `R(t)`. So decay CAN become a pure caller-clock rank-time function (C6-A) WITHOUT losing the testing effect, by (a) making the existing rank-time decay always-on against a caller-supplied clock and (b) keeping reinforcement as a separate, explicitly-triggered event rather than a recall side-effect. Naive "never write back" is NOT safe â€” finding #5 shows the write-back is a used learning signal, the one place internal genuinely diverges from aionforge's stance.
- **Q7 (ANSWERED):** There is **NO** session/spec-folder dominance cap (finding #6). Result assembly truncates with two bare `slice(0, limit)` steps (`stage2-fusion.ts:1395`, `stage4-filter.ts:308`); the only "diversity" present is embedding-similarity MMR (`apply_n`), an orthogonal axis (finding #8). A cap slots in cleanly just before the stage-4 final slice with aionforge's spill-if-underfilled semantics (C7-A), keyed on `spec_folder`/`sessionId`. iter-1's "internal result-assembly/cap location not yet read" is now resolved.
