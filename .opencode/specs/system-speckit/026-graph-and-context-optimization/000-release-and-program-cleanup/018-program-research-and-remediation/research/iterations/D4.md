### Angle 49: Multi-session shared-index race
- **Framing:** Concurrent sessions writing to the shared `memory_index` database cause data corruption, lost writes, or stale reads that a per-session index or commit-discipline tooling would prevent.
- **Repo evidence:** `system-spec-kit/mcp_server/core/db-state.ts:409` (`acquireIndexScanLease` — single-writer lease on `scan_started_at`); `system-spec-kit/mcp_server/core/config.ts:126` (`INDEX_SCAN_COOLDOWN: 30000` ms); `system-spec-kit/mcp_server/stress_test/durability/index-scan-coalescing-stress.vitest.ts:5-10` (64-wide concurrent burst test: exactly one writer admitted, rest get structured `lease_active` backoff); `system-spec-kit/mcp_server/stress_test/durability/README.md:23` (stress test proves single-writer, coalescing, expired-lease reclaim); `system-spec-kit/mcp_server/handlers/memory-index.ts:352` ("Reserve scan_started_at up front to avoid check-then-set race windows"); `system-spec-kit/mcp_server/core/db-state.ts:510` ("parallel and corrupt scan_started_at — signal a short retry/coalesce window").
- **Findings:** The shared-index race is real but well-mitigated. The `memory_index_scan` handler uses a single-writer lease pattern: `acquireIndexScanLease()` atomically reads `last_index_scan` and `scan_started_at` in one transaction, reserves the slot, and rejects overlapping scans with a structured `lease_active` reason. The 64-wide stress test proves this works under contention. The launcher's front-proxy (`launcher-session-proxy.cjs`) adds another layer: multiple MCP clients share one daemon process, and the proxy handles transparent recycle without exposing race windows to clients. However, the `memory_save` path (non-scan writes) does NOT use the same lease — saves go through the daemon's single-process SQLite connection with WAL mode, which serializes writes at the database level. The real race exposure is NOT concurrent index scans (solved by the lease) but concurrent `memory_save` calls from different sessions hitting the same daemon — SQLite WAL handles this correctly but with potential `SQLITE_BUSY` under extreme contention. The `SPECKIT_LAUNCHER_BRIDGE_DISABLED` env var exists as a rollback flag for intentional parallel runs.
- **Next experiment:** Run 4 concurrent `cli-opencode` sessions, each saving 20 spec-docs to the same shared daemon, and measure: (1) any `SQLITE_BUSY` errors, (2) any lost writes (count rows vs expected), (3) any stale reads (save-then-search latency to consistency). If 0 errors across 80 saves, the shared daemon is safe; if >0, a per-session index or save-queue would help.
- **Verdict:** The single-writer lease on `memory_index_scan` is battle-tested (64-wide stress). The shared daemon's WAL-based write serialization for `memory_save` is architecturally correct but lacks explicit contention testing. The real risk is not index-scan races (solved) but daemon-level SQLite_BUSY under extreme concurrent writes. A per-session index would add complexity without clear benefit given the current single-daemon architecture. **LOW CONFIDENCE** that a real race exists under normal workloads.
