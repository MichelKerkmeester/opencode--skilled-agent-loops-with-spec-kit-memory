## Iteration 009 — Distribution & Portability: what sk-code/skills can borrow from ponytail

### Evidence base

**Ponytail's model** (embedded source): ONE behavior source (`skills/*/SKILL.md` + compact `AGENTS.md`) → THIN, host-specific adapters for 9 hosts. Adapter rule: *point capable hosts at the shared `skills/`+`hooks/`; only hosts limited to project-instructions get copied rule text, kept aligned to `AGENTS.md`*. Alignment enforced in CI by `scripts/check-rule-copies.js` — byte-equality for compact copies + canary-phrase presence for the longer SKILL.md.

**Our repo, two orthogonal portability axes:**

1. **STACK/PROJECT axis (sk-code §1):** `sk-code` is declared "the **only** skill end users should edit when adopting this template" — replace `references/{webflow,opencode,motion_dev}/` + `assets/{…}/`, then "Update `STACK_FOLDERS` (§2) + `RESOURCE_MAP` to match." Every other skill stays codebase-agnostic "to keep upstream pulls clean." This **is** ponytail's single-source philosophy, but applied to *which stack the template targets*, not *which host runs it*.

2. **HOST/RUNTIME axis (agents):** `.opencode/agents/*.md` is canonical; `.claude/agents/*.md` and `.codex/agents/*.toml` are mirrors (`README.txt`: "Sibling runtimes…"). Crucially these are **full ~44–55 KB copies**, not thin adapters — the review trio shows near-identical bodies differing only in frontmatter format (md frontmatter vs `developer_instructions = '''…'''` TOML wrapper) and the canonical-path string (`.opencode/agents/*.md` → `.codex/agents/*.toml`). The `.codex` file even carries `# Converted from: .opencode/agents/review.md` — proving a conversion step existed, but I found **no live generator script**; the conversion looks one-time/manual.

**The repo already has the ponytail analog — but mis-scoped.** `.opencode/skills/deep-improvement/scripts/lib/mirror-sync-verify.cjs` is a working `check-rule-copies.js` equivalent: it hardcodes the 3-runtime `RUNTIME_MIRRORS`, extracts the agent body per format (strips md frontmatter / extracts codex `developer_instructions`), `normalizeRuntimeSpecificText()` collapses the only legit per-runtime difference (the agent-path strings), then compares **token sets** (deliberately *not* byte equality). `scan-integration.cjs` does a lighter 2-of-3 signal match. `mirror_drift_policy.md` makes all-3-mirror parity a **hard gate before agent-definition promotion**.

The limitation: this lives **inside the `deep-improvement` skill** and only fires **during a guarded agent-improvement promotion**. A hand edit to `.claude/agents/code.md` that diverges from canonical is caught by **nothing** at commit/CI time — unlike comment-hygiene, which has three independent gates (`scripts/hooks/claude-posttooluse.sh`, `.opencode/hooks/pre-commit`, `.github/workflows/comment-hygiene.yml`, per SKILL.md §3).

### Answers

**1. Does sk-code already embody ponytail's single-source? What's missing?** Yes, on the **stack axis** — `STACK_FOLDERS`+`RESOURCE_MAP` retarget is a genuine single-source-of-truth customization surface, arguably cleaner than ponytail because skills physically live **once** in `.opencode/skills/` and are `Read()` by every runtime, so skills need *no* host mirror at all. What's missing is a **validator binding the declaration to disk**: nothing checks that each `STACK_FOLDERS` key has a real `references/<surface>/`+`assets/<surface>/` tree (or that on-disk trees are all declared). `verify_alignment_drift.py` checks *code-file* integrity rules, not the `STACK_FOLDERS`↔`references/` contract.

**2. Adapter-sync-check for skills across 3 runtimes — where?** For **skills**, not needed (single physical copy — the scalable inverse of ponytail's copy-per-host). For **agents**, the check already exists; the borrow is to **promote `mirror-sync-verify.cjs` out of `deep-improvement` into a repo-wide gate** that runs on every agent edit, sitting beside the existing comment-hygiene gates (`.opencode/hooks/pre-commit` + a `.github/workflows/` job). That is exactly how ponytail runs `check-rule-copies.js` in CI rather than only at release.

**3. Concrete sk-code portability improvement.** Add a **retarget checklist asset** + a small `assets/scripts/verify_stack_folders.py` that parses the `STACK_FOLDERS` dict from SKILL.md §2 and asserts (a) every declared surface has `references/<surface>/`+`assets/<surface>/`, (b) no orphan reference tree is undeclared, (c) `RESOURCE_MAP` targets exist. This is the sk-code analog of `check-rule-copies.js`: instead of "compact copies byte-equal + SKILL.md canary present," it's "declared `STACK_FOLDERS` ↔ on-disk `references/` stay in sync." Directly closes the §1 instruction ("Update `STACK_FOLDERS` to match") with enforcement.

**4. Negative knowledge — what does NOT scale to sk-code.** Ponytail fans out to 9 hosts *because it is tiny*. For a large multi-file router these mechanisms break: (a) **byte-equality copies** — sk-code is hundreds of KB across `references/`/`assets/`/`scripts/`; the repo already abandoned byte-equality for **token-set comparison with path normalization** for exactly this reason. (b) **Always-on injection** (OpenCode plugin injecting the full ruleset every turn) — sk-code is loaded **on-demand** post-detection; injecting its full body each turn would blow the context budget. Its progressive disclosure is the correct opposite design. (c) **Copy-rule-text-into-each-host-project-file** (Cursor/Windsurf/Cline/Copilot copies) — fine for a one-file ruleset, unmaintainable for a multi-file router; the repo's "skills live once, `Read()` by all" is the scalable inverse. (d) **9-platform fan-out** — bounded by *runtime capability*, not file format: the agents depend on the MCP daemon stack (spec-memory/code-index/skill-advisor) that Cursor/Windsurf/Kiro/Pi don't run, so more host adapters multiply maintenance (already 3 full copies/agent) with no payoff.
