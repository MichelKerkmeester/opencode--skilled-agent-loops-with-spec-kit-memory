### Angle 25: Does maxDepth truncate genuinely-relevant blast-radius nodes?

- **Framing:** Testable claim — the BFS `maxDepth` hard cap silently discards reverse-import dependency nodes beyond the depth boundary without surfacing a completeness-gap warning to the caller.

- **Repo evidence:** `.opencode/skills/system-code-graph/mcp_server/handlers/query.ts:1096-1097` — `if (nextDepth > maxDepth || visited.has(importerFilePath)) { continue; }`. The default non-transitive depth is 1 (`query.ts:1299`), transitive default is 3 with handler clamp at 20 (`query.ts:1232-1233`). Granularity classification uses only depth-1 fanout (`query.ts:1032-1050`) — deeper truncation affects the reported fanout set but not risk classification. The `buildDepthGroups` function (`query.ts:1012-1025`) groups only collected nodes; it has no post-hoc completeness check.

- **Findings:** The BFS strictly enforces `maxDepth` as a hard cut — nodes at depth `maxDepth+1` are never visited, with no reachability gap detection. For deep dependency chains (e.g., a library re-exported through 4+ intermediate modules), files genuinely in the blast radius are invisible. The caller gets no signal distinguishing "no deeper importers exist" from "deeper importers exist but were truncated." The risk classification is calibrated on depth-1 only, which is intentionally conservative but means depth-truncated nodes don't even affect the high/medium/low signal. The `limit` (result-count) truncation *does* emit an `overflowed` boolean (`query.ts:1126`), but the depth truncation has no equivalent — it's a silent loss.

- **Next experiment:** Construct a 5-hop transitive import chain (A→B→C→D→E) and run blast_radius on A with `includeTransitive:true, maxDepth:2`. Confirm nodes at depth 3-4 are absent. Then measure recall against a ground-truth dependency graph on a real codebase at default depth 3 vs max allowed depth 20 — what % of the full transitive closure is missed?

- **Verdict:** Yes, maxDepth silently truncates relevant nodes with no completeness signal. Recommendation: emit a `depthTruncated: true` boolean (mirroring `overflowed`) when the BFS frontier is non-empty at `maxDepth`, and consider raising the default from 3 to at least 5 for transitive queries. **LOW CONFIDENCE** on the default-tuning recommendation — needs the recall measurement first.

### Angle 32: Model-benchmark correctness gate — does it reliably prevent saturation from crowning a wrong winner?

- **Framing:** Testable claim — when a correctness column is fully saturated (all frameworks score 1.0 on correctness), the benchmark gate correctly refuses to rank by correctness and cannot crown a correctness-based WINNER.

- **Repo evidence:** `.opencode/skills/deep-improvement/scripts/model-benchmark/tests/sweep-acceptance.vitest.ts:155-227` (the "SATURATION CANNOT WIN" acceptance suite) asserts: `agg.correctness_saturated` must be true when every group has `correctness_mean === 1`; `agg.ranking_key` must NOT be `'correctness'`; the verdict on a single sample must be TIE or INCONCLUSIVE. The gate is also independently tested in `sweep-stats-ci.vitest.ts:302-316` and `sweep-runtime.vitest.ts:199+`. The `correctnessGate.threshold` is a user-configurable numeric in [0,1] validated by `sweep-foundation.vitest.ts:151-158`.

- **Findings:** The gate is demonstrably solid for the specific attack it was designed for: a perfectly-correct responder that saturates correctness at 1.0. In that scenario, the framework drops correctness from the ranking key, forces a TIE/INCONCLUSIVE on a single sample, and surfaces every saturated fixture with `action: "promote-or-demote-to-smoke"`. With genuine differentiation (terse vs verbose responders on efficiency), the gate lets a trusted WINNER through on format/efficiency with a margin vs noise_floor check. The test coverage spans the full path: mock responder → sweep run → aggregate assertion → synthesis file order check.

- **Next experiment:** The gate protects against a fully-saturated correctness column, but what about a *near-saturated* column (e.g., 0.98 vs 0.99)? The saturation flag may remain `false` while the margin is noise-level. Stress-test with a fixture pack where 4 of 5 frameworks score 0.97-0.99 on correctness and verify whether a 0.02 delta can crown a spurious winner.

- **Verdict:** The gate reliably prevents a fully-saturated correctness winner. The edge case worth checking is near-saturation (close scores with a narrow, noise-level margin where saturation flag doesn't fire). The test suite is thorough for its design contract.

### Angle 33: Adversarial round-2 verify — measured false-positive reduction of the "downgrade 4 of 5 P1s" pattern

- **Framing:** Testable claim — the claim adjudication + adversarial self-check protocol (Hunter/Skeptic/Referee) produces a measurable severity-downgrade rate for initially-reported P0/P1 findings across real review runs.

- **Repo evidence:** The adversarial self-check is mandated by SKILL.md Rule 10 (`.opencode/skills/deep-review/SKILL.md:452`) and the iteration checklist step 5. The claim adjudication packet schema is defined in `loop_protocol.md:349-376` with required fields: `findingId`, `claim`, `evidenceRefs`, `counterevidenceSought`, `alternativeExplanation`, `finalSeverity`, `confidence`, `downgradeTrigger`, `transitions`. The `downgradeTrigger` field (`state_format.md:432`) is a concrete future-condition predicate. The gate is load-bearing: a missing/failing adjudication packet vetoes STOP via `claimAdjudicationGate` (`blocked_stop` event at convergence.md:117). The adversarial self-check and claim adjudication are *separate* mechanisms: self-check is "is the blocker real?", adjudication is "record the evidence trail and final severity."

- **Findings:** The infrastructure exists to measure the downgrade rate — every transition from asserted severity to final severity is recorded in the `transitions` array inside the claim adjudication packet, and both the original and adjudicated severity are preserved. However, no automated metric aggregation over the JSONL history exists in the shipped code. The "downgrade 4 of 5 P1s" phrase doesn't appear literally in the codebase (only `downgrade` appears in relation to P0/P1 severity adjustment), making it a conceptual test scenario: run a review where 5 P1 findings are asserted, have the adversarial verify pass downgrade 4 of them, and check that convergence reaches a clean stop with only 1 active P1. The mechanism to do this is fully wired — the `claimAdjudicationGate` in the blocked_stop event checks `activeP0P1` — but no end-to-end measurement of the false-positive reduction rate was automated as a metric.

- **Next experiment:** Run a deep-review on a known target with seeded false-positive P1s, capture `claimAdjudicationGate` in the JSONL `blocked_stop` events, and compute the precision: (# initially-asserted P0/P1) / (# surviving after adjudication). Measure across 5+ review runs for a distribution.

- **Verdict:** The protocol is well-specified and wired into the convergence gate, but no automated false-positive reduction metric exists. The infrastructure supports the measurement — it just hasn't been run systematically. Recommendation: add a `claimAdjudicationGate` metric to the convergence report that surfaces the adjudication precision (asserted → confirmed ratio).

### Angle 34: Convergence detection — is the 0.10 threshold well-calibrated?

- **Framing:** Testable claim — the `convergenceThreshold` of 0.10 for deep-review either stops loops too early (causing missed findings) or too late (wasting iterations), relative to deep-research's 0.05.

- **Repo evidence:** `.opencode/skills/deep-review/references/convergence/convergence.md:42-63` — `convergenceThreshold` = 0.10 compares severity-weighted new findings against accumulated findings (P0=10, P1=5, P2=1). The composite convergence uses a 3-signal weighted vote (convergence.md:204-258): Rolling Average (0.30, threshold 0.08), MAD Noise Floor (0.25), Dimension Coverage (0.45). The composite needs `stopScore >= 0.60`. Sibling defaults: deep-research = 0.05, deep-review = 0.10, deep-ai-council (proposed) = 0.20 (convergence.md:58-64). The document explicitly warns: "Carrying threshold expectations across siblings will cause unexpected iteration counts."

- **Findings:** The 0.10 threshold for deep-review is *twice* the sensitivity of deep-research's 0.05, meaning the review loop stops earlier — it accepts a 10% new-findings ratio where research would keep going to 5%. This asymmetry makes sense: review findings are severity-weighted (a single P0 finding dwarfs many P2 refinements), so the signal-to-noise ratio is naturally sharper. However, the composite convergence score uses `rollingStopThreshold` = 0.08 (not 0.10) for the rolling average signal, and the MAD noise floor has no fixed threshold — it compares against the statistical noise floor. The 0.10 `convergenceThreshold` itself appears in the description but not directly in the composite formula — it feeds the `newFindingsRatio` computation that the rolling average and MAD consume. There is also a `semanticNovelty` signal (convergence.md:260-284) that catches the case where `newFindingsRatio` stays moderate due to severity-weighted refinements but semantically new defect categories are exhausted.

- **Next experiment:** Compute the distribution of stopping iteration numbers across real review runs. For each run that stopped at convergence (not maxIterations), retroactively compute what the stop would have been at thresholds 0.05, 0.08, 0.10, 0.15. Count how many runs would have stopped earlier (1+ iterations saved) vs later (1+ iterations gained) at each threshold.

- **Verdict:** The 0.10 threshold is plausibly well-calibrated for review (vs 0.05 for research), given the severity-weighted signal and the multi-signal composite that gates on rolling average (0.08) and MAD noise. However, no empirical calibration data exists in the shipped code. Recommendation: add an auto-tuning experiment that collects actual iteration-count vs threshold curves across stored runs.

### Angle 21: Deep-loop fan-out — real speedup of async concurrency vs serial spawnSync

- **Framing:** Testable claim — the shipped async-concurrency remediation (replacing `spawnSync` with `spawn` + `runCappedPool`) achieves roughly K× speedup for K concurrent executor lineages vs the prior serial path.

- **Repo evidence:** The original synchronous path: `executor-audit.ts:596` — `const result = spawnSync(input.command, input.args, {...})` on the single-executor path. The async path: `fanout-run.cjs:186-251` — `runLineageProcess` uses `spawn` (not spawnSync) inside a Promise, with explicit commentary at lines 173-176: "Resolves a spawnSync-shaped result... so the pool's K-in-flight cap actually overlaps lineages. spawnSync blocked the single thread until each child exited, serializing every lineage regardless of the concurrency cap." The pool: `fanout-pool.cjs:138-199` — `runCappedPool` uses a pump pattern with at most `concurrency` items in flight at once. The fan-out config defaults concurrency to 2 (`executor-config.ts:306`). The test: `fanout-run.vitest.ts:315-353` — two ~1s sleep stubs run concurrently with concurrency 2, asserting wall-clock < 1.9s (serial would be ~2s).

- **Findings:** The remediation shipped and is tested. The `runLineageProcess` function correctly yields the event loop with `await` on each spawned child, enabling `runCappedPool` to achieve genuine K-at-a-time concurrency. The test proves 2× speedup for 2 concurrent lineages (~1s vs ~2s). The concurrency mechanism also applies to the council multi-seat dispatcher (`council/multi-seat-dispatch.cjs`), which uses the same pool pattern. The `fanout-run.cjs` script also handles per-kind state-dir isolation (`SPECKIT_STATE_ENV_BY_KIND`, line 436-441) so same-kind replicas don't collide on lockfiles. However, the measured speedup is only validated at K=2 with sleep stubs — no measurement against real executor costs (network-bound API calls, model inference latency) exists.

- **Next experiment:** Run a real fan-out with K=3 vs K=1 on a research task that takes ~60s per iteration per lineage. Measure wall-clock time and compute the speedup ratio. Profile the overlap ratio: (sum of individual lineage durations) / wall-clock time. Repeat for K=2, 3, 4 to find the effective concurrency ceiling (beyond which network/model contention dominates).

- **Verdict:** The async concurrency remediation shipped correctly — the test proves overlap. The expected speedup is K× for truly independent, I/O-bound lineages, but the shipped test only measures a stub. Recommendation: add a CI benchmark that measures real fan-out speedup at K={1,2,3} against a lightweight echo-based baseline to catch regressions in the pool/pump mechanism. **LOW CONFIDENCE** on real-world speedup magnitude — needs the proposed measurement.
