# Iteration 004 — Track A (mimo)

## Focus
Widen validator mechanical reach (stability-gating beyond colors; banned-word check; split values-vs-claims score; font allowlist)

## Findings
1. **[P0] A6: stability-gating blind to non-color tokens** — checkStabilityGating (validate.ts:100-123) only iterates tokens.colorTokens for L4 detection. ShadowToken (types.ts:448), RadiusToken (455), ComponentGroup (461), and TypographyLevel (440) all carry stability?:StabilityClassification with layer:'content' possible, but the validator never checks them. A fabricated 'layered depth system' built on a one-off image-derived shadow (L4) passes validation because the shadow's CSS string is real — checkPhantomColors doesn't trace shadow strings at all, and checkStabilityGating ignores shadows. Gradients (types.ts:394-399) lack stability fields entirely despite cluster.ts classifying color stability for gradient-sourced colors.
   - Recommendation: 1) In types.ts, add stability?:StabilityClassification to the gradients array items in DesignTokens (line 394). 2) In validate.ts, rename checkStabilityGating to checkStabilityGatingColors (keep existing logic). Add three new check functions: checkShadowStabilityGating (scan md for box-shadow CSS strings, cross-reference tokens.shadowTokens where stability.layer==='content' → fail), checkGradientStabilityGating (scan md for linear-gradient/radial-gradient CSS strings, cross-reference tokens.gradients where stability.layer==='content' → fail), checkComponentStabilityGating (scan component section prose for component type names, cross-reference tokens.components where stability.layer==='content' → fail). 3) Register all four in validateDesignMd. 4) Add 'content-shadow', 'content-gradient', 'content-component' to CRITICAL_FAILURE_TYPES (line 372).
   - Evidence: validate.ts:100-123 (checkStabilityGating loop), types.ts:443-462 (stability fields on Shadow/Radius/Component), types.ts:394-399 (gradients missing stability), cluster.ts:539-563 (classifyShadowStability/classifyRadiusStability/classifyComponentStability exist)
2. **[P0] A7: no code enforces banned-word / interpretive-adjective discipline** — quality_checklist.md DQ-01 (line 193) lists 14 banned words: sleek, modern, clean, minimalist, beautiful, stunning, elegant, seamless, intuitive, cutting-edge, state-of-the-art, leverage, utilize, robust. anti_patterns.md AP-02 (line 62) adds: professional, polished, sophisticated. AP-09 (line 233) adds filler phrases: 'Let's dive into', 'Moving on to', 'Now let's look at', 'As we can see', 'It's worth noting'. NO validate.ts function checks any of these. The validator passes prose like 'A clean, modern design with elegant typography' with zero friction.
   - Recommendation: Add checkProseDiscipline(md): {passed, warnings}. 1) BANNED_ADJECTIVES = new Set(['sleek','modern','clean','minimalist','beautiful','stunning','elegant','seamless','intuitive','cutting-edge','state-of-the-art','leverage','utilize','robust','professional','polished','sophisticated']). 2) FILLER_PHRASES = ['let\'s dive into','moving on to','now let\'s look at','as we can see','it\'s worth noting','without further ado']. 3) For each banned word occurrence, check the surrounding sentence (±50 chars) for a numeric value (\d+px|\d+rem|#[0-9a-f]{6}|\d+:\d+|\d+\.\d+). If a numeric anchor is present in the same sentence, suppress the warning (this is value-bound prose, not empty filler). 4) For filler phrases, always warn — they carry zero design information. 5) All matches produce warnings (type:'banned-word' or 'filler-phrase'), not failures — preserving the design doc's ability to use these words when grounded in data. 6) Register in validateDesignMd with weight 1pt per warning, matching existing warning scoring.
   - Evidence: quality_checklist.md:192-195 (DQ-01), anti_patterns.md:46-62 (AP-02), anti_patterns.md:233-234 (AP-09 filler), validate.ts:287-310 (checkContent exists but only checks typography table and color count)
3. **[P0] A10: single score conflates value fidelity with prose provenance** — validateDesignMd (line 347) computes score = 100 - failures*5 - warnings*1. A DESIGN.md with all real hexes, complete sections, and proper format — but prose full of 'A sleek, modern design system with elegant depth principles' — scores 99/100 and passes (≥80, no critical failures). The score gives no signal that the prose is unverified filler. There is no mechanical check that prose claims (color roles, typography rationale, depth philosophy) trace back to actual token data. The prose-provenance gap is the primary hallucination vector the validator misses.
   - Recommendation: Split into two scores in ValidationResult: { valuesScore: number, claimsScore: number, score: number }. valuesScore = existing scoring logic (hex presence, sections, format, stability gating — all mechanical). claimsScore = 100 minus penalties from: (a) checkProseDiscipline warnings (A7 above), (b) checkProseClaims — scan descriptive paragraphs for numeric assertions (hex, px, rem, ratio) and verify each against tokens; each unverified assertion is a failure (-5), (c) checkSemanticGrounding — verify color role names in prose ('primary', 'accent', 'surface') map to actual token roles, not invented labels. score = Math.round(valuesScore * 0.6 + claimsScore * 0.4). Update isPass to require both valuesScore ≥ 80 AND claimsScore ≥ 60. Add 'unverified-claim' and 'ungrounded-semantic' to failure types (not critical — they affect claimsScore only, not hard-fail).
   - Evidence: validate.ts:347 (score calculation), validate.ts:378-380 (isPass: score≥80 && !critical), quality_checklist.md:203-206 (DQ-03: characteristics must include specific values), anti_patterns.md:116-129 (AP-05: over-inference from single occurrence)
4. **[P1] A12: checkUnknownFonts permissive skip-list masks fabricated fonts** — checkUnknownFonts (validate.ts:125-178) has 15+ skip heuristics. Line 171 has an explicit allowlist of 'common system fonts not always in extraction': Georgia, Times New Roman, Arial, Helvetica, Verdana, Courier, Inter, Fira Code, JetBrains Mono, DM Sans, Noto Sans, Roboto, system-ui. A doc that claims 'Typography uses Helvetica Neue for headings' would pass if 'Helvetica' is in the skip-list — the 'Neue' suffix would still be caught, but 'Helvetica' alone would not. More critically, the 15 heuristic skips (no uppercase → skip, looks like CSS → skip, camelCase → skip, ALL-CAPS → skip, quoted → skip, underscore → skip, comma-separated → skip) create a wide bypass surface. A fabricated font like 'Source Serif Pro' would NOT be caught if it appeared without backticks. However, the actual risk is moderate: the heuristic requires backtick wrapping AND uppercase letters AND reasonable length, so random fabrication is caught. The real gap is that the skip-list at line 171 treats 'Georgia' as always-valid even when the extracted data has zero Georgia usage.
   - Recommendation: Invert the check to strict-allowlist mode: 1) knownFonts = union of tokens.fontInfo.fontFaces + tokens.fontInfo.loadedFonts + tokens.cssVariables font-family values (already computed lines 128-133). 2) SYSTEM_FONT_ALLOWLIST = ['system-ui','-apple-system','BlinkMacSystemFont','Segoe UI','Roboto','Helvetica Neue','Arial','Noto Sans','sans-serif','Apple Color Emoji','Segoe UI Emoji','Times New Roman','Georgia','Courier New','Courier','monospace','serif']. 3) For each backtick-wrapped candidate passing the existing heuristic filters (lines 145-169), check: if normalized ∈ knownFonts → pass. Else if normalized ∈ SYSTEM_FONT_ALLOWLIST → pass. Else → warn. 4) This eliminates the 'Georgia always passes' risk: if tokens show zero Georgia usage AND the doc claims Georgia, it warns. The existing heuristic skips (no uppercase, CSS property, etc.) remain as pre-filters to avoid false positives on non-font backtick content. Severity remains warning (not failure) because font names are lower-risk than phantom hexes.
   - Evidence: validate.ts:125-178 (full checkUnknownFonts), validate.ts:171 (explicit system font skip-list), types.ts:362-367 (fontInfo structure with fontFaces + loadedFonts)

## Questions Answered
- Can checkStabilityGating be mechanically extended beyond colorTokens? Yes — ShadowToken/RadiusToken/ComponentGroup/TypographyLevel all carry stability fields; gradients need the field added.
- Can banned-word discipline be enforced without false positives? Yes — sentence-level numeric-anchor detection suppresses warnings for value-bound prose while catching empty filler.
- Can the single score be split without breaking existing consumers? Yes — ValidationResult gains valuesScore/claimsScore alongside the existing score field; isPass adds a claimsScore gate.
- Is checkUnknownFonts' skip-list a real fabrication risk? Moderate — the explicit system-font allowlist masks legitimate 'font not in extraction' warnings, but the heuristic pre-filters catch most random fabrication.

## Summary
The validator's mechanical reach is narrow: stability gating only covers colors (A6), banned-word discipline exists only as prose checklists with zero code enforcement (A7), and the single score lets prose-only hallucinations pass at 99/100 (A10). The font check's explicit skip-list is a moderate risk (A12). All four findings have concrete mechanical fixes — the banned-word check uses sentence-level numeric anchors for false-positive suppression, the dual score splits value fidelity from prose provenance, and the stability extension reuses existing stability classification infrastructure already present in cluster.ts.
