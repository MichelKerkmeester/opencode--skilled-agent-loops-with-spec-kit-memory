# Iteration 6: Regression-Safe Canonical-First Remediation Ranking

## Focus

This iteration ranked remediation candidates for a universal canonical-first contract without implementing them. “Universal” retains the prior narrow meaning: canonical wins unqualified ambiguity, explicit qualified paths remain authoritative, and unique legacy-only packets remain readable during migration. Ranking is by regression-adjusted coverage: direct failure removal first, then broader precedence alignment, then removal of the compatibility alias.

## Findings

1. **The evidence-ranked candidates are: (1) call-site-specific automatic-writer fixes; (2) canonical-first shared-resolver order with legacy fallback; (3) tracked-symlink retirement after dependency removal; (4) a relative symlink only as a temporary compatibility bridge; and (5) canonical-only cutover, rejected.** Rank 1 should canonicalize the Stop hook's generated packet target, make post-save phase-pointer refresh consume the already resolved absolute packet, and make untracked `spec/create.sh` creation canonical by default. It has the smallest behavior blast radius and closes failures that resolver reordering cannot: the Stop route preserves an explicit `specs/...` argument before shared fallback, phase-pointer refresh re-resolves that original argument, and `spec/create.sh` constructs `specs/` directly. Its known regression is only for callers relying on *implicit* legacy writes; deliberate explicit legacy paths should remain supported behind migration-only compatibility. Rank 2 aligns remaining unqualified scripts-side selection, but changes the winner for divergent duplicate relative IDs, so it requires a duplicate inventory and collision gate first. Rank 3 removes clone/worktree/plain-file dependence only after ranks 1–2 prove the alias is unnecessary. Rank 4 repairs the current absolute payload but cannot support `core.symlinks=false`, so it is a bridge, not an endpoint. Rank 5 would orphan unique legacy-only packets and is not regression-safe. 2. **Data migration is a prerequisite classification pass, not a bulk rewrite.** A read-only inventory should classify every relative packet identity as (a) canonical only, (b) legacy only, (c) both spellings resolving to the same inode, or (d) divergent duplicates. Class (c) needs no data movement; class (b) remains readable through fallback and can be migrated packet-by-packet; class (d) requires an explicit winner or merge before canonical-first precedence changes; class (a) is already compliant. Existing packet IDs and workflow folder identities do not need root-spelling rewrites because inspected identities are metadata-backed or root-relative. The prerequisite is therefore an auditable collision/move manifest with packet-local validation and backup locations, not a database-wide or metadata-wide ID conversion. [SOURCE: .opencode/specs/system-speckit/000-migration-from-soa-and-cleanup/008-spec-root-resolution-hardening/research/iterations/iteration-002.md:11-17,19-23] [INFERENCE: the four migration classes follow from same-inode deduplication, retained fallback, and the duplicate-winner regression]

3. **The regression-safe rollout is staged and each stage must be independently reversible.** Stage A builds fresh source/dist artifacts and adds the dual-root fixtures before behavior changes; Stage B applies the three rank-1 call-site fixes while preserving explicit-root inputs; Stage C runs the migration inventory and resolves divergent duplicates; Stage D reorders only the shared default list to canonical-first while retaining legacy enumeration/fallback; Stage E proves all writer/reader scenarios with the alias absent; only then may Stage F remove the tracked alias or document it as unsupported local convenience. This order prevents a resolver-only patch from falsely appearing complete and prevents symlink removal from exposing known bypass paths. 4. **Rollback must restore behavior and data separately; restoring the symlink alone is not a valid rollback.** Keep call-site changes, resolver order, and alias policy in separable changes so Stage B can revert generated target behavior and Stage D can restore legacy-first precedence without undoing unrelated fixes. Before moving any class-(b) or merging class-(d) packet, preserve a manifest and recoverable packet copy; rollback then restores the packet at its recorded root and returns precedence to legacy-first. Re-adding a relative alias can be a temporary Unix-compatible bridge, but it cannot recover symlink-disabled checkouts, where Git materializes link text as a plain file, and it must never be used to undo or mask migrated data. [SOURCE: .opencode/specs/system-speckit/000-migration-from-soa-and-cleanup/008-spec-root-resolution-hardening/research/iterations/iteration-003.md:12-15,22-24] [SOURCE: .opencode/specs/system-speckit/000-migration-from-soa-and-cleanup/008-spec-root-resolution-hardening/research/iterations/iteration-004.md:21-27,31-35] [INFERENCE: separable behavior rollback plus manifest-backed packet restoration is required because alias state and packet location fail independently]

5. **The acceptance matrix must cross root state, packet state, and entrypoint, and must assert both write location and reader visibility.** Root states: valid relative alias, absent alias, dangling alias, distinct plain `specs/` directory, and symlink-disabled plain file. Packet states: canonical-only, legacy-only, same-inode alias, unique leaf, duplicate leaf, and divergent duplicate relative ID. Entrypoints: Stop autosave, bare/canonical-qualified/legacy-qualified `generate-context`, phase-child save with parent-pointer refresh, tracked and untracked `spec/create.sh`, scripts-side detection, MCP indexing/resume, and Gate-3 binding. Required assertions are: implicit writes land canonically; explicit legacy input remains explicit during the migration window; legacy-only reads remain available; divergent duplicates fail preflight rather than silently switch; child and parent metadata update the same resolved packet; MCP and scripts observe the intended packet; and every applicable case passes with and without the alias. Fresh compiled artifacts are a prerequisite because the configured Stop route executes dist, while iteration 5 established parity only for the previously inspected branches, not global dist freshness. ## Reflection

- What worked and why: Reusing exact iteration 2–5 anchors avoided blocked broad scans and made candidate ranking depend on demonstrated bypass paths, duplicate behavior, and portability failure rather than patch size.
- What did not work and why: Treating the shared resolver reorder as the obvious first patch fails coverage analysis because the highest-severity automatic writers either preserve an explicit legacy path or construct one directly.
- What I would do differently: Before implementation, generate the collision manifest first so the resolver-order decision can be measured against actual duplicate and legacy-only counts rather than only control-flow risk.
