- **NEW — verdict CHANGED/REFUTED (supersedes the iteration 1-6 engine-gap list): `PRAGMA synchronous = NORMAL` IS accepted and honored at v0.7.0-pre.6.** The pragma write handler parses `"0"/OFF → SyncMode::Off`, `"1"/NORMAL → SyncMode::Normal`, anything else → `SyncMode::Full` (comment: "SQLite defaults to NORMAL for invalid values, but we want to default to a higher durability level so deviating here"), then calls `connection.set_sync_mode(mode)`; the read handler returns the stored mode. `SyncMode {Off=0, Normal=1, Full=2}` is defined in core, and `get_sync_mode()` is consulted in the pager's commit/sync path, so the value is plumbed into I/O behavior, not merely stored. The previously established "synchronous=NORMAL unsupported" engine gap no longer holds at the vendored version. [SOURCE: external/turso-main/core/translate/pragma.rs:650-672, 1499-1506; core/lib.rs:335-339; core/storage/pager.rs:2981,2999]
- **NEW — the question's premise is doubly moot: `ensure_vector_shard_schema` never sets `synchronous` at all.** Its pragma sequence on the attached shard schema is exactly `journal_mode = WAL`, `cache_size = -8192`, `mmap_size = 33554432`, `temp_store = DEFAULT`, `wal_autocheckpoint = 256` — no synchronous pragma. `synchronous = NORMAL` is issued only once, on the main connection at open time, alongside `busy_timeout/foreign_keys/cache_size/mmap_size/wal_autocheckpoint/temp_store`. [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:498-503, 1579-1586]
- **NEW — degradation analysis even under a hypothetical OFF/FULL-only engine: graceful, non-fatal, durability-up.** Because Turso's pragma parser coerces unrecognized synchronous values to `SyncMode::Full` rather than erroring, a `.pragma('synchronous = NORMAL')` call can never throw out of the init sequence — worst case is silent escalation to FULL (slower commits, higher durability), not a failed bootstrap. Combined with the previous finding, the spec-kit init sequence has no synchronous-related failure mode on Turso at v0.7.0-pre.6. [SOURCE: external/turso-main/core/translate/pragma.rs:656,667]
- **PARTIAL — residual uncertainty: schema-prefixed pragmas in the shard path.** `ensure_vector_shard_schema` issues schema-qualified pragmas (`<schema>.journal_mode = WAL` etc.) against an ATTACH-ed shard; this iteration verified the connection-level `synchronous` path but did not verify Turso's handling of the `schema.` prefix on pragma targets (SQLite scopes journal_mode per-database; Turso's pragma table shows `PragmaFlags::SchemaOpt` only on some pragmas). Earlier iterations' mmap_size/wal-related gaps still apply to this block. [SOURCE: external/turso-main/core/pragma.rs:95-109; vector-index-store.ts:498-503]
- **KNOWN — UNCHANGED: experimental-feature production warnings persist in the vendored manual.** `--experimental-encryption`, `--experimental-views`, `--experimental-multiprocess-wal`, and `mvcc` journal mode each carry "the feature is not production ready so do not use it for critical data right now." Consistent with iterations 1-6 (MVCC experimental, .tshm sidecar opt-in). [SOURCE: external/turso-main/docs/manual.md:156-158, 592]
