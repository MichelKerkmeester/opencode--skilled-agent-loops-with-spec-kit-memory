## Iteration 007 — Ship-then-question & severity (seat opus-4)

**The core distinction this angle forces.** Ponytail bundles two behaviors that map to *different skills*:
1. **"Ship the lazy version + question the requirement in one response, never stall"** — an **implementer** behavior (how to respond while *building*). A reviewer ships nothing, so this cannot live in sk-code-review.
2. **"YAGNI-extremist / deletion before addition"** — has a **reviewer** projection: flag code that *should not exist*, not just code that is *too complex*.

Treating these as one thing is the trap. They go to two different files.

**Q1 — New finding class or severity-lens for over-engineering/YAGNI-violation?**

**No new `findingClass`, and no new severity tier.** Two precise reasons:

- The `findingClass` enum (`SKILL.md` §1 line 98 / `review_core.md` §7: `instance-only | class-of-bug | cross-consumer | algorithmic | matrix/evidence | test-isolation`) is a **fix-scope / propagation axis** — "how widely must the fix spread." A YAGNI finding is almost always `instance-only` by scope; what's distinctive about it is the **action direction** (the fix is *deletion*, not *modification*), which is already carried by the `recommendation` field. Adding "over-engineering" to `findingClass` would conflate two orthogonal axes and is the wrong home. *(This is the cleanest negative result of the iteration.)*

- Severity is **already fully specified**: `code_quality_checklist.md` §7 lines 140-143 state the exact rule the angle proposes — *"P2 default for stylistic duplication/complexity. Escalate to P1 if duplication or complexity introduces behavior/regression risk."* `review_core.md` §2 confirms P2 = "non-blocking improvement … maintainability follow-up," and line 35's escalation rule ("low confidence but high impact → classify toward the higher severity") already lets a reviewer push a risky unneeded-code finding up. So **P2-by-default, P1-if-it-adds-risk is the existing contract** — nothing to add.

What *is* genuinely missing is narrower: KISS asks **"is this the simplest form of the thing?"** It does not crisply ask **"should this newly-added thing exist at all, given the stated requirement?"** A feature can be *simply implemented yet entirely unrequested* — it slips between KISS (simple enough → no flag) and `removal_plan.md` (which is scoped to **pre-existing dead code** with "no active consumers," not to fresh diff additions that *do* have a consumer but no requirement). That needed-ness-of-new-additions question, wired to a **removal** recommendation (not a "simplify" note), is the only real gap.

**Q2 — Does "ship-then-question" belong in sk-code, not sk-code-review?**

**Yes — sk-code, the implementer.** Exact home: `sk-code/SKILL.md` §4 RULES, adjacent to the Escalation Discipline / Four-Laws bullets that iteration-001 F5 already pointed at. But there's a real tension the angle surfaces: ponytail's **"never stall"** collides head-on with sk-code's **SCOPE-LOCK** (Four Laws #2) and the repo **Logic-Sync HALT** ("HALT → Report → ask which truth prevails"). The reconciliation: *ship the simplest correct implementation of the requirement **as specified**; if part of the requirement looks unnecessary, surface a scope-amendment recommendation in the **same** response — never silently cut scope, and don't block solely to ask when a safe minimal version exists.* That keeps ponytail's anti-stall value without breaching SCOPE-LOCK. This **sharpens iteration-001 F5** (which routed "challenge the requirement" through escalation) by adding the anti-stall dimension — I flag the partial overlap honestly.

**Q3 — Already covered? (negative knowledge, stated plainly)**

**Largely yes — ~65-70% redundant.**
- Implementer-side over-engineering avoidance: covered by the repo-root **CLAUDE.md ANTI-PATTERNS table** (over-engineering, gold-plating, scope creep, wrong-abstraction — each with a Socratic response like *"Is this solving a current problem or a hypothetical one?"*) and the **ANALYSIS LENSES** (CLARITY, SCOPE).
- Reviewer-side YAGNI: covered by `code_quality_checklist.md` §7 KISS line 130 (*"…added for speculative future use without a current requirement (YAGNI)"*) with the correct severity rule.
- Dead-code deletion: covered by `removal_plan.md`.

The residual, non-redundant delta is only: **(a)** a needed-ness prompt on *new* additions (existence, not just simplicity), **(b)** wiring such findings to a *removal* recommendation via `removal_plan.md` rather than a "simplify" note, and **(c)** the implementer anti-stall rule reconciled with SCOPE-LOCK. Everything else would be duplication — and restating the CLAUDE.md table inside the checklist would create a second drifting source of truth (same mistake iteration-001 F4 warned against).

**Q4 — Concrete wording + placement**

*Reviewer side* — add **one prompt line** to `code_quality_checklist.md` §7 "KISS Checks" (not a new section, not a new class):
> *"New code not traceable to a current requirement (a feature, branch, parameter, or config nothing in the stated scope asked for) — recommend **removal**, not just simplification, and cross-reference `removal_plan.md`. Severity P2 by default; P1 when the unneeded code adds attack surface, contract obligations, or regression/maintenance risk (per §7 severity guidance + `review_core.md` §2 escalation)."*

And one review prompt under the same section:
> *"Was this code asked for? If the requirement were dropped, would anything break? If not → removal candidate."*

*Implementer side* — add **one bullet** to `sk-code/SKILL.md` §4 RULES (✅ ALWAYS, near Escalation Discipline):
> *"Build the simplest correct implementation of the stated requirement — don't stall. If part of the requirement looks unnecessary (YAGNI), implement it as specified **and** raise a scope-amendment recommendation in the same response; never silently cut scope (SCOPE-LOCK), never block solely to ask when a safe minimal version exists."*
