# Iteration 001 — Track A (deepseek)

## Focus
Concrete validator+extraction fixes for the two known prose hallucinations (focus-consistent hard check; named-principle token-binding; claims-to-token provenance mechanism)

## Findings
1. **[P0] focus-consistent-fabrication** — extractFocusIndicator returns {consistent:true} when focusStyles.length===0, i.e. when NO interaction data was captured. Interaction capture is OFF by default (extract.ts:191: '--no-interaction … default: skipped'; --fast forces it off). The empty style + true boolean survives through to DESIGN.md §9 Focus Indicators, where the AI writes 'focus indicator is consistent' with zero evidence. validate.ts has NO check that reads the Accessibility Contract prose against a11yTokens.focusIndicator.
   - Recommendation: THREE changes: (1) In a11y-extract.ts:88-103, change the zero-length branch to return `{ style: {}, consistent: false, captured: false }` — add a `captured` field to A11yTokens.focusIndicator (types.ts:271-274). (2) In validate.ts, add a new function `checkA11yFocusProvidence(md, tokens)` called from `validateDesignMd` that: locates the §9 Focus Indicators subsection text; scans for prose describing focus styles or using 'consistent'; if any exists, gates on `tokens.a11yTokens.focusIndicator.captured===true && Object.keys(tokens.a11yTokens.focusIndicator.style).length>0`. If captured===false, ANY focus description beyond 'Not observed' is a failure. (3) If the word 'consistent' appears, additionally require `consistent===true` vs `===false` — but with the gate above, consistent can only be true when there ARE focus styles (the new code only sets consistent true when styles exist).
   - Evidence: a11y-extract.ts:101-103 (returns consistent:true on empty array); extract.ts:191 (interaction capture skipped by default); validate.ts:318-350 (6 checks, none inspecting focus prose); design_md_format.md §13 Focus Indicators (line 796-803, prose format section); types.ts:271-274 (A11yTokens.focusIndicator.consistent: boolean with no captured flag)
2. **[P0] depth-principle-solicited-no-binding** — Two format authorities conspire to demand an unverifiable named principle: (A) design_md_format.md §9 (Depth & Elevation, line 544) REQUIRES a 'Named principle — state the depth philosophy upfront using one of these categories (or a custom name)'. The 'or a custom name' clause explicitly authorizes invention. (B) writing_style_guide.md §16 (Named Principle Pattern, lines 589-631) provides Pattern B '[technique]-as-[metaphor]' with 10+ examples cross-referenced from the Shadow Philosophy Pattern (§12) — together these solicit 'gradient-as-depth', 'chromatic-depth', 'shadow-as-border' as the expected voice. The AI, obligated to produce a named principle, invents 'gradient-as-depth' even when tokens.gradients[] is empty or has zero depth-relevant entries. validate.ts' 6 checks never read §6 prose for truth against tokens.shadowTokens[] or tokens.gradients[].
   - Recommendation: TWO changes: (1) In validate.ts, add function `checkDepthPrincipleBinding(md, tokens)` that: extracts the named principle from §6 prose (first bold phrase after 'Named principle' or first bold term in the section); maps principle keywords to token requirements — 'gradient' → `tokens.gradients.length > 0` else FAIL; 'shadow[-as]' → `tokens.shadowTokens.length > 0` else FAIL; 'chromatic' → at least one `shadowTokens[]` value contains non-neutral RGBA (e.g., not pure (0,0,0,x) or (0,0,0,x)); for ANY named principle, require at least one `shadowToken.type==='elevation'` or FAIL with 'principle references depth but no elevation-type shadows found'. (2) In design_md_format.md §9 line 544, change '(or a custom name)' to '(or a custom name derived from and traceable to a specific token entry)', and add a sub-rule: 'A named principle MUST be traceable to a concrete token entry via an inline source comment or validator-verifiable pattern.'
   - Evidence: design_md_format.md §9:534-575 (named principle REQUIRED, line 544 authorizes custom names); writing_style_guide.md §16:589-631 (Pattern B solicits 'technique-as-metaphor'); writing_style_guide.md §12:473-487 (Shadow Philosophy Pattern reinforces naming); validate.ts:318-350 (zero depth-prose checks); tokens.json DesignTokens.type:394-399 (gradients array — empty when no gradients extracted)
3. **[P1] claims-to-token-provenance-gap** — All 6 validate.ts checks are mechanical: hex-tracing (phantom-color, stability-gating), font-name-tracing (unknown-fonts), format linting (format-consistency), section heading counting (section-completeness), and lightweight content inspection (content-checks: typography table existence, color count threshold). NONE reads a prose SENTENCE to verify the claim it makes. Both known hallucinations ('focus indicator is consistent', 'gradient-as-depth replaces shadow elevation') passed the 99/100 score because no check maps a prose assertion to a token entry. The WRITE phase has no mechanism to declare which token(s) back each claim, so validation cannot cross-check.
   - Recommendation: Adopt INLINE `<!-- source: tokens.x -->` sentinels over a claims.json sidecar. Rationale: (a) validate.ts already has `stripHtmlComments()` at line 51-53 — stripping is an existing, working path; (b) single-file provenance — no sidecar to desync; (c) the sentinel lives adjacent to the claim it grounds, making manual review possible. Implementation: (1) WRITE-phase prompt must embed `<!-- source: tokens.<path> -->` before every claim-containing sentence (e.g., `<!-- source: tokens.a11yTokens.focusIndicator -->`). (2) Add `checkSourceProvenance(md, tokens)` to validate.ts that: parses all html comments in raw md (before stripHtmlComments); extracts `source: tokens.<path>` entries; resolves each dotted path against the DesignTokens object; verifies the referenced key exists and its value is non-empty/non-null; fails any claim whose source path leads to an empty array, zero-length object, or falsy key that is semantically required (e.g., `consistent:false` without `captured:true`). This would have caught BOTH cases: the focus claim would reference `tokens.a11yTokens.focusIndicator.style` which is empty `{}`, and the gradient-depth claim would reference `tokens.gradients[]` which would be empty `[]`.
   - Evidence: validate.ts:51-53 (stripHtmlComments exists); validate.ts:318-350 (6 checks, zero sentence-level verification); cardinal_rules_card.md:33-44 (fidelity checklist has zero prose-verification items); design_md_prompt_template.md:40-56 (cardinal rules mention only value-copy fidelity, not claim-to-token binding)

## Summary
Two P0 bugs and one P1 structural gap enable prose hallucinations that sail past a 99/100 validator score. The extraction fabricates `consistent:true` when interaction capture is OFF (the default), and the format+style specs actively require a named depth principle with no token binding. The validator's six checks are entirely mechanical (hex-tracing, heading-counting, format-linting) and never read a prose sentence against token ground truth. The recommended fix for all three is: add sentence-level provenance checks to validate.ts — structurally for focus consistency and depth principles, and systematically via inline `<!-- source: tokens.x -->` sentinels for general claim-to-token binding.
