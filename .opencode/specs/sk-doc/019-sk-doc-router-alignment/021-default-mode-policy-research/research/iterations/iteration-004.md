# Iteration 4: Fable-5 diverse-lens review

## Focus

Independent verification and adversarial challenge of iterations 1-3 by a different model lens. All primary artifacts were re-read from scratch (seven `hub-router.json` files, seven parent `SKILL.md` routing contracts, five mode registries, the replay implementation, the create-skill template and schema canon, the shipped route-gold fixtures), and the git history of every `defaultMode` was mined — a lens the SOL iterations did not use. The challenge targets: (a) the 1-keep/4-null split, (b) the defer-as-burden-of-proof prior, (c) the `zeroSignalPolicy` encoding, and (d) what SOL missed.

## Actions Taken

1. Re-verified iteration 1's replay-semantics claims directly against `router-replay.cjs` (telemetry construction, intent selection, hub-router projection, resource assembly, surface-router bypass).
2. Re-read all seven `hub-router.json` files and cross-tabulated three fields SOL treated in isolation: `defaultMode`, `defaultResource`, and the placement of the `hub-identity` vocabulary class.
3. Re-read the five auto-defaulting parents' `SKILL.md` routing contracts plus both null controls, checking specifically what each contract says happens on a true zero-signal request.
4. Ran `git log -S '"defaultMode"'` per hub and `--grep` sweeps to date the introduction of each default and find any deliberate default-mode decisions.
5. Read the create-skill hub-router template and `parent_hub_router_schema.md` as they exist today, then dated them against the SOL iterations (file mtimes 18:28-18:56 on 2026-07-17 versus commits `5dd0a330a4` 15:50 and `ee592c7290` 17:06 the same day).
6. Located the shipped defer route-gold fixtures and every hub surface router's `DEFAULT_RESOURCE` declaration.

## Findings

1. **Iteration 1's replay semantics are CONFIRMED, line for line.** `defaultApplied = selectedModes.length === 0 && policy.defaultMode != null` [SOURCE: .opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/router-replay.cjs:342]; `workflowMode` stays `null` on zero signal [SOURCE: router-replay.cjs:350]; `deferReason` is `no-mode-scored` / `ambiguous-multi-axis` [SOURCE: router-replay.cjs:343-345]; selection keeps every positive scorer within the delta [SOURCE: router-replay.cjs:476-480]. A named-child default and `null` are replay-indistinguishable except for the `defaultApplied` flag. No disagreement.

2. **`defaultMode` on a keyword-routed hub is NOT merely a zero-signal fallback — it anchors the catch-all vocabulary architecture, and SOL evaluated it as if it were only a fallback.** The template states it outright: catch-all `hub-identity` "belongs on the DEFAULT mode only … hub-generic words then route to the default (or defer)" [SOURCE: .opencode/skills/sk-doc/create-skill/assets/parent_skill/parent_skill_hub_router_template.json:5] and "This is defaultMode; it alone carries hub-identity so hub-generic words route here" [SOURCE: parent_skill_hub_router_template.json:37]. The schema canon repeats the rule and names the exception (detection-routed, `defaultMode: null`, e.g. sk-code) [SOURCE: .opencode/skills/sk-doc/create-skill/references/parent_skill/parent_hub_router_schema.md:225-229]. Iteration 3 cited line 37 of the template yet still characterized the default as a mere "presumption" — the architectural function (giving hub-generic vocabulary exactly one scoring home so specialized modes don't co-fire) was in the cited evidence and dropped from the analysis. In sk-prompt this mechanism is live: `hub-identity` (including the bare keyword `"prompt"`) scores only `prompt-improve` [SOURCE: .opencode/skills/sk-prompt/hub-router.json:17,21], so most "generic" requests never reach the zero-signal branch at all — they route to the default *by scoring*.

3. **Two same-day commits — postdating none of the artifacts SOL read but uncited by SOL — materially change the evidence.** `5dd0a330a4` (2026-07-17 15:50, "stop prompt-models over-ranking the default mode") deliberately re-affirmed sk-prompt's default-as-catch-all design: hub-identity stripped from `prompt-models`, its weight raised 5→6 so a named-model signal decisively escapes the default, route-gold going BLOCKED (0/4) → PASS (4/4). `ee592c7290` (17:06, "stop the hub-router scaffold minting over-emitting routers") fixed the template after over-emission from shared hub-identity "just blocked the route-gold gate on six shipped hubs." [SOURCE: git log/show for both commits] The SOL iterations (written 18:28-18:56 the same day per file mtimes) cite the *post-fix* weight of 6 as static config [iteration-002.md finding 2] without noting it was minted hours earlier as a deliberate default-protection decision. Consequence: sk-prompt's KEEP is not "provisional pending corpus" — it is the single default in the fleet with fresh, deliberate, benchmark-receipted evidence behind it.

4. **No hub in the fleet auto-executes a child on zero signal — the live authored semantic is already universally "defer, optionally with a suggestion." This reframes the whole question.** Per contract: sk-prompt's fallback loads `prompt-improve/SKILL.md` *as context* and returns `UNKNOWN_FALLBACK` [SOURCE: .opencode/skills/sk-prompt/SKILL.md:70-72]; cli "does not silently default to cli-opencode on genuine ambiguity" [SOURCE: .opencode/skills/cli-external-orchestration/SKILL.md:60]; deep-loop returns a disambiguation checklist when no mode dominates [SOURCE: .opencode/skills/system-deep-loop/SKILL.md:55-56]; mcp-tooling declares its default "the defer-time suggestion offered alongside disambiguation" [SOURCE: .opencode/skills/mcp-tooling/hub-router.json:24; SKILL.md:59,65]; sk-design requires intake before any route and defers below confidence 0.5 [SOURCE: .opencode/skills/sk-design/SKILL.md:53-55,96-103]. So "keep vs flip" is not a choice between auto-routing and deferring — nobody auto-routes. The honest fleet semantic of a non-null `defaultMode` is *defer-time suggestion pointer*; the decision per hub is whether that suggestion is worth keeping, plus (finding 2) whether the hub needs a catch-all scoring anchor.

5. **The five defaults split into "resource-backed" and "vestigial" — a split SOL never made, and it is the strongest evidence for two of the four flips.** Backed: sk-prompt (`defaultResource: ["prompt-improve/SKILL.md"]` [SOURCE: sk-prompt/hub-router.json:13]), cli (`["cli-opencode/SKILL.md"]` [SOURCE: cli-external-orchestration/hub-router.json:13]), mcp-tooling (`["mcp-chrome-devtools/SKILL.md"]`, fallback-only [SOURCE: mcp-tooling/hub-router.json:20-24]). Vestigial: system-deep-loop's `defaultResource` is the neutral hub `README.md` — not the research packet [SOURCE: system-deep-loop/hub-router.json:13]; sk-design's is four neutral `shared/` vocabulary docs — not the interface packet [SOURCE: sk-design/hub-router.json:13-18]. For deep-loop and sk-design the named default neither routes (prose defers), nor scores (no default-only identity class), nor even suggests (defaultResource does not back it). It is fully dead metadata; flipping to null is a zero-behavior-change truth reconciliation. For the three backed hubs, flipping `defaultMode` alone silently *keeps* the child bias in the `defaultResource` channel — SOL's verdicts and its proposed `zeroSignalPolicy` object are both silent on `defaultResource`.

6. **sk-design's default sits on top of a differently broken layout: `hub-identity` is a scoring class on ALL SIX modes — the exact over-emission bug the canon was patched for the same day.** [SOURCE: sk-design/hub-router.json:34,46,57,69,81,88] A hub-generic design prompt does not fall to `defaultMode` at all — it co-fires all six modes within the ambiguity delta (replay: `deferReason: ambiguous-multi-axis`, six selected intents), which `ee592c7290` calls "over-emission that fails the route-gold gate." sk-code shares this layout (hub-identity on all four modes [SOURCE: sk-code/hub-router.json:19,24,29,34]) but is exempt as detection-routed/subset-scored [SOURCE: parent_hub_router_schema.md:229]. sk-design is not exempt. So NULL alone does not fix sk-design; the coherent fix is the mcp-tooling pattern — `defaultMode: null` AND hub-identity moved to discovery-only classes [SOURCE: mcp-tooling/hub-router.json:25-29,94-101] — which matches its intake-first hard constraint. SOL's verdict is directionally right but incomplete as a change-spec.

7. **cli-external-orchestration: NULL is right, and stronger than SOL argued — a static default is *unrepresentable* here.** The correct defer-time suggestion is runtime-dependent: each mode refuses self-dispatch when the current runtime IS the target CLI [SOURCE: cli-external-orchestration/SKILL.md:31], so `cli-opencode` is precisely wrong for every OpenCode-runtime session. No static `defaultMode` value can be correct across runtimes; this is a detection problem (exclude the current runtime's own CLI), not a default problem. Additionally, the hub has no hub-identity/catch-all class at all [SOURCE: cli-external-orchestration/hub-router.json:32-51], so the default protects no scoring architecture; and Layer-0 resolves provider-named traffic before the hub runs (the advisor's executor-delegation scorer sources this hub's registry [SOURCE: cli-external-orchestration/SKILL.md:62-64; mode-registry.json:5]), leaving the hub's zero-signal branch to genuinely provider-less requests where the choice is material. Flip — but also neutralize or fallback-only the `defaultResource` (finding 5).

8. **The zero-signal branch is rare by construction, which undercuts BOTH sides of SOL's expected-loss framing.** Most mode traffic is pre-disambiguated before hub scoring ever matters: per-mode slash commands exist for all seven deep-loop modes and five sk-design modes [SOURCE: system-deep-loop/mode-registry.json command fields; sk-design/mode-registry.json command fields], the advisor resolves executor names at Layer-0 (finding 7), and mode vocabularies are broad enough to catch vague-but-directional asks (bare `"prompt"` [SOURCE: sk-prompt/hub-router.json:21]; `"bolder"`, `"quieter"`, `"clarify"`, `"delight"` as interface-aliases [SOURCE: sk-design/hub-router.json:123-127]). So the residual zero-signal population is small AND genuinely ambiguous — which means the mis-default risk SOL guards against is small, but the disambiguation-annoyance cost SOL's critics would raise is *equally* small. The empirically measured cost driver in this fleet is the opposite branch entirely: over-emission on *scored* routes, which actually blocked route-gold on six hubs [SOURCE: ee592c7290 commit message]. SOL's decision rule spends its rigor on the rarer, cheaper branch.

9. **The `zeroSignalPolicy` encoding is over-specified: roughly 70% of iteration 3's proposed surface already ships.** Defer gold with `workflowMode: null` + `routeOutcome: "defer"` + `forbiddenWorkflowModes` exists today [SOURCE: .opencode/skills/system-deep-loop/deep-improvement/assets/skill_benchmark/fixtures/sk_design_dispatch/sk_design_dispatch_neither_001.private.json; fixtures/deep_loop_workflows/dlw_context_001.private.json]; `routeOutcome`/`workflowMode`/`forbiddenWorkflowModes`/`minimalPairGroup` are documented gold fields [SOURCE: .opencode/skills/system-deep-loop/deep-improvement/references/skill_benchmark/scenario_authoring.md:29-42]; `defaultResourceSemantics: "fallback-only"` exists, parses, and is documented [SOURCE: router-replay.cjs:122-125; skill-benchmark/README.md:183]; replay already emits `defaultApplied` and `deferReason` [SOURCE: router-replay.cjs:342-345]. The named-child-vs-null observability gap closes with ONE new gold assertion (assert `routeTelemetry.defaultApplied` — better first renamed `defaultConfigured`, resolving iteration 1's naming tension) rather than a nine-field `expected_route` tuple. The three-strategy enum duplicates derivable state: `detect-and-bundle` ⇔ surface-axis + `outcomes.surfaceBundle` presence, already bound by canon [SOURCE: parent_hub_router_schema.md:88,229,284]. The simpler deliberate-choice forcing: a validator requiring every hub to match ONE of three archetypes (see finding 10) + `defaultResourceSemantics: "fallback-only"` mandatory whenever `defaultMode` is non-null + one zero-signal route-gold fixture per hub whose gold matches the declared archetype. Fixtures are the evidence burden; `corpusRef`/`costModelRef` prose fields would go stale unverified.

10. **The current canon blesses only TWO archetypes; SOL's 4-null verdict needs a third canonized or the flipped hubs become schema-orphans.** As of `ee592c7290`, canon knows: keyword-routed (non-null default, hub-identity on the default only — sk-prompt) and detection-routed (`defaultMode: null`, mode from surface markers — sk-code) [SOURCE: parent_hub_router_schema.md:85,225-229]. The four flips create keyword-routed hubs with `defaultMode: null`, which the schema table does not describe. mcp-tooling is the live, route-golded proof of the missing third archetype — defer-routed: `defaultMode: null` (post-flip), hub-identity as discovery-only defer evidence, suggestion carried by fallback-only `defaultResource` [SOURCE: mcp-tooling/hub-router.json:20-29]. sk-doc is the same archetype minus the annotations (null default, no identity class at all [SOURCE: sk-doc/hub-router.json:5,36-49]). Canonize defer-routed as pattern three and every verdict in this packet becomes expressible; skip it and `ee592c7290`'s freshly hardened template will quietly pressure future authors back toward named defaults.

11. **Minor confirmed defects worth carrying into implementation.** (a) sk-prompt's routing pseudocode has an unwritten branch: the fallback triggers on "confidence low OR contradictory OR (defaultMode null AND no mode wins)" — with a non-null default and no winner and no low-confidence flag, control falls through the mode loop to a `return single or orderedBundle` with zero modes [SOURCE: sk-prompt/SKILL.md:70-81]. Nearly unreachable (finding 2's catch-all), but the contract hole exists and a null flip — or an explicit non-null branch — would close it. (b) The shipped defer-gold representation is already inconsistent with its own doc: `workflowMode` is `null` in one fixture, `[]` in another, and "a string for `single` and `defer` outcomes" per the authoring doc [SOURCE: sk_design_dispatch_neither_001.private.json; dlw_context_001.private.json; scenario_authoring.md:42]. Normalize before adding fields. (c) All five defaults were introduced by bulk scaffold/merge/rename commits with no per-hub dominance reasoning (`7fe9e7278ca` sk-design, `6323b843425` deep-loop, `1a3c30cbabc` mcp-tooling + cli merge, `c425653a1e4` sk-prompt sweep, `0950bd9f215` cli rename) — confirming SOL's "the configuration is the hypothesis, not evidence" — with the sole exception that sk-prompt's was later deliberately re-affirmed (`5dd0a330a4`).

## Where I Disagree / Refine

1. **The question is mis-framed as keep-vs-flip of an auto-route.** No hub's authored contract auto-executes a child on zero signal (finding 4). The genuine decision is: re-specify non-null `defaultMode` as a *defer-time suggestion* (never a route), or null it. Under that framing "KEEP sk-prompt" and "FLIP mcp-tooling" are almost the same policy — both hubs defer with a suggested child — and SOL's hard 1-vs-4 boundary overstates the behavioral difference.
2. **sk-prompt's keep is under-claimed, and for the wrong reason.** SOL calls it "provisional … still needs traffic validation." It is the one default with deliberate, benchmark-receipted, same-day evidence (finding 3), and its primary function is the catch-all scoring anchor (finding 2), which SOL never analyzed. The corpus SOL demands would validate a branch (zero-signal fallback) that the catch-all architecture makes nearly unreachable.
3. **The defer prior is right, but SOL's expected-loss inequality is the wrong justification.** It is unfalsifiable in practice (no zero-signal corpus exists or is planned) and unnecessary: defer-with-suggestion strictly dominates both naked defer (same safety, better UX) and auto-default (same UX hint, no mis-route) for keyword hubs, no corpus needed. The rigorous-looking inequality lends false precision; the archetype rule plus one fixture per hub is the enforceable version.
4. **`zeroSignalPolicy` (three strategies + new object + nine-field tuple) is over-engineered.** ~70% ships already; one telemetry assertion + one rename + two validator rules + the third archetype achieve the same deliberate-choice forcing with zero new objects (finding 9). The `evidence.corpusRef/costModelRef` fields in particular would become unverified prose.
5. **SOL's verdicts are incomplete as change-specs.** Flipping `defaultMode` alone leaves the cli-opencode bias in `defaultResource` (cli), leaves the six-way hub-identity co-fire (sk-design), and would drop mcp-tooling's authored suggestion were it not independently preserved by fallback-only `defaultResource`. Each flip is a two-field decision (`defaultMode` + `defaultResource`/identity-placement), not one.
6. **SOL missed the hub-identity coupling entirely** despite citing the exact template line that states it (finding 2) — and missed that the fleet's real, measured routing failure (six hubs blocked at the route-gold gate) lives on the scored path the zero-signal analysis never touches (finding 8).

## Open Risks

1. **Archetype gap:** if the defer-routed archetype is not canonized, the four flipped hubs sit outside `parent_hub_router_schema.md`'s two blessed patterns, and the freshly hardened template (`ee592c7290`) will pressure future editors to re-add named defaults "per canon."
2. **Regression invisibility:** until a telemetry assertion lands, re-adding a `defaultMode` to a flipped hub changes nothing observable in current gold (`defaultApplied` only) — the flip cannot defend itself.
3. **sk-prompt catch-all breadth:** the keep leans on the bare keyword `"prompt"` being safe for everything that reaches this hub; only `prompt-models` has a weight-gap escape. Any future third mode must repeat the weight-6 maneuver or the catch-all becomes a mis-router.
4. **Live-agent divergence cuts against complacency on the flips:** an LLM reading `hub-router.json` may treat a non-null `defaultMode` as an auto-route license regardless of the SKILL.md prose, so the deterministic replay (where the default is inert) *understates* live mis-route risk — this is the strongest surviving argument FOR flipping promptly, and it is inferred, not measured. [INFERENCE: no live-trace corpus of hub-level zero-signal requests was found in the packet or benchmark assets]
5. **defaultResource legacy union:** hubs without `defaultResourceSemantics` keep "legacy always-union" replay behavior [SOURCE: skill-benchmark/README.md:183; router-replay.cjs:594-597]; in practice all seven hubs are shielded because replay routes resources through their surface routers [SOURCE: router-replay.cjs:665-689; all seven ship `shared/references/smart_routing.md`], but any hub that loses its surface router silently re-activates the union path.

## Reflection

- What worked: cross-tabulating `defaultMode` against `defaultResource` and hub-identity placement exposed structure invisible when each field is read alone; git dating showed the research and the canon moved on the same day without seeing each other.
- What did not work: no live-trace evidence of hub-level zero-signal frequency exists anywhere in the repo, so finding 8's "rare by construction" remains an inference from entry-surface architecture rather than a measurement.
- What I would do differently: an implementation iteration should start from the archetype validator and the `defaultConfigured` rename — the two smallest deltas that make every verdict in this packet both expressible and regression-guarded.
