# Iteration 002: Reducer And Convergence Guarantees

## Focus

Inspect the deep-review reducer, state format, convergence references, graph convergence code, and available tests to distinguish documented review rigor from machine-checked guarantees. This pass specifically checked whether deep-review enforces deeper bug-finding artifacts such as producer/consumer coverage, invariant hypotheses, bug-class saturation, graph evidence, and claim adjudication, or whether it mostly enforces artifact shape and stop-gate bookkeeping.

## Findings

1. The reducer preserves discovered findings and stop-blocker history, but it does not enforce rich bug-search artifacts. Structured finding deltas keep `id`, severity, status, title, file, `findingClass`, and claim/evidence text [SOURCE: .opencode/skills/deep-review/scripts/reduce-state.cjs:495], then the registry tracks active/resolved findings and severity transitions [SOURCE: .opencode/skills/deep-review/scripts/reduce-state.cjs:543]. That is useful once a finding exists. The gap is that `findingClass` is optional and no reducer branch requires producer/consumer matrices, invariant hypotheses, bug-class hypotheses, or affected-surface completeness before an iteration counts.

2. Dimension coverage is machine-reduced from shallow signals: iteration `dimensions` arrays and checked strategy rows. `buildDimensionCoverage()` marks a dimension covered when an iteration record lists that dimension or when strategy text is checked [SOURCE: .opencode/skills/deep-review/scripts/reduce-state.cjs:696]. It does not verify that the dimension included a minimum file set, call graph, producer/consumer pair, failing test search, or invariant enumeration. This supports the prior hypothesis that deep-review can satisfy coverage with taxonomy rotation rather than adversarial bug-class coverage.

3. Post-dispatch validation checks schema presence more than semantic depth. The canonical review iteration field list includes `findingDetails`, `findingsNew`, `filesReviewed`, and `newFindingsRatio` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:112], and the validator type-checks `filesReviewed`, `dimensions`, `findingDetails`, and `newFindingsRatio` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:376]. It does not validate that `findingDetails` entries include `findingClass`, `scopeProof`, `affectedSurfaceHints`, counterevidence, or evidence references. The state-format reference says `findingDetails` should carry those richer fields [SOURCE: .opencode/skills/deep-review/references/state_format.md:219], but that contract is not enforced deeply.

4. There is a concrete schema drift risk around `findingsNew`. The state-format reference requires `findingsNew` as a severity-count object with P0/P1/P2 keys [SOURCE: .opencode/skills/deep-review/references/state_format.md:218], but the prompt-pack JSON example shows `"findingsNew":[]` [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:81]. The validator checks the presence of `findingsNew` but does not type-check it [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:366]. This is a machine-check gap: malformed but present state can pass dispatch validation and later distort convergence math or reducer assumptions.

5. Convergence has serious legal-stop gates, but most gates are still post-discovery and stop-time checks. The YAML requires claim adjudication before STOP for active P0/P1 findings [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:444], blocks STOP on unresolved P0, weak evidence density, hotspot saturation, and security-fix replay gaps [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:477]. These are valuable guarantees, but they do not force the reviewer to generate candidate findings through invariant sweeps, API producer/consumer tracing, or negative-test searches before declaring a dimension reviewed.

6. Graph convergence enforces useful structural signals, but its current review model is still coarse. The handler weights review signals as dimension coverage, finding stability, P0 resolution, evidence density, and hotspot saturation [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/coverage-graph/convergence.ts:67]. The scoped review signal computation only looks at DIMENSION, FINDING, FILE, COVERS, CONTRADICTS, EVIDENCE_FOR, RESOLVES, and hotspot metadata [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/coverage-graph/convergence.ts:316]. This can block empty or underconnected review graphs, but it does not model producer nodes, consumer nodes, invariant nodes, test nodes, bug-class nodes, or required path enumeration.

7. Graph data is optional enough to disappear from stop rigor. The convergence reference says `graphEvents` are optional in iteration records and omitted from gate evaluation when absent [SOURCE: .opencode/skills/deep-review/references/convergence.md:677]. The prompt pack also says to include an optional `graphEvents` array and omit it when no graph events are produced [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:69]. That means graph-assisted convergence is a guard when graph data exists, not a universal proof that the review searched the relevant structure.

8. Tests and fixtures prove blocked-stop and graph-blocking behavior, not deep bug-finding behavior. A live test verifies review graph convergence can return `STOP_BLOCKED` for unresolved P0 and uncovered dimension blockers [SOURCE: .opencode/skills/system-spec-kit/mcp_server/tests/graph-aware-stop.vitest.ts:116]. A reducer fixture preserves `blocked_stop` with `dimensionCoverageGate` and `p0ResolutionGate` failures [SOURCE: .opencode/skills/deep-review/scripts/tests/fixtures/blocked-stop-session/review/deep-review-state.jsonl:5]. A gap-query test verifies uncovered DIMENSION and FILE sources are reported [SOURCE: .opencode/skills/system-spec-kit/mcp_server/tests/coverage-graph-signals.vitest.ts:39]. I did not find comparable machine checks that require producer/consumer inventory, invariant hypotheses, bug-class coverage, or negative-test search before an iteration or stop can pass.

9. The deepest documented producer/consumer guarantee appears only in security-sensitive fix replay. The convergence reference requires security-sensitive fix reruns to validate producer/consumer/matrix coverage [SOURCE: .opencode/skills/deep-review/references/convergence.md:388], and the YAML mirrors that for `fixCompletenessReplayGate` [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:484]. That is too narrow for bug discovery. Deep-review needs a general pre-stop or per-dimension evidence matrix, not only a remediation replay gate after risky fixes.

## Questions Answered

- Does the reducer preserve enough detail from iteration artifacts to force deeper follow-up? Partly. It preserves findings, severity transitions, blocked-stop history, graph convergence rollups, and dimension coverage, but it mostly trusts iteration self-reporting for coverage and does not require rich bug-search fields.
- Does graph convergence enforce meaningful review edges? Partly. It can block STOP based on dimension coverage, P0 resolution, evidence density, and hotspot saturation, but only if graph data exists and only using a coarse review graph vocabulary.
- Are there tests proving producer/consumer inventories, invariant hypotheses, or bug-class saturation evidence? I found tests for blocked stops, graph blockers, coverage gaps, and schema presence. I did not find tests that enforce producer/consumer inventory, invariant hypothesis rotation, bug-class saturation, or negative-test search as required review outputs.
- Why can focused deep-research find more bugs? Because it can keep a narrow bug-finding question active until evidence is exhausted, while deep-review can satisfy its machine gates by emitting dimension, file, severity, and stop-gate bookkeeping without proving that likely bug classes were enumerated.

## Questions Remaining

- Which first implementation change gives the best leverage: stricter JSONL schema validation, mandatory per-dimension bug-class matrix, required producer/consumer graph events, or prompt-pack changes that force hypothesis rotation?
- Should graph convergence add explicit node kinds for producer, consumer, invariant, test, and bug-class, or should those remain delta-only fields checked by post-dispatch validation?
- How often do real deep-review packets omit `graphEvents`, malformed `findingsNew`, or shallow `findingDetails` while still passing loop validation?
- Should deep-review stop be illegal when graph data is absent for code/spec targets above a complexity threshold?
