# Iteration 035 — Track D (deepseek)

## Focus
Minimum-viable typed-token schema (section->DTCG $type map, additive typed layer, deterministic-render bridge)

## Findings
1. **[P0] section→$type backbone** — §2 Color Palette→color. §3 Typography→typography composite (fontSize via dimension, fontFamily, fontWeight, lineHeight). §5 Layout→dimension (spacingScale + radiusTokens). §6 Depth→shadow composite. §6.5 Motion→duration + cubicBezier + transition composite. §12 Iconography→dimension (icon sizes) + number (strokeWidth) + strokeStyle. §4 Components→border composite + transition composite. §9 Accessibility→number (contrastRatio). §2.5 DarkMode→color. §1/§7/§8/§10/§13→no DTCG $type (qualitative/prose). Gradients→gradient composite (if split from §6 decorative-depth).
   - Recommendation: Add a `$type` union: 'color'|'dimension'|'fontFamily'|'fontWeight'|'duration'|'cubicBezier'|'number'|'shadow'|'typography'|'gradient'|'border'|'transition'|'strokeStyle' to existing interfaces. ColorToken.$type='color'; TypographyLevel.$type='typography'; ShadowToken.$type='shadow'; RadiusToken/SpacingScale entries.$type='dimension'; MotionSystem entries.$type='duration'/'cubicBezier'. This is a 1-field addition per interface—minimum blast-radius.
   - Evidence: types.ts lines 409-475 (current interfaces), design_md_format.md §§2-12 (data needs), DTCG draft spec §3.1 $type vocabulary
2. **[P0] per-token deterministic id** — No token in the current system has a stable id the DESIGN.md can cite. ColorToken has only hex/rgba; TypographyLevel has no id; ShadowToken has only value string. The validator (validate.ts lines 57-93) checks phantom colors by hex, but can't verify that a DESIGN.md claim like 'weight 300 headlines' traces to a specific TypographyLevel row.
   - Recommendation: Add `id: string` to ColorToken, TypographyLevel, ShadowToken, RadiusToken, and motion entries. Generate as stable hash of the token's canonical properties (e.g. `sha256(hex)` for colors, `sha256(fontFamily+fontSize+fontWeight+lineHeight)` for typography). This costs 0 schema rewrites—one field per interface—and gives the validator a `<token-id>` anchor for every DESIGN.md citation.
   - Evidence: types.ts: ColorToken (line 409, no id), TypographyLevel (line 428, no id), ShadowToken (line 443, no id), RadiusToken (line 451, no id); validate.ts line 59 uses Set<string> of normalized hex, no token-id citation path
3. **[P1] composite typography→deterministic table render** — DESIGN.md §3 Hierarchy table requires: Role, Font, Size, Weight, LineHeight, LetterSpacing, Features, Notes. The current TypographyLevel has all raw data fields (fontFamily, fontSize, fontWeight, lineHeight, letterSpacing, textTransform, fontFeatureSettings, typicalTags, sampleTexts) but NO DTCG alias support. The Role column is currently hand-written by the MD generator. With DTCG $type='typography' and an id, a table row can be auto-rendered: Role derived from typicalTags+fontSize bracket, Font from fontFamily, Size from fontSize, Weight from fontWeight as numeric, etc.
   - Recommendation: Add `roleLabel?: string` to TypographyLevel populated by bucketing fontSize into Display/H1/H2/H3/Body/Caption during extraction (already implicit in sorted-by-size output). Then the §3 Hierarchy table becomes deterministically renderable: iterate sorted TypographyLevel[], render rows from fields. Keep the existing fields; add DTCG projection fn that converts TypographyLevel→{$type:'typography',$value:{fontFamily,fontSize,fontWeight,lineHeight,letterSpacing}}.
   - Evidence: design_md_format.md lines 394-422 (§3 Hierarchy table columns), types.ts TypographyLevel (line 428: has all columns except Role and Notes)
4. **[P1] composite shadow→deterministic elevation table render** — DESIGN.md §6 Depth table (Level, Treatment, Frequency, Use) partially maps to ShadowToken (value, frequency, type, typicalElements). The 'Level' name (Flat/Ring/Elevated/etc.) and the 'Use' column are currently heuristic. DTCG shadow composite ($value: {color, offsetX, offsetY, blur, spread}) would enable automatic classification: blur=0 spread=1→Ring, blur>0→Elevation level by blur magnitude.
   - Recommendation: Keep ShadowToken.value as the original CSS string. Add parsed shadow parts as optional derived fields (color, offsetX, offsetY, blur, spread) in a DTCG projection—not in the schema, generated by a projection function. This preserves the existing extractor while allowing the §6 table to auto-classify shadow levels from parsed offsets. Add a `shadowLevelLabel?: string` field populated from the parsed classification.
   - Evidence: design_md_format.md lines 542-573 (§6 table), types.ts ShadowToken (line 443: value string + type enum but no parsed parts), tokens.json shadowTokens[0] value='rgba(0, 0, 0, 0.04) 0px 1px 2px 0px'
5. **[P1] motion→deterministic §6.5 tables** — DESIGN.md §6.5 requires: Duration Scale table (Token label, Duration, Frequency, Use) and Easing Functions list (value, frequency). The current MotionSystem has durationScale[] with label/value/frequency and timingFunctions[] with value/frequency. With DTCG $type='duration' on each scale entry and $type='cubicBezier' on each easing, the §6.5 tables become deterministically renderable: label column from durationScale[].label, Duration from .value, Frequency from .frequency. No gap in current data shape.
   - Recommendation: Add optional DTCG projection: durationScale entries get {$type:'duration',$value:'150ms',id:hash} views. timingFunctions entries get {$type:'cubicBezier',$value:'cubic-bezier(.4,0,.2,1)',id:hash}. No data-shape change needed—the projection is a pure function over the existing MotionSystem.
   - Evidence: design_md_format.md lines 596-638 (§6.5 tables), types.ts MotionSystem (line 260: durationScale+timingFunctions arrays), tokens.json motionSystem has exact §6.5 column data
6. **[P2] additive typed layer vs full DTCG rewrite** — Iteration-024 judged a full DTCG migration as SCOPED/high-cost because it would require rewriting every extraction path (cluster.ts ~1742 lines, multiple specialized parsers), every validator (validate.ts ~436 lines), and every MD generator pipeline. The additive path adds: (a) 1 field per interface ($type), (b) 1 field per token (id), (c) a projection function dtcgProject(tokens: DesignTokens): DTCGView that returns a typed alias layer without touching the extractors. This is ~80 lines of new code vs. a ~3000+ line rewrite. The provenance benefit (validator can cite token-ids, claim-to-token traceability) is delivered entirely by (a)+(b). The deterministic rendering benefit (§3 typography table, §6 shadow table, §6.5 motion tables) is delivered by (c) plus roleLabel/shadowLevelLabel fields. Full DTCG replacement can be deferred indefinitely.
   - Recommendation: Phase 1 (additive, ~80 LOC): add $type + id to types.ts interfaces, populate in cluster.ts at token creation, add DTCGView projection type + dtcgProject() function, add token-id citations to validate.ts. Phase 2 (deferred, ~3000+ LOC): full DTCG token file output, alias resolution engine, composite value normalization. Phase 1 alone satisfies the 'claim-provenance' requirement.
   - Evidence: cluster.ts ~1742 LOC extraction pipeline, validate.ts ~436 LOC validator, types.ts 493 LOC interfaces; all need zero changes for Phase 1 beyond $type/id field additions

## Summary
Add `$type` (DTCG enum) + `id` (content-hash) to each token interface in types.ts—no shape changes, pure additive fields. Then add a `DTCGView` projection type over the existing `DesignTokens` that maps each section's data to DTCG vocabulary (color→§2, typography composite→§3, shadow composite→§6, duration/cubicBezier→§6.5, dimension→§5+§12). This delivers claim-provenance (validator cites token-ids) and deterministic rendering for the typography/shadow/motion tables at ~80 LOC, deferring the full DTCG alias-resolution engine indefinitely.
