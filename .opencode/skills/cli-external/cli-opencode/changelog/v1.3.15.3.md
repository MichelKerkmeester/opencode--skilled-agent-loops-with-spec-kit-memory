## Fix Detail

### F1 — Rule 5 vs Rule 16 pkill self-contradiction (safety bug)

ALWAYS Rule 5 states "DO NOT auto-kill external operator-owned opencode sessions … exclude `opencode run` from pkill (per 2026-05-23 operator directive)." ALWAYS Rule 16's cleanup step instructed `pkill -9 -f "opencode run"` — a blanket pattern match that kills every `opencode run` process on the machine, including operator-owned sessions Rule 5 explicitly protects. Rule 16 now captures the dispatched process's own PID at launch and kills only that captured PID directly and its own orphan children (`kill -9 "$OC_PID" 2>/dev/null; pkill -9 -P "$OC_PID" 2>/dev/null`), with an inline note: "Kill only the dispatch you started, by captured PID; never a blanket `pkill -9 -f "opencode run"` pattern — see Rule 5." The cleanup intent (reap the dispatcher's own orphans between single-dispatch-discipline runs) is preserved; only the kill scope changed.

### F2 — stale cli-codex row in the sibling-boundary table

The README "Sibling Boundaries" table listed `cli-opencode` on two rows. The third row (`cli-opencode` / OpenAI / "Sandboxed coding, repo analysis, PR review, live web research") was the retired `cli-codex` skill's description, left behind from the packet-122 cli-codex deprecation. Deleted; the table now lists exactly the two live cli-* skills (`cli-opencode` / OpenCode, `cli-claude-code` / Anthropic). The surrounding sentence ("The cli-X skills each dispatch to a different provider and never overlap") required no change — it holds for two skills as written.

### F3 — parallel-session quick start omitted the share-confirmation gate

The README Step 4 quick start ran `opencode run --share --port 4096 …` without surfacing that `--share` requires prior operator confirmation (the skill's own `share-requires-confirmation` hard_rule, plus SKILL.md's NEVER rule 2 / CHK-033). Added a one-line note directly above the Step 4 code block: "Confirm before you share. `--share` publishes the session and exposes its contents at a browser-accessible URL — get operator confirmation first, per this skill's `share-requires-confirmation` hard_rule."

### F4 — default-dispatch recipe contradicted the no-top-level-`--agent` contract

The README Step 3 "default dispatch" recipe passed `--agent context` at the top level, directly contradicting SKILL.md ALWAYS Rule 3 ("no `--agent`; current opencode rejects a top-level `--agent`; put any agent-profile request in the prompt body") — and contradicting the very next sentence in the README itself, which states `context` is a subagent that should not be passed as a top-level `--agent`. The recipe now omits `--agent` entirely and folds the role into the prompt body ("Act as a context-retrieval agent: search memory context and return the top findings."), staying runnable while matching the documented pattern.

### F5 — 6 corrupted filename links

A concurrent session's blanket `sk-prompt-models` → `sk-prompt/prompt-models` find/replace had split a filename segment inside 6 link targets, turning the real files `07--prompt-templates/deepseek-v4-direct-with-sk-prompt-models.md` and `07--prompt-templates/kimi-k2-7-direct-with-sk-prompt-models.md` into non-existent paths (`.../with-sk-prompt/prompt-models.md`). Restored all 6 link targets to the real on-disk filenames; the surrounding descriptive prose ("through sk-prompt/prompt-models + sk-prompt") was left as-is since it is prose, not a filename. No files were renamed.

## Caveats

- F1's cleanup kills the captured PID directly (`kill -9 "$OC_PID"`) plus its children (`pkill -9 -P "$OC_PID"`), not a process group. The process-group form `kill -9 -- -"$OC_PID"` was wrong here: a backgrounded `opencode run &` is not a process-group leader (no `setsid`/`set -m`), so it targets a nonexistent group and silently misses the process — the direct-PID form `kill -9 "$OC_PID"; pkill -9 -P "$OC_PID"` is the correct scoped cleanup. If a dispatch is launched through an intermediate shell wrapper, `$!` is the wrapper's PID rather than `opencode run`'s — capture the actual `opencode run` PID before relying on the cleanup in a scripted context.
- This is a documentation/instruction-content fix only — no runtime script or automation code changed. `graph-metadata.json`'s `causal_summary` (which still says "the four sibling cli-* skills," last updated 2026-06-03, predating the cli-codex retirement) was flagged by this review as a related-but-out-of-scope drift; it was not touched here because it was not one of the five findings in scope for this fix pass.
