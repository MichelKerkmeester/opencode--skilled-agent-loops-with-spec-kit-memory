# Iteration 015 — Track D (deepseek)

## Focus
Component/state fidelity from design-to-code tools (semantic mapping over geometry, evidence-gated variants, §11 hard-gate to interaction capture)

## Findings
1. **[P0] Semantic component tagging (Locofy/Builder.io pattern)** — Industry tools use explicit component MAPPING (class-name conventions, ARIA role, data-attributes) rather than geometric heuristics. Our dom-collector.ts:169-171 already captures className, role, ariaLabel, nearestLandmark, and structuralRegion on every ElementStyle — but cluster.ts:1134-1230 uses NONE of them for detection. Instead it relies on brittle geometric thresholds: Badge requires radiusPx>=100 AND height<30 (misses non-pill badges); Card requires padding>=12 AND width 200-800 (misses full-width cards). The semantic tags that would identify these components reliably are already collected and unused.
   - Recommendation: In cluster.ts §7 Component Identification (line 1134), add a pre-pass that matches element.className against known patterns (/.btn/, /.card/, /.badge/, /.nav/, /.input/) and element.role against ARIA widget roles BEFORE falling through to geometric heuristics. Place className/role lookup as the primary identification path, retain geometry only as a fallback for untagged elements. The Locofy tagging pattern means: if class contains 'btn', it IS a button; geometry is only a confidence booster, never the primary signal.
   - Evidence: cluster.ts:1164-1229 (geometric-only detection), dom-collector.ts:193-263 (already-collected semantic fields unused for detection)
2. **[P1] Variant classification: evidence-gate on 'Primary' default** — classifyVariant() in cluster.ts:1234-1257 defaults to 'Primary' for ALL elements that don't match Ghost/Destructive/Secondary — regardless of whether distinguishing style features exist. This masks real variant diversity: Outline buttons (transparent bg + border, not 'Ghost' which requires no-bg), Tertiary/Text-only (no bg, no border, not Ghost either because Ghost requires alpha<0.05 which is often true for text-only), Subtle vs Bold, Size variants. The default-to-Primary erases all unclassified variants into a single bucket.
   - Recommendation: gate classifyVariant at cluster.ts:1256: replace `return 'Primary'` with an evidence-gated sentinel like `return 'unclassified'` OR add explicit detection branches for Outline (border present, bg transparent), Tertiary (no bg, no border, no shadow), before the Primary catch-all. When no named variant has distinguishing features, output 'unclassified' so the writer documents the dominant variant honestly rather than inventing a 'Primary' label. The count column already shows how many instances fall in each bucket; let the data speak.
   - Evidence: cluster.ts:1256 (fallthrough return 'Primary'), cluster.ts:1259-1281 (all unclassified elements grouped under Primary with inflated counts)
3. **[P0] §11 State Matrix hard-gate (Builder.io/Anima interaction fidelity)** — Builder.io and Anima both gate interaction state output on actual data capture — they never invent states. In our case: when --with-interaction is NOT passed (the DEFAULT), extract.ts:81 sets noInteraction=true, and all ComponentVariant state fields (hoverChanges, focusVisibleChanges, focusChanges, activeChanges, disabledStyle) are null (cluster.ts:1347-1351). The DESIGN.md §11 format spec marks it as a REQUIRED section, and the LLM writer, faced with null state diffs but a required section, INVENTIONS plausible values. The manual playbook at 08--interaction explicitly documents this: 'the default is --no-interaction... the State Matrix section may only contain an absence note' — but the format spec requires the section to exist and the writer interprets 'required' as 'must produce content.'
   - Recommendation: Three changes: (1) In extract.ts:81, flip the default from `noInteraction = true` to `false` so interaction capture runs by default (match the cost: §11 is required, so the data should be collected by default). The --fast flag can still override. (2) Add a hard-gate in the DESIGN.md format spec (§11): when ALL state diffs across ALL components are null, §11 MUST render only 'Interaction state capture was not enabled for this extraction. Re-run with --with-interaction.' — no table, no invented cells. (3) In validate.ts, add a check: if §11 contains specific CSS values but tokens.json components[].variants[*].*Changes are all null, FAIL with 'fabricated-interaction-data'.
   - Evidence: extract.ts:81 (default noInteraction=true), cluster.ts:1347-1351 (all state fields null when interaction off), manual_testing_playbook/08--interaction/interaction-state-matrix.md:18 ('the State Matrix section may only contain an absence note'), design_md_format.md:§15 lines 881-920 (section 11 required)
4. **[P2] Auto Layout → Flexbox/Grid structural mapping (Anima pattern)** — Anima maps Figma Auto Layout to CSS Flexbox/Grid with explicit direction/wrap/align/gap inference. Our ElementStyle (dom-collector.ts:245-252) already collects display, flexDirection, justifyContent, alignItems, gridTemplateColumns, gap — every token Anima needs. But cluster.ts:1368-1405 only uses gridTemplateColumns for column-count and maxWidth+margin for centering. The flex-direction, justify-content, align-items, and gap statistics are collected but never analyzed. §5 'Layout Principles' in DESIGN.md is forced to write prose because the tokens have no structural layout data.
   - Recommendation: In cluster.ts §8 Layout Patterns (line 1368), add aggregation: (1) frequency distribution of display values (flex/grid/block), (2) top flex-direction values with counts, (3) top justify-content + align-items pairs with counts — this reveals the dominant layout axis (row vs column) and alignment strategy (center/start/stretch), (4) gap mode and frequency. Output as a structured layoutProfile object (displayMode, primaryAxis, primaryAlignment, crossAxisAlignment, gapMode) on DesignTokens.layoutPatterns. This gives §5 real structural tokens so the writer describes observed layout patterns with evidence instead of prose.
   - Evidence: dom-collector.ts:246-252 (flex/grid properties collected), cluster.ts:1368-1405 (only grid columns + centering analyzed), types.ts:381-386 (layoutPatterns has only maxContentWidth, commonColumnCounts, sectionSpacing, contentAlignment — no flex/grid structural data)

## Questions Answered
- Q1: Semantic component signals ARE already collected (className, role, ariaLabel, nearestLandmark in dom-collector.ts:193/194/195/263/262) but cluster.ts detection at line 1134 ignores them entirely — the borrowable signal is class-name pattern matching + ARIA role before geometric fallback, not instead of it
- Q2: classifyVariant at cluster.ts:1256 needs an evidence-gate — the fallthrough `return 'Primary'` should become `return 'unclassified'` so the writer doesn't treat all unclassified elements as a named variant with inflated counts
- Q3: §11 hard-gate requires three changes: flip default to --with-interaction in extract.ts:81, add null-state rendering rule to format spec, add fabrication detection to validate.ts — all three are needed because the current system incentivizes invention at every layer
- Q4: Anima's Auto Layout mapping IS borrowable because dom-collector.ts already collects flex/grid properties — they just aren't aggregated into structural tokens in cluster.ts:1368. Adding a 15-line aggregation block turns collected-but-unused CSS into §5-evidenced layout tokens

## Summary
Four borrowable techniques identified from industry design-to-code tools: (1) Locofy/Builde's semantic component TAGGING via className+ARIA role — our dom-collector already captures these but cluster.ts ignores them in favor of brittle geometric heuristics; (2) evidence-gating variant classification so unknown variants aren't silently mapped to 'Primary'; (3) hard-gating §11 State Matrix to only render when --with-interaction captured real data (flip the default flag, add null-detection to the writer format spec, add fabrication-detection to validate.ts); (4) Anima's Auto Layout→Flexbox structural mapping — our ElementStyle already collects flex/grid tokens but never aggregates them for §5. Highest-value single adoption: semantic className+role pre-pass in cluster.ts §7 (line 1134), because it fixes component identification (P0) AND requires zero new data collection.
