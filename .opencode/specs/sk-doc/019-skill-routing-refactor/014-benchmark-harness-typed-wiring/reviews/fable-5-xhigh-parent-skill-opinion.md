# Parent-skill routing architecture — Fable-5 (xhigh)

Independent architectural opinion on the best method for parent-skill routing. The failure was reproduced live, not trusted from the brief.

> **VERDICT: Land surface-authoritative + the coherence invariant now, but reframe the end-state: the hub was never a second classifier to "collapse later," it was a category error to score it at all. A parent hub should hold policy and an address table; the ONE classifier is intent → leaf-pairs, and the mode is a projection of the selected leaves, never an independent decision.**

## Receipts (verified live)
`"make an ascii diagram of the approval loop"` → hub ties all 11 authoring modes at 4/3 (shared `authoring-actions` class, `hub-router.json:37`); `selectIntents` keeps everything within `AMBIGUITY_DELTA=1` (`router-replay.cjs:33,404-408`); `cappedWorkflowModes` slices insertion-order top-2 = `[create-skill, create-skill-parent]` (`executor-dispatch.cjs:55-58`); surface pass correctly resolves the two flowchart leaves; the pair filter (`:81`) drops them to **zero typed pairs**. Full-inventory case confirmed: hub scores nothing, `workflowModes=[]`, 111 surface resources → 0. `routerPolicy.tieBreak` is genuinely dead (zero references in the skill-benchmark scripts) while `parent_hub_router_schema.md` §7 promises "Sort by score, then by `routerPolicy.tieBreak`" — the published schema and the implementation contradict each other. That contradiction, not the cap, is the tell that the design drifted.

## Q1 — Verdict on the method
Surface-authoritative + coherence invariant is correct as the landing move, and it's not a mitigation — it's the *repair of a category error*. The system answers one question ("what does this prompt want?") twice, with two taxonomies over two vocabularies, then takes the **intersection** — which multiplies error (a pair survives only if both are right) and whose failure mode is *silent emptiness*, the worst possible shape. The mode was never an open question: every leaf already carries its mode through `leaf-manifest.json` and the registry. The right architecture for a parent hub: **one scored decision (intent → leaf-pairs), two lookups (leaf→mode via manifest; mode→packet entrypoint via registry), one policy layer** (outcomes, delta, bundle rules, defer contract — what `hub-router.json` should shrink to).

## Q2 — Is a second classifier ever worth keeping?
A second independent keyword pass over the *same prompt* answering the *same question* is a split-brain liability — keep it only as shadow telemetry to measure divergence, never on the routing path. Two legitimate things wear the "hub classifier" costume: (1) the *is-this-hub-relevant-at-all* gate — already exists one level up as the skill advisor; duplicating it inside the hub is redundant; (2) **veto-only checks** that can ABSTAIN/DEFER but never re-rank (leaf→mode must be a registered mode; negation guard; mode-count cap) — checks that can only say "no" cannot split the brain. The single taxonomy's shape: keyed intents like today's `INTENT_SIGNALS`, each mapping to `{pairs: [(workflowMode, leafResourceId)], policyHints}` — type the pairs *at authoring time* in `smart_routing.md` rather than re-deriving them. Intents must be allowed to span modes and be sub-mode: `AGENT_COMMAND` spans `create-agent`+`create-command`; `HVR`/`INSTALL_GUIDE`/`OPTIMIZATION` are sub-mode slices. A mode-shaped taxonomy (the hub's) literally cannot express either. Bonus the collapse forces you to see: `create-benchmark`, `create-diff`, `create-skill-parent` currently have **no surface intent at all** — unreachable except via FULL_INVENTORY. The hub pass was masking coverage holes; one taxonomy plus a CI rule "every registered mode reachable by ≥1 intent" surfaces them.

## Q3 — The backstop (four layers)
1. **Abstain on non-discriminative wins** — require the top intent to have matched ≥1 intent-owned keyword; shared verbs alone (create/make/new) contribute zero margin and route to UNKNOWN_FALLBACK, which `smart_routing.md` already documents and the replay must actually honor.
2. **Fail-closed structural invariants**: every emitted pair ∈ manifest (exists today); advertised `workflowModes == orderedUnique(pairs[*].workflowMode)` (the coherence invariant); and — the tripwire this bug demanded — **empty contract with non-empty raw resources is an ERROR, never a silent zero**. That single assertion would have caught all 5 zeroed fixtures and SD-015 on day one.
3. **A frozen held-out corpus** never used to author keywords, with CI failing on any routing delta without a version bump — the anti-overfit tripwire, non-negotiable because the current surface keywords are visibly fitted to 19 fixtures.
4. The runtime truth: this router executes inside an LLM that reads SKILL.md. The deterministic layer should be *advisory-with-abstain*; the agent's own semantic reading is the real backstop, so make abstain cheap (quick_reference + a 3-line disambiguation checklist) instead of making the keyword layer pretend to be infallible.

## Q4 — Parent-hub-specific hazards
(a) *Shared vocabulary classes are a design defect*: a class referenced by every sibling adds a constant to every score — zero discriminative value, guaranteed tie plateau. Lint: no vocabulary class referenced by more than ~2 sibling signals; hub-wide verbs belong in a hub-identity prior, never per-mode scoring. (b) *Multi-mode prompts are legitimate* (agent+command; create-then-quality) — the taxonomy must emit cross-mode pair sets; the 2-mode cap stays the structural default; named `bundleRules` are the sanctioned exception path. (c) *Full-inventory must be a first-class intent inside the one taxonomy* — the fatal design keyed the bypass off a scenario flag (`executor-dispatch.cjs:74`) the production path can never set, while the hub had no such mode; drive the bypass from the router's own FULL_INVENTORY intent. (d) *Negation/mention-without-intent*: substring scoring is blind to "don't create a changelog, just explain it" — a cheap window guard (negator within N tokens suppresses the hit) plus adversarial held-out fixtures; don't build more than that. (e) *Shared leaves force false single-ownership*: `leaf-aliases.json` types `shared/references/hvr_rules.md` to exactly one mode; a second consumer misattributes or drops it — allow multi-ownership or a mode-neutral type. (f) *Disambiguate vs guess*: the default when the parent genuinely can't tell is **DEFER with a 2–3 option question naming the top candidate children and the one fact that would decide** — which `defaultMode: null` and `outcomes.defer` already declare and the implementation ignores. Guessing is permitted only on a mode-owned-vocabulary win with real margin; never on shared verbs, and never on JSON insertion order.

## Q5 — Biggest risk of my method
Abstain-by-default raises short-term DEFER friction, and the predictable operator response — fattening intent keywords until every prompt "routes" — quietly rebuilds the overfitted mush with one head instead of two; the frozen held-out corpus is the only guardrail, so if it doesn't land in the same change as the collapse, it never lands.
