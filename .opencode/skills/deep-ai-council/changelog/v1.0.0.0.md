First release of `deep-ai-council` as a dedicated skill. Council deliberation moves out of `system-spec-kit` and into its own package, the runtime agent is renamed from `multi-ai-council` to `deep-ai-council` across all three runtimes, the advisor learns to route council prompts to the new skill, and the planning-only boundary is enforced through scoped-write helpers that only ever touch packet-local `ai-council/**` artifacts. Two spec packets ship behind this release: 101/001 created the skill surface, 101/003 expanded references and tests to fully document the agent contract.

> Spec folders: `.opencode/specs/skilled-agent-orchestration/101-deep-multi-ai-council-skill/{001-deep-ai-council-skill-creation, 003-deep-ai-council-reference-expansion}` (Level 1 each)

## New Features

The headline change is that AI council deliberation now has a real home. Before this release, the council was a multi-strategy planning idea threaded through `system-spec-kit`, but the references, persistence scripts, and routing all lived inside an unrelated skill, which made it hard to evolve council behavior without touching spec-folder governance. This release lifts council out into its own package with its own runtime agent.

&nbsp;

#### Dedicated Skill Package

The new `deep-ai-council` skill owns council activation triggers, smart routing across eight council intents, the planning-only operational boundary, and the rules that make council reports reproducible. The skill stays codebase-agnostic, so any project can dispatch a council without inheriting Webflow or OpenCode-specific patterns.

&nbsp;

#### Renamed Runtime Agent

The runtime agent now ships as `@deep-ai-council` across all three runtime mirrors (OpenCode, Claude, Codex). The agent body documents the 9-step workflow from RESOLVE through DELIVER, the dispatch depth rules (Depth 0 parallel via `Task`, Depth 1 sequential via `sequential_thinking` MCP), and the persistence protocol that calls writer functions in a fixed order so every artifact lands with an audit event.

&nbsp;

#### Advisor Routing

Natural prompts like "Run an AI council deliberation to compare implementation plans" used to fall through to a generic explainer because the source scorer's read-only guard tripped on phrases like "compare implementation plans". The advisor source scorer now allows explicit `deep-ai-council` council phrases through that guard, returning the new skill with confidence 0.95 and uncertainty 0.12 on the regression suite.

## Architecture

The release adds a clean library boundary for everything the council writes to disk, and it preserves the additive-only state-event policy so existing council JSONL state stays readable.

&nbsp;

#### Council Persistence Library

A new library at `scripts/lib/persist-artifacts.js` exposes seven named writer functions, one per artifact type (`writeConfig`, `writeStrategyMd`, `writeStateJsonl`, `writeSeat`, `writeDeliberation`, `writeCritique`, `writeReport`). Every writer resolves the destination under `<packet>/ai-council/`, refuses to write outside that root with an `OUT_OF_SCOPE_WRITE` error, then appends an `artifact_written` audit event with byte count and SHA-256 checksum to the append-only state log. The previous CLI wrapper at `scripts/persist-artifacts.cjs` is now a thin entry point that delegates to the library, so non-council callers still work unchanged.

&nbsp;

#### Append-Only State Schema

The state log keeps the v1 event types (round_start, seat_returned, deliberation_synthesized, round_end, council_complete) and adds v1.2 audit events (`artifact_written`, `rollback`, `artifact_superseded`) without breaking older readers. The protocol string stays `multi-ai-council` for compatibility with existing council state rows produced before the rename — the policy is additive only, never rewrite history.

&nbsp;

#### Round Rollback Forensics

Failed rounds move into `ai-council/failed/round-NNN-<timestamp>/` with matching `artifact_superseded` markers in the state log. Operators can inspect the failed round folder for forensic context, then dispatch a fresh round; in-place retry is intentionally out of scope so the audit trail stays clean.

## Documentation

The release ships ten reference files that fully document the agent contract, plus a human-facing README, a 13-section operator playbook, and an 18-scenario manual testing matrix.

&nbsp;

#### Skill References

Six references cover the day-to-day surface: `output_schema.md` (the markdown report contract the parser enforces), `folder_layout.md` (packet-local artifact tree), `state_format.md` (append-only JSONL events), `command_wiring.md` (caller-owned post-dispatch persistence), `seat_diversity_patterns.md` (lens combinations and vantage targets), and `convergence_signals.md` (two-of-three-agree rule and escape hatches). Four further references were added in 101/003 to lift agent-body logic into the reference layer: `scoring_rubric.md` documents the 5-dimension rubric, multi-round deliberation, and the Hunter / Skeptic / Referee adversarial cross-critique; `depth_dispatch.md` codifies Depth 0 vs Depth 1 detection and no-recursive-counciling rules; `failure_handling.md` walks through timeouts, all-seats-fail, contradiction-without-resolution, and insufficient vantage diversity; `anti_patterns.md` lists the eleven council anti-patterns with detection cues and recovery actions.

&nbsp;

#### Human-Facing README

`README.md` follows the HVR (Human-Visible Reference) shape used by `deep-research`, `deep-review`, and `deep-agent-improvement`: ten numbered sections from Overview through Related Documents, three concrete usage examples (top-level dispatch, `@orchestrate` at Depth 1, CLI-skill capture), the runtime parity table for all three mirrors, and a troubleshooting matrix.

&nbsp;

#### Manual Testing Playbook

The playbook follows the canonical `manual_testing_playbook.md` plus numbered category folders shape. Eighteen scenarios across seven categories (DAC-001 through DAC-018) cover runtime routing, council deliberation and seat diversity, artifact persistence and state format, convergence and rollback, scope boundaries, depth detection and resume semantics, and the writer library contract. Every per-feature file carries the required nine-column scenario table, realistic operator prompts, and source anchors back to the implementation.

## Upgrade

No migration is required. The state schema stays additive-only, so existing `ai-council/` artifacts produced before the rename remain readable. Callers that previously dispatched `@multi-ai-council` should switch to `@deep-ai-council`; no shim is provided because no active runtime consumer of the old name was found at rename time.

If a downstream consumer still depends on the old runtime agent name, surface concrete evidence of the dependency before adding a compatibility shim — the project policy is to add shims only when a real consumer requires one.
