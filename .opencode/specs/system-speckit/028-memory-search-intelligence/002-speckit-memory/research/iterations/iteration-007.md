### Galadriel mechanic-by-mechanic net-new assessment

**(a) Zero-token local retrieval — SATURATED (framing only, does not transfer).**
Galadriel's headline is "retrieval costs **zero Anthropic tokens** … all retrieval runs locally in ChromaDB + SQLite" [SOURCE: galadriel README.md:108, 118-119]. This is an *LLM-token-budget* property of an **in-process** agent harness: galadriel calls the LLM, so any retrieval it does *inside the prompt loop* would cost tokens unless pushed to a local store. **The internal Memory MCP is already out-of-band** — it is an MCP server queried over IPC/stdio, not a block of the model's context window; its vector + FTS5 + BM25 + causal channels already run locally in SQLite (`lib/search/*`, confirmed across iters 1-5). So "zero-token local recall" is **already true by architecture** for the internal system and yields **no candidate** — the only transferable artifact is *positioning language* ("recall is budget-free / off the model's token budget"), not a mechanism. [INFERENCE: based on galadriel README.md:108-130 being an in-process-agent property vs the internal system being a separate MCP server; confirmed by the absence of any in-prompt retrieval path in iters 1-5.]

**(b) Bi-temporal KG triples (valid_from/valid_to) + timeline query — REINFORCES C3, does not extend it.**
Galadriel's KG stores `subject --[predicate]--> object` triples with `valid_from`, invalidates by setting `valid_to` (never deletes), and exposes `kg_timeline(entity)` returning `vf → vt` chronological history [SOURCE: galadriel palace.py:465-493 (kg_add valid_from), 557-565 (kg_invalidate sets valid_to), 568-587 (kg_timeline)]; README frames it as "when a fact changes, the old triple gets a `valid_to` date and the new one goes in … Nothing is overwritten, only superseded" [SOURCE: galadriel README.md:129, 151]. **This is a strictly LIGHTER version of what aionforge already drove into C3.** The internal substrate is already *richer*: the lineage table carries `valid_from`/`valid_to` columns with a documented "**Lineage state active projection and asOf resolution**" feature and a per-memory history log [SOURCE: lib/search/vector-index-schema.ts:24-26, 184-185, 2038-2085], the causal-edge layer already has `supersedes`/`contradicts` + `invalid_at` edge-close (C3-A, confirmed iter-2 contradiction-detection.ts:99-110), and `lib/cognitive/temporal-contiguity.ts` already builds spec-folder timelines [SOURCE: lib/cognitive/temporal-contiguity.ts:5, 193]. Galadriel's only marginal contribution is the **recall-surfaced** angle — exposing a `kg_timeline`-style "full history of entity X" *as a first-class recall verb* — which is exactly what **C3-C's `TemporalMode=History`** already proposes [iter-2 C3-C; iter-6 row 16c]. So galadriel **confirms C3-C is the right shape** (an independent second system converged on per-entity timeline-on-recall) but adds **no new candidate** beyond it. Confirmed: substrate present; the only "gap" galadriel highlights (timeline as a recall verb) is already C3-C.

**(c) Ambient "goodnight" consolidation cadence — NET-NEW (C-G1), the one genuine galadriel contribution.**
Galadriel runs consolidation on a **clock**, not on a write. Two mechanisms: (i) **goodnight** (21:00 CET, daily) fires `archive_daily_logs()` which mines the daily-log dir so "today's entries become palace-searchable," using mtime-dedup so re-mining only files new content [SOURCE: galadriel palace.py:277-290 (archive_daily_logs, mtime dedup), README.md:464 (goodnight 21:00 mines daily logs), 670 (goodnight fires archive_daily_logs)]; and (ii) **ambient reflection** (11/14/17/20 CET, silent) fires a turn that "files anything worth keeping to the memory palace … recording what would otherwise be lost between turns," routed so nothing reaches the user — "continuity of attention" [SOURCE: galadriel README.md:507-545, 628-635]. **The internal system has NO cadence-driven consolidation.** Every internal consolidation path is **event-triggered**: `reconsolidation-bridge.ts` runs reconsolidation *on save* (`isSaveReconsolidationEnabled`, save-planner-mode) [SOURCE: handlers/save/reconsolidation-bridge.ts:1-3, 14-18 — it is the SAVE bridge], and the only internal *timers* are **retention/cleanup sweeps**, not consolidation: `session-manager.ts` runs `cleanupInterval`, `staleCleanupInterval`, `retentionSweepInterval` (`setInterval`) [SOURCE: lib/session/session-manager.ts:199, 203, 208] — these expire/evict, they do not *mine or enrich*. So galadriel surfaces a real gap: **a low-priority background "consolidation tick" that periodically drains the deferred/`complete_with_pending_vectors` backlog and runs reconsolidation/enrichment on a cadence, independent of save volume** — which is precisely the missing half of **C4-C** (iter-2/iter-6 row 16d: "the fast/deferred seam exists; the missing piece is a durable cursor + per-item state machine"). C4-C supplies the *cursor*; C-G1 supplies the *clock that advances it*. **This is the strongest net-new galadriel candidate.** Confirmed: internal consolidation is save/sweep-triggered only; no reflection-cadence path exists.

  - **C-G1 (M): cadence-driven ambient consolidation tick.** Add a low-priority interval (reuse the existing `session-manager.ts` interval-registration pattern [SOURCE: lib/session/session-manager.ts:199-208]) that periodically advances the C4-C consolidation cursor — draining pending vectors/BM25/causal-extraction from the deferred-save backlog and running `reconsolidation-bridge` enrichment on items in `raw`/`in_progress` state — so enrichment completes on a clock even when saves are sparse, instead of only when a new save triggers the bridge. mtime/`content_hash`-dedup so re-ticks are idempotent no-ops (mirrors galadriel's mtime-dedup, palace.py:281-283). Effort **M** — interval plumbing + cursor advance is small; it **depends on C4-C** (needs the durable cursor + per-item state to advance) and pairs with C4-A (idempotency so a tick that re-touches a half-saved item is a clean replay). Galadriel evidence: palace.py:277-290, README.md:507-545. Internal seam: session-manager.ts:199-208 (timer infra) + reconsolidation-bridge.ts (enrichment) + C4-C cursor.

**(d) Memory-palace humanist organization (wings/rooms/drawers/halls) — MODEST net-new facet (C-G2).**
Galadriel organizes memory with four nested concepts: **wing** (top-level namespace, ~one per agent), **room** (folder-based grouping, exactly one per drawer — `memory`/`harness`/`tower`/…), **drawer** (the ~200-1000-token atomic chunk), and crucially **hall** (a **keyword-based, auto-classified topic that cross-cuts rooms** — a drawer about a harness bug lives in `room=harness` AND `hall=problems`) [SOURCE: galadriel README.md:140-147; palace.py:57-74 (search filters wing/room/hall), 641-689 (taxonomy: wing→room counts + hall counts)]. The composable query `palace_search("retry logic", room="harness", hall="problems")` reads as "bug-tagged content from the code area" [SOURCE: galadriel README.md:147]. **The internal system has the room/wing analogues but NOT the hall analogue.** Internal records already carry `spec_folder` (≈ room: folder-based, one per record), `contextType` (≈ a coarse room/type), and `importance_tier` [SOURCE: handlers/chunking-orchestrator.ts:44, 246-247, 481; handlers/memory-bulk-delete.ts:151]. What is **missing is the cross-cutting, auto-classified keyword topic ("hall")** — a derived label like `decisions`/`problems`/`milestones` that is *orthogonal* to the spec-folder and is auto-assigned at index time, queryable as an independent filter facet. This is a **modest** candidate because (i) the internal `artifact-routing.ts` already has *keyword* classification machinery (`keywords: ['plan','strategy','phases','milestones','timeline']` at `lib/search/artifact-routing.ts:179`) that could *seed* the auto-classifier, and (ii) galadriel's halls overlap heavily with the aionforge query-class axis (C2-A) and with `contextType` — so the net-new surface is narrow. Confirmed: internal has folder/type/tier facets, no cross-cutting auto-topic facet; keyword machinery for one exists at artifact-routing.ts:179.

  - **C-G2 (M): cross-cutting auto-classified topic facet ("hall").** Add a derived, index-time-assigned `topic` label (small fixed vocabulary: decisions/problems/milestones/…), orthogonal to `spec_folder`/`contextType`, queryable as an independent recall filter (`recall(query, topic="problems")`). Seed the classifier from the existing keyword machinery [SOURCE: lib/search/artifact-routing.ts:179]. Effort **M** — additive column + auto-classifier + filter plumbing. **Low leverage / rank below the C2-C…C4-A spearhead** — it overlaps `contextType` and the C2-A retrieval-class axis, so confirm it earns its keep against those before building. Galadriel evidence: README.md:140-147; palace.py:57-74. Internal seam: chunking-orchestrator.ts:246-247 (where contextType/tier are written) + artifact-routing.ts:179 (keyword seed).

**(e) Archive-before-clear lifecycle — SATURATED (internal continuity is the equivalent, arguably richer).**
Galadriel archives the conversation to the palace *before* `/new` / `/compact` / max_tokens-recovery wipes it, so "nothing is lost at the moment of wipe" [SOURCE: galadriel README.md:106, 234-275 (archive_conversation), 651-664 (1.12.1 archive-before-recovery)]. The internal analogue is the **continuity snapshot + memory-save discipline**: `authored-continuity-snapshot.ts` builds a resume-ladder-derived snapshot and upserts it into the markdown continuity record *before* a session boundary [SOURCE: lib/continuity/authored-continuity-snapshot.ts:1-3, 8-9 (buildResumeLadder), 38-40], and the whole spec-kit save path (`/memory:save`, `memory_save`) is an explicit archive-before-loss discipline. So galadriel's archive-before-clear is **already covered** by the internal continuity/save model — and the internal version is *richer* (structured resume ladder + thin continuity record, not just a raw conversation mine). **No candidate.** Confirmed: internal continuity-snapshot covers the archive-before-boundary need.

**(f) Wake-up snapshot injection — SATURATED (internal continuity-snapshot is the equivalent).**
Galadriel injects a compact ~800-token L0+L1 wake-up snapshot into the dynamic system-prompt block on every call, cached to a file by a subprocess [SOURCE: galadriel README.md:110, 672; palace.py:299-316 (read_wake_up_text, "called on every API call, must stay cheap — just a file read"), 429-460 (refresh_wake_up_cache subprocess)]. The internal analogue is **exactly** `authored-continuity-snapshot.ts` + the `_memory.continuity` frontmatter that the resume ladder reads — a compact, file-cached continuity snapshot surfaced at session start [SOURCE: lib/continuity/authored-continuity-snapshot.ts:1-3, 30-37 (AuthoredContinuitySnapshotResult: docsUpdated, recoveryContext)]. The internal system *already* has the "walk into every session with continuity" property the wake-up snapshot provides. The only galadriel nuance — caching the snapshot via a *subprocess* to keep the embedder out of the hot read path — is an in-process-agent concern that does not apply to the out-of-band MCP server (the read path is already a cheap file/DB read). **No candidate.** Confirmed: internal continuity-snapshot is the wake-up equivalent.

Each row names **what speaks the old contract** (the migration/compat risk — a consumer, stored row, or cache that assumes the pre-change shape).
