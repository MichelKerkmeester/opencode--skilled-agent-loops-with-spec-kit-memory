# Iteration 11 (DeepSeek lineage): Cognee retriever internals — the un-mined retrievers

> Model: **DeepSeek v4 Pro** (`deepseek/deepseek-v4-pro --variant high`, read-only, via `opencode run`). Orchestrator-written. newInfoRatio **0.7**. Deep-extracted the Cognee retrievers NOT covered in iter-8 (agentic, summary, lexical/jaccard, natural-language). 5 candidates. ## Candidates
1. **CG-agentic-tool-loop → NET-NEW (H/L).** ReAct-style agent loop: LLM emits a structured `AgentStep` (`tool_call` | `final_answer`), tools execute with ACL gating, results append to context, iterate to `max_iter`, then force a best-effort final answer. `agentic_retriever.py:419-478`. Maps to `handlers/memory-context.ts` strategy router (:1291-1311 dispatches quick/deep/focused/resume — an "agentic" strategy is a natural seam). NET-NEW: `memory_context` is a static mode router with zero tool-calling. **The headline iter-11 find** (H leverage, L effort because the router seam already exists).
2. **CG-post-retrieval-summarization → NET-NEW (L/S).** After resolving retrieved edges→text, call `summarize_text()` with a dedicated prompt to compress before returning. `graph_summary_completion_retriever.py:50-70`. Maps to `stage2-fusion.ts` enrichment (:1398-1408). NET-NEW: we enrich with metadata/anchors/validation but never LLM-summarize retrieved content (closest analog = token-budget truncation, which *drops* not compresses).
3. **CG-jaccard-lexical-channel → EXTENDS CG-pluggable-lexical-architecture (L/S).** Token-set Jaccard + frequency-aware multiset variant `Σmin/Σmax` + stopword filtering. `jaccard_retrival.py:45-60`. Maps to `hybrid-search.ts` channel collection (:1358-1424). EXTENDS: we have BM25+FTS5 but neither set-Jaccard nor its multiset variant.
4. **CG-pluggable-lexical-architecture → EXTENDS CG-jaccard (M/M).** Lexical retriever parameterized by injectable `tokenizer`/`scorer` callables + lazy-init/caching + heapq top-k. `lexical_retriever.py:15-26`. Maps to `hybrid-search.ts` channel execution. EXTENDS: our channels are hardcoded (vectorSearchFn/ftsSearch/bm25Search/graphSearchFn) with no injection API; this is the generalization (a ChannelPlugin interface). **(Circular EXTENDS with #3 — jaccard is the concrete instance, this is the generalization.)**
5. **CG-query-generation-retry-loop → NO-TRANSFER / needs-reframe (M/L).** NL query → LLM-generated Cypher from schema introspection → execute → on empty/error, feed prior attempt back as negative feedback → retry to `max_attempts`. `natural_language_retriever.py:35-105`. **NO-TRANSFER as-is**: assumes entity-node graph introspection (`MATCH (n) UNWIND keys(n)`) which our memory-ID graph can't support; the retry-with-feedback pattern is transferable but schema introspection would need an SQL-table-reflection reframe. Lower priority per the systemic caveat.

## Net effect
2 NET-NEW (agentic-tool-loop = the standout; post-retrieval-summarization), 2 EXTENDS (jaccard + pluggable-lexical, paired), 1 NO-TRANSFER. The agentic-tool-loop is the highest-leverage Cognee retriever idea found so far — it maps onto an *additional `memory_context` strategy*, dodging the memory-ID-vs-entity-node mismatch entirely.
