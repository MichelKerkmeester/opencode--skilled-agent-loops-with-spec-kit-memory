## Q1 — Memory-index move-reconciliation under rapid concurrent spec-folder moves

- **Findings:**
  1. The reconciler lives in `lib/storage/incremental-index.ts:512-628` (`reconcileMoves`). The pre-pass categorization (`categorizeFilesForIndexing`, `incremental-index.ts:267-312`) splits discovered files into `toIndex/toUpdate/toSkip/toDelete`; a vanished old path is added to `toDelete` and a new file path is added to `toIndex`. The reconciler then pairs them before the delete+reindex cycle. [HIGH] Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:512-628` and `.opencode/skills/system-spec-kit/mcp_server/handlers/memory-index.ts:692-700`.
  2. The match is **grandparent + basename only** — i.e., it only fires for a one-level sibling folder rename (e.g. `specs/012-old/spec.md` → `specs/012-new/spec.md`); a cross-grandparent move (e.g. `specs/parent-a/012-x/spec.md` → `specs/parent-b/012-x/spec.md`) or a multi-level rename fails the `path.dirname(path.dirname(oldPath)) === newGrandparent` test and falls through to delete+reindex, losing the embedding. [HIGH] Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:567-573` (the `siblingDeleted` filter).
  3. The new-side eligibility gate is the presence of a `packet_id` in the **new folder's** `graph-metadata.json` (read with `fs.existsSync`/`fs.readFileSync`, not the DB); the old folder's `graph-metadata.json` is NOT consulted. A packet that was moved but the operator forgot to write `graph-metadata.json` on the new side is silently excluded from the in-place repath. [HIGH] Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:520-543`.
  4. The DB uniqueness guard is `SELECT … WHERE (file_path = ? OR canonical_file_path = ?) AND embedding_status != 'failed' LIMIT 2` followed by `if (rows.length !== 1) continue;` — so a duplicate-row anomaly (two live rows for the same old path) is treated as "skip" and falls back to delete+reindex, which loses embeddings on the surviving row. [HIGH] Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:558-582`.
  5. The `stored document_type` cross-check (when the column is present) only blocks the repath when the new-path-derived `extractDocumentType(newPath)` is *different* from the stored value (`incremental-index.ts:588-591`); a row stored as the generic `'memory'` kind at a `spec.md` path is therefore incorrectly treated as a match (the stored `'memory'` ≠ derived `'spec'`, so the row is **skipped**). The test `incremental-index-move-reconcile.vitest.ts:175-199` documents the protective direction; an inverted mismatch is silently treated as "skip" rather than "match" by `continue`. [MEDIUM] Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:588-591`.
  6. The `UPDATE memory_index SET file_path=?, canonical_file_path=?, spec_folder=?, file_mtime_ms=?, updated_at=datetime('now') WHERE id = ? AND embedding_status != 'failed'` runs as a single prepared statement with no enclosing transaction in `reconcileMoves` — the function iterates NEW-paths in a JS loop and each UPDATE is a separate auto-commit. A crash mid-loop leaves the index in a half-reconciled state where some renamed rows point to the new path and others still point to the old path. [HIGH] Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:603-612`.
  7. Race exposure: the **single-writer gate** is the `scan_started_at` row in the `config` table, acquired at the start of `handleMemoryIndexScan` via `acquireIndexScanLease` (`handlers/memory-index.ts:354`); a lease heartbeat at `max(10000, leaseExpiryMs/3)` (≥10s, typically 20s for the 60s default) keeps the lease alive during long multi-batch scans (`handlers/memory-index.ts:400-408`). Two concurrent `memory_index_scan` calls are **coalesced** onto the first scan's lease window instead of running in parallel (`handlers/memory-index.ts:358-381`). [HIGH] Evidence: `.opencode/skills/system-spec-kit/mcp_server/handlers/memory-index.ts:351-408` and `.opencode/skills/system-spec-kit/mcp_server/core/db-state.ts:88-90,409-537`.
  8. Failure-mode under SQLite contention: `acquireIndexScanLease` returns `acquired:false, reason:'contention'` with `waitSeconds=LEASE_CONTENTION_RETRY_SECONDS=1` (a 1-second hint), failing closed so a second scan cannot race the first (`db-state.ts:387,508-522`). This protects the reconciler from intra-DB races but does not protect against the `reconcileMoves` UPDATE loop being interrupted by a power loss between successive auto-commits. [HIGH] Evidence: `.opencode/skills/system-spec-kit/mcp_server/core/db-state.ts:387,508-522`.
  9. Race exposure: a concurrent **`memory_save`** into the same target spec folder is serialised by `withSpecFolderLock` + an interprocess lockdir at `os.tmpdir()/mk-spec-memory-save-locks/<sha1(specFolder)>` with a 30s timeout and 5-minute stale-lock recovery (`handlers/save/spec-folder-mutex.ts:14-86`). So a `memory_index_scan`-driven reconcile and a `memory_save` for the same folder cannot interleave. **However, a `memory_save` for the OLD folder and a `memory_index_scan` that observed the move in the same window can still race** because the interprocess lock is keyed on the (already-renamed) new spec-folder string, while the scan is a global single-writer. [MEDIUM] Evidence: `.opencode/skills/system-spec-kit/mcp_server/handlers/save/spec-folder-mutex.ts:14-86` and `.opencode/skills/system-spec-kit/mcp_server/handlers/memory-index.ts:349-381`.

The reconciler reads `graph-metadata.json` with a plain `fs.readFileSync` (synchronous, blocking the event loop) and trusts the parsed `packet_id` field without any fsync barrier — a partial write of `graph-metadata.json` (e.g., rename-in-progress) yields either a parse error (skipped via the inner `catch {}`) or a half-populated object (e.g. `packet_id: undefined` → skipped via the `typeof !== 'string'` guard). The defensive `catch` means a half-written file is silently dropped from the in-place repath, falling back to delete+reindex. [MEDIUM] Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:520-543`.

- **Verdict:** The move reconciler is **correct for the single-sibling-rename case** under the global single-writer scan lease, but it (a) only handles a one-level sibling rename (cross-parent moves fall through to a delete+reindex and lose the embedding), (b) runs the row UPDATE outside a transaction so a crash mid-loop leaves half the rows repointed, (c) gates eligibility on the new folder's `graph-metadata.json` only, and (d) treats duplicate-row anomalies as "skip" (no embedding preservation). Concurrent moves within the scan lease window are safe; concurrent `memory_save` for the same target folder is also safe via `withSpecFolderLock`, but moves racing with `memory_save` for the *old* folder path are not explicitly fenced. [HIGH]

## Q2 — Shared-daemon lease model on ungraceful launcher death

- **Findings:**
  1. The launcher maintains **two** coordination files: an **owner-lease** at `<dbDir>/.spec-memory-owner.json` (`{ownerPid, ppid, executablePath, startedAtIso, lastHeartbeatIso, ttlMs, canonicalDbDir}`) and a **pid-lease** at `<dbDir>/.mk-spec-memory-launcher.json` (`{pid, startedAt, ownerPid, childPid, …}`). They are independent and key on different processes. [HIGH] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:84-86, 314-324, 802-806`.
  2. The owner-lease is written via `writeOwnerLeaseFileExclusive` (`fs.openSync(path, 'wx', 0o600)` + `fs.fsyncSync(fd)` + `fs.renameSync(tmp, current)`) so a racing launcher cannot both win the lease. The reclaim path re-reads after the atomic rename to confirm "did I actually win?". [HIGH] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:281-403`.
  3. The `classifyOwnerLease` decision tree, in order: (a) `processLiveness(ownerPid) === 'dead'` → `'stale-pid'`; (b) `processLiveness === 'unknown-eperm'` (sandbox returns EPERM) → `'unknown-eperm'`; (c) the actual `ppid` differs from the recorded `ppid` and is `1` (init) → `'ppid-1-orphan'`; (d) `now - lastHeartbeat > 2*ttlMs` → `'stale-heartbeat-reclaim'`; otherwise `'live-owner'`. The TTL defaults to 60_000 ms. [HIGH] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:314-363` and `.opencode/bin/lib/model-server-supervision.cjs:274-284`.
  4. SIGKILL-9 / power loss / OOM-kill: the launcher's `process.on('exit', …)` and signal handlers do **not** run. The owner-lease file is left on disk. On the next secondary launch: `classifyOwnerLease` runs `processLiveness(ownerPid)` (`process.kill(pid, 0)`) — ESRCH returns immediately, so the lease is classified as `'stale-pid'` and reclaimed in the same `acquireOwnerLeaseFile` call. **Re-election latency = secondary launcher startup + a single signal-0 roundtrip** (sub-millisecond on the same host), not the 2×TTL window. [HIGH] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:346-374` and `.opencode/bin/lib/model-server-supervision.cjs:274-284`.
  5. Graceful shutdown via SIGTERM/SIGINT/SIGHUP/SIGQUIT: `installSignalHandlers` (line 1307-1325) calls `shutdownLauncherForSignal` (line 1263-1305), which sets `launcherShutdownInProgress = true`, clears heartbeats, sends SIGTERM to the context-server child, waits up to `RESPAWN_REAP_GRACE_MS = 7000ms`, then SIGKILLs if still alive, then `clearAllLeaseFiles()` which `unlinkSync`s both the owner-lease and pid-lease. The `process.on('exit', clearAllLeaseFiles)` is the last-resort path. Secondary launcher sees no lease and immediately claims. **Re-election latency ≈ SIGTERM→exit (≤7s) + secondary startup.** [HIGH] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:1263-1336` and `.opencode/bin/lib/model-server-supervision.cjs:19`.
  6. **Zombie-launcher exposure:** a launcher's child finishes but the launcher is still alive as a zombie (defunct process not yet reaped by its parent). `process.kill(pid, 0)` returns 0 (the PID exists in the kernel), so `processLiveness` reports `'alive'`. The owner-lease heartbeat stops being refreshed. The `ppid-1-orphan` check only fires when the parent **is** init (PPid = 1); for a normal parent that hasn't called `wait()` yet, PPid is still the original parent, so this branch is **false** and the lease is reported as `'live-owner'` even though the launcher can do nothing. The only reclaim path is the **2×TTL = 120s heartbeat staleness** floor. **Worst-case re-election latency for a zombie launcher = ~120s.** [HIGH] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:346-363` (dead < EPERM < ppid-1-orphan < stale-heartbeat), `.opencode/bin/lib/model-server-supervision.cjs:274-284` (zombie ⇒ signal-0 ok).
  7. **EPERM / sandbox exposure:** `processLiveness` returns `'unknown-eperm'` when `kill(pid, 0)` raises EPERM (process exists but we lack permission to signal — common in containers/seccomp/sandbox). `classifyOwnerLease` treats this as a live lease (the branch returns `'unknown-eperm'`, not `'stale-pid'`, and `acquireOwnerLeaseFile` then treats `'unknown-eperm'` as **non-reclaimable** alongside `'live-owner'` at `mk-spec-memory-launcher.cjs:371`). A second launcher that also has EPERM cannot prove the first is dead; **the system can deadlock with a live-but-uncontactable owner for the full 2×TTL = 120s** until the heartbeat finally expires. [HIGH] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:346-374`.
  8. **Heartbeat interval is ttlMs/2, floored to 1s** (`startOwnerLeaseHeartbeat`, line 427-443). Default ttlMs=60_000 → interval 30s. The "stale" threshold for reclaim is `2 * ttlMs = 120s`, so a heartbeat-skip of one interval (30s) is within tolerance. Two consecutive missed beats (60s) still leaves a margin of 60s before reclaim. **Single-beat heartbeat gaps are not the failure mode; the zombie and EPERM paths above are.** [HIGH] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:427-443`.
  9. **Respawn path** (`respawnAfterDeadSocket`, line 671-752): when a secondary sees the lease held by a dead-pid launcher but the IPC socket is gone, it takes `acquireBootstrapLock({requireLock:true})` AND `acquireRespawnLockFile()` (`.mk-spec-memory-respawn.lock`, `RESPAWN_LOCK_STALE_MS = 60_000 ms`), reaps the previous owner with SIGTERM→7s→SIGKILL, reclaims the owner-lease with `writeOwnerLeaseFileExclusive`, reaps the dead context-server child, and only then spawns a new server. Worst-case respawn latency is therefore: 60s (stale respawn-lock) + 7s (reap grace) + 1s (SIGKILL grace) + child spawn + heartbeat. [HIGH] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:613-752` and `.opencode/bin/lib/model-server-supervision.cjs:19-20`.
  10. **Race in `clearOwnerLeaseFile`** (line 450-464): the function `readOwnerLeaseFile()` twice, checks both reads return the same `ownerPid === ownerLeasePid`, then `unlinkSync`. A concurrent reclaim that writes a new owner-lease between the two reads will see different `ownerPid` values and abort the unlink — but the second launcher has already taken ownership. Conversely, a racing reclaim that runs between the "ownerPid matches" check and the `unlinkSync` could unlink a freshly written lease belonging to a different owner, leaving a window where no owner-lease exists. The residual sub-syscall window is acknowledged in the comment at `mk-code-index-launcher.cjs:409-411`. [MEDIUM] Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:450-476` and `.opencode/bin/mk-code-index-launcher.cjs:406-411`.
  11. **Liveness probe (deep-probe) for respawn decisions**: the secondary launcher uses `probeDaemon` with `deepProbe: true` to require a JSON-RPC `initialize` reply (not just a socket accept), so a hung-but-listening daemon is correctly classified as dead. Probe timeout default 5000ms, capped at 6999ms (`launcher-ipc-bridge.cjs:9-10, 122-127, 356-365`). This is the daemon-side liveness check that the owner-lease liveness check is not sufficient for. [HIGH] Evidence: `.opencode/bin/lib/launcher-ipc-bridge.cjs:9-10, 124-128, 356-365`.
  12. **Skill-advisor launcher does NOT have an owner-lease model** — it only has the simple `leaseHeldFromFile` (pid-lease) check at `.opencode/bin/mk-skill-advisor-launcher.cjs:215-225`. A SIGKILL'd skill-advisor launcher leaves only a pid-lease; a secondary relies on signal-0 → ESRCH to detect and bridge/respawn. **Asymmetric vs the spec-kit and code-graph launchers.** [HIGH] Evidence: `.opencode/bin/mk-skill-advisor-launcher.cjs:201-225`.

- **Verdict:** For the spec-kit (memory) launcher, the design is mostly robust: SIGKILL'd launchers are reclaimed **immediately** (sub-ms) via signal-0 ESRCH, and graceful shutdown reclaims within the SIGTERM→SIGKILL grace (≤7s). The two real failure modes are (a) **zombie launchers** (parent not yet reaped) which look alive to signal-0 and require the full 2×TTL = 120s heartbeat staleness floor, and (b) **EPERM sandboxed owners** which are treated as live and likewise wait out the 120s heartbeat. Respawn-after-dead-socket adds up to 60s (respawn lock staleness) + 7s (reap grace) on top. The skill-advisor launcher uses a simpler pid-only lease model — it has neither the heartbeat-fence nor the ppid-1-orphan reaper, so its failure modes are not directly comparable. [HIGH]

[HIGH] Evidence: `.opencode/skills/system-skill-advisor/references/graph/skill_graph_query_cookbook.md:138,166`.

The propagator can *propose* a richer set on demand, but until an operator runs it, the advisor's recommendations use only the author-declared graph. The hook fires on every `UserPromptSubmit` and queries the current graph each time; if the graph is asymmetric, the hook returns recommendations that respect the declared (possibly asymmetric) graph. [HIGH] Evidence: `.opencode/skills/system-skill-advisor/hooks/claude/user-prompt-submit.ts:116-204` and `.opencode/skills/system-skill-advisor/mcp_server/lib/cross-skill-edges/detect-inbound-enhances.ts:177-232`.

The two output counters in the result (`applied`, `skipped_existing`, `candidates`) plus the per-candidate `applyable` flag (true only when `weight !== null && context !== null && blockers.length === 0`) gate the writes; a candidate that scores 0.80 on the three rules but lacks a deterministic weight or context is **never** auto-applied. The `inferEdgePayload` helper (`context-template.ts`) is the gatekeeper, and only fires when the source has a same-family exemplar already declaring the `weight` and `context_template`. [HIGH] Evidence: `.opencode/skills/system-skill-advisor/mcp_server/lib/cross-skill-edges/detect-inbound-enhances.ts:208-227` and `.opencode/skills/system-skill-advisor/mcp_server/lib/cross-skill-edges/index.ts:46-66`.

The `applyEnhanceEdge` patcher enforces the same boundary at write time. [HIGH] Evidence: `.opencode/skills/system-skill-advisor/mcp_server/handlers/skill-graph/propagate-enhances.ts:45-53`.

- **Verdict:** The mechanism is a **carefully bounded authoring helper**, not a runtime improvement loop. It detects missing inbound `enhances` edges via a deterministic three-rule composite scorer and writes only on explicit trusted-caller `apply` intent with the `applyable` gate as the final filter. **It measurably improves unprompted skill discovery only when an operator actually runs it** — and today nothing runs it automatically. The asymmetric `enhances`/`enhanced_by` problem the cookbook flags is real, but the propagator is opt-in tooling for it, not a self-maintaining system. The design is correct, the test coverage is strong (F-08-001 malformed-asset, F-06-004 malformed-JSON, F-03-001 workspace-escape), and the value-vs-complexity verdict is that the feature is **net-positive if the operator discipline is present** (a single `propagate_enhances mode:apply applyAllHighConfidence` run per refactor closes the asymmetry) but **invisible in steady state** because the advisor hook has no plumbing to schedule or invoke it. [HIGH]
