## Findings (numbered, with citations)

1. Probe 14 was in the FTS top-20 and almost certainly in the dense top-20 at bench time. Live FTS ranks put `structural-indexer.ts` in the lexical lane at ranks 9-12, with the exact `.opencode` mirror at rank 10. The bench rerank `pre_score` for the first structural-indexer candidate is `0.021620337372297457`; under Iter 3's RRF formula, that exact value equals either vector rank 1 + FTS rank 9 or vector rank 9 + FTS rank 1. Since live FTS shows structural-indexer at rank 9 while the stress test owns FTS rank 1, the consistent reconstruction is `.gemini/.../structural-indexer.ts` at dense rank 1 and FTS rank 9. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py:42-87`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:88-132`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/baseline-bge.rerank-scores.jsonl:6`; SQLite query: probe 14 FTS ranks 9-12 are structural-indexer mirrors.

2. Probe 14 survived fusion and dedup into the rerank input, but as a mirror-selected `.gemini` representative rather than the literal `.opencode` path. The rerank score logs are written by the adapter over the rerank head, so any candidate listed there is already post-fusion/post-dedup input to model reranking. Both baseline and Jina list `.gemini/skills/system-code-graph/mcp_server/lib/structural-indexer.ts` twice in that head, with `pre_score` values `0.021620337372297457` and `0.010769230769230769`. The exact `.opencode` chunk shares `source_realpath`, `content_hash`, and line range with the `.gemini` mirror, so `_dedup_and_rank_hybrid_rows()` is expected to keep only one alias per chunk-line key. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:247-272`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:481-515`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/baseline-bge.rerank-scores.jsonl:6`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/jina-v3.rerank-scores.jsonl:6`; SQLite query: ids `19820` and `44683` share the same structural-indexer `source_realpath` and line range.

3. Baseline BGE lost probe 14 inside reranking, not before it. The BGE rerank row scores the structural-indexer chunks at `0.02532653696835041` and `0.006373446434736252`, while the stress-test chunk scores `0.07645358890295029`, `query-intent-classifier.ts` scores `0.051814135164022446`, and indexer-types chunks score above the structural implementation. The bench output therefore misses the normalized expected source path even though the implementation was in the rerank input. This is exactly the lexical-cue-density failure described by the Phase 2 README: the stress test uses the walker language more directly than the implementation header/import chunk. Citations: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/README.md:5-9`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/baseline-bge.rerank-scores.jsonl:6`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/baseline-bge.results.jsonl:6`.

The expected target is `.opencode/skills/system-spec-kit/scripts/dist/memory/generate-context.js`, but the live project settings exclude `**/dist`, and read-only SQLite returned `0` vec rows and `0` FTS rows for that exact dist path. 6. The current live DB cannot exactly replay Phase 2's dense vector top-20 as written. The bench script says it pinned `sbert/BAAI/bge-code-v1`, but the current vec table schema is `embedding float[768]`, and the current cached `BAAI/bge-code-v1` encoder produces 1536 dimensions. That mismatch blocks an exact read-only dense KNN replay against the live DB. The practical conclusion is still strong: Phase 2's own rerank logs prove structural-indexer had reached the model head for probe 14, and SQLite proves probe 10's expected dist file is unindexed. Citations: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/run-phase2-smoke.sh:48-51`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:579-596`; SQLite query: `sqlite_master` reported `embedding float[768]`, and dense replay failed with `Expected 768 dimensions but received 1536`.
