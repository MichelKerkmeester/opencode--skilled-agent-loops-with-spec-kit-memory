`mcp-tooling` gains a third workflow mode: `mcp-aside-devtools` brings AI-browser automation via the Aside browser (agentic `aside` CLI, deterministic `aside repl` evidence capture, `aside mcp` via Code Mode fallback) into the hub.

---

## Why

The hub covered Chrome-family debugging (`mcp-chrome-devtools`), ClickUp task management (`mcp-click-up`), and the Figma Desktop transport (`mcp-figma`) — but had no bridge for agentic AI-browser automation. Aside's developer surface (natural-language CLI tasks + deterministic REPL + a built-in stdio MCP server) fills that gap with the same CLI-primary/MCP-fallback architecture the hub already routes.

## What Changed

#### New workflow packet (`mcp-aside-devtools/`)

- 41-file nested packet authored from a converged 10-iteration deep-research fan-out (3 lineages); `packetKind: "workflow"`, `backendKind: "cli-plus-mcp"`, `mutatesWorkspace: true`.
- Live-probed MCP inventory documented as version-pinned (one `repl` tool at CLI 1.26.626.1517, protocol 2024-11-05) with runtime rediscovery mandated.

#### Hub surfaces

- `mode-registry.json`: fourth `modes[]` entry, workflow axis; count-coupled prose updated.
- `hub-router.json`: `mcp-aside-devtools` signal (weight 4; `aside-devtools-aliases` + `agentic-browser` + `hub-identity`), mirrored vocabulary classes, tie-break slot after `mcp-click-up` and before the transport.
- `SKILL.md`, `description.json`, `graph-metadata.json`: mode table row, keywords, trigger example, intent signals, domains, packet entity, key files.
- `manual-testing-playbook/hub-routing/aside-browser-automation.md`: routing scenario guarding the Aside-vs-Chrome vocabulary boundary.
- `.utcp_config.json`: `aside` manual registered (stdio, `aside mcp`, credential-free `env: {}`); single-manual posture — the dual-manual layout stays an open question gated on an isolation test.
