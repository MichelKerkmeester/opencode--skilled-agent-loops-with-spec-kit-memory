# Iteration 012 — Track D (deepseek)

## Focus
Static CSS analysis (project-wallace/CSS Stats/Parker) — dedup, occurrence clustering, hybrid static+computed

## Findings
1. **[P0] hybrid-static-computed-var-resolution** — Our dom-collector.ts stylesheet walk captures --var names and their raw authored values (e.g. var(--color-primary)), but never resolves var() chains to their computed values. cluster.ts:662-675 can only cross-reference a CSS variable to a color token when the raw value is a direct color string (hex/rgb/hsl). CSS variables storing indirection (var() references, fallback chains) produce zero matches in the cross-reference loop, meaning var-named tokens appear in cssVariables[] but have no colorToken linkage. The downstream DESIGN.md carries variable names without resolved hex provenance. A hybrid approach — walk stylesheets for declaration metadata (name, selector, raw value) PLUS resolve each --var via getComputedStyle(document.documentElement).getPropertyValue(name) in the browser runtime — closes this gap: every CSS variable gets a name + resolved-value pair. This is the single highest-fidelity improvement available.
   - Recommendation: Add a resolution pass in dom-collector.ts after the stylesheet walk: for each extracted CSSVariable, call getComputedStyle(document.documentElement).getPropertyValue(name) to capture the resolved value. Store it in a new field (resolvedValue) on the CSSVariable type (types.ts). In cluster.ts:662-675, use resolvedValue (falling back to raw value) for color matching.
   - Evidence: dom-collector.ts:105-143 (stylesheets walk, value=style.getPropertyValue(prop).trim() — raw authored), cluster.ts:662-675 (cross-reference uses parseColor(v.value) against raw, fails on var() chains), types.ts:44-49 (CSSVariable has no resolvedValue field)
2. **[P1] occurrence-band-pre-clustering** — Our cluster.ts:698-724 applies greedy OKLCH perceptual clustering (deltaE<3) sorted by frequency-desc: the highest-frequency color always wins as the cluster representative, and any lower-frequency color within deltaE<3 merges into it. This is purely perceptual-space and can merge a true brand color (#0033cc, freq=2000) with a one-off decorative blue (#0033ca, freq=4) that happens to sit just 1.8 deltaE away, losing the occurrence signal that would keep them distinct. CSS Stats' occurrence-based clustering does the inverse — it groups by how many times a value appears before applying any perceptual merge. A combined two-pass approach would: (1) bucket colors by occurrence frequency percentile (top-quartile → high-confidence brand/system candidates; bottom-quartile → suspect one-offs), (2) perceptual-cluster within each band independently, (3) cross-band: only merge low-freq into high-freq when deltaE < 1.0 (very tight). This preserves the occurrence signal that our current perceptual-only approach discards.
   - Recommendation: Refactor the clustering loop in cluster.ts:574-724 to add a pre-pass that partitions the color entries into frequency bands (e.g. top 30%, middle 40%, bottom 30%) before the greedy deltaE merge. Run perceptual clustering within each band, then do one cross-band pass at a tighter threshold (deltaE < 1.0) where low-freq merges into high-freq.
   - Evidence: cluster.ts:698-724 (greedy perceptual clustering, no occurrence-band pre-filtering), cluster.ts:400-537 (stability classifier uses frequency but only post-clustering, so incorrectly-merged clusters already lost their individual frequency signals)
3. **[P1] role-typed-css-var-dedup** — Our CSS variable extraction (dom-collector.ts:105-143) is a flat concat: every --custom-property from :root/html/body/theme selectors is dumped into one array with no grouping by role (color, spacing, font, shadow, radius). The consistency analysis in cluster.ts:1423-1445 does per-name cross-page consistency but no structural grouping. get-custom-properties-style extraction would group by name prefix (--color-* → colorTokens, --spacing-* → spacingSystem, --font-* → typographyLevels, --shadow-* → shadowTokens, --radius-* → radiusTokens) and deduplicate at the selector level (same name on same selector → keep last; same name on different selectors → record both). This maps directly to our 17-section DESIGN.md structure and eliminates the noisy mix of unrelated variables that currently floods the cssVariables[] array. Additionally, our scope is too narrow — we only check :root/html/body/theme, missing component-level custom properties (.button { --btn-radius: 8px }) which are common in shadow-DOM-free component libraries.
   - Recommendation: Replace the flat extraction in dom-collector.ts:105-143 with a prefix-grouped extractor that partitions by role prefix (--color-, --spacing-, --font-, --shadow-, --radius-, --motion-, --layout-) and removes the :root-only selector scope to capture all selectors. Each group feeds directly into the corresponding token builder in cluster.ts.
   - Evidence: dom-collector.ts:118-126 (selector filter checks only :root/html/body/theme, missing component-level selectors), types.ts:44-49 (CSSVariable has no role/group field), cluster.ts:662-675 (flat O(n×m) cross-reference without role awareness)
4. **[P2] static-property-type-discovery** — project-wallace/css-analyzer does static CSS parse to categorize every declaration by property type (colors under color/background-color/border-color, sizes under font-size/width/height, etc.) and emits occurrence counts for each distinct value AS-FOUND. Our css-analyzer.ts already has a full css-tree AST parse (line 487) but only extracts pseudo-classes, media queries, transitions, and animations — it ignores the vast bulk of property declarations for token discovery. A static-property-type pass would discover tokens that never render at runtime (unused theme variants, disabled states, print stylesheets) and provide a secondary cross-validation signal against the computed-style census: if a color appears 200 times in static CSS but only 3 times in computed elements, it's likely a special-state token (focus ring, error state) that deserves inclusion.
   - Recommendation: Add a static-token-discovery pass in css-analyzer.ts that walks the css-tree AST and collects all color/border/shadow/radius/font/spacing declarations grouped by property type, emitting occurrence counts and selectors. Feed this into cluster.ts as a secondary validation signal (not primary — keep computed-style as ground truth).
   - Evidence: css-analyzer.ts:485-492 (full AST parse exists, only extracts pseudo-classes/media/transitions/animations), cluster.ts (no static-source validation signal consumed)

## Summary
The highest-value technique to adopt is hybrid static-computed CSS variable resolution: resolve every extracted --custom-property's var() chain to a computed value so the token dictionary carries both autho red name and runtime hex provenance. This fixes the cross-referencing gap in cluster.ts:662-675 and is implementable in a single file (dom-collector.ts) with a getPropertyValue() pass. Second-priority is augmenting the perceptual clustering with occurrence-band pre-filtering (cluster.ts) to prevent high-freq system colors from absorbing genuine one-off colors at deltaE<3, and third is role-typing the CSS variable extraction to map directly to DESIGN.md sections instead of dumping a flat list.
