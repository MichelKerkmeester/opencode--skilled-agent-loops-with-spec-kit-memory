## [**1.0.0.0**] - 2026-05-15

This first release establishes `system-code-graph` as a standalone skill that owns structural code intelligence. The package was extracted from `system-spec-kit`, given a clean MCP server identity (`mk-code-index`), fully decoupled from spec-kit's source tree, and wired into every install + diagnostic surface so it can be discovered, installed, and repaired the same way as the other MCP servers. The release covers ten public tools (`code_graph_scan/query/context/status/verify/apply`, `detect_changes`, `code_graph_status/reindex/feedback`) reachable through the `mcp__mk_code_index__*` namespace.

#### Standalone MCP server identity

- Need: structural code intelligence lived inside `system-spec-kit`, mixing a graph subsystem with spec-kit's memory and lifecycle code.
- Change: the package now ships as `.opencode/skills/system-code-graph/` with a private `@spec-kit/system-code-graph` runtime, a dedicated `.opencode/bin/mk-code-index-launcher.cjs`, and a local SQLite graph at `mcp-server/database/code-graph.sqlite`.
- Why it matters: code-graph can evolve, be installed, and be debugged on its own schedule without dragging spec-kit along.

&nbsp;

#### Structural code intelligence — 10 MCP tools

- Need: callers wanted relationship answers ("what imports this?", "what calls X?", "what does this diff touch?") without falling back to plain text search.
- Change: the standalone server registers `code_graph_scan`, `code_graph_query`, `code_graph_context`, `code_graph_status`, `code_graph_verify`, `code_graph_apply`, `detect_changes`, and the `code_graph_status/reindex/feedback` CocoIndex bridge.
- Why it matters: graph reads are false-safe — when the index is stale, missing, or scope-mismatched, handlers return blocked payloads with a required next action instead of plausible empty answers.

&nbsp;

#### Spec-kit decoupling (isolation arc)

- Need: spec-kit and code-graph still shared imports across modules, so the two skills could not be released, refactored, or tested independently.
- Change: zero `from 'system-spec-kit'` imports remain in `system-code-graph` source; cross-skill types live in `@spec-kit/shared`; a hardened MCP env allowlist, atomic marker writes, and path validation closed every P0/P1 finding from the 017 deep-review. A CI isolation check (`.github/workflows/isolation-check.yml`) prevents regressions.
- Why it matters: code-graph can be removed, reinstalled, or upgraded without touching spec-kit, and spec-kit can be packaged without pulling code-graph internals.

&nbsp;

#### End-user-safe defaults with maintainer opt-in

- Need: a wide-open structural index is great for maintainers but unwanted disk + startup cost for end users.
- Change: the launcher ships with all five `SPECKIT_CODE_GRAPH_INDEX_*` flags committed as `false`. Maintainers set `SPECKIT_CODE_GRAPH_MAINTAINER_MODE=true` in `.env.local` (gitignored) and the launcher flips all five to `true` on startup. Per-call `code_graph_scan` arguments still override.
- Why it matters: new clones stay quiet by default; full-repo indexing is one env var away when needed.

&nbsp;

#### Install guide + `/doctor:mcp` coverage

- Need: there was no skill-level install guide, the master install README enumerated five of six MCP servers without listing `mk_code_index`, and `/doctor:mcp install` / `/doctor:mcp debug` only registered four servers.
- Change: authored `INSTALL-GUIDE.md` (sk-doc `skill_reference_install_guide` template, nine sections); added `mk_code_index` to the master install README (§7.1 Component Matrix, §7.3 Bundles, new §10.4 Phase-3 subsection, §19 Related Documents) and to `mcp-doctor.sh`, `doctor_mcp_install.yaml`, `doctor_mcp_debug.yaml`, and `mcp.md`. Fixed the latent `diagnose_mk-spec-memory` hyphen typo in `mcp-doctor.sh` while there.
- Why it matters: discovery, install, and runtime diagnostics now follow the same surface as `mk-spec-memory`, `mk_skill_advisor`, and the other MCP servers.

&nbsp;

#### Documentation alignment

- Need: README, ARCHITECTURE, feature catalog, and manual testing playbook needed to reflect the standalone identity and the rename.
- Change: README + ARCHITECTURE.md follow the sk-doc skill template; `feature-catalog/feature-catalog.md` and `manual-testing-playbook/manual-testing-playbook.md` are kept package-local with stable cross-skill references; `mcp-server/{handlers,lib,tools,database,tests}/README.md` describe each code surface.
- Why it matters: a single read of the package answers what it owns, how to operate it, and where each piece of code lives.

## Upgrade

No migration required for new installs. Existing configs that still reference `system_code_graph` should rename the entry to `mk_code_index`, point `command`/`args` at `.opencode/bin/mk-code-index-launcher.cjs`, and use `mcp__mk_code_index__*` for all tool references. See `INSTALL-GUIDE.md` §8 Troubleshooting for the full rename steps.
