# Established root cause (from iterations 001-004 + orchestrator-verifications — READ them; do not re-derive)
Daemon liveness is session-coupled with NO always-on supervisor. (1) re-election is spec-memory-ONLY and is release-for-adopt + reactive-respawn, NOT autonomous resurrect; an unadopted released daemon idle-times-out and dies. (2) advisor + code-index kill-child-on-shutdown (no release) and can kill what concurrent sessions bridge to. (3) the warm-probe trap: prompt-time/SessionStart hooks are warm-only → never cold-spawn → a session starting with a dead daemon reports it down all session; only a NON-warm CLI/launcher invocation cold-spawns. (4) stale sockets reaped only reactively on a fresh non-warm bind. (5) ungraceful death (SIGKILL/OOM/crash-loop give-up) → no resurrector. (6) lease is process-based, not runtime-aware.

# YOUR TASK: design FIX B = "uniform launcher-hardening, NO OS dependency"
The idea: fix this WITHIN the existing launcher/lease model (pure Node, cross-platform), WITHOUT depending on launchd/systemd (that is Fix A, covered separately). Design it concretely and map it against every failure mode.

# Design questions (be concrete — name functions/files to change)
1. UNIFORM RELEASE/ADOPT: extend spec-memory's release-for-adopt + stale-reclaim (adopt-live / reap-respawn-dead) to advisor + code-index, replacing their kill-child-on-shutdown. 2. PROACTIVE STALE-SOCKET SWEEP: add an explicit unlink-dead-same-uid-socket step at launcher start (before relying on EADDRINUSE), so a leftover socket from an unclean death is reclaimed even by the FIRST binder. Where.
3. RESURRECTOR WITHOUT A FRESH SESSION: the hard part. Without OS supervision, what brings a fully-dead daemon back when sessions are warm-only? Options to evaluate: (a) make the warm-only prompt-time path FIRE-AND-FORGET a background cold-spawn (detached) instead of only throwing 75 — "warm-probe but kick a cold start"; (b) a lightweight resident watchdog the first session detaches that outlives the session and KeepAlive-restarts the daemon; (c) escalate to non-warm after N consecutive warm failures. Pick the best + say why; note the chicken-and-egg (who keeps the watchdog alive).
4. HF SIDECAR REFCOUNT: make the shared HF model-server refcounted/owner-agnostic so spec-memory's release can't kill the sidecar advisor still needs. Where.
5. IDLE-TIMEOUT: should an UNADOPTED released daemon idle-die at all, or stay warm? Tradeoffs (resource vs availability); what to tune.
