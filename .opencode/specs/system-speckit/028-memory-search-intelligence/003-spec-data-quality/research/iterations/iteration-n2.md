# Iteration N2 - First-Class Rollup Node: Mechanism Already Exists, Content Is The Gap

## TITLE

Cohort N2 logic. Angle designs the first-class rollup-node mechanism concretely: how to materialize a track-level and a phase-parent-level rollup record that aggregates children, indexed with an embedding as a first-class result rather than the keyword-only fallback; the broad-versus-specific gate; and a measurement plan that can prove the parent-grain top-3 payoff on this corpus. The decisive finding flips the prior framing: the first-class embedded rollup record is not a green-field build. It already exists, escapes the fallback channel, and is even boosted on broad queries. The defect is the content the rollup carries, not the mechanism that carries it.

## FINDINGS

The rollup record already exists as a first-class embedded index row. A graph-metadata.json file is detected as document type graph_metadata (`memory-parser.ts:443-444`), flattened by `graphMetadataToIndexableText()` (`graph-metadata-parser.ts:1311-1353`), and indexed as a memory_index record with contextType planning and its own full embedding through the standard parse path (`memory-parser.ts:326-364`). It does not ride the community fallback lane. The community 0.02-multiplier trap lives entirely in `community-search.ts` (keyword-only `scoreSummary` at `:91-99`, member rows projected at `:285-314`); the prior iterations cite `memory-search.ts:1286` for the 0.02 multiplier, but no `memory-search.ts` exists in the tree and no literal 0.02 community multiplier exists in code — the 0.02 is a characterization of the keyword-fallback ceiling, not a live constant. The point still holds and is in fact stronger: a graph_metadata record never enters that lane at all, so the parent-grain rollup is already on the first-class side of the line the loop has been trying to cross.

Two rollup grains already have nodes on disk. The phase-parent grain: `028/graph-metadata.json:208` carries a real aggregated `causal_summary` that enumerates its nine children ("nine top-level children span four subsystem phase parents..."). The track grain: `system-spec-kit/graph-metadata.json` exists at the track root with packet_id `system-spec-kit` and children_ids listing the member packets (`:3-11`). Both are indexed today. The track node is therefore the literal track-level rollup record the angle asks for — it is just carrying the wrong content.

The defect is aggregation content, two concrete failures. First, the flattener writes children as bare IDs and stops: `Children: ${metadata.children_ids.join(', ')}` (`graph-metadata-parser.ts:1327-1329`) emits "Children: system-spec-kit/026..., 027..." with no child key_topics, no child causal summaries, no status distribution. A broad query embedding has nothing child-derived to match against. Second, the track node's own causal_summary is a stale single-packet artifact: `system-spec-kit/graph-metadata.json:50` reads "Two-bug fix to mcp-coco-index daemon..." with empty key_files and empty entities (`:48-49`) — it describes one historical packet, not the track. The track derive path produced this because `deriveCausalSummary()` (`graph-metadata-parser.ts:995-1008`) reads only the node's own spec docs, and a track root has no aggregating spec.md. So the broad query "what is the state of spec-kit retrieval" matches a node whose embedding is about a daemon bug fix.

The broad-versus-specific gate is already half-wired and is the right seam. `isPacketOrientedQuery()` (`stage1-candidate-gen.ts:324-339`) tests for terms like packet, parent, children, resume, dependency; when true, `boostGraphMetadataCandidates()` (`:341-364`) adds +0.12 to graph_metadata candidates and subtracts 0.12 from migrated ones. This is exactly a broad-query lever that lifts rollup-grain records without a new routing pipeline. It currently keys on relationship vocabulary, not breadth, so it is the correct extension point rather than a new mechanism.

The payoff is unmeasured because the eval harness cannot see the floor. Confidence truncation runs only on the production path: `if (!evaluationMode)` gates the `truncateByConfidence` call (`hybrid-search.ts:2046-2049`), with DEFAULT_MIN_RESULTS 3 (`confidence-truncation.ts:35`). The eval harness in `lib/eval/` (`eval-metrics.ts`, `rank-metrics.ts`, `k-value-analysis.ts`) measures recall@K and nDCG@K with the tail intact, so any metric the harness reports is blind to the exact 3-result window the truncation law actually taxes. That is the structural reason the parent-grain top-3 payoff is currently unmeasured: the measurement plan must report rank@1-3 specifically, not recall@K.

## CONCRETE CHANGE

(a) Aggregation, not a new record type. Change `graphMetadataToIndexableText()` (`graph-metadata-parser.ts:1311-1353`) so that for any node with non-empty children_ids it loads each child's graph-metadata.json (already on disk, already a known path shape via `resolveChildrenIds` at `:681-690`) and appends a CHILDREN ROLLUP block: each child's causal_summary first sentence, its top key_topics, and its status, plus a track-level status distribution count. This is cheap, hash-stable, zero-LLM concatenation; it reuses fields already in every child file. Separately, fix the track-root derive: when a node is a track root (children present, no own spec.md), `deriveCausalSummary()` must aggregate children rather than fall through to the stale single-packet summary — replace the `system-spec-kit/graph-metadata.json:50` daemon-bug text with a children-derived synthesis. The embedding then carries child signal and a broad query can match it. LLM synthesis of the rollup paragraph is additive on top, not required for v1.

(b) Breadth gate. Extend `isPacketOrientedQuery()` (`stage1-candidate-gen.ts:324-339`) with a breadth signal: short queries (<= 4 content tokens after stopword strip, reusing `extractQueryTerms` from `community-search.ts:105-117`) OR queries containing track/overview vocabulary ("state of", "overview", "what is", track name) also enable the graph_metadata boost; narrow queries naming a specific file, symbol, or three-digit packet number do not. Keep the existing +0.12 / -0.12 deltas. This prevents the track rollup from crowding leaf packets on narrow queries while letting it surface on broad ones. The gate is caller-cheap and already returns into a boost the pipeline applies.

(c) Measurement plan. Build a fixed query set on this corpus split into BROAD (10 queries like "state of spec-kit retrieval", "memory search work", "028 phase children") and NARROW (10 queries naming one packet, file, or symbol). Run each through the production hybrid-search path with evaluationMode off so truncation is live, and record the rank position of the relevant track/phase rollup node and of the correct leaf for every query. The single go-metric is BROAD parent-grain hit@3 before versus after the aggregation change, paired with a NARROW guardrail: leaf hit@3 on narrow queries must not regress. Use `lib/eval/rank-metrics.ts` for the position math but a new harness wrapper that runs with truncation on (the existing harness runs with `evaluationMode` true and is blind to the floor). Baseline first: the current track node carries daemon-bug content, so its expected BROAD hit@3 is near zero, which is the number to beat.

## EVIDENCE

- Rollup record already first-class and embedded: `memory-parser.ts:443-444` (graph_metadata detection), `:326-364` (parse to indexed planning record with embedding), `graph-metadata-parser.ts:1311-1353` (`graphMetadataToIndexableText`).
- Two grains exist on disk: phase parent `028/graph-metadata.json:208` aggregated causal_summary + `:6-16` children_ids; track root `system-spec-kit/graph-metadata.json:3-11` children_ids, `:50` stale daemon-bug causal_summary, `:48-49` empty key_files/entities.
- Aggregation defect: `graph-metadata-parser.ts:1327-1329` writes bare children IDs only; `:995-1008` `deriveCausalSummary` reads own docs only.
- Broad-vs-specific gate half-wired: `stage1-candidate-gen.ts:324-339` `isPacketOrientedQuery`, `:341-364` `boostGraphMetadataCandidates` +0.12/-0.12.
- Truncation law taxes retrieval only, eval blind: `hybrid-search.ts:2046-2049` `if (!evaluationMode)`, `confidence-truncation.ts:35` DEFAULT_MIN_RESULTS 3; eval infra `lib/eval/rank-metrics.ts`, `eval-metrics.ts`, `k-value-analysis.ts`.
- Community 0.02 is keyword-only fallback, not where rollup lives: `community-search.ts:91-99` `scoreSummary`, `:285-314` member projection; no `memory-search.ts` in tree (prior `memory-search.ts:1286` citation is stale).
- Brief: GraphRAG community hierarchy + summaries at `stage-0-external-findings.md:31`, candidate 6 at `:95`.

## READER

Retrieval primary. The track and phase rollup records compete at candidate generation as first-class embeddings; the change moves their content from per-node noise to child-aggregated signal so a broad query lands a rollup in the live top-3 instead of three arbitrary leaves. Logic secondary: the parent/child hierarchy in graph-metadata.json becomes machine-readable on the production path, which is the GraphRAG community purpose, achieved without a community-detection rebuild or a new index lane.

## ON-WRITE OR RETROACTIVE

Both, and cheaply. On-write: `graphMetadataToIndexableText` already runs on every graph-metadata save, so the aggregation block ships automatically with the next save of each parent/track node — no new write trigger. Retroactive: a one-time re-derive of existing track-root and phase-parent nodes (re-run the existing graph-metadata refresh over nodes with non-empty children_ids) regenerates their indexable text and re-embeds; this is the same refresh path `refreshGraphMetadataForSpecFolder` (`:1251-1296`) already uses, with its idempotent-write guard (`:1281-1287`) preventing churn on leaf nodes. The breadth gate (b) and the truncation-on harness (c) are pure code, no data migration.

## RISK

Go on (a) aggregation and (b) the breadth gate; no-go on a separate rollup record type or a new index lane — that work is already done and a parallel lane would duplicate the graph_metadata path and risk a self-referential cycle (a rollup record indexed as a member of its own community). Cheap-concat aggregation may read as a flat list rather than a synthesis; LLM synthesis is the additive follow-up, gated behind a measured win, not v1. The breadth heuristic can misclassify: a short query that is actually narrow ("envelope.ts") could pull the +0.12 boost and waste a top-3 slot — the NARROW guardrail in (c) is the gate that catches this before default-on, and the gate must stay caller-controllable. The stale track-root causal_summary (daemon-bug text) proves the derive can produce confidently-wrong content for aggregating nodes; the track-root fix in (a) must be verified to not regress leaf derives, since `deriveCausalSummary` is shared. Final go-decision is conditional on (c): if BROAD parent-grain hit@3 does not improve with truncation live, the content change is a no-op against the floor and should not default-on — measure before flipping.
