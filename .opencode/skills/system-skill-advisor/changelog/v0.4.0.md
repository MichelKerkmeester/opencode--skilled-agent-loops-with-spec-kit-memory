This release repurposes `mcp_server/scripts/check-prompt-quality-card-sync.sh` from a 3-mirror hash-comparison sync checker into a focused duplication guard covering all 4 `cli-*` prompt-quality cards. The script now asserts that the canonical framework table and the CLEAR scoring table are not inlined in any per-CLI card; it exits 0 only when none of the four cards contain either table. The old hash-comparison logic is gone.

---

## Changed

#### `check-prompt-quality-card-sync.sh` repurposed as a duplication guard

The original script compared MD5 hashes across three mirror paths to detect divergence between the canonical prompt-quality card and its copies. That design required maintaining synchronized copies and a known-good hash baseline, and it fired false positives whenever a deliberate update had not yet propagated to all mirrors.

The script now serves a different contract: it scans all 4 `cli-*` prompt-quality cards (`cli-claude-code`, `cli-codex`, `cli-devin`, `cli-opencode`) and asserts that neither the canonical framework comparison table nor the CLEAR scoring table has been inlined into any of them. Exit 0 means no duplication was found. Any non-zero exit means at least one card contains content that belongs only in the canonical card, and CI should reject the change.

The per-card scan uses targeted substring matches against the table header text unique to each canonical block, which is stable across minor wording changes and does not require an updated hash baseline after every canonical-card edit.

&nbsp;

#### Scope of the guard

| Card scanned | Path |
| ------------ | ---- |
| cli-claude-code | `.opencode/skills/cli-claude-code/references/prompt-quality-card.md` |
| cli-codex | `.opencode/skills/cli-codex/references/prompt-quality-card.md` |
| cli-devin | `.opencode/skills/cli-devin/references/prompt-quality-card.md` |
| cli-opencode | `.opencode/skills/cli-opencode/references/prompt-quality-card.md` |

The canonical framework table and CLEAR table remain in their single authoritative location and are referenced by the per-CLI cards, never copied into them.

## Backward Compatibility

The script's exit-code contract is preserved: 0 for pass, non-zero for fail. Any CI step that ran the old sync check continues to invoke the same script path. The semantics of a non-zero exit have changed (duplication detected vs. hash mismatch), but the operational response is the same: review and fix the flagged card before merging.

---

## Upgrade

No migration required. The script is self-contained. Remove any stored hash baselines that the old sync-check relied on; they are no longer read.
