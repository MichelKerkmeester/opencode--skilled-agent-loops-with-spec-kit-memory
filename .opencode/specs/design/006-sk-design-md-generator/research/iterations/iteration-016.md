# Iteration 016 — Track D (deepseek)

## Focus
Token storage/provenance disciplines (typed collections+manifest, aliasing/references, composite tokens)

## Findings
1. **[P0] Multi-file typed collections + Manifest (Penpot, Tokens Studio)** — Penpot exports one JSON file per mode with a Manifest.json linking them; Tokens Studio splits tokens by collection (color/type/space/etc.) with a manifest registry. Both make each token slot individually addressable by file+key rather than requiring a linear scan of a monolithic array. Our tokens.json currently dumps all token types (colors, typography, shadows, radii, spacing, components, gradients, breakpoints, a11y, darkMode, cssVariables) into one 4600-line flat file. A writer must traverse the entire colorTokens array to find a specific hex; there is no stable key to cite.
   - Recommendation: Split cluster.ts output: write color-tokens.json, type-tokens.json, space-tokens.json, shadow-tokens.json, gradient-tokens.json, motion-tokens.json each with a deterministic sort key (e.g. hex-key or frequency-rank index) plus a manifest.json listing collection paths, token counts, and extraction metadata. Impact: ~80 LOC in extract.ts (fs.writeFile loop), ~30 LOC in cluster.ts (return structure change to emit per-type maps). Phase: extract→write pipeline in tool/scripts/.
   - Evidence: Penpot native export manifest spec; Tokens Studio GitHub sync collection-per-file convention; DTCG 2025.10 multi-file spec §2.1
2. **[P1] Semantic aliasing with provenance chains (Specify SDTF, DTCG $ref)** — Specify's SDTF supports 50+ token types with composition and aliasing via `$ref` pointers (semantic token → primitive token). DTCG defines a `$value` with a `$ref` mechanism. Our tokens.json infers roles heuristically: colorTokens[].usedAs.{textColor,bgColor,borderColor,...} are frequency counts, not authoritative roles. A color 'used mostly as text' (textColor:294, borderColor:735) has ambiguous identity — the writer doesn't know if this is a text token or a border token. The colorRelationships.scales detect OKLCH hue-groups but don't name them. CSS variable names (cssVariableNames[]) are collected but never used to create alias chains. There is no semantic→primitive reference layer.
   - Recommendation: Add an `aliases` section to tokens.json (or a separate semantic-tokens.json): when cssVariableNames contains a pattern like --color-text-primary or --color-bg-surface, create a named alias entry `{ "key": "text.primary", "$ref": "#/colorTokens/hex:#061b31", "provenance": "--color-text-primary", "confidence": "css-var" }`. When no CSS variable exists, create inferred aliases from usedAs dominance with confidence: 'heuristic'. This gives the writer an explicit role vocabulary with provenance tags. Impact: ~60 LOC in cluster.ts (CSS var name → alias mapping pass). Phase: post-cluster in cluster.ts.
   - Evidence: Specify SDTF composition/aliasing spec; DTCG 2025.10 §3.4 Reference Tokens; W3C DTCG Editor's Draft §4.2 Aliases
3. **[P1] Composite typed tokens (Specify typography/shadow/gradient objects)** — Specify's SDTF represents typography as a single typed composite: `{ "$type": "typography", "$value": { "fontFamily": "Inter", "fontSize": "16px", "fontWeight": 400, "lineHeight": 1.5, "letterSpacing": "0%" } }`. Similarly, shadow tokens are structured `{ offsetX, offsetY, blur, spread, color }` objects, not raw CSS strings. Our tokens.json typographyLevels keeps fontFamily/fontSize/fontWeight as separate flat fields with no composite token identity; shadowTokens stores raw CSS `box-shadow` strings; gradients are flat `{ type, value, elementTag, location }` entries. A writer must re-parse these to reassemble prose descriptions, which is the assembly step where fabrication risk enters.
   - Recommendation: Transform shadowTokens from raw CSS strings into structured objects: `{ type, offsetX, offsetY, blur, spread, color, inset }` — cluster.ts already parses these values in classifyShadow() but discards the parsed components. Transform gradients into `{ type, stops: [{color, position}], angle }` — extractGradientColors() already parses stops but only collects colors. For typography, add a composite key (e.g., `heading.1` or `body.large`) mapped from typicalTags + fontSize rank. Impact: ~100 LOC in cluster.ts (structured parsing in shadow/gradient/typography passes). Phase: cluster.ts token assembly.
   - Evidence: Specify SDTF typed token schema (typography/shadow/gradient/border); DTCG 2025.10 §3.2 Composite Types
4. **[P2] Re-extract diff / drift detection (borrowable from proof.ts pixel-sampling discipline)** — Tokens Studio's GitHub two-way sync diffs the JSON structure to detect changes. Our proof.ts already does a one-way fidelity check: it screenshots the original site, extracts pixels, and computes OKLCH ΔE coverage against the extracted palette. This same pixel-sampling discipline can be repurposed: re-run extraction on the same URL, run proof.ts against the old tokens.json, and the coverage score delta becomes a drift signal. Additionally, a structural diff between two tokens.json files (new color tokens appeared, frequency rankings shifted, a hex changed by ΔE > 3) would detect CSS changes. This is a borrowable pattern: proof.ts's scoreColorCoverage() + deltaE() + image-region exclusion provides the fidelity machinery already.
   - Recommendation: Add a diff.ts script (sibling to proof.ts) that: (1) re-extracts tokens from the same URL, (2) diffs against the prior tokens.json structurally (new/missing/changed tokens by ΔE threshold), (3) re-runs proof.ts pixel sampling against the old palette to compute a coverage-drift score, (4) writes drift-report.json. This gives provenance across time: 'this site changed; these tokens drifted.' Impact: ~150 LOC new file (tool/scripts/diff.ts) reusing existing parseColor/deltaE/scoreColorCoverage from cluster.ts and proof.ts. Phase: new validate-phase tool.
   - Evidence: Existing proof.ts pixel-sampling pipeline at .opencode/skills/sk-design-md-generator/tool/scripts/proof.ts:75 (scoreColorCoverage with ΔE threshold); cluster.ts:348 (deltaE) — both reusable

## Questions Answered
- Would splitting into typed collections + manifest let a writer cite specific token slots? Yes — the manifest gives stable file+key addresses (e.g. color-tokens.json#idx-3) vs. scanning a 4k-line array.
- Does aliasing give a safer role vocabulary than OKLCH heuristics? Yes — cssVariableNames already carry the site author's semantic intent (--color-text-primary); aliasing captures this as provenance rather than discarding it after cross-referencing hexes.
- Does compositing reduce the writer's reassembly step? Yes — structured shadow objects eliminate CSS-string parsing; typography composites eliminate the fontFamily+size+weight reassembly; both close the fabrication gap.

## Summary
The highest-value single change is the multi-file typed-collection + manifest structure (P0): it gives every token a stable file+key address, which is the prerequisite for aliasing (P1) and composites (P1). Aliasing with CSS-variable provenance would then replace the fragile usedAs heuristics with authoritative role names. Extraction code impact is concentrated in cluster.ts (~270 LOC across all three P0/P1 changes) and extract.ts (~80 LOC for multi-file output), reusing the existing OKLCH/deltaE/WCAG machinery already present.
