# Iteration 032 — Track D (mimo)

## Focus
Design-system-vs-content boundary detection (cross-page prevalence gate, filter-before-cluster)

## Findings
1. **[P0] Post-hoc classification without prevalence gate leaks one-off content** — classifyColorStability (cluster.ts:400-435) scores colors AFTER all are collected and clustered. A single-page, bg-only gray like #646464 scores: freq>=50 → +10, achromatic → +10, pages<0.2 → -20, bg-only → -15 = -15 → 'content'. BUT a single-page gradient color with freq=60 scores: freq>=50 → +10, gradientColor-only → -15, single-page → -20 = -25 → 'content'. The real leak is upstream: these values enter clustering at line 698-724, get merged into cluster representatives, and inflate token counts + color relationship scales BEFORE stability is ever checked. The classification labels them 'content' but they already corrupted the system palette. Coverage-based election (design-extract, Superposition) filters BEFORE clustering — a value on 1 page never enters the token set.
   - Recommendation: Add prevalence gate in clusterTokens at cluster.ts:675 (after colorMap build, before OKLCH clustering). Filter: remove colorMap entries where pages.size < 2 AND frequency < 10 AND (usedAs dimension count <= 1). This eliminates single-page one-offs before they enter deltaE clustering.
   - Evidence: cluster.ts:400-435 (classifyColorStability), cluster.ts:698-724 (clustering), design-boundary-detect.ts:92-107 (scoreColorOverlap uses raw color lists)
2. **[P0] Frequency-only scoring allows content to masquerade as system** — classifyColorStability Signal 4 (cluster.ts:419-421): frequency >= 50 adds +20 regardless of page coverage. A color appearing 200 times on a single page (hero section, gradient stops, repeated card bg) gets +20. Combined with achromatic +10, that's +30, minus -20 (single-page) minus -15 (bg-only) = -5 → 'campaign'. But if it's bg+border (2 dimensions), the -15 vanishes: score = +30 - 20 = +10 → 'system'. A single-page, 2-dimension achromatic color at freq=50 becomes a 'system' token. Cross-page prevalence is the missing hard gate.
   - Recommendation: Replace frequency bonus at cluster.ts:419-421 with page-coverage-gated frequency: only add frequency bonus when pagesCoverage >= 0.3. Alternatively, add a hard requirement: score >= 30 requires pagesCoverage >= 0.3 (cluster.ts:429).
   - Evidence: cluster.ts:419-421 (frequency signal), cluster.ts:429-431 (layer thresholds)
3. **[P1] OKLCH clustering merges content colors into system clusters** — The greedy clustering at cluster.ts:700-724 sorts by frequency descending and merges any color within deltaE < 3 of an existing cluster. A high-frequency system blue (#0066CC, freq=500) can absorb a one-off content teal (#0077AA, freq=1) if they're within deltaE 3. The merged cluster inherits the system blue's frequency + pages, making the content teal invisible. Superposition-style pre-filtering would prevent this: unused/rare colors never enter the merge pool.
   - Recommendation: Apply prevalence gate (pages.size >= 2 OR frequency >= 10) to the sortedByFreq array at cluster.ts:701 before the clustering loop. Content colors below threshold are excluded from clustering entirely, not merged silently.
   - Evidence: cluster.ts:700-724 (clustering loop), cluster.ts:698-699 (sort by frequency)
4. **[P1] Pipeline order: classify-after-cluster loses provenance** — Current pipeline: collect → cluster (deltaE<3) → build tokens → classify stability. The cluster step merges pages sets and frequencies, so a 1-page content color merged into a 5-page system color inherits pagesCoverage=5/N. classifyColorStability then sees 5 pages and scores +30 for coverage. The original single-page provenance is erased. Running a prevalence filter BEFORE clustering (Superposition-style) preserves the boundary signal.
   - Recommendation: Restructure clusterTokens: add filter step between line 675 (CSS var cross-ref) and line 678 (OKLCH conversion). Candidates: entries with pages.size >= 2 OR (frequency >= 10 AND usedAs dimension count >= 2). Log filtered-out entries for audit.
   - Evidence: cluster.ts:569-675 (collection), cluster.ts:678-724 (clustering), cluster.ts:1639-1641 (classifyTokenStability called after everything)
5. **[P1] design-boundary-detect.ts operates on page-groups, not individual tokens** — detectBoundaries (design-boundary-detect.ts:228-311) compares PageGroups pairwise, answering 'do these page sets share a design system?' It does NOT answer 'is this specific value a design token or content?' The scoreColorOverlap (line 92-107) uses raw hex lists with rgbDistance < 15 threshold — it sees ALL colors including content. A page group with 200 content colors dilutes the overlap signal. The boundary detection and the stability classification are solving different problems with no shared signal.
   - Recommendation: Wire pagesCoverage from cluster.ts colorTokens into design-boundary-detect.ts scoreColorOverlap. Filter to only colors with pagesCoverage >= 0.3 before computing overlap. This makes boundary detection token-aware rather than raw-color-aware.
   - Evidence: design-boundary-detect.ts:92-107 (scoreColorOverlap), cluster.ts:737-753 (pagesCoverage computed but not used by boundary detect)
6. **[P2] Hybrid design: prevalence-gate + perceptual-cluster + role-by-usage** — Optimal 3-stage pipeline: (1) PREVALENCE GATE — value must appear on >=2 pages OR have frequency>=20 across >=2 usage dimensions. Eliminates ~60-80% of content values cheaply. (2) PERCEPTUAL CLUSTER — deltaE<3 greedy merge on survivors only. No content colors contaminate system clusters. (3) ROLE-BY-USAGE — classifyColorStability runs on filtered+clustered tokens. With content already removed, the scoring signals (CSS vars, achromaticity, structural tags) become reliable discriminators between infrastructure/system/campaign rather than fighting noise. This mirrors Superposition (exclude unused) + CSS Stats (cluster by occurrence) + our OKLCH stability in sequence.
   - Recommendation: Implement as 3 functions: filterContentCandidates(colorMap, minPages=2, minFreq=10) → clusterTokens(filtered) → classifyTokenStability(result). Call chain in clusterTokens at line 675.
   - Evidence: cluster.ts:569-1641 (current monolithic pipeline), types.ts:326-330 (StabilityClassification layer model)

## Questions Answered
- classifyColorStability uses 5 signals (page coverage, usage dimensions, CSS vars, frequency, achromaticity) but has no hard prevalence gate — frequency bonus can override single-page penalty, leaking content as system
- Cross-page prevalence IS a stronger primary gate than OKLCH-frequency score: it's binary (present on N pages or not), can't be gamed by high local frequency, and directly answers the design-token question
- Boundary detection SHOULD run before clustering: Superposition-style pre-filtering prevents content colors from contaminating system clusters via deltaE merge, preserving provenance
- Hybrid pipeline: prevalence-gate (pages>=2 OR freq>=20 across 2+ dimensions) → OKLCH deltaE<3 cluster → stability classification. Each stage narrows the signal for the next

## Summary
The current boundary detection is post-hoc: classifyColorStability labels tokens AFTER clustering has already merged content colors into system clusters via deltaE<3 proximity. The #646464 L4 leak and one-off gradient contamination both trace to the absence of a prevalence gate before clustering. The highest-value fix is a pre-cluster filter at cluster.ts:675 requiring pages.size>=2 OR (frequency>=10 AND 2+ usage dimensions), which eliminates single-page content before it enters OKLCH merge — matching the Superposition/CSS-Stats consensus that cross-page prevalence is the primary system-vs-content signal.
