# Iteration 1: Root cause of over-routing + the two structural slicing approaches (H1, H2)

## Focus
Diagnose WHY `sk-code` over-routes (D3=42, routed ~16-20 vs gold ~5-8), then weigh the two STRUCTURAL slicing hypotheses build-ready:
- **H1 — surface×concern slicing**: load only the DETECTED surface's slice for the matched concern instead of the union across surfaces.
- **H2 — phase-gated loading**: defer heavy implementation references until the §2 routing phase is past planning.

This iteration is deliberately narrow: it establishes the mechanism of waste from primary evidence so H3 (lazy/progressive) and H4 (cap/rank) in later iterations build on a verified root cause, not a guess.

## Actions Taken
1. Read the full machine-readable router (`smart_routing.md` §11 + the §11 preamble that documents the three intentional-lossiness points).
2. Read the per-surface prose maps (§4 Webflow, §5 Motion.dev, §6 OpenCode) and the load tiers (§3).
3. Read `SKILL.md` §2 (surface + phase detection algorithm, `STACK_FOLDERS`, language sub-detection).
4. Read `stack_detection.md` (surface precedence, non-Webflow guards) and `phase_detection.md` (Phase 0-3 lifecycle, transitions).
5. Read the live benchmark report (`skill-benchmark-report.md`) AND the raw `skill-benchmark-report.json` for per-scenario `routedCount` / `wastedCount` / `observedReads`.
6. Read `d4-ablation.json` (skill-on vs skill-off, n=2).
7. Read the phase `spec.md` and confirmed iteration/state scaffolding.

## Findings

### F1 — Root cause is RESOURCE_MAP cross-surface UNION, not a routing-accuracy defect
The `RESOURCE_MAP` keys on **intent only**; each intent value is a flat list that unions Webflow + Motion.dev + OpenCode + universal resources under one key. `IMPLEMENTATION` alone lists 20 paths spanning `references/webflow/implementation/*`, `references/opencode/shared/*`, and five `assets/opencode/checklists/*` authoring checklists. [SOURCE: .opencode/skills/sk-code/references/smart_routing.md:327-349] The §11 preamble states this explicitly: "Surface is flattened. Webflow, Motion.dev, and OpenCode resources for a shared intent are unioned under one key... a replay that sees prompt text alone will over-route relative to a single surface." [SOURCE: .opencode/skills/sk-code/references/smart_routing.md:295] So the flat dictionary itself is the structural waste source — surface is supposed to pre-filter, but the projection that the live trace exercises cannot express that filter.

### F2 — The waste is large and concentrated in the unioned IMPLEMENTATION/LANGUAGE_STANDARDS keys (primary evidence)
Per-scenario `routedCount`/`wastedCount` from the live run: SD-001 routed=20, wasted=12 (d3=0.40); LS-001 routed=16, wasted=11 (d3=0.3125); CS-001 routed=16, wasted=10 (d3=0.375); RD-002 routed=7, wasted=7 (d3=0). [SOURCE: .opencode/skills/sk-code/benchmark/live-final/skill-benchmark-report.json:84-92,162-169,330-337,230-237] Gold slice per scenario is therefore ~5-8 (routed minus wasted: SD-001=8, LS-001=5, CS-001=6). The two biggest absolute-waste scenarios (SD-001, CS-001) both have IMPLEMENTATION/ANIMATION as dominant intent, which pulls the 20-path IMPLEMENTATION union; LS-001 is a language refactor that pulls the 22-path LANGUAGE_STANDARDS union (which lists BOTH webflow/* and opencode/* language folders). [SOURCE: .opencode/skills/sk-code/references/smart_routing.md:425-447]

### F3 — D4 ablation: the over-routed slice is net-NEGATIVE precisely on the routine task, net-positive on the domain-pattern task
`LS-001` (routine TypeScript/language refactor) skill-on=0.82 vs skill-off=0.95 — the broad load HURT by 0.13. `CS-001` (cross-stack Motion.dev domain pattern) skill-on=0.88 vs skill-off=0.78 — the slice HELPED by 0.10. [SOURCE: .opencode/skills/sk-code/benchmark/live-final/d4-ablation.json:8-30] This corroborates F2 mechanistically: the same union that adds useful domain context (CS-001 Motion patterns) adds pure noise on a routine task (LS-001 loading webflow/* CSS/HTML guides alongside opencode/typescript). The harm is the cross-surface half of the union, not the volume per se.

### F4 — Surface detection ALREADY produces the exact key needed to slice the union (H1 is structurally cheap)
`SKILL.md` §2 runs surface detection FIRST via `STACK_FOLDERS` (early-return precedence: OPENCODE target/CWD > WEBFLOW markers > UNKNOWN), and the live traces confirm `observedSurface` was correct on every scenario (SD-001 WEBFLOW, LS-001 OPENCODE, CS-001 WEBFLOW; surfaceMatch=true). [SOURCE: .opencode/skills/sk-code/SKILL.md:75-110] [SOURCE: .opencode/skills/sk-code/benchmark/live-final/skill-benchmark-report.json:104-106,182-185,349-352] The surface key is available and reliable at RESOURCE_MAP-lookup time; the map simply discards it. H1 = re-key the map from `intent` to `(surface, intent)` so the lookup returns only the detected surface's slice plus surface-agnostic `universal/*`.

### F5 — H1 mechanism: nest RESOURCE_MAP under surface; Motion.dev stays an additive peer overlay
Build-ready shape: `RESOURCE_MAP[surface][intent]`, where `surface ∈ {WEBFLOW, OPENCODE}` and a separate `MOTION_DEV` overlay list is appended only when MOTION_DEV intent fires (it is already a peer category, not a surface — §5, stack_detection §1). [SOURCE: .opencode/skills/sk-code/references/smart_routing.md:141-143] [SOURCE: .opencode/skills/sk-code/references/stack_detection.md:24] Worked estimate against the live data: LS-001 (OPENCODE + LANGUAGE_STANDARDS) drops all 9 `references/webflow/{css,html,javascript}/*` paths from the 22-path LANGUAGE_STANDARDS union, leaving the opencode/* language folders + universal — directly attacking the 11 wasted paths. SD-001/CS-001 (WEBFLOW) drop the `references/opencode/shared/*` and `assets/opencode/checklists/*` authoring entries from the 20-path IMPLEMENTATION union. [SOURCE: .opencode/skills/sk-code/references/smart_routing.md:328-349,425-447]

### F6 — H1 breaks on genuinely cross-surface tasks; the union currently masks this with over-load
The one place the flat union is load-bearing: a task that legitimately touches BOTH surfaces (e.g. a `.opencode/` preview server that ships Webflow animation libs — the exact mixed-marker case §2 precedence is built for). [SOURCE: .opencode/skills/sk-code/SKILL.md:112] Because §2 precedence forces a SINGLE surface (OPENCODE wins), H1 would load only the OPENCODE slice and STARVE the Webflow-animation half. The current union accidentally covers this by loading everything. So H1 needs an explicit cross-surface escape hatch (multi-surface flag or an additive secondary-surface overlay), NOT just a single-surface lookup — otherwise H1 trades D3 waste for a D2 recall regression on mixed tasks.

### F7 — H2 phase-gating is real but lower-leverage and detection is the weak link
The lifecycle is well-specified (Phase 0 Research -> 1 Implementation -> 1.5 Quality -> 2 Debugging -> 3 Verification) with explicit transitions. [SOURCE: .opencode/skills/sk-code/references/phase_detection.md:38-45,89-95] H2 would defer the WEBFLOW "implementation trio" (`animation_workflows.md`, `implementation_workflows.md`, `performance_patterns.md`) and per-language style guides until Phase 1 is entered. BUT: (a) phase detection is the softest signal — the intent scorer only gives a `+5` boost for *unambiguous* verification/debugging/testing/quality phases (`smart_routing.md` §2 scoring); the §11 projection drops that boost entirely, and there is no positive "implementation phase" signal, so a fresh implementation prompt has no reliable phase marker to gate on. [SOURCE: .opencode/skills/sk-code/references/smart_routing.md:296] (b) The implementation trio is a HARD contract ("not a guideline") that exists to prevent SD-001-style partial coverage, so deferring it directly fights an existing anti-regression invariant. [SOURCE: .opencode/skills/sk-code/references/smart_routing.md:118-126]

### F8 — H2 attacks a smaller share of the waste than H1
The waste in the high-D3-loss scenarios is dominated by CROSS-SURFACE union members (F2/F5), not by phase-inappropriate-but-same-surface heavy refs. Phase-gating only helps the within-surface, wrong-phase fraction (e.g. loading verification checklists during implementation). For SD-001 the trio is 3 of 20 paths and is gold-required anyway; deferring it cannot reclaim the 12 wasted paths, most of which are opencode/* entries that H1 removes outright. So H2 is complementary at best and should be sequenced AFTER H1, gated on a reliable phase signal that does not yet exist.

## Recommendations (build-ready)
1. **Adopt H1 (surface-nested RESOURCE_MAP) as the primary remediation.** Re-key `RESOURCE_MAP[intent]` -> `RESOURCE_MAP[surface][intent]`, with `universal/*` and the `DEFAULT_RESOURCE` preamble surface-agnostic, and a `MOTION_DEV` overlay appended only on MOTION_DEV intent. This uses an already-reliable signal (surface, surfaceMatch=true on 3/3 live), is the documented intended behavior the §11 projection cannot express, and directly removes the cross-surface waste that F3 proves is net-negative on routine tasks. Estimated D3 impact: removes 9 webflow/* paths from LS-001's slice and the opencode/* authoring block from WEBFLOW IMPLEMENTATION — i.e. attacks the exact `wastedCount` paths.
2. **Pair H1 with a mandatory cross-surface non-starvation safeguard (do NOT ship H1 alone).** Add a `secondarySurface` / `crossSurface=true` path: when the mixed-marker case fires (a task touching both surfaces, or an explicit Motion-on-Webflow cross-stack prompt), append the secondary surface's intent slice as an additive overlay. This is the D2-recall regression guard the spec REQ-003 demands and is the only thing the current union does that H1 must preserve.
3. **Treat the drift guard as a hard constraint on any H1 change.** `sk-code-router-sync.vitest.ts` fails if any RESOURCE_MAP path is missing on disk or any routable doc stops being covered. [SOURCE: .opencode/skills/sk-code/references/smart_routing.md:291] A surface-nested map MUST keep every doc reachable under at least one (surface,intent) cell or the sync test red-flags it — this doubles as a free coverage/D2 guard.
4. **Defer H2 to a complementary follow-on, not the primary lever.** Only pursue phase-gating after H1 lands AND a positive implementation-phase signal exists; never defer the implementation trio (it is a hard contract). Frame H2 as "stop loading verification/quality checklists during implementation," a small within-surface win, not a cross-surface one.

- What is the live D3 score if we replay LS-001/SD-001/CS-001 against a surface-nested map? Need the gold sets per scenario to compute the recall-vs-efficiency frontier (REQ-002) — the benchmark stores `wastedCount` but not the explicit gold path list; recover from the scenario fixtures.
- For the cross-surface overlay (Rec 2): should it append the FULL secondary-surface intent slice, or a capped/top-ranked subset (the H4 question)? This is where H1 and H4 intersect.

## Ruled-out directions
- **Re-litigating that the §11 flattening exists** — out of charter; the strategy explicitly says investigate how to tighten loading GIVEN it, not whether it should exist. [SOURCE: .opencode/specs/skilled-agent-orchestration/122-deep-improvement-skill-benchmark-mode/011-sk-code-routing-efficiency-remediation/research/deep-research-strategy.md:9]
- **H2 as the primary fix** — ruled out this iteration: it attacks a smaller, within-surface slice of the waste (F8), and its trigger (phase detection) is the weakest signal in the router (F7), with no positive implementation-phase marker and a hard implementation-trio contract working against it.
- **Blanket resource cap independent of surface (a naive H4)** — not pursued here because F3 shows the harm is cross-surface contamination, not raw count; a count cap that kept the wrong-surface paths would still hurt LS-001. (Full H4 evaluation deferred to a later iteration.)
- **Changing surface detection** — unnecessary: surfaceMatch=true on all live scenarios (F4), so the routing key is already correct; the fix is in the map, not the detector.
