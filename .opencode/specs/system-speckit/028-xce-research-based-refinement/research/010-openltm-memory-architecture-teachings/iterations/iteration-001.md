## Mechanism
OpenLTM recall is lexical-candidate-first. `recall()` tokenizes the query, quotes every token, joins them with `OR`, then runs `memories_fts MATCH ? ORDER BY rank LIMIT 50` against FTS5; it records BM25-ish ranks as `Math.exp(rank)` because SQLite FTS5 rank is negative and closer to zero is better (`packages/openltm-core/src/db.ts:425-442`). If the FTS candidate set is smaller than the requested `limit`, semantic fallback runs when `ltm.semanticFallback !== false`, calling `getSimilarMemories(query, limit * 2, 0.5)` and merging semantic ids/scores into the same id set (`packages/openltm-core/src/db.ts:444-456`, `packages/openltm-core/src/config.ts:64-72`). Semantic search uses sqlite-vec KNN when available, otherwise brute-force cosine over stored embeddings, with a `similarity >= threshold` filter (`packages/openltm-core/src/embeddings.ts:261-301`).

The merged set is filtered by tags, category, project scope, workspace, agent, and `status = 'active'` (`packages/openltm-core/src/db.ts:464-510`). The hot path excludes the embedding blob from result rows (`packages/openltm-core/src/db.ts:521-533`). The actual query-result sorter does **not** sort by the explainer’s 40/35/15/10 blended score; it sorts by materialized `decay_score` or `computeDecayScore()` after FTS/semantic candidate generation (`packages/openltm-core/src/db.ts:545-550`). Recall also mutates access telemetry by updating `last_used_at`, `last_recalled_at`, `recall_count`, and `first_recalled_at` for returned ids (`packages/openltm-core/src/db.ts:552-560`).

Decay is fixed half-life based: importance 5 is infinite, 4 is 180 days, 3 is 90, 2 is 30, 1 is 14; score is `importance × confidence × 0.5^(days_since/half_life)` (`packages/openltm-core/src/db.ts:200-227`). Below `0.25`, memories can be soft-deprecated unless protected by importance 5 or `confirm_count >= 5` (`packages/openltm-core/src/db.ts:209-210`, `packages/openltm-core/src/db.ts:247-260`).

The explainer is deterministic and side-effect-free. It computes `totalScore = fts*0.40 + semantic*0.35 + importance*0.15 + recency*0.10`, or assigns 75% to the available retrieval channel when only FTS or semantic exists (`packages/openltm-core/src/recall/explainer.ts:1-9`, `packages/openltm-core/src/recall/explainer.ts:57-66`). It also emits `importanceBoost`, 90-day linear `recencyBoost`, and a temperature bucket: hot at `recallCount >= 10` or recalled within 7 days, warm at `>=3` or 30 days, cool at `>=1` or 90 days (`packages/openltm-core/src/recall/explainer.ts:30-43`, `packages/openltm-core/src/recall/explainer.ts:45-75`). `recall()` attaches this explainer after sorting (`packages/openltm-core/src/db.ts:570-580`).

`categorise.ts` is a content/category classifier, not verified as a recall query router. It scores six memory categories by regex keyword hits, uses best-hits/total-hits as confidence, accepts the heuristic at threshold `0.6`, otherwise calls Anthropic Haiku with the first 500 chars; on failure it returns the best heuristic guess (`packages/openltm-core/src/recall/categorise.ts:21-78`, `packages/openltm-core/src/recall/categorise.ts:86-138`). UNKNOWN whether any recall path consumes it; scoped source shows it exported as a utility (`packages/openltm-core/src/index.ts:40-42`).

The docs describe the intended posture as FTS5 BM25 primary, semantic fallback when FTS returns `< 3`, merge/dedup, then decay plus importance boost (`docs/03-architecture.md:472-483`). That conflicts with code, where fallback threshold is `< limit`, default `10` (`packages/openltm-core/src/db.ts:417-420`, `packages/openltm-core/src/db.ts:444-445`). Verdict: OpenLTM’s posture is better than pure trigger-first for ad-hoc paraphrased recall, but not better than system-spec-kit’s full fused search; the transferable gap is explainability and deficit-triggered rescue.

## Negative knowledge
- Do not replace system-spec-kit fused search with OpenLTM’s FTS-primary fallback wholesale; it is a simpler candidate-generation policy, not a superior scoring architecture.
- Do not expose an explanation score that is not the actual rank score.
- Do not downgrade FSRS to fixed half-lives.
- Do not copy the cloud classifier fallback into default local memory.
- Do not quote-and-OR every query token if preserving user FTS operators or phrase intent matters.

## Open questions
- Should `memory_match_triggers()` get semantic rescue only when trigger confidence is below threshold, or always when result count is below N?
- What exact `why_ranked` schema should unify vector, BM25/FTS5, trigger, causal, FSRS, importance, recency, and state-tier contributions?
- Should recall explanations feed the existing shadow feedback reducers so bad rankings become debuggable training data?
