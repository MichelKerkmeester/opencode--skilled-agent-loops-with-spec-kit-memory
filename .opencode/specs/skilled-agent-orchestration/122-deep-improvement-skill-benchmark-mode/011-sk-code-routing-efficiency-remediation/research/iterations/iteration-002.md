# Iteration 2: The dynamic approaches (H3 lazy, H4 cap/rank) + the quantified recall-vs-efficiency frontier

## Focus
With iteration 1's root cause fixed (RESOURCE_MAP is an intent-only cross-surface UNION; H1 surface-nesting is the high-leverage fix; a cross-surface non-starvation safeguard is mandatory), this iteration evaluates the two DYNAMIC hypotheses and quantifies the frontier from primary data:
- **H3 — lazy/progressive loading**: DEFAULT_RESOURCE preamble + top-ranked slice first, fetch more on demand. What does "on demand" mean for a *stateless router projection* vs the *live skill*? Where would the trigger live, and does it add a harmful round-trip on broad tasks?
- **H4 — anti-over-routing heuristic**: cap/rank resources per intent, drop low-signal docs for narrow tasks. What ranking signal exists? What is the recall risk, and how does H4 intersect H1's cross-surface overlay (full slice vs capped subset)?
- **The recall-vs-efficiency frontier (REQ-002)**: recover the per-scenario GOLD resource sets so D3/D2 trade-offs are computed, not hand-waved. Define the D1/D2 regression guard (REQ-003) concretely.

## Findings (cited)

### F9 — The scorer's gold for D3/D2 is `expectedResources` ONLY; `expectedAssets` is parsed but never scored
`scoreScenario` builds `expected.resources` from `scenario.expectedResources` alone. [SOURCE: .opencode/skills/deep-improvement/scripts/skill-benchmark/score-skill-benchmark.cjs:58] The playbook loader fills `expectedResources` from the "Expected references loaded" block and `expectedAssets` from a separate "Expected assets loaded" block, and they are never merged. [SOURCE: .opencode/skills/deep-improvement/scripts/skill-benchmark/load-playbook-scenarios.cjs:125-134,153-154] Consequence for the frontier: **every `assets/*` path the router emits is counted as waste**, because no scenario's gold contains an `assets/*` entry — D3 structurally penalizes the RESOURCE_MAP for listing checklists/recipes/templates regardless of whether they are genuinely useful. SD-001's IMPLEMENTATION slice alone carries 5 `assets/opencode/checklists/*` + `assets/opencode/recipes/spec_folder_write.md` + 4 `assets/*/README.md` = pure scored-waste under this harness. [SOURCE: .opencode/skills/sk-code/references/smart_routing.md:339-348] This is a measurement seam, not a routing defect, and it shapes what H3/H4 should optimize.

### F10 — D3/D2 use EXACT string set-membership; a path is gold-hit only if the routed string equals a gold string verbatim
`setRecall` counts a hit only when the gold path is in `new Set(actual)`, and `wasted` counts a routed path only when it is NOT in `expected.resources`. [SOURCE: .opencode/skills/deep-improvement/scripts/skill-benchmark/score-skill-benchmark.cjs:24-27,107] No prefix/glob matching. So a ranking heuristic (H4) that keeps a "close" path does not partially score — recall is all-or-nothing per path. This means H4 dropping ANY gold-listed path is a hard D2 hit, and the RESOURCE_MAP slice and the playbook gold must agree path-for-path to score well.

### F12 — NEW & load-bearing: D2 recall is ALREADY broken on SD-001 (0.727) and CS-001 (0.60); the union does NOT achieve full recall
This overturns the iteration-1 working assumption that the union "masks" recall by over-loading. SD-001 D2=0.727 and CS-001 D2=0.60 mean the routed set is MISSING gold paths even while over-routing. [SOURCE: .opencode/skills/sk-code/benchmark/live-final/skill-benchmark-report.json] Mechanism: LS-001's gold lists `references/opencode/shared/{code_organization,universal_patterns}.md`, which live in IMPLEMENTATION, not LANGUAGE_STANDARDS — yet LS-001 routes LANGUAGE_STANDARDS, so those two gold paths should be misses, but LS-001 scored D2=1.0; that is because the substring scorer also fired IMPLEMENTATION/CODE_QUALITY keywords (e.g. "standards") pulling `code_organization`/`universal_patterns` in via a co-selected intent. Conversely SD-001/CS-001 route gold paths (`webflow/css/quality_standards.md`, `motion_dev/scroll_and_gestures.md`, `webflow/verification/*`) that are NOT in the fired intents' RESOURCE_MAP slices, so they are genuine recall misses. **Implication for the regression guard (REQ-003): the guard must be "D2 must not DROP below the current per-scenario baseline" (0.727 / 0.60 / 1.0 / 1.0), NOT "D2 must stay at 1.0" — because it was never 1.0.** A naive H4 cap that also happens to fix recall is possible, but the guard floor is the measured baseline, not perfection.

### F13 — H3 (lazy/progressive) has NO trigger surface in the stateless router projection; "on demand" is undefined for the replay
The benchmark D3/D2 are produced by `routeSkillResources`, which is a single synchronous pass: score intents -> select within `AMBIGUITY_DELTA` -> union DEFAULT_RESOURCE + RESOURCE_MAP[intent] -> return the full list. [SOURCE: .opencode/skills/deep-improvement/scripts/skill-benchmark/router-replay.cjs:169-185] There is no notion of a follow-up turn, no observed-need feedback, no "fetch more" event — the router returns one set and is scored on it. So **H3's "on demand" can only live in the LIVE skill (the AI reading SKILL.md across turns), never in the replay that produces the D3 number.** Two hard consequences: (a) H3 cannot improve the benchmark D3 at all unless the lazy preamble itself emits a *smaller routed set* in the single replay pass — i.e. H3 collapses into "ship a smaller default slice," which is H4 by another name; (b) on a genuinely broad task the live AI would need a second read round-trip to recover the deferred slice, and the harness has no way to credit or debit that, so H3's real cost (an extra turn) is invisible to the metric and its only measurable effect is whatever it removes from the first pass. H3 is therefore NOT independently measurable on this benchmark and should not be ranked as a distinct lever; its measurable core is identical to H4's "smaller first slice."

### F14 — H4 (cap/rank) IS the only dynamic lever that moves the scored D3, and it has exactly two viable ranking signals
The replay exposes two signals usable as a ranking key without new detection: (1) **intent score** — `scoreIntents` already returns a numeric per-intent score; `selectIntents` keeps everything within `AMBIGUITY_DELTA=1` of the top, so a tie-broken or thresholded selection (drop intents strictly below top, not top-1) directly shrinks the union. [SOURCE: .opencode/skills/deep-improvement/scripts/skill-benchmark/router-replay.cjs:146-163] (2) **surface match** — once H1 nests the map, surface is the dominant rank: drop wrong-surface paths first (this is H1, and it is where 9 of LS-001's 11 wasted paths die, iteration-1 F5). The +5 phase boost the projection drops (iteration-1 F7) is NOT available as a ranking signal in the replay, so phase cannot rank here. **Ranking recommendation: H4 = surface-rank (subsumed by H1) THEN intent-score-rank (tighten `AMBIGUITY_DELTA` or drop sub-top intents), applied only to references; do NOT cap by raw count.** A raw-count cap risks dropping gold (F10's all-or-nothing recall), whereas surface+intent ranking drops precisely the cross-surface / weak-intent paths that F9/F11 show are the waste.

### F15 — H1×H4 intersection (the open question from iteration 1, now answered): the cross-surface overlay must be the FULL secondary-surface intent slice, NOT a capped subset
Because the scorer recall is exact and all-or-nothing (F10) and CS-001's gold REQUIRES all 4 `motion_dev/*` references plus the webflow implementation + verification refs (D2 already only 0.60, F12), a capped/top-ranked overlay would drop gold and push CS-001's D2 BELOW its 0.60 baseline — violating the REQ-003 guard. CS-001 is the canonical cross-surface scenario and it is the one with the WORST recall today. [SOURCE: .opencode/skills/sk-code/manual_testing_playbook/07--cross-stack-routing/webflow-plus-motion-dev.md:34-45] [SOURCE: .opencode/skills/sk-code/benchmark/live-final/skill-benchmark-report.json] Therefore: **when the cross-surface flag fires, append the secondary surface's FULL intent slice (H1's overlay wins); apply H4 intent-score ranking only WITHIN a single surface's selection, never to prune the cross-surface overlay.** This is the precise boundary between H1 and H4: H4 ranks intra-surface; H1's overlay is unranked and additive.

### F16 — RD-002 is a negative/advisor scenario and is OUT of the D3 frontier; it should not be optimized against
RD-002 is the sk-code-vs-sk-doc anti-pattern (a doc-edit prompt that must route to sk-doc, advisor top-1 != sk-code). [SOURCE: .opencode/skills/sk-code/manual_testing_playbook/manual_testing_playbook.md:202-207] Its `classKind` is `advisor`, `expectedSurface` is None, `firstFailingStage=activated-inter`, and its D3 (routed 7, wasted 7, score 0) is the negative-activation branch where "wasted = everything routed" is the intended suppression signal, not an over-routing defect. [SOURCE: .opencode/skills/sk-code/benchmark/live-final/skill-benchmark-report.json] [SOURCE: .opencode/skills/deep-improvement/scripts/skill-benchmark/score-skill-benchmark.cjs:112-113] Any H1/H4 change must leave RD-002 untouched (it is governed by the `sk-code -2 / sk-doc +3` anti-signal pre-filter the projection drops, iteration-1 F-context). The frontier is the 3 positive routing scenarios (SD-001, LS-001, CS-001) only.

## Recommendations (build-ready)
1. **Rank H4 as the SECOND lever, scoped to references + intra-surface only; subsume H3 into it.** H4's measurable core = "shrink the first routed slice." H3 (lazy fetch) adds an invisible-to-metric round-trip with no replay trigger (F13), so do not implement H3 as a separate mechanism; implement its only measurable benefit (smaller first slice) as H4's intent-score ranking. H4 ranking key = surface (from H1) > intent score (`scoreIntents`) ; drop sub-top intents instead of keeping the full `AMBIGUITY_DELTA=1` band, OR tighten the delta. [SOURCE: router-replay.cjs:146-163]
2. **The cross-surface overlay is unranked and FULL (H1 wins the H1×H4 boundary).** When `crossSurface=true` (CS-001 class), append the entire secondary-surface intent slice; apply H4 ranking only within a single surface's selection. This protects CS-001's already-fragile D2=0.60 baseline (F15).
3. **REQ-003 regression guard, concrete:** after any H1/H4 change, re-run the deterministic router-replay over SD-001/LS-001/CS-001 and assert **D2 per-scenario >= {SD-001:0.727, LS-001:1.0, CS-001:0.60}** (the measured baseline floor, NOT 1.0 — F12) AND **D1-intra surfaceMatch stays true** AND the `sk-code-router-sync.vitest.ts` coverage test stays green (every routable doc reachable under >=1 (surface,intent) cell — iteration-1 Rec 3). Target: D3 up from {0.40,0.312,0.375} toward >=0.6 without any D2 drop.
4. **Decide whether to fix the asset-scoring seam (F9) BEFORE optimizing, or accept it.** Because `expectedAssets` is never scored, the harness will keep counting every routed `assets/*` path as waste. Two options for the follow-on build phase: (a) treat all `assets/*` as deferrable (move them out of the first routed slice — a clean H4 win that costs zero real recall since they are not gold), or (b) fold `expectedAssets` into the scored gold so checklists/recipes can score as hits. Option (a) is the lower-risk efficiency win and aligns with H4; option (b) is a benchmark-fidelity change outside this remediation's scope. Recommend (a) for the remediation, flag (b) as a separate benchmark-quality finding.

## Ruled-out directions
- **H3 as an independent, separately-ranked approach** — ruled out (F13): the stateless replay has no "on demand" trigger; H3's only measurable effect is identical to H4's smaller-first-slice, and its real cost (an extra live round-trip) is invisible to the benchmark. Fold it into H4; do not build a separate lazy-fetch mechanism for this remediation.
- **Raw-count resource cap (naive H4)** — ruled out (F10/F15): exact all-or-nothing recall means a count cap can drop gold and breach the D2 floor, especially on CS-001 (D2=0.60 baseline). Rank by surface+intent, never by count.
- **Capping/top-ranking the cross-surface overlay** — ruled out (F15): CS-001 needs the FULL motion_dev + webflow slice; a capped overlay pushes its D2 below baseline.
- **Optimizing RD-002 into the frontier** — ruled out (F16): it is a negative advisor scenario governed by the doc-only anti-signal, not an over-routing case; its D3=0 is the intended suppression signal.
- **Re-deriving H1 / re-litigating the §11 flattening** — out of charter and already settled in iteration 1.
