# Iteration 023 — Track D (mimo)

## Focus
Agent-consumption format (structured tokens primary, tokens.css output, MCP endpoint, DESIGN.md as rendered view)

## Findings
1. **[P0] Invert primary artifact: tokens.json → deterministic DESIGN.md (doc-as-token-view)** — Every analyzed platform (Supernova, Backlight, Zeroheight, Knapsack) renders docs FROM tokens as live-bound views — the doc is a projection of data, not a free-write. Current pipeline has AI WRITE the entire 17-section DESIGN.md including value-tables (colors, typography, shadows, radii, spacing), which is the primary hallucination vector: validate.ts catches phantom hex codes but NOT fabricated fontSize/fontWeight/lineHeight values. Diez DocsGen generates token documentation deterministically from typed token data with zero inference. The fix: split WRITE into (1) AUTO-RENDER: deterministic formatters.ts generates Sections 2-8 as markdown tables directly from tokens.json (zero AI), (2) AI-ANNOTATE: AI writes only narrative sections (0, 1, 5, 7, 8, 9-16) where judgment is required. tokens.json becomes the source-of-truth primary deliverable; DESIGN.md becomes a rendered view. DTCG $type/$value wrapping of tokens.json unblocks type-gated validation + downstream emit interop (Style Dictionary, Terrazzo, Theo all consume DTCG natively).
   - Recommendation: New tool/scripts/dtcg-wrap.ts post-cluster (maps ColorToken→{$type:'color',$value:hex,...}). New tool/scripts/formatters.ts with formatColorTable(), formatTypographyTable(), formatShadowTable(), formatComponentTable(). Modify WRITE phase in SKILL.md + design_md_format.md to split into auto-render + AI-annotate. Update validate.ts to type-filter before checking. Phase: extraction output refactor (tool/scripts/).
   - Evidence: iteration-011 P0 findings (DTCG + deterministic tables), iteration-017 P0 (doc-as-token-view: Supernova/Backlight/Zeroheight/Knapsack), iteration-020 P1 (value-only export: ~70% context reduction for agents), iteration-004 P0 (prose-provenance gap as primary hallucination vector)
2. **[P1] Emit tokens.css alongside tokens.json (CSS custom properties output)** — extract-design-system (arvindrk) emits tokens.json + tokens.css as dual output — tokens.css is a :root block of CSS custom properties that agents can @import directly. The transform from tokens.json→tokens.css is trivial (iterate tokens, emit --color-primary: #533afd; etc.). Open Design's design system format already uses tokens.css as its structured token deliverable (proven in anobel.com/004-bento-visuals: 120+ CSS custom properties covering brand colors, ramps, typography, radii, surfaces). CSS custom properties are directly consumable by any frontend agent without JSON parsing — the agent just references var(--color-primary). This gives agents a zero-friction import path while DESIGN.md remains the human-readable documentation.
   - Recommendation: New tool/scripts/emit-css.ts: read tokens.json, emit tokens.css with :root { } block. Group by category (--color-*, --font-*, --shadow-*, --radius-*, --spacing-*). Include semantic aliases (from iter-011 P1 aliasing finding). Run as Phase 1b after extract.ts. Output alongside tokens.json in extraction directory. File: tool/scripts/emit-css.ts (new), tool/scripts/types.ts (add CSS output config).
   - Evidence: extract-design-system (arvindrk) dual-output pattern (iter-018 P1), anobel.com/004-bento-visuals tokens.css (120+ custom properties proven in production), Open Design design system format (tokens.css as :root block)
3. **[P1] Cite-the-token guardrail + evidence quotas for AI prose sections** — extract-design-system avoids hallucination by never asking the LLM to write prose — output is structured data only. Dembrandt is pure CLI with zero AI. Our hybrid architecture (CLI extract → AI write → CLI validate) keeps AI prose as value proposition but the WRITE prompt (design_md_prompt_template.md) has only 6 cardinal rules with weak anti-fabrication. Current Sections 1 (Visual Theme: 200-300 words of LLM-invented metaphors), 7 (Content & Voice: 'vibe paragraph' from LLM interpretation), and 0 (Brand Context: LLM-selected personality adjectives) are high-fabrication-risk. The fix: every prose claim must parenthetically cite >=2 token values. Every named design principle must cite >=3 token paths from tokens.json. Section 7's 'vibe paragraph' replaced with observed-data-only format (actual UI text samples, tone derived ONLY from tokens).
   - Recommendation: Modify assets/design_md_prompt_template.md: add 'Claim Evidence Required' as cardinal rule #7. Modify tool/resources/design_md_format.md: add hard evidence quotas to Sections 0, 1, 7. Enhance tool/scripts/validate.ts to parse citation annotations and verify against tokens.json. Modify Section 7 to observed-data-only format. File: assets/design_md_prompt_template.md, tool/resources/design_md_format.md (Sections 0, 1, 7), tool/scripts/validate.ts.
   - Evidence: iteration-018 P0 (fabrication-surface analysis: 3 high-risk sections), iteration-018 P1 (extract-design-system prompt-driven grounding), iteration-017 P1 (cite-the-token guardrail from Zeroheight), iteration-009 (comparative framing as fabrication mandate)
4. **[P2] MCP token endpoint for structured agent queries** — Supernova exposes an AGENT-READY MCP endpoint so consuming agents read tokens directly rather than trusting prose descriptions. If consuming agents (sk-code, Cursor, Windsurf) could query tokens.json via a structured endpoint instead of parsing markdown, prose drift becomes irrelevant for numeric lookups — the doc becomes documentation-of-record for humans; agents read the source. Lower priority because our agents already read tokens.json directly in practice, and the P0 deterministic-tables change eliminates most prose-trust issues for value sections. A CLI fallback (query-token.ts --key 'colors.primary' --format json) provides 80% of the value without MCP infrastructure.
   - Recommendation: Phase 1: add tool/scripts/query-token.ts CLI that reads tokens.json and returns structured lookups (--key path, --format json|css). Phase 2 (if agent demand warrants): expose as lightweight MCP tool via existing mcp-open-design transport or standalone. File: tool/scripts/query-token.ts (new), optional MCP registration.
   - Evidence: iteration-017 P2 (Supernova AGENT-READY MCP endpoint), iteration-023 prompt (MCP endpoint question), mcp-open-design/SKILL.md (existing MCP transport that reads whole files but not per-token queries)
5. **[P2] Dual-layer document: structured summaries atop prose sections** — Open Design's DESIGN.md is consumed by 4+ coding agents with a DETERMINISTIC RE-RENDER schema — same input produces same output regardless of which agent reads it. Our prose-heavy sections (1, 7, 0) produce different interpretations per agent: Claude Code might extract 'shadow-as-border' as a technique; Codex might extract it as a design philosophy. Section 13 (Agent Prompt Guide) already proves structured consumption works: self-contained prompts work identically for any agent. The fix for remaining prose sections: add a 3-5 line key-value 'Agent-Oriented Summary' block at the top of each prose-heavy section, with parseable key-value pairs that any agent can consume without NLP.
   - Recommendation: Modify tool/resources/design_md_format.md: add 'Structured Summary' requirement for Sections 1, 3, 5, 6 — a 3-5 line key-value block at each section's top. Prose follows as human context. This makes DESIGN.md dual-purpose: machine-consumable summary + human-readable detail. File: tool/resources/design_md_format.md (Sections 1, 3, 5, 6 header format).
   - Evidence: iteration-018 P1 (multi-agent output structuring: Open Design deterministic re-render), iteration-018 P1 (Section 13 Agent Prompt Guide as proven pattern), vercel/DESIGN.md prose metaphors as agent-confusion examples

## Questions Answered
- Is prose-heavy DESIGN.md the wrong primary artifact for AI agent consumption? → YES — tokens.json should be primary; DESIGN.md a deterministic render + AI annotation overlay. (iter-011, iter-017, iter-020)
- Should sk-design-md-generator emit tokens.css? → YES — trivial transform from tokens.json, directly importable by agents, proven in anobel.com production. (extract-design-system pattern, anobel/004-bento-visuals)
- Would an MCP token endpoint eliminate the prose-trust problem? → Partially — it eliminates numeric lookup trust but is lower priority than fixing the WRITE phase to be deterministic for value sections. CLI query-token.ts gives 80% of value. (iter-017 P2)
- What borrowable patterns exist from extract-design-system's skill architecture? → Prompt-driven grounding: front-load token constraints so every prose claim cites evidence; never let the LLM free-write numeric values. (iter-018 P1)

## Summary
The highest-value consumption-format change is inverting the artifact hierarchy: make tokens.json the primary deliverable with DESIGN.md as a deterministic render (auto-generated value-tables from typed tokens + AI-written narrative overlay). This eliminates the two primary hallucination vectors (phantom colors AND fabricated typography/shadow values) by keeping AI out of value-table generation entirely. The pipeline impact is a WRITE-phase split into auto-render (formatters.ts over DTCG-typed tokens) and AI-annotate (evidence-quoted prose for narrative sections only), with tokens.css as a trivially-generated secondary output for direct agent import.
