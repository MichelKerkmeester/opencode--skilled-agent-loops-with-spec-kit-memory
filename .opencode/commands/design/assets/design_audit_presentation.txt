# Design Audit Presentation Contract

Presentation source of truth for `/design:audit`. The router owns mode selection and asset loading. Workflow YAML owns execution. This file owns visible prompts, result displays, and next-step wording.

## 1. Startup Presentation

For `:confirm`, or no suffix with incomplete `$ARGUMENTS`, the consolidated setup prompt is the first visible response. Ask all applicable questions once, then wait.

For `:auto`, or no suffix with complete `$ARGUMENTS`, do not show the consolidated prompt. Resolve setup through the Auto Resolution Table below.

### Auto Resolution Table

| Field | Required | Default | Targeted-ask when |
|-------|----------|---------|--------------------|
| target | Yes | none | Always, when missing |
| scope | No | full | Never - defaults silently |
| score | No | false | Never - defaults silently |
| register | No | auto (declared field -> task cue -> surface in focus) | Register is genuinely mixed or unresolved |

### Consolidated Prompt Template

```text
Design Audit setup

Q1. What is the audit target (URL, component, screen, or file), and what scope/score do you want?
Q2. Is this Brand or Product register?

Reply format: "<target>, <scope or ->, <score y/n>, <brand|product>"
```

Never split these questions into separate visible prompts.

### Auto Fail-Fast Display

```text
Design Audit auto setup failed
Missing required input: audit target
Provide the target or rerun with :confirm.
STATUS=FAIL ERROR="missing required setup inputs"
```

## 2. Results Display

### Success

```text
STATUS=OK PRODUCES="Design Quality Audit Report" NEXT=/design:foundations,/design:interface,/design:motion PROOF=target,evidenceInventory,severityFindings,qualityScore
```

### Missing Input

```text
What is the audit target (URL, component, screen, or file), and what scope/score do you want?
STATUS=ASK MISSING=<input>
```

### Failure

```text
STATUS=FAIL ERROR=<named-cause>
```

### Defer

```text
STATUS=DEFER ROUTE=<hub|sibling>
```

## 3. Next-Step Suggestions

| Condition | Suggested Command | Reason |
|-----------|--------------------|--------|
| Findings need static token, contrast, or system decisions before another pass | `/design:foundations` | Fix static-system findings |
| Findings point to a changed interface direction rather than local fixes | `/design:interface` | Revisit direction |
| Findings involve animation, timing, state feedback, or reduced-motion behavior | `/design:motion` | Fix motion findings |
| Accepted design findings move to implementation | sk-code handoff via `.opencode/skills/sk-design/shared/sk-code-handoff.md` | Hand off to build |

This command never silently chains; it emits options only (`HANDOFF_REQUIRED=false`, recommend-only).

---

## 4. Example Usage

```
/design:audit src/components/Checkout.tsx --scope a11y --score
```

Returns: a findings-first design quality report with evidence, scores, owners, and verification notes.
