# Iteration 6: Interface structure, enrichment, assets, and delivery contract

## Focus

This iteration deep-mapped Hallmark's interface-facing structure and asset packet to sk-design's existing `design-interface` mode, private procedure cards, shared creation contract, real-UI loop, content gate, and mechanical preflight. It also sampled representative subordinate macrostructure and component files to test whether the useful unit is the catalogue, the packet schema, or the code snippets.

The selected interpretation is surgical reuse: improve existing Interface assets and handoffs without adding a public Hallmark-style builder, a preset theme picker, or a parallel component library. Roadmap-only capability ranking remains deferred to a later iteration.

## Findings

1. **Hallmark's strongest structural contribution is a whole-page choice before component styling, not its list of 21 names.** `macrostructures.md` binds heading placement, body composition, dividers, controls, imagery, reveal, navigation, and footer into a single page fingerprint, then loads only the selected packet. sk-design already rejects the median at candidate index 0 and asks wireframes to differ structurally, but it does not require a named page-shape record that carries regions, proof placement, navigation/footer treatment, and responsive collapse into handoff. Adapt the idea into `design-interface/references/design-process/variation-diversity.md` and `procedures/wireframe-exploration.md`: each candidate should declare `pageShape`, `regions`, `proofPlacement`, `navFooterTreatment`, `responsiveCollapse`, and `signatureMove`; load one matching private recipe only after selection. Do not copy all Hallmark names or its CSS stamp. 5. **Hero enrichment exposes a direct sk-design contract bug: mandatory media can itself be the AI default.** Hallmark defaults to typography-only, requires media to answer a brief signal, and uses a tiered source decision only after image need is established. sk-design's preflight currently requires every hero to have a real visual, and its content gate says a pure-text page is incomplete. Replace both absolute checks with an evidence-based rule: `signature role resolved = typographic | owned media | real product proof | generated asset | placeholder`; require source, subject, crop/aspect, mobile collapse, and reduced-motion behavior when media is present. Typography-only passes when the chosen page shape and brief rationale make typography the signature; a generic text-plus-gradient blob still fails. This is a high-value, low-to-medium-effort correction in `assets/interface-preflight-card.md`, `references/design-process/copy-and-mock-data.md`, and `procedures/aesthetic-direction.md`. ## Licensing and Provenance Boundary

Hallmark's repository-level MIT license permits copying, modification, and redistribution of covered source text/code if the copyright and permission notice accompany copies or substantial portions. Therefore `COPY` would be legally possible for covered files with notice retention, but this matrix deliberately prefers `ADAPT` or `LEARN` because sk-design already has stronger ownership and proof contracts. Hosted imagery, linked libraries, photographs, logos, quotes, and generated outputs are separate artifacts; the repository license does not establish redistribution rights for them. No matrix row recommends copying those assets. ## Ruled Out

- Copying all 21 macrostructures and roughly 50 component packets as a second component library. The durable value is the packet schema and selection distinctions; the raw catalogue would create parallel doctrine and implementation ownership.
- Preserving the Hallmark stylesheet stamp or diversification log. sk-design already carries accepted decisions and evidence in typed envelopes; source comments should explain durable implementation reasons, not workflow history.
- Requiring imagery on every hero or forbidding typography-only interfaces. This creates the same generic-media pressure the anti-slop workflow is meant to remove.
- Hotlinking or redistributing the Hallmark imagery kit. The manifest is a placeholder and the hosted binaries are not covered by the checked-in source alone.
- Importing exact vendor recommendations, price claims, fixed tilt/icon/file-size numbers, or universal medium priority. These are volatile or brief-dependent.
- Copying sample HTML/CSS/JS as Interface output. The shipped mode is advisory; implementation belongs to an accepted `sk-code` handoff.

## Dead Ends

No research path was fully dead. Broad catalogue import was eliminated as an implementation direction, but the load-on-demand packet architecture remains productive.

## Edge Cases

- Ambiguous input: `custom-craft.md` can be read as an asset policy or an implementation cookbook. This iteration treated its durable value as asset-selection and proof logic; code recipes remain downstream examples.
- Contradictory evidence: Hallmark says typography-only is always acceptable, while sk-design currently requires hero imagery; Hallmark favors hand-built CSS/SVG while sk-design broadly bans hand-rolled SVG fallback. The resolution is intent and evidence: a designed, brief-grounded signature can be typographic or custom-crafted, while ungrounded placeholder craft still fails.
- Missing dependencies: Hallmark's imagery manifest names hosted outputs that are not checked into the external snapshot. The fallback was to assess the documented manifest and hosting contract, not the binaries.
- Partial success: The Interface-facing asset set is mapped. ROADMAP capabilities and final cross-iteration prioritization remain for later iterations, so this iteration answers only the Interface portion of the complete-asset question.

## Reflection

- What worked and why: Comparing output schemas and proof ownership before comparing recipe content separated transferable decision logic from Hallmark-specific code and taste doctrine. Sampling packets from multiple families verified that the load-on-demand architecture, not any single archetype, is the scalable contribution.
- What did not work and why: Broad heading inventories hid internal contradictions and could not distinguish a deliberate custom asset from a fallback. Narrow line reads against sk-design's exact preflight and content gates exposed the actionable contract changes.
- What I would do differently: In the roadmap pass, start from existing sk-design private procedures and command outputs, then classify each proposed capability as a missing public route, a private card extension, or a non-goal before evaluating implementation cost.

## Recommended Next Focus

Map Hallmark's ROADMAP and remaining new-capability proposals—variant generation, brand-first authored design artifact, structural cookbook, data visualization, tactile-rebellion, and image-generation hooks—against existing sk-design commands and private procedures, then rank only the genuinely missing capabilities.
