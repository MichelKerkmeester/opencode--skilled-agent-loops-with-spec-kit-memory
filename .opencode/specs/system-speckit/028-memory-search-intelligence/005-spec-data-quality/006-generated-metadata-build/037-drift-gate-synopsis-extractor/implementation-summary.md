## What Was Built

Status COMPLETE. Both behaviors shipped behind the default-OFF `SPECKIT_GENERATED_METADATA_DRIFT_GATE` flag with a grandfather report mode. With the flag off, generated bytes are unchanged.

### Shared synopsis extractor

One shared `derivePacketSynopsis(specContent, field)` helper now lives in `mcp_server/lib/description/packet-synopsis.ts` with explicit precedence (Overview paragraph, then Problem/Purpose first sentence, then frontmatter description, then title heading, then first body line) and field-specific length limits (`SYNOPSIS_FIELD_LIMITS`: description 150, causal_summary 600). With the flag on, `extractDescription` in `folder-discovery.ts` and `deriveCausalSummary` in `graph-metadata-parser.ts` both route through it, so the two fields derive from the same precedence over the same source doc, each honoring its own ceiling. A unit assertion confirms flipping the spec.md Overview moves both fields together. With the flag off both fields keep their legacy local extractors, byte-identical. This implements proposal 11.

### Generated-metadata drift gate

`checkGeneratedMetadataDrift(specFolder)` lives in `mcp_server/lib/graph/generated-metadata-drift.ts` (re-exported from the parser and the api barrel). It reads the stored `description` and `causal_summary`, re-derives them fresh through the shared extractor, and returns a drift report. It never writes the folder, proven by a byte-snapshot no-write assertion. It is paired with `source_doc_hashes`, a sha256-per-source-doc freshness key persisted on `graph-metadata.json` under `derived` (flag-gated, optional in the schema): when the stored hashes match the current docs the re-derive is skipped, the cheap first check. `resolveGeneratedMetadataDrift` maps drift to a non-blocking `info` under grandfather mode and a blocking `error` when enforced. This implements proposal 10.

### Guarded rollout and report-only wiring

The drift gate is wired as a report-only read into strict validation (a new `GENERATED_METADATA_DRIFT` rule in `validate.sh` and the validator registry, via the `scripts/validation/generated-metadata-drift.ts` bridge) and into dry-run backfill (a `drift` array on the backfill summary). The bridge sets grandfather mode from `!isGeneratedMetadataDriftGateEnabled()`, so with the flag off drift resolves to `info` and the strict exit code is unchanged, and only with the flag on does it become a hard error. The calibration re-fit recommendation from 031 was documentation-only and is intentionally not applied: the research proved the curve cannot move the verdict.

## How It Was Delivered

The shared `derivePacketSynopsis` helper was authored first, then both fields routed through it behind the flag, then `checkGeneratedMetadataDrift` plus `computeSourceDocHashes` and the optional `source_doc_hashes` schema field, then the report-only wiring into the strict validator and the backfill summary. Each behavioral change is gated on `isGeneratedMetadataDriftGateEnabled()` so a flag-off derive stays byte-identical. The drift-detection, no-write, shared-extractor precedence, and grandfather report-mode proofs all land in the vitest.

<!-- ANCHOR:decisions -->
## Key Decisions

| Decision | Why |
|----------|-----|
| Ship behind a default-OFF flag and a grandfather report mode | Many existing folders carry drifted or divergently-extracted synopsis text, so a hard gate flipped on at once would mass-fail them |
| The gate reports drift and never writes | A re-derive that wrote would churn the generated files the 031 research exists to stop churning |
| One shared `derivePacketSynopsis` for both fields | The two fields use two extractors today, so they diverge from the same source doc, and one helper with field-specific limits makes them agree |
| Persist `source_doc_hashes` as the freshness key | A re-derive needs a cheap first check, so a doc edit changes the hash and the gate skips re-deriving folders whose hashes are unchanged |
| Wire the gate into strict validation and dry-run backfill only | Those are the read surfaces that should surface staleness, and neither should mutate the folder it reads |
| Defer the scoped migration and the hard flip | Cleaning the existing folders and flipping the flag ON is a follow-on decision once the report mode confirms the drift population |

<!-- ANCHOR:limitations -->
## Known Limitations

1. **Migration deferred.** The existing folders that carry drifted or divergently-extracted synopsis text are not cleaned by this phase, the grandfather report mode reports them until a scoped migration runs.
2. **Hard flip deferred.** Flipping `SPECKIT_GENERATED_METADATA_DRIFT_GATE` on (which routes the live generators through the shared extractor and enforces drift) stays a follow-on decision until the report mode confirms the drift population is clean.
3. **Calibration unchanged by design.** The 031 calibration re-fit recommendation is documentation-only: the research proved the curve cannot move the verdict, so the curve was not changed.
4. **Hash scope resolved to in-file.** `source_doc_hashes` persists inside `graph-metadata.json` under `derived`, optional and flag-gated, so a flag-off derive and legacy files omit it cleanly and no non-source change churns it.
<!-- /ANCHOR:limitations -->
