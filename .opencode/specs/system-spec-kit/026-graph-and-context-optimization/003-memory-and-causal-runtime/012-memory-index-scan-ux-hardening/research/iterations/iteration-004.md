# Iteration 4: A4 Embedder Resilience and Degraded-Mode Indexing

## Focus

This iteration defines the degraded-mode contract for the fragile embedder dependency. It builds on the prior A1-A3 decisions: scans should be idempotent async jobs, lexical indexing should commit first, vector embedding should drain separately, and concurrent writers should coordinate through a DB-backed job/claim model. The goal here is a caller-visible success state where text/FTS/BM25 search works immediately while vectors lag truthfully and recover automatically.

## Actions Taken

- Read the deep-research quick reference, current state log, strategy, and iterations 001-003 to preserve the answered A1-A3 decisions.
- Traced the shared embedding circuit breaker, retry-manager provider circuit breaker, retry claim/retention behavior, and failure-to-retry transitions.
- Traced the deferred lexical indexing path through async embedding, record creation, and vector-index mutations.
- Traced embedding batch controls, hf-local cold-load/readiness behavior, crash-loop cooldown, and documented environment knobs.
- Designed the A4 failure taxonomy, degraded job state, circuit composition, bounded backlog policy, and caller-facing UX fields.

## Findings (file:line evidence)

1. Lexical commit can be made independent of the provider, but the current scan design must force the deferred path to make that invariant explicit.

   In async embedding mode, the save pipeline checks the persistent cache and otherwise records `Deferred: async_embedding requested` without calling the provider [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:140] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:151]. When no embedding is present, `createMemoryRecord()` routes to `vectorIndex.indexMemoryDeferred()` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/create-record.ts:281] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/create-record.ts:327], passing `contentText` and metadata into the deferred path [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/create-record.ts:327] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/create-record.ts:341]. The deferred vector mutation writes or updates `memory_index` with `embedding_status = 'pending'`, stores `content_text`, resets retry metadata, updates the active projection, refreshes interference scores, and logs that the row is BM25/FTS5 searchable without an embedding [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:337] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:345] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:351] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:354] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:376] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:378].

   Sync embedding errors are already mostly non-fatal to lexical upsert: if `generateDocumentEmbedding()` returns null, the pipeline records a sanitized failure reason [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:189] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:191], and if the provider throws, the catch stores a sanitized failure reason instead of rethrowing [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:194] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:200]. It then returns `embedding: null`, `status: 'pending'`, and the failure reason [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:204] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:208]. Remaining abort paths are not provider failures; they are storage failures in the DB transaction or cache-write path. Deferred indexing itself is a SQLite transaction [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:337] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:383], so disk-full/readonly DB errors still correctly mean lexical indexing did not commit.

   Recommendation: the future scan lexical phase should always call the save/index path with deferred vectors for scan work. Treat provider errors as vector backlog metadata, not scan failure. Treat DB write failures as the one hard failure class because searchability was not achieved.

3. The current retry path can double-penalize outages unless the two circuit breakers compose behind one provider-state preflight.

   Shared embeddings returns null without a provider call while its circuit is open [SOURCE: .opencode/skills/system-spec-kit/shared/embeddings.ts:676] [SOURCE: .opencode/skills/system-spec-kit/shared/embeddings.ts:679]. Retry-manager separately checks its own provider circuit before calling `generateDocumentEmbedding()` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:707] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:713]. If shared embeddings returns null, retry-manager records provider failure behavior and increments retry count through the null-embedding path [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:735] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:742]. Provider exceptions also increment retry count and can mark rows failed after `MAX_RETRIES = 3` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:800] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:810].

   Recommendation: expose a single read-only embedding provider state used by both inline and background vector drain. `providerState` should include `available`, `loading`, `sharedCircuitOpenUntil`, `retryCircuitOpenUntil`, `crashLoopRetryAfter`, and `lastProbeAt`. The vector drain should compute `nextVectorAttemptAfter = max(openUntil/retryAfter values)` and skip claims while blocked. Recovery should be automatic through the existing half-open behavior: shared embeddings clears after cooldown and allows one probe [SOURCE: .opencode/skills/system-spec-kit/shared/embeddings.ts:72] [SOURCE: .opencode/skills/system-spec-kit/shared/embeddings.ts:79], while retry-manager clears after its cooldown [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:422] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:430]. Do not layer 60s + 120s + crash-loop cooldown serially unless all blockers are truly active; report the maximum active blocker as the caller-visible wait.

4. Bounded vector backlog needs a scan-owned drain, not the default retry loop alone.

   The embedder reindex job already uses a bounded default batch size of 50, with env bounds 1-1000 [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:74] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:76] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:135] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:137]. The execution router's provider batch upper bound is 256 texts and ~768 KiB per request body [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/execution-router.ts:39] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/execution-router.ts:43] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/execution-router.ts:194] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/execution-router.ts:212], and ENV docs explicitly warn that larger reindex batches raise throughput at higher peak memory [SOURCE: .opencode/skills/system-spec-kit/mcp_server/ENV_REFERENCE.md:422] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/ENV_REFERENCE.md:423]. The background retry loop defaults to only 5 items every 5 minutes [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:357] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:364], which is safe for incidental failed rows but too slow as the only completion path for a large forced scan backlog.

   Recommendation: after lexical commit, start a scan-associated vector drain with one active vector batch per DB/provider profile by default. Claim at most 50 rows per tick, then let the execution router sub-chunk to the provider's 256-count and byte caps. Keep `maxInFlightVectorRows`, `maxInFlightBytes`, and `maxConcurrentProviderPosts` explicit in job telemetry; default `maxConcurrentProviderPosts` to 1 for local providers to avoid spawn storms and local RSS spikes. The retry loop remains the safety net for old incidental rows, while scan-created backlog drains through the faster bounded job path.

5. Caller-facing degraded signal should be a first-class success state, not an error envelope.

   Recommended scan/status fields:

   | Field | Meaning |
   |---|---|
   | `status` | `queued`, `running`, `lexical_complete`, `complete_with_pending_vectors`, `complete`, `failed_storage`, `cancelled`. `complete_with_pending_vectors` is success, not error. |
   | `phase` | `walk`, `commit_lexical`, `vector_deferred`, `vector_draining`, `complete`. |
   | `searchableNow` | True only after lexical rows/projections/FTS content committed. False for storage failures. |
   | `degraded` | True when vectors lag due to provider state, circuit, load, crash-loop, or slow drain. |
   | `degradedReasons` | Stable enums such as `provider_loading`, `shared_circuit_open`, `retry_circuit_open`, `crash_loop_cooldown`, `loading_wedged`, `slow_provider`, `storage_blocked`. |
   | `vectorBacklog` | `{ pending, retry, failed, claimed, succeeded, total }`, with clean pending counted separately from attempted retry rows. |
   | `providerState` | `{ available, loading, circuitOpen, crashLoopCooldown, model, profileKey, lastProbeAt }`. |
   | `nextVectorAttemptAfter` | ISO timestamp derived from active circuit/cooldown/probe backoff. Null when drain is active or complete. |
   | `nextPollAfterMs` | Caller poll hint; short while running, aligned to `nextVectorAttemptAfter` while blocked. |
   | `warnings` | Human-readable but non-fatal explanations: `embeddings lagging but lexical search is available`. |

   This contract makes the desired UX explicit: `indexed + searchable now, N vectors pending, provider degraded, will retry at T`. The only hard error class should be validation or storage failure where lexical rows did not commit.

6. Ruled out: treating provider outage as scan failure, using only retry-manager defaults, or converting all pending rows to retry during outages.

   Provider outage as scan failure contradicts the deferred lexical path that already stores searchable pending rows [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:363] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:378]. Relying only on the default retry loop would make a 1,000-row backlog take roughly many hours at 5 rows per 5 minutes [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:357] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:364]. Converting clean pending rows to retry during a circuit-open skip is unsafe because retention intentionally excludes never-attempted clean pending rows but prunes rows with `embedding_status = 'retry'` or non-zero retry count by age/cap [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:493] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:498] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:499] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:527].

## Questions Answered

- A4 is answered. The scan should always commit lexical rows first when storage is healthy, return `complete_with_pending_vectors` as a success state when vectors lag, and surface `degraded` plus `nextVectorAttemptAfter` for provider trouble.
- Cold load, shared circuit-open, retry-manager circuit-open, hf-local crash-loop cooldown, model-load wedge, and slow POSTs should not fail the scan wholesale. They should leave clean rows `pending`, avoid retry-count consumption before real provider attempts, and auto-resume through half-open/readiness probes.
- ENOSPC/storage failures split into two classes: non-critical provider/cache/supervisor metadata write trouble can degrade vectors, but DB transaction failure blocks lexical search and must be a hard per-file/job storage failure.
- The vector backlog should drain through a scan-owned bounded worker using a 50-row default tick and provider-side count/byte chunking; the 5-items/5-min retry loop should remain only a safety net.
- The two circuit breakers should compose through one provider-state surface so callers see one backoff timestamp and rows are not double-penalized.
