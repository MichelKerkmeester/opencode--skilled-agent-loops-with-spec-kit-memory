## Review Analysis — seat `opus-launcher-lifecycle`

**Angle:** Does `mk-code-index-launcher.cjs` wrongly *exit* on child SIGTERM instead of *respawning* (reconnecting proxy), and how does that compare to spec-memory's transparent recycle?

### What the code actually does (traced, not guessed)

**Owner model (the assigned hypothesis):** The owner launcher spawns the daemon with `stdio: 'inherit'` and is **not** detached (`mk-code-index-launcher.cjs:875-879`). Its child-exit handler clears leases and dies with the child — mirror-signal or `process.exit(code)` (`:893-902`). There is no in-place relaunch, no RSS recycle, no `superviseChildExit`. This is **intentional and documented**, not an oversight:
- `:374-377` — "this launcher exits with its child rather than supervising relaunches"
- `:152-156`, `:176-178` — the reconnecting proxy is deliberately scoped to *secondary* clients only; the classifier tool-set is "the only diff" vs spec-memory.

**spec-memory contrast (verified):** spec-memory spawns the daemon **detached, stdio all-ignored, `unref()`** under re-election (`mk-spec-memory-launcher.cjs:208-212,1317-1327`), **supervises and relaunches** the child in place with a crash-loop guard (`:1337-1376`), and **releases** the daemon on owner shutdown (`:196-214`). That is what makes spec-memory's recycle transparent even for the owner.

**The reconnecting proxy itself** (`lib/launcher-session-proxy.cjs`) only **reconnects to an existing socket** — it never respawns a backend. On daemon death it runs a bounded reattach loop (40 attempts, `:725-753`) then fails closed with retryable `-32001`. Daemon *respawn* happens only on a **fresh** launcher invocation via the dead-socket path (`launcher-ipc-bridge.cjs:421-423` → `respawnAfterDeadSocket` `mk-code-index-launcher.cjs:639-688`).

### Verdict on the hypothesis

**Refuted.** "Exits instead of respawning" is the documented design for a structural index: queries are replayable, scans/applies are client-redriven (`:157-170`), and a dead daemon is respawned on the next invocation. The proxy reconnect is correct for the common owner-churn case (daemon orphaned-but-alive → secondary reattaches to the surviving socket). The difference from spec-memory is real but **intentional**, so it is not a P0/P1.

### One concrete defect found (correctness/traceability, P2)

The child-exit signal-mirror is **defeated by the launcher's own signal handlers** for catchable signals. When the daemon child is killed by an external **SIGTERM/SIGINT/SIGHUP/SIGQUIT**, `:897` runs `process.kill(process.pid, signal)`. Because the launcher registered handlers for exactly those four signals (`:911-929`), the self-signal is intercepted rather than terminating. In that handler `childProcess.killed` is `false` (the child died externally, not via our `.kill()`), so it takes the if-branch: a `once('exit')` that can never fire again, a no-op `childProcess.kill()` on a dead pid, and an **`unref()`'d** 5s timeout. With the heartbeat timer also `unref()`'d, the event loop drains and the launcher exits **0** instead of mirroring **143/130/129/131**. (SIGKILL is unaffected — uncatchable — so it mirrors correctly.) Impact is low: a runtime that classifies daemon termination by exit code mis-reads a signal-killed daemon as a clean exit; the connection severs and respawns-on-next-call regardless. spec-memory guards its handler with `launcherShutdownInProgress` (`:1338`); code-index has no equivalent flag.

No P0/P1 found. The epic's daemon-lifecycle design holds up under this lens.
