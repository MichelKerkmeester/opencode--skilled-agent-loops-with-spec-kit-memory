## Topology (by-slice / by-model / hybrid â€” when each, lineage count)

- **By-slice = default.** Codebase partitioned into disjoint regions (top-level dirs, modules, or feature boundaries); each lineage owns one slice and produces slice-local reuse-catalog + coverage-gap evidence. Rationale: aligns with the brief's "highest-value section = REUSE catalog" because reuse entries are naturally scoped to a file:symbol region, and a disjoint partition makes dedup trivial (same `file:symbol` cannot appear in two slices unless it is a cross-slice dependency, which the merge treats as a *relation* not a *duplicate*). Default N = `ceil(|target paths| / slice_budget)`; observed target in this repo is `.opencode/skills/*/SKILL.md` + `.opencode/agents/*.md` â€” partition by skill directory gives Nâ‰ˆ15 (one per skill), or by feature surface (config / workflow / runtime / agents) gives N=4â€“5.
- **By-model = secondary axis, on demand.** Two-to-three model lenses (e.g., "structural reader", "reuse-extender", "risk-surfacer") applied to the *same* slice when a single model lens produces low-relevance or low-confidence findings. Triggered only after one by-slice pass with low coverage-coverage-saturation; never first because model-only fan-out wastes the partitioning win.
- **Hybrid = nested fan-out, capped.** N slices Ã— M model lenses with K-in-flight pool ceiling; the existing `fanout-pool.cjs` `concurrency` cap is the only thing that needs to be reused (`deep-loop-runtime/scripts/fanout-run.cjs:401-484`). Default cap: K = 4 for sub-4GB-class small models (SWE-1.6, MiniMax-M3, Kimi-K2.6); see `sk-prompt-small-model/SKILL.md` model profiles for the per-model pressure rule.
- **One-lineage default for trivial targets.** When `|target paths| â‰¤ slice_budget` and gather-subject is narrow, run a single lineage and skip the pool â€” matches the brief's "REUSE catalog is the highest-value report section" (no need to over-engineer fan-out for small surfaces).

## MergeRules (what merge reconciles + dedup)

- **Reuse-catalog dedup key = `file:symbol`** (not `id || title` as in `fanout-merge.cjs:97`); within a by-slice partition, the same `file:symbol` cannot appear twice in one slice, but the *same* file in two slices (cross-slice dependency) is a *cross-reference edge* in the merged report, not a duplicate. Conflict resolution: keep entry with **max relevance score**; union `_lineages[]` attribution across the contributors; if `recommended_extension` differs across lineages, escalate to a `contradictions[]` row, do NOT silently pick one.
- **Coverage-gap reconciliation** = set-union with stable gap-ids (`gap-<slice>-<ordinal>`); gaps already answered by another slice's reuse-catalog entry are dropped, gaps unique to one slice are kept verbatim with `_lineages=[label]`.
- **Cross-lineage contradiction detection** = run pairwise over the unioned reuse-catalog: if two lineages assert different contracts for the same `file:symbol`, emit a `contradiction` node in the coverage graph (analogous to review's `CONTRADICTS` relation `coverage_graph_schema.md:84`) and surface it in the merged report's "Gaps & Unknowns" section (mirroring the @context Context Package contract, `.opencode/agents/context.md:259`).
- **Relevance gate** = per-slice self-reported relevance (0.0â€“1.0) is *not* merged; the host computes a cross-slice relevance rollup (median across lineages that mention the same symbol) and gates the final Context Report: a reuse entry with cross-slice median relevance < threshold (proposed 0.40) is dropped to "uncertain" with a documented reason. The threshold lives in `deep-context-config.json` so the convergence.cjs can read it (analogous to `convergenceThreshold` per loop type â€” see `deep-research/SKILL.md:27-37`).
- **Merged registry name** = `deep-context-slice-registries/*.json` (one per slice) + `deep-context-report.json` (host-merged) + `context-report.md` (rendered); mirrors the `deep-{review,research}-findings-registry.json` convention at `fanout-merge.cjs:318-320`.
- **Attribution file** = `fanout-attribution.md` already produced by `fanout-merge.cjs:261-288`; deep-context reuses the same shape, adding a `relevance_median` column per lineage.

## LineagePromptContract (mandatory fields the worker prompt must carry)

The current `buildLoopPrompt` at `fanout-run.cjs:122-157` only carries `spec_folder`, `config.fanout_lineage_artifact_dir`, `session_id`, `executor`, `loop_type`, and an optional `config.maxIterations`. **It omits the gather-subject, the slice boundary, the known-context, and the output schema â€” the worker is told to read the SKILL.md and run the loop, but never told *what to look for*.** A worker running blind against a read-only codebase will produce generic findings, not a slice-local reuse catalog. The fix:

| Field | Source | Required? | Why it is mandatory |
|---|---|---|---|
| `gather_subject` | operator / orchestrator input | YES | The "query" the loop is trying to answer; without it the worker is blind. Concrete form: "What existing functions/utilities in this slice extend cleanly for `<gather_subject>`?" |
| `slice_boundary` | host partitioner (N disjoint slices) | YES | Hard cap on read paths; the worker prompt must include an explicit list of allowed globs/dirs and an explicit denylist. Without it, slices overlap and dedup becomes ambiguous. |
| `known_context` | `handover.md` + `spec.md` + `resource-map.md` of the target spec folder | YES | Smaller than full repo; gives the worker a "what's already known" anchor so it does not re-discover. Mirrors the @context Layer-1 (`agents/context.md:159-172`). |
| `output_schema` | the Context Report slice JSON shape | YES | The worker must emit a typed slice registry. Proposed shape: `{ slice_id, reuse_entries: [{file, symbol, relevance, recommended_extension, evidence, _lineage}], coverage_gaps: [{id, question, _lineage}], contradictions: [{file, symbol, claim_a, claim_b, _lineages}] }`. Without an explicit schema, small models emit free prose and the merge cannot dedup. |
| `lineage_label` | host partitioner | YES | For `_lineages` attribution downstream. |
| `session_id` | host (`fanout-{label}-{runId}`) | YES | Coverage-graph namespace; reuses the existing `session_id` PK pattern in `coverage_graph_schema.md:113`. |
| `artifact_dir` | host (per-lineage sub-dir) | YES | Write boundary; same as current. |
| `read_only` | boolean `true` | YES | Reinforces the agent-level invariant from `agents/context.md:38-46`; small-model workers are explicitly LEAF read-only analyzers, not implementers. |
| `executor`, `iterations`, `model` | existing | YES | Unchanged. |
| `prompt_pack_template` | path under `.opencode/skills/deep-context/assets/` | YES | Rendered via the existing `renderPromptPack` (`lib/deep-loop/prompt-pack.ts:55-73`); declared variables are validated at render time so a missing `gather_subject` throws `PromptPackError` *before* the subprocess spawns. |

## StateOwnership (host-writes vs executor-writes + rationale)

- **Executor (lineage) writes** â€” bounded to the lineage's `artifact_dir`:
  - `iterations/iteration-NNN.md` (write-once narrative, same as deep-research at `SKILL.md:287`)
  - `<slice-id>-state.jsonl` (append-only, with `loop-lock` and `atomic-state` per `lib/deep-loop/atomic-state.ts:36-53`)
  - `deep-context-slice-registry.json` (one per slice, write-once per iteration via reducer)
  - `graphEvents[]` for the lineage's own coverage-graph `session_id`
  - **No writes outside `artifact_dir`.** Enforced by the prompt's "do not touch any path outside {lineageDir}" clause (`fanout-run.cjs:153`) and by the existing sandbox posture in `fanout-run.cjs:414-426`.
- **Host (workflow/reducer) writes** â€” owns the merged and machine surfaces:
  - `deep-context-report.json` (the merged reuse-catalog, deduped, with cross-slice relevance rollup)
  - `context-report.md` (rendered Context Report â€” the only file `/speckit:plan` and `/speckit:implement` consume)
  - `fanout-attribution.md` (reuses the existing builder at `fanout-merge.cjs:261-288`)
  - The cross-lineage `coverage_nodes` / `coverage_edges` upserts via `scripts/upsert.cjs` (which requires `--loop-type context` â€” see Open Questions)
  - Convergence decisions via `scripts/convergence.cjs` (which requires `--loop-type context`)
- **Rationale (Gate-3 / reliability):**
  1. **Iterate-once / reducer-owned pattern is established** â€” `manual_testing_playbook/07--command-flow-stress-tests/write-boundary-reducer-owned-files.md:24-27` ("only the reducer mutates reducer-owned files (registry, dashboard, strategy) and that iterations stay write-once for their own narrative"); deep-context reuses it verbatim, calling the merged registry / report / attribution the "reducer-owned" surfaces.
  2. **Multi-agent Gate-3 safety** â€” sub-agents MUST NOT write the merged Context Report because that is the artifact the orchestrator later passes to `/speckit:plan`; a polluted merged file breaks the next phase. The host is the only writer with the full view of N lineages.
  3. **Cross-lineage atomicity** â€” the dedup-by-`file:symbol` + cross-slice relevance rollup requires reading all slice registries; only the host can sequence that after all lineages settle, and the existing `loop-lock` (`lib/deep-loop/loop-lock.ts`) plus `writeStateAtomic` (`lib/deep-loop/atomic-state.ts:36`) are the durability primitives.
  4. **Convergence-gating requires deterministic session_id** â€” the convergence script reads `session_id` to scope graph queries (`references/script_interface_contract.md:64-68`); the host mints one per slice *and* one for the merged pass, ensuring the coverage graph can distinguish per-slice from cross-slice signals without colliding.
  5. **Failure containment** â€” if a single lineage subprocess exits non-zero, the host still emits a partial Context Report from the surviving slices (matches `fanout-run.cjs:495-497` exit-code matrix: 0/2/3); if a *lineage* wrote the merged file, a partial failure could corrupt the merged surface before the host knows to skip it.

## OpenQuestions

1. **Coverage-graph schema migration cost.** `loop_type='context'` requires (a) extending the `loop_type` enum validated in `scripts/upsert.cjs` and `scripts/convergence.cjs` (currently hard-rejects anything outside `research`/`review` per `fanout-merge.cjs:297`), (b) new node kinds (proposed: `SLICE`, `SYMBOL`, `REUSE_ENTRY`, `GAP`, `CONTRADICTION`) added to `coverage-graph-db.ts` allow-list, (c) new relation kinds (proposed: `EXTENDS`, `CROSS_SLICE`, `CONTRADICTS`, `SATISFIES`). Per `SKILL.md:211` this is a backward-incompatible change and requires a new ownership ADR before any code is written â€” escalate to user.
2. **Convergence metric for context** â€” `newInfoRatio` (research, 0.05) and severity ratio (review, 0.10) do not fit. Options: (a) `coverage_gap_delta` per iteration (drops below threshold = stop), (b) `reuse_catalog_entropy` (Shannon entropy of the `recommended_extension` distribution saturates when no new patterns appear), (c) hybrid. Need iteration-4 follow-up to choose; current proposal is (a) for parity with existing Bayesian scorer in `lib/deep-loop/bayesian-scorer.ts`.
3. **`known_context` shape** â€” should it be (a) the active spec folder's `handover.md` + `spec.md` + `resource-map.md` only, (b) a wider window across all spec folders in `.opencode/specs/`, or (c) include `memory_match_triggers` output? (a) is the smallest, most-cache-friendly choice and matches the @context Layer-1 pattern; (b) is more thorough but expensive for a small-model worker.
4. **Per-slice self-reported relevance vs host-computed relevance** â€” does the worker emit a `relevance` per reuse entry and the host *only* rolls up, or does the host compute relevance from evidence density? The former trusts small-model judgment (cheap, but noisy); the latter requires the host to re-read evidence (expensive, more accurate). Lean toward former + a host-side sanity check (relevance < 0.2 entries dropped to a "low-confidence" bucket in the report).
5. **Hybrid concurrency cap (K)** â€” `fanout-pool.cjs` takes a `concurrency` arg (`fanout-run.cjs:402`), but there is no documented default for the small-model mix. Suggested K=4 with the per-kind `SPECKIT_<KIND>_STATE_DIR` isolation pattern already at `fanout-run.cjs:107-115`; needs validation against the `sk-prompt-small-model` per-model budget tables.
6. **Slice-boundary collision detection** â€” when a `file:symbol` is reachable from two slices (e.g., a shared utility), how does the host decide which slice "owns" it for the dedup key? Proposed rule: the slice whose boundary contains the file's directory wins; the other slice emits a `cross_slice` edge. Need to confirm with deep-review precedent (the `IN_FILE` relation at `coverage_graph_schema.md:89` is per-file, not per-symbol).
7. **No new MCP tools** â€” per `SKILL.md:201` and the isolation ADR, deep-context MUST NOT add MCP tools; all surface is via the existing `.cjs` script contract. Iteration-5 deliverable is a new ownership ADR + a `lib/deep-loop/` extension to the existing `upsert.cjs` / `convergence.cjs` to accept `--loop-type context`, not a new tool family.
