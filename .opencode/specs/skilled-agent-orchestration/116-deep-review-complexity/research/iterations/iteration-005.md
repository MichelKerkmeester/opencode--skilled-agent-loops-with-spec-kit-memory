# Iteration 005: Research Mechanics To Transfer Into Review

## Focus

Identify which deep-research mechanics should transfer into deep-review. This pass compared state strategy, prompt pack, convergence, reducer, and agent behavior for the mechanics that drive novelty, narrow focus, ruled-out directions, evidence enumeration, and bug-search depth.

## Findings

1. Transfer the research loop's question-led state strategy into review as candidate-led state, not as a replacement for severity review. Deep-research state keeps key questions as the reducer-owned focus substrate: unchecked questions drive next focus, newly answered questions are moved to answered state, and next focus is based on remaining questions and successful approaches [SOURCE: .opencode/skills/deep-research/references/state_format.md:562] [SOURCE: .opencode/skills/deep-research/references/state_format.md:575] [SOURCE: .opencode/skills/deep-research/references/state_format.md:578]. Deep-review, by contrast, injects dimension, review target, scope files, and prior severity counts into the prompt pack [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:11]. The transfer should be a `candidateQuestions` or `candidateMatrix` lane under each dimension: bug class, invariant, producer, consumer, negative test searched, evidence, result, and ruled-out reason.

2. Transfer the research prompt pack's novelty-producing delta vocabulary. Research requires a markdown narrative, an appended canonical iteration record, and a per-iteration delta file whose examples include `finding`, `invariant`, `observation`, `edge`, and `ruled_out` records [SOURCE: .opencode/skills/deep-research/assets/prompt_pack_iteration.md.tmpl:41] [SOURCE: .opencode/skills/deep-research/assets/prompt_pack_iteration.md.tmpl:53]. Review's delta example allows finding, classification, traceability-check, and ruled_out records, but its narrative is organized around dimension, files, severity, traceability, verdict, and next dimension [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:76] [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:86]. Deep-review should add required pre-finding delta records: `hypothesis`, `invariant`, `producer`, `consumer`, `negative_test`, and `candidate_edge`, then let severity findings reference candidates selected from that ledger.

3. Transfer research's narrow-focus behavior before adding more review breadth. The deep-research agent must choose the narrower evidence-backed interpretation when multiple focus candidates tie [SOURCE: .opencode/agents/deep-research.md:124], and it must read exhausted approaches before choosing focus so `BLOCKED` categories are not retried [SOURCE: .opencode/agents/deep-research.md:117]. Deep-review recovery pivots to the least-covered dimension when stuck [SOURCE: .opencode/skills/deep-review/references/convergence.md:514], which helps breadth but not necessarily depth inside the same bug class. The review improvement is a two-level focus rule: first pick the dimension, then pick one narrow candidate cluster and exhaust or promote it before moving on.

4. Transfer research's negative-knowledge reducer path. Deep-research reducer promotes iteration `Ruled Out` and `Dead Ends` entries into `ruledOutDirections`, then writes them back to strategy "RULED OUT DIRECTIONS" and "EXHAUSTED APPROACHES" [SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:554] [SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:630] [SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:701]. This is exactly the missing "absence of finding is evidence" mechanic for review. Deep-review should not only list confirmed-clean surfaces; it should persist ruled-out bug hypotheses with evidence so later iterations do not repeat shallow scans or falsely count a clean dimension without search proof.

5. Transfer research's evidence enumeration into review's clean-path proof. Deep-research computes `coverageBySources` from iteration-file sources and JSONL `sourcesQueried` [SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:217] [SOURCE: .opencode/skills/deep-research/scripts/reduce-state.cjs:566], and its state format distinguishes primary, secondary, and tentative evidence for coverage purposes [SOURCE: .opencode/skills/deep-research/references/state_format.md:205]. Deep-review's evidence density gate covers active findings at stop time [SOURCE: .opencode/skills/deep-review/references/convergence.md:409], but there is no equivalent source-density gate for clean candidate rows. Review needs enumeration counts for searched files, producers, consumers, tests, and docs even when no P0/P1 is found.

6. Transfer research's convergence pressure from "coverage of named questions" to "coverage of named candidate classes." Deep-research convergence uses question entropy as a 0.35-weighted signal and treats answered/total question coverage as a stop vote [SOURCE: .opencode/skills/deep-research/references/convergence.md:187]. The legal-stop bundle then requires every key question to have an evidence-backed answer and quality sub-gates to pass [SOURCE: .opencode/skills/deep-research/references/convergence.md:215]. Deep-review's highest-weight stop signal is dimension coverage at 0.45, meaning configured dimensions and required traceability protocols are covered and aged [SOURCE: .opencode/skills/deep-review/references/convergence.md:211]. Add a review stop gate that mirrors question coverage: every required candidate class for the target risk profile has either an active finding, a cited ruled-out record, or an explicit deferred/blocker record.

7. Transfer reducer-owned state discipline from research into review. The deep-research agent is explicitly barred from writing reducer-owned strategy, registry, and dashboard files [SOURCE: .opencode/agents/deep-research.md:53] [SOURCE: .opencode/agents/deep-research.md:226]. Deep-review currently makes the agent update strategy after writing findings [SOURCE: .opencode/agents/deep-review.md:198], and its prompt pack even lists strategy and findings registry as allowed write paths [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:61]. That gives the review agent more bookkeeping surface during a bug-finding pass. A less surface-level review loop should make the agent write only narrative, state append, and delta records; the reducer should own strategy, registry, dashboard, and candidate-coverage rollups.

8. Transfer research's novelty accounting, but adapt it to concepts rather than severity. Research's `noveltyJustification` explains the newInfoRatio for post-hoc calibration [SOURCE: .opencode/skills/deep-research/references/state_format.md:185]. Deep-review's `newFindingsRatio` is severity-weighted, and a new P0 forces a floor of 0.50 [SOURCE: .opencode/skills/deep-review/references/convergence.md:363]. That is useful for release gating, but it does not measure whether a clean iteration searched a new class of bug. Make `semanticNovelty` or `candidateNovelty` required in review state for non-trivial code/spec targets, using new candidate classes, new invariants, new producer/consumer paths, and new ruled-out directions as the numerator.

9. The research mechanism that should not transfer wholesale is web/source freedom. Deep-research allows WebFetch and external source gathering [SOURCE: .opencode/agents/deep-research.md:13], while deep-review explicitly denies WebFetch and remains code-only [SOURCE: .opencode/agents/deep-review.md:13]. Keep that boundary. The transferable part is not web research; it is the state shape that forces narrowed hypotheses, evidence rows, and ruled-out directions.

10. Prior packet evidence supports making candidate-generation required before severity adjudication. Iteration 003 found rich review fields are uneven, with non-empty `graphEvents` in only 40 of 885 surveyed completed/legacy iteration records and zero structured `bug-class` detail mentions [SOURCE: .opencode/specs/skilled-agent-orchestration/116-deep-review-complexity/research/iterations/iteration-003.md:16] [SOURCE: .opencode/specs/skilled-agent-orchestration/116-deep-review-complexity/research/iterations/iteration-003.md:28]. Iteration 004 then found the review prompt and agent validate candidates after discovery but do not force bug-class hypotheses, invariant sweeps, producer/consumer tracing, negative-test searches, or adversarial matrix construction first [SOURCE: .opencode/specs/skilled-agent-orchestration/116-deep-review-complexity/research/iterations/iteration-004.md:17] [SOURCE: .opencode/specs/skilled-agent-orchestration/116-deep-review-complexity/research/iterations/iteration-004.md:31]. The next implementation should therefore add candidate generation as a first-class review artifact, not merely tighten P0/P1 adjudication.

## Questions Answered

- Which deep-research mechanics should transfer? Question-led focus should become candidate-led review focus; structured `invariant`/`observation`/`edge`/`ruled_out` deltas should become structured review candidate records; negative knowledge should become reducer-owned; source/evidence enumeration should apply to clean paths; convergence should require candidate-class saturation before STOP.
- Why can focused deep-research surface more bugs? It externalizes the search process: focus, remaining questions, evidence sources, ruled-out directions, and novelty all survive across fresh-context iterations. Deep-review often externalizes the report: dimension, file, severity, traceability, verdict, and active findings.
- How should deep-review become less surface-level? Add a required candidate-generation matrix before severity adjudication, reduce agent-owned bookkeeping, make reducer-owned candidate coverage drive strategy, and block stop when required candidate classes lack findings, ruled-out evidence, or explicit blockers.
- What should not transfer? Deep-review should not become web-enabled research by default and should not discard severity, release-readiness, or P0/P1 adjudication discipline.

## Questions Remaining

- What exact minimal schema should represent candidate rows: JSONL-only delta records, a `candidateMatrix` array on the iteration record, or both?
- Which target classes should require full candidate matrices: all code/spec reviews, only complexity-scored targets, only security/correctness dimensions, or only packets with implementation diffs?
- Should graph convergence gain explicit node kinds for bug class, invariant, producer, consumer, and negative test, or should post-dispatch validation enforce candidate rows first and graph support follow later?
- How should legacy review packets remain readable while new packets fail closed on missing candidate-generation evidence?
