# Iteration 033 — Track D (deepseek)

## Focus
Composite/semantic tokens reduce re-assembly fabrication (typography/shadow/border/transition tokens + aliasing)

## Findings
1. **[P0] ComponentVariant.style as flat Record<string,string>** — §4 Component Stylings (Buttons, Cards, Badges, Inputs, Navigation, etc.) requires the writer to parse box-shadow strings into color/offset/blur/spread, border into color+width+style from 9 scattered ElementStyle fields, typography from fontFamily+fontSize+fontWeight flat keys, and transition from a single string. ComponentVariant.style: Record<string,string> at types.ts:467 forces re-assembly of shadow+border+typography+transition primitives into the structured variant descriptions demanded by design_md_format.md §7 (Background, Text, Padding, Radius, Font weight+size, Shadow, Hover, Focus, Transition, Use). This is the single largest re-assembly fabrication point — every interactive component variant passes through it.
   - Recommendation: Add decomposed composite interfaces to types.ts: ComponentVariantStyle: { backgroundColor, textColor, typography (ref to TypographyLevel), padding, borderRadius, border: {color, width, style} | null, shadow: {color, offsetX, offsetY, blur, spread} | null, transition: {duration, delay, timingFunction, property} | null, additional?: Record<string,string> }. Populate during extraction from ElementStyle and InteractionCapture. The writer then transcribes each sub-composite deterministically instead of parsing flat strings.
   - Evidence: types.ts:464-475 (ComponentVariant.style Record<string,string>), types.ts:443-449 (ShadowToken.value flat string), types.ts:160-166 (TransitionInfo already composite but not referenced), design_md_format.md §7 lines 445-459 (variant format requires Background/Text/Font/Shadow/Transition properties assembled from flat record)
2. **[P0] Typography role naming without provenance** — §3 Typography Rules 'Hierarchy' table (design_md_format.md §6, line 396) demands a 'Role' column with invented names ('Display Hero', 'Body Large'). TypographyLevel (types.ts:428-441) already composites family+size+weight+lineHeight+letterSpacing — matching W3C DTCG typography composite — but has no role/alias field. The writer invents role names from typicalTags[] and sampleTexts[]. No CSS-variable provenance exists to map a TypographyLevel to a semantic role like 'heading.xxl'.
   - Recommendation: Add TypographyRole alias type: { role: string, token: TypographyLevel, sourceVariables: string[] }. The role name derives from CSS custom property names when available (e.g., --heading-xl → 'Heading XL') or from tag-frequency heuristics. This turns role from invention into typed reference with provenance.
   - Evidence: types.ts:428-441 (TypographyLevel composite, no role field), design_md_format.md §6 lines 396-411 (Hierarchy table requires invented Role name, 12-20 rows)
3. **[P1] ShadowToken.value as flat box-shadow string** — ShadowToken (types.ts:443-449) stores shadow as a single string value (e.g., 'rgba(50,50,93,0.12) 0px 16px 32px 0px'). §6 Depth & Elevation (design_md_format.md §9) requires the writer to classify shadows by type (Ring, Elevation, Inset) and describe them in a shadow scale table with Level/Treatment/Frequency/Use columns. The writer must parse offsetX, offsetY, blur, spread, and color from the string to generate the classification and the Philosophy narrative. W3C DTCG shadow composite is {color, offsetX, offsetY, blur, spread}.
   - Recommendation: Decompose ShadowToken in types.ts to { color: string, offsetX: number, offsetY: number, blur: number, spread: number, frequency, type, typicalElements, stability? }. The writer can deterministically render box-shadow strings and classify elevation levels from structured fields. Keep a computed 'value' getter for backward compat.
   - Evidence: types.ts:443-449 (ShadowToken.value flat string), design_md_format.md §9 lines 551-569 (shadow scale table requires per-component shadow analysis)
4. **[P1] Color naming without typed aliasing** — §2 Color Palette & Roles requires 'Descriptive brand-specific name, never Blue 1' and a 'Personality sentence' per color (design_md_format.md §4, line 181). ColorToken (types.ts:409-426) has hex, rgba, frequency, usedAs breakdown, cssVariableNames[], and sourcePages — but no name or role field. The writer invents 'Stripe Purple' and 'A saturated blue-violet that reads as financially authoritative' from raw hex #533afd. cssVariableNames[] provides provenance (e.g., --color-brand) but the format spec ignores it for naming. W3C DTCG aliasing: a semantic token references a primitive with provenance.
   - Recommendation: Add ColorRole alias type: { role: string, token: ColorToken, sourceVariable: string | null }. When a CSS variable like --color-brand maps to a hex, the role becomes 'Brand Primary' (derived from variable name), not an invented 'Stripe Purple'. The personality sentence remains writer-fabricated (requires design judgment) but the name becomes provenance-traced.
   - Evidence: types.ts:409-426 (ColorToken has cssVariableNames[] but no role/name), design_md_format.md §4 lines 179-200 (format demands invented name + personality sentence per color)
5. **[P2] ComponentVariant.transition as flat string while TransitionInfo is composite** — TransitionInfo (types.ts:160-166) already decomposes into { property, duration, timingFunction, delay } — matching W3C DTCG transition composite. But ComponentVariant.transition: string|null (types.ts:473) stores transition as an opaque string. The §6.5 Motion System writer must parse this string to populate the Duration Scale table and Easing Functions list (design_md_format.md §10). The composite already exists in the pipeline; it's just not threaded through.
   - Recommendation: Replace ComponentVariant.transition: string|null with transition: TransitionInfo | null. During extraction, populate TransitionInfo from the same CSS parsing that already produces it in CSSAnalysis.transitions[]. The writer transcribes the decomposed fields directly.
   - Evidence: types.ts:160-166 (TransitionInfo composite), types.ts:473 (ComponentVariant.transition flat string), design_md_format.md §10 lines 604-620 (Duration Scale + Easing tables)
6. **[P2] No BorderToken composite — border properties scattered** — Borders are distributed across ElementStyle as 9 separate fields (borderTopColor through borderLeftColor at types.ts:62-65, borderTopWidth through borderLeftWidth at types.ts:85-88, borderStyle at types.ts:89). No BorderToken composite exists matching W3C DTCG border composite {color, width, style}. The §4 writer assembles border descriptions from fragments. The §6 writer must distinguish border-shadows (zero-blur box-shadow used as border) from true borders.
   - Recommendation: Add BorderToken { color: string, width: string, style: string } to types.ts. Populate from ElementStyle during extraction, normalizing the 9 fields into a unified border when all four sides match. Write border descriptions deterministically.
   - Evidence: types.ts:62-65,85-89 (border fields scattered), design_md_format.md §9 lines 568-569 (border-shadow classification rule requires distinguishing border from shadow), W3C DTCG border composite definition

## Summary
Six re-assembly fabrication points identified across five DESIGN.md sections. §4 Component Stylings is the highest-value target: ComponentVariant.style as flat Record<string,string> (types.ts:467) forces the writer to parse shadow, border, typography, and transition from opaque strings for every interactive component variant. Adopting a decomposed ComponentVariantStyle composite (with sub-composites for ShadowToken, BorderToken, TransitionInfo, and a TypographyLevel reference) would let the writer transcribe pre-assembled entities instead of re-assembling primitives — eliminating the single largest source of fabrication in the write pipeline.
