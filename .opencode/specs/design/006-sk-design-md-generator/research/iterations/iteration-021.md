# Iteration 021 — Track D (deepseek)

## Focus
Fabrication-safe reverse-engineering methodology (deterministic inference, functional naming, value-tables-over-prose)

## Findings
1. **[P0] Step 5 — Role Naming (LLM)** — The 6-step methodology maps as: (1) Playwright crawl [DETERMINISTIC], (2) computed CSS extraction [DETERMINISTIC], (3) OKLCH clustering deltaE<3 + frequency dedup [DETERMINISTIC], (4) L1-L4 stability scoring [DETERMINISTIC], (5) role assignment via LLM per color_role_taxonomy.md heuristics [FABRICATION RISK], (6) DESIGN.md composition with personality sentences + named principles + comparative framing [FABRICATION RISK]. Step 5 is where the LLM interprets usage frequency, context, and CSS variable names to assign semantic roles (primary/secondary/accent/border etc.). Even with the 5-tier priority heuristic (CSS var name > framework convention > usage frequency+context > visual position > hue), the LLM still exercises judgment on "usage frequency + context combination" and the anti-pattern AP-03 (semantic misattribution) proves this failure mode is real — calling blue 'primary' when it appears on 3% of elements while gray dominates 62% of text.
   - Recommendation: Adopt a DETERMINISTIC role-assignment rule in cluster.ts (not the LLM phase): rank colors by usage-as dimension (textColor, bgColor, borderColor, shadowColor, gradientColor, iconColor) and assign roles algorithmically. Most-frequent bgColor+interactiveContext → primary. Most-frequent text color → text-primary. Most-frequent border color → border. CSS variable name match overrides the algorithmic assignment. Reserve the LLM only for narrating WHY the role assignment is notable (deviation from expected frequency), not for DECIDING the role. File: tool/scripts/cluster.ts (add a classifyColorRole() function).
   - Evidence: color_role_taxonomy.md §2 Role Assignment Priority (5-tier heuristic, LLM-interpreted); anti_patterns.md AP-03 Semantic Misattribution (hue-based not frequency-based); cluster.ts only outputs hex/rgba/frequency/usedAs/confidence, leaving role assignment entirely to the LLM in the WRITE phase
2. **[P0] Step 6 — Narrative Prose (LLM)** — The WRITE phase injects LLM-generated narrative across ALL 14+ sections: personality sentences on every color entry (writing_style_guide.md §6: 'One sentence about why this specific color was chosen and what it feels like'), named design principles ('chromatic depth', 'whispered authority' — all LLM-invented), comparative framing ('where most systems use X, this system uses Y'), metaphors ('content emerges from darkness like starlight'), and atmosphere paragraphs. These are ALL fabrication — the LLM generates claims about design intent from raw hex values. The consensus in design-system methodology is that a design-system doc should be value-tables + minimal factual annotation, not prose essays. The gold-standard examples (stripe, vercel, linear, supabase) ARE prose-heavy but were generated by the same LLM methodology and are thus subject to the same fabrication risk.
   - Recommendation: Restrict narrative to exactly 2 sections: §0 Brand Context and §1 Visual Theme & Atmosphere. All other sections (§2-§17) should be value-tables with factual annotations only: hex, frequency, usage breakdown, CSS variable, and a machine-generated functional-role label. Personality sentences become an optional 'Interpretive Notes' subsection clearly demarcated with '⚠️ Interpretive — not extracted from source.' File: tool/resources/design_md_format.md (amend v2 format spec), tool/resources/writing_style_guide.md (add fabrication-safe mode).
   - Evidence: writing_style_guide.md §3 Opening Paragraph Rules (metaphor requirement), §6 Color Descriptions (personality sentence requirement), §16 Named Principle Pattern (LLM-invents principle names); design_md_format.md §3-§17 mandate narrative prose across all sections; anti_patterns.md AP-02 Generic Description (bans adjectives but LLM substitutes OTHER adjectives)
3. **[P1] Step 4 — System Inference (L1-L4)** — The standard methodology's Step 4 (token vs incidental) is implemented as L1-L4 stability classification in cluster.ts:classifyColorStability(). The score uses 5 signals — page_coverage (0.8/0.5/0.2), usage dimensions, CSS variable presence, absolute frequency (500/50/5), and chromaticity. Key gaps vs best methodologies: (a) absolute frequency thresholds don't scale with site size — a percentile-based threshold would be more defensible; (b) cross-page consistency (Step 9 in cluster.ts) is computed but NOT fed into stability scoring — best methodologies use it as a primary signal; (c) weighting is hardcoded without calibration against known design systems.
   - Recommendation: Replace absolute frequency thresholds with within-site percentile ranks (top 10% by frequency → +20, top 25% → +10, bottom 10% → -20). Feed cross-page consistency into stability scoring: consistent CSS var → +25, inconsistent → -25. Add a calibration dataset of known design systems (Material Design, Polaris) with known L1/L2/L3/L4 assignments to validate thresholds. File: tool/scripts/cluster.ts:classifyColorStability() (lines 400-435).
   - Evidence: cluster.ts:400-435 (stability scoring with absolute thresholds 500/50/5); cluster.ts:1407-1445 (cross-page consistency computed but siloed); manual_testing_playbook/09--cluster/oklch-clustering.md (notes degenerate clustering failure mode)
4. **[P1] Step 5 — Functional vs Evocative Naming** — Methodologies that avoid fabrication use functional/positional names: 'color used as 88% of borders' instead of 'Nobel Navy'. Our color_role_taxonomy.md already provides a functional taxonomy (primary, secondary, accent, background, surface, text-primary, border, focus, destructive, success, warning, info, highlight, overlay — 19 functional roles). This IS the anti-fabrication discipline. However, the writing_style_guide.md §6 mandates a 'personality sentence' on every color entry (e.g. 'A saturated blue-violet that reads as financially authoritative') — re-introducing fabrication. The gold-standard examples MIX both: 'Stripe Purple' (evocative name) + functional role + personality sentence.
   - Recommendation: Formalize color_role_taxonomy.md §1's identification signals into a deterministic role classifier in cluster.ts. Use the functional role as the canonical name (e.g. 'Primary CTA' not 'Stripe Purple'). Evocative brand-name labels become an optional 'displayName' field in tokens.json. Personality sentences are gated behind an 'interpretiveNotes' field with a fabrication-warning prefix. File: tool/scripts/cluster.ts (add classifyColorRole()), tool/resources/color_role_taxonomy.md (bless functional names as canonical).
   - Evidence: color_role_taxonomy.md §1 (19 functional roles with deterministic identification signals); writing_style_guide.md §6 (mandates personality sentence); anti_patterns.md AP-12 Vague Color Names (encourages functional names like 'Action Default')
5. **[P2] Step 3 — OKLCH Clustering Threshold** — The OKLCH clustering uses deltaE < 3 with SCALED Euclidean distance (dl×100, dc×100, dh with √(a·b) product). The effective perceptual threshold varies across OKLCH space: near-achromatic colors at high L would merge at visually distinct values; saturated mid-L colors would separate at visually similar values. The methodology flaw: fixed threshold but non-uniform perceptual meaning.
   - Recommendation: Use a chroma-scaled threshold: threshold = max(1.5, 3.0 × (0.3 / max(c_avg, 0.3))) so low-chroma colors get a tighter merge threshold. Alternatively, switch to CIEDE2000 with threshold 2.3. File: tool/scripts/cluster.ts:deltaE() and clustering loop at line 708.
   - Evidence: cluster.ts:348-354 (deltaE with hardcoded ×100 scaling); cluster.ts:698-724 (greedy clustering with fixed deltaE < 3); manual_testing_playbook/09--cluster/oklch-clustering.md (notes single-cluster failure)
6. **[P2] Step 6 — Narrative Scope Consensus** — The design_md_format.md mandates narrative prose across 14+ sections — personality sentences, named principles, comparative framing, metaphors, vibe paragraphs. The consensus in published methodology (context.dev + similar) is that a design-system doc should be value-tables + minimal factual annotation at the token level, reserving interpretive narrative for the system-level introduction. Our format spec is the OPPOSITE. The gold-standard examples follow the narrative-heavy pattern but were authored by the same LLM methodology — self-reinforcing, not independently validated. The fabrication-safe consensus: §0 Brand Context + §1 Visual Theme → narrative appropriate; §2-§17 → value-tables with factual annotations only.
   - Recommendation: Amend design_md_format.md to create a two-tier format: (1) MEASURED sections (§0, §2-§17) contain only values copied verbatim from tokens.json with machine-generated functional labels and frequency annotations — zero LLM prose; (2) INTERPRETIVE section (§1 Visual Theme & Atmosphere) is the ONLY section with LLM-generated narrative, carrying a header disclaimer: '⚠️ This section is interpretive.' File: tool/resources/design_md_format.md (v3 format spec).
   - Evidence: design_md_format.md §3-§17 (mandate narrative prose in all sections); writing_style_guide.md (all 20 sections mandate LLM-generated narrative); anti_patterns.md AP-02 (bans adjectives but doesn't ban narrative — LLM just substitutes other adjectives)

## Questions Answered
- Step 4 (system inference — L1-L4) is DETERMINISTIC in our pipeline, unlike LLM-based methodologies. The defensible inference rule is: CSS variable naming (strongest) > cross-page consistency > within-site percentile frequency > usage-dimension diversity > chromaticity. Our pipeline uses all except cross-page consistency (computed but not scored) and percentile ranking (uses absolute thresholds).
- Step 5 (role naming) — the anti-fabrication discipline from published methodologies IS functional/positional naming, and our color_role_taxonomy.md already defines 19 functional roles with deterministic identification signals. The gap: cluster.ts doesn't execute these signals algorithmically — it defers to the LLM. Closing this gap is the single highest-value anti-fabrication move.
- Narrative is appropriate ONLY in §0 Brand Context and §1 Visual Theme & Atmosphere. The consensus is value-tables + factual annotations for the token catalog. Our current format spec mandates narrative in ALL sections, which is the OPPOSITE of the fabrication-safe consensus.
