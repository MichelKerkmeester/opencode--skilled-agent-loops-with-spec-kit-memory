## Findings

### F-speckit-runtime-agent-path-undefined (P0, commands)

All 6 `/speckit` workflow YAMLs that use the `[runtime_agent_path]` token (`speckit_complete_auto.yaml:357,363,387`; `speckit_complete_confirm.yaml:330,336,360`; `speckit_implement_auto.yaml:252,267`; `speckit_implement_confirm.yaml:216,231`; `speckit_plan_auto.yaml:297`; `speckit_plan_confirm.yaml:303`) never define a `runtime_agent_path_resolution:` block anywhere in the file, in any sibling speckit asset, or in any shared/included fragment (confirmed no include/anchor mechanism exists in this codebase's YAML — `grep -n "^\s*<<:\|_anchors\|!include\|extends:"` on `speckit_complete_auto.yaml` returns nothing). This is distinct from and more foundational than the already-known singular-`.opencode/agent` typo (iteration 7 finding, confined to the 4 `/create` files that DO define the block) — the speckit family's 12 `[runtime_agent_path]` interpolation sites (`deep-research.md` x3, `review.md` x4, `debug.md` x4, and one further `deep-research.md`) have **no resolution definition at all**, meaning `[runtime_agent_path]` is a dangling, literally-unresolvable bracket token when a speckit workflow YAML is the only file loaded (the `/create` family's block is not in scope/loaded during a `/speckit:complete`, `/speckit:implement`, or `/speckit:plan` run). No code-level resolver exists anywhere in the repo (confirmed via full-tree grep across `.ts/.js/.cjs/.mjs`) — this token is exclusively resolved by hand-authored per-file YAML blocks, and speckit never carries one.

**Fix**: Add an identical `runtime_agent_path_resolution:` block (`default: .opencode/agents` — using the corrected plural form, not perpetuating the iteration-7 singular typo — and `claude: .claude/agents`) to the top of all 6 speckit workflow YAML files, matching the `/create` family's existing pattern (`create_agent_auto.yaml:44-46`).

### F-cli-opencode-standalone-skill-framing-4x (P1, cross-surface — expansion of recon seed #4)

Recon seed #4 flagged `deep_research_auto.yaml:1012,1016` framing `cli-opencode` as a standalone top-level "skill." The mechanical sweep found this exact phrasing pattern — "The cli-opencode skill SKILL.md §SELF-INVOCATION PROHIBITED contract is the authoritative gate" — repeated verbatim (with minor prose variation) in **4 total locations**, not 1:
- `.opencode/commands/deep/assets/deep_research_auto.yaml:1016`
- `.opencode/commands/deep/assets/deep_research_confirm.yaml:765`
- `.opencode/commands/deep/assets/deep_review_auto.yaml:1073`
- `.opencode/commands/deep/assets/deep_review_confirm.yaml:840`

All four say "the cli-opencode skill" as if `cli-opencode` were addressable as its own top-level skill; its actual `SKILL.md` lives at `.opencode/skills/cli-external/cli-opencode/SKILL.md` — a nested mode under the `cli-external` parent hub (confirmed against the current hub structure: `cli-external{cli-opencode,cli-claude-code}`). This is a load-bearing dispatch-safety guard citing a path/skill-identity that no longer matches the current hub topology; a reader trying to verify the SELF-INVOCATION PROHIBITED contract by looking for a standalone `cli-opencode` skill directory would not find one.

**Fix**: Reword all 4 sites from "the cli-opencode skill SKILL.md" to "the `cli-external/cli-opencode` mode's SKILL.md (`.opencode/skills/cli-external/cli-opencode/SKILL.md`)" to match current hub structure, consistent with how `cli-claude-code` (the sibling mode) would be referenced.

### F-deep-family-agent-file-comment-inconsistent-with-dispatch-mechanism (P2, commands, cross-surface)

The 4 `/deep` family `agent_file:` sites (`deep_research_auto.yaml:83`, `deep_research_confirm.yaml:99`, `deep_review_auto.yaml:74`, `deep_review_confirm.yaml:72`) hardcode `.opencode/agents/deep-research.md` / `.opencode/agents/deep-review.md` with the comment "Canonical path. Runtime-specific directories (e.g. `.claude/agents/`) reference this source." This differs structurally from the `/create` and `/speckit` families' `[runtime_agent_path]` token approach — the deep family's actual dispatch mechanism (verified at `deep_research_auto.yaml:173,862`: `dispatch: { agent: deep-research }`) never interpolates this `agent_file` field at all; it dispatches by bare agent name. The `agent_file:` field in `agent_config:` is therefore purely descriptive/documentary in this family, not a live dispatch target — unlike create/speckit where `agent_file:`'s value is the thing actually substituted into a Task-tool call. This is not a broken reference (the path is correct), but it is an inconsistent schema: three command families use the same field name (`agent_file:`) for two different semantics (live interpolation target vs. static documentation-only pointer), which risks a future editor "fixing" the deep family's `agent_file` value to a `[runtime_agent_path]` token under the mistaken assumption it needs cross-runtime resolution like the other two families.

**Fix**: No dead-reference fix needed (the path is correct and consistent with `.claude/agents` being real files, not the create/speckit runtime_agent_path pattern). Recommend a one-line schema note in the `agent_config:` block of all 4 files clarifying that `agent_file:` here is documentation-only (dispatch is by bare `agent:` name), to prevent future confusion with the create/speckit `[runtime_agent_path]`-interpolated usage of the same key name.

This closes the "beyond `/create`, `/deep`, `/speckit`" branch of the open question: there is nothing to find in the other three families because they contain zero `agent_file`/`agent_availability` declarations, dead or otherwise.
