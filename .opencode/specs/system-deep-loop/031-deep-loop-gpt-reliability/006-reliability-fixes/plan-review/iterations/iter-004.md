## GAPS

1. **HMAC key transits a command-string channel the threat model never analyzes — forgeability collapses.**
   - Evidence: iter-012.md:61 states the key is "passed only as workflow-private binding to validation, never written to prompt/state and never passed through `buildExecutorDispatchEnv`" — and `buildExecutorDispatchEnv` (executor-audit.ts:537-553) only filters the *child's* env. But iter-012.md:192 interpolates `receiptKey: '{dispatch_receipt.key}'` into the `if_cli_opencode.command` heredoc body, and the runtime must materialize that `command:` string to execute it — either as a `sh -c` argv (the *exact* ancestry/cmdline surface F-013's `detectFromAncestry(kind, context.ancestryCmdlines)` at executor-audit.ts:495 already inspects) or as a temp script file in a dir the child can read. The design's entire premise is "unforgeable by construction" (F-041, spec 004 §2), yet its only leak-control addresses env, not command-string/ancestry/fs transit. A compromised executor reading ancestor cmdlines (`ps`, `/proc`) or the temp script recovers the key and self-issues valid receipts, defeating F-010/F-011/F-012 outright. No `crypto`/`hmac` exists in executor-audit.ts today (verified), so this is the design's hole, not a built system.
   - Severity: **blocker**
   - Amendment: Keep the key in the engine process. Have `writeDispatchReceipt` compute the HMAC in the YAML-engine step using facts known pre-dispatch, and have the audited wrapper *return* post-dispatch facts (child pid, exit status, actual sessionId) for the engine to countersign — never pass `receiptKey` into `runAuditedExecutorCommand` or any `command:` string. Add a threat-model section proving the key's only residence is engine heap, with an explicit assertion that no `command:` template ever receives it.

2. **`runAuditedExecutorCommand` is not reachable from any CLI branch today — REQ-003 is a 12-command migration, not a re-wire.**
   - Evidence: F-013 claims the wrapper is "bypassed"; verified current state confirms none of the three CLI branches call it. `if_cli_copilot` (deep_review_auto.yaml:817-869) calls `spawnSync('copilot', built.argv)` directly; `if_cli_claude_code` (880-903) shells `claude` directly; `if_cli_opencode` (904-914) shells `opencode run` directly. Phase 004 REQ-003 ("route the auto-YAML CLI branches through the audited executor wrapper") and iter-012 deliverable 2 only redesign the `cli_opencode` branch — copilot (with its `@PROMPT_PATH` rewrite at 859) and claude-code (with `CLAUDE_CONFIG_DIR` at 882 and `</dev/null` at 914) are undesigned. Across review/research/context/ai-council auto YAMLs that is ~9-12 command rewrites, each with branch-specific arg construction. Spec 004 risks row names none of this; effort is "M" with no per-branch tasking.
   - Severity: **major**
   - Amendment: Scope REQ-003 per-branch-per-YAML with a migration sub-task for each of {copilot, claude-code, opencode} × {review, research, context, ai-council}; spec the wrapper's input contract generically enough for all three arg styles (positional vs `@path` vs `-p`); add a regression acceptance cell asserting each branch still produces identical dispatch behavior post-wrap.

3. **The native Task path cannot bind a real child identity into a pre-dispatch receipt — receipts prove parent intent, not child work.**
   - Evidence: The receipt is written in `pre_dispatch_receipt` (iter-012.md:100-114), i.e. before dispatch. The native branch is `if_native: dispatch: agent: deep-review` (deep_review_auto.yaml:812-816) — a YAML-engine primitive whose child sessionId/pid is assigned by the runtime only once the Task call executes, unknowable at receipt-write time. The schema nonetheless lists `child.sessionId` + `sessionIdSource: 'native_dispatch'` as required fields (iter-012.md:24-28). The only honest native source is the parent's `dispatchId` proxy. So the receipt attests "the engine intended dispatch X," not "child process Y performed this work" — strictly weaker than the "unforgeable + child-bound" framing in spec 004 §2 and F-041. Spec 004 success criterion #2 ("native dispatch pass") defines no test asserting the native receipt binds to the actual subagent session. This is the native/CLI asymmetry the brief asks about.
   - Severity: **major**
   - Amendment: Split the receipt into a pre-dispatch *intent* record (engine-signed, no child id) and a post-dispatch *completion* countersign that the engine writes after the native Task returns the real sessionId/pid; require the validator to bind the two. Add an acceptance test: a native dispatch whose post-completion child id differs from the runtime-assigned one is rejected. Document that `sessionIdSource:'native_dispatch'` is unachievable pre-dispatch and remove it or redefine it as post-hoc.

4. **No failure mode for receipt-write I/O failure on a legitimate dispatch — transient errors convert success to rejection.**
   - Evidence: REQ-001 ("emit a dispatch_receipt") specifies no atomic-write, lock, retry, or idempotency for `writeDispatchReceipt` appending to `{artifact_dir}/dispatch-receipts.jsonl`. Success criterion #1 ("inline work without a receipt is rejected") provides no carve-out for a dispatch that occurred but whose receipt write failed (disk full, `EACCES`, concurrent-append line interleave from sibling seats in the same artifact_dir per F-016/F-017 parallel sweep/council). iter-012 deliverable 3 makes a missing receipt a hard `dispatch_receipt_missing` failure whenever route proof is configured. The two outcomes are both bad: hard-fail aborts legitimate completed work; soft-continue rejects it at validation. There is no recovery path.
   - Severity: **major**
   - Amendment: Add a REQ that `writeDispatchReceipt` use an atomic write (`O_APPEND` single write of a pre-serialized line, or temp-file+rename for the whole file under a per-dispatch lock), with idempotent retry on transient I/O error and a deterministic `dispatch_receipt_write_failed` failure class distinct from `dispatch_receipt_missing`. Define behavior for concurrent seats: per-dispatch receipt files keyed by dispatchId, or an append lock, so post_dispatch_validate line-parse can't corrupt.

5. **Demoting route-proof fields breaks undocumented downstream consumers across 4 YAMLs and the validator.**
   - Evidence: Verified consumers of `target_agent`/`resolved_route`/`agent_definition_loaded`: post-dispatch-validate.ts:619-666 enforces presence+equality (hard `route_proof_missing`/`route_proof_mismatch`), called at :1384-1394; and `assert_jsonl_fields` lists them as REQUIRED jsonl fields in deep_review_auto.yaml:932, deep_context_auto.yaml, deep_research_auto.yaml, and deep_ai-council_auto.yaml. REQ-002 ("demote model-written route fields to advisory") and iter-012 deliverable 3 only patch the validator's route-proof check and show one yaml excerpt's field list — they do not touch the four YAMLs' `assert_jsonl_fields`, nor audit any scorecard/dashboard/reducer that reads these fields for agent attribution. Leaving them required in 4 YAMLs turns every post-004 dispatch into `jsonl_missing_fields` failures.
   - Severity: **major**
   - Amendment: Add a migration REQ that enumerates every `assert_jsonl_fields` and every reader of the three route fields (grep-verified list), removes them from required-field lists in all 4 auto YAMLs, and either drops the validator's `validateRouteProofRecord` or repoints it at receipt-derived values exclusively. Add an acceptance assertion: a dispatch whose state log omits the three model-written fields still passes when a valid receipt exists.

6. **No key lifecycle for resumable/multi-dispatch runs — receipts become unverifiable exactly when phase 009 lands (couples GAP-13).**
   - Evidence: post_dispatch_validate needs `receipt_key` to verify (iter-012.md:216 `receipt_key: "{dispatch_receipt.key}"`). The key is "workflow-private … never written to prompt/state" (iter-012.md:61) — i.e. in-memory for one process lifetime. Phase 009 (closes F-032) splits the heaviest command into resumable sub-invocations sharing one session id + ledger = N dispatches across process boundaries; any crash/resume must re-read the key to validate earlier receipts. If the binding is in-memory only, a resumed run sees all prior receipts as `dispatch_receipt_invalid`. If persisted to disk for resume, the child — which has read+write on `artifact_dir` (it writes iteration files there) — can read it and forge. F-041/iter-012 assume single-process lifetime; spec 004 names neither storage nor derivation. This is the same surface as the threat model but on the storage axis.
   - Severity: **major**
   - Amendment: Specify key derivation as a function of a per-run secret the engine holds plus the dispatchId (e.g. HMAC over dispatchId with a run-master key), so receipts are re-verifiable on resume without persisting any per-dispatch key; OR define a sealed key store the child cannot read (mode 0600, parent-owned) and a resume protocol that re-opens it. Add 004 as an explicit dependency of 009 (sharpening registry GAP-13) with a resume-mid-run acceptance test asserting all sub-invocation receipts validate.

7. **Receipt log shares the child's r/w artifact_dir — delete/truncate DoS and non-atomic concurrent append.**
   - Evidence: `receiptLogPath: "{artifact_dir}/dispatch-receipts.jsonl"` (iter-012.md:58,79) sits in the same directory the child legitimately writes iteration files to (deep_review_auto.yaml iteration_pattern under artifact_dir). The child cannot forge a MAC (no key) but *can* `unlink`/truncate the receipt file → `dispatch_receipt_missing` (silent DoS that looks like absorption). Concurrent seats in one artifact_dir (F-016 council round_started/seat_started, F-017 context parallel sweep) appending to one JSONL file risk non-atomic line interleave that breaks post_dispatch_validate's line parse. The replay test-case (iter-012.md:309) relies on `previousStateLogSize`, a value the child influences by appending — fragile binding. iter-012 specifies no `O_APPEND`/lock/per-dispatch file.
   - Severity: **major**
   - Amendment: Write receipts to a parent-owned, child-unwritable path (e.g. `{artifact_dir}/.receipts/` mode 0700 parent uid), or one file per dispatchId (`{dispatchId}.receipt.jsonl`) to eliminate both the truncate-DoS and the concurrent-append race; have the validator tolerate a missing receipt file only when a sibling intent+completion pair exists for the dispatchId. Add an acceptance test: a child that deletes its receipt file is classified as `dispatch_receipt_destroyed` (failure), not silently absorbed.
