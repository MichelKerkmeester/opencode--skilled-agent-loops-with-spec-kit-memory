# v3.5.0.0, The Self-Maintaining Memory Runtime, the Forkable Toolkit, and a Program That Closed Itself Out

v3.4.2.0 was a hardening-and-honesty release: it built the comment-hygiene detector, closed the embedding-stack review backlog, and repaired the memory database. v3.5.0.0 is the release where the memory and MCP substrate stops needing a babysitter. Checkpoints become real files that survive a crash. The memory index learns to reconcile and self-heal. The MCP launcher grows a front-proxy that hides a daemon recycle from every client. Post-save enrichment, reconsolidation, and causal relation inference flip on by default and run in the background. And the long-running 026 graph-and-context-optimization program lands its final memory-runtime sprint, then documents every single one of its ~634 phases into one canonical changelog tree before closing.

Alongside the runtime work, the skill toolkit was re-architected to be forkable: a three-layer prompt-knowledge split moves the model registry and benchmarks into `sk-prompt-small-model` so `sk-prompt` can be lifted out cleanly, every skill's playbook gets global sequential numbering, and the deep-improvement skill grows a third lane that benchmarks skills for real-world usefulness. The tree also got materially lighter: a reorg baseline purged the vendored hybrid-RAG research repos, removing roughly 2.1 million lines that never belonged in the shipping tree.

The throughline mirrors the prior release: build the durable mechanism before you trust the convenience, re-validate every deep-review finding against HEAD before you touch it, verify a fix through the path that actually failed, and when you document history, make the document match the code even when the honest entry is "shipped, then removed."

## Checkpoint-v2: Checkpoints That Survive a Crash

The pre-v3.5 checkpoint path took an in-process snapshot. A restore that failed partway could leave the live database half-applied, and a sharded runtime was not reliably selected into the v2 path. This release rebuilds checkpoints as durable files across a phased landing.

- **Phase 1 (`966a75c3be`)** introduced schema v29 and exposed `includeEmbeddings` on `checkpoint_create`, touching the handler, the vector-index schema, the storage layer, and the tool schema.
- **Phase 2 (`f86a80ca65`)** made CREATE file-based via `VACUUM INTO`, a single 605-line change in the checkpoint storage layer with its own create suite.
- **Phase 3 (`40f647e4ae`)** added the file-swap RESTORE path and a reopen coordinator, so a restore swaps a complete file in rather than mutating the live DB in place.
- **Phases 4-7 (`b9820541e9`)** landed the journal crash-safety remediation: a durable restore journal across the context server, db-state, reindex, extraction adapter, and vector-index store, so a crash mid-restore is recoverable.
- **The `.needs-rebuild` sentinel (`6c76793f25`, 013/005)** is a durable post-restore self-heal marker: if a restore cannot finish cleanly, the next boot sees the sentinel and rebuilds rather than trusting a partial state.
- Two correctness fixes hardened the journal ordering: `fsync` the stale-`.bak` removal *before* the restore journal records it (`83e0661e5f`), and demote the restore journal to swap-pending *before* an in-process revert (`29160c0e50`).
- `cce4fe931d` fixed v2 selection on sharded runtimes by gating on `vec_memories` only, and `253db8b438` updated the downgrade-guard test for the subsequent SCHEMA_VERSION 30 bump.

Schema then moved to **v30** in `203fb19cbc` (013/004) as a memory_save replay-enrichment marker with repair-on-replay, landing through the 013 phase-parent restructure.

## The Self-Maintaining Memory Index

`memory_index_scan` graduated from a manual trigger to a self-maintaining surface across three phases plus a uniqueness guard.

- **Phase 1 (`98330d18fc`)**: a coalescing contract (redundant concurrent scans collapse), a `health.index` field on `memory_health`, and an orphan sweep.
- **Phase 2 (`be23d4a295`)**: an async scan with a drain circuit guard, so a long scan does not block and a failing provider cannot wedge the queue.
- **Phase 3 (`757f5235db`)**: move reconciliation (a relocated spec folder is re-pointed rather than orphaned) and a scan heartbeat for liveness.
- **`942ad78d9c`**: an active-row uniqueness guard plus multi-tenant scope isolation across the save pipeline (24 files), closing the door on duplicate active rows and cross-scope bleed.

The 013 packet that carried this work was restructured into a phase parent (`db88702658`) holding the self-maintaining-index child and the checkpoint-v2 child, then reconciled to shipped and strict-validated.

## MCP Front-Proxy and Launcher Reliability

The recurring failure mode behind much of the prior release's friction was a daemon recycle surfacing as a hard client error. This release inserts a front-proxy between clients and the daemon.

- **`598be61b05`** built the MCP front-proxy: an in-launcher session proxy that survives a daemon recycle transparently, across the launcher session-proxy lib, the launcher, the context server, and the idle-timeout path (11 files, +1,402).
- **`5f79a9667f`** added a fail-closed re-handshake on protocol-version drift, so a client and a recycled daemon that disagree on protocol version re-negotiate instead of silently mismatching.
- **`2d25ed42ea`** made multi-client reconnect transparent via a per-client proxy.
- **`c68c7bf5e3` (026/018)** hardened the front-proxy against an externally-killed or slow-booting daemon, and **`1f1e52ca8e` (026/020)** stored the owner's actual IPC socket path in the lease so a divergent-`SPECKIT_IPC_SOCKET_DIR` secondary launcher bridges to the live daemon instead of reporting `no-bridge-socket`.
- **`d1183dc07d` (026/024)** un-skipped and de-flaked the launcher-lease integration suite, which had been failing 100% deterministically because the fixture copied the launcher `.cjs` but not its `lib/` tree (every spawned launcher died `MODULE_NOT_FOUND` before writing a lease). The fix copies the whole `lib/` tree, isolates each test's socket dir, and adds an end-to-end socketPath bridge test.

This sits on top of the daemon-lifecycle root-cause heal from the start of the window (`d626b5aa5f`, 032/001): a boot-time FTS auto-heal plus a launcher clean-close barrier, backed by a substrate test.

## Upgrade

- **No breaking API changes.** New capability is additive. The MCP server reports `1.8.0`.
- **Checkpoint schema is now v30.** Checkpoints created on older schema versions are handled by the downgrade guard; the v2 file-based path activates automatically on the next checkpoint create/restore. A restore that is interrupted now leaves a `.needs-rebuild` sentinel and self-heals on the next boot.
- **Post-insert enrichment is default-on and async.** Saves return immediately with `enrichmentStatus: deferred` and a bounded background scheduler runs enrichment after commit. Set `SPECKIT_POST_INSERT_ENRICHMENT_SYNC=true` to force the legacy synchronous path. Reconsolidation remains **opt-in** via `SPECKIT_RECONSOLIDATION_ENABLED`.
- **Causal relation backfill is available but conservative.** `memory_causal_stats({ backfill: { dryRun: false } })` promotes deterministic structural signals into `caused`/`supports` edges. The similarity and `contradicts` collectors are **opt-in** (`similarity: true` / `contradicts: true`) and gated by the SEC-001 conflict guard, which skips any auto edge whose pair already carries a valid conflicting edge.
- **Embedding resolution is local-first.** With no explicit `EMBEDDINGS_PROVIDER`, resolution is `persisted-ollama → hf-local`, and cloud (OpenAI/Voyage) is only reached as an explicit choice or a genuine last-resort cascade. A running daemon must recycle to pick up the rebuilt `shared/dist`.
- **Worktree-per-session isolation is opt-in via the launch path.** The guard is wired into the Claude SessionStart chain; `sk-git` creates numbered `wt/{NNNN}-{name}` worktrees.
- **`sk-prompt` is now forkable.** The model registry and benchmarks live in `sk-prompt-small-model`; consumers that referenced the old in-`sk-prompt` registry paths should repoint to the hub.
- **026 changelog history is canonical.** The per-program index lives at `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/changelog/README.md`, with one rollup per track and one leaf per shipped phase.
