Runtime-robustness parity for `deep-context`. The loop now inherits the same durability and validation guarantees that `deep-research` and `deep-review` get from the shared `deep-loop-runtime`, so a crashed or interrupted run no longer leaves half-written state, malformed seat output no longer slips silently into the merge, two runs cannot clobber one packet, and a CLI seat cannot recursively launch another loop. All wiring is gated to `loop_type='context'`, so research and review behavior is unchanged.

## Why This Release Matters

The first release shipped `deep-context` as a host-driven loop that used the runtime's coverage-graph, convergence, and council dispatch, but skipped the runtime's durability and validation layer. Its reducer wrote the registry and dashboard with plain non-atomic writes, read the JSONL state log without repairing a crash-corrupted tail, merged seat findings without validating them, locked the packet with prose-only semantics, and dispatched CLI seats without a recursion guard. The mature loops avoid all of those failure modes by using the shared runtime helpers.

This release closes that gap. The reducer and a new lock wrapper now call the runtime helpers directly (in-process via the tsx CJS register, with contract-equivalent inline fallbacks so the dual-use scripts stay runnable and importable), and the loop workflows wire single-writer locking and the executor-audit recursion guard.

---

## New Features

#### Atomic state writes

`reduce-state.cjs` writes `findings-registry.json` through the runtime `writeStateAtomic` helper and the dashboard through an atomic temp-fsync-rename text write. A crash mid-write leaves the previous complete file intact, never a half-written artifact for a downstream reader.

&nbsp;

#### JSONL tail repair

Before reading `deep-context-state.jsonl`, the reducer runs the runtime `repairJsonlTail` to truncate a crash-corrupted trailing line, then surfaces `{repaired, droppedBytes}` as `registry.stateLogRepair`. An interrupted final append no longer aborts the reduce.

&nbsp;

#### Post-dispatch seat validation

Each seat finding is validated before merge (known kind, a path or symbol present, numeric relevance). Invalid findings are surfaced in `registry.seatValidationWarnings` rather than silently merged or dropped, mirroring the runtime post-dispatch-validate discipline.

&nbsp;

#### Single-writer loop lock

A new `scripts/loop-lock.cjs` wraps the runtime `acquireLoopLock`/`refreshLoopLock`/`releaseLoopLock`. The `:auto` and `:confirm` workflows invoke it in `step_acquire_lock` and `step_release_lock` for stale-lock reclaim and clean owner-scoped release, replacing the prose-only advisory lock.

&nbsp;

#### Executor-audit recursion guard

The CLI seat dispatch contract now sets the runtime executor-audit recursion-guard env (`SPECKIT_CLI_DISPATCH_STACK` via `buildExecutorDispatchEnv`), so a dispatched CLI seat cannot recursively launch another deep-context loop, and seat provenance follows the runtime audit contract.

&nbsp;

#### Packet restructure + docs

Packet 134 is now a phase parent: `001-context-loop-foundation` (the shipped foundational build) and `002-runtime-robustness-parity` (this release). The feature catalog and manual testing playbook gained a `07--runtime-robustness` category (five features, five `RUNTIME-*` scenarios), and the SKILL, README, agent, and command docs document the inherited robustness layer.
