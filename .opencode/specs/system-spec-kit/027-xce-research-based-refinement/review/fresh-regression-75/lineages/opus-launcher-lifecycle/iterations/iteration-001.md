## Review — seat `opus-launcher-lifecycle`

**Angle:** re-election lease liveness; hunt a window where two launchers both write.

**What I traced.** The spec-memory single-writer guarantee rests on three layered locks: the owner lease (`.spec-memory-owner.json`), the pid lease (`.mk-spec-memory-launcher.json`, carrying `childPid`), and the exclusive respawn lock. I traced both reclaim paths — `main()` stale-reclaim (lines 1556–1696) and `respawnAfterDeadSocket` (774–855) — against the `launchServer` owner gate (1300–1303).

**The good news (no P0/P1 double-writer).** The DB single-writer is preserved across every path I examined:
- `respawnAfterDeadSocket` re-claims the owner lease with `writeOwnerLeaseFileExclusive` (O_EXCL, line 829) *after* taking the respawn lock, so even two launchers cannot both pass it.
- The `main()` stale-reclaim path is gated by `launchServer`'s `ownsOwnerLeaseFile(process.pid)` (1300), and the owner-lease file holds exactly one pid (read fresh), so only that one pid spawns the daemon.
- `recycleDaemonInPlace` deliberately keeps the lease across an in-place recycle (1200–1204) to avoid the no-lease window. Correct.

**Two real gaps found (both P2, both in the re-election serialization).**

1. **Respawn lock is not actually exclusive under staleness.** `acquireRespawnLockFileAt` (supervision 666–680) reclaims a stale/empty lock by `unlinkSync` + reopen, *without* atomically claiming it first. Two launchers racing a stale respawn lock (crashed prior holder, dead pid) can interleave: A unlinks → A reopens (holds) → B unlinks (deletes **A's** fresh lock) → B reopens (holds). Both now believe they hold the "exclusive" lock the launcher comment (1616–1617) relies on to serialize reap+respawn. The bootstrap lock solves exactly this with an atomic rename-claim (1272–1279); the respawn lock does not. For spec-memory the DB double-writer is masked by the owner-lease gate, but the lock's documented contract is breached and the model-server respawn path relies on this same lock alone.

2. **Owner-lease reclaim reread over-claims, yielding a no-daemon liveness hole.** The comment at line 478 ("Re-read after reclaim so two launchers racing a stale lease cannot both act as owner") is unsound: `writeOwnerLeaseFile` + reread (479–489) only catches an interleaved writer, not sequential execution — A writes/rereads A (acquired), then B writes/rereads B (acquired); both return `acquired:true` with the file ending at pid B. Because the owner lease and the respawn lock are independent and unordered, the launcher that wins the respawn lock and does the reap (say A, which finished its owner-acquire first) is then **blocked** by `launchServer`'s `ownsOwnerLeaseFile` gate (file holds B), while B loses the respawn lock and bails. Net: no daemon spawned this round, and A exits **silently** (main line 1687 `if (!launched) return;`) — no retryable JSON-RPC error to its host, unlike every other bail path. Self-heals on the host's next retry, so availability-only, but the host-A session is dead until then and the invariant comment is misleading.

Both are narrow concurrent-cold-start races, no data loss, single-writer intact — hence P2. The respawn-lock non-atomic reclaim is the one I'd fix (mirror the bootstrap lock's atomic claim) since it's a lock documented as exclusive that isn't.
