## [**1.5.0.0**] - 2026-05-11

First major iteration of `sk-doc` after the v1.3.0.0 series. Packet 102 ships five sequential phases that restructure the skill surface: reference relocation, a dedicated skill README asset, an agent identity rename across three runtime mirrors, manual playbook coverage for the renamed agent, and a single transactional remediation pass that closes every P1 and P2 finding from the phase-parent deep-review. All five phases land as additive child packets in `skilled-agent-orchestration/102-sk-doc-skill-readme-and-structure`. The `/create:*` command family stays surface-stable.

> Spec folder: `.opencode/specs/skilled-agent-orchestration/102-sk-doc-skill-readme-and-structure/{001-sk-doc-reference-relocation, 002-sk-doc-skill-readme-asset, 003-markdown-agent-rename, 004-sk-doc-playbook-markdown-agent-coverage, 005-deep-review-p1-p2-remediation}` (Level 2 phase parent, 5 phase children).

## New Features

### Reference Relocation (Phase 001)

The six sk-doc creation guides moved from `.opencode/skills/sk-doc/references/specific/` to the references root. SKILL.md smart routing, README related-documents, asset template links, manual playbook expected-resources, every `/create:*` command doc and YAML asset, and `graph-metadata.json` source-doc paths all point at the new root locations. The empty `specific/` directory is gone. Exact-match grep across the skill, agents, and commands surfaces returns zero stale `references/specific` matches.

&nbsp;

### Skill README Template Asset (Phase 002)

A new `assets/skill/skill-readme-template.md` scaffold lands in the skill-creation flow alongside the SKILL.md and reference templates. The template structure aligns with `skill_asset_template.md` and opens with a short H1 intro, then `## 1. OVERVIEW`, `Purpose`, `Usage`, and `Location & Naming` before the content model. The `SKILL_CREATION` intent in SKILL.md now loads four resources instead of three, and `references/skill-creation.md` carries new README guidance. Folder README authoring continues to use `assets/readme/readme-template.md`. Skill-package README authoring uses the new dedicated asset.

&nbsp;

### Markdown Agent Identity Rename (Phase 003)

The dedicated documentation executor agent renames from `@create` to `@markdown` across all three runtime mirrors: `.opencode/agents/markdown.md`, `.claude/agents/markdown.md`, and `.codex/agents/markdown.toml`. The Codex registry block updates from `[agents.create]` to `[agents.markdown]` with `config_file = "agents/markdown.toml"`. Orchestrator and code-agent references route documentation and component authoring through `@markdown`. The `/create:*` command family (`/create:agent`, `/create:sk-skill`, `/create:feature-catalog`, `/create:testing-playbook`, `/create:folder_readme`, `/create:changelog`) keeps every invocation unchanged because this is an agent identity rename, not a command-family rename.

&nbsp;

### Manual Playbook Coverage For @markdown (Phase 004)

The manual testing playbook adds a new `06--agent-dispatch/` section with three `dispatch_real` scenarios that actually invoke `@markdown` across the three named CLI runtimes. SD-018 covers `@markdown` via cli-claude-code on Claude Opus 4.7. SD-019 covers `@markdown` via cli-codex on `gpt-5.5 / fast` with sandbox network access. The playbook preamble, Categories row, and Scenario Index updates close the prior gap where the preamble admitted three CLI runtimes while listing only two. All three scenarios executed sequentially with persisted evidence under `evidence/`.

## Architecture

### Path Move (Phase 001)

The `references/specific/` subdirectory was the only directory under references that grouped sk-doc creation guides separately from global standards. The relocation flattens that layer so creation guides sit at the same level as global standards under `references/`, with `global/` retained for the global-standards subset. YAML key names like `specific_creation` stay stable to avoid behavior changes outside path relocation.

&nbsp;

### Skill Asset Category (Phase 002)

A new `assets/skill/` category joins the existing four asset categories (`changelog_template.md`, `frontmatter_templates.md`, `llmstxt_templates.md`, `readme/`, `flowcharts/`, `agents/`, `feature-catalog/`, `testing_playbook/`, `template_rules.json`). The category exists to separate skill-package README scaffolds from folder README scaffolds because they target different content models.

&nbsp;

### Agent Identity Decoupling (Phase 003)

The rename decouples agent identity (`@markdown`) from command-family identity (`/create:*`). Before this phase the two were conflated, so renaming the agent risked breaking command callers. The decoupling lets the documentation executor evolve under one name while every `/create:*` invocation keeps working. The Codex TOML mirror needed the same identity rename because `.codex/agents/create.toml` existed and was wired through `[agents.create]` in `.codex/config.toml`.

&nbsp;

### Inline-Contract Workaround For Codex SpawnAgent (Phase 005)

The cli-codex `@markdown` typed-agent dispatch gap reflects a two-registry architecture in codex v0.130.0. The `.codex/config.toml` user-defined agent block IS read for CLI context, but the `codex_core::tools::router` SpawnAgent runtime allowlist is a separate baked-in list that does NOT propagate user-defined agents. The resolution rewrites SD-019 with `execution_mode: dispatch_inline_contract`, an explicit SpawnAgent forbid, a Gate 3 pre-answer of "D) Skip", a BINDING-trace requirement, and contract-acknowledgment fields in success criteria. Codex `gpt-5.5` then reads `.codex/agents/markdown.toml` developer_instructions and follows them itself without dispatching a sub-agent. SD-018 and SD-020 still verify real typed-agent dispatch, so the three CLIs test the same agent identity through three different mechanisms.

## Upgrade

No migration is required for callers of `/create:*` commands. The command surface stays identical.

Downstream agents and orchestrators that previously typed `@create` as the documentation executor MUST switch to `@markdown`. The rename applies to all three runtime mirrors. Specifically:

1. Update any orchestrator routing rule that named the previous documentation executor.
2. Update any Codex registry consumer that read `[agents.create]` to read `[agents.markdown]`.
3. Update any saved prompt or template that hard-coded `agents/create.md` or `agents/create.toml` paths.

Real typed-agent dispatch via codex `SpawnAgent` remains upstream-blocked in codex v0.130.0. SD-019 carries `expected_skip_in_non_interactive: true` with a documented rationale, and resolves the contract via the inline-contract execution mode. A future codex release that maps `.codex/config.toml` user-defined agents into the `codex_core::tools::router` SpawnAgent allowlist would let SD-019 flip back to `dispatch_real` and exercise true typed-agent dispatch under `codex exec`.

If a downstream consumer depended on the `references/specific/` path prefix for sk-doc creation guides, switch to the new root-level paths: `references/skill-creation.md`, `references/agent_creation.md`, `references/readme_creation.md`, `references/install_guide_creation.md`, `references/feature_catalog_creation.md`, `references/manual_testing_playbook_creation.md`.
