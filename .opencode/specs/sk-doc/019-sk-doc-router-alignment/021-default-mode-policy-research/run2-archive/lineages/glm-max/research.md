# Research Synthesis — glm-max lineage (defaultMode Run 2, divergent)

**Executor:** cli-opencode / GLM-5.2 (`zai-coding-plan/glm-5.2`). One of 4 parallel-detached fan-out lineages (sol-ultra, luna-max, glm-max, terra-max), each converging independently. This is the GLM-5.2 lens.
**Method:** 5 iterations, `stopPolicy: max-iterations`, `convergenceThreshold: 0.05` (telemetry only — the loop ran all 5 by contract; convergence was deliberately not allowed to synthesize early). Each iteration opened a distinct thread from the spec's Divergent Exploration Agenda and was explicitly forbidden from re-deriving Run 1's verdict.
**Baseline diverged FROM:** Run 1 (4 iterations, in `run1-archive/`) — keep sk-prompt; flip cli-external, system-deep-loop, mcp-tooling, sk-design to null; null's fallback loads the routing helper; add a "defer-routed keyword hub" third archetype.

## Bottom line

Run 1's core verdict **survives** an adversarial GLM lens — but with one overturned flip, one epistemic downgrade, and a constructive replacement of its encoding.

1. **One per-hub disagreement (the load-bearing delta):** `cli-external-orchestration` should be **detection-defaulted**, not pure-defer null. Its best default ("the executor of the runtime you're inside") is environmentally determined and self-correcting; nulling throws away cli's one genuine signal. Run 1 correctly killed the *static* bias but left cli's best behavior unencoded.
2. **One epistemic downgrade:** the three pure-defer nulls (system-deep-loop, mcp-tooling, sk-design) stand as *directional-pending-measurement*, not *proven* — Run 1's deliberate no-corpus method leaves them unfalsifiable in the over-deferring direction. They are truth-reconciliation, not validated behavior improvement (the benchmark is blind to keep-vs-null, Run 1's own finding).
3. **One constructive replacement of the encoding:** Run 1's uniform "third archetype" becomes a falsifiable **four-shape decision rule**, where each shape is nominated by a *testable condition* (is there an environmental signal for this axis?), not by imitation.

## The constructive output: the four-shape decision rule

The encoding deliverable. Each shape is nominated by a **falsifiable condition**, replacing Run 1's single "third archetype." The shapes are **per-axis** (a hub can be more than one — sk-code already is), not per-hub.

| Shape | Default resolution | Falsifiable nomination condition | Example |
|---|---|---|---|
| **keyword-with-default** | static named child (catch-all anchor) | one mode is the genuine dominant case AND serves as the legal home for hub-generic vocabulary (canon's hub-identity-on-default rule) | sk-prompt → prompt-improve |
| **detection-defaulted** | `null` static + env-detected default | an environmental signal exists that resolves the *default* (not a specific mode) | cli-external-orchestration (runtime) |
| **detection-routed (axis)** | `null` + env-detected mode | an environmental signal exists that resolves a specific mode/axis | sk-code surface axis (CWD/markers) |
| **pure-defer** | `null` + disambiguation-card fallback | NO environmental signal AND no genuine catch-all | deep-loop, mcp-tooling, sk-design |

**The discriminant that picks the shape:** *is the routing signal for this axis environmental (machine-detectable) or intentional (lexical)?* Environmental → detect (shapes 2/3). Intentional + one dominant → default (shape 1). Intentional + no dominant → defer (shape 4). This is testable per-axis, which is why the rule is falsifiable where Run 1's "imitate the third archetype" is not.

## Key findings (by thread)

### Q1 — Layer-0 ↔ Layer-1 coupling (thread 2)
Flipping to null moves **zero** disambiguation work to Layer-0. The hub's defer is a *terminal branch* at Layer-1: it asks the user via `UNKNOWN_FALLBACK_CHECKLIST`, never re-invokes the advisor (no infinite loop). The advisor already ships the same defer-with-clarification pattern Run 1 prescribed for hubs (`DEFAULT_RESOURCE_SEMANTICS = "fallback-only"`; `clarifying_question` on low deep-routing confidence), and already pre-resolves modes for **one** hub (system-deep-loop, via a bespoke regex-gated `_apply_deep_skill_routing_layer` that writes `recommendation["mode"]`). That layer is hand-maintained and not generalizable without duplicating each hub's registry logic into the advisor. **[iter 1]**

### Q2 — detection-routed universal? (thread 6)
No. Detection requires an *environmental* signal; sk-design's 5 judgment modes have only lexical/intent discriminators (color, motion, audit — all request vocabulary, nothing on disk). The archetype is **per-axis**, not per-hub: sk-code is already a hybrid (detects surface from CWD, null-defaults workflow). The four flips have no environmental signal, so null+mode-map is their *terminal* correct state, not a waypoint to detection. **[iter 2]**

### Q3 — second-order anti-patterns of null+mode-map (thread 7)
The canon **already names the disease mechanism**: `hub-identity` vocabulary belongs on the default mode ONLY (template `_routingPrecisionNote`), because a shared class makes all modes co-fire on hub-generic words (the over-emission Run 1 cured). This makes flipping to null the eviction of hub-identity's only scoring home — the point, and the seed of four second-order costs:
- **A1 — name becomes unrouteable:** the skill's own name (`design`, `prompt`) scores no mode → always defers. Bounded friction (one extra turn, a menu not a dead end), but the canon must name it as the *expected* cost so authors don't re-add a default to suppress it.
- **A2 — menu-vs-scorer split (the central refinement):** Run 1's "load `smart_routing.md` + `mode-registry.json`" is correct ONLY with a semantics split. `smart_routing.md` is a safe *scored* resource (`DEFAULT_RESOURCE = []` — emits nothing on zero-signal). `mode-registry.json` must be a **PRESENTED disambiguation-card**, never a scored resource (loading it as a scorer with hub-identity vocab would recreate the bias). Introduce `defaultResourceSemantics: "disambiguation-card"` distinct from `"fallback-only"`.
- **A3 — registry drift becomes user-visible:** under null+mode-map, `mode-registry.json` becomes the user-facing menu; drift is no longer internal. Needs the existing bidirectional-alignment checker run as a **gate**, not a lint.
- **A4 — load a compressed card, not the raw registry:** one line per mode (purpose + 2-3 triggers), derived from the registry. Bounds token cost and removes the temptation to score the menu. **[iter 3]**

### Q4 — edge cases (thread 9)
- **cli is detection-defaulted** (the 1+3 split — see bottom line). Detection can target the *default axis*, not just surface bundling, when an env signal for the default exists.
- **Single-mode threshold:** a one-mode entity is a standalone skill (system-skill-advisor has no `mode-registry.json`), not a hub. Canon threshold for adopting `routerPolicy` should be **≥2 modes** — a one-mode "hub" inherits the maintenance coupling (A3) for nothing.
- **Empty/drifting map:** the disambiguation card must be **derived** from the registry (single source), and the alignment checker must gate. Empty-map = "no modes configured; cannot route," not a silent empty menu. **[iter 4]**

### Q5 — contrarian steelman (thread 12)
The strongest *falsifiable* steelman: a default is justified when self-correcting (env-determined) OR genuinely catch-all. This overturns the cli flip (→ detection-defaulted), concedes the three pure-defer nulls (they satisfy neither condition), and downgrades them to *directional-pending-measurement* because Run 1's no-corpus method leaves the over-deferring tradeoff unmeasured. A categorical "keep all five" steelman is unsupported. The constructive output is the four-shape rule above, each shape with a named falsifier. **[iter 5]**

## Recommendations (encoding deltas vs Run 1)

### A. Per-hub router changes (the 1+3 split)
1. **sk-prompt** — KEEP (agrees with Run 1).
2. **cli-external-orchestration** — **DETECTION-DEFAULTED, not null.** Flip the *static* `defaultMode: null` AND add a runtime-detection block (`defaultResolution: "runtime-detected"`) that resolves the default from the active runtime. Neutralize the static `defaultResource: ["cli-opencode/SKILL.md"]` child-bias. This is the one real divergence from Run 1.
3. **system-deep-loop, mcp-tooling, sk-design** — **FLIP to null + disambiguation-card fallback** (agrees with Run 1's flip, refines the fallback per B below). Treat as *directional-pending-measurement*: ship behind the route-gold gate, watch for over-deferring friction.

### B. Fallback semantics (the menu-vs-scorer split — refines Run 1's "load both")
- `defaultResource` for a pure-defer hub loads a **compressed disambiguation card** derived from `mode-registry.json` (one line/mode: purpose + 2-3 triggers), NOT the raw registry.
- Introduce `defaultResourceSemantics: "disambiguation-card"` (presented to user, never scored) as distinct from `"fallback-only"` (a scored suggestion). Forbid `hub-identity` vocabulary in any resource loaded as a fallback scorer.
- `smart_routing.md` remains a safe *scored* resource; the card is the *presentation* layer. Never conflate.

### C. Canon encoding (four shapes, not three)
- Replace Run 1's single "defer-routed keyword hub" archetype with the **four-shape decision rule** (keyword-with-default / detection-defaulted / detection-routed-axis / pure-defer), each nominated by the testable environmental-vs-intentional discriminant. Archetypes are **per-axis**: declare `routerPolicy` strategy per axis (workflow default vs surface), not one hub-level `defaultMode`.
- Add `defaultResolution: "static" | "runtime-detected"` as a typed option distinct from `defaultMode`.
- Set the canon threshold for adopting `routerPolicy` at **≥2 modes** (single-mode = standalone skill).
- Promote the `hub-router.json ↔ mode-registry.json` bidirectional-alignment check from lint to **gate** (drift is now user-visible).
- Keep Run 1's `defaultApplied → defaultConfigured` rename and the one-zero-signal-fixture-per-hub rule; add a fixture per shape (especially a detection-defaulted fixture for cli asserting the runtime-resolved default).

## Honest scope and confidence

- **What's solid (0.80+):** the per-axis archetype reframe; the canon's hub-identity-on-default mechanism (line-cited); the menu-vs-scorer split; the single-mode threshold; the drift-as-gate upgrade; the epistemic downgrade of the three nulls (direct consequence of Run 1's no-corpus stance).
- **The one contested claim (0.55):** cli = detection-defaulted rather than null. It depends on runtime-detection being robustly implementable across supported runtimes — plausible (the runtime is the process) but **not verified in this run**. Falsifier: if runtime detection is unreliable, cli falls back to pure-defer null and Run 1's original flip stands unchanged.
- **What this run did NOT do:** no live-model routing experiment (thread 3 of the agenda — "does a real model disambiguate cleanly or freeze?"); no migration ordering analysis (thread 4); no full `defaultConfigured` rename ripple (thread 5); no multi-dominant/bundling deep-dive (thread 8). Those remain open for other lineages or a future run.
- **Sibling lineages** (sol-ultra, luna-max, terra-max) converge independently; their findings are not read here. A cross-lineage merge (Run 2 synthesis) is the operator's next step, not this lineage's.
- All router-change recommendations are **proposals** — changing a shipped `defaultMode`/detection block is a live-routing behavior change. Each ships behind the route-gold gate.
