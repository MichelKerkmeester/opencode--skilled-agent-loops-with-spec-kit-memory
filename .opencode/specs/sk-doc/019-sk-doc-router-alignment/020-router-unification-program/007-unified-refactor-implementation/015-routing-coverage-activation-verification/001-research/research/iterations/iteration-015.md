# Iteration 15 — gpt-5.6-sol ultra — focus: activation

The current dependency graph is not cutover-safe. It contains a P2↔P3 cycle, lacks a gate making compiled output machine-authoritative, and allows P4 to pass without closing catalogs, playbooks, durable evidence, or unset-default verification.

Verified claims:

- The live advisor still computes recommendations first and only appends `compiledRoute` as optional metadata; no downstream authority change occurs there. The P4 plan enumerates predicate/directive changes and parity checks but no machine-consumption gate, while the activation packet still describes advisor-hook enforcement as incomplete. `.opencode/skills/system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:357-373`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/plan.md:172-179`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/010-live-activation/spec.md:126`.

- The planned graph is cyclic: the canary depends on a parity harness, P3 waits for P2 canary evidence, and the benchmark parity harness itself depends on P3 eligibility. `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/plan.md:334-340`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/plan.md:349-353`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/014-benchmark-alignment/spec.md:163-169`.

- The benchmark plan always forces `SPECKIT_COMPILED_ROUTING=1`. That proves explicit opt-in, not the P4 contract where an unset variable means compiled and `=0` is the kill-switch. `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/014-benchmark-alignment/spec.md:99-107`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/014-benchmark-alignment/spec.md:130-137`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/decision-record.md:73-75`.

- The accepted eligibility rule says any valid fresh manifest confers eligibility, but four non-hub routers use a different activation archetype and were excluded from the seven-hub rollout. At least `sk-git` and `system-code-graph` also retain classified legacy/compiled semantic differences, so manifest discovery needs an explicit archetype boundary. `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/decision-record.md:182-184`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/010-live-activation/spec.md:74`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/009-non-hub-rollout/001-sk-git/implementation-summary.md:123-125`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/009-non-hub-rollout/002-system-code-graph/implementation-summary.md:96-98`.

- The create-skill contract intentionally generates exact default-off wording and acknowledges that P4 must update its scaffold with all seven directives. The P4 plan names hub directives but not the two generator templates or their contract tests. `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/013-create-skill-alignment/spec.md:88-100`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/013-create-skill-alignment/spec.md:162-168`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/plan.md:174-178`.

- P4 currently requires only route-gold parity, serving status, fallback behavior, scorer hashes, and sibling packets being “available.” It does not require completed catalogs, manual scenarios, durable benchmark reports, or LUNA evidence. `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/plan.md:174-178`; `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/plan.md:247-253`.

Recommended dependency graph:

`P0 stable interfaces + authority → P1 CI/Lane-C enforcement → P2 explicit-on canary + LUNA → P3 consumer adoption/onboarding/coverage closure → P4 cohort cutover`

1. Move the read-only manifest-discovery, archetype discriminator, status probe, and flag parser interfaces into P0. Keep P3 for switching consumers and removing the hardcoded sets. This breaks the benchmark/canary cycle.

2. Add a P0 authority gate naming the machine consumer of compiled decisions. Byte-identical behavior remains mandatory, but the served target must originate from compiled output rather than an unconsumed sibling field.

3. Implement Lane C during P1 against the stable P0 interface. Test three environment modes: explicit `1`, unset-default, and explicit `0`; add invalid-value behavior without touching the frozen scorers.

4. Require P2 to join deterministic Lane C evidence with a compiled-serving LUNA HIGH playbook result. Transport timeout must remain a separate result from routing correctness.

5. Make P3’s exit gate require implemented—and green—create-skill alignment, catalogs, playbooks, durable report pairs, and sk-code alignment. “Packet available” is not sufficient.

6. Make P4 atomic per cohort: runtime predicates, hub directive, create-skill templates/tests, ENV documentation, catalog wording, and durable evidence move together. Explicitly exclude or separately qualify all four non-hub archetypes.
