The latest commit says it checkpoints “pre-existing uncommitted work,” so repository evidence identifies the commit that captured the current link but not the exact command or actor that created it in the working tree. 2. **The alias is intentional, but its current absolute payload does not implement the stated portable intent.** The latest adding commit explicitly lists “root specs -> .opencode/specs symlink,” and `spec/create.sh` calls `.opencode/specs/<track>` canonical while describing root `specs` as the legacy symlink. The stated design is therefore an intentional compatibility alias; the committed developer-specific absolute target is an implementation defect, not evidence that the alias was intended to bind to one machine. [SOURCE: command `git show -s --format=... 2f6f6abcfb97aef79e97d5fa28169c6229dca341`] [SOURCE: .opencode/skills/system-spec-kit/scripts/spec/create.sh:811-819]
3. **The only active maintenance mechanism found is ordinary Git checkout of a tracked mode-`120000` entry; no production repair/recreation path was found.** `git ls-files --stage -- specs` reports mode `120000`, while targeted searches for `ln -s`, `symlinkSync`, and root-specs symlink creation under `.opencode/skills` and `.opencode/scripts` found only test-fixture creation and no `.opencode/scripts` match. Production `spec/create.sh` consumes the alias but does not create it. Thus a removed or malformed link is restored by a checkout/reset of the tracked entry, not by Spec Kit startup or maintenance code. 4. **Fresh clones and symlink-disabled checkout modes are unsafe in two distinct ways.** With normal symlink support, a clone checks out the tracked absolute payload and therefore points back to the original developer path rather than the clone's own `.opencode/specs`; it is broken when that path is absent and can alias an unrelated checkout when it exists. With `core.symlinks=false`, Git documents that symlinks are checked out as small plain files containing link text, so root `specs` ceases to be a directory alias altogether. Repository CI evidence does not establish a supported-platform matrix: the inspected spec-kit workflow checks out only on `ubuntu-latest`, and the symlink integration test treats `EPERM` as an early successful return rather than exercising fallback behavior. [SOURCE: command `git cat-file -p HEAD:specs` => absolute developer path] [SOURCE: https://git-scm.com/docs/git-config#Documentation/git-config.txt-coresymlinks] [SOURCE: .github/workflows/strict-pass-freshness-sweep.yml:11-23] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/tests/folder-discovery-integration.vitest.ts:180-217]
5. **Linked worktrees are specifically unsafe because the absolute payload crosses worktree boundaries.** Git documents that `git worktree add <path>` checks a commit into a new linked working tree. Since the committed `specs` blob names the main checkout's absolute canonical directory, each linked worktree at another path receives a link back to `/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs`, not to that worktree's own `.opencode/specs`. Reads and writes through root `specs` can therefore leak across branches/worktrees even when the link is not broken. [SOURCE: https://git-scm.com/docs/git-worktree] [SOURCE: command `git cat-file -p HEAD:specs` => `/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs`] [INFERENCE: checkout preserves the tracked link payload while only the worktree root changes]
6. **Git tar archives preserve the unsafe symlink and absolute payload.** The read-only pipeline `git archive --format=tar HEAD specs | tar -tvf -` emitted `lrwxrwxrwx ... specs -> /Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs`; Git's archive documentation states that archives contain the named tree's structure. Extracting that tar elsewhere therefore produces the same broken or cross-checkout alias. ZIP/extractor-specific realization was not directly tested, so the defensible conclusion is that repository archives are not universally portable and tar is conclusively unsafe. [SOURCE: command `git archive --format=tar HEAD specs | /usr/bin/tar -tvf -`] [SOURCE: https://git-scm.com/docs/git-archive]
