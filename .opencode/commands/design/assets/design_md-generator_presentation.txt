# Design MD-Generator Presentation Contract

Presentation source of truth for `/design:md-generator`. The router owns mode selection and asset loading. Workflow YAML owns execution. This file owns visible prompts, result displays, and next-step wording.

## 1. Startup Presentation

For `:confirm`, or no suffix with incomplete `$ARGUMENTS`, the consolidated setup prompt is the first visible response. Ask all applicable questions once, then wait.

For `:auto`, or no suffix with complete `$ARGUMENTS`, do not show the consolidated prompt. Resolve setup through the Auto Resolution Table below.

### Auto Resolution Table

| Field | Required | Default | Targeted-ask when |
|-------|----------|---------|--------------------|
| live_url | Yes | none | Always, when missing |
| output | Yes | none | Always, when missing |
| register | No | auto (declared field -> task cue -> surface in focus) | Register is genuinely mixed or unresolved |

### Consolidated Prompt Template

```text
Design MD-Generator setup

Q1. Which live URL should I extract, and where should the DESIGN.md be written?
Q2. Is this Brand or Product register?

Reply format: "<live-url>, <output-dir>, <brand|product>"
```

Never split these questions into separate visible prompts.

### Auto Fail-Fast Display

```text
Design MD-Generator auto setup failed
Missing required input: [live URL and/or output directory]
Provide the missing input or rerun with :confirm.
STATUS=FAIL ERROR="missing required setup inputs"
```

## 2. Results Display

### Success

```text
STATUS=OK PRODUCES="Style Reference DESIGN.md" NEXT=/design:foundations,/design:interface,/design:audit PROOF=sourceUrl,extractedTokensDigest,fidelityScore
```

### Missing Input

```text
Which live URL should I extract, and where should the DESIGN.md be written?
STATUS=ASK MISSING=<input>
```

### Failure

```text
STATUS=FAIL ERROR=<named-cause>
```

### Defer

```text
STATUS=DEFER ROUTE=<hub|sibling>
```

## 3. Next-Step Suggestions

| Condition | Suggested Command | Reason |
|-----------|--------------------|--------|
| Measured CSS should be distilled into a static visual-system plan | `/design:foundations` | Turn measurement into a static system |
| Measured evidence should inform a new or reshaped interface direction | `/design:interface` | Ground a direction in evidence |
| The captured surface needs quality review or hardening based on measured evidence | `/design:audit` | Validate quality before build |
| Extracted evidence informs implementation | sk-code handoff via `.opencode/skills/sk-design/shared/sk-code-handoff.md` | Hand off to build |

This command never silently chains; it emits options only (`HANDOFF_REQUIRED=false`, recommend-only).

---

## 4. Example Usage

```
/design:md-generator https://stripe.com --output design/reference
```

Returns: a Style Reference DESIGN.md and extracted token data under the output directory.
