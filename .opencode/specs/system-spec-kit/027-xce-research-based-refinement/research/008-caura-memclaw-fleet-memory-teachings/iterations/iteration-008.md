## Mechanism

Write flow: `ParallelEmbedEnrich` decides whether embedding/enrichment run inline or defer. Fast mode always defers enrichment, and SaaS-style fast mode can defer embedding; deferred work is explicitly documented as later `EMBED_REQUESTED` / `ENRICH_REQUESTED` events consumed by `core-worker` and PATCHed back onto the row (`core-api/src/core_api/pipeline/steps/write/parallel_embed_enrich.py:7`, `core-api/src/core_api/pipeline/steps/write/parallel_embed_enrich.py:16`, `core-api/src/core_api/pipeline/steps/write/parallel_embed_enrich.py:26`). The hot write persists the memory first; if embedding is absent it stores `metadata.embedding_pending=true` (`core-api/src/core_api/pipeline/steps/write/write_memory_row.py:36`). If fast-mode enrichment is absent it stores `metadata.enrichment_pending=true` (`core-api/src/core_api/pipeline/steps/write/merge_enrichment_fields.py:65`).

Emission: after the row exists, `ScheduleBackgroundTasks` publishes background embed/enrich work through service helpers rather than blocking the request (`core-api/src/core_api/pipeline/steps/write/schedule_background_tasks.py:119`, `core-api/src/core_api/pipeline/steps/write/schedule_background_tasks.py:144`, `core-api/src/core_api/pipeline/steps/write/schedule_background_tasks.py:205`). `_schedule_embed_or_reembed` runs in-process retry in inline/local mode, but in deferred mode publishes `EMBED_REQUESTED` (`core-api/src/core_api/services/memory_service.py:1544`, `core-api/src/core_api/services/memory_service.py:1561`). `_schedule_enrich_or_inline` similarly either runs in-process background enrichment or publishes `ENRICH_REQUESTED` (`core-api/src/core_api/services/memory_service.py:1635`, `core-api/src/core_api/services/memory_service.py:1662`, `core-api/src/core_api/services/memory_service.py:1680`). The publishers validate payload schemas before transport, build an `Event`, and publish fire-and-forget (`common/events/memory_embed_publisher.py:21`, `common/events/memory_embed_publisher.py:30`, `common/events/memory_enrich_publisher.py:27`, `common/events/memory_enrich_publisher.py:50`).

Transport model: the event envelope carries `event_id`, `event_type`, `tenant_id`, `correlation_id`, timestamp, and payload (`common/events/base.py:14`). `publish()` is explicitly fire-and-forget: it returns once accepted by the transport, not once every subscriber has processed it (`common/events/base.py:61`). `subscribe()` states Pub/Sub is at-least-once and handlers must be idempotent, using `event.event_id` as a natural dedup key when needed (`common/events/base.py:68`). The Pub/Sub implementation repeats that handlers must tolerate duplicate events because ack failures and handler exceptions cause redelivery (`common/events/pubsub.py:44`, `common/events/pubsub.py:49`). There is no application outbox in the inspected files; Pub/Sub provisioning, retry policies, and dead-letter topics are external infrastructure (`common/events/pubsub.py:7`).

Delivery and failure behavior: Pub/Sub `publish()` queues to the SDK batch queue and deliberately does not wait on the returned future; shutdown drains in-flight batches, but per-message publish errors do not surface to callers (`common/events/pubsub.py:340`, `common/events/pubsub.py:348`). Subscriber loops pull messages, decode them, dispatch handlers, ack successful batches, and nack failed handler batches via `modify_ack_deadline(..., 0)` (`common/events/pubsub.py:513`, `common/events/pubsub.py:526`, `common/events/pubsub.py:535`, `common/events/pubsub.py:542`, `common/events/pubsub.py:548`). Malformed messages are ack-dropped to avoid infinite poison-message redelivery (`common/events/pubsub.py:621`). If any handler raises, `_dispatch_all` returns failure so Pub/Sub redelivers, and already-successful handlers must tolerate the duplicate (`common/events/pubsub.py:639`, `common/events/pubsub.py:646`, `common/events/pubsub.py:673`). Permanent subscription/config errors halt the pull loop and mark the bus unhealthy (`common/events/pubsub.py:563`, `common/events/pubsub.py:580`). The worker and API register subscribers before `bus.start()` because pull loops are created from the current registry (`core-worker/src/core_worker/app.py:90`, `core-api/src/core_api/consumer.py:217`).

Handler idempotency: embed redelivery re-PATCHes the same embedding and uses content-hash cache to avoid re-calling the provider when possible (`core-worker/src/core_worker/consumer.py:25`, `core-worker/src/core_worker/consumer.py:144`). Enrichment redelivery re-PATCHes fields and treats write-after-write as safe, while protecting agent-provided fields from being overwritten by worker defaults (`core-worker/src/core_worker/clients/storage_client.py:473`, `core-worker/src/core_worker/consumer.py:368`). Lifecycle pipeline actions use an audit row plus a recent-success dedup window for LLM-heavy periodic work (`common/events/lifecycle_handlers.py:127`, `common/events/lifecycle_handlers.py:133`, `common/events/lifecycle_handlers.py:157`, `common/events/lifecycle_handlers.py:314`). Back-channel contradiction detection is guarded by a lock/CAS pattern so both `EMBEDDED` and `ENRICHED` can fire without double-writing contradictions (`core-api/src/core_api/consumer.py:11`, `core-api/src/core_api/services/contradiction_detector.py:186`).

Store/index consistency and reconciliation: MemClaw’s vector “index” is not a separate consumer-owned index in the inspected path; the worker PATCHes `embedding` back onto the canonical memory row and clears `metadata.embedding_pending=false` in the same PATCH (`core-worker/src/core_worker/clients/storage_client.py:168`, `core-worker/src/core_worker/clients/storage_client.py:187`). Enrichment similarly clears `metadata.enrichment_pending=false` on successful worker PATCH (`core-worker/src/core_worker/consumer.py:431`). Until embedding lands, search keeps rows visible through FTS fallback: rows without embeddings bypass vector similarity threshold if `has_embedding=False` (`core-api/src/core_api/pipeline/steps/search/post_filter_results.py:21`). Reconciliation exists as a NULL-embedding scan that republishes the same `EMBED_REQUESTED` events; it is restart-idempotent because successfully patched rows stop matching `embedding IS NULL` (`core-worker/src/core_worker/backfill.py:1`, `core-worker/src/core_worker/backfill.py:22`, `core-worker/src/core_worker/clients/storage_client.py:237`, `core-worker/src/core_worker/clients/storage_client.py:249`). This is the strongest transferable consistency pattern: detect stale/missing derived state from canonical store columns, then feed repair through the same idempotent handler path.

## Negative knowledge

Do not copy distributed Pub/Sub provisioning into Spec Kit Memory. MemClaw assumes externally managed topics/subscriptions, ack deadlines, retry policies, and dead-letter topics (`common/events/pubsub.py:7`). A local SQLite system should prefer an in-process queue or durable job table with retry counters and a reconciliation scan.

Do not copy fire-and-forget publisher error blindness wholesale. MemClaw intentionally does not wait for Pub/Sub publish futures, accepting that per-message publish errors only land in SDK logs (`common/events/pubsub.py:340`, `common/events/pubsub.py:348`). In a local store, publish/enqueue should usually be in the same SQLite transaction as the row insert, or at least recorded durably before returning, because there is no managed broker to absorb the gap.

Do not copy multi-tenant worker-pool complexity unless Spec Kit gains real multi-user daemon deployment. MemClaw’s per-tenant scoping, tenant config payloads, per-tenant storage slots, service-account tokens, and subscription prefixes solve fleet isolation problems (`common/events/factory.py:40`, `core-worker/src/core_worker/consumer.py:544`, `core-worker/src/core_worker/backfill.py:69`). Spec Kit should keep the pattern but collapse infrastructure to row/job identity and local locks.

Do not copy back-channel event fanout before subscribers exist. MemClaw’s `EMBEDDED`/`ENRICHED` events exist because contradiction detection needs a post-PATCH trigger and can be idempotently invoked from either ordering (`core-api/src/core_api/consumer.py:11`). Spec Kit should first identify concrete local subscribers; otherwise, status columns plus reconciliation are enough.

Do not treat lifecycle audit/dedup windows as a generic local requirement. MemClaw uses audit rows and 23-hour dedup windows for fleet cron actions (`common/events/lifecycle_handlers.py:100`, `common/events/lifecycle_handlers.py:133`). Spec Kit may need job attempt history, but not fleet-cron dedup semantics unless it adds scheduled global lifecycle jobs.
