# Iteration 14: W5 adversarial: trigger, MVCC, CDC family

## Focus
Adversarial re-verification of eight CHANGED/REFUTED verdicts about Turso v0.7.0-pre.6 covering the trigger family (default availability, RAISE(ABORT), INSTEAD OF rejection, changes() reliability), MVCC interactions (AUTOINCREMENT via sequences, custom index-module rejection, Tantivy FTS exclusivity), the experimental multi-process WAL sidecar, the CDC pragma surface, Busy/BusySnapshot error semantics with busy_timeout, and the window-function ceiling. Evidence was independently re-located in core translation code, VDBE execution, MVCC tests, sqltest suites, fuzz tests, and JS/Rust bindings — deliberately not relying on the earlier iterations' COMPAT.md citations, though COMPAT.md was cross-checked where it conflicts with tests.

## Findings
- **KNOWN — CONFIRMED — CLAIM-1 (CREATE/DROP TRIGGER default-on incl. RAISE(ABORT)):** The CREATE TRIGGER translation path has no experimental gate — its only rejections are system tables, virtual tables, duplicates, and INSTEAD OF [SOURCE: core/translate/trigger.rs:143-173]. DROP TRIGGER is fully wired: `translate_drop_trigger` [SOURCE: core/translate/trigger.rs:470,586] dispatched from [SOURCE: core/translate/mod.rs:300-303]. No `experimental_triggers`/`enable_triggers` symbol exists anywhere in core (exhaustive grep, zero hits), and the JS bindings' experimental-feature match block has **no** `"triggers"` arm — the `'triggers'` entry in [SOURCE: bindings/javascript/packages/common/types.ts:1] falls through to the no-op `_ => core_opts` arm [SOURCE: bindings/javascript/src/lib.rs:275-288], i.e. it is a stale type-level leftover, not a gate. RAISE(ABORT): translated at [SOURCE: core/translate/expr/translator.rs:2914], positively tested in `trigger-raise-abort-before-insert` / `trigger-raise-abort-after-insert` [SOURCE: testing/sqltests/tests/trigger.sqltest:1429-1454], and fuzz-tested across all four RAISE types (IGNORE/ABORT/FAIL/ROLLBACK) [SOURCE: tests/fuzz/raise.rs:17-39].
- **PARTIAL — NUANCED — CLAIM-2 (INSTEAD OF fails; changes() partial):** The INSTEAD OF half is CONFIRMED: an unconditional `bail_parse_error!("INSTEAD OF triggers are not supported yet")` in the CREATE TRIGGER path [SOURCE: core/translate/trigger.rs:168-173] — every INSTEAD OF trigger is rejected at creation, so view triggers cannot exist at all. The changes() half is NUANCED: COMPAT.md still marks `changes()`/`total_changes()` "🚧 Partial — Still need to support update statements and triggers" [SOURCE: COMPAT.md:269,316], but the live test suite asserts **correct** counts for multi-row INSERT, UPDATE (including the index-write exclusion case, changes()=6, and rowid-moving UPDATE, changes()=1), and DELETE, all under `@cross-check-integrity` [SOURCE: testing/sqltests/tests/changes.sqltest:34-80]. So "wrong for UPDATE statements" is stale; the only documented residual gap is trigger-related counting plus the missing `sqlite3_total_changes64` C-API [SOURCE: COMPAT.md:599]. Characterization: counters are reliable for plain INSERT/UPDATE/DELETE; treat them as suspect only where trigger sub-statements are involved.
- **KNOWN — CONFIRMED — CLAIM-3 (AUTOINCREMENT works under MVCC via atomic sequences):** `test_autoincrement_works_in_mvcc` creates an AUTOINCREMENT table and inserts successfully in MVCC mode [SOURCE: core/mvcc/database/tests.rs:10415-10436]; the line above it ("not supported … due to sqlite_sequence") is a stale doc comment immediately corrected by "AUTOINCREMENT is supported in MVCC mode via atomic sequences" [SOURCE: core/mvcc/database/tests.rs:10412-10413]. Pre-existing WAL-mode tables also work via sequences [SOURCE: core/mvcc/database/tests.rs:10438-10449]. The mechanism is real: a hidden per-table sequence `__turso_internal_autoincrement_<table>` [SOURCE: core/schema.rs:184-189] and an explicit "AUTOINCREMENT MVCC rowid-generation path" [SOURCE: core/translate/sequence.rs:167], plus MVCC rowid-update semantics tests [SOURCE: core/mvcc/database/tests.rs:6515-6524].
- **KNOWN — CONFIRMED — CLAIM-4 (custom index modules / Tantivy FTS excluded under MVCC):** The rejection site is `if connection.mvcc_enabled() && using.is_some() { bail_parse_error!("Custom index modules are not supported in MVCC mode") }` in CREATE INDEX validation [SOURCE: core/translate/index.rs:65-67]. FTS is exactly such a module (`FTS_INDEX_METHOD_NAME = "fts"` [SOURCE: core/index_method/fts.rs:42]) and is Tantivy-backed (`use tantivy::…` [SOURCE: core/index_method/fts.rs:26-37]; `fts = ["dep:tantivy"]`, tantivy 0.26.0 [SOURCE: core/Cargo.toml:43,71]). Added nuance: index methods are additionally gated behind `--experimental-index-method` even outside MVCC [SOURCE: core/translate/index.rs:58-63].
- **KNOWN — CONFIRMED — CLAIM-5 (opt-in multi-process WAL via .tshm sidecar):** `DatabaseOpts.enable_multiprocess_wal` with builder `with_multiprocess_wal()` [SOURCE: core/lib.rs:222,279-280], default off in both core and bindings [SOURCE: bindings/rust/src/lib.rs:168; core/incremental/cursor.rs:399]. The sidecar is real: "Cross-process shared WAL coordination backed by the `.tshm` file", magic `TSHMWAL\0` [SOURCE: core/storage/shared_wal_coordination.rs:1,38-39], path formed as `{db_path}-tshm`/`{wal_path}-tshm` [SOURCE: core/storage/wal.rs:116-118]. Exposed as experimental: JS feature string `"multiprocess_wal"` [SOURCE: bindings/javascript/src/lib.rs:285], Rust builder `experimental_multiprocess_wal()` [SOURCE: bindings/rust/src/lib.rs:226], and `experimental_multiprocess_wal_enabled()` [SOURCE: core/connection.rs:2471]. A dedicated multi-process test module exists, including an AUTOINCREMENT burst child-process test [SOURCE: core/multiprocess_tests.rs:122-131,2273-2280].
- **KNOWN — CONFIRMED — CLAIM-6 (CDC de-unstabled: capture_data_changes_conn):** Stable pragma name `capture_data_changes_conn` is the canonical constant [SOURCE: sync/engine/src/database_tape.rs:33], with the CHANGELOG explicitly recording "add stable PRAGMA capture_data_changes_conn name and keep old one around for compatibility" [SOURCE: CHANGELOG.md:509]; `unstable_capture_data_changes_conn` survives only as an alias [SOURCE: COMPAT.md:224-225]. Modes parse as exactly `off|id|before|after|full` with that list in the error message [SOURCE: core/lib.rs:2812-2823]; default is off (`Option<CaptureDataChangesInfo>`, "off mode is None" [SOURCE: core/vdbe/mod.rs:739; core/lib.rs:1989]). Per-connection: stored on `Connection` [SOURCE: core/connection.rs:324,2219-2224]. Default capture table `turso_cdc` plus companion `turso_cdc_version` [SOURCE: core/translate/pragma.rs:1772-1773], schema maintained by a dedicated insn [SOURCE: core/vdbe/insn.rs:1914-1921]. Integration tests exercise `'full'`, `'id'`, `'before'` [SOURCE: tests/integration/functions/test_cdc.rs:75,107,152].
- **KNOWN — CONFIRMED — CLAIM-7 (Busy/BusySnapshot, busy_timeout, no SQLITE_BUSY in JS):** `BusySnapshot` is a distinct error variant [SOURCE: core/error.rs:71], deliberately returned instead of `Busy` where "Retrying with busy_timeout will NEVER HELP" [SOURCE: core/storage/wal.rs:3174-3183]. busy_timeout exists twice over: `PRAGMA busy_timeout` [SOURCE: core/pragma.rs:146; core/translate/pragma.rs:344-350] and `Connection::set_busy_timeout` [SOURCE: core/connection.rs:3923]. The JS binding wires an open-option `timeout` (ms) into `set_busy_timeout` [SOURCE: bindings/javascript/src/lib.rs:255,267-268,330-332] and its tests use `PRAGMA busy_timeout=100` [SOURCE: bindings/javascript/sync/packages/native/promise.test.ts:240]. On the JS error surface, `StepResult::Busy` maps to a generic "database is locked" error and no `SQLITE_BUSY` symbol exists anywhere under bindings/javascript/src [SOURCE: bindings/javascript/src/lib.rs:212] — consistent with "no SQLITE_BUSY code in the JS surface".
- **KNOWN — CONFIRMED — CLAIM-8 (window functions = row_number() + aggregates OVER only):** Although a fuller `WindowFunc` enum is declared (Rank/DenseRank/PercentRank/CumeDist/Ntile/Lag/Lead/FirstValue/LastValue/NthValue) [SOURCE: core/function.rs:470-495], the name resolver constructs only `Self::Window(WindowFunc::RowNumber)` [SOURCE: core/function.rs:1580] and the VDBE implements only `WindowFunc::RowNumber` accumulator arms — no Rank/Lag/Lead execution exists [SOURCE: core/vdbe/execute.rs:6396,6428]. The planner comment confirms "…functions such as ROW_NUMBER() are supported" [SOURCE: core/translate/plan.rs:3104] while aggregate-over-window flows through `AggValue` [SOURCE: core/translate/window.rs:528]. Extra restriction beyond the claim: ORDER BY inside a window function's argument list is asserted unsupported [SOURCE: core/translate/window.rs:428-431], and `VALUES` clauses with windows are rejected [SOURCE: core/translate/window.rs:109]. The enum variants are dormant plumbing, not capability.

## Ruled Out
- Triggers being behind an experimental opt-in: the JS `'triggers'` ExperimentalFeature type entry is dead code (no match arm in the feature dispatch; no corresponding core flag exists).
- The MVCC AUTOINCREMENT rejection still being live: the "not supported" text at core/mvcc/database/tests.rs:10412 is a stale leftover comment line sitting directly above the corrected doc and a passing positive test.
- "changes() wrong for UPDATE statements" (per COMPAT.md): contradicted by passing, integrity-cross-checked UPDATE tests in changes.sqltest; the residual caveat narrows to trigger interactions.

## Dead Ends
- Compound shell commands (multi-`rg` batches with `;` and pipes) were rejected by the sandbox permission layer, burning the first six tool calls with zero evidence; all searches had to be re-issued as single `rg` invocations.
- The broad case-insensitive `instead` grep over core returned ~120 prose-comment hits with only 2 relevant lines (trigger.rs) — a word-boundary `INSTEAD OF` pattern would have been far cheaper.

## Reflection
- What worked: Re-locating every claim in at least two independent surfaces (core translation/VDBE + tests or bindings) caught both a stale COMPAT.md row (changes()/UPDATE) and a stale test comment (MVCC AUTOINCREMENT) that naive citation-trusting would have propagated; the enum-vs-execution split in function.rs vs vdbe/execute.rs was the decisive adversarial check for the window-function ceiling.
- What failed: The sandbox rejected all batched compound shell commands, costing six wasted calls and pushing total tool usage (~22 effective) past the 12-call target; one over-broad grep (`instead`) produced mostly noise.
- Confidence: high — all eight verdicts rest on direct rejection sites, parse functions, or passing positive tests in the vendored source, with file:line citations independent of the earlier iterations' evidence; the only medium-confidence element is the exact trigger-related failure mode of changes(), which is documented (COMPAT.md) but not pinned to a failing test.

## Recommended Next Focus
Pin down the precise trigger-interaction failure mode of changes()/total_changes() (e.g., whether trigger sub-statement writes are counted into the outer statement's count, against SQLite semantics) with a targeted look at trigger_exec.rs's n_change handling and any skipped/expected-fail tests.
