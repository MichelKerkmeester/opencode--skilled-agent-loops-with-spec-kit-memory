# Iteration 10 — synthesis: ranked optimization plan + measurable impact estimates + trade-offs

## Summary

The strongest path for the new 246 MB Ollama/Jina baseline is not to constrain embedding models; it is to shrink always-resident search/index state, bound SQLite residency, and keep local model runtimes behind unloadable or process-isolated boundaries. Ranked by near-term RSS return, the plan is: stop eagerly retaining the JS BM25 index, apply explicit SQLite memory profiles, gate startup/background work, bound profile-aware caches, then tackle vector-shard and sidecar architecture for future model swaps.

## Findings

### Finding 1: Rank 1 — make SQLite FTS5 the default BM25 engine and stop warming the JS BM25 index
- Evidence: `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1706-1711` schedules an in-memory BM25 rebuild at startup whenever `bm25Index.isBm25Enabled()` is true; `.opencode/skills/system-spec-kit/mcp_server/lib/search/bm25-index.ts:100-104` enables that path by default; `.opencode/skills/system-spec-kit/mcp_server/lib/search/bm25-index.ts:262-263` retains per-document `tokens: string[]` plus `Map<string, number>` term frequencies; `.opencode/skills/system-spec-kit/mcp_server/lib/search/sqlite-fts.ts:162-203` already provides weighted FTS5 BM25; `.opencode/skills/system-spec-kit/mcp_server/lib/search/hybrid-search.ts:465-469` already delegates the FTS lane to that SQLite BM25 path.
- Memory impact: Iteration 8's read-only rebuild probe measured current BM25 at roughly +116 MB heapUsed and +278 MB RSS in a fresh process; a packed term-id prototype reduced it to about 12 MB heapUsed plus 6 MB arrayBuffers. For PID 4791's 246 MB baseline, the expected steady win is 80-110 MB if BM25 warmup completes in the live environment, or near 0 MB if `ENABLE_BM25=false` or warmup has not run.
- Proposed change: in `.opencode/skills/system-spec-kit/mcp_server/lib/search/bm25-index.ts`, add `SPECKIT_BM25_ENGINE=sqlite|packed-inmemory|legacy-inmemory|auto`, default `auto`. In `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1706-1711`, replace unconditional `bm25.rebuildFromDatabase(database)` with `if (bm25Index.shouldWarmInMemoryBm25(database)) ...`, where `auto` returns false when FTS5 is available. In `.opencode/skills/system-spec-kit/mcp_server/lib/search/hybrid-search.ts:1251-1257`, keep the `bm25` source label but route to SQLite FTS5 BM25 when the in-memory index is cold; implement packed term-id storage before allowing `packed-inmemory` as a warm resident engine.
- Trade-off: rankings can shift because SQLite FTS5 tokenization/weights are not identical to the current JS stemmer/synonym path. This needs golden-query comparison, but it preserves lexical search, hybrid search, vector search, and every embedder choice.
- Effort: M

### Finding 2: Rank 2 — apply a connection-level SQLite memory profile before broader storage changes
- Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:800-806` hard-codes `cache_size=-64000`, `mmap_size=268435456`, and `temp_store=MEMORY` on every opened vector DB connection; `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:376-380` retains connections in a process-level map keyed by path. A read-only active DB probe during this iteration measured the Ollama/Jina store at 139 MB with table hot spots: `memory_lineage` 41.3 MB, `memory_index` 33.2 MB, `embedding_cache` 19.0 MB, `vec_memories_vector_chunks00` 16.8 MB, and `memory_fts_data` 11.3 MB.
- Memory impact: Iteration 4 measured balanced pragmas (`cache_size=-16384`, `mmap_size=67108864`, `temp_store=DEFAULT`) saving about 30-35 MB RSS on the active store, and low-memory pragmas (`cache_size=-8192`, `mmap_size=33554432`) saving about 50-55 MB. The saving compounds if multiple profile/custom-path connections remain open.
- Proposed change: replace the pragma block in `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:800-806` with `applySqliteRuntimePragmas(new_db, target_path)`. Use env-backed profiles: `SPECKIT_SQLITE_MEMORY_PROFILE=balanced|low-memory|throughput`, default `balanced`; balanced = `SPECKIT_SQLITE_CACHE_KIB=16384`, `SPECKIT_SQLITE_MMAP_BYTES=67108864`, `SPECKIT_SQLITE_TEMP_STORE=DEFAULT`, `SPECKIT_SQLITE_WAL_AUTOCHECKPOINT_PAGES=512`; low-memory = `8192`, `33554432`, `DEFAULT`, `512`; throughput keeps the current values. Log effective pragmas plus `page_size`, `page_count`, and `freelist_count` once per opened connection.
- Trade-off: lower mmap/cache budgets make repeated large scans colder. Iteration 4's synthetic probe saw balanced about 30% slower and low-memory about 65-70% slower, so default to balanced and make low-memory explicit for constrained sessions.
- Effort: S

### Finding 3: Rank 3 — gate startup and background work so idle startup stays small
- Evidence: `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1494-1496` opens the DB at startup; `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1597-1684` immediately runs integrity checks and initializes search/storage consumers; `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1738-1746` resumes embedder reindex jobs; `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1748-1756` starts the retry manager; `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1899-1902` schedules a background scan; `startupScan()` calls `indexSingleFile(filePath, false)` at `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1292-1295`, which can generate embeddings synchronously for changed files.
- Memory impact: this is a timing win, not a feature removal. Deferring DB open plus BM25/retry/scan can keep true idle startup below the SQLite/BM25 resident cost, plausibly avoiding 40-130 MB before the first memory-owning tool call. It also prevents accidental local-provider model load during dirty startup scans or resumed reindex work.
- Proposed change: split `main()` into a thin MCP bootstrap and `ensureMemoryRuntimeInitialized(reason)`. Leave server construction, tool schemas, validation, runtime detection, and signal handlers in startup; move `vectorIndex.initializeDb()`, integrity/dimension validation, consumer `init()` calls, BM25 warmup, `resumeReindexJobs()`, retry start, and `startupScan()` behind the guard. Call the guard only from memory-owning tools and mutation paths. Add `tryGetDb()` to `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts` so health/session priming can report "database not opened yet" without forcing initialization.
- Trade-off: the first memory call pays DB initialization and possible schema validation latency. Startup health becomes less complete unless it reads a lightweight summary file, but all features remain available on first use.
- Effort: L

### Finding 4: Rank 4 — make persistent embedding caches byte-, profile-, and task-bounded
- Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/cache/embedding-cache.ts:101-149` caps `embedding_cache` at 10,000 rows globally, not bytes or profile; `.opencode/skills/system-spec-kit/mcp_server/lib/cache/embedding-cache.ts:153-167` defines TTL eviction but production paths do not maintain it; `.opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:114-130` computes a cache key from normalized content while ignoring the model argument; `.opencode/skills/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts:202-208` writes cache rows after quality gate. This iteration's active DB probe measured 4,127 cache rows, 16.9 MB raw embeddings, and 19.0 MB table pages; the historical llama-cpp profile has 10,000 rows and 30.7 MB raw embeddings.
- Memory impact: immediate active-baseline saving is likely 5-20 MB after eviction plus `PRAGMA shrink_memory`, depending on the chosen byte cap and hit rate. Prevention value is higher: 10,000 rows at 2048 dims are about 78 MB raw before SQLite overhead, so a Voyage/OpenAI profile can otherwise become a large hot-DB surface.
- Proposed change: in `.opencode/skills/system-spec-kit/mcp_server/lib/cache/embedding-cache.ts`, add columns `profile_key TEXT NOT NULL DEFAULT ''` and `input_kind TEXT NOT NULL DEFAULT 'document'`, then migrate the primary key from `(content_hash, model_id, dimensions)` toward `(content_hash, profile_key, input_kind)`. Replace `MAX_CACHE_ENTRIES` with `SPECKIT_EMBED_CACHE_MAX_BYTES`, `SPECKIT_EMBED_CACHE_PROFILE_MAX_BYTES`, `SPECKIT_QUERY_EMBED_CACHE_MAX_BYTES`, and secondary entry caps. Run byte-budget maintenance after `storeEmbedding()`, call `db.pragma('shrink_memory')` when rows are deleted, and wire query embedding cache around `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-queries.ts:629-657` only after `input_kind='query'` is available.
- Trade-off: older cache entries are recomputed sooner; for Voyage/OpenAI that can mean API cost, so cloud defaults should be disk-generous but still bounded. Schema migration and call-site updates are the cost of keeping task-specific embeddings correct.
- Effort: M

### Finding 5: Rank 5 — protect pluggability by isolating local model runtimes, not by narrowing model choices
- Evidence: Ollama is already out of process: `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/adapters/ollama.ts:242-275` and `.opencode/skills/system-spec-kit/shared/embeddings/providers/ollama.ts:308-325` call HTTP endpoints. The remaining in-process local provider, `HfLocalProvider`, stores `extractor` and `loadingPromise` on the provider instance at `.opencode/skills/system-spec-kit/shared/embeddings/providers/hf-local.ts:200-221` and imports `@huggingface/transformers` inside `getModel()` at `.opencode/skills/system-spec-kit/shared/embeddings/providers/hf-local.ts:224-267`. The registry already models future backends but throws for `api` and `sentence-transformers` at `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/registry.ts:125-130`; backend kinds explicitly include `sentence-transformers` at `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/types.ts:15-19`.
- Memory impact: immediate active Ollama/Jina PID saving is near 0 MB because no model is loaded in-process. Conditional impact is the largest safety net: hf-local is documented as a ~274 MB model class, and local GGUF options on this machine are 313 MB Q8, 584 MB BF16, and 1.1 GB F32 before runtime overhead. Sidecar plus idle kill keeps PID 4791 small immediately and can recover hundreds of MB whole-machine RSS after local embedding idle.
- Proposed change: create `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/sidecar-client.ts` implementing `EmbedderAdapter` over JSONL stdio and `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/sidecar-worker.ts` for concrete local backends. Add `SPECKIT_EMBEDDER_EXECUTION=auto|direct|sidecar`, default `direct` for Ollama/Voyage/OpenAI and `sidecar` for hf-local, future sentence-transformers, and any reintroduced llama-cpp. Route both `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-queries.ts:629-657` and reindex jobs at `.opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:252-281` through a shared execution client; keep DB writes in the MCP process.
- Trade-off: first local embedding after idle pays process spawn plus model load, and logs/debugging span one more process. Whole-machine RSS only drops after idle eviction; while the worker is active, the memory still exists outside the MCP PID.
- Effort: L

### Finding 6: Rank 6 — split full per-profile SQLite stores into canonical metadata plus vector/cache shards
- Evidence: `.opencode/skills/system-spec-kit/shared/embeddings/profile.ts:75-77` currently derives a full `context-index__<profile>.sqlite` for each embedding profile; `.opencode/skills/system-spec-kit/shared/embeddings/profile.ts:302-307` returns that profile DB as the active DB path. Vector search joins active vector tables directly from the same DB at `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-queries.ts:81-99` and `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-queries.ts:301-317`; default vector writes target `vec_memories` at `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:281-286`. Current files show the duplication cost: active Ollama/Jina profile DB is 139 MB, historical llama-cpp profile DB is 716 MB.
- Memory impact: active-profile immediate RSS saving is below the pragma/BM25 levers, roughly the hot vector/cache portion that can stop living in the canonical DB: about 35-40 MB in the active store (`embedding_cache` + vector chunk pages before indexes). The larger value is preventing old profile metadata/lineage/checkpoint copies from being opened, mmap'd, or maintained during model experiments.
- Proposed change: change profile pathing so the canonical metadata DB is stable, for example `context-index.sqlite`, and add `EmbeddingProfile.getVectorShardPath(baseDir): vectors/context-vectors__${slug}.sqlite`. Add `attachActiveVectorShard(database, profile)` in `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts` after open, with low-memory pragmas for the shard. Move `vec_<dim>`, `vec_memories*`, and profile-specific `embedding_cache` into the shard; keep `memory_index`, lineage, FTS, projection, governance, checkpoints, and session state in the canonical DB. Update `activeVectorSource()` and mutation/reindex writes to qualify attached shard tables.
- Trade-off: this is a real migration: backup/restore, checkpoint compatibility, WAL behavior, integrity checks, and `checkDatabaseUpdated()` all need tests. It is worth doing after the small RSS levers land, not before.
- Effort: L

### Finding 7: Rank 7 — add byte-aware health and cache telemetry before using V8 old-space caps
- Evidence: startup logs only RSS at `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1895-1896`; `memory_health` exposes only PID, RSS, and uptime at `.opencode/skills/system-spec-kit/mcp_server/handlers/memory-crud-health.ts:315-319`; the launcher spawns `node dist/context-server.js` without child-specific V8 flags at `.opencode/bin/mk-spec-memory-launcher.cjs:296-302`. Tool cache is count-bounded at `.opencode/skills/system-spec-kit/mcp_server/lib/cache/tool-cache.ts:56-60` and reports counts, not bytes, at `.opencode/skills/system-spec-kit/mcp_server/lib/cache/tool-cache.ts:459-471`; trigger matcher retains regexes on entries at `.opencode/skills/system-spec-kit/mcp_server/lib/parsing/trigger-matcher.ts:23-33` and `.opencode/skills/system-spec-kit/mcp_server/lib/parsing/trigger-matcher.ts:532-535`.
- Memory impact: telemetry itself saves 0 MB, but it prevents wrong fixes. The likely follow-on wins are small-to-medium: trigger regex retention is probably several MB to tens of MB under the active 12k-trigger store, tool cache can retain content-heavy payloads, and in-process embeddings are single-digit MB raw.
- Proposed change: add `.opencode/skills/system-spec-kit/mcp_server/lib/telemetry/heap-profiler.ts` with heap/RSS marks from `process.memoryUsage()` and `node:v8.getHeapStatistics()`, plus opt-in `v8.writeHeapSnapshot()` via `SPECKIT_HEAP_SNAPSHOT_DIR` and signal. Extend `memory_health({ reportMode: "full" })` with `heap_mb`, `external_mb`, `array_buffers_mb`, `toolCache.approxBytes`, `triggerMatcher.regexEntryCount`, and embedding-cache byte estimates. Add `SPECKIT_CONTEXT_SERVER_MAX_OLD_SPACE_MB` to `.opencode/bin/mk-spec-memory-launcher.cjs:296-302`, but leave it unset until BM25/caches are reduced; use 192-256 MB as a canary range, not a default.
- Trade-off: heap snapshots pause the server and can contain sensitive memory contents, so snapshots must be disabled by default and written to a 0700 diagnostics directory. Old-space caps can turn leaks into crashes if applied too early.
- Effort: S-M

## Cross-references

- Builds on Iteration 3 by keeping embedding-cache byte/profile caps, but ranks them below BM25 and SQLite because the active cache is ~19 MB table pages, not a 100 MB resident root.
- Builds on Iteration 4 by preserving the balanced/low-memory SQLite pragma plan as the fastest low-risk implementation.
- Builds on Iteration 5 and Iteration 9 by treating llama-cpp/hf-local as conditional local-runtime risks; the active Ollama path should stay direct because it is already out of process.
- Builds on Iteration 6 by adding task-aware query/document cache keying as a prerequisite for safe Voyage/OpenAI disk query caching.
- Builds on Iteration 7 by keeping code graph outside this ranked PID 4791 plan; only passive RPC batching/marker gating belongs here.
- Builds on Iteration 8 by making BM25 the top immediate target and by placing V8 heap caps after representation/caching fixes.
- Confirms Iterations 1-2 are historical for the old 1080 MB baseline: node-llama-cpp unload remains important, but it is not the current 246 MB active-resident root.

## Negative knowledge (ruled-out)

- Restricting embedding models is the wrong control surface. Profile-aware caches, sidecars, and runtime memory profiles preserve Voyage, llama-cpp, Ollama, BGE, mxbai, and future adapters.
- Replacing SQLite wholesale with LMDB/RocksDB is not the next move. `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-queries.ts:301-317` relies on SQL joins and ranking filters, and package dependencies are currently `better-sqlite3` plus `sqlite-vec` at `.opencode/skills/system-spec-kit/mcp_server/package.json:52-60`.
- Code graph is not an in-process mirror to shrink. `.opencode/skills/system-spec-kit/mcp_server/lib/code-graph-boundary.ts:4-5` documents a process boundary; marker reads are local and request-time graph reads cross MCP at `.opencode/skills/system-spec-kit/mcp_server/lib/code-graph-boundary.ts:272-305`.
- WAL and `page_size` are not primary RSS levers. WAL files are single-digit MB in the active store, and `page_size` is an offline rebuild choice rather than a safe runtime fix.
- The in-process embedding LRU is not a leading root. `.opencode/skills/system-spec-kit/shared/embeddings.ts:106-107` caps it at 1,000 vectors, which is single-digit MB raw at 1024 dims; fix it for correctness/profile hygiene, not first-order RSS.
- `ps -p 4791` remains unavailable in this sandbox (`operation not permitted`), so live PID deltas need launcher telemetry or an unsandboxed probe.

- Is BM25 actually warm in the live 246 MB PID, or is the baseline measured before the scheduled warmup completes? A heap mark after `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1706-1711` would settle the priority.
