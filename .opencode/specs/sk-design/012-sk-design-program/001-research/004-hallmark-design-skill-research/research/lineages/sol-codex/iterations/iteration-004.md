# Iteration 4: Motion, Microinteraction, and State Contracts

## Focus

This iteration compared Hallmark's `motion.md`, `microinteractions.md`, and `interaction-and-states.md` with sk-design's MOTION mode, shared motion vocabulary, timing laws, pattern cards, interaction-state procedure, performance guidance, and Brand-vs-Product register. The comparison covers duration tiers, easing vocabulary, distance and transform rules, microinteraction patterns, interaction states, reduced motion, and theme coupling.

The narrow interpretation is file-level improvements to the shipped MOTION system. A broader redesign/variant command analysis remains deferred because it is outside this iteration's motion focus and is already a blocked parity-only direction in strategy.

## Findings

## Ruled Out

- Copying Hallmark's named-theme multiplier table into sk-design: it would turn a useful semantic concept into preset coupling and would require MIT notice retention for close reuse.
- Replacing sk-design's rendering-cost hierarchy with a universal transform/opacity-only ban: Hallmark's own recipes contradict that absolute rule, while sk-design already handles bounded paint and layout-like transitions more accurately.
- Adopting Hallmark's global `150ms` reduced-motion override: it conflicts with Hallmark's own `0.01s` path and is weaker than sk-design's instant/static semantic-parity contract.
- Requiring all eight Hallmark states on every element: states must be applicable; explicit N/A is more accurate than synthetic success/error states on controls that cannot enter them.
- Adding a new public motion or redesign command: `/interface:motion` and the existing redesign lane already own those jobs.

## Dead Ends

- Exact named-theme values are not a portable source of truth for sk-design's extracted corpus. The transferable concept is a semantic motion-character record, not Hallmark's ten theme rows.
- Universal distance values are not justified across Brand and Product surfaces. They remain examples until target evidence or a token system assigns them.

## Edge Cases

- Ambiguous input: "per-theme timings" could mean importing Hallmark's presets or adding theme-aware capability. This iteration selected the narrower capability interpretation because sk-design's register and extracted corpus are not Hallmark theme identities.
- Contradictory evidence: Hallmark says to animate only transform/opacity but later recommends grid-row, color, border, shadow, blur, and value animation; it also alternates between `150ms` and `0.01s` reduced-motion defaults. Both claims are preserved above, and sk-design's cost-tier plus semantic-parity model is better supported.
- Missing dependencies: none. All target files were available locally.
- Partial success: none. This iteration answers the motion/theme portion of the open extraction/motion/theme question; the broader curated-theme and roadmap analysis remains open.
