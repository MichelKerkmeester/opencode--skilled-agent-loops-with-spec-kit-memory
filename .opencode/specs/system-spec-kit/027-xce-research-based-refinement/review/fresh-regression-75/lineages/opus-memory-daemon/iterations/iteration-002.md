## Review: opus-memory-daemon seat — consolidation correctness & concurrency

**Scope traced:** `consolidation.ts` (full), its real dependencies in `causal-edges.ts` (`updateEdge`, `getStaleEdges`, `countEdgesForNode`, module-global `db`), the caller `handlers/save/response-builder.ts:852`, and the write-safety/n3lite tests. `better-sqlite3` runs `database.transaction(fn)()` as a SAVEPOINT when a transaction is already open (the manual `BEGIN IMMEDIATE` at `consolidation.ts:489`), rolling back to the savepoint and re-throwing on error — that re-throw is what the partial-failure findings hinge on.

**What's solid (not findings):** The cadence gate is genuinely race-safe — `BEGIN IMMEDIATE` takes a RESERVED lock before the `last_run_at` read, so two processes can't both pass the due-check (no double-run window). Strengthen vs. decay query windows are disjoint (≤7d vs. ≥30d), so no edge is double-processed. The auto-edge strength cap in the hebbian path matches `insertEdge` via the shared `isAutoEdgeCreator`. These are correct.

**Defects found** (all P2 — real correctness/concurrency issues, but the feature is behind the default-off `SPECKIT_CONSOLIDATION` flag and runs as a weekly batch, so none is a blocker):

1. **Counters ignore `updateEdge`'s return value** — `strengthened++`/`decayed++` fire unconditionally even when `updateEdge` returns `false`, so `ConsolidationResult.hebbian` over-reports actual mutations.

2. **Swallowed Hebbian failure is recorded as a successful cycle** — `runHebbianCycle`'s own try/catch eats the re-thrown savepoint error; the outer transaction then commits `last_run_at = now`, suppressing retries for 7 days even though all strengthen/decay writes were rolled back.

3. **Read-only O(n²) scan held inside the write lock** — the up-to-5s contradiction scan and N+1 bounds count run inside `BEGIN IMMEDIATE`, blocking every other writer (including the epic's new CLI front-door processes) for the cycle's duration. It's a tradeoff against the double-run guard, but the *read-only* scan does not need the write lock.

4. **Threaded `database` param diverges from the module-global used for writes** — reads use the threaded handle; `updateEdge`/`countEdgesForNode`/`getStaleEdges` (and `detectStaleEdges(_database)`, which ignores its arg outright) use `causal-edges`'s module `db`. Safe only because the daemon shares one connection; if those ever differ, `runHebbianCycle`'s "atomic" wrapper wouldn't cover its own writes and the `BEGIN IMMEDIATE` connection would self-conflict.
