## 5. Structural Layout Finding
`system-deep-loop/runtime/` should remain infrastructure, not become a seventh workflow mode. The existing workflow hub routes modes through `mode-registry.json`, while runtime modules are backend support for research/review/council/improvement. Adding runtime as a mode would blur workflow and infrastructure boundaries.

Evidence: `.opencode/skills/deep-loop-workflows/SKILL.md:12`, `.opencode/skills/deep-loop-workflows/mode-registry.json:19`.

## 9. Fallback-Router Finding
`fallback-router.ts` is a real, tested decision primitive but has no production dispatch caller. Current fanout behavior retries the same lineage label/model until `maxRetries` exhaustion, then records `retry_exhausted`; it does not swap models.

Fallback wiring should remain optional/deferred because it is behavior expansion requiring registry/schema work, canonical-id to provider-slug normalization, explicit caller approval, a `fanout-run.cjs` integration point, and separate fallback lineage attribution.

Evidence: `.opencode/skills/deep-loop-runtime/lib/deep-loop/fallback-router.ts:357`, `.opencode/skills/deep-loop-runtime/scripts/fanout-pool.cjs:628`, `.opencode/skills/sk-prompt-models/assets/model_profiles.json:269`.
