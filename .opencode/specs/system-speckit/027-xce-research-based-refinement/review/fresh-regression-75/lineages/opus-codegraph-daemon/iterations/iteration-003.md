## Review Analysis — Seat `opus-codegraph-daemon`

**Angle:** Accuracy of the `why_included` provenance breadcrumb chain under multi-hop.

### What the code actually does
`buildContext` (line 128) resolves anchors and, per anchor, calls `expandAnchor` (symbol anchors) or builds an inline file-anchor section. `expandAnchor` does **strictly 1-hop** expansion in all three modes (`neighborhood`/`outline`/`impact`) — confirmed: no recursion, `depth` is only ever `0` (anchor) or `1` (direct neighbor), and `recordWhyIncluded` is always called with `depth: 1`. So the `edgeChain` array + `depth` field model is forward-looking scaffolding; a true multi-hop chain is never emitted. The breadcrumb is keyed per-file in `whyIncludedByFile` with a "shallowest-depth-wins" dedup.

### Findings (all traceability/hygiene; no P0/P1)

**1. `truncationReason` is stamped per-entry but driven by section/result-level aggregates — complete breadcrumbs get falsely flagged as truncated.** This is the on-lens finding. Three stamping paths over-report:
- `finalize` (lines 412–421): if `deadlineExceeded` OR `omittedWhyIncluded > 0`, **every** entry in the section is stamped `'deadline'`/`'trace_limit'`, including entries that were fully and accurately captured before the limit was hit. A file dropped purely because the `CONTEXT_WHY_INCLUDED_LIMIT` (25) cap was reached causes *all 25 surviving, complete* entries to be labeled `'trace_limit'`.
- `stampContextTraceTruncation` (lines 282–295), called at lines 216–220: when the **text brief** (`textBrief` string) is truncated/omits sections, every `why_included` entry across **all** sections is stamped `'budget'` — even though `why_included` is structured data returned independently of `textBrief` and is never itself truncated by the token budget. A fully-rendered first section's breadcrumbs get `truncationReason:'budget'` despite being complete.

The breadcrumb *content* (edges/files/depth/confidence) stays accurate; only the per-entry completeness flag is wrong, and it errs toward over-reporting incompleteness (graceful degradation), so this is P2 — but it's a genuine provenance-accuracy defect in the exact artifact under review.

**2. `ambiguous` is hard-coded `false` for every non-anchor entry**, ignoring edge `confidence`/`evidenceClass`. The test at `code-graph-context-handler.vitest.ts:398` codifies this: an edge with `evidenceClass:'INFERRED'`, `detectorProvenance:'heuristic'`, `confidence:0.8` yields `ambiguous:false`. Defensible if `ambiguous` means *node-identity* ambiguity (the target row is exact) rather than *edge-reliability* ambiguity — hence P2/suggestion, not a clear bug. But a consumer reading the breadcrumb cannot distinguish a resolved edge from a heuristic one via this flag.

**3. Multi-edge-to-same-file keeps only the first edge.** `recordWhyIncluded` (line 459) returns early when `previous.depth <= input.depth`. Since all neighbors are depth 1, a file reachable via two depth-1 edges (e.g. anchor both CALLS and IMPORTS symbols in file X) retains only the first edge in `edgeChain`; the second reason is silently dropped. Representative-path provenance is acceptable, so P2.

**4. Dead code: file-anchor branch in `expandAnchor` (lines 487–499) is unreachable.** `expandAnchor` is called only at line 198, which is reached only after the `if (!anchor.symbolId)` block at lines 174–196 has `continue`d past all file anchors. So `anchor.symbolId` is always truthy on entry; the `!anchor.symbolId` branch never executes. P2 maintainability.

No P0/P1: no crash, data loss, security hole, or broken core behavior. The breadcrumb data itself is accurate; the defects are in completeness-metadata fidelity and dead code.
