# Embedding Worker Diagnostic - Seat Statements

All three seats are simulated inline by cli-codex, per the binding trace. No external CLI or subagent participated.

## Seat A - Systems Lens

The strongest systems-level wedge is not one bug but a feedback loop: each Memory MCP daemon owns its own in-process embedding singleton, and the retry loop is deliberately slow. In `shared/embeddings.ts`, the provider is a lazy module singleton (`providerInstance`, `providerInitPromise`) at lines 365-417. The llama-cpp provider then adds its own module singleton (`cachedRuntime`, `runtimePromise`) at `shared/embeddings/providers/llama-cpp.ts:71-73`, and the first real embedding call loads the GGUF and creates one embedding context at lines 195-229. That means every live MCP daemon can independently mmap/load the same 328 MB model and hold its own SQLite connection.

SQLite is WAL-enabled with a 10 second busy timeout in `mcp_server/lib/search/vector-index-store.ts:800-805`, so multiple stale daemons are not automatically fatal, but they can stretch write latency and make llama-cpp inference hit the 30 second per-embedding timeout in `llama-cpp.ts:31` and `llama-cpp.ts:338-341`. After repeated misses, the shared embedding circuit breaker opens after 3 failures and returns `null` immediately for 60 seconds (`shared/embeddings.ts:47-99`, `shared/embeddings.ts:643-646`, `shared/embeddings.ts:699-702`). The retry manager has a separate 5-failure, 2-minute circuit (`retry-manager.ts:365-399`, `retry-manager.ts:600-603`). My systems diagnosis: stale multi-daemon contention triggered timeouts and circuit-breaker flapping; after cleanup, the surviving daemon can embed, but recovery is bounded by the background job.

## Seat B - Empirical Lens

The observed drain rate matches the code almost too neatly. `BACKGROUND_JOB_CONFIG` is hard-coded to `intervalMs: 5 * 60 * 1000` and `batchSize: 5` in `mcp_server/lib/providers/retry-manager.ts:342-347`. `startBackgroundJob()` runs one immediate batch and then one interval batch (`retry-manager.ts:849-861`), and `runBackgroundJob()` processes at most that batch size (`retry-manager.ts:881-903`). The theoretical steady-state drain is 5 rows per 5 minutes, or 60 rows/hour. A visible rate of roughly 1-2/minute is consistent with one successful batch every 3-5 minutes plus measurement jitter, not with a healthy inline path.

The initial burst from 2523 to 2632 vectors in about a minute is also explainable from code. Successful saves trigger opportunistic retry of 2 extra items at `handlers/save/response-builder.ts:708-712`, and async pending saves trigger an immediate `claimAndRetryEmbedding()` via `setImmediate()` at `response-builder.ts:401-415`. Those paths can create a short recovery burst once stale locks disappear. When the shared circuit breaker reopens or writes stop succeeding, the system falls back to the 5-minute retry loop. If vec_rows missing is 274 while queue is about 443, the queue is not only old vector holes; new pending/retry rows are still being added faster than the retry loop can clear them. The data supports H1 strongly and supports H2 as the trigger for why inline embedding stopped dominating.

## Seat C - Contrarian / Skeptic Lens

H1 and H2 could both be incomplete. The code does not prove llama-cpp itself is healthy in this process. `LlamaCppProvider.canLoad()` only checks model readability and imports `node-llama-cpp` (`llama-cpp.ts:294-305`); it does not run inference. Provider creation also runs with `warmup: false` by default (`shared/embeddings.ts:397-401`), so startup can report a plausible provider while the first real `getEmbeddingFor()` fails. The optional dependency is `node-llama-cpp` `^3.15.1` in `mcp_server/package.json:63-65`; the install hint in `llama-cpp.ts:162-164` mentions 3.17.1, which is a small but real compatibility smell. A Q8_0 GGUF or Apple Silicon runtime incompatibility would look like repeated inference failures, shared circuit opens, and search failures without requiring WAL contention.

Another alternative is error-envelope opacity rather than provider failure. `memory_save` defaults to E081 in `lib/errors/core.ts:52-58`, while unmatched errors become "An unexpected error occurred" at `core.ts:180-198`; that can make validation, path, DB, and embedding-adjacent failures indistinguishable from the API. The cheapest falsifier is a single-process, same-env embedding smoke test that bypasses `memory_save` and SQLite: import the shared embedding provider, call `generateQueryEmbedding("health check")`, assert a 768-length vector, and run it while all but one MCP daemon are stopped. If that fails, H1/H2 are secondary.
