# Iteration 5 — gpt-5.6-sol high — focus: activation

Headline finding: the P0→P4 direction is sound, but two blockers make it non-executable as written. The advisor launcher removes the feature flag before spawning the live daemon, and the two default predicates are fleet-global—so they cannot support a hub-by-hub cutover.

Verified claims:

- `mk-skill-advisor-launcher.cjs` constructs the daemon environment exclusively from `CHILD_ENV_ALLOWLIST`, which does not include `SPECKIT_COMPILED_ROUTING`; both build helpers and the live server receive that filtered environment. A P2 canary set in the parent profile therefore does not reach the live advisor daemon (`.opencode/bin/mk-skill-advisor-launcher.cjs:99`, `.opencode/bin/mk-skill-advisor-launcher.cjs:267-270`, `.opencode/bin/mk-skill-advisor-launcher.cjs:1267-1271`). The environment reference already documents this propagation requirement and fresh-session adoption for another advisor flag, establishing the correct operational pattern (`.opencode/skills/system-spec-kit/mcp-server/ENV-REFERENCE.md:639`).

- The two predicates are global checks: the advisor enriches nothing unless the process flag is exactly `1` (`.opencode/skills/system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:357-373`), while the resolver independently accepts only exact `1` before checking hub authority (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/011-runtime-engine/lib/resolve.cjs:20-24`, `:40-42`). Because all seven manifests already say `servingAuthority:"compiled"`, changing absent-variable semantics activates the fleet at once, not one hub at a time (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/010-live-activation/activation/sk-code/manifest.json:1`, `activation/mcp-tooling/manifest.json:1`, `activation/system-deep-loop/manifest.json:1`, `activation/cli-external-orchestration/manifest.json:1`, `activation/sk-prompt/manifest.json:1`, `activation/sk-design/manifest.json:1`, `activation/sk-doc/manifest.json:1`).

- P4 therefore needs a tri-state policy rather than two literal default flips: `0` forces fleet-wide legacy, `1` force-enables eligible hubs for canary use, and unset consults a per-hub `defaultEnabled` rollout field. Both runtime consumers must use that policy with `hubId`; each P4 step then flips one hub’s field and its matching `SKILL.md` directive together. The seven directives currently require explicit `=1` at `.opencode/skills/sk-code/SKILL.md:52`, `.opencode/skills/mcp-tooling/SKILL.md:44`, `.opencode/skills/system-deep-loop/SKILL.md:40`, `.opencode/skills/cli-external-orchestration/SKILL.md:41`, `.opencode/skills/sk-prompt/SKILL.md:36`, `.opencode/skills/sk-design/SKILL.md:56`, and `.opencode/skills/sk-doc/SKILL.md:50`.

- P3 must not simply delete `HUB_CHILD`. That object is both an eligibility list and the engine-location map used to load each hub’s compiler harness and router (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/011-runtime-engine/lib/compiled-route.cjs:23-31`, `:35-50`). Separate eligibility from artifact location, ideally by promoting engines/manifests to durable hub-owned paths and deriving both through one manifest index.

- The existing routing drift workflow is the natural P1 gate, but its path filters cover registries, hub routers, advisor sources, and parent-skill tooling—not activation manifests, compiled engines, hub `SKILL.md` files, or the launcher (`.github/workflows/routing-registry-drift.yml:9-16`, `:19-26`). Without expanding those filters, relevant edits can bypass drift CI entirely.

Ranked executable sequence:

1. P0: promote the resolver, add causal serving status, document the flag, add it to the advisor child allowlist, and test daemon restart/adoption.
2. P1: extend `routing-registry-drift.yml` and add a deterministic command that validates all seven manifest hashes plus the promoted resolver.
3. P2: set `=1` only in one named profile, restart the daemon, and verify through `advisor_recommend`, not only the direct CLI.
4. P3: introduce one durable manifest registry while keeping engine location separate from eligibility.
5. P4: land the tri-state policy once, then flip `defaultEnabled` and the corresponding directive hub-by-hub; malformed flag values should fail closed, and `=0` must override every per-hub default.
