# Iteration 8 — minimax/MiniMax-M3 — focus: unnamed

# Iteration 8 — UNNAMED gaps (hooks/CI/launchers, advisor-hook thread, 4 non-hub routers)

## Headline
The advisor-hook machine-enforcement layer named in 011-runtime-engine/implementation-summary.md:23,111,147 as "in progress / NOT done" is structurally prevented from happening by **two separate compiledRoute-strip layers** (bridge `buildNativeBrief` + plugin `brief` consumption) AND by **two separate SPECKIT_COMPILED_ROUTING-strip allowlists** (launcher + bridge), neither of which is documented in `ENV-REFERENCE.md`. None of the 5 non-hub single-skill routers carry a compiled-routing directive or an explicit "ineligible by design" clause, and the 009 Phase Map is incomplete (3 of 4 children unlisted, mcp-code-mode missing entirely). Combined, even a P4 hub-by-hub flip with `SPECKIT_COMPILED_ROUTING=1` exported through `opencode.json` cannot reach the compiled router from a Claude/Codex/OpenCode runtime that calls the MCP path.

## Verified claims (with file:line)

**CompiledRoute-strip, layer 2 — bridge `buildNativeBrief` rebuilds recommendations without it.**
- `system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:357-374` — `enrichCompiledRoutes` attaches `compiledRoute` per recommendation when `SPECKIT_COMPILED_ROUTING === '1'`.
- `system-skill-advisor/mcp-server/schemas/advisor-tool-schemas.ts:221` — `compiledRoute: z.record(z.string(), z.unknown()).optional()` is in the public schema.
- `system-skill-advisor/mcp-server/plugin-bridges/mk-skill-advisor-bridge.mjs:540-551` — `buildNativeBrief` rebuilds the recommendations list with ONLY `skill`, `kind`, `confidence`, `uncertainty`, `score`, `passes_threshold`, `reason`, `ambiguousWith`. `compiledRoute` is **not copied** even though `data.recommendations` carries it (line 519-528 returns the full object). The bridge's `response()` envelope (line 95-102) preserves arbitrary `metadata`, so it could carry a top-level compiledRoute summary, but does not.
- `.opencode/plugins/mk-skill-advisor.js:771-773` — plugin consumes ONLY `response.brief` (string). F-3-3 already named this; the new finding is that **bridge layer 2 is the upstream cause**, not just the plugin's `brief` rendering.

**SPECKIT_COMPILED_ROUTING-strip, layer 2 — bridge CHILD_ENV_ALLOWLIST also excludes it.**
- `.opencode/bin/mk-skill-advisor-launcher.cjs:99-150` — launcher's allowlist; F-5-1 confirmed `SPECKIT_COMPILED_ROUTING` is absent.
- `system-skill-advisor/mcp-server/plugin-bridges/mk-skill-advisor-bridge.mjs:58-87` — bridge's allowlist also lacks it. `createChildEnv()` at line 210-214 filters via this set, so the CLI fallback path `runCliRecommend` (line 672-735) — which spawns `node .opencode/bin/skill-advisor.cjs advisor_recommend` with `env: createCliChildEnv(env)` — strips the flag before the CLI subprocess sees it. The in-process native path (`loadNativeAdvisorModules`, line 425) reads `process.env` directly, so it inherits, but **only if OpenCode was launched with the flag set** (chain inheritance: OpenCode → plugin → bridge → native compat).
- `system-spec-kit/mcp-server/ENV-REFERENCE.md` — `grep -n "SPECKIT_COMPILED_ROUTING"` returns zero hits. The flag is **undocumented** in the env reference, so operators have no authoritative list of which variables need forwarding and which get stripped.

**Per-prompt Node child spawn — no batched / cached compilation evaluation.**
- `system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:332-345` — `compiledRouteForRecommendation` uses `execFileSync(process.execPath, [...compiled-route.cjs, --hub, skillId, --prompt, prompt], { timeout: 5_000, maxBuffer: 1_048_576 })`. With topK=3 recommendations and 7 COMPILED_ROUTING_HUBS, **up to 3 fresh Node process spawns per `advisor_recommend` call**. Each spawn re-resolves the resolver from the spec tree (per F-2-4), reads the activation manifest, and runs the hub's canary/plain router. No in-process cache, no batching, no concurrency limit.

**Serving-authority observability gap — no tool reads the per-hub manifests.**
- `.opencode/plugins/mk-skill-advisor.js:826-861` — `spec_kit_skill_advisor_status` reports only plugin-side metrics (cache hit rate, bridge status, last duration). It does **not** enumerate hub serving authority.
- `system-skill-advisor/mcp-server/handlers/advisor-status.ts` — `grep -n "servingAuthority"` returns zero hits. `advisor_status` reports freshness/trustState/generation but never reads `010-live-activation/activation/<hub>/manifest.json`.
- The 7 manifests (`specs/.../010-live-activation/activation/<hub>/manifest.json`, e.g. `sk-code/manifest.json:1` = `{"servingAuthority":"compiled","shadowOnly":false,...}`) are consumed only by the resolver at hot-path time (`resolve.cjs:26-34`). No observability surface ships a "hub X is currently compiled-serving" read to an operator. ADR-002's data-driven eligibility needs that readout to confirm the derived predicate; today the only way to learn a hub's state is to run `flip-serving.cjs --status` (CLI-only).

**Non-hub single-skill routers — incomplete eligibility story.**
- 5 single-skill routers exist (`sk-git/SKILL.md:39` smart-router; `system-spec-kit/SKILL.md:95` registry-less; `mcp-code-mode/SKILL.md:71` standalone single-mode; `system-code-graph/SKILL.md:98` singleton-mode; `system-skill-advisor/SKILL.md:87` standalone single-mode). All 5 are absent from `COMPILED_ROUTING_HUBS` (`advisor-recommend.ts:41-49`) and absent from `HUB_CHILD` (`specs/.../011-runtime-engine/lib/compiled-route.cjs:23-31`).
- `specs/.../009-non-hub-rollout/spec.md:25-28` Phase Documentation Map lists only `001-sk-git`. The directory actually contains 4 children (`001-sk-git/`, `002-system-code-graph/`, `003-system-skill-advisor/`, `004-system-spec-kit/`) with full packet docs each. `mcp-code-mode` has no entry at all. ADR-002 (`012-default-on-decision/decision-record.md:153-252`) plans to derive eligibility from manifest freshness; the 5 non-hub routers have no manifest and no eligibility clause, so they remain permanently ineligible by absence — which is the intended design but never stated.

**The advisor-hook machine-enforcement thread is structurally unreachable.**
- `011-runtime-engine/implementation-summary.md:23,111,147` declares advisor-hook machine-enforcement as "in progress / NOT done".
- The only path from the prompt to the compiled router is: prompt → `mk-skill-advisor.js:824` `appendAdvisorBrief` → bridge subprocess → `mk-skill-advisor-bridge.mjs:519` `handleAdvisorRecommend` → `advisor-recommend.ts:362` `enrichCompiledRoutes` → `compiledRouteForRecommendation` (line 326-355) → `compiled-route.cjs:39` legacy sentinel on any drift. This 6-hop chain **discards compiledRoute at hop 5** (bridge rebuild), so even if every gate fires, the model sees only the legacy brief text — never the compiled decision. There is no second consumer of `compiledRoute` in the tree (`grep -rn "compiledRoute" .opencode/` returns only the resolver, the engine, the schema, the enricher).

## Ranked recommendations

1. **(P0, advisor-hook thread)** Either (a) thread `compiledRoute` through `buildNativeBrief.metadata` as a top-level summary (so the plugin's existing metadata path picks it up — `mk-skill-advisor-bridge.mjs:97-99` already preserves `metadata`), AND add `SPECKIT_COMPILED_ROUTING` to both `mk-skill-advisor-launcher.cjs:99-150` and `mk-skill-advisor-bridge.mjs:58-87` allowlists; or (b) downgrade the implementation-summary claim and treat the compiled router as unreachable from the prompt until both layers are patched. F-3-3's gap is necessary but not sufficient — the bridge strip and the launcher/bridge env strip must both be fixed before the model can observe compiled routing.
2. **(P0, observability)** Add an `advisor_status` field or a dedicated `compiled_routing_status` MCP tool that reads `010-live-activation/activation/<hub>/manifest.json` for each hub and reports `{hubId, servingAuthority, generation, manifestMtime}`. Without it, ADR-002's data-driven eligibility has no operator-visible readout.
3. **(P0, governance docs)** Add `SPECKIT_COMPILED_ROUTING` to `system-spec-kit/mcp-server/ENV-REFERENCE.md` with default OFF, daemon-propagation caveat (must reach the MCP-server child — currently stripped by launcher + bridge allowlists), and per-hub override semantics.
4. **(P1, perf)** Replace `execFileSync` per recommendation with either a batched `--multi` CLI invocation or an in-process resolver import. The 3-spawns-per-prompt cost is a regression risk at advisor-recommend frequency.
5. **(P1, non-hub coverage)** Reconcile `009-non-hub-rollout/spec.md:25-28` Phase Map to list all 4 children (add `002-system-code-graph`, `003-system-skill-advisor`, `004-system-spec-kit`) and either add `005-mcp-code-mode` or document its omission. Add a one-line "non-hub routers are ineligible by design; eligibility is hub-only" clause to `012-default-on-decision/decision-record.md` ADR-002 consequences so ADR-002's "any future skill" claim (line 235) doesn't over-promise.
6. **(P2, cache-shape)** Confirm whether `enrichCompiledRoutes` re-invocation per cache hit (line 477) is the intended behavior — currently it IS, but a comment in the code would prevent a future maintainer from "optimizing" it to skip the re-invocation and breaking the machine-enforcement contract.
