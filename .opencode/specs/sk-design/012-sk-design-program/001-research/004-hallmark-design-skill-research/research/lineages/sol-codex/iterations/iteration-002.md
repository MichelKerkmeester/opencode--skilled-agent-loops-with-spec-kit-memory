# Iteration 2: Audit and anti-slop coverage diff

## Focus

This iteration compared Hallmark's canonical 58-gate presentation test, named anti-pattern catalog, six-axis pre-emit critique, and read-only audit verb with sk-design's shipped audit procedure, fingerprint catalog, broader audit contract, and shared polish gate. The narrow interpretation is deliberate: Hallmark's design-authoring rules are treated as candidate audit probes, not as a replacement release gate.

## Findings

1. **The useful unit of reuse is a selective probe library, not Hallmark's 58/58 all-or-nothing gate.** Hallmark requires every answer to be “no” and mixes genre exceptions into the same checklist; sk-design instead treats tells as evidence-backed hypotheses, keeps legitimate brand choices, and grades user impact through P0-P3 plus five audit dimensions. Verdict: ADAPT individual probes into existing references and the machine-readable fingerprint registry; SKIP wholesale checklist replacement. ## Ruled Out

- Copying the complete 58-gate document into sk-design: it duplicates broader checks, imports brittle style doctrine, and would require preserving Hallmark's MIT notice for copied or substantial text.
- Replacing P0-P3 with critical/major/minor: the two systems measure different things.
- Adding Hallmark CSS stamps or .hallmark/log.json to the audit workflow: audit is read-only and sk-design already has evidence/source surfaces.
- Promoting font lists, zero-chroma neutrals, 5% accent area, 1.3× hero padding, or fixed family-count ceilings to universal blockers: they are useful prompts only when tied to the brief and rendered evidence.
- The attempted direct registry enumeration used the wrong assumed top-level JSON shape. The catalog reference itself documents the registry mirror and parity checker, so this did not block the file-level target recommendation; exact row-schema work remains implementation-phase verification.

## Dead Ends

None. The selective-probe route is productive; only wholesale import and Hallmark-specific state/stamp mechanics are exhausted.

## Edge Cases

- Ambiguous input: Hallmark calls some rules universal while also supplying genre exceptions and project-history conditions. This iteration treated them as presentation probes with applicability conditions, not universal truths.
- Contradictory evidence: Hallmark's “critical” means “ships as slop,” while sk-design's P0/P1 means user/task/access harm. The conflict is resolved by retaining sk-design severity and mapping tells into Anti-Patterns.
- Missing dependencies: Exact machine-registry row schema was not enumerated because the initial jq selector assumed an object instead of the checked-in top-level shape. Existing catalog-to-registry documentation was sufficient for architectural targeting, but an implementation pass must inspect the schema before editing.
- Partial success: The gate comparison and target plan are complete; implementation-level detector expressions and fixtures were intentionally deferred because this is research-only.
