This release fixes the hub's doc surface so it stops describing itself as a passive "thin sentinel" and instead reads as the per-model prompt-craft hub it actually became after the model registry and profiles moved in. The SKILL.md, README and the nine per-model profiles now follow the sk-doc templates, so the docs match the skill's real job. No dispatch behavior changed.

> Spec folder: `.opencode/specs/skilled-agent-orchestration/130-prompt-knowledge-layering/010-hub-doc-alignment-and-router/` (Level 1)

---

## What Changed

#### Architecture

- **Model-keyed smart router** -- SKILL.md §2 declared "this skill has no runtime router of its own", which no longer matched reality. We replaced the no-op with a real router faithful to `skill_smart_router.md`: runtime discovery, an alias map that resolves a named model to its canonical id, guarded loads and three-tier fallback. The routing key is the model id and the resource is `references/models/<id>.md`, so dispatch picks the right profile reliably. The LOC cap was relaxed from 200 to 300 to fit the inline router.

&nbsp;

#### Documentation

- **README rewrite** -- the README still carried the stale hub identity and ephemeral spec references (arc name, phase numbers, research-iteration links, spec-folder links), which rot over time. We rewrote it to the correct hub identity, stripped every ephemeral reference and aligned structure plus Human Voice Rules to `skill_readme_template`, so the README stays accurate and durable.
- **Nine profiles conformed** -- the per-model profiles and `_index.md` lacked a consistent opening and section style. Each now opens with `## 1. OVERVIEW` (Purpose / When to Use / Core Principle) and uses ALL-CAPS numbered H2 sections per `skill_reference_template`, with code-block scaffold headers and content preserved. The verification confirmed seven numbered ALL-CAPS sections per profile, so the profiles are now scannable and uniform.
