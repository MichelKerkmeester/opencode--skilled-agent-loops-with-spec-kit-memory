**Standard consequence:** Equality at the Rust struct level is insufficient. The release-blocking assertion is:

```text
typescript_oracle(input).as_bytes() == rust_boundary(input).as_bytes()
```

Tolerance is zero. Parsing both outputs and comparing semantic JSON is only a diagnostic because it can conceal key-order, number-rendering, omission, escaping, and newline differences.

### Test Layers

#### Unit Tests

Place focused tests beside implementation code in `#[cfg(test)] mod tests`. Rust convention puts unit tests in `src/`, where they can test private implementation details without including test code in normal builds. [SOURCE: https://doc.rust-lang.org/book/ch11-03-test-organization.html]

Use unit tests for:

- Six-decimal rounding at boundaries immediately below, at, and above half-way cases.
- Comparator chains with equal primary scores, equal confidence, equal RRF rank, Unicode IDs, and duplicate IDs.
- Deterministic SHA-256 input composition and truncation.
- Canonical serialization helpers.
- Explicit map-to-sequence canonicalization.
- Panic-free handling of malformed boundary DTOs.

Do not mock the TypeScript oracle in unit tests. Unit tests establish local invariants; they cannot prove cross-language parity.

#### Integration Tests

Put public-contract tests in top-level `tests/*.rs`; Cargo compiles each file as a separate crate using only the public API. Shared helpers belong in `tests/common/mod.rs`, not `tests/common.rs`, to avoid Cargo treating the helper as a separate empty integration-test crate. [SOURCE: https://doc.rust-lang.org/book/ch11-03-test-organization.html]

Recommended responsibilities:

- `parity_golden.rs`: replay every committed input against the pure Rust entry point and compare exact expected bytes.
- `determinism.rs`: repeat cases under different insertion orders and process runs.
- `property_parity.rs`: generated differential testing against the TypeScript oracle.
- `boundary_native.rs`: napi-rs addon DTO and serialized-output tests.
- `boundary_wasm.rs`: `wasm-bindgen-test` coverage for the actual WASM export.
- `error_parity.rs`: malformed input, nullability, overflow, and error-envelope equality.

For WASM exports, use `#[wasm_bindgen_test]` and execute through `wasm-pack test --node`; this exercises the generated JavaScript/WASM boundary rather than only the host-native Rust core. [SOURCE: https://wasm-bindgen.github.io/wasm-bindgen/wasm-bindgen-test/usage.html]

For napi-rs, retain a Node-side test that loads the generated `.node` addon and compares its result with the same TypeScript oracle. napi-rs produces the native addon, JavaScript loader, and TypeScript declarations as package artifacts, so a Rust-only test cannot validate the delivered Node boundary. [SOURCE: https://napi.rs/docs/introduction/getting-started]

#### Documentation Tests

Use executable doc tests for public, deterministic Rust APIs such as DTO constructors, canonicalization, ID generation, and pure ranking functions. `rustdoc` compiles and runs fenced Rust examples, and assertions make examples behavior tests. [SOURCE: https://doc.rust-lang.org/rustdoc/write-documentation/documentation-tests.html]

Do not use doc tests as napi-rs or WASM parity evidence. They do not replace loading the generated Node/WASM artifact. Use `no_run` only when execution genuinely requires unavailable infrastructure; ordinary examples should run. [SOURCE: https://doc.rust-lang.org/rustdoc/write-documentation/documentation-tests.html]

### Property Testing as the Parity Oracle

Prefer `proptest` over `quickcheck` for this surface. `proptest` supports per-value generation and shrinking, which is better suited to constrained DTOs, nearly tied scores, malformed boundary objects, and graph structures than type-wide generation alone. When a failure occurs it shrinks toward a minimal reproducer. [SOURCE: https://proptest-rs.github.io/proptest/intro.html]

The primary property is differential, not merely algebraic:

```rust
proptest! {
    #[test]
    fn matches_typescript_oracle(input in boundary_input_strategy()) {
        let expected = typescript_oracle_bytes(&input)?;
        let actual = rust_boundary_bytes(&input)?;
        prop_assert_eq!(actual, expected);
    }
}
```

The generator must deliberately overweight:

- Values around six-decimal rounding boundaries.
- `-0.0`, finite extremes, and non-finite values if the TypeScript boundary accepts them.
- Equal and nearly equal scores that exercise every tie-break.
- Permutations of logically identical map/set insertion orders.
- Unicode, combining characters, empty strings, separators used by deterministic IDs, and long paths.
- Empty, singleton, duplicate, cyclic, dangling, and depth-truncated graph inputs.
- Missing versus explicit `null` fields and optional arrays.
- Inputs that produce empty, abstaining, degraded, or error envelopes.

**Recommended execution design:** batch generated JSONL inputs through one long-lived Node oracle process instead of spawning Node once per case. Compare the oracle's raw stdout record with raw Rust output. Keep process protocol framing separate from payload bytes so line delimiters cannot accidentally become part of the product contract.

Proptest defaults to 256 successful cases and exposes case count, RNG algorithm, and seed through `Config` and environment variables. CI should use a fixed minimum case count and print the seed on failure. [SOURCE: https://docs.rs/proptest/latest/proptest/test_runner/struct.Config.html]

Proptest persists failing seeds under `proptest-regressions` and recommends committing them so collaborators and CI replay the failures. [SOURCE: https://proptest-rs.github.io/proptest/proptest/failure-persistence.html]

**Additional repo rule:** after shrinking a TS↔Rust mismatch, promote the minimized concrete input and TypeScript output into the committed golden corpus. Persisted seeds alone can change meaning when strategies evolve; the concrete fixture becomes the durable migration contract.

For WASM, prefer generating the parity corpus on the host and replaying it through `wasm-bindgen-test`. Direct proptest-on-WASM support is documented as highly experimental and requires disabling unsupported default features. [SOURCE: https://proptest-rs.github.io/proptest/proptest/wasm.html]

### Snapshot Testing

Use `insta` for stable, human-reviewable serialized diagnostics and complex DTOs. It supports raw string snapshots and structured serializer snapshots, stores changed snapshots as pending files, and provides an explicit review flow. [SOURCE: https://insta.rs/docs/quickstart/]

For parity-protected output:

- Prefer `assert_snapshot!` over structured JSON/YAML snapshot macros when exact product bytes are the subject.
- Snapshot the already-canonicalized string; do not let insta choose the serialization format.
- Use structured snapshots only for diagnostic views whose exact wire bytes are not contractual.
- Set `CI=true` and `INSTA_UPDATE=no`; CI must fail rather than create or accept replacements. [SOURCE: https://insta.rs/docs/quickstart/]
- Reject unreferenced snapshots in CI rather than silently deleting them. `cargo insta test` supports explicit unreferenced-snapshot policies. [SOURCE: https://insta.rs/docs/cli/]

Goldens and snapshots serve different purposes: goldens are the TypeScript-owned migration oracle; insta snapshots are Rust test-review artifacts. A snapshot must not become an independently invented second oracle.

### Determinism Tests

Run every parity case repeatedly after permuting source insertion order. Rust's default `HashMap` is randomly seeded and iteration order is arbitrary, so any map iteration that reaches serialization, ranking, ID composition, or traversal output is a parity defect unless explicitly sorted or converted to an ordered representation. [SOURCE: https://doc.rust-lang.org/std/collections/struct.HashMap.html]

Required metamorphic properties:

```text
output(input) == output(clone(input))
output(input) == output(permuted_map_insertion(input))
output(input) == output(permuted_equivalent_edges(input))
output(input) == output(run_in_fresh_process(input))
```

The final property does not require that semantically unordered input produce the same output if TypeScript itself preserves input order. The TypeScript oracle decides whether order is semantic; tests must encode that decision explicitly.

### Benchmarking

Use Criterion with `[[bench]] ... harness = false`. Criterion supplies a stable-compatible custom harness, warm-up, repeated sampling, statistical comparison, unique benchmark IDs, and `black_box` protection against optimization removing the measured work. [SOURCE: https://bheisler.github.io/criterion.rs/book/getting_started.html]

Do not use Rust's built-in `#[bench]` harness: `#[bench]` remains unstable and nightly-only, while Cargo explicitly identifies Criterion as a stable-channel alternative. [SOURCE: https://doc.rust-lang.org/cargo/commands/cargo-bench.html]

Benchmark three distinct costs:

- `kernel`: pure scoring, sorting, hashing, traversal, or fusion with pre-decoded Rust values.
- `serialization`: DTO decode, canonical serialization, and byte production.
- `boundary_batch`: Node↔napi-rs or Node↔WASM crossing at realistic batch sizes.

Never report a kernel-only speedup as an addon/WASM speedup. Boundary copying, conversion, and serialization are part of the architecture being evaluated.

Use fixed, versioned workloads derived from the golden corpus, including small typical inputs, tie-heavy adversarial inputs, and large break-even inputs. Benchmark IDs must encode workload and size, such as `rank/tie-heavy/1000`.

Criterion supports named saved baselines and comparison against an existing baseline. [SOURCE: https://bheisler.github.io/criterion.rs/book/user_guide/command_line_options.html]

Recommended CI policy:

- Every PR: `cargo test --benches` to prove benchmark targets compile and execute without full measurement. Criterion documents this as its CI smoke path. [SOURCE: https://bheisler.github.io/criterion.rs/book/user_guide/command_line_options.html]
- Dedicated pinned runner: save the base-branch baseline, run the candidate on the same machine, and report percentage plus confidence interval.
- Gate only sustained, contract-relevant regressions above a documented threshold; do not hard-fail ordinary shared-runner noise.
- Keep correctness parity separate and absolute: benchmark variance may use thresholds, parity never does.
