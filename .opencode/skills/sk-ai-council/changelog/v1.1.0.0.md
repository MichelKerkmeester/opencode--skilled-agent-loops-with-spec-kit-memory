## New Features

### Dedicated Council Graph (Phase 003)

A derived SQLite projection now indexes council deliberation as graph rows. Four MCP tools own the surface — `council_graph_upsert`, `council_graph_query`, `council_graph_status`, `council_graph_convergence` — registered as a distinct family from the existing `deep_loop_graph_*` research/review graph. Eight node kinds (SESSION, ROUND, SEAT, CLAIM, EVIDENCE, DISAGREEMENT, DECISION, RECOMMENDATION), ten relation kinds (PARTICIPATES_IN, PROPOSES, SUPPORTS, CONTRADICTS, DERIVES_FROM, AGREES_WITH, RESOLVES, ESCALATES, EVIDENCE_FOR, RECOMMENDS), five query modes (`unresolved_disagreements`, `evidence_chain`, `decision_support`, `convergence_blockers`, `hot_nodes`), and three convergence buckets (`STOP_ALLOWED`, `CONTINUE`, `STOP_BLOCKED`) cover the full council deliberation lifecycle. ADR-001 documents the dedicated-projection decision over deep-loop graph reuse. Artifacts under `ai-council/**` stay authoritative — the graph rebuilds from them.

### Playbook Now Covers Graph Behavior (Phases 004 + 005)

The manual testing playbook grew from 18 to 32 scenarios across 9 categories. Phase 004 added 8 functional graph integration scenarios (DAC-019..DAC-026) one-to-one with the shipped MCP surface — idempotent upsert + self-loop rejection, empty-input no-op, hostile-metadata redaction, five query modes, three convergence buckets, recovery payload, derived-projection replay contract, tool-family separation. Phase 005 added 6 value-comparison scenarios (DAC-027..DAC-032) that contrast the no-graph operator baseline workflow against the with-graph workflow on real-world situations: unresolved-disagreement triage, decision provenance audit, convergence safety under critical disagreement (the graph blocks a stop the naive two-of-three baseline would allow), stalled-council blocker ranking, hot-topic discovery, mid-run interruption recovery. Phase 005 also repaired two pre-existing vitest failures (Claude translated frontmatter parity assertion, stale persist-artifacts HELPER_PATH).

### Measured Value Ratios (Phase 006)

Phase 006 turned the value-comparison scenarios from contract-only docs into a fixture-driven vitest. Each test seeds a synthetic artifact tree plus a temp council graph, runs both workflows, asserts the graph returns the expected answer, and writes measured baseline-vs-graph ratios to `council-graph-value-report.json` on every run. Speed-driven scenarios show 7×-13× fewer file reads; safety-driven (DAC-029) and structure-driven (DAC-032) scenarios show their value through correctness rather than read count.

## Architecture

### Phase 003 Storage Layer

The new `lib/council-graph/` package owns SQLite-backed derived storage. `council-graph-db.ts` exposes idempotent upsert with strict-schema validation (rejects self-loop edges, treats empty input as explicit no-op success, redacts arbitrary metadata to an allowlist of scalar fields before serializing). `council-graph-query.ts` exposes the five query modes with bounded outputs and prompt-safe metadata. `handlers/council-graph/` exposes four MCP entrypoints — upsert, query, status, convergence — that wrap the lib with input validation and prompt-safe response shaping. The graph is a derived projection by design: rows can be deleted scoped to `(specFolder, sessionId)` and replayed from `ai-council/**` artifacts without touching the artifacts themselves.

### Phase 006 Test Harness

The value-scenario harness uses shared seed helpers (`seed-helpers.ts`) plus per-scenario fixtures (`dac-027.ts` through `dac-032.ts`) plus shared seed-data JSON consumed by both vitest and the operator seeder (`scripts/seed-council-value-fixture.cjs`). Each test opens a fresh in-memory SQLite DB, seeds the graph through the same upsert handler the runtime uses, runs both the no-graph baseline (`fs.readdirSync` + grep predicates) and the with-graph workflow (MCP handler calls), then asserts answer equivalence and writes measured metrics to `council-graph-value-report.json`. Temp directories clean up between tests for isolation.

### Phase 007 Test-Gate Topology

The council test gate is local-only by design — no GitHub Actions. Three invocation paths share one matrix definition: `npm run test:council` (fastest, vitest only), `bash test-council-matrix.sh` (vitest + sk-doc + strict spec validate, ~5 seconds), opt-in pre-push hook (matrix when council files changed, silent skip otherwise). The pre-push template lives at `.github/hooks/scripts/pre-push-council.sh` and stays opt-in via symlink.
