### Cluster A — confirm-gating and rollback ordering in apply (tri-025, tri-027, tri-026, tri-077, tri-028)

**tri-025 / tri-027 (P1) — destructive ops without confirm outside hard-stale.** Verified current: the only confirm gate in `applyCodeGraph` is `apply-orchestrator.ts:415` (`classification.state === 'hard-stale' && args.confirm !== true`). An explicit `recover-sqlite-corruption` or `rollback-bad-apply` request on a fresh/soft-stale graph passes the fresh-noop check at `:400` (which requires `args.operation === undefined`), skips the confirm gate, and reaches `recovery-procedures.ts` `moveTriplet`/restore at `:174` and `:263-267`. The playbook (`code-graph-apply-sub-operations.md:56`) promises confirm refusal unconditionally — the code only delivers it when staleness happens to classify hard-stale.
*Risk if fixed wrong:* an op-specific confirm gate must sit BEFORE the known-good snapshot at `:446` (gating after snapshot still pollutes the snapshot chain, worsening tri-028) and must not change the fresh-noop or soft-stale rescan paths that doctor flows rely on. **Fix: code-small** (one pre-dispatch predicate + abort return).

**tri-026 / tri-077 (P1/P2) — repair-nodes skip reports committed.** Verified current: `apply-orchestrator.ts:314-320` returns `repairNodes.skippedReason` from dispatch when `crashRootCauseAddressed !== true`; control then flows through postflight to `:508-519` which records and returns `status:'committed'`. The playbook (`:48`) expects a refusal with `requiredAction`. Note a post-banking addition: `:336-341` now throws `repair-nodes requires confirm=true` when eligible rows exist — but that only fires AFTER the crash-root-cause check passes, so the misleading committed-no-op path is unchanged.
*Risk if fixed wrong:* automation reading `apply.lastResult`/`batteryPassRate` from `code_graph_status` will see a new terminal status; the abort must still write audit events and apply-metadata so the status surface stays consistent. **Fix: code-small** (pre-dispatch abort with `requiredAction`, before the `:446` snapshot).

**tri-028 (P1) — rollback restores the snapshot it just took.** Verified current: `:446-451` runs `snapshotKnownGoodTriplet` unconditionally before every non-dry dispatch, including `rollback-bad-apply`. `rollbackBadApply` then quarantines the live triplet and restores `findLatestKnownGood` (`recovery-procedures.ts:135-148`, `.sort().at(-1)`) — which is lexicographically the snapshot created milliseconds earlier from the very state the operator is trying to roll back. Manual rollback is therefore a no-op restore of the suspect state, not the prior baseline.
*Risk if fixed wrong:* this is snapshot-ordering surgery. Skipping the snapshot for rollback ops loses the safety copy of the pre-rollback state; selecting "second-latest" naively breaks when the latest known-good is legitimately from a prior committed apply. Target selection needs run-scoped exclusion (e.g., pass the current run's snapshot dir into rollback and exclude it). Interacts with tri-151 retention — pruning must never delete the rollback target. **Fix: code-careful.**
