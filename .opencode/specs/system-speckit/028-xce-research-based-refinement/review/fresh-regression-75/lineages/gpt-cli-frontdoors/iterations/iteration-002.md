# Seat gpt-cli-frontdoors — iteration 2 (global #45)

- Model: `openai/gpt-5.5-fast` · Executor: opencode · Angle: gpt-cli-frontdoors
- Verdict: **CONDITIONAL** · Findings: P0=0 P1=1 P2=1

## Analysis

Findings:

1. P1 security: prompt-time markers only imply warm-only; they do not block trusted mutation commands.
Evidence: the spec safety model says `advisor_rebuild` and `skill_graph_scan` are “blocked from prompt-time paths entirely” and that the CLI cannot sneak a rebuild into a hook (`before-vs-after.md:375`). In the CLI, prompt-time env vars only feed `defaultWarmOnly()` (`skill-advisor-cli.ts:353-361`), mutation gating only checks `trusted` (`skill-advisor-cli.ts:659-664`), and validated tools are invoked afterward (`skill-advisor-cli.ts:1289-1296`). A prompt-time process with a warm daemon plus `--trusted` or `MK_SKILL_ADVISOR_CLI_TRUSTED=1` can still run the mutation.

2. P2 security: `--trusted=false` is parsed as trusted.
Evidence: option parsing strips `=...` before dispatch (`skill-advisor-cli.ts:398`), then any `trusted` raw flag sets `trusted = true` without inspecting the inline value (`skill-advisor-cli.ts:428-430`). Generated wrappers commonly emit `--flag=false`; doing that here silently grants mutation authority.

No shell-style argument injection was found in the front-door path I traced: the shim forwards via `spawnSync(process.execPath, [cliDist, ...argv])` without a shell (`.opencode/bin/skill-advisor.cjs:114`), and the CLI forwards validated JSON-RPC arguments rather than shell-building commands.
