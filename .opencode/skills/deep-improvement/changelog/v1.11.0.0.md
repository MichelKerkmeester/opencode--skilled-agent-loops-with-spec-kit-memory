Lane C now measures whether a skill is actually **useful**, not just well-routed. v1.10.0.0 shipped the live "Mode B" benchmark but its D4 was a *hallucination*-grader proxy (which by its own rubric does not score correctness), n=2, and unscored in every aggregate — so routine-task usefulness was effectively UNKNOWN. This release adds the **D4-R task-outcome instrument** that measures real usefulness (does the skill help the model *do* a routine task?), a **deferred-asset scoring lane** that stops asset deferral from inflating D3, and **real-phrasing intent synonyms** so the router recognizes realistic user wording. The weighted dimensions and the v1 report schema are unchanged — every addition is advisory.

## What Changed

### D4-R — a real task-outcome usefulness instrument

- **A separate instrument, never collapsed with the hallucination D4.** `d4-ablation.cjs` gains `buildTaskOutcomePrompt` / `gradeTaskOutcome` / `runD4RAblation`: the model is asked to *do* a routine task (a minimal patch plan + verification commands, not a routing list) skill-on vs skill-off, and both answers are graded by a new task-outcome rubric (`model-benchmark/scorer/grader/prompts/system-grader-task-outcome.md`: correctness / verification-fit / focus / hallucination-risk) reused through the Lane B grader harness with distinct cache keys. Score = `0.5 + (on − off) / 2`; stamped `attribution: approximate` with the same skill-off contamination guard.
- **Wired into the orchestrator behind an opt-in flag.** `run-skill-benchmark.cjs` gains `--d4` (live only) → an async `augmentWithD4R` pass that runs the ablation over routine scenarios and writes an advisory `D4_task_outcome` to the report. Kept out of the synchronous `run()` so the deterministic CI path is unchanged.

### Deferred-asset scoring lane (D3 honesty)

- **Assets scored on their own lane.** `expectedAssets` now feeds a separate `assetRecall` (live: recall vs the model's stated assets; router: deferred + *visible*), and `live-executor.parseLiveResult` keeps assets on a distinct `observedAssets` channel. Previously a stated, useful asset was merged into `observedResources` and counted as D3 over-routing waste — a measurement artifact. D2/D3 now score references only.

### Intent-signal recall (sk-code)

- **Real-phrasing synonyms.** `sk-code` `INTENT_SIGNALS` gains `in-view` / `motion cdn` (MOTION_DEV) and `smooth-scroll` / `intersectionobserver` (IMPLEMENTATION), so the router routes the real gold for realistic CS-001/SD-001 wording. Distinctive tokens only — bare `gate`/`gated` were rejected (they over-match `investigated`/`delegated`). SD-001 D2 0.455→0.636, CS-001 0.200→0.500 (D3 0.286→0.417); zero D2 regressions across 24 scenarios.

### Report (additive only)

- `score-skill-benchmark.cjs` adds an `advisorySignals` block (`D4_task_outcome` + `assetRecall`), surfaced but **not** folded into the weighted aggregate; `build-report.cjs` renders an "Advisory signals (NOT in the weighted aggregate)" section. `schemaVersion: skill-benchmark-report.v1` is preserved.

## Result

The D4-R ablation ran live on five routine sk-code scenarios (gpt-5.5-fast on/off, claude-sonnet grader): **`D4_task_outcome` aggregate 54/100.** The default scenario set is codified as `DEFAULT_D4R_SCENARIOS` in `scripts/skill-benchmark/run-skill-benchmark.cjs` (`LS-001`, `LS-002`, `LS-003`, `LS-004`, `SD-002`). The durable finding: the skill helps most where the base model is weak (LS-001 .35→.72, LS-004 .22→.44) and can hurt where it is already strong (LS-002 .85→.72, SD-002 .45→.27) — over-routing noise. This confirms round-1's n=2 "task-dependent usefulness" hypothesis at n=5 with the right instrument, and motivates further D3 efficiency work. Artifact + README: `sk-code/benchmark/d4r-live/`.
