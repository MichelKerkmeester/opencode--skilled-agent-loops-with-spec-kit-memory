## Findings
1. **Adopt one authority per plane, not dual authority over style content.** Flat style artifacts remain the authoring and content source of truth; the published SQLite generation becomes the sole runtime authority for discovery, eligibility, ranking, cards, and retrieval metadata. SQLite is a rebuildable materialized projection, not an independently editable copy of style content. This split preserves the current deterministic manifest contract, which reads and hashes the corpus, rejects a moving corpus, and derives one generation hash, while also removing corpus walks from the query path. It also preserves hydration's stronger guarantee: a request must match a committed generation, the live corpus must still serialize identically, paths must remain contained, rights must pass, and bytes must match the recorded digest. A database-only content authority would invalidate those guarantees; dual writes would create conflict resolution without a canonical winner. 2. **Use a five-stage migration with an immutable baseline.** Stage 0 freezes schema/normalization versions and captures the checked manifest, source aggregate hash, fixture-query outputs, and a rollback checkpoint. Stage 1 bootstraps a side database from one two-pass-verified, non-moving corpus snapshot and publishes nothing. Stage 2 keeps legacy file retrieval primary while every source change feeds shadow indexing and sampled dual reads. Stage 3 canaries the persistent adapter for selected callers while legacy remains an explicit rollback mode. Stage 4 makes SQLite the only normal query input after all gates pass; flat artifacts remain hydration and rebuild inputs. Stage 5 retires legacy query scanning only after the rollback window, but keeps an offline parity/repair command. The bootstrap must restart rather than publish if the post-build fingerprint changes. 3. **Make parity multidimensional and separate compatibility from semantic improvement.** Census parity is exact equality of active IDs, UUID-to-slug mapping, statuses, artifact hashes, record count, and source generation. Eligibility parity is zero symmetric difference for every fixture/filter case. Legacy-adapter parity is byte-equivalent ordered cards, refusal/error codes, and generation mapping for the existing deterministic lexical endpoint. Native-v2 quality is measured separately with eligible-set Recall@K, top-5 overlap/order (or NDCG against curated judgments), per-channel attribution, and lexical-only fallback; semantic additions must not be misclassified as migration divergence. Operational parity tracks source-to-published-generation lag, shadow query errors, p50/p95 latency, hydration success/refusal-code parity, tombstone backlog, and vector coverage. 4. **Cut over only on explicit, repeatable gates.** Require: exact census/eligibility/legacy-card parity over the complete deterministic fixture suite; zero unexplained generation mismatches or critical shadow errors over a defined observation window; no unconfirmed missing-style or tombstone backlog; lexical availability with vectors absent/degraded; p95 query latency materially below the current corpus-walk path; successful staged rebuild, cancel, crash-restart, checkpoint restore, and rollback drills; and operator sign-off on a machine-readable parity report tied to source hash, DB generation, schema version, and retrieval-profile version. The gate fails closed if the corpus changes during bootstrap or if the staged row count is incomplete, matching both current manifest verification and staged-shard publication safeguards. 5. **Rollback is an adapter/pointer reversal, never a reverse content write.** Before cutover, retain the prior DB generation, active-generation pointer, compatibility manifest, and checkpoint. A retrieval regression switches the adapter to legacy file retrieval or a prior DB generation only when that generation's source aggregate still matches the live corpus; otherwise rebuild from current files or use legacy current-file retrieval. Do not export stale DB rows back over flat files. Hydration remains fail-closed during rollback because it rechecks generation, live manifest, containment, rights, digest, and byte cap. The staged-reindex pattern demonstrates why publication must swap a complete adjacent file atomically and leave the active shard untouched on cancel/failure. 6. **Repair derived state in place only when bounded; rebuild the complete database beside the active one otherwise.** Health/repair may regenerate FTS triggers/content, vector status, or a single style projection from authoritative files under one transaction. Structural corruption, schema migration, broad parity failure, or unknown drift requires a clean staged rebuild, integrity/census/parity checks, then an atomic generation-pointer/file switch. A checkpoint restore is not final truth when files have advanced: restore under a maintenance barrier, then reindex from files and retain a needs-rebuild sentinel until every derived surface succeeds. This follows system-spec-kit's transaction-wrapped clear-and-restore behavior and its explicit post-restore rebuild sentinel. 7. **Delete through the source plane and represent absence as a lifecycle, not an immediate hard delete.** An operator delete changes/removes the flat style artifacts and authoritative crawl membership; a DB-only delete is prohibited because a rebuild would resurrect the style. One missing scan records `pending_missing`; a second independent confirmed scan or explicit source delete creates an idempotent tombstone (`deleted_at`, source generation, reason), excludes it from retrieval, and preserves identity/provenance/relationship evidence. Hard purge follows a retention period, checkpoint, no-live-reference check, and successful rebuild/parity gate. Bulk destructive operations require explicit confirmation and should checkpoint first; soft-delete rows are naturally excluded by `deleted_at IS NULL`. ## Ruled Out
- **SQLite as independently editable style-content authority:** incompatible with current file-backed hydration and creates an unimplemented authoring/export contract.
- **Dual-write content authority:** no deterministic conflict winner; partial writes can make retrieval and hydrated bytes disagree.
- **Permanent dual reads or corpus-walk fallback:** conceals index failures and preserves the performance/freshness problem.
- **DB-only deletion:** authoritative files would resurrect the row on rebuild.
- **Blind checkpoint restore as final state:** restored derived rows may be older than authoritative files and require reindex/parity verification.

## Dead Ends
- Treating “source of truth” as one store for every plane is definitively eliminated for the current system. Content/authorship and runtime retrieval have different guarantees; forcing either flat files or SQLite to own both either retains query scans or breaks hydration.
- Retiring flat files from hydration is not part of this migration. It requires a separately specified content-store contract and cannot be inferred from successful retrieval cutover.

## Edge Cases
- Ambiguous input: “after cutover” could mean query cutover or complete content migration. This iteration selects the narrower evidenced interpretation: query cutover; complete content migration is deferred.
- Contradictory evidence: none. Existing code consistently treats file bytes plus a checked generation as hydration authority, while SQLite patterns are derived/index lifecycle evidence.
- Missing dependencies: no implemented style database or production traffic baseline exists, so numeric observation-window, latency, and relevance thresholds remain implementation-plan parameters; the required metric dimensions and zero-divergence invariants are resolved.
- Partial success: none; Q5 is answered at architecture-policy level, while Q1 remains for a later iteration.

## Reflection
- What worked and why: Tracing the current manifest and hydration guards before applying system-spec-kit lifecycle patterns exposed the correct authority boundary: database retrieval can be promoted without weakening file-byte verification.
