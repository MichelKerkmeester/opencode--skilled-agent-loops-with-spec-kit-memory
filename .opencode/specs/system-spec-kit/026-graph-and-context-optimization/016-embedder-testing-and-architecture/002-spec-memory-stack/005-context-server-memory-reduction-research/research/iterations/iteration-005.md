# Iteration 5 — node-llama-cpp memory patterns (model unload after idle, context shrinking, F16 vs Q8 quantization)

## Summary

At the new 246 MB Ollama baseline, node-llama-cpp is not resident in the spec-memory MCP process; the memory risk returns when provider resolution selects `llama-cpp` for save, retry, migration, or query embedding paths. The compiled llama-cpp provider still keeps the native model and embedding context cached forever, creates an `"auto"` embedding context with option names that do not match the installed node-llama-cpp API, and treats dtype mostly as profile metadata rather than model-file selection. The measurable levers are idle disposal (~313-584+ MB recoverable after idle), dynamic context shrinking (~36 MiB raw KV cache, likely ~40-60 MB RSS when loaded), and dtype/path alignment (Q8 vs BF16 vs F32 swings local model footprint by ~271-800 MB).

## Findings

### Finding 1: The llama-cpp runtime is cached forever and has no production dispose path
- Evidence: `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js:22` defines module-level `cachedRuntime`; `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js:132` returns that runtime when the model path matches; `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js:178` stores `{ llama, model, context }` back into `cachedRuntime`. The only reset path is test-only and clears references without awaiting native disposal at `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js:342`. The shared provider interface has no lifecycle hook in `.opencode/skills/system-spec-kit/shared/dist/types.d.ts:31`, and context-server cleanup currently disposes the local reranker but not embeddings in `.opencode/skills/system-spec-kit/mcp_server/context-server.ts:1380`.
- Memory impact: Local model files on this machine are Q8_0 = 313M, BF16 = 584M, and F32 = 1.1G under `~/.cache/huggingface/gguf/embeddinggemma-300m/`. Once llama-cpp is selected, the in-process RSS can regain most of the old 1080 MB baseline shape. For Q8_0, idle disposal should recover roughly 300-450 MB depending on mmap residency and allocator retention; BF16 recovery is closer to 550-700 MB; F32 can exceed 1 GB.
- Proposed change: Add an optional provider lifecycle to the shared embedding layer, then implement it for llama-cpp. In `.opencode/skills/system-spec-kit/shared/types.ts`, extend `IEmbeddingProvider` with `dispose?(): Promise<void>` and `isLoaded?(): boolean`. In `.opencode/skills/system-spec-kit/shared/embeddings.ts`, export `disposeEmbeddingProvider()` that awaits `providerInstance.dispose?.()`, clears `providerInstance` and `providerInitPromise`, and clears any in-process embedding cache. In the llama-cpp provider source, add `disposeCachedRuntime(reason)` that awaits `cachedRuntime.context.dispose()` and `cachedRuntime.model.dispose()` before nulling `cachedRuntime`, `runtimePromise`, and `runtimePromisePath`. Track `activeInferenceCount` so idle disposal never races an embedding call. Schedule disposal after `LLAMA_CPP_EMBEDDINGS_IDLE_UNLOAD_MS`, default `600000`, with `0` disabling the idle timer. Finally, call `_embeddings.disposeEmbeddingProvider()` from context-server cleanup immediately after the local reranker cleanup step.
- Trade-off: The first embedding after an idle period pays model load latency again. It also adds lifecycle state that must be tested around concurrent embedding calls, shutdown, and hot-reload.
- Effort: M

### Finding 2: The embedding context likely allocates for the full 2048-token train window
- Evidence: The provider creates its embedding context with `contextSize: 'auto'`, `minContextSize: 512`, `maxContextSize: trainContextSize`, and `batchSize: Math.min(512, trainContextSize)` in `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js:153`. The installed node-llama-cpp type definition accepts `contextSize?: "auto" | number | { min?: number; max?: number }`, not separate `minContextSize` or `maxContextSize`, in `.opencode/skills/system-spec-kit/mcp_server/node_modules/node-llama-cpp/dist/evaluator/LlamaEmbeddingContext.d.ts:19`. The implementation forwards only `contextSize`, `batchSize`, `threads`, `createSignal`, and `ignoreMemorySafetyChecks` to model context creation in `.opencode/skills/system-spec-kit/mcp_server/node_modules/node-llama-cpp/dist/evaluator/LlamaEmbeddingContext.js:70`. GGUF metadata for the local embeddinggemma files reports `contextLength = 2048`, `embeddingLength = 768`, `blockCount = 24`, `attentionHeadCount = 3`, and `attentionKvHeadCount = 1`.
- Memory impact: Raw F16 KV cache for this embedding model is approximately `context * layers * kvHeads * headDim * K/V * bytes`, where `headDim = 768 / 3 = 256`. At 2048 tokens this is `2048 * 24 * 1 * 256 * 2 * 2 = 50,331,648 bytes`, about 48 MiB. At 512 tokens it is about 12 MiB. Shrinking the idle/default context from 2048 to 512 should save about 36 MiB raw KV cache, likely 40-60 MB RSS after allocator, graph, and batch buffers. This does not replace model unload; it is the second lever while the model remains loaded.
- Proposed change: Replace the one permanent embedding context with a small default context plus grow/shrink behavior. In the llama-cpp provider, load and cache the model separately from the context, tokenize input before context creation, and choose a context size class from the token count. Use the API shape node-llama-cpp actually supports, for example `model.createEmbeddingContext({ contextSize: { min: targetContextSize, max: targetContextSize }, batchSize: resolvedBatchSize })`. Suggested defaults: `LLAMA_CPP_EMBEDDINGS_CONTEXT_MIN=512`, `LLAMA_CPP_EMBEDDINGS_CONTEXT_STEP=256`, `LLAMA_CPP_EMBEDDINGS_CONTEXT_MAX=0` meaning model train context, and `LLAMA_CPP_EMBEDDINGS_BATCH_SIZE=128` or `256`. For long documents, grow to the nearest class up to 2048 so no feature is removed. After `LLAMA_CPP_EMBEDDINGS_CONTEXT_SHRINK_IDLE_MS=60000`, dispose the oversized context and recreate the 512-token context on the next short request.
- Trade-off: Long inputs pay context creation cost when they exceed the resident size. The provider becomes more complex because model lifetime and context lifetime are separate. The upside is that the common short-query path no longer reserves a full-context KV cache all day.
- Effort: M

### Finding 3: `LLAMA_CPP_EMBEDDINGS_DTYPE` does not currently select the GGUF file
- Evidence: The compiled provider defaults to `embeddinggemma-300M-Q8_0.gguf` at `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js:17`. `normalizeModelPath()` only uses `LLAMA_CPP_EMBEDDINGS_MODEL_PATH` or that default path in `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js:44`. `resolveLlamaCppDtype()` reads and normalizes `LLAMA_CPP_EMBEDDINGS_DTYPE` in `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js:51`, but the value is surfaced through profile metadata at `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js:319`; it does not change the selected GGUF file. The local model files are Q8_0 = 313M, BF16 = 584M, and F32 = 1.1G, and all three report 768 embedding dimensions and 2048 context length.
- Memory impact: Accidentally using BF16 instead of Q8_0 adds about 271 MB of model-file footprint before allocator and mmap effects. F32 adds about 800 MB over Q8_0. Either choice can erase the entire 246 MB post-POST-006 process improvement if loaded in-process. Conversely, changing the dtype env var alone does not reduce memory today because it does not select a different model file.
- Proposed change: Add a single `resolveLlamaCppModelSelection()` helper used by availability checks, provider construction, metadata, and logs. If `LLAMA_CPP_EMBEDDINGS_MODEL_PATH` is set, use it and infer dtype from GGUF metadata or basename; warn when it conflicts with `LLAMA_CPP_EMBEDDINGS_DTYPE`. If no path is set, map dtype to file name: `q8` or `q8_0` -> `embeddinggemma-300M-Q8_0.gguf`; `bf16` or `f16` -> `embeddinggemma-300M-BF16.gguf` until a true F16 file exists; `f32` -> `embeddinggemma-300M-F32.gguf`. Add `bf16` to the allowed dtype list. On load, log `provider=llama-cpp model_file=<basename> dtype=<inferred> size_mb=<model.size/1048576> context_size=<actual> gpu_layers=<value>`.
- Trade-off: This makes dtype selection more explicit and slightly more code-heavy. It also reveals when a user has configured a high-memory model, but it does not restrict that choice.
- Effort: S

### Finding 4: The llama-cpp memory path is source/dist drifted, so durable fixes need a source-owned provider first
- Evidence: `context-server.js` imports the compiled shared factory from `@spec-kit/shared/embeddings/factory` in `.opencode/skills/system-spec-kit/mcp_server/dist/context-server.js:40`, and the compiled factory still supports `llama-cpp` in `.opencode/skills/system-spec-kit/shared/dist/embeddings/factory.js:45`. The current TypeScript factory excludes `llama-cpp` from `SupportedProviderName` in `.opencode/skills/system-spec-kit/shared/embeddings/factory.ts:66` and from `SUPPORTED_PROVIDERS` in `.opencode/skills/system-spec-kit/shared/embeddings/factory.ts:97`. The compiled llama-cpp provider source map points to `../../../embeddings/providers/llama-cpp.ts`, but that source file is absent in the current tree.
- Memory impact: This does not add RSS by itself, but it is a delivery risk for Findings 1-3. Patching compiled JS would be overwritten by the next TypeScript build; patching only current TS would not affect the live compiled llama-cpp path unless the missing provider is restored or the factory is reconciled.
- Proposed change: Before implementing memory changes, reconcile shared embeddings source and dist. Reintroduce `.opencode/skills/system-spec-kit/shared/embeddings/providers/llama-cpp.ts` from the compiled provider semantics, then apply the lifecycle, context-sizing, and dtype-selection changes there. Align `shared/embeddings/factory.ts`, `shared/dist/embeddings/factory.d.ts`, and the compiled output so `voyage`, `ollama`, and `llama-cpp` are all explicit pluggable providers. Do not hand-edit `dist/` as the only source of truth.
- Trade-off: This is architectural cleanup before memory work lands. It increases implementation scope, but without it the llama-cpp fixes are easy to lose or impossible to validate through the normal build.
- Effort: M

## Cross-references

This iteration updates Iteration 1's llama-cpp idle-unload finding for the new 246 MB baseline: the recommendation still matters, but the RSS savings are now conditional on the user selecting llama-cpp rather than always present. It also builds directly on Iteration 2's proposed provider lifecycle hook; the concrete target is now `IEmbeddingProvider.dispose?()`, shared `disposeEmbeddingProvider()`, and context-server shutdown wiring. Iterations 3 and 4 already covered persistent embedding-cache and SQLite RSS levers, so this file intentionally avoids repeating cache-size, mmap-size, and WAL tuning except to note that those remain independent of llama-cpp model residency. The POST-006 Ollama change is complementary: Ollama keeps model memory outside `context-server.js`, while these changes prevent the in-process provider from recreating the old resident-memory profile.

## Negative knowledge (ruled-out)

- Do not count the current 246 MB RSS as proof that llama-cpp is optimized. It only proves the active Ollama path is not loading node-llama-cpp in-process.
- Do not rely on `LLAMA_CPP_EMBEDDINGS_DTYPE` alone for memory reduction. In the compiled provider, dtype changes metadata but not the model path.
- Do not make smaller context size a hard cap. That would reduce features for long documents. The safe version is dynamic grow/shrink up to the model train context.
- Do not default to experimental KV-cache quantization as the main optimization. node-llama-cpp exposes experimental cache type knobs, but context sizing is lower risk and enough to recover the obvious 40-60 MB while preserving model choice.
- Do not use `useMlock` for this daemon. It intentionally pins model memory and works against idle recovery. `useDirectIo` is also not a good default for this environment because node-llama-cpp documents it as unsupported on macOS.
- Do not patch only `.opencode/skills/system-spec-kit/shared/dist/embeddings/providers/llama-cpp.js`. The compiled artifact is currently the live path, but the missing TypeScript source means such a patch would be brittle.

## Open questions

- How much RSS does `context.dispose()` plus `model.dispose()` return on macOS after a real Q8_0 load? The expected recovery is hundreds of MB, but allocator retention and mmap page cache should be measured with an unsandboxed harness.
- Should the default idle unload be 10 minutes, 2 minutes, or tied to MCP activity? Ten minutes avoids repeated cold starts during active editing sessions; two minutes maximizes memory recovery.
- Should dynamic contexts keep only one current context, or an LRU of size classes? One context minimizes RSS; a tiny LRU reduces latency for alternating short and long documents.
- Should llama-cpp be represented in the newer embedder registry alongside Ollama, or remain in the shared factory only? Either can preserve model pluggability, but the current split makes memory behavior harder to reason about.
