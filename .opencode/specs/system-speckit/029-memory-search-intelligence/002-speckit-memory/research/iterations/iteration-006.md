### Q10 PROPER — Four cross-cutting spines -> sibling-subsystem generalizations
These are the seed for the **parent roadmap** (028 parent). Non-goal reminder (strategy line 46): this packet does NOT touch the siblings; it maps *where* each spine would land so the parent can scope sibling phases.

**Spine (i) — Determinism / reproducibility** [members: C5-A/B serialization, C-X1 bonus-over-channels, C6-A caller-clock decay, C4-B content-addressed IDs]
- **-> code-graph:** code-graph also blends multiple ranked signals (callers, imports, degree, recency) and renders result sets a model caches against. The same two patterns transfer verbatim: (a) the **content-derived render-order stage** (C5-A/B) so identical structural queries render byte-identically (stable prompt-cache prefix on graph results), and (b) the **G2 "apply-exactly-once" invariant** [F5-14] as a single choke point so a signal is applied at fusion-time XOR post-fusion, never both. Content-addressed `derived_id` (C4-B) maps directly to code-graph's derived edges (call edges, import edges) so re-scans are idempotent no-ops.
- **-> skill-advisor:** the advisor's recommendation ranker also fuses signals (trigger-phrase match, graph proximity, recency). C-X1's `{bonusOverChannels:'active'|'configured'}` option is the *transferable caution*: any RRF-style multi-source bonus inherits "turning a channel off is not score-neutral." A shared `fuseResultsMulti` option lets advisor profiles zero channels without distorting survivors. Same caller-clock determinism (C6-A) applies to any recency term the advisor computes.
- **-> deep-loop:** convergence detection is itself a determinism problem — `newInfoRatio` and reducer state must be reproducible across re-runs. Content-derived ordering (C5) and the apply-once invariant generalize to how deep-loop reducers fold iteration records (stable ordering of findings -> stable convergence verdict).

**Spine (ii) — Bi-temporal edge-based currentness** [members: C3-A flip-on, C3-B four-timestamp, C3-C TemporalMode+provider, C3-D tombstone-vs-temporal-close separation]
- **-> code-graph:** the strongest sibling match. Code structure is intrinsically bi-temporal — a symbol/edge is valid for a range of commits (event-time) and was indexed at a scan-time (transaction-time). The internal causal-edge `valid_at`/`invalid_at` + `supersedes`/`contradicts` model (C3-A) is directly the template for "this call edge was superseded by a refactor at commit X." C3-D's separation (temporal-close = "not current in HEAD, History-readable" vs tombstone = "file deleted, off-state") maps cleanly to code-graph's renamed/deleted-symbol handling. The four-timestamp window (C3-B) and `TemporalMode` (C3-C: AsOf-commit / History) would let code-graph answer "what did the call graph look like at commit X" — a genuinely new capability seeded here.
- **-> skill-advisor:** lighter touch — skill recommendations have a currentness notion (a skill SKILL.md is superseded by an edit). `supersedes` edges + edge-presence currentness apply if the advisor graph tracks skill-version edges.
- **-> deep-loop:** marginal — deep-loop state is append-only by design; the currentness model is less applicable, though the `supersedes` pattern could mark a later iteration's finding as superseding an earlier contradicted one.

**Spine (iii) — Query-class routing** [members: C2-A classifier, C2-B per-class weights, C2-C graph-gating]
- **-> code-graph:** code-graph queries also have *retrieval shapes* — "find exact symbol" (single-hop, graph-expansion OFF) vs "trace impact/callers" (multi-hop, graph-expansion ON) vs "find by entity name." The C2-A classifier + C2-C graph-gating pattern maps almost 1:1: a code-graph query-class router would gate transitive expansion by class, fixing the same "indiscriminate graph expansion hurts single-hop precision" problem [iter-1 #12] in the structural-search ranker. **This is the highest-value cross-subsystem transfer** — code-graph has the same graph-over-expansion failure mode.
- **-> skill-advisor:** advisor prompts also have shapes (single-skill lookup vs multi-skill workflow vs "how do I"). Per-class weight profiles (C2-B) into the advisor's fusion would tune trigger-vs-graph-vs-recency weighting by prompt shape. Additive third axis, same as memory.
- **-> deep-loop:** marginal/none — deep-loop dispatches by mode (research/review/council), not by query shape; the routing axis doesn't generalize cleanly.

**Spine (iv) — Graceful degradation** [members: C9 embedder-degrade]
- **-> code-graph:** code-graph ALREADY degrades (the `memory-context.ts:1513` "code graph unavailable -> hybrid degrades to semantic-only" path [iter-3 #8] is the *consumer* of code-graph's own degrade). The C9 pattern — detect-unavailable, drop the dependent channel, report a flag instead of throwing — is the reusable contract: code-graph should report `graph_available:false` rather than fail when its index is cold/missing. Daemon exit-75 (retryable IPC unavailability, per the CLI front-door contract) is the same "degrade, don't throw" discipline at the transport layer.
- **-> skill-advisor:** the advisor daemon's warm-only/exit-75 fallback is *already* this spine in practice. C9 formalizes the in-process version: when the advisor's embedding/graph component is unavailable, degrade to trigger-phrase-only matching + report a flag, never throw.
- **-> deep-loop:** generalizes as iteration-level resilience — a deep-loop iteration whose tool/source is unavailable should degrade to a narrower evidence path and report status (`timeout`/`stuck`) honestly rather than failing the whole loop. This agent's own status-honesty contract is the deep-loop instance of spine (iv).

### Cross-cutting summary (seed for the parent roadmap)
- **code-graph is the heaviest beneficiary** — it is touched by ALL FOUR spines, and is the strongest match for spine (ii) bi-temporal currentness (commit-time = event-time) and spine (iii) query-class routing (same graph-over-expansion failure mode). The parent roadmap should prioritize a code-graph phase.
- **skill-advisor is touched by all four** but more lightly — mostly via the shared `fuseResultsMulti`/RRF substrate (C-X1, C2-B) and the already-live degrade/exit-75 discipline (C9).
- **deep-loop is touched by (i) and (iv)** meaningfully (determinism of convergence; iteration-level degradation), marginally by (ii)/(iii).
- The single most-transferable artifact is the **shared `fuseResultsMulti` `{bonusOverChannels}` option (C-X1)** plus the **apply-exactly-once G2 invariant** — both live in `shared/algorithms/rrf-fusion.ts` and `stage2-fusion.ts`, are already shared-library code, and underpin determinism across every sibling ranker.
