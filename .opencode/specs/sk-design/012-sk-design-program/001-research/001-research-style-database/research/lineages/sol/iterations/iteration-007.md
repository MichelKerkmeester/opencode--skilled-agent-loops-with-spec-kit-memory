5. Style retrieval presently needs only: (a) direct curated `SIMILAR_TO` neighbors with rationale, (b) shared-facet/axis/industry joins, and (c) semantic nearest neighbors from embeddings. Similarity is not transitive, so “A similar to B similar to C” must not become a default relevance claim. Direct neighbors and shared attributes are one-hop SQL joins; vector similarity is already a ranked projection. A normalized `style_relationships(source_style_id, relation, target_style_id NULL, raw_target_label, rationale, resolution_status, confidence, source_hash, generation_id)` table with indexes on `(source_style_id, relation)` and `(target_style_id, relation)` is sufficient now. 6. **Technology recommendation:** ship SQLite + FTS5 + profile-addressed embeddings, with normalized relational edges in the same published SQLite generation. This minimum topology keeps eligibility, metadata, FTS, relationship identity, and generation publication in one transactional snapshot; the vector shard remains a rebuildable generation/profile-bound projection. Compared with it, a dedicated graph database adds native variable-path querying but also a second availability domain, schema/migration surface, backup/restore path, authentication/runtime dependency, projection lag, and cross-store cutover failure modes without a demonstrated style query that benefits. A hybrid with a graph projection preserves relational authority but pays those duplication costs, so it is an optional future topology, not the launch topology. ## Ruled Out
- Dedicated graph database in the minimum topology: no current style query needs native variable-depth traversal, and the corpus relationship targets are mostly unresolved.
- Treating every `similar.business` label as a canonical edge: 60.96% are ambiguous or unresolved by unique canonical name.
- Materializing shared facet or embedding similarity as authoritative graph edges: both are derived query signals and would create avoidable churn.
- Default transitive similarity expansion: the relationship semantics do not support transitivity.

## Dead Ends
- “Graph data requires a graph server” is eliminated by both named in-repo implementations: they use indexed SQLite adjacency tables and bounded application/SQL queries.
- Evaluating native graph algorithms before canonical relationship resolution and a concrete multi-hop product query would benchmark an invented workload.

## Edge Cases
- Ambiguous input: “graph database” could mean any node/edge model or a dedicated graph engine. The comparison uses the dedicated-engine meaning; the in-repo SQLite graph projections are treated as evidence for the relational-edge option.
- Contradictory evidence: none.
- Missing dependencies: none; no external graph-service benchmark was needed because selecting one is a stated non-goal and no qualifying workload exists.
- Partial success: none.

## Reflection
- What worked and why: reading the graph storage and query modules together separated graph data modeling from graph-engine necessity; quantifying the corpus's actual `similar` payload prevented a topology recommendation based on assumed relationship quality.
- What did not work and why: a broad relationship-key search was noisy because `similar` appears in nearly every canonical file; the bounded parser converted that noise into resolution and topology measurements.
- What I would do differently: benchmark a concrete bounded SQLite relationship query only after target normalization rules and a user-facing multi-hop query are specified; benchmarking synthetic traversal now would manufacture demand.
