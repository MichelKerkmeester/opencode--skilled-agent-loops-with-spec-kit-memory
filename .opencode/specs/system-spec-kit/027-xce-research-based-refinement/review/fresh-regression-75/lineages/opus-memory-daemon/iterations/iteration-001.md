### Areas that held up (no finding)

- **Fail-closed secret scrubbing** — `shared/parsing/secret-scrubber.ts` throws `SecretScrubberError` on internal failure (`scrubSecretsDetailed` lines 220-226), restores the prototype chain so `instanceof` survives transpilation (line 27), and both durable call sites let it propagate to refuse the write: `atomicSaveMemory` (`handlers/memory-save.ts:3799`, uncaught at entry) and `parseMemoryContent` (`lib/parsing/memory-parser.ts:276`, uncaught). `memory-crud-update.ts:187` explicitly catches and refuses. Scrub runs before content-hash/embedding/FTS. Contract honored. No ReDoS in the patterns (all single-class quantifiers anchored by literals).
- **Causal-cache generation bump** — both mutators bump *before* clearing per-db caches (`causal-edges.ts:215`, `causal/sweep.ts:216`), exactly one bump per mutation (batch/bulk/delete paths pass `shouldInvalidateCache=false`/`invalidateCaches:false` then bump once). Process-local scope is documented intent, not a bug.
- **Tombstone-sweep rollback** — `sweepCausalEdges` inserts all tombstones then deletes all edges inside one `runInTransaction` (atomic; partial failure rolls back both). `nextLifecycleGeneration` is computed sequentially so same-identity edges in one sweep get unique, monotonic generations. Full edge snapshot persisted in `restore_metadata`. Sound.
- **Idempotency receipts** — `ON CONFLICT DO NOTHING` first-writer race, conflict-vs-replay branching, and best-effort eviction are carefully handled.

### Finding (P1) — MCP-server save mutex has no liveness gate or heartbeat

`handlers/save/spec-folder-mutex.ts` exists to enforce single-writer per spec folder (header: "prevent concurrent indexing races (TOCTOU)"). Its stale-reap is **time-only**:

- `isStaleLock` (lines 37-44) reaps purely on `Date.now() - stats.mtimeMs > LOCK_STALE_MS` (5 min, line 18).
- It writes `owner.json` with `pid` (lines 63-67) but **never reads it back** — no `process.kill(pid, 0)` liveness check (confirmed by repo-wide grep).
- **No heartbeat**: the lock dir mtime is set once at `mkdir` and never refreshed, so the 5-min clock starts at acquisition and never resets while a save runs.

Consequence: a save that legitimately holds the lock past 5 min (large/chunked embedding under a slow or retrying provider — all of which run *inside* `fn()` per `atomic-index-memory.ts:320-411`) has its lock **reaped from a live owner** (lines 74-77) by a second process, which then `mkdir`s and proceeds — two concurrent indexers on the same spec folder, the exact TOCTOU the module is meant to prevent. This epic *added* the multi-process conditions that make it reachable: daemon re-election (two daemons briefly coexist) and the CLI front-doors (separate writer processes).

This is not speculative: the project already confirmed and fixed the **identical defect class** in the sibling save lane. Commit `7c17b07e01` ("save-lock liveness") rewrote `scripts/memory/generate-context.ts` to do exactly what's still missing here — pid liveness via `process.kill(ownerPid, 0)` (`getCanonicalSaveLockOwnerState`, lines 424-436), a `utimesSync` heartbeat (`startCanonicalSaveLockHeartbeat`, lines 448-461), and reap only when `ownerState === 'dead'` or `'unknown' && stale` (line 485). Its commit message states the bug verbatim: "a 30s mtime-stale reap could steal the lock from a LIVE owner mid-save (locks were written once, never refreshed), defeating the single-writer guarantee." That fix did **not** touch `spec-folder-mutex.ts` (last modified 2026-05-14, a rename; the fix landed 2026-06-11) — the MCP-server lane was missed. Same defect, fixed in one lane, left in the other.

Recommendation: port the generate-context.ts pattern into `spec-folder-mutex.ts` — read `owner.json.pid`, gate reap on `process.kill(pid, 0)` (reap immediately on `ESRCH`/dead; only fall back to the time threshold when owner state is unreadable/unknown; never reap an alive pid), and start a `utimesSync` heartbeat on the lock dir for the duration of `withSpecFolderLock`'s `fn()`.
