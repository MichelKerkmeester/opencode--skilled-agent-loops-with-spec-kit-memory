# /create:command Presentation Contract

This file is the single source of truth for user-facing presentation in `/create:command`: startup questions, setup/status dashboards, and result display. The command router owns asset routing only. The workflow YAML owns execution behavior.

## 1. Phase 0 Verification Display

Run this automatic self-check before setup. This is not a user question.

```text
SELF-CHECK: Are the create-command packet resources available?

Required resources:
- .opencode/skills/sk-doc/create-command/SKILL.md
- .opencode/skills/sk-doc/create-command/assets/command-template.md
- .opencode/skills/sk-doc/create-command/assets/command-presentation-template.md
- .opencode/skills/sk-doc/create-command/references/router-presentation-split.md
- .opencode/skills/sk-doc/create-command/references/argument-hints-and-modes.md

If all resources exist:
- create_command_verified = true
- Continue to setup

If any resource is missing or uncertain:
- Stop before loading workflow YAML
- Display the hard-block message below
```

Hard-block message:

```text
CREATE-COMMAND PACKET REQUIRED

This command requires verified create-command resources for:
- Command invocation and path rules
- Argument-hint and allowed-tools standards
- Router/presentation separation
- Canonical command and presentation templates

Missing resource: [path]

STATUS=FAIL ERROR="Create-command packet resource missing"
```

## 2. Auto Setup Resolution

For `:auto`, resolve setup using the system auto-mode contract, then load `create-command-auto.yaml` only when every required field is available.

Resolution tiers:

| Tier | Use When | Output |
| --- | --- | --- |
| Tier 1 | `$ARGUMENTS`, flags, pre-bound answers, and defaults resolve all required fields | Persist config and load auto workflow |
| Tier 2 | One or two required fields remain genuinely ambiguous | Ask one narrow question per missing field |
| Tier 3 | Required fields remain unresolved | Emit missing-inputs error and do not load YAML |

Pre-bound setup schema:

```yaml
PRE-BOUND SETUP ANSWERS:
  command_invocation: /create:example
  command_request: Create a split workflow command that ...
  command_path: .opencode/commands/create/example.md
  package_shape: split_4_file_set
  spec_choice: existing
  spec_path: <spec-folder>
  execution_mode: AUTONOMOUS
  memory_choice: skip
```

Default resolution table:

| Field | Required | Resolves Via | Default | Tier-2 Candidate |
| --- | --- | --- | --- | --- |
| `command_invocation` | Yes | Positional invocation or marker `command_invocation` | none | No |
| `command_request` | Yes | Remaining positional text or marker `command_request` | none | Yes, when purpose is ambiguous |
| `command_path` | Yes | Derived from invocation, `--path`, or marker `command_path` | derived path under `.opencode/commands/` | No |
| `package_shape` | Yes | marker `package_shape` or workflow default | `split_4_file_set` | No |
| `spec_choice` | Yes | marker `spec_choice` or targeted choice | none | Yes, when folder choice is ambiguous |
| `spec_path` | Conditional | marker `spec_path`, derived from `spec_choice`, or null when skipped | none | No |
| `execution_mode` | Yes | `:auto`, `:confirm`, or marker `execution_mode` | `AUTONOMOUS` under `:auto` | No |
| `memory_choice` | No | marker `memory_choice`, prior-session detection, or default | `skip` when no prior continuity exists | No |

## 3. Consolidated Startup Prompt

Ask one prompt containing only applicable questions. Do not split these questions across turns.

```text
Before proceeding, please answer:

Q0. Command Invocation (if not provided)
What OpenCode slash command should be created or updated?
Format: /command or /namespace:action, for example /create:widget or /doctor:cache.

Q1. Command Workflow
What should this command do, which arguments should it accept, and which tools does it need?

Q2. Spec Folder
A) Use existing: [suggest if related found]
B) Create new spec folder
C) Update related spec: [if partial match found]
D) Skip documentation

Q3. Execution Mode (if no :auto/:confirm suffix)
A) Interactive - confirm at each step
B) Autonomous - execute without prompts

Q4. Prior Session Context (if existing spec has continuity)
A) Load handover plus implementation-summary continuity
B) Load canonical spec docs up to a recent cap
C) Skip

Reply with answers, for example: /create:widget, "builds widget command with :auto/:confirm", A, A, C
```

Hard stops:

- Do not proceed until all applicable questions are answered.
- Do not auto-create spec folders without explicit confirmation.
- Do not auto-select execution mode without suffix or explicit choice.
- Do not infer command purpose from context, screenshots, or conversation history.
- Do not wire resource paths unless they exist on disk.

## 4. Setup Dashboard

After parsing setup, display or maintain this dashboard internally before loading YAML.

| Field | Required | Value | Source |
| --- | --- | --- | --- |
| `create_command_verified` | Yes | `[value]` | Phase 0 self-check |
| `command_invocation` | Yes | `[value]` | Q0 or `$ARGUMENTS` |
| `command_request` | Yes | `[value]` | Q1 or `$ARGUMENTS` |
| `command_path` | Yes | `[value]` | Derived invocation or `--path` |
| `package_shape` | Yes | `[value]` | Workflow default or marker |
| `spec_choice` | Yes | `[value]` | Q2 |
| `spec_path` | Conditional | `[value]` | Derived from Q2 |
| `execution_mode` | Yes | `[value]` | Suffix or Q3 |
| `memory_loaded` | Conditional | `[value]` | Q4 when applicable |

Proceed only when every required field has a value.

The derived package uses a kebab-case router stem and sibling assets named `<stem>-auto.yaml`, `<stem>-confirm.yaml`, and `<stem>-presentation.txt`.
