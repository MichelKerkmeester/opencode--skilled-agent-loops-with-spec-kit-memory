# Iteration 2: system-spec-kit persistence and retrieval lifecycle

## Focus
This iteration traced the in-repo system-spec-kit metadata database, FTS5 projection, vector shard, embedding cache/status lifecycle, incremental scanner, transactional mutations, and hybrid query pipeline. The requested `mcp_server` spelling resolves in this checkout to the canonical code path `mcp-server`; `mcp_server/database/` contains runtime SQLite artifacts rather than schema source, so schema ownership was traced through the directly connected TypeScript modules.

## Findings
1. **Schema ownership is explicit, centralized, and migration-versioned.** `vector-index-schema.ts` owns a current schema version of 41 and records the evolution of indexing, lineage, idempotency, tombstone, and retention features; `ensure_schema_version()` creates a singleton version row, runs migrations, validates backward compatibility, and only then advances the version. A style database should reuse this contract—one authoritative schema module, monotonic migrations, compatibility checks, and a persisted version—but should own a separate style schema rather than importing the memory store's domain-heavy `memory_index`. 2. **The primary row is metadata/source text; FTS and vectors are derived stores keyed by the same integer row id.** `memory_index` carries path identity, content hash, mtime, embedding status/model, source text, tombstone state, and provenance; FTS5 is an external-content table linked by `content_rowid='id'`, while sqlite-vec uses the row id in `vec_memories`. For styles, the reusable invariant is one stable canonical style id joined across projections, but the current UUID-first style contract means the style table should preserve UUID as domain identity and keep slug/path as unique locators rather than copying the memory store's path-derived logical identity. 3. **FTS consistency is database-enforced, but its document shape is memory-specific.** Insert/update/delete triggers maintain the external-content `memory_fts` projection atomically, and queries join it back to active `memory_index` rows before applying folder/tombstone filters and weighted SQLite `bm25()`. This trigger-backed projection is reusable; the mismatch is its four memory fields (`title`, `trigger_phrases`, `file_path`, `content_text`). A style-owned FTS table needs intentional columns for title/name, summary/description, tags/facets/capabilities, and searchable design text so column weights represent style semantics rather than memory semantics. [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/vector-index-schema.ts:399-432] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/sqlite-fts.ts:150-205]
4. **Embedding lifecycle separates metadata state, model profile, cached payloads, and the active vector shard.** The metadata row records `pending|success|failed|retry|partial`; deferred records remain FTS/BM25-searchable; an attached shard stores profile metadata, a content-addressed embedding cache keyed by hash/profile/input-kind/model/dimensions, a dimension-specific fallback table, and a sqlite-vec virtual table when available. The embedder helper persists an active provider/model/dimension pointer transactionally and validates dimensions. Styles can reuse these profile/status/cache contracts, but should embed a deterministic retrieval document assembled from style metadata and selected artifacts—not raw memory text—and must invalidate by both style content hash and embedding profile. 5. **Incremental ingest is a two-level freshness protocol with retry safety.** It first classifies paths by existence and mtime, but even an unchanged mtime is checked against SHA-256 when a stored hash exists; pending/failed embeddings force reindex. Discovery also compares indexed paths to the current scan to identify deletions. Crucially, batch mtimes are updated only after successful indexing, so failures remain retryable. The style mismatch is granularity: each style record is assembled from multiple artifacts and a manifest projection, so its content hash must be a canonical aggregate over all retrieval-relevant files/fields rather than only one file's bytes; mtime can remain a fast hint but not the authority. 6. **Primary SQL/vector mutations are transactional; some secondary indexes are deliberately rebuildable.** Insert/update writes metadata, active projection, vector payload, and cache invalidation inside `better-sqlite3` transactions; FTS follows through SQL triggers. Hard delete transactionally removes vector payload, ancillary rows, embedding-cache entries, and the source row, while in-memory BM25 cleanup is best-effort after commit. The public delete path normally tombstones rows in a transaction and preserves graph edges when soft-delete semantics are enabled. A style store should copy the boundary: source row + persistent FTS/vector metadata update atomically, with in-memory ranking caches treated as disposable and health-checked/rebuildable; it must choose and document whether removed flat styles tombstone first or hard-delete only after a retention window. [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/vector-index-mutations.ts:367-420] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/vector-index-mutations.ts:534-711] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/vector-index-mutations.ts:739-800] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/handlers/memory-crud-delete.ts:90-185]
7. **Query fusion is channel-oriented and fail-open, not a single SQL ranking expression.** Vector, FTS, BM25, trigger, graph, and degree lanes execute sequentially because `better-sqlite3` is synchronous; channel failures are recorded without blocking other lanes. FTS and BM25 are deduplicated into a keyword lane, then adaptive intent/document weights or multi-list RRF combine ranked lists. For styles, the reusable API shape is explicit lane options plus per-result source attribution; the mismatch is that memory-specific intent, recency, cognitive, graph-degree, and trigger machinery is excessive for the initial migration. The narrow first contract should apply existing eligibility/rights constraints as hard prefilters, fuse persistent FTS5 and vector lanes, expose scores/provenance, and keep graph as an optional later lane. ## Ruled Out
- Reusing `memory_index` directly for styles: its schema and lifecycle encode memory-specific governance, cognition, chunking, lineage, and retention concerns.
- Treating mtime as authoritative freshness: system-spec-kit itself retains a content-hash check and retry states even on the mtime fast path.
- Making vector availability a prerequisite for style retrieval: deferred memory rows demonstrate a valid lexical-only searchable state.
- Updating an external FTS table manually in application code: external-content FTS triggers provide a stronger atomic consistency boundary.

## Dead Ends
- Inspecting `mcp_server/database/` as schema source yielded only live SQLite databases, vector shards, lock/journal files, and logs; schema definitions are owned by `mcp-server/lib/search/vector-index-schema.ts` and connected modules.
