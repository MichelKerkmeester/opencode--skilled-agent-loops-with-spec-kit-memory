# Iteration 9 — gpt-5.6-sol high — focus: catalogs

Headline finding: each cataloged hub needs one hub-level **Compiled Hub Routing** entry describing its effective serving state—not duplicated child-mode entries or implementation history. Its wording must change during the same P4 stage as that hub’s directive because the current directives still describe opt-in, default-off behavior, while P4 changes the default and requires verified rollback after each hub (`.opencode/skills/sk-code/SKILL.md:52-56`, `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/plan.md:172-179`).

The canonical root entry should contain:

- **Description:** “Selects this hub’s workflow mode through the compiled resolver when the effective activation and eligibility gates permit it; otherwise retains the legacy hub router.”
- **Current Reality before P4:** “Compiled routing is additive and opt-in with `SPECKIT_COMPILED_ROUTING=1`; legacy remains authoritative when disabled or when the compiled front door returns its legacy sentinel.” This matches every current hub directive, including `sk-prompt` and `cli-external-orchestration` (`.opencode/skills/sk-prompt/SKILL.md:36-40`, `.opencode/skills/cli-external-orchestration/SKILL.md:41-45`).
- **Current Reality after that hub’s P4 cutover:** “Compiled routing is the default authority for a fresh eligible manifest. `SPECKIT_COMPILED_ROUTING=0` is the fleet kill-switch; missing or stale eligibility routes to the classified legacy fallback.” The planned observable states are `compiled-serving`, `legacy-fallback`, and `drifted` (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/plan.md:98-107`).
- **Source Files:** link to the promoted stable front door, shared eligibility/status implementation, the hub’s root `SKILL.md`, and its durable activation manifest. Do not anchor the catalog to the current resolver beneath `.opencode/specs/`; catalog review explicitly rejects mutable spec/phase history as a runtime source (`.opencode/skills/sk-doc/create-feature-catalog/SKILL.md:343-351`).
- **Validation:** one automated parity/status test row and one manual-playbook row; link the durable Lane C benchmark report under related references. Catalogs describe behavior and link to manual execution detail rather than embedding scenario matrices (`.opencode/skills/sk-doc/create-feature-catalog/SKILL.md:311-324`).

The per-feature file should be `routing-and-activation/compiled-hub-routing.md`, titled `Compiled Hub Routing`, with H1 `# Compiled Hub Routing (compiled-route.cjs)`. Use three short `HOW IT WORKS` paragraphs: core routing outcome, activation/rollback, then status/failure classification. Its required implementation table, validation table, metadata, and root-entry linkage follow `.opencode/skills/sk-doc/create-feature-catalog/SKILL.md:274-305`. Use hub-qualified triggers such as `"<hub> compiled hub routing"`, `"<hub> compiled route"`, and `"SPECKIT_COMPILED_ROUTING <hub>"`.

A catalog leaf cannot be dropped into a hub as an isolated cross-cutting document. The workflow requires a root catalog, category, root entry, and one linked per-feature file, while the root presents itself as the canonical current inventory (`.opencode/skills/sk-doc/create-feature-catalog/SKILL.md:211-224`, `.opencode/skills/sk-doc/create-feature-catalog/assets/feature-catalog-template.md:93-115`). For hubs without a complete root catalog, either fund the full inventory package or document compiled routing centrally in `system-skill-advisor`; do not create a one-feature pseudo-catalog.

The central `advisor_recommend` leaf also needs a precise update. Its current response description omits `compiledRoute` (`.opencode/skills/system-skill-advisor/feature-catalog/mcp-surface/advisor-recommend.md:20-26`), while the handler attaches that field only for the seven allowlisted hubs, only when the flag equals `1`, and suppresses it on legacy sentinel or execution failure (`.opencode/skills/system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:41-49`, `.opencode/skills/system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:326-371`).

Ranked actions:

1. Add the phase-specific root entry and leaf contract above to the implementation spec; make its post-P4 wording part of each hub’s lockstep cutover.
2. Update `advisor-recommend.md` with the exact optional-field conditions and corresponding automated/manual anchors.
3. Decide explicitly which hubs receive complete catalog packages; prohibit orphan leaves and incomplete “complete inventory” roots.
4. Fix a newly found template/runtime contradiction: the snippet template says feature-catalog triggers drive advisor routing, but the actual harvester scans only `references/` and `assets/`. Either extend harvesting to canonical `feature-catalog/` trees or correct the template claim (`.opencode/skills/sk-doc/create-feature-catalog/assets/feature-catalog-snippet-template.md:35-54`, `.opencode/skills/system-skill-advisor/feature-catalog/auto-indexing/doc-frontmatter-harvest.md:20-37`).
