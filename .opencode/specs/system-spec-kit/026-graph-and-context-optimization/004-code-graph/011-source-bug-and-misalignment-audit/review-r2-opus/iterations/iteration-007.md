why: The BUG-01 fix added `|| readiness.verificationGate === 'fail'` to shouldBlockReadPath in query.ts (875) and context.ts (73). getVerificationGate (ensure-ready.ts:359-391) is independent of freshness: a fully-fresh graph whose last gold battery `passed===false` yields freshness:'fresh', verificationGate:'fail' — a real, plausible state. In that case shouldBlockReadPath returns true, but buildFallbackDecision (query.ts:878-898) / buildContextFallbackDecision (context.ts:92-114) both return null (neither the freshness==='error' nor the freshness!=='fresh' branch matches). buildBlockedReadPayload therefore ships message `code_graph_full_scan_required`, blockReason:'full_scan_required', and NO fallbackDecision. The operator is told to run code_graph_scan, but a re-scan does not re-run gold verification and cannot clear a failed gate (the parallel Opus review IT02-CORR-01 [P1] confirms verificationGate is captured once pre-scan at ensure-ready.ts:632 and reused stale post-scan at 729/763, so freshness stays 'fresh' and the gate stays 'fail' — an infinite wrong-remediation loop). detect-changes.ts handles the identical state correctly: readinessRequiresBlock(105) blocks, and the block message (256-262) surfaces blockedFreshness='fresh-with-failed-verification' so operators see the true cause. The three read handlers are NOT aligned on this state despite the fix comments claiming a 'single contract matching detect_changes'. The block itself is the correct safe behavior (no wrong structural answer is returned); only the recovery hint is wrong, hence P2.

fix: In query.ts buildFallbackDecision and context.ts buildContextFallbackDecision, add a branch for readiness.freshness==='fresh' && readiness.verificationGate==='fail' that surfaces a distinct blockReason (e.g. 'verification_failed') and a fallback that does NOT recommend code_graph_scan as the cure (point to re-running gold verification / clearing the gate). Mirror detect-changes.ts's 'fresh-with-failed-verification' label in the block message so all three read handlers expose the same vocabulary for this state. Note: the underlying inability of a scan to clear the gate is already tracked as the parallel-review P1 IT02-CORR-01 (ensure-ready.ts stale verificationGate) — do not duplicate that; this finding is narrowly the handler-side message/fallback defect.
