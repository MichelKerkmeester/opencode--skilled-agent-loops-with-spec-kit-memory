# Iteration 9 — architectural alternatives (subprocess pool, separate embed-only daemon, on-disk vector index like LMDB/RocksDB)

## Summary

The best architectural memory reduction is to keep `context-server` as the metadata/search orchestrator and move heavyweight local embedding runtimes behind a generic child-process boundary. Replacing SQLite wholesale with LMDB/RocksDB is not the first lever: current query logic depends heavily on SQL joins, filters, FTS, lineage, and projection tables, so the safer storage architecture is a canonical metadata DB plus profile-specific vector/cache shards.

## Findings

### Finding 1: Add an embed-worker boundary for any in-process local provider, with Ollama/API staying direct
- Evidence: `context-server.ts:1494-1510` initializes the DB and active embedder in the long-lived MCP process, while `shared/embeddings.ts:365-418` keeps a lazy singleton `providerInstance`. The remaining in-process local fallback still imports and caches `@huggingface/transformers`: `shared/embeddings/providers/hf-local.ts:199-221` stores `extractor`, `loadingPromise`, and loaded state; `hf-local.ts:237-267` imports the pipeline and stores it on `this.extractor`. By contrast, Ollama is already model-out-of-process: `mcp_server/lib/embedders/adapters/ollama.ts:242-275` and `shared/embeddings/providers/ollama.ts:308-325` call HTTP endpoints and keep only request metadata in Node. The current registry has a pluggable manifest surface but only wires `ollama`; `api` and `sentence-transformers` throw `NotImplementedError` at `mcp_server/lib/embedders/registry.ts:125-130`.
- Memory impact: active Ollama/Jina saves little inside PID 4791 because the model is already outside `context-server`. The regression risk is provider fallback or future local BGE/mxbai/llama-cpp: `hf-local.ts:17` documents a ~274 MB model class before transformer runtime/MPS overhead, and prior Iteration 5 measured llama-cpp model-file choices at ~313 MB Q8, ~584 MB BF16, and ~1.1 GB F32. A child process keeps that model RSS out of the MCP PID immediately, and idle-killing the child recovers the same order of memory at whole-machine level after inactivity.
- Proposed change: create `mcp_server/lib/embedders/sidecar-client.ts` implementing `EmbedderAdapter` over JSONL stdio, and `mcp_server/lib/embedders/sidecar-worker.ts` that loads a concrete backend (`hf-local`, future `sentence-transformers`, reintroduced `llama-cpp`) on demand. Add env `SPECKIT_EMBEDDER_EXECUTION=auto|direct|sidecar`, defaulting to `sidecar` for local in-process backends and `direct` for `ollama`, `voyage`, and `openai`. Change `registry.ts:getAdapter()` so `sentence-transformers` returns `new SidecarEmbedderAdapter(manifest)` instead of throwing, and add a shared-provider equivalent for `EMBEDDINGS_PROVIDER=hf-local` so `shared/embeddings.ts:getProvider()` receives a sidecar provider instead of constructing `HfLocalProvider` in the MCP process. The protocol should include `ready`, `embed({texts,inputType,profile})`, `stats`, and `dispose`; the client should enforce `SPECKIT_EMBEDDER_WORKER_IDLE_MS` default 300000-600000 and kill the child after idle.
- Trade-off: first local embedding after idle pays process spawn plus model load. Debugging gets one more process and log stream. Whole-machine RSS does not drop while the worker is actively loaded; the main win is keeping PID 4791 small and making idle recovery deterministic.
- Effort: L

### Finding 2: Run embedder reindex jobs through the worker, not through the MCP event loop
- Evidence: `embedder-set.ts:66-70` queues reindex directly after ensuring the target vector table. `lib/embedders/reindex.ts:252-284` constructs the adapter in-process, reads `memory_index` batches, calls `adapter.embed(...)`, writes vectors, and repeats until complete. `resumeReindexJobs()` restarts queued/running jobs on daemon startup at `reindex.ts:372-386`; `context-server.ts:1738-1746` calls that during startup. The reindex batch default is 50 rows at `reindex.ts:73-75`.
- Memory impact: batch arrays are small (`50 * 1024 * 4` is about 0.2 MB raw for 1024-dim vectors), but the adapter's model/runtime can be hundreds of MB if the selected backend is local in-process. Moving only query embeddings to a worker is incomplete: a model swap or resumed reindex can still load the same heavy provider into the MCP PID and hold it for the whole job.
- Proposed change: introduce an `EmbedderExecutionClient` used by both `vector-index-queries.ts:629-657` and `reindex.ts:246-308`. In `reindex.ts`, replace `const adapter = getAdapter(initialJob.toName)` plus `adapter.embed(rows.map(memoryText))` with `const embedder = getEmbedderExecutionClient(initialJob.toName)` and `embedder.embed({ texts: rows.map(memoryText), inputType: 'document', jobId })`. Keep DB writes in the MCP process initially (`writeVectors(...)`) to preserve the current single-writer behavior; only the embedding computation crosses the process boundary. Add progress telemetry from the worker: loaded profile, worker RSS if available, batch latency, and idle deadline.
- Trade-off: IPC copies input text and embedding buffers across processes. Reindex throughput may fall slightly, but it becomes measurable and schedulable. A later optimization can use `Transferable`/binary framing if JSONL buffer cost shows up.
- Effort: M

### Finding 3: Replace per-profile full SQLite stores with canonical metadata DB plus active vector/cache shard
- Evidence: profile pathing currently derives a complete SQLite DB per embedding profile: `shared/embeddings/profile.ts:75-78` returns `context-index__${slug}.sqlite`, and `resolveActiveProfileDbPath()` returns that profile file at `profile.ts:302-307`. `vector-index-store.ts:376-380` keeps a connection map by DB path, and each opened DB receives the same heavy pragmas at `vector-index-store.ts:800-806`. Current files show the cost of whole-DB profile copies: the active Ollama/Jina store is 139 MB and the historical llama-cpp store is 716 MB. A read-only `dbstat` probe on the active store attributed about 41.3 MB to `memory_lineage`, 33.2 MB to `memory_index`, 19.0 MB to `embedding_cache`, 16.8 MB to `vec_memories_vector_chunks00`, and 11.3 MB to FTS data.
- Memory impact: profile-specific full DBs duplicate hot metadata and lineage across model experiments. On the current active DB, moving vectors/cache out of the metadata DB would relocate roughly 35-40 MB of active vector/cache pages before index overhead; on the old 716 MB profile, the embedding cache alone has 10,000 rows / 30.7 MB raw vectors and `vec_768` has 13,184 rows / about 38.6 MB raw vectors. The bigger RSS win is that old profile shards do not need to be opened or mmap'd when only metadata/FTS/lineage is needed.
- Proposed change: change path resolution so `MEMORY_DB_PATH` / `SPEC_KIT_DB_DIR` points to a canonical metadata DB, e.g. `context-index.sqlite`, while `EmbeddingProfile` gains `getVectorShardPath(baseDir): vectors/context-vectors__${slug}.sqlite`. Add `attachActiveVectorShard(database, profile)` in `vector-index-store.ts` after connection open, applying low-memory pragmas to the shard connection/attachment. Move `vec_<dim>`, `vec_memories*`, and profile-specific `embedding_cache` into the shard. Then update `activeVectorSource()` in `vector-index-queries.ts:81-99` to return `active_vec.vec_<dim>` / `active_vec.vec_memories`, and update mutation writes at `vector-index-mutations.ts:281-286` plus reindex writes at `reindex.ts:210-233` to target the attached shard. Keep `memory_index`, `memory_lineage`, FTS, projection, governance, checkpoints, and session state in the canonical DB.
- Trade-off: migration is large. Cross-DB transactions, WAL files, backup/restore, integrity checks, and `checkDatabaseUpdated()` all need tests. The payoff is that embedder swap stops meaning "new copy of the whole memory database"; it becomes "new vector/cache shard for the same metadata graph."
- Effort: L

### Finding 4: Do not replace SQLite with LMDB/RocksDB as the first on-disk vector-index move
- Evidence: the vector path is SQL-coupled, not a standalone vector-store lookup. `vector-index-queries.ts:217-326` joins the vector table to `memory_index` and `active_memory_projection`, applies tier/scope/expiry filters, computes FSRS/half-life decay, and orders by vector distance plus effective importance. The runtime abstraction exists (`lib/interfaces/vector-store.ts:13-47`, `SQLiteVectorStore` at `vector-index-store.ts:887-945`), but the production query functions still depend on SQL table names from `activeVectorSource()` at `vector-index-queries.ts:81-99`. Package dependencies currently include `better-sqlite3` and `sqlite-vec`, not LMDB/RocksDB (`mcp_server/package.json` dependencies).
- Memory impact: LMDB uses mmap by design, so it can reproduce the same "mapped file looks like RSS" ambiguity Iteration 4 already found for SQLite. RocksDB would need an explicit block cache, compaction policy, and native dependency. Given the active vector payload is about 16.8 MB in `vec_memories_vector_chunks00` plus 19.0 MB in `embedding_cache`, a new engine is unlikely to beat the simpler shard/pragma approach by enough to justify losing SQL pushdown.
- Proposed change: keep SQLite as the default. If alternate vector storage is still desired after shard/pragma/BM25 work, add a narrow `VectorCandidateStore` interface that returns only `{ id, distance }[]` for `search(embedding, { limit, dim, profile })`, then keep metadata filtering/ranking in SQLite in a two-phase query. Gate it with `SPECKIT_VECTOR_STORE_ENGINE=sqlite|external`, default `sqlite`. Implement any LMDB/RocksDB prototype as a sidecar process first, not a native module loaded into `context-server`, so its mmap/block-cache RSS does not become PID 4791 RSS.
- Trade-off: two-phase vector search can change ranking because SQL filters may be applied after candidate retrieval unless the candidate pool is over-fetched. A sidecar also adds failure modes and consistency work. This is a research prototype path, not the next production optimization.
- Effort: M for the interface and SQLite-preserving two-phase path; L for a real LMDB/RocksDB backend

### Finding 5: The subprocess pool should be size-1 by default; multi-model warm pools increase total RSS
- Evidence: embedders are selected by active metadata in `vec_metadata` (`schema.ts:41-43`, `schema.ts:137-139`), and model swaps run as queued reindex jobs (`embedder-set.ts:69-90`). Current startup already reports one active embedder at `context-server.ts:1500-1504`. The launcher currently spawns exactly one context server child with `node dist/context-server.js` at `.opencode/bin/mk-spec-memory-launcher.cjs:296-302`; there is no worker supervision or pool policy today.
- Memory impact: a pool with one loaded worker per model would preserve latency but can multiply model RSS: hf-local/BGE class workers are ~274 MB plus runtime, while prior llama-cpp figures range from ~313 MB to >1 GB depending on dtype. A size-1 local pool with idle eviction caps loaded local-model memory to one profile. In the active Ollama baseline, leave pool size at zero/direct because Ollama already owns model residency.
- Proposed change: implement a small `EmbedderWorkerPool` with defaults `SPECKIT_EMBEDDER_WORKER_POOL_SIZE=1`, `SPECKIT_EMBEDDER_WORKER_MAX_IDLE=1`, and `SPECKIT_EMBEDDER_PRELOAD=false`. Pool key should be the full profile (`provider`, `model`, `dim`, `dtype`, `baseUrl`), but eviction should kill the least-recently-used worker before loading another local profile. Add launcher support for optional worker stdio tagging and cleanup on context-server exit, but let the MCP child own worker lifecycle first; later, the parent launcher can supervise shared workers if multiple MCP clients need them.
- Trade-off: alternating between two local models will cold-load each switch. That is acceptable for memory-first defaults and can be overridden for benchmarking with pool size >1.
- Effort: M

## Cross-references

This builds on Iteration 2 Finding 4 and Iteration 5 by turning provider lifecycle/unload into a process-boundary design instead of a provider-specific cleanup patch. It builds on Iteration 4 by treating SQLite memory as a residency-policy/topology issue, not a reason to abandon SQL. It builds on Iteration 7's process-boundary conclusion for code graph: boundaries help when the heavy state is genuinely in another process, but launcher-level RSS must be measured separately. It incorporates Iteration 8's BM25 warning by keeping large retained indexes out of the long-lived MCP process where possible, while leaving BM25-specific representation changes to Iteration 8.

It also updates Iteration 5's llama-cpp discussion: in this checkout, `rg` found no current `llama-cpp` provider source or compiled provider file under `shared/embeddings/providers`; the current provider set is Voyage, OpenAI, Ollama, and hf-local. Future llama-cpp support should be reintroduced behind the generic sidecar adapter rather than as a direct in-process provider.

## Negative knowledge (ruled-out)

- Worker threads are not the right isolation primitive. They have separate V8 heaps but share the same process address space, so native model memory still contributes to PID 4791 RSS. Use child processes for memory isolation.
- A new embed-only daemon is unnecessary for the active Ollama/Jina path. Ollama is already the embed daemon; adding another wrapper around HTTP calls would add latency without reducing `context-server` RSS.
- A multi-model warm pool should not be the default. It improves first-query latency but intentionally keeps multiple model weights resident, which conflicts with the memory-reduction goal.
- Replacing the whole memory DB with LMDB/RocksDB would remove too much existing SQL behavior: FTS, lineage, governance, session state, projection joins, decay scoring, and integrity checks. If explored, use it only as a vector-candidate sidecar.
- Moving vectors to a sidecar store does not remove the need for SQLite pragma tuning. Metadata, lineage, FTS, and caches still use SQLite, and attached vector shards still need explicit cache/mmap budgets.

## Open questions

- Is the target metric PID 4791 RSS or launcher-level total RSS across `mk-spec-memory`, worker children, Ollama, skill-advisor, and code-graph? Sidecars improve PID RSS immediately, but whole-machine memory only drops after idle eviction.
- What idle timeout is acceptable for local embeddings: 2 minutes, 5 minutes, or 10 minutes? The right default depends on normal editing/search cadence.
- Does `sqlite-vec` behave correctly and performantly in an attached database schema for `JOIN active_vec.vec_<dim>` / `JOIN active_vec.vec_memories`, or does it require a main-schema virtual table?
- Should the worker open the database read-only for reindex batches, or should the MCP process send texts and remain the only DB reader/writer? The latter is simpler; the former reduces IPC text copying but adds connection/cache pressure.
- What migration path should consolidate existing full profile DBs into a canonical metadata DB plus vector shards without losing old profile vectors or checkpoint restore compatibility?
