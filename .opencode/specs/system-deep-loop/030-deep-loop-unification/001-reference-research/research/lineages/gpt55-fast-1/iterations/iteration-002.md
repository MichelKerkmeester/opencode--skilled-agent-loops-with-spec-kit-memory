# Iteration 2: Bidirectional Path-Coupling Inventory

## Focus
This iteration inventoried concrete path couplings that a `deep-loop-runtime` + `deep-loop-workflows` merge must repair. The focus was not every textual mention, but migration-critical references that affect command execution, agent routing, test discovery, advisor routing, compiled command contracts, and operational diagnostics.

## Actions Taken
- Read the current state log, strategy, and registry before selecting the path-coupling focus.
- Grepped command, agent, advisor, and system-spec-kit surfaces for `deep-loop-runtime` and `deep-loop-workflows` references.
- Read representative files at the highest-risk coupling points: `/deep:*` YAML contracts, doctor route manifest, orchestrator route contract, advisor deep-routing code, TypeScript alias projection, system-spec-kit test config, optimizer manifest, and runtime integration-points doc.

## Findings
1. `/deep:*` command YAML is the highest-risk rewrite surface because it embeds both sides of the split: mode packet paths under `deep-loop-workflows` and runtime script/library paths under `deep-loop-runtime`. `deep_research_auto.yaml` references workflow skill docs/assets [SOURCE: .opencode/commands/deep/assets/deep_research_auto.yaml:64-77], runtime fanout/lock/convergence/prompt/audit/validation/upsert/merge paths [SOURCE: .opencode/commands/deep/assets/deep_research_auto.yaml:160] [SOURCE: .opencode/commands/deep/assets/deep_research_auto.yaml:229-243] [SOURCE: .opencode/commands/deep/assets/deep_research_auto.yaml:616] [SOURCE: .opencode/commands/deep/assets/deep_research_auto.yaml:819-859] [SOURCE: .opencode/commands/deep/assets/deep_research_auto.yaml:1017-1039] [SOURCE: .opencode/commands/deep/assets/deep_research_auto.yaml:1290-1297]. This must be migrated atomically with the directory move.
2. The compiled command contract is a second command-surface coupling, not just generated noise: `deep_research.contract.md` says it was generated by `deep-loop-runtime/scripts/compile-command-contracts.cjs` and carries source digests for `/deep:research`, both YAML modes, `mode-registry.json`, hub `SKILL.md`, and deep-research packet files [SOURCE: .opencode/commands/deep/assets/compiled/deep_research.contract.md:1-80]. Any path move must regenerate compiled contracts after source rewrites.
3. `/doctor deep-loop` directly routes diagnostics to runtime script paths and declares its route as read-only diagnostic over coverage/council graphs [SOURCE: .opencode/commands/doctor/_routes.yaml:100-110]. A merge must update doctor route manifests and doctor assets, or diagnostics will keep pointing to the old runtime skill.
4. Agent routing is coupled through `orchestrate.md`: deep routes must include a `Deep Route:` line whose source of truth is `.opencode/skills/deep-loop-workflows/mode-registry.json`, and dispatch correctness is verifiable by diffing against that registry [SOURCE: .opencode/agents/orchestrate.md:180-206]. The registry path is therefore an agent prompt contract, not just a documentation link.
5. The skill advisor has two hardcoded coupling layers. Python deep routing projects legacy deep skill IDs to `{skill: deep-loop-workflows, mode}` and sets `MODE_REGISTRY_PATH` to the workflows registry [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/scripts/skill_advisor.py:2570-2587]. TypeScript alias scoring similarly resolves old deep-loop aliases to `deep-loop-workflows` and points comments/readers to `deep-loop-workflows/mode-registry.json` [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/aliases.ts:95-165]. Reference migration must include advisor source, tests, graph metadata, and generated skill graph records.
6. `system-spec-kit` is a bidirectional consumer, not only an external dependency. Its Vitest config discovers `../deep-loop-runtime/tests/**/*.{vitest,test}.ts` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/vitest.config.ts:15-22], while its optimizer manifest lists deep-research/deep-review config files under `deep-loop-workflows` [SOURCE: .opencode/skills/system-spec-kit/scripts/optimizer/optimizer-manifest.json:7-18] [SOURCE: .opencode/skills/system-spec-kit/scripts/optimizer/optimizer-manifest.json:30-39]. Moving either side without updating system-spec-kit tests and optimizer metadata will silently degrade verification.
7. The runtime integration-points document already records additional consumers beyond the main command loop: ai-council command/orchestration, doctor route/update, system-code-graph playbooks, legacy MCP READMEs, and deep-improvement docs [SOURCE: .opencode/skills/deep-loop-runtime/references/integration_points.md:158-170]. This is an existing migration checklist and should be used as a coverage oracle.
