## Analysis: Self-healing / Control-theory loops → agent self-correction

This angle is **not exhausted** — the SRE/autonomic-computing world has spent a decade industrializing exactly the feedback-control primitives that agentic loops reinvent ad hoc. Five concrete, verified artifacts transfer directly.

**1. Reconciliation loops (the canonical control loop).** `kubernetes-sigs/controller-runtime` is the battle-tested library behind every Kubernetes operator. Its core primitive — `Reconcile(actual) -> (desired, requeue)` — is a pure control loop: observe actual state, compute the delta vs. desired, apply a corrective action, and schedule a re-queue *with exponential backoff*. This is structurally identical to a deep-loop iteration: it is **idempotent by contract** (the reconciler must tolerate being called with stale state), it never trusts a single observation, and the *requeue-after* knob is literally the loop cadence. The transferable lesson for `runtime/convergence` + `runtime/locks-recovery`: define the iteration body as a reconciler over an externalized desired-state spec, so any crash resumes by re-running Reconcile against the last persisted state rather than replaying an event log. Note `kubernetes/kubernetes` itself (124k★) is the same pattern at full scale (kube-controller-manager is a fleet of these loops).

**2. Circuit breakers = bang-bang feedback controllers on failure rate.** `resilience4j/resilience4j` (10.7k★, v2.4.0 Mar 2026) is the cleanest production treatment of feedback control applied to *failure*. Its CircuitBreaker is a closed-loop controller: the sliding-window failure rate is the measured variable, the threshold is the setpoint, and OPEN→HALF_OPEN→CLOSED is a hysteresis/bang-bang actuator. The most transferable *design* lesson is buried in its "Golden Rule": **one circuit-breaker instance per backend, never shared** — because a shared instance lets one failing service trip the breaker for all. For deep-loop this means: **each fan-out target (each source, each sub-agent) must own its own failure-rate tracker and its own bulkhead**, so a single flaky source cannot stall or poison the whole loop. Maps to `runtime/fan-out-fan-in`, `runtime/locks-recovery`, `runtime/budget-cost`.

**3. Remediation playbooks = declarative self-healing.** `robusta-dev/robusta` (3.1k★, 0.45.0 Jul 2026) and `keephq/keep` (12.1k★, v0.54.2 Jul 2026) are the two leading open-source "observe→correlate→auto-remediate" engines. Robusta's playbooks (`trigger → action`) and Keep's YAML workflows (`triggers → steps → actions`, with alert dedup + AI correlation) are the SRE answer to "what does the corrective half of the control loop *do*." The transferable idea: **separate the detection policy (when to act) from the remediation action (what to do), both declarative** — deep-improvement's promotion/suppression rules are the same split, and Keep's AI-correlation + dedup is a ready pattern for `runtime/dedup-novelty` and `runtime/gauges-observability`.

**4. Observe→diff→signal as a first-class loop.** `dgtlmoon/changedetection.io` (32.3k★, 0.55.8 Jul 2026) is a pure control loop reduced to its essence: poll → diff against baseline → emit signal, now with **LLM-powered change summarization that suppresses irrelevant diffs** ("notify only when price < $50"). This is a concrete instance of novelty filtering driving loop gating — directly maps to `runtime/dedup-novelty` and the reducer/aggregator design thread.

**5. The formal PID/feedback-control-for-LLM framing exists.** Search surfaced `arXiv:2604.22273` "Self-Correction as Feedback Control: Error Dynamics, Stability..." (Apr 2026), which explicitly models iterative LLM self-correction as a dynamical system with error dynamics and stability conditions — i.e., it asks *when does the feedback loop converge vs. diverge*. This is the theoretical mirror of the directive. (Caveat: I only verified the abstract snippet via search, not the full PDF — confidence **med** on exact details.) Adjacent work (SmartControl 2025, MDPI adaptive-PID servo 2025) runs the *reverse* mapping (LLM tuning PID gains), which is less directly useful but shows the bridge is two-way.

**Contradiction worth logging:** control theory says a well-tuned feedback loop converges (stability is the goal), but resilience4j's philosophy is to **suspend** the loop (open the circuit, fail fast) precisely when the system is distressed. The right move under load isn't always to tighten feedback — sometimes it's to stop iterating. This directly tensions deep-loop's "keep converging" assumption and feeds the open *when-NOT-to-loop* thread.
