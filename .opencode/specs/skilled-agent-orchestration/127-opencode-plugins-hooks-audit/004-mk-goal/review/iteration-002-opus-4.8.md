## Net assessment

All 13 prior findings hold at the mechanism level, corroborated by internal evidence (the info.id asymmetry, the unused part/info.tokens candidate lists) and by reading the tests, which use synthetic event shapes that structurally hide F1/F2/F3. I downgraded five over-rated severities (F4,F5,F6,F7,F8 from P1→P2/refinement) because they are opt-in, narrow-window, unreachable-in-normal-flow, or fail-open-by-design; F1/F2/F3 remain the genuinely load-bearing P1 correctness bugs. I add four new findings: an LLM-verifier session-pollution/feedback defect (O1), a demonstration that the existing `stale` flag cannot mitigate the F4 race (O2), an auto-turn budget leak on failed sends (O3), and a token double-charge on ledger eviction (O4).

**Prior findings adjudicated:** 13 - 9 confirmed, 4 adjusted. **New findings this pass:** 4.

**Parity (Opus view):** Confirmed: .claude/settings.json registers no goal-related hooks, so there is no Claude-runtime counterpart to drift against — prior parity assessment stands. The capability gap is real and intentional: goal state is keyed by OpenCode session id and hex-named per session, a namespace Claude Code hooks cannot reproduce; the only cross-runtime risk is silent divergence if a Claude Stop/PostToolUse goal shim is ever added without mirroring the OpenCode session-id keying and the native-event parsing gaps flagged in F1-F3.

### Detail

#### O1 - LLM verifier submits its judge-prompt into the user's live session, polluting the conversation and creating a feedback loop
- **Severity / Category / Confidence:** P2 / bug / med
- **Location:** `.opencode/plugins/mk-goal.js:1871-1880`
- **Evidence:** defaultLlmSupervisorVerifier calls promptAsync with path:{ id: sessionID } — the user's OWN session — injecting a text part ('Judge whether the active goal is complete...'). Because it is fire-and-forget into the same session (see F5), the model's reply becomes a new assistant turn that itself emits message.updated/session.idle, re-triggering accounting and another verify/continue cycle.
- **Impact:** In MK_GOAL_VERIFIER=llm mode the verifier corrupts the visible transcript with meta-prompts and can self-drive additional turns; it also charges the user's own token budget for verifier calls. Distinct from F5 (which is about mis-parsing the return): even a correct parse would still pollute the session.
- **Proposed fix:** Run the verifier against a dedicated ephemeral session or a direct model/completions call, never path:{id: userSessionID}; validate a structured response from that isolated call.

#### O2 - The `stale` flag plumbed to continuation cannot detect F4-style evidence staleness
- **Severity / Category / Confidence:** P2 / bug / med
- **Location:** `.opencode/plugins/mk-goal.js:1973-2018`
- **Evidence:** maybeVerifyGoal sets stale = !resultApplied (2017), and resultApplied becomes true whenever goalId+status still match (1977-1978) — it does NOT consider whether evidence/updatedAtMs changed during the verify await. maybeContinueGoal's guard `if (verifierResult?.stale)` (2175) therefore never trips for the exact race F4 describes, and line 1987 unconditionally writes result.evidence back over any newer current.lastEvidence even on not_met verdicts.
- **Impact:** The existing staleness plumbing gives false assurance: a verdict computed on superseded evidence is treated as fresh by both the state write and the downstream continuation gate, and newer evidence is silently regressed.
- **Proposed fix:** Capture a monotonic revision (updatedAtMs or lastActivityMessageID) at read time and, inside the queued mutation, treat the result as stale (skip apply, set stale=true) when it changed; never overwrite current.lastEvidence with older snapshot evidence.

#### O3 - A failed continuation promptAsync still consumes an auto-turn (no rollback of the reserved turn)
- **Severity / Category / Confidence:** P2 / bug / med
- **Location:** `.opencode/plugins/mk-goal.js:2222-2243`
- **Evidence:** reserveContinuationTurn increments autoTurnsUsed and stamps lastContinuationAtMs BEFORE the network call (2092-2094). If promptAsync throws, the catch records 'prompt_async_failed' with suppress=true (2234-2241) but never decrements autoTurnsUsed or restores the pre-reservation state.
- **Impact:** Transient send failures permanently burn the limited auto-turn budget (default 8); repeated flaky sends can exhaust the cap without a single successful continuation, and the wall/cap accounting is skewed by phantom turns.
- **Proposed fix:** On promptAsync failure, roll the reservation back (decrement autoTurnsUsed, clear lastContinuationMessageId) before recording the suppressed reason, or reserve the turn only after a successful submit.

#### O4 - accountedMessageUsage eviction (>64 message ids) can double-charge a re-emitted message
- **Severity / Category / Confidence:** refinement / bug / low
- **Location:** `.opencode/plugins/mk-goal.js:700-714`
- **Evidence:** normalizeAccountedMessageUsage and rememberAccountedMessage cap the per-message ledger to MAX_ACCOUNTED_MESSAGES (64) via slice(-64). Charging uses previousMessageTokens = accountedMessageUsage[messageID] || 0 (1646/1738). If an evicted message id ever re-emits a cumulative-token message.updated, previous resolves to 0 and the FULL cumulative tokenDelta is charged again.
- **Impact:** Over-counting of tokensUsed on long sessions (>64 distinct accounted messages) if any older message updates again, which can spuriously trip budget_limited. Low likelihood (latest message usually the one updating) → refinement.
- **Proposed fix:** Track the last-accounted cumulative per message with an LRU that is safe against re-charge, or charge deltas from a session-global cumulative counter rather than a bounded per-message map.
