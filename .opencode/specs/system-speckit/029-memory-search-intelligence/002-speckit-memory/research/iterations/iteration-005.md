## Findings (file:line)

### Q1 — Definitive 8→5 signal map (CONFIRMED, supersedes iter-1's draft table)

1. **The internal channels are exactly 5 and named identically at the router boundary**: `type ChannelName = 'vector' | 'fts' | 'bm25' | 'graph' | 'degree'` [SOURCE: lib/search/query-router.ts:36, 68]. RRF's `SOURCE_TYPES` adds `keyword` as a sixth label but the routed set is these 5 [SOURCE: shared/algorithms/rrf-fusion.ts:12-19; query-router.ts:68]. This pins the target side of the map.

| # | aionforge signal | internal channel/seam (file:line) | relationship |
|---|---|---|---|
| 1 | Lexical (BM25) | `bm25` channel + `fts` (FTS5) channel | **1:1 (split)** — internal splits SQLite FTS5 vs in-memory BM25; one aionforge signal → two internal channels [rrf-fusion.ts:12-19] |
| 2 | LexicalAnchor (exact surface match) | `fuseScoresAdvanced` term-match bonus, `0.05`/match | **PARTIAL/GAP** — a *post-fusion bonus* exists (`termMatches * termMatchBonus`, `rrf-fusion.ts:413-422`), but it is NOT a first-class channel and is NOT class-gated like aionforge's `exact_rerank` flag |
| 3 | Dense (vector ANN + exact rerank) | `vector` channel + `stage3-rerank` | **1:1** — present in two stages [iter-1 #4] |
| 4 | Support (1-hop SUPPORTS expansion) | causal-boost BFS + co-activation spreading (`stage2-fusion.ts:24-25` steps 2/2a) | **ANALOGOUS** — different edge semantics (causal vs SUPPORTS) |
| 5 | Graph (Personalized PageRank) | `graph` channel + graph-signals + `degree` channel | **1:1 (algorithm differs)** — internal uses curated causal-edge BFS + degree-centrality, not PPR; one aionforge signal → two internal channels (`graph`, `degree`) |
| 6 | Recency (re-rank) | recency-fusion step 1a + `IntentWeights.recency` | **1:1** [stage2-fusion.ts:24] |
| 7 | Importance (decay-at-rank) | `fsrs.ts` structural freshness (rank-time, pure) | **1:1 on read; DIVERGES on write** — internal ALSO has `strengthenOnAccess` write-back (iter-4 #2); aionforge never writes [iter-4 #4] |
| 8 | Trust (reliability-folded) | none found | **GAP** — no reliability/provenance-trust signal in the pipeline [iter-1 #49] |

   So: **5 of 8 map cleanly** (Lexical, Dense, Graph, Recency, Importance — though Graph/Lexical are 1→2 channel splits), **Support is analogous**, and **LexicalAnchor (partial) + Trust (full GAP)** are the two genuine gaps iter-1 flagged — now confirmed [SOURCE: rrf-fusion.ts:12-19, 413-422; query-router.ts:36; stage2-fusion.ts:24-25; iter-1 #49].

2. **The G2 double-counting guard is specifically about INTENT WEIGHTS, not arbitrary signals.** The module invariant: "Every score modification in the pipeline happens exactly once here … Intent weights are NEVER applied to hybrid results (G2 double-weighting guard) … Hybrid search already applies intent-aware scoring internally (RRF / RSF fusion). Post-search intent weighting is therefore ONLY applied for non-hybrid search types … Applying it to hybrid results would double-count" [SOURCE: lib/search/pipeline/stage2-fusion.ts:9-11, 37-40]. The function-level docblock repeats it: "G2 PREVENTION: This function is ONLY called for non-hybrid search types. Hybrid search (RRF / RSF) already incorporates intent-weighted signals during fusion. Calling this on hybrid results would double-count intent" [SOURCE: stage2-fusion.ts:624-627].

3. **The G2 guard's actual call-site is a single boolean branch.** `const isHybrid = config.searchType === 'hybrid'` [SOURCE: stage2-fusion.ts:1008]; then `// G2 PREVENTION: Only apply for non-hybrid search types. … if (!isHybrid && config.intentWeights) { … applyIntentWeightsToResults(results, config.intentWeights) }` [SOURCE: stage2-fusion.ts:1252-1257]. Session boost and causal boost are the mirror image — gated `if (isHybrid && …)` [SOURCE: stage2-fusion.ts:1013, 1075]. So the architecture's rule is: **a signal is applied in exactly ONE of {fusion-time (hybrid), post-fusion (non-hybrid)} — never both.**

4. **`fuseResultsMulti` ALREADY satisfies aionforge's "zero-weight elides cleanly" contract — this is NOT a conflict.** Channel weight resolves as `list.weight ?? (list.source === GRAPH ? graphWeightBoost : 1.0)` using `??` "so explicit weight=0 is honoured (`||` would treat 0 as falsy)" [SOURCE: shared/algorithms/rrf-fusion.ts:299-303], and a zero/negative weight short-circuits the whole channel: `if (weight <= 0 || list.results.length === 0) { continue; }` [SOURCE: rrf-fusion.ts:304-306]. A skipped channel contributes nothing AND is not counted — exactly aionforge's "a zero weight elides the signal entirely — it contributes nothing" [SOURCE: external/aionforge-memory-development/docs/retrieval.md:89]. **Per-class zero-weights are mechanically safe at the per-channel-score level today.**

5. **THE PRECISE CONFLICT SURFACE (two prongs, both narrower than iter-1's "every signal must contribute" framing):**
   - **Prong A — convergence-bonus reshaping (the hidden one).** The multi-source convergence/overlap bonus is computed against the *active* channel count: zero-weighting a channel runs `continue` BEFORE `activeChannelCount += 1` [SOURCE: rrf-fusion.ts:304-307], and the calibrated overlap bonus uses `overlapRatio = (channelsHit - 1) / Math.max(1, totalChannels - 1)` where `totalChannels = activeChannelCount` [SOURCE: rrf-fusion.ts:345, 359-360], or the flat bonus uses `convergenceBonus * (uniqueSourceCount - 1)` [SOURCE: rrf-fusion.ts:383]. **So per-class zero-weights silently change the multi-source agreement bonus** — a candidate that hit 3-of-5 channels gets a *different* bonus once a class turns one channel off (now 3-of-4), even though its own per-channel RRF contributions are unchanged. This is a real, non-obvious coupling: turning a channel off is not score-neutral for the *surviving* channels' overlap reward. aionforge's "re-ranks never widen recall" assumes signal independence; the internal convergence bonus makes channels **interdependent**. A class profile that zeroes channels must be designed knowing the bonus denominator moves [SOURCE: rrf-fusion.ts:304-388; aionforge retrieval.md:22, 49].
   - **Prong B — double-application against G2.** If a retrieval-shape weight vector is injected as `RankedList.weight` *and also* re-applied as a post-fusion re-rank (the way `applyIntentWeightsToResults` re-weights), it re-introduces exactly the double-count G2 prevents [SOURCE: stage2-fusion.ts:624-627, 1252-1257]. The class weights MUST enter ONCE, at the pre-fusion `RankedList.weight` seam, and be treated as fusion-time signals (the hybrid branch) — never replayed post-fusion.

6. **What breaks if you "naively add signals":** adding a new aionforge-style signal (e.g., a Trust channel, or a first-class LexicalAnchor channel) as a 6th `RankedList` is mechanically clean for scoring (fusion just sums one more `1/(k+i+1)` term) — BUT it (a) increases `activeChannelCount`, again shifting every multi-source candidate's convergence bonus denominator [rrf-fusion.ts:360, 383], and (b) if that signal duplicates an existing one's evidence (e.g., LexicalAnchor over the same tokens FTS5/BM25 already matched), the convergence bonus will **reward the same surface match twice as "multi-source agreement"** — a soft double-count that the G2 *intent* guard does NOT catch (G2 only guards intent weights, not channel-overlap inflation). [INFERENCE: based on rrf-fusion.ts:299-388 mechanics + the fact that LexicalAnchor (#1 above) covers the same lexical surface as bm25/fts.] Confirming would need a fusion-bonus unit test with a synthetic candidate hitting both bm25 and a new anchor channel.

### Q2 — Query-class router design (CONCRETE seam + integration)

7. **The retrieval-shape classifier lives alongside `query-router.ts`, as an ADDITIVE THIRD AXIS, not a replacement.** The two existing classifiers are orthogonal: complexity tier (`simple|moderate|complex`, drives channel *subset*) and task intent (`add_feature|fix_bug|…`, drives `IntentWeights`) [SOURCE: iter-1 #9; lib/search/query-classifier.ts:13; lib/search/intent-classifier.ts:13]. aionforge's classes (`SingleHopFactual|MultiHop|Temporal|Entity|Quote`) are a *retrieval-shape* axis — about candidate-set shape, not difficulty or task — so a new `retrieval-class-classifier.ts` co-located with `query-router.ts` would emit a `retrievalClass` field added to `RouteResult` [SOURCE: lib/search/query-router.ts:46-52; aionforge retrieval.md:129-130]. The router already aggregates classification + channels + plan into `RouteResult`, so it is the assembly point.

8. **The per-class weight vector enters at the pinned seam `RankedList.weight` → `fuseResultsMulti`.** `interface RankedList { …; weight?: number }` [SOURCE: shared/algorithms/rrf-fusion.ts:83-86, 111] consumed by `fuseResultsMulti` [SOURCE: rrf-fusion.ts:272, 302]. A class → `Record<ChannelName, number>` profile (aionforge's `RetrievalProfile` per-signal weights, `retrieval.md:129`) is applied by populating each channel-list's `.weight` BEFORE fusion. `weight:0` elides (finding #4), HEAVY/MODERATE/LIGHT map to e.g. `1.0/0.6/0.3` weights (aionforge 4-level scale, iter-1 #11). **This is a pre-fusion injection, so it lands on the hybrid/fusion-time side of the G2 boundary — correct by construction** (finding #3/#5B).

9. **Graph-expansion gating ATTACHES to an EXISTING router primitive — `query-router.ts:238-254`.** The router already computes a graph-preservation decision: `preserved`/`includeDegree` flags set when intent is `find_spec|find_decision` (`:242-244`) OR when `densityScore >= ENTITY_DENSITY_ACTIVATION_THRESHOLD` (`:247-252`) [SOURCE: lib/search/query-router.ts:238-254, 86]. This is an embryonic retrieval-shape signal: entity-density routing already turns the graph/degree channels on for entity-heavy queries. **aionforge's `graph_expansion` flag (per-class, off for SingleHopFactual) slots directly here** — the retrieval class would feed this same `preserved`/`includeDegree` computation: `SingleHopFactual → graph_expansion:false` (set `graph` weight 0, don't preserve), `MultiHop/Entity → graph_expansion:true` [SOURCE: query-router.ts:238-254; aionforge retrieval.md:129-130 "graph_expansion"; iter-1 #12 "indiscriminate graph expansion hurts single-hop precision, helps multi-hop recall"]. Turning the causal-graph channel off for single-hop factual = setting its `RankedList.weight` to 0 (finding #4 makes this clean for the channel score; **mind Prong A** — it shifts the bonus for the surviving channels).

Effort **M** — new classifier (regex/feature most-specific-first per aionforge `retrieval.md` class order) + `RouteResult` field + tests; the feature-extraction scaffold is reusable from `query-classifier.ts` [iter-1 #9].

Likely wants a `convergenceBonus`-aware profile or a bonus computed over *configured* (not active) channels — a small `fuseResultsMulti` option.

11. **K=40 vs K=60 — does k matter, and is changing it safe given the guard?** Internal `DEFAULT_K=40` is "optimized for ~1000-memory corpus," overridable via `SPECKIT_RRF_K` [SOURCE: shared/algorithms/rrf-fusion.ts:37-38, 150-164; iter-1 #1]; aionforge uses validated `k=60` [SOURCE: external/aionforge-memory-development/docs/retrieval.md:81]. **k is purely the rank-smoothing constant in `1/(k+i+1)`** [SOURCE: rrf-fusion.ts:310] — a larger k flattens the curve (top ranks matter relatively less; deeper ranks retain more weight), which suits aionforge's larger corpus. **Changing k is SAFE w.r.t. the G2 guard** — the guard is about *intent double-application* (findings #2-3), entirely orthogonal to the smoothing constant. The real coupling for k is the **convergence bonus**: `convergenceBonus`/`CALIBRATED_OVERLAP_MAX` are tuned relative to the `1/(k+i+1)` magnitudes [SOURCE: rrf-fusion.ts:310, 377-379, 383], so retuning k without revisiting the bonus constants would change the bonus-to-base ratio. Recommendation: **k is a tuning knob, not a correctness lever; leave at 40 for the ~1000-memory corpus unless a benchmark says otherwise, and treat k + convergence-bonus constants as a co-tuned pair** [INFERENCE: based on rrf-fusion.ts:37-38, 310, 377-388 — bonus constants are absolute, not k-relative].

### Q10 cross-cutting hooks (newly discovered this iteration)

12. **Two reusable cross-cutting patterns surfaced:**
    - **"Apply-exactly-once across fusion-time vs post-fusion" (the G2 invariant)** is a general determinism/double-count guard pattern [stage2-fusion.ts:9-11, 1008, 1252-1257] that the code-graph and skill-advisor rankers (which also blend multiple signals) could adopt verbatim — a single architectural choke point where "every signal applied once" is enforced by a hybrid/non-hybrid branch. Generalizes to Q10's determinism spine.
    - **Convergence-bonus channel-interdependence (Prong A)** is a transferable *caution*: any subsystem that adds a multi-source agreement bonus on top of RRF inherits the "turning a channel off is not score-neutral" coupling. Worth a shared `fuseResultsMulti` option `{ bonusOverChannels: 'active' | 'configured' }` so per-class profiles in ANY ranker (memory, code-graph, skill-advisor) can zero channels without distorting survivors [seam: rrf-fusion.ts:345-388]. This is a genuinely new candidate (C-X1) for the Q10 synthesis.
