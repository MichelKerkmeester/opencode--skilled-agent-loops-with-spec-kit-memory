# Iteration 003 — Track D (deepseek)

## Focus
Borrowable anti-fabrication architectures from the external landscape (coverage-based token election, DTCG provenance, evidence-gated narrative)

## Findings
1. **[P0] Coverage-based token election (design-extract/designlang)** — design-extract crawls multiple canonical pages and ELECTS tokens by site-wide page-coverage, not just element frequency. A token appearing on only 1 of 8 pages is excluded regardless of how many elements use it on that page. The current L1-L4 classifier in cluster.ts uses additive scoring across 5 signals (pageCoverage, usage dimensions, CSS var presence, frequency, chromaticity) — a one-off hero gradient can score high on frequency (500+ elements on one page) and leak into L2 despite 12.5% page coverage. Coverage-based election makes page-level prevalence the PRIMARY gate, directly preventing one-off campaign gradients from being presented as system tokens that the AI then builds interpretive prose around.
   - Recommendation: In cluster.ts classifyColorStability(), make pageCoverage the dominant gate: tokens with <30% page coverage are capped at L3 regardless of other scores. Add a pageDistribution [perUrl] frequency map to ColorToken. In the WRITE phase prompt template, add 'Only tokens present on ≥30% of crawled pages may be described as system conventions.' The VALIDATE phase should add a coverage-gating check similar to checkStabilityGating that flags any hex in DESIGN.md whose source token has pageCoverage <0.3.
   - Evidence: design-extract (github Manavarya09, 3.3k stars): multi-URL synthesis with coverage-based token election, DTCG tokens, WCAG audit, MCP server. Mentioned in project catalog as 'ELECTS tokens by COVERAGE (site-wide vs one-off).'
2. **[P0] DTCG $type/$value token schema (Style Dictionary, Diez, Terrazzo, W3C DTCG 2025.10)** — Every token has mandatory $type (color, typography, shadow, etc.) and $value. This creates structural provenance: a prose claim about 'depth' must reference a shadow or gradient token with $type: shadow — it cannot be an untyped narrative invention. The current tokens.json shape uses flat arrays (colorTokens[], shadowTokens[], radiusTokens[]) without typed cross-references. A claim like 'gradient-as-depth replaces shadow elevation' could reference a gradient hex and a shadow hex from DIFFERENT arrays with no structural link proving the claimed relationship. DTCG's type system would force the claim to anchor to specific token paths (shadows.elevation.medium.$value, gradients.hero.$value), making fabrication of false relationships auditable because each $type has defined semantics.
   - Recommendation: In the EXTRACT phase, emit a second output file tokens.dtcg.json alongside tokens.json using the W3C DTCG 2025.10 $type/$value schema. In the WRITE phase prompt, add 'Every named design principle (chromatic depth, shadow-as-border, etc.) must be accompanied by the DTCG token path(s) that constitute its evidence.' In the VALIDATE phase, add a dtcgProvenance check that parses DESIGN.md for named principles, extracts the claimed token paths, and verifies each exists in tokens.dtcg.json with the correct $type. Cross-references between $types (e.g., 'gradient' replacing 'shadow') that don't exist in the data become validation failures.
   - Evidence: W3C DTCG 2025.10 specification with 20+ reference implementations. Style Dictionary, Diez, Terrazzo all enforce $type/$value as zero-invention transformation frameworks. design-extract outputs DTCG tokens natively.
3. **[P1] Thin-evidence omission / unused-style filtering (Superposition, CSS Stats occurrence clustering)** — Superposition analyzes computed CSS + DOM and EXCLUDES unused styles. CSS Stats clusters colors by OCCURRENCE, implicitly filtering noise. The shared pattern: when evidence is thin, the tool OMITS rather than fabricating. sk-design-md-generator's writing_style_guide.md Section 16 (Named Principle Pattern) requires naming a design principle for every major decision, even when the data is sparse (e.g., 2 shadow tokens, one of which is the default browser shadow). AP-25 (Unnamed Principles) treats MISSING a principle name as a defect. This creates an obligate-fabrication pressure: the AI MUST name principles even when evidence is insufficient. The external tools show that omission is the correct anti-fabrication posture.
   - Recommendation: In writing_style_guide.md Section 16, add a gate: 'A named principle requires at minimum 3 distinct token instances across at least 2 crawled pages. If fewer than 3 tokens of a given type (shadow, gradient, radius) exist, write "Insufficient data to identify a [type] system — only N tokens extracted." instead of inventing a principle.' In the WRITE prompt template, add: 'If a design concept cannot be traced to at least 3 concrete tokens in tokens.json, do not describe it as a system characteristic.' In validate.ts, add a thinEvidence check that counts token categories mentioned in DESIGN.md principles and flags any that have <3 backing tokens.
   - Evidence: Superposition: renders page, analyzes computed CSS + DOM, EXCLUDES unused styles, converts to tokens. CSS Stats: computed-style introspection; color clustering by OCCURRENCE; specificity graphs. Both demonstrate that filtering by actual presence is the correct anti-fabrication pattern.
4. **[P1] Deterministic section-schema gating with evidence quotas (Open Design, DesignMD)** — Open Design uses a deterministic 9-section re-render schema. DesignMD has a 9-section markdown schema with AI-agent-ready prompts that constrain what each section may contain. The current format spec (design_md_format.md) defines 17 sections but the writing_style_guide.md Section 13 (Section Structure Checklist) sets soft 'Density Targets' (e.g., '6-10 values in 4 paragraphs') without enforcement. A section can have 0 token values and still pass all validation checks. Open Design's deterministic re-render means every section has a contract — the renderer produces the same output for the same input because the schema itself constrains what prose is permissible. Making section contracts HARD (required value counts, required token references per paragraph) would make fabrication structurally impossible because ungrounded prose would fail the section contract.
   - Recommendation: In design_md_format.md, convert 'Density Target' from soft guidance to hard quotas per section: Atmosphere requires ≥6 token values, Key Characteristics require ≥2 values per item, Shadow Philosophy requires ≥3 shadow token references, etc. In validate.ts, add a checkEvidenceQuotas function that counts token references per section and fails sections that don't meet minimums. This is complementary to DTCG provenance — quotas check quantity, provenance checks relationship accuracy.
   - Evidence: Open Design (nexu-io, 57.4k stars): DESIGN.md 9-section schema with deterministic re-render. DesignMD (designmd.cc): URL -> DESIGN.md via headless browser, live @media + interaction states, 9-section markdown schema with AI-agent-ready prompts.
5. **[P2] Explicit component hierarchy mapping (Builder.io Visual Copilot, Anima, Locofy)** — These design-to-code tools work via explicit hierarchy/component MAPPING rather than freeform interpretation. Every visual element maps to a specific component in a specific hierarchy — 'this shadow belongs to this card component, which has this variant, at this position in the tree.' sk-design-md-generator's cluster.ts already identifies components (Button, Input, Card, etc.) and captures their styles with interaction states, but the WRITE phase prose treats design characteristics as abstract system properties rather than component-grounded facts. The 'gradient-as-depth' fabrication would be impossible if every depth claim had to cite: 'Component X uses shadow Y at elevation Z, and no component uses gradient-based depth — depth is shadow-only (0 gradient-depth tokens across N components).'
   - Recommendation: In the WRITE phase, add a rule: 'Every claim about a design system pattern must cite at least one specific component and its token values as evidence. Abstract claims without component grounding are fabrication.' In validate.ts, extend section coverage checks to verify that prose claims in the Depth & Elevation section cite componentIDs that exist in tokens.components. A 'shadow philosophy' claim about how the system uses depth should reference the shadow values in specific ComponentVariant styles.
   - Evidence: Builder.io Visual Copilot / Anima / Locofy: design→code via explicit hierarchy/component MAPPING (not regeneration). Catalog notes: 'explicit component mapping; spec-driven export.' The mapping pattern forces claims to anchor to concrete, enumerable entities.

## Questions Answered
- Which 3–5 techniques most directly prevent AI-prose-fabrication (ranked)? → Coverage-based election (#1), DTCG $type/$value schema (#2), thin-evidence omission (#3), deterministic section quotas (#4), explicit component mapping (#5)
- Does coverage-based token election reduce L4 leakage better than OKLCH+L1-L4 gating? → Yes. Current gating uses additive scoring where high element-frequency can override low page-coverage. Coverage-based election makes page-level prevalence the primary gate, catching one-off page tokens that L1-L4 scoring misclassifies as L2 due to element count. The current pagesCoverage field contributes only 20-30 points out of a possible 60-100 — a token at 0% page coverage can still reach L2 if it has high frequency + CSS variable presence + high usage dimensions.
- Does DTCG $type/$value give stronger provenance anchoring than current tokens.json? → Yes. Current tokens.json has flat arrays (colorTokens[], shadowTokens[], radiusTokens[]) with no typed cross-references. DTCG's mandatory $type prevents category errors, enables explicit token references ({ $value: '{color.brand.primary}' }), and makes 'gradient-as-depth' claims auditable because each $type has defined semantics. The current shape captures MORE raw data (frequency distributions, usage contexts) but has WEAKER structural provenance for prose claims.
- Does any tool constrain or omit narrative when evidence is thin? → Yes. Superposition EXCLUDES unused styles entirely. CSS Stats clusters purely by occurrence with no interpretive overlay. Both demonstrate that omission is the correct anti-fabrication posture. sk-design-md-generator's current writing guide REQUIRES named principles (AP-25 treats unnamed principles as a defect) and encourages interpretive prose even with sparse data — the opposite approach.

## Summary
The root cause of AI-prose-fabrication in sk-design-md-generator is a structural disconnect: the VALIDATE phase checks hex presence but the WRITE phase is instructed to produce interpretive prose (named principles, comparative framing, metaphors) with no evidence-quota requirement and no token-path provenance anchoring. The three highest-leverage borrowable techniques are: (1) coverage-based token election from design-extract, which would prevent one-off page tokens from leaking into L2 where they become grounds for false system claims; (2) DTCG $type/$value schema from the W3C standard + Style Dictionary ecosystem, which would add structural provenance to every token reference in DESIGN.md prose; and (3) thin-evidence omission from Superposition/CSS Stats, which would replace the current obligate-naming pressure (AP-25) with a threshold below which the tool states 'insufficient data' instead of fabricating a principle. The single highest-value adoption is coverage-based token election because it requires the smallest architectural change (modify cluster.ts scoring weights) while directly addressing the most common fabrication pattern: one-off campaign tokens masquerading as system conventions.
