## 2. PROBLEM & PURPOSE

Once the shared substrate exists — the transition-authorized typed ledger core, the shared evidence and control services, the compatibility/shadow/rollback bridge, durable fan-out/fan-in, and the frozen shared mode contracts from the preceding phases — each deep-loop mode still runs its own loop with its own in-flight state. Migrating them all at once would race on shared write-sets and strand the runtime mid-flight.

This phase is the per-mode fan-out. Each of the eight modes becomes an independent fractal parent that migrates that mode's full run behavior onto the shared typed event-ledger substrate: a typed event schema, deterministic reducers, sealed reference artifacts, certificates and receipts, a resume adapter, a shadow-parity harness, and a rollback-guarded mode gate. The migrations run behind the frozen shared contracts and land additive and dark until each mode's gate proves shadow parity. Ordering is load-bearing: deep-improvement-common owns the evaluator/canary/promotion services its three benchmark variants reuse, so it lands first; deep-alignment reuses the review-loop backbone shared with deep-review.
