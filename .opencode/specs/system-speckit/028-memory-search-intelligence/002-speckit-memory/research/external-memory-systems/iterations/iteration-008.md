# Iteration 8 (MiMo lineage): Graphiti community detection + Cognee retriever zoo

> Model: **MiMo v2.5 Pro** (`xiaomi/mimo-v2.5-pro --variant high`, 1M ctx, read-only, via `opencode run`). Orchestrator-written. newInfoRatio **0.8** (freshly-mined surface: Graphiti's community layer + Cognee's retriever classes — neither touched in iters 1-6). **8 candidates, 5 NET-NEW.**

> **⚠ Carry the iter-6 systemic caveat:** the internal causal graph is **memory-ID→memory-ID, not entity-node**. Every "community"/"entity-cluster" candidate below assumes an entity-node graph and needs that reframe before it maps cleanly. The Cognee *retriever-pipeline* candidates are reframe-light (they wrap query→retrieve→answer loops, not node identity).

## Graphiti community layer (NET-NEW cluster — gated on the entity-node reframe)
1. **GR-community-label-propagation → NET-NEW (H/M).** Label-propagation over entity `RELATES_TO` edges clusters entities into communities; LLM pairwise-reduce then writes a `CommunityNode` (summary+name); communities become first-class searchable nodes (cosine+BM25+MMR+cross-encoder). `graphiti_core/utils/maintenance/community_operations.py:93-138,174-213,30-90`; `search/search.py:763-862`. Internal has **no clustering layer** on the causal graph; `entity_catalog` merges names but never clusters into summary groups.
2. **GR-community-pairwise-summary → NET-NEW (M/M).** Community summaries built via tournament-style pairwise LLM reduction (merge in parallel pairs until one remains) → compressed community-level abstraction. `community_operations.py:141-155,174-199`. No internal summarize-on-cluster exists; `memory_save` writes individual docs but never synthesizes group summaries.
3. **GR-community-search-channel → EXTENDS adaptive-fusion (H/M).** Dedicated community search (parallel fulltext+cosine over Community nodes, RRF/MMR/cross-encoder rerank) — communities participate alongside entity edges + episodes. `search/search.py:763-862`; `search_config.py:48-51,74-77,104-109`. The multi-channel RRF already exists; a community-derived channel is the new retrieval dimension.

## Cognee retriever zoo (the richest NET-NEW vein this iteration)
4. **CG-cot-validate-retrieve-loop → NET-NEW (H/M).** Chain-of-thought retriever iteratively validates the answer + generates a follow-up question, fetching new graph triplets per round (max_iter=4), merging triplets across rounds. `cognee/modules/retrieval/graph_completion_cot_retriever.py:36-95,144-176,208-217`. Internal `memory_context` is single-pass; no validate-then-retrieve-again loop.
5. **CG-query-decomposition → NET-NEW (H/M).** Decomposes a complex query into focused subqueries (structured LLM output), retrieves triplets per subquery in parallel, merges+dedups, then either combines context or answers each subquery before synthesizing. `cognee/modules/retrieval/graph_completion_decomposition_retriever.py:26-30,85-107,150-187`. Internal search has no decomposition stage; complex queries hit one path.
6. **CG-question-type-router → EXTENDS intent (M/S).** `BEAMRouter` classifies into 10 question types (information_extraction, temporal_reasoning, multi_session_reasoning, contradiction_resolution, summarization…) and routes each to an optimal retriever + type-specific prompt. `cognee/eval_framework/answer_generation/beam_router.py:22-90,98-119`. Internal `memory_context` has `intent=auto` but does NOT select different retrieval strategies per type — beam_router shows 10 distinct type→retriever+prompt mappings.
7. **CG-temporal-query-extraction → NET-NEW (H/M).** Extracts time intervals from NL queries (structured `QueryInterval`), filters graph events by timestamp range, then vector-ranks the filtered events; falls back to triplet search when no bounds found. `cognee/modules/retrieval/temporal_retriever.py:19-30,84-104,119-154`. Internal search has no time-interval extraction; records have timestamps but aren't searched by extracted time range. **High cross-fit with C3-x currentness.**
8. **CG-context-extension-convergence → EXTENDS banked CG-iterative-context-extension (M/S).** The concrete implementation of the banked candidate: use the generated completion as the next search query, merge with **convergence detection** (stop when no new triplets / max rounds). `cognee/modules/retrieval/graph_completion_context_extension_retriever.py:14-17,56-98,102-129`. Adds the specific convergence-stop + completion-as-query idiom.

## Net effect
5 NET-NEW (2 community + 3 Cognee retrievers), 2 EXTENDS, 1 EXTENDS-of-banked. The Cognee retriever zoo (CoT loop, query-decomposition, temporal extraction) is the strongest fresh vein — it maps to a **multi-strategy `memory_context`** rather than to graph node identity, so it dodges the memory-ID-vs-entity-node mismatch. The Graphiti community cluster is high-leverage but **gated on first deciding whether to introduce entity nodes** into the causal graph.
