## Finding 4 — The self-invocation guard is a `cli-opencode`-skill-specific ADR-001 convention, not a runtime-wide GPT-vs-Claude barrier

The guard that blocked 2 of phase 005's 4 smokes is explicitly scoped: "**Run the self-invocation guard before dispatch** (ADR-001): Layer 1 env-var lookup for any `OPENCODE_*`, Layer 2 process-ancestry probe for `opencode` parent, Layer 3 `~/.opencode/state/<id>/lock` probe." [SOURCE: .opencode/skills/cli-opencode/SKILL.md:319] It fires specifically when the *cli-opencode skill itself* is about to shell out to a nested `opencode run` process from inside an already-running OpenCode session — i.e., it is a guard against **OpenCode-invoking-OpenCode**, model-agnostic by construction (it would trip identically whether the nested dispatch targeted GPT or a hypothetical Claude-via-OpenCode-CLI run). It says nothing about, and does not gate, native Task-tool dispatch of a custom agent (`agent: deep-research`, `subagent_type: "general"`) inside the same session, nor does it gate a different CLI skill (`cli-claude-code`) being invoked from an OpenCode session (cross-CLI dispatch is explicitly the intended use — "cli-X skills are for cross-AI delegation only" [SKILL.md:16]).

**Correction to iteration 1's proposed next step:** the missing baseline in phase 005 does not require reproducing the exact blocked path. It requires running the SAME 4 command-owned smokes with a different executor binding — either `executor: native` inside a session whose driving model is confirmed Claude, or `executor: cli-claude-code` (this exact lineage's own executor) — neither of which is gated by ADR-001's `cli-opencode` self-invocation check. Two of phase 005's four failures (`GENERAL AGENT REQUIRED failure` for research, and the `context`/`ai-council` pre-launch blocks) were *not even* the ADR-001 guard specifically (`GENERAL AGENT REQUIRED` reads as a different, upstream agent-resolution failure — worth a follow-up read of what emits that exact string, deferred to a later iteration if time permits) — meaning some of phase 005's 4/4 failure count may have collapsed multiple distinct failure mechanisms under one table, another point neither prior lineage separated out.

## Finding 6 — KQ5's plugin feasibility claim is real and stronger than either prior lineage's cited evidence, with one unresolved technical gap

Both prior lineages justify plugin feasibility by analogy: gpt-fast-high cites `experimental.chat.system.transform` / `experimental.chat.messages.transform` usage in `mk-code-graph.js` [gpt-fast-high/research.md:82, citing mk-code-graph.js:442-518]; glm-max cites the same class of precedent more briefly. **Neither cites the actual hook that would be load-bearing for a pre-dispatch enforcement plugin.** The OpenCode plugin SDK exposes a distinct hook, `tool.execute.before`, with signature `(input: {tool, sessionID, callID}, output: {args: any}) => Promise<void>` [SOURCE: .opencode/node_modules/@opencode-ai/plugin/dist/index.d.ts:235-241]. This hook fires before ANY tool call executes (including, presumably, the `task`/Task-dispatch tool used for custom-agent dispatch) and receives a mutable `output.args` object — this is the actual mechanism a "pre-dispatch hook asserting route-proof fields resolve against mode-registry.json" (glm-max/research.md:66) would attach to, not the chat-system-transform hooks both lineages cited (those inject text into the system prompt *before generation*, which cannot inspect or act on a specific tool call's arguments after the model has already decided to make it).

This **strengthens** KQ5's feasibility verdict (a real, args-mutating pre-tool-call hook exists) but also surfaces an unresolved technical gap neither lineage flagged: the `tool.execute.before` signature has **no explicit deny/block output field** (contrast with `permission.ask`, which returns `{status: "ask"|"deny"|"allow"}` [index.d.ts:225-227]). It is unconfirmed from the type surface alone whether a plugin can actually **reject** a mismatched dispatch (e.g., by throwing inside the hook) versus only **rewrite** `output.args` to something safe/inert. "Fails-closed... on mismatch" (glm-max/research.md:66) is stated as settled; it is not settled by the evidence gathered so far — it is the single most important open implementation question for KQ5's fail-closed variant, and should be flagged as a residual for the phase-009 implementation, not asserted as proven.
