## 7. Gaps & Unknowns

Both seats independently surfaced the same gap cluster (all agreement 2/2). These are the seeded imperfections ‚Äî recorded as context for a consumer, **not** as fixture fixes:

- **Truncation can reintroduce an edge hyphen** ‚Äî `src/slugify.js:24-26`: the length cap `str.slice(0, maxLen)` runs *after* edge-hyphen stripping (line 21), so a cut landing on a hyphen boundary leaves a trailing `-`, violating spec.md requirements 4 & 5 (no edge separators; truncation must keep the slug valid). *(highest-relevance gap, 0.90)*
- **Incomplete input validation** ‚Äî `src/slugify.js:13`: only `number` is stringified; `null`/`undefined`/`object`/`boolean` fall through to `.toLowerCase()` (line 15) and throw `TypeError`. This **contradicts** `tasks.md`'s "Input validation complete" claim (spec-vs-code conflict).
- **`maxLen` falsy coercion** ‚Äî `src/slugify.js:10`: `maxLen || 60` treats a legitimate `0` as unset and does not guard a negative `maxLen` (which mis-slices from the string end).
- **No Unicode transliteration** ‚Äî `src/slugify.js:18`: accented/non-ASCII chars are dropped as separators, not transliterated (`"Caf√©" ‚Üí "caf"`). Matches the spec's `caf-menu` example but limits multilingual reuse.
- **Undefined empty-result behavior** ‚Äî `src/slugify.js:18,28`: all-non-alphanumeric input (e.g. `"!!!"`) returns `""`; the spec defines no behavior for an empty slug ‚Äî an open contract question for callers using slugs as identifiers.

**Verify before reuse**: decide the empty-slug contract and whether callers must pre-validate input types before calling `slugify`.
