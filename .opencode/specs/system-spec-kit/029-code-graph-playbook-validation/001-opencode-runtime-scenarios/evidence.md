## Harness-validated facts (smoke test)

- **deepseek-direct reaches the `mk_code_index` MCP runtime without `--pure`.** Confirmed via scenario 006: the dispatched session invoked tool `mk_code_index_code_graph_status` successfully. `--pure` is NOT used and must NOT be (it would disable the MCP runtime these scenarios require).
- **Live graph is empty in this environment** (`totalNodes: 0`, `freshness: "empty"`, `dbFileSize: 106496`, `schemaVersion: 5`). Scan-dependent scenarios (001, 002, 003, 004) must run their own scan inside a disposable workspace.
- `--agent general` triggers a harmless "subagent, not primary — falling back to default" warning on this opencode build; dropped for subsequent dispatches.
- **F-RUNTIME-1 (scope-safety, expected):** `code_graph_scan` enforces `rootDir` **within the workspace root (the repo)** — it rejects `/tmp/...` paths. Disposable workspaces must live UNDER the repo (e.g. gitignored `tmp/`); `/tmp` is refused. This is good scope-containment behavior, but it means scan-family scenarios index files into the shared repo graph DB (source of the graph-metadata churn) rather than a fully isolated DB.
- **F-RUNTIME-2 (parser quarantine — KEY FINDING):** After several successful scans (Batch A indexed 52 files/1996 nodes cleanly), subsequent fresh-workspace scans began returning `zero_node_scan_rejected` with "tree-sitter parser globally quarantined, all files parse_error." Once quarantined, no new workspace scan indexes anything, blocking every scenario whose precondition is a freshly-scanned disposable workspace (002, 005, and any of 007/008/022/024 attempted afterward). Recommend investigating the parser-quarantine trigger threshold / tree-sitter native-module stability under repeated scans, and whether quarantine should auto-recover. Scan *mechanics* themselves are validated by 003/004 (parser healthy).
