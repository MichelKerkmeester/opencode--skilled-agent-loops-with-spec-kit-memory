## [**1.1.3.0**] - 2026-07-10

This release adds the GitKraken MCP server (`gitkraken/mcp`, `@gitkraken/gk`, stdio) as a second remote-collaboration transport alongside GitHub MCP. Unlike GitHub MCP, GitKraken MCP reaches GitHub, GitLab, Azure DevOps, Bitbucket, and Jira from one tool, plus GitLens AI workflows (commit composer, launchpad triage, PR review, work-start) with no local equivalent. The skill now documents, routes to, and safety-gates it the same way it already does for GitHub MCP.

---

## What Changed

#### New Reference Doc

- **`references/gitkraken-mcp-integration.md`** -- Tool selection guide, the full verified tool surface (captured live from `gk mcp --list-tools` against an authenticated install, not the incomplete upstream README), usage examples, and error handling. Mirrors `github_mcp_integration.md`'s structure so the two remote-transport docs read as one family.

#### Router and Rules

- **New `GITKRAKEN_MCP` intent** -- Added to the `INTENT_SIGNALS`/`RESOURCE_MAP` router pseudocode, plus `gitkraken`/`gitlens` keyword triggers in the owned-keywords list and a references-table row in §5, so cross-platform PR/issue and GitLens-shaped requests route to the new doc automatically.
- **New ALWAYS rule #12: local-mutation carve-out** -- GitKraken MCP exposes `git_add_or_commit`, `git_push`, `git_branch`, `git_checkout`, `git_worktree`, and `git_stash`, which duplicate mutations this skill already gates behind the ask-first worktree rule, the no-direct-branch rule, and the conventional-commit format. The new rule states plainly: use the existing Bash workflow for those, reserve GitKraken MCP for GitLens AI workflows and cross-platform issue/PR/repository work with no local equivalent. The former rule #12 (operator direct-push authorization) renumbered to #13, and its one cross-reference elsewhere in the file was corrected to match.
