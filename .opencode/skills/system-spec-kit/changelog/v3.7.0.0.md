# v3.7.0.0, Honest Gaps, Clean Records

The center of this release is the moment a search comes up short. Memory search now tells you the truth about what it could not find. The low-confidence warning fires only when there is genuinely no strong match, instead of second-guessing a search that found exactly what you asked for, and the warning and the verdict can no longer disagree.

Around that, the records the framework keeps about itself got disciplined. The spec metadata that powers memory search and graph traversal is now type-safe and reproducible, the whole tree was migrated to the new contracts, and every skill document carries its own version. A new design tool reads a live website's CSS so an agent builds against the real colors, fonts and spacing rather than guessing them.

The through-line is that the changes were earned rather than assumed. Three search benchmarks ran against the real corpus this cycle and two of them ended in a no. A reproducible-ranking option was measured, found to change results without improving them, then removed. The evidence-gap fix itself was caught mid-flight when a live check disagreed with a green benchmark.

None of this changes how you call the system. `memory_search`, `memory_save`, the `/deep:*` commands and the skill and agent names all behave as before. The work happened in the honesty of the answer and the cleanliness of the records, not on the path you use every day.

## Search

Last release made the confidence score honest. This one makes the warning honest too: a search should tell you clearly when it does not have a good answer, and never contradict itself.

#### Honest Evidence-Gap Warnings

The "low confidence" warning now means what it says. It had been driven by a peakedness statistic (how far the top result stood above the pack) rather than by relevance, which got it backwards. A strong, tightly clustered set of good matches looked suspicious and got the warning, while a flat spread of weak off-topic matches slipped through clean. A search for a real term like `graph` could carry the warning even though it found solid results. The signal now keys off the same absolute-relevance measure the verdict uses, so the warning appears only when the search genuinely lacks a relevant match, and a healthy search stays quiet. A benchmark of the old behavior found it raised a false warning on two of every three strong queries, the redesign brought that to zero.

&nbsp;

#### A Safety Check That Was Off Is On

A verdict is supposed to drop from "good" to "weak" when the evidence behind it is thin, so a confident answer cannot ride out on a gap. That downgrade had been wired to a field nothing ever set, so it never fired, and a known gap could surface beside a confident verdict. The downgrade is connected now, and because it reads the same relevance signal as the warning, the two always agree. Where a strong-but-thin result once showed a contradiction, the answer is now consistent end to end.

&nbsp;

#### Scores on Every Result Row

Every result line shows a relevance number. Rows that came from the relationship and degree channels had been rendering a blank dash because the displayed score only knew one of the internal score names, so a reader could not tell how relevant those rows were. The displayed value now resolves through the same chain the ranker uses, so a graph or degree hit shows its score like any other row.

&nbsp;

#### A Truthful Status Field

The envelope no longer reports a status that means something other than what it looks like. A "weights off" field had been read by anyone as "the smart routing is off," but it was reporting a different, unrelated mechanism that is always off for a normal search. The retrieval-profile state now reports on its own field, so the status you read is the status you get.

&nbsp;

#### Cleaner Result Display

The result dashboard adds up. The header count now matches the rows actually shown rather than a pre-trim total, and a long folder path renders as its leaf title instead of being cut off mid-string, so a dense result set reads cleanly.

## Spec Records

The metadata the framework generates about each spec folder (the `description.json` and `graph-metadata.json` files that feed memory search and graph traversal) was hand-driftable and noisy. This release makes it type-safe, idempotent and reproducible, then migrates the whole tree onto the new rules.

#### Type-Safe Status

A folder's generated status is now a closed set of values (planned, draft, complete and the rest) instead of free-running prose, and a first-class validator checks both generated files before a bad value can propagate. Legacy hand-edited statuses no longer leak through. The check runs in a non-blocking report mode through the migration window and enforces after it.

&nbsp;

#### Scoped Operations, No Side Effects

Refreshing one packet touches only that packet. The backfill tool now takes a folder argument and defaults to that single scope, so a one-folder intent no longer quietly dirties the whole tree, and a repo-wide run takes an explicit `--all`. Archived and future-staged folders are kept out of the descriptions cache by a default-on exclusion, because keeping them out is a pure safety win.

&nbsp;

#### Reads That Stay Clean

Reading spec metadata stopped rewriting it. The description generator skips a write when the real content is unchanged (ignoring volatile timestamp fields), a targeted cache upsert updates one folder without rescanning the tree, and access and freshness telemetry moved to an index-layer store so a read no longer dirties an unchanged file. The tree stays clean and the diffs stay readable.

&nbsp;

#### Consistent Lineage and Synced Summaries

A folder's parent and children now reconcile across partial rescans, so a valid parent relationship survives when one subscan misses it and children are treated as append-only. Separately, a shared extractor unifies how a packet's overview populates the description and causal-summary fields, and a drift detector re-derives those fields on read and reports when they have diverged from the source docs, backed by a stored source-doc freshness key.

&nbsp;

#### A Clean Migration and a Flag Reckoning

The whole tree was regenerated onto the new contracts, 2,049 spec folders including archives, with a byte-stable second run proving the output is deterministic and zero validation violations afterward. A graduation benchmark then measured every default-off metadata flag against the migrated tree and applied the verdicts: twelve flags earned their way to default-on or enforcing, one purely informational flag was deleted for earning no measured win, and the two that needed a field the first migration had not written were graduated after a re-run wrote it.

## Design

Last release split design judgment from transport and shipped the judgment skill plus two transports. This release adds the piece that was missing: a way to capture what a real design actually is.

#### A Live CSS Measurement Tool

`sk-design-md-generator` reads a live website's CSS and writes it into a structured seventeen-section design reference an agent can build against. Instead of an agent inventing colors, fonts, spacing and shadows, it works from what the page actually ships. With the judgment skill deciding what something should look like and the transports moving design data to and from tools, the family now closes the loop: measure the real thing, make a deliberate choice, execute it. The skill is fully opt-in and needs a one-time browser install before first use.

&nbsp;

#### One Naming Family

The interface-design judgment skill is now `sk-design-interface`, renamed from `sk-interface-design` so everything design starts with `sk-design-`. The behavior and content are identical, this is a clarity rename, and the live docs and skill graph now use the new name.

## Skill Corpus

Two changes make the skill documents themselves more honest and more buildable.

#### Every Document Carries Its Version

A skill document now self-reports its version. Each one carries a four-part `version` field in its frontmatter, anchored to the skill's own version plus a per-document edit count, so when you open a reference you can tell whether you are holding the fresh copy or a stale one. The standard is written into the doc skill, the templates emit it, validators check it, and a deterministic engine populated it across all 2,210 in-scope documents. Before this, only twenty-one files carried a version and they were hand-maintained. New docs created through the `/create` commands are born versioned.

&nbsp;

#### The Parent-Skill Pattern, Now a Standard

The hub-over-modes shape that the deep loops proved last release is now a written, tooled convention rather than a one-off. The routing identity model is hardened and registry-driven instead of hardcoded, a drift-guard test keeps the registry honest against the code, the pattern is taught in prose with templates, and a read-only `/doctor:parent-skill` validator checks that a parent skill follows the rules. Anything you already built keeps working, the next hub-shaped skill just does not have to be hand-assembled.

## Also in This Release

#### Citation Is Model-Robust

A 144-cell benchmark drove `/memory:search` with four models (MiMo, Kimi, DeepSeek and gpt-5.5 fast) across twelve deliberately vague queries. The citation surface held up everywhere: every model obeyed the cite-only-on-a-good-verdict rule on every query. The new verdict tiers (good, weak, gap and cite-with-caveat) read coherently across all four models, and a post-graduation re-run confirmed the noise-floor and lexical-grounding flags that graduated this cycle close an off-corpus false-confidence on terms like `kubernetes`. DeepSeek came out the most efficient and stable driver for search, and Kimi over-explores expensively on the vaguest queries.

## Internal Seams (No User-Facing Change)

These are structural changes worth naming for the maintainers who will work in these files next.

- **Generator fingerprints.** The graph-metadata generator persists a source-doc fingerprint that survives an idempotent re-derive, so the system can prove a generated field is fresh without re-reading the doc. Phase-child detection moved behind one shared helper.
- **Metadata identity resolver.** A shared spec-folder identity resolver reconciles parent and child relationships across re-derives behind a default-off flag, with a review flag marking folders where the new contract differs from legacy state.
- **The determinism deletion.** The reproducible-ranking flag and all its gated branches were removed after the benchmark declined it, reverting to the default recency-aware ranking. The always-on result-ordering tie-break that shipped alongside it was kept, it is a pure determinism win that needs no flag.

## Upgrade Notes

No migration is required for the common path. The interface-design judgment skill was renamed from `sk-interface-design` to `sk-design-interface`, update any hardcoded reference, the old name still resolves for now. The relevance-aware evidence gap is on for everyone and needs no action. The determinism flag was removed, so if you had set `SPECKIT_DETERMINISTIC_RANKING` it no longer does anything and the system ranks by recency as it did by default. New skill documents created through the `/create` commands are born versioned. The spec-metadata flags that graduated are on by default, and the ones still default-off stay off until you enable them.

## Verification

This release follows the same shipping rule as the last: a structural protection that only ever makes the system safer or cleaner ships on for everyone, while anything that changes results ships off by default behind a named flag until a benchmark earns it on. The relevance-aware evidence gap, the dead-cap fix and the type-safe metadata validator are live for everyone, the determinism option was measured and removed, and the remaining metadata flags graduate only as the benchmark clears them.

#### Three Benchmarks, Two Said No

The search work was decided by real-corpus measurement, not by assumption. The vague-query benchmark proved the citation surface model-robust. The determinism benchmark measured the reproducible-ranking option, found it reshuffled the top results on roughly 40% of queries with no evidence the new order was better, and the option was removed. The gap-threshold benchmark found the evidence-gap detector was measuring the wrong thing entirely (peakedness, not relevance) and that no threshold could separate a good search from a thin one, which is what drove the relevance-aware redesign rather than a tuning tweak.

&nbsp;

#### A Live Check That Caught a Masked Bug

The evidence-gap redesign nearly shipped a regression. Its first benchmark passed green, but a full-handler live dispatch showed the warning firing on every single search. The benchmark had fed the detector clean absolute-relevance scores while production fed it a different, compressed score, so the bug only appeared on the real path. The fix threads the production-correct relevance signal into the detector, the re-benchmark was rebuilt to run through the real pipeline so it could not mask the same class of bug again, and a second live dispatch confirmed the warning now fires only on a genuine gap.

&nbsp;

#### The Flag Reckoning and Per-Packet Validation

The metadata work landed under a graduation benchmark that measured thirteen default-off flags against the freshly migrated tree, graduated twelve and deleted one. Each packet shipped under its own strict spec validation, and the skill-corpus versioning was applied by a deterministic engine and checked across all 2,210 documents. The benchmark harnesses ran against read-only copies of the live corpus and never touched a production shard or a live daemon.
