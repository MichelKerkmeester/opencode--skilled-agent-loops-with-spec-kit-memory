## Review: checkpoint create/restore integrity

**What's solid (verified, not re-reported):** The v1 restore is transaction-wrapped with `clearExisting`-mode rollback-on-any-error (3186-3196), zod envelope validation (2932), per-row `validateMemoryRow` (3008), merge-mode savepoint isolation per table (`restoreMergeTableAtomically`, 1764), and the FK-cascade-avoidance INSERT-OR-IGNORE+UPDATE strategy (2974-2987). The v2 swap path has a well-reasoned journal demotion protocol (swap-pending/swap-done) with crash/in-process symmetry (2707-2722, 2769-2783), fsync of files and parent dirs, and `.bak` rollback. Snapshot files come from `VACUUM … INTO`, which is a consistent point-in-time copy. The catalog-merge (`mergeCheckpointCatalogRows`, 2742) correctly preserves checkpoints created after the snapshot. Shard naming (`context-vectors__${slug}.sqlite`, vector-index-store.ts:430) matches the `deriveEmbedderSlug` regex, so the embedder guard normally fires.

**Three real, lower-severity gaps in restore faithfulness:**

1. **Embedder-mismatch guard is bypassed when the snapshot's `embedderSlug` is `null`** (2611). The guard reads `manifest.embedderSlug !== null && manifest.embedderSlug !== liveEmbedderSlug`. But `vecTables` is set independently of the slug (`vecIncluded ? [...] : []`, 2311), so a snapshot can have `embedderSlug: null` yet `shouldRestoreVec: true`. In that state the vec shard is copied over the live shard with **no** embedder/dimension compatibility check. Realism is low because normal shard names resolve a slug, but if shard naming ever deviates the guard becomes dead and any embedder change is silently restored over — wrong-dimension embeddings.

2. **v2 reports `result.restored = manifest.memoryCount`** (2755) — a count captured from the *live* DB after `VACUUM` (2313), never verified against the swapped-in snapshot's actual rows. Contrast `workingMemoryRestored`, which uses an actual `countTableRows(newDb, …)` (2756). A count drift (cross-process write between VACUUM and count) or a truncated-but-openable snapshot yields a falsely-confident "restored N memories." Traceability gap, not data loss.

3. **Catalog merge resurrects deleted checkpoints.** `VACUUM main INTO` copies the entire `checkpoints` table (the `skip` manifest only governs v1 row loops, not the v2 file copy), so snapshot-main holds the catalog as of create time. `mergeCheckpointCatalogRows` only *adds* rows (964-998); it never removes. Any checkpoint deleted between this checkpoint's creation and the restore reappears as a phantom catalog row pointing at a snapshot dir that `deleteCheckpoint` already removed (3257). Such a row shows in `listCheckpoints` and fails `loadCheckpointV2Manifest` on a restore attempt. Self-limiting via MAX_CHECKPOINTS pruning, hence P2.

None are P0/P1 — no data-loss or crash path; the heavy lifting (atomicity, rollback, validation) is correct.
