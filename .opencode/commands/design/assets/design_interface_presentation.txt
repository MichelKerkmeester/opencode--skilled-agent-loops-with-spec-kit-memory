# Design Interface Presentation Contract

Presentation source of truth for `/design:interface`. The router owns mode selection and asset loading. Workflow YAML owns execution. This file owns visible prompts, result displays, and next-step wording.

## 1. Startup Presentation

For `:confirm`, or no suffix with incomplete `$ARGUMENTS`, the consolidated setup prompt is the first visible response. Ask all applicable questions once, then wait.

For `:auto`, or no suffix with complete `$ARGUMENTS`, do not show the consolidated prompt. Resolve setup through the Auto Resolution Table below.

### Auto Resolution Table

| Field | Required | Default | Targeted-ask when |
|-------|----------|---------|--------------------|
| target | Yes | none | Always, when missing |
| register | No | auto (declared field -> task cue -> surface in focus) | Register is genuinely mixed or unresolved |
| mode | No | direction | Never - defaults silently |

### Consolidated Prompt Template

```text
Design Interface setup

Q1. Which interface surface (target), and is this Brand or Product register?
Q2. Task lane, if not the default: A) direction  B) directions  C) redesign  D) preflight  E) build  F) aesthetic

Reply format: "<target>, <brand|product>, <lane-letter or ->"
```

Never split these questions into separate visible prompts.

### Auto Fail-Fast Display

```text
Design Interface auto setup failed
Missing required input: interface target
Provide the target or rerun with :confirm.
STATUS=FAIL ERROR="missing required setup inputs"
```

## 2. Results Display

### Success

```text
STATUS=OK PRODUCES="Interface Direction Spec" NEXT=/design:foundations,/design:motion,/design:audit PROOF=target,register,designDials,preflightResult
```

### Missing Input

```text
Which interface surface, and is this Brand or Product register?
STATUS=ASK MISSING=<input>
```

### Failure

```text
STATUS=FAIL ERROR=<named-cause>
```

### Defer

```text
STATUS=DEFER ROUTE=<hub|sibling>
```

---

## 3. Next-Step Suggestions

| Condition | Suggested Command | Reason |
|-----------|--------------------|--------|
| Direction needs a static token, type, spacing, or theming system | `/design:foundations` | Turn direction into a static visual system |
| Direction needs purposeful animation, transitions, or micro-interactions | `/design:motion` | Specify temporal behavior |
| Direction is ready for findings-first review and scoring | `/design:audit` | Validate quality before build |
| Accepted direction moves to implementation | sk-code handoff via `.opencode/skills/sk-design/shared/sk_code_handoff.md` | Hand off to build |

This command never silently chains; it emits options only (`HANDOFF_REQUIRED=false`, recommend-only).

---

## 4. Example Usage

```
/design:interface dashboard-shell --mode redesign
```

Returns: a deliberate interface direction with rationale, visual choices, critique, and handoff notes.
