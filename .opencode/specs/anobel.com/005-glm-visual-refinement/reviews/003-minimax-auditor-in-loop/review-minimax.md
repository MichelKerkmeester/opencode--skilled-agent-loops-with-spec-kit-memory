## Verdict
**AGREE-WITH-CHANGES — confidence 0.70** — mechanism is sound in principle, but the cost/benefit vs. a single tighter-prompt round-1 regen is unproven, and three load-bearing assumptions (GLM round-2 instruction compliance, MiniMax rescore stability, audit finding accuracy) are asserted, not verified.

## What GPT-5.5 got right
- **Generator ≠ auditor is settled correctly** — GLM-5.2 confabulation in vision-audit-benchmark.md §3 is real; an external critique channel is required.
- **Adopt-if multi-gate shape is right** — combining proof_check + contrast_check + geometry_check + MiniMax rescore is the correct defense against false-fixes (REQ-004).
- **Failure-only routing is the correct default** — skipping the 27 SHIP sentinels is the right cost discipline; the "SKIP_SHIP" early-exit is clean.
- **Hard predecessor gate on phase 001** is correct — without the deterministic surface, adopt_if cannot block overflow/contrast false-fixes.
- **T0 vs. T1 A/B framing** is the right experimental instinct — a comparator arm proves the structure earns its place.

## Gaps / risks / errors
1. **GLM round-2 compliance on instructions round-1 ignored is the load-bearing assumption that breaks the whole phase.** Round-1's contract (`gen-tile.mjs:22`) already says "Size the instrument to leave room for the bottom title+desc block" — GLM ignored it on tile 4. Round-2's "use linear/stacked/bounded-lane primitives" is the same kind of instruction. **RC-3 root cause is partly "instruction-density mismatch"** — A4 ADDS a typed-JSON block on top, increasing prompt density further. The 5-7 conversion prediction has no empirical basis if round-2 compliance matches round-1 compliance.
2. **Cheaper alternative not considered: a single augmented-prompt round-1 regen.** Extract the 18 issues verbatim → 5-8 distilled rules → re-run `gen-tile.mjs` on the 18 FIX tiles with a 10-line prompt addition. No adapter, no JSON schema, no T0/T1 arm, no T0-vs-T1 cost. Same +1-3 SHIP potential at ~1.0x marginal cost (just the 18 extra GLM calls), vs. A4's 1.5-2x (18 round-2 GLM + 18 MiniMax rescore).
3. **`classifyFixes` regex is fragile on real audit text.** Sampled `audit-accountbeheer.json`: tile-2's "introduces an off-brand red accent not in palette" needs semantic understanding of "red accent" — pure regex requires exact phrase variants. Tile-4's "MS Vesta row and 'Live overname' status text collide" mixes overflow, collision, AND z-order in one sentence; the 3-fix cap collapses them. The 18 FIX findings likely collapse to ~8-10 unique fixes, not 18.
4. **The 18 FIX finding accuracy is asserted, not verified.** No inter-rater reliability check; no ground-truth adjudication. A wrongly-FIX tile (audit false positive) gets repaired needlessly; a wrongly-SHIP tile (audit false negative — the subtle-clip case in vision-audit-benchmark.md §3) is never repaired. A4 has no audit-correction mechanism; REQ-006 only BLOCKS the wrong-SHIP from being adopted, it does not REPAIR it.
5. **T0/T1 A/B is structurally weak — it tests prompt structure, not the value of round-2 itself.** Both arms are GLM round-2. A proper A/B is 3 arms: T0 (raw MiniMax issue verbatim), T1 (typed fix-JSON), T2 (fresh round-1 with distilled rules). T0 vs. T1 alone cannot answer "is the typed JSON doing the work, or is any round-2-with-feedback doing it?" — this is the **single most important question** the proposal punts on.
6. **MiniMax rescore is the adopt_if signal but has unmeasured variance.** GLM is at temp 0.4 (`gen-tile.mjs:30`); MiniMax rescore temperature is unspecified. A real fix may score differently on rescore from inter-call noise, not actual regression. The `false_fix_rate ≤ 1/18` criterion (5.5%) is meaningless without a rescore-variance baseline.
7. **Open questions in spec.md §7 are unanswered in plan.md.** "Should A4 run on borderline SHIP?" and "If marginal is +1-3, is 1.3-1.5x latency justified?" are the two highest-leverage questions in the whole spec, and the plan punts both. The plan's Definition of Done ("T1 beats T0") is also non-decisional if both arms fail equally.
8. **The "9 hard invariants" partly duplicate the round-1 contract** (560×480, bottom-left title, palette, 21px weight). Re-asserting them does not address RC-3's instruction-density root cause; it amplifies it. A4 will likely make round-2 compliance *worse* on dense tiles.
9. **The "max 3 fixes per tile" cap is unjustified.** Why 3? Audit JSON shows tiles have 1-2 distinct issues typically. If the cap is a real budget, it should be 2. If it's a safety bound against prompt bloat, it should be justified by GLM's context window math (24000 tokens / ~500 per fix ≈ 48 fixes possible).
10. **The "free 2D absolute positioning forbidden" rule in the round-2 prompt is the real prize** — but it should be a phase-001 prompt-level invariant, not a per-tile fix. A4 is patching a round-1 prompt defect, not solving the primitive problem. Phase 004 (A5/A7 skeleton) addresses this properly; A4 is a temporary band-aid.

## Strongest improvement or alternative
**Drop the adapter layer and run a 3-arm T0/T1/T2 A/B on the 18 FIX tiles:**
- **T0** = round-2 with the raw MiniMax `issue` strings verbatim + the original round-1 contract
- **T1** = round-2 with typed fix-JSON (the current A4)
- **T2** = fresh round-1 with the original contract + a 10-line "lessons learned" block distilled from the 18 issues
- Same harness, same `adopt_if` gates, same cost ceiling (~1.4x baseline)

If T0 ≈ T1, the typed-JSON structure is decorative — adopt T0, save the schema overhead. If T1 ≫ T2, the typed feedback earns its place. If T2 ≈ T1, the whole A4 phase is unnecessary — a single augmented round-1 prompt regen gets the same result at lower latency, lower cost, and no MiniMax round-2 dependency. **This single test answers the open questions spec.md §7 punts on, replaces the underspecified T0/T1 design, and pre-empts the "is A4 even needed" question before ~3 weeks of implementation.**
