# Iteration 10 — gpt-5.6-sol high — focus: benchmark drift classification + per-hub serving-status readout

**Headline finding:** The Lane C verdict enum and report renderer have no slot for **degraded/expected** — the state 012 REQ-004 says a drift-to-legacy fallback must report as. The new `compiledRouting` block (014 spec:113-118) will land as a second boolean gate beside `routeGold.gate.failed` (score-skill-benchmark.cjs:1767); combining the two via OR collapses three observably distinct states — *compiled-serving with route-gold violation*, *drifted legacy-fallback*, *broken resolver* — into one BLOCKED signal. The companion per-hub serving-status readout 012 REQ-004 mandates (line 99) has no implementation target: `compiled-route.cjs` only resolves `(hub, prompt) → decision`, never `(fleet) → status[]`. Every existing observability surface — `advisor_status` (advisor-status.ts:230-338), `spec_kit_skill_advisor_status`, and the bridge's metadata path — never reads the seven activation manifests at `.opencode/specs/.../010-live-activation/activation/<hub>/manifest.json`.

---

## Verified claims with `file:line`

### A. The drift-classification gap in the verdict enum

- **score-skill-benchmark.cjs:1765-1773** declares the verdict enum: `BLOCKED-BY-STRUCTURE | BLOCKED-BY-REGISTRY | BLOCKED-BY-ROUTE-GOLD | BLOCKED-BY-ROUTING | BLOCKED-BY-TOOL-SURFACE | NO-SCENARIOS | PASS | CONDITIONAL | FAIL`. None of these is `DEGRADED` or `LEGACY-FALLBACK` or `DRIFTED`.
- **012 spec.md:99** mandates "the verify harness must score drift-to-legacy as *degraded/expected*, not *broken*"; REQ-004 names three states `compiled-serving | legacy-fallback | drifted`. The verdict enum ships zero of these slots.
- **014 spec.md:134-136** plans `report.compiledRouting` with `drift rows` and `breakages` but says "routing divergence or breakage produces a distinct blocked verdict" — no distinct slot for drift. The expected degradation path has no rendered verdict.
- **012 checklist.md:90-92** treats `compiled-serving`, `legacy-fallback`, and `drifted` as three separate CHK rows, each with their own fixture requirement — but the Lane C verdict enum has one slot.

### B. The per-hub serving-status readout is unimplemented

- **012 tasks.md:86 (T007)** "Add the per-hub `compiled-serving | legacy-fallback | drifted` serving-status readout". No follow-up commit / no module path; no entry in any bin or handler.
- **.opencode/bin/** contains `compiled-route.cjs` (per-hub × per-prompt decision), `skill-advisor.cjs`, `mk-skill-advisor-launcher.cjs` — none ships a `compiled-status.cjs` or `--all-hubs` fleet readout. **verified** by `ls .opencode/bin/`.
- **advisor-status.ts:230-338** never reads the seven activation manifests; it inspects `skill-graph.db` (skill_nodes / vec_metadata) for freshness/trust, never the runtime serving authority. **confirmed by read**.
- **F-8-4** (prior iteration) already named this gap; my read confirms no code path between activation manifests and any status surface.

### C. `isHubTypeSkill` misaligns with compiled-eligibility

- **run-skill-benchmark.cjs:68-70**: `isHubTypeSkill = fs.existsSync(hub-router.json)`. **Structural**, not compiled-eligibility.
- **advisor-recommend.ts:41-49** `COMPILED_ROUTING_HUBS` is the compiled-eligibility allowlist (7 hubs). A hypothetical 8th hub with `hub-router.json` but no entry in this set would be hub-type but NOT compiled-eligible — `route-gold: auto` → ENFORCED, `compiledRouting: auto` (014 spec line 242) → off; operators must reason about both flags to know what was scored.

### D. `compiledRouting` telemetry is sibling, not parallel

- **score-skill-benchmark.cjs:745-791** `reduceRouteTelemetry` emits `telemetryMissingRate/routeMissRate/aliasMissRate/bundleMissRate` for the legacy replay only (it reads `row.routeTelemetry` set on line 1541 from `routerResult.routeTelemetry`).
- **014 spec.md:107** plans `compiledRouting` as a separate report block, not as additional rows in `routeTelemetry`. An operator reading `report.advisorySignals.routeTelemetry.routeMissRate` for a hub where compiled is *drifted* sees a healthy legacy number while the compiled-serving side is silently failing — the metric vocabulary mixes the two regimes under the same shape, with no `compiled.telemetryMissingRate` parallel.

### E. Pin-table duplication across three callers

- **lib/activate-hub.cjs:46-49**: `PINNED_SCORER_DIGESTS = { router-replay.cjs, score-skill-benchmark.cjs, load-playbook-scenarios.cjs }` with hardcoded digests.
- **lib/flip-serving.cjs:42-45**: same three names, **same** digests, copy-pasted.
- **014 spec.md:132 (REQ-001)**: planned third copy inside `compiled-routing-parity.cjs`. Three-source-of-truth with no shared file; if any one drifts, the activation pass can pass while the parity run silently fails (or vice versa) and the operator sees two different error messages naming the same drift.

### F. Funnel renderer cannot point at the new lane

- **score-skill-benchmark.cjs:1644-1652** declares `const funnel = {}; for (const r of rows) ...` and renders `headlineBottleneck` from the same counts.
- **build-report.cjs:178-181** renders `## Funnel` from this dict. New compiledRouting parity rows (one per scenario when 014 lands) have no `firstFailingStage` slot and will either be silently dropped from the funnel or double-counted by scenarioId. `headlineBottleneck` will never name the compiled-parity lane even when it is the dominant failure mode.

## Ranked actionable recommendations

1. **Add a `DEGRADED | LEGACY-FALLBACK | COMPILED-SERVING | BROKEN | DRIFTED` lane to the Lane C verdict enum** (score-skill-benchmark.cjs:1765-1773 + build-report.cjs:81-86) before 014's parity lane lands. Without a distinct verdict slot, REQ-004 of 012 cannot be observed — drift looks identical to breakage. The new `compiledRouting.blockedReason` should be a *sub-cause* of an outer verdict, not an OR-combined boolean. (F-10-1, F-10-5)
2. **Ship `compiled-route-status` as a new `.opencode/bin/compiled-route-status.cjs` CLI** with `--hub <id>` and `--all` modes that read the seven activation manifests and emit `{hubId, servingAuthority, shadowOnly, selectedPolicy.generation, effectivePolicyHash, fenceEpoch, manifestFingerprint}`. This is the only way the operator can answer "is hub X currently compiled-serving?" without reading seven files by hand. (F-10-3, F-10-7)
3. **Extract the frozen-scorer pin into a single `.opencode/bin/compiled-routing-scorer-pins.json`** so `lib/activate-hub.cjs`, `lib/flip-serving.cjs`, and the planned `compiled-routing-parity.cjs` all `require()` the same digests; a drift on any one digest fails all three callers with the same error string naming the same file. (F-10-6)
4. **In the planned `compiledRouting` lane, attach a `lane: 'legacy'|'compiled'` field to each row's `firstFailingStage` and extend `reduceRouteTelemetry` with a `compiledTelemetry` parallel block** so the existing `headlineBottleneck` can name "compiled-parity" when that is the dominant failure. (F-10-2, F-10-7)
5. **Add a `compiledEligibility` field to the `routeGold` summary** so a hub-type scenario whose target is not in `COMPILED_ROUTING_HUBS` is reported as `n/a` rather than scored; the gap between hub-type detection (file existence) and compiled-eligibility (manifest + allowlist) must be surfaced, not papered over by `auto`. (F-10-4)
