# Iteration 3 — gpt-5.6-sol high — focus: playbooks

The new-router manual coverage belongs in eight root playbooks, not all 39: the seven compiled-eligible hub playbooks plus `system-skill-advisor`. Child-mode playbooks should remain focused on packet behavior. The fixed eligibility set is `sk-code`, `mcp-tooling`, `system-deep-loop`, `cli-external-orchestration`, `sk-prompt`, `sk-design`, and `sk-doc` (`system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:41-49`). Hub playbooks already define the correct ownership boundary: advisor identity followed by hub-mode resolution, excluding child internals (`cli-external-orchestration/manual-testing-playbook/manual-testing-playbook.md:11-19`, `mcp-tooling/manual-testing-playbook/manual-testing-playbook.md:11-19`, `sk-prompt/manual-testing-playbook/manual-testing-playbook.md:11-19`).

The central `system-skill-advisor` package should own four canonical operational cases. It already owns native MCP, hooks, compatibility, daemon, and lifecycle validation (`system-skill-advisor/manual-testing-playbook/manual-testing-playbook.md:17-27`) and requires full JSON, exit-code, hook stdout/stderr, and verdict evidence (`system-skill-advisor/manual-testing-playbook/manual-testing-playbook.md:72-90`).

| Proposed file | State | Required assertions |
|---|---|---|
| `compiled-routing/flag-off-kill-switch.md` | Flag absent or `0` | Legacy `skillId`/`workflowMode` remain unchanged; `compiledRoute` is absent; status reports `legacy-fallback` with cause `flag-disabled`. |
| `compiled-routing/flag-on-fresh-seven-hub-matrix.md` | Flag `1`, fresh manifest | Each of seven touchstone prompts keeps the same advisor winner; `compiledRoute` exists with expected action, targets, policy hash, and generation. |
| `compiled-routing/drift-falls-back-to-legacy.md` | Flag `1`, stale routing input in disposable workspace | Status is `drifted` with re-mint-required cause; advisor omits `compiledRoute`; the legacy recommendation still serves. |
| `compiled-routing/serving-status-classification.md` | Seven-hub fleet readout | Every hub reports exactly `compiled-serving`, `legacy-fallback`, or `drifted`, including cause, manifest identity, policy hash, and generation. |

Each of the seven hub roots should then gain one thinner `compiled-routing/luna-high-route-follow.md` scenario. Reuse a shipped playbook prompt, execute flag-off and flag-on probes, and assert identical top-level skill/mode plus the expected compiled target. This belongs at the hub root because those playbooks already validate live advisor-plus-mode behavior—for example `sk-code` has a dedicated advisor category (`sk-code/manual-testing-playbook/manual-testing-playbook.md:220-226`), `sk-design` explicitly captures the advisor vote and resolved mode (`sk-design/manual-testing-playbook/manual-testing-playbook.md:47-53`), and `system-deep-loop` does the same (`system-deep-loop/manual-testing-playbook/manual-testing-playbook.md:39-44`). Author each new file through the `create-manual-testing-playbook` template: one canonical per-feature file carrying exact prompt, command, signals, evidence, verdict, and triage (`sk-doc/create-manual-testing-playbook/SKILL.md:207-231`).

Two blockers must land before those model-run scenarios can count as end-to-end proof.

First, the prompt-time advisor hook currently cannot transport the compiled decision. The native response schema accepts `compiledRoute` (`system-skill-advisor/mcp-server/schemas/advisor-tool-schemas.ts:208-222`) and the native handler attaches it (`system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:362-371`), but the hook subprocess recommendation type has no such field (`system-skill-advisor/mcp-server/lib/subprocess.ts:16-23`) and its rendered brief only tells the model to use the top skill (`system-skill-advisor/mcp-server/lib/render.ts:197-200`). A LUNA run through the current hook can therefore pass while never observing the compiled mode decision. Add compiled action/targets/status to the hook contract and a hook-level assertion before accepting real-model evidence.

Second, `sk-code` cannot yet prove the required `code-opencode` surface bundle through the public compiled front door. The front door passes only `{prompt}` to the evaluator (`011-runtime-engine/lib/compiled-route.cjs:73-75`), while the compiled composite fixture requires both a risk slice and a live certificate (`006-parent-hub-rollout/001-sk-code/fixtures/canary-cases.v1.json:60-66`); the same fixture explicitly clarifies when the certificate is absent (`006-parent-hub-rollout/001-sk-code/fixtures/canary-cases.v1.json:119-124`). Extend the runtime input contract or establish a safe certificate context before requiring the sk-code scenario to return a `surfaceBundle` containing `code-opencode`.

The serving-status scenario also needs a new causal interface. Today missing/unparseable manifests become `legacy` (`011-runtime-engine/lib/resolve.cjs:26-33`), while flag-off, legacy authority, and engine failure all collapse to `null` (`011-runtime-engine/lib/resolve.cjs:40-47`) and finally the same legacy sentinel (`011-runtime-engine/lib/resolve.cjs:64-65`). A manual playbook cannot distinguish kill-switch behavior from drift or breakage until a machine-readable status probe preserves the cause.

One previously unnamed documentation gap blocks the `sk-doc` hub case: its live skill is now an eleven-packet registry router (`sk-doc/SKILL.md:46-105`), but its playbook still declares the old flat `RESOURCE_MAP`/`INTENT_MODEL` surface as authoritative (`sk-doc/manual-testing-playbook/manual-testing-playbook.md:9-13`). Realign that playbook before adding compiled-routing coverage; otherwise its expected modes and prompts target a retired contract.

Use this exact routed-model shape after hook enforcement exists:

`SPECKIT_COMPILED_ROUTING=<0|1> codex exec --model gpt-5.6-luna -c model_reasoning_effort="high" -c service_tier="fast" -c approval_policy=never --sandbox read-only -o <stdout-evidence> "<route-only prompt>" 2> <stderr-evidence>`

The model and effort are valid (`cli-codex/SKILL.md:235-242`), and the CLI contract requires explicit model, effort, and service tier (`cli-codex/SKILL.md:267-268`). Keep routing validation read-only and tell LUNA to return the injected advisor skill, compiled action, target mode/packet, and serving status without executing the underlying user task. Treat timeout as `SKIP` with an infrastructure blocker, not routing failure: earlier LUNA routing runs hit the existing 150-second ceiling repeatedly (`010-live-activation/real-model/run-log.txt:50-57`, `:95-102`, `:143-150`).

Ranked recommendations:

1. P0 — carry `compiledRoute` and serving status through the prompt-time hook; otherwise LUNA never sees the decision being tested.
2. P0 — expose causal serving status and an injectable disposable-workspace fixture path for drift testing.
3. P0 — repair the sk-code runtime input boundary so a real `code-opencode` surface bundle can route.
4. P0 — add the four canonical state scenarios to `system-skill-advisor`.
5. P1 — add one flag-off/flag-on LUNA-high route-follow scenario to each eligible hub root; do not duplicate them in child playbooks.
6. P1 — realign the stale sk-doc playbook before adding its case.
7. P1 — extend the named alignment gate: `verify_alignment_drift.py` currently scans language files and dispatches extension-specific checks (`sk-code/code-opencode/assets/scripts/verify_alignment_drift.py:480-515`), while the router contract it must protect lives in Markdown `RESOURCE_MAP` (`sk-code/code-opencode/SKILL.md:43-51`). It presently cannot prove parent/child router-map equality.
