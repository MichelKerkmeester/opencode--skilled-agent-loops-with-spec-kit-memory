# Iteration 011 — Track D (mimo)

## Focus
Token-transform frameworks (Style Dictionary/Theo/Terrazzo/Diez/DTCG) — typed tokens + deterministic value-table generation

## Findings
1. **[P0] DTCG $type/$value adoption in tokens.json** — Current tokens.json is a flat bag of category arrays (colorTokens[], typographyLevels[], shadowTokens[], radiusTokens[]) with extraction metadata (frequency, confidence, usedAs). DTCG wraps each token as {"$type":"color","$value":"#533afd","$description":"..."}. Concretely improves: (1) claim provenance — each token self-declares its type, so a dimension can't slip into a color slot; (2) validator binding — validate.ts currently uses ad-hoc heuristics (regex hex matching, backtick font-name detection) that could become type-gated assertions; (3) downstream emit — Style Dictionary, Terrazzo, and Theo all consume DTCG natively, enabling deterministic multi-platform output without transform code. What breaks: cluster.ts (1742 lines) builds flat arrays — needs a DTCG wrapper layer (~200 LOC); types.ts needs DTCG-aware interfaces; validate.ts's checkPhantomColors/checkUnknownFonts become type-filtered passes. Extraction code (extract.ts, dom-collector.ts) is unaffected — it feeds cluster.ts, not tokens.json directly.
   - Recommendation: Add a dtcg-wrap.ts module post-cluster that maps ColorToken→{$type:'color',$value:hex,...}, TypographyLevel→{$type:'typography',$value:{fontFamily,...}}, ShadowToken→{$type:'shadow',$value:...}, RadiusToken→{$type:'dimension',$value:...}. Write tokens.json in DTCG shape. Update validate.ts to type-filter before checking. File: tool/scripts/dtg-wrap.ts (new), tool/scripts/cluster.ts (emit DTCG), tool/scripts/types.ts (add DTCG interfaces), tool/scripts/validate.ts (type-gate checks). Phase: extraction output refactor.
   - Evidence: Style Dictionary DTCG support, Terrazzo DTCG 2025.10 parser, Theo declarative transforms, W3C DTCG spec — all converge on $type/$value as the interop contract.
2. **[P0] Deterministic value-table generation from typed tokens** — Current pipeline: AI reads tokens.json and writes DESIGN.md value-tables (typography scale, color palette table, shadow/radius tables, component matrix). This is the primary hallucination vector — the validator's checkPhantomColors catches fabricated hex codes, but a fabricated fontSize, fontWeight, or lineHeight in a typography table passes undetected. Diez's DocsGen generates token documentation deterministically from typed token data — zero inference, zero fabrication. Applying this: Sections 2 (Color Palette & Roles), 3 (Typography Rules), 4 (Component Stylings value-tables), 6 (Depth & Elevation tables), 6.5 (Motion System tables), 17 (Design Tokens Dictionary) can all become deterministic formatters over typed tokens. Sections 0 (Brand Context), 1 (Visual Theme), 5 (Layout Principles prose), 7 (Content & Voice), 8 (Do's/Don'ts), 9-16 require AI authorial judgment and stay AI-written.
   - Recommendation: Create formatters.ts with deterministic table generators: formatColorTable(tokens), formatTypographyTable(tokens), formatShadowTable(tokens), formatComponentTable(tokens). These emit markdown tables directly from DTCG-typed tokens. AI writes only narrative sections (0,1,5,7,8,9-16). Report-gen.ts already does this for HTML — port the pattern to markdown. File: tool/scripts/formatters.ts (new). Phase: WRITE phase refactor — split into 'deterministic tables' then 'AI narrative overlay'.
   - Evidence: validate.ts:57-93 (checkPhantomColors) catches hex fabrication but not typography/shadow/component fabrication. Diez DocsGen pattern: tokens→deterministic docs.
3. **[P1] Style Dictionary transform/format pipeline separation** — Current pipeline is extract→write (monolithic). Style Dictionary's model: source tokens → transforms (name/attribute/value math) → formats (CSS vars, iOS, Android, JS). Separation means each stage is independently testable and replaceable. For sk-design-md-generator: 'extraction → typed tokens → formatted DESIGN.md' is cleaner than 'extraction → AI-write-everything'. The transform stage becomes a no-op for the primary path (values are already measured, not derived), but the FORMAT stage is where deterministic table generation lives. This also opens a future path: if tokens.json is DTCG-shaped, any Style Dictionary formatter can consume it for non-DESIGN.md outputs (CSS custom properties file, Tailwind config, etc.).
   - Recommendation: Don't import Style Dictionary (2.8k deps) — adopt its architecture pattern instead. tokens.json is the 'source' file. formatters.ts is the 'format' step. The AI narrative overlay is an optional 'post-format annotation'. This keeps the tool lean while gaining the pipeline separation. Phase: architecture refactor (same as above, no new files).
   - Evidence: Style Dictionary architecture (source→transform→format), Terrazzo plugin emitters pattern.
4. **[P1] Aliasing for semantic color roles vs free authorial naming** — Current DESIGN.md color roles are AI-inferred: the AI reads tokens.colorTokens and assigns semantic labels (primary, secondary, surface, text) based on frequency and usage heuristics. This is fragile — the same extraction may produce different role assignments on different runs. DTCG aliasing creates explicit semantic mappings: {"color.primary": {"$value": "{color.brand.500}", "$type": "color"}}. Concretely: tokens.json could include a 'semanticAliases' section that maps role names to extracted token indices, computed deterministically from frequency+usage rules (e.g., highest-frequency bgColor = surface, highest-chromatic bgColor = primary). The AI then uses aliases instead of re-inferring each time.
   - Recommendation: Add semanticAliases to tokens.json output: {primary: colorTokens[4], surface: colorTokens[0], text: colorTokens[2], ...} computed by deterministic rules in cluster.ts. AI uses these aliases in DESIGN.md writing. File: tool/scripts/cluster.ts (add alias computation), tool/scripts/types.ts (add SemanticAliases interface). Phase: extraction output.
   - Evidence: W3C DTCG aliasing spec, Style Dictionary reference resolution.

## Summary
The single highest-value adoption is DTCG $type/$value wrapping of tokens.json (P0) — it unblocks type-gated validation, deterministic table generation, and downstream emit interop in one change. Combined with deterministic value-table formatters (P0), this eliminates the two primary hallucination vectors (phantom colors AND fabricated typography/shadow values) by keeping AI out of value-table generation entirely. The pipeline becomes: extract → DTCG-typed tokens → deterministic tables + AI narrative → type-gated validate.
