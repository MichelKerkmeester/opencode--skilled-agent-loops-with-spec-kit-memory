## Mechanism
OpenLTM’s memory source of truth is the SQLite `memories` row: the module says it “replaces skills/learned/*.md with structured SQLite + FTS5,” the schema defines `content`, `dedup_key UNIQUE`, `confirm_count`, and `status` on `memories`, and the markdown dump is explicitly “auto-generated” and “do not edit directly” (`packages/openltm-core/src/db.ts:1`, `packages/openltm-core/src/schema.sql:35`, `packages/openltm-core/src/schema.sql:45`, `packages/openltm-core/src/db.ts:709`, `packages/openltm-core/src/db.ts:727`). This depends on the row-write model.

`learn()` scrubs content, derives a normalized `dedupKey`, looks up an existing row by `dedup_key`, then either updates that row by incrementing `confirm_count`, `last_confirmed_at`, and `confidence`, or inserts a new row and returns `created` with `confirm_count: 1` (`packages/openltm-core/src/db.ts:304`, `packages/openltm-core/src/db.ts:307`, `packages/openltm-core/src/db.ts:314`, `packages/openltm-core/src/db.ts:317`, `packages/openltm-core/src/db.ts:323`, `packages/openltm-core/src/db.ts:347`, `packages/openltm-core/src/db.ts:352`, `packages/openltm-core/src/db.ts:414`). This is row-coupled reinforcement, not document editing.

OpenLTM provenance and audit are per-row tables: `memory_provenance.memory_id` is a FK to `memories(id)`, while `memory_audit.memory_id` intentionally is not a FK so audit survives row deletion (`migrations/008_add_memory_provenance.sql:6`, `migrations/008_add_memory_provenance.sql:8`, `migrations/009_add_memory_audit.sql:1`, `migrations/009_add_memory_audit.sql:2`, `migrations/009_add_memory_audit.sql:6`). The DAO snapshots row fields from `memories` and writes before/after JSON audit rows (`packages/openltm-core/src/dao/provenanceAudit.ts:15`, `packages/openltm-core/src/dao/provenanceAudit.ts:18`, `packages/openltm-core/src/dao/provenanceAudit.ts:37`). This is mostly row-write audit machinery.

OpenLTM’s cited `dedup.ts` is not a content hash; it lowercases, strips punctuation, collapses whitespace, trims, and truncates content to 200 chars (`packages/openltm-core/src/dedup.ts:5`, `packages/openltm-core/src/dedup.ts:8`, `packages/openltm-core/src/dedup.ts:11`). A separate janitor also deduplicates row memories by embedding similarity and can merge rows by updating content, marking one row superseded, repointing relations, and boosting `confirm_count` (`packages/openltm-core/src/janitor/dedup.ts:64`, `packages/openltm-core/src/janitor/dedup.ts:90`, `packages/openltm-core/src/janitor/dedup.ts:244`, `packages/openltm-core/src/janitor/dedup.ts:265`, `packages/openltm-core/src/janitor/dedup.ts:324`). This is row identity management.

## Row-coupled findings (negative knowledge)
- `created|reinforced` is negative knowledge: it treats duplicate row text as confirmation evidence, which authored docs do not imply (`packages/openltm-core/src/db.ts:96`, `packages/openltm-core/src/db.ts:321`).
- `confirm_count` as truth-strength is negative knowledge: it is a row counter, not a document-quality or decision-validity signal (`packages/openltm-core/src/schema.sql:48`, `packages/openltm-core/src/db.ts:324`).
- Per-memory provenance rows are negative knowledge for canonical authored memory: source traceability belongs in the doc, continuity metadata, and git, not a shadow row table (`migrations/008_add_memory_provenance.sql:1`, `migrations/008_add_memory_provenance.sql:8`).
- Row merge/supersede dedup is negative knowledge: it mutates row identity, relations, tags, and counters instead of asking an author to reconcile documents (`packages/openltm-core/src/janitor/dedup.ts:264`, `packages/openltm-core/src/janitor/dedup.ts:276`, `packages/openltm-core/src/janitor/dedup.ts:324`).
