## Mechanism
`EvaluateSession` resolves a transcript from hook input or Claude history, parses JSONL, and exits if the transcript is missing or has fewer than five messages (`hooks/src/EvaluateSession.ts:123`, `hooks/src/EvaluateSession.ts:138`, `hooks/src/EvaluateSession.ts:146`). It mines assistant text, tool uses, error tool results, modified-file paths, then writes generated session-pattern markdown and a learned-patterns summary (`hooks/src/EvaluateSession.ts:86`, `hooks/src/EvaluateSession.ts:159`, `hooks/src/EvaluateSession.ts:170`, `hooks/src/EvaluateSession.ts:207`, `hooks/src/EvaluateSession.ts:219`, `hooks/src/EvaluateSession.ts:291`). Its memory-capture path is proposal-first: error blocks become `MemoryProposal`s, optional LLM extraction runs only when `ltm.evaluateSessionLlm` is enabled, and merged proposals are written to `${PLUGIN_DATA}/proposals/<session>.json` rather than directly learned (`hooks/src/EvaluateSession.ts:223`, `hooks/src/EvaluateSession.ts:236`, `hooks/src/EvaluateSession.ts:240`, `hooks/src/EvaluateSession.ts:245`, `hooks/src/EvaluateSession.ts:246`).

`llmExtract` has two materially different paths: `extractAndLearn()` calls an LLM, parses decisions/gotchas/patterns/progress, and calls `learn()` for each extracted item (`hooks/lib/llmExtract.ts:30`, `hooks/lib/llmExtract.ts:40`, `hooks/lib/llmExtract.ts:50`, `hooks/lib/llmExtract.ts:53`); `extractProposals()` uses the same extraction prompt but returns proposal objects without writing to the DB (`hooks/lib/llmExtract.ts:69`, `hooks/lib/llmExtract.ts:90`, `hooks/lib/llmExtract.ts:94`, `hooks/lib/llmExtract.ts:100`). The row dependency is explicit in `learn()`: it scrubs content, deduplicates by key, reinforces an existing `memories` row, or inserts a new `memories` row (`packages/openltm-core/src/db.ts:304`, `packages/openltm-core/src/db.ts:307`, `packages/openltm-core/src/db.ts:317`, `packages/openltm-core/src/db.ts:324`, `packages/openltm-core/src/db.ts:352`).

`GitCommit` is the clearest auto-mine path: the file describes a post-commit hook that auto-extracts learnings from diffs, is controlled by `config.ltm.gitLearnEnabled`, collects commit metadata/diff, applies ignore/min-size gates, spawns a detached extractor, and that extractor calls `extractAndLearn()` with source `git-commit:<hash>` (`hooks/src/GitCommit.ts:3`, `hooks/src/GitCommit.ts:10`, `hooks/src/GitCommit.ts:40`, `hooks/src/GitCommit.ts:48`, `hooks/src/GitCommit.ts:92`, `hooks/src/GitCommit.ts:118`, `hooks/src/GitCommit.ts:126`, `hooks/src/GitCommit.ts:132`, `hooks/src/GitCommit.ts:138`). The automation is opt-in by default: `evaluateSessionLlm` and `gitLearnEnabled` default to false (`src/config.ts:71`, `src/config.ts:73`).

`UpdateContext` records mid-session progress by reading the transcript, detecting `Write`/`Edit`/`MultiEdit`, building a progress line, then writing to DB context APIs when a DB exists, with markdown fallback only as generated progress state (`hooks/src/UpdateContext.ts:11`, `hooks/src/UpdateContext.ts:56`, `hooks/src/UpdateContext.ts:101`, `hooks/src/UpdateContext.ts:106`, `hooks/src/UpdateContext.ts:116`, `hooks/src/UpdateContext.ts:128`, `hooks/src/UpdateContext.ts:133`). `proposalQueue` itself is storage-light JSON, but OpenLTM’s accept path turns a proposal into a row via `learn()`; reject only removes it from the JSON file (`hooks/lib/proposalQueue.ts:11`, `packages/openltm-core/src/proposals.ts:52`, `packages/openltm-core/src/proposals.ts:78`, `packages/openltm-core/src/proposals.ts:87`, `packages/openltm-core/src/proposals.ts:105`). A safer automation model exists in `PreCompact`: it builds a bounded context summary from DB or files and writes a snapshot, not a new learned fact (`hooks/src/PreCompact.ts:12`, `hooks/src/PreCompact.ts:29`, `hooks/src/PreCompact.ts:56`, `hooks/src/PreCompact.ts:61`).

## Row-coupled findings (negative knowledge)
- `extractAndLearn()` should not be ported as-is: it turns LLM-mined snippets directly into `learn()` row writes with category/importance metadata.
- `GitCommit` auto-learning should not be ported: commit diffs are evidence for authored release notes or summaries, not standalone source-of-truth memories.
- OpenLTM proposal acceptance should not be ported directly: “accept” calls `learn()` and admits a row, while system-spec-kit admission must be a document edit plus index refresh.
- `UpdateContext` DB progress writes should not be ported directly: silent context rows bypass the authored continuity ladder.
- `confirm_count` reinforcement should not drive doc memory admission: reinforcing duplicate rows has no clean analogue to deliberate markdown source documents.

## Open questions
- Should system-spec-kit add an opt-in “propose handover update from transcript/git” command that runs before `/memory:save` but never indexes until accepted into docs?
- What retention/privacy rules should apply to generated proposal files or snapshots before they become authored docs?
- Should PreCompact snapshots be indexed, or remain session-recovery artifacts outside canonical memory?
