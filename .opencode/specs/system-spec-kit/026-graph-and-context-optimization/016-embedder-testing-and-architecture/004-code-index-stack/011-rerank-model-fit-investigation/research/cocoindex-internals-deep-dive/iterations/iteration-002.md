# Iter 2 — indexer.py deep-read — chunking, exclusions, path_class assignment, structural-indexer.ts chunk inspection

## TL;DR (2-3 sentences)

`indexer.py` indexes whole source files through `RecursiveSplitter` with runtime defaults of `chunk_size=1500`, `chunk_overlap=200`, and `min_chunk_size=250`, then stores both vector rows and FTS rows per chunk. The actual DB has `.opencode/skills/system-code-graph/mcp_server/lib/structural-indexer.ts` indexed as 67 TypeScript implementation chunks spanning lines 1-2234; the query-relevant concepts are present, but they are spread across early import/module chunks, parsing chunks, file-walker chunks, and final `indexFiles` chunks rather than one coherent `StructuralIndexer` class chunk. Sequential-thinking MCP was attempted six times for this iteration, but every call returned `user cancelled MCP tool call`, so the status reflects that instead of pretending the tool completed.

## Findings (numbered, with citations)

1. The chunking strategy is simple and retrieval-significant: the indexer reads the full file text and lets `RecursiveSplitter` produce overlapping chunks, with default target size 1500, overlap 200, and minimum chunk size 250. In typical long TypeScript files this means chunks are line-range slices, not semantic class/function documents; a large file can be split into many overlapping windows and query terms may land in different candidate rows. The actual `structural-indexer.ts` result confirms this: 67 chunks span lines 1-2234, with sample chunks for imports, parser conversion, file walking, phase orchestration, and `indexFiles()` separated across the file. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:33-39`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:273-283`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/config.py:370-387`; SQLite query: `COUNT(*)=67`, `MIN(start_line)=1`, `MAX(end_line)=2234`.

2. The indexed file set is code-only by default, then project settings and canonical exceptions refine it. Defaults include common source extensions but deliberately exclude markdown/text/rst from CocoIndex Code; live project settings include `py`, `js`, `ts`, `tsx`, shell, SQL, C-family, Rust, Go, Java, PHP, and Lua, while excluding hidden/system noise plus spec folders and changelog paths. The edge case is canonical skill material: selected `.opencode/skills/*/references/**` and asset trees can bypass hidden-directory exclusion, so `.opencode/skills/system-code-graph/.../structural-indexer.ts` is indexable despite living under a hidden top-level directory. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/settings.py:17-73`, `.cocoindex_code/settings.yml:1-46`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:218-242`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:351-362`.

3. `path_class` is coarse but deterministic, and `structural-indexer.ts` falls through to `implementation`. The order matters: vendor/generated/spec-research/tests/docs checks run before the default implementation class. `structural-indexer.ts` is under `.opencode/skills/...`, not under `specs`, not in `tests`, not markdown docs, and not vendor/build/dist, so the DB result `path_class=implementation` is expected. This also means the Phase 2 path-class boost could not distinguish the desired `.opencode` implementation from the mirrored `.gemini`, `.codex`, or `.claude` implementation copies; all four mirrors have identical 67-chunk implementation classification. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:55-93`; SQLite mirror query returned `.claude`, `.codex`, `.gemini`, `.opencode` each with `chunks=67` and `path_class=implementation`.

4. The relevant probe tokens are present, but no single chunk contains the whole natural-language intent "filesystem walker that emits typed structural symbols and import edges." The opening chunks include the module title and typed symbol imports (`CodeNode`, `CodeEdge`), chunks around lines 1221-1263 include `parseFile`, chunks around 1376-1411 and 2089-2123 include `findFiles`, and final chunks around 2194-2234 include `indexFiles` and `runPhases`. That supports Iter 1's warning about chunking as a failure surface: the file is indexed, but the highest-scoring candidate depends on which slice best matches the query, and competing stress-test chunks can still outrank it. Citations: `.opencode/skills/system-code-graph/mcp_server/lib/structural-indexer.ts:9-26`, `.opencode/skills/system-code-graph/mcp_server/lib/structural-indexer.ts:1221-1259`, `.opencode/skills/system-code-graph/mcp_server/lib/structural-indexer.ts:1391-1465`, `.opencode/skills/system-code-graph/mcp_server/lib/structural-indexer.ts:2077-2154`, `.opencode/skills/system-code-graph/mcp_server/lib/structural-indexer.ts:2194-2234`; DB chunk samples `44655`, `44701`, `44708`, `44684`, `44686`, `44717`, `44712`.

5. Phase 2's probe-14 behavior now looks less like a missing-index problem and more like a candidate selection/rerank/mirror problem. The DB proves the expected `.opencode` file and FTS row set exist; Phase 2 baseline and BGE+path-class missed probe 14, while Jina v3 hit it at top-5 but still ranked `indexer-types.ts` above the expected file. The bge-path-class rerank-score row also shows `.gemini/.../structural-indexer.ts` candidates in the head, so mirror alias competition is active before final output. Citations: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/baseline-bge.results.jsonl:6`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/bge-path-class.results.jsonl:6`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/jina-v3.results.jsonl:6`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/bge-path-class.rerank-scores.jsonl:6`.
