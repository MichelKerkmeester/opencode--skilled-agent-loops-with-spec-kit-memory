## Mechanism
OpenLTM stores a materialized `decay_score` on `memories`, defaults it to `1.0`, and indexes it descending for relevance-order reads (`migrations/011_add_decay_score_and_archive.sql:4`, `migrations/011_add_decay_score_and_archive.sql:6`). Its janitor refresh is two SQL `UPDATE`s in one transaction: first recompute active-memory scores, then mark low scores deprecated (`packages/openltm-core/src/janitor/decay.ts:14`, `packages/openltm-core/src/janitor/decay.ts:31`, `packages/openltm-core/src/janitor/decay.ts:40`). The score is `importance × confidence × 0.5^(age/halfLife)`, where `importance=5` and `confirm_count>=10` are non-decaying, importance 4/3/2/else use 180/90/30/14 days, and age is based on `COALESCE(last_used_at, last_confirmed_at, created_at)` (`packages/openltm-core/src/janitor/decay.ts:16`, `packages/openltm-core/src/janitor/decay.ts:20`, `packages/openltm-core/src/janitor/decay.ts:26`). The docs expose the same half-life table and formula, and say scores below `0.25` are soft-deprecated and not session-start injected (`docs/02-how-it-works.md:58`, `docs/02-how-it-works.md:70`, `docs/02-how-it-works.md:76`). The deprecation SQL enforces `status='active'`, `importance<5`, `confirm_count<10`, and `decay_score<0.25` (`packages/openltm-core/src/janitor/decay.ts:31`). Recall/query speed is helped because default recall omits the large embedding blob and can `ORDER BY decay_score DESC` in SQL (`packages/openltm-core/src/db.ts:522`, `packages/openltm-core/src/db.ts:524`). Access refreshes recency: recall updates `last_used_at`, `last_recalled_at`, and `recall_count`, and the temporal metadata migration adds recall timestamps/counts plus supersession fields (`packages/openltm-core/src/db.ts:552`, `migrations/006_temporal_metadata.sql:3`). Tests exercise the exported formula, including immortal importance 5, decreasing score with age, 180-day half-life for importance 4, and positive nonzero scores (`src/__tests__/decay.test.ts:52`, `src/__tests__/decay.test.ts:76`, `src/__tests__/decay.test.ts:94`, `src/__tests__/decay.test.ts:108`). Retention is staged: deprecated low-importance, rarely recalled memories with `decay_score<0.10` are copied as JSON into `memory_archive` before deletion, max 100 per run (`packages/openltm-core/src/janitor/archive.ts:4`, `packages/openltm-core/src/janitor/archive.ts:49`). UNKNOWN: docs claim deprecated memories can still be recalled explicitly, but the core recall query I read unconditionally adds `status = 'active'` (`docs/02-how-it-works.md:76`, `packages/openltm-core/src/db.ts:510`).

## Negative knowledge
Do not copy hard `confirm_count>=10` immortality; repeated access should increase FSRS stability, not make stale facts permanent.

Do not copy the apparent deprecated-recall ambiguity; define explicit `includeDeprecated` or `tier=deprecated` semantics.

Do not copy the parallel JS/SQL threshold drift: exported JS deprecation protects `confirm_count>=5`, while SQL janitor uses `>=10`.

Should stale FSRS memories become `deprecated`, `DORMANT`, or merely receive lower ranking?
