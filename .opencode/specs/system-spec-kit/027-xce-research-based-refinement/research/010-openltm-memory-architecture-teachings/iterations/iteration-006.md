## Mechanism
OpenLTM implements redaction as a fixed ordered `PATTERNS` array of `{id, regex, replacement}` objects, replacing matches with typed placeholders such as `[REDACTED:aws-access-key]` and `[REDACTED:jwt]` (`packages/openltm-core/src/secretsScrubber.ts:17`, `packages/openltm-core/src/secretsScrubber.ts:21`, `packages/openltm-core/src/secretsScrubber.ts:62`). The pattern set covers AWS access keys, contextual 40-char AWS secret keys, GitHub tokens, OpenAI/Anthropic/Google/Stripe/Slack tokens, JWTs, bearer tokens, authenticated DB/Redis connection strings, PEM private keys, and generic quoted API-key-like assignments (`packages/openltm-core/src/secretsScrubber.ts:18`, `packages/openltm-core/src/secretsScrubber.ts:26`, `packages/openltm-core/src/secretsScrubber.ts:31`, `packages/openltm-core/src/secretsScrubber.ts:36`, `packages/openltm-core/src/secretsScrubber.ts:41`, `packages/openltm-core/src/secretsScrubber.ts:46`, `packages/openltm-core/src/secretsScrubber.ts:51`, `packages/openltm-core/src/secretsScrubber.ts:56`, `packages/openltm-core/src/secretsScrubber.ts:61`, `packages/openltm-core/src/secretsScrubber.ts:66`, `packages/openltm-core/src/secretsScrubber.ts:71`, `packages/openltm-core/src/secretsScrubber.ts:76`, `packages/openltm-core/src/secretsScrubber.ts:81`).

The scrubber is sequential and fail-open: it resets each regex `lastIndex`, applies `replace`, records deduplicated pattern IDs in a `Set`, and returns original text with no redactions if anything throws (`packages/openltm-core/src/secretsScrubber.ts:90`, `packages/openltm-core/src/secretsScrubber.ts:93`, `packages/openltm-core/src/secretsScrubber.ts:96`, `packages/openltm-core/src/secretsScrubber.ts:100`, `packages/openltm-core/src/secretsScrubber.ts:101`). The write path invokes `scrubSecrets(input.content)` immediately in `learn()`, before deduplication, existing-row lookup, insert, markdown export, embedding enqueue, and auto-relation work (`packages/openltm-core/src/db.ts:304`, `packages/openltm-core/src/db.ts:307`, `packages/openltm-core/src/db.ts:314`, `packages/openltm-core/src/db.ts:317`, `packages/openltm-core/src/db.ts:352`, `packages/openltm-core/src/db.ts:394`, `packages/openltm-core/src/db.ts:401`). It logs only count and pattern IDs, not secret values (`packages/openltm-core/src/db.ts:309`). It derives fallback titles from scrubbed content, but caller-provided `title`, `source`, `project_scope`, and `provenanceMetadata` are not scrubbed in the shown path (`packages/openltm-core/src/db.ts:350`, `packages/openltm-core/src/db.ts:361`, `packages/openltm-core/src/db.ts:382`, `packages/openltm-core/src/db.ts:383`).

I found no dedicated scrubber test in the scoped `packages/openltm-core/src/__tests__/*`; the files visible there are CLI/install and extension capability tests (`packages/openltm-core/src/__tests__/cli.test.ts:1`, `packages/openltm-core/src/__tests__/extensions.test.ts:1`). In system-spec-kit, direct `memory_save` parses file content and persists `parsed.content` as `contentText` into `memory_index`, while the existing redaction gate is scoped to extracted working-memory summaries and stores only a redaction flag, not redacted content (`.opencode/skills/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:244`, `.opencode/skills/system-spec-kit/mcp_server/handlers/save/create-record.ts:316`, `.opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-mutations.ts:301`, `.opencode/skills/system-spec-kit/mcp_server/lib/extraction/extraction-adapter.ts:246`, `.opencode/skills/system-spec-kit/mcp_server/lib/cognitive/working-memory.ts:449`).

## Negative knowledge
- Do not copy OpenLTM’s fail-open behavior unchanged; returning original text on scrubber failure can persist the exact secret.
- Do not scrub only `content`; titles, trigger phrases, provenance metadata, and routing summaries can also carry secrets.
- Do not copy the broad generic API-key regex without calibration; it can over-redact whole quoted assignment fragments.
- Do not redact emails by default in long-term spec memory unless a privacy policy requires it; project docs often contain legitimate contact identifiers.

## Open questions
- Should redacted canonical spec files be rewritten, or should only the indexed `content_text`, embeddings, and FTS rows be scrubbed?
- Should redaction block the save when the redaction ratio is high, matching our extraction skip-threshold pattern?
- Should secret-redaction metadata become searchable/auditable as a governance field without storing matched values?
