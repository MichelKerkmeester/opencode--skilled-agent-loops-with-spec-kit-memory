## What Changed

#### Eight playbook SKIPs promoted to runnable

The original SKIP rationale was wrong or out of date on eight scenarios. DV-003 (stdin redirect), DV-012b (skills show), DV-013b (mcp add/get/remove lifecycle), DV-015 (resume by id), DV-021 / DV-022 / DV-023 / DV-024 (cross-AI dispatch from each sibling CLI) all run cleanly on the real binary with the right invocation. Wave-2 evidence (`evidence/playbook-run-wave2-2026-05-15.md`) captures the working commands. cli-devin shifts from 16 PASS / 13 SKIP after v1.0.0.0 to 24 PASS / 3 SKIP after v1.0.2.0.

---

#### Six binary-surface corrections

Empirical testing revealed gaps the v1.0.1.0 docs still carried. Session ids are human-friendly kebab-case slugs like `paint-bean`, not UUIDs. `devin list --format json` is the canonical non-interactive parseable list shape. `devin mcp` has a full lifecycle (add / get / list / remove / login / logout / enable / disable), not just the three commands originally documented. `devin mcp add` requires a `--` separator before stdio commands: `devin mcp add <NAME> -- <CMD> [ARGS]`. `devin cloud drs` (Declarative Repo Setup — environment blueprints, sandbox sessions, builds) is the real cloud subcommand surface and is accessible under Codeium / Windsurf basic Pro. Empirical testing also showed Devin does NOT exhibit the silent stdin-theft failure mode that cli-codex / cli-opencode docs describe — the `</dev/null` redirect is now documented as a portability convention rather than a load-bearing requirement.

---

#### DV-018 cloud handoff split into surface + round-trip

DV-018 in v1.0.1.0 conflated two different tests: (a) is the cloud surface reachable at all (`devin cloud --help` returning the subcommand list) and (b) does an async laptop-close-to-PR round trip actually work. The first is shell-runnable in seconds on any tier that includes basic cloud access. The second requires interactive TUI handoff initiation, cloud-VM entitlement, and a multi-hour wait. v1.0.2.0 splits these into DV-018 (the original round-trip scenario, now explicitly marked operator-driven manual) and the new DV-027 (cloud surface accessibility, shell-runnable).

---

#### Three legitimate SKIPs reaffirmed

DV-017 (5-check gate negative), DV-019 (self-invocation guard), and DV-020 (cloud-handoff exception) remain SKIP. These test cli-devin's smart-router logic, which runs at the calling-AI orchestrator layer rather than via the `devin` binary. They are meaningful operator-runnable manual tests, not shell-automatable binary tests. The scenario files now state this explicitly so future automated runs do not retry the same wrong execution path.

---

#### SWE-1.6 prompt-quality contract (NEW in v1.0.2.0)

SWE-1.6 is coding-specialized but smaller than the complex-task models (DeepSeek v4, GLM 5.1, Kimi k2.6) — it relies on the calling AI doing structural decomposition upfront rather than figuring it out mid-loop. EVERY `--model swe-1.6` dispatch now MUST be composed through `sk-prompt` (using STAR / RCAF / BUILD framework + CLEAR 5-check) AND must include an explicit `<pre-plan>` block in the prompt body — ordered steps with per-step acceptance criteria, stop conditions, and a verification approach — BEFORE handing control to SWE-1.6. The canonical pre-planned template is `assets/prompt_templates.md §2`. SKILL.md adds this as ALWAYS rule #12, and the prompt-quality card now marks STAR / RCAF / BUILD as the three canonical SWE-1.6 frameworks with the contract requirement called out inline.

When pre-planning reveals the task is more complex than SWE-1.6's clearly-defined zone (ambiguous requirements, multi-step reasoning, large refactor scope), the calling AI should escalate to `--model deepseek-v4` rather than throwing a longer freeform prompt at SWE-1.6. This is the single largest improvement to cli-devin's effective output quality this release ships.

## Upgrade

**No invocation breaking changes.** Existing v1.0.1.0 dispatches keep working — v1.0.2.0 corrections are doc-accuracy improvements that match what the binary already accepts.

**Operationally significant for SWE-1.6 work**: the new ALWAYS rule #12 requires production `--model swe-1.6` dispatches to be composed through `sk-prompt` (STAR / RCAF / BUILD framework + CLEAR 5-check) AND include an explicit `<pre-plan>` block. The canonical pre-planned SWE-1.6 template is in `assets/prompt_templates.md` §2. Existing simple/test SWE-1.6 dispatches keep working; this rule applies to production-style coding work. When pre-planning reveals a task is more complex than SWE-1.6's clearly-defined zone, escalate to `--model deepseek-v4` rather than throw a longer freeform prompt at SWE-1.6.

**Playbook scenario updates** (operators consuming the playbook for automated runs): pick up the corrected invocations in the v1.0.2.0 scenario files — `devin -p "<prompt>"` for positional prompts, `devin list --format json` for session-id capture (session ids are kebab-case slugs like `paint-bean`, not UUIDs), `devin mcp add <NAME> -- <CMD> [ARGS]` separator syntax. The new shell-runnable DV-027 cloud surface accessibility check should run BEFORE attempting the operator-driven DV-018 round-trip — DV-027 confirms tier reachability in seconds without committing to the multi-hour async round-trip.
