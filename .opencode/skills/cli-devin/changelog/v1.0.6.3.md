# cli-devin v1.0.6.3 — RULE #14 truth-up against the binary

**Released**: 2026-05-18
**Status**: Documentation-only correction. SKILL.md RULE #14 brought into agreement with `references/agent-config-recipes.md` and the actual Devin binary behavior.

---

## Summary

`SKILL.md` §4 ALWAYS rule #14 (Sequential_thinking MCP mandate) was telling callers to configure the MCP server "via the `--agent-config` recipe's `mcp_servers` field". The Devin binary v2026.5.6+ has rejected that shape since v1.0.4.0 (`untagged enum McpServer` parser error + `ACP: agent_config mcp_servers are not yet supported in the ACP path and will be ignored` self-log). `references/agent-config-recipes.md` has correctly documented the 2-layer working pattern (user-scope `devin mcp add` registration + `system_instructions` mandate) for several versions, but RULE #14 itself was never updated. This addendum fixes that contradiction so callers reading the rule first don't waste a dispatch attempt on a config-shape rejection.

The bug class manifested during the daemon-corruption arc (packets 006-009): the first cli-devin dispatch for packet 006 failed with `Failed to parse agent config: mcp_servers: data did not match any variant of untagged enum McpServer at line 40 column 18` because the dispatching agent followed RULE #14 verbatim. The fix at the time was empirical (remove the `mcp_servers` block and rely on `devin mcp list` already-registered server); this addendum upstreams that fix into the rule itself.

## Why this lives in changelog/ and not as a phase reference in SKILL.md

Per the project convention (`feedback_skill_docs_no_phase_references.md`): SKILL.md states the present-tense desired logic (the 2-layer pattern). The history of WHY the rule was wrong and HOW we discovered it (failed dispatch in packet 006) stays here in `changelog/`.

---

## Caveats

- The recipe-level `mcp_servers` field MAY become functional in a future Devin release. When it does, RULE #14 + the recipes ref doc should be updated together to prefer the recipe-level form (it's more reproducible and less profile-dependent than user-scope registration). Both surfaces now consistently say "RESERVED but not yet wired", so the upgrade path is unambiguous.
- This addendum does NOT change the 5-thoughts threshold, the structural-output requirement, or the model-agnostic scope of the rule — those remain in force.
