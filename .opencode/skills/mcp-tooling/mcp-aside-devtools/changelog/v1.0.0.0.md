## [**1.0.0.0**] - 2026-07-16

This first release delivered the Aside AI browser as the workflow packet for natural-language browser tasks, deterministic Playwright-compatible evidence capture, and MCP composition through Code Mode. It combined the registered `aside` manual, the installed CLI surface, live discovery evidence, Chrome-DevTools-parity guidance, operator validation material, and template-conformant package documentation without overstating authenticated or browser-bound behavior.

#### Three lanes, one router

- Need: Browser work divided into outcome-oriented tasks, deterministic proof steps, and multi-tool composition, and each activity required a different surface.
- Change: The packet routed weighted TASK, REPL, MCP, INSTALL, and TROUBLESHOOT intents across `aside "<task>"`, `aside --session <id> "<task>"`, `aside repl "<code>"`, and the Aside MCP `repl` tool through Code Mode. The MCP lane served as the composition fallback while the CLI remained the primary operator surface.
- Why it mattered: Agents selected the simplest suitable lane and kept repeatable evidence work out of the nondeterministic task lane.

&nbsp;

#### Registered backend and live discovery

- Need: The packet required one authoritative account of the registered manual, discovery naming, and helper inventory.
- Change: The release documented the registered stdio manual that launched `aside mcp` with an intentionally empty `env`, preserved its byte-true snapshot in `assets/utcp-aside-manual.md`, and treated manual absence as a diagnostic error. A dated 2026-07-16 fixture recorded the single registry tool as `aside.aside.repl` and the TypeScript callable as `aside.aside_repl(args)`, with per-session rediscovery still required because the tool list could change.
- Why it mattered: Consumers could distinguish the dotted discovery name from the underscore call form and diagnose wiring drift without re-registering or editing operator-owned configuration.

&nbsp;

#### CLI, REPL, and session contract

- Need: Operators needed a version-pinned command surface and clear browser/session boundaries.
- Change: The references covered task starts and continuation, account selection and status commands, `aside mcp`, `aside repl`, the three-layer session model, permission and concurrency rules, and the discovered REPL helpers for tabs, snapshots, display, pages, screenshots, PDF, filesystem utilities, sleep, and authenticated fetch. The docs required `console.log` for returned values, neutral-session tab attachment, explicit timeouts, independent artifact verification, and a single-writer posture while multi-client isolation remained unresolved.
- Why it mattered: Deterministic calls used the observed helper contract, while browser binding, permission inheritance, bound-page result shapes, and parallel isolation stayed fail-closed instead of becoming assumptions.

&nbsp;

#### Chrome DevTools parity

- Need: The packet needed an honest map from the discovered Playwright `page` API to the browser-debugging jobs covered by `mcp-chrome-devtools`.
- Change: Six feature leaves and six matching playbook scenarios documented DOM inspection, screenshots and PDF, console and network event capture, cookies and storage, performance timing, and navigation with multiple tabs. The patterns used `page` and `page.context()` APIs confirmed as reachable by the discovery fixture, and bound-session executions remained SKIP-valid until an authorized browser session could verify their live result shapes.
- Why it mattered: The packet exposed practical parity without claiming native HAR export, Lighthouse audits, raw-CDP passthrough, dedicated console or network tools, or parallel isolated browser instances.

&nbsp;

#### Evidence, catalog, and operator validation

- Need: Capability claims, installation posture, and recovery behavior required one auditable package rather than scattered prose.
- Change: The release shipped an 18-capability feature catalog across the five router intents and the DevTools-parity family, a 21-scenario manual testing playbook with routing-recall holdouts and recovery cases, three example workflow scripts, a macOS installer wrapper, a read-only doctor with an MCP handshake probe, and CLI/MCP server-package pointers. The dated online-research reference corroborated the public CLI/MCP/REPL surface, verbatim `mcp.json` wiring, permission model, hidden-password behavior, and installer-only distribution while retaining CONFIRMED, INFERRED, and UNKNOWN labels.
- Why it mattered: Operators could validate installation, registration, discovery, routing, evidence capture, and failure recovery end to end, with authenticated smoke calls and browser-bound probes explicitly operator-gated or SKIP-valid.

&nbsp;

#### Package and template conformance

- Need: The packet had to match the create-skill and create-readme contracts without changing its runtime behavior.
- Change: `SKILL.md`, `README.md`, `INSTALL-GUIDE.md`, references, assets, examples, feature catalogs, playbook scenarios, server pointers, and script documentation used kebab-case filesystem names and resolved links. Reference and asset documents used an OVERVIEW-first Purpose and Usage structure, and `scripts/README.md` used the code-folder README sections for architecture, directory tree, key files, entrypoints, validation, and related resources, grounded in the actual mutation and exit-code behavior of `install.sh` and `doctor.sh`.
- Why it mattered: The package remained tool-compatible, navigable, and strictly aligned with its authoring templates while preserving its technical and epistemic boundaries.
