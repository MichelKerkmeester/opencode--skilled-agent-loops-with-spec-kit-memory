# Iteration 2: A2 Timeout Hardening

## Focus

This iteration designs the concrete timeout-proof execution model for `memory_index_scan`: a bounded scan-job worker with a lexical-first commit boundary, resumable queue/checkpoints, continuation cursors for compatibility sync mode, and failure semantics that park vector work without failing the scan. It also begins A3 by defining what concurrent callers see under scan-key coalescing.

## Actions Taken

- Read the deep-research state log, strategy, quick reference, and prior iteration to preserve the A1 recommendation and focus only on A2.
- Re-read `memory_index_scan`, scan lease state, embedder re-index jobs, embedder status, execution-router batching, retry manager, IPC bridge, and save/indexing paths.
- Verified where current scan work is unbounded, where deferred lexical indexing already exists, and where retry retention can become unsafe for a large attempted backlog.
- Designed one recommended execution model and documented tradeoffs.

   Recommended queue contract: add `index_scan_jobs` keyed by `scanKey` plus `index_scan_work_items`. A scan job records normalized scope/options, `phase`, `status`, `totalFiles`, `filesDiscovered`, `filesQueued`, `filesIndexed`, `filesSkipped`, `filesFailed`, `staleDeleted`, `pendingVectors`, `retryVectors`, `failedVectors`, `cursor`, `startedAt`, `updatedAt`, and `errorSummary`. A work item is one canonical file for lexical work because file-level parse/validate/upsert is the smallest unit that maps to current indexing semantics and mtime safety. Each work item stores canonical path, original path, spec folder, file hash/mtime snapshot, kind, status (`queued`, `running`, `indexed`, `skipped`, `failed`), attempt count, last error, and committed memory id if one exists.

   Checkpoint semantics: discovery may run as a bounded producer that inserts work items in deterministic canonical-path order and updates `filesDiscovered`/`cursor` after each producer page. 4. Vector drain should be separate from lexical drain and should not rely solely on the current default retry loop for large scan backlogs.

   Recommended vector contract: the scan job enqueues or wakes a dedicated bounded vector-drain step after lexical commit. 5. Failure and retry semantics need one refinement: do not move a large clean backlog from `pending` to `retry` just because the provider circuit is open.

   Recommended semantics: vector drain must check provider/circuit state before claiming clean pending rows. 6. Continuation cursors should be a compatibility escape hatch, not the primary model.

   The primary contract should return a job handle immediately. If a caller explicitly requests synchronous bounded work, `memory_index_scan` should accept `mode: 'bounded'` or `waitForCompletionMs` and return a `continuationCursor` whenever it reaches a cap. The cursor should encode only a server-stored job id plus phase token, not raw file lists, so it survives path dedup, stale cleanup, process restart, and queue reordering. Each continuation processes another bounded slice: discovery page, lexical file-item batch, or vector row batch. When the cap is reached, the response includes `status: 'in_progress'`, `phase`, `jobId`, `continuationCursor`, `processedThisCall`, `progress`, and `nextPollAfterMs`. A cursor call that finds the job already completed returns the final job summary.

7. A3 initial concurrency answer: concurrent callers should coalesce on `scanKey`, not compete for the lease.

   The lease currently distinguishes active lease from cooldown in DB state [SOURCE: .opencode/skills/system-spec-kit/mcp_server/core/db-state.ts:443] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/core/db-state.ts:456], but the handler collapses both to one `E429` response [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/memory-index.ts:245] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/memory-index.ts:247]. Under the new model, lease acquisition is a worker-start guard only: request N computes the same `scanKey`, atomically finds or creates the active job, and returns that job with `coalesced: true` if it already exists. Different scan keys may queue behind the single writer. IPC secondary clients are already capped to 8 by default [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/ipc/socket-server.ts:13] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/ipc/socket-server.ts:49] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/ipc/socket-server.ts:127], and excess secondary connections are refused at the bridge layer [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/ipc/socket-server.ts:133] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/ipc/socket-server.ts:135]. Scan coalescing should sit below clients in the shared DB/job layer so primary and secondary callers observe the same active job.

## Questions Answered

- A2 is answered. The recommended model is an idempotent async scan job with a lexical-first phase boundary, persisted file work items, bounded drain ticks, checkpointed progress, and a separate vector drain that leaves clean pending rows pending during provider outages.
- The exact phase boundary is lexical DB/search commit: after the per-file row/projection/BM25/FTS/mtime/hook work has committed, the scan is searchable and successful even if vectors lag.
- The work item should be a canonical file for lexical scan work, plus memory row ids for vector drain. Batches are worker implementation details, not durable identity.
- Continuation cursors are useful only for bounded synchronous compatibility; the default UX should be job id plus polling/coalescing.
- A3 has an initial answer: all concurrent callers with the same normalized scan scope/options receive the same job, and lease/cooldown never surface as raw user-facing `E429`.
