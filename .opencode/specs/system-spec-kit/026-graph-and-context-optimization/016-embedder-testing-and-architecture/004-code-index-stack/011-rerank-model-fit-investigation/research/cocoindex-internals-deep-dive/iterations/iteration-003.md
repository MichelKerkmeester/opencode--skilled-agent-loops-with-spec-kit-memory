# Iter 3 — fusion.py deep-read — RRF + weights + top-K cutoff to rerank window

## TL;DR (2-3 sentences)

`fusion.py` uses weighted reciprocal-rank fusion, not raw score blending: each lane contributes `weight * (1 / (k + rank))`, with defaults `k=60`, `vector_weight=0.7`, and `fts_weight=0.7`. The reranker does not see "all fused candidates"; it sees the first `config.rerank_top_k` candidates after fusion, heuristic boosts, and mirror dedup, with the default head size set to 20. The cutoff is stable when input ranks and scores are unequal, but exact ties can inherit unordered `set` iteration and stable-sort ordering, so the implementation is deterministic in intent but not fully tie-stable across Python processes.

## Question (restate the scoped RQ)

Deep-read `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py`. Capture the exact RRF formula and the `k=60` implications, explain how vector weight `0.7` and FTS5 weight `0.7` combine, trace the top-K cutoff that determines how many candidates reach rerank, and answer: if dense plus FTS5 fusion produces 100 candidates and only top-20 go to rerank, what governs which 20, and is that cutoff stable or stochastic?

## Findings (numbered, with citations)

1. The exact RRF formula is `score = vector_weight/(k + vector_rank) + fts_weight/(k + fts_rank)`, omitting any lane where the chunk is absent. With defaults, a vector rank-1 only hit contributes `0.7 / 61 = 0.0114754098`, an FTS rank-1 only hit contributes the same, and a dual-lane rank-1 hit contributes `1.4 / 61 = 0.0229508197`. The normalized raw vector and FTS scores are not blended into this score; they are stored as `vector_score` and `fts_score` and used only as secondary sort keys after `rrf_score`. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py:28-39`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py:42-87`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/config.py:15-19`.

2. Equal weights of `0.7` do not mean the two lanes are averaged; they mean each lane has the same reciprocal-rank voting power, and a candidate present in both lanes can roughly double a same-rank single-lane candidate. With `k=60`, early rank differences are intentionally compressed: vector rank 1 is `0.0114754`, vector rank 20 is `0.00875`, and vector rank 40 is `0.007`. That makes lane agreement more important than tiny rank deltas, while still allowing a high single-lane item to compete with a low dual-lane item. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py:55-67`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/config.py:389-406`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:94-132`.

3. If dense plus FTS fusion produces 100 candidates, the 20 that reach rerank are governed by two stages, not `fusion.py` alone. First, `rrf_fuse()` sorts the union of vector and FTS chunk IDs by RRF score, then normalized vector score, then normalized FTS score. Second, `_dedup_and_rank_hybrid_rows()` converts fused rows to `QueryResult` objects, starts from `fused.rrf_score`, applies implementation/canonical path boosts or penalties, sorts by final `QueryResult.score`, and removes mirror aliases. The reranker receives `candidates[:20]` only after that stage when default `COCOINDEX_RERANK_TOP_K` is in effect. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py:79-87`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:399-444`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:481-515`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:675-683`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/reranker.py:153-183`.

4. The candidate supply is smaller than the phrase "100 fused candidates" might imply for the default CLI/MCP top-10 search. `query_codebase()` derives `fetch_k = (limit + offset) * 4`; with `limit=10, offset=0`, each lane asks for 40 candidates, so the fused union can be at most 80 before dedup. A fused set of 100 would require a larger requested result window or offset, or enough per-lane candidates under a different caller configuration. The formula matters for Phase 2 because the smoke bench captured top-5 outputs but the underlying search default and rerank default still let only a bounded head reach model scoring. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:594-640`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:690-695`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/run-phase2-smoke.sh:14-24`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/run-phase2-smoke.sh:48-51`.

5. The cutoff is mostly stable, but not perfectly tie-stable. Vector and FTS input lists are ordered by database scores, and the main RRF sort is deterministic for unequal `(rrf_score, vector_score, fts_score)` tuples. The caveat is `chunk_ids = {vector ids} | {fts ids}` in `rrf_fuse()`: exact ties that survive all sort keys preserve the iteration order of that set through Python's stable sort, and `_dedup_and_rank_hybrid_rows()` later sorts only by final score, preserving earlier order for final-score ties. So repeated runs with identical DB, Python hash seed, environment, and input ordering should pick the same head, but exact ties can drift across processes or runtimes unless explicit chunk-id/path tie-breakers are added. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py:53-87`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:178-205`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:234-245`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:121-132`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:502-515`.

6. Phase 2's `pre_score` logs are consistent with the code path: many logged candidates have scores around `0.0114754098`, exactly the rank-1 single-lane default-weight RRF contribution, and probe 14's head includes `.gemini/.../structural-indexer.ts` before rerank rather than the desired `.opencode` mirror. That supports Iter 2's candidate-selection/mirror diagnosis: the reranker can reorder only the head it receives, and path-class boost cannot distinguish mirrors when they share the same `implementation` class. Citations: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/bge-path-class.rerank-scores.jsonl:6`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/jina-v3.rerank-scores.jsonl:6`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/cocoindex-internals-deep-dive/iterations/iteration-002.md:31-35`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:247-272`.
