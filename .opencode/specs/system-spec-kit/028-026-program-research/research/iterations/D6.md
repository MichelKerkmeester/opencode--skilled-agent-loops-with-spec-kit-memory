### Angle 16: Front-proxy transparency — adversarial timing analysis
- **Framing:** There exists a bounded temporal window during daemon recycle where a client observes a `-32001` (retryable) or `-32002` (non-retryable) error rather than transparent recovery.
- **Repo evidence:** `launcher-session-proxy.cjs:11-17` (reattach backoff ladder `[100,250,500,1000,1500]ms` × 40 attempts ≈ 41s; cold-start capped at 30 attempts at line 17, reduced from 120 in 018-hardening). `mk-spec-memory-launcher.cjs:1053-1102` (recycleDaemonInPlace — kills children, holds lease, launcher stays alive). `launcher-ipc-bridge.cjs:124-210` (deepProbe requires JSON-RPC `initialize` reply; hung-daemon detection at line 127-130).
- **Findings:** Transparency holds for replayable reads and idempotent writes (`REPLAYABLE_TOOL_NAMES` at line 33-47) — these are replayed after reattach with no client-visible error. The client observes an error in three scenarios: (a) a non-replayable tool call (e.g. `memory_delete`) queued during REATTACHING — gets `-32001` immediately because the proxy cannot safely replay it; (b) the respawned daemon returns a different `protocolVersion` — the proxy fails-closed with `-32002` requiring client reconnect; (c) the reattach ladder exhausts (pathological — daemon fails to respawn within ~41s) — all queued requests fail with `-32001`. The reattach backoff is intentionally bounded (formerly ~176s, hardened to ~41s) to avoid indefinite client hangs. The deepest adversarial window is the cold-start path: if the daemon crashes during the recycle itself (double-fault), cold-start attempts kick in, extending the window without raising the ceiling since both ladders are serial.
- **Next experiment:** Run a stress harness that sends concurrent read + write MCP requests while sending SIGTERM/SIGKILL to the daemon at randomized offsets within the recycle grace window. Measure: (a) percentage of `-32001` vs transparent replay; (b) whether any write is silently lost (acceptance: 0%); (c) maximum client-visible outage duration across 1000 cycles.
- **Verdict:** Transparency is well-engineered but not absolute — the contract is "retryable errors instead of silent hangs." The design tradeoff (bounded ~41s ladder vs. infinite patience) is correct. The cold-start-reduces-from-120-to-30 hardening closed the highest-risk window. Recommend: add a metric counter for `-32001` responses so operators can monitor degrade-to-error rates.

### Angle 17: Shared-daemon lease model — ungraceful lease-holder death
- **Framing:** When the lease-holder launcher dies ungracefully (SIGKILL, OOM, crash), secondary launchers must detect staleness, reclaim the lease, re-elect, and respawn the daemon without data corruption or competing writers.
- **Repo evidence:** `mk-spec-memory-launcher.cjs:346-363` (`classifyOwnerLease` — three staleness detectors: dead PID, ppid-1-orphan, heartbeat >2×ttlMs=120s). `mk-spec-memory-launcher.cjs:365-403` (`acquireOwnerLeaseFile` — exclusive `'wx'` create, re-read-after-write to verify no race). `mk-spec-memory-launcher.cjs:671-752` (`respawnAfterDeadSocket` — bootstrap lock, re-read leases, reap owner, reap child with clean-close barrier, respawn). `model-server-supervision.cjs` line 19: `RESPAWN_REAP_GRACE_MS=7000`. Heartbeat every `ttlMs/2`=30s (line 427-443).
- **Findings:** Three re-election paths exist with different latencies. (1) Clean SIGTERM death: owner runs `shutdownLauncherForSignal`, clears owner lease, secondary's next bridge probe detects no owner and re-elects — <1s + respawn. (2) Dead-socket detection: secondary's bridge probe returns `action:'respawn'` — acquires bootstrap lock, re-reads leases under lock, reaps old owner (SIGTERM→7s→SIGKILL), reaps child with `.unclean-shutdown` marker verification, spawns replacement — <15s if the daemon responds cleanly to SIGTERM. (3) Heartbeat-based staleness: owner dies silently (SIGKILL), heartbeat expires after 2×ttlMs=120s, secondary's periodic bridge check classifies as `stale-heartbeat-reclaim` — ~120s worst case. The ppid-1-orphan detector (line 351-354) catches the scenario where the launcher dies but its daemon child is reparented to init — faster than waiting for heartbeat timeout. The re-read-after-write pattern (line 392-400) prevents two concurrent reclaimers from both acting as owner. The `.unclean-shutdown` marker ensures replacement daemon knows whether to rebuild FTS at boot.
- **Next experiment:** Induce ungraceful lease-holder death via `kill -9` and `kill -19` (SIGSTOP to simulate hung process) at three points: mid-request, idle, and during child respawn. Measure: (a) time until secondary starts serving; (b) any dual-writer window (two processes opening the SQLite DB simultaneously); (c) data loss on the in-flight request.
- **Verdict:** The model is sound — exclusive file creation + heartbeat + dead-socket probing + ppid-1-orphan detection forms a defense-in-depth staleness classifier. The 120s heartbeat timeout is the longest-duration failure path. Mitigation exists: the dead-socket probe path (faster than waiting for heartbeat staleness) fires whenever a secondary actively needs the daemon. A watchdog on the lease-holder's own heartbeat failure (`shutdownLauncherForSignal` at line 436) elegantly self-terminates a degraded owner, kicking re-election immediately. **One gap: there is no external health-check daemon that actively monitors the lease holder** — if no secondary launcher connects during the 120s stale window, the dead daemon simply remains down until demand arrives.
