# Deep Research Strategy: Literal Interface Command Prompts

## 1. Research Topic

Rewrite the five `/interface:*` commands as literal, self-contained design prompts while preserving the split where commands own intake and lifecycle, `sk-design` modes own taste and judgment, transports own realization, and `sk-code` owns application-code mutation.

## 2. Non-Goals

- Do not modify commands, presentation assets, skills, workflows, tests, or packet docs.
- Do not write anywhere outside this detached `sol-high-fast` lineage.
- Do not reopen the settled five-command namespace or stable workflow-mode mapping without contradictory evidence.
- Do not duplicate palettes, typography prescriptions, style menus, reference inventories, or mode-owned taste doctrine into commands.
- Do not settle unrelated styles-library structure or database-fate work.

## 3. Stop Conditions

- Complete exactly five iterations because `stopPolicy=max-iterations`.
- Produce concrete, substantially self-contained command-body proposals for all five `/interface:*` commands.
- Resolve the runtime-supported include/shared-fragment mechanism with direct repository evidence.
- Provide a durable authority/anti-duplication rule and a testable implementation plan.

## 5. Answered Questions

- The shipped bodies preserve valuable command-specific intake, sibling discrimination, route proof, stable mode mappings, typed outputs, and mutation boundaries, but outsource the creation experience across presentations and the shared contract. The missing layer is a command-visible, evocative, actionable brief with experiential stakes, anti-generic constraints, real context/content, and a mode-specific quality bar. (iteration 1)
- OpenCode command templates support direct `@./path` file-content expansion. The recommended mechanism is literal, mode-specific creation prose in each command plus one `@./.opencode/skills/sk-design/shared/creation-contract.md` expansion for shared normative lifecycle and authority content. This yields runtime-expanded self-containment; checked-in source-byte duplication would require generation and is unnecessary. (iteration 2)
- Five concrete body cores now define the command-visible mission, intake, grounding, stable route, ordered work, targeted revision, proof ceiling, typed output, and handoff for design, foundations, motion, audit, and design-reference. Public orchestration stays literal, universal schemas arrive through the shared `@` contract, and values/judgment remain mode-owned. (iteration 3)
- Adversarial review preserved the missions and mode-specific quality rules but moved universal intake schemas, common blocks, targeted revision, evidence labels, and handoff continuity back to the shared contract. It restored `ASK`, required literal suffix control, made audit review-only, and restricted design-reference token tables to measured values. The safe shipment is test-first and atomic across wrappers, presentation/YAML ownership declarations, and metadata projections. (iteration 4)
- OpenCode 1.18.4 source confirms `@.opencode/skills/sk-design/shared/creation-contract.md`: leading-dot references are accepted and non-home references resolve from `ctx.worktree`. The final grammar, acceptance matrix, atomic rollout, and whole-change rollback are complete; an end-to-end sentinel is the first implementation gate, not an unresolved spelling decision. (iteration 5)

## 6. What Worked

- Exact cross-command anchor comparison separated the shared routing skeleton from meaningful domain-specific intake and exposed the missing prompt layer without discarding working architecture. (iteration 1)
- Exact command-authoring and renderer-symbol tracing distinguished native `@` expansion, `$ARGUMENTS` substitution, shell rendering, and generated-router patterns. (iteration 2)
- A common mission-to-handoff grammar plus one decisive mode-specific quality rule produced five concrete prompts without flattening their distinct jobs. (iteration 3)
- Phrase-level contract comparison and the 15-test baseline exposed regressions that thematic prompt review missed. (iteration 4)
- Matching the installed OpenCode version to tagged parser/resolver source resolved include rooting without violating the lineage-only boundary. (iteration 5)
<!-- /ANCHOR:what-worked -->

<!-- ANCHOR:what-failed -->
## 7. What Failed

- Broad repository search for generic include/template terms was noisy; exact command-runtime and renderer symbols are required for the next pass. (iteration 1)
- Historical compiled-stub descriptions were insufficient; direct live-command reads were required to establish the current runtime contract. (iteration 2)
- A broad all-command read truncated before all five routers; exact current anchors plus the prior synthesis covered the missing comparisons, but iteration 4 must narrowly verify authority-bearing phrases against each exact router. (iteration 3)
- Official documentation proves `@` content expansion but does not explicitly define the base of the exact `@./.opencode/...` spelling; the lineage-only boundary prohibits creating a scratch command outside this packet. (iteration 4)
- Broad local `@` examples could not prove OpenCode custom-command semantics; only tagged OpenCode parser/resolver source was authoritative. (iteration 5)

## 8. Exhausted Approaches

- Replacing mode judgment with command-owned taste: already rejected by the authority split established in prior research.
- Treating the current shared contract alone as the requested literal-prompt deliverable: contradicted by the verified gap analysis.
- Wholesale router replacement: exhausted because current prerequisites, route mappings, proof outputs, and handoff boundaries are useful and should be retained. (iteration 1)
- Presentation-only enrichment: exhausted because it leaves the literal command body dependent on external fragments and preserves the experiential gap. (iteration 1)
- Shell-output compilation for this rewrite: exhausted as disproportionate registry, drift, security, and manifest machinery when native `@` expansion exists. (iteration 2)
- Hand-duplicating the full lifecycle in five commands: exhausted because it creates normative drift without improving runtime self-containment. (iteration 2)
- Applying anti-generic creative rhetoric to `design-reference`: exhausted because extraction quality is source fidelity and provenance, not originality. (iteration 3)
- Wrapper-only literal enrichment while retaining presentations as normative prompt authority: exhausted because it creates competing command-package sources of truth. (iteration 4)
- Alternate canonical spelling `@./.opencode/...`: exhausted as path-equivalent but unnecessarily divergent from the documented project-relative form. (iteration 5)
<!-- /ANCHOR:exhausted-approaches -->

<!-- ANCHOR:ruled-out-directions -->
## 9. Ruled-Out Directions

- Command-owned palettes, typography recipes, style menus, or fixed visual doctrine: ruled out because modes own taste and design judgment. (iteration 1)
- Markdown links/read imperatives and presentation/YAML assets as loader includes: ruled out because they do not provide native file-content expansion. (iteration 2)
- Copying reference inventories, transport mechanics, exact palettes/fonts, token recipes, or validation procedures into command bodies: ruled out as mode/transport-owned doctrine. (iteration 3)
- Dropping `STATUS=ASK`, letting audit apply fixes, or allowing brief-provided values into design-reference token tables: ruled out by shared and mode contracts. (iteration 4)
- Treating the end-to-end include sentinel as an unresolved research decision: ruled out because tagged source settles spelling; the sentinel is an integration regression gate. (iteration 5)
