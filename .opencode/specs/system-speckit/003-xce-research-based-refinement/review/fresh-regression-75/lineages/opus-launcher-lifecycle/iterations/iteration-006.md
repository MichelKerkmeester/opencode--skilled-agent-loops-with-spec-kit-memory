## Analysis — seat `opus-launcher-lifecycle`

**Scope traced:** `mk-skill-advisor-launcher.cjs` lifecycle paths — the owner-lease heartbeat self-shutdown (`startOwnerLeaseHeartbeat`, 537-551), the respawn reap escalation (`reapOwnerBeforeRespawn`, 406-435), the model-server/sidecar shutdown escalation (`shutdownModelServerRoot`, 935-960 + `reapProcessTreeGroups`, 334-357), the signal handlers (1185-1217), and the child-exit lease-wipe (`childProcess.on('exit')`, 1156-1173) with `clearAllLeaseFiles`/`clearOwnerLeaseFile` (805-819, 558-572). I also confirmed the advisor child's own signal handling (advisor-server.ts:326-330).

### What is solid (no finding)
- **Exit-handler lease-wipe is well-guarded.** `clearOwnerLeaseFile` short-circuits on `!Number.isInteger(ownerLeasePid)` (559) and double-reads `ownerPid === ownerLeasePid` (562-563); `clearLeaseFile` gates on `lease.pid === process.pid` (808). A bridging launcher (never acquired the lease) and a launcher that lost ownership to a successor both fail these guards, so neither wipes the live owner's/successor's lease files. The signal-mirror branch correctly clears leases *before* `process.kill(self, signal)` because `process.on('exit')` doesn't fire under SIGKILL (1159-1162). This adoption edge is handled correctly.
- **Respawn reap escalation ordering is correct:** SIGTERM → `waitForPidExit(7000)` → SIGKILL → `waitForPidExit(1000)` → `owner-reap-timeout` (415-433). Sidecar descendant reaping (`reapProcessTreeGroups`) correctly SIGTERMs then SIGKILLs after a 1s `unref()`'d timer (347-356).

### Findings

**1. Heartbeat-failure self-shutdown lacks SIGKILL escalation — orphans a duplicate advisor child (P2, correctness).** At 546-548 the heartbeat-refresh-failure handler does `childProcess.kill('SIGTERM')` then *synchronously* `process.exit(128)`, with no `waitForChildExit` and no SIGKILL escalation. The advisor child handles SIGTERM via an **async** `shutdownAdvisor(...).finally(() => process.exit(0))` (advisor-server.ts:329-330), so the child is still tearing down (DB/IPC flush) when its supervising launcher vanishes — it gets reparented to init with no escalation backstop. The successor launcher that took ownership will spawn a *fresh* advisor child concurrently. This path exists precisely to "preserve single ownership," yet a slow/hung `shutdownAdvisor` leaves two advisor children alive. Contrast the signal-handler path (1192-1198), which does `kill(signal)` → `waitForChildExit(5000)` → `SIGKILL`. The heartbeat path should use the same escalation.

**2. Respawn/heartbeat reap signals a recorded PID with no executable verification (P2, correctness).** `reapOwnerBeforeRespawn(targetPid)` (406-435) and the `pendingBootstrapReapPid` reap in `main` (1244) issue `process.kill(targetPid, 'SIGTERM'/'SIGKILL')` based only on `processLiveness`. `targetPid` comes from a recorded `childPid`/`ownerPid` in a stale lease file. The owner lease carries `executablePath` (332, 379) but `classifyOwnerLease` (457-474) never compares it, and the respawn reap path verifies neither executable nor start-time. If the recorded PID was recycled by the OS to an unrelated process between recording and reap, the launcher SIGKILLs an innocent process. Lower likelihood (recorded PIDs are short-lived), and may be an accepted systemic tradeoff, but it's the one kill-target gap on the escalation path with no liveness-beyond-PID guard.

Both are P2 — no P0/P1. The lease-wipe adoption logic the seat targeted is correct; the gaps are in escalation *consistency* on the emergency-bail path.
