This release lands the sk-code-review portion of the packets 108 plus 110 auto-review uplift program, applying 5 reusable teachings (H-1, H-2, H-4, M-1, M-2) distilled from a deep-research pass over the upstream `dzianisv/opencode-plugins` auto-review package. Every mechanism cleared a 5-of-5 unanimous deep-ai-council review by `cli-codex gpt-5.5 xhigh fast` before any code shipped, which caught real bugs and reshaped scope. The net effect is a review surface that emits a parseable verdict on its final line, refuses to repeat findings across iterations, and can short-circuit on already-reviewed pull-request state or low-evidence diffs without ever skipping security, authentication, configuration, persistence, dependency, sandbox, or public-facing changes.

## New Features

The five mechanisms below all reached the SKILL.md and one new reference file. None of them change the public review API; they change what reviewers must do and what the dispatcher gets back.

#### Final-line exact-string verdict

Reviews used to end with prose that downstream consumers had to guess at. The dispatcher had no parseable signal for "approved" versus "changes requested" versus "commented", so retry and gating logic was unreliable. Every sk-code-review prompt now requires the rendered output to end with a plain-text final line of the form `Review status: APPROVED`, `Review status: REQUESTED_CHANGES`, or `Review status: COMMENTED`. The verdict line is plain text rather than Markdown bold (the council called the bold variant a misread risk) and the SKILL.md NEVER section forbids any other final line. Consumers can now grep the last line of the review output for a deterministic verdict.

&nbsp;

#### CODE-REVIEW marker at the dispatcher boundary

Reviewers occasionally drifted out of role when the rendered prompt was long enough that the role banner scrolled off-screen. The fix injects a `CODE-REVIEW` marker at the start of the rendered-prompt boundary (in the dispatcher YAML, not in reference resources that carry YAML frontmatter where a stray marker would corrupt parsing). The dispatcher then runs a marker-scan step that fails fast if the marker is missing after rendering. This places the role banner where the reviewer is guaranteed to see it without polluting any document that ships frontmatter.

&nbsp;

#### Anti-repetition rule

Multi-iteration reviews used to surface the same finding under slightly different wording, which inflated finding counts and hid genuine new evidence. The SKILL.md NEVER section now prohibits restating an issue already reported in an earlier iteration of the same review. Reviewers must either omit the finding or extend the existing entry with new evidence. This is enforced by prompt rule rather than by post-processing so the reviewer carries the intent end-to-end.

&nbsp;

#### Pull-request-state content-hash dedup

Re-running a review on a pull request that has not changed used to spend tokens re-deriving the same findings. The new PR-state dedup cache (rooted at `.opencode/.sk-code-review-cache/`) keys cached verdicts by `sha256(repo + pr_number + head_sha + base_sha + changed_files_digest)`. A cache hit short-circuits the review and returns the prior verdict with provenance. The cache is invalidated automatically when any of the keying inputs change, so a force-push or a rebase produces a fresh review. Full keying and invalidation rules live in the new `references/pr_state_dedup.md` reference.

&nbsp;

#### Opt-in minimum-evidence gate with conservative skip taxonomy

Reviewers used to apply the same depth to a one-line whitespace fix and to a 500-line refactor. The new opt-in gate, controlled by `SK_CODE_REVIEW_MIN_CHANGED_LINES`, can short-circuit a review when the diff is too small to warrant deep analysis. The skip taxonomy is deliberately conservative: it NEVER skips changes that touch security, authentication, configuration, persistence, dependencies, sandbox boundaries, or public-facing paths, even when the line count falls below the threshold. The gate must be explicitly enabled by the caller; the default is unchanged from prior releases.

## Council-Gated Workflow

Both packets ran through a 5-seat deep-ai-council deliberation before any source change. The council was dispatched via `cli-codex gpt-5.5 xhigh fast` and returned REQUEST-CHANGES on the first pass for each packet with HIGH confidence. The revisions that landed in this release include: H-1 rendered as plain text rather than Markdown bold; H-2 markers placed at the dispatcher boundary rather than inside frontmatter-bearing reference templates; M-2 skip taxonomy expanded to cover seven sensitive change classes that must never short-circuit. Implementation followed via `cli-opencode + deepseek-v4-pro --pure`.

## Upgrade

No migration required. The dedup cache is created on first hit and is safe to delete at any time. The minimum-evidence gate is opt-in via `SK_CODE_REVIEW_MIN_CHANGED_LINES`; leaving it unset preserves prior behavior. Reviewers picking up the new prompts will start emitting the final-line verdict immediately. Consumers that ignore the final line continue to work unchanged. Source commits: 73e9e361e, 74782acfb, baaeaad52.
