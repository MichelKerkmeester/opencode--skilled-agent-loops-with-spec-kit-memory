# Iteration 022 — Track D (deepseek)

## Focus
Anti-fabrication pattern taxonomy + DTCG spine (ranked value/cost; minimal viable adoption set)

## Findings
1. **[P0] (a) Computed-style sampling, not screenshots** — dmdg (Playwright CSSOM), Style Dictionary, Token Studio. Guarantee: HIGH — reads real computed values from browser engine, not pixel-estimation from bitmaps. Adoption cost: LOW (sunk — already core to dmdg's extract.ts). Foundational pattern; all downstream fidelity depends on it.
   - Recommendation: adopt/maintain. No change needed; extend to capture cascade-layered values (e.g., @layer overrides) for DTCG $type enrichment.
   - Evidence: dmdg extract.ts (Playwright page.evaluate computed styles), Style Dictionary platform transforms, Token Studio Figma plugin
2. **[P0] (b) Typed-schema (DTCG) zero-invention** — dmdg tokens.json uses a flat custom schema (colorTokens, typographyLevels, shadowTokens) with no $type enforcement. Token Studio, Style Dictionary, Amazon Style Dictionary, and Specify all emit DTCG-typed output. Guarantee: HIGHEST — a token CANNOT exist without a typed slot; the schema IS the provenance. Adoption cost: MEDIUM — requires extract.ts to emit DTCG-typed tokens.json, validate.ts to check $type conformance, and DESIGN.md claim types to cite DTCG slots. This is the single highest-leverage structural change.
   - Recommendation: adopt as spine. Migrate tokens.json to DTCG 2025.10 format: every token gets $type, $value, $description. Composite tokens for scales (e.g., a color scale is a composite of color tokens with aliasing). validate.ts adds DTCG schema conformance checks.
   - Evidence: W3C DTCG 2025.10 spec ($type: color/dimension/fontFamily/fontWeight/duration/cubicBezier/number/shadow/gradient/typography/border/transition/strokeStyle), 20+ reference implementations, Token Studio (Figma plugin), Specify, Style Dictionary v4
3. **[P0] (f) Deterministic value-table rendering vs AI prose** — dmdg DESIGN.md uses tables for colors, typography, shadows, and states — these are deterministic. But prose sections (§0 Brand Context, §1 Visual Theme, §7 Content & Voice, named strategies in §3, §6) are LLM-generated and are the largest remaining hallucination surface. Style Dictionary renders programmatic output; Token Studio auto-generates documentation from token metadata. Guarantee: HIGH for table sections; LOW for prose sections. Adoption cost: MEDIUM — fully deterministic prose requires anchoring every narrative claim to a DTCG $description or $value slot (e.g., §1's 'three expansion paragraphs' become auto-composed from token $description fields).
   - Recommendation: adopt: link every DESIGN.md prose claim to a DTCG $description anchor. Short-term: auto-generate §1 expansion paragraphs from token $description + frequency data. Long-term: fully programmatic DESIGN.md rendering from DTCG token graph.
   - Evidence: dmdg design_md_format.md §1 (Visual Theme requires 'specific hex values... font name, weight, size, letter-spacing'), §3 (Named Strategies need bold name + values + rationale), anti_patterns.md AP-02 (Generic Description), Style Dictionary outputReferences
4. **[P0] (h) Functional/positional naming vs evocative** — dmdg anti_patterns.md AP-12 (Vague Color Names) and AP-14 (Cross-Section Terminology Drift) guide functional naming, but enforcement is procedural (LLM prompt guidance), not structural. Token Studio enforces naming conventions via token group structure. Style Dictionary supports functional naming via CTI (Category/Type/Item) transform. Guarantee: MEDIUM (procedural, LLM-dependent). Adoption cost: LOW — DTCG adoption structurally enforces this: $type dictates naming scope; composite token paths create positional naming hierarchies (e.g., color.semantic.action.default).
   - Recommendation: adopt alongside DTCG migration. Replace evocative names ('Stripe Purple', 'Deep Navy') with positional DTCG paths (color.brand.primary, color.text.heading) while preserving evocative names in $description for human prose. Validate functional-vs-evocative naming in validate.ts.
   - Evidence: dmdg anti_patterns.md AP-12, AP-14, AP-27 (Frequency Ignorance); Token Studio token group hierarchy; Style Dictionary CTI transform; DTCG composite token naming convention
5. **[P1] (c) Coverage/occurrence-based election of system-vs-one-off** — dmdg cluster.ts assigns L1-L4 stability classes based on frequency + pagesCoverage + CSS variable presence. Style Dictionary uses token sets/filters. Token Studio uses token scopes. Guarantee: HIGH (deterministic classifier). Adoption cost: LOW — already implemented; DTCG migration would allow $type-aware classification (e.g., a color with frequency 2 but $type=shadow might be L2 because shadows are structural).
   - Recommendation: adopt/enhance. Extend cluster.ts to use DTCG $type as a classification weight (dimensions are more stable than colors; shadows more stable than gradients). Add confidence thresholds calibrated per $type.
   - Evidence: dmdg cluster.ts (L1-L4 classifier), tokens.json frequency + pagesCoverage + confidence fields, anti_patterns.md AP-05 (Over-inference), AP-27 (Frequency Ignorance)
6. **[P1] (d) Unused-style exclusion** — dmdg L4 (content-level) exclusion via cluster.ts. PurgeCSS, unused-css, CSS Stats. Guarantee: MEDIUM — excludes content-level tokens but doesn't guarantee system-level coverage is exhaustive. Adoption cost: LOW — already implemented; DTCG migration adds granularity (a composite token reference with no value-level usage is detectable).
   - Recommendation: adopt/maintain. DTCG composite tokens create audit trail: a token referenced in a composite but never used directly is a dead system token. Add dead-token detection to validate.ts.
   - Evidence: dmdg cluster.ts L4 exclusion, tokens.json stabilityClass field, design_md_format.md §22 (Stability principle)
7. **[P1] (e) Explicit component mapping not heuristic** — dmdg component_taxonomy.md + DESIGN.md §4 'Use:' lines with real button labels from observed DOM text. Chromatic visual tests, Storybook test-runner. Guarantee: MEDIUM — mapping is DOM-observed (element type + text content) but cross-component token sharing (a color used in both buttons and links) is not structurally captured. Adoption cost: LOW — already implemented; DTCG composite tokens could track token→component→occurrence edges.
   - Recommendation: adopt/enhance. DTCG composite tokens add component-context metadata: a 'button.primary' composite references color/typography/shadow/spacing tokens with source-page + element-selector provenance.
   - Evidence: dmdg component_taxonomy.md, design_md_format.md §7 (Use: lines requirement), tokens.json usedAs fields (textColor/bgColor/borderColor/shadowColor/gradientColor/iconColor)
8. **[P2] (g) Measured contrast (CIEDE2000)** — dmdg colorRelationships.contrastPairs uses WCAG 2.1 ratio (relative luminance). CIEDE2000 (perceptually uniform delta-E) is better for non-text contrast judgments (e.g., icon-to-surface, gradient-stop adjacency). Contrast, Polypane, axe-core. Guarantee: HIGH (CIEDE2000), MEDIUM (WCAG ratio for non-text). Adoption cost: LOW-MEDIUM — add CIEDE2000 calculation to css-analyzer.ts alongside WCAG ratio; surface both in tokens.json contrastPairs.
   - Recommendation: adopt. Add CIEDE2000 to contrastPairs as a complementary metric: WCAG ratio for text accessibility; CIEDE2000 for visual-distinction validation. Enables auto-flagging of near-identical colors that should be merged.
   - Evidence: dmdg tokens.json colorRelationships.contrastPairs (WCAG ratio + meetsAA/meetsAAA), CIEDE2000 CIE standard, Polypane contrast tools

## Questions Answered
- DTCG as spine: tokens.json→DTCG migration enables (b) structurally and anchors (f). DTCG $type authority proves provenance without procedural trust. Composite tokens with aliasing create auditable token graphs.
- DESIGN.md claim→DTCG slot mapping: §2/§2.5→color; §3→fontFamily+fontWeight+dimension; §4→color+dimension+shadow+transition; §5→dimension; §6→shadow; §6.5→duration+cubicBezier; §11→color+shadow+transition; §12→dimension+number. Prose sections (§0/§1/§7/§8) cite token $description as claim-anchor.
- Top 3-4 patterns to reduce prose-fabrication: (b)+(f)+(h)+(c). They compose tightly — DTCG types tokens (b) → coverage classifies them (c) → functional naming labels them (h) → deterministic rendering surfaces them as tables + $description citations (f). The composition eliminates the LLM as a value-invention surface.
- Minimal viable adoption: DTCG-typed tokens.json + $description field on every token + prose sections cite $description. Full ideal adds CIEDE2000, composite token graphs, and fully programmatic DESIGN.md generation.
