## Mechanism
Chunking: MemClaw has two “split into smaller memory units” paths, not a general chunk-fingerprint incremental index. Large ingest/direct writes use a structure-aware chunker: `Section` carries `text`, `breadcrumb`, `token_count`, and block types (`core-api/src/core_api/services/ingest_chunking.py:70`), markdown parsing preserves headings/lists/code/tables (`core-api/src/core_api/services/ingest_chunking.py:127`), and `chunk_blocks()` greedily closes sections on H1/H2 boundaries or soft-token overflow while preserving lists/tables/code as atomic blocks (`core-api/src/core_api/services/ingest_chunking.py:335`). Each section then goes through LLM fact extraction via `_chunk_content()`, with breadcrumb context injected into the prompt (`core-api/src/core_api/services/ingest_service.py:255`). Enrichment also has `atomic_facts`: the prompt asks for separate child facts only when distinct claims would be searched by different vocabulary (`common/enrichment/_prompts.py:127`), and the schema says each fact becomes a child memory with its own embedding (`common/enrichment/schema.py:50`).

Embedding trigger/versioning: write mode controls whether embedding/enrichment happen inline or are deferred. Strong writes embed and enrich inline; fast writes always defer enrichment and may defer embedding depending on deployment settings (`core-api/src/core_api/pipeline/steps/write/parallel_embed_enrich.py:7`). Deferred embedding publishes `EMBED_REQUESTED` with `memory_id`, `tenant_id`, `content`, and optional `content_hash` (`common/events/memory_embed_publisher.py:21`; `common/events/memory_embed_request.py:21`). `_schedule_embed_or_reembed()` chooses inline `_reembed_memory()` or publishes the event (`core-api/src/core_api/services/memory_service.py:1544`). MemClaw has provider/model/dimension configuration, but not durable per-row embedding model versioning in the embed request: `MemoryEmbedRequest` contains only identity, tenant, content, and content_hash (`common/events/memory_embed_request.py:38`). Model/dim are process/env/schema concerns: `VECTOR_DIM = 1024` (`common/constants.py:3`), OpenAI model comes from `OPENAI_EMBEDDING_MODEL` (`common/embedding/constants.py:21`), and the registry validates base-url/dim/truncation compatibility (`common/embedding/_registry.py:267`). Docs say provider switches do not require migration if schema dim stays fixed, though re-embed may be desirable for vector-space consistency (`docs/local-embedder.md:307`).

Idempotent re-embed: deferred embedding is explicitly at-least-once safe: a redelivered event re-PATCHes the same deterministic embedding, and `content_hash` avoids duplicate provider calls (`core-worker/src/core_worker/consumer.py:25`). The worker first looks up an existing embedding by content_hash and reuses it on hit (`core-worker/src/core_worker/consumer.py:144`), while storage lookup failures degrade to “miss” rather than nacking (`core-worker/src/core_worker/clients/storage_client.py:134`). The PATCH clears `metadata.embedding_pending=false` in the same storage update (`core-worker/src/core_worker/clients/storage_client.py:187`). Inline `_reembed_memory()` retries with backoff and refuses to overwrite an embedding already present, treating that as a race guard (`core-api/src/core_api/services/memory_service.py:1723`; `core-api/src/core_api/services/memory_service.py:1750`).

Enrichment schema + flow: `EnrichmentResult` includes `memory_type`, `weight`, `title`, `summary`, `tags`, `status`, temporal bounds, PII fields, `retrieval_hint`, `atomic_facts`, and `llm_ms` (`common/enrichment/schema.py:64`). The prompt is the schema source and defines retrieval hints as query-vocabulary augmentation metadata, plus atomic facts as optional child-memory candidates (`common/enrichment/_prompts.py:108`; `common/enrichment/_prompts.py:127`). `enrich_memory()` runs a provider/fallback chain and “never raises”; it always returns an `EnrichmentResult` or heuristic fallback (`common/enrichment/service.py:272`). The worker routes ORM fields and metadata fields through explicit allowlists and import-time exhaustiveness checks (`core-worker/src/core_worker/consumer.py:217`; `core-worker/src/core_worker/consumer.py:284`). Redelivered enrichment skips agent-provided fields (`core-worker/src/core_worker/consumer.py:368`), uses `llm_ms > 0` to prevent heuristic fallback from clearing real LLM-derived fields (`core-worker/src/core_worker/consumer.py:393`; `core-worker/src/core_worker/consumer.py:445`), clears `enrichment_pending`, PATCHes storage, then publishes `ENRICHED` best-effort (`core-worker/src/core_worker/consumer.py:431`; `core-worker/src/core_worker/consumer.py:473`; `core-worker/src/core_worker/consumer.py:555`). Important correction: hint-based re-embedding is disabled because it made write vectors use hinted text while queries used raw text, hurting recall; both hot path and background now embed raw content (`core-api/src/core_api/pipeline/steps/write/parallel_embed_enrich.py:29`; `core-api/src/core_api/services/memory_service.py:2094`).

## Teachings for Spec Kit Memory (027/003)
1. **Claim** · Keep a durable per-chunk fingerprint/cache gate before embedding, but scope it to local chunk identity rather than tenant/content-only hashes. **Evidence** · MemClaw hashes tenant/fleet/content (`core-api/src/core_api/services/memory_service.py:91`), sends content_hash in embed requests (`common/events/memory_embed_request.py:31`), and the worker reuses cached embeddings on content_hash hit (`core-worker/src/core_worker/consumer.py:144`). **Maps-to** · 027/003. **Verdict** · ADAPT. **Risk** · Over-broad hashes can reuse embeddings across semantically different chunk contexts; under-broad hashes miss reuse. **Confidence** · High. **Why it transfers (or not)** · Spec Kit Memory already has chunk fingerprints; the transferable lesson is the idempotent “fingerprint hit skips embed” gate, not MemClaw’s tenant/fleet hash shape.

2. **Claim** · Persist explicit “pending/stale/cleared” embedding/enrichment state transitions in metadata, not just absence of a vector. **Evidence** · MemClaw clears `embedding_pending` when the worker PATCHes the vector (`core-worker/src/core_worker/clients/storage_client.py:187`) and clears `enrichment_pending` after enrichment succeeds (`core-worker/src/core_worker/consumer.py:431`). **Maps-to** · 027/003. **Verdict** · ADAPT. **Risk** · Extra state can become inconsistent unless updates are atomic with vector/enrichment writes. **Confidence** · High. **Why it transfers (or not)** · A local SQLite/vector-store pipeline still benefits from resumable “dirty/pending/complete” state, but it should be one local transaction or a recoverable two-store commit marker, not Pub/Sub status.

3. **Claim** · Add durable embedding provenance/version fields even though MemClaw mostly lacks them per row. **Evidence** · MemClaw’s embed request has no provider/model/version fields (`common/events/memory_embed_request.py:38`), while provider/model/dim are resolved from env/schema (`common/embedding/constants.py:21`; `common/constants.py:3`) and docs note provider switches may justify re-embedding for vector-space consistency (`docs/local-embedder.md:307`). **Maps-to** · 027/003 or new sub-packet. **Verdict** · ADOPT. **Risk** · Schema churn and unnecessary re-embed if version semantics are too sensitive. **Confidence** · Medium. **Why it transfers (or not)** · Spec Kit Memory’s embedder is fixed to ollama nomic 768d, so this should not support model swapping; it should record `embedder_id`, `dim=768`, prompt/text-shape version, and enrichment/schema version so stale rows are detectable after pipeline changes.

4. **Claim** · Do not embed retrieval-hint-prefixed content unless query embeddings use the same surface; store hints as retrieval metadata instead. **Evidence** · MemClaw explicitly removed hint-based re-embed because hinted write text vs raw query text reduced identical content-query cosine similarity and capped recall (`core-api/src/core_api/pipeline/steps/write/parallel_embed_enrich.py:158`); inline enrichment now persists retrieval_hint only for debugging/auditability, not embedding shape (`core-api/src/core_api/services/memory_service.py:2063`). **Maps-to** · 027/003. **Verdict** · ADAPT. **Risk** · Pure raw-content embeddings may miss concept-level queries that a hint could help. **Confidence** · High. **Why it transfers (or not)** · With fixed nomic-768d, keep chunk text as the embedding source unless Spec Kit also applies the same augmentation to queries or stores a separate hint vector/channel.

5. **Claim** · Use explicit enrichment field routing and “real enrichment vs fallback” guards before mutating indexed metadata. **Evidence** · MemClaw separates ORM enrichment fields from metadata fields (`core-worker/src/core_worker/consumer.py:217`), fails import on unrouted schema fields (`core-worker/src/core_worker/consumer.py:284`), skips agent-provided fields on redelivery (`core-worker/src/core_worker/consumer.py:368`), and uses `llm_ms > 0` to stop heuristic fallback from clearing real LLM metadata (`core-worker/src/core_worker/consumer.py:445`). **Maps-to** · new sub-packet. **Verdict** · ADOPT. **Risk** · If fallback markers are unreliable, stale metadata may be preserved or good fallback output may be ignored. **Confidence** · High. **Why it transfers (or not)** · Spec Kit Memory can reuse the principle locally: only deterministic/validated enrichment should update searchable metadata; fallback/heuristic enrichment should not erase stronger prior values.

## Negative knowledge
Do not copy Pub/Sub worker topology, at-least-once delivery machinery, DLQs, per-tenant concurrency slots, tenant config payloads, or encrypted credential-on-event patterns. Those exist because MemClaw is fleet/multi-tenant and distributed (`core-worker/src/core_worker/consumer.py:31`; `common/events/memory_enrich_request.py:10`; `core-worker/src/core_worker/consumer.py:544`).

Do not copy pgvector/VECTOR_DIM migration logic or model-switch guidance as a feature direction. Spec Kit Memory’s embedder is fixed to ollama nomic 768d. MemClaw’s `VECTOR_DIM=1024`, TEI/bge-m3 defaults, truncation knobs, and provider-switch docs are negative knowledge except for the general need to record dimension/provenance (`common/constants.py:3`; `docs/local-embedder.md:271`; `common/embedding/_registry.py:312`).

Do not copy hint-prefixed re-embedding. MemClaw tried it and removed it because it created write/query embedding-surface asymmetry (`core-api/src/core_api/pipeline/steps/write/parallel_embed_enrich.py:29`).

Do not copy tenant-scoped content hashes directly. Spec Kit Memory needs local file/chunk identity, line spans, chunk kind, content fingerprint, dependency DAG inputs, and embedder/schema version, not `tenant_id:fleet_id:content` (`core-api/src/core_api/services/memory_service.py:91`).

Do not treat MemClaw’s async worker atomic-fact gap as a pattern to emulate. The worker explicitly logs that async child-memory creation is not implemented (`core-worker/src/core_worker/consumer.py:520`), while the synchronous path creates child memories with embeddings (`core-api/src/core_api/services/memory_service.py:2105`).
