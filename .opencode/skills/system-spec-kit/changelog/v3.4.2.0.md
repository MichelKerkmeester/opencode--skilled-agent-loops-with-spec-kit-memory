# v3.4.2.0, The Comment Rule Gets Teeth, and the Review Backlog Closes

v3.4.1.0 shipped the substrate. v3.4.2.0 is a hardening-and-honesty release. The sk-code rule that bans ephemeral-artifact pointers in code comments finally has a machine that enforces it, and that machine immediately proved the rule had been ignored at roughly ten times the scale anyone thought. The same release closes the actionable findings from the 20-iteration embedding-stack deep review, lands the coordinated single-writer/durability cluster the review flagged as must-fix-together, and repairs the memory database that had been failing every write since session start.

The throughline: build the detector before you trust the cleanup, re-validate every finding against HEAD before you touch it, and verify a fix through the path that actually failed, not the layer that was always fine.

## The Comment Rule, Now Enforced

The sk-code rule is simple: a code comment may keep its durable WHY but must never embed an ephemeral-artifact pointer, a spec folder or number, a task/checklist/requirement id, a review-finding id, an ADR id, or a ticket. Those get renamed or archived on their own schedule and rot into dangling pointers.

An earlier pass cleaned 27 files using hand-written grep patterns. This release built the detector that should have come first.

#### A precision-first detector

`scripts/validation/ephemeral-pointer-audit.mjs` is a standalone Node ESM script with no dependencies. It inspects comment regions only, never string or code content, so it never flags a runtime path constant or test fixture value. Each forbidden pattern is narrowed to the shape of a real tracking id and paired with an explicit allow-list: HTTP status codes, embedding dimensions, token-budget tiers, rolling-window sizes, schema-version tags, JSDoc example shapes, externally-versioned standards, and the runtime `.opencode/specs/` paths the code actually reads. It exits nonzero on any hit, so CI, pre-commit, or an sk-code verification step can gate on it.

#### The cleanup it forced

Run against the tree, the guard reported 274 true violations across 116 files, against the 27 the first pass had touched. The classes the hand patterns missed: `P0-`/`P1-`/`P2-` review-finding ids, two-digit and hyphenated task ids, no-hyphen checklist ids, `#NN` issue refs, `NNN-NNN` spec pairs, and `SPRINT N` labels.

A parallel sweep brought the whole tree to zero, 261 fixes across 119 files via fan-out agents that each self-verified to guard-clean, plus an orchestrator pass that closed eight stragglers in directories the batching had missed. Every edit kept the explanatory text and dropped only the id. Both `@spec-kit` workspaces build clean and the diff is comment and string text only, so the entire sweep is behaviorally inert.

## Deep-Review Backlog: Re-Validated, Then Closed

The 20-iteration embedding-stack deep review produced a backlog. This release works it, with one rule: re-validate every finding against the current HEAD before editing, because line numbers had shifted under the comment sweep and several findings turned out to be already-fixed or mis-located.

#### Documentation reconciliation

Three documentation findings landed first. The 005 file-change matrix was attributing idle-eviction to the wrong file; it lives in `model-server-supervision.cjs`, not `hf-model-server.cjs`. ENV_REFERENCE was missing four embedding knobs (`EMBEDDER_REINDEX_BATCH_SIZE`, `EMBEDDING_DIM`, `HF_EMBEDDINGS_PREFIX_QUERY`/`DOC`). The four daemon-reliability packet ledgers (009/010/012/013) had verification tables left open while their summaries claimed "implemented"; all four strict-validate cleanly now, so the open items are marked done with real evidence.

#### Correctness edges, with two honest no-changes

Three isolated correctness findings landed with regression tests. The execution-router adapter cache now includes the resolved dimension in its invalidation check, so a `dimensionsOverride` can never silently reuse a wrong-dimension adapter. The bootstrap auto-selector now moves an explicit `EMBEDDINGS_PROVIDER` to the front of the local-first cascade. The retention sweep re-validates `delete_after` inside the delete transaction, closing a multi-process data-loss window.

Two findings were correctly dispositioned rather than patched. One described a function that does not exist; the real lock fallback is an intentional graceful degradation to an in-process queue, so failing closed would have been a regression. The other was already fixed at HEAD. Re-validation is what caught both.

## The Single-Writer / Durability Cluster

The deep review flagged a cluster of launcher and durability defects that share resources, a respawn lock, a root-liveness authority, lease files, and a shutdown marker, and warned they must land together or piecemeal fixes would re-collide. This release lands them as one coordinated change set behind deterministic tests.

- **Respawn lock** now bounds staleness to listener liveness, so a live demand listener cannot have its lock declared stale by wall-clock alone; and the lazy listener re-arms on a failed spawn instead of stranding.
- **Idle eviction** now reaps the root process, not just its descendants, through the shared root-liveness authority.
- **Clean-shutdown marker** is deleted only after a confirmed-successful close, so a failed detach or close can no longer leave the next open wrongly trusting the DB.
- **Same-dimension reindex** writes to a staging shard and atomic-swaps on success, so a mid-loop failure can no longer half-overwrite the live shard; the worker also re-reads its cancel flag mid-run.
- **Direct startup** applies the same socket-ownership and live-resident perimeter guard the supervised path uses, and tcp targets enforce loopback or auth.
- **Standalone-save lease** honors a live recorded child pid in a stale-looking launcher lease, so a second writer is not admitted.

The owner-lease election race the review listed was re-validated as already O_EXCL-remediated at HEAD, so it needed verification, not a change. Verification: both workspaces plus the scripts package build clean, `node --check` passes on the touched launchers, and four new deterministic suites (26 tests, injecting time, liveness, spawn, and signal rather than sleeping) plus three regression suites (29 tests) and the leases suite (9 tests) all pass.

## Memory Database Repair

The spec-memory MCP had been failing every write with `SQLITE_CONSTRAINT_PRIMARYKEY` since the session began, blocking `memory_save`, `memory_index_scan`, and `memory_match_triggers`.

The investigation root-caused it to the FTS5 shadow desyncing after an unclean shutdown, and an honest correction belongs in the record: the first repair attempt verified at the SQLite-CLI layer and declared success prematurely, while the MCP write path, the one that actually failed, was never exercised. On re-examination the on-disk database was healthy at rest (`quick_check` ok, `memory_index` and `memory_fts` row counts matching), and the failing daemon had already died. The real fault was a stale, unclean daemon lifecycle, not standing corruption.

A clean daemon restart cleared it. `memory_health` reports HEALTHY with FTS integrity ok and zero constraint errors, and the repair was verified the right way this time, through real MCP writes: `memory_index_scan` succeeded across the session's new packets with zero errors. The diagnostic-only `/doctor memory` references in the packet docs were corrected to point at the guarded `memory_health` autoRepair path, and the three packet docs that had drifted into mutual contradiction were reconciled to the verified outcome.

## Upgrade

- No breaking changes. This release is hardening, cleanup, and a database repair.
- **Comment hygiene is now detectable.** `node .opencode/skills/system-spec-kit/scripts/validation/ephemeral-pointer-audit.mjs <paths>` reports ephemeral-artifact pointers in code comments and exits nonzero on any hit. Wiring it into pre-commit or CI is a follow-on opt-in.
- **Memory subsystem is healthy again.** No action required; the next daemon boot reopens a clean DB.
- **Deferred follow-ons:** the graph-metadata churn code fix (the daemon rewriting timestamps tree-wide) and the worktree-per-session automation that would prevent the multi-session contention behind much of this session's friction are designed and documented, to land on a quiet tree.
