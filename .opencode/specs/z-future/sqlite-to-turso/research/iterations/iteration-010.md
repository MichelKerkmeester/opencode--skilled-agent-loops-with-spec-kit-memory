# Iteration 10: W4 deep-dive: compat-mode pilot scoping for spec-kit

## Focus
Enumerate the actual better-sqlite3 API surface the three skills (system-spec-kit, system-code-graph, system-skill-advisor) consume, validated against the vendored turso-main v0.7.0-pre.6 compat-mode gaps: (1) which not-implemented compat methods are actually called and where, (2) confirm the bare-VACUUM sites and whether VACUUM INTO substitutes, (3) map all `.changes` call sites and rank which flows break under an unreliable change counter, (4) verify from `core/` whether the vendored engine supports schema-qualified PRAGMA against ATTACHed DBs (the `ensure_vector_shard_schema` pattern).

## Findings
- NEW — verdict CHANGED (better than baseline assumed): ZERO call sites of the compat-mode not-implemented surface across all three skills. A grep for `backup( / serialize( / .function( / .aggregate( / .table( / loadExtension / .iterate( / .raw( / .pluck( / .columns( / unsafeMode / defaultSafeIntegers` over `.opencode/skills/system-spec-kit/mcp_server/lib`, `system-code-graph/mcp_server/lib`, `system-skill-advisor/mcp_server/lib` returned no real hits. A looser sweep found only two false positives: `serialize()` is a method on the `WorkingSetTracker` class (JSON state, not a DB handle) [SOURCE: .opencode/skills/system-code-graph/mcp_server/lib/working-set-tracker.ts:85], and `merge_checkpoint_catalog_from_backup` is a local function name [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:934,1029]. Per-site workaround needed: NONE — the compat pilot's method-gap surface for these three skills is empty.
- NEW: the reason `backup()` is unused: spec-kit checkpoints already snapshot via schema-qualified `VACUUM main INTO ...` and `VACUUM active_vec INTO ...` with a busy-retry loop [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/storage/checkpoints.ts:2281,2284,2288-2296]. This pattern is translation-supported in the vendored engine (see VACUUM finding below), pending VDBE-runtime confirmation for non-main schema.
- KNOWN — verdict UNCHANGED: the two bare in-place VACUUM sites are exactly `canonical.exec('VACUUM')` after `DETACH shard/legacy` and `shard.exec('VACUUM')` on a fresh connection, both in the one-time legacy→shard migration, each on a short-lived exclusive connection closed in `finally` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/db-shard-migration.ts:274,282,272-285].
- NEW: VACUUM INTO CAN substitute at both sites. Vendored `translate_vacuum` emits `Insn::VacuumInto { schema_name, dest_path }` unconditionally (no experimental flag) [SOURCE: external/turso-main/core/translate/vacuum.rs:28-37], while bare VACUUM requires `--experimental-vacuum` [vacuum.rs:39-44], is incompatible with multiprocess WAL [vacuum.rs:45-49], and schema-qualified in-place VACUUM is hard-rejected ("only supported for the main database") [vacuum.rs:51-57]. Since both migration sites run on exclusive just-opened connections immediately before close, `VACUUM INTO <tmp>` + atomic rename is a drop-in workaround; the file already has a `moveWalCompanions` helper for the rename dance [SOURCE: db-shard-migration.ts:287-292].
- NEW: third vacuum-family site (non-blocking): `memory-retention-sweep.ts` queries `pragma('auto_vacuum')` and only runs `pragma('incremental_vacuum')` when mode===2, with both calls try/catch-wrapped warn-and-continue [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/governance/memory-retention-sweep.ts:349-363]. Vendored engine: `auto_vacuum` setter is gated behind `--experimental-autovacuum` [SOURCE: core/translate/pragma.rs:509-513], the query returns pager mode (default None=0) [pragma.rs:1373-1378], and `incremental_vacuum` early-halts when auto-vacuum is off [pragma.rs:576-584]. Net effect: the sweep's vacuum branch self-disables gracefully under Turso. No other bare VACUUM/incremental_vacuum sites exist in the three skills.
- NEW (extends the iter-1..9 "changes reliability partial" gap): ~60 `.changes` call sites across the three skills, splitting into reporting counters (tolerant) and control-flow dependencies (break if the counter is unreliable). Critical control-flow sites: (a) HIGHEST severity — skill-advisor daemon lease acquisition/renewal treats `result.changes === 0` as lease-lost [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/daemon/lease.ts:350,359]; an inaccurate counter undermines single-writer mutual exclusion (the surviving lease model from iter-9) — could admit two writers or wedge takeover; (b) job-claim at-most-once semantics `claimed: result.changes > 0` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/providers/retry-manager.ts:326, plus 507,530,872-876]; (c) optimistic-concurrency guard `if (changes !== 1) continue` [SOURCE: lib/storage/incremental-index.ts:643]; (d) upsert-fallback branching on `changes === 0` [SOURCE: lib/storage/checkpoints.ts:3137; lib/storage/reconsolidation.ts:528,577]; (e) dedup-send decision `shouldSend = insertResult.changes > 0` after INSERT-OR-IGNORE [SOURCE: lib/session/session-manager.ts:609]; (f) boolean mutation-success returns feeding callers [SOURCE: lib/storage/access-tracker.ts:164,177; lib/cognitive/attention-decay.ts:151; lib/storage/causal-edges.ts:595,818,1107; lib/search/vector-index-mutations.ts:584,697,819,912,942; lib/ops/job-queue.ts:381; lib/search/entity-linker.ts:931; lib/learning/corrections.ts:359]. Reporting-only sites (deletedCount/insertedTurns/migration logs, e.g. session-manager.ts:777-841, code-graph-db.ts:791,1194, embedding-cache, sweep, analytics) tolerate inaccuracy. Per-site workaround for critical sites: rewrite to `UPDATE/DELETE ... RETURNING`-count or SELECT-verify; or gate the pilot on the upstream changes-counter fix.
- NEW — verdict CHANGED (risk retired): the vendored engine fully supports schema-qualified PRAGMA against ATTACHed DBs at the translation layer. `translate_pragma` accepts `ast::QualifiedName`, resolves `database_id = resolver.resolve_database_id(name)` and tracks `schema_was_explicit = name.db_name.is_some()` [SOURCE: core/translate/pragma.rs:212-234]; the JournalMode setter emits `Insn::JournalMode { db: database_id, .. }` [pragma.rs:372-389]; WalCheckpoint routes with database_id + schema_was_explicit [pragma.rs:427-436]. This covers the actual call sites: `PRAGMA shard.journal_mode = WAL` [SOURCE: db-shard-migration.ts:245], `pragma('${ACTIVE_VECTOR_SCHEMA}.journal_mode = WAL')` inside `ensure_vector_shard_schema` [SOURCE: lib/search/vector-index-store.ts:498-499], and `<schema>.wal_checkpoint(TRUNCATE)` — the latter already try/catch best-effort at every site [vector-index-store.ts:1681,1697,1731-1732,1743-1744]. Caveat: Turso is WAL-native, so `journal_mode = WAL` is accepted but runtime behavior is normalize-to-WAL; semantics are compatible with the skills' intent.
