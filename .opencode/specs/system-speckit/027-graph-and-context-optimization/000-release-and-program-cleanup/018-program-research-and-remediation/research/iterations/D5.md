### Angle 4: Formal correctness of SEC-001 conflict guard against transitive (3+ node) contradiction cycles

- **Framing:** The SEC-001 guard prevents backfill from overwriting valid edges on the same (source, target) pair — but it has no logic to detect or prevent transitive contradiction cycles spanning 3+ nodes.

- **Repo evidence:** `mcp_server/lib/causal/relation-backfill.ts:692-709` (`hasConflictingValidEdge` queries only `WHERE source_id = ? AND target_id = ?`), `mcp_server/lib/graph/contradiction-detection.ts:47-56` (`relationsConflict` is directional, per-pair), `mcp_server/tests/relation-backfill-conflict.vitest.ts:238-270` (3-node test creates three independent pairwise conflicts, not a transitive inference chain). Deep review iteration-004 (`iteration-004.md:77`) explicitly ruled "NOT A NEW DEFECT — contradiction-detection.ts has no transitive-cycle logic and never claimed to."

- **Findings:** The guard is formally correct for what it claims: reciprocal (same-pair) conflict between `caused`/`supports`/`enabled` and `contradicts`. It inserts non-conflicting edges first (lineage, spec-chain), then checks each conflict-prone candidate via `hasConflictingValidEdge` which queries only the direct (source, target) pair. For a 3-node cycle like A→B (caused), B→C (caused), A→C (contradicts), there is **no guard at all** — `detectContradictions` would see A→C as a fresh pair with no existing conflicting edge and insert it, without checking whether it contradicts an inferred transitive relationship. This is acknowledged out-of-scope; iteration-018 (`iteration-018.md:86-89`) carries forward "whole-graph reciprocal/transitive contradiction reconciliation" as future work.

- **Next experiment:** Create a fixture with A→B (caused), B→C (caused), and run auto-backfill to see if it emits A→C (contradicts). Verify that the resulting contradictory 3-cycle goes undetected by both the guard and the detector, then model whether a bounded BFS path-traversal on `caused`/`enabled` edges would catch it. Measure: false-positive rate of such transitive inferences on a real memory_index.

- **Verdict:** The guard is correct within its stated scope (2-node reciprocal pairs). Transitive-cycle detection is not implemented anywhere in the codebase and is a known gap carried forward to future work. Recommend adding a bounded path-traversal (`maxDepth=3`) that checks whether a new `contradicts` edge A→C conflicts with a transitive `caused`/`enabled` path A→...→C — but measure the false-positive rate first, since not all transitive chains are semantic contradictions.

### Angle 10: checkpoint-v2 durability — crash windows in restore journal + .needs-rebuild sentinel

- **Framing:** After the Opus cross-review P1 fix, does any crash-timing window in the two-phase journal (`swap-pending` → `swap-done`) + `.needs-rebuild` sentinel leave the database in an unrecoverable state?

- **Repo evidence:** `mcp_server/lib/storage/checkpoints.ts:113-118` (journal/sentinel constants), `:2660-2699` (swapFn closure with atomic journal transitions), `:2714-2721` (P1 fix: demote journal to swap-pending before in-process rollback), `:2739-2794` (post-open failure catch with journal demotion). Commit `5aec7864b1`/`29160c0e50` (2026-06-02) shipped the fix. Tests: `checkpoints-v2-restore.vitest.ts:260-296` ("demotes the restore journal to swap-pending before rolling back a failed swap reopen"), `:298-319` (swap-pending boot recovery), `:321-338` (swap-done keeps restored DB + writes sentinel), `:340-389` (re-merge post-snapshot catalog rows from .bak).

- **Findings:** Pre-fix, there was a genuine unrecoverable crash window: if `swapFn` completed (journal=`swap-done`, live=snapshot, .bak=original), then `reopen()` threw, the rollback's `rmSync→renameSync` gap between deleting the snapshot and restoring the .bak could crash — leaving journal=`swap-done` (never demoted) and boot recovery would `rmSync` the .bak, destroying both copies. The fix writes `swap-pending` journal BEFORE the rollback reopen at both recovery sites (swap-failure at line 2714 and post-open failure at line 2775), so any crash-in-rollback finds `swap-pending` and correctly restores from .bak. All other windows are benign: crash after `swap-pending` write but before .bak rename → boot clears journal (no DB touched); crash in `swap-pending` with .bak present → boot restores from .bak; crash after clearRestoreJournal → no journal, DB is the restored snapshot, .bak is stale but swept on next create.

- **Next experiment:** Inject a process.exit() between `rmSync(liveMainPath)` and `renameSync(backupMainPath→liveMainPath)` inside a rollback reopen, then verify boot recovery finds `swap-pending` journal and restores from .bak. Also: test the orphan-snapshot-dir-on-create-crash path (Window H) — inject a crash between `renameSync(tmpDir, finalDir)` and catalog INSERT, verify `sweepOrphanCheckpointSnapshotDirs()` cleans it on next create.

- **Verdict:** After the P1 fix, **no unrecoverable crash windows remain**. The two-phase journal protocol provides deterministic recovery for all crash-timing scenarios. The `.needs-rebuild` sentinel adds a secondary safety net for post-restore derived rebuild failures. Residual concerns (orphan snapshot dirs, pruned dir timing) are benign — disk space waste only, recovered by sweep on next create.

### Angle 19: WAL-checkpoint-on-close data-loss window — formal bound and autocheckpoint

- **Framing:** What is the maximum uncheckpointed WAL data surviving a crash, and does `wal_autocheckpoint=256` close the unbounded data-loss window?

- **Repo evidence:** `mcp_server/lib/search/vector-index-store.ts:1554-1562` (main DB PRAGMAs: `journal_mode=WAL`, `synchronous=NORMAL`, `wal_autocheckpoint=256`, `cache_size=-64000`), `:498-503` (active_vec shard: same autocheckpoint=256), `:1617-1665` (`close_db` — dual-schema TRUNCATE checkpoint → detach → close → marker removal), `:1688-1692` (`checkpointAllWal` 5-minute periodic TRUNCATE). `context-server.ts:1558-1634` (`fatalShutdown` — ordered drain: fileWatcher → ingestWorker fence → closeDb, bounded by `SHUTDOWN_DEADLINE_MS=5000`). `context-server.ts:2175` (5-min periodic checkpoint registered). Launcher: `bin/mk-spec-memory-launcher.cjs:633-634` (SIGTERM→SIGKILL escalation after `RESPAWN_REAP_GRACE_MS=7000`, giving daemon 2000ms margin over its 5000ms deadline). Packet 010 benchmark: `.opencode/specs/.../010-at-rest-wal-durability/scratch/wal-durability-benchmark-results.json` — pre-autocheckpoint live WAL was ~9.2 MiB (2349 pages); after autocheckpoint=256, forced TRUNCATE returned `checkpointed: 0` (already flushed).

- **Findings:** The formal worst-case bound on uncheckpointed data at crash is **≤256 WAL pages (~1 MiB)** from autocheckpoint, plus writes accumulated within the 5-minute periodic TRUNCATE window. The benchmark shows post-autocheckpoint, forced TRUNCATE checkpoints zero pages (autocheckpoint already flushed everything), confirming the passive bound works. The `synchronous=NORMAL` setting means in a power-loss scenario (not SIGKILL), committed data since the last OS fsync may be lost — the `FULL`+`fullfsync` upgrade was benchmarked at +19.576ms p99 (7x p50 increase) and deferred as a benchmark-gated follow-up. The ingest worker is fenced BEFORE closeDb (`job-queue.ts:237-242` — `acquireWorkerDb` throws when `shuttingDown`), preventing DB reopen that would re-dirty the WAL. On crash, the `.unclean-shutdown` marker persists and boot recovery runs FTS5 integrity-check with auto-heal.

- **Next experiment:** Run the benchmark with `synchronous=FULL`+`fullfsync` on a production-like workload (mixed read/write, concurrent ingest) to quantify the real latency cost. Also: inject a SIGKILL during peak ingest and measure bytes actually replayed at next boot vs the 256-page bound — validate that autocheckpoint=256 holds under sustained write pressure.

- **Verdict:** **Yes, bounded autocheckpoint closes the unbounded window.** Pre-010, live WAL was ~9.2 MiB with no bound. Post-010, the ceiling is ~1 MiB (autocheckpoint passive) plus the 5-minute write delta. The remaining window is the `synchronous=NORMAL` power-loss gap — a deliberate tradeoff whose cost/benefit should be re-measured.

### Angle 20: Socket-path canonicalization and TOCTOU — residual symlink/race attack surface

- **Framing:** After three-layered hardening (directory fence, fresh-bind symlink reject, post-bind chmod guard), does any residual symlink or TOCTOU race surface remain on the unix-domain socket bind path?

- **Repo evidence:** `shared/ipc/socket-server.ts:264-282` (directory fence: `mkdirSync 0o700` + `statSync` uid/mode rejection), `:283-295` (fresh-bind symlink reject: `lstatSync(socketPath)` + `isSymbolicLink()` → refuse), `:389-396` (post-bind chmod guard: `lstatSync` + symlink/socket check before `chmodSync 0o600`), `:142-178` (EADDRINUSE reclaim fence: `lstatSync` uid+socket+root check). `:85-98` (`canonicalizePath` — walks to nearest existing ancestor, `realpathSync` on ancestor, re-appends raw tail; made fail-closed in 017 remediation). - **Findings:** Cross-uid attack surface is **CLOSED** — the directory ownership fence (lines 264-282) rejects any socketDir not owned by the daemon's uid or with group/world-writable bits, which blocks an attacker from planting a symlink in the socket path. The fresh-bind symlink reject (lines 283-295) adds a defense-in-depth layer checking the tail node directly. - **Verdict:** The hardening is **formally complete for cross-uid attacks**. The residual same-uid self-race on `chmodSync` is a P2 defense-in-depth item (acknowledged in the 017 remediation), survivable because no privilege boundary is crossed. The three-layered fence (directory → fresh-bind → post-bind) provides overlapping protection that holds even if one layer is bypassed. **LOW CONFIDENCE** on whether the same-uid race window is practically exploitable — needs the stress-test above.
