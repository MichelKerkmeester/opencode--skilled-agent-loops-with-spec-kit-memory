# Iteration 13: W5 adversarial: driver/API and pragma family

## Focus
Adversarial re-verification of seven driver/API and pragma-family verdicts against the vendored Turso v0.7.0-pre.6 tree. I deliberately avoided COMPAT.md and the earlier citations, instead reading the core translation layer (`core/translate/pragma.rs`, `attach.rs`, `vacuum.rs`), the WAL checkpoint-mode enum (`core/storage/wal.rs`), core unit tests (`core/connection.rs`), fuzz tests, and the JS bindings source (`bindings/javascript/packages/common/{promise,compat}.ts`, package manifests). All paths below are relative to the vendored root `.opencode/specs/z_future/sqlite-to-turso/external/turso-main/`. The headline result: both pragma contradictions resolve AGAINST iteration 1, six claims survive, and CLAIM-4 (ATTACH/DETACH "fully supported") is overturned — SQL-level ATTACH/DETACH is experimental-flag-gated.

## Findings
- **KNOWN — CONFIRMED — CLAIM-1 (JS .pragma() in both APIs):** The async API implements `async pragma(source, options)` building `` `PRAGMA ${source}` `` and preparing it [SOURCE: bindings/javascript/packages/common/promise.ts:410-421]; the sync better-sqlite3 compat API implements a non-async `pragma(source, options)` with the same construction [SOURCE: bindings/javascript/packages/common/compat.ts:148-159]. Both live in `packages/common`, shared by the native and wasm packages. Verified in source, not docs.
- **NEW — CONFIRMED (iteration-5 side); iteration-1 OVERTURNED — CLAIM-2 (wal_checkpoint(TRUNCATE)):** The engine-level pragma handler parses a named argument via `CheckpointMode::from_str` and defaults to PASSIVE when no argument is given [SOURCE: core/translate/pragma.rs:891-917]; the mode enum defines all four SQLite forms — `Passive`, `Full`, `Restart`, `Truncate` [SOURCE: core/storage/wal.rs:153-167]. TRUNCATE is exercised and `.unwrap()`-asserted in core unit tests [SOURCE: core/connection.rs:4587], fuzz tests [SOURCE: tests/fuzz/mvcc_rowid_allocator.rs:411,475,550], and JS binding tests for both native and wasm [SOURCE: bindings/javascript/packages/wasm/promise.test.ts:184; bindings/javascript/packages/native/compat.test.ts:156]. The parameterized form is supported at engine level; iteration 1's "unsupported" verdict is refuted. Exact forms: no-arg → PASSIVE; `wal_checkpoint(PASSIVE|FULL|RESTART|TRUNCATE)` all parse (unknown names raise "Unknown Checkpoint Mode").
- **NEW — CONFIRMED (iteration-9 side); iteration-1 OVERTURNED — CLAIM-3 (synchronous=NORMAL):** The write handler explicitly accepts NORMAL: numeric `"0"`→Off, `"1"`→Normal, and keyword matching `b"OFF"|b"0"`→Off, `b"NORMAL"|b"1"`→Normal, with everything else coerced to Full — a deliberate deviation from SQLite noted in-source ("SQLite defaults to NORMAL for invalid values, but we want to default to a higher durability level") [SOURCE: core/translate/pragma.rs:650-672]. The value is honored via `connection.set_sync_mode(mode)` and read back by the query side [SOURCE: core/translate/pragma.rs:1499-1506]. Accepted values: OFF/0, NORMAL/1, FULL (and any unrecognized token silently becomes FULL — no EXTRA tier, no error). Iteration 1's "only OFF/FULL" is refuted.
- **NEW — OVERTURNED — CLAIM-4 (ATTACH/DETACH fully supported):** A complete translation exists (`translate_attach`/`translate_detach` emitting `ScalarFunc::Attach`/`Detach`) [SOURCE: core/translate/attach.rs:13-178], BUT both statements hard-fail unless the experimental flag is enabled: "ATTACH is an experimental feature. Enable with --experimental-attach flag" [SOURCE: core/translate/attach.rs:21-25] and the identical gate for DETACH [SOURCE: core/translate/attach.rs:125-129]. The CLI exposes it as opt-in `experimental_attach` wired through `with_attach(...)` [SOURCE: cli/app.rs:95,244; core/lib.rs:269-272]. Core tests attach via the programmatic Rust API `conn.attach_database(...)` (which bypasses the SQL gate) [SOURCE: core/connection.rs:4578], and `rg experimental_attach` finds zero hits under `bindings/javascript`, so the JS SDK has no visible switch to enable SQL-level ATTACH. "Fully supported" is wrong as stated: supported-when-flagged, default off.
- **KNOWN — CONFIRMED — CLAIM-5 (VACUUM INTO unconditional, bare VACUUM gated):** `VACUUM INTO 'path'` emits `Insn::VacuumInto` with no flag check [SOURCE: core/translate/vacuum.rs:30-38], while bare in-place VACUUM bails with "VACUUM is an experimental feature. Enable with --experimental-vacuum flag" [SOURCE: core/translate/vacuum.rs:39-42]. Module doc states this design directly [SOURCE: core/translate/vacuum.rs:11-12]. Additional nuances found: in-place VACUUM is also rejected under experimental multiprocess WAL [SOURCE: core/translate/vacuum.rs:45-48], and schema-qualified VACUUM (any schema other than main) is unsupported even with the flag [SOURCE: core/translate/vacuum.rs:51-56].
- **KNOWN — CONFIRMED — CLAIM-6 (schema-qualified PRAGMA):** `translate_pragma` takes `&ast::QualifiedName`, resolves it via `resolver.resolve_database_id(name)` and tracks `schema_was_explicit = name.db_name.is_some()`, passing the resolved `database_id` into the handler dispatch [SOURCE: core/translate/pragma.rs:212-246]; the checkpoint handler emits `Insn::Checkpoint { database: database_id, .. }` [SOURCE: core/translate/pragma.rs:907-911]. Proven end-to-end against an attached database: `conn.execute("PRAGMA aux.wal_checkpoint(TRUNCATE)").unwrap()` in a passing core test [SOURCE: core/connection.rs:4585-4587]. The mechanism is generic at the entry point, so qualified pragmas like `<schema>.journal_mode` route through the same database-id resolution.
- **KNOWN — CONFIRMED — CLAIM-7 (./compat sync API with transaction variants):** Both native package manifests export `"./compat": "./dist/compat.js"` [SOURCE: bindings/javascript/packages/native/package.json:14; bindings/javascript/sync/packages/native/package.json:14]. The compat implementation provides `transaction(fn)` and attaches `.deferred`/`.immediate`/`.exclusive` variants via `wrapTxn("DEFERRED"/"IMMEDIATE"/"EXCLUSIVE")` with the better-sqlite3 property-propagation trick [SOURCE: bindings/javascript/packages/common/compat.ts:116-144]. The API is sync (non-async `pragma`, `db.exec(...)` called without await in its tests [SOURCE: bindings/javascript/packages/native/compat.test.ts:156]).

## Ruled Out
