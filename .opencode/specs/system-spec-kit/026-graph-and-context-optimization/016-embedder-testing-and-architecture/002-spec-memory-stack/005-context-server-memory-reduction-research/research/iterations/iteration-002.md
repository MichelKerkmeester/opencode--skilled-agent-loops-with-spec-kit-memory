# Iteration 2 — lazy-load and on-demand init opportunities (defer model load until first embed call, defer code-graph load until first query)

## Summary

The embedding provider path is already mostly lazy: provider construction is deferred until `generateEmbedding()`/`generateDocumentEmbedding()`/`generateQueryEmbedding()`, and local model weights are loaded inside provider-specific first-use methods. The larger lazy-init opportunity is around startup side effects: `context-server` opens and migrates the 716 MB vector DB, can run auto-migration, starts an immediate retry batch, and schedules a startup scan before the user issues an embedding/search request. Code-graph is not resident inside `context-server`; startup reads a marker file, while full graph queries are already RPC-on-demand.

## Findings

### Finding 1: `context-server` eagerly opens the hot vector DB and initializes search modules before the first tool call
- Evidence: `mcp_server/context-server.ts:1568-1571` calls `vectorIndex.initializeDb()` during `main()`; `mcp_server/context-server.ts:1597-1615` immediately runs integrity and dimension checks; `mcp_server/context-server.ts:1677-1704` initializes checkpoint/access/hybrid/session/causal graph consumers; `mcp_server/context-server.ts:1706-1711` can rebuild BM25 at startup. The DB opener sets `cache_size = -64000`, `mmap_size = 268435456`, and `temp_store = MEMORY` in `mcp_server/lib/search/vector-index-store.ts:800-806`.
- Memory impact: this does not load the embedding model, but it makes SQLite/`sqlite-vec` resident immediately. On the live store, `context-index__llama-cpp__unsloth-embeddinggemma-300m-gguf__768__q8.sqlite` is 716 MB; its hot vector/cache tables include `memory_index` ~70.5 MB, `vec_1024` ~58.0 MB, `vec_768` ~51.6 MB, and `embedding_cache` ~39.2 MB. Combined with the current 64 MiB SQLite page-cache target and 256 MiB mmap ceiling, a true idle server can plausibly retain 100-300 MB before any model is loaded.
- Proposed change: split `main()` into a thin MCP bootstrap plus an idempotent `ensureMemoryRuntimeInitialized(reason)` function. Keep `Server`, `ListToolsRequestSchema`, static tool definitions, runtime detection, and signal handlers in startup; move `vectorIndex.initializeDb()`, `verifyIntegrity()`, `validateEmbeddingDimension()`, DB consumer `init()` calls, BM25 rebuild, retry job start, reindex resume, and startup scan scheduling behind `ensureMemoryRuntimeInitialized()`. Call that guard at the start of memory-owning tool handlers and mutation hooks. Add a non-initializing DB accessor, e.g. `tryGetDb()` in `vector-index-store.ts`, so session priming and health snapshots can report "database lazy/not opened" without accidentally calling `get_db()`/`initialize_db()`.
- Trade-off: the first real memory call pays DB open, schema validation, and module-init latency. Startup health logs become less complete unless a lightweight summary file is used for instructions. The feature set stays intact; the timing of initialization changes.
- Effort: L

### Finding 2: startup auto-migration can become a hidden first embedder load
- Evidence: `mcp_server/context-server.ts:1553-1559` calls `getStartupEmbeddingProfile()` and `runAutoMigrationIfNeeded()` before DB startup completes. `shared/embeddings/factory.ts:456-549` checks for a llama-cpp target plus an hf-local source store, loads the migration runner via dynamic import, and runs `runMigration()` from source to target. That migration path is designed to re-embed rows, not just copy metadata.
- Memory impact: if this branch triggers, it can load the local embedding backend during startup. Current measured model references: the llama-cpp GGUF file is 313 MB on disk and the benchmark artifact reports ~500 MB loaded RSS for the baseline; Ollama-backed Jina is ~495 MB loaded but outside `context-server` RSS; Nomic is ~578 MB in the Ollama runner. Deferring this path avoids a startup jump of roughly 500 MB in `context-server` for llama-cpp, plus transient batch buffers and SQLite write pressure.
- Proposed change: change `runAutoMigrationIfNeeded()` into a metadata-only startup check by default. At startup, write or update a pending migration row/marker such as `embedder_jobs` or `.auto-migration-pending.json` with source, target, row counts, and reason. Start the actual migration only when an explicit embedder/migration tool is called, or when the first user-initiated embedding call proves the provider is already loaded. Keep an override like `MEMORY_AUTO_MIGRATE_HF_TO_LLAMA=eager` for operators who want current behavior; keep `false` as opt-out.
- Trade-off: migration completion is no longer guaranteed before the first query after a provider switch. The query path must keep using the currently active profile until the queued migration completes, and health/status should surface the pending migration clearly.
- Effort: M

### Finding 3: retry and startup-scan jobs can load the model immediately after startup even though provider construction is lazy
- Evidence: `mcp_server/context-server.ts:1748-1757` starts the embedding retry manager during startup, and `mcp_server/lib/providers/retry-manager.ts:872-877` immediately runs an initial background batch before waiting for the 5 minute interval. That batch reaches `generateDocumentEmbedding()` at `retry-manager.ts:628-629` on cache miss. Separately, `context-server.ts:1899-1902` schedules `startupScan(DEFAULT_BASE_PATH)` with `setImmediate()`, `context-server.ts:1294-1297` calls `indexSingleFile()`, `handlers/memory-save.ts:2699-2705` defaults `asyncEmbedding` to `false`, and `handlers/save/embedding-pipeline.ts:152-155` generates document embeddings on cache miss.
- Memory impact: the current live DB has `pending_retry=0`, so this path probably does not explain the observed PID right now. In a dirty workspace or after async saves, however, the first retry or updated-file scan can load the local model without a user search/save request. For llama-cpp this is the same ~500 MB loaded-model class as Iteration 1; for hf-local q8 the source comments estimate ~310 MB vs ~620 MB fp32; for Ollama the loaded model is in the Ollama runner process, not the MCP server.
- Proposed change: add an embedder activation gate shared by retry and startup scan. For example, introduce `shouldAllowBackgroundEmbedding()` in `lib/providers/retry-manager.ts` or `shared/embeddings.ts` that returns true only after `markExplicitEmbeddingUse()` has been called by `memory_search`, `memory_context`, `memory_save`, `memory_index_scan`, or `embedder_start_reindex`, or after a configurable grace period such as `SPECKIT_BACKGROUND_EMBEDDING_GRACE_MS`. Change `startBackgroundJob()` to schedule its first run after the interval/grace instead of immediate execution, and change startup scan to pass `asyncEmbedding: true` for scan-origin indexing unless the activation gate is open. Existing retry and full indexing still happen; they stop being the accidental first model loader.
- Trade-off: newly discovered or pending memories may remain lexical-only briefly. Vector recall catches up after explicit embedding use or the grace timer. Debugging gets one extra state: DB indexed, embeddings pending because background embedding is gated.
- Effort: M

### Finding 4: provider/model lazy loading is already mostly correct; the missing abstraction is generic unload/loaded-state, not another provider restriction
- Evidence: `shared/embeddings.ts:365-418` keeps a lazy singleton provider and calls `createEmbeddingsProvider({ warmup: false })` only on first embedding. `shared/embeddings/providers/hf-local.ts:224-227` starts with `extractor = null`, and `hf-local.ts:230-245` imports `@huggingface/transformers` only in `getModel()`. `shared/embeddings/providers/llama-cpp.ts:206-249` loads node-llama-cpp, the GGUF model, and embedding context only in `loadRuntime()`, which is reached from `generateEmbedding()` via `llama-cpp.ts:356-368`. `shared/embeddings/factory.ts:1108-1115` returns success for local API-key validation without loading a model.
- Memory impact: provider construction alone is not the 1 GB root. The expensive boundary is first inference/model load: ~500 MB for the current llama-cpp baseline per benchmark docs, ~495-578 MB for measured Ollama runners outside the MCP process, and low single-digit MB for Voyage/OpenAI in-process state.
- Proposed change: extend `IEmbeddingProvider` in `shared/types.ts:54-63` with optional `dispose?(): Promise<void>` and `isLoaded?(): boolean`, then implement no-op/lightweight versions for Voyage/OpenAI/Ollama-style adapters and real unload for in-process local providers. In `shared/embeddings.ts`, expose `disposeEmbeddingProvider()` and `getProviderLoadedState()` so retry/health/status can observe loaded state without forcing `getEmbeddingProfileAsync()` or model warmup. This builds on Iteration 1's llama-cpp idle-unload finding but keeps the contract model-agnostic for Voyage, llama-cpp, Ollama, BGE, mxbai, and future adapters.
- Trade-off: slightly broader provider interface and tests across every backend. The upside is that memory policy becomes a provider capability rather than a llama-cpp special case.
- Effort: M

### Finding 5: code-graph is not a resident `context-server` structure; only auto-enrichment can spawn query-time RPC work
- Evidence: `mcp_server/lib/code-graph-boundary.ts:1-5` documents the process boundary. Startup/session priming uses marker reads: `code-graph-boundary.ts:153-187` reads `.code-graph-readiness.json`, and `hooks/memory-surface.ts:487-494` uses `getCodeGraphStatusSnapshot()` to build the prime package. Full graph reads use `callCodeGraphTool()` at `code-graph-boundary.ts:272-305`, which creates an MCP client/stdio transport and closes it in `finally`. The automatic dispatch enrichment path calls `code_graph_context` in `context-server.ts:541-546` and is reached after successful tool dispatch at `context-server.ts:970-973` when arguments contain file paths.
- Memory impact: direct RSS saving inside PID 3216 is effectively 0 MB because the graph database and graph server are not held in this process. The code-graph DB on disk is currently 53 MB, and any runtime RSS belongs to the separate `mk_code_index` process/launcher. `ps` measurement was blocked in this sandbox (`operation not permitted`), so process RSS attribution still needs an external probe.
- Proposed change: do not try to "lazy-load code graph" inside `context-server`; it already is marker-only until an RPC. If whole-machine RSS is the target, move the policy to the runtime config/launcher layer: avoid launching `mk_code_index` in lockstep unless a code-graph tool is called, and keep `context-server` auto-enrichment marker-only unless `SPECKIT_CODE_GRAPH_AUTO_ENRICH=true` or the tool call is structurally explicit. The in-process change is small: gate `resolveDispatchGraphContext()` at `context-server.ts:970-973` behind a structural intent flag or explicit graph-enabled caller context, while still returning marker status and recommended graph calls in session priming.
- Trade-off: automatic file-path enrichment may arrive one call later unless auto-enrich remains enabled. Explicit `code_graph_context`, `code_graph_query`, and bootstrap flows are preserved.
- Effort: S for context-server gating; M if runtime MCP lazy launch is included.

## Cross-references

This builds on Iteration 1 Finding 1 by separating "first model load is already lazy" from "model remains resident after first use"; idle unload remains the right follow-up after these startup gates. It builds on Iteration 1 Finding 2 by showing that SQLite memory pressure starts at `main()` before any tool call, not only under search workload. It agrees with Iteration 1's code-graph negative knowledge: there is no in-process code-graph mirror in `context-server`, so code-graph RSS must be attributed to the standalone `mk_code_index` process.

## Negative knowledge (ruled-out)

- `resolveStartupEmbeddingConfig()` is not the local model loader for llama-cpp or hf-local. For local providers, `validateApiKey()` returns without model load; the only external validation call is for Voyage/OpenAI, which is not a material RSS contributor.
- `embedder_list` does not load every model. Its Ollama readiness path calls `/api/tags`, and the llama-cpp baseline adapter reports readiness from `isModelLoaded()` plus dimension metadata; it does not call `embed()`.
- Moving `VoyageProvider` construction later is not a meaningful memory optimization. It stores API key/base URL/model counters; its large work is remote.
- Deferring code-graph marker reads is not useful. The marker is a small JSON file; the memory-sensitive boundary is the RPC to `mk_code_index`, not `getCodeGraphStatusSnapshotFromMarker()`.
- Shrinking the in-memory embedding LRU remains low priority for this dimension. Iteration 1 already showed the raw vectors are single-digit MB at 1,000 entries.

## Open questions

- Should startup dynamic instructions be generated from a lightweight `memory-startup-summary.json` instead of opening the hot DB for `memory_stats`?
- What should the default background embedding grace be: first explicit embed only, a fixed delay, or a hybrid "delay unless pending queue exceeds N" policy?
- Does the auto-migration path still need to be default-eager after the newer embedder swap/job system, or can all provider migrations be represented as resumable `embedder_jobs`?
- For whole-machine RSS, which runtime currently launches `mk_code_index` and `mk_skill_advisor` lockstep with `mk-spec-memory`: Codex config, OpenCode config, or an external supervisor?
