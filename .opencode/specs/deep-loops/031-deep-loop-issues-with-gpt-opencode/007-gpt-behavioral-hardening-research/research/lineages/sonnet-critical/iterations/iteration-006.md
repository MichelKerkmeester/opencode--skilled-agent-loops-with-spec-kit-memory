## Finding 15 — "Wait" survives adversarial pressure on the self-assessment-bias axis specifically, because a non-GPT lineage reached the same conclusion independently

The charter (§9.2) asks specifically to check whether `gpt-fast-high`'s conclusions look "softer, more deferential, or more self-protective" than the evidence supports. If "wait, do not unpark FIX-5" were primarily an artifact of GPT avoiding a conclusion that indicts its own dispatch reliability, the non-GPT lineage (`glm-max`, GLM-5.2, which has no such conflict of interest researching GPT's failures) should plausibly have reached a different, more aggressive conclusion. It did not — glm-max independently reaches the same "WAIT... do NOT unpark now; do NOT close as sufficient" verdict [glm-max/research.md:119], and its supporting reasoning is if anything more explicit about *why* waiting is not evasion: "Why not never (CONFIRMED): 'never' is only viable if cheaper layers prove sufficient — exactly what is unproven" [glm-max/research.md:124]. Two independently-run lineages, one with a structural conflict of interest and one without, converging on the same substantive verdict is evidence *against* self-protective bias being the primary driver of this specific conclusion — this is exactly the "agreed with each other for the wrong reason (shared blind spot)" possibility the charter also asks to check for (§9.3.1), and on this KQ the more likely explanation is convergent, correct reasoning, not a shared blind spot.

## Finding 16 — "Wait" is additionally defensible on evidence this round itself produced: FIX-5 does not fix Mode D or the latency symptom, so unparking it now would not resolve the full confirmed symptom set anyway

Iteration 3 established a CONFIRMED (not merely inferred) instance of Mode D — the `GENERAL AGENT REQUIRED` self-check firing during phase 005's research-mode smoke. FIX-5 (host hard identity / process isolation) is architecturally scoped to preventing wrong-agent dispatch; both the predecessor research and this packet's own phase 006 decision record are explicit that hard identity "prevents wrong-agent dispatch but does nothing about GPT halting on an advisory step it reads as a gate" [glm-max/research.md:35, itself citing `../001/research.md §8b`]. Given iteration 3's confirmed Mode D instance, and given both lineages attribute the "slow as orchestrate" symptom to role-resolution latency rather than identity mismatch [consolidated research.md §0], **unparking FIX-5 today would leave at least 2 of the operator's 4 confirmed symptoms (stuck-on-flows, slowness) unaddressed**, while the cheaper prompt-layer fixes (Mode D self-check rewrite from iteration 3, the corrected NDP-safe orchestrate routing fix from iteration 5) target exactly those two symptoms directly. This is a substantive, non-bias-driven argument for sequencing cheap-fix-first that neither prior lineage stated this precisely (both gesture at "cheaper layers first" as a blast-radius/reversibility argument, not as a coverage argument — i.e., "FIX-5 wouldn't even fix 2 of the 4 symptoms" is a stronger, different claim than "FIX-5 is expensive and unproven").

## Finding 17 — But the "gate cannot fire, indefinitely parked" residual risk (both lineages' top concern) is more pessimistic than iterations 1-2's own findings support

The consolidated synthesis names this the **highest-priority residual risk**: "the entire 010-through-012 chain is unreachable without a confirmed `OPENCODE_PID`-free external shell. Confirm this before committing to the plan above" [consolidated research.md §4], echoing glm-max's "if no OPENCODE_PID-free shell exists, the gate cannot fire and FIX-5 stays parked indefinitely — the same trap as 005/006" [glm-max/research.md §12, §1.3 of consolidated]. Iteration 2 (Finding 4-5) already established that the specific guard responsible for 2 of phase 005's 4 blocked attempts (the `cli-opencode` ADR-001 self-invocation guard) is scoped to nested `opencode`-invoking-`opencode` dispatch specifically, and does **not** gate native Task dispatch or cross-CLI dispatch (e.g. `cli-claude-code`, the exact executor this very lineage runs under). **The KQ1/KQ6 GPT-vs-Claude comparison does not strictly require a genuine `OPENCODE_PID`-free external shell for its Claude/native leg** — that leg can run via `cli-claude-code` or native Task dispatch inside the existing environment, right now, without waiting for an elusive clean external shell. Only the GPT leg (which must go through `cli-opencode` specifically, per the packet's own established dispatch model for exercising GPT) needs the external-shell condition. This halves the "unreachable" framing: the comparison's non-GPT half is already runnable, and this lineage's own clean execution (iterations 1-6, no failures) is itself informal evidence toward it (per iteration 2 Finding 5, with the stated caveats).

**Correction to KQ9/§1.3's framing:** "the gate cannot fire... indefinitely parked" should be revised to "the GPT leg of the gate requires a genuine external shell; the Claude/native leg does not and can be run now to establish half the comparison." This is a more actionable, less fatalistic framing of the same residual risk, and it directly answers glm-max's own open question ("Confirm an OPENCODE_PID-free external shell is available" [glm-max/research.md §12]) with a partial "you don't need one for the whole thing."
