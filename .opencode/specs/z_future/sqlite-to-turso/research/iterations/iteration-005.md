# Iteration 5: C5 MVCC and concurrency model

## Focus
Revalidate the baseline (v0.5.0-era) claims about Turso's MVCC/BEGIN CONCURRENT maturity, the restrictions that drove Gap 7 (AUTOINCREMENT) and Gap 8 (triggers), the WAL/multi-process story underpinning the daemon/lease single-writer architecture (skill-advisor `lease.ts`, code-graph WAL probes, spec-kit `wal_checkpoint(TRUNCATE)` recovery), and the contention-error surface that must replace the `SQLITE_BUSY` predicate in `runDaemonStateMutation` — all against the vendored Turso v0.7.0-pre.6 source.

## Findings
- **NEW — CHANGED (Gap 7 premise): AUTOINCREMENT now works under MVCC via atomic sequences.** The vendored test file literally strikes the old claim: "AUTOINCREMENT is not supported in MVCC mode due to sqlite_sequence / AUTOINCREMENT is supported in MVCC mode via atomic sequences" [SOURCE: core/mvcc/database/tests.rs:10412-10414], with passing-by-design tests `test_autoincrement_works_in_mvcc` [tests.rs:10416], `test_autoincrement_insert_works_for_preexisting_table` (pre-MVCC tables migrate) [tests.rs:10438-10441], `test_concurrent_autoincrement_inserts` [tests.rs:10498-10501], and `sqlite_sequence` reconciliation after checkpoint [tests.rs:10536-10539]. Baseline 003 Gap 7 ("AUTOINCREMENT does not work under MVCC", research/003:399-414) and 001:215 are out of date. Residual caveat: a comment flags an edge where "MVCC AUTOINCREMENT [is] able to re-emit a rowid already in use" in a recovery scenario [SOURCE: core/connection.rs:3555].
- **NEW — CHANGED (Gap 8 premise): no trigger-under-MVCC rejection exists in the engine anymore.** An exhaustive sweep of MVCC `bail`/unsupported errors finds rejections only for: indexes without rowid [core/mvcc/cursor.rs:1211], custom index modules [core/translate/index.rs:66, core/schema.rs:1751], REINDEX [index.rs:619], CDC [core/translate/pragma.rs:595], virtual tables [core/translate/schema.rs:1692], WITHOUT ROWID tables [core/vdbe/execute.rs:1237], and ClearBtree [execute.rs:11320] — triggers are absent from that list, and MVCC tests actively create and recover triggers [tests.rs:1384-1396, 9309]. Verdict vs baseline "triggers do not fire under BEGIN CONCURRENT" (003:448): CHANGED, likely lifted — though this is inferred from absence-of-rejection plus test coverage, not an executed firing test.
- **NEW — cross-gap interaction: Tantivy FTS and MVCC are mutually exclusive.** "Custom index modules are not supported in MVCC mode" [SOURCE: core/translate/index.rs:66; core/schema.rs:1751]. Since Turso FTS is delivered as a custom `index_method` module (established iteration finding), the baseline's W8b plan ("Tantivy removes trigger maintenance", 003:457) cannot be combined with MVCC mode at this version.
- **PARTIAL — C5a maturity verdict: MVCC remains explicitly experimental.** It is now a journal mode whose literal is `experimental_mvcc` (serialized "mvcc") [SOURCE: core/storage/journal_mode.rs:26], enabled in tests via `PRAGMA journal_mode = 'experimental_mvcc'` [tests.rs:10484]; the repo's own agent guide labels it "experimental multi-version concurrency (WIP)" [SOURCE: external/turso-main/CLAUDE.md, Guides section]. Hardening is very active: BEGIN IMMEDIATE/CONCURRENT interlock [CHANGELOG.md:155], shuttle tests [CHANGELOG.md:119], MVCC REPL [CHANGELOG.md:110], recovery-heavy Whopper profiles [CHANGELOG.md:16-17], plus a `PRAGMA mvcc_checkpoint_threshold` tuning surface [COMPAT.md:230]. Baseline 002:99's re-evaluation criterion "MVCC stable with trigger support" is therefore half-met: trigger support appears present, stability is still self-declared experimental.
- **NEW — REFUTED (baseline architecture assumption): "Turso will never support multi-process."** v0.7.0-pre.6 ships experimental **multi-process WAL coordination via a `.tshm` sidecar** ("cross-process WAL readers and writers") [SOURCE: external/turso-main/README.md:61]. Mechanism: sidecar at `{db_path}-tshm` [core/storage/wal.rs:116-118], an mmap'd "authority" (`MappedSharedWalCoordination`) that per-connection coordination delegates to and which "serializes across processes" [wal.rs:1277-1293]; readers register with the authority [wal.rs:2125]; open mode `SharedWalCoordinationOpenMode::MultiProcess` [wal.rs:2263, core/lib.rs:298]. The old in-code claim "we will never support multi process… don't need WAL's index file" survives only as a stale comment [wal.rs:2662-2663].
- **PARTIAL — C5b verdict: the daemon/lease single-writer model SURVIVES (still required).** Multiprocess WAL is opt-in (`DatabaseOpts.enable_multiprocess_wal` / `with_multiprocess_wal(true)`, default off) [SOURCE: core/lib.rs:222,279-280], rejected for in-memory paths, unsupported IO backends, and unsupported filesystems [lib.rs:802-812], fails fast on mixed legacy/multiprocess opens ("Database is already open with[out] experimental multiprocess WAL in another process") [lib.rs:865,898], and — decisively — is **not exposed in the JavaScript bindings at all** (zero hits in bindings/javascript). Also COMPAT.md:69: "We don't support mixed SQLite and Turso in multi-process scenarios." So Node consumers remain effectively single-process-per-DB: `lease.ts` heartbeat-lease single-writer and code-graph daemon ownership remain necessary post-migration (their *synchronous transaction* idiom is a separate gap-16 issue, not a concurrency-model one).
- **KNOWN/UNCHANGED — `wal_checkpoint(TRUNCATE)` recovery path survives for local files.** `PRAGMA wal_checkpoint(TRUNCATE)` is exercised directly in vendored tests [SOURCE: core/mvcc/mod.rs:321; core/mvcc/persistent_storage/logical_log.rs:4658; core/connection.rs:4587 (aux schema)], and checkpoint modes incl. Full exist [wal.rs:5842]. Spec-kit's `rollbackBadApply` checkpoint (context-report.md:19) ports as-is for local-file DBs; the context-report's "no WAL checkpoint concept" caveat applies only to remote-replica deployments.
- **NEW — C5c: there is no `err.code === 'SQLITE_BUSY'` in the Turso JS surface; the retry predicate must be rewritten.** Core errors: `LimboError::Busy` → "Database is busy" [SOURCE: core/error.rs:66-67], `BusySnapshot` → "Database snapshot is stale. You must rollback and retry the whole transaction." [error.rs:70-71], `WriteWriteConflict` → "Write-write conflict" (MVCC write contention) [error.rs:84-85], `TableLocked` → "Runtime error: database table is locked" [error.rs:62-63]; a numeric `SQLITE_BUSY = 5` exists only as a core-internal C-compat constant [error.rs:281-285]. The JS binding maps `StepResult::Busy` to a **generic napi error with message "database is locked" and no code property** (`Error::new(Status::GenericFailure, …)`) [SOURCE: bindings/javascript/src/lib.rs:212,225-227]. `runDaemonStateMutation`'s fresh-DB-per-attempt/3-retry predicate must match on message substrings ("database is locked", "Database is busy", and under MVCC "Write-write conflict"/"snapshot is stale") or on a wrapper-normalized code — message-matching is fragile and should be centralized.
- **NEW — mitigations for the retry predicate exist natively:** the JS binding accepts a `busy_timeout`-style open option wired to `Connection::set_busy_timeout` [SOURCE: bindings/javascript/src/lib.rs:255-268,330-331; core/connection.rs:3923], and `PRAGMA busy_timeout` is implemented [core/pragma.rs:146] — short contention can be absorbed before errors surface. Counter-nuance: at least one WAL contention class is explicitly non-retryable: "Retrying with busy_timeout will NEVER HELP." [wal.rs:3176] — the new predicate should distinguish retryable Busy from fatal lock states rather than blind-retrying 3×.

## Ruled Out
- "Turso is permanently single-process; the WAL has no cross-process story" — refuted by the shipped experimental `.tshm` multi-process WAL coordination (README.md:61; core/lib.rs:222ff); the wal.rs:2662 "never support multi process" comment is stale.
- Gap 7's premise "AUTOINCREMENT does not work under MVCC" — refuted by atomic-sequence implementation and four dedicated tests (tests.rs:10412-10539); the W7b "switch to UUIDs" contingency (003:422-433) is no longer needed for MVCC compatibility.
- Any expectation that Turso JS errors carry better-sqlite3-style `code: 'SQLITE_BUSY'` — the binding emits message-only generic napi errors (lib.rs:212,225-227).
- COMPAT.md as a source of MVCC maturity signal — it contains essentially no MVCC content (single hit: `PRAGMA mvcc_checkpoint_threshold`, COMPAT.md:230); maturity evidence lives in journal_mode.rs, the agent guides, tests, and CHANGELOG.

## Dead Ends

## Reflection
- What worked: directory-wide `rg` over the vendored core for `bail`/"not supported in MVCC" strings produced a complete, citable restriction list in one pass; the MVCC test file's own corrected doc-comment (10412-10414) was the single strongest piece of evidence for the Gap 7 verdict; chasing `tshm`/`authority` from a single wal.rs hit unlocked the entire multi-process story.
- What failed: WebSearch/WebFetch permissions were denied, so the "current web state" half of the mandate is unfulfilled; sandbox rejected `;`-chained and `cd` compound commands, costing three tool calls; trigger-under-MVCC support is established by absence-of-rejection plus test coverage rather than a positive documentation statement or an executed firing test.
- Confidence: **high** for C5b and C5c (direct source citations for the multiprocess WAL gate, JS error shape, and checkpoint/busy_timeout survival); **medium-high** for C5a (AUTOINCREMENT verdict is airtight; the trigger verdict is well-evidenced inference); **no coverage** of post-vendor web state.
