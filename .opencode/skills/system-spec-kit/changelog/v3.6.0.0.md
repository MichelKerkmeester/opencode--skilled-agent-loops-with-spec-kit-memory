# v3.6.0.0, Harder Core, Longer Reach

The center of this release is the part you rely on without thinking about it. Memory and search got smarter about telling you the truth, and the background services that hold them up got tougher under stress. A genuine match is cited again, search keeps working when its index is damaged, and the daemons (the long-running background services behind memory and search) now recover instead of duplicating themselves or corrupting their own data.

Around that core, two more things moved. The five separate deep-loop skills folded into one hub running on a shared backend, and a new set of terminal-driven design tools arrived. Both changes are real, but neither asks anything new of you.

None of this changes how you actually call the system. `memory_search`, `memory_save`, the `/deep:*` commands and the agent names all behave exactly as before. The work happened in the failure paths and the corners that never fit the happy case, not on the path you use every day.

## Search

Search picked up a handful of fixes this release, and most of them share a theme: a real match should be findable and the score on it should be honest.

#### Honest, Citable Answers

Search cites the matches it finds. A genuine 0.89 match reads as a good match, cites its results and reports a real confidence figure. A scoring mismatch had been holding good verdicts back, since the confidence step compared fusion scores (which cluster low near 0.03 because RRF, a method that blends two ranked result lists, compresses them) against thresholds tuned for cosine similarity at 0.7 and 0.4. A resolver now reads the right score for confidence while leaving the ranking order alone.

&nbsp;

#### Memory-Safe Keyword Fallback

When the semantic path is unavailable, search falls back to keyword matching, and that fallback now stays light on memory. It packs its data tightly and weights title and trigger matches above body text, so the right entry ranks first. The payoff is concrete: a realistic warmup that once spiked to 687MB now holds at 137MB, well under budget, with the ranking unchanged.

&nbsp;

#### Scoped Queries Return Their Full Set

A scoped search gives you everything that actually matches. The keyword engine gathers the full candidate set first, applies your filters and only then trims to your limit, so a narrow query comes back complete rather than short.

&nbsp;

#### Scope Isolation

A scoped search stays inside its scope and cannot pull in results from other scopes. The broaden-and-rescue path (the lexical backfill and sibling injection that widens a thin result set) had re-queried the index without re-applying the spec-folder scope filter, so a scoped search could surface rows from outside its scope. Every injected row now re-applies the scope filter, backed by a regression test. This is separate from the full-set fix above: that one keeps a scoped query from coming back short within its scope, this one keeps it from leaking across scopes.

&nbsp;

#### Safe Query Handling

A `/memory:search` query that contains shell-special characters now resolves exactly as typed, since the query argument is quoted, so a search for something with punctuation behaves predictably.

&nbsp;

#### Cold and Archived Results

Older and archived entries stay within reach. Results from cold and deprecated tiers flow back into the keyword and trigger channels under a freshness-aware ranking from FSRS (a spaced-repetition scheduler that scores how relevant something still is), and the semantic lane fills in missing projections so cold entries surface when they match.

&nbsp;

#### One Score Everywhere

The same match reads the same way on every surface. Rendered output reports a single score name on a plain 0-to-1 scale to two decimals, and the older confidence values and percentages are gone from what you see, so two callers reading the same result see one number.

&nbsp;

#### Ranking Explanation Trace

When a result ranks where it does and you want to know why, you can now ask. An opt-in trace returns a breakdown of how the ranker scored a result, pulled from its own intermediate values, and an inline warning flags when one memory contradicts or supersedes another.

## Memory Safety

A retrieval system you cannot trust to be honest, or to leave your edits alone, is worse than a slow one. Four protections close that gap, and the structural ones are on for everyone.

#### Secret Scrubber

Credentials should never reach the index, even by accident. A scrubber now sits at the very front of the pipeline, ahead of hashing, embedding and keyword indexing, and redacts API keys, tokens, bearer values and private-key blocks into typed redaction markers. It fails closed: if scrubbing cannot finish, the write is refused rather than stored in the clear.

&nbsp;

#### Provenance Guard

When you edit a memory by hand, an automated process should not be able to quietly overwrite it. A new provenance column records who wrote each field, human or automated, and the write guard uses it to skip automated overwrites of human-authored content before the change lands. A companion tagger marks the automated writers, so the rule holds everywhere rather than at one checkpoint.

&nbsp;

#### Safer Retention

The cleanup that ages out old entries checks an entry's tier and pinned state before deleting anything. A constitutional or critical entry that hits its expiry earns a logged refusal instead of being removed, so the cleanup stays within what it should touch.

&nbsp;

#### Continuity Across Compaction

A long session resumes from fresh state instead of stale notes. An opt-in snapshot (the flag `SPECKIT_AUTHORED_CONTINUITY_SNAPSHOT`, off by default) refreshes the handover and continuity notes right before a long session is compacted, and startup now reports what it restored. It stays off until you turn it on.

&nbsp;

#### Vector Self-Heal

The system detects, quarantines and rebuilds a damaged vector shard (a slice of the semantic search index), and the repair survives a restart. A persisted marker and a real completeness check decide between resuming the repair and clearing a stale flag, so a corrupted slice gets rebuilt rather than silently replaced with an empty one, even across a restart.

## Resilient Daemons

A daemon is a background service that stays running and answers requests over a socket. The 026 release gave them a second way in. This release makes sure that when a connection drops, a daemon dies or a burst of traffic hits a limit, the system recovers instead of duplicating itself or corrupting its data. These protections are on for everyone, and the advisor parity takes effect on your next fresh session.

#### Advisor Launcher Parity

The skill-advisor launcher now recovers the way the memory and code services already do. A new session bridges the live daemon instead of spawning a rival, and a hung daemon is reaped and replaced under a lock. Until this release it could decide to reconnect on a dead socket and then act as if it had not, so a second session could start a second writer. That gap is closed.

&nbsp;

#### Single-Writer Lock

A single-writer lock keeps two processes from writing the same database at once. The second writer loses the lock and exits cleanly, so the shared index stays intact.

&nbsp;

#### Scriptable Command-Line Front Doors

All three daemons now expose their full tool set as scriptable command-line clients over the warm socket: 37 tools for spec-memory, 8 for code-index and 9 for skill-advisor. The normal connection stays primary and these are additive. They are read-only at prompt time, maintenance and mutation are blocked there, and advisor changes require an explicit trusted flag.

&nbsp;

#### Higher Connection Cap

Busy fan-outs now have room: the limit on simultaneous connections to a daemon rose from eight to 64, so a burst of parallel work no longer runs into the ceiling.

&nbsp;

#### Orphaned Helper Cleanup

Helper processes clean up after themselves. They exit when their input closes, when the connection drops or after a short watchdog timeout, so they no longer pile up once their parent is gone. Sixteen stranded processes were cleared in the work that landed the fix.

&nbsp;

#### Config Alignment

Daemon re-election is now on by default in the launcher itself rather than only when a config turns it on, and the four runtime configs were cleaned and brought into line, including a fix for two of them that carried a JSON error that left them unparseable.

&nbsp;

#### Advisor State Stays Out of Specs

The skill advisor keeps its working files at the repo root, out of your spec folders. When it ran from inside a packet, its workspace resolver could fall back to that packet and drop a stray `.advisor-state` directory into it. The resolver now refuses to land inside a `specs/` subtree, so the advisor writes its state where it belongs and your spec folders stay clean. A regression test pins the new behavior, and the directories that had already leaked were cleaned up.

## One Hub for the Deep Loops

The deep loops had grown into five separate skills that shared a backend and kept drifting apart. This release folds them into one without changing a single thing you type.

#### Five Skills, One Hub

The five public identities (context, research, review, ai-council and improvement) now live in one hub that sits over a shared, frozen backend. The hub holds no per-mode logic of its own. It reads which mode you want and routes to it, and each mode keeps its own behavior. Advisor routing collapsed five skill identities into one plus a mode alias, and the old five-skill surface is gone. The change is invisible where it counts: every `/deep:*` command and every agent name is unchanged, and the only thing you might notice is that diagnostics now check both of the loop's databases instead of missing one.

&nbsp;

#### A Reusable Parent-Skill Pattern

The hub-over-modes shape is now a documented convention with a tool that builds it for you. The `/create:sk-skill-parent` command scaffolds a parent skill with its mode packets, a README and the agent mirrors, the doc skill gained matching templates, and a routing-drift check keeps the registry honest against the code. So the next hub-shaped skill does not have to be hand-assembled.

## Design at the Terminal

For most of its life this framework lived inside the spec folder. Three new skills walk interface work out to the terminal, where you can script, gate and verify it like anything else. They split judgment from transport on purpose, so neither erodes the other.

#### The Design Judgment Skill

`sk-design-interface` carries the design judgment. It is a standalone, Apache-2.0-only skill that keeps the Anthropic frontend-design principles as its base, and its method is the opposite of templated defaults: ground the work in the live subject, name the real-world median and deliberately depart from it, and critique before building. Design references are read live rather than copied into the tree. This is the non-negotiable taste layer that the two transports below are required to defer to.

&nbsp;

#### The Open Design Transport

`mcp-open-design` drives the installed Open Design desktop app from the terminal through its command-line tool and a small server. Generation is a conversation rather than one shot: a run reaches a question, your answer triggers the actual build and a final step writes the file. The design-judgment skill is a hard precondition here, so the transport refuses to do interface work without it. In the same pass the `mcp-magicpath` hosted-canvas skill was removed, leaving `mcp-open-design` as the active design transport.

&nbsp;

#### The Figma Transport

`mcp-figma` is the second transport, driving Figma Desktop from the terminal for install, connect, read, author and export. Command safety is tiered: read-only commands run freely, changes ask first and destructive commands are off by default. The connection ships safe by default, using a plugin with no patch to the app, while the more invasive option is gated and paired with a rollback. An optional path supplies design context for code generation.

## Also in This Release

#### Kimi K2.7 and Shared Setup

Kimi K2.7 lands as a small-model option with a full prompt-craft profile, and a bakeoff settled on its best prompt framework. The five MCP skills now share one install-and-doctor setup: a single install guide, a verify-only install script and a read-only doctor check. The Figma skill reached its first stable release at 1.0.0.0, and cross-references tie the two design transports to the judgment skill.

&nbsp;

#### Sharper Reviews and Provenance Honesty

The `sk-code` review rules got sharper: a review that recommends removing something now has to name what replaces it, a marker calls out deliberate simplification, and a rule forbids quietly cutting scope. Operators can name the review depth directly with `SK_CODE_REVIEW_DEPTH=lite|full|ultra`, which names the existing depth tiers without changing the review contract. On the dispatch side, the executor audit now fails loud when the model that actually ran differs from the model that was requested (behind the flag `SPECKIT_PROVENANCE_CHECK=1`), so a silent model swap cannot produce a dishonest record.

&nbsp;

#### Written-Down Discipline

The operating discipline that drove this whole release is now written into the framework docs and constitutional rules rather than living in one reviewer's head: confirm a claim before you make it, capture a baseline before claiming no regressions, treat a finding as a hypothesis until you open the cited code, and name a rollback before anything irreversible. That doctrine grew out of a Fable 5 review practice, and a typed agent contract that travels with dispatched work backs it. Both are advisory and never block a valid exchange.

## Internal Seams (No User-Facing Change)

These are structural changes with no behavior change for callers, worth naming for the maintainers who will work in these files next.

- **Storage adapter.** A five-port adapter now sits over the database layer for testability and cleaner seams. No behavior changes for callers.
- **Faster graph traversal.** Two graph walks whose query defeated the index were replaced by a shared in-app traversal. Same results, fewer full scans.
- **Prose docs out of code search.** Markdown files were dropped from the default code-graph indexing, so prose docs are no longer treated as code nodes. A maintainer flag re-enables them, and search over real code is unchanged.
- **Command presentation split.** Command families had their routing separated from their presentation contracts. A structural refactor with unchanged behavior.
- **Runtime dependency seam.** The deep-loop runtime's scattered reach-ins into shared modules were pulled behind one seam, and its lock reclaim and merge steps were made atomic. The runtime's contract is unchanged.

## Upgrade Notes

No migration is required for the common path. If you set `SPECKIT_SKILL_ADVISOR_HOOK_DISABLED`, rename it to `MK_SKILL_ADVISOR_HOOK_DISABLED`. The `mcp-magicpath` skill was removed, so switch any design work to `mcp-open-design`. The advisor recovery fix takes effect on your next fresh session. The new structural protections are on by default, while the flag-gated features stay off until you enable them.

## Verification

Two review programs back this release, and they follow the same rule for shipping: structural protections that only ever make the system safer ship on for everyone, while anything that changes results or behavior ships off by default behind a named flag until it earns its way on. So the secret scrubber, the provenance guard, vector self-heal and the single-writer lock are live for everyone, and idempotency, soft-delete tombstones, the acceptance-coverage gate, completion-freshness checking, semantic trigger fallback and the learning reducers stay default-off and opt-in.

#### The 027 Review

The memory and daemon work went through a large tri-model review: 75 read-only review seats across three models swept fifteen angles and produced 113 unique findings after removing duplicates. A second adversarial pass narrowed sixteen candidate issues down to five real ones, and all five were fixed and verified, covering a stale-cache bug on delete, a save lock that only reaps a provably dead owner, a history rebuild wrapped in a transaction, a write that carries human authorship onto its successor and a startup lock that reclaims on owner death instead of waiting out a timeout.

&nbsp;

#### Per-Packet Validation

The skills-and-tooling work each shipped under its own strict spec validation. The design playbooks ran live across three small models and passed 12 of 13, with the one open scenario (WIRE-001) waiting on a fresh Code Mode session to confirm a live tool list. The Kimi prompt recommendation came out of its own bakeoff, and the three daemon command-line surfaces stayed green throughout. The vector-durability and advisor-lifecycle fixes were verified against sandboxed fixtures that never touch a live shard or a real daemon.
