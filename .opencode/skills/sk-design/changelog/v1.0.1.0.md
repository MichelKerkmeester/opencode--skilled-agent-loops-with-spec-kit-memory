# v1.0.1.0 - Routing authority and pseudocode reconciliation

**Released:** 2026-06-25

Maintenance release that reconciles the umbrella router's §2 routing authority with the design family as it actually ships, repairs defects in the illustrative router pseudocode, and aligns the family-identity surfaces. No change to the umbrella-router architecture: the parent stays a thin router over five independently invokable children.

---

## What Changed

### Routing authority points at a child that exists

- The SPEC / `DESIGN.md` intent now routes to `sk-design-md-generator`, the current shipped child, in both the §2 routing table and the `ROUTE_TO_CHILD` pseudocode. Previously it named `sk-design-spec`, which is not on disk.
- The §7 child-skills list and the §2 table now agree, and both keep a brief note that a dedicated `sk-design-spec` child may split this responsibility out later.

### Router pseudocode hygiene

- Zero-signal prompts (no design keyword matched) now return the `UNKNOWN_FALLBACK` disambiguation checklist instead of silently defaulting to the interface child. Routing Rule #2 is preserved: generic "make it look good" prompts still score and route to `sk-design-interface`.
- Removed the dead low-confidence branch that compared integer keyword weights against a `0.5` float threshold with an unreachable `or [0]` fallback.
- `LOAD_LEVELS` is now read: the happy-path result carries the per-intent load level instead of defining the map and never using it.

### Family identity and metadata

- Added a `metadata.family: sk-code` block to the SKILL.md frontmatter so the family value agrees with `graph-metadata.json`, which already used the valid `sk-code` family.
- Refreshed the graph-metadata `last_updated_at` timestamp.

### Resource-loading table

- The ALWAYS row now reflects that the shared `anti_slop_principles.md` baseline always loads, matching the `DEFAULT_RESOURCE` load in the pseudocode (the table previously said "None").

---

## Known Follow-ups

- The weighted reverse graph edges from `sk-design-interface` and `sk-design-md-generator` back to this parent are still expressed only through `manual.related_to`, not through a weighted `enhances`/`siblings` edge. Adding those reciprocal edges belongs to the child skills and is left for a family-wide metadata pass.
