9. **Success metrics should be contract-level and behavior-level.** Contract metrics: 100% of invalid schema fixtures fail with precise reasons; 100% of valid ledger fixtures pass; all existing post-dispatch, graph-aware-stop, coverage-graph-signals, prompt-pack, reducer-schema, and contract-parity tests still pass. Operator metric: a maintainer can audit why a PASS is clean from structured rows alone, without rereading every iteration narrative.

## Questions Answered

- Which verification layer should ship first? Post-dispatch schema validation, because it is the earliest reliable place to reject shallow iteration records before reducer, graph, dashboard, or synthesis can normalize them.
- What exact tests prove the candidate ledger is real? Add post-dispatch invalid/valid fixture tests, prompt-pack render assertions, reducer fixture assertions, graph candidate-coverage stop blockers, and a sandboxed manual CP-058 seeded-bug scenario.
- How should reducer fixtures change? Add a candidate-ledger fixture that persists ruled-out candidates, linked findings, search debt, and candidate-coverage blocked stops through registry and dashboard output.
- How should graph tests change? Add review node kinds for bug class, invariant, producer, consumer, and test; add convergence blockers for missing candidate coverage; keep existing unresolved-P0 and uncovered-dimension blockers green.
- What makes this less surface-level in measurable terms? A PASS or clean dimension must carry auditable evidence of searched bug classes, invariants, producer/consumer paths, and negative tests, not only a severity count and dimension label.

## Questions Remaining

- Should the minimum candidate-row count be fixed per dimension, scaled by target size, or derived from targetSelection risk categories?
- Should `candidateMatrix` and `searchLedger` be synonyms during migration, or should one canonical field be chosen immediately?
- Should graph expansion ship in the same packet as schema enforcement, or follow once the text/JSON ledger has stabilized?
- What is the exact legacy policy: warn for old packets, enforce for new `reviewDepthSchemaVersion >= 1`, or enforce only when `reviewTargetType` is `spec-folder`, `skill`, `agent`, `track`, or `files`?
