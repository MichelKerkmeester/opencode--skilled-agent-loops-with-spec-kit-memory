# Iteration 15: W5 adversarial: vector, FTS, and call-site claims

## Focus

Adversarial re-verification of seven CHANGED/REFUTED verdicts from iterations 10–11 covering (a) call sites of the Turso compat not-implemented surface across the three skills' production TypeScript, (b) the state of native vector indexing in vendored Turso v0.7.0-pre.6, (c) dense f32 blob layout and cosine-distance semantics vs sqlite-vec, (d) JS-SDK enablement of the `index_method` experimental flag, (e) FTS index-time weights and the NoMergePolicy, and (f) two specific call-site dependencies in `vector-index-store.ts` and `reindex.ts`. All evidence was re-located independently from core Rust implementation, bindings source, binding tests, and our production TypeScript — not from COMPAT.md.

## Findings

- **CLAIM-1** — KNOWN, **CONFIRMED**. Fresh grep for `\.backup\(|\.serialize\(|\.function\(|\.aggregate\(|\.table\(|loadExtension\(|\.iterate\(|\.raw\(|\.pluck\(` across `system-spec-kit/mcp_server/lib`, `system-code-graph/mcp_server/lib`, `system-skill-advisor/mcp_server/lib` (production `*.ts`, excluding tests/dist) returned **zero hits**. A sanity grep proves the search works and re-locates the known vector-layer exception, which manifests as `sqliteVec.load(...)` (the npm wrapper around extension loading), never as a literal `loadExtension(`: [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:430], [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/db-shard-migration.ts:242], [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:1090,1571]. All four sites are inside the vector/embedder layer slated for replacement. Caveat: patterns required a leading dot (better-sqlite3 method-call shape); bare-identifier forms would not have matched, but none plausibly exist.
- **CLAIM-2** — KNOWN, **CONFIRMED**. `core/index_method/mod.rs` registers exactly three index-method modules: `backing_btree`, `fts`, `toy_vector_sparse_ivf` [SOURCE: external/turso-main/core/index_method/mod.rs:14-17]. The only vector index method documents itself as "Simple inverted index for sparse vectors", created via `USING toy_vector_sparse_ivf`, handling **jaccard** similarity over sparse-encoded vectors only [SOURCE: external/turso-main/core/index_method/toy_vector_sparse_ivf.rs:29-36]. Dense cosine search has no index path — `vector_distance_cos` is a plain scalar function [SOURCE: external/turso-main/core/vector/mod.rs:122-133], so `ORDER BY vector_distance_cos(...) LIMIT k` is a full linear scan at 0.7.0-pre.6.
- **CLAIM-3** — KNOWN, **CONFIRMED**. `vector_serialize` emits Float32Dense as `Value::from_blob(x.bin_eject())` with **no header and no trailing type byte** — unlike Float64Dense (trailing `2`), Float32Sparse (dims + trailing `9`), Float1Bit (trailing `3`), Float8 (trailing `4`) [SOURCE: external/turso-main/core/vector/operations/serialize.rs:6-51]. The f32 bin data is a raw little-endian concatenation: `floats.iter().flat_map(|f| f.to_le_bytes()).collect()` [SOURCE: external/turso-main/core/vector/vector_types.rs:520], with a `bin_len == dims * 4` invariant [SOURCE: external/turso-main/core/vector/vector_types.rs:639]. That is byte-identical to sqlite-vec float32 blobs. Cosine: Float32Dense routes to simsimd `f32::cosine` (cosine *distance*) [SOURCE: external/turso-main/core/vector/operations/distance_cos.rs:23-26,85-87]; the in-repo Float8 path makes the formula explicit, `1.0 - dot / (norm1 * norm2).sqrt()` [SOURCE: external/turso-main/core/vector/operations/distance_cos.rs:77], and a test asserts opposite vectors yield distance 2.0 with the comment "Normal cosine distance [0, 2]" [SOURCE: external/turso-main/core/vector/vector_types.rs:698-723]. So `vector_distance_cos = 1 − cosine_similarity`, matching sqlite-vec's `vec_distance_cosine` ascending-order semantics.
- **CLAIM-4** — KNOWN, **CONFIRMED**. The NAPI connect options struct exposes `pub experimental: Option<Vec<String>>` [SOURCE: external/turso-main/bindings/javascript/src/lib.rs:179] and maps `"index_method" => core_opts.with_index_method(true)` [SOURCE: external/turso-main/bindings/javascript/src/lib.rs:273-280]. The TS type includes it: `ExperimentalFeature = 'views' | … | 'index_method' | …` [SOURCE: external/turso-main/bindings/javascript/packages/common/types.ts:1,21], and a binding test exercises exactly the claimed form: `connect(":memory:", { experimental: ["index_method"] })` [SOURCE: external/turso-main/bindings/javascript/packages/native/promise.test.ts:246]. JS-SDK enableable, not CLI-only.
- **CLAIM-5** — KNOWN, **NUANCED** (weights + NoMergePolicy confirmed; per-segment-stats mechanism not directly read). Weights are declared in the CREATE INDEX `WITH` clause — doc comment `WITH (tokenizer='default',weights='col1=1.0,col2=2.0')` [SOURCE: external/turso-main/core/index_method/fts.rs:1335], parsed at attachment creation from `cfg.parameters.get("weights")` via `parse_field_weights` [SOURCE: external/turso-main/core/index_method/fts.rs:1377-1386,1226-1281] and stored as fixed `field_weights` applied as per-field boosts [SOURCE: external/turso-main/core/index_method/fts.rs:1483,1816] — i.e., fixed at index definition, not per-query like FTS5 `bm25(fts, w1, w2)`. `NoMergePolicy` is confirmed: imported [SOURCE: external/turso-main/core/index_method/fts.rs:31] and set on the writer, `writer.set_merge_policy(Box::new(NoMergePolicy))` [SOURCE: external/turso-main/core/index_method/fts.rs:2956], so segments never merge. The final link — BM25 statistics being computed per-segment and therefore drifting as unmerged segments accumulate — follows from the tantivy-style segment architecture but I did not read the scorer code itself this pass; treat that sub-clause as well-supported inference, not re-verified fact.
- **CLAIM-6** — KNOWN, **CONFIRMED** (it is the fallback, not the primary path). The code at the cited location does exactly what was claimed: `SELECT sql FROM sqlite_master WHERE name='vec_memories'` followed by `vec_sql?.match(/FLOAT\[(\d+)\]/i)` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:258-273]. That regex only matches sqlite-vec vec0 virtual-table DDL (`embedding FLOAT[768]`), so it breaks under a native-table design with different column DDL. Nuance: it is the *secondary* discovery source — the primary reads `vec_metadata` key `'embedding_dim'` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:241-256] and survives migration if the metadata table is carried over.
- **CLAIM-7** — PARTIAL, **CONFIRMED with corrected line attribution**. The staging-shard reindex flow does depend on a TRUNCATE checkpoint before the atomic rename, but the pragma itself lives at line 454, not inside 511–590: `try { shard.pragma('wal_checkpoint(TRUNCATE)'); } catch (_err) { /* best-effort */ }` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:454]. The 511–590 range contains the staging + swap flow whose comment explicitly states the dependency — stale `-wal`/`-shm` sidecars beside the active shard are unlinked because the renamed pair is "already TRUNCATE-checkpointed at staging close" [SOURCE: reindex.ts:511-518,579-589], ending in `fs.renameSync(stagingShardPath, activeShardPath)` [SOURCE: reindex.ts:589]. The checkpoint is wrapped in best-effort try/catch, so a Turso port must preserve equivalent truncate-checkpoint semantics (cross-reference the W5-seat-13 `wal_checkpoint` findings) or the swap can rename a shard whose WAL still holds unapplied frames.
