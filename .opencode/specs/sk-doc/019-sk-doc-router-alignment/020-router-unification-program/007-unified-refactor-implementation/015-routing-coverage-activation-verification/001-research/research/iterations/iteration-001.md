# Iteration 1 — gpt-5.6-sol high — focus: catalogs

Headline finding: update three existing catalog packages, not all 24. Ownership belongs to the routed hub, the advisor response surface, and flag governance. The remaining catalogs describe child modes, transports, or unrelated systems and should not duplicate parent-hub routing.

Verified claims:

- `sk-design/feature-catalog/` is the only existing catalog that explicitly inventories an eligible hub and its mode-routing shell (`.opencode/skills/sk-design/feature-catalog/feature-catalog.md:15`, `.opencode/skills/sk-design/feature-catalog/feature-catalog.md:25`). Add a root entry under `## 2. MANAGER SHELL` named “Compiled routing, legacy fallback, and kill switch,” linking to a new `manager-shell/compiled-routing-and-legacy-fallback.md`.
- That new leaf should follow the four required template sections: overview, behavior, source files, and metadata (`.opencode/skills/sk-doc/create-feature-catalog/assets/feature-catalog-template.md:193`, `.opencode/skills/sk-doc/create-feature-catalog/assets/feature-catalog-template.md:201`, `.opencode/skills/sk-doc/create-feature-catalog/assets/feature-catalog-template.md:219`, `.opencode/skills/sk-doc/create-feature-catalog/assets/feature-catalog-template.md:237`). Its `HOW IT WORKS` content should cover eligibility/default polarity, `route|clarify|defer|reject`, fail-safe legacy fallback, serving-status observability, drift handling, and the `=0` kill switch.
- The Skill Advisor catalog already owns `advisor_recommend`; compiled routing is an optional field of that existing feature, not a separate MCP feature. The public schema defines `compiledRoute` on each recommendation (`.opencode/skills/system-skill-advisor/mcp-server/schemas/advisor-tool-schemas.ts:221`), while the existing leaf does not mention it in its response description (`.opencode/skills/system-skill-advisor/feature-catalog/mcp-surface/advisor-recommend.md:22`). Extend that leaf’s frontmatter triggers, `HOW IT WORKS`, implementation table, and validation table; do not add a 43rd root entry.
- The exact advisor text must state that enrichment is gated, omission is the safe fallback, and the field is additive: the handler returns unchanged output when disabled (`.opencode/skills/system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:362`) and merges `compiledRoute` without replacing the recommendation (`.opencode/skills/system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:371`).
- The Spec Kit catalog already has a canonical “Feature flag governance” entry and leaf (`.opencode/skills/system-spec-kit/feature-catalog/feature-catalog.md:4480`, `.opencode/skills/system-spec-kit/feature-catalog/governance/feature-flag-governance.md:25`). Extend those rather than creating another flag feature. Document `SPECKIT_COMPILED_ROUTING`, its current/default state, staged rollout, explicit kill switch, eligibility rule, drift response, and serving-status readout. Because the touched leaf currently uses an unstructured source list (`.opencode/skills/system-spec-kit/feature-catalog/governance/feature-flag-governance.md:49`), convert it to the template’s separate implementation and validation tables.
- Do not update child-mode catalogs. Compiled routing selects the hub mode before packet loading (`.opencode/skills/system-deep-loop/SKILL.md:38`, `.opencode/skills/system-deep-loop/SKILL.md:40`), while per-mode catalogs belong to the selected packets (`.opencode/skills/system-deep-loop/SKILL.md:94`). For example, the deep-research catalog explicitly inventories loop lifecycle and convergence, not parent-hub routing (`.opencode/skills/system-deep-loop/deep-research/feature-catalog/feature-catalog.md:9`). The same boundary applies to transport catalogs such as mcp-chrome-devtools, whose catalog owns its CLI/MCP selection (`.opencode/skills/mcp-tooling/mcp-chrome-devtools/feature-catalog/feature-catalog.md:15`).
- Catalog wording must follow rollout phase. The template requires shipped current reality, not planned behavior (`.opencode/skills/sk-doc/create-feature-catalog/assets/feature-catalog-template.md:30`), and explicitly requires a flag-gated compatibility layer to be labelled (`.opencode/skills/sk-doc/create-feature-catalog/references/common-pitfalls.md:35`). P0–P3 text must say default-OFF; P4 must change the same passages to default-ON with `=0` as the kill switch.
- A census hazard was previously unnamed: the mcp-click-up tree uses uppercase `FEATURE-CATALOG.md` (`.opencode/skills/mcp-tooling/mcp-click-up/feature-catalog/FEATURE-CATALOG.md:2`), contrary to the lowercase canonical filename (`.opencode/skills/sk-doc/create-feature-catalog/assets/feature-catalog-template.md:41`). Any automation globbing only `feature-catalog.md` sees 23 roots and silently misses the twenty-fourth tree.
- Durable catalog source tables must not preserve the current resolver’s spec-tree path. The front door currently imports from that mutable location (`.opencode/bin/compiled-route.cjs:16`), while the catalog template explicitly requires packet-history-free stable anchors (`.opencode/skills/sk-doc/create-feature-catalog/assets/feature-catalog-template.md:33`). Author the leaves only after the resolver is promoted, or point them to the stable promoted path during the same change.

Ranked recommendations:

1. P0: encode the three-owner matrix in the implementation spec: new sk-design leaf, existing advisor leaf, existing spec-kit governance leaf. Explicitly exclude the other 21 catalog trees.
2. P0: make catalog publication phase-coupled—default-OFF wording through P3, then an atomic P4 update to default-ON plus the kill switch.
3. P1: require every leaf’s source tables to anchor the stable front door/resolver, eligibility manifest, drift verifier, automated parity tests, and matching manual-playbook scenario.
4. P1: update `advisor-recommend.md` in place; adding a separate compiled-router advisor leaf would incorrectly model an optional response field as another public tool.
5. P2: make catalog discovery accept or explicitly report case-variant roots, then migrate mcp-click-up separately rather than folding unrelated cleanup into the router cutover.
