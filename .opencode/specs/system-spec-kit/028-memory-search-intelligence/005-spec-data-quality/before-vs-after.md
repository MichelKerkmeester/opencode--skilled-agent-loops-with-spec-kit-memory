# Spec-Kit Data Quality by Default: The Planned Program and the Shipped Build (028 Track C)

> Track C, the `005-spec-data-quality` parent, is the seventh track under packet 028. It opened as research only, ran an official five-lineage deep research across thirty-seven substantive iterations on opus seeded by a Stage 0 external sweep of about thirty cited sources across seven angles, and turned the converged result into a tiered, dependency-ordered program of twenty-eight Level-2 phases. Every one of those twenty-eight carries a full Level-2 doc set marked PLANNED, no code landed for them and their completion sits at zero. What that program became is not a delivered feature but a costed, fidelity-aware build plan that a future packet inherits, where every candidate already knows whether it runs at the write surface before any truncation applies or has to prove a gain on the prod retrieval path first. The lineage then continued past that scaffold and shipped. A second wave of children, 029 through 040, built and graduated the generated-metadata quality program described in Section 3, and a third wave, 041 through 044, landed the search-quality fix wave and the evidence-gap redesign described in Section 4, so the track is a planned program at its head and a delivered build at its tail.

---

## THE ORGANIZING PRINCIPLE

The track holds the same discipline as the rest of 028. Ship for correctness and reversibility, not for an unproven win, and let the truncation law decide where the weight goes. The prod default route does not cap a query at three results. It guarantees a never-cut-below-three minimum, returns the longer tail up to a twenty-wide window and trims that tail only at a relevance cliff and under a running token budget, so the token budget and not the number three is the real prod limiter. A candidate that runs at the write surface ships on cost because it executes before any truncation applies. A candidate whose value depends on the served result window stays gated until a prod-mode measurement proves the tail it adds is actually served to the reader. That single split, write-time is cheap and retrieval must be proven, decides which of the twenty-eight phases is buildable now and which waits.

## 1. THE WRITE-SURFACE PROGRAM AND ITS INFRASTRUCTURE

This is the half of the program whose value lands before any truncation applies, the on-write checks, the retroactive automation, the novel floor-bypassing capabilities and the shared engine and governance layer that order them.

**Before**

The authored spec docs and the two metadata JSONs had no automated quality program of their own. A live default-ON quality loop already ran, a pure quality scorer and a non-mutating post-save reviewer with an auto-fix path, but it fired only on the memory-save path and never reached the authored write surface, so a spec doc or a metadata file got none of the gates a saved memory got. The two metadata JSON schemas validated through hand-rolled checks at warn severity while the real zod schemas sat dormant, and a never-triggered legacy-grandfathered bypass quietly weakened strict mode. Trigger phrases lived in one place and were not propagated into description.json, field values were not enum-constrained so a typo passed, provenance and freshness signals were computed but hidden, and there was no shared safe-fix path so any future front door would hand-roll its own notion of a safe fix. Nothing organized this scattered surface into one program.

**After**

The research produced a tiered, dependency-ordered slate of write-surface phases, all PLANNED. The keystone is A1 at phase 001, which wires the existing pure scorer and the non-mutating reviewer to the authored spec-doc and metadata-JSON write surface and adds a CONTENT_QUALITY rule to validate.sh, reusing the live machinery rather than the destructive quality loop. The one measured GO is A4 at phase 004, which promotes the DESCRIPTION_SHAPE and GRAPH_METADATA_SHAPE rules from warn to error against the real zod schemas and deletes the dead legacy-grandfathered bypass. A4 is unconditional as a decision but flip-gated as an action, because the warn-to-error flip is the last beat of a four-beat migration and only fires once the live corpus re-measures to zero schema failures, and that count is not zero today, a real run of the target graph-metadata schema fails sixteen to twenty-four files, sixteen excluding archives and twenty-four including, not the parse-only subcount of eleven an earlier census reported. The rest of Tier A is scoped on cost rather than a measured win, trigger propagation into description.json at 002, enum-constraint of the two schemas at 003, trigger-coherence assertion between spec.md and description.json at 005, HVR-style autofix at 006, EARS constraints and requirement-coverage at 007, surfaced provenance fields at 008, content-hash integrity at 009 and per-surface gates at 010. Tier B is retroactive automation across 011 through 013, a scheduled data-quality sweep with guarded auto-fix, a `/doctor` data-quality route and a retrieval-feedback edge. A separate novel slate across 019 through 025 carries seven floor-bypassing capabilities kept from twelve judged, cross-doc contradiction detection, embedding-drift monitoring, example and test generation, a context-budget assembler, a typed-relation knowledge graph, a freshness-decay queue and per-doc quality SLAs. Two infra phases anchor the build, the shared safe-fix engine at 026 that is the one engine, one detector registry and one frozen fixClass allow-list the five front doors mount on rather than each diverging on what counts as a safe fix, and the governance and rollout layer at 028 that orders the build through a seventeen-stage sequence, codifies a four-beat migration discipline and two structural safety invariants and freezes an eighteen-item NO-GO list with its ten anti-patterns. The feature flags the program names, an authored-surface quality-loop flag, a scheduled-sweep flag, an EARS-lint flag, a retrieval-gap-detect flag and the C-tier retrieval flags, are proposed and not built.

**Impact**

A future build packet inherits a costed write-surface program rather than a pile of ad-hoc fixes. The cheap real wins run before any truncation applies, so they ship on cost the moment the engine they depend on lands. The buildable-now subset is four phases, the shared safe-fix engine at 026, the A1 keystone at 001, the A4 schema gate at 004 and the A3 enum-constraint at 003, each carrying its own evidence and none waiting on a prod retrieval read. The one measured GO is isolated with its undercount caught before any flip, the rest of the slate is sequenced behind its named engine or gate dependency, and the seven novel capabilities are recorded as design rather than promoted, most of them thin and unproven and deferred until a measurement supports them.

**Why**

A write-time candidate runs before any truncation applies, so it ships on cost and never has to clear a retrieval benchmark. That is why Tier A, Tier B, the novel slate and the infra phases are scoped on cost and reuse-first, every one of them leaning on machinery that already ships rather than standing up a second quality system, which the NO-GO list names as the central anti-pattern next to swapping the database and adding cryptographic signing. A4 is the lone phase that earns the word measured, and even it ships as a validation tightening rather than a ranking move, with its error flip held behind a re-measure-to-zero backfill so the live undercount is repaired before the gate hardens. The shared engine ships before A1, B1 and B2 because those three reuse it as their fix path, and the governance layer ships the ordering, the migration discipline and the NO-GO freeze so no later phase reinvents a safe-fix policy or jumps an inviolable edge.

## 2. THE GATED RETRIEVAL SLATE AND THE FLOOR EXPERIMENT

This is the half whose value depends on the served result window, the Tier-C retrieval candidates, the prod-mode gate that fences them and the floor experiment the whole constraint rests on.

**Before**

The truncation law was a measurement finding and nothing more. Section 7 of the root had named it and Section 6 had priced flags against it, but no standing program organized a whole slate of retrieval-side data-quality work around the same eval-versus-prod fidelity discipline. The dual-mode run-eval-v2 harness reported an eval-versus-prod fidelity delta but ran no baseline comparison and shipped only single-target goldens that saturated, so there was no prod-window recall gate at all and every retrieval-side candidate stayed an unprovable hypothesis. Embedded chunks lost their header path and global identity, metadata sat beside the text instead of inside the embedding, and no instrument could tell an eval-mode gain apart from a gain the prod path would actually serve.

**After**

The retrieval work is fenced as Tier C across phases 014 through 018, every item default-off and marked CONDITIONAL, chunk-prefixing at 014, the prod-mode recall gate at 015, answerable-questions tags at 016, metadata fusion at 017 and an LLM-judge scorer at 018. The unblocker is C2 at phase 015, the prod-mode recall gate every other Tier-C candidate waits behind, and its scope was widened during the round-2 remediation from a completeRecall@3-only read to the completeRecall@3, @5 and @8 columns the harness already emits plus an order-sensitive NDCG@K companion with a top1 guard, so a promotion now needs both a recall rise and a ranking-quality hold and reads only the prod lens, never the eval column. The floor experiment at phase 027 is the operator-agreed measurement the whole frozen Tier-C constraint rests on, a sweep that raises the result budget beyond the current floor of three and reports whether results four through ten are signal or noise on this corpus. It is proposed and not yet run, and it changes no prod default, it measures the tail and returns a verdict. No Tier-C candidate can promote until the 015 gate goes green, and the gate has not been run.

**Impact**

The retrieval slate is not abandoned and it is not forced through, it is fenced behind a measurement that has to land first. A future packet that widens the served prod window or moves the floor inherits five specified retrieval lanes and a prod-mode gate that will tell it honestly whether each one transfers, rather than an eval-mode number the reader never sees. The measurability instrument is sequenced before every consumer, exactly as the gate-zero coverage guard is the precondition for trusting any recall number elsewhere in 028.

**Why**

A retrieval feature earns a keep only by proving a gain on the prod path the reader actually sees, never on an eval-mode number the prod truncation cuts, the same discipline the new-feature arc in Section 7 of the root held for its tail-additive seams. The eval path skips truncation while the prod path applies it, so the external brief's recall@K numbers do not survive the prod fidelity gap and external validation alone cannot promote a retrieval candidate. That is why the C2 gate ships first as the precondition for trusting any Tier-C number, why the floor experiment is the named unblock condition for the truncation-law constraint rather than a default change, and why every Tier-C phase keeps its scaffold but stays unbuildable until that prod-mode read confirms the tail it adds is served.

## 3. THE GENERATED-METADATA BUILD THAT SHIPPED

This is the half of the lineage that left the scaffold and landed, children 029 through 040, the generated-metadata quality build, the full-repo migration that earned its flag flips and the benchmark that graduated them.

**Before**

Every `description.json` and `graph-metadata.json` in the tree was written under the old contract. Status was free prose rather than an enum, paths were stored unprefixed, stamps were wall-clock so a re-derive always looked dirty, and the two schemas validated at warn while the real zod schemas sat dormant. There was no shared identity resolver so a scoped re-derive could erase a parent or drop children, no idempotent-write guard so a save rewrote files whose only delta was the stamp, no freshness key so nothing could tell a stale derived field from a fresh one, and no fingerprint so the integrity rule had nothing to check. The staging tree z_future could seed generated JSON. The search verdict scored a lone high-cosine off-corpus hit as good because banding read raw relevance with no grounding floor, no noise-floor subtraction and no evidence-gap bridge, and the verdict and render switches that would close those holes existed only as default-off code awaiting a measured before-and-after.

**After**

The generated-metadata build landed across children 033 through 038, an identity resolver with merge safety, the scoped-backfill z-exclusion, idempotent description writes with a targeted cache upsert, the metadata validator and status enum, the drift gate with a shared synopsis extractor and the generator hardening that persists a source fingerprint. Child 039 then ran the full-repo migration, a scoped per-folder driver that restamped every `description.json` and `graph-metadata.json` in the whole tree to the new enum-clean, prefixed-path and content-hashed format, z_archive included and z_future excluded by operator decision, gated on a byte-stable second run and a verify pass that read 2049 folders at zero violations. That restamp earned the migration-gated flips, because the reason those switches shipped off is that the live files still spoke the old contract the new rule rejects. Child 040 then ran the graduation benchmark, the same earn-or-delete reckoning the original 028 flags faced, on the thirteen built switches. Twelve graduated to default-ON or enforcing on a measured effect, the identity, idempotent, drift-gate, hardening, lexical-grounding, noise-floor, cite-with-caveat, evidence-gap and envelope-fidelity flags plus the grandfather flip to a hard error, the z-exclusion that ships on cost and the false-confirm ceiling set to zero. One switch, grounding-signal, set its display field correctly but moved no retrieval metric and was deleted with its code. Children 029 through 032 frame the wave, a vague-query model benchmark that found the quality and citation verdict model-robust, two research phases that drove the build and the small change that makes z_future always ignored.

**Impact**

The tree now carries clean generated metadata that the integrity rule enforces rather than narrates, enum-valid status, prefixed paths, preserved lineage, content-hashed freshness keys and a source fingerprint on every folder. The search verdict no longer scores an off-corpus term as good, the false-confirm rate that the lexical-grounding and noise-floor flags target fell from 0.833 to 0 and now holds enforced. Archived spec content stays discoverable and deprioritized rather than excluded, and the staging tree stays out of the generated cache. The original twenty-eight research phases are unchanged by this wave, they remain the forward plan, but the data-quality lineage they belong to has now shipped its first concrete generation.

**Why**

The build held the same discipline as the rest of 028. Each behavioral switch dark-launched off and reversible, the migration restamped the tree before any enforcing flag flipped, and every flip waited on a measured before-and-after rather than an assumption. The migration earned the flips because a switch that enforces a new shape cannot flip while the live files still carry the old one, and the benchmark graduated them because a switch that only changes output without moving a real metric does not earn default-ON. Grounding-signal failed that bar and was removed, so the tree carries no switch a future reader has to reason about without a verdict behind it.

## 4. THE SEARCH-QUALITY FIX WAVE AND THE EVIDENCE-GAP REDESIGN

This is the search-quality arc that followed the metadata build, children 041 through 044, the six fixes the 029 deep-research surfaced and the three benchmark-gated decisions that chased the deepest of them to a relevance-aware redesign.

**Before**

The evidence-gap cap was dead code on the live path. The `SPECKIT_EVIDENCE_GAP_VERDICT` cap had graduated to default-ON but never fired, because the search handler set the evidence-gap warning that drives the banner yet never set the boolean the verdict cap reads, so an off-corpus query rendered `good` beside a gap banner, the banner and the verdict disagreeing. Underneath that wiring gap the evidence-gap detector keyed off a Z-score that measures distributional peakedness, the top score's distance from the pack, rather than relevance, so it rewarded a lonely weak off-corpus top and punished a rich strong cluster. The `citeCorrect` benchmark metric was binary and read about 0.55 where a three-tier reading reads 1.0, the dashboard left graph and degree rows with a blank score, the envelope carried no honest `retrievalProfileWeightsEnabled` field, ranking carried wall-clock recency inputs with no reproducibility switch, and the presentation contract did not assert the displayed count against the rows shown.

**After**

`041-search-quality-fixes` landed all six. The keystone bridged the Stage-4 `evidenceGapDetected` signal into the verdict path, so the cap fires live and an off-corpus query caps `good` to `weak` with the banner and verdict in agreement. The other five made the `citeCorrect` metric three-tier-aware, added the honest `retrievalProfileWeightsEnabled` envelope field, resolved a `score` on graph and degree rows, added a default-off `SPECKIT_DETERMINISTIC_RANKING` flag with an always-on trigger id tie-break, and tightened the presentation contract to the `shown of total` form. Three benchmark-gated decisions followed. `042-deterministic-ranking-benchmark` held the determinism flag default-off, since flag-ON ranking is reproducible at determinism 1.0 but removing wall-clock recency diverges from the default on 5 of 12 queries (top-K overlap 0.69, Kendall tau 0.73) and recency is load-bearing with no ground-truth that the deterministic order is better. Per the operator directive to remove benchmark-rejected stay-off flag code, `SPECKIT_DETERMINISTIC_RANKING` and its gated branches were then deleted, reverting to the default recency-on ranking, with the always-on trigger id tie-break kept as the pure-win part and the behavior identical to the prior default. `043-gap-threshold-calibration-benchmark` found the gap detector mis-designed not mis-tuned, the Z-score peakedness statistic over-caps strong queries (should-good false-positive 0.67) and misses flat off-corpus ones (should-gap detection 0.33) and no threshold separates them (optimal 0.8 reaching only 0.50). `044-relevance-aware-evidence-gap` then redesigned the gap relevance-aware behind a `SPECKIT_RELEVANCE_AWARE_GAP` flag, keying the decision off the verdict banding's noise-floor-subtracted relevance, `gapDetected = max(0, topRelevance - noiseFloor) < LOW_THRESHOLD`, which dropped the should-good false-positive rate from 0.67 to 0.00 and raised agreement with the verdict gap band from 0.61 to 1.00, firing only on the one query the verdict rates gap. The first graduation attempt almost shipped a regression, the relevance branch was fed `resolveEffectiveScore` (RRF-magnitude around 0.03) while its band expects absolute cosine, so it flagged every query with the evidence-gap banner, a bug the original benchmark had masked by feeding hand-picked absolute-relevance scores and that a full-handler live dispatch caught. It was fixed by threading the verdict banding signal `resolveCalibrationScore` into the detector through a `relevanceScores` option, fail-closed to the Z-score path when absent, re-benchmarked through the production `executePipeline` (graph and kubernetes return no-gap, only oauth returns gap) and confirmed by a full-handler live re-verify, then graduated to default-on with zero test collateral and 161 gap and pipeline tests passing.

**Impact**

The gap signal now means what it says. The banner and the verdict can never disagree again, because both read one shared relevance definition, and the cap is driven by relevance rather than peakedness, so a rich strong cluster is no longer falsely capped and a flat weak cluster is. The off-corpus over-confidence the 029 benchmark caught, `good` beside a gap banner, is resolved at the source and proven by the fast-subset rerun at 6 of 6 off-corpus cells capped and 0 `good`-beside-banner contradictions where the prior failure mode was 19 of 144. The `citeCorrect` metric reads honestly at 1.0 across all four models, the dashboard shows graph and degree scores rather than blanks, and the determinism flag is gone, its code deleted on a measured no-harm failure so production keeps recency-on with only the trigger id tie-break retained. The relevance-aware gap is graduated default-on, keying off the banding absolute relevance and firing only on a genuine gap, after a production-scoring fix that a live re-verify caught and validated.

**Why**

The arc held the same discipline as the rest of the track. The keystone is the smallest change that fixes the real bug, a bridge of an existing signal rather than a threshold change, so it activates the already-correct cap without touching gap detection. The determinism flag failed the no-harm test, recency doing real work on the queries with competing matches, so it earned no keep and its code was deleted rather than left dormant, the pure-win trigger id tie-break kept because it needs no flag. The gap redesign reuses the verdict banding's relevance instead of a new metric, which is what makes the verdict agreement reach 1.00 by construction rather than by coincidence, and its graduation went through a production-scoring fix, the detector was fed RRF-magnitude scores rather than the absolute cosine its band expects, caught by a live dispatch the benchmark had masked and fixed by threading `resolveCalibrationScore` into the detector before the flip to default-on.

## THE REVIEW AND THE CURRENT STATE

Track C did not stop at scaffolding. Before any code landed it went through its own round-2 deep review and a scoped remediation. The review returned a conditional verdict. The remediation corrected the truncation-law framing from cap to floor, widened the C2 gate from a completeRecall@3-only read to the @3, @5 and @8 columns plus the order-sensitive companion, re-sized the A4 census from the parse-only eleven to the real sixteen-to-twenty-four and reframed the buildable-now subset to the honest single-digit count of four. It also reconciled a completion-bookkeeping fork, where some docs claimed full completion while others still read five percent and pointed at starting the research loop, and corrected a handful of scaffold inaccuracies, a plan that named an export that already existed, a seam that had to cover both metadata files rather than one and a reuse that crossed a code boundary. Every correction held the program research-only.

The current state splits in two. The original twenty-eight research phases stay scaffolded as the operator's record of intent, none deleted, every one marked PLANNED at completion zero, a ready-to-build forward plan a later packet inherits. The shipped lineage spans children 029 through 044. The generated-metadata build, the full-repo migration that restamped the whole tree at 2049 folders and zero violations and the graduation benchmark that kept twelve flags and deleted one landed first, then the search-quality fix wave bridged the dead evidence-gap cap live and the two benchmarks deleted the determinism flag on a measured no-harm failure and named the gap detector mis-designed, and the relevance-aware redesign graduated default-on after a production-scoring fix a live re-verify caught. All of it is delivered and committed, and every phase across the track now carries a changelog. So Track C is no longer a research-only track. Its head is a planned program and its tail is a delivered build, and the line between them is the migration that earned the flips.
