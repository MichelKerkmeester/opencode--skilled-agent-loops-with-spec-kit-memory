deep-agent-improvement is now a genuine two-lane skill end to end. Lane A (improve an agent file) and Lane B (benchmark a model or prompt framework) are co-equal at the command, doc, agent, and on-disk layers, where Lane B used to be a model-benchmark mode bolted onto an agent-improvement skill.

## What Changed

#### Command layer asks the lane (phase 008)

- **`/deep:start-agent-improvement-loop` now resolves a use-case lane.** When an agent path is supplied it auto-resolves to Lane A and runs exactly as before (CMD-1 behavioral identity), so existing invocations are untouched. A `--lane=model-benchmark` flag or a `--profile` argument routes to Lane B, and a lane question appears only when the lane is genuinely ambiguous.
- **New dedicated `/deep:start-model-benchmark-loop` command** plus its gemini mirror and two Lane B workflow YAMLs (`deep_start-model-benchmark-loop_{auto,confirm}.yaml`) that run `loop-host.cjs --mode=model-benchmark`, reduce, journal, and offer mode-aware promotion.
- **Advisor registration:** a `deep-model-benchmark` alias group plus bounded command-level disambiguation so benchmark phrasing routes to the new command rather than the Lane A one.
- **Latent fix:** `loop-host.cjs` never forwarded `--scorer` and `--grader` to `run-benchmark.cjs`, so selecting `5dim` or a `mock`/`llm` grader through any command path silently fell back to `pattern`/`noop` while the report disagreed with the journal. Both flags are now forwarded in the model-benchmark branch only, with a regression test. The agent-improvement branch is byte-identical so TST-1 stays green.

#### SKILL.md two co-equal lanes (phase 009)

- **The skill doc presents Lane A and Lane B as parallel sections** instead of modes 1 to 3 plus a trailing Mode 4. A WHEN TO USE two-lane table was added, and the smart router gained a `MODEL_BENCHMARK` intent at weight 5 with a matching RESOURCE_MAP entry. DQI stayed in the excellent band at 97.

#### references and assets lane-separated (phase 010)

- **`references/` and `assets/` are now organized by lane** (`agent-improvement/`, `model-benchmark/`, `shared/`) rather than by function. This supersedes the v1.8.0.0 function grouping (scoring, promotion-gates, integration, workflow), which was an intermediate step. Every literal was repointed: SKILL.md RESOURCE_MAP, DEFAULT_RESOURCE, RUNTIME_ASSETS, both lanes of command YAMLs, both command docs, README, graph-metadata, scripts, the feature catalog and playbook, and the benchmark profile's internal fixtureDir.

#### agent lane awareness (phase 011)

- **The `@deep-agent-improvement` awareness note went from "Mode awareness" to "Lane awareness"** across all 4 runtime mirrors, byte-identical (mirror-drift 0). The note makes clear there is one agent and Lane B dispatches models through `dispatch-model.cjs`, never a second Claude agent.

#### catalog, playbook, and dashboard visibility (phase 012)

- **The feature catalog and manual testing playbook label each entry by lane.**
- **F-P2-5 closed:** `reduce-state.cjs` now surfaces the `mode` metadata it already recorded. The registry carries per-profile and global `modes`, and the dashboard prints a "Lane (mode) mix" line in the global summary and each profile section. This is display only, routing is untouched, and a missing mode counts as agent-improvement so legacy ledgers still total. A new regression test covers it.

#### scripts lane-separated (phase 013)

- **The 16 scripts moved into `scripts/{agent-improvement,model-benchmark,shared}/`** (with `scorer/` under model-benchmark), while `lib/` and `tests/` stay at the scripts root. `loop-host.planInvocation()` was kept byte-identical and a new `resolveScriptPath()` helper does lane resolution at spawn time, so TST-1 holds. `dispatch-model.cjs` path joins were re-depthed (its config load is positively asserted to resolve to a real file, closing a silent-fallback risk). All references were rewritten, including the 13 vitest path literals.

## Upgrade

No migration required for callers of `/deep:start-agent-improvement-loop` with an agent path, which behave exactly as before. New work that benchmarks a model or prompt framework should use `/deep:start-model-benchmark-loop` or pass `--lane=model-benchmark`. Internal scripts moved into lane subdirs, so any external caller hardcoding `scripts/<name>.cjs` should update to `scripts/<lane>/<name>.cjs`.
