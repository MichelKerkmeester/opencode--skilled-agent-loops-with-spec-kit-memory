# Iteration 025 — Track D (deepseek)

## Focus
Confidence/grading as anti-fabrication signal (per-section evidence grade, per-token coverage confidence, measured a11y grade)

## Findings
1. **[P0] cluster.ts:752,911 — per-token confidence from coverage-election** — ColorToken.confidence and TypographyLevel.confidence already computed as 'high'|'medium'|'low' from frequency thresholds (≤2=low, ≥5=high) and page-coverage ratio (≥0.5=high). This is pure coverage-election: a color found on 1 page 2 times is low-confidence; a color found on 80% of pages 500+ times is high. The field exists in the schema (types.ts:424,439) and is populated at extraction time, but the DESIGN.md writer does not currently surface it. Fidelity mechanism: deterministic, runs on element-frequency histograms + crawled-page intersection, no ML, no guesswork.
   - Recommendation: Plumb to DESIGN.md writer (Phase 2): for each token claim, emit confidence level. Strong claims ("The primary brand color is...") only from high-confidence tokens. Low-confidence tokens → stamp "ABSENT — insufficient evidence" instead of inventing context. File: extract.ts (already computes) → report-gen.ts or the WRITE phase prompt to consume.
   - Evidence: cluster.ts:752,911 (confidence assignment); types.ts:424,439 (schema)
2. **[P0] cluster.ts:400-565 — stability classification (L1-L4) with float confidence** — Every token (color, typography, shadow, radius, component) receives a stability classification: infrastructure(L1)→system(L2)→campaign(L3)→content(L4), each with a 0.0-1.0 confidence computed from multi-signal scoring (page coverage, usage dimensions, CSS-var presence, frequency, chromaticity). L4 tokens are hard-excluded from DESIGN.md; L3 tokens require "Subject to change" annotation. The validator enforces this via checkStabilityGating() (validate.ts:100-123) — any L4 hex appearing in DESIGN.md is a critical failure regardless of score.
   - Recommendation: Already active as anti-fabrication gate in validate.ts. Extend: surface the stability confidence float per token in the report-gen.ts HTML report so a human reviewer can audit borderline L2/L3 cases. File: cluster.ts (classifier), validate.ts:100-123 (gating enforcement).
   - Evidence: cluster.ts:400-565 (stability classifiers); validate.ts:100-123,372-376 (CRITICAL_FAILURE_TYPES gate)
3. **[P0] validate.ts:57-93,318-349 — graded validator with critical-failure gating** — The validator already produces a graded score (0-100: -5 per failure, -1 per warning) AND enforces CRITICAL_FAILURE_TYPES (phantom-color, content-color, missing-section) that hard-fail regardless of score. A single fabricated hex triggers FAIL even if all other checks pass. This is already an evidence-completeness validator, not purely binary: it distinguishes between format issues (warnings: -1) and fabrication (failures: -5, gate-failing). The score degradations per-error type create a natural severity scale.
   - Recommendation: Add per-section evidence-completeness checks: for each DESIGN.md section, compare the writer's token claims against the backing tokens.json and flag sections that make claims about dimensions with ZERO backing data (e.g., §7 Content & Voice, §8 Do's and Don'ts, §13 Agent Prompt Guide have no token backing — any claim there is unverified assertion). This would add an evidence_level dimension to the existing pass/fail/warning system. File: validate.ts (add checkEvidenceCompleteness()).
   - Evidence: validate.ts:318-349 (validateDesignMd); validate.ts:371-376 (critical failure gate)
4. **[P1] proof.ts:75-100 — fidelity pixel-sampling coverage** — The proof module captures a live screenshot, samples 2,000 pixels from CSS-rendered areas (excluding img/video/svg/background-image regions), converts each to OKLCH, and computes delta-E perceptual distance against the extracted palette. Pixels within deltaE<12 count as matched. This produces a single coverage percentage (e.g., 94.2%) — a measured answer to "how much of the visual surface does our palette explain?" Low coverage signals the extractor missed tokens or the site has extensive image-driven color.
   - Recommendation: Augment the proof output to break down coverage per DESIGN.md dimension (color palette coverage vs. typography coverage vs. shadow accuracy) rather than a single aggregate. This would produce a per-section evidence-completeness report card. File: proof.ts (add per-dimension scoring).
   - Evidence: proof.ts:75-100 (scoreColorCoverage); proof.ts:58-64 (deltaE)
5. **[P1] a11y-extract.ts:257-357 — measured WCAG data, no aggregate grade** — The a11y extractor computes real WCAG contrast ratios from DOM color pairs (not templated), checks AA/AAA per pair, measures min touch-target sizes, detects focus-indicator consistency, and checks reduced-motion support. This is measured, transcribable data — every number traces to a real DOM element. However, no aggregate a11y grade is computed. The data sits in tokens.json a11yTokens but §9 of DESIGN.md gets raw data, not a synthesized grade.
   - Recommendation: Compute an aggregate a11y score from the existing metrics: contrast-pass-rate × focus-consistency × touch-target compliance. Place in §9 as a measured grade (e.g., "A11y Grade: B — 38/44 pairs pass AA, focus indicators consistent, min touch target 44px"). This is transcribable, not inventable. File: a11y-extract.ts (add synthesizeGrade()).
   - Evidence: a11y-extract.ts:257-357 (extractA11y); types.ts:270-300 (A11yTokens schema)
6. **[P2] design-boundary-detect.ts:228-311 — cross-site similarity scores per dimension** — The boundary detector computes similarity scores per dimension (font, color, spacing, radius, component, shadow) between site groups. These are inherently confidence signals: a site with a single page scores 100 on everything (no variance = low evidence of system-ness). Multi-domain extractions produce real similarity scores that reveal where a design system is consistent vs. fragmented — the thin dimensions signal invention-prone areas.
   - Recommendation: Pipe dimension scores into the DESIGN.md as a system-coherence annotation. A low similarity score on a dimension (e.g., shadow=15%) warns the writer that no coherent shadow system exists and to stamp ABSENT rather than invent one. File: design-boundary-detect.ts (already computes), feed to report-gen.ts or WRITE phase prompt.
   - Evidence: design-boundary-detect.ts:228-311 (detectBoundaries)

## Summary
The sk-design-md-generator already has multiple grading/confidence mechanisms — per-token confidence (cluster.ts:752), 4-layer stability classification (cluster.ts:400-565), graded validation with critical gating (validate.ts:371), and pixel-level fidelity proof (proof.ts). The highest-value mechanism is per-token confidence from coverage-election: it is already computed, deterministic, and covers every color and typography token. The gap is plumbing — the DESIGN.md writer (Phase 2) does not surface these grades, and the validator (Phase 3) does not produce per-section evidence-completeness scores. The fix needs no new computation, only wiring existing cluster.ts confidence/stability outputs into the writer prompt and validator checks.
