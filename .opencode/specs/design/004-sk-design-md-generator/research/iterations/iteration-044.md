1. **[P0] findInteraction key mismatch — fix verified** — SOUND-WITH-CAVEAT. Verified: interactionLookup builds key as `${tag}|${classes}|${componentType}` (cluster.ts:1288) using classifyComponent result (e.g. 'link' for <a>, 'button' for <div role=button>). findInteraction constructs `${el.tag}|${el.className}|${el.tag}` (cluster.ts:1294) — the third slot uses raw el.tag, never 'link'/'button'. For <a>: lookup='a|cls|link', query='a|cls|a' → MISS. For <div role=button>: lookup='div|cls|button', query='div|cls|div' → MISS. Confirmed: ElementStyle.role (dom-collector.ts:194) and DiscoveredElement.role (interaction-capture.ts:100) both use el.getAttribute('role')||'' — identical source. classifyComponent('a','') returns 'link' (interaction-capture.ts:120), so a function classifyByTagRole(tag,role) matching classifyComponent's logic would work. CAVEAT: both key slots use full class strings, but representative's className (dom-collector.ts:193, getAttribute('class')) may differ from captured element's classes (interaction-capture.ts:88, el.className property) due to inline class mutations between discovery and clustering. The linear-scan fallback (cluster.ts:1298) checks el.className.includes(cap.element.classes) — this fails when representative has fewer classes (common: 'btn-primary' doesn't include 'btn-primary large'). Proposed robust fix: key both lookup AND query by `${tag}|${firstClassPrefix}|${classifyComponent(tag,role)}`, using only the first class token (the prefix used for group dedup). This avoids full-class-string mismatch.
   - Recommendation: Use first-class-prefix in key: extract prefix = className.split(/\s+/)[0] at both build and lookup sites. Regression test: extract a site with <a> tags that have :hover CSS — verify DESIGN.md §Components shows hoverChanges/hoverDiff for link components instead of null. Test also <div role='button'> elements. Test that a <button class='btn-primary'> and <button class='btn-secondary'> (same prefix, different variant) each get their correct interaction data.
   - Evidence: cluster.ts:1288,1294,1297-1301; interaction-capture.ts:118-126; dom-collector.ts:193-194; iteration-019.md:10-12

3. **[P0] deltaE clustering threshold — safety verified** — SOUND-WITH-CAVEAT. Confirmed: current deltaE<3 (cluster.ts:708) in ×100-scaled OKLCH corresponds to unscaled ΔE<0.03 — ~30× tighter than JND. Brand-blue pair #3B82F6 (L≈0.66,C≈0.22,h≈260) → #60A5FA (L≈0.73,C≈0.18,h≈260): dl≈7, dc≈4, dh≈0 → deltaE≈√(49+16+0)=8.06. At threshold 3 this pair does NOT merge, splitting brand scale into separate tokens with diluted frequency/pagesCoverage (iter-005 confirms). Raising to deltaE<8 would merge this pair. CAVEAT: without a usage-dimension gate, deltaE<8 could merge functionally different colors. Example: primary-blue bg #0066CC and accent-teal text #0077AA could have deltaE<8 if their chroma/hue differ while lightness is similar. The greedy single-pass (cluster.ts:703-724) has no iterative re-centering, amplifying the risk. SAFE PATH: deltaE<8 AND colors must share ≥1 usage dimension (usedAs overlap on bg/text/border). This prevents a bg-only system blue from absorbing a text-only accent teal. Alternative: raise threshold conservatively to 5 (not 8-12) and add a post-merge de-dilute step that recomputes pagesCoverage from union of pre-cluster page sets.
   - Recommendation: Raise threshold to deltaE<8 only when paired with usage-dimension gate: only merge if colors share ≥1 usage dimension. Without the gate, use deltaE<5 as a safer intermediate. Regression test: extract a site with a brand palette (e.g. Tailwind UI demo) — verify that light/dark scale variants of the same brand color are merged into ONE token, and that tokens with the same frequency but different roles (bg vs text accent) remain separate. Add a golden-test JSON file with known-color pairs and expected merge/split outcomes.
   - Evidence: cluster.ts:348-354,700-724,741; iteration-005.md:7-8,11-12; iteration-032.md:13-14

(b) Brand accent on hero button only (1 page, freq=15, bg+border=2 dims): pages<2 PASSES, freq≥10 FAILS, dims≥2 FAILS → NOT excluded. This gate is strictly additive — it removes entries from the clustering pool and cannot affect already-clustered tokens. It does not modify classification thresholds or scoring weights.
