This release rebinds deep-research onto the new `deep-loop-runtime` peer skill (arc 118, FULL_ISOLATE_NO_MCP). The 13 lib files plus the 4 graph operations (convergence, upsert, query, status) previously hosted inside `system-spec-kit/mcp_server/` now live in `deep-loop-runtime/`. Deep-research workflow YAMLs invoke `.cjs` scripts directly instead of the removed `mcp__mk_spec_memory__deep_loop_graph_*` MCP tools.

> Spec folder: `.opencode/specs/skilled-agent-orchestration/116-deep-skill-evolution/` (Level 3, phased — 8 children)

---

## Why This Release Matters

Before arc 118, deep-research's workflow YAML dispatched graph operations through `mcp__mk_spec_memory__deep_loop_graph_*` tools registered with the spec-memory MCP server. That meant two consumer skills (`deep-research`, `deep-review`) depended on internals of a third skill (`system-spec-kit/mcp_server/`), and every workflow step that needed graph state went through an MCP marshalling layer. The arrangement was accumulated drift; no ADR justified it.

Arc 118 (user-directive override of the 117 AI Council SPLIT ruling) consolidated all deep-loop runtime infrastructure into a new peer skill (`deep-loop-runtime`), deleted the 4 MCP tools, and switched the workflow YAML to direct `.cjs` script invocation — the same canonical pattern `reduce-state.cjs` already used. Operators tracing a deep-research bug now stay inside `deep-loop-runtime/` + `deep-research/` instead of traversing into a different skill's MCP server source tree.

## What Changed for Deep-Research

#### Workflow YAML invocations switched to scripts

Both `deep_research_auto.yaml` and `deep_research_confirm.yaml` were updated. Every graph operation goes through a script:

```yaml
# Before (deleted in arc 118):
mcp_tool:
  name: mcp__mk_spec_memory__deep_loop_graph_convergence
  parameters: { specFolder: "{spec_folder}", loopType: "research", sessionId: "{session_id}" }

# After (canonical):
bash: 'node .opencode/skills/deep-loop-runtime/scripts/convergence.cjs --spec-folder "{spec_folder}" --loop-type "research" --session-id "{session_id}"'
outputs:
  - graph_decision
  - graph_signals_json
  - graph_blockers_json
  - graph_blockers_csv
  - graph_stop_blocked
```

Output bindings stay byte-equivalent — the scripts emit the same JSON-on-stdout shape the removed MCP tools used.

#### Dependency rebinding

Any deep-research-adjacent code that previously imported from `system-spec-kit/mcp_server/lib/deep-loop/*` or `system-spec-kit/mcp_server/lib/coverage-graph/*` now imports from `deep-loop-runtime/lib/{deep-loop,coverage-graph}/*`. The SQLite database (`deep-loop-graph.sqlite`) moved with the schema-owner code and now lives at `.opencode/skills/deep-loop-runtime/storage/`.

#### Test infrastructure

The `mcp_server/vitest.config.ts` include glob extended to discover tests at `../deep-loop-runtime/tests/**/*.{vitest,test}.ts`, so a single `pnpm vitest run` exercises both surfaces. The only test that stayed in `mcp_server/tests/deep-loop/` is `review-depth-reducer.vitest.ts`, which tests `deep-review/scripts/reduce-state.cjs` (lives in `deep-review/`, not the migrated runtime).

#### Version frontmatter reconciled

The deep-research `SKILL.md` frontmatter version was stale at `v1.6.2.0` while the changelog history already extended to `v1.11.0.0`. This release bumps the frontmatter to `v1.12.0.0` so the SKILL.md and changelog history are aligned.

## Upgrade

No deep-research consumer changes required beyond what arc 118 shipped. Operators directly running `node deep-loop-runtime/scripts/<X>.cjs` should pass `--spec-folder`, `--loop-type`, `--session-id`, plus any optional flags documented in `.opencode/skills/deep-loop-runtime/references/script_interface_contract.md`. The four `mcp__mk_spec_memory__deep_loop_graph_*` tools are gone and have no aliases; any external code that hardcoded those tool names must switch to the script paths.
