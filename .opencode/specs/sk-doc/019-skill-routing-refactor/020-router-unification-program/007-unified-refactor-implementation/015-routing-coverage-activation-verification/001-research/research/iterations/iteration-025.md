# Iteration 25 — gpt-5.6-sol high — focus: closeout

The synthesis is not implementation-ready. Beyond F-14 through F-24, eight unresolved decisions remain. The largest blocker is ownership: the decision packet explicitly permits documentation only, while its task list assigns runtime, CI, environment, and hub changes to the same packet.

Verified residuals:

- The P0–P4 runtime cutover has no valid implementation scope. Phase 012 limits its change set to packet documentation and explicitly excludes runtime files, skills, and manifests (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/spec.md:109-117`), yet its tasks require modifying the verifier, CI, eligibility resolver, runtime defaults, and seven hubs (`.../012-default-on-decision/tasks.md:84-93`). The synthesis must create or designate implementation children before work starts.
- The resolver decision contradicts its handoff. The accepted decision is to promote the resolver beside the runtime shim (`.../012-default-on-decision/decision-record.md:264`, `:287-289`), but the implementation summary still reopens “promote or approve residual coupling” (`.../012-default-on-decision/implementation-summary.md:160`, `:170`). Promotion should be treated as settled.
- Non-hub policy is unresolved. Phase 012 explicitly pins only `sk-git` legacy (`.../012-default-on-decision/spec.md:97`), while all four non-hub candidates remain non-serving or legacy-authoritative: `001-sk-git/implementation-summary.md:126`, `002-system-code-graph/implementation-summary.md:36`, `003-system-skill-advisor/implementation-summary.md:46`, and `004-system-spec-kit/implementation-summary.md:119`. Meanwhile create-skill plans to reject compiled onboarding for standalone skills (`.../013-create-skill-alignment/checklist.md:68`). The other three candidates therefore lack a durable disposition.
- The status schema is internally incomplete. The decision contract exposes only `compiled-serving | legacy-fallback | drifted` (`.../012-default-on-decision/spec.md:133`), while the benchmark contract requires a fresh-manifest resolver failure to classify as `broken` (`.../014-benchmark-alignment/spec.md:103`). The shared interface needs either a fourth state or separate authority and cause fields.
- Invalid environment values have no specified post-cutover meaning. Verification asks for unset, `0`, `1`, and invalid-value tests (`.../012-default-on-decision/checklist.md:76`), but the contract specifies only explicit enablement and the `=0` kill-switch (`.../012-default-on-decision/tasks.md:84`). The synthesis must define a strict truth table and diagnostic behavior.
- The P2 canary remains unassigned: neither its environment profile nor promotion owner has been selected (`.../012-default-on-decision/checklist.md:66`; `.../012-default-on-decision/implementation-summary.md:169`).
- The required GPT-5.6 LUNA HIGH acceptance has no implementing workstream. Phase 014 expressly excludes real models and network calls (`.../014-benchmark-alignment/spec.md:85`) and describes itself as deterministic Mode A only (`.../014-benchmark-alignment/implementation-summary.md:99`). Existing LUNA shortcomings therefore cannot be closed by Lane C.
- Drift failure ownership remains undecided. Phase 014 leaves open whether Lane C or the separate drift-CI job owns the non-zero exit (`.../014-benchmark-alignment/implementation-summary.md:98-109`). Until settled, drift could be double-gated or merely reported by both.

No implementation claim is currently verified: phase 012 says no runtime, eligibility, CI, canary, or directive has changed (`.../012-default-on-decision/checklist.md:238`); phase 013 reports 0/17 P0 checks verified (`.../013-create-skill-alignment/checklist.md:135`); and phase 014 lists every implementation verification as Planned (`.../014-benchmark-alignment/implementation-summary.md:80-88`).

Ranked actions:

1. Create scoped implementation children for P0 through P4, with explicit owners for runtime, CI, canary, content coverage, and cutover.
2. Freeze the shared contracts first: stable resolver location, eligibility/discovery API, four-state-or-two-axis status schema, and complete environment-value truth table.
3. Decide the four non-hub candidates’ fate explicitly: permanently legacy with an archetype discriminator, or onboarded through a separate non-hub serving front door.
4. Select the P2 canary profile, owner, observation window, promotion thresholds, and rollback trigger.
5. Keep deterministic Lane C separate from a mandatory live GPT-5.6 LUNA HIGH acceptance stage that records model, variant, serving authority, manifest identity, and all-scenario verdicts.
6. Assign exactly one non-zero drift gate, then make other consumers report the same result without independently redefining exit policy.
7. Carry the prior catalog, playbook, archiving, hook-consumption, launcher, template, and `RESOURCE_MAP` findings into named implementation workstreams; none may be treated as implicitly covered by phases 012–014.
