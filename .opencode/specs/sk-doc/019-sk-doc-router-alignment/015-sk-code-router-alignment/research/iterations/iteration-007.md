## Findings

1. **The benchmark's existing stage split is necessary but not sufficient for anti-gaming.** The loader can represent `routing`, `holdout`, and `negative`, and aggregation excludes holdouts from the fitted headline while reporting fitted-minus-holdout gap. However, sk-code's index-table loader currently assigns every non-negative scenario to `routing`; the committed report therefore has 23 fitted rows, zero holdouts, and a null generalization gap. A valid optimization trial must freeze independently authored holdouts before candidate output exists; output-derived holdouts or thresholds selected after seeing results are invalid. 3. **The gates must run as a fail-closed sequence, not as one weighted score:** freeze independent fixtures/digests and thresholds; capture a same-revision baseline; run fitted, holdout, and negative candidate replay; enforce D3 semantics; enforce typed-contract and topology/D5 gates; enforce exact 18/18 surface equality; then run a fixed live holdout sample. Any hard-gate failure rejects the candidate even if fitted aggregate rises. This follows the existing distinction between weighted rows and structural/registry verdict blockers. [SOURCE: .opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/run-skill-benchmark.cjs:214-255] [SOURCE: .opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/score-skill-benchmark.cjs:1633-1661] [INFERENCE: anti-gaming properties are conjunctive invariants and cannot safely be averaged away]

4. **Contract-only and router-changing candidates need different invariants.** An identity-only typed-contract remediation may change manifest digest, unresolved counts, typed eligibility/recall, error class, and verdict, but must preserve raw selected resources, flat expected-leaf hits, D1/D2, D3, and all 18 surface outputs exactly. A router optimization must instead demonstrate new independently authored leaf hits in both fitted and holdout partitions while passing every negative, D3, topology, typed, live, and surface invariant. [SOURCE: .opencode/specs/sk-doc/019-sk-doc-router-alignment/015-sk-code-router-alignment/research/iterations/iteration-005.md:7-14] [SOURCE: .opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/score-skill-benchmark.cjs:1202-1257] [INFERENCE: separating identity projection from raw selection prevents a contract cleanup from being advertised as a routing-quality gain]

5. **The current artifacts cannot yet pass this matrix.** The baseline has no holdout rows, live scoring treats model-stated resources as primary while unordered reads are only corroboration, and current D3 has no gold-completeness discriminator. Therefore no implementation should claim generalization or actual leaf-load improvement until independent holdouts, explicit completeness/budgets, and ordered live provenance are present; the matrix is complete as a specification, but evidence for a candidate remains intentionally absent. ## Ruled Out

- Accepting fitted-score improvement without a non-empty sealed holdout partition; the current null holdout cannot prove generalization. [SOURCE: .opencode/skills/sk-code/benchmark/baseline/skill-benchmark-report.json:13-20]
- Selecting thresholds after seeing candidate output; this turns acceptance criteria into fixture fitting. [INFERENCE: a post-hoc threshold is conditional on the candidate result and therefore is not independent evidence]
- Treating generated report JSON, manifest edits, or typed-error removal as raw recall gain; those artifacts can change without selecting a missing expected leaf. [SOURCE: .opencode/specs/sk-doc/019-sk-doc-router-alignment/015-sk-code-router-alignment/research/iterations/iteration-006.md:15-17]
- Using current `raw.observedReads` as actual ordered-load evidence; it is a deduplicated extraction from tool inputs without causal provenance or order metrics. [SOURCE: .opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/live-executor.cjs:279-327]

## Dead Ends

- Aggregate-only acceptance is definitively invalid: conjunctive surface, topology, suppression, and contract invariants can be averaged away.
- A holdout created or edited after candidate inspection is contaminated and must be regenerated independently rather than relabeled.

## Reflection

- What worked and why: Following the loader, scorer, runner, live parser, and current report in sequence exposed which anti-gaming properties already have executable fields and which require new preregistered evidence.
- What did not work and why: The current corpus cannot instantiate the holdout gate, and current live telemetry cannot establish ordered actual-load metrics; both are evidence absences rather than reasons to weaken the matrix.
- What I would do differently: In a candidate-evaluation iteration, capture fixture, manifest, router-source, model/settings, and 18-case-vector digests before the baseline run so every acceptance input is mechanically immutable.
