# Iteration 010 — Track B (deepseek)

## Focus
Feed visual/structural sections (gradient decomposition, breakpoint key-changes, section coverage report)

## Findings
1. **[P0] B6: Raw gradient strings — no decomposition** — Confirmed. GradientInfo (types.ts:124-130) stores raw linear-gradient(...) strings only: {type, value, elementTag, elementClasses, rect}. DesignTokens.gradients (types.ts:394-399) preserves the same flat {type, value, elementTag, location} shape. cluster.ts:1449-1483 aggregates gradients but never decomposes value into angle/type/stops/role — location is heuristically inferred from rect position (hero/button/card/decorative) but isDecorative is not modeled. extractGradientColors (cluster.ts:244-261) only regex-extracts color substrings from the raw value for the color-map, never parses stop positions. No GradientToken shape with decomposed stops[{color,position}] exists anywhere in the codebase.
   - Recommendation: Add GradientToken interface to types.ts: {type:'linear'|'radial'|'conic', angle?:string, stops:{color:string,position:string}[], usageLocation:string, isDecorative:boolean}. Add parseGradient() in cluster.ts to regex-parse linear-gradient(angle, color pos, color pos...) into decomposed stops. Feed the stops array to DESIGN.md §6 so depth claims are evidence-backed and the validator can bind any 'gradient depth' narrative to real stop data.
   - Evidence: types.ts:124-130, types.ts:394-399, cluster.ts:1449-1483, cluster.ts:244-261
2. **[P1] B8: Breakpoints lack per-breakpoint key changes** — Confirmed. MediaBreakpoint (types.ts:153-158) has only {query, type, value, ruleCount}. No per-breakpoint layout-diff data exists. Extraction runs single-viewport DOM collection at 1440px (extract.ts:298) — all CSS analysis and DOM element capture happen at one width. The crawl produces multi-viewport screenshots (types.ts:20 — 1920/1440/768/375/320) proving Playwright can resize, but DOM collection never runs at alternate widths. cluster.ts:1487-1504 deduplicates breakpoints by type:value key summing ruleCount only. To populate per-breakpoint keyChanges, multi-viewport DOM re-extraction at each breakpoint boundary width is architecturally required but not implemented.
   - Recommendation: Two-tier fix: (a) add optional multi-viewport DOM re-extraction — for each unique breakpoint boundary, resize viewport and re-collect DOM, then diff element positions/display/font-size/grid-cols against baseline 1440px, populating keyChanges:{property:string, from:string, to:string}[] on MediaBreakpoint. (b) If multi-viewport extraction is too heavy for default runs, add a keyChangesAbsent:true flag so the writer knows not to invent specific breakpoint behaviors — §10 stays descriptive rather than fabricating diffs.
   - Evidence: types.ts:153-158, extract.ts:298, extract.ts:471-484, cluster.ts:1487-1504, types.ts:20
3. **[P0] B15: No token→section coverage map exists** — Confirmed. None of report-gen.ts, preview-gen.ts, or proof.ts produce a section-by-section token coverage map. report-gen.ts:627-676 generates an HTML dashboard with palette/typo/validation/proof but no per-section token-gap analysis. preview-gen.ts:249-254 renders token swatches in HTML, never maps to DESIGN.md sections. proof.ts:301-443 evaluates color palette pixel coverage (ΔE < 12 threshold) — this is fidelity measurement, not section coverage. validate.ts:241-285 (checkSectionCompleteness) checks whether 15-17 required section headers exist in the markdown text — it validates markdown structure, not whether those sections have backing token data. validate.ts:57-93 (checkPhantomColors) detects hex colors in DESIGN.md not found in tokens.colorTokens — a color-level check, not section-level. The exact invention sites (sections whose backing token set is empty) are currently undetectable by any tool.
   - Recommendation: Design a SectionCoverageReport with shape: {sections:{number,title,tokenField,tokenCount,isEmpty,inventionRisk:none|low|medium|high}[], emptySections:string[], summary:{totalSections,backedSections,emptySections,thinSections}}. Implement as a new check (checkSectionCoverage) in validate.ts — it already reads both DESIGN.md and tokens.json. Map each of the 17 DESIGN.md sections to its DesignTokens fields (§2→colorTokens+colorRelationships, §3→typographyLevels+fontInfo, §6→shadowTokens+gradients, §6.5→motionSystem, §10→breakpoints, §11→components[*].variants[*].hoverChanges, §12→iconSystem, §2.5→darkMode, etc.). Any section with an empty/null backing token set is flagged HIGH invention risk. This is a near-zero-cost mechanical hallucination detector that catches 100% of empty-section fabrication sites.
   - Evidence: report-gen.ts:627-676, preview-gen.ts:249-254, proof.ts:301-443, validate.ts:241-285, validate.ts:57-93

## Questions Answered
- B6: Is gradient decomposition possible from raw strings? Yes — the raw linear-gradient(...) string contains angle and color-stop data that a regex parser can extract. Current code only regex-extracts colors (extractGradientColors in cluster.ts:244-261), leaving angle and stop positions unused.
- B8: Can extraction diff computed layout across viewports to populate breakpoint keyChanges? Architecturally feasible (Playwright already resizes for screenshots at 5 widths), but single-viewport DOM collection (extract.ts:298) means no baseline exists for diffing. Multi-viewport re-extraction or explicit ABSENT flag needed.
- B15: Does any existing tool produce a token→section coverage map? No — report-gen, preview-gen, proof, and validate each serve different purposes (HTML dashboard, token preview, fidelity proof, hex/section-header validation) and none maps DESIGN.md sections to their backing token set cardinality.
