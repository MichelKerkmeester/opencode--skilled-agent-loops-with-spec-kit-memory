## Findings [file:line]

### Q3 — Fan-out recovery: durable state, classification, retry budget
1. There IS a per-branch durable status ledger, but it is observability-only, not retry state. `fanout-run.cjs:376` opens `orchestration-status.log`; the pool emits `started` / `completed` / `failed` JSONL events with `duration_ms` via `appendStatusLedger` (`fanout-pool.cjs:81-117,235-238`), and a final `orchestration-summary.json` records `total/succeeded/failed/all_failed` (`fanout-pool.cjs:207-220`, `fanout-run.cjs:495-502`). [SOURCE: fanout-run.cjs:376,495-502; fanout-pool.cjs:81-117,207-220,235-238]
2. There is NO retry and NO transient/fatal classification anywhere in the pool. `settleItem` catches a worker rejection and records a single terminal `rejected` result (`fanout-pool.cjs:100-118`); `runCappedPool` dispatches each item exactly once (`nextIndex += 1` with no requeue, `fanout-pool.cjs:174-194`). A lineage that throws is failed permanently for that run. [SOURCE: fanout-pool.cjs:100-118,174-194]
3. The worker collapses every non-zero / timed-out CLI exit into one undifferentiated `throw` — a 4-hour timeout (`computeLineageTimeoutMs` cap, `fanout-run.cjs:164-168`), a rate-limit 429, and a genuine fatal config error all become the same `Error` with `{exitCode, timedOut}` (`fanout-run.cjs:472-489`). No code reads exit semantics to decide "retryable vs terminal." `timedOut` (`signal === 'SIGTERM'`, `:473`) is the only distinguishing flag and it is not used to gate a retry. [SOURCE: fanout-run.cjs:164-168,472-489]
4. Recovery today is the post-hoc file salvage from iteration 1 only: after the (single) run, `runSalvageSweep` reconstructs missing iteration `.md` files from captured stdout (`fanout-run.cjs:468-470`). It cannot re-execute a branch; it only rescues text a weak CLI printed but failed to write. [SOURCE: fanout-run.cjs:462-470; iter-001 finding 11]
5. Q3 GAP (inference): when a lineage fails transiently (timeout / rate-limit), the only recovery is to re-run the whole fan-out invocation — every sibling lineage re-executes from scratch because the pool has no notion of "this branch is `failed`, re-dispatch just this branch from durable state with a bounded attempt count." The ledger has the data (per-label `failed` events) but nothing consumes it as resumable state. [INFERENCE: Findings 1-4 — ledger is write-only telemetry, no read-back-to-requeue path]

### aionforge Q3 mapping
6. aionforge classifies pass failures and reads the attempt count from a DURABLE audit trail, not memory: `PassError::Transient` leaves the episode `raw` to retry as oldest-pending; `Fatal` or transient-exceeding-`max_retries` (default 5) marks `failed`, retained + audited; the attempt count comes from the `consolidation_failed` audit rows "so a crash does not hand a failing episode a fresh retry budget on every restart." [SOURCE: consolidation.md:54,61-68 (via iter-001 finding 12)]

### Q4 — Cost ceiling + backpressure
7. Cost guards are advisory count caps, not an enforced ceiling. `DEFAULT_COUNCIL_COST_GUARDS` is `{max_rounds_per_topic:3, max_topics_per_session:5, seats_per_round:3, saturation_threshold:0.2}` (`cost-guards.cjs:15-20`). `evaluateCouncilCostGuards` only RETURNS `{continue_allowed, stop_reasons, upper_bound}` (`:114-140`) — it never aborts, kills, or throws; honoring the stop is left entirely to the caller's loop. There is no hard kill-switch. [SOURCE: cost-guards.cjs:15-20,114-140]
8. The only signals are static counts + a verdict-delta saturation check (`:122-133`). There is NO live throughput/latency/backlog gauge: no lag-since-last-progress, no in-flight/pending count, no failure-rate gauge. `computeCouncilCostUpperBound` (`:83-90`) pre-computes a worst-case `max_rounds`/`max_seat_outputs` budget but nothing meters actual consumption against it at runtime. [SOURCE: cost-guards.cjs:83-90,114-133]
9. The fan-out layer DOES already capture the raw material for backpressure gauges — per-lineage `duration_ms` and `started/completed/failed` counts in `orchestration-status.log` (`fanout-pool.cjs:90,108,235-238`) — but it is summarized after the run, never sampled live to trip a ceiling or warn on lag. [SOURCE: fanout-pool.cjs:90,108,207-220,235-238]
10. Q4 GAP (inference): the budget model is "count to a cap and hope the caller stops," with no runtime backpressure. A fan-out where K lineages each silently run to the 4-hour timeout cap consumes the full budget with zero mid-flight warning, because nothing meters elapsed lag/pending against a ceiling. [INFERENCE: Findings 7-9]

### aionforge Q4 mapping
11. aionforge per-tick observability emits live backlog-age gauges `consolidation_lag_seconds`, `consolidation_episodes_pending`, `consolidation_episodes_failed` plus per-tick supersession/contradiction/quarantine/summary counters; when the oldest pending episode's `ingested_at` exceeds `lag_ceiling` (default 5s) the scheduler WARNS — observable backpressure with a tripwire, exactly what the count-only guards lack. [SOURCE: consolidation.md:192-201]

### Q5 — Cross-iteration continuity
12. `prompt-pack.ts` is a pure checked-placeholder renderer: `renderPromptPack` reads a template and substitutes `{var}` tokens from a `variables` map, throwing `PromptPackError` on any missing token (`prompt-pack.ts:55-73`); `extractTemplateTokens` / `validatePromptPackTemplate` just diff expected vs present tokens (`:29-40,82-96`). It has ZERO awareness of iterations, prior findings, or open questions — it cannot thread anything; continuity is entirely whatever the caller stuffs into `variables`. [SOURCE: prompt-pack.ts:29-40,55-73,82-96]
13. The only callers of `renderPromptPack` are the module itself and its vitest (`grep -rn renderPromptPack` → 2 files). So in the shipped runtime, cross-iteration continuity is NOT carried by the prompt pack; it is carried by the reducer rewriting strategy.md machine-owned anchors (`key-questions`, `answered-questions`, `next-focus`, etc., per iter-001 finding 16) which the next iteration re-reads. Continuity is "reducer folds findings into strategy.md → next iteration reads strategy.md cold," not a threaded reflection turn. [SOURCE: grep callers = {prompt-pack.ts, tests/unit/prompt-pack.vitest.ts}; iter-001 finding 16 reducer anchors]
14. galadriel ambient reflection is the threading model the deep loop lacks: a silent, side-effect-only turn (`_send_agent_silent`, README:520-521) that takes stock of "an open question worth keeping… a fact that has changed" and files it to durable memory (README:514-518); the explicit next-stage trajectory is "reflection that reads its own recent diary + open-questions and *threads* across ticks, so a thought begun at 11:00 can be picked up at 14:00 rather than starting fresh each time" (README:536-538). [SOURCE: galadriel README:514-524,536-538]
15. Q5 ASSESSMENT (inference): deep-loop iterations DO thread forward, but only the reducer's distilled strategy.md anchors — they do not cheaply thread the iteration's OWN raw open-questions/findings as a self-curated reflection; each iteration effectively reconstructs its working set from the strategy snapshot rather than carrying its prior internal thread. This is the "reactive, reconstruct cold" mode galadriel's threading explicitly improves on. The reducer is the cheap continuity spine already present; the galadriel-style add is a per-iteration "open-questions carried forward" block the iteration writes for itself, distinct from the reducer's machine-owned fold. [INFERENCE: Findings 12-14]

## Questions Answered
- **Q3 — ANSWERED.** Durable per-branch state exists as the `orchestration-status.log` ledger (`fanout-run.cjs:376`, `fanout-pool.cjs:235-238`) BUT it is write-only telemetry: the pool dispatches each lineage exactly once (`fanout-pool.cjs:174-194`), catches a rejection as a single terminal `rejected` result (`:100-118`), and collapses every non-zero/timeout exit into one undifferentiated throw with no transient/fatal split (`fanout-run.cjs:472-489`). Recovery is post-hoc FILE salvage only (`:468-470`); the sole real recovery for a failed branch is re-running the whole fan-out. **Candidate (M):** consume the ledger's per-label `failed` events as resumable state and add transient-vs-fatal classification keyed on `timedOut`/exit-code + a durable per-branch attempt count read from the ledger (mirroring aionforge `max_retries=5`, count-from-audit, `consolidation.md:61-68`), so a transient branch is re-dispatched alone instead of re-running all siblings. Smaller first step (S): classify `timedOut`/429 as `transient` and surface a `retryable` flag in the summary without auto-retry.
- **Q4 — ANSWERED.** Cost guards are advisory: `evaluateCouncilCostGuards` only returns `stop_reasons` and never enforces (`cost-guards.cjs:114-140`); signals are static counts + a verdict-delta check (`:122-133`) with no live lag/pending/failed gauge. The fan-out ledger already records per-lineage `duration_ms` + status counts (`fanout-pool.cjs:90,108`) but only post-run. **Candidate (M):** add live backpressure gauges (in-flight count, oldest-pending lag, failure rate) sampled from the pool's existing `started/completed/failed` ledger events, plus an enforceable `lag_ceiling`-style tripwire (aionforge `consolidation.md:192-201`, default-5s warn) that the loop honors as a hard stop — converting the count-only advisory caps into a metered ceiling. Smaller step (S): emit a `lag_seconds`/`pending` gauge line per ledger tick (no enforcement) for observability parity.
- **Q5 — ANSWERED.** Continuity is reducer-mediated, not prompt-pack-mediated: `prompt-pack.ts` is a stateless `{var}` substituter (`:55-73`) with no iteration awareness, and its only callers are itself + its test. Cross-iteration threading happens by the reducer folding findings into strategy.md anchors that the next iteration re-reads cold (iter-001 finding 16). galadriel's threaded ambient reflection (README:536-538) is the cheaper-continuity model: an iteration that reads its OWN prior open-questions and threads them rather than reconstructing from the distilled snapshot. **Candidate (S):** have each iteration emit a lightweight self-owned "open-questions carried forward" block (separate from the reducer's machine-owned fold) so the next iteration threads its own thread cheaply instead of starting from the reduced strategy alone — reuses the existing JSONL/strategy spine, no new model call.
