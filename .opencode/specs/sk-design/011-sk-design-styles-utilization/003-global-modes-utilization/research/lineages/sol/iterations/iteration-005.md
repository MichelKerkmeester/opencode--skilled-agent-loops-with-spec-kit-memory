## Findings

1. **`FOUNDATIONS_AXIS_SOURCE v1` should join request and selection without treating selected tokens as target-ready.** The request is `{surface, systemRole, register, briefDigest, ownedSystemDigest?, targetPlatforms[], accessibilityBar, densityBar, requestedAxes[], pinnedRoles[], pinnedValues[], requiredRelationships[], exclusions[], corpusGeneration}`. Higher authority is resolved before querying: owned-system and user-pinned roles/values are immutable inputs, not facets a reference may replace. Selection returns `{selectionMode: coherent-anchor|bounded-synthesis|none, references[], axisOwners[], fallback}`; each reference carries `{styleId, contentHash, sourceUrl, originalUrl, styleUuid, capturedAt, rightsStatus, rightsEvidenceScope, selectedAxes, rationale}`. `coherent-anchor` is the default; `bounded-synthesis` is allowed only for an explicit uncovered axis, at most three styles globally, and exactly one owner per accepted independent axis. Foundations still owns system role, register, axis needs, selection, and refusal. [SOURCE: .opencode/skills/sk-design/design-foundations/SKILL.md:113-146] [SOURCE: .opencode/specs/sk-design/011-sk-design-styles-utilization/001-research-utilization/research/lineages/sol/research.md:98-145] [SOURCE: .opencode/specs/sk-design/011-sk-design-styles-utilization/001-research-utilization/research/lineages/sol/research.md:149-168] [INFERENCE: the schema binds existing foundations inputs to phase-001 generation and ownership rules]

2. **The safe compatibility model is a typed graph plus a relationship-preserving blueprint, not an axis checklist.** `FOUNDATIONS_AXIS_COMPATIBILITY v1` contains `axisNodes[{axis, owner, targetRoles[], sourceRelationships[], pinnedBy}]`, `edges[{from, to, relation: preserve|recalculate-for-target|target-derived|incompatible, reason, evidence}]`, `conflicts[]`, and `verdict: coherent-anchor|bounded-compatible|refuse-synthesis`. Every node has zero or one source owner, every cross-axis edge is reconciled, and any overlapping owner or unexplained edge fails. For each accepted node, `RELATIONSHIP_BLUEPRINT v1` records role topology, ordering/grouping constraints, identity locks, do-not constraints, target adaptation reason, and the transformation policy; it deliberately omits a copyable source-value array. This preserves useful relationships while letting foundations recalculate actual color, type, spacing, and breakpoint values for the target register, platform, density, and existing system. [SOURCE: .opencode/skills/sk-design/design-foundations/SKILL.md:115-170] [SOURCE: .opencode/skills/sk-design/design-foundations/references/layout/adaptation_matrix.md:22-60] [SOURCE: .opencode/skills/sk-design/design-foundations/references/layout/adaptation_matrix.md:97-130] [SOURCE: .opencode/specs/sk-design/011-sk-design-styles-utilization/001-research-utilization/research/lineages/sol/research.md:153-168] [INFERENCE: typed edges expose dependencies that coarse axis presence cannot]

3. **Kobu and 19-86 prove that schema regularity supports discovery but not compatibility or independent ownership.** Both token files expose `color`, `font`, `typography`, and `spacing`, yet their optional shapes and role cardinalities differ: Kobu has seven named colors, three font records, many typography steps, and a 5-119px extracted spacing list; 19-86 has two colors, one hairline family, two typography steps, a 6px declared unit, and one flat surface. Their rationales make those values relational: Kobu couples warm parchment, photography-only color, dual type roles, and 60-80px section rhythm, while 19-86 couples black/white, one hairline family, 24px ledger rhythm, no imagery, and a 490px watermark. Neither spacing set is a safe direct fill for foundations' target scaffold, whose 4px baseline, role-first naming, register, dark-mode, and downstream checks are target requirements. Thus top-level axis co-presence is only an availability signal. The pair is a required negative stress fixture: a proposed cross-source composition must refuse incompatible identity edges or mark affected relationships `target-derived`; it cannot call color, type, or spacing independent merely because the JSON keys differ. [SOURCE: .opencode/skills/sk-design/styles/kobu/design-tokens.json:1-55] [SOURCE: .opencode/skills/sk-design/styles/kobu/design-tokens.json:56-318] [SOURCE: .opencode/skills/sk-design/styles/kobu/DESIGN.md:1-95] [SOURCE: .opencode/skills/sk-design/styles/19-86/design-tokens.json:1-89] [SOURCE: .opencode/skills/sk-design/styles/19-86/DESIGN.md:1-62] [SOURCE: .opencode/skills/sk-design/styles/19-86/DESIGN.md:105-141] [SOURCE: .opencode/skills/sk-design/design-foundations/assets/token_starter.md:30-145] [INFERENCE: common structural keys are necessary for bounded hydration but insufficient for semantic compatibility]

4. **Transformation, provenance, unverified proof, and handoff need separate but linked records.** Each `TRANSFORMATION_LEDGER` row is `{axis, sourceRef, sourcePathOrToken, sourceRole, targetRole, targetTokenName, operation: preserve-relationship|rename-role|recalculate-for-target|derive-from-target|reject, targetReason, exactReuseAllowed, proofStatus}`; averaging, interpolation, synthetic means, silent overlap, and detached signature motifs are invalid operations. Provenance proves lineage only: both fixture `source.md` files identify Refero/original URLs, UUIDs, capture time, and included artifacts, but do not establish reuse rights or target fitness. `FOUNDATIONS_CORPUS_PROOF v1` therefore records request/selection digests, compatibility verdict, transformations, refusals, source hashes, rights status, and explicit downstream states for contrast, gamut, baseline rhythm, naming, measure, responsive behavior, and touch targets. Until target values exist and the owning downstream step runs deterministic checks, each applicable state is `not-assessed` or `blocked-with-reason`, never `pass`. `FOUNDATIONS_CORPUS_HANDOFF v1` then extends the shared envelope with target-decided locked values, axis-owner ledger, source lineage, transformation reasons, validation owners, and unresolved risks; `sk-code` implements those accepted target tokens and must neither rehydrate the corpus nor invent roles. 5. **Refusal is fail-closed for source influence while fallback preserves the ordinary foundations workflow.** Refuse the affected reference or synthesis when generation/hash changes, provenance is absent, a required axis has overlapping owners, a cross-axis edge is incompatible or unexplained, exact reuse lacks rights evidence, a source contradicts pinned/owned-system roles, or target values cannot be traced to one owner and reason. Unknown system role, brand source, contrast bar, or target platform blocks the affected design decision rather than inviting corpus inference. If retrieval is unavailable or no source fits, record `selectionMode:none` plus `corpusGrounding: unavailable|no-fit` and continue from target evidence through the baseline foundations workflow; never use stale hydration or manufacture replacement values. If one coherent source covers the request, do not synthesize. ## Questions Answered

- `How should \`design-foundations\` turn captured token axes and relationships into safe starting systems, compatibility maps, and implementation-ready evidence without averaging values or claiming accessibility proof?` Answer: use generation-bound `FOUNDATIONS_AXIS_SOURCE v1`, a typed compatibility graph, relationship blueprints, an explicit transformation/provenance ledger, downstream checks held at `not-assessed` until run on target values, and a target-decision-only `FOUNDATIONS_CORPUS_HANDOFF v1`; refuse overlap/conflicts and fall back to the ordinary target-derived foundations workflow.

## Reflection

- What worked and why: comparing the target token starter with both token JSON and rationale made the distinction between structural regularity and semantic compatibility concrete; the typed-edge model emerged from actual cross-axis couplings rather than an abstract synthesis rule.
- What did not work and why: treating color, typography, and spacing as independent merely because they are separate top-level JSON keys failed immediately; both references bind those axes through identity, composition, density, and platform assumptions.
- What I would do differently: freeze one positive coherent-anchor fixture and the Kobu/19-86 negative synthesis fixture before implementing any adapter, then require every compatibility edge, transformation, and downstream proof state to round-trip through the handoff schema.
