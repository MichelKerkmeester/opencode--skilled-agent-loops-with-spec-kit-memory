# v3.5.0.5, A Second Door to the Same Daemon

The MCP transport has been the memory system's only door, and it is a fragile one. When a transport drops mid-session, Claude Code never reconnects it, so the 37 tools simply vanish for the rest of the session. Every session also pays the tool-schema token overhead up front, and every runtime needs its own MCP registration. This release ships the dual-stack answer: `node .opencode/bin/spec-memory.cjs`, a daemon-backed CLI that fronts the identical 37-tool surface over the same unchanged daemon. The MCP registration stays exactly where it was — the CLI is the resilience and universal surface for hooks, cron, CI and transport-down recovery, not a replacement.

The pairing rule mattered as much as the CLI itself: a CLI nobody's runtime calls does not close the transport-down incident class. So the release lands with the runtime integrations attached — warm-only hook fallbacks for Claude Code and Codex, and a new OpenCode plugin for a surface that, until now, could only reach memory over MCP. Sibling releases ship the same pattern for the other two system daemons: system-code-graph v1.2.0.0 (`code-index`, 8 tools) and system-skill-advisor v0.7.0 (`skill-advisor`, 9 tools plus a trust gate).

## The Transport Was the Single Point of Failure

#### What went wrong

The mk-spec-memory daemon was always sturdier than the path to it. The 026/027 launcher work made the daemon survive owner disposal, re-election and lease races — but a session whose MCP transport died still lost its memory tools permanently, because the client never re-establishes the transport. The incident class was never the daemon; it was the single door.

#### The fix

The CLI is a second IPC client over the daemon that already existed. Nothing about the daemon changed: the same handler queue serializes writes, the same launcher owns the lifecycle, and `spec-memory list-tools --format json` returning `count: 37` straight from `TOOL_DEFINITIONS` makes surface parity against the MCP registration a one-command check. The shim guards before it dispatches — it defaults the socket directory, rejects socket paths over the Darwin `sun_path` limit, and maps a spawn failure to exit `75` so callers can retry instead of guessing.

## The Runtimes Actually Use It

A shared fallback helper (`spec-memory-cli-fallback.ts`, with `code-index-cli-fallback.ts` beside it for the sibling) wraps the CLI in a socket probe plus `--warm-only --timeout-ms` invocation. No socket means a fast fail-open return — measured around 1 ms, against 117–198 ms for a warm call — and the hook result simply omits the CLI-backed extras rather than blocking the prompt. Cold spawn stays confined to SessionStart, explicit prewarm, cron and non-prompt maintenance. The one-shot native bridge, measured at 824.8 ms, stays banned from the prompt path.

On the OpenCode side, the new `mk-spec-memory.js` plugin gives that runtime its first non-MCP route to memory. The bridge speaks CLI/IPC only; the earlier in-process import approach was rejected precisely because a second process importing the database directly is a dual-writer waiting to happen.

## Proven Under Load, Then Reviewed

The suites lock the contract from several directions: 37-tool parity and help behavior, parser and IPC coverage, dual-spawn hardening with re-election on and off, lifecycle reap gating, and a dual-client test running real MCP and CLI clients concurrently against one daemon. A deep-review pass then hardened what the first wave missed: the natural-exit/drain-await fix for pipe truncation, softened continuity overclaims (the runtime transport-down drills are still pending, deliberately), and stress scenarios for concurrent dual-client load, large-payload pipe integrity and numeric-coercion edge arguments now live in the manual-testing playbook.

---

## Still Open

The dual-stack window is open, not closed. The runtime transport-down drills run during real production use, and the measured MCP reference surface (93 files, 1,041 references) migrates only after the window proves the CLI in practice. MCP removal is a separately-gated decision after that. Until then, both doors stay open.

---

## Upgrade

No migration. The MCP registration is untouched and every existing caller keeps working. The CLI is available immediately: `node .opencode/bin/spec-memory.cjs list-tools --format text` answers offline, and any tool call auto-spawns the daemon through the launcher unless `--warm-only` forbids it. If the shim refuses with exit `69`, rebuild the dist (`npm run build --workspace=@spec-kit/mcp-server`). The new env flags are documented in `mcp-server/ENV-REFERENCE.md`.
