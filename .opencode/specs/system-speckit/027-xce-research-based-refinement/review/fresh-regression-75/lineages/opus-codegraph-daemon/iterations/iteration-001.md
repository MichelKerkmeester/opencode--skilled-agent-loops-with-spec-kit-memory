## Analysis — opus-codegraph-daemon (tombstone audit correctness)

**Scope traced:** `code-graph-db.ts` tombstone subsystem (record/prune/summary functions, lines 227–410, 1459–1532), the write paths that emit tombstones (`replaceNodes` 921–955, `replaceEdges` 972–1019, `pruneDanglingEdges` 1027–1036, `removeFile` 1134–1155, `cleanupOrphans` 1505–1532), and the two handlers that surface the summary (`scan.ts`, `status.ts`). I also read the tombstone vitest and the feature-catalog doc to separate documented design from defects.

**What is correct (not findings):**
- `recordEdgeTombstonesForSymbols/ForSources/Dangling` each record the *exact* predicate the subsequent `DELETE` uses, and run inside the caller's `d.transaction()` (`replaceNodes`, `replaceEdges`, `removeFile`). Record-before-delete with matching predicate → accurate happy-path lineage.
- The capped "recent retained" semantics (prune to `getCodeGraphTombstoneLimit()`, `byKind`/`byReason` reflecting **retained** rows, tail-of-scan truncation, within-op eviction of earlier node tombstones by later higher-id edge tombstones) are **documented** as "retained deletion lineage" in `feature_catalog/.../code-graph-status.md:24`. Documented design — not bugs.
- ISO-8601 `deleted_at` lexical ordering is chronologically valid (fixed `…Z` format), tie-broken by autoincrement `id`. Deterministic.
- WAL single-writer + `busy_timeout=5000` + per-op transactions serialize concurrent scans; `getTombstoneSummary` reads a consistent WAL snapshot. No torn accounting under concurrent scans on the transactional paths.

**Genuine defects found — both isolated to `cleanupOrphans` (lines 1505–1532):**

1. **Edge-tombstone undercount from statement ordering.** `recordDanglingEdgeTombstones(d, 'cleanup_orphans', …)` (1508) records only edges whose source/target are *already* missing from `code_nodes`. But the orphan nodes are still present at that point — they aren't deleted until line 1521. Edges attached to those orphan nodes only become dangling *after* the node DELETE, and are then removed by the final dangling sweep (1525–1529) **with no tombstone recorded**. So `byReason['cleanup_orphans']` captures node lineage (via the explicit INSERT at 1512–1517) but silently drops the edge lineage caused by its own node removal — contradicting the doc's "deletion lineage" claim. This is the same record-before-delete discipline that `replaceNodes`/`removeFile` get right (they record edges for the affected symbols *before* deleting), inverted here.

2. **No transaction wrapper (atomicity asymmetry).** `removeFile` carries an explicit CG-003 `d.transaction()` fix (1138–1154); `replaceNodes` (929) and `replaceEdges` (980) are transactional too. `cleanupOrphans` runs its tombstone INSERTs, node DELETE, and edge DELETE as four independently auto-committed statements. A crash mid-function leaves an inconsistent intermediate (orphan nodes gone, dangling edges not yet swept) and, on retry, re-records the same dangling edges → duplicate tombstone rows.

**Severity / confidence calibration:** Both are audit-trail-only (tombstones are observe-only; the live graph remains correct), the feature is **default-OFF** (`SPECKIT_CODE_GRAPH_TOMBSTONES` unset → `false`), and `cleanupOrphans` has **no non-test caller** anywhere in `.opencode` (verified via grep) — it is exported but currently unreferenced. That caps real-world impact at latent, so both are **P2**, not P1. No P0/P1 in this slice. The prior remediation correctly hardened `removeFile` (CG-003) but did not extend the same record-ordering + transaction discipline to `cleanupOrphans`.
