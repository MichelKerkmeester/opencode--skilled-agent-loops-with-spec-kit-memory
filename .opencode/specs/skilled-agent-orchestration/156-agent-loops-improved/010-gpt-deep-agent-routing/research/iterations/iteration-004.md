## Findings

1. **`opencode.json` defines NO `models`, `agents`, agent, or default-model binding.** It contains only `permission`, four `mcp` servers (sequential_thinking, mk-spec-memory, mk_skill_advisor, mk_code_index, code_mode), and `experimental.disable_paste_summary`. There is therefore no config-level artifact that could make the `@orchestrate` surface prefer one agent/model over the build-primary surface — both surfaces inherit whatever model the session loaded. [SOURCE: opencode.json:1-101 (no models/agents keys present)]

2. **`@orchestrate` dispatches EVERY custom agent — including `@deep-research` — with `subagent_type: "general"`.** The Agent Selection table lists `@deep-research` at priority 2, tier LEAF, `subagent_type: "general"`; `@context` at priority 1 also uses `"general"`; the Task Format template hard-codes `Subagent Type: "general" (ALL dispatches use "general")`. The Agent Loading Protocol (§2) and Prompt/Agent Consistency Guard require READ-ing the agent file, INJECTING its content into the Task prompt, and setting `subagent_type: "general"` — explicitly warning "Telling a general agent 'you are @debug' is NOT equivalent to loading `debug.md`." **Consequence: in this repo, agent identity is carried SOLELY by prompt content (the injected agent file), NOT by a per-agent runtime type.** There is no runtime type that distinguishes "run as @deep-research" from "run as @general" — both are the `general` runtime wrapper dressed by prompt text. [SOURCE: .opencode/agents/orchestrate.md:99-105,157-166,168-176,207]

3. **There is NO `general.md` or `build.md` agent file.** Glob of `.opencode/agents/*.md` returns exactly 13 agents (orchestrate, review, code, design, deep-research, ai-council, deep-review, deep-context, context, prompt-improver, deep-improvement, markdown, debug) — none named `general` or `build`. `@general` is therefore a BUILT-IN runtime agent with no specialized file: the "build primary agent" surface is just the default session model running with no injected agent identity. [SOURCE: glob .opencode/agents/*.md; orchestrate.md:207 `"built-in" for @general`]

4. **ROOT CAUSE IS SURFACE-AGNOSTIC (KQ5 answered).** The two surfaces differ in *which prompt carries the dispatch prose* — `@orchestrate` injects it into a Task-tool dispatch prompt (Surface A), the `/deep:research` command injects it via the command YAML's `step_dispatch_iteration` render (Surface B, per iteration-001 F4) — but they SHARE the same two-part root cause: **(i) Task-tool dispatch with the universal `subagent_type: "general"` wrapper** (no per-agent type to enforce identity), **combined with (ii) soft YAML/prose dispatch instructions** ("dispatch the deep LEAF agent", "call the task tool with subagent: X") that are followed-or-ignored at model discretion. The mis-route (Mode A: general absorbs the leaf role; Mode B: re-dispatch regression) is the model's response to that prose in EITHER surface. Surface-specific richness (`@orchestrate`'s §2 Agent Selection table, Agent Loading Protocol, Prompt/Agent Consistency Guard) is a MITIGANT that a GPT-backed model under-weights relative to the literal prose dispatch — NOT a different root cause. [INFERENCE: based on F2 (identity-is-prompt-only) + F1 (no config binding) + iteration-001 F4/F5 + iteration-003 F15 taxonomy]
