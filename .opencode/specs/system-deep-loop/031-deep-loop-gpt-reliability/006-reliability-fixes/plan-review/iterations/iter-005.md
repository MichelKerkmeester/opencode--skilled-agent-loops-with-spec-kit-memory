## GAPS

**1. Reducer safety rule is missing from 005's requirements and success criteria.** *Severity: blocker.*
`005-progress-records/spec.md:73` (REQ-001) defines the schema and "step-transition-only semantics" but never names the critical code-side guard that `iter-014.md:92` calls out as DELIVERABLE 3: "completion reducers must use an explicit allowlist for completion-bearing records and ignore `type:'progress'` / `event:'progress_record'` for completion math." Without an allowlist requirement in REQ-001 and a verification step in the success criteria, any reducer that keys on record-type prefix (e.g. `type !== 'progress'`) will silently misclassify heartbeat-shaped records. `005/spec.md:84` only states the negative invariant ("stalled cells still classified as stalls") with no mechanism. **Recommended amendment:** Add REQ-005 mandating an explicit allowlist (or the equivalent schema-version-1.3 discriminator check) in every reducer / validator that derives completion from `ai-council-state.jsonl` and `*-state.jsonl` files, and add a checklist row that exercises a reducer fed only `type:'progress'` lines and asserts it returns null for completion math. Cite `iter-014.md:92` and the council compatibility rule in `state_format.md:167` as the reference pair.

**2. IMB-001-high "partial credit" in 005's success criteria is logically unreachable from 005's declared scope.** *Severity: major.*
`005-progress-records/spec.md:64` declares in scope "council + context auto YAML + loop protocol; the review prompt pack" and explicitly excludes improvement. But `005-progress-records/spec.md:76` (REQ-004) and `:87` (success criteria) claim "Review/improvement emit progress after each sub-step" and "IMB-001-high partial credit." `findings-registry.md:39` (F-031) is anchored at `assets/prompt_pack_iteration.md.tmpl:57-59, 75-117` — the **review** prompt pack. `iter-014.md` shows no improvement-prompt-pack insertion point. The improvement budget-edge death is the F-032/F-033/F-034 cluster owned by `009-pacing-and-resume`. The "IMB-001-high partial credit" cell is therefore not produced by anything 005 touches. **Recommended amendment:** Either remove "IMB-001-high partial credit" from `005/spec.md:87` and move it entirely to `009-pacing-and-resume/spec.md:87`, or expand 005's in-scope list to "review + improvement prompt pack" and add the corresponding iter-014 insertion point (none exists today — that is a real design gap to either produce or escalate).

**3. Council "persist each seat stepwise" (REQ-002) has no implementation path against `persist-artifacts.cjs`.** *Severity: major.*
`005-progress-records/spec.md:74` requires "Council persists each seat stepwise." `persist-artifacts.cjs:255-285` (`parseCouncilReport`) and `:264-268` (missing-section list) require **all** of `Council Composition`, `Per-seat sections`, `Recommended Plan`, and `Plan Confidence` to be present in a single input; `renderArtifacts` at `:432-443` emits the full seat set as one batch and `writeArtifacts` at `:634-667` writes them in one transaction. A stepwise call that delivers only one seat's markdown will fail validation (`missing: ['Per-seat sections', 'Recommended Plan', 'Plan Confidence']`, exit 1 at `:735`). The iter-014 council after-block (`iter-014.md:50-56`) demonstrates the schema for stepwise records but does not name the writer; for context it points to `state_paths.seat_dir/{seat.label}.json` writes — a different file than the seat markdown artifact that `persist-artifacts.cjs` produces. The spec is silent on which path is the stepwise-persistence target. **Recommended amendment:** REQ-002 must name the writer (either a new `append-seat.cjs` companion, an `--incremental` flag in `persist-artifacts.cjs` that re-runs validation per seat, or a separate `state_paths.seat_dir/*.json` contract distinct from the markdown contract) and add a contract test asserting the chosen path is consumed by the harness as a "per-seat settle" event.

**4. F-018 is scope-creep inside a liveness-records phase.** *Severity: major.*
`005-progress-records/spec.md:74` (REQ-002) closes F-015, F-016, and F-018 together. F-018 in `findings-registry.md:26` is "Single bounded referee pass per persisted round (two-of-three + max-round escape); adjudicator-stability math behind an optional flag" — a convergence-rule change to `deep_ai-council_auto.yaml:10-11, 102-103` and `convergence_signals.md:54-58`, not a liveness-record change. Bundling it under "Step-Transition Progress Records" (the spec title) and the per-step-record semantic of the parent phase conflates two distinct mechanisms: observable progress (records) and bounded internal evaluation (referee math). 005's success criteria at `:84-85` are written entirely in liveness terms; F-018's "bounded referee pass" has no corresponding success-criterion line, so its acceptance cell is not specified. **Recommended amendment:** Either split F-018 out of 005 (move it to a new phase, or to 007-agent-executor-contracts which is the more natural home for council-mode rule changes), or add a sub-success-criterion for F-018 that names the convergence-rule artifact (e.g. "F-018 closed: a single bounded referee pass is enforced per round, auto-yaml:10-11 no longer permits an internal adjudicator-stability loop unless the optional flag is on, and ACB-005 hidden-loop detector flips to pass").

**5. The 60s threshold in REQ-001 is asserted, not derived.** *Severity: minor.*
`005-progress-records/spec.md:73` requires started/completed pairs "for any step expected >60s without another write." The benchmark watchdog at `033-deep-loop-behavior-benchmarks/001-framework-and-harness/spec.md:73` is 120s no-progress. The observed council stalls are 12.3m / 13.4m / 13.8m (`004-rollout-council-improvement/scorecard.md:18-20, 50`) and context stalls 4.3m / 4.5m (`003-rollout-research-context/scorecard.md:26, 66`). None of these reference 60s as a natural unit; 60s is half the watchdog, but the design (`iter-014.md:24`) and 005's spec give no rationale for the ratio. A 30s threshold or a 90s threshold would yield the same observable behavior at the cell level, and the value is unanchored. **Recommended amendment:** In REQ-001 or the design-fidelity notes, derive the threshold from the watchdog window (120s no-progress) and the minimum useful lead time, e.g. "started must be emitted no later than t = (watchdog_window − per-event_watchdog_safety_margin) / 2; with watchdog=120s the threshold is 60s." If the threshold is instead a usability floor, say so and cite a measured LLM sub-step latency distribution. Either way, the number should not be free-floating.

**6. The "no timer heartbeats, step transitions only" rule is unenforceable at the schema level and masks trivial no-op steps.** *Severity: major.*
The schema in `iter-014.md:5-20` is a free-form `type:"progress"` record with `status:"started" | "completed"`. Nothing in the schema distinguishes "transition" from "timer" or "no-op." `005-progress-records/spec.md:73` asserts "step-transition-only semantics," and `iter-014.md:26` says "do not emit timer-only heartbeats; every record must correspond to a real step transition" — but the validator that ingests the JSONL has no field to check this against. A literal executor (precisely the failure mode 034 is designed around) can emit a `started`/`completed` pair around a 1-second no-op that touches a single token and satisfies the 60s rule on its face, while the watchdog's mtime check (`iter-014.md:96`) is satisfied by the JSONL append alone. The reducer safety rule (gap 1) protects completion math, but does not prevent the watchdog from being reset by no-op heartbeats on an otherwise hung run. **Recommended amendment:** Add a schema field that anchors the record to real work — e.g. `unit_id` (already present) plus a `progress_delta` (e.g. seat count, evidence count, bytes written) that a validator cross-checks against the artifact produced — and require the validator to reject `started`/`completed` pairs whose `progress_delta` is zero or whose `artifact_path` does not exist on disk. Make this a REQ, not a design note.

**7. In-CLI council seat simulation does not produce a natural per-seat dispatch boundary for `started`/`completed` records.** *Severity: major.*
`loop_protocol.md:25` (and the synthesis at `synthesis.md:5`) states the council "Dispatch or simulate distinct seats within one CLI round." When the host simulates seats in-CLI, the dispatch step is a single internal model action that produces all seats' markdown in one write — there is no per-seat host-side event for a `started`/`completed` pair to bracket. `005-progress-records/spec.md:74` (REQ-002) requires per-round liveness records, and the iter-014 design at `iter-014.md:38-56` shows per-seat records in the after-block — but the council auto YAML (`deep_ai-council_auto.yaml:765-816` cited in `findings-registry.md:23`) has a single dispatch step, not a per-seat sub-step. A literal executor following the existing dispatch code will produce one `seat_started` for the whole STEP 2 and one `seat_completed` for the whole STEP 2, which fails the liveness intent (the watchdog can be reset at 120s but the per-seat granularity is lost). The synthesis's "unified thesis" (`synthesis.md:5`) explicitly calls out in-CLI council as a different failure surface from dispatch modes; 005 must address it. **Recommended amendment:** Add a REQ to 005 mandating that when the council runs in-CLI, the host MUST break STEP 2 into per-seat sub-steps with explicit started/completed records (or, equivalently, an internal iterator that emits the JSONL lines between simulated seats), and add a contract test that runs an in-CLI council and asserts the count of `seat_started`/`seat_completed` records equals the configured `seats_per_round`. If the design cannot guarantee this for in-CLI mode, restrict the stepwise-liveness requirement to CLI-dispatched mode and explicitly accept that in-CLI mode is bounded by the watchdog only.
