`mcp-tooling` gains its third design transport: `mcp-mobbin` brings read-only app/screen/flow design research (Mobbin MCP via Code Mode) into the hub, completing the three-mode expansion program (aside, refero, mobbin).

---

## Why

Mobbin's MCP server (real-app mobile UX patterns: screens, flows, elements) had no bridge, no registration, and no routing vocabulary. With `mcp-refero` covering design-reference search, `mcp-mobbin` completes the design-research transport pair — both read-only, both paired with `sk-design` for judgment.

## What Changed

#### New transport packet (`mcp-mobbin/`)

- 31-file nested packet authored from a converged 10-iteration deep-research fan-out (3 lineages, all first-attempt); `packetKind: "transport"`, `backendKind: "code-mode-remote-mcp"`, `mutatesWorkspace: false`, Write/Edit/Task forbidden.
- Honest epistemics: OAuth/DCR/PKCE documented as mechanism (no API key env var exists — negative finding preserved); callable name INFERRED pending live discovery; 10 research open questions carried verbatim.

#### Hub surfaces

- `mode-registry.json`: sixth `modes[]` entry; `transport-axis` now declares three transports, all `crossHubPairing` → `sk-design`.
- `hub-router.json`: `mcp-mobbin` signal (weight 4; `mobbin-aliases` + `app-design-research` + `hub-identity`), mirrored vocabulary, tie-break appended last.
- `SKILL.md` (v1.3.0.0), `description.json`, `graph-metadata.json`: mode table row, transport enumerations, keywords, trigger example, intent signals, domains, packet entity.
- `manual_testing_playbook/hub_routing/mobbin_app_research.md`: routing scenario guarding the Mobbin-vs-Refero boundary (generic phrasing may defer).
- `.utcp_config.json`: `mobbin` manual registered (stdio `npx -y mcp-remote https://api.mobbin.com/mcp`, empty env — OAuth flows through mcp-remote's browser round-trip on first use).
