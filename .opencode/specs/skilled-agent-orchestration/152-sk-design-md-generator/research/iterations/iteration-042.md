## Focus
WRITE-phase construction trace (raw-JSON manual write; prompt-builder/formatters structural fix)

## Findings
1. **[P0] No WRITE-phase prompt-builder exists** — The WRITE phase is fully manual. There is no script in tool/scripts/ that builds a constrained prompt from tokens.json. The grep for 'prompt.?builder|compose|prompt.?gen|buildPrompt|renderPrompt|formatPrompt' across all 19 TypeScript modules returned zero results. The SKILL.md pipeline diagram (lines 218-224) describes the WRITE phase as 'Read tokens.json ... Compose 17-section DESIGN.md' but has no script invocation. cli.ts:11 confirms: 'For full DESIGN.md generation, use the Claude Code skill /design-md.' The extraction_workflow.md (lines 51-58) prescribes only manual actions: read the format spec, read tokens.json, compose.
   - Recommendation: Create a prompt-builder script (e.g., tool/scripts/build-write-prompt.ts) that: (a) reads tokens.json and the 17-section format spec, (b) pre-renders deterministic value-tables for each section (color hex tables, typography scales, shadow/radius/spacing tables), (c) explicitly marks each section's data availability as PRESENT/ABSENT/EMPTY, (d) filters tokens by stability class (L1+L2 in main sections, L3 with caution annotation, L4 excluded), (e) produces a constrained markdown prompt where the model fills prose context but values are pre-populated from tokens.json. This is the formatters.ts concept (iter-38) applied at the WRITE boundary.
   - Evidence: SKILL.md:218-224, SKILL.md:257 (no Phase 2 command), tool/scripts/cli.ts:11, references/extraction_workflow.md:51-58, grep of tool/scripts/*.ts for prompt-builder/compose returned 0 hits
2. **[P0] Raw JSON exposure enables fabrication at scale** — The WRITE-phase agent receives the raw DesignTokens object (~15 top-level sections: colorTokens[], typographyLevels[], shadowTokens[], radiusTokens[], components[], darkMode, a11yTokens, etc.) as an unprocessed JSON blob. The cardinal rules in assets/design_md_prompt_template.md are declarative text ('every hex must be verbatim') — that is a wish, not a structural constraint. The model must: (a) navigate deeply nested arrays to find values for each of 17 sections, (b) manually filter by stabilityClassification.layer to exclude L4 tokens, (c) notice silent absences (e.g., darkMode.supported:false means 'omit the section' — no explicit ABSENT signal). Every one of these is a fabrication surface where the model can: invent a hex not in the tokens, miscategorize an L4 token as L2, or write a dark-mode section from inverted light values because it didn't parse the boolean gate.
   - Recommendation: The prompt-builder eliminates these surfaces by pre-rendering value-tables: the model receives a prompt like '## Colors

PRE-RENDERED (from tokens.json, DO NOT EDIT):
| Hex | Role | Frequency |
|-----|------|---------- |
| #1a1a2e | bg-primary | 142 |
...' — the model adds prose, not values. Each section gets a DATA_AVAILABILITY block: 'DARK_MODE: ABSENT — no section should be written' before the model reaches that part of the document.
   - Evidence: types.ts:334-407 (DesignTokens interface — 15 nested sections), design_md_prompt_template.md:31-57 (cardinal rules are prose, not structural constraints)
3. **[P1] Validation is post-hoc only — no upfront structural guard** — validate.ts (436 lines) runs AFTER DESIGN.md is written. It checks phantom colors (hex values in DESIGN.md that don't exist in tokens.json), section completeness (all 17 sections present), and format consistency. This is a post-hoc safety net — it catches fabrication, it does not prevent it. The WRITE phase has zero upfront constraints: the model can write 100 fabricated hex values, and validate.ts will catch them, but the model already wasted tokens and time on hallucinated output. Worse, the validation failures trigger a human-escalation loop (SKILL.md:310-311: 'ESCALATE IF validation reports hex mismatches ... Ask whether to correct tokens.json before re-validating'), which is slow and manual.
   - Recommendation: Move the constraint upstream: the prompt-builder pre-renders values so the model cannot fabricate them in the first place. validate.ts remains as a defense-in-depth post-check, but the primary anti-fabrication mechanism becomes the prompt structure itself rather than a post-hoc grep. This also reduces escalation frequency because the model never sees raw hex values to fabricate — it only sees pre-formatted value-tables.
   - Evidence: validate.ts:57-80 (phantom color check), validate.ts:372-376 (critical failure types), SKILL.md:310-311 (escalation path for validation failures)
4. **[P1] Stability-class filtering is a cognitive load on the model** — Every token in colorTokens[], shadowTokens[], radiusTokens[], and components[] has an optional stability?: StabilityClassification field (types.ts:326-330) with layer values 'infrastructure'|'system'|'campaign'|'content' (L1-L4). The WRITE template (SKILL.md:273-279) requires the model to: include L1+L2 in main sections, annotate L3 with 'Subject to change', and exclude L4 entirely. This is a filtering task performed by the model on raw JSON — the model must iterate arrays, check stability.layer fields, and decide per-token what to include. A single misread (e.g., including an L4 content color as a brand color) produces a fabricated design token that validate.ts may not catch (if the hex exists in tokens.json but the stability class is wrong, it passes the phantom-color check).
   - Recommendation: The prompt-builder pre-filters tokens by stability class before building the prompt. Each section receives a typed sub-prompt: L1+L2 tokens are pre-rendered in the main sections, L3 tokens get a separate 'Campaign Tokens (Subject to Change)' appendix with the annotation pre-applied, and L4 tokens are simply absent from the prompt. The model never sees L4 tokens, cannot accidentally include them, and doesn't waste context on filtering decisions.
   - Evidence: types.ts:326-330 (StabilityClassification interface), types.ts:425/440/449/455/462 (stability? fields on ColorToken/TypographyLevel/ShadowToken/RadiusToken/ComponentGroup), SKILL.md:270-280 (stability class table)
5. **[P2] Silent absences create fabrication risk for conditional sections** — Several DESIGN.md sections are conditional on tokens.json presence: dark mode (darkMode.supported===true), icon system (iconSystem!==null), motion system (motionSystem!==null), a11y data (a11yTokens fields may be sparse). The WRITE template says 'include a dark-mode section ONLY if the tokens contain a detected dark palette' (SKILL.md:243-244), but tokens.json expresses absence via boolean flags and null/empty values — the model must read these programmatic signals from raw JSON and translate them into section omission decisions. A model that skims the JSON or misses a boolean gate can write a fabricated dark-mode section. No explicit ABSENT signal exists in the prompt.
   - Recommendation: The prompt-builder emits explicit DATA_AVAILABILITY blocks before each conditional section: e.g., `DARK_MODE: PRESENT — 12 dark color variables detected` or `DARK_MODE: ABSENT — no dark palette detected, omit this section`. This converts a programmatic signal (boolean field in JSON) into a natural-language instruction the model cannot miss. Same pattern for iconSystem, motionSystem, and a11yTokens sub-fields.
   - Evidence: types.ts:222-229 (DarkModeData with boolean supported field), types.ts:388-390 (nullable iconSystem/motionSystem), SKILL.md:243-244 (conditional dark-mode rule)
6. **[P2] The prompt template is a copy-paste artifact, not a programmable pipeline stage** — assets/design_md_prompt_template.md is a 77-line markdown document with three `<PLACEHOLDER>` slots. It is described as a 'copy-paste prompt' (line 7). It encodes cardinal rules as prose but performs zero transformation of tokens.json. There is no script that fills these placeholders with actual paths and domain names — the agent must do it manually. This means every WRITE-phase invocation starts from an untemplated, human-dependent prompt assembly step, introducing variability and the risk of placeholder leakage (e.g., the model receives `<TOKENS_JSON_PATH>` instead of a real path).
   - Recommendation: The prompt-builder replaces the copy-paste template entirely. It reads the format spec, reads tokens.json, fills all placeholders programmatically, and outputs a complete, machine-generated prompt. The assets/cardinal_rules_card.md and assets/design_md_prompt_template.md become inputs to the builder script, not outputs of a human copy-paste workflow.
   - Evidence: design_md_prompt_template.md:7 ('Copy-paste prompt'), design_md_prompt_template.md:63-71 (placeholder guide — human fills <DOMAIN>, <TOKENS_JSON_PATH>, <DESIGN_MD_PATH>)

## Questions Answered
- How does the WRITE phase actually work? — Confirmed: it is fully manual. There is NO script-based prompt-builder, compose script, or formatter. The agent reads tokens.json + the format spec and composes DESIGN.md by hand. The 'prompt template' at assets/design_md_prompt_template.md is a copy-paste text block with placeholder slots, not a programmatic builder.
- Where does fabrication enter? — (a) The model receives raw, unfiltered JSON with L1-L4 tokens mixed together; it must filter by stability class manually. (b) Empty conditional sections (dark mode, icons, motion) are expressed as boolean/null fields, not explicit ABSENT signals. (c) All 17 section values must be manually found in nested JSON arrays. (d) The 'cardinal rules' are prose, not structural constraints — the model can ignore them. (e) Validation is post-hoc (validate.ts), not upfront.
- Should there be a prompt-builder script? — Yes. It is the single highest-leverage structural fix. It composes with iter-30 (doc-as-view) and is the concrete instantiation of iter-38's formatters.ts: per-section deterministic rendering functions that transform typed tokens into pre-rendered value-tables, with explicit DATA_AVAILABILITY markers for conditional sections.

## Summary
The WRITE phase is fully manual — there is no prompt-builder, compose script, or formatter in tool/scripts/. The agent receives raw tokens.json (~15 nested sections with L1-L4 tokens mixed together) and a copy-paste text template with placeholder slots. The 'cardinal rules' are prose in a prompt, not structural constraints. Fabrication enters at five surfaces: (1) the model must navigate raw JSON to find per-section values, (2) it must filter by stability class manually (L4 tokens are embedded alongside L1/L2), (3) empty conditional sections are expressed as boolean/null fields with no explicit ABSENT signal, (4) there is no upfront structural guard — validate.ts runs post-hoc, and (5) the prompt template is a human-filled copy-paste artifact with no programmatic transformation of tokens.json. The highest-leverage fix is a deterministic prompt-builder script that pre-renders value-tables from tokens.json, injects explicit DATA_AVAILABILITY markers (PRESENT/ABSENT) for conditional sections, pre-filters by stability class, and produces a constrained prompt where the model writes prose around pre-populated values rather than interpreting raw JSON. This is the formatters.ts concept (iter-38) applied at the WRITE-phase boundary.
