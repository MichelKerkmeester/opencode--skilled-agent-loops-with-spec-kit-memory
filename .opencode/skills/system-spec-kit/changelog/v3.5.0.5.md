# v3.5.0.5, Daemon Re-election On By Default

v3.5.0.4 shipped daemon re-election as a flag-gated, default-off foundation pending live validation. This release turns it on by default for every session, backs it with a real integration test, and closes the last loose ends from the anti-disconnection investigation. The change is the difference between a shared daemon that dies with whichever session happened to own it and one that survives a session ending so the others keep working.

---

## What's New at a Glance

| | |
|---|---|
| **Re-election on by default** | The committed runtime configs now set `SPECKIT_DAEMON_REELECTION=1` for mk-spec-memory, so a disposing session releases the shared daemon for a live secondary to adopt instead of killing it. Set `0` to restore the old behavior. |
| **Live release test** | A hermetic integration test drives the real release-vs-kill decision and real process semantics: flag-on releases the detached daemon and it survives, flag-off kills it. |
| **Orphan-sweep template shipped** | The LaunchAgent template the runbook references now exists, dry-run by default, so loading it is non-destructive. |
| **cli session scoping documented** | A dispatched cli sub-session must omit sessionId when calling memory tools. The guard that rejects a foreign id is intended. |

---

## Why On By Default

The investigation found that the earlier fixes made a daemon's death clean but did not keep it alive. With re-election off, the session that owns the shared daemon takes it down on the way out, and every other live session loses MCP transport until a fresh session respawns it. Re-election is the piece that keeps the daemon alive across an owner's exit.

Turning it on for everyone is safe because the downside is bounded. If a live secondary adopts the released daemon, the backend simply survives. If none adopts it, the released daemon is ownerless and the existing idle self-exit reaps it, which lands back on the prior behavior rather than anything worse. The release-vs-kill decision itself is now covered by an integration test.

## Honest Status

Full multi-session secondary adoption is still under live observation rather than proven by a CI test, because a real launcher cannot be spawned in a test without touching the shared production lease and database (both are derived from the launcher's own location with no environment override). The shipped test validates the real decision functions and the real OS release semantics with a hermetic stand-in, and the idle-timeout bound covers the unadopted case. The launcher's code default stays off, so the runtime configs are the single on-switch and a one-character edit reverts it.

## The Loose Ends, Closed

#### Orphan-sweep LaunchAgent template

The runbook and a manual-test scenario referenced a LaunchAgent template that was never committed, so the scenario always failed its lint step. The template now exists at `.opencode/scripts/launchagents/com.michelkerkmeester.orphan-sweep.plist`. It runs the sweeper in dry-run with a log path and does not fire on load, so installing it is a deliberate, non-destructive step. The scenario now passes end to end.

#### cli session scoping

A dispatched cli sub-session that called a memory tool with a parent session id was rejected with a session-scope error. That is an intended guard against session spoofing, not a defect. The shared cli memory-handback reference now says to omit the session id and reuse the one the server returns.

---

## Upgrade

**Activates on a fresh session.** The runtime config change is read when a launcher starts, so a running daemon keeps its old behavior until the next session spawns it. No rebuild is required. To opt out, set `SPECKIT_DAEMON_REELECTION` to `0` in the runtime config. For faster cleanup of an unadopted daemon, also enable the orphan sweeper, which stays opt-in.
