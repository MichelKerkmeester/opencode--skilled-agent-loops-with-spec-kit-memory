## What Was Built

The passive goal feature now has a root command. `/goal` is a thin router that selects one plugin tool call, and the tools own session resolution, state mutation, status rendering, and the exact injection preview.

### Root Command Router

`.opencode/commands/goal_opencode.md` resolves `$ARGUMENTS` into `QUERY`, `FIRST`, and `REST`, then calls `mk_goal` or `mk_goal_status`. The command never edits `.goal-state`, never infers a session id, and never runs shell commands from an objective.

### Goal Tools

`mk_goal` supports `set`, `show`, `clear`, `complete`, and `pause`. `mk_goal_status` reads the current session goal and returns the same status envelope, including `injection_preview` generated by the injection renderer.

## How It Was Delivered

The command shipped as markdown over plugin-owned tools, keeping all stateful behavior in `mk-goal.js` and all user command routing in `.opencode/commands/goal_opencode.md`.

<!-- ANCHOR:decisions -->
## Key Decisions

| Decision | Why |
|----------|-----|
| Keep `/goal` state-free. | The plugin already owns state normalization, session resolution, and failure envelopes. |
| Return terse status lines. | Command callers and tests can parse stable fields without inspecting JSON files. |
| Include `injection_preview` in status output. | Users can see exactly what the passive system transform will inject. |
