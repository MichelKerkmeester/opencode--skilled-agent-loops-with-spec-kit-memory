## Mechanism
OpenLTM uses an idempotent root DDL plus numbered SQL migrations. The root schema defines `schema_migrations(version, applied_at, checksum)` in `schema.sql`, but the actual runner tracks applied files in `_schema_version(version, name, applied_at, checksum)`, so the naming contract is internally inconsistent. Evidence: `packages/openltm-core/src/schema.sql:5`, `packages/openltm-core/src/schema.sql:7`, `packages/openltm-core/src/schema.sql:11`, `packages/openltm-core/src/migrations.ts:25`, `packages/openltm-core/src/migrations.ts:27`, `packages/openltm-core/src/migrations.ts:32`.

FTS5 is external-content and trigger-maintained, not a batch-only index. `memories_fts` maps `rowid` to `memories.id` and indexes `title, content`; insert/delete/update triggers synchronously mirror base-table writes using FTS5 delete rows. Evidence: `packages/openltm-core/src/schema.sql:127`, `packages/openltm-core/src/schema.sql:129`, `packages/openltm-core/src/schema.sql:132`, `packages/openltm-core/src/schema.sql:137`, `packages/openltm-core/src/schema.sql:142`, `packages/openltm-core/src/schema.sql:147`. `context_items_fts` repeats the same pattern for project context. Evidence: `packages/openltm-core/src/schema.sql:155`, `packages/openltm-core/src/schema.sql:157`, `packages/openltm-core/src/schema.sql:164`, `packages/openltm-core/src/schema.sql:169`, `packages/openltm-core/src/schema.sql:174`.

Migration `020_fts_coverage.sql` is the clearest FTS discipline: it drops/recreates `memories_fts`, repopulates from `memories`, then recreates triggers; it also creates/repopulates `context_items_fts` and adds its triggers. Evidence: `migrations/020_fts_coverage.sql:7`, `migrations/020_fts_coverage.sql:9`, `migrations/020_fts_coverage.sql:17`, `migrations/020_fts_coverage.sql:20`, `migrations/020_fts_coverage.sql:24`, `migrations/020_fts_coverage.sql:34`, `migrations/020_fts_coverage.sql:43`, `migrations/020_fts_coverage.sql:50`, `migrations/020_fts_coverage.sql:53`.

The migration runner discovers `*.sql`, requires `NNN_name.sql`, sorts by filename, computes SHA-256, strips comments, splits optional `-- DOWN`, and applies pending migrations in transactions before inserting `_schema_version`. Evidence: `packages/openltm-core/src/migrations.ts:44`, `packages/openltm-core/src/migrations.ts:47`, `packages/openltm-core/src/migrations.ts:53`, `packages/openltm-core/src/migrations.ts:72`, `packages/openltm-core/src/migrations.ts:202`, `packages/openltm-core/src/migrations.ts:239`, `packages/openltm-core/src/migrations.ts:242`. `001_baseline.sql` is a no-DDL sentinel that records the baseline created by `schema.sql`. Evidence: `migrations/001_baseline.sql:1`, `migrations/001_baseline.sql:4`, `migrations/001_baseline.sql:7`.

Connection setup is WAL-first with FK enforcement and busy timeout; `shared-db.ts` executes `schema.sql`, then runs pending migrations, then backfills vec indexes if sqlite-vec exists. Evidence: `packages/openltm-core/src/shared-db.ts:31`, `packages/openltm-core/src/shared-db.ts:36`, `packages/openltm-core/src/shared-db.ts:37`, `packages/openltm-core/src/shared-db.ts:39`, `packages/openltm-core/src/shared-db.ts:42`, `packages/openltm-core/src/shared-db.ts:45`.

## Negative knowledge
Do not copy the split `schema_migrations`/`_schema_version` naming; it creates source-of-truth ambiguity.

Do not trust same-file rollback unless `DOWN` SQL is executable; `parseMigration()` strips comments, while migrations like `002` have commented-only downs.

Do not copy single-file DB assumptions; system-spec-kit’s index and vector shard must be backed up, checked, and restored as one unit.

Do not replace FSRS/causal-edge schema with OpenLTM’s simpler decay/relation model; our current cognitive and causal surfaces are richer.

Should `memory_health` add an explicit FTS5 `integrity-check`/rebuild report for `memory_fts`?

Should migration checksums be stored per migration while preserving our existing single `schema_version` compatibility gate?
