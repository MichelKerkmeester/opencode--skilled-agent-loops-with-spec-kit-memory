# Getting Started With the Fixture Docs Corpus

A practical walkthrough for reading this fixture's `docs/` tree, written the way a real project guide would explain itself.

---

## 1. OVERVIEW

This guide orients a reader — human or an audit leaf-agent — inside the `fx_001_alignment_target` fixture's `docs/` corpus: what each subfolder is for, how the files were built, and how the same corpus could be regenerated by hand if it ever needed to be. Nothing in this guide is fixture-only machinery; it reads like an ordinary project guide on purpose, because that is exactly the shape a conformance audit needs to check it against. A guide that only made sense inside the benchmark would not exercise the same code paths as a real documentation tree.

---

## 2. QUICK START

Three things worth knowing before reading further:

- The corpus lives under `docs/`, relative to the fixture root, with `references/`, `skill-shaped/`, and `guides/` subfolders classifying by path the same way a real skill tree does.
- Every file here reads on its own; none depends on state left over from a previous run or from any other file in the corpus.
- Nothing in this corpus is auto-generated boilerplate — each file's content is specific to what it is seeded to demonstrate, not a template stamped out four times.

```bash
find docs -name "*.md" | sort
```

Running that from the fixture root lists every markdown file currently in the corpus, in the same order a path-based directory walk would discover them.

---

## 3. STRUCTURE

| Subfolder | Classifies as | Why |
|---|---|---|
| `docs/` (root) | `readme` | Exact `README.md` filename match |
| `docs/references/` | `reference` | Path contains `/references/` |
| `docs/skill-shaped/` | `skill` | Exact `SKILL.md` filename match |
| `docs/guides/` | falls back | No special path segment, so classification falls through to the default case |

The table mirrors the real classifier's own detection order rather than inventing a fixture-only taxonomy — a genuine audit should see the same shape a real project's `docs/` tree would have, not a simplified stand-in.

---

## 4. TROUBLESHOOTING

If re-running an audit over this corpus produces a different set of findings than expected, check these first:

- Confirm the lane's `scope` actually points at `docs/` and not a sibling directory like `docs-clean/` — the two are scoped separately on purpose, and mixing them up changes which findings are even possible.
- Confirm no file was edited in place; a read-only audit should leave every byte in this corpus unchanged, including this file.
- Re-run the authority's real validator directly against the suspect file rather than trusting a visual scan of the rendered markdown — a document can look irregular at a glance and still be fully conformant, or look tidy and still fail a structural check.
