# Iteration 005 - K1.4 chunked embeddings and stable identity

## Focus

K1.4 asks whether spec-doc embeddings should be chunked by frontmatter and section boundaries with per-chunk fingerprints, so editing one section of a long spec does not re-embed the whole document. This iteration also addresses the stable-identity blocker from iteration 004: chunk identity must survive heading renames, section reorders, and split/merge operations.

Verdict preview: yes with adaptation. The repo already has a partial chunking path, but it is gated to large files and full-auto saves, and its durable identity is not strong enough for section-level incremental embedding.

## Findings

### CocoIndex chunking example

The canonical example chunks every supported file before embedding. `process_file()` reads file text, detects language from filename, splits with `RecursiveSplitter.split(text, chunk_size=1000, min_chunk_size=300, chunk_overlap=300, language=language)`, then maps `process_chunk` over the chunks (`main.py:89-107`). Each chunk is embedded independently and stored with filename plus `start_line` and `end_line` (`main.py:69-85`).

CocoIndex does not expose a semantic chunk key in that example. The primary key is generated by `IdGenerator.next_id(chunk.text)` (`main.py:77-80`). `IdGenerator` is explicitly designed to produce distinct IDs for repeated chunk contents while keeping the sequence stable across runs (`id.py:89-99`, `id.py:126-149`). That is stable for deterministic splitter output, but it is still content-plus-ordinal identity. Edits that change chunk boundaries before a later chunk can shift IDs unless the higher-level pipeline memoization can reuse unchanged upstream outputs.

The shipped splitting API has two relevant strategies: `SeparatorSplitter`, which splits by regex separators and returns position-aware chunks (`text.py:39-84`), and `RecursiveSplitter`, which recursively splits by syntax/structure and returns chunks with byte, char, line, and column positions (`text.py:120-180`, `text.py:183-209`). The `Chunk` object itself carries text plus positions only; no chunk ID, heading path, AST path, or fingerprint is part of the resource (`chunk.py:25-36`).

### Our current embedding granularity

The normal parser still returns one `ParsedMemory` per file. `parseMemoryContent()` computes a single `contentHash` over full file content and returns full `content` for ordinary spec/memory docs (`memory-parser.ts:342-385`). The normal save embedding path then computes one cache key from the normalized whole document and generates one weighted document embedding (`embedding-pipeline.ts:119-155`). That means ordinary saves still behave as whole-doc embedding units.

Chunking exists, but only on one gated path. `memory-save.ts` computes `shouldUseChunkedIndexing(routedParsed.content, plannerMode)` before embedding, but still calls `generateOrCacheEmbedding()` for the whole routed doc first (`memory-save.ts:2190-2217`). Later, if `shouldChunkContent` is true, it calls `indexChunkedMemoryFile()` (`memory-save.ts:2339-2348`). The chunk gate itself requires `plannerMode === 'full-auto'` and `needsChunking(content)` (`chunking-orchestrator.ts:121-126`), and `needsChunking()` requires content longer than 50,000 characters (`anchor-chunker.ts:60-62`, `anchor-chunker.ts:303-307`). So current behavior is "large-doc fallback chunking", not general incremental spec-section embedding.

The chunking orchestrator already embeds children independently and uses a persistent content-hash embedding cache per normalized chunk (`chunking-orchestrator.ts:266-286`). That is the strongest evidence that chunk-level re-embed skipping is technically cheap. The gap is stable row identity: chunk rows get `parent_id`, `chunk_index`, and `chunk_label` (`chunking-orchestrator.ts:326-337`), but `content_hash` remains the parent full-doc hash, not the chunk fingerprint (`chunking-orchestrator.ts:328-331`).

### Schema support and retrieval behavior

`memory_index` already supports multiple rows per document. The table has `parent_id`, `chunk_index`, and `chunk_label` columns (`vector-index-schema.ts:2412-2414`), and migration v16 added those columns plus parent/chunk indexes (`vector-index-schema.ts:863-898`). The table also has `anchor_id` and a uniqueness constraint on `(spec_folder, file_path, anchor_id)` (`vector-index-schema.ts:2360-2364`, `vector-index-schema.ts:2418-2419`), which is close to what stable chunk identity needs.

Retrieval can already collapse chunk hits back to document-level results. Hybrid search has an aggregation stage that identifies results with `parentMemoryId` and `chunkIndex`, collapses chunk hits, and returns the parent memory id with `_chunkHits` metadata (`hybrid-search.ts:1358-1389`). That means the UX does not have to choose between granular embedding and noisy chunk-only output; we can embed and search chunks while aggregating back to docs by default.

### Stable identity blocker

Headings alone are not stable enough. The 027 spec tree shows many files with consistent `<!-- ANCHOR:* -->` markers immediately before H2 sections, while headings include numbering and sometimes level labels such as `L2:` (`rg survey`, e.g. `013-cocoindex-memory-port-research/spec.md:50-205` and `tasks.md:35-92`). A heading slug survives section reorder but breaks on rename; a token/window offset breaks on edits before the section; a chunk index breaks on insertion, split, merge, and reorder.

The existing anchor chunker is a better starting point. It extracts paired `<!-- ANCHOR:id -->` / `<!-- /ANCHOR:id -->` regions and labels chunks by anchor id when possible (`anchor-chunker.ts:70-118`, `anchor-chunker.ts:153-176`). But it only chooses anchor chunking if at least two paired anchors are present (`anchor-chunker.ts:288-295`). Many spec docs use opening `<!-- ANCHOR:* -->` markers before headings without explicit closing anchors, so the current extractor can fall back to structural chunks and lose the available stable anchor signal.

Split/merge needs a two-layer identity. For a normal section, `chunk_id = anchor:<anchor_id>` is stable across heading rename and reorder. For oversized anchored sections, use `chunk_id = anchor:<anchor_id>#part:<part_ordinal>` plus `chunk_fingerprint = sha256(normalized chunk content)`. If content inside part 2 changes because part 1 grew, ordinal identity may shift; that is acceptable only as a fallback. A stronger later option is a within-anchor content-defined chunker, but the first practical step is anchor-level identity for normal sections and explicit invalidation for oversized section rechunking.

## Chunking Strategy Decision Matrix

| Strategy | Identity | Re-embed cost on minor edit | Pros | Cons |
|----------|----------|----------------------------|------|------|
| Whole-doc (current normal path) | path/canonical path plus optional anchor for routed saves | Full doc | Simple; existing parser/save flow | Wasteful for long specs; one changed section invalidates weighted whole-doc embedding |
| Per-H2-section, identity=heading-slug | normalized heading slug | Only changed section if headings unchanged | Author-friendly; no marker discipline | Heading renames break identity; duplicate headings need ordinal fallback |
| Per-H2-section, identity=anchor-id | `<!-- ANCHOR:* -->` / generated stable section anchor | Only changed anchored section | Stable across heading rename/reorder; already common in spec docs; maps to existing `anchor_id` | Requires marker presence or bootstrap migration; split/merge policy must be explicit |
| Per-fixed-token-window | start/end token offsets or sequence | Affected windows | Mechanical; no markdown parser needed | Brittle on earlier edits; poor semantic boundaries; hard to explain in retrieval results |
| Per-frontmatter-block + per-section | `frontmatter:<name>` plus `anchor:<id>` | Metadata block plus affected sections | Separates continuity/metadata churn from narrative sections; fits spec docs | More parser/schema work; needs stable block IDs for YAML/frontmatter regions |

## Verdict on K1.4 and Sketch

YES-WITH-ADAPTATION. Spec-doc embeddings should become chunked units with per-chunk fingerprints, but the identity scheme must be anchor-first, not heading-first or position-first. Current code proves the table and retrieval surfaces can handle parent/child rows, and chunk embeddings already use a per-chunk cache; the missing piece is a stable logical chunk key plus chunk fingerprint stored durably.

Recommended schema: extend `memory_index` rather than adding a separate `memory_chunks` table for the first pass. Add `parent_doc_id` as an alias/clarifier only if needed; existing `parent_id` already represents parent document row. Add `chunk_id TEXT`, `chunk_fingerprint TEXT`, `chunk_kind TEXT CHECK(chunk_kind IN ('frontmatter','continuity','section','overflow','fallback'))`, `chunk_start_line INTEGER`, `chunk_end_line INTEGER`, and indexes on `(parent_id, chunk_id)` and `(spec_folder, canonical_file_path, chunk_id)`. Keep `chunk_index` as display/order metadata, not identity.

Identity scheme:

1. `frontmatter:<block>` for YAML frontmatter and `_memory.continuity` blocks.
2. `anchor:<anchor_id>` for anchored sections. Prefer explicit anchors; bootstrap missing section anchors from existing H2 headings by inserting or recording generated anchors only when a migration/formatter is allowed.
3. `anchor:<anchor_id>#part:<n>` for oversized anchored sections split internally.
4. `heading:<slug>#ordinal:<n>` as bootstrap fallback for legacy docs without anchors.
5. `window:<seq>` only as last resort and marked `chunk_kind='fallback'`, because it is not stable across earlier edits.

Migration: do not re-embed everything on day 1. Add nullable columns first. Existing rows remain valid parent/doc rows. On the next save or index scan, create chunk rows lazily only for docs over a threshold or docs in spec folders touched by the operation. If an existing whole-doc embedding is fresh and no chunk rows exist, keep returning it. Backfill chunks gradually through `memory_index_scan({ incremental:true })`, using the chunk fingerprint cache to skip provider calls for unchanged normalized chunk text.

Retrieval: store/search chunks, aggregate by parent by default. Return chunk snippets and `_chunkHits` when useful, but keep the current doc-level UX unless the caller asks for `granularity:'chunk'`. This matches the existing MPAB-style chunk collapse path in hybrid search (`hybrid-search.ts:1358-1389`).

## Questions Answered

- K1.4 answered: yes, chunk spec-doc embeddings with per-chunk fingerprints.
- Stable identity answered: use anchor IDs as logical chunk identity; heading slugs and positions are bootstrap/fallback only.
- Schema answered: extend `memory_index` child rows with `chunk_id` and `chunk_fingerprint`; existing `parent_id`/`chunk_index`/`chunk_label` are insufficient as identity.
- Migration answered: lazy nullable-column migration plus incremental backfill avoids all-doc re-embedding on rollout.
- Retrieval answered: search chunks and aggregate back to parent docs by default.
