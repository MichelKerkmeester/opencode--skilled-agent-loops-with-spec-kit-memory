## Findings
1. **Expose a native v2 request that separates membership from ranking.** `queryStylesV2` should accept `{apiVersion:2,text,eligibility:{requiredFacets,exclusions,usage,exactReuse},signals:{axes,needs},mode,channels:{fts,vector},page:{size,candidateK,cursor},generationHash,maxCardBytes}`. Preserve the current 1,000-character/12-term lexical bounds, `reference|exact-reuse` usage semantics, 768-2,048-byte card range, and five-card page default/maximum; use `candidateK` (default 50, hard maximum 100) only as each channel's fusion depth. This separation prevents `axes` and `needs`, which are ranking signals today, from silently becoming hard filters. [SOURCE: .opencode/skills/sk-design/styles/_engine/rank-fts.mjs:18-24] [SOURCE: .opencode/skills/sk-design/styles/_engine/rank-fts.mjs:46-54] [SOURCE: .opencode/skills/sk-design/styles/_engine/cards.mjs:9-12] [SOURCE: .opencode/skills/sk-design/styles/_engine/cards.mjs:68-81] [INFERENCE: a nested v2 contract can distinguish existing membership and scoring semantics while a flat-request adapter preserves v1]

2. **Apply hard eligibility in SQL before every retrieval channel.** The eligible CTE/materialized ID set must enforce active/non-tombstoned generation membership, normalized required-facet inclusion, exclusion absence, `provenance.status='known'`, and exact-reuse rights (`rightsKnown` plus `allowed|licensed|public-domain`). FTS and vector queries join this set before top-k, rather than filtering their returned lists, so an ineligible high-scoring style can neither consume channel depth nor re-enter through fusion. The current code establishes this ordering and its tests prove lexical ranking cannot restore an excluded style. [SOURCE: .opencode/skills/sk-design/styles/_engine/eligibility.mjs:15-48] [SOURCE: .opencode/skills/sk-design/styles/_engine/eligibility.mjs:55-80] [SOURCE: .opencode/skills/sk-design/styles/_engine/__tests__/eligibility-first.test.mjs:15-31]

3. **Pin one published generation and one retrieval profile for the entire request.** Begin a read transaction, resolve `active_generation`, and bind source rows, FTS projection, vector profile/retrieval hash, fusion-profile version, cards, and cursor to that snapshot. A caller-supplied generation must match an available retained generation; continuation must reopen the cursor's exact generation or return `generation-expired`, never switch to current. Projection rows whose generation/retrieval hash/profile differs are stale and excluded channel-by-channel. This replaces the current rebuild-and-byte-compare guard without weakening its same-generation guarantee. [SOURCE: .opencode/skills/sk-design/styles/_engine/style-library.mjs:119-155] [SOURCE: .opencode/skills/sk-design/styles/_engine/hydrate.mjs:225-291] [INFERENCE: iteration-4 atomic publication plus one SQLite read snapshot provides the persistent equivalent of the live-manifest equality guard]

4. **Fuse ranked lists, not incomparable raw scores.** Produce three deterministic lists after eligibility: `structured` ordered by the existing required-facet/axis/need score, weighted FTS5 ordered by BM25, and vectors ordered by cosine similarity. Fuse active lists with versioned weighted RRF `sum(weight[channel] / (40 + rank[channel]))`, initial weights `structured=1, fts=1, vector=1`, no adaptive intent or graph lanes, and final tie-break by UUID. Return each rank and reciprocal-rank contribution. This adopts system-spec-kit's robust rank fusion and attribution while avoiding its memory-specific dynamic routing, graph, cognition, and post-fusion layers. [SOURCE: .opencode/skills/system-spec-kit/shared/algorithms/rrf-fusion.ts:21-42] [SOURCE: .opencode/skills/system-spec-kit/shared/algorithms/rrf-fusion.ts:70-88] [SOURCE: .opencode/skills/system-spec-kit/shared/algorithms/rrf-fusion.ts:445-510] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/hybrid-search.ts:1593-1649]

5. **Return explicit channel health and score attribution without changing legacy cards.** The native response should be `{ok,apiVersion,generationHash,queryFingerprint,rankingMode,degraded,channels,eligibility,page,cards}`. Each channel reports `{state:used|disabled|unavailable|stale|empty,generationHash,candidates}`, and each native card keeps all current fields while adding `retrieval:{rank,fusedScore,structuredScore,sourceRanks,sourceContributions,channels}` plus `hydrationToken`. The v1 adapter strips native-only additions and preserves the current top-level `ok/generationHash/degraded/rankingMode/eligibility/cards` shape and current card `score:{deterministic,lexical,total}` field. 6. **Use generation-bound keyset pagination over a fixed fusion window.** `page.size` remains at most five, while `candidateK` fixes the independently retrieved top-k pool for every page. An opaque cursor carries version, generation hash, query/eligibility/signals hash, fusion-profile version, candidateK, and the last `(fusedScore,id)` order key; continuation rejects any request mismatch. Recomputing the same bounded lists under the same immutable generation yields the same order, and keyset comparison avoids offset drift. The pressure test produced tied hybrid scores for A/C, deterministically ordered A then C, and resumed at D without duplication. [SOURCE: command: bounded Node RRF/filter/cursor pressure test, 2026-07-19] [SOURCE: .opencode/skills/system-spec-kit/shared/algorithms/rrf-fusion.ts:229-242] [INFERENCE: generation plus request/fusion fingerprints make stateless recomputation stable without storing query sessions]

7. **Treat hydration tokens as locators, never authorization.** A native card's opaque versioned token should encode style UUID, generation hash, content hash, requested mode, usage, and permitted include names. `hydrateStyleV2` validates/decodes it, then re-runs generation availability, mode allowlist, exact-reuse rights, root/style containment, artifact digest, and byte-cap checks. The compatibility adapter continues accepting the closed legacy `{id,generationHash|cardGenerationHash,mode,includes,maxBytes,usage}` request; it consumes the new token before invoking the existing guard because the legacy validator intentionally rejects unknown keys. [SOURCE: .opencode/skills/sk-design/styles/_engine/hydrate.mjs:29-56] [SOURCE: .opencode/skills/sk-design/styles/_engine/hydrate.mjs:164-199] [SOURCE: .opencode/skills/sk-design/styles/_engine/hydrate.mjs:315-370] [SOURCE: .opencode/skills/sk-design/styles/_engine/__tests__/hydrate-guard.test.mjs:20-50]

8. **Define a deterministic projection-degradation ladder with no normal-path corpus walk.** Use hybrid when same-generation FTS and vectors are healthy; use structured+FTS when vectors are disabled/unavailable/stale; structured+vector when FTS is unavailable/stale; and structured-only SQL when both accelerators are unavailable. Empty text naturally uses structured-only. Every result names inactive channel reasons. If the authoritative database generation itself is absent, invalid, or known stale, fail `index-unavailable|index-stale` because rights and eligibility cannot safely degrade; do not walk files at read time. This keeps the reference implementation's fail-open channel behavior but makes the terminal fallback persistent and bounded. [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/hybrid-search.ts:650-695] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/hybrid-search.ts:1630-1655] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/hybrid-search.ts:2483-2553] [INFERENCE: structured-only SQL is safer than stale source metadata and cheaper than the current source scan]

9. **Keep corpus walking only behind an explicit migration/rollback adapter.** `runQuery`, `runHydrate`, and CLI `query|hydrate` remain the public v1 boundary. A backend selector (`legacy`, `shadow`, `persistent`) can initially default to legacy, dual-read in shadow, and later default to persistent; only explicit legacy/rollback mode may invoke `buildManifest`, disposable FTS, or `rankWithSourceScan`. In persistent mode, old `useFts` maps to channel disablement, `acceleratorGenerationHash` becomes an expected projection-generation diagnostic, cards are adapted to their current bounded shape, and hydration maps to the native token path while retaining old refusal codes. This makes cutover reversible without preserving read-time walking as a hidden degraded mode. ## Ruled Out
- Raw weighted addition of BM25, cosine, and current deterministic scores: their scales are incomparable and would make attribution/profile changes unstable.
- Post-retrieval eligibility filtering: it can starve top-k and lets forbidden candidates influence fusion depth.
- Offset pagination or unpinned cursors: either can duplicate/skip cards across generation or profile changes.
- Treating a hydration token as a rights capability: existing hydration deliberately rechecks rights, containment, and hashes.
- Falling back to corpus walking after persistent cutover: that recreates the latency and freshness problem this design removes.

## Dead Ends
None. The existing source-scan fallback remains useful only as a time-bounded migration/rollback backend and should not be promoted into the persistent API.
