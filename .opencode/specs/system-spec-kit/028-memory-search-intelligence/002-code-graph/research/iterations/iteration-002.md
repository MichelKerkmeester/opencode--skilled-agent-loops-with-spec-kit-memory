## Findings [file:line]

### Q1 — Bi-temporal mapping onto code_edges

1. **aionforge models currentness as edge presence + closed windows, never deletes.** The `BiTemporal` block (`valid_from`/`valid_to`/`ingested_at`/`expired_at`) lives on the edge, not the node; supersession closes the old window (`valid_to <- valid_from`) and writes `old -SUPERSEDED_BY-> new`, retaining the prior fact [SOURCE: bi-temporal-model.md:14-39, 94-101]. This is the direct analog of the iter-1 §12 mapping: scan-time = transaction-time (`ingested_at`/`expired_at`), commit-time = event-time (`valid_from`/`valid_to`). [CONFIRMED by doc read.]

2. **All five physical edge-delete sites in code-graph are confirmed** (currentness = physical presence, no temporal columns): `replaceNodes` deletes node-touching edges [code-graph-db.ts:941-943]; `replaceEdges` deletes per-source [code-graph-db.ts:985]; deferred dangling delete [code-graph-db.ts:1012]; `pruneDanglingEdges` [code-graph-db.ts:1031]; `removeFile`/symbol-removal delete [code-graph-db.ts:1149]; and the orphan GC sweep [code-graph-db.ts:1546]. Node deletes at :936 (`replaceNodes`) and :1540 (GC). [CONFIRMED via grep.]

3. **CANDIDATE Q1-C1 (M, low-conflict): add `valid_at`/`invalid_at` (nullable INTEGER, transaction-time generation) to `code_edges` and supersede instead of delete.** Replace the four reindex DELETEs (:941, :985, :1012, :1031) with an UPDATE that sets `invalid_at = <current scan generation>` on edges no longer re-asserted, and INSERT new edges with `valid_at = <generation>, invalid_at = NULL`. The default read path filters `invalid_at IS NULL` (= aionforge `Current`); an as-of-generation read filters `valid_at <= G AND (invalid_at IS NULL OR invalid_at > G)` (= aionforge `AsKnownAt`). This enables "as-of-last-green-scan impact queries" when a fresh scan is partial or broken. [INFERENCE from bi-temporal-model.md:46-83 mapped onto code-graph-db.ts:177-184 schema (no temporal cols today).] Conflict risk: the dangling-prune contract (code-graph-db.ts:957-968) and cross-file CALLS resolver must learn to skip already-invalidated edges; SCHEMA_VERSION bump from 5 required.

4. **CANDIDATE Q1-C2 (S, low-conflict): represent path-coupled-ID renames as `supersedes` instead of delete+create.** Because the symbol ID is `sha256(filePath::fqName::kind)` [indexer-types.ts:101-102], a pure file/symbol rename mints a NEW id and the old one is deleted — aionforge would instead close the old node's window and write `old -SUPERSEDED_BY-> new` (bi-temporal-model.md:94-101). A code-graph `SUPERSEDES` edge type (or reuse of the existing tombstone machinery, code-graph-db.ts:227-288, currently opt-in/off) keyed on matching `contentHash` (`sha256(content)`, indexer-types.ts:108-109) across the rename would preserve rename lineage for impact queries. [INFERENCE.] This is strictly additive — it does not require Q1-C1's column change, but composes with it.

5. **Generation watermark (Q6 preview).** aionforge's maintained sets are generation-checked end-to-end: a stale provider surfaces as an **error**, never a stale set; the `generation` is "a live watermark, not a hint" [bi-temporal-model.md:146-154]. This is the concrete refinement for code-graph's binary `freshness !== 'fresh'` block (handlers/query.ts:903-915, iter 1 finding 15): a generation number would let as-of-generation reads replace all-or-nothing block. Deferred to Q6 per strategy; flagged here because Q1-C1's `valid_at`/`invalid_at` columns are the storage substrate a generation watermark needs.

### Q2 — Content-addressed IDs + Transient/Fatal quarantine

6. **aionforge IDs are content-addressed and idempotent; that is what makes re-extraction/replay a no-op.** `fact_id` hashes the canonical triple + episode + rule version; entity ids hash namespace + type + normalized name [consolidation.md:84-90, 106-108]. Re-running after a crash reconstructs identical artifacts [consolidation.md:55-59]. [CONFIRMED by doc read.]

7. **code-graph edge IDs are NOT content-addressed — they are path-coupled via the symbol IDs.** Edge endpoints are `source_id`/`target_id` = the path-coupled symbol IDs [indexer-types.ts:101-102; code-graph-db.ts:177-184]. So a rename that changes a file path changes every incident edge's endpoints → the rename looks like delete+create, exactly the fragility aionforge avoids. [CONFIRMED — combines iter-1 finding 8 with the symbol-ID grep this iteration.]

8. **The parser skip-list does NOT classify Transient vs Fatal.** `SkipListErrorClass = 'B1' | 'B2' | 'OTHER'` are tree-sitter WASM **crash cohorts**, not a retry policy [parser-skip-list.ts:9, 27-35]. Add is an idempotent upsert that only bumps `attempt_count` [parser-skip-list.ts:69-91]; there is NO `max_retries` ceiling, NO transient-leaves-pending semantics, and removal is an intentional permanent no-op (`recordSuccess` is `@deprecated`, body empty, "must not auto-unskip... or imply self-heal") [parser-skip-list.ts:93-97]. A single crashing file is therefore permanently skipped with no bounded-retry escape. [CONFIRMED by full read.]

9. **aionforge's quarantine model is the candidate refinement.** `PassError::Transient` (down embedder, rate limit, timeout) leaves the episode `raw` so it retries; `PassError::Fatal` OR transient-exceeds-`max_retries` (default 5) marks it `failed` — retained, audited, excluded, and later ticks **proceed past it** so a poison pill never wedges the pipeline; the attempt count is read from the durable failed-audit trail, not memory, so a crash does not refresh the retry budget [consolidation.md:60-68]. [CONFIRMED by doc read.]

10. **CANDIDATE Q2-C1 (M, medium-conflict): add a Transient/Fatal classification + bounded retry to the skip-list.** Split today's permanent-skip into: (a) `TRANSIENT` (e.g. WASM OOM, timeout, deadline-abort) → re-attempt on next scan until `attempt_count >= max_retries` (default 5), then promote to (b) `FATAL` (the current permanent-skip, manual-review-only). Reuse the existing `attempt_count` column (parser-skip-list.ts:79-87) as the durable budget — it already persists across restarts via SQLite, matching aionforge's "durable audit trail, not in-memory" property [consolidation.md:66-68]. Conflict risk: directly contradicts the current "must not auto-unskip / no self-heal" stance (parser-skip-list.ts:93-97), so it is a deliberate policy change, not a pure addition — needs explicit owner sign-off. The B1/B2/OTHER cohort labels become orthogonal to the new transient/fatal axis.

11. **CANDIDATE Q2-C2 (L, high-conflict): content-address edge endpoints (decouple symbol ID from path).** Re-key symbol IDs on `sha256(contentHash::fqName::kind)` or `sha256(fqName::kind::signature)` so a rename keeps the same ID and edges survive intact (aionforge `fact_id` analog, consolidation.md:84-90). This would make renames truly non-destructive WITHOUT needing the Q1-C2 `SUPERSEDES` edge. Marked L/high-conflict because the path-coupling is currently load-bearing for file-scoped `replaceNodes(fileId, ...)` (code-graph-db.ts:921) and the `DELETE WHERE file_id = ?` reindex unit; decoupling forces a different reindex granularity and risks cross-file fq_name collisions. [INFERENCE.] Q1-C2 (`SUPERSEDES` edges) is the lower-risk alternative to this for the same rename-lineage goal.

### Q3 — Impact ranking is a flat edge-walk; PPR assessment

12. **CONFIRMED: impact/neighborhood ranking is a flat single-hop edge enumeration, no centrality.** `code_graph_context` impact mode walks reverse `CALLS`/`IMPORTS` edges at depth 1 only, pushing each incoming edge with `formatContextEdge` (confidence = `metadata.confidence ?? edge.weight`) and a hard 400 ms latency budget that truncates with `omittedEdges` — there is no recursive spread, no mass propagation, no re-rank by graph position [SOURCE: code-graph-context.ts:627-671, 343-360, 297-300]. Zero `pagerank|personaliz|teleport|damping|ppr` tokens exist in `handlers/`, `lib/`, or `graph/` [SOURCE: grep, empty result]. The shared BFS primitive `traverseGraphBfs` (bfs-traversal.ts:76-174, iter-1 finding 12) is visited-set + depth-bound only. [CONFIRMED.]

13. **aionforge's `Graph` signal IS native Personalized PageRank**, seeded on query-named entity nodes, spreading mass across the associative graph, best-first by PageRank score, with the reached fact set intersected with current-support membership (PageRank is not current by construction); fusion reads only the rank [SOURCE: retrieval.md:193-198]. Crucially, aionforge GATES it: PPR/expansion runs ONLY for the `MultiHop` and `Entity` query classes because "indiscriminate graph expansion measurably hurts simple single-hop precision while it helps multi-hop recall" [retrieval.md:108-118, 187-206]. [CONFIRMED.]

14. **CANDIDATE Q3-C1 (L, medium-conflict): add PPR-seeded ranking for the `impact` query mode, gated like aionforge.** Seed PPR on the subject symbol(s), spread over reverse `CALLS`/`IMPORTS` (and forward for neighborhood), weight teleport by existing static edge weights (config-defaults.ts:45-59) × `metadata.confidence` (the unwired rank-time hook from iter-1 finding 7). Order impact results by PPR score instead of raw edge enumeration order, so the most-central blast-radius callers surface first under the 400 ms budget rather than arbitrary truncation. [INFERENCE from retrieval.md:193-198 mapped onto code-graph-context.ts:627-671.] Conflict/risk: (a) must respect the existing latency budget — PPR over a large component can blow 400 ms, so it needs a bounded-iteration / power-method cap; (b) apply the aionforge lesson and GATE it to impact/multi-hop modes only, never the cheap neighborhood/outline default, to avoid hurting single-hop precision (retrieval.md:111-113); (c) intersect the PPR-reached set with the (future Q1-C1) `invalid_at IS NULL` current set so superseded edges do not pollute ranking. Marked L because it is genuinely new ranking machinery; a cheaper S/M precursor is to first wire `metadata.confidence` into a weighted sort of today's flat enumeration (Q4 territory) and measure whether PPR is even needed.
