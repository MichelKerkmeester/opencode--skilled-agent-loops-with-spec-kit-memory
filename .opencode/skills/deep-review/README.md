## 1. OVERVIEW

### Purpose

`deep-review` solves a problem single-pass review tools cannot: thorough, dimension-balanced code audits where context degradation across long sessions would otherwise bias the verdict. Each iteration dispatches a clean `@deep-review` agent that reads strategy and state from disk, picks one focus dimension, emits findings to JSONL, and exits. The orchestrating command checks a three-signal convergence vote between iterations and stops the loop only when severity-weighted findings stabilize, all required dimensions are covered, and three binary quality gates pass.

The agent is read-only against the review target. All state lives in a dedicated `{spec_folder}/review/` packet: an immutable config, an append-only JSONL log, a mutable strategy file, reducer-owned dashboards and registries, and per-iteration markdown notes. Recovery, pause-resume, and forensic replay all read from disk, not from memory.

When the review hands control back to general packet work, `/speckit:resume` rebuilds context in the canonical order (`handover.md`, `_memory.continuity`, then the rest of the packet docs). Memory artifacts generated by the loop stay secondary.

### Usage

Read this README to learn what the skill does, how to invoke it, how to read its outputs, and how to recover when something goes wrong. The runtime contract for the loop itself lives in `SKILL.md` and the four `references/` deep-dives. This README links into them at every point where a reader needs the next level of detail.

| Capability | `deep-review` | `sk-code-review` |
|---|---|---|
| Review pattern | Iterative multi-pass until convergence | Single pass |
| Context model | Fresh per iteration, state on disk | One context for the whole review |
| Severity model | P0 / P1 / P2 weighted convergence | Findings list without weighted stop |

### Key Features

- **Three-signal convergence vote** combining severity-weighted new findings, rolling stop average, and dimension coverage. STOP is legal only when all three vote together and the quality-gate bundle passes.
- **P0 override** forces at least one more iteration on any new P0, regardless of other signals.
- **Adversarial self-check** on every P0 before it enters the active findings registry.
- **Semantic coverage graph** persists per-iteration `graphEvents` so structural blind spots block premature STOP.
- **Resource-map emission** writes `{spec_folder}/review/resource-map.md` from converged delta evidence. Operators can disable per run with `--no-resource-map`.
- **Fail-closed reducer** refuses to write derived state when JSONL corruption is detected.
- **Auto-resume** detects existing state on re-invocation and continues from the last completed iteration.
- **Pause sentinel** halts between iterations when the operator creates `.deep-review-pause`.

## 2. QUICK START

**Step 1: Invoke the loop.**

Pick `:auto` for unattended runs or `:confirm` for approval gates between iterations:

```bash
# Autonomous audit of a skill package
/deep:start-review-loop:auto "skill deep-review"

# Interactive audit with approval at each step
/deep:start-review-loop:confirm "<spec-folder>"

# Autonomous audit of every spec folder in a track
/deep:start-review-loop:auto "track skilled-agent-orchestration"
```

Expected result: the YAML workflow creates `{spec_folder}/review/` and dispatches the first iteration.

**Step 2: Monitor or pause.**

Watch the dashboard between iterations:

```bash
cat {spec_folder}/review/deep-review-dashboard.md
```

Pause cleanly by creating the sentinel:

```bash
touch {spec_folder}/review/.deep-review-pause
```

Delete the sentinel to resume.

**Step 3: Read the verdict.**

After convergence (or max iterations), the loop writes the synthesized report:

```bash
grep -E "^Release Readiness Verdict|^## " {spec_folder}/review/review-report.md
```

Expected result: nine section headers plus a verdict line of `PASS`, `CONDITIONAL`, or `FAIL`.

## 3. FEATURES

### 3.1 FEATURE HIGHLIGHTS

The loop runs in four phases (initialization, iteration, synthesis, save) with state externalized to seven files in `{spec_folder}/review/`. Each iteration is a single LEAF dispatch that focuses on one review dimension, emits findings with file:line evidence, appends a JSONL delta, and updates the strategy file. The orchestrating command evaluates convergence between iterations using three signals (severity-weighted new findings, rolling stop average, dimension coverage) that vote together. STOP is legal only when the vote passes AND three binary quality gates (evidence, scope, coverage) plus the claim-adjudication gate all clear.

Every P0 finding runs through an adversarial Hunter / Skeptic / Referee self-check before it enters the active registry. New P0s force at least one more iteration regardless of other signals. The reducer fails closed on JSONL corruption, refusing to publish derived state until the contract is repaired. A semantic coverage graph keyed by `loop_type='review'` lets convergence cross-check structural blind spots against the dimension vote.

Synthesis dedups findings across iterations using content-hash plus file:line, reconciles claim-adjudicated severities, and writes the nine-section `review-report.md`. A convergence-time `resource-map.md` is emitted by default from converged delta evidence. Operators can disable per run with `--no-resource-map`. Auto-resume reads existing state on re-invocation and continues without re-running prior iterations.

| Priority | ID | Checks |
|---|---|---|
| 1 | `correctness` | Logic, off-by-one, return types, broken invariants, edge cases |
| 2 | `security` | Injection, auth bypass, secrets exposure, unsafe deserialization, trust boundaries |
| 3 | `traceability` | Spec/code alignment, checklist evidence, cross-reference integrity |
| 4 | `maintainability` | Codebase patterns, documentation quality, clarity, follow-on change cost |

### Target Types

| Type | Description |
|---|---|
| `spec-folder` | One spec folder + referenced implementation + tests + release artifacts |
| `skill` | Skill package: `SKILL.md`, references, assets, scripts, runtime integrations |
| `agent` | Named agent across runtime-specific definitions, with capability-parity surfaces |
| `track` | Spec track spanning multiple child folders + shared coordination artifacts |
| `files` | Explicit file set + immediate cross-references required to evaluate shipped behavior |

### Tuning Guide

| Goal | Adjustment |
|---|---|
| Deeper audit | Lower `--convergence` to 0.05, raise `--max-iterations` to 10 |
| Faster gate check | Raise `--convergence` to 0.15, lower `--max-iterations` to 4 |
| Security-focused audit | Pass `--dimensions correctness,security` to restrict scope |
| Skip the resource map | Pass `--no-resource-map` for the run |

### Security-Sensitive Override

Runs that target security, path handling, env precedence, schema boundaries, persistence, or shared policy automatically raise the bar: `minStabilizationPasses=2`, closed-finding replay is required, and the fix-completeness gate is enforced. STOP is not legal until the review report contains a closed-gate replay table that marks each prior active or remediated P0/P1 as `PASS`, `FAIL`, or `carried forward`, with file:line or command evidence. Full contract in [`references/convergence.md`](./references/convergence.md).

### Pause and Resume

Create a file at `{spec_folder}/review/.deep-review-pause` during a running loop to halt between iterations. Delete the file to resume. Auto-resume detects existing JSONL state on re-invocation and continues from the last completed iteration without re-running prior work.

## 7. TROUBLESHOOTING

| What you see | Likely cause | Fix |
|---|---|---|
| Loop stuck on one dimension | Strategy `next-focus` not advancing after the dimension completed | Read `{spec_folder}/review/deep-review-strategy.md`, if the dimension shows as complete, manually update `completed-dimensions` and re-run. If not, dispatch one more iteration. |
| P0 findings keep blocking convergence | Real P0s surviving adversarial check, or misclassified P1s elevated to P0 | Open the relevant `iterations/iteration-NNN.md`, read the Hunter / Skeptic / Referee output. If misclassified, downgrade severity in the JSONL entry and re-run convergence evaluation. |
| `review-report.md` missing sections | Synthesis step interrupted before all 9 sections wrote | Re-run synthesis manually, JSONL state is intact, synthesis reads from it without re-running iterations. |
| Loop runs to max iterations without converging | Convergence threshold too tight, or target has genuine structural issues | Raise `--convergence` to 0.15 and re-run with auto-resume. If findings keep appearing, the target needs remediation before review can converge. |
| Agent attempts a sub-agent dispatch | LEAF constraint violated in the agent definition | Verify `@deep-review` agent definition forbids the `Task` tool. Reload the runtime mirror file for the active CLI. |
| Verdict appears wrong | P0 severity misclassified or active-findings count incorrect | Confirm adversarial self-check ran on the P0 in the relevant iteration file. Check the JSONL `synthesis_complete` event for `activeP0` accuracy. |
| Reducer refuses to write derived state | JSONL corruption detected by `reduce-state.cjs` | Read the corruption-warning output. Repair the JSONL line, or re-run with `--lenient` to skip the corrupt entry, `corruptionWarnings` stays visible in reducer state. |
| `resource-map.md` not produced at convergence | `--no-resource-map` was passed, or `config.resource_map.emit == false` | Re-run without `--no-resource-map`, or flip the config flag and re-run synthesis. |

## 8. FAQ

**Q: How is `deep-review` different from `sk-code-review`?**

A: `sk-code-review` is a single-pass review. `deep-review` runs multiple iterations until convergence, tracks dimension coverage, runs adversarial self-checks on P0 findings, and produces a structured nine-section `review-report.md` with a three-way verdict. Use `sk-code-review` for quick feedback during active development. Use `deep-review` for release gates and audits where missed findings are expensive.

**Q: Can I review individual files?**

A: Yes. Set the target type to `files` and pass the explicit file list at invocation. The loop scopes all findings and cross-reference checks to the declared file set plus their immediate cross-references.

**Q: What triggers the P0 override?**

A: Any new P0 finding within an iteration sets `newFindingsRatio >= 0.50` regardless of other signals. This forces at least one more iteration. The override is a hard convergence guard and cannot be suppressed.

**Q: Does the review modify the target code?**

A: No. The `@deep-review` agent is strictly read-only against the review target. It writes only to the dedicated `{spec_folder}/review/` packet: strategy, JSONL state, dashboard, iteration files, the final report, and (by default) `resource-map.md`.

**Q: What happens after a FAIL verdict?**

A: Release is blocked. Run `/speckit:plan` to create a remediation plan from the findings in `{spec_folder}/review/review-report.md`, implement the fixes, then re-run `deep-review` to confirm resolution. The re-run uses auto-resume and continues from prior state.

**Q: What does CONDITIONAL mean?**

A: Active P0 is zero but active P1 findings remain. The code can proceed to planning and remediation but should not ship until P1s are resolved. Run `/speckit:plan` with the review report as input.

**Q: Can I pause a running autonomous review?**

A: Yes. Create a file at `{spec_folder}/review/.deep-review-pause` during the loop. The orchestrator checks for this sentinel between iterations and halts cleanly. Delete the file to continue from the last completed iteration.

**Q: How do I extend the review after convergence?**

A: Raise `--max-iterations` above the number of completed iterations and re-invoke. Auto-resume detects the existing JSONL state and continues without repeating prior iterations.

**Q: Where does the `graphEvents` data go?**

A: Each iteration appends `graphEvents` to its JSONL record. The coverage-graph reducer at [`.opencode/skills/deep-loop-runtime/lib/coverage-graph/coverage-graph-db.ts`](../deep-loop-runtime/lib/coverage-graph/coverage-graph-db.ts) ingests them under `loop_type='review'`, keyed by `(spec_folder, session_id, iteration)`. Convergence can then cross-check the graph for unresolved coverage gaps before allowing STOP.
