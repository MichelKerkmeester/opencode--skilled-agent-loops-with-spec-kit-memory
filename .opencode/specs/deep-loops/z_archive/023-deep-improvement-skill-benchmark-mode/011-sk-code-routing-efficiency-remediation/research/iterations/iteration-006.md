## Findings
1. **f-i6-01: Post-remediation D4 is still unmeasured, so the routine-usefulness lift is UNKNOWN.** The live-remediated report has aggregate `79` but `D4.score: null` and `status: "unscored-mode-a"` (`.opencode/skills/sk-code/benchmark/live-remediated/skill-benchmark-report.json:11-39`). The scorer hardcodes aggregate D4 as null/unscored for all reports (`.opencode/skills/deep-improvement/scripts/skill-benchmark/score-skill-benchmark.cjs:254-264`). The only saved D4 artifact is the live-final ablation with `D4_aggregate: 49`, `scoredCount: 2`, `total: 2`, and `attribution: "approximate"` (`.opencode/skills/sk-code/benchmark/live-final/d4-ablation.json:1-7`).

2. **f-i6-02: The current D4 ablation is not a direct usefulness instrument; it reuses a hallucination grader.** `d4-ablation.cjs` calls `grader.gradeD4` on skill-on and skill-off text, then computes `0.5 + (onScore - offScore) / 2` (`.opencode/skills/deep-improvement/scripts/skill-benchmark/d4-ablation.cjs:47-52`). The grader system prompt says its job is “D4 Hallucination” and tells it not to score correctness, paths, or planning structure (`.opencode/skills/deep-improvement/scripts/model-benchmark/scorer/grader/prompts/system-grader.md:3-4`, `:22-41`). Therefore LS-001’s stored `score: 0.435` means “skill-on had lower hallucination-grade output than skill-off,” not “the skill made the routine task less useful” (`.opencode/skills/sk-code/benchmark/live-final/d4-ablation.json:10-17`).

3. **f-i6-03: The live benchmark prompt prevents measuring what the model does with the slice.** The live executor reframes scenarios as “routing only,” says “do NOT edit files,” and asks for a fenced JSON block listing surface/resources/assets/agent (`.opencode/skills/deep-improvement/scripts/skill-benchmark/live-executor.cjs:35-52`). The playbook’s acceptance model, however, requires the AI’s user-visible outcome to be sound (`.opencode/skills/sk-code/manual_testing_playbook/manual_testing_playbook.md:43-49`, `:109-119`). Example: SD-002’s desired outcome is an actual TypeScript-aware fix plus verification evidence, while its routing test metadata says not to actually edit during routing tests (`.opencode/skills/sk-code/manual_testing_playbook/01--surface-detection/opencode-detection.md:46-49`, `:102-107`). Current D4 cannot see that outcome gap.

4. **f-i6-04: Current live D3 is also partly a measurement artifact because it scores stated routing, not actual observed exploration cost.** `parseLiveResult` sets `observedResources` to the union of stated JSON `resources` and stated JSON `assets`, while `observedReads` are stored only under `raw` (`.opencode/skills/deep-improvement/scripts/skill-benchmark/live-executor.cjs:149-168`). `scoreScenario` then computes D3 from `routerResult.resources`, `routedCount`, and expected-resource waste (`.opencode/skills/deep-improvement/scripts/skill-benchmark/score-skill-benchmark.cjs:102-118`). In live-remediated CS-001, the row scores D3 at `0.6667` from `routedCount: 15`, `wastedCount: 5`, but raw observed reads include broad `references/**/*.md`, `assets/**/*.md`, and `assets/motion_dev/snippets/*` globs (`.opencode/skills/sk-code/benchmark/live-remediated/skill-benchmark-report.json:320-368`). That means the current D3 gain is real for stated first-slice routing, but not yet proven real for actual model exploration cost.

5. **f-i6-05: The repository already has enough routine-task scenarios to fix D4’s n=2 problem without inventing a new corpus.** The playbook defines four OPENCODE language-sub-detection scenarios LS-001 through LS-004 (`.opencode/skills/sk-code/manual_testing_playbook/manual_testing_playbook.md:189-198`) and lists 24 total scenarios across seven categories (`.opencode/skills/sk-code/manual_testing_playbook/manual_testing_playbook.md:291-322`). LS-002 is a clean third routine candidate: “add a `--json-output` flag” to `skill_advisor.py`, expected `OPENCODE` + `PYTHON`, expected python/shared refs, expected python/universal assets, and no TypeScript/Shell/Config refs (`.opencode/skills/sk-code/manual_testing_playbook/02--language-sub-detection/opencode-python.md:14-39`). LS-003 and LS-004 provide shell/config follow-ups (`opencode-shell.md:12-36`, `opencode-config.md:12-36`).

6. **f-i6-06: A build-ready D4-R instrument should be action-scored, not recall-scored.** The current playbook already defines the needed acceptance axes: exact prompt, advisor/surface/resource evidence, loaded refs/assets, agent dispatch, and user-visible outcome (`.opencode/skills/sk-code/manual_testing_playbook/manual_testing_playbook.md:75-84`, `:109-125`). The D4-R harness should run skill-on/skill-off in a no-write “produce a minimal patch plan or unified diff plus verification commands” mode, then score: task-action correctness, surface/language contamination, verification fit, and hallucinated symbol/path risk. This would measure whether the slice helps the model produce a better routine answer, while D2/D3 continue measuring routing recall/efficiency.

7. **f-i6-07: The third routine scenario should be LS-002 first, not a synthetic new file.** LS-002 differs from LS-001 by language, target file, and expected reference family while staying routine and non-browser (`.opencode/skills/sk-code/manual_testing_playbook/02--language-sub-detection/opencode-python.md:16-39`). With current n=2, each scenario contributes 50 percentage points to the D4 mean; adding LS-002 drops that to 33.3 points per scenario. If the three scores were LS-001 `0.435`, CS-001 `0.55`, and a neutral LS-002 `0.50`, D4 would be `49.5`, effectively unchanged from the stored `49`; if LS-002 matched LS-001, D4 would fall to `47.3`. That shows the currently reported D4 magnitude is too sample-sensitive to guide remediation.

## Implications for D3/D4
- **D4 real-vs-artifact:** The stored `49` is mostly a measurement artifact for this question. It is sourced from a hallucination grader and two scenarios, not from a post-remediation routine-task usefulness run. The real post-remediation D4 is UNKNOWN until an action-scored D4-R suite is run.
- **D3 real-vs-artifact:** The stated-resource D3 improvement is real inside the current scorer, but actual exploration-cost improvement is not proven. Live rows can still include broad `glob` reads that the D3 formula ignores.
- **Expected D4 magnitude:** Adding LS-002 as a third routine probe will not by itself “lift” D4; it will make the estimate less volatile. The likely immediate gain is measurement honesty, not a score bump.
- **Concrete next build lever:** Add D4-R as a separate reported dimension or D4 subscore: `D4_hallucination_delta` for the existing grader and `D4_task_outcome_delta` for routine usefulness. Do not collapse them into one number until both are populated.
- **Routine suite proposal:** Run LS-001, LS-002, LS-003, LS-004, plus SD-002 in no-write patch-plan mode. Use CS-001 separately as a domain-pattern usefulness probe, not as evidence for routine OPENCODE usefulness.
- **Pass condition for a real D4 lift:** Skill-on must improve task-action correctness or verification fit versus skill-off, without simply naming more expected resources and without increasing observed broad globs. If the only improvement is better path naming or fewer hallucinated symbols, classify it as benchmark-fidelity improvement, not routine usefulness.
