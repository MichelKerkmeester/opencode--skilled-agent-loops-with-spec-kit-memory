## Findings
1. **Use an explicit resumable state machine: `DISCOVER -> VERIFY -> PARSE_VALIDATE -> COMMIT -> VECTOR_DRAIN -> PUBLISH`, with failures returning the style to the earliest safe phase.** Discovery sorts non-private directories and reads the crawl manifest; mtime/size/ctime are hints for candidate selection, but the existing engine rereads bytes, computes per-style content hashes, and rejects a corpus that changes between its data scan and verification fingerprint. Therefore a style is unchanged only when its recomputed aggregate hash matches committed state, not merely when mtimes match. [SOURCE: .opencode/skills/sk-design/styles/_engine/manifest.mjs:584-637] [SOURCE: .opencode/skills/sk-design/styles/_engine/manifest.mjs:640-705] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/storage/incremental-index.ts:283-321]
2. **Parse and validate outside the write lock, then atomically upsert all relational source rows plus one deterministic retrieval document inside one `BEGIN IMMEDIATE` transaction.** The transaction should replace normalized child rows, upsert `retrieval_documents`, set `aggregate_hash`, `retrieval_hash`, parser/schema versions, and mark the vector projection `pending` only after all constraints pass. External-content FTS triggers then update insert/update/delete rows in the same SQLite transaction; the scan mtime marker advances only after that commit, preserving automatic retry after parse, validation, or write failure. [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/storage/ports/contention-policy.ts:202-234] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/search/vector-index-schema.ts:399-432] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/handlers/memory-index.ts:1503-1509] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/handlers/memory-index.ts:1684-1692]
3. **Model vectors as an idempotent derived queue keyed by `(style_id, retrieval_hash, profile_key)`, not as part of source publication.** A unique queue key coalesces duplicate work; workers claim `pending/retry`, use bounded backoff, consult a cache keyed by content hash and active profile, and commit vector payload plus success status atomically. Before that commit they must compare the queued retrieval hash/profile with the current row and discard superseded results, a style-specific CAS guard required because the reference retry path otherwise selects by row id. Lexical retrieval remains available while vector work is pending or failed. 4. **Treat absence as a quarantined observation, then tombstone; reserve hard delete for explicit retention or a verified rebuild.** One scan adds an `absence_suspect` with first-seen generation/time. A later independently completed scan may tombstone only after every canonical path is known absent and the quarantine elapsed; unknown filesystem state fails safe, and reappearance clears the suspect. Tombstoning removes the style from active/FTS/vector retrieval in one transaction but retains identity, provenance, hashes, and deletion generation for rollback. Hard delete occurs only after a retention window, no live references, and at least one newer published full rebuild or operator-confirmed maintenance snapshot. 5. **Publish corpus generation in the same relational commit as the changed styles and tombstones; readers bind one SQLite snapshot and one generation.** The transaction writes an immutable `corpus_generations` row containing parent generation, crawl hash, deterministic aggregate over active `(style_id, aggregate_hash)`, counts, and completion time, then advances a singleton current-generation pointer. A query begins a read transaction, reads that pointer, and returns cards stamped with it; hydration requires the same generation and still verifies requested artifact hashes. This replaces per-query corpus rescans without weakening the current stale-generation refusal. 6. **A full rebuild must stage a complete generation-addressed database/shard, verify it, checkpoint WAL, and only then switch new readers.** Rebuild into sibling staging files from a stable corpus snapshot, validate schema/foreign keys/FTS integrity, active counts, aggregate generation hash, and vector coverage policy, then checkpoint/truncate WAL. Publish through an atomic pointer/rename after detaching old file bindings; the in-repo vector rebuild explicitly keeps the active shard untouched on partial failure, validates staged row count, atomically renames, and notes that an attached SQLite connection otherwise remains bound to the old inode. Existing readers finish on the old generation; new connections resolve the new pointer. [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/embedders/reindex.ts:484-530] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/embedders/reindex.ts:586-630] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/embedders/reindex.ts:661-705]
7. **Crash recovery is deterministic because durable phase evidence, not process memory, controls resumption.** A crash before relational commit leaves no source change and no mtime success marker; after commit but before vector completion leaves a searchable lexical row and durable pending job; during rebuild leaves only disposable staging artifacts; after file swap but before pointer/status completion is detected by reconciling staged/active metadata and generation hashes before serving. Write-lock helpers roll back failed bodies and failed commits, while the vector rebuild cleans staging and leaves the last-good active shard on cancellation/error. [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/storage/ports/contention-policy.ts:206-233] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/handlers/memory-index.ts:1503-1509] [SOURCE: .opencode/skills/system-spec-kit/mcp-server/lib/embedders/reindex.ts:764-780]
8. **Rollback is a generation-pointer operation first, data restoration second.** Keep the immediately previous generation-addressed DB/shard and manifest until the new generation passes query/hydration smoke checks and vector reconciliation. Rollback blocks new mutations, switches the durable pointer back to the verified prior generation, reopens reader connections, and leaves the failed generation quarantined for diagnosis; only if in-place source corruption occurred is checkpoint restore needed. This is why tombstones and prior files must outlive publication and why in-place rebuild of the active corpus is unsafe. ## Ruled Out
- In-place rebuilding of the active database or vector shard: a crash can expose partial state, and open SQLite attachments can remain bound to an orphaned old inode.
- Hard deletion after one missing-directory observation: transient or unknown filesystem state is not sufficient destructive evidence.
- Accepting a vector worker result by style id alone: content/profile changes can make an otherwise valid embedding stale.
- Publishing a generation in a second transaction after row mutation: readers could observe mixed old-generation metadata and new rows.
