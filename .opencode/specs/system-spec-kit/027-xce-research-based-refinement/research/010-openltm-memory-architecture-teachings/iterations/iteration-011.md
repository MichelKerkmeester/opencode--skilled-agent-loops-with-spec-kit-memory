## Mechanism

OpenLTM’s canonical memory is a SQLite `memories` row, not an authored document: DB-SPEC calls `memories` “the core table” for every durable insight from `learn`, GitLearn, EvaluateSession, or import, and the schema stores `content`, `category`, `importance`, `confidence`, `dedup_key`, `confirm_count`, `status`, embeddings/recall/supersession metadata as row columns (`docs/internal/DB-SPEC.md:43-46`, `packages/openltm-core/src/schema.sql:35-67`). Its markdown artifacts are projections/fallbacks: `context-summary.md` is “never the source of truth; the DB is,” and the generated memory dump says to edit via `/learn`, `/forget`, `/relate`, not directly (`docs/03-architecture.md:123-124`, `packages/openltm-core/src/db.ts:723-727`).

`learn()` is the row write path: it scrubs secrets before write/dedup (`packages/openltm-core/src/db.ts:304-314`), computes a content-normalized `dedup_key`, reinforces an existing row by incrementing `confirm_count`, refreshing `last_confirmed_at`, and raising `confidence`, or inserts a new row into `memories` (`packages/openltm-core/src/db.ts:317-347`, `packages/openltm-core/src/db.ts:352-366`). It then attaches tags/relations, writes provenance/audit rows, exports a generated markdown dump, queues/executes embedding generation, and auto-detects relations (`packages/openltm-core/src/db.ts:370-405`). The `WriteQueue` serializes queued DAO writes through a promise tail and optional `BEGIN IMMEDIATE`, but `learn()` itself performs direct `UPDATE`/`INSERT`; embedding DAO writes do use the queue (`packages/openltm-core/src/lib/writeQueue.ts:10-24`, `packages/openltm-core/src/dao/embeddings.ts:26-52`).

Subsystem filter: recall is **DOC-ANALOG** because FTS/semantic retrieval transfers, but `last_recalled_at`/`recall_count` row mutations must become index metadata (`packages/openltm-core/src/db.ts:425-461`, `packages/openltm-core/src/db.ts:552-560`). Decay/janitor are **DOC-ANALOG** for rank freshness, but row `status='deprecated'`, auto-promotion, archive, and dedup actions are row-coupled mutations (`packages/openltm-core/src/janitor/decay.ts:14-38`, `packages/openltm-core/src/janitor/promote.ts:40-100`). Hooks are split: session injection/snapshots **TRANSFER**, while transcript/git fact mining into `learn()` rows is **ROW-COUPLED** (`hooks/src/SessionStart.ts:213-237`, `hooks/src/GitCommit.ts:81-97`, `hooks/lib/llmExtract.ts:50-62`). Graph is **DOC-ANALOG**: typed edges transfer, but endpoints must be document anchors instead of memory row IDs (`packages/openltm-core/src/schema.sql:95-104`, `packages/openltm-core/src/graph.ts:51-62`). Secrets are **TRANSFERS** (`packages/openltm-core/src/secretsScrubber.ts:17-105`). Provenance is **DOC-ANALOG**; row audit snapshots are row-coupled (`migrations/008_add_memory_provenance.sql:6-23`, `migrations/009_add_memory_audit.sql:6-20`). Embeddings **TRANSFER** as a derived side index, already split from the hot row (`migrations/010_memory_embeddings_split.sql:5-25`, `packages/openltm-core/src/embeddings.ts:261-301`). Learn/reinforce is **ROW-COUPLED**.

## Row-coupled findings (negative knowledge)

- `learn()/reinforce`: dedup, `confirm_count`, `confidence`, and `status` mutate a memory row, so there is no authored source document to preserve.
- Auto-mining via GitCommit/EvaluateSession: extracted facts become `learn()` rows, which conflicts with deliberate `/memory:save` authored-doc creation.
- Row audit/time travel: `before_json`/`after_json` snapshots reconstruct row state, not document revision history.
- Janitor auto-promotion/dedup/deprecation: it creates pending rows and mutates row status; only the ranking/health ideas should be adapted.
- Per-row recall telemetry: useful as index metadata, but should not mutate canonical spec documents on every retrieval.

## Open questions

- Should doc-based recall telemetry live only in the derived index, or should any access signal ever update continuity frontmatter?
- Should auto-mined transcript/git insights be rejected entirely, or converted into human-reviewable proposed doc-save drafts?
