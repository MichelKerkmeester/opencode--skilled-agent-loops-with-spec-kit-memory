# v1.5.0.0 - Smarter Mobbin/Refero reference routing

**Released:** 2026-06-17

The skill now decides for itself whether to consult a real-world reference (Mobbin or Refero), instead of treating them as a purely passive ON_DEMAND resource. The decision is a hybrid: take the initiative when it clearly helps and a subscription is connected, ask the user when it is borderline, and fall back to the generic process otherwise.

---

## What Changed

### A decision gate in the reference doc

`references/design-grounding/design_references_mcp.md` §3 gained a "Deciding whether to consult (initiative or ask)" gate:

- **Does it help?** A reference is worth pulling when the brief sits in a convention-heavy category where the real-world default is strong: checkout, onboarding/auth, settings, dashboards, social feeds, pricing, data tables, calendars, messaging, search/filters. It helps little for novel/bespoke briefs or a pinned brief.
- **Initiative or ask:** take the initiative (pull ONE reference, name the default, cite the URL) when the category benefits AND a subscription is connected; ask the user one line when borderline or subscription status is unknown; fall back to the generic anti-default process otherwise (non-blocking).
- **Source pick by surface:** Mobbin for native/iOS or in-app screens and flows; Refero for web pages and visual-style direction (styles first).

### Surfaced in routing and rules

- `SKILL.md` §2 reframed the Mobbin/Refero resource row from `ON_DEMAND` to `INITIATIVE / ASK` with the source split.
- `SKILL.md` §4 added ALWAYS rule 7: decide at the critique step whether a real-world reference would sharpen the default, by initiative or by asking.

---

## Notes

- The existing guardrails are unchanged: one reference only, read live, never copied, never a chooser, grounding stays upstream and non-negotiable, and a pinned brief always wins.
- The tool catalogs (`mobbin_tools.md`, `refero_tools.md`) are unchanged; only the routing logic was added.
