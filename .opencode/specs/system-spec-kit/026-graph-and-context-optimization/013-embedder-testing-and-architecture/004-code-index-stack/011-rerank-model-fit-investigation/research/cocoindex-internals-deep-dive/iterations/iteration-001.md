# Iter 1 — CocoIndex retrieval pipeline map — entry → hybrid → rerank → output

## TL;DR (2-3 sentences)

The CocoIndex Code retrieval path is daemon-mediated: CLI and MCP requests both end up as `SearchRequest` IPC messages, then `ProjectRegistry.search()` calls `query_codebase()` against `.cocoindex_code/target_sqlite.db`. The core ranking pipeline is query embedding → dense vec0 retrieval → optional FTS5 lexical retrieval → weighted RRF fusion → dedup/heuristic boosts → optional rerank of a configured head window → final pagination/output projection. The most exposed failure surfaces are not only the reranker; chunking, embedding prompt/model, FTS tokenizer/query normalization, RRF weights/window sizing, dedup, and result formatting can all move relevant code above or below the final cut.

## Question (restate the scoped RQ)

Map the full CocoIndex retrieval pipeline by reading the fork code under `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/`: trace entry → tokenize/embed → dense vector search → FTS5 lexical search → fusion (RRF) → top-K cutoff → rerank → output. For each stage, cite file, function, and line range, and identify all failure-surface candidates: chunking, embedder, FTS5 tokenizer, fusion weights, top-K window, reranker, and output formatting.

## Findings (numbered, with citations)

1. Entry has two front doors, but both converge on daemon search. CLI `ccc search` creates `query_str`, optional path/language filters, and calls `_search_with_wait_spinner()` before printing results; MCP `search` exposes the same semantic search contract, optional refresh, filters, limit, and offset, then calls `client.search()` in an executor. The failure surface here is request shaping: default CWD-derived path filters, user-provided `limit`, `offset`, and `paths`, and whether refresh runs before search can change what reaches ranking. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/cli.py:235-269`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/cli.py:381-416`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/server.py:129-231`.

2. Search transport is IPC, not an in-process query from the CLI. `DaemonClient.search()` encodes `SearchRequest`, sends it to the daemon, handles wait notices, and returns `SearchResponse`; daemon `_dispatch()` either waits for load-time indexing or calls `ProjectRegistry.search()` directly. The failure surface is latency and stale index state: a request can wait on indexing, and the daemon decides whether search observes loaded project state or opens the target DB read-only. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/client.py:125-180`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/daemon.py:737-760`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/daemon.py:783-804`.

3. Chunking and document embedding happen during indexing, before retrieval. `process_file()` reads text, skips decode failures and empty files, detects language, classifies `path_class`, uses configurable chunk size/min/overlap with `RecursiveSplitter`, declares rows containing content, line ranges, content hash, path class, and `await embedder.embed(chunk.text)`, then populates FTS for each chunk. This makes chunking a first-order failure surface: if the relevant symbol is split badly, too small, too large, filtered by path matchers, misclassified by language, or assigned the wrong path class, all downstream dense/FTS/rerank stages inherit that error. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:33-39`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:245-317`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:320-365`.

4. Embedder selection affects both stored chunk vectors and query vectors. `Project.create()` provides the same embedder to the indexing/query environment; `shared._build_embedder()` selects SentenceTransformers versus LiteLLM and sets `query_prompt_name`; `query_codebase()` embeds the user query with `await embedder.embed(query, query_prompt_name)`. Failure candidates include model family, query prompt name, device fallback, LiteLLM/Ollama availability, and query/document prompt asymmetry. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/project.py:104-122`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/shared.py:143-186`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:579-595`.

5. Dense vector retrieval has two modes: vec0 KNN when filters allow it, and full scan when path filters are present. `_knn_query()` and `_knn_query_with_ids()` use `embedding MATCH ? AND k = ?`, optionally partitioned by language; `_full_scan_query*()` computes `vec_distance_L2(embedding, ?)` with SQL-level path/language filtering. The failure surface is the candidate window: `fetch_k = (limit + offset) * 4`, so a relevant chunk excluded from the first `fetch_k` vector/filtered candidates cannot be rescued later unless FTS retrieves it in hybrid mode. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:109-245`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:533-553`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:594-655`.

6. FTS5 lexical search is a separate recall lane with its own tokenizer and query normalization. The FTS table uses SQLite `unicode61 remove_diacritics 2`; the app also tokenizes the user query through a Python regex allowing letters, digits, underscore, dot, slash, colon, and hyphen, then joins quoted tokens with `OR`; `query_fts()` orders by `-bm25()`. Failure candidates include regex token loss, OR broadening, tokenizer mismatch against code punctuation, BM25 score distribution, path/language filters, and FTS sync drift. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:25-37`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:64-85`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:88-132`.

7. Hybrid fusion is weighted RRF over the vector and FTS ranked lists, then heuristic boosts are applied. `query_codebase()` builds `RankedRow` lists for vector and FTS lanes, calls `rrf_fuse()` with env-configured `hybrid_rrf_k`, `hybrid_vector_weight`, and `hybrid_fts5_weight`, fetches missing FTS-only records, then `_dedup_and_rank_hybrid_rows()` applies implementation-intent and canonical-resource boosts/penalties. Failure candidates include RRF `k`, channel weights, tie-breakers, min-max audit scores not used for score composition, path-class heuristics, canonical boosts, and dedup collapsing mirrored aliases. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:399-444`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:481-515`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:613-674`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py:42-87`.

8. The reranker is optional and only sees a bounded head window. In hybrid mode, `query_codebase()` calls `reranker.rerank()` only when `config.rerank_enabled` is true and passes `top_k=config.rerank_top_k`; `CrossEncoderRerankerAdapter.rerank()` loads the model lazily, scores `(query, candidate.content)` pairs for the first `top_k`, applies optional path-class multipliers, sorts the head by reranker score, and appends the tail unchanged. Jina v3 follows the same head/tail contract but uses `model.rerank(query, docs)` and has a document-character clipping knob. Failure candidates include model load fallback, RAM gate, model family, adapter selection, top-K too small, document truncation, path-class score multiplication, and score logging side effects. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:675-683`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/reranker.py:20-47`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/reranker.py:98-183`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/rerankers_jina_v3.py:127-190`.

9. Output formatting can hide or expose ranking telemetry, and the final cutoff happens after rerank. `_window_results()` slices candidates with `offset : offset + limit` after dedup/rerank; daemon and MCP/CLI surfaces project the same fields into protocol/Pydantic/Typer outputs, including `rankingSignals`, `fts5_score`, `rrf_score`, `pre_rerank_score`, and `reranker_score`. The failure surface is diagnostic, not only ranking: if callers ignore telemetry or if output parsing extracts top paths incorrectly, a bench can misclassify why a result won or lost. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:518-530`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:690-711`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/protocol.py:96-119`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/cli.py:160-192`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/server.py:205-231`.
