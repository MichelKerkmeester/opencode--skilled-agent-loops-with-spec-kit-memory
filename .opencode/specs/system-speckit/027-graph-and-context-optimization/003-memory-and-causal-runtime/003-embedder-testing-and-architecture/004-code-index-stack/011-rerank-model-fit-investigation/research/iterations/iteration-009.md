# Iter 9 — Quantitative Decision Rubric + Stacking Analysis

## TL;DR (2-3 sentences)

Path-class boost remains the highest-ranked first move under a quantitative rubric because it directly targets the two cleanest failures, costs little, and adds effectively no inference latency. Once implementation cost and latency uncertainty are included, **mxbai becomes the more practical model-swap experiment than Qwen3**, even though Qwen3 still has the stronger semantic-quality prior. Fusion recalibration is useful only as an instrumented side experiment, and prompt injection into the current bge reranker should be rejected.

## Question (restate the scoped RQ)

Build a quantitative decision rubric across iterations 1-8. For each candidate, score: (a) probability of fixing at least two of four failure probes, (b) probability of no regression on control hits, (c) implementation cost, (d) inference latency delta in milliseconds, and (e) maintenance burden. Produce an ordered recommendation and a stacking analysis that identifies complementary versus redundant combinations.

### 1. Quantitative rubric: fix probability is necessary but not sufficient

The fix-probability column uses Iter 8's adversarial estimates as priors: path-class boost 0.72, Qwen3 0.66, mxbai 0.58, fusion 0.24, and bge prompt injection 0.06. Those numbers are not replay results; they are estimates derived from prior evidence and should be treated as decision priors until probe replay exists. <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/iterations/iteration-008.md" lines="96-106" /> <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/iterations/iteration-002.md" lines="84-98" /> <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/007-ollama-and-bge-promotion/003-bge-code-v1-confirmation-and-promote/pre-confirmation-margin-analysis.md" lines="45-70" />

Latency score is 1.0 for no extra inference work, 0.8 for published faster-but-local-unknown, and 0.4 for local latency unknown. Maintenance score is Low=0.8, Medium=0.6, High=0.3. These scores are intentionally coarse because the control set and local latency are not measured yet. <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/iterations/iteration-005.md" lines="97-105" /> <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/iterations/iteration-004.md" lines="101-108" /> <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/iterations/iteration-006.md" lines="132-153" />

### 2. Ordered recommendation: path-class first, then mxbai and Qwen3 in a replay matrix

The first recommended action is **path-class boost behind a flag or config**, not a blind production default. ### 3. Stacking analysis: model swap plus path-class is complementary; model swap plus model swap is redundant

The strongest stack is **better reranker + conservative path-class post-processing**. The model swap attacks the semantic scoring failure; the path-class layer adds metadata the model currently does not see. This is complementary because current reranking uses only `(query, candidate.content)` and then sorts by reranker score, while `path_class` is already available on each result. <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/reranker.py" lines="126-150" /> <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/schema.py" lines="24-40" /> <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py" lines="55-93" />

| Combination | Relationship | Rationale | Recommendation |
|---|---|---|---|
| Path-class + mxbai | Complementary | mxbai changes relevance scoring; path-class adds metadata-aware guardrails | Test after measuring each alone |
| Path-class + Qwen3 instruction | Complementary | Qwen3 can receive task intent; path-class handles residual path-role mistakes | High-upside stack, but more moving parts |
| mxbai + Qwen3 | Redundant | Both are alternative reranker replacements for the same stage | Do not stack except as offline comparison |
| Fusion recalibration + any reranker fix | Weakly complementary | Fusion shapes candidate set before rerank but does not fix reranker scoring | Instrument first; stack only if channel logs justify it |
| Prompt injection + bge-v2-m3 | Not useful | Current bge is not instruction-tuned, so prefixing is noise, not control | Reject |
| Prompt instruction + Qwen3 | Complementary | Qwen3 is instruction-aware; this is the model family where prompt guidance belongs | Include in Qwen3 replay lane |

Fusion is the main "maybe" stack: it could remove a distractor from the top-K window before rerank, but there is no dense-vs-BM25 attribution for the failure probes. The implementation exposes RRF channel ranks and scores, and config exposes weights, so the right next step is logging and replay rather than choosing dense-heavy or sparse-heavy by guesswork. <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py" lines="42-87" /> <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/config.py" lines="310-340" /> <ref_file file="/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/iterations/iteration-006.md" lines="132-153" />

### 4. The two unresolved fixture cases limit confidence in every candidate

The rubric should not reward a candidate too much for "fixing all four" until probe 10 and probe 18 are audited. Probe 10 expects a generated `.js` artifact even though the TypeScript source is arguably more semantically correct; probe 18 expects a test while a reference doc wins, so an implementation-favoring rule is not enough.
