# Iteration M3 - on-write vs retroactive vs automated mechanisms (opus, cross-cutting)

## TITLE

Cohort M3. Model opus via claude2. Wave-3 mechanism design that maps every confirmed recommendation onto a shared on-write generator plus validate.sh gate plus retroactive backfill slot, with re-embeds gated behind the active-shard coverage guard, and a fourth async slot for an LLM-as-judge content scorer.

## FINDINGS

Two findings sharpen the picture. `community-search.ts` already implements GraphRAG-style rollup summaries but as a keyword-only fallback channel gated `SPECKIT_COMMUNITY_SEARCH_FALLBACK` that fires only on weak or zero primary results, and `causal_summary` already exists in every `graph-metadata.json` but is consumed only by the parser and schema, never embedded or searched. Both confirm the candidate-6 thesis. Every confirmed recommendation decomposes into the same three-slot mechanism, an on-write generator where the field is born, a `validate.sh` strict gate where it is enforced, and a retroactive backfill sweep where the 2022 existing files catch up, with re-embeds gated behind the active-shard coverage guard (`embedding-reconcile.ts:376`). The two retrieval wins additionally need a cache-key version bump (`embedding-cache.ts:397-416`) so the re-embed is forced and reversible. The corpus proves the pattern is viable, `graphMetadataSchema` is generated, parsed (`generate-context.ts:562`) and pluggably validated, but description.json and frontmatter have no schema, `causal_summary` is born metadata-only, and community rollups are keyword-only, so the work is wiring what half-exists into the prod write plus gate plus index path not inventing infrastructure.

## CONCRETE CHANGE

Mechanism 1 schema-validation gate adherence and logic. `graphMetadataSchema` exists and is enforced in code (`graph-metadata-schema.ts:61`, parsed at `generate-context.ts:562`) but `validate.sh` only checks the boolean `data.legacy_grandfathered === true` (`validate.sh:175-183`) and never structurally parses either JSON, and description.json has no zod schema. On-write extend `generate-context.js` to construct and parse a new `descriptionSchema` (new `lib/description/description-schema.ts`) before `atomicWriteJson`. Gate add a `run_schema_validation_check()` strict validator to `validate.sh` following the existing plug pattern that emits tab-separated `rule/status/message/detail` dispatched from `run_strict_validators()` (`validate.sh:828-836`, mirroring `run_continuity_freshness_check` at `validate.sh:716` and `run_evidence_marker_lint_check` at `validate.sh:772`). Tier graph and description to hard error under strict after a one-time sweep, frontmatter to warn-then-error staged because the legacy spec.md files carry hand-authored frontmatter, respecting `detect_legacy_grandfathered` exactly as `validate.sh:927` branches on strict and not grandfathered. No re-index, this is the cheapest win.

Mechanism 2 EARS plus three-tier constraint template adherence and logic. The L2 template (`templates/examples/level_2/spec.md:85-106`) is an ANCHOR-delimited bare ID and Requirement and Acceptance table with no EARS shapes and no always or ask-first or never block. Reshape the requirement column inside the requirements anchor to EARS patterns and add a constraints anchor with the three tiers, L2 plus required and L1 optional. Add a warn-tier linter using the same `run_*_check` plug that flags requirement rows not matching an EARS regex and a missing constraints anchor, warn-only because no format guarantees adherence. Add a `tasks-cover-requirements` cross-artifact analyze gate where every REQ-id in spec.md must be referenced by at least one task in tasks.md. No re-index on write but the EARS content change shifts the content_hash so the next save re-embeds naturally.

Mechanism 3 deterministic header-path prefix retrieval, highest-ROI and externally validated. The embed path destroys structure before vectorizing, `content-normalizer.ts:216-231` runs stripYamlFrontmatter, stripAnchors, stripHtmlComments, stripCodeFences, normalizeMarkdownTables, normalizeMarkdownLists then normalizeHeadings so a header collapses to its bare text and the header path is gone before the SHA-256, and the cache key (`embedding-cache.ts:397-416`) is `(content_hash, profile_key, input_kind, model_id, dimensions)` over normalized content with no chunk-strategy version. On-write prepend a deterministic header-path plus global-id prefix to each chunk injected upstream of normalizeHeadings in `content-normalizer.ts` so the path is captured from the live heading stack. Add `chunk_strategy_version` to the cache-key tuple at `embedding-cache.ts:397-416` so the re-embed is forced and rollback is a version decrement. Re-index is a one-time bulk re-embed gated behind the active-shard coverage guard, `embedding-reconcile.ts:376` throws `ActiveShardGuardError` unless `verifyActiveShard` passes in apply mode, so rollout is dry-run reconcile then verify coverage then apply and a partial re-embed fails closed.

Mechanism 4 rollup nodes logic and retrieval, conditional. Two rollup artifacts already exist but neither competes, `causal_summary` is written into every `graph-metadata.json` but consumed only by the parser and schema, and `community-search.ts` implements community summaries only as a keyword fallback gated `SPECKIT_COMMUNITY_SEARCH_FALLBACK`. Both are keyword-only so they match one channel, `channelCount=1`, earning zero cross-channel bonus, `hybrid-search.ts:879` computes `crossChannelBonus = min(0.06, 0.02*(channelCount-1))` which is the real mechanism behind the paraphrased 0.02 multiplier, there is no separate penalty constant, the penalty is the forgone bonus that embedded records earn. Materialize the rollup as a first-class embedded `memory_index` record not a fallback string, v1 embeds the existing `causal_summary` for zero LLM cost so it earns a vector score plus the cross-channel bonus. Add a broad-vs-specific query gate so rollups surface for broad queries and stay out of precise lookups. v2 is LLM-aggregated child summaries only if v1 broad-query lift justifies the token cost.

Mechanism 5 new metadata fields retrieval and logic. `graphMetadataDerivedSchema` (`graph-metadata-schema.ts:40-59`) has `created_at`, `last_save_at` and `last_accessed_at` but no embedding-provenance block and no shelf-life or stale-after temporal class. Add an `embedding` block of `{model_id, dimensions, chunk_strategy_version, content_fingerprint, embedded_at}` and a `temporal` block of `{last_verified, shelf_life_class, stale_after}`, born on-write in `generate-context.js`. Add an automated drift-triggered re-embed, a scheduled sweep recomputes each chunk fingerprint vs the stored `content_fingerprint` and enqueues a re-embed when cosine drift exceeds 0.05 or `content_hash` changed, gated behind the coverage guard, making staleness visible instead of a silent failure. Both blocks must be `.optional()` exactly as `last_active_child_id` at `graph-metadata-schema.ts:57` until backfill populates them.

Cross-cutting LLM-as-judge quality scorer retrieval and adherence and logic. `eval-quality-proxy.ts` is a deterministic retrieval-quality proxy scoring search results not document content, there is no per-doc content judge, but the host infra exists including `eval-db.ts`, `eval-logger.ts`, `shadow-scoring.ts`, `reporting-dashboard.ts` and a working `backfill-research-metadata.ts` template. Add a new `lib/eval/quality-judge.ts` scoring each doc 1 to 5 Likert on clarity, completeness, requirement-coverage and freshness with a one-line rationale, the deterministic proxy stays as a cheap pre-filter and the judge runs on flagged packets or a sampling cadence. Store `derived.quality_score`, `quality_rationale`, `scored_at` and `scorer_model` in `graph-metadata.json` via Mechanism 5 schema, fire the judge async after `generate-context.js` completes so it never blocks the save and writes metadata-only with no re-embed. Backfill via a `backfill-quality-scores.ts` cloned from `backfill-research-metadata.ts` with metadata-only re-index. The score enters ranking as a fusion boost reusing the exact pattern at `stage2-fusion.ts:278` where `specLevel` contributes `(specLevel-1)*0.02` capped at 0.06, and a low score flags rework and surfaces in `reporting-dashboard.ts`.

## READER

Mechanism 1 write-time author plus retrieval through clean trigger_phrases and key_topics. Mechanism 2 the AI agent consuming the spec for adherence plus logic. Mechanism 3 retrieval via vector plus the cross-channel bonus path. Mechanism 4 logic because the rollup is the packet thesis plus retrieval on broad queries. Mechanism 5 retrieval via freshness prior and drift trigger plus logic via provenance. LLM-judge all three, retrieval boost, adherence rework flag, logic rationale.

## ON-WRITE OR RETROACTIVE

Mechanism 1 on-write born-valid plus retroactive one-time validate sweep. Mechanism 2 on-write template plus soft retroactive linter warning. Mechanism 3 on-write new chunks plus retroactive one bulk re-embed of the existing corpus behind the coverage guard. Mechanism 4 on-write embed the already-born `causal_summary` plus retroactive backfill-embed of existing causal_summaries, incremental re-embed only. Mechanism 5 on-write new derived fields plus automated drift sweep. LLM-judge on-write async post-save plus retroactive metadata-only backfill. The decisive ordering ships Mechanism 1 first for zero re-index and to unblock Mechanism 5 schema fields, Mechanism 4 v1 second as the cheapest incremental retrieval win, Mechanism 3 last among retrieval as the most expensive full re-embed, with Mechanism 2 and the LLM-judge riding alongside as warn-tier and async.

## RISK

Mechanism 1 legacy-frontmatter blast radius is real, flipping frontmatter to error before the sweep fails closed on grandfathered packets, the gate must respect `detect_legacy_grandfathered`. Mechanism 2 structure alone cannot guarantee adherence, keep the EARS regex warn-tier so legitimate prose is not punished. Mechanism 3 the bulk re-embed is the single most expensive action in the packet, the cross-corpus magnitude is unproven on this index, gate apply behind the coverage guard, measure HitRate against the eval fixtures before and after, keep `chunk_strategy_version` decrementable for instant rollback. Mechanism 4 is conditional, real only if embedded first-class, breadth-gated and v1 cheap-concat measured before paying for v2, the mirage to avoid is leaving it as the keyword fallback and claiming a win. Mechanism 5 the 0.05 drift threshold is borrowed from the source not tuned to this corpus, treat as a hypothesis. LLM-judge cost and non-determinism mean it must be sampled not run synchronously every save, store `scorer_model` and `scored_at` for re-score on model change, and validate the quality boost does not regress HitRate on eval fixtures before enabling. Externally validated is the metadata-fusion and header-path effect plus the deterministic facts that the prod embed path strips headers, that `causal_summary` and community rollups are keyword-only and that the 3-result floor taxes count not vector quality, hypothesis is the specific HitRate magnitude of each re-embed on this corpus, the 0.05 threshold and the quality-boost weight, and every re-embed is reversible via `chunk_strategy_version` and fails closed via the active-shard guard so the cost of being wrong is bounded.
