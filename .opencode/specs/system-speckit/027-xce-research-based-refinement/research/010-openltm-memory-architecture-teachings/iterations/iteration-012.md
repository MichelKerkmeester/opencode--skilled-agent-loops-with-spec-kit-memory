## Mechanism
OpenLTM’s closest “document-ish” continuity layer is still row-first. Its schema states a two-table design where `context_items` are per-project context and `memories` are learned insights, and `context_items` explicitly “replaces the 4 per-project Markdown context files” (`packages/openltm-core/src/schema.sql:1`, `packages/openltm-core/src/schema.sql:15`). The rows hold `goal|decision|progress|gotcha`, content, session ID, permanence, optional promoted `memory_id`, and status (`packages/openltm-core/src/schema.sql:17`, `packages/openltm-core/src/schema.sql:20`, `packages/openltm-core/src/schema.sql:23`, `packages/openltm-core/src/schema.sql:24`, `packages/openltm-core/src/schema.sql:25`, `packages/openltm-core/src/schema.sql:26`). By contrast, durable `memories` rows carry category, importance, confidence, dedup key, confirmation counters, embeddings, recall metadata, and status (`packages/openltm-core/src/schema.sql:35`, `packages/openltm-core/src/schema.sql:39`, `packages/openltm-core/src/schema.sql:41`, `packages/openltm-core/src/schema.sql:42`, `packages/openltm-core/src/schema.sql:45`, `packages/openltm-core/src/schema.sql:47`, `packages/openltm-core/src/schema.sql:48`, `packages/openltm-core/src/schema.sql:51`, `packages/openltm-core/src/schema.sql:56`). So `context_items` are rows, not authored docs; they differ from `memories` mainly by being short-term, project-scoped continuity rows with mixed lifecycle and optional promotion.

The row semantics are explicit: `goal` deletes/reinserts one row, `decision` and `gotcha` append permanent rows, and `progress` deduplicates by `session_id` (`packages/openltm-core/src/context.ts:42`, `packages/openltm-core/src/context.ts:58`, `packages/openltm-core/src/context.ts:67`, `packages/openltm-core/src/context.ts:74`). The DAO repeats this with queued row writes and trims progress to the 20 most recent entries (`packages/openltm-core/src/dao/contextItems.ts:23`, `packages/openltm-core/src/dao/contextItems.ts:36`, `packages/openltm-core/src/dao/contextItems.ts:43`, `packages/openltm-core/src/dao/contextItems.ts:55`). `context-summary.md` is generated from those rows as a human-readable snapshot and fallback, not truth (`packages/openltm-core/src/context.ts:132`, `packages/openltm-core/src/context.ts:148`, `docs/03-architecture.md:123`, `docs/03-architecture.md:124`, `docs/03-architecture.md:411`, `docs/03-architecture.md:413`).

Session injection is more portable. `SessionStart` caps injected context at 60 lines, regenerates `context-summary.md` from DB, reads it, trims it, and emits a status-bearing session block plus optional LTM memories, conflicts, and backfill hints (`hooks/src/SessionStart.ts:17`, `hooks/src/SessionStart.ts:182`, `hooks/src/SessionStart.ts:204`, `hooks/src/SessionStart.ts:205`, `hooks/src/SessionStart.ts:224`, `hooks/src/SessionStart.ts:226`, `hooks/src/SessionStart.ts:237`). The LTM subsection is row-backed but conceptually analogous to indexed spec-doc retrieval: it embeds the summary prefix, fetches similar global/project memories, optionally adds graph insights, and truncates the section (`hooks/src/SessionStart.ts:37`, `hooks/src/SessionStart.ts:40`, `hooks/src/SessionStart.ts:41`, `hooks/src/SessionStart.ts:49`, `hooks/src/SessionStart.ts:51`, `hooks/src/SessionStart.ts:62`, `hooks/src/SessionStart.ts:63`). UNKNOWN: docs say the block is `## Restored Project Context`, but the source I read emits an `## LTM Session` status line (`docs/03-architecture.md:243`, `docs/internal/PRD.md:199`, `hooks/src/SessionStart.ts:224`).

`PreCompact` is the strongest doc-analog: it writes a compact Markdown snapshot before compaction, first from DB rows and then from legacy markdown files if DB is unavailable (`hooks/src/PreCompact.ts:12`, `hooks/src/PreCompact.ts:17`, `hooks/src/PreCompact.ts:23`, `hooks/src/PreCompact.ts:29`, `hooks/src/PreCompact.ts:38`, `hooks/src/PreCompact.ts:56`, `hooks/src/PreCompact.ts:61`). The hook docs frame this as automatic lifecycle continuity: `SessionStart` injects context, `UpdateContext` saves progress to `context_items`, `EvaluateSession` extracts patterns into `memories`, and `PreCompact` snapshots `context-summary.md` (`docs/06-hooks.md:3`, `docs/06-hooks.md:15`, `docs/06-hooks.md:16`, `docs/06-hooks.md:17`, `docs/06-hooks.md:18`).

## Row-coupled findings (negative knowledge)
`context_items` as canonical continuity storage should not be ported: it replaces Markdown with rows, while system-spec-kit’s authored docs are the canonical memory.

Goal/progress row mutation rules should not be ported: delete/reinsert, session-ID dedup, and trim-by-row-ID are row lifecycle policies, not document authorship policies.

Decision/gotcha promotion into `memories` should not be ported: it depends on `memory_id`, `learn()`, and row status transitions.

Automatic `UpdateContext` progress capture should not be ported directly: it writes transcript-derived progress rows automatically, while system-spec-kit memory writes are deliberate save/document updates.

FTS triggers over `context_items` should not be copied as source-of-truth behavior: indexing transfers, but only as a derived index over authored docs.

## Open questions
The docs/source mismatch on the literal injection heading needs synthesis: OpenLTM’s architecture promises `Restored Project Context`, while current source emits `LTM Session`.

The right system-spec-kit budget is unresolved: OpenLTM’s 60-line context cap and 30-line LTM cap are useful defaults, not proven fit.

The best place for pre-compact continuity in system-spec-kit remains a policy question: `handover.md`, `_memory.continuity`, or MCP-only session bootstrap snapshot.
