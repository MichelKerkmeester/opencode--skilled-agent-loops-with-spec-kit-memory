## [**1.0.1.0**] - 2026-07-12

Smart-routing section-contract conformance. The section-1 header previously merged discovery and routing as `WHEN TO USE + SMART_ROUTING`, where the underscore prevented the canonical section matcher from recognizing a distinct `SMART ROUTING` heading. The packet now presents the two concerns as separate, correctly named H2 sections so it satisfies the required-section contract enforced by the shared skill packager.

---

## What Changed

#### SKILL.md structure

- Split the merged `WHEN TO USE + SMART_ROUTING` header into two sections: `## 1. WHEN TO USE` (use / do-not-use boundaries) and `## 2. SMART ROUTING` (keyword triggers plus the sibling-packet hand-off rule). No prose was removed; the routing keyword-trigger line moved under the new SMART ROUTING section.
- Renumbered the remaining H2 sections contiguously so numbering runs 1 through 6.
- Added a `## 5. SUCCESS CRITERIA` section stating when a command produced through this packet is complete, clearing the recommended-section gap.

#### Rationale

- A space-separated `SMART ROUTING` heading is what the section matcher recognizes; the underscore form read as a single combined heading missing the routing section. Naming the section explicitly keeps the packet discoverable and contract-clean without changing any authoring guidance.

## Upgrade

No migration required. Structural normalization only — all authoring workflow, rules, and references are unchanged.
