# Iteration 018 — Track D (deepseek)

## Focus
DESIGN.md schema peers (Open Design 9-section incl anti-patterns/decisions/references; spec-honesty 17->9-10 sections; skill grounding pattern)

## Findings
1. **[P0] Fabrication-Surface Analysis (17-vs-9-section)** — Our 17-section DESIGN.md has 3 high-fabrication-risk sections that Open Design's 9-section schema avoids: Section 1 (Visual Theme & Atmosphere) requires 200-300 words of interpretive prose, 'comparative framing' metaphors, and named design principles — concepts invented by the LLM, not extracted from tokens. Section 7 (Content & Voice) requires a 'vibe paragraph' (2-3 sentences of brand personality from copy) — pure LLM interpretation with no token grounding. Section 0 (Brand Context) requires 3-5 personality adjectives 'tied to observable design decisions' — the LLM selects the adjectives then reverse-engineers the tie to values. By contrast, Open Design's 9-section (palette, typography, spacing, motion, voice, anti-patterns, components, decisions, references) has ZERO sections that require the LLM to invent metaphors or atmosphere. The 'voice' section in Open Design is observed content (actual UI text), not interpretive prose. Our extra 8 sections beyond Open Design's 9: Brand Context (MIXED value), Visual Theme & Atmosphere (HIGH fabrication), Dark Mode (HIGH value, conditional), Depth & Elevation (HIGH value, extractable), Content & Voice (HIGH fabrication), Accessibility Contract (HIGH value, computed), Responsive Behavior (MEDIUM), State Matrix (HIGH value), Iconography (MEDIUM), Agent Prompt Guide (HIGH value, self-contained). Evidence: design_md_format.md sections 2-3 define the atmosphere prose requirements (comparative framing, metaphors, 'could this describe 3 other sites?' test) with zero evidence-quota enforcement; the Vercel example DESIGN.md devotes 20 lines to invented atmosphere prose including 8 metaphors.
   - Recommendation: In design_md_format.md: (a) Replace Section 7's 'vibe paragraph' requirement with observed-data-only format — actual UI text samples, casing patterns, tone descriptors derived ONLY from tokens.sampleTexts — eliminating the LLM-invented personality summary. (b) Add hard evidence quotas to Section 1: each prose paragraph MUST cite >=2 specific token values; each named principle MUST cite >=3 token paths from tokens.json. (c) Section 0's personality adjectives MUST be drawn from a closed taxonomy derived from observed patterns, not LLM-selected.
   - Evidence: design_md_format.md:100-146 (Section 1 spec), design_md_format.md:648-725 (Section 7 spec), mcp-open-design/SKILL.md:224 ('9-section prose'), vercel/DESIGN.md:6-24 (20 lines of LLM-invented atmosphere prose), iteration-003.md:13-15 (thin-evidence omission finding)
2. **[P0] DECISIONS Section Gap (Open Design)** — Open Design's 9-section DESIGN.md includes a DECISIONS section — a dedicated provenance-anchored record of WHY each design decision was made, citing specific token values or constraints as evidence. Our DESIGN.md has NO decisions section. Decision rationale is scattered across prose sections: 'Weight 300 as the signature headline weight — light, confident, anti-convention' (stripe/DESIGN.md:17) embeds a decision in atmosphere prose with no structured provenance. 'Shadow-as-border philosophy... zero-offset, zero-blur, 1px-spread shadow that creates a border-like line without box model implications' (vercel/DESIGN.md:12) is the strongest example — it names the technique, provides the CSS value, and explains the architectural reason — but it lives buried in atmosphere prose, not in an auditable decisions table. A DECISIONS section would make each claim auditable: 'Claim: shadow-as-border replaces CSS borders. Evidence: 0px 0px 0px 1px shadows appear 4x more than elevation shadows; zero components use solid border CSS. Tradeoff: loses border-color transitions but gains box-model immunity.' This prevents fabrication because every decision MUST cite tokens — a claim with no token evidence can't be entered. Our anti_patterns.md catalog (28 anti-patterns) is a SEPARATE writer-resource doc, not a DESIGN.md output section. Open Design bakes anti-patterns INTO the output, making them part of the design system's contract that consumers (multi-agent) read.
   - Recommendation: In design_md_format.md: add a new Section ~8.5 'Design Decisions & Provenance' (between Do's/Don'ts and Accessibility). Format: decision name (bold), claim (1 sentence), evidence (>=2 token values or frequency ratios from tokens.json), tradeoff (what was sacrificed), and anti-pattern link (what NOT to do that this decision prohibits). This section replaces the scattered-in-prose decision rationale currently embedded in Sections 1 and 6. The anti_patterns.md catalog should be partially promoted INTO the DESIGN.md output (the 5-8 most system-specific anti-patterns) as a subsection of this DECISIONS block, making the output self-contained for multi-agent consumers.
   - Evidence: mcp-open-design/SKILL.md:224 ('DESIGN.md 9-section prose' with DECISIONS), stripe/DESIGN.md:17 (embedded decision in atmosphere prose), vercel/DESIGN.md:12 (shadow-as-border decision in atmosphere prose), anti_patterns.md (28-writer-side anti-patterns, not output)
3. **[P1] extract-design-system Skill Architecture (Prompt-Driven Grounding)** — extract-design-system (arvindrk) is an AI-agent SKILL — meaning its entire extraction logic lives IN THE PROMPT, not in separate tool code. The skill instructs the AI to: read the page's computed CSS → classify by W3C DTCG type → emit tokens.json + tokens.css. There is NO interpretive WRITE phase — the output is machine tokens, not prose. Dembrandt takes the opposite architecture: pure Node CLI, static+computed CSS hybrid, zero AI involvement. Our architecture is hybrid: TypeScript CLI extracts → AI writes DESIGN.md prose → TypeScript CLI validates. The key vulnerability in our architecture is the AI-operated WRITE phase — the same LLM that hallucinates is tasked with composing prose. extract-design-system's prompt-driven architecture avoids this by never asking the LLM to write prose — it outputs structured data only. Dembrandt avoids it by being fully programmatic. The borrowable pattern is NOT to eliminate the WRITE phase (the DESIGN.md prose IS our value proposition), but to front-load token constraints MORE aggressively in the WRITE prompt so the LLM writes grounded prose. Currently, the WRITE prompt template (design_md_prompt_template.md) has 6 cardinal rules but only #1 addresses hallucination directly. The template should include a 'Claim Evidence Required' rule: 'Every conceptual claim (principle name, metaphor, comparative framing) in Sections 1, 6, and 7 MUST be followed by a parenthetical citing >=2 token values or frequency ratios from tokens.json. A claim without parenthetical evidence is a fabrication.' Additionally, Dembrandt's static+computed hybrid approach (parsing both stylesheet CSS AND runtime computed values) produces more complete tokens than our computed-only approach — CSS custom properties defined in stylesheets but not applied to elements would be missed by our Playwright computed-style extraction.
   - Recommendation: In SKILL.md assets/design_md_prompt_template.md: add 'Claim Evidence Required' as cardinal rule #7: every prose claim about system characteristics must parenthetically cite specific tokens. In SKILL.md Section 3 (How It Works) WRITE phase description: add 'Before composing any prose section, enumerate the token values available for that section's domain. If fewer than 3 tokens exist for a domain (e.g., 2 shadow tokens), restrict the section to documenting only those values without interpretive claims.' In cluster.ts (future): adopt Dembrandt's static+computed hybrid to capture CSS custom properties from stylesheets that aren't applied to any element.
   - Evidence: design_md_prompt_template.md:40-56 (current WRITE prompt with 6 cardinal rules, none requiring evidence citations), iteration-003.md:6-21 (coverage-election and DTCG provenance as anti-fabrication), design_md_format.md:534-574 (Section 6 Depth & Elevation requires named principle but no evidence quota)
4. **[P1] Multi-Agent Output Structuring (Open Design)** — Open Design's DESIGN.md is consumed by 4+ coding agents (Claude Code, Codex, Cursor, OpenCode) with a DETERMINISTIC RE-RENDER schema — the same input produces the same output regardless of which agent reads it. This is achieved by a structured section contract where each section's content type is constrained. Our DESIGN.md is consumed by sk-code, sk-design-interface, and potentially other agents, but our prose-heavy sections (1, 7, 0) produce different interpretations per agent: Claude Code might extract 'shadow-as-border' as a technique; Codex might extract it as a design philosophy; Cursor might parse it as a component pattern. Our Section 13 (Agent Prompt Guide) already solves this by providing self-contained copy-paste prompts — the prompts work identically for any agent. But Sections 1-12 are freeform prose with no structured consumption contract. Evidence: the Vercel example's Section 1 includes claims like 'The Geist design system treats the interface like a compiler treats code' — this is a compelling metaphor for Claude but may confuse a code-generation agent that needs parseable rules. The fix: add a structured 'Agent-Oriented Summary' block at the TOP of each prose-heavy section, with key-value pairs that any agent can parse: 'Primary font: Geist. Headline weight: 600. Headline tracking: -2.88px at 48px. Shadow technique: box-shadow as border (0px 0px 0px 1px rgba(0,0,0,0.08)).' This creates a dual-layer document: structured header for agents, freeform prose for humans.
   - Recommendation: In design_md_format.md: add a 'Structured Summary' requirement for Sections 1, 3, 5, and 6 — a 3-5 line key-value block at each section's top that any agent can parse without NLP. The prose follows as human context. This makes the DESIGN.md dual-purpose: machine-consumable summary + human-readable detail, directly addressing the multi-agent consumption need without sacrificing the prose value.
   - Evidence: mcp-open-design/SKILL.md:224 ('deterministic re-render'), vercel/DESIGN.md:8-12 (prose metaphors), vercel/DESIGN.md:301-332 (Section 13 self-contained prompts — proven pattern for agent consumption)
5. **[P2] REFERENCES Section Gap (Open Design)** — Open Design's 9-section DESIGN.md includes a REFERENCES section for citations and further reading. Our DESIGN.md has no references section. The SKILL.md references 10+ resource docs (design_md_format.md, writing_style_guide.md, anti_patterns.md, etc.) but none of these make it into the DESIGN.md output. A consumer reading our DESIGN.md has no provenance trail for where the values came from beyond the 'Generated: 2026-04-05' header. A REFERENCES section would cite: the source URL(s), the extraction tool version, the token stability class definitions, and any external standards referenced (WCAG level, DTCG format). This is lower priority because it adds provenance without directly preventing fabrication, but it improves trust and debuggability for multi-agent consumers.
   - Recommendation: In design_md_format.md: add a REFERENCES section (before Agent Prompt Guide) with: source URL(s), extraction date, tool version, stability class legend (L1-L4), and standards referenced. Low implementation cost, high trust benefit.
   - Evidence: mcp-open-design/SKILL.md:224 ('9-section prose' with REFERENCES), design_md_format.md:54-63 (file header has only date/URL/pages/framework — no tool version or standards)
6. **[P2] Section Count Inflation — Spec vs Reality** — The design_md_format.md specifies 14 core + 1 half-section + 4 optional = up to 19 sections. But ALL four gold-standard examples (Vercel: 10 sections, Stripe: 9 sections, Linear: 9 sections, Supabase: ~9 sections) output only 9-10 sections. The spec demands sections (0 Brand Context, 7 Content & Voice, 9 Accessibility Contract, 11 State Matrix, 12 Iconography) that are NEVER written in practice. This creates two problems: (1) the section count claim ('17-section DESIGN.md') is misleading to consumers who expect all 17; (2) the WRITE phase is contractually obligated to produce sections it cannot fulfill from tokens.json, creating fabrication pressure. The 'required' designation for sections 7 and 11-12 in a single-page extraction where no content text or component states were sampled forces the LLM to invent content.
   - Recommendation: Reclassify sections in design_md_format.md: promote the 9-10 sections actually produced in examples to REQUIRED; demote 7 (Content & Voice), 11 (State Matrix), 12 (Iconography) to CONDITIONAL (only when tokens contain sampleTexts, state data, or icon detections). Demote 0 (Brand Context) to CONDITIONAL (only for multi-page extractions with enough data to derive brand personality from evidence rather than LLM interpretation). This reduces the nominal section count from '17' to '13 core + conditional expansions,' which is honest about actual output and reduces fabrication pressure.
   - Evidence: vercel/DESIGN.md:10 sections (grep count of '## N.' headings), stripe/DESIGN.md:9 sections (by pattern), linear/DESIGN.md:9 sections (by pattern), design_md_format.md:27-48 (section table showing all as 'yes' required)

## Questions Answered
- Q1: Open Design's ANTI-PATTERNS and DECISIONS sections are higher-signal than our philosophy-heavy sections because they require token evidence per entry (a DECISION without cited tokens can't exist; an ANTI-PATTERN without a specific violated rule is meaningless). Our Sections 1 (Atmosphere) and 7 (Content & Voice) invite fabrication because they require LLM-invented concepts (metaphors, vibe) with no token-evidence quota. ANTI-PATTERNS should be promoted from our internal anti_patterns.md resource doc INTO the DESIGN.md output as a new section; DECISIONS is a missing section entirely that would replace scattered-in-prose rationale with auditable provenance.
- Q2: Of our 8 extra sections beyond Open Design's 9, the value-adding ones are: Dark Mode (2.5, conditional/data-driven), Accessibility Contract (9, computed ratios), State Matrix (11, extractable states), Agent Prompt Guide (13, self-contained utility). The fabrication-inviting ones are: Brand Context (0, personality adjectives from LLM), Visual Theme & Atmosphere (1, 200-300 words of metaphors), Content & Voice (7, vibe paragraph). Recommended revision: demote 0, 7, 11, 12 to CONDITIONAL (only when tokens contain sufficient data); add DECISIONS section with token-evidence requirement; promote ANTI-PATTERNS into output. Net section count: 13 core + 5 conditional = honest 13-18 output, not claimed 17.
- Q3: extract-design-system's prompt-driven architecture eliminates the interpretive WRITE phase entirely — the skill prompt IS the extraction, outputting tokens.json + tokens.css with zero AI-prose. Dembrandt is pure CLI, similarly avoiding AI prose. Our hybrid architecture (CLI extract → AI write → CLI validate) keeps the AI-prose phase as the value proposition but should adopt their grounding pattern: front-load token constraints in the WRITE prompt so EVERY prose claim must cite token evidence (adoptable via prompt template change in SKILL.md). Dembrandt's static+computed hybrid would improve token completeness (capturing unused CSS custom properties).
- Q4: YES — multi-agent consumption strongly argues for MORE structured output. Our Section 13 (Agent Prompt Guide) already proves this: self-contained prompts work identically for any agent. The prose sections (1, 7) produce divergent interpretations across agents. The fix is a dual-layer document: structured key-value summary at the top of each prose section (machine-consumable) + freeform prose below (human context). Open Design's deterministic re-render achieves this by constraining section content types — we should add structured summaries to Sections 1, 3, 5, 6.
