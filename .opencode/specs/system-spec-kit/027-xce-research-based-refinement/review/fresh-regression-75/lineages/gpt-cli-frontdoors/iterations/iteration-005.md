- Verdict: **CONDITIONAL** · Findings: P0=0 P1=1 P2=0

## Analysis

Reviewed the three CLI shims, compiled CLI sources, shared bridge/proxy helpers, warm hook fallback helpers, launcher bridge paths, socket-server hardening, and relevant docs. I did not modify files or run write-capable commands.

**Finding: P1 Security**
The client side of the daemon CLI path does not verify ownership of predictable `/tmp/mk-*` socket directories before probing/connecting. The server bind path has owner hardening, but warm-only hooks and CLIs can still be pointed at a pre-planted local fake daemon if an attacker owns a traversable, non-world-writable `/tmp/mk-<service>` directory and exposes `daemon-ipc.sock`.

This is not a speculative naming concern: the three CLIs all rely on the same bridge probe/connect path, and their local `ensureSocketEnvironment()` checks only group/world-writable bits, not `uid`. The warm hook fallbacks are even thinner: they check socket existence/length and then deep-probe. Because `probeDaemon()` treats any JSON-RPC reply for the probe id as alive, the fake endpoint can pass warm-only gating and receive CLI JSON-RPC payloads.
