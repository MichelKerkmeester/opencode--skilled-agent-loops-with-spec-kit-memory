## [**1.1.2.1**] - 2026-06-06

This patch encodes the GitHub release-notes H1 contract. The release title field on GitHub already renders `vX.X.X.X — Title`, so a release body that opens with the same H1 duplicates it on the Releases page. The H1 belongs only to the changelog markdown file in the repo, where it is the file's identity. The skill now states this as a binding rule and ships the exact strip mechanics, replacing an older line in the finish workflow that wrongly claimed changelog files carry no version header at all.

---

## What Changed

#### New ALWAYS Rule

- **GitHub release bodies never start with an H1** -- New rule #11 in the SKILL.md rules list (subsequent rules renumbered). When publishing from a changelog file, strip the leading H1 and following blank lines into a temp notes file before `gh release create/edit --notes-file`. The repo changelog md keeps its heading untouched.

#### Finish Workflow (Step 6: Create Release)

- **H1 contract replaces a contradictory claim** -- The "Important" list previously said changelog files start directly with the summary paragraph and can be used as-is for the release body; that contradicted the house changelog style, which opens every release file with `# vX.X.X.X — Title`. The line now states the contract explicitly and includes the strip command: `tail -n +2 changelog/vX.X.X.X.md | sed '/./,$!d' > /tmp/release-body.md`.

- **Common Mistakes sharpened** -- The "--notes-file with unprocessed changelog" entry gained the concrete fix (strip into a temp file, never edit the H1 out of the repo file itself) and a post-publish detection check: `gh release view vX.X.X.X --json body --jq .body | head -1` must not start with `# `.
