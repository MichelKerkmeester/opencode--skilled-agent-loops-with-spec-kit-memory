This release lands the deep-agent-improvement portion of the packets 108 plus 110 auto-review uplift program. One reusable teaching from the upstream `dzianisv/opencode-plugins` auto-review package now applies to the mutation-coverage pipeline: a signature-based dedup that prevents the same mutation from being proposed twice across iterations of the same improvement arc. The signature is stored alongside each mutation in `mutation-coverage.json` — the council was explicit in section 10.6 that the authoritative storage IS `mutation-coverage.json`, NOT `agent-improvement-state.jsonl`, so the replay consumer and the coverage reducer share one source of truth. Every mechanism cleared a 5-of-5 unanimous deep-ai-council review by `cli-codex gpt-5.5 xhigh fast` before any code shipped.

## New Features

#### Mutation signature dedup in the coverage pipeline

The mutation-coverage pipeline used to propose the same mutation under multiple iterations of a single improvement arc when the reducer rebuilt state from `mutation-coverage.json`. There was no equality predicate strong enough to recognize "we already tried this" across iterations. The fix introduces `computeMutationSignature(mutation)` in `scripts/mutation-coverage.cjs`. The signature is `sha256(dimension + ":" + mutationType + ":" + targetSection + ":" + normalized_body.slice(0,64))` — dimension and mutation type pin the proposal axis, target section pins the location, and the first 64 characters of the normalized body capture the proposal substance. A pre-propose check now compares each candidate signature against the `signatures` field on every existing entry in the `mutations[]` array. A match skips the proposal and records the skip with `EXHAUSTED-FROM: iter-NNN` so the trail back to the original proposal is preserved.

&nbsp;

#### Authoritative storage in `mutation-coverage.json`

The naive port would have stored signatures in `agent-improvement-state.jsonl`. The council's section 10.6 review rejected that path because the JSONL state file is event-sourced and append-only, which means dedup queries would have to scan the entire history on every proposal. The signatures live on each mutation entry in `mutation-coverage.json`, which is the reducer's canonical output and is sized to the actual mutation count rather than the iteration count. The replay consumer and the reducer now read signatures from the same place.

&nbsp;

#### Reducer update for the new `signature` field

`scripts/reduce-state.cjs` recognizes the new `signature` field on every coverage entry it sees. The reducer is backward-compatible: entries that predate this release have no `signature` field, and the reducer falls through to the legacy equality check rather than dropping them. New coverage files written after this release will always carry signatures because `mutation-coverage.cjs` populates the field before any append.

&nbsp;

#### Bypass via `DEEP_AGENT_IMPROVEMENT_SKIP_DEDUP=1`

There are corner cases where an operator genuinely wants to retry an exhausted mutation — for example, after manually editing the target section, or after a prior failure that the operator believes was environmental rather than substantive. The `DEEP_AGENT_IMPROVEMENT_SKIP_DEDUP=1` env-var bypasses the pre-propose signature check for the duration of the run. The bypass is intentionally not a permanent config setting; operators must opt in per invocation so the default behavior stays safe.

&nbsp;

#### `EXHAUSTED-FROM:` skip-reason format

When the signature check skips a proposal, the skip is recorded with `EXHAUSTED-FROM: iter-NNN` where `iter-NNN` is the iteration that originally proposed the mutation. The format is grep-friendly and lets operators trace any exhausted proposal back to its origin without scanning the whole coverage file.

## Council-Gated Workflow

Packet 110 ran through a 5-seat deep-ai-council deliberation before any source change. The council returned REQUEST-CHANGES on the first pass with HIGH confidence and reshaped scope on the storage decision. The original draft proposed `agent-improvement-state.jsonl` as the signature store; the council's section 10.6 ruled that storage had to live on the mutation entry in `mutation-coverage.json` so the replay consumer and reducer share one source of truth. The skip-reason format and the env-var bypass also surfaced from the council deliberation rather than from the original packet draft. Implementation followed via `cli-opencode + deepseek-v4-pro --pure`.

## Upgrade

No migration required. Existing `mutation-coverage.json` files that predate this release have no `signature` field on their mutation entries; the reducer recognizes this and falls through to the legacy equality check, so historical coverage data continues to function. New mutation proposals after this release will populate the `signature` field on every entry, and dedup will engage on the next iteration. Set `DEEP_AGENT_IMPROVEMENT_SKIP_DEDUP=1` if you need to bypass the check for a specific run; leave it unset for the safe default. Source commit: b29640496.
