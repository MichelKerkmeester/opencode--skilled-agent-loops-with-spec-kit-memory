# Research Synthesis: Spec-Kit Data Quality by Default

This is the canonical synthesis of the spec-kit data-quality deep-research. It is the official multi-lineage result. Five distinct deep-research lineages ran thirty-seven substantive iterations against the live retrieval and validation code, and they all converged on one program. The prior content of this file was a manual first-pass approximation. This replaces it. The unit of this packet is a verdict, not a build. Every retrieval verdict below stays a hypothesis until a prod-mode measurement runs on this corpus.

## 1. EXECUTIVE VERDICT

The program is a correctness-and-adherence program wearing a retrieval costume, and the truncation law is what undresses it. The prod retrieval path runs a truncation stack the eval harness skips. `DEFAULT_MIN_RESULTS = 3` (`confidence-truncation.ts:35`) is a never-cut-below-3 MINIMUM guarantee, not a cap, and confidence truncation is cliff-conditional, firing only on a relevance gap past position 3 and otherwise returning the full set, so a prod page runs 3 to 20 results rather than always 3. The real prod-limiting stage is token-budget truncation. The whole stack runs only on the prod seam (`if (!evaluationMode)` at `hybrid-search.ts:2049`), so 028 already measured a 5.9x eval-versus-prod fidelity gap on this exact corpus. The DIRECTION is what carries the program, and it survives. Truncation taxes one class of candidate only. It cannot cut what is not a retrieval result, so adherence and logic and write-time candidates bypass it entirely and ship on cost, while every retrieval candidate pays the truncation tax plus a re-index cost before it can even be measured. That inverts naive ROI. The cheapest real wins are the write-time gates, not the headline retrieval techniques. The keystone that anchors the whole build is a discovery, not a design. A verify-fix-verify quality loop with auto-fix already runs LIVE and default-ON on the memory-save path (`quality-loop.ts`, with `SPECKIT_QUALITY_LOOP` and `SPECKIT_QUALITY_AUTO_FIX` both default TRUE). The program is overwhelmingly wiring that shipped machinery to the authored spec-doc and metadata-JSON surface it never reached, not building anything green-field. The retrieval slate stays frozen behind a prod-mode completeRecall@3 read it does not yet have. The honest headline across all five lineages is one measured unconditional GO, a cluster of floor-bypassing GO-on-cost gates and fields, a small CONDITIONAL retrieval slate that is hypothesis-until-prod-measured, seven genuinely novel floor-bypassing capabilities the reuse-first work missed, plus a consolidated NO-GO list whose every entry is an instance of one of ten governance anti-patterns.

## 4. THE BUILD-READY IMPLEMENTATION

The five keystones collapse onto one small shared core so they cannot diverge. There is ONE engine, three front doors and ONE registry.

`detector-registry.ts` is the single source of truth. Each entry declares `{id, surface, detect, fixClass, fix}` where `fixClass` is one of `safe`, `risky` or `none`, deny-by-default. Adding a detector is one entry. Granting it `safe` requires editing the frozen allow-list, and that edit is itself a `guarded`-class change that re-checks the two structural invariants, so the registry guards itself with the same invariants it enforces. `dq-engine.ts` is the pure core. `runDetectors(target, opts)` returns `{issues, applied, skipped}`, never writes in report mode, and in apply mode executes only `fix()` for detectors whose `fixClass` is in `opts.allowFixClass` (always just `['safe']`), with the content_hash idempotency guard and atomic writes. It reuses the shipped scorers verbatim and adds no scoring logic of its own.

The three front doors call the SAME engine. A1 is on-write and report-only. B1 is scheduled and headless, report in CI plus operator-local safe-apply. B2 is interactive, report plus `--confirm` safe-apply. B3 is a detector that emits queue rows the front doors surface. C2 is the gate every retrieval-class detector's promotion must pass.

The exact seams per keystone, verified to file:line:
- A1 is a two-file hook. H1 scores the JSONs at `generate-context.ts:398 atomicWriteJson`. H2 extends the non-mutating reviewer where it already runs at `scripts/core/workflow.ts:1854`. H3 adds a `CONTENT_QUALITY` rule to `validate.sh` via a `scripts/validation/content-quality.ts` registered in `validator-registry.json` next to the existing shape rules.
- B1 is `scripts/sweep/dq-sweep.ts` plus `.github/workflows/dq-corpus-sweep.yml` (`schedule` plus `workflow_dispatch`) plus an opt-in `post-merge` hook that the existing globbing installer wires with no installer change.
- B2 appends a `data-quality` route to `_routes.yaml` and authors `doctor_data-quality.yaml` with the inverted `validate_targets` lists, then re-runs `route-validate.py`.
- B3 captures an aggregate `impression_count` and `min_rank_seen` at the search seam behind `SPECKIT_RETRIEVAL_GAP_DETECT` default-off, then `detect-retrieval-gaps.ts` writes edge-tagged rows into a `refinement_queue` table mirrored on `learned_feedback_audit`.
- C2 authors `spec-corpus-golden.json` (multi-target, all three measurability classes) and `run-spec-recall-gate.mjs` (PROMOTION and REGRESSION) around the unchanged `run-eval-v2.mjs`.

The frozen `fixClass` allow-list, consolidated. Safe: `desc.shape` regenerate from frontmatter, `enum.tier_status_ctype` case-normalize, `triggers.propagate` additive subset copy capped at 12, `hvr.style` length-neutral fence-aware swap, `anchor.unclosed` append closer. Risky and suggest-only: `desc.generic`, `graph.child_aggregation`, `req.ears_coverage`, the edge-a recall-gap enrich-triggers action. None and advisory: `budget.overlength`, the edge-b below-floor row. The single invariant across all of it: a detector that touches an authored-doc BODY is never `safe`.

## 5. THE GOVERNANCE AND ROLLOUT LAYER

The order is the safety property. The consolidated sequence is the topological sort of five inviolable dependency edges. Census before gate, engine before front doors, backfill before error, coverage guard before retrieval trust, plus C2 before retrieval promotion. The master sequence is 17 stages across seven phases. Phases I through V ship the floor-bypassing reuse-first half on cost. Phase VI adds the novel correctness and adherence slate (report-only and additive, so it parallelizes with III through V). Phase VII builds the measurement instrument first, then touches the retrieval half.

The migration plan is a four-beat discipline per gate that does not compress. WARN (land default-off, report, never block), then BACKFILL (dry-run report then batched apply through the existing additive write paths), then RE-MEASURE TO ZERO (confirm the corpus-wide failing count is 0), then ERROR (flip warn to error, drop the dormant `legacy_grandfathered` bypass, verify a corrupted scratch packet now exits 2). The Stage-0 census is the prerequisite for every count-to-zero. Known starting numbers: 24 invalid graph files against the target schema (16 live after excluding archives), 0 grandfathered packets so the bypass deletes with zero blast radius, the frontmatter and continuity gaps from the dry-run, the version-grammar census (20 SKILL.md, 2 grammars), plus the mixed-embedding-regime chunk census. The retrieval migration is a SEPARATE migration gated on the coverage guard (`embedding_context_version` plus a coverage readout) that does not exist yet. The single legacy-corpus invariant: no new hard rule is introduced until its backfill report reads zero.

The safety model classifies every fix by blast radius with two structural invariants. INV-1, a fix touching an authored body is never `safe`, made mechanical by the `computeAuthoredDocQuality` wrapper that throws on full-auto and by quarantining the budget-trim to memory-save. INV-2, a retrieval-class change is never promoted without a prod@3 read through the C2 gate. Four human boundaries escalate from auto-safe-fix (local `--apply` and CI report) through human-gated guarded fixes (`--confirm`, backfill apply, doctor APPLY) and always-human-authored body edits to the release-reviewer gate that MUST read the prod-mode column. CI never auto-commits, because the governance boundary is not whether a fix is safe but whether its blast radius is human-reviewed. Four standing drift guards keep the corpus honest. Embedding drift, coverage drift, storage drift via read-time content_hash, plus cross-copy drift on the triple-copy trigger vocabulary.

The measurement plan is one reader, one metric, no cross-credit. Write-time earns its keep on a conformance count driven to zero plus a scratch-packet regression-catch proof. The scheduled sweep earns its keep only if its first run finds at least one defect in each of the three escape classes (path-filter escapes, backfill blind spots, cross-surface coherence) that the shipped change-triggered tiers demonstrably missed. If it finds nothing they missed, it is downgraded to on-demand. Retrieval earns its keep ONLY on a prod-mode completeRecall@3 RISE, with external @5 and @10 and @20 numbers and eval-mode @K explicitly inadmissible because the eval harness runs with the tail intact while prod truncation can cut it. Each novel item proves a distinct non-retrieval metric of its own.

## 6. PROVE-FIRST CAVEATS

This is a research deliverable, not a build. Nothing here is shipped.

The truncation law is the spine. The prod truncation stack taxes retrieval candidates only and is blind in the eval harness, which runs with the tail intact. Its confidence floor never cuts below 3 and fires only on a relevance cliff, and the token-budget stage is what actually limits the prod page. Both run only outside eval mode. Every retrieval verdict inherits its unblock condition from this law. The recall@K trap is the most seductive in the packet. The external brief's numbers are all measured at K of 5 or wider, so they justify spending the re-index to find out and they cannot stand in for the find-out. A release reviewer who reads the eval-mode column instead of the prod-mode column repeats the 028 saturation mistake. The one net-negative caution is the graph-metadata rollup. It is the single candidate that can actively harm a narrow query by displacing a real child, so its build is justified only if the narrow-query regression is measured in the SAME pass as the broad-query win. Prod-mode for retrieval is non-negotiable. Every Tier C item stays a hypothesis until a full re-index runs at 100-percent coverage under the new version and the `run-eval-v2.mjs` dual-mode read shows the prod-mode completeRecall@3 number move.
