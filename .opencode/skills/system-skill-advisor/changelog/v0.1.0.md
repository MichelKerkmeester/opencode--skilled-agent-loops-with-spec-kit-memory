## [**0.1.0**] - 2026-05-15

This first preview release establishes `system-skill-advisor` as a standalone MCP server that owns prompt-time skill recommendation. The package was extracted from `system-spec-kit`, given a dedicated MCP identity (`mk_skill_advisor`), decoupled from spec-kit's source tree and wired into every install + diagnostic surface. The release covers eight public tools (`advisor_recommend/rebuild/status/validate`, `skill_graph_scan/query/status/validate`) reachable through the `mcp__mk_skill_advisor__*` namespace. The 0.1 line keeps shadow-mode + parity gates in place while corpus and holdout metrics mature.

#### Standalone MCP server identity

- Need: advisor recommendation lived inside `system-spec-kit`, mixing skill graph and advisor scoring with spec-kit's memory and lifecycle code.
- Change: the package now ships as `.opencode/skills/system-skill-advisor/` with its own runtime, a dedicated `.opencode/bin/mk-skill-advisor-launcher.cjs` and a package-local SQLite database at `mcp_server/database/skill-graph.sqlite`.
- Why it matters: the advisor can be installed, tuned or replaced independently. Rollback to the Python CLI shim is one env var away.

&nbsp;

#### Native advisor recommendation, 4 advisor tools

- Need: prompts needed prompt-safe skill recommendations with cache state, lifecycle redirects and freshness trust gates that the legacy Python shim could not deliver directly inside MCP.
- Change: the standalone server registers `advisor_recommend`, `advisor_rebuild`, `advisor_status` and `advisor_validate`. `advisor_recommend` returns prompt-safe `recommendations[]` plus cache + lifecycle metadata and `advisor_validate` returns real corpus, holdout, parity, safety and latency measurements.
- Why it matters: every runtime can call the same MCP-native advisor surface with fail-open semantics, instead of forking subprocess shims per host.

&nbsp;

#### Skill graph, 4 graph tools

- Need: maintainers wanted to inspect, query and validate the skill graph directly without bouncing through advisor scoring.
- Change: `skill_graph_scan`, `skill_graph_query`, `skill_graph_status` and `skill_graph_validate` expose the package-local SQLite graph. Lifecycle, freshness and trust state mirror the advisor tools.
- Why it matters: skill-graph rebuilds, audits and integration checks have a stable MCP surface that other skills (and tests) can depend on.

&nbsp;

#### Spec-kit decoupling (isolation arc)

- Need: spec-kit and the advisor still shared imports across modules. The skill advisor could not evolve or be released independently.
- Change: Phase 1 isolation lifted the advisor out of `system-spec-kit` runtime imports. Phase 2 + 3 hybrid decoupling moved shared types into `@spec-kit/shared` and inlined what stayed local. The CI isolation check sweeps both skills.
- Why it matters: the advisor can ship preview iterations without coupling to the spec-kit memory release cadence.

&nbsp;

#### Compat shims for Python and OpenCode plugin bridge

- Need: existing prompt hooks and CLI consumers depended on the Python `skill_advisor.py` shim and the OpenCode plugin bridge.
- Change: `skill_advisor.py` probes the native advisor first and translates the response back to the legacy JSON-array shape. If the native probe is unavailable, it falls back to the local Python scorer. `--force-native` and `--force-local` testing toggles ship for rollback debugging. The OpenCode plugin bridge mirrors the same fail-open pattern.
- Why it matters: existing automation keeps working while the native advisor moves toward 1.0.

&nbsp;

#### Install guide + `/doctor:mcp` coverage

- Need: the skill had an `INSTALL_GUIDE.md`, but the master install README and `/doctor:mcp install` / `/doctor:mcp debug` enumerated four MCP servers without registering `mk_skill_advisor`.
- Change: added new Â§10.5 Skill Advisor subsection to the master install README and registered `mk_skill_advisor` in `mcp-doctor.sh` (new `diagnose_mk_skill_advisor()` function), `doctor_mcp_install.yaml`, `doctor_mcp_debug.yaml` and `mcp.md`. Inserted the previously-missing `mk_skill_advisor` block into `.vscode/mcp.json` for runtime parity across all six configs.
- Why it matters: discovery, install and runtime diagnostics now follow the same surface as the other MCP servers in the repository.

&nbsp;

#### Documentation alignment

- Need: README, ARCHITECTURE, feature catalog and manual testing playbook needed to describe the standalone identity and the public tool surface.
- Change: README + ARCHITECTURE.md follow the sk-doc skill template; `feature_catalog/feature_catalog.md` and `manual_testing_playbook/manual_testing_playbook.md` are kept package-local; `mcp_server/**/README.md` describe each code surface.
- Why it matters: one read of the package surfaces what it owns, how to operate it and where each piece of code lives.

## Upgrade

No migration required. Pre-1.0 preview: public tool ids (`advisor_recommend`, `advisor_rebuild`, `advisor_status`, `advisor_validate`, `skill_graph_*`) are stable. Existing configs that referenced `system_skill_advisor` should rename the entry to `mk_skill_advisor` and point `command`/`args` at `.opencode/bin/mk-skill-advisor-launcher.cjs`. The Python shim (`skill_advisor.py`) remains available. Set `SPECKIT_SKILL_ADVISOR_FORCE_LOCAL=1` to fall back to the legacy local scorer for diagnosis.
