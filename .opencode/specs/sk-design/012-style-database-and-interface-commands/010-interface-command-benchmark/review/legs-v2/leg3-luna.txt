Creation/Remediation Artifact - Interface Direction Spec

Direction: “Signal Path”

The empty screen should feel like an unwired operations console: the dashboard already understands the team’s delivery model, but needs its first repository connection to activate the signal.

Contrastive directions considered:

1. “Signal Path” — a structured metric pipeline with four inactive nodes. Selected because it teaches the product model while preserving dashboard credibility.
2. “Control Room” — dense empty metric matrix with subdued outlines. Strong operational feel, but risks looking broken before setup.
3. “Warm Start” — large onboarding panel with supporting metric explanations. More approachable, but less distinctive for engineering teams.

Selected direction: “Signal Path”.

Authored color tokens:

| Token | Hex | Use |
|---|---:|---|
| `ink-950` | `#0D1117` | page background |
| `ink-900` | `#151B24` | primary surfaces |
| `ink-850` | `#1C2531` | raised cards and active zones |
| `grid-700` | `#2A3645` | borders, dividers, inactive paths |
| `text-100` | `#F2F5F8` | primary text |
| `text-300` | `#A7B4C2` | secondary text |
| `signal-teal` | `#62D0C7` | primary action and connected path |
| `signal-amber` | `#F2B86B` | attention and first-run cue |
| `signal-red` | `#F27878` | error state |
| `on-signal` | `#0D1117` | text on teal or amber controls |

No gradients. Teal is the primary product signal; amber and red are reserved for status semantics.

Screen composition:

1. Header  
   Workspace name on the left; “First workspace” status and account controls on the right.

2. Page heading  
   Eyebrow: `DELIVERY SIGNAL`  
   Heading: `Your delivery signal starts here.`  
   Body: `Connect a repository to turn pull requests, review time, cycle time, and deploy frequency into one operating view.`

3. Four-metric rail  
   Each card shows one real metric name and a state-specific message:

   - `Pull Requests` — `Awaiting repository history`
   - `Review Time` — `Needs review events`
   - `Cycle Time` — `Needs merged-workflow timestamps`
   - `Deploy Frequency` — `Needs deployment events`

4. Signal Path setup card  
   A horizontal route connects four nodes:

   `Pull Requests → Review Time → Cycle Time → Deploy Frequency`

   Each node contains the metric label, a muted inactive indicator, and one short explanation. The path becomes teal only after data arrives.

5. Primary action  
   `Connect a repository`

   Secondary action: `See how these metrics work`

Representative states:

- Empty:

  `No repositories connected`  
  `Connect GitHub or GitLab to build your first delivery view.`  
  CTA: `Connect a repository`

- First-run:

  `Start with one repository`  
  `We’ll inspect pull requests, review events, merged-workflow timestamps, and deploys. You can change the connection later.`  
  CTA: `Choose repository`

- Loading:

  `Reading repository history`  
  `Scanning merged pull requests, review events, cycle timestamps, and deploys…`

  Progress sequence:

  `Repository connected` → `Pull requests found` → `Review events found` → `Deploy events found`

- Error:

  `We couldn’t read delivery history`  
  `The repository connection is valid, but the provider returned no event access. Check the integration scope and try again.`  
  Actions: `Reconnect repository` and `View integration settings`

Motion intent:

- Primary reveal: `220ms`, `cubic-bezier(0.2, 0.8, 0.2, 1)`
- State transition: `160ms`, `ease-out`
- Progress movement: `1600ms`, linear
- Hover/focus response: `120ms`, ease-out
- Animate only the signal path and status changes; no bouncing, parallax, or decorative motion.
- Reduced motion: remove path drawing and use immediate state replacement. Preserve `aria-live` status updates.
