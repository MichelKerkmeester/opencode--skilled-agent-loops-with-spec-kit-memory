## Iteration 6 Report — sk-code:code-opencode alignment

**Headline finding**: The `code-opencode` surface is governed by **three disconnected drift guards** that each inspect a different layer of the routing contract, with no single end-to-end test covering the chain that the compiled-router + advisor hook claim to enforce. The code-opencode SKILL.md RESOURCE_MAP is locally parseable, the parent `smart-routing.md` flat union is drift-guard-validated by `sk-code-router-sync.vitest.ts:119-192`, and the code files pass `verify_alignment_drift.py` — but the **compiled-router output vocabulary (`targetQualifiedIds`) and the surface RESOURCE_MAP vocabulary (`references/...` paths) share no typed-pair contract**, and neither guard covers the compiled→surface agreement that the 015 sk-code-router-alignment spec implicitly assumes.

**Verified claims**

1. `verify_alignment_drift.py` does NOT inspect Markdown RESOURCE_MAP. Its SUPPORTED_EXTENSIONS (line 39-51) covers `.ts/.tsx/.mts/.js/.mjs/.cjs/.py/.sh/.rs/.json/.jsonc` only; the alignment-verification-automation.md contract at `.opencode/skills/sk-code/code-opencode/references/shared/alignment-verification-automation.md:48-51` explicitly states "The verifier is intentionally lightweight. It does not replace the standards checklists and does not scan markdown prose." [CONFIRMED; refines F-3-8]

2. `code-opencode/SKILL.md:45-51` claims "the parent sk-code hub RESOURCE_MAP is the union of this map (re-prefixed with code-opencode/) and the sibling code-webflow map plus the parent-owned universal/shared tier; a drift guard enforces that equality." The cited drift guard is `sk-code-router-sync.vitest.ts` (in `system-deep-loop/.../scripts/skill-benchmark/tests/`), NOT `verify_alignment_drift.py`. The latter is the named verifier of the OPENCODE surface in `references/shared/alignment-verification-automation.md:34`, but it is silent on the parent-child RESOURCE_MAP equality the SKILL.md asserts.

3. `sk-code-router-sync.vitest.ts:119-192` enforces: (a) every child RESOURCE_MAP path exists at the child root; (b) **parent surface map == union(re-prefixed children) + PARENT_TIER_ALLOWLIST**. PARENT_TIER_ALLOWLIST (lines 124-134) explicitly enumerates 9 hub-owned paths; the code-opencode surface child is tested for parseability and on-disk existence but not for any contract with the compiled-router.

4. The compiled-router emits a different vocabulary than the surface RESOURCE_MAP. The compiled `compiledRoute()` at `.opencode/specs/.../lib/compiled-route.cjs:73-85` returns `{ hubId, action, selectionKind, targets: <normalized qualifiedIds>, effectivePolicyHash, generation }`; the on-disk gold `.opencode/specs/.../006-parent-hub-rollout/001-sk-code/compiled/route-gold.typed.json` uses `targetQualifiedIds` like `sk-code/code-review/code-review/workflow/review-cache`. The code-opencode SKILL.md RESOURCE_MAP (lines 73-156) emits paths like `references/shared/universal-patterns/naming-and-commenting.md`. These two vocabularies share no typed-pair contract — `router-replay.cjs`'s `buildResourceContract` (lines 235-262) converts RESOURCE_MAP strings to `(workflowMode, leafResourceId)` pairs via `leaf-manifest.json` only, never via compiled output.

5. The `code-opencode/benchmark/router-mode-a/skill-benchmark-report.json:342,430,518,606,694,782,870,958` reports `routeTelemetry.reason: "no-hub-router"` for every scenario — because `code-opencode` is a surface packet (no `hub-router.json`), the router-replay loads its inline RESOURCE_MAP directly rather than the hub union. The benchmark measures the surface RESOURCE_MAP in isolation; it does NOT measure the parent-hub-union contract that 015 plans to type, and it has no path to invoke `compiled-route.cjs` because the compiled router resolves at the HUB level (the spec-tree `compiled-route.cjs:23-31` `HUB_CHILD` maps hub ids to `006-parent-hub-rollout/00N-<hub>`, with no entry for surface packets).

6. `code-opencode/benchmark/router-mode-a/skill-benchmark-report.json:9` records `targetSkill.root: "/Users/.../0030-018-rust/.opencode/skills/sk-code/code-opencode"` — a stale worktree path from another iteration. Confirms F-4-5.

7. `hub-router.json:32-36` declares the `code-opencode` mode pointing at `code-opencode/SKILL.md` only (one resource). When `router-replay.cjs` loads the hub router, it routes a `code-opencode` request to that single SKILL.md; the per-intent surface paths only enter via `loadSurfaceRouter()` (`.opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/router-replay.cjs:545-565`). So the hub-router level is lossy by design: a compiled-route enabled hub still has a hub-router.json that points to a single packet SKILL.md regardless of intent.

8. The advisor hook `advisor-recommend.ts:41-49` `COMPILED_ROUTING_HUBS` Set and the spec-tree `compiled-route.cjs:23-31` `HUB_CHILD` map both enumerate the same 7 hubs (`sk-code, mcp-tooling, system-deep-loop, cli-external-orchestration, sk-prompt, sk-design, sk-doc`), but in different shapes (Set vs path map). The two are NOT cross-checked; a hub added to one but not the other causes silent drift (advisor attaches a `compiledRoute` for a hub whose compiled engine throws "unknown hub", or vice versa).

9. The 015 spec at `.opencode/specs/sk-doc/019-sk-doc-router-alignment/015-sk-code-router-alignment/spec.md` plans (Phase 3 line 91-99 of plan.md) a leaf-manifest + typed-gold derivation that **packet-qualifies flat `references/...` entries** so the shared tier carries mode identity on the typed-pair surface. The pre-requisite contract (Phase 4 line 100-103) re-runs the drift-guard. But no contract is added for compiled-output → typed-pair parity; the leaf-manifest's leaves are all `references/...` or `assets/...` paths, while compiled output is `qualifiedIds` — different schemas.

10. `code-opencode/SKILL.md` has a leaf-manifest at `.opencode/skills/sk-code/leaf-manifest.json:4-71` enumerating 67 leaves under `workflowMode: "code-opencode"` — including all 47 of code-opencode's `references/...` files plus all 17 of its `assets/...` files. So the typed-pair surface IS already populated for code-opencode at the leaf level — but the RESOURCE_MAP paths inside code-opencode/SKILL.md are NOT packet-qualified, so the `buildResourceContract` derivation has to do the prefixing in code. There's no test that this prefixing matches `registryPacketRoots(skillRoot)` (`router-replay.cjs:571-578`).

11. `verify_stack_folders.py` (lines 1-47) validates language folder presence (typescript/javascript/python/shell/rust/config) — disjoint from `verify_alignment_drift.py` (code-file syntax) and disjoint from `sk-code-router-sync.vitest.ts` (Markdown RESOURCE_MAP). Three scripts, three coverage bands, no overlap.

12. `code-opencode/SKILL.md` lacks a `feature-catalog/` subtree (line 168 enumerates the asset and reference trees, but the directory is empty: `ls .opencode/skills/sk-code/code-opencode/feature-catalog/` returns no output). The 24 feature-catalog trees the prior iteration references are at OTHER skill roots; the code-opencode surface has none. The 015 spec plans leaf-manifest generation but does not plan a feature-catalog for code-opencode.

**Ranked actionable recommendations**

1. **Add a cross-vocabulary drift test.** Either extend `sk-code-router-sync.vitest.ts` with a fifth `it()` that loads the parent `smart-routing.md` flat RESOURCE_MAP union (via `loadSurfaceRouter`), parses it, then asserts that for every `code-opencode/` qualifiedId in the parent-hub compiled route-gold there is a corresponding `code-opencode/SKILL.md` RESOURCE_MAP entry whose `leafResourceId` matches via `leaf-manifest.json` — OR document in `references/shared/alignment-verification-automation.md` that the verifier is explicitly **not** the parent-child equality guard and the named guard is `sk-code-router-sync.vitest.ts`. Both claims currently live in `code-opencode/SKILL.md:51`.

2. **Cross-check the duplicated compiled-hub allowlists.** `advisor-recommend.ts:41-49` `COMPILED_ROUTING_HUBS` (Set) and the spec-tree `compiled-route.cjs:23-31` `HUB_CHILD` (path map) should be the same set of 7 hubs. Add a Vitest in `system-skill-advisor/mcp-server/tests/handlers/` that loads the spec-tree `compiled-route.cjs` and asserts `Array.from(COMPILED_ROUTING_HUBS).sort() === Object.keys(HUB_CHILD).sort()` — otherwise an advisor call can attach a `compiledRoute` field that the resolver cannot compute (or vice versa).

3. **Cover the `code-opencode` surface in the 015 leaf-manifest derivation.** The 015 plan Phase 4 (lines 100-103) re-runs the drift-guard after packet-qualifying shared-tier paths; extend it with a sibling `it()` that loads `.opencode/skills/sk-code/leaf-manifest.json`, extracts the `code-opencode` mode's leaves, and asserts that the union of `code-opencode/SKILL.md` `RESOURCE_MAP` paths (after stripping the `references/...` prefix from each entry to make them leaf-relative) equals the manifest's code-opencode leaves minus the deferred-asset and playwright-only entries. This closes the gap between the inline router and the typed-pair surface.

4. **Move the compiled-router resolver from the mutable spec tree.** F-2-4 already flagged this; the recommendation to keep `compiled-route.cjs:16-21` resolving into `.opencode/specs/.../011-runtime-engine/lib/resolve.cjs` is the load-bearing risk. Action: either ship a copy of `resolve.cjs` under `.opencode/skills/sk-code/` (or a sibling `.opencode/lib/`) and have `.opencode/bin/compiled-route.cjs` resolve there, OR add a Vitest that fails when the resolver path does not begin with `.opencode/skills/` (rejects the spec tree). The 015 spec at line 91-99 needs this stability before its Phase 5 live-mode sample runs.

5. **Refactor the alignment-verification contract documentation.** The `references/shared/alignment-verification-automation.md` claims authority for `verify_alignment_drift.py` but does not link to the actual parent-child equality guard `sk-code-router-sync.vitest.ts`. Add a "Related drift guards" section pointing at the Vitest file path AND the orchestrator hook (`run-skill-benchmark.cjs:219,308`) for Lane C. This was confirmed-asked in F-3-8 but the documentation has not been updated.
