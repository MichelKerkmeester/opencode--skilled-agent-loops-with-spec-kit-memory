## Cross-model notes

- **GPT-5.6-Sol:** F1 and O3 share one required cache-provenance redesign and should land together; O4 should be included in that schema migration. Implement F2 before consumer changes so both runtimes can represent degraded checks without calling them fresh. Then update OpenCode cache invalidation/retention (F3, F7), followed by Claude hardening and parity (F4, F5, F6, O1). O5 must be resolved before build attestation is enabled so version-2 records cover exactly the real build inputs. O2 is not safely fixable as a same-dispatch warning through the installed OpenCode hook API; command mutation or throwing would regress the explicit warn-only contract.
- **Opus 4.8:** Shared root causes: F1 and O3 are the same family — the freshness cache is checker-written and not anchored to build/dist identity. Land them together as one cache-schema bump (add dist identity + an origin field). O3's dist-identity check is the self-contained, low-risk half (catches dist revert/tamper); F1's full fix requires build-written hashes (postbuild wiring across 8 packages) and is the higher-risk half — O3 reduces but does not eliminate F1's narrow content-changed-with-preserved-older-mtime gap. Ordering/coupling: do O3 first (defines the schema), then F1 layers the origin semantics on top; O4 also touches writeStoredSourceHash so sequence O3→O4→F1 to avoid three separate rewrites of that function. F2+F5 both harden the Claude hook and should ship together (error-surfacing + input validation), and both need the check-dist-staleness.sh payload guard. F4 (shared subprocess deadline) reduces the SIGKILLs that create O4 orphans, so F4 lightens O4. Parity map: F1/F2/O3/O4/O5 are lib-level (dist-freshness.cjs) and automatically reach BOTH runtimes; F3/F7/O2 are OpenCode-plugin-only (Claude is stateless/per-edit); F4/F5/F6 are Claude-hook-only; F2 and O1 are the two that require touching BOTH surfaces deliberately. Non-issues / judgment calls: O2 is effectively a documentation refinement — the next-turn, non-blocking, no-stdout behavior is the correct contract given the TUI-safety invariant, so no functional change is warranted unless the product explicitly wants dispatch-time blocking. O1 is a genuine contract divergence rather than a defect; the recommended fix (SessionStart check-all mirror on the Claude side) is optional-but-correct, and the minimum acceptable resolution is documenting the intentional edited-file-scoped PostToolUse contract. The prior audit's O4 suggestion to 'write to an OS temp dir before rename' is rejected as an EXDEV cross-device-rename regression, and the prior F1 suggestion to 'treat any hash mismatch as stale regardless of mtime' is rejected as a false-stale regression on every legitimate rebuild unless build-written hashes back it.

## Per-finding fix designs

### F1 [P1 (GPT P1 / Opus P2)] A source-hash mismatch can be silently blessed as fresh
`.opencode/skills/system-spec-kit/scripts/lib/dist-freshness.cjs:430` — **Status: both models agree**

**Opus 4.8** — risk: med · confidence: med
- Root cause: Confirmed at dist-freshness.cjs:428-488. When the stored hash != current sourceHash (source content changed since the cache was written), the code falls to the mtime heuristic (445-471). If the changed source's newest mtime is NOT greater than distMtime, lines 473-487 overwrite the cache with the new hash and return fresh. Because the cache is CHECKER-written (330,473), not build-written, the stored hash only records 'the last state the checker blessed', so a genuine content change whose mtime is preserved/older than dist is silently blessed. Same root-cause family as O3 (cache not anchored to build identity).
- Fix approach: The prior audit fix ('treat any hash mismatch as stale regardless of mtime') would REGRESS: on every legitimate rebuild (edit source at T1, build dist at T2>T1) the stored hash still differs from current until the checker re-blesses via the mtime branch we'd be deleting, so it would false-stale every rebuild. The mtime branch is load-bearing. The correct fix is to make the cache authoritative by having the BUILD write it: add a `postbuild` step in each watched package that calls the already-exported `writePackageSourceHashCache`, and tag build-written entries with an origin marker. Then in checkPackageFreshness, a hash mismatch against a build-origin cache => stale; a checker-origin or absent cache => fall back to today's mtime heuristic (bootstrap/degraded, unchanged).
- Exact change: In dist-freshness.cjs: extend the cache payload written by writeStoredSourceHash to include `origin:'build'|'checker'` and dist identity (see O3), and have writePackageSourceHashCache stamp origin:'build'. In checkPackageFreshness, after computing sourceHash: read stored; if stored.origin==='build' and stored.sourceHash!==sourceHash => return stale (source changed since last build); otherwise keep the existing mtime path. Add a `postbuild`/`build` chain in the 8 DIST_PACKAGES package.json build scripts invoking `node <path>/dist-freshness.cjs`'s writePackageSourceHashCache (or a thin CLI subcommand) for each entry. Do NOT delete the mtime branch.
- Acceptance: Regression: edit a source file, rebuild dist (postbuild writes build-origin hash), run check => fresh. Bug: with a build-origin cache present, change source content while preserving an mtime <= distMtime => check returns stale. Bootstrap: delete the cache, run check with dist newer than sources => fresh (mtime path), no false stale.
- Side effects / parity: Cross-package: touches all 8 package.json build scripts; a postbuild failure must not fail the build (wrap fail-open). Both OpenCode plugin and Claude hook consume checkPackageFreshness unchanged, so both inherit the fix. Ship together with O3 (shared cache-schema change).
- Test: Unit test in the cjs's own harness: seed a build-origin cache (origin:'build') via writePackageSourceHashCache, mutate a source file's bytes but reset its mtime older than dist, assert checkPackageFreshness returns status:'stale'. Add a bootstrap test asserting no false-stale when only a checker-origin/absent cache exists.

**GPT-5.6-Sol** — risk: med · confidence: high
- Root cause: Confirmed: checkPackageFreshness computes the current source hash, but an existing mismatch only falls through to the mtime comparison; when no source mtime exceeds the dist mtime, it writes the changed hash into the cache and returns fresh (.opencode/skills/system-spec-kit/scripts/lib/dist-freshness.cjs:428-487). The checker therefore acts as its own build attestor and can bless an uncompiled, old-mtime edit.
- Fix approach: Make freshness records build-attested rather than checker-authored. A check may compare records but must never promote a mismatching source hash. Use a two-phase build record so a source edit occurring during compilation cannot be attested accidentally.
- Exact change: In dist-freshness.cjs, replace the version-1 source-only cache with a validated version-2 build record. Add prepare-build and record-build CLI operations: prepare-build records the pre-build source hash for every registered entry; record-build runs only after successful compilation, verifies the source hash is unchanged from prepare-build, captures the dist identity, and atomically promotes the record. Remove the writeStoredSourceHash call from checkPackageFreshness at current line 473. Treat a missing/version-1 record as status unverified with stale:true and the rebuild command; treat any source-hash mismatch against a version-2 record as stale regardless of mtimes. Add the prepare/record lifecycle commands to the npm build flow of all seven packages declared at dist-freshness.cjs:21-161, preserving existing build and postbuild steps.
- Acceptance: After a successful registered npm build, the package is fresh. Editing a watched source, forcing its mtime older than the dist entry, and checking again returns stale and leaves the build record unchanged. Editing a source between prepare-build and record-build refuses promotion. A legacy cache never produces fresh until a registered build establishes version-2 provenance.
- Side effects / parity: This is shared by both runtimes: the OpenCode plugin and Claude wrappers consume the same result automatically. Every rebuildCommand target in DIST_PACKAGES must install the build lifecycle integration. Coordinate with O3 and O4; migrate tests that directly call writePackageSourceHashCache to the two-phase protocol.
- Test: Extend dist-freshness.vitest.ts with old-mtime source mutation, legacy-cache migration, successful two-phase build, source-changed-during-build, and failed-build/no-promotion cases. Add a registry test proving every DIST_PACKAGES entry's npm build lifecycle invokes prepare-build and record-build.

### F2 [P1 (GPT P1 / Opus P2)] Checker errors are reported as fresh and suppressed by both consumers
`.opencode/skills/system-spec-kit/scripts/lib/dist-freshness.cjs:354` — **Status: both models agree**

**Opus 4.8** — risk: low · confidence: high
- Root cause: Confirmed. freshnessError returns stale:false,status:'error' (dist-freshness.cjs:354-363). check-all aggregation derives status solely from stale (590-592), so an all-error run => status:'fresh', exit 0. The plugin's stalePackages filters `.filter(r=>r.stale)` (mk-dist-freshness-guard.js:58) dropping errors, and check-dist-staleness.sh acts only on `payload.stale===true` (line 57). Errors (missing source path, missing package root, checker crash) are silently indistinguishable from fresh in every consumer.
- Fix approach: Keep execution fail-open but make 'error' a first-class status distinct from fresh, and surface it (bounded, non-blocking) in all three consumers. Adopt the prior fix, adding the standalone check-all CLI aggregation which the prior fix missed.
- Exact change: (1) dist-freshness.cjs check-all block (590-592): compute `errors = results.filter(r=>r.status==='error')`; set result.status = stale.length? 'stale' : errors.length? 'error' : 'fresh', and attach `errors` count/list; keep exit 0 for the error status (fail-open, only STALE_EXIT_CODE stays fatal). (2) Plugin: have stalePackages/refreshStale also collect `status==='error'` results; buildBrief emits a distinct bounded line e.g. '[dist-freshness-guard] CHECK ERROR (could not verify freshness): <pkg> — <message>' and appendGuardLog records it; still never throws/never stdout. (3) check-dist-staleness.sh: after the stale branch, add `elif payload.get('status')=='error': print('DIST FRESHNESS CHECK ERROR: '+message)` to stdout so the Claude hook surfaces it; still exit 0.
- Acceptance: Force an error (point a package root at a missing dir): OpenCode injects/logs a CHECK ERROR brief; Claude hook prints a CHECK ERROR line; CLI `check-all` reports status:'error' with exit 0; none crash or block.
- Side effects / parity: Both runtimes must be updated together (parity: both currently suppress). Bounded output only — cap error lines to avoid flooding the system-context injection.
- Test: Plugin test: add a fixture whose package root/source path is missing, assert exactly one CHECK ERROR line is logged and injected and console stays empty. cjs test: assert check-all with an error result yields status:'error'. Shell test: feed check-file JSON with status:'error', assert the warning prints and exit 0.

**GPT-5.6-Sol** — risk: med · confidence: high
- Root cause: Confirmed: freshnessError returns status:error with stale:false (.opencode/skills/system-spec-kit/scripts/lib/dist-freshness.cjs:354-363); check-all considers only item.stale and therefore labels an all-error run fresh (.opencode/skills/system-spec-kit/scripts/lib/dist-freshness.cjs:590-592). The plugin filters out every non-stale result (.opencode/plugins/mk-dist-freshness-guard.js:57-59), and the Claude wrapper prints only payload.stale (.opencode/skills/sk-code/code-quality/scripts/check-dist-staleness.sh:52-60).
- Fix approach: Keep execution fail-open but make diagnostic state explicit and visible. Errors must not become stale build claims, but they also must never be represented as fresh.
- Exact change: In check-all aggregation, collect error results separately and use precedence: status stale when any result is stale, otherwise degraded when any result is error, otherwise fresh. Include degraded:true and an errors array when applicable; retain stale:false and exit 0 for an error-only run. Update non-JSON printing to emit bounded DIST FRESHNESS CHECK ERROR lines. In the plugin, cache diagnostics rather than only stale entries, create a synthetic error diagnostic if checkAllFreshness throws, and make buildBrief render separate stale and check-error lines. In check-dist-staleness.sh, print an actionable error warning when the shared payload or an aggregate child has status:error/degraded.
- Acceptance: An all-error check-all invocation returns status:degraded, stale:false, exits 0, and never prints or serializes fresh. Mixed stale/error results retain stale status and expose both categories. OpenCode injects and logs a bounded check-error brief; Claude displays the same actionable warning while allowing the edit or command to continue.
- Side effects / parity: Update both OpenCode and Claude consumers together. Any code parsing check-all.status must accept degraded. Hermetic plugin fixtures currently omit six package roots, so they must construct all registered package fixtures or intentionally assert the resulting diagnostics rather than assuming missing roots are fresh.
- Test: Add shared-library tests for error-only, mixed, and fresh aggregation plus CLI exit behavior. Add plugin tests proving check errors are injected without console output. Add Claude-wrapper tests proving error JSON produces a warning and exit 0.

### F3 [P1 (GPT P1 / Opus P2)] OpenCode can retain a fresh cache for two minutes after a source edit
`.opencode/plugins/mk-dist-freshness-guard.js:109` — **Status: both models agree**

**Opus 4.8** — risk: low · confidence: high
- Root cause: Confirmed. staleForInjection reuses staleCache for STALE_CACHE_TTL_MS=120000 (mk-dist-freshness-guard.js:28,109-114). Force-refresh happens only on session.created (134) and risky-bash (122); tool.execute.before does not invalidate the cache for write/edit/patch tools, so a source edit that newly makes a package stale is not reflected for up to 2 minutes.
- Fix approach: Invalidate the cache in tool.execute.before when an editor tool (write/edit/patch/multiedit) targets a watched source file, so the next experimental.chat.system.transform recomputes from the post-write disk state. This aligns OpenCode with the Claude hook, which already re-checks on every Write|Edit.
- Exact change: In tool.execute.before, after the bash branch, add an editor-tool branch: if `input.tool` is one of {write,edit,patch,multiedit} (case-insensitive), read the target path from output.args (filePath/path), and if `isWatchedSourceFile(path, ...)` (or unknown path) set `staleCache=null` (mark dirty) so staleForInjection refreshes next turn. Fail-open in try/catch. Import isWatchedSourceFile or packageForSourceFile from the lib (add to its exports if not already — packageForSourceFile is exported; use it, unmatched => still invalidate cheaply or skip).
- Acceptance: With a fresh cache, invoke the edit hook on a watched source path, then invoke the transform hook => it refreshes and reflects the new staleness without waiting for the TTL. Editing an unwatched/non-source path does not force needless rehash (or forces at most one).
- Side effects / parity: Parity: Claude PostToolUse is already per-edit immediate; this closes the gap. Minor extra hashing (once per turn after an edit, not per file). No Claude change needed.
- Test: Plugin test: fresh fixture, cache warm; write a source file to make dist stale; call the edit before-hook with that file_path; call transform; assert a stale brief is injected (previously it would be empty within the TTL window).

**GPT-5.6-Sol** — risk: low · confidence: high
- Root cause: Confirmed: staleForInjection reuses staleCache for 120000 ms (.opencode/plugins/mk-dist-freshness-guard.js:25-28,109-114). tool.execute.before refreshes only selected Bash calls and ignores source-mutating tools (.opencode/plugins/mk-dist-freshness-guard.js:117-125), while Claude performs a post-edit check immediately (.opencode/skills/sk-code/code-quality/scripts/hooks/claude-posttooluse.sh:90-105).
- Fix approach: Invalidate before a recognized mutation and refresh after it. Refreshing after execution observes the resulting filesystem state; invalidating before execution prevents a missed after-hook from preserving an apparently fresh entry.
- Exact change: Add a case-insensitive mutating-tool classifier for write, edit, and apply_patch/apply-patch. In tool.execute.before, clear staleCache and staleCheckedAt for those tools before returning. Add tool.execute.after, supported by the SDK at .opencode/node_modules/@opencode-ai/plugin/dist/index.d.ts:249-258, and call refreshStale('source-mutation') after those tools. Refreshing all recognized file mutations is preferable to fragile patch-text parsing; the next system transform then uses the post-write result without waiting for the TTL.
- Acceptance: Prime a fresh cache, mutate a watched source with an old preserved mtime through each supported mutating tool, and invoke the next system transform within two minutes. It must inject stale immediately. An unrelated read tool must retain the cache. A failed/no-op mutation may recheck but must not generate a false stale result.
- Side effects / parity: OpenCode gains parity with Claude's immediate post-edit observation. No Claude change is required for Write/Edit, although any future Claude mutation matcher must be added to .claude/settings.json and its input parser at the same time.
- Test: Extend mk-dist-freshness-guard.test.cjs with fresh-cache-then-edit, apply-patch, tool.execute.after, no-op mutation, and unrelated-tool cases; assert no console writes and no TTL delay.

### F4 [P1 (GPT P1 / Opus P2)] Sequential subprocess timeouts exceed the enclosing Claude hook timeout
`.opencode/skills/sk-code/code-quality/scripts/hooks/claude-posttooluse.sh:68` — **Status: both models agree**

**Opus 4.8** — risk: low · confidence: high
- Root cause: Confirmed. claude-posttooluse.sh runs the comment checker with timeout=8 (line 72) then the dist checker with timeout=8 (line 97) sequentially; the enclosing PostToolUse hook is capped at 10s (.claude/settings.json:92). Worst case 16s > 10s => the harness SIGKILLs the hook, losing warnings and (per O4) possibly killing a mid-write cache rename.
- Fix approach: Enforce a single shared monotonic deadline under the 10s hook budget and pass only the remaining budget to each subprocess (reject the 'just raise the timeout' option — the 10s cap is intentional and raising it worsens per-edit latency). Concurrency is a valid alternative but adds Popen/poll complexity to a fail-safe hook; the shared-deadline approach is smaller and guarantees < 10s.
- Exact change: In claude-posttooluse.sh add `import time; START=time.monotonic(); BUDGET=9.0`. Before the comment subprocess: `t=min(8, BUDGET-(time.monotonic()-START))`; pass `timeout=max(0.5,t)`. Before the dist subprocess: `rem=BUDGET-(time.monotonic()-START); if rem < 0.5: skip (log to stderr); else timeout=min(8, rem)`. Keep both inside their existing try/except so a TimeoutExpired stays fail-safe.
- Acceptance: Simulate a slow comment checker (sleep 8): the dist checker gets <=1s or is skipped, and the whole hook returns in <10s (no SIGKILL). Normal fast case: both run, total well under budget.
- Side effects / parity: Claude-hook-specific; the OpenCode plugin runs in-process with no subprocess-timeout stacking, so no plugin change. Interacts with O4 (fewer SIGKILLs => fewer orphan .tmp files).
- Test: Shell/integration test invoking the hook with a stub comment checker that sleeps 8s; assert total wall time < 10s and the hook exits 0.

**GPT-5.6-Sol** — risk: med · confidence: high
- Root cause: Confirmed: the comment checker may run for 8 seconds and the dist checker for another 8 seconds sequentially (.opencode/skills/sk-code/code-quality/scripts/hooks/claude-posttooluse.sh:68-75,93-100), but the enclosing PostToolUse hook is killed after 10 seconds (.claude/settings.json:85-93). The line-72 claim that one timeout is well under the hook timeout ignores the second subprocess.
- Fix approach: Run the independent checks concurrently under one bounded budget instead of increasing every edit's worst-case latency to approximately 20 seconds.
- Exact change: Refactor claude-posttooluse.sh to resolve both checkers first, launch available checks concurrently, and collect their results in deterministic comment-then-dist display order. Give each worker at most 7 seconds, leaving margin under the 10-second host limit. Reduce check-dist-staleness.sh's nested Node timeout from 8 to 6 seconds so its child is terminated before the parent worker deadline. Preserve exit 0 and isolate each worker's exception so one failure cannot suppress the other result.
- Acceptance: With both fake checkers sleeping for six seconds, the complete hook returns in under nine seconds and displays both results in stable order. A hung checker is terminated before the Claude 10-second limit while the other result is still surfaced. The hook always exits 0.
- Side effects / parity: Claude-only scheduling change; OpenCode has no enclosing Claude hook timeout. F5's top-level fail-open boundary must include worker creation and result collection. Keep .claude/settings.json at 10 seconds unless measured startup overhead proves the bounded implementation cannot meet it.
