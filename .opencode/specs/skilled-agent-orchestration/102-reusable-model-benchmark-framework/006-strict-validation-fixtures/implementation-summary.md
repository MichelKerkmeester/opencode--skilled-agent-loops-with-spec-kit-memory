## What Was Built

Three boolean strict-VALIDATION benchmark fixtures plus a capability profile, designed to discriminate two near-equal frontier models by strictness on input validation. Each fixture's oracle is DOMINATED by adversarial-invalid inputs (subtle malformed cases a lax-but-plausible validator wrongly accepts), so a validator missing ANY rule scores strictly below 1.0.

### Files Changed

| File | Action | Purpose |
|------|--------|---------|

| `assets/model-benchmark/benchmark-fixtures/validate-date.json` | Created | `isValidDate(s)` strict ISO `YYYY-MM-DD`, leap-year-correct — 26 cases (7/19) |

<!-- ANCHOR:how-delivered -->
## How It Was Delivered

Oracle-correctness-first, in three phases. A throwaway `/tmp` harness held a reference implementation and a deliberately-lax implementation for each validator. Every oracle case's expected boolean was generated by running the reference impl over the spec-listed valid/invalid input lists (never hand-authored), then split into a visible `tests` spread (3 valid + 3 adversarial-invalid) and `hidden_tests` (the remainder). Both impls were scored through the real `code-task-scorer.cjs`; only after the reference scored 1.0 and the lax impl scored strictly < 1.0 was the fixture JSON written. The v3 profile was copied from v2 with the fixture list + sample count adjusted, then validated through `profile-validator.cjs`. Vitest was extended with additive describe blocks and the full suite re-run. The `/tmp` harness was deleted. The orchestrator then ran the **live n=5 benchmark** (n=2 de-risk → full 40-cell run, skill-compliant dispatch, 0 pollution); the verdict + per-sample evidence live in `eval/synthesis.md`.

<!-- ANCHOR:decisions -->
## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Generate every oracle `expect` from a reference impl (never hand-author) | Eliminates the "wrong oracle value → garbage benchmark" risk by construction |
| Over-represent invalid cases (≥60%) | A lax validator only fails on malformed inputs; invalid dominance is what pushes it below 1.0 |
| Each lax impl skips exactly one real rule the invalid set exercises | Proves the fixture discriminates (ipv4 leading-zero, date leap/days, semver numeric-leading-zero + empty-id) |
| Keep `hard-roman-to-int` in v3 | It discriminated in prior phases; retained as the 4th fixture per the task contract |
| Orchestrator ran the live n=5 benchmark (de-risk → full) | The fixtures' purpose is the verdict; M3 1.0 (gate-eligible) vs MiMo 0.891 — see `eval/synthesis.md` |
| Read the gate, not the framework's "TIE" label | The reporter prints TIE-on-format under saturation; the correctness gate + reproducibility (004+006) are the real signal |

<!-- ANCHOR:limitations -->
## Known Limitations

1. **Verdict is reliability-driven, not a capability chasm** — MiMo solves these tasks ~0.8–0.94 of the time but ~1-in-5 emits a fully-failing answer; M3 never did (32/32 cells). The ~0.11 gap reproduces across 004 + 006.
2. **2 of 4 fixtures still saturated** (date, semver — both models 1.0); the signal comes from ipv4 + roman. `promote-or-demote-to-smoke` applies to the saturated two.
3. **Lax impls are illustrative** — they demonstrate one missed-rule failure mode per fixture; a real model may miss a different subset, but the invalid-dominant oracle still drops any partial validator below 1.0.
4. **Framework "TIE-on-format" is misleading under saturation** — the correctness gate cleanly separates the pair (M3 eligible, MiMo not). Recommended P2: report the sole gate-eligible model as the correctness winner.
