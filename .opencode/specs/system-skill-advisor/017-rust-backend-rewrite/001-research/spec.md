## Phase Context

This is the **scoring-and-ranking half** of a two-phase Rust-backend-rewrite research packet. A future single-lineage **16-round** `/deep:research` run will study the CPU-bound decision path that turns one prompt into ranked skill recommendations: prompt normalization, curated phrase/token matching, lexical and derived matching, graph-causal propagation, optional BM25F scoring, lane fusion, confidence/uncertainty, ambiguity handling, and executor-delegation overrides. The handler calls the scorer only after freshness and exact-cache checks, then maps and truncates the ranked output [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/handlers/advisor-recommend.ts:369-432].

The current live projection is small but non-trivial: SQLite supplies the skill rows and edges, then six command bridges are appended [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/projection.ts:58-149] [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/projection.ts:596-633]. At charter time the advisor reported 18 candidates, consistent with 12 indexed skills plus six bridges, and the graph reported 41 edges. Those counts are a baseline to re-measure during research, not evidence of a speedup.

**Scope Boundary**: Research and synthesis only. **No Rust is written, no crate or native module is scaffolded, no loop is launched by this charter-authoring task, and no backend source is modified.** Embeddings, vector-index/search internals, skill-graph tools, and transport/daemon/CLI architecture belong to Phase 002.

**Dependencies**:
- `advisor_recommend` orchestration, status/cache boundary, and scorer call [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/handlers/advisor-recommend.ts:369-482].
- The scorer core under `mcp_server/lib/scorer/`, especially `fusion.ts` (876 LOC), `projection.ts` (733 LOC), `executor-delegation.ts` (499 LOC), and lane modules.
- The existing benchmark with p95 gates of 50 ms for a reused projection and 60 ms when the projection is reloaded [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/bench/scorer-bench.ts:15-17] [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/bench/scorer-bench.ts:43-62].
- The native dependency boundary: `better-sqlite3` and `@huggingface/transformers` are direct dependencies [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/package.json:13-19].

**Deliverables**:
- `research/research.md` — merged, cited synthesis across all rounds.
- An **improvement matrix**: scoring component × {latency / memory / tail latency / throughput / determinism / packaging} → {big win / marginal / none / already-native}, with every verdict tied to the execution residency.
- A **new-feature-feasibility matrix**: candidate capability × {possible in TypeScript / practical in TypeScript / Rust unlocks / native dependency already supplies it}.
- A **risk register** covering ranking parity, calibration drift, FFI overhead, migration cost, test gaps, and operational complexity.
- A **ranked recommendation**: full rewrite / targeted native module (`napi-rs` or WASM) / Rust sidecar / do-not-rewrite, with the first measurable next step.

## 2. PROBLEM & PURPOSE

### Problem Statement
The scorer is the backend's strongest pure-compute rewrite candidate, but even this path crosses native boundaries. `scoreAdvisorPrompt` builds five lane outputs, indexes their matches, loops over every projected skill, computes confidence and uncertainty, sorts the candidates, applies ambiguity/abstention gates, and finally applies executor delegation [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/fusion.ts:605-715] [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/fusion.ts:727-875]. Most text, graph, fusion, and decision math is JS-resident. However, projection rows come from SQLite through `better-sqlite3`, prompt embeddings come from an embedder such as ONNX, and stored vectors are loaded through the database [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/projection.ts:596-635] [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/lanes/semantic-shadow.ts:92-149] [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/lanes/semantic-shadow.ts:266-310]. Rewriting JavaScript callers cannot accelerate those already-native primitives.

Without a residency-aware latency decomposition, a Rust proposal can wrongly count SQLite or ONNX time as a rewrite win, optimize an 18-candidate scorer whose absolute cost is already negligible, or break calibrated ranking behavior for theoretical throughput. Conversely, the current matcher repeatedly constructs token sets, regular expressions, maps, arrays, and sorted graph walks in V8; at larger candidate/vocabulary sizes those operations may justify a narrow native kernel rather than a full backend rewrite [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/text.ts:29-99] [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/lib/scorer/lanes/graph-causal.ts:36-134].

### Purpose
Produce a decision-ready, source-cited answer about the scoring core alone: quantify how much routing latency is genuinely JS-resident; identify whether current and plausible future candidate-set sizes make that cost material; preserve ranking, ambiguity, and delegation semantics; and select the smallest architecture that yields a measured benefit.

## 3. SCOPE

### In Scope
- Map the `advisor_recommend` cache-miss path from prompt input through `scoreAdvisorPrompt` to ranked public recommendations.
- Measure or design measurement for JS-resident normalization, matching, graph propagation, fusion, sorting, confidence, uncertainty, ambiguity, and delegation work.
- Quantify the current 18-candidate baseline and model larger candidate, edge, vocabulary, phrase, and prompt sizes.
- Assess full Rust, targeted `napi-rs`/WASM, sidecar, and do-not-rewrite options for the scorer boundary only.
- Preserve byte-stable ordering, thresholds, attribution evidence, abstention behavior, and prompt-safe output.

- The embedder stack, model selection, ONNX inference implementation, vector-index/search algorithms, or vector similarity architecture.
- Skill-graph MCP tools, graph indexing/rebuild behavior, daemon lifecycle, MCP transport, and CLI transport.
- Writing or compiling Rust, scaffolding a crate, wiring FFI, or changing backend source.

## 6. RISKS & DEPENDENCIES

| Type | Item | Impact | Mitigation |
|------|------|--------|------------|
| Risk | Current scorer has only 18 projected candidates | Native-call overhead may exceed JS compute savings | Measure absolute lane and decision costs before discussing percentages |
| Risk | SQLite and ONNX time is attributed to the scorer rewrite | Inflated recommendation | REQ-003 and REQ-005 force per-stage residency labels |
| Risk | Existing benchmark mixes projection reuse and reload but not all handler stages | Misleading p95 interpretation | Add a research-only measurement design for scorer-only and handler-level spans [SOURCE: .opencode/skills/system-skill-advisor/mcp_server/bench/scorer-bench.ts:43-62] |
| Risk | Rust changes tuned ranking behavior | Routing regressions despite lower latency | Require corpus replay, exact ordering/rounding checks, and threshold parity |
| Risk | FFI copies prompt/projection/evidence data | Targeted module loses its theoretical gain | Include serialization and boundary-crossing costs in every native-module estimate |
