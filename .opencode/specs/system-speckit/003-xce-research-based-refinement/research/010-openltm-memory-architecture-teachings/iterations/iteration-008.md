## Mechanism
OpenLTM’s runtime embedding abstraction is small and provider-gated: `EmbeddingProviderName` is only `"gemini" | "openai" | "ollama" | "disabled"`, with `name`, `model`, `dim`, `available()`, and `generate()` as the provider contract, and the default config is `provider: "disabled"` with `confidenceThreshold: 0.6` (`packages/openltm-core/src/providers/embeddingProvider.ts:7`, `packages/openltm-core/src/providers/embeddingProvider.ts:17`, `packages/openltm-core/src/providers/embeddingProvider.ts:22`). The factory dynamically loads Gemini/OpenAI/Ollama and falls back to `DisabledProvider` for anything else (`packages/openltm-core/src/providers/embeddingProvider.ts:30`, `packages/openltm-core/src/providers/embeddingProvider.ts:47`). The disabled provider is a true no-op: `available()` is false and `generate()` returns null (`packages/openltm-core/src/providers/disabled.ts:7`, `packages/openltm-core/src/providers/disabled.ts:12`, `packages/openltm-core/src/providers/disabled.ts:16`).

Provider implementations degrade quietly. Ollama defaults to `nomic-embed-text`, dim `768`, and `http://localhost:11434`; availability probes `/api/tags` with a 2s timeout, and generation catches failures as null (`packages/openltm-core/src/providers/ollama.ts:9`, `packages/openltm-core/src/providers/ollama.ts:14`, `packages/openltm-core/src/providers/ollama.ts:18`, `packages/openltm-core/src/providers/ollama.ts:27`). OpenAI uses dim `1536` and returns null on non-OK responses (`packages/openltm-core/src/providers/openai.ts:9`, `packages/openltm-core/src/providers/openai.ts:29`). Gemini uses dim `768`, requires an API key, and returns null on non-OK responses (`packages/openltm-core/src/providers/gemini.ts:10`, `packages/openltm-core/src/providers/gemini.ts:19`, `packages/openltm-core/src/providers/gemini.ts:31`). `cohere` and `openrouter` are not in the runtime `src/providers` abstraction; they exist in the janitor provider layer, whose type union includes them (`packages/openltm-core/src/janitor/providers/types.ts:149`).

Recall is FTS5-first. Query terms are quoted and OR-joined, then searched against `memories_fts` with `ORDER BY rank LIMIT 50`; BM25’s negative rank is normalized with `Math.exp(rank)` (`packages/openltm-core/src/db.ts:426`, `packages/openltm-core/src/db.ts:433`, `packages/openltm-core/src/db.ts:439`). Only when FTS returns fewer IDs than the requested limit does OpenLTM attempt semantic augmentation, guarded by `ltm.semanticFallback !== false`, hard-coded to `threshold = 0.5`, and wrapped in `try/catch` (`packages/openltm-core/src/db.ts:444`, `packages/openltm-core/src/db.ts:448`, `packages/openltm-core/src/db.ts:452`, `packages/openltm-core/src/db.ts:457`). If no provider is configured, `embedText()` returns null and semantic search returns `[]` (`packages/openltm-core/src/embeddings.ts:191`, `packages/openltm-core/src/embeddings.ts:194`, `packages/openltm-core/src/embeddings.ts:261`). The explainer states that when semantic is absent, FTS absorbs the semantic weight: `fts * 0.75 + importance * 0.15 + recency * 0.10` (`packages/openltm-core/src/recall/explainer.ts:5`, `packages/openltm-core/src/recall/explainer.ts:60`). However, query results are sorted by `decay_score`, not by that explainer score; the explainer is attached after sorting (`packages/openltm-core/src/db.ts:545`, `packages/openltm-core/src/db.ts:548`, `packages/openltm-core/src/db.ts:570`).

The split-table migration creates `memory_embeddings(memory_id PRIMARY KEY, embedding BLOB, model, dim, created_at)`, copies non-null legacy blobs from `memories.embedding`, drops the old partial index, and drops the inline column (`migrations/010_memory_embeddings_split.sql:5`, `migrations/010_memory_embeddings_split.sql:16`, `migrations/010_memory_embeddings_split.sql:21`, `migrations/010_memory_embeddings_split.sql:25`). Migration 003 was only an inline-embedding presence index (`migrations/003_embedding_index.sql:3`). The DAO writes the side table and dual-writes sqlite-vec when available (`packages/openltm-core/src/dao/embeddings.ts:35`, `packages/openltm-core/src/dao/embeddings.ts:48`), and recall explicitly selects slim columns to exclude the blob from MCP hot paths (`packages/openltm-core/src/db.ts:522`, `packages/openltm-core/src/db.ts:527`). `sqlite-vec` is a derived, lazy, no-throw index over `memory_embeddings`, default dim `768`, with `similarity = 1 - distance`; when unavailable, callers fall back to JS cosine or FTS-only paths (`packages/openltm-core/src/vec/index.ts:4`, `packages/openltm-core/src/vec/index.ts:10`, `packages/openltm-core/src/vec/index.ts:20`, `packages/openltm-core/src/vec/index.ts:93`). UNKNOWN: `docs/04-configuration.md` documents `semanticFallback`, but not the source-level `embeddings` provider block (`docs/04-configuration.md:7`, `docs/04-configuration.md:29`, `packages/openltm-core/src/config.ts:17`).

## Negative knowledge
- Do not copy provider proliferation as a product goal; system-spec-kit should keep ollama-nomic and abstract availability/degradation around it.
- Do not copy unwired knobs like `confidenceThreshold`; OpenLTM defines it but recall uses a hard-coded `0.5` semantic threshold.
- Do not copy the ranking/explainer split; OpenLTM computes FTS/semantic scores but sorts query results by decay score.
- Do not copy schema drift: fresh `schema.sql` still declares `memories.embedding`, while migration 010 drops it and some older paths still query it.
- Do not copy docs/config drift; requested docs expose `semanticFallback` but not the source-level `embeddings` config block.

## Open questions
- Should system-spec-kit add a compact SQL `memory_embeddings` metadata table even though vectors already live in a separate store?
- Should degraded search responses expose “vector unavailable/skipped” in trace output by default?
- Should embedder health be driven by Ollama reachability, vector coverage, or active-shard reconciliation as the canonical signal?
