### Angle 2: Crash-safety of the bounded background enrichment scheduler
- Framing: If the daemon dies between `markEnrichmentPending` and the deferred `setImmediate` enrichment, no data is lost because the row is left in `pending` and recovered by the existing replay/backfill path — verify this is actually true end-to-end.
- Repo evidence: `memory-save.ts:2633` writes `markEnrichmentPending(database, memoryId, POST_INSERT_ENRICHMENT_VERSION)` **inside the commit transaction**; `memory-save.ts:2864-2897` `scheduleBackgroundEnrichment` re-resolves the DB handle at run time (`requireDb()` → returns null on recycle-in-flight) and aborts silently on a deprecated/archived row. `handlers/save/enrichment-state.ts:159` defines `markEnrichmentPending`; the recovery path is `runPostInsertEnrichmentRepairBackfill` in `handlers/memory-index.ts:538-558` and the broader `runEnrichmentBackfill`. Stress test exists: `stress_test/durability/enrichment-marker-backfill-stress.vitest.ts:128-196`.
- Findings: Crash safety is real but **narrow**: the pending marker is durable (written in the same transaction as the row), and recovery drains via the backfill repair. What is *not* in the recovered row after a crash is the `enrichmentResult` itself — only the marker is recovered; the enrichment bundle reruns. There is also a subtle gap: if the crash happens **after** `recordEnrichmentResult` but **before** the response is returned to the caller, the row is correctly marked complete (idempotent on re-run), but the caller observed `deferred`; this is the documented "use SYNC if read-after-write is required" caveat. `SPECKIT_POST_INSERT_ENRICHMENT_SYNC=true` is the only path that gives the caller a strict consistency story.
- Next experiment: A stress fixture that (1) writes a row with `SPECKIT_POST_INSERT_ENRICHMENT_SYNC=true` then kills the process before the HTTP response is read, then (2) restarts and asserts the row's `post_insert_enrichment_status` is `complete` with the expected entity/edge count (no double-enrichment, no lost enrichment).
- Verdict: Crash safety is correctly engineered around `markEnrichmentPending` + backfill drain. The single durable invariant is "a pending marker means a row needs the bundle run exactly once." No evidence of silent data loss. **SAFE** for the documented async-default contract.

### Angle 8: Self-maintaining index move-reconciliation under concurrent folder moves
- Framing: `reconcileMoves` (`lib/storage/incremental-index.ts:512-628`) pairs old→new paths by `grandparent + basename + document_type` inside a single `incrementalIndex` instance. Under two concurrent `memory_index_scan` calls each observing different in-flight moves (e.g. `012-old` → `012-new` and `013-old` → `013-new` across sessions), what is the correctness / race exposure?
- Repo evidence: `incremental-index.ts:512-628` performs read-then-UPDATE for each candidate pair (`db.prepare(...).run(...)` at line 603-612). The function is **not wrapped in a transaction** — it iterates and issues N separate UPDATEs. The DB write itself is per-row atomic in SQLite, but the *toDelete/toIndex bookkeeping* is not protected. Test fixture `tests/incremental-index-move-reconcile.vitest.ts` runs against a single in-process db with no concurrency. `memory-index.ts:688-696` invokes `reconcileMoves` once per scan, after a `listStaleIndexedPaths`/`listIndexedRecordIdsForDeletedPaths` pass. The single-writer claim is at the scan-job layer (`db-state.ts` lease), not the move layer.
- Findings: Two scenarios are safe and one is not. (1) **Same-packet renames in one scan** — the `reconciledOldPaths`/`reconciledNewPaths` sets (`incremental-index.ts:545-547`) gate iteration; safe in-process. (2) **Concurrent scans seeing the same move** — the lease/heartbeat (db-state.ts) is the job-layer single-writer, so only one scan runs at a time; the other coalesces. (3) **Move between two scan runs** — the old path is in `toDelete` from the pre-move file walk and the new path is in `toIndex`; if both run interleaved across processes (e.g. two `opencode` sessions each calling `memory_index_scan` against the same DB), the second scan may see an `oldPath` that the first scan already UPDATEd, leading to `rows.length !== 1` (line 582) and a benign no-op — but if both scans see the new path as `toIndex` and the old as `toDelete` simultaneously, two parallel UPDATEs on the same `id` are individually atomic but the response counts are non-deterministic.
- Next experiment: A vitest with two `reconcileMoves` calls running via `Promise.all` against the same in-memory db, each holding different `oldPath`/`newPath` pairs, asserting (a) no `UNIQUE` violation, (b) `reconciled.length` equals the union of intended moves, (c) the per-row `id` is unchanged. Add a second test where both calls race on the *same* old/new pair and assert idempotency (only one UPDATE actually changes a row).
- Verdict: In-process correctness is good (the sets protect iteration). The **cross-process / cross-session** race is the real exposure and is **not covered by existing tests**; the production safety net is the scan-job lease, but the lease is per-scan, not per-row-update, so a folder move that lands between two scan windows is the only scenario where the index could go briefly inconsistent (until the next scan). LOW CONFIDENCE on the exact failure rate in production without an end-to-end move-during-scan fixture.

### Angle 9: Multi-tenant scope isolation past the active-row uniqueness guard
- Framing: Phase-4 (commit `942ad78d9c`) shipped a v28 partial unique index and a deprecate-before-insert protocol to keep one active row per `(spec_folder, canonical_file_path, anchor, tenant, user, agent, session)` — but are any read paths past the *write* guard still scope-blind?

- Findings: The active-row uniqueness guard prevents two writes from the same logical key within a scope, but it does **not** prevent a read in a different scope from seeing the row. That is true *only* as long as the production DB stays null-scoped. A future multi-tenant rollout would need the BM25/FTS lane plus the `get_by_path`/`get_by_id` prepared statements to thread the scope through.
- Next experiment: A vitest that seeds two rows with the same `file_path` but different `tenant_id` values, then issues `memory_search({ query, tenantId: 'A' })` and asserts (a) the tenant-B row is **not** in `bm25_hits`, (b) `fts5Bm25Search` SQL EXPLAIN QUERY PLAN shows a `tenant_id IS NULL OR tenant_id = 'A'` predicate, (c) `get_by_path('file_path', { tenantId: 'A' })` returns only the A row. - Verdict: **Multi-tenant scope isolation is structurally correct at the write layer and dormant-incomplete at the read layer.** The active-row uniqueness guard holds, but two read paths (`get_by_path`/`get_by_id` prepared statements and the FTS5 BM25 lane) are scope-blind.
