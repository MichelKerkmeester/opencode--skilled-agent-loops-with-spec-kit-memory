## Analysis: Streaming / Event-Driven / Actor Loop Patterns → deep-loop runtime

All five repos verified live with current star counts and recency. None are in the prior-176 catalogue.

**The core mapping problem.** deep-loop's `runtime/state-jsonl-checkpointing` is an append-only event log, and `runtime/fan-out-fan-in` dispatches parallel sub-agents. The actor/streaming ecosystem has spent 15+ years solving exactly these two problems at scale. The transferable value is not deployment topology (deep-loop runs in-process on small state) but the *coordination patterns* — specifically how these systems get **consistent distributed snapshots without stopping the world**, **stateful workers that survive failure by identity**, and **incremental aggregates that don't recompute from scratch**.

### 1. Orleans virtual actors (dotnet/orleans — 10.8k★, v10.2.1 Jun 2026)
The Virtual Actor Model: grains have a **stable identity** (user-defined key), are **auto-activated on demand**, and **auto-deactivated when idle**. State is kept in memory while active, written to storage on `WriteStateAsync`. The caller never learns *which machine* hosts the grain — placement is transparent. **Reminders** are durable timers that fire even when the grain is deactivated.

**Why it transfers to fan-out-fan-in:** deep-loop currently dispatches parallel Task agents as anonymous ephemeral workers. If fan-out workers were **addressable by stable identity** (e.g. `iteration:38:angle:actor-model`), three things become free: (a) re-visiting an angle in a later iteration naturally re-activates the same logical worker and its accumulated context; (b) a crashed worker is transparently re-activated elsewhere with state restored from its last `WriteStateAsync` — no manual retry bookkeeping; (c) idle workers free memory automatically. This **fuses fan-out and checkpointing**: the worker's activation boundary *is* the checkpoint boundary. The reminder primitive is a model for durable scheduled-continuation: "re-check convergence in N minutes" that survives process death.

### 2. Flink async barrier snapshotting (apache/flink — 26.2k★, 38k commits, active)
Flink's checkpointing is **Chandy-Lamport consistent snapshots without global pause**: a coordinator injects a barrier marker into the stream; each operator snapshots its state on receipt and forwards the barrier; the distributed snapshot is complete when barriers reach all sinks. `flink-state-backends` lets state live in RocksDB or heap. Plus the **iterate operator** makes bounded-stream iteration a first-class primitive.

**Why it transfers to state-jsonl-checkpointing under concurrency:** deep-loop's JSONL is currently single-writer (sequential iterations) so exactly-once is free. The moment fan-out runs *concurrently*, you need a **consistent cut** across all parallel workers to snapshot a coherent iteration state. The barrier pattern maps directly: emit a checkpoint-barrier into the event stream; each parallel sub-agent snapshots partial state on receipt; the iteration state is durably checkpointed only when all barriers are collected. This is the formal mechanism for the open "determinism under concurrency (fan-out correctness)" thread — and it's *exactly* the cost prior threads assumed.

### 3. Kafka Streams changelog state stores (apache/kafka — 33.2k★, 17.8k commits; `streams/` module)
Each stateful operator has a **local store (RocksDB) backed by a Kafka changelog topic**; on failure the store is rebuilt by replaying the changelog. Exactly-once = consume offset + produce output + commit offset **in one transaction**.

**Why it transfers to state-jsonl-checkpointing:** The dual-store pattern validates deep-loop's design (hot state + append-only log) but adds the **transactional-commit insight**: state-write + output-emission + checkpoint-advance must be one atomic unit, else retry causes double-processing. For deep-loop this means: append the finding to JSONL, emit the synthesis, advance the checkpoint cursor — all-or-nothing. The changelog-replay model is also the canonical answer to "how do I rebuild mutable gauge state after a crash?"

### 4. Pekko Persistence (apache/pekko — 1.6k★, 28k commits, 62 tags; fork of Akka pre-BSL)
EventSourced actors **persist events to an append-only journal** and periodically **save snapshots**; recovery replays events *since the last snapshot*. `persistence-query` provides the CQRS read-side projection. **Cluster sharding** distributes actors across nodes by consistent-hashing on entity ID.

**Why it transfers to state-jsonl-checkpointing + dedup-novelty:** This is the textbook event-sourcing model that deep-loop's JSONL already implements implicitly. The key missing piece is the **snapshot cadence**: replaying all 45 iterations of full state on every resume is O(iterations); periodic snapshots collapse resume cost to O(snapshot_age). For `runtime/dedup-novelty`, pure event sourcing (replay all "finding-emitted" events to reconstruct the novelty union set) is O(N) per resume — the snapshot resolves this but introduces snapshot-staleness. This is a concrete, real tension to resolve in the novelty gauge.

### 5. RisingWave incremental materialized views (risingwavelabs/risingwave — 9.2k★, v3.0.1 Jun 2026, Rust)
Explicitly repositioned as **"event streaming platform for agentic AI."** Core mechanism: **incremental computation** — when upstream data changes, only the affected portion of a materialized view is recomputed (freshness <100ms), state in cheap object storage, recovery in seconds.

**Why it transfers to gauges-observability + convergence + dedup-novelty:** deep-loop's gauges (novelty score, convergence metric, budget consumed) are recomputed each iteration by scanning accumulated state — O(total) per gauge update. The incremental-MV lesson: define each gauge as a **stream-materialized incremental fold** over the event log so updates are O(delta). Concretely: the novelty "seen-set" maintained as an incremental union (O(1) add + membership check, not O(N) scan); the convergence metric as an incremental EWMA/variance over per-iteration deltas. This makes the observability layer **scale-independent of iteration count**, which matters as the loop grows past 45 iterations.

### Contradictions surfaced
1. **Concurrency's coordination cost vs. its throughput gain.** Streaming exactly-once (barriers, 2PC transactions, idempotent producers) is *expensive* and only net-positive at Kafka scale (thousands of partitions). deep-loop's fan-out is 3–8 parallel sub-agents. The barrier/transaction overhead that Flink/Kafka justify may be **net-negative at agent-loop scale** — suggesting deep-loop should keep single-writer sequential JSONL and treat fan-out results as *commutative merges* into a single-writer log, rather than adopting true multi-writer concurrency. This contradicts the thread that "we need concurrency for throughput."
2. **Event sourcing vs. mutable set membership.** Pure event sourcing ("never mutate, only append, reconstruct by replay") directly conflicts with `runtime/dedup-novelty`'s need for *mutable union-set* membership tests. Replaying all emitted-finding events to rebuild the novelty set each resume is O(N). Pekko's snapshot resolves it but reintroduces staleness — a real design fork: fast novelty checks (mutable set + snapshot) vs. audit purity (pure replay). deep-loop should make this tradeoff explicit rather than implicit.

### Next divergent angles
- **Supervision-tree / "let it crash" as a deliberate failure model** (Erlang OTP, Pekko supervisor strategies: one-for-one / all-for-one / rest-for-one): model each iteration step as a supervised process that crashes-and-restarts-from-last-checkpoint, replacing try/catch-around-every-LLM-call.
- **Watermark-based lateness windows for fan-out**: in streaming, a watermark declares "no events older than T arrive" — map to declaring a fan-in watermark ("sub-agents older than wall-clock T are presumed failed") to bound the fan-in close-window. Connects to the open bounded-staleness/IS-correction thread.
- **Backpressure as budget control**: streaming systems shed load via backpressure rather than hard rejection — map to `runtime/budget-cost` gracefully degrading (slow fan-out) instead of hard-stopping when budget is exhausted.
