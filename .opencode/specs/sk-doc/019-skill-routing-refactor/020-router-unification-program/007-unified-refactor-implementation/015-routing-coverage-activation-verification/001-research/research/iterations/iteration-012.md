# Iteration 12 — Observability + Archiving for Compiled Routing

## Headline

No per-hub serving-status readout exists today (no CLI, no MCP tool, no plugin row reads the seven activation manifests), and there is no durable hub-local archive for compiled-routing parity evidence — every per-hub audit artifact (`activation-record.json`, `serving-flip-record.json`) is overwritten on the next run, silently losing re-mint history. `spec_kit_skill_advisor_status` (`mk-skill-advisor.js:826-861`) is the lowest-blast place to add the fleet readout, and `.opencode/skills/<hub>/benchmark/compiled-routing/<run-label>/` is the natural durable shape — but 014 leaves both unmapped.

## Verified claims

### Current per-hub state IS on disk, just not surfaced

All seven hubs already carry `servingAuthority: compiled, shadowOnly: false` after the P4b flip (verified by reading each `manifest.json` under `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/010-live-activation/activation/<hub>/manifest.json`). Each hub also has a `fence-state.json` (epochs 2–4), a `serving-flip-record.json` (per-flip audit), and an `activation-record.json` (with `realModelVerification.perModelPass` for `luna`/`minimax`/`sol`). These are durable, byte-stable files — but they live only inside the spec tree, and no module path joins them into one machine-readable snapshot.

### The runtime answers (hub, prompt) but never (fleet → status[])

`.opencode/bin/compiled-route.cjs` (lines 23-40) is purely `(hub, prompt) → decision | legacy-sentinel`. The shim `compiled-route.cjs:36-39` collapses every failure into `{ servingAuthority: 'legacy' }` with no `cause` field (drift vs broken), no live-hash, no eligibility verdict. `resolve.cjs:22-48` gates on the flag and the manifest but only returns a single boolean/null. There is no `compiled-route.cjs --fleet` and no shared `serving-snapshot.cjs` that walks the seven manifests, recomputes each live hash, and emits the `compiled-serving | legacy-fallback | drifted` triple the 012 spec mandates (`spec.md:133`, `plan.md:158`, `tasks.md:86` T007). F-8-4 / F-10-3 already flagged the absence; my contribution is naming the augmentation surface.

### `spec_kit_skill_advisor_status` is the natural front door

`mk-skill-advisor.js:826-861` already returns ~22 lines of prompt-safe plugin status (`cache_hit_rate`, `bridge_invocations`, `advisor_lookups`, etc.) — but never reads the seven manifests at `.opencode/specs/.../010-live-activation/activation/<hub>/manifest.json`. Adding ~10 lines emitting `hub_serving[]` (one row per hub from each manifest + flag + live-hash recompute + parity baseline) gives operators a single tool to inspect fleet state without inventing new CLI surface.

### The 014 Lane C spec has no archive mapping

`014-benchmark-alignment/spec.md:79` says "JSON and Markdown report additions" inside `report.compiledRouting` (a Lane C parity block), and the harness writes to caller-provided `--outputs-dir` (per `run-skill-benchmark.cjs` and the existing `sk-code/benchmark/live-final/` shape). 014 does not name where the per-hub serving snapshot persists across runs — the natural mapping `<hub>/benchmark/compiled-routing/<run-label>/{serving-snapshot.json, serving-snapshot.md}` parallels the Lane C convention (`sk-code/benchmark/README.md:73-83`) but no spec calls for it. A LUNA-high routed-subject run today would either pollute the parity outputs dir or vanish.

### Per-hub audit history is silently overwritten

Both writers overwrite singular files: `activate-hub.cjs:244` writes `activation-record.json`, and `flip-serving.cjs:137` writes `serving-flip-record.json`. No `flip-history.jsonl` or re-mint audit exists. system-deep-loop exposes this concretely: `manifest.json` at fence 3 shows `effectivePolicyHash: 82d7f3af...` with `generation: 3`, but `serving-flip-record.json` at fence 3 (before 2, after 3) shows `effectivePolicyHash: 7b5491c4...` — a silent re-mint between the recorded flip and the current manifest with no audit row.

### Activation manifest is missing fields the readout needs

`010-live-activation/activation/<hub>/manifest.json` carries `{ schemaVersion, selectedPolicy:{ effectivePolicyHash, generation }, servingAuthority, shadowOnly }` (one line each). It does NOT carry `liveConfigHash` (so a probe cannot detect drift without recomputing it from hub inputs, F-16-3) and does NOT carry `engineLocator` / `resolverPath` (so a probe cannot report "promoted vs spec-tree resolver" status from the manifest alone, F-16-2 / ADR-003). The 012 P0 remediation of the spec-tree resolver coupling leaves no manifest-side reflection of the resolved path, so even an authoritative readout would still need to inspect `.opencode/bin/compiled-route.cjs:16-21` separately.

### Real-model evidence is per-run, not per-serving-state

`.opencode/specs/.../010-live-activation/real-model/sk-code/verdict.json` records `perModelPass: { luna: "1/2", sol: "2/2", minimax: "2/2" }` and 6 raw prompt transcripts in `raw/luna-1-1.txt` etc. — but it does NOT include the `servingAuthority` or `effectivePolicyHash` at the moment of the LUNA query, so a reader cannot reconstruct "was this hub compiled-serving under the flag when LUNA answered?". F-11-2 already noted the harness scores the model, not the compiled-router; F-12-4 notes the durable archive shape loses the same data the parity gate would need.

### Existing plugin pattern can be lifted

`.opencode/plugins/mk-dist-freshness-guard.js` detects compiled-output staleness for MCP-server dist artifacts; a parallel `mk-compiled-routing-snapshot.js` plugin shape could emit the per-hub serving snapshot at session start without touching the runtime hot path. This pattern is not named anywhere in the 012/013/014 specs.

## Ranked recommendations

1. **Author `serving-snapshot.json` schema per hub** (sibling to `skill-benchmark-report.json`) and author the rendering template in `create-benchmark`'s `skill-benchmark-storage-guide.md`. Required fields: `hubId`, `capturedAt`, `flag`, `manifest: { selectedPolicyHash, generation, fenceEpoch, servingAuthority, shadowOnly }`, `liveConfigHash`, `freshness: 'fresh'|'stale'|'missing-manifest'|'engine-error'`, `engineResolverPath`, `parityBaseline: { routeGoldHash, parity: 'equal'|'divergent'|'unknown' }`, `realModelLast: { luna, minimax, sol }`. Source of truth: spec tree (manifest + fence + flip + real-model), recomputed at probe time.

2. **Author the per-hub durable archive mapping** as `<hub>/benchmark/compiled-routing/<run-label>/{serving-snapshot.json, serving-flip-history.jsonl, route-gold.json}` following the Lane C sibling-folder convention (`sk-code/benchmark/README.md:75-83`, `create-benchmark` §10 storage guide). Add a `benchmark/README.md` row to all seven hubs pointing at the latest run, with status `current | superseded | frozen`. Without this mapping, 014's parity harness produces evidence that has no home.

3. **Add a `compiledRoutingFleetStatus` block to `spec_kit_skill_advisor_status`** (`mk-skill-advisor.js:826-861`) — read each of the seven manifests, recompute each live hash via the same `loadHubEngine()` path the runtime uses (`compiled-route.cjs:42-58`), classify each hub into one of `compiled-serving | legacy-fallback | drifted`, and emit `hub_serving[]` (7 rows) plus a one-line `fleet_summary` (counts per state). This is a session-start, read-only surface — no hot-path impact. The runtime front door stays unchanged.

4. **Add an append-only `flip-history.jsonl` per hub** so re-mints between flips are auditable. Each line: `{ at, fenceEpoch, kind: 'activate'|'flip'|'re-mint', priorHash, newHash, by }`. This closes the silent-overwrite gap exposed by the system-deep-loop manifest/flip-record hash mismatch and lets the snapshot prove "manifest is currently at fence N, hash X, last re-mint Y days ago".

5. **Extend `create-skill` (013) to mint the manifest into BOTH the spec-tree canonical location and a hub-local `.opencode/skills/<hub>/activation/manifest.json` copy** so the fleet probe can scan one location for "all known hubs including freshly-created-but-not-yet-activated" without walking the spec tree. A create-skill run before activation currently leaves the new hub invisible to the per-hub probe.

6. **Resolve the archive home for `compiled-routing-parity.cjs` output in 014 spec** by adding a `reportPathConvention` paragraph naming `<hub>/benchmark/compiled-routing/<run-label>/skill-benchmark-report.{json,md}` as the durable sibling of `router-final/` and `live-final/`. Without this, parity evidence either overwrites Lane C parity dirs or is dropped after the run.

7. **(novel, archived) Add `mk-compiled-routing-snapshot.js`** as a session-start plugin that runs the per-hub probe and writes/updates `.opencode/skills/<hub>/benchmark/compiled-routing/latest/serving-snapshot.json` with the schema in #1. Lifts the `mk-dist-freshness-guard.js` pattern and gives the 012 P0 observability contract a concrete module path (T007) — none currently exists.
