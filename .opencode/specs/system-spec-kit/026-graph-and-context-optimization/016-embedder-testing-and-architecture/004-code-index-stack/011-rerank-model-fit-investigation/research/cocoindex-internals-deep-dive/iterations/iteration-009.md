# Iter 9 — Stacking analysis — A+B+C+D combinations: complementary, redundant, sequenced?

## TL;DR (2-3 sentences)

Options A (jina-v3) and B (candidate-set fix) are strategically complementary across pipeline stages — A fixes reranker scoring, B fixes which chunks enter the rerank head — but tactically redundant for probes 14+18 where A alone flips both misses to hits with the existing head. Option C (jina+path-class) is wholly redundant over A because path-class boost is a dead variable when all head candidates share `implementation` class. The recommended sequence is D (bench fix) → A (jina-v3 promotion) → B (chunking improvements), with D enabling confident measurement of A's probe-5 regression and B amortizing infrastructure improvements across all future reranker choices.

## Question (restate the scoped RQ)

Stacking analysis across options A/B/C/D. Which combinations are COMPLEMENTARY (B fixes the candidate set, A+B then maximizes both candidate quality + rerank quality)? Which are REDUNDANT (B alone may render A unnecessary)? Which must be SEQUENCED (D first → cleaner measurements → A or B with confidence)? Specifically: if iter 6 finds that hybrid weight imbalance is load-bearing, does jina-v3 (option A) win evaporate after recalibration? Or does jina-v3 still rank better on the recalibrated candidate set? Output: a 2x2 dependency matrix + recommended sequencing.

## Findings (numbered, with citations)

### Pairwise Relationship Analysis

1. **A + B: STRATEGICALLY COMPLEMENTARY (different pipeline stages)**. A (jina-v3 reranker) operates on the post-fusion candidate head — it scores and reorders already-retrieved candidates. B (candidate-set fix) operates on the pre-rerank pipeline — it changes which chunks enter the head, and which content the reranker evaluates. They affect different pipeline stages: B cannot fix a reranker that scores wrong, A cannot fix candidates that never reach the head. Together they provide defense-in-depth: B ensures the right candidates with rich content enter the head, A ensures those candidates are scored highly once there. Citation: pre_score identity at `baseline-bge.rerank-scores.jsonl:6` vs `jina-v3.rerank-scores.jsonl:6` proves A operates on unchanged head; Iter 2+4 evidence at `iteration-002.md:29-35` and `iteration-004.md:13-24` proves B operates on chunk boundaries and query expansion before rerank.

2. **A vs B for probes 14+18: TACTICALLY REDUNDANT (A alone fixes them)**. A (jina-v3) flips probes 14 and 18 from MISS to HIT using the IDENTICAL candidate head that BGE fails on. B would change the head composition (different chunks of structural-indexer.ts, different file candidates entirely). But for these specific probes, the right file IS already in the head — structural-indexer.ts is at pre_score rank 1 (probe 14) and test_refresh_split.py is at pre_score rank 10 (probe 18). The failure is inside the reranker, not before it. A fixes it decisively at S cost; B might also fix it at M cost but is UNPROVEN. B is not globally redundant however — future probes may fail at pre-rerank stage where the right file never enters the head, and A cannot rescue candidates it never sees. Citation: `phase2-comparison.md:7-14`, `baseline-bge.rerank-scores.jsonl:6,8`, `jina-v3.rerank-scores.jsonl:6,8`, Iter 8 cost table at `iteration-008.md:37`.

3. **C vs A: C is WHOLLY REDUNDANT with A**. Option C (jina-v3 + path-class boost) adds path-class scoring on top of jina-v3. However, path-class boost is a dead variable for the known failure probes: probe 14's head candidates all share `implementation` class (multiplier 1.00 — identity), probe 18 has `docs` vs `tests` classes with equal 0.85 multipliers (zero relative reordering), probe 5 has both expected and stress tests as `tests` class (equal 0.85). C adds zero incremental failure-fixing power over A. C also adds complexity: two env vars (`COCOINDEX_RERANK_ADAPTER` + `COCOINDEX_RERANK_PATH_CLASS_BOOST`), interaction testing, documentation. The Iter 7 utility gap of 0.008 (A=0.759 vs C=0.751) is driven entirely by the downstream-unblock score penalty for wasted complexity. Verdict: deploy A alone; C is speculative overhead. Citations: `reranker.py:20-47`, `config.py:20-30`, `phase2-comparison.md:26`, `iteration-007.md:21-22`.

4. **D MUST precede B for confident measurement**. B requires a full re-index (1.2 GB DB discard, 15-40 minute wall-clock wait) followed by a full 8-probe bench re-run. Without D's bench fixes: (a) probe 10's fixture-truth mismatch (`.js` dist vs `.ts` source) wastes a bench lane on an unfixable probe, (b) parser artifacts (import statements, mock data strings misidentified as paths) add noise to hit/miss classification, (c) remaining fixture-truth bugs in probes 1-8 are un-audited. Running B before D means investing 1-2 engineer-days in chunking experiments whose results are partially contaminated by measurement noise. D is zero-cost (S dev, 0 min, 0 operator) and should be sequenced first. Citation: `baseline-bge.results.jsonl:4,6,8` (probe 10 fixture-truth + parser artifacts), `iteration-008.md:37-38` (re-index cost), `phase2-bench/README.md:7` (probe 10 ambiguity documented).

5. **D and A can proceed in PARALLEL**. D (bench fix) modifies only the bench script and fixture files — it does not touch the daemon, DB, or CocoIndex code. A (jina-v3 promotion) modifies only the reranker adapter chain in `reranker.py:191-223` and optionally the jina adapter module header. These are independent subsystems with zero code overlap. A's promotion decision is already well-supported by Phase 2 bench data — the pre_score identity across lanes proves the reranker difference is real, and the probe-14/18 flips are unequivocal. D improves confidence in A's probe-5 regression diagnosis and probe-1 shared-miss analysis, but does not gate A's deployment. Both can proceed concurrently. Citation: `reranker.py:191-223` (adapter dispatch, zero dependency on bench harness), `run-phase2-smoke.sh:117-168` (parser logic, zero dependency on daemon).

6. **Hybrid weight recalibration sub-question: PRE-ANSWERED — not load-bearing**. The Iter 6 adversarial synthesis already ranked hybrid weight imbalance at probability 5% (second-lowest of 6 hypotheses). The RRF defaults at `fusion.py:42-87` use equal 0.7/0.7 weights and `k=60`. The structural-indexer BENEFITS from being dual-lane: its pre_score `0.0216` is ~2x the stress test's single-lane `0.0115`. Changing weights would alter the absolute scores but not the pre_score ordering between structural-indexer and the stress test — structural-indexer would still be rank 1. The reranker receives the same head regardless of weight tuning. jina-v3's win does NOT evaporate after hybrid recalibration because: (a) the head composition is unchanged, (b) jina's scoring advantage is a property of its cross-encoder architecture, not of the pre_score distribution, (c) the pre_score ordering is stable under weight variation for this query. Citations: `fusion.py:42-87`, `config.py:15-17`, `iteration-006.md:40-42`.

7. **jina-v3 advantage persists after chunking improvements (B does not render A unnecessary)**. If B improves chunking so a content-rich structural-indexer chunk (e.g., `findFiles` at lines 1391-1465) reaches pre_score rank 1 instead of the 9-line import header, BGE might score it higher — perhaps `0.05` to `0.06` instead of `0.025`. However, the stress test `walker-dos-caps.vitest.ts` scores `0.076` under BGE because it LITERALLY contains the query terms "walker", "filesystem", "edges" in test code. BGE's cross-encoder is fundamentally lexical-cue-driven; richer implementation code may add more semantic tokens but will not add the exact query words that the stress test already has. Jina v3 scores structural-indexer at `0.083` to `0.110` even with the 9-line header chunk — its advantage is architectural (less lexical-cue-dependent), not content-dependent. With richer chunks, jina could score structural-indexer even higher. Conclusion: A does NOT evaporate after B; at worst, B narrows the gap under BGE (partial win), while A remains the decisive margin for the failure probes. Citations: `baseline-bge.rerank-scores.jsonl:6` (BGE structural-indexer 0.025 vs stress test 0.076), `jina-v3.rerank-scores.jsonl:6` (jina structural-indexer 0.083-0.110 vs stress test 0.110), `iteration-002.md:29-35` (chunk 44683 is 9-line import header), `iteration-006.md:29-47` (two-factor root cause: lexical-cue-density times chunk boundary).

8. **A + B together provide probe-5 regression mitigation**. Probe 5 (the observed jina-v3 control regression) is a different failure class: the expected file IS in the head but jina scores it below the stress test. Under BGE, both score similarly (~0.77 each) because both are `tests`-class files with similar lexical content. Under jina, the stress test scores 3.2x higher (0.468 vs 0.147). If B improves chunking so the expected `code-graph-status-readiness-snapshot.vitest.ts` chunk contains richer, more distinctive content (e.g., more specific test assertions, unique function names), jina might score it higher relative to the stress test. B does not fix probe 5 by itself (BGE already hits it), but B + A together provide a mitigation pathway for the regression. This makes A + B complementary for robustness: A wins the failure probes, B reduces the regression risk. Citation: `baseline-bge.rerank-scores.jsonl:3` (BGE scores both ~0.77), `jina-v3.rerank-scores.jsonl:3` (jina scores stress test 3.2x higher), `iteration-002.md:29-35` (chunking affects which test assertions appear in top-ranked chunk).
