# Iteration 11: W4 deep-dive: vector + FTS replacement design

## Focus
This iteration produces concrete replacement designs for the two extension-dependent search subsystems that block a Turso (tursodb v0.7.0-pre.6) migration: (1) the sqlite-vec `vec0` virtual table `vec_memories` — exact replacement DDL, the KNN query rewrite onto `vector_distance_cos ... ORDER BY ... LIMIT`, the `reindex.ts` `writeVectorsToKnn` rewrite, compatibility of the staging-shard atomic-rename flow, and a quantized vector8 two-stage variant with concrete SQL; and (2) the FTS5 external-content `memory_fts` table — Turso Tantivy FTS DDL, rewrites for the bm25-index.ts gates and the 3 sync triggers, the JS-SDK surface for the `index_method` experimental flag, the blast radius if FTS is still experimental at adoption, and the ranking drift between Tantivy's index-time weights and the query-time `bm25()` column weights used today.

## Findings

**Q1 — vec_memories replacement design**

- **KNOWN / UNCHANGED** — Current DDL is `CREATE VIRTUAL TABLE active_vec.vec_memories USING vec0(embedding FLOAT[${profile.dim}])` created in the attached shard [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/search/vector-index-store.ts:516-522] and mirrored in the staging shard writer [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/embedders/reindex.ts:437-443], alongside a plain canonical blob table `vec_<dim>(id INTEGER PRIMARY KEY, vec BLOB NOT NULL)` [vector-index-store.ts:509-514; reindex.ts:383-394]. Turso has no vector index — linear scan only, vectors as plain BLOBs [SOURCE: external/turso-main/docs/sql-reference/functions/vector.mdx:392-394 Note; :40-47].
- **NEW** — Exact replacement DDL: `CREATE TABLE IF NOT EXISTS active_vec.vec_memories (rowid INTEGER PRIMARY KEY, embedding BLOB NOT NULL)`. vec0's implicit rowid becomes an explicit `INTEGER PRIMARY KEY` (rowid alias), so all existing `rowid`-addressed reads/writes keep their shape. Raw `Float32Array` buffers from `to_embedding_buffer()` insert directly — the f32-blob layout is sqlite-vec-compatible (established iter 1–9) and `vector_distance_cos` accepts raw f32 BLOBs of matching dimension; no `vector32(json)` round-trip needed [vector.mdx:130-161 plus established].
- **NEW** — KNN query rewrite shape: `SELECT rowid, vector_distance_cos(embedding, :queryVec) AS distance FROM active_vec.vec_memories ORDER BY distance ASC LIMIT :k` [SOURCE: vector.mdx:339-347 canonical example]. Distance is `1 − cosine_similarity`, so existing similarity conversion is `similarity = 1 − distance` (today sqlite-vec's cosine distance has the same convention; no score-semantics change).
- **PARTIAL / CHANGED** — Dimension discovery: the fallback that regexes `FLOAT\[(\d+)\]` out of `sqlite_master.sql` [vector-index-store.ts:258-273] dies with the vec0 DDL; the authoritative source is already `vec_metadata.embedding_dim` checked first [vector-index-store.ts:213-256], so the migration only needs to delete the regex branch, not redesign discovery.
- **NEW** — `writeVectorsToKnn` rewrite [reindex.ts:365-381]: today it does `DELETE FROM vec_memories WHERE rowid = ?` then `INSERT INTO vec_memories (rowid, embedding) VALUES (?, ?)` per row with `BigInt(row.id)` inside `db.transaction()`. On a plain table this collapses to a single `INSERT OR REPLACE INTO vec_memories (rowid, embedding) VALUES (?, ?)` (vec0 forbade OR REPLACE; plain tables don't). `transaction(fn)` exists in the compat surface (established iter 1–9). Verification item: BigInt parameter binding through the compat layer is untested. The whole `sqliteVec.load()` / `vecAvailable` branching in `writeVectorsToShard` [reindex.ts:424-447] and the temp-view INSTEAD-OF alias layer [vector-index-store.ts:551-612] become deletable.
- **NEW (real gap)** — Staging-shard atomic-rename flow [reindex.ts:511-590]: batches write to `<active>.reindex-<jobId>.staging`, the shard is closed after `shard.pragma('wal_checkpoint(TRUNCATE)')` [reindex.ts:454], active `-wal`/`-shm` sidecars are unlinked, then `fs.renameSync(staging, active)` [reindex.ts:579-589]. The rename itself is filesystem-level and engine-agnostic, **but** the flow's correctness depends on the staging WAL being empty at close, and COMPAT.md marks `PRAGMA wal_checkpoint` as 🚧 Partial with "Not Needed calling with param (pragma-value)" — i.e. the parameterized `wal_checkpoint(TRUNCATE)` form is unsupported [SOURCE: external/turso-main/COMPAT.md:215]. Adjacent shard pragmas also break: `wal_autocheckpoint` ❌ [COMPAT.md:214; used at vector-index-store.ts:503] and `mmap_size` ❌ [COMPAT.md:183; used at :501]; `journal_mode`, `cache_size`, `temp_store` are ✅ [COMPAT.md:177,148,205]. Design fix: use the bare `PRAGMA wal_checkpoint;` (the supported form) and verify the returned frame counts, or rename the `-wal`/`-shm` sidecars together with the main file instead of relying on an empty WAL. This is a new P1 item for the Path C-prime adapter, on top of the established bare-VACUUM blocker at db-shard-migration.ts:274/:282.
- **NEW** — Quantized two-stage variant (concrete SQL), per the established vector8 lever: store a companion `embedding_q8 BLOB` (1 byte/dim → coarse scan reads 384 B/row instead of 1,536 B/row), then:
  ```sql
  WITH coarse AS (
    SELECT rowid, embedding
    FROM active_vec.vec_memories
    ORDER BY vector_distance_cos(embedding_q8, :queryVecQ8) ASC
    LIMIT :k * 4
  )
  SELECT rowid, vector_distance_cos(embedding, :queryVec) AS distance
  FROM coarse ORDER BY distance ASC LIMIT :k;
  ```
  Caveat: the vendored doc defines `vector8(json_array)` as taking integers 0–255 [vector.mdx:90-105], so it may not auto-quantize floats — the adapter would quantize in JS (affine scale f32→u8, same transform on the query vector). Whether `vector8()` accepts float arrays and scales internally (as libSQL's FLOAT8 does) is a verification item. At the current corpus scale (low thousands of memories) the plain f32 linear scan is likely fine and the two-stage variant is an optimization to keep in reserve.

**Q2 — memory_fts replacement design**

- **KNOWN / UNCHANGED** — Today: FTS5 external-content table `CREATE VIRTUAL TABLE memory_fts USING fts5(title, trigger_phrases, file_path, content_text, content='memory_index', content_rowid='id')` plus 3 sync triggers `memory_fts_insert` / `memory_fts_update` / `memory_fts_delete` using the FTS5 `'delete'` command protocol [SOURCE: vector-index-schema.ts:1255-1281, migration v14]. Gates: `isMemoryFtsAvailable` probes `sqlite_master WHERE type='table' AND name='memory_fts'` [bm25-index.ts:478-480] and `shouldUseSqliteLexicalEngine` routes engines, throwing in `sqlite` mode without the table and falling back to in-memory JS BM25 in `auto` mode [bm25-index.ts:520-538, 508-509]. Weighted query: `SELECT m.*, -bm25(memory_fts, 10.0, 5.0, 2.0, 1.0) AS fts_score FROM memory_fts JOIN memory_index m ON m.id = memory_fts.rowid WHERE memory_fts MATCH ?` [SOURCE: sqlite-fts.ts:193-196; weights at bm25-index.ts:71-78].
- **NEW** — Replacement DDL: Turso FTS is an index on the base table, not a separate content table:
  ```sql
  CREATE INDEX memory_fts ON memory_index
  USING fts (title, trigger_phrases, file_path, content_text)
  WITH (tokenizer = 'default',
        weights = 'title=10.0,trigger_phrases=5.0,file_path=2.0,content_text=1.0');
  ```
  [SOURCE: external/turso-main/docs/fts.md:88-134]. Consequence: **all 3 sync triggers are deleted outright** — INSERT/UPDATE/DELETE on `memory_index` maintain the index automatically (UPDATE = delete+insert internally, commits batched at 1,000 docs) [fts.md:264-275]. Gate rewrites: both `sqlite_master` probes (bm25-index.ts:478; sqlite-fts.ts:115) must check `type='index'` instead of `type='table'` (the DDL lives in the schema table as an index [fts.md:283-286]). Query rewrite: `SELECT m.*, fts_score(m.title, m.trigger_phrases, m.file_path, m.content_text, :q) AS fts_score FROM memory_index m WHERE fts_match(m.title, m.trigger_phrases, m.file_path, m.content_text, :q) ORDER BY fts_score DESC LIMIT :k` — no JOIN, no `-bm25` sign flip (fts_score is already higher-is-better), and function-recognition mode tolerates extra WHERE filters [fts.md:152-243].
- **NEW** — `index_method` from the JS SDK / embedded use: confirmed first-class in the SDK, not CLI-only. `connect(":memory:", { experimental: ["index_method"] })` is exercised in the bindings' own FTS test [SOURCE: external/turso-main/bindings/javascript/packages/native/promise.test.ts:246-252]; `'index_method'` is in the `ExperimentalFeature` union [bindings/javascript/packages/common/types.ts:1]; the NAPI layer maps it to `core_opts.with_index_method(true)` [bindings/javascript/src/lib.rs:280]; the Rust sync Builder also gained `experimental_index_method` [COMPAT/CHANGELOG.md:138]. **PARTIAL**: whether the better-sqlite3-style compat entrypoint (`./compat`) forwards the `experimental` option array is unverified in the vendored tree — verification item for the adapter pilot.
- **NEW** — What breaks if FTS is still experimental at adoption: (1) the flag is per-connection — any process opening the DB without it (scripts, doctor tools, other runtimes) hits the `USING fts` DDL in the schema as an unknown index method, the schema-poisoning failure mode; (2) established mutual exclusion with MVCC forecloses the concurrent-writer lane for the whole DB; (3) unconditional NoMergePolicy means segments accumulate per 1,000-doc commit, requiring a new scheduled `OPTIMIZE INDEX` maintenance job that FTS5 never needed [fts.md:421-444, 450-457]; (4) no read-your-writes inside a transaction [fts.md:456] — any save-pipeline step that inserts a memory and lexically searches it pre-COMMIT silently misses it (post-insert enrichment/dedup paths need an audit); (5) no cross-version on-disk stability promise — though a full rebuild from `memory_index` is already a supported, cheap flow at this corpus size. **De-risk lever unique to this codebase:** `engine='auto'` already degrades to the in-memory JS BM25 when `memory_fts` is absent [bm25-index.ts:508-509], so the safe adoption sequence is: migrate without Turso FTS (JS BM25 carries lexical search), enable the FTS index behind the flag later.

**Q3 — Tantivy weighted scoring vs FTS5 query-time bm25() weights**

- **NEW** — Structural difference: FTS5 weights are query-time arguments (`bm25(memory_fts, 10,5,2,1)` chosen per call); Turso weights are **index-time** (`WITH (weights='...')`, fixed at CREATE INDEX; changing them = DROP + re-CREATE = full re-index) [fts.md:117-134], with query-time `field:term^boost` syntax as a per-query escape hatch [fts.md:258]. Today's weights are a build-time constant (`BM25_FTS5_WEIGHTS`, never varied per query) [bm25-index.ts:71-78], so encoding `title=10.0,trigger_phrases=5.0,file_path=2.0,content_text=1.0` at index time is a lossless mapping for current behavior.
- **NEW** — Expected ranking drift, largest first: (a) **per-segment statistics under NoMergePolicy** — Tantivy computes BM25 over per-segment term stats, and with merging disabled, identical documents in different segments score differently and IDF never converges to corpus-global until `OPTIMIZE INDEX`; (b) **default boolean operator** — Tantivy QueryParser treats multi-term queries as OR [fts.md:252] while FTS5 MATCH defaults to implicit AND, changing the candidate set, not just its order — compensate by having `sanitizeFTS5Query`'s successor emit explicit `AND` between tokens; (c) minor score perturbation from Tantivy's quantized (byte) fieldnorms vs FTS5's exact column-length normalization, with the same k1=1.2/b=0.75 defaults on both sides.
- **NEW** — Compensation plan: encode today's weights index-time; schedule `OPTIMIZE INDEX` after bulk ingest/reindex to stabilize IDF; keep fusion rank-based (RRF tolerates absolute-score drift — the code already documents that BM25 scores are unbounded and must be normalized/RRF'd before mixing [bm25-index.ts:24-27]); re-validate any absolute fts_score thresholds; and regression-compare top-k overlap/NDCG (not raw scores) using the existing benchmark fixtures under `mcp_server/benchmarks/`.

## Ruled Out
- Porting the vec0 virtual table or the temp-view INSTEAD-OF alias layer to Turso — no extension loading in the compat surface (established: `loadExtension()` not implemented), and the plain-table replacement makes the entire alias layer (vector-index-store.ts:551-612) deletable rather than portable.
- Porting the 3 FTS5 sync triggers — Turso FTS indexes the base table directly; the FTS5 `'delete'`-command external-content protocol has no equivalent and no need.
- Relying on the `FLOAT[(\d+)]` sqlite_master regex for dimension discovery post-migration — `vec_metadata` is already the primary source and must become the only one.
