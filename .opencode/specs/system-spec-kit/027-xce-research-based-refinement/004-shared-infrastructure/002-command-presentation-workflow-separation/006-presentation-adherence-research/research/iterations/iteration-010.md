# Iteration 010 (deepseek-v4-pro)

**Summary:** Memory's four routers fail render-contract adherence for two interacting reasons: they lack the 'Presentation Boundary' exclusion-zone section that all 13 non-memory routers have, and they embed inline routing detail that lets mid-tier models produce output without ever reading the presentation file. The minimal patch: add Presentation Boundary sections + HARD RULE binding output to the exact parseable template in the presentation contract — ~8 lines per memory router, highest ROI. The deeper fix (creating YAML workflow assets per memory command) addresses the architectural root cause but requires ~4 new YAML files with 'read presentation contract → execute → render from contract' steps.

## [P1][DOC-DRIFT] Memory routers lack Presentation Boundary section — all other families have it
- Evidence: 13 routers across doctor/speckit/deep/mcp contain a 'Presentation Boundary' section (e.g. doctor/speckit.md:65, speckit/resume.md:42, deep/start-research-loop.md:121). The 4 memory routers (search.md, save.md, manage.md, learn.md) have none. This exclusion-zone section is the strongest adherence signal because it tells the model 'I do NOT own these display elements — go read the presentation file.'
- Detail: Every command family with proven good adherence (doctor, speckit, deep) includes an enumerated Presentation Boundary section listing what display content lives exclusively in the presentation file. Memory routers use only advisory language like 'read the presentation asset and use it as the display source of truth' (search.md:18), which mid-tier models treat as optional. The Presentation Boundary creates a hard semantic cut — the router declares it owns zero display text, forcing the model to acquire templates from the contract.
- Fix sketch: Add a '## Presentation Boundary' section to all 4 memory routers enumerating owned display types per command, matching the doctor/speckit pattern exactly.

## [P1][BUG] search.md inline routing detail creates a self-sufficient path that bypasses the presentation contract
- Evidence: search.md:40-47 lists explicit tool calls ('Prefer memory_context({...})', 'Use memory_quick_search for fast query-only fallback') followed only by 'Render compact, parseable output using the presentation asset.' A mid-tier model can follow the tool-calling instructions, get results, and emit free prose without ever reading the presentation file because the routing instructions are complete enough to produce output.
- Detail: Doctor/speckit routers separate tool selection from display by delegating tool calls to YAML workflows, creating a natural stop point where the model needs display instructions. Search.md combines both in one file. The presentation reference at line 47 is the 7th instruction in a list — models often skip the last step after completing the concrete tool-calling sequence. The parseable template at search_presentation.md §2 (MEMORY:SEARCH header + STATUS=OK line) is never enforced.
- Fix sketch: Add a HARD RULE to search.md §5: 'Result display MUST follow the exact parseable template at presentation asset §2 with the MEMORY:SEARCH header and STATUS= line; never emit free prose for retrieval results.'

## [P1][DOC-DRIFT] Memory family is the only command family without YAML workflow assets — root cause of the architectural gap
- Evidence: All 4 memory routers document this explicitly: search.md:15, save.md:15, manage.md:15, learn.md:15 each state 'Missing upstream asset — Keep routing in this file until a workflow YAML is introduced by a separate workflow-asset change.' Doctor/speckit/deep each have YAML assets with structured execution steps that separate display from logic.
- Detail: The YAML workflow architecture is the root cause of good adherence in non-memory families because YAML enforces ordered execution steps, each owning a specific concern. A YAML step that must render output naturally forces the model to pause and seek the presentation contract. Without YAML, the model processes a single .md file with mixed routing, tool-calling, and display instructions, allowing it to short-circuit the display phase. The routers' own table documents this as a known architectural gap.
- Fix sketch: Create minimal YAML workflow assets for each memory command with 'read presentation contract' as the first step and 'render output from presentation contract using the resolved result' as the final step.

## [P2][REFINEMENT] search.md §2 startup routing creates a circular dependency for the startup question
- Evidence: search.md:22 says 'ask one open-ended question from the presentation asset' but search_presentation.md §1 (lines 9-11) is the source of that question text ('What would you like to retrieve or analyze?'). If the model follows the routing instruction without first reading the presentation asset (the sequence in the .md: routing §2 comes right after the advisory line at §1), it has no question text and may invent one.
- Detail: Doctor routers explicitly order steps: 'Execution order: 1. Read presentation.md, 2. Read routes.yaml, 3. Parse target...' (doctor/speckit.md:43-52). Search.md has no such ordering — its startup routing section (§2) references the presentation asset but does so before establishing that the file must be read FIRST. Mid-tier models that skip the advisory line at §1 and jump to §2 have no startup question text, leading to invented wording.
- Fix sketch: Add numbered execution order to search.md matching doctor's pattern: step 1 reads the presentation asset, step 2 parses $ARGUMENTS using the now-loaded startup question.

## [P2][REFINEMENT] Presentation contracts lack self-enforcing authority language about their own format exclusivity
- Evidence: search_presentation.md §2 (lines 26-47) defines parseable templates but has no statement like 'This is the ONLY valid output format. Never emit free prose for retrieval results.' Compare with how doctor presentation (doctor_speckit_presentation.md:3) declares 'This file is the single source of truth for user-facing presentation.' Search_presentation §2 gives the template but does not claim it is the exclusive format.
- Detail: Presentation contracts can strengthen adherence by self-declaring their authority. When a mid-tier model reads search_presentation.md §2, it sees a template but may still interpret it as just an example. Adding 'This is the ONLY valid output format for retrieval results' makes the contract self-enforcing. The doctor presentation wins because it declares single-source-of-truth status upfront plus the router's Presentation Boundary reinforces it. Memory has neither.
- Fix sketch: Add a line to search_presentation.md §1 and §2 header: 'This section is the ONLY valid output format. Do not emit free prose.'

## [P3][REFINEMENT] Existing YAML workflows in doctor/speckit/deep lack an explicit 'Read presentation contract' step
- Evidence: doctor_memory.yaml (242 lines) contains detailed phase-by-phase workflow steps but never includes 'Read .opencode/commands/doctor/assets/doctor_speckit_presentation.md.' The output contract at lines 214-224 defines required report sections but does not reference the presentation file. Current good adherence depends entirely on the router's Presentation Boundary, not on YAML-level enforcement.
- Detail: The YAML workflows that 'work' do so because the router .md enforces the presentation boundary, not because the YAML workflow reinforces it. If a future model or execution path bypasses the router and loads YAML directly, it would have no instruction to read the presentation contract. Adding a 'Read presentation' step to every YAML makes the contract durable regardless of entry point.
- Fix sketch: Add 'Read the presentation contract at <path>' as the first workflow step in every YAML asset across doctor, speckit, and deep families.

## [P2][DOC-DRIFT] All 4 memory routers share the same weak presentation-reference pattern — fix should be uniform
- Evidence: search.md:18, save.md:18, manage.md:18, learn.md:18 each use identical or near-identical advisory language ('Before rendering..., read the presentation asset and use it as the display source of truth.') but none has a Presentation Boundary section, numbered execution order, or a HARD RULE binding output format to the presentation contract. The pattern is uniform across the family.
- Detail: The uniform weakness means a single-template fix applies to all 4 routers. Save.md has a particularly high risk because it controls writes — a model that invents its own save-plan display could misreport what was saved. Manage.md has 11 sub-modes each with their own presentation template in manage_presentation.md; each mode has a specific parseable output format that the model can only discover by reading the contract file.
- Fix sketch: Apply the Presentation Boundary section + numbered execution order pattern uniformly to save.md, manage.md, and learn.md.
