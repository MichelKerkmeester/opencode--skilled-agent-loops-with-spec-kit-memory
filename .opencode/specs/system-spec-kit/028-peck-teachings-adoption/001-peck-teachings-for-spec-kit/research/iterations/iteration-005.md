# Findings

## Finding 1: The rollout template should be "warn-only first, enforce later" for new quality gates, but T3 can skip that because it is template guidance only.

Spec-kit already has a concrete warn-only precedent: `SPECKIT_SAVE_QUALITY_GATE` is default-on but can be disabled, records an activation timestamp, and treats the first 14 days after activation as warn-only so scores are logged without blocking saves. [SOURCE: `.opencode/skills/system-spec-kit/mcp_server/lib/validation/save-quality-gate.ts:226-258`] [SOURCE: `.opencode/skills/system-spec-kit/mcp_server/tests/save-quality-gate.vitest.ts:182-215`] The runtime gate explicitly logs would-reject events during warn-only and returns `pass: true`; after the warn-only period, normal pass/fail semantics apply. [SOURCE: `.opencode/skills/system-spec-kit/mcp_server/lib/validation/save-quality-gate.ts:753-848`] There is also a per-call `qualityGateMode: 'enforce' | 'warn-only'` path for spec-doc indexing, where V-rule, sufficiency, and template-contract failures can be logged instead of rejected. [SOURCE: `.opencode/skills/system-spec-kit/mcp_server/handlers/memory-save.ts:2130-2145`] [SOURCE: `.opencode/skills/system-spec-kit/mcp_server/handlers/memory-save.ts:2167-2213`]

Cross-teaching implication: T4 and T1 should reuse this graduated pattern because they add validation semantics. T2 should stay diagnostic/read-only and does not need a blocker rollout. T3 is additive template text and should instead be verified by scaffold + strict validation; no warn-only period is needed unless implementation chooses tracked `##` sections, which prior research ruled out for the first pass. [SOURCE: `.opencode/specs/system-spec-kit/028-peck-teachings-adoption/002-self-check-templates/spec.md:93-97`] [SOURCE: `.opencode/specs/system-spec-kit/028-peck-teachings-adoption/002-self-check-templates/spec.md:121-125`] [SOURCE: `.opencode/specs/system-spec-kit/028-peck-teachings-adoption/001-peck-teachings-for-spec-kit/research/iterations/iteration-001.md:17-28`]

## Finding 2: `WARNING` is not safely advisory during completion; strict mode turns it into a blocker.

The validation reference defines `WARNING` as exit 0 in normal mode but exit 2 in strict mode, while `INFO` stays exit 0 in both modes. [SOURCE: `.opencode/skills/system-spec-kit/references/validation/validation_rules.md:27-35`] `validate.sh` encodes that behavior in JSON pass/fail and process exit: warnings fail `passed` under strict and exit 2 when strict mode is active. [SOURCE: `.opencode/skills/system-spec-kit/scripts/spec/validate.sh:930-945`] [SOURCE: `.opencode/skills/system-spec-kit/scripts/spec/validate.sh:1056-1059`]

Cross-teaching implication: the initial T4 rule should be `INFO` unless a specific strict-mode escape hatch or rollout flag prevents completion failures. This matches iteration 002's conclusion and the phase 003 open question. [SOURCE: `.opencode/specs/system-spec-kit/028-peck-teachings-adoption/003-current-state-discipline/spec.md:147-160`] [SOURCE: `.opencode/specs/system-spec-kit/028-peck-teachings-adoption/001-peck-teachings-for-spec-kit/research/iterations/iteration-002.md:14-16`] Any future T1 `AC_COVERAGE` rule can start as `INFO` or `WARNING` only if it is explicitly outside completion strict mode; otherwise its first default-on release should be opt-in or warn-only to avoid breaking unrelated packets.

## Finding 4: The planned order T3 -> T4 -> T2 -> T1 is risk-based, not technically dependency-based.

The parent spec states that phases 002-004 implement the three lowest-risk improvements in ascending-risk order and that T1 is excluded because it has the largest blast radius. [SOURCE: `.opencode/specs/system-spec-kit/028-peck-teachings-adoption/spec.md:64-68`] The parent open questions confirm the order is fixed by ascending risk and that the phases are technically independent; the order warms up the pattern rather than satisfying dependencies. [SOURCE: `.opencode/specs/system-spec-kit/028-peck-teachings-adoption/spec.md:128-133`] The source analysis recommends the same sequence: T3 trivial/additive, T4 advisory reuse of an existing rule, T2 additive lifecycle diagnostic, and T1 last because it needs assertion syntax, checklist changes, a new rule, warn-only rollout, and per-level opt-in. [SOURCE: `.opencode/specs/system-spec-kit/028-peck-teachings-adoption/001-peck-teachings-for-spec-kit/peck-teachings-analysis.md:289-301`]

Cross-teaching implication: no phase technically blocks another, but the safest operational order remains T3 -> T4 -> T2, with T1 deferred. T3 establishes low-risk template-edit discipline and scaffold validation. T4 exercises advisory validation and strict-mode policy. T2 adds a non-mutating review surface. T1 should consume the lessons from all three before introducing a completion gate.

## Finding 5: Verdict ownership should differ by blast radius: self-attestation is enough for T3/T4/T2 implementation checks, but not enough for T1 coverage verdicts.

T3 completion can be self-verified by checking generated scaffolds and strict validation because the change is static template guidance and the phase acceptance criteria are structural.
