## [**1.3.1.0**] - 2026-07-14

Safety hardening of the owner-first worktree tooling after an adversarial review reproduced correctness/safety defects that the original harnesses (which never exercised adversarial inputs) had missed. No public interface, branch grammar, or command surface changed — every fix is internal hardening plus a permanent regression test, so the shipped safety guarantees are now backed by code rather than assertion.

---

## What Changed

#### Allocator — race-safe number reservation

- **`worktree-naming.sh` lock is now genuinely serial under stale-lock contention.** The previous dead-lock reclaim could let two contenders proceed as owner and mint the same number (N contenders collapsing to fewer than N distinct numbers). Reclaim now uses an atomic rename-steal — only one contender can win renaming the lock directory aside, and it discards the stale lock and re-races a fresh `mkdir` rather than re-inserting it — so N concurrent allocations always yield N distinct, monotonically increasing numbers. A high-water persistence failure now aborts allocation without emitting a number, and the validators enforce the documented `0001..9999` bound and the exact `.worktrees/<pair>` path. Verified across 100 consecutive stale-lock-contention runs, all clean.

#### Reaper — reaps only a proven-inactive exact wrapper pair

- **`worktree-reaper.sh` no longer trusts a malformed marker or a mis-shaped branch.** The whole session-marker file must match a PID grammar (a garbage marker is treated as ambiguous and the worktree is kept — absence of proof is never proof of absence); a `work/*` branch is reaped only when it is an exact `work/<runtime>/<slug>` triple whose runtime/slug equals the directory basename; and `--reap-daemons` signals a daemon only when the worktree/DB path it references is actually gone and the PID identity still matches.

#### Session wrapper — hardened inputs

- **`worktree-session.sh` rejects adversarial inputs before they can act.** A runtime identity that cannot form a legal Git ref is refused before `worktree add`; the child exec no longer leaks the diagnostic reason into the runtime argv; and any `SPECKIT_WORKTREE_SHARED_PATHS` entry with `..`/absolute components is rejected, with source and destination canonicalized and containment-checked so a shared-path copy can never delete or symlink outside the worktree.

#### Pre-push — fail-open, version-controlled owners

- **The `pre-push` naming validator is tri-state.** A genuine invalid name still blocks, but an internal validator error (for example a failed owner-discovery scan) fails **open** so a tooling fault never blocks a legal push, and the authorized-owner set is derived only from version-controlled `SKILL.md` files — an untracked skill can no longer authorize a remote owner.

#### Tests + alignment

- **Every reproduced defect became a permanent regression case** (shared-path containment + exec argv, malformed marker, mis-paired wrapper, live-worktree daemon, stale-lock contention, persistence failure, numeric/path bounds, internal-error fail-open, untracked owner). The sourceable allocator adopts a guarded strict-mode pattern so it passes the sk-code OPENCODE/SHELL alignment gate without imposing `set -e` on callers that source it.

## Migration Notes

None. Internal hardening only; no branch grammar, tooling interface, or command surface changed. The related closeout docs for the first ship were reconciled to describe the hardened behavior (and to correct a blanket "0-ahead" cleanup phrasing to reachability-based losslessness).
