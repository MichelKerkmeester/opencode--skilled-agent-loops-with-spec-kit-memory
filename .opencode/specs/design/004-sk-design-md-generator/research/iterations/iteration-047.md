## Focus
Phase-1 pre-mortem (failure modes + guards for the riskiest changes)

## Findings
1. **[P0] Interaction default-ON breaks CI and gold-standard snapshots** — Flipping noInteraction true→false at extract.ts:81 activates captureInteractions() for every extraction. Per-page budget is PAGE_TIMEOUT=30s (interaction-capture.ts:52), per-element PER_ELEMENT_TIMEOUT=6s (interaction-capture.ts:51), MAX_ELEMENTS=50 (interaction-capture.ts:53). Worst case 8 pages × 30s = 4min added. Real risk: (a) CI pipelines with <5min timeout gates will hard-fail. (b) Gold-standard examples (Vercel/Stripe/Linear/Supabase) were generated without interaction data — their tokens.json snapshots now gain hoverDiff/focusVisibleDiff/activeDiff fields, breaking any snapshot-diff test. (c) JS-heavy sites (Webflow, SPAs) produce flaky interaction captures — the existing try-catch at extract.ts:323-327 swallows failures silently, producing partial interaction data that looks complete. Likelihood: HIGH (every existing user pipeline affected). Blast radius: ALL extractions, ALL downstream consumers.
   - Recommendation: Add --fast-no-interaction as explicit opt-out (already in research). Gate default-ON behind a MAJOR version flag or environment variable (e.g., SK_DESIGN_INTERACTION_DEFAULT=on). Before shipping: re-run all 4 gold-standard extractions with interaction ON, diff tokens.json against stored snapshots, update snapshots. Add a CI-timing assertion: if extraction time > 2× baseline, warn. Log interaction capture success/failure counts per page in extraction-report.json so users can audit partial data.
   - Evidence: extract.ts:81, interaction-capture.ts:51-53, extract.ts:322-328
2. **[P0] deltaE <3→<10 collapses intentionally-close brand palette shades** — Raising threshold at cluster.ts:708 from <3 to <10 means colors with ΔE 3-10 (perceptible but intentionally distinct) merge. Concrete: Tailwind blue-500 (#3B82F6) vs blue-400 (#60A5FA) has ΔE≈8.1 — these would merge, destroying a 2-step scale. Material Design's blue-600/700/800 have ΔE 4-7 between adjacent stops. The greedy single-pass algorithm (cluster.ts:703-724) commits to first match with no recentering — after merging color A into cluster X, color B (ΔE 9 from A but 12 from X's center) may now incorrectly merge into X because it's within 10 of the original center. No post-merge recomputation exists. Likelihood: HIGH on any site with a blue/gray scale. Blast radius: Brand palette fidelity — the core value proposition.
   - Recommendation: Use threshold <7 as compromise (captures same-hue merges, spares adjacent stops). Add post-merge recentering: after each merge, recompute cluster centroid as frequency-weighted average. Add regression test: cluster.test.ts must assert that Tailwind blue-400/500/600 remain distinct, and that #333333/#444444 (ΔE≈4.5) merge. Log merge decisions in extraction-report.json for audit. Add a --delta-e-threshold CLI flag so power users can tune.
   - Evidence: cluster.ts:703-724, cluster.ts:708, cluster.ts:1664
3. **[P1] Section coverage check false-ABSENT on legitimately-present conditional sections** — New checkSectionCoverage (T7) maps each section to DesignTokens backing fields. Risk: §11 State Matrix depends on interaction data (tokens.components[].variants[].hoverChanges etc.), §12 Iconography depends on tokens.iconSystem.totalCount, §6.5 Motion depends on tokens.motionSystem. If extractA11yAsync wiring (T2) fails silently on a page (timeout at a11y-extract.ts:366-388 each wrapped in try-catch), or if interaction capture returns partial data (some elements captured, some timed out), the validator sees non-null but incomplete data → marks section PRESENT → AI writes prose from incomplete data → partial fabrication. Conversely, if the heuristic checks a field that's populated but the check looks at the wrong path (e.g., checking tokens.a11yTokens.tabOrder vs tokens.a11yTokens.focusIndicator.style), it falsely marks ABSENT → section wrongly omitted. Likelihood: MEDIUM (depends on heuristic accuracy). Blast radius: Missing sections in DESIGN.md.
   - Recommendation: Define a SectionDataMap type mapping each conditional section to exact dot-paths in DesignTokens. Unit-test each mapping against a known-populated tokens.json AND an empty one. For partial data: add a completeness heuristic (e.g., §11 requires interaction data from ≥2 pages AND ≥5 captures, not just non-null). Log section coverage decisions in validation output with the exact field paths checked.
   - Evidence: validate.ts:241-285, cluster.ts:1506-1568, a11y-extract.ts:361-391
4. **[P1] extractA11yAsync partial failure produces silently incomplete a11y tokens** — Wiring extractA11yAsync (a11y-extract.ts:361-391) per-page at extract.ts introduces 4 async calls per page (tabOrder, langAttribute, skipLinkDetected, altTextCoverage), each wrapped in individual try-catch. If a page times out during tab-order extraction (page.evaluate hangs on JS-heavy DOM), the result has tabOrder=undefined but altTextCoverage populated. Downstream: tokens.a11yTokens.tabOrder is undefined, §9 validator can't distinguish 'not captured' from 'page has no tabbable elements'. The altTextCoverage.percentage could be 100% (no images found) vs 100% (all images have alt) — indistinguishable. Additionally, cssAnalyses is NOT passed as 3rd arg to extractA11y in current code (extract.ts:445 calls extractA11y(domCollections, interactionSets) — only 2 args), so reducedMotionSupport is always undefined. The research says to fix this, but if the wiring misses the 3rd arg, reducedMotionSupport stays dead. Likelihood: HIGH (guaranteed on any page with JS-heavy DOM). Blast radius: §9 Accessibility Contract accuracy.
   - Recommendation: Add a 'captureStatus' field to each a11y sub-result: 'captured' | 'failed' | 'not-applicable'. In extractA11yAsync, set status explicitly on each try-catch path. Pass cssAnalyses as 3rd arg to extractA11y — add a type-check assertion that the function signature matches. Add integration test: mock a page that times out on tabOrder, verify partial result has captureStatus='failed' for tabOrder and 'captured' for altTextCoverage. Log per-page a11y capture status in extraction-report.json.
   - Evidence: a11y-extract.ts:361-391, extract.ts:445, a11y-extract.ts:257-261
5. **[P1] Focus fix consistent:true→captured:false breaks downstream validator and template** — Changing a11y-extract.ts:101-103 from {style:{}, consistent:true} to {style:{}, captured:false} changes the A11yTokens shape. Current downstream consumers: (1) cluster.ts:1558-1561 builds a11yTokens.focusIndicator with style/consistent fields — this is the dead-code duplicate that gets overwritten at extract.ts:469, but if the overwrite is removed (as research recommends), the cluster path must also change. (2) The DESIGN.md prompt template reads focusIndicator.consistent to decide whether to write 'consistent' or 'varies' prose. (3) validate.ts has no check on focusIndicator shape, but the new checkA11yFocusProvenance gate (T1) will check for captured:false — if this check doesn't exist yet when the fix ships, the validator still passes docs that fabricated focus prose from the old consistent:true. Likelihood: HIGH (shape change is breaking). Blast radius: §9 prose generation, validator gate.
   - Recommendation: Ship the captured:false shape change AND the checkA11yFocusProvenance validator gate in the same commit. Update cluster.ts:1557-1561 dead-code path to use captured field. Update prompt template to read captured field. Add test: extract with no interaction data → verify tokens.a11yTokens.focusIndicator.captured === false AND style === {}.
   - Evidence: a11y-extract.ts:101-103, cluster.ts:1557-1561, extract.ts:469
6. **[P2] New validator prose-discipline check false-positives on legitimate design vocabulary** — checkProseDiscipline (C1) scans for 17+ banned adjectives. Words like 'bold', 'medium', 'clean', 'modern', 'elegant' have legitimate uses: 'font-weight: bold', 'medium spacing', 'a clean separation between layers'. Numeric-anchor suppression (±50 chars) helps but doesn't cover: (a) backtick-wrapped values (`bold` → triggers, but ±50 char window may not reach the backtick context), (b) table cells where 'bold' appears in a font-weight column. The research says WARNING-tier only (weight 1pt), but if the new checkSectionCoverage also fires (e.g., 5 missing sections × 5pt = 25pt deduction), combined with prose warnings, previously-passing gold-standard examples could drop below 80 score threshold. Likelihood: MEDIUM (depends on how many banned words appear in legitimate prose). Blast radius: All existing passing examples.
   - Recommendation: Implement as WARNING-tier only, never FAILURE-tier. Add context-aware suppression: skip matches inside backticks, inside markdown table cells, and within 50 chars of a numeric value. Test against all 4 gold-standard DESIGN.md outputs: zero false-positive matches required. Add a --strict-prose flag to enable; default off for Phase 1.
   - Evidence: validate.ts:318-350 (no prose checks exist yet), research §3 C1
