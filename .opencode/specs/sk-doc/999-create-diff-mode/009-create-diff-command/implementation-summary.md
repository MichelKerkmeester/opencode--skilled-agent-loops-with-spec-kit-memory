## How It Was Delivered

The four OpenCode artifacts were authored by copying the shape of the sibling `flowchart` command and retargeting every step at the create-diff engine, so the command stays a thin orchestration layer over `create_diff.py` and `validate_report.py`. The engine is consumed as a CLI, never re-implemented: the workflow resolves the operator's inputs, shells out to `compare`/`compare-pair`, validates the produced report, and presents the result — the source document is read-only throughout. Every `--flag` in the two YAMLs (`--before/--after/--report/--view/--json/--label-before/--label-after/--state-dir`) was checked real against the engine's `build_parser`. After the router and assets existed, the Codex stub was generated (not hand-written) via `sync-prompts.cjs`, and the `mode-registry.json` `command` field was flipped from `null` to `"/create:diff"`. Verification then confirmed — structurally — routing, asset existence, the tool-surface subset, the engine-never-bypassed invariant, the registry wiring, stub idempotency, and recursive strict validation. All verification was structural (shape, parse, asset resolution, flag validity, registry/canon gates, strict spec validation); live end-to-end invocation in an actual runtime is the one deferred acceptance step (see Known Limitations).

<!-- ANCHOR:decisions -->
## Key Decisions

| Decision | Why |
|----------|-----|
| Follow the full sibling pattern (router + presentation.txt + auto.yaml + confirm.yaml) even though create-diff is engine-backed | Consistency with all ten existing `/create:*` commands: one predictable shape for operators and canon gates. The operator explicitly chose the full pattern over a minimal engine-only wrapper. |
| Keep the command a thin layer over the engine; never re-implement diffing | The comparison, extraction, snapshotting, and rendering are already shipped and hardened (phases 002–008); duplicating any of it in the command layer would create a second, drift-prone source of truth. REQ-003 freezes this as a P0 invariant. |
| Generate the Codex stub rather than hand-author it | `sync-prompts.cjs` already emits Codex prompt stubs from OpenCode commands; a generated stub stays idempotent and free of hand-edit drift, matching how the other commands' Codex prompts are produced. |
| `:confirm` is the default mode when no mode token is given | Matches the sibling grammar (`flowchart` and the other `/create:*` commands default to the interactive, checkpointed workflow); `:auto` is opt-in for gate-free runs. |
| `allowed-tools` is a strict subset of the mode surface with no `Task` | The router must not widen the create-diff mode's capability surface; excluding `Task` preserves the LEAF, no-sub-agent boundary the mode declares. |

<!-- ANCHOR:limitations -->
## Known Limitations

1. **Live end-to-end invocation not exercised.** Running `/create:diff` in an actual OpenCode/Codex session (a baseline or explicit-pair run that produces a validated report end to end) is a user-runtime acceptance step that is not reproducible from this authoring environment. All verification recorded here is structural — shape, parse, asset resolution, flag validity, registry/canon gates, and strict spec validation. The command is built and structurally sound; only the live-runtime acceptance run remains for the operator to confirm.
2. **Cross-links are operator-owned.** The parent `spec.md` phase map and the sibling 008 `Successor` field are updated separately; this packet only creates files inside `009-create-diff-command/`.
3. **Generated metadata pending.** `description.json` and `graph-metadata.json` for this child are produced by the operator's `generate-context.js` / backfill afterward, so a metadata/continuity-freshness warning is expected until then.
4. **Fidelity is the engine's, not the command's.** The command surfaces whatever fidelity tier the engine reports; it neither raises nor masks it, and a limited-fidelity comparison is never presented as complete.
