## Focus
Doc-as-view-over-tokens target architecture (deterministic/grounded-annotation/bounded-prose section classes + deterministic renderer)

## Findings
1. **[P0] Three-class section partition** — Sections partitioned into DETERMINISTIC (auto-rendered value tables, zero AI), GROUNDED-ANNOTATION (value table + 1-line token-cited AI note), and BOUNDED-PROSE (labeled [INFERRED], gated/optional). Class (a) covers §2 Colors, §2.5 Dark Mode, §3 Typography, §5 Spacing+Radii, §6 Shadow table, §6.5 Duration/Easing tables, §9 Contrast table, §10 Breakpoints, §12 Icon sizing, §17 Token Dictionary — sections where every value has a tokens.json row. This removes the LLM as a value-invention surface for fontSize/fontWeight/lineHeight/shadow/radius fabrication that validate.ts currently cannot catch (only hex values are checked). Class (b) covers §4 Component Stylings, §5 Whitespace Philosophy, §6 Named Principle, §6.5 Motion Philosophy+Choreography, §8 Do's/Don'ts, §11 State Matrix — the AI writes 1-line annotations that MUST cite token values. Class (c) covers §0 Brand Context, §1 Visual Theme, §7 Content & Voice, §13 Prompt Guide, §14-16 — AI prose labeled [INFERRED], each paragraph citing >=2 token values.
   - Recommendation: Modify tool/resources/design_md_format.md to add per-section classification tags (DETERMINISTIC/GROUNDED/BOUNDED). Create tool/scripts/formatters.ts with formatColorTable(), formatTypographyTable(), formatShadowTable(), formatSpacingTable(), formatRadiusTable(), formatDarkModeSection(), formatMotionTable(), formatContrastTable(), formatBreakpointTable(), formatIconTable(), formatTokenDictionary(). Modify SKILL.md WRITE phase spec to: (1) run formatters.ts first → pre-rendered markdown, (2) AI receives pre-rendered tables + tokens.json, writes only class-(b) annotations and class-(c) prose, (3) assemble final DESIGN.md. Phase: tool/scripts/ + tool/resources/ + assets/.
   - Evidence: iteration-023 P0 (doc-as-token-view: Supernova/Backlight/Zeroheight/Knapsack render docs FROM tokens), iteration-022 P0(f) (deterministic value-table rendering vs AI prose), iteration-018 P0 (fabrication-surface analysis: 3 high-risk prose sections), design_md_format.md sections 3-17 (value-table format specs), validate.ts:283-310 (checkContent only checks color count and typo table presence — never validates fontSize/shadow values against tokens)
2. **[P0] Deterministic renderer (formatters.ts)** — The core architectural change: a TypeScript script at tool/scripts/formatters.ts that reads tokens.json (post-cluster, post-DTCG-wrap) and emits markdown for all class-(a) sections. This runs BEFORE the AI WRITE phase. The AI then receives pre-rendered sections as its input and is contractually forbidden from modifying them — it only appends class-(b) annotation lines and class-(c) [INFERRED] prose blocks. This structurally removes the LLM from value-table generation, eliminating the primary hallucination vector that validate.ts currently cannot catch: fabricated fontSize (e.g., 14px typed as 16px), fabricated fontWeight (400 typed as 500), fabricated lineHeight, fabricated shadow offsets/blurs, fabricated border-radius values. validate.ts only checks hex colors against tokens.json (checkPhantomColors at line 57) — all other token types pass unchecked.
   - Recommendation: Create tool/scripts/formatters.ts with one exported function per class-(a) section, each accepting typed token slices and returning markdown strings. Wire into the WRITE phase of SKILL.md: before handing control to the AI, run `npx ts-node tool/scripts/formatters.ts <tokens.json> --output <pre-rendered.md>`. The AI WRITE prompt is amended to say 'Sections 2, 2.5, 3, 5(spacing+radii), 6(shadow table), 6.5(duration+easing), 9(contrast), 10(breakpoints), 12(sizing), 17 are pre-generated in <pre-rendered.md>. DO NOT modify these sections. Append your annotations and prose after each pre-rendered block.' Phase: tool/scripts/formatters.ts (new), tool/scripts/types.ts (add section renderer output types), SKILL.md (WRITE phase spec update).
   - Evidence: iteration-023 P0 (Diez DocsGen generates token documentation deterministically from typed token data with zero inference), iteration-022 P0(f) (Style Dictionary renders programmatic output from token metadata), design_md_format.md:149-239 (Section 2 color entry format — every field maps to tokens.json fields), design_md_format.md:366-425 (Section 3 hierarchy table — every column maps to token fields), validate.ts:57-93 (checkPhantomColors only validates hex — no check for fontSize/weight/lineHeight/shadow fabrication)
3. **[P0] Token-citation gating in validate.ts** — For class-(b) annotations and class-(c) prose: every AI-written claim must parenthetically cite >=2 token values from tokens.json. The current validate.ts (lines 57-350) has no citation-awareness — it checks hex provenance (phantom colors) and format consistency but never verifies that prose claims are grounded in token evidence. A new validation gate, checkCitationGating(), parses citation annotations in markdown (e.g., 'Light weight as signature: Weight 300 at 56px [token: typographyLevels[0].weight, §3]') and verifies each cited token path exists in tokens.json. This closes the gap that iteration-018 identified: Section 1's 'named principles' and Section 7's 'vibe paragraph' currently have zero evidence-quota enforcement.
   - Recommendation: Add checkCitationGating() to tool/scripts/validate.ts: parse DESIGN.md for citation markers (proposed format: `[token: <path>, <src-section>]`), resolve each path against tokens.json, flag unreferenced claims. For class-(c) sections (§0, §1, §7), enforce minimum 2 token citations per paragraph. For class-(b) sections (§4, §5 whitespace, §6 principle, §6.5 philosophy, §8, §11), enforce minimum 1 citation per annotation line. Uncited prose claims in class-(b)/(c) sections are validation FAILURES (not warnings). Also update design_md_format.md to require citation syntax in all groundable sections. Phase: tool/scripts/validate.ts (add function + wire into validateDesignMd), tool/resources/design_md_format.md (citation syntax spec).
   - Evidence: iteration-018 P1 (extract-design-system prompt-driven grounding — every claim must cite evidence), iteration-017 P1 (Zeroheight's cite-the-token guardrail), iteration-023 P1 (evidence quotas for AI prose sections), design_md_prompt_template.md:40-56 (current WRITE prompt has 6 cardinal rules but ZERO evidence-citation requirements)
4. **[P1] DTCG-typed tokens.json spine** — Migration of tokens.json from flat custom schema to W3C DTCG 2025.10 format ($type, $value, $description per token) enables: (a) type-gated validation — validate.ts can check that a DTCG-typed token's $value matches what appears in DESIGN.md for $type=duration (not just $type=color), (b) deterministic formatters can render $type-appropriate markdown (color → hex table, dimension → px+rem scale, duration → ms table, cubicBezier → CSS function), (c) $description fields provide prose anchors for class-(b) annotations and class-(c) claims, and (d) downstream interop with Style Dictionary, Terrazzo, Theo. This is the structural backbone that makes the three-class partition enforceable at the type level rather than convention.
   - Recommendation: Create tool/scripts/dtcg-wrap.ts: reads tokens.json post-cluster, wraps every token in DTCG envelope (ColorToken → { $type: 'color', $value: hex, $description: role+frequency }). Composite tokens for type scales: shadow scale as composite of shadow tokens, spacing scale as composite of dimension tokens. Update types.ts with DTCGShape interface. Phase: tool/scripts/dtcg-wrap.ts (new), tool/scripts/types.ts (add DTCGShape), run as Phase 1c after cluster.ts.
   - Evidence: iteration-022 P0(b) (DTCG typed-schema zero-invention: token cannot exist without typed slot), iteration-022 P0(f) (DTCG $description anchors prose claims), W3C DTCG 2025.10 spec ($type color/dimension/fontFamily/fontWeight/duration/cubicBezier/number/shadow), validate.ts:57 (only hex-checked, no type-gated validation for non-color tokens)

## Questions Answered
- How to partition the 17 sections into three fabrication-risk classes: Class (a) DETERMINISTIC = §2,§2.5,§3,§5(spacing+radii),§6(shadow table),§6.5(duration+easing),§9(contrast),§10(breakpoints),§12(sizing),§17 (sections where every field has a tokens.json row and zero AI judgment is needed). Class (b) GROUNDED-ANNOTATION = §4,§5(whitespace philosophy),§6(named principle+contrast para),§6.5(motion philosophy+choreography),§8,§11 (sections where AI writes 1-line annotations that must cite >=1 token value each). Class (c) BOUNDED-PROSE = §0,§1,§7,§13(prompts),§14-16 (AI prose labeled [INFERRED], each paragraph citing >=2 token values).
- Where the deterministic renderer lives and what it replaces: tool/scripts/formatters.ts — a new TypeScript script that reads tokens.json (post-cluster, post-DTCG-wrap) and emits markdown tables for all class-(a) sections. It runs BEFORE the AI WRITE phase as a pre-generation step. The AI receives the pre-rendered tables as input and is forbidden from modifying them — it writes only class-(b) annotation lines and class-(c) prose. This replaces the current WRITE phase where the AI generates ALL 17 sections including value tables.
- How the token-citation requirement is enforced: validate.ts gains a new checkCitationGating() function that parses citation markers (format: `[token: <json-path>, <src-section>]`) in DESIGN.md prose and verifies each cited path exists in tokens.json. Class-(c) sections require >=2 citations per paragraph; class-(b) sections require >=1 citations per annotation line. Uncited claims in these sections are hard FAILURES. This composes with existing phantom-color and stability-gating checks — the full gate chain is: (1) pre-render filter (formatters.ts applies L1/L2 only), (2) hex check (checkPhantomColors — no fabricated colors), (3) format check (lowercase 6-digit, numeric weights), (4) citation check (every prose claim grounded), (5) section completeness (three-tier required/conditional/optional).
- How TIER-1 fixes compose: focus-check (hex format) remains as post-generation gate for class-(c) sections. section-cut (completeness) is updated to three-tier: class-(a) ALWAYS present (formatters guarantee), class-(b) REQUIRED when data exists, class-(c) OPTIONAL with [SKIPPED] marker when insufficient data. AP-29 (stability gating) is wired into formatters.ts as a pre-render filter — only L1+L2 tokens enter class-(a) tables, and validate.ts checks that no L4 colors appear anywhere (covering both auto-rendered and AI-written sections).
- Migration path: Phase A (minimal viable, ~20% effort for ~60% gain) = formatters.ts for §2+§3 only + citation gating for §1+§7. Phase B (full architecture) = all class-(a) formatters + DTCG-typed tokens.json + complete citation gating + tokens.css emission + structured section summaries.

## Summary
The target architecture inverts the artifact hierarchy: tokens.json (with DTCG-typed spine) becomes the primary deliverable, and DESIGN.md becomes a deterministic render where all value-bearing sections (§2 Colors, §3 Typography, §5 Spacing/Radii, §6 Shadows, §6.5 Motion tables, §9 Contrast, §10 Breakpoints, §12 Icon sizing, §17 Token Dictionary) are auto-generated by formatters.ts as markdown tables directly from typed tokens with zero AI involvement. AI prose is reduced to thin, token-cited annotation on component styling and behavioral sections (§4, §8, §11) and explicitly [INFERRED]-labeled narrative prose (§0, §1, §7), with validate.ts upgraded to verify that every AI-written claim cites token evidence — structurally eliminating the entire value-fabrication surface that the current hex-only validator cannot detect.
