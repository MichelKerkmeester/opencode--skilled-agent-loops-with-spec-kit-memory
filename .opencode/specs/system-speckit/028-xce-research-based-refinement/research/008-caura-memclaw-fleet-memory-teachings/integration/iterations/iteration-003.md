## UX/automation notes
- The integration should prefer hidden defaults over flags: receipts on all retryable writes, source-kind inferred from command/agent/tool caller, feedback logged server-side, stale-exclusion audited in default search, and ownership linting in pre-commit/doctor.
- Manual commands remain escape hatches and dashboards, not primary workflows: `/memory:manage`, `/doctor memory`, and `/doctor skill-budget` should explain what automation observed rather than ask users to trigger the observation.
- Response envelopes are the lowest-friction UX surface: existing `hints`, `meta.autoSurface`, and `postMutationHooks` fields can carry receipt replay, stale-exclusion warnings, audit-health summaries, and auto-surface counts without changing command syntax. See `.opencode/skills/system-spec-kit/mcp_server/hooks/response-hints.ts:86-128` and `.opencode/skills/system-spec-kit/mcp_server/hooks/mutation-feedback.ts:6-58`.
- Avoid new public feedback-write tools. Feedback should be captured from search/use behavior only; active mutation reducers remain deferred until diagnostics prove ledger quality.
- Pre-commit is appropriate for static drift only. Runtime correctness belongs in on-write hooks, startup hooks, and `/doctor`; pre-commit already blocks comment-hygiene issues at `.opencode/hooks/pre-commit:16-34` and can add ownership-map drift without becoming a runtime safety substitute.
