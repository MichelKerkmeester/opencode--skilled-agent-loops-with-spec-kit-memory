### FINDINGS

1. **Gate 3 is worded as a priority hard stop, not an interactive-only safeguard.** `CLAUDE.md` says Gate 3 “Overrides Gates 1-2” and must be asked “BEFORE any analysis/tool calls” when file modification is detected, then reinforces “Ask first, then act” with “No Read/Edit/Write/Bash ... before answer” (`CLAUDE.md:126-138`). The recovery block gives the exact observed halt prefix: `Before I proceed, I need to ask about documentation:` (`CLAUDE.md:178-180`).

2. **The machine classifier explicitly treats deep-loop/autonomous command text as Gate-3-triggering write flow.** Resume triggers include `/deep:research`, `/deep:review`, `/deep:ai-council`, `:auto`, `deep-research`, `deep-review`, `deep-loop`, and `convergence` (`.opencode/skills/system-spec-kit/shared/gate-3-classifier.ts:108-141`). `:auto` matches when paired with `speckit` or `/deep:` (`.opencode/skills/system-spec-kit/shared/gate-3-classifier.ts:197-200`). The decision order says resume/memory-save triggers require Gate 3, and the code returns `resume_match` when a workflow invocation marker such as `run`, `use`, `start`, `continue`, or `launch` is present (`.opencode/skills/system-spec-kit/shared/gate-3-classifier.ts:238-240`, `.opencode/skills/system-spec-kit/shared/gate-3-classifier.ts:274-337`).

3. **The autonomous YAMLs are genuinely write-producing, so a literal executor is not hallucinating the write risk.** Deep review declares `workflow_execution: autonomous` and `approvals: none`, but also says “Never hold findings in memory; write everything to files” (`.opencode/commands/deep/assets/deep_review_auto.yaml:8-14`, `.opencode/commands/deep/assets/deep_review_auto.yaml:20-24`). It creates packet directories and writes config/state/registry/strategy files (`.opencode/commands/deep/assets/deep_review_auto.yaml:202-204`, `.opencode/commands/deep/assets/deep_review_auto.yaml:378-439`). Deep research has the same autonomous/no-approval shape and write mandate (`.opencode/commands/deep/assets/deep_research_auto.yaml:20-27`, `.opencode/commands/deep/assets/deep_research_auto.yaml:34-37`), then writes state artifacts and may create or edit `spec.md` (`.opencode/commands/deep/assets/deep_research_auto.yaml:296-347`, `.opencode/commands/deep/assets/deep_research_auto.yaml:375-421`).

4. **The presentation contract tells `:auto` not to prompt, but it never states that auto setup satisfies Gate 3.** For `:auto`, it says “do not emit the consolidated prompt by default” and to resolve setup via the three-tier flow before loading YAML (`.opencode/commands/deep/assets/deep_review_presentation.txt:11-23`). It allows `spec_folder` to be bound from a marker, flag, scope extraction, or target path (`.opencode/commands/deep/assets/deep_review_presentation.txt:27-58`, `.opencode/commands/deep/assets/deep_review_presentation.txt:62-68`). That supports Claude’s likely interpretation that a pre-bound `spec_folder` is the standing ownership answer, but the text only tolerates that reading; it does not explicitly override the CLAUDE.md hard-block language.

5. **The strongest literal text points GPT toward the generic Gate-3 question, while Claude appears to weigh command intent and resolved ownership.** `CLAUDE.md` says to stop and ask Gate 3 if gates are being skipped, unless the user already answered Gate 3 for the same task (`CLAUDE.md:178-180`), and the self-check is binary: “File modification? Asked spec folder question?” (`CLAUDE.md:182-187`). The command contract separately says all required values, including `spec_folder`, can be resolved in autonomous setup before YAML load (`.opencode/commands/deep/assets/deep_review_presentation.txt:17-23`, `.opencode/commands/deep/assets/deep_review_presentation.txt:67-68`). There is no explicit precedence bridge between these two mechanisms, so GPT follows the hard-block/recovery text and Claude likely treats the pre-bound command setup as sufficient.
