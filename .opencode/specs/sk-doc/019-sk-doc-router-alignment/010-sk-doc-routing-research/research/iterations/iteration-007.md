# Iteration 7: Typed public-ID edge-case hardening

## Focus

Pressure-test the settled `(workflowMode, leafResourceId)` design against duplicate packet-local filenames, shared aliases, N-to-1 mode-to-packet multiplexing, sk-code's second-layer router, migration compatibility, and regression safety. The namespace choice remains fixed.

## Actions Taken

1. Enumerated repeated packet-local resource names across all sk-doc packets and checked whether shared resources are exposed through packet symlinks.
2. Traced sk-doc's registry/router mapping, including the two workflow modes that resolve to one physical packet.
3. Compared sk-code's surface-first, intent-second resource router with the proposed sk-doc second layer.
4. Traced benchmark fixture loading, exact-resource scoring boundaries, D5 connectivity behavior, and the prior two-clean-row baseline.
5. Executed the current sk-doc D5 scanner as a read-only baseline: score `100`, `gateFailed=false`.

## Findings

### 1. Composite uniqueness is the required invariant

Packet-local collisions are real and benign: `references/README.md` appears in ten sk-doc packets, while `references/common_pitfalls.md`, `references/examples.md`, and `references/worked_example.md` each appear in multiple packets. Therefore `leafResourceId` alone cannot be globally unique. Uniqueness must be enforced on the pair `(workflowMode, leafResourceId)`; duplicate leaf IDs across different modes are valid. [SOURCE: file:.opencode/skills/sk-doc/create-agent/references/README.md:1] [SOURCE: file:.opencode/skills/sk-doc/create-changelog/references/README.md:1] [SOURCE: file:.opencode/skills/sk-doc/create-agent/references/common_pitfalls.md:1] [SOURCE: file:.opencode/skills/sk-doc/create-command/references/common_pitfalls.md:1]

The manifest validator should reject duplicate composite keys, not duplicate leaf names or duplicate physical targets. A collision test must create two modes with the same `references/README.md` leaf and prove both round-trip independently.

### 2. N-to-1 mode multiplexing must fan out, not collapse

`create-skill` and `create-skill-parent` are distinct public workflow modes that both resolve to packet `create-skill`. A manifest generated by packet directory alone would collapse their public identities. Generation must iterate registry modes, then enumerate each selected packet, producing two composite key sets that may resolve to the same physical files while retaining the initiating `workflowMode`. [SOURCE: file:.opencode/skills/sk-doc/mode-registry.json:18] [SOURCE: file:.opencode/skills/sk-doc/mode-registry.json:30] [SOURCE: file:.opencode/skills/sk-doc/mode-registry.json:34]

Required compatibility test: `(create-skill, references/parent_skill/parent_skills_nested_packets.md)` and `(create-skill-parent, references/parent_skill/parent_skills_nested_packets.md)` both resolve successfully, resolve to the same physical path, and remain distinct public keys.

### 3. Shared aliases require an explicit many-to-one declaration

The changelog fixture expects public leaf `assets/changelog_template.md` while naming `shared/assets/changelog_template.md` as the physical source. The optimization fixture similarly expects a packet-local `assets/llmstxt_templates.md` alias for a shared physical asset. No packet-local reference/asset symlinks were found, so recursive filesystem discovery cannot infer these aliases. [SOURCE: file:.opencode/skills/sk-doc/manual_testing_playbook/agent_dispatch/markdown_agent_cli_opencode.md:9] [SOURCE: file:.opencode/skills/sk-doc/manual_testing_playbook/agent_dispatch/markdown_agent_cli_opencode.md:94] [SOURCE: file:.opencode/skills/sk-doc/manual_testing_playbook/intent_detection/optimization.md:8]

The manifest needs authored alias declarations whose keys are typed public pairs and whose values are physical shared paths. Many public keys mapping to one physical path is valid; reverse lookup from physical path must never choose a public key without mode context.

### 4. sk-code validates the layering, not the legacy serialization

sk-code routes surface first and intent second, then selects exact resources from the narrowed family. That validates the sk-doc design's separation between a stable discriminator and a packet-local leaf map, plus its rule against directory placeholders and inventory dumps. [SOURCE: file:.opencode/skills/sk-code/shared/references/smart_routing.md:34] [SOURCE: file:.opencode/skills/sk-code/shared/references/smart_routing.md:114] [SOURCE: file:.opencode/skills/sk-code/shared/references/smart_routing.md:175] [SOURCE: file:.opencode/skills/sk-code/shared/references/smart_routing.md:269]

It does not invalidate typed sk-doc IDs. sk-code serializes selected resources as hub-root-qualified strings such as `code-webflow/references/...`; that convention combines selection identity and load address. Reusing that serialization would preserve the ambiguity the sk-doc fix is meant to remove. The reusable convention is staged selection and exact maps, not the output string shape.

### 5. Migration must be dual-read, single-write, and fail closed

Current benchmark fixtures expose `expected_resources` as strings, and the loader carries those strings forward without topology resolution. Typed IDs therefore change an existing caller boundary even if the physical files do not move. [SOURCE: file:.opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/load-playbook-scenarios.cjs:295] [SOURCE: file:.opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/load-playbook-scenarios.cjs:321]

A bounded migration bridge should accept:

- a typed `{ workflowMode, leafResourceId }` value;
- a legacy packet-qualified hub load address only when its packet prefix matches the scenario's workflow mode through the registry;
- a legacy packet-local leaf string only when workflow mode is already explicit;
- a declared shared alias only through the manifest.

All new emitters write typed pairs. The bridge rejects ambiguous local strings without a mode, mismatched packet prefixes, undeclared shared targets, and payloads where typed and legacy forms disagree. It should record legacy-form telemetry so removal is evidence-based.

Compatibility fixtures should cover duplicate leaf names, both create-skill modes, shared aliases, legacy packet-qualified input, legacy packet-local input with mode context, conflicting dual forms, and unknown aliases.

### 6. Regression safety requires two frozen baselines

The two previously clean benchmark rows must be copied into a compatibility fixture that scores both legacy and typed representations through the same canonical pair normalizer. The typed run must preserve exact expected/observed set equality and zero waste; this proves the bridge is not a blanket suppression rule. The previous classification identifies exactly two clean rows. [SOURCE: file:.opencode/specs/sk-doc/019-sk-doc-router-alignment/010-sk-doc-routing-research/research/iterations/iteration-003.md:84]

The current D5 baseline executed in this iteration is score `100` with `gateFailed=false`. Manifest-aware nested coverage must be additive to the existing structural checks: run the current scanner before and after, require the same or higher score, preserve every existing pass, and fail if any current finding regresses. Then separately assert manifest completeness, composite-key uniqueness, alias target existence, and smart-router leaf reachability. ## Questions Answered

- Duplicate local filenames do not collide when uniqueness is defined on `(workflowMode, leafResourceId)`.
- Shared aliases and N-to-1 packet multiplexing require legal many-to-one physical resolution; neither permits public-key collapse.
- sk-code supports the staged-routing architecture but not reuse of its packet-qualified string serialization.
- Existing callers need a dual-read/single-write migration bridge with explicit rejection cases.
- Clean-row and D5 baselines can be protected with canonical-pair fixtures and additive structural assertions.

## Reflection

What worked: comparing actual topology against a sibling second-layer router separated reusable routing structure from incompatible serialization. What failed: filesystem alias discovery returned no packet symlinks, which rules it out as the shared-alias source.

Ruled out: global uniqueness of `leafResourceId`; packet-directory-keyed manifest generation; inferring shared aliases from symlinks; adopting sk-code's packet-qualified string output as the new public namespace.
