# Iteration 9: Operational Migration, Rollback, Rebuild, and CI Sequence

## Focus

This iteration converted the ranked remediation and R1–R9 acceptance model into a release sequence with explicit write freezes, preflight classifications, compatibility semantics, stop conditions, rollback boundaries, clean-build order, and operating-system CI lanes. It specifically challenged iteration 6's call-site-writer-first order: changing implicit writers before migrating a unique legacy-only packet can create a new canonical packet beside the legacy packet, while retaining unrestricted explicit legacy writes after migration can recreate split state. The operational contract below therefore makes the compatibility window reader-compatible but writer-closed.

## Findings

1. **Writer cutover must follow packet canonicalization, not precede it; the earlier call-site-first ranking is unsafe as a deployment order unless writers remain frozen.** The current shared scripts resolver selects the first existing legacy root, and untracked `spec/create.sh` writes directly to `specs/`; iteration 8's intended writers instead target canonical. If a unique legacy-only packet still exists when an implicit writer becomes canonical-only, that writer can create or update the same relative ID under `.opencode/specs` while the old packet remains under a distinct `specs/`, producing the R6 split state that the rollout is meant to prevent. The safe order is: deploy/read-run collision preflight, freeze automatic writers, canonicalize packet ownership, verify zero in-root legacy packets, deploy canonical writers plus per-write collision guards, then unfreeze. A compatibility period may preserve explicit legacy **reads**, but ordinary or automatic legacy **writes** must be blocked; a narrowly named migration override is safe only while the global write freeze is active. 3. **The point of no return is the first successful canonical write after S3 unfreezes, not alias removal or reader reordering.** Before that write, S2's manifest and quarantined originals can reverse every packet move under the freeze. After it, restoring old legacy packets can discard or fork new canonical changes, so rollback must be behavior-only: refreeze, restore the previous source+dist bundle atomically, preserve canonical data, and either prove a same-inode compatibility bridge before resuming or roll forward. This separates three independent rollback domains: packet data (reversible only through frozen S2), executable behavior (reversible as a source+dist bundle), and alias metadata (reversible only as a non-universal convenience). Restoring the symlink alone never repairs stale parent pointers, divergent content, or a symlink-disabled checkout. 4. **Clean rebuild order is scripts first, MCP server second, followed by behavior—not freshness-marker—execution against both dist surfaces.** `@spec-kit/scripts` exposes `tsc --build` but no clean script and its runtime main is `dist/memory/generate-context.js`; CI must therefore remove its candidate `dist` and TypeScript build-info in the isolated checkout, run typecheck and targeted source tests, build scripts, then run the R1–R9 and generator dist cases. Only after scripts dist is green should CI run MCP source typecheck/tests and `npm run rebuild`, whose declared sequence cleans MCP dist, prepares freshness metadata, compiles, and finalizes dist; then it must run the built Stop-hook and MCP-reader cells against the same expected-result table. Promotion hashes both rebuilt dist trees and deploys them with source as one unit. Building MCP first is unsafe because its Stop hook invokes the scripts generator runtime, and a source-only green result does not prove the configured compiled path. ## Ruled Out

- **Writer fixes before data preflight/migration:** this can strand the old packet and create a canonical twin for the same relative ID. [INFERENCE: based on the current legacy-first/direct-legacy writers and the intended canonical writer destinations]
- **A read/write legacy compatibility window:** read fallback protects legacy-only discoverability, but write fallback allows automatic or stale clients to recreate split state after migration. Compatibility is therefore read-only, with migration writes allowed only under the global freeze. [INFERENCE: based on R2 fallback and R6 divergent-duplicate behavior]
- **Rolling packet data back after writer unfreeze:** the quarantine is older after the first canonical write, so restoration is potentially destructive. [INFERENCE: based on the S2/S3 write boundary]
- **MCP-first or source-only build validation:** the configured hook reaches scripts dist, while both packages declare compiled runtime entrypoints. [SOURCE: .opencode/skills/system-spec-kit/scripts/package.json:5-19] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/package.json:5-31]
- **One Ubuntu symlink lane as portability proof:** the current workflow is Ubuntu-only, and mandatory absent/plain-file behavior is distinct from optional symlink capability. [SOURCE: .github/workflows/strict-pass-freshness-sweep.yml:11-27]
