# Iter 6 — Adversarial synthesis — real root cause of probe 14 failure (6 hypotheses ranked)

## TL;DR (2-3 sentences)

The Phase 2 bench refutation of the "BGE + path-class boost fixes lexical-cue density" hypothesis was correct but mis-targeted: path-class boost was a dead variable because structural-indexer.ts and its distractors share `implementation` class. The real root cause is a two-factor interaction: (1) BGE-reranker-v2-m3's cross-encoder scores lexical token overlap over semantic role, and (2) the top-ranked structural-indexer chunk contains only 9 lines of imports/header (chunk 44683, lines 1-9), starving the cross-encoder of content. The pre-rerank pipeline (embedder, FTS, RRF, dedup) is NOT the bottleneck — structural-indexer reaches the rerank head as the highest-pre_score candidate in ALL THREE lanes, and Jina v3 proves the identical head can produce a top-5 hit with a different scoring function.

## Question (restate the scoped RQ)

Adversarial synthesis of iters 1-5. The iter 2 deep research hypothesis was "BGE reranker rewards lexical-cue density over semantic role; tests outrank implementation." Phase 2 bench refuted this: BGE+path-class boost made ZERO probe outcome changes because the expected implementation files (e.g. structural-indexer.ts for probe 14) were NOT in the rerank window. Reconstruct the REAL root cause of probe 14 failure by ranking these 6 hypotheses in probability of being load-bearing: (a) chunking too small/too big, (b) embedder choice limits semantic recall (bge-code-v1 misses the semantic intent), (c) FTS5 tokenization splits the query terms wrongly, (d) hybrid weight imbalance favors one channel, (e) rerank top-K=20 cutoff is too narrow, (f) something else (specify). For each: probability + supporting evidence from iters 1-5 + falsifiable test.

## Findings (numbered, with citations)

### Hypothesis Ranking (most → least load-bearing)

1. **Hypothesis (f) — Reranker model lexical-cue-density preference (probability: 95%)**. The BGE-reranker-v2-m3 cross-encoder scores `(query, chunk_content)` pairs by token-level lexical overlap, not abstract semantic role. The structural-indexer chunk (9 lines of imports/header) has minimal query-term overlap with "filesystem walker that emits typed structural symbols and import edges." The stress test `walker-dos-caps.vitest.ts` contains "walker", "filesystem", "edges" literally in test code, earning a BGE score 3x higher (0.076 vs 0.025). Jina v3, with identical input, scores structural-indexer at 0.083-0.110 — proving the reranker model's scoring function is the proximate differentiator. Falsifiable test: re-run probe 14 with an alternative cross-encoder (e.g., `mixedbread-ai/mxbai-rerank-base-v1`, `BAAI/bge-reranker-v2-gemma`) without changing chunking — if structural-indexer hits top-5 under a different model, the BGE model is confirmed as the bottleneck. Supporting evidence: `baseline-bge.rerank-scores.jsonl:6`, `jina-v3.rerank-scores.jsonl:6`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/reranker.py:139-183`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/README.md:8`.

2. **Hypothesis (a) — Chunking boundary suboptimality amplifies reranker weakness (probability: 70%)**. The `RecursiveSplitter` with defaults (1500/200/250) produces 67 chunks for structural-indexer.ts, but the chunk that reaches dense rank 1 (id 44683, lines 1-9) is a token-sparse import/header slice. The implementation-rich chunks (`findFiles` at lines 1391-1465, `parseFile` at 1221-1259, `indexFiles` at 2194-2234) rank lower because text-splitting boundaries don't respect class/function declarations. If the top-ranked chunk contained the full class body with JSDoc, BGE might have found sufficient lexical overlap to score it above the stress test. Chunking is not the sole cause (Jina succeeds with the same chunks), but it is a strong amplifier — it determines WHICH content the reranker evaluates. Falsifiable test: re-index with larger `chunk_size=4000` or a code-aware splitter that preserves export boundaries, then re-run probe 14 under BGE. If structural-indexer now hits top-5, chunking is confirmed as the primary amplifier. Supporting evidence: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:33-39`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:273-283`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/cocoindex-internals-deep-dive/iterations/iteration-002.md:29-35`.

3. **Hypothesis (b) — Embedder choice limits semantic recall quality (probability: 30%)**. bge-code-v1 successfully ranked structural-indexer at dense rank 1 — the retrieval system found the right file. However, a different embedder might have ranked an implementation-rich chunk (not the 9-line header) to position 1, or brought more diverse structural-indexer chunks into the top-20 pool. The current embedder picked the match based on vector similarity, but the similarity was to the concept of "structural indexer" (the title) rather than to the query's behavioral description. A code-aware embedder (e.g., `voyage-code-2`) might produce embeddings that better capture behavioral semantics. Falsifiable test: re-index with `voyage-code-2` or `codegemma-embed`, re-run probe 14, and check dense top-20 composition — if implementation-body chunks now appear at rank 1-3, embedder is load-bearing. Supporting evidence: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:579-595`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/shared.py:143-202`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/run-phase2-smoke.sh:48-51`.

4. **Hypothesis (c) — FTS5 tokenization OR-broadening favors distractors (probability: 20%)**. The regex tokenizer `[A-Za-z0-9_./:-]+` extracts tokens and joins them with OR. The query word "walker" matches `walker-dos-caps.vitest.ts` directly in filename and content, giving the stress test FTS rank 1. However, the structural-indexer STILL reaches the rerank head with the highest pre_score, so FTS tokenization does not explain why the reranker fails. FTS tokenization contributes to which candidates enter the head, but the reranker's scoring of those candidates is the bottleneck. Falsifiable test: run probe 14 in vector-only mode (disable hybrid, COCOINDEX_HYBRID=false) — if structural-indexer still misses top-5 under BGE, FTS tokenization is not load-bearing. Supporting evidence: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:11-12`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:88-132`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/cocoindex-internals-deep-dive/iterations/iteration-005.md:23-27`.

5. **Hypothesis (d) — Hybrid weight imbalance (probability: 5%)**. Both `hybrid_vector_weight` and `hybrid_fts5_weight` default to 0.7, giving equal voting power. The structural-indexer BENEFITS from being dual-lane: its pre_score (0.0216) is double the stress test's single-lane pre_score (0.0115). If weights favored either lane, the structural-indexer's relative advantage would change but it would still be the top-ranked candidate. Weight imbalance is not causal. Falsifiable test: run with weights 1.0/0.0 (dense-only) and 0.0/1.0 (FTS-only) — structural-indexer should still reach the rerank head in at least the dense-only path, and BGE should still score it below the stress test, confirming weights are irrelevant. Supporting evidence: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fusion.py:42-87`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/config.py:389-406`.

6. **Hypothesis (e) — Rerank top-K=20 cutoff too narrow (probability: 5%)**. Default `COCOINDEX_RERANK_TOP_K=20` and the probe 14 head has 9 candidates, all of which are in the rerank window. The structural-indexer IS scored by both BGE and Jina. Increasing top-K would add MORE candidates (beyond rank 20 in the fused list) but wouldn't change how BGE scores the ones already present. The bottleneck is within-model scoring, not window size. Falsifiable test: run with top-K=100 — structural-indexer's relative rank in the reranked head will be unchanged because BGE still scores it below the stress test. Supporting evidence: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/reranker.py:153-183`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/config.py:415-420`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/run-phase2-smoke.sh:73-90`.

### Synthesis: the two-factor interaction

7. The root cause is best modeled as an interaction between (f) and (a): BGE-reranker-v2-m3's cross-encoder rewards lexical token overlap between query and candidate text (factor f), and the top-ranked structural-indexer chunk delivers minimal lexical surface (9-line import header) because default chunking doesn't respect code structure (factor a). The pre-rerank pipeline is functioning correctly — it retrieves the right file and ranks it highest before rerank. The failure occurs inside the reranker, where the cross-encoder sees an information-poor candidate and scores it below information-rich distractors. Jina v3 succeeds with the same information-poor chunk because its cross-encoder architecture is less lexical-cue-dependent. Fixing EITHER factor would likely resolve probe 14: (path A) swap the reranker to a model with better semantic scoring, or (path B) improve chunking so the top-ranked slice contains richer content. Fixing both would be the most robust solution. Supporting evidence: `baseline-bge.rerank-scores.jsonl:6`, `jina-v3.rerank-scores.jsonl:6`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/indexer.py:33-39`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/reranker.py:139-183`.
