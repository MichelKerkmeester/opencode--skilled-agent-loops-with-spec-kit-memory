## Mechanism

OpenLTM’s authoritative memory unit is a DB row: `memories` stores `id`, `content`, `title`, confidence, status, and recall metadata, while `context_items` are also rows and explicitly “replace” per-project markdown context files (`packages/openltm-core/src/schema.sql:15`, `packages/openltm-core/src/schema.sql:17`, `packages/openltm-core/src/schema.sql:35`). Its lexical index is a derived FTS5 surface over those rows: `memories_fts` and `context_items_fts` are external-content FTS tables tied to rowids, with insert/delete/update triggers that mirror base-table changes (`packages/openltm-core/src/schema.sql:127`, `packages/openltm-core/src/schema.sql:129`, `packages/openltm-core/src/schema.sql:136`, `packages/openltm-core/src/schema.sql:147`, `packages/openltm-core/src/schema.sql:155`, `packages/openltm-core/src/schema.sql:174`). Migration 020 shows the rebuild pattern: drop/recreate FTS, repopulate from base rows, then recreate triggers (`migrations/020_fts_coverage.sql:7`, `migrations/020_fts_coverage.sql:16`, `migrations/020_fts_coverage.sql:20`, `migrations/020_fts_coverage.sql:24`, `migrations/020_fts_coverage.sql:43`, `migrations/020_fts_coverage.sql:50`).

Embeddings are row-keyed. Migration 010 splits vectors into `memory_embeddings(memory_id, embedding, model, dim, created_at)` with no verified content fingerprint (`migrations/010_memory_embeddings_split.sql:5`, `migrations/010_memory_embeddings_split.sql:6`, `migrations/010_memory_embeddings_split.sql:8`). Backfill detects only active rows with no embedding row (`packages/openltm-core/src/dao/embeddings.ts:63`, `packages/openltm-core/src/dao/embeddings.ts:69`, `packages/openltm-core/src/dao/embeddings.ts:71`). New `learn()` inserts enqueue/inline embedding by memory id (`packages/openltm-core/src/db.ts:352`, `packages/openltm-core/src/db.ts:396`, `packages/openltm-core/src/db.ts:401`, `packages/openltm-core/src/db.ts:403`), and the queue payload is only `{ memoryId }` (`packages/openltm-core/src/queue/index.ts:18`, `packages/openltm-core/src/queue/index.ts:35`, `packages/openltm-core/src/queue/worker.ts:54`). This is insertion/missing-row freshness, not edited-document freshness. VERIFIED content edit paths update `memories.content` but do not show embedding invalidation in the same block (`src/graph-server.ts:1123`, `src/graph-server.ts:1124`, `packages/openltm-core/src/janitor/dedup.ts:255`, `packages/openltm-core/src/janitor/dedup.ts:258`); UNKNOWN whether another path re-embeds afterward.

The stronger transferable signal is that derived vector surfaces are rebuildable from an embedding source table and model/dim are treated as compatibility boundaries: `setEmbedding()` stores model/dim and dual-writes vec when available (`packages/openltm-core/src/dao/embeddings.ts:36`, `packages/openltm-core/src/dao/embeddings.ts:45`), while `rebuildVecIndex()` rebuilds from `memory_embeddings` filtered by active dim/model (`packages/openltm-core/src/vec/index.ts:107`, `packages/openltm-core/src/vec/index.ts:119`, `packages/openltm-core/src/vec/index.ts:123`).

## Row-coupled findings (negative knowledge)

- OpenLTM’s `learn()` reinforcement path updates `confirm_count` and confidence for an existing `dedup_key`; that is row lifecycle, not authored-doc freshness (`packages/openltm-core/src/db.ts:317`, `packages/openltm-core/src/db.ts:321`, `packages/openltm-core/src/db.ts:324`).

- “Needs embedding” equals “no `memory_embeddings` row”; edited docs can have an existing but stale vector, so this should not be ported as-is (`packages/openltm-core/src/dao/embeddings.ts:63`, `packages/openltm-core/src/dao/embeddings.ts:69`).

- Embedding jobs keyed only by `memoryId` are insertion-row coupled; doc systems need versioned jobs keyed to the edited document snapshot (`packages/openltm-core/src/queue/index.ts:18`, `packages/openltm-core/src/queue/index.ts:40`).

- `context_items` as rows replacing markdown context files is anti-fit for system-spec-kit, where markdown continuity docs remain source of truth (`packages/openltm-core/src/schema.sql:15`, `packages/openltm-core/src/schema.sql:17`).
