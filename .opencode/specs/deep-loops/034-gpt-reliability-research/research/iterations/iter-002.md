### FINDINGS

1. The setup contract reaches the executor as a referenced asset, not as an inlined template. The presentation assets declare themselves the visible-prompt source of truth (`.opencode/commands/deep/assets/deep_review_presentation.txt:3`, `.opencode/commands/deep/assets/deep_context_presentation.txt:3`, `.opencode/commands/deep/assets/deep_agent-improvement_presentation.txt:3`), while the command doc only tells the executor to load that asset before showing startup questions (`.opencode/commands/deep/review.md:15`) and lists the asset path (`.opencode/commands/deep/review.md:80-84`). Mechanism: GPT only sees the verbatim prompt if it follows the “load the presentation contract” instruction; otherwise it reconstructs from the router prose.

2. The full consolidated setup prompt exists as a fenced human-instruction block, not a machine-bound render template. Review defines `EXECUTE THIS SINGLE CONSOLIDATED PROMPT` at `.opencode/commands/deep/assets/deep_review_presentation.txt:94-99`, then question text at `.opencode/commands/deep/assets/deep_review_presentation.txt:163-202`; context does the same at `.opencode/commands/deep/assets/deep_context_presentation.txt:66-69` and `.opencode/commands/deep/assets/deep_context_presentation.txt:133-161`; agent-improvement does the same at `.opencode/commands/deep/assets/deep_agent-improvement_presentation.txt:60-65` and `.opencode/commands/deep/assets/deep_agent-improvement_presentation.txt:117-148`. Mechanism: Claude is likely copying the loaded block by convention; GPT is satisfying the semantic halt rule but paraphrasing/omitting pieces because no “render these exact lines” structure is enforced.

3. Current `:auto` semantics conflict with “bare auto should show consolidated setup.” The presentation files say `:auto` must not emit the consolidated prompt by default (`.opencode/commands/deep/assets/deep_review_presentation.txt:11`, `.opencode/commands/deep/assets/deep_context_presentation.txt:11`, `.opencode/commands/deep/assets/deep_agent-improvement_presentation.txt:13`) and route missing positional inputs to Tier 3 fail-fast (`.opencode/commands/deep/assets/deep_review_presentation.txt:21-23`, `.opencode/commands/deep/assets/deep_context_presentation.txt:21-23`, `.opencode/commands/deep/assets/deep_agent-improvement_presentation.txt:23-25`). The shared auto contract also says missing input is not ambiguity and Tier 3 must not emit setup questions (`.opencode/skills/system-spec-kit/references/workflows/auto_mode_contract.md:47-56`, `.opencode/skills/system-spec-kit/references/workflows/auto_mode_contract.md:116-130`). Mechanism: GPT has to reconcile a full setup-prompt block with a separate fail-fast rule, so partial rendering is predictable.

4. Confirm-mode YAML does not bind or carry the presentation text; it consumes already-resolved setup values. `deep_review_confirm.yaml` only declares placeholder inputs (`.opencode/commands/deep/assets/deep_review_confirm.yaml:26-37`) and later validates required bindings before file writes (`.opencode/commands/deep/assets/deep_review_confirm.yaml:179-186`). The router says YAML loads only after required inputs are bound (`.opencode/commands/deep/review.md:94-97`). Mechanism: the halt-time presentation happens before YAML; YAML cannot make GPT render the full setup prompt.

5. Exact gap: presentation ownership is documented, but no structural enforcement exists at halt time. The router says presentation content “lives only” in the asset (`.opencode/commands/deep/review.md:118-126`, `.opencode/commands/deep/context.md:107-115`, `.opencode/commands/deep/agent-improvement.md:148-156`), while the shared fail-fast template lives elsewhere (`.opencode/skills/system-spec-kit/references/workflows/auto_mode_contract.md:112-130`). Mechanism: the executor must assemble command name, missing fields, mode, applicable questions, and stop behavior from multiple prose sources; Claude follows the convention, GPT partially reconstructs.
