# Iteration 14 Report — Reversibility & Kill-Switch Fleet-Wide Semantics

## Headline finding

The activation/flip toolchain **claims byte-exact reversibility but the serving-prior rollback target is silently invalidated by any subsequent `activate-hub.cjs` re-mint**, and the kill-switch is **bi-state (`'1'` vs everything-else) rather than the tri-state the P4 default-on target requires**. Combined with a missing operator rollback command on `activate-hub.cjs` and three layered env-strip paths, the cutover cannot currently be reversed or killed cleanly under fleet-wide P4 semantics.

## Verified claims

**Reversibility hazard — stale serving-prior after re-activation.** `flip-serving.cjs:125` captures the pre-flip manifest only on the FIRST flip ever: `if (!fs.existsSync(servingPriorPath)) fs.writeFileSync(servingPriorPath, fs.readFileSync(manifestPath));`. The guard never refreshes `manifest.serving-prior.json`. Meanwhile `activate-hub.cjs:214` unconditionally swaps `manifest.json ← manifest.candidate.json` on a new P4a re-activation, with no corresponding refresh of `manifest.serving-prior.json`. The current `system-deep-loop` state is one activate-hub call away from triggering this: `manifest.json` is gen3 compiled (hash `82d7f3...`), `manifest.candidate.json` is gen4 (hash `7b5491...`), and `manifest.serving-prior.json` is still gen3 legacy. Sequence: activate-hub swaps manifest → gen4 legacy; flip-serving skips re-save (file exists) → flips to gen4 compiled; operator runs `--rollback` → restores gen3 legacy (WRONG; correct target is gen4 legacy).

**Missing operator rollback on P4a activation.** `activate-hub.cjs:79-92` `parseArgs` accepts only `--hub`, `--child`, `--verify`, `--json`. The word "rollback" appears only inside `proveRollback` (line 161-180), an in-memory byte-exact proof run inside a scoped temp dir that proves rollback is *possible* but produces no operator command and writes no artifact. This is asymmetric with `flip-serving.cjs:101-111`, which DOES implement `--rollback` for the serving flip. To undo a bad activation, an operator must hand-edit JSON — an untenable recovery posture for fleet-wide P4.

**Kill-switch is bi-state, not tri-state.** Both read sites use exact-string `'1'` match: `resolve.cjs:23` (`process.env[FLAG] === '1'`) and `advisor-recommend.ts:362` (`process.env.SPECKIT_COMPILED_ROUTING !== '1'`). Unset, `'0'`, `'false'`, `'off'`, `''`, `'true'`, `'yes'`, `'on'`, `'2'` ALL collapse to legacy. There is no code path that distinguishes "operator explicitly killed" (`'0'`) from "compiled routing is unset because P4 hasn't shipped" — yet 012's P4 target requires exactly that distinction. Refines F-15-3 (which flagged benchmark coverage of this gap) by naming where the production resolver itself must change.

**Silent fail-safe on absent spec tree.** `compiled-route.cjs:33-38` wraps the resolver require in `try { ... } catch { route = null; }` and emits only `{ servingAuthority: 'legacy', hubId }`. The resolver path is hardcoded into the spec tree (`compiled-route.cjs:16-21`). If the spec tree is moved/deleted (e.g., during the 019-program cutover the 007 phase gets archived), the front door fails safe to legacy with NO stderr line, NO log, NO telemetry. For P4 default-on, an archived spec tree becomes an invisible fleet-wide rollback.

**Fence epoch is direction-ambiguous.** `flip-serving.cjs:105` (rollback) and `flip-serving.cjs:128` (forward) both do `nextFence = fence + 1`. A state reached by `flip → rollback → flip` has the same monotonic shape as `flip → flip`; nothing in `fence-state.json` records direction. The audit record (`serving-flip-record.json`) captures `before/after` once and is then overwritten on the next flip (already flagged by F-12-4), so for P4 the fence alone cannot answer "did we go forward or backward to get here?".

**`.env` interaction with the launcher allowlist.** `mk-skill-advisor-launcher.cjs:70-76` loads `.env.local` / `.env` into `process.env` at module top-level, BEFORE any spawn. `createChildEnv()` (line 267-286) then filters through `CHILD_ENV_ALLOWLIST` (line 99-150), which does NOT contain `SPECKIT_COMPILED_ROUTING` (confirmed: zero matches). The standard operator guidance "put it in `.env`" therefore silently fails: launcher sees the flag, spawned MCP server child does not. This compounds F-5-1 (launcher allowlist) and F-8-2 (bridge allowlist) with the `.env` interaction that makes it operator-invisible.

## Ranked actionable recommendations

1. **P0 — Make serving-prior refresh atomic with manifest swap.** Either (a) `activate-hub.cjs` re-writes `manifest.serving-prior.json` whenever it swaps `manifest.json` at line 214, or (b) `flip-serving.cjs` removes the `if (!fs.existsSync(...))` guard at line 125 and unconditionally re-saves pre-flip. Option (b) is cheaper and also repairs rollback-after-rollback cycles.
2. **P0 — Add `--rollback` to `activate-hub.cjs` symmetric with `flip-serving.cjs`.** Reuse the byte-exact hash-validated restore path already proven by `proveRollback`. Without it, P4a recovery is unscripted.
3. **P0 — Convert the kill-switch to tri-state before P4.** Define `unset → on (post-P4 default)` / `'1' → on (explicit)` / `'0' | 'false' | 'off' → off (explicit kill)`. Update both `resolve.cjs:23` and `advisor-recommend.ts:362` and add the flag to BOTH `CHILD_ENV_ALLOWLIST` sets (launcher line 99-150, bridge line 58-87).
4. **P1 — Emit a stderr breadcrumb when the front door fail-safes.** One line in `compiled-route.cjs:36-38` (`process.stderr.write('compiled-route: resolver unavailable, using legacy\n')`) preserves fail-safe semantics while giving operators a fighting chance.
5. **P1 — Encode cutover direction in the fence record.** Either add `direction: 'forward' | 'rollback'` to `fence-state.json` writes, or stop incrementing the fence on rollback so `manifest.json` fence value can be reconciled against `serving-flip-record.json.before`.
6. **P1 — Document the `.env` strip in ENV-REFERENCE.md.** Operators need to know `SPECKIT_COMPILED_ROUTING` in `.env` is invisible to the spawned advisor child until the allowlist adds it.
