## Review: document-helpers + five-port storage adapter seam

**Scope traced:** `document-helpers.ts`; all five ports (`vector-store`, `lexical-search`, `graph-traversal`, `maintenance`, `contention-policy`) plus `common.ts`/`index.ts`; and the real dependencies the ports call — `vector-index-mutations.ts` (`writeActiveVectorPayload`/`deleteActiveVectorPayload`), `vector-index-queries.ts` (`get_memory`), and `vector-index-store.ts` (`activeVectorSource`).

**What is sound (verified, not assumed):**
- `document-helpers.ts` — both functions are pure; weights and the spec-type predicate are correct. No defects.
- Vector identity contract — `memory_index.id === vec_memories.rowid` holds: `writeActiveVectorPayload` inserts `vec_memories(rowid=id)` (mutations.ts:84), and `getEmbeddingForId` reads back `WHERE rowid = BigInt(id)` (vector-store.ts:395-400). Consistent.
- `search` port↔legacy overload similarity scaling round-trips correctly (`minScore*100` in, `similarity/100` out). The advisory-id semantics are thoroughly documented in the interface JSDoc.
- `contention-policy` retry math (`retryAttempts`, `retryDelayMs`), `isSqliteContentionError` code+message matching, and `defaultSleepSync` (Atomics.wait on a zeroed buffer = blocking sleep) are all correct.
- `maintenance.checkpoint` interpolates `wal_checkpoint(${mode})` but `mode` is constrained by the `CheckpointOptions` union and the local trusted DB — not an injection boundary.

**Two latent boundary-correctness asymmetries (both P2 — real code defects, but on paths with no live caller today):**

1. **Async write-lock leaks an open transaction on COMMIT failure.** In `runWithWriteLock` (contention-policy.ts:210-219), the promise success handler runs `database.exec('COMMIT')` *unguarded*; if COMMIT throws (e.g. `SQLITE_BUSY` on a WAL commit), the promise rejects with no `ROLLBACK`, leaving the connection mid-transaction. The **sync** path (line 221, inside the outer `try`) does roll back via the `catch → rollbackQuietly`. Confirmed dormant: `withWriteLock` has zero production callers and no async operation is ever passed to it (only two sync-op tests), so the promise branch is entirely unexercised — which is exactly why prior passes wouldn't have caught it.

2. **`VectorStore.clear()` orphans per-embedder shard tables.** `clear()` (vector-store.ts:304-312) deletes `vec_memories` + `active_memory_projection` + `memory_index`, but when a non-default embedder is active, `writeActiveVectorPayload` *also* writes the embedding to a per-dim shard (`vecTableNameForDim`, mutations.ts:87-90), and single-record `deleteActiveVectorPayload` correctly cleans that shard (mutations.ts:95-98). The bulk `clear()` does not, contradicting its own JSDoc ("Full memory-store reset … removes all vector records"). Dormant: no production caller of the port `clear()` and the orphan only manifests under a non-default embedder.

Neither rises to P0/P1: no data-loss/crash/security on any live path, and the epic's hardened behavior is intact. These are defensive-hardening items at the adapter seam matching this seat's correctness angle.
