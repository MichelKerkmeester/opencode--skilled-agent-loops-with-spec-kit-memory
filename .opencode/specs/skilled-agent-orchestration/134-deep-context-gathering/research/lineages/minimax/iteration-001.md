## PriorArt
- **Aider repo-map** | tree-sitter extracts per-file symbols (classes, methods, signatures with types) → files become nodes in a graph with edges for imports/calls → a PageRank-style graph-ranking algorithm is run with the chat's currently-edited files as seeds to greedily fill a `--map-tokens` budget (default 1k tokens), emitting only the most-referenced symbols | Cheap, deterministic, language-aware, no LLM at index time; degrades to noise on monorepos with weak tree-sitter grammars and ignores runtime/behavioral context; static — does not see call frequency or recent edits | **Q3** — make the coverage-frontier a graph-ranked candidate list of *symbols* (not files) seeded by task entry-points (modified paths, public exports, test files), filling a token budget; skip text-only assets. **Q1** — let convergence track "% of task-relevant typed-signature surface covered" rather than "% of files visited". **Q2** — make the first section of the Report a typed symbol skeleton (defs + signatures) ranked by graph centrality, not raw source bodies.
- **Universal ctags / etags** | Regex-based extraction emitting a flat tags index (definitions + references); typically used as a coarse pre-filter before a heavier AST/graph pass | Fast, language-agnostic, near-zero infra cost; no type info, no scoped call graph, frequently misses dynamic features | **Q3** — use as the first pass to enumerate candidate symbols across all files before the more expensive tree-sitter + graph step; cheap enough to re-run on every iteration.
- **RAPTOR (Sarthi et al. 2024, arXiv:2401.18059)** | Recursively chunks text, clusters (UMAP + clustering) and abstractively summarizes bottom-up to build a tree at multiple granularities; retrieval can hit chunk, cluster summary, or root | Recovers cross-document themes flat chunks miss (+20% absolute on QuALITY with GPT-4); higher levels lose exact details and can fabricate when summary abstracts past actual code behavior | **Q3** — chunk the codebase at multiple granularities (symbol → file → module → subsystem) and let the sweep route a query to the right level. **Q2** — the Report should carry a "zoom ladder" from cluster-level summary down to pointers, with the LLM-driven slice step filling only the gaps the ladder doesn't already cover.
- **Microsoft GraphRAG (Edge et al. 2024, arXiv:2404.16130)** | LLM extracts entities + relationships from chunks → builds a knowledge graph → Leiden community detection partitions it → communities are LLM-summarized hierarchically (leaf→root); retrieval has "local" (entity-neighborhood) and "global" (community-summary reasoning) modes | Powerful for "what themes run through this corpus" global questions; very expensive two-pass LLM indexing; brittle schema-drift on code (code is *already* graph-shaped) and community summaries are the canonical stale-summary failure mode | **Q2** — use a community/summary-style section in the Report for thematic knowledge (auth model, error-handling convention, layering rules) that grep/RAG cannot surface. **Q1** — include a "summary-staleness check": re-validate every hierarchical summary against the slices it was derived from, regenerate when content-hash drifts.
- **Agentic/iterative retrieval vs one-shot embedding RAG (Anthropic Research, 2025)** | Replace one-shot top-k embedding retrieval with a planner agent that decides what to retrieve next based on intermediate findings; orchestrator-worker pattern showed +90.2% over single-agent Opus 4 on Anthropic's research eval, with token usage explaining 80% of BrowseComp variance | Recovers non-cosine-similar connections, adapts to surprises, parallelizes naturally; 4–15× the token cost of a single call, and gains evaporate without disciplined task decomposition and tool ergonomics | **Q1** — convergence = "no subagent returns more than K new reuse-candidates in the last N rounds" (novelty saturation), not "% of slices visited" or "context filled". **Q3** — slice dispatch must be orchestrator-driven and re-routable, not a fixed sweep, so the loop can pivot to hot modules the moment an entry-point is found.
- **Anthropic sub-agent architecture (2025)** | Lead agent plans and dispatches N parallel subagents, each with its own context window; subagents explore with tools and return a condensed 1k–2k token summary; lead synthesizes and either dispatches more or terminates | Each subagent can burn tens of thousands of tokens without polluting the lead; separation of concerns reduces path-dependency | **Q2** — enforce a strict per-subagent output schema (`{slice_id, symbols_seen, reuse_candidates:[{path,symbol,signature,why_reusable,confidence}], summary, gaps}`), not free-form text, so the lead can dedupe, score, and assemble the Report mechanically.
- **Context compaction (Anthropic Claude Code, 2025)** | When working context nears the window limit, summarize it into a high-fidelity digest and reinit with summary + last-N accessed slices; the model is prompted to maximize recall first, then precision | Preserves coherence across long horizons; over-aggressive compaction silently loses subtle details whose importance appears only later | **Q2** — the Report itself is the durable persisted compaction artifact: slices are working memory, the Report is the digest, re-running the sweep is "redo", not incremental mutation. **Q1** — amortize via per-slice summary cache keyed by content hash so the same slice is never re-summarized within a run.
- **Anthropic tool design for agents (Sep 2025)** | Tight tool descriptions, token-budgeted responses, pagination/filter/range parameters, "concise" vs "detailed" `response_format` enums, truncation default (Claude Code caps tool responses at 25k tokens) | Cuts context rot from oversized tool results; misrouted agents when tool descriptions are vague | **Q3** — expose the slice-reading tool with two modes: `skeleton` (signatures + docstring + first line, ~50–200 tokens per symbol) and `body` (full source). Subagents default to `skeleton` and only pull `body` for symbols they actually cite. **Q2** — the Report records which mode produced each entry so downstream consumers can re-pull on demand.
- **"Just-in-time" context (Anthropic context-engineering, Sep 2025)** | Don't pre-load everything; keep lightweight identifiers (paths, symbol names) and let agents pull data on demand via grep/glob/head | Mirrors human filesystem use; avoids stale indexing and full pre-processing cost; slower at runtime if tools are misdesigned | **Q3** — the coverage-frontier is a *list of pointers* (path + qualified symbol + signature + 1-line purpose), not source bodies. The Report ships pointers; the planning/implementation agent that consumes the Report fetches bodies.
- **Hybrid pre-load + just-in-time (Anthropic Claude Code pattern)** | `CLAUDE.md`-style files are naively pre-loaded; primitives like `grep`/`glob` handle dynamic lookup | Beats either pure pre-load (stale) or pure runtime (slow) for codebases; requires curated conventions | **Q2** — ship a "Conventions / implicit rules" section in the Report (preferred libs, error idiom, layering) derived from observed patterns across slices, so the planner inherits decisions without re-deriving them.
- **Chroma Context Rot (Hong/Troynikov/Huber, Jul 2025)** | Across 18 LLMs incl. GPT-4.1, Claude 4, Gemini 2.5, Qwen3: performance degrades *non-uniformly* with input length on constant tasks; distractors degrade non-uniformly (some are far worse); structurally *coherent* haystacks hurt vs. shuffled ones; needle position has little effect; a single semantically-adjacent distractor measurably drops retrieval | Empirical proof "more context ≠ better" and that ordering/structure matters; the coherence-hurts finding is counter-intuitive and damning for "dump whole files in canonical order" | **Q1** — never use "context length filled" or "visited all slices" as convergence; use *marginal-utility-of-adding-context* (new-info-per-token). **Q2** — order Report sections by *reuse-candidate cluster*, not by file narrative, to avoid the coherent-haystack penalty. **Q3** — chunk slices by semantic unit (symbol + its immediate callers/callees), not by file boundary, so a subagent never receives a coherent story that can mislead attention.
- **LongMemEval + RAG pitfall (Chroma 2025)** | Adding irrelevant context forces the model to do retrieval + reasoning in one call; performance on focused (~300 tok) vs full (~113k tok) inputs drops sharply even on capable models; Claude family abstains more (conservative) and GPT family hallucinates more (confident) under ambiguity | Confirms static-RAG is structurally weaker than focused-input reasoning, and that ambiguity-handling differs by family | **Q2** — the Report must be *focused*: REUSE catalog + a few "rule" lines, not a 100k-token corpus dump. **Q1** — if a subagent's slice exceeds a per-slice token budget, force it to return a skeleton-only summary, never the full body.
- **Aider `CONVENTIONS.md`** | A persistent small markdown file always loaded into context listing project conventions (preferred libs, type conventions) | Cheap, very high signal, works because it is explicit | **Q2** — auto-derive a "Conventions" section of the Report from observed cross-slice patterns (import choices, error idioms, type-annotation style) so the consumer inherits them without re-derivation.

## NegativeKnowledge
- **Naive whole-repo embedding RAG** — Chroma 2025 + NoLiMa prove semantic similarity ≠ reuse relevance, and similarity-tier mismatch gets *worse* as context grows; also dead-on-arrival for symbol-level reuse (you want the exact signature of `parseFoo`, not topically-similar functions).
- **Pre-baking raw source into the Report** — direct path to context rot; the Report's job is *typed pointers*, not bodies. The consumer pulls bodies just-in-time.
- **Stale hierarchical / community summaries** — GraphRAG's known failure mode; LLM-summaries of code modules drift within days. Re-validate summaries against their source slices via content-hash, or regenerate.
- **"Visited N% of slices" as convergence** — confuses coverage with utility. Use *new reuse-candidates per round* or *novelty-per-token*; a slice can be visited and yield nothing.
- **Coherent narrative ordering of slices** — Chroma shows structurally coherent inputs *hurt* long-context accuracy; group by reuse-candidate cluster, not by file order or call-graph story.
- **One big LLM call with all slices** — loses on every axis: context rot, no parallelism, no orchestrator for adaptive routing, single point of hallucination.
- **"Distractors don't matter because the model is smart"** — Chroma 2025 disproved: even one near-needle distractor measurably degrades retrieval, and distractor impact is non-uniform (some are ~3× worse than others).
- **Treating a large context window as a license to stuff it** — Anthropic: "context must be treated as a finite resource with diminishing marginal returns"; sub-agents with focused contexts outperform one agent with a 1M-token window.
- **Confidence scores without source-pointers** — without path+symbol+line refs, the Report cannot be re-validated; a reuse-candidate is useless if the consumer cannot locate it.
- **Run-once static index** — codebases change daily; static pre-computed embeddings/summaries go stale; the slice sweep should be content-hash keyed, with incremental re-use of unchanged slices.

## TopRecommendations
1. **Coverage-frontier = graph-ranked symbol list, not file list** — use tree-sitter (or ctags as a fast first pass) to extract typed symbols, run PageRank over the import/call graph seeded from task entry-points, greedily fill a token budget. Skip text-only assets; this is the only way to keep the frontier useful on large repos.
2. **Slice dispatch is orchestrator-driven, not a fixed sweep** — adapt the Anthropic multi-agent pattern; convergence = "≤K new reuse-candidates in the last N subagent rounds" (novelty saturation), never "% slices visited". The frontier re-ranks each round based on what subagents returned.
3. **Per-subagent output schema is structured JSON, not free text** — `{slice_id, symbols_seen, reuse_candidates:[{path,symbol,signature,why_reusable,confidence,evidence_refs}], summary, gaps}` — so the lead can dedupe, score, and assemble the Report mechanically, and Q1 can be computed mechanically.
4. **Two-mode slice tool: `skeleton` (signatures + docstring) and `body` (full source)** — subagents default to `skeleton` and only pull `body` for cited symbols. The Report ships pointers + signatures; the planning/implementation agent pulls bodies just-in-time. This is the single biggest defense against context rot.
5. **Convergence uses marginal-utility, not coverage** — track novelty-per-token and "ranked-signature-surface covered %"; stop when marginal-utility < threshold across N consecutive rounds. Re-rank the frontier when novelty drops. Re-validate hierarchical summaries against their source slices by content-hash to prevent the GraphRAG-staleness failure mode.
