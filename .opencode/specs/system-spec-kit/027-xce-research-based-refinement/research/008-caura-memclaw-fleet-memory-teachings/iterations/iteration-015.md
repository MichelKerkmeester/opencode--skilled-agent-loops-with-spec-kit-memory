## Verified mechanism (deep)
Re-embed trigger:
MemClaw embeds on create unless cached or deferred. It computes `content_hash`, checks storage for an existing embedded row, and reuses that vector when found (`core-api/src/core_api/services/memory_service.py:565`, `core-api/src/core_api/services/memory_service.py:570`, `core-api/src/core_api/services/memory_service.py:577`). If no vector is available, the write stores `embedding=NULL`, sets `metadata["embedding_pending"]=True`, and schedules `_schedule_embed_or_reembed(...)` with the hash (`core-api/src/core_api/services/memory_service.py:838`, `core-api/src/core_api/services/memory_service.py:961`).

Deferred mode is explicit: `deployment_mode="deferred"` stores `embedding=NULL`, publishes `EMBED_REQUESTED` and `ENRICH_REQUESTED`, and lets `core-worker` patch the row later (`core-api/src/core_api/config.py:63`, `core-api/src/core_api/config.py:65`, `core-api/src/core_api/config.py:66`).

Content updates trigger immediate re-embedding and re-hashing, not a stale-state transition. The update path detects `content_changed`, calls `get_embedding(data.content, tenant_config)`, computes a new hash, checks duplicate constraints, and patches `content`, `embedding`, and `content_hash` (`core-api/src/core_api/services/memory_service.py:2281`, `core-api/src/core_api/services/memory_service.py:2285`, `core-api/src/core_api/services/memory_service.py:2287`, `core-api/src/core_api/services/memory_service.py:2320`, `core-api/src/core_api/services/memory_service.py:2322`).

Bulk creates compute hashes per item, use them for duplicate detection, persist the hash, and queue bulk re-embedding only for rows whose embedding is `None` (`core-api/src/core_api/services/memory_service.py:1094`, `core-api/src/core_api/services/memory_service.py:1101`, `core-api/src/core_api/services/memory_service.py:1257`, `core-api/src/core_api/services/memory_service.py:1485`, `core-api/src/core_api/services/memory_service.py:1509`).

Backfill is NULL-driven, not version-driven. It scans rows whose `embedding IS NULL`, fetches content plus `content_hash`, and republishes embed requests (`core-worker/src/core_worker/backfill.py:22`, `core-worker/src/core_worker/backfill.py:65`, `core-worker/src/core_worker/backfill.py:96`, `core-worker/src/core_worker/backfill.py:122`, `core-worker/src/core_worker/backfill.py:126`).

Fingerprint computation:
The memory fingerprint is `sha256(f"{tenant_id}:{fleet_id or ''}:{content}")`; it includes tenant and fleet scope plus exact content (`core-api/src/core_api/services/memory_service.py:91`, `core-api/src/core_api/services/memory_service.py:92`). It is stored as `Memory.content_hash` and indexed by tenant plus hash (`common/models/memory.py:33`, `common/models/memory.py:88`).

Fingerprint comparison:
The worker uses the request `content_hash` to call `find_embedding_by_content_hash`, then skips the provider call on hit (`core-worker/src/core_worker/consumer.py:144`, `core-worker/src/core_worker/consumer.py:146`, `core-worker/src/core_worker/consumer.py:147`, `core-worker/src/core_worker/consumer.py:161`). Storage only returns rows where tenant and hash match, embedding is non-NULL, and row is not deleted (`core-storage-api/src/core_storage_api/services/postgres_service.py:646`, `core-storage-api/src/core_storage_api/services/postgres_service.py:655`, `core-storage-api/src/core_storage_api/services/postgres_service.py:656`, `core-storage-api/src/core_storage_api/services/postgres_service.py:657`, `core-storage-api/src/core_storage_api/services/postgres_service.py:658`).

Model-version invalidation:
Correction to 085: MemClaw does not durably record document-vector embedder provenance on memory rows. `Memory` has `embedding`, `content_hash`, `metadata`, `client_request_id`, and lifecycle fields, but no `embedding_provider`, `embedding_model`, `embedding_version`, `embedding_dim`, `embedded_at`, or `embedding_state` columns (`common/models/memory.py:23`, `common/models/memory.py:24`, `common/models/memory.py:28`, `common/models/memory.py:33`, `common/models/memory.py:39`, `common/models/memory.py:56`). Storage `memory_update_embedding` writes only `embedding` and optionally wholesale metadata (`core-storage-api/src/core_storage_api/services/postgres_service.py:581`, `core-storage-api/src/core_storage_api/services/postgres_service.py:588`, `core-storage-api/src/core_storage_api/services/postgres_service.py:589`).

Provider/model config exists at runtime, not as durable per-row provenance. The provider protocol exposes `provider_name` and `model` (`common/embedding/protocols.py:33`, `common/embedding/protocols.py:37`). The OpenAI provider cache keys on API key, model, base URL, dimensions flag, query instruction, and truncation (`common/embedding/_registry.py:93`, `common/embedding/_registry.py:95`, `common/embedding/_registry.py:97`, `common/embedding/_registry.py:99`). Tenant config can supply an embedding model (`core-api/src/core_api/services/organization_settings.py:436`, `core-api/src/core_api/services/organization_settings.py:442`). But persisted memory rows do not store that tuple.

Search query embeddings do have cache-versioning: query cache hashes model, vector dim, query instruction, tenant, and normalized query, and bumps key prefix when inputs change (`core-api/src/core_api/services/memory_service.py:2641`, `core-api/src/core_api/services/memory_service.py:2651`, `core-api/src/core_api/services/memory_service.py:2652`, `core-api/src/core_api/services/memory_service.py:2655`). That mechanism does not invalidate stored document embeddings.

Worker idempotency:
Embed worker idempotency is content-hash reuse plus safe overwrite. The consumer doc explicitly states redelivery re-PATCHes the same embedding and content-hash lookup reduces redelivery cost (`core-worker/src/core_worker/consumer.py:25`, `core-worker/src/core_worker/consumer.py:26`, `core-worker/src/core_worker/consumer.py:28`). It validates payloads, drops malformed poison messages, looks up cached vectors, calls provider only on miss, and patches storage (`core-worker/src/core_worker/consumer.py:96`, `core-worker/src/core_worker/consumer.py:99`, `core-worker/src/core_worker/consumer.py:146`, `core-worker/src/core_worker/consumer.py:162`, `core-worker/src/core_worker/consumer.py:171`). The storage client clears `metadata.embedding_pending` on successful patch (`core-worker/src/core_worker/clients/storage_client.py:187`, `core-worker/src/core_worker/clients/storage_client.py:198`).

Enrich worker idempotency is merge-and-skip based. It snapshots agent-provided fields so redelivery cannot overwrite agent choices (`core-worker/src/core_worker/consumer.py:368`, `core-worker/src/core_worker/consumer.py:370`, `core-worker/src/core_worker/consumer.py:390`, `core-worker/src/core_worker/consumer.py:415`). It clears `metadata.enrichment_pending` with a JSON metadata patch (`core-worker/src/core_worker/consumer.py:431`, `core-worker/src/core_worker/consumer.py:440`). It only allows real LLM output to overwrite always-write metadata fields, using `llm_ms > 0` to prevent heuristic fallback redelivery from clobbering prior LLM output (`core-worker/src/core_worker/consumer.py:445`, `core-worker/src/core_worker/consumer.py:456`). Storage applies `metadata_patch` as an atomic JSONB merge, not a whole metadata replacement (`core-storage-api/src/core_storage_api/services/postgres_service.py:432`, `core-storage-api/src/core_storage_api/services/postgres_service.py:507`, `core-storage-api/src/core_storage_api/services/postgres_service.py:531`).

Derived-state machine:
Correction to 085: MemClaw does not have explicit `pending/stale/embedded` state-machine columns. The effective state is inferred from `embedding IS NULL`, JSON metadata flags `embedding_pending` and `enrichment_pending`, and ordinary `status` values such as `active`/`deleted`/`outdated` (`common/models/memory.py:24`, `common/models/memory.py:28`, `common/models/memory.py:56`, `core-api/src/core_api/services/memory_service.py:385`, `core-api/src/core_api/services/memory_service.py:386`). ## Concrete Spec-Kit-native adoption sketch (027/003)
Record shape for SQLite single-daemon incremental indexing:
`memo_chunk_index(chunk_id TEXT PRIMARY KEY, memo_id TEXT NOT NULL, doc_path TEXT NOT NULL, chunk_key TEXT NOT NULL, chunk_kind TEXT NOT NULL, line_start INTEGER NOT NULL, line_end INTEGER NOT NULL, content_hash TEXT NOT NULL, dependency_hash TEXT NULL, embedder_id TEXT NOT NULL DEFAULT 'ollama:nomic-embed-text', embedder_dim INTEGER NOT NULL DEFAULT 768, embedder_version TEXT NOT NULL, embedding_state TEXT NOT NULL CHECK embedding_state IN ('pending','embedded','stale','failed'), enrichment_state TEXT NOT NULL CHECK enrichment_state IN ('pending','enriched','stale','failed','none'), embedded_at TEXT NULL, enriched_at TEXT NULL, failure_count INTEGER NOT NULL DEFAULT 0, last_error TEXT NULL, updated_at TEXT NOT NULL)`.

Use `chunk_id = sha256(doc_path + '\0' + chunk_key)` where `chunk_key` is stable semantic identity such as heading path, memo record id, frontmatter id, or chunk-kind plus local ordinal fallback. Store `line_start` and `line_end` as location metadata, but do not include line numbers in `content_hash` unless the chunk has no stable key. This avoids re-embedding unchanged text that merely moved.

Use `content_hash = sha256(canonical_chunk_text)` for embedding reuse. Include `chunk_kind` in a separate comparison gate, not necessarily in the embedding hash, because identical text under different semantic kinds may need different enrichment but not necessarily a different vector. If 027/003 wants stricter semantics, use `embedding_hash = sha256(chunk_kind + '\0' + canonical_chunk_text)` and `content_hash = sha256(canonical_chunk_text)`.

Use explicit fixed embedder provenance despite fixed default: `embedder_id='ollama:nomic-embed-text'`, `embedder_dim=768`, `embedder_version=<ollama model digest or configured local model version>`. The fixed nomic-768d default stays fixed operationally, but the row records the exact active tuple so future model refreshes are controlled rather than invisible.

Change-detection algorithm:
On each daemon scan, parse docs into chunk candidates with `doc_path`, `chunk_key`, `chunk_kind`, `line_span`, `content_hash`, and dependencies. Load existing rows by `doc_path + chunk_key`. If no row exists, insert `embedding_state='pending'` and `enrichment_state='pending'` as needed. If row exists and `content_hash` is unchanged and embedder tuple matches, update only `line_start`, `line_end`, dependency data, and `updated_at`; reuse embedding. If `content_hash` changed, set `embedding_state='stale'` or `pending`, clear `embedded_at`, keep old vector usable only if retrieval allows stale reads, and set enrichment to `stale` only for enrichment kinds affected by content. If only dependency DAG changed, update dependency rows and set graph-derived state stale, not vector state. If a prior chunk disappears, mark deleted or tombstone instead of immediate hard delete until dependent DAG cleanup completes.

Embedding/enrichment idempotency:
Use a single local work loop with compare-and-swap style updates. When selecting work, capture `(chunk_id, content_hash, embedder_id, embedder_dim, embedder_version)`. Before writing the vector, run `UPDATE ... WHERE chunk_id=? AND content_hash=? AND embedder_id=? AND embedder_dim=? AND embedder_version=?`. If zero rows update, the chunk changed while work was in flight; discard the result. If duplicate work is queued, first check whether state is already `embedded` for the same tuple and no-op. This ports MemClaw’s redelivery safety without Pub/Sub.

Minimal embedder-change invalidation:
At daemon startup, compute `active_embedder_tuple=('ollama:nomic-embed-text', 768, version)`. Mark only rows where `(embedder_id, embedder_dim, embedder_version) != active_tuple` as `embedding_state='stale'`. Do not touch rows already matching. Do not mark enrichment stale unless the enrichment prompt/version or chunk content changed. For fixed nomic-768d, most runs will mark zero rows; a future Ollama model digest change marks exactly the rows with old provenance.

Spec Kit should adopt MemClaw’s content-hash reuse but improve on MemClaw by adding durable provenance and explicit derived-state columns. That directly serves 027/003’s memo records, dependency DAG, chunk fingerprints, chunk kinds, and chunk line-spans before handler scan changes.

## Negative knowledge (confirmed)
Do not port Pub/Sub topics, DLQs, event envelopes, worker fleet deployment, per-tenant concurrency slots, Cloud Run ID-token signing, tenant credential payload transport, or multi-tenant scoping. These exist to make distributed SaaS workers safe, while Spec Kit Memory is a local single-user daemon.

Do not port Postgres/pgvector-specific storage machinery. MemClaw’s `Vector(VECTOR_DIM)` column, PostgreSQL JSONB merge, partial unique indexes, and `ON CONFLICT` bulk-write idempotency solve different storage constraints (`common/models/memory.py:24`, `core-storage-api/src/core_storage_api/services/postgres_service.py:344`, `core-storage-api/src/core_storage_api/services/postgres_service.py:347`, `core-storage-api/src/core_storage_api/services/postgres_service.py:531`).

Do not port MemClaw’s lack of durable stored-vector provenance. That is confirmed absence, not a teaching to copy.

Do not port hint-based re-embedding. MemClaw’s actual write path says it was removed and raw content vectors are retained (`core-api/src/core_api/services/memory_service.py:2094`).
