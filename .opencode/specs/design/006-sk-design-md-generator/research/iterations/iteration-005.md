# Iteration 005 — Track B (deepseek)

## Focus
OKLCH clustering + L1-L4 stability gating + variant/component classification accuracy

## Findings
1. **[P0] B3 — deltaE<3 under-clusters brand colors, corrupts stability signals** — CONFIRMED with mechanism. The deltaE threshold of 3 in ×100-scaled OKLCH (lines 348–354, 708) is equivalent to an unscaled perceptual difference <0.03 — far tighter than the 1–3 DE2000 range for just-noticeable difference. A brand-blue pair #3B82F6 (L≈0.66,C≈0.22,h≈260) and its lighter scale variant #60A5FA (L≈0.73,C≈0.18,h≈260) yields deltaE≈8.1 and will NOT merge. The greedy single-pass loop (lines 703–724) sorts by frequency and commits to the first match, with no iterative re-centering, so once split, a brand color stays split. Each fragment gets diluted frequency and pagesCoverage, pulling its stability score below the L1 cutoff (≥60) even though the unified token would qualify. The per-page frequency is also divided (`Math.round(c.frequency / c.pages.size)` at line 741), further degrading the classification.
   - Recommendation: Tune the threshold to deltaE < 8–12 (unscaled ~0.08–0.12) based on calibration against known brand palettes, or adopt density-based clustering (DBSCAN in OKLCH) to let cluster radius adapt to local color density. Add a post-merge step that re-flattens frequency/pages across merged members. After clustering, recompute pagesCoverage from the union of original (pre-cluster) page sets stored in the ColorEntry map, not the split cluster residual.
   - Evidence: cluster.ts:348–354 (deltaE fn), cluster.ts:700–724 (greedy loop), cluster.ts:741 (per-page frequency division)
2. **[P1] B3s — Primary-sort greed corrupts cluster representativeness** — CONFIRMED. The clustering loop (lines 701–724) sorts by frequency descending then greedily assigns each color to the first representative within deltaE<3. The first-seen high-frequency color becomes the permanent centroid for its region. If a later-seen color has higher perceptual quality (higher chroma, more typical of the cluster mean) it cannot become the representative. This means the cluster's hex/OKLCH values in the output ColorToken (line 739–753) are those of the highest-frequency member, which may not be the best exemplar. The downstream DESIGN.md displays this hex as the canonical value for that color role.
   - Recommendation: After greedy assignment, recompute each cluster representative as the frequency-weighted mean in OKLCH space, or the medoid (member with minimum average deltaE to all others in the cluster). Store the per-cluster member list to enable this post-hoc selection.
   - Evidence: cluster.ts:701 (frequency sort), cluster.ts:706–718 (first-match merge), cluster.ts:737–753 (cluster → ColorToken mapping)
3. **[P2] B4 — Hue circular-distance formula is correct; modulo is harmless dead code** — REFUTED. The expression `Math.abs(color.h - other.h) % 360` followed by `if (hueDist > 180) hueDist = 360 - hueDist` at lines 781–783 correctly computes circular distance for all h values in [0,360]. Example: h=350 vs h=10 → Math.abs(340)%360=340 → 340>180 → 360−340=20 ✓. The `% 360` is redundant since Math.abs output ≤360, but harmless. The main deltaE function (line 351–352) uses `Math.sin(dhRad/2)` which is also circularly correct for any angle. No failure case exists for the stated inputs.
   - Recommendation: No fix needed for correctness. Optionally remove the dead `% 360` and use the standard single-expression form `(((a.h - b.h) % 360) + 360) % 360` for clarity and to handle potential out-of-range inputs from future color-space changes.
   - Evidence: cluster.ts:781–783 (hue distance in light-scale grouping), cluster.ts:351–352 (hue in deltaE via sin)
4. **[P0] B10 — Frequency underweighted; L4/one-off colors can score into L2** — CONFIRMED with mechanism. The score ceiling per signal is: pagesCoverage +30, text/border +25, multi-dim +15, cssVar +20, frequency +20, achromatic +10 = 120 max. Frequency contributes at most ±20 (lines 419–421), while pagesCoverage + cssVar + usage-type alone can contribute 75 points. A one-off content-derived color (e.g., a hero gradient stop bound to --hero-accent) that: (a) appears on ≥50% of pages because its parent section is shared → +20 pagesCoverage, (b) is mapped to a CSS variable → +20, (c) is used as text → +25, (d) has frequency ≤5 → −20. Total = 45 ≥ 30 → classified L2 (system). The SKILL.md contract (lines 275–278) requires L4 = 'excluded entirely' but this token gets included in main DESIGN.md sections as a 'system' token, enabling fabrication from a content-derived value. The frequency penalty window (−20 for ≤5, 0 for 6–49, +10 for 50–499) is too narrow relative to the other signals.
   - Recommendation: Re-weight: make frequency the dominant signal by increasing frequency bonus/penalty range to ±40 (e.g., freq≥500→+40, freq≥100→+25, freq≤3→−40). Add a 'content-signal' detector: if usedAs is bg/gradient-only AND frequency≤10 AND pagesCoverage<0.3, force L4 regardless of other signals. Gate cssVar bonus behind frequency≥20 to prevent a single-use CSS variable from hijacking the classification. Add an explicit 'no single-page-only L2' rule: if pagesCoverage<0.2, cap layer at L3.
   - Evidence: cluster.ts:400–435 (classifyColorStability), SKILL.md:275–278 (L1-L4 contract), cluster.ts:419–421 (frequency scoring)
5. **[P1] B10b — No boundary-disambiguation; L2/L3 cliff misclassifies borderline tokens** — CONFIRMED. The thresholds at lines 429–432 are hard cutoffs: score 30 → L2, score 29 → L3. The SKILL.md contract line 309 says 'ESCALATE IF a token straddles the L2/L3 boundary and the classifier cannot disambiguate.' No escalation or boundary-aware logic exists — a score of 29 produces L3 with confidence 0.69, while 30 produces L2 with confidence 0.70, an identical confidence band across a classification cliff. The confidence normalization (line 434) maps score −40..+60 to 0..1 linearly, so confidences near thresholds are always ~0.5–0.7 and never trigger a 'low confidence → escalate' path.
   - Recommendation: Add a boundary zone (score 25–35) where the token is tagged 'boundary' and its confidence is clamped to ≤0.5 with a flag that the DESIGN.md writer uses to surface the token for manual review. Implement the SKILL.md escalation contract: when confidence < 0.6 AND score is within ±5 of a threshold, emit a warning in the extraction report listing the token, its signals, and the straddled boundary.
   - Evidence: cluster.ts:429–432 (hard cutoffs), cluster.ts:434 (confidence normalization), SKILL.md:309 (escalation contract)
6. **[P1] B11 — classifyVariant default-to-Primary masks real variant diversity** — CONFIRMED with mechanism. The function at lines 1234–1257 has only 4 classification branches: Ghost (transparent bg), Destructive (hard RGB red check), Primary (dark bg + light text), Secondary (light bg + dark text + shadow/border). The final `return 'Primary'` (line 1256) is a catch-all fallthrough that captures: (a) light bg + light text without shadow/border → 'Primary' (should be Ghost-on-light or Outline), (b) medium-brightness bg (0.3 ≤ L ≤ 0.7) with any text → 'Primary', (c) dark bg + dark text → falls through past the Primary check (needs textLum>0.5) → hits default 'Primary'. This flattens 3+ real variant classes (Outline, Tertiary, Link, Subtle) into 'Primary', causing the downstream DESIGN.md to over-report Primary variants and under-report or fabricate missing ones.
   - Recommendation: Replace fallthrough with 'Unknown' or a named classification after analyzing remaining conditions. Add: (1) 'Outline' — transparent bg with colored border (detect via borderColor + borderWidth>0 + bg.a<0.1), (2) 'Tertiary' — medium bg (0.3≤L≤0.7) with contrasting text, (3) 'Link' — anchor tag without bg/border, (4) keep 'Unknown' for genuinely ambiguous cases and log them. Remove the final `return 'Primary'` default.
   - Evidence: cluster.ts:1234–1257 (classifyVariant), cluster.ts:1256 (fallthrough return 'Primary')
7. **[P0] B11b — Hard RGB Destructive heuristic is fragile and false-positives on red brand colors** — CONFIRMED with mechanism. Line 1245: `bg.r > 180 && bg.g < 100 && bg.b < 100` detects only red-spectrum backgrounds as 'Destructive'. Failure modes: (a) Non-red destructive colors (amber #EAB308 → r=234>180,g=179≥100 → NOT caught; orange #EA580C → r=234>180,g=88<100,b=12<100 → caught, but both are orange-ish), (b) Sites where red IS the primary brand color — ALL red buttons get classified as Destructive, even when they are the primary CTA. The heuristic cannot distinguish semantic intent from color alone. (c) Dark red destructive buttons (#991B1B: r=153<180 → not caught). (d) No check for text color or border — a red-bg Card component (not a button) would also get Destructive classification on its variant.
   - Recommendation: Replace the hard RGB heuristic with: (1) Check if the element's role/aria-label/textContent contains destructive keywords ('delete', 'remove', 'danger', 'destructive'), (2) Use a perceptually-tuned red/orange hue range in OKLCH (h in [20,50] with C>0.1) instead of RGB thresholds, (3) If neither semantic nor perceptual signal is clear, classify as 'Colored' variant rather than 'Destructive'. Only apply Destructive label when ≥2 signals agree.
   - Evidence: cluster.ts:1245 (hard RGB check), cluster.ts:1234–1257 (entire classifyVariant)
8. **[P1] B12 — Component geometric heuristics are site-specific and will fail for common designs** — CONFIRMED with mechanism. Badge heuristic (line 1191): `radiusPx>=100` requires fully-pill border-radius — misses badges with radius 8–24px (standard pill badge on smaller elements), `height<30` misses 32–40px status badges, `hasBg` misses border-only badge styles. Card heuristic (lines 1204–1214): `width 200–800` excludes sidebar cards (<200px) and full-width dashboard cards (>800px), `padding>=12` excludes compact card designs (padding 8–10px common in dense UIs), `childrenCount>=2` is the most fragile — a simple card `<div class='card'><p>Text</p></div>` has childrenCount=1 and will be skipped entirely, missing the most common card pattern. The `radiusPx>0` gate excludes sharp-corner card designs (common in brutalist/material-flat systems). These thresholds are tuned for a specific design density and break across different information-architecture choices.
   - Recommendation: Relax thresholds and add multiple detection paths: Badge — use `<30 OR (el.rect.width < 60 && hasBg && (textContent length < 15))` to catch small labeled elements. Card — lower width floor to 120, remove upper width bound, lower padding floor to 6, reduce childrenCount to ≥1, and add a text-content-length heuristic (≥20 chars) as an alternative to childrenCount. Add a confidence field to ComponentGroup indicating how many signals matched, and surface low-confidence components for manual review. For Badge, also check for prominent border-radius-to-height ratio (radius > height*0.3) as an alternative to the absolute 100px threshold.
   - Evidence: cluster.ts:1191 (Badge: radiusPx>=100 && height<30), cluster.ts:1204–1214 (Card: width 200-800, padding>=12, childrenCount>=2)
9. **[P2] B12b — Double-defined classifyShadow function masks the exported version** — CONFIRMED. `classifyShadow` is defined twice: an exported version at line 374 and a shadowing local function at line 1047 inside `clusterTokens`. The local version is identical in logic but shadows the exported one within the function scope. No downstream caller inside `clusterTokens` can reach the exported version, making the export dead code for the single-extraction path. The merge path (`mergeTokenSets` at line 1648) does NOT call either classifyShadow — it merges pre-classified ShadowTokens by exact value match only. This is low-severity but indicates code that evolved without cleanup.
   - Recommendation: Remove the local duplicate at line 1047 and call the exported `classifyShadow` (line 374) directly. Verify the exported version handles the same edge cases (it does — the logic is identical). This also ensures the merge path can re-classify shadows if needed.
   - Evidence: cluster.ts:374 (exported classifyShadow), cluster.ts:1047 (local shadowing classifyShadow)

## Summary
The clustering pipeline has two P0 issues: deltaE<3 is ~30× too tight for perceptual color grouping (splitting brand colors, diluting frequency/coverage that drive stability), and frequency is underweighted in stability scoring (a one-off CSS-variable-bound text color on 50%+ pages reaches L2 despite frequency≤5). Two P1 issues: classifyVariant defaults unknown combinations to 'Primary' masking real variant diversity, and the hard RGB Destructive check fails for non-red destructive colors and false-positives on red-brand sites. Component geometric heuristics are tuned for a single site density and will miss cards with childrenCount=1 (the most common pattern). B4 (hue circular distance) is refuted — the formula is correct.
