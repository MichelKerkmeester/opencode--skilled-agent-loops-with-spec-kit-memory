# Design Motion Presentation Contract

Presentation source of truth for `/design:motion`. The router owns mode selection and asset loading. Workflow YAML owns execution. This file owns visible prompts, result displays, and next-step wording.

## 1. Startup Presentation

For `:confirm`, or no suffix with incomplete `$ARGUMENTS`, the consolidated setup prompt is the first visible response. Ask all applicable questions once, then wait.

For `:auto`, or no suffix with complete `$ARGUMENTS`, do not show the consolidated prompt. Resolve setup through the Auto Resolution Table below.

### Auto Resolution Table

| Field | Required | Default | Targeted-ask when |
|-------|----------|---------|--------------------|
| component_state | Yes | none | Always, when missing |
| library | No | none | Never - defaults silently |
| register | No | auto (declared field -> task cue -> surface in focus) | Register is genuinely mixed or unresolved |

### Consolidated Prompt Template

```text
Design Motion setup

Q1. Which component or state transition should the motion describe, and which library?
Q2. Is this Brand or Product register?

Reply format: "<component-state>, <library or ->, <brand|product>"
```

Never split these questions into separate visible prompts.

### Auto Fail-Fast Display

```text
Design Motion auto setup failed
Missing required input: component or state transition
Provide the missing input or rerun with :confirm.
STATUS=FAIL ERROR="missing required setup inputs"
```

## 2. Results Display

### Success

```text
STATUS=OK PRODUCES="Motion Design Spec" NEXT=/design:interface,/design:foundations,/design:audit PROOF=componentState,motionPurpose,timingModel,reducedMotionPath
```

### Missing Input

```text
Which component or state transition should the motion describe, and which library?
STATUS=ASK MISSING=<input>
```

### Failure

```text
STATUS=FAIL ERROR=<named-cause>
```

### Defer

```text
STATUS=DEFER ROUTE=<hub|sibling>
```

## 3. Next-Step Suggestions

| Condition | Suggested Command | Reason |
|-----------|--------------------|--------|
| Motion choices change or depend on the overall interface direction | `/design:interface` | Align motion with direction |
| Motion constraints affect static layout, token, or responsive decisions | `/design:foundations` | Reconcile with the static system |
| The motion spec is ready for performance, accessibility, or quality review | `/design:audit` | Validate quality before build |
| Accepted motion behavior moves to implementation | sk-code handoff via `.opencode/skills/sk-design/shared/sk-code-handoff.md` | Hand off to build |

This command never silently chains; it emits options only (`HANDOFF_REQUIRED=false`, recommend-only).

---

## 4. Example Usage

```
/design:motion modal-open-close --library framer-motion
```

Returns: a motion plan with purpose, timing, easing, interaction states, performance notes, and reduced-motion behavior.
