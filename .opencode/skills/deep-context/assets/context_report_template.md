# Deep Context: Context Report Template

The schema for the deliverable emitted at convergence. Ships verified **pointers**, not source bodies — open the cited `file:symbol` to read code just-in-time.

---

## 1. OVERVIEW

### Purpose

Defines the section order and field schemas for the `context/context-report.md` (and `context-report.json`) written by the host at the end of a converged `deep-context` run. The REUSE catalog section appears first because it is the highest-value section — it tells the planner and implementer what existing code they can extend rather than rewrite.

### Usage

The host synthesizes this template at convergence:
1. Copy this template into `{spec_folder}/context/context-report.md`.
2. Replace each placeholder row with real entries from `context/findings-registry.json`.
3. Every `file:symbol` reference MUST be code-graph-verified before the report is finalized; label anything that could not be verified.
4. The corresponding `context-report.json` carries the same findings in machine-readable form for downstream tooling.

Consumers: `/speckit:plan` loads `context-report.md` in place of its ad-hoc exploration dispatch; `/speckit:implement` uses the REUSE catalog and Touch List to determine which symbols to extend.

## 2. REPORT HEADER

```markdown
# Context Report: [FEATURE / SCOPE]

> Generated by `deep-context`. Ships verified **pointers**, not source bodies —
> open the cited `file:symbol` to read code just-in-time. Sections are ordered
> by value-per-token. Each finding carries confidence (by cross-executor
> agreement) and freshness (code-graph-verified vs unverified).

**Scope**: [what was gathered]
**Executors**: [pool, e.g. native×2 + mimo + gpt + deepseek]
**Iterations**: [N]
**Stop**: [converged | cap | blocked — and blocking gate if applicable]
```

## 3. REUSE CATALOG (SECTION 1 — HIGHEST VALUE)

Existing functions/utilities/patterns to extend — *check here before writing new code*.

```markdown
## 1. REUSE Catalog (highest value)

| id | symbol (`file:line`) | signature | reuse | confidence | agree | freshness | notes |
|----|----------------------|-----------|-------|------------|-------|-----------|-------|
| reuse-001 | `fqSymbol` (`path/to/file.ts:42`) | `funcName(arg: Type): ReturnType` | extend\|compose\|wrap\|import | 0.0–1.0 | k/N | verified\|unverified | limitations or usage notes |
```

**Field guide**:

| Field | Meaning |
|-------|---------|
| `id` | Stable short id (`reuse-NNN`); referenced in Touch List and integration points |
| `symbol (file:line)` | Fully-qualified symbol with file path and line number (code-graph-verified) |
| `signature` | Function/class signature — just enough to know how to call it |
| `reuse` | One of: `extend` (subclass/override), `compose` (call it), `wrap` (adapter), `import` (use as-is) |
| `confidence` | `maxRelevance` from the findings registry (0.0 – 1.0) |
| `agree` | `k/N` = how many of N executors found this entry |
| `freshness` | `verified` (code-graph-confirmed) or `unverified` (needs manual check) |
| `notes` | Limitations, constraints, or usage hints |

## 4. INTEGRATION POINTS (SECTION 2)

Where new code must connect. Mark hard-required vs optional.

```markdown
## 2. Integration Points

- **[HARD]** `entry/exit symbol` (`path/to/file.ts:line`) — why a miss breaks the contract.
- **[soft]** `hook/registry` (`path/to/file.ts:line`) — optional connection; failure degrades gracefully.
```

---

## 5. TOUCH LIST (SECTION 3)

Files to change, in implementation order.

```markdown
## 3. Touch List

1. `path/to/file.ts` — why this file changes and what to change.
2. `path/to/other.ts` — dependency on item 1; extend existing function.
```

---

## 6. CONVENTIONS (SECTION 4)

Codebase idioms to follow, each cited by `file:line` exemplar (not prose).

```markdown
## 4. Conventions

- **Naming**: `path/to/file.ts:12` — pattern observed.
- **Error handling**: `path/to/file.ts:88` — error boundary exemplar.
- **Logging**: `path/to/logger.ts:5` — logging wrapper used throughout.
- **Test layout**: `path/to/__tests__/file.test.ts:1` — co-located tests, vitest runner.
```

---

## 7. PRUNED DEPENDENCY SUBGRAPH (SECTION 5)

Only edges within the touch radius; the full graph is noise.

```markdown
## 5. Pruned Dependency Subgraph

- `A → imports → B` — why this edge is in scope.
- `B → imports → C` — hotspot: 3 callers; blast-radius risk.
```

---

## 8. PRIOR ART / DECISIONS (SECTION 6)

Related specs/ADRs/memory, date-stamped — prevents re-debating settled choices.

```markdown
## 6. Prior Art / Decisions

- **[2025-05-01]** Decision to use X pattern — see `.opencode/specs/...`.
- **[2025-04-15]** ADR: Y approach rejected because ... — see `.opencode/specs/...`.
```

---

## 9. GAPS AND UNKNOWNS (SECTION 7)

What could not be found or verified, and what to verify before deciding.

```markdown
## 7. Gaps & Unknowns

- `symbol X` — could not verify in the code graph; check manually before using.
- `path/to/file.ts` — marginal relevance near-miss; may be relevant if scope expands.
- **Verify**: confirm the initialization order of `Y` before wiring `Z`.
```
