## Findings
1. **[P1] checkProseClaims validator — banned interpretive phrases** — SCOPED — adopt with two caveats. Caveat 1: banned-word lists are inherently brittle (e.g., 'robust' can describe a validation pipeline legitimately; 'clean' can describe a separation of concerns). The numeric-anchor suppression window (±50 chars) mitigates but doesn't eliminate false positives — MUST be WARNING-tier only, never hard-failure, with weight ≤1pt. Caveat 2: the inline `<!-- source: tokens.x -->` sentinel mechanism IS implementable without burdening the writer — stripHtmlComments() already exists at validate.ts:51-53, HTML-comment parsing is a ~30 LOC addition to validate.ts — but it should be a separate structural check, not bundled with the banned-word heuristic. The claims.json sidecar was correctly rejected by iteration 013 (C8) as a desync risk.
   - Recommendation: Adopt source sentinels as a structural provenance check (new checkSourceSentinels in validate.ts). Adopt banned-word list as a low-weight warning check with numeric-anchor suppression. Keep them separate checks with different severity tiers.
   - Evidence: validate.ts:51-53 (stripHtmlComments already handles HTML comment stripping); validate.ts:318-350 (validateDesignMd extension point); iteration-001.md:14 (source sentinel design); iteration-004.md:10-12 (banned-word check design with numeric-anchor suppression); iteration-013.md:51-56 (inline sentinels preferred over sidecar)
2. **[P0] focus-consistent hard check** — REAL — the root-cause line still exists at a11y-extract.ts:101-103: `if (focusStyles.length === 0) { return { style: {}, consistent: true }; }`. This returns consistent:true when interaction capture was never run (it's OFF by default per extract.ts:81 `noInteraction = true`). The fix is clean and low-false-positive: add a `captured: boolean` field on A11yTokens.focusIndicator (types.ts:271-274), change the zero-length branch to return `{ style: null, consistent: null, captured: false }`, and add a gate in validate.ts that flags any §9 Focus Indicators prose beyond 'Not observed' when captured===false. No legitimate case produces zero focusStyles with captured:true.
   - Recommendation: Implement three changes: (1) add `captured:boolean` to focusIndicator type, (2) return captured:false when zero focusStyles, (3) add checkA11yFocusProvidence to validate.ts that fails any focus prose when captured===false. Also flip interaction capture default from OFF to ON (extract.ts:81) since §9 is a required section.
   - Evidence: a11y-extract.ts:101-103 (returns consistent:true on empty array); extract.ts:81 (noInteraction=true default); extract.ts:191 (interaction capture skipped by default); validate.ts:318-350 (6 checks, none inspecting focus prose); iteration-001.md:7-8 (original bug discovery); iteration-009.md:85-93 (captured sentinel proposal)
3. **[P1] named-principle must bind to a token / make philosophy conditional-on-evidence** — SCOPED — removing mandatory philosophy entirely WOULD lose genuine value. When a named principle is grounded in actual tokens (e.g., 'Shadow-as-border' with `0px 0px 0px 1px rgba(0,0,0,0.08)` appearing 4x more than elevation shadows), it communicates the WHY behind the data and is the single most useful part of a design doc for an implementing developer. The adversarial concern is valid: iteration 021's proposal to quarantine ALL narrative to §0-§1 only would throw out genuinely valuable, evidence-backed principles along with the fabricated ones. The balance is: make named principles CONDITIONAL ON EVIDENCE — require ≥3 distinct backing tokens across ≥2 pages before a principle may be named. When evidence is thin, stamp ABSENT. This preserves the high-value principles while eliminating fabrication pressure. Evidence from iteration 006 (C3): the format spec's unconditional 'Named principle — state the depth philosophy upfront' clause IS the fabrication driver, not the principle concept itself.
   - Recommendation: In design_md_format.md §9 line 544: change 'Named principle — state the depth philosophy upfront using one of these categories (or a custom name)' to 'Named principle — REQUIRED ONLY when ≥3 shadow/gradient/depth-related tokens exist across ≥2 crawled pages. When absent, stamp: **Depth philosophy: ABSENT — insufficient depth tokens extracted.**' Apply same evidence-gate to §1 design principles and §3 named strategies. Do NOT abolish principles entirely — they are the highest-value prose when evidence-backed.
   - Evidence: design_md_format.md:534-575 (§9 unconditional named principle requirement); design_md_format.md:122-128 (§1 unconditional 2-4 named principles); iteration-006.md:7-8 (C3: mandatory philosophy solicits fabrication); iteration-003.md:13-15 (thin-evidence omission from Superposition/CSS Stats); iteration-021.md:10-11 (narrative quarantine proposal — overbroad)
4. **[P0] adopt DTCG $type/$value schema** — SCOPED — HIGH-disruption migration with MEDIUM anti-fabrication return. The core prose fabrication problem (inventing 'gradient-as-depth' philosophy, 'consistent focus indicators', vibe paragraphs) is NOT caused by token type absence — it's caused by the WRITE phase being instructed to produce interpretive prose with no evidence quotas. DTCG $type/$value creates structural provenance (a shadow claim must reference a $type:'shadow' token) which enables type-gated validation, but it does NOT prevent the AI from writing 'The gradient creates atmospheric depth' — that's a prose control issue solved by evidence quotas + deterministic table generation, not token typing. The REAL value of DTCG adoption is downstream interoperability (Style Dictionary, Tokens Studio consumers) and cleaner validator architecture. The cost is substantial: ~200 LOC in cluster.ts for DTCG wrapping, new interfaces in types.ts, validate.ts refactoring to type-gated checks, and a breaking change to tokens.json format. Migration should be phased: (1) evidence quotas + deterministic tables first (higher anti-fabrication ROI), (2) DTCG wrapping as a parallel output (tokens.dtcg.json alongside tokens.json), (3) full migration once the format stabilizes.
   - Recommendation: Phase 1: add DTCG output as a PARALLEL file (tokens.dtcg.json) — zero breaking changes to existing pipeline. Phase 2: add type-gated validation checks in validate.ts that read the DTCG file. Phase 3: migrate tokens.json to DTCG natively once the ecosystem adopts it. Do NOT block anti-fabrication fixes on DTCG migration.
   - Evidence: cluster.ts:569-843 (current flat token assembly — would need ~200 LOC wrapper); types.ts:409-475 (current types need DTCG interfaces); validate.ts:57-93 (checkPhantomColors — hex-only, no type awareness); iteration-007.md:11-18 (DTCG provenance anchoring proposal); iteration-011.md:7-9 (DTCG adoption with explicit what-breaks analysis); iteration-021.md:10-11 (deterministic tables as higher-value than DTCG for fabrication)

Iteration 018 audited the 4 gold-standard examples: Vercel outputs 10 sections, Stripe 9, Linear 9, Supabase ~9. The format spec (design_md_format.md §1 table, lines 27-48) marks 15 sections as required (0-13 + 6.5), but validate.ts:244-260 enforces all 15. The sections that are frequently absent from actual outputs are: §0 Brand Context, §7 Content & Voice, §11 State Matrix, §12 Iconography, and often §9 Accessibility Contract. These sections have zero or near-zero backing token data in single-page extractions — the validator REQUIRES their headers (checkSectionCompleteness at line 241-285 is a pure string-match for `## N. SectionName`), forcing the AI to fabricate content. The spec should honestly reflect what the pipeline CAN produce: core 9-10 sections (1-6, 6.5, 8, 10, 13) as REQUIRED, and demote 0, 7, 9, 11, 12 to CONDITIONAL (only when backed by ≥N tokens). This reduces fabrication pressure by removing 'must produce' obligations for data-poor sections.
   - Recommendation: In design_md_format.md §1 table (lines 27-48): demote §0, §7, §11, §12 from 'Required: yes' to 'Required: conditional (data-driven)'. In validate.ts checkSectionCompleteness (line 241): make section requirements data-driven — §2.5 requires darkMode.supported===true, §6.5 requires motion data present, §12 requires iconSystem.detected===true. This eliminates the header-enforced fabrication pressure. The nominal section count changes from '17' to '10 core + 5 conditional' — an honest statement of capability.
   - Evidence: design_md_format.md:27-48 (section table showing all as 'yes' required); validate.ts:241-285 (checkSectionCompleteness — header-only, no token data inspection); iteration-018.md:22-24 (section count inflation — spec demands 17, examples produce 9-10); iteration-008.md:7-9 (Section 0-1-6-7 as primary fabrication vectors); iteration-013.md:57-74 (C12: checkSectionCompleteness is header-only, not data-driven)
6. **[P1] coverage-based token election over OKLCH+L1-L4** — SCOPED — additive, not replacement. Coverage-based election (page-level prevalence as primary gate) would NOT regress current behavior — it would CORRECT current misclassifications. The current classifyColorStability (cluster.ts:400-435) uses a 5-signal additive score where pageCoverage contributes only ±30 of a max ~120. A one-off hero gradient on 1 of 8 pages (12.5% coverage) but with high element-frequency (500+), a CSS variable, and multi-dimensional usage can score ≥30 and reach L2 (system). Coverage election as a HARD PRE-GATE (token on <30% of pages → capped at L3 regardless of other signals) would correctly demote these one-off campaign tokens. However, coverage election does NOT replace the OKLCH+L1-L4 scoring — it precedes it. The OKLCH clustering (deltaE<3 for perceptual merging) and the 5-signal stability scoring (pagesCoverage, usage dimensions, CSS var, frequency, chromaticity) still provide value within each coverage band: they distinguish infrastructure vs system vs campaign WITHIN the set of multi-page tokens. The coverage gate prevents single-page tokens from leaking into the stability classifier at all, solving the root-cause misclassification while preserving the nuanced scoring for multi-page tokens.
   - Recommendation: Implement as an ADDITIVE pre-gate in cluster.ts:569 BEFORE the stability pass at line 1640. Rule: if totalPages ≥ 2 AND pagesCoverage < 0.3 → cap stability layer at L3. Also fix single-page coverage inflation: when totalPages === 1, compute coverage as frequencyPercentile instead of pagesCoverage=1.0. Do NOT replace the existing OKLCH+L1-L4 scoring — it serves a different purpose (perceptual clustering + nuanced stability grading within a coverage band).
   - Evidence: cluster.ts:400-435 (classifyColorStability — pageCoverage ±30 of max ~120); cluster.ts:738 (pagesCoverage = c.pages.size / totalPages — single-page extraction always 1.0); cluster.ts:539-563 (classifyTokenStability entry point at line 1640); iteration-003.md:7-8 (coverage election from design-extract); iteration-005.md:16-17 (B10: frequency underweighted, L4 leakage to L2 confirmed with mechanism); iteration-007.md:7-9 (coverage gate as pre-filter before stability pass)

## Questions Answered
- Does the focus-consistent bug root-cause line still exist? YES — a11y-extract.ts:101-103 returns consistent:true on empty interactions, interaction capture is OFF by default (extract.ts:81), and validate.ts never checks the Accessibility Contract prose against tokens.
- Would removing mandatory philosophy lose genuine value? YES — evidence-backed named principles (e.g., 'shadow-as-border' with real shadow frequency data) are the highest-value prose in a design doc. The fix is evidence-gating, not removal.
- Does DTCG adoption directly fix prose fabrication? NO — token typing addresses structural provenance, but prose claims like 'gradient creates atmospheric depth' are a prompt/writer discipline issue, not a type system issue. DTCG has high interop value but should not block anti-fabrication fixes.
- Are 5 required sections never written in practice? YES — validated across 4 gold-standard examples (Vercel 10 sections, Stripe 9, Linear 9, Supabase ~9). The validator enforces 15 section headers but ~5 have zero backing token data in typical extractions.
- Would coverage-based election regress current behavior? NO — it would CORRECT misclassifications where single-page tokens leak into L2 via high element-frequency overriding low page-coverage. It should be additive (pre-gate), not replacement.
