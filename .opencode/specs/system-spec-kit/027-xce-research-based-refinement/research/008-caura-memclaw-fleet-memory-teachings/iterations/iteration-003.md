## Mechanism
MemClaw’s governance is built for fleet deployment, not a personal local store. The public contract states the product is “multi-tenant, multi-agent” and “built for fleets, not single agents,” with scoped memory, cross-agent propagation, and fleet-wide trust tiers as core architecture (`README.md:30`, `README.md:36`).

Scope model: every memory row has `tenant_id`, optional `fleet_id`, `agent_id`, and `visibility`; visibility defaults to `scope_team` (`common/models/memory.py:19`, `common/models/memory.py:20`, `common/models/memory.py:21`, `common/models/memory.py:61`, `common/models/memory.py:64`). The agent-facing skill states the write-time meanings: `scope_agent` private, `scope_team` default fleet, `scope_org` all fleets in tenant (`static/skills/memclaw/SKILL.md:67`, `static/skills/memclaw/SKILL.md:69`, `static/skills/memclaw/SKILL.md:70`).

Tenant and cross-tenant enforcement: auth stores the current tenant and optional readable tenants in context vars, then sets Postgres GUCs `app.tenant_id` and `app.readable_tenant_ids` for RLS-aware deployments (`core-api/src/core_api/db/session.py:29`, `core-api/src/core_api/db/session.py:33`, `core-api/src/core_api/db/session.py:52`, `core-api/src/core_api/db/session.py:70`, `core-api/src/core_api/db/session.py:74`, `core-api/src/core_api/db/session.py:79`, `core-api/src/core_api/db/session.py:84`). MCP auth similarly resolves `X-Tenant-ID`, verified `X-Agent-ID`, readable tenants, and capabilities into request context (`core-api/src/core_api/mcp_server.py:145`, `core-api/src/core_api/mcp_server.py:165`, `core-api/src/core_api/mcp_server.py:189`, `core-api/src/core_api/mcp_server.py:195`, `core-api/src/core_api/mcp_server.py:206`, `core-api/src/core_api/mcp_server.py:213`).

Read filtering is default-deny for private agent memories: list/search returns `scope_org` and `scope_team`, plus `scope_agent` only when the caller owns it; otherwise `scope_agent` is excluded (`core-api/src/core_api/repositories/memory_repository.py:114`, `core-api/src/core_api/repositories/memory_repository.py:117`, `core-api/src/core_api/repositories/memory_repository.py:136`, `core-api/src/core_api/repositories/memory_repository.py:140`, `core-api/src/core_api/repositories/memory_repository.py:143`, `core-api/src/core_api/repositories/memory_repository.py:149`). ## Teachings for Spec Kit Memory
1. **Claim** · ADAPT trust tiers into provenance trust, not access-control trust. **Evidence** · MemClaw trust levels gate fleet powers (`docs/integration-without-plugin.md:167`, `docs/integration-without-plugin.md:171`) and enforcement lives in fleet read/write/delete gates (`core-api/src/core_api/services/agent_service.py:139`, `core-api/src/core_api/services/agent_service.py:166`, `core-api/src/core_api/services/agent_service.py:263`). **Maps-to** · new sub-packet: `provenance-trust-weighting-for-derived-memory`. **Verdict** · ADAPT. **Risk** · Over-importing authz semantics would add fake multi-agent complexity to a single-user store. **Confidence** · High. **Why it transfers (or not)** · The transferable concept is “not all memories deserve equal authority.” Spec Kit already has importance tiers and constitutional always-surface memories; the missing analog is trust weighting for source/provenance, e.g. human-authored > validated automated > shadow-derived > stale/low-confidence.

2. **Claim** · ADAPT append-only audit for automated memory mutations. **Evidence** · MemClaw’s settings path writes a flat diff audit row after updates (`core-api/src/core_api/services/organization_settings.py:801`, `core-api/src/core_api/services/organization_settings.py:824`, `core-api/src/core_api/services/organization_settings.py:846`), and generic audit events include tenant, agent, action, resource, and detail (`core-api/src/core_api/services/audit_service.py:50`, `core-api/src/core_api/services/audit_service.py:56`). **Maps-to** · new sub-packet: `automated-mutation-audit-trail`. **Verdict** · ADAPT. **Risk** · A full fleet audit queue is unnecessary and can become operational noise. **Confidence** · High. **Why it transfers (or not)** · Single-user Spec Kit does not need forensic tenant accounting, but it does need explainability for self-maintaining index changes, shadow feedback promotions, tier changes, bulk deletes, retention sweeps, and causal-edge backfills.

3. **Claim** · REJECT agent/fleet/cross-tenant scope machinery; keep only a local default-deny analog for sensitive auto-derived state. **Evidence** · MemClaw scopes are explicitly fleet concepts (`static/skills/memclaw/SKILL.md:69`, `static/skills/memclaw/SKILL.md:70`, `static/skills/memclaw/SKILL.md:72`, `static/skills/memclaw/SKILL.md:73`), and query filters prevent `scope_agent` leaks across callers (`core-api/src/core_api/repositories/memory_repository.py:136`, `core-api/src/core_api/repositories/memory_repository.py:143`, `core-api/src/core_api/repositories/memory_repository.py:149`). **Maps-to** · 027 child or new sub-packet: `local-visibility-for-ephemeral-derived-state`. **Verdict** · REJECT fleet mechanism / ADAPT concept. **Risk** · If copied directly, Spec Kit would gain meaningless “agents/fleets” and confuse retrieval policy. **Confidence** · High. **Why it transfers (or not)** · There is no cross-team leak risk in a local single-user database. The only useful analog is default-deny surfacing for draft, shadow, quarantined, or low-confidence derived memories until promoted.

4. **Claim** · DEFER PII quarantine; adopt only lightweight sensitive-content tagging if there is a concrete local need. **Evidence** · README claims PII quarantine before crossing fleet boundaries (`README.md:253`), while visible code in this pass shows PII detection metadata and counts (`core-api/src/core_api/services/memory_service.py:630`, `core-api/src/core_api/services/memory_service.py:632`, `core-api/src/core_api/repositories/memory_repository.py:927`). **Maps-to** · new sub-packet if pursued: `local-sensitive-memory-tagging`. **Verdict** · DEFER. **Risk** · PII quarantine can become false-positive friction in a personal/local store; it solves cross-fleet exposure, which Spec Kit lacks. **Confidence** · Medium. **Why it transfers (or not)** · The concept of marking sensitive generated memories may transfer for operator awareness or export hygiene, but quarantine-as-is is a fleet boundary mechanism.

5. **Claim** · REJECT suppression as org lifecycle enforcement; ADAPT “suppression” only as a local disabled/archived memory state. **Evidence** · Suppression mirrors enterprise soft-delete events into tenant rows (`common/events/org_suppression_event.py:3`, `common/events/org_suppression_event.py:7`) and blocks authenticated requests for suppressed tenants (`core-api/src/core_api/auth.py:267`, `core-api/src/core_api/auth.py:282`, `core-api/src/core_api/auth.py:285`). **Maps-to** · existing Spec Kit retention/archive surfaces or new sub-packet: `memory-suppression-state-for-retrieval`. **Verdict** · REJECT fleet mechanism / ADAPT concept. **Risk** · Importing tenant suppression would create dead configuration and fake security posture. **Confidence** · High. **Why it transfers (or not)** · A single-user local store needs no org soft-delete boundary. It may benefit from explicit suppression flags for “do not retrieve this memory” without deleting evidence.

## Negative knowledge
Multi-tenant RLS context, `tenant_id` GUCs, and readable-tenant GUCs do not transfer directly; they exist to isolate tenants and widen reads only for authorized cross-tenant credentials (`core-api/src/core_api/db/session.py:29`, `core-api/src/core_api/db/session.py:52`, `core-api/src/core_api/db/session.py:70`, `core-api/src/core_api/db/session.py:84`).

Agent-scoped credentials, gateway-injected `X-Agent-ID`, `X-Readable-Tenant-IDs`, `X-Capabilities`, and gateway shared-secret perimeter checks are fleet ingress machinery, not local-memory needs (`core-api/src/core_api/mcp_server.py:189`, `core-api/src/core_api/mcp_server.py:195`, `core-api/src/core_api/mcp_server.py:210`; `core-api/src/core_api/auth.py:385`, `core-api/src/core_api/auth.py:395`, `core-api/src/core_api/auth.py:401`).

Fleet read/write ladders should not be ported as authorization. Trust >=2 for cross-fleet read and trust >=3 for cross-fleet write/delete are meaningful only when multiple fleets and agents share a tenant (`core-api/src/core_api/services/agent_service.py:139`, `core-api/src/core_api/services/agent_service.py:166`, `core-api/src/core_api/services/agent_service.py:263`).

Per-tenant concurrency bulkheads and rate limiting are noisy-neighbor controls. Spec Kit’s local SQLite/vector store has different bottlenecks and no tenant starvation problem (`core-api/src/core_api/middleware/per_tenant_concurrency.py:1`, `core-api/src/core_api/middleware/per_tenant_concurrency.py:4`, `core-api/src/core_api/middleware/per_tenant_concurrency.py:150`; `core-api/src/core_api/middleware/rate_limit.py:8`).

Org suppression eventing, tenant-suppression mirrors, Pub/Sub redelivery, DLQ behavior, and fail-open suppression cache are enterprise lifecycle controls. They do not map to a single-user local memory system except as a simple retrieval-suppressed state (`common/events/suppression_handlers.py:9`, `common/events/suppression_handlers.py:101`; `core-api/src/core_api/suppression.py:24`, `core-api/src/core_api/suppression.py:87`).

Cross-tenant read audit should not transfer as-is. Spec Kit can audit automated changes, but it does not need “who read from my tenant” forensic rows (`core-api/src/core_api/services/audit_service.py:81`, `core-api/src/core_api/services/audit_service.py:84`, `core-api/src/core_api/services/audit_service.py:99`).

PII quarantine before cross-fleet exposure is mostly negative knowledge for Spec Kit. If local export/sync features later appear, sensitive tagging may become relevant; until then, quarantine is solving a fleet leak boundary (`README.md:253`; `core-api/src/core_api/services/memory_service.py:630`).
