# Iteration 18 — gpt-5.6-sol ultra — focus: verification

Headline finding: the missing component is a promotion-grade evidence chain. Lane C, manual playbooks, and LUNA currently measure different contracts, and their outputs cannot prove that the same compiled-serving build passed every layer.

Verified claims:

- Lane C is not a manual-playbook executor. The playbook template requires exact commands, expected signals, evidence, pass/fail criteria, and triage (`.opencode/skills/sk-doc/create-manual-testing-playbook/assets/manual-testing-playbook-snippet-template.md:80-120`), but the frozen loader sets `passCriteria: null` and `critical: false` (`.opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/load-playbook-scenarios.cjs:560-595`). The runner merely dispatches and scores the loaded prompt (`.opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/run-skill-benchmark.cjs:194-220`).

- The `sk-doc` playbook contract contradicts its template. The skill permits only `PASS`, `FAIL`, or blocked `SKIP` (`.opencode/skills/sk-doc/create-manual-testing-playbook/SKILL.md:269-279`), while the root template introduces per-scenario `PARTIAL` and allows release readiness when noncritical scenarios remain partial (`.opencode/skills/sk-doc/create-manual-testing-playbook/assets/manual-testing-playbook-template.md:248-269`).

- Exact LUNA HIGH provenance is not report-verifiable. The live executor passes `DEFAULT_VARIANT` into the dispatch but records only `chosenModel` in successful and failed observations (`.opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/live-executor.cjs:358-380`). A prior LUNA comparison therefore recorded model, effort, and tier out-of-band because report JSON did not carry the subject configuration (`.opencode/specs/system-deep-loop/034-skill-benchmark-codex-executor/transport-comparison.md:1-5`).

- Existing LUNA acceptance is weaker than the requested gate. The activation packet declares real-model verification complete at 40/42 because its two non-passes were transport timeouts (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/010-live-activation/spec.md:108-113`); the archived `sk-prompt` verdict records LUNA at only `1/2` (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/010-live-activation/real-model/sk-prompt/verdict.json:23-29`, `:64-67`).

- Previous LUNA evidence is fitted-only. The prior analysis explicitly says no evaluated suite had holdouts (`.opencode/specs/system-deep-loop/034-skill-benchmark-codex-executor/tier2-luna-routing-analysis.md:3`), and the sk-code report confirms `holdoutScore: null` and `holdoutCount: 0` (`.opencode/specs/system-deep-loop/034-skill-benchmark-codex-executor/artifacts/tier2-sk-code-luna-opencode.report.json:13-20`).

- Durable storage is documented but not enforced. The storage contract says every run gets an immutable sibling label (`.opencode/skills/sk-doc/create-benchmark/references/skill-benchmark/skill-benchmark-storage-guide.md:36-40`), while the runner writes directly over the caller-selected JSON and Markdown paths (`.opencode/skills/system-deep-loop/deep-improvement/scripts/skill-benchmark/run-skill-benchmark.cjs:295-298`).

- The raw log proves a post-flip sweep ran (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/010-live-activation/real-model/run-log.txt:87-88`, `:173`), but the sibling runtime summary still says it was not rerun (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/011-runtime-engine/implementation-summary.md:111`). The evidence is recoverable, but completion metadata is stale.

Recommended acceptance pyramid:

| Layer | Required P4 acceptance |
|---|---|
| Automated | Before P4, prove unset/`0` legacy and `1` compiled; on the P4 candidate, prove unset/`1` compiled and explicit `0` legacy. Exercise fresh, missing, stale, malformed, resolver-failure, eligible-hub, and legacy-by-construction cases through the runtime front door, advisor, native bridge, hook, and CLI fallback. Every injected divergence must turn the intended assertion red. The planned state matrix is at `012-default-on-decision/checklist.md:76-95`; mutation/falsifier verification is required by `.opencode/skills/sk-code/code-opencode/references/workflow-verify.md:125-127`. Run `verify_alignment_drift.py`, plus semantic tests for resolver location and eligibility equality because its current scope is structural only (`.opencode/skills/sk-code/code-opencode/references/shared/alignment-verification-automation.md:37-59`). |
| Benchmark parity | For every newly authored compiled-router typed-gold row on every eligible hub, require both legacy and compiled observations to pass the same gold and compare exactly on normalized routing fields. Require zero mismatches, breakages, parse failures, or unexpected exclusions; deliberate stale fixtures alone may produce the expected degraded state. Preserve all frozen hashes. These are the intended dual-parity rules (`014-benchmark-alignment/spec.md:132-157`) and negative-control matrix (`014-benchmark-alignment/plan.md:157-164`). |
| Manual playbook | Execute the actual command sequence and inspect the expected signals; a Lane C run does not satisfy this layer. Every cutover scenario must be critical and `PASS`; `PARTIAL`, `SKIP`, missing evidence, or missing serving-status receipts block P4. Cover default-unset compiled serving, explicit-`0` kill switch, stale-manifest fallback, resolver breakage, and correct per-hub status. |
| LUNA live | For each eligible hub, run one authored playbook prompt and one gold-bearing held-out paraphrase, twice each, using the exact GPT-5.6 LUNA model and `high` variant. Record executor, model, variant, CLI version, flag state, serving status, observed skill/leaf call, and compiled decision. All four runs must pass; timeout, missing telemetry, or parse failure is incomplete—not “zero wrong-hub.” After fleet cutover, run an explicit-`0` LUNA control proving legacy fallback. |
| Promotion join | Gate each hub’s P4 promotion on one immutable evidence envelope containing an evidence-run ID, git SHA, built-runtime digest, activation-manifest digest, scenario-corpus digest, flag/default semantics, and every layer’s receipt. All receipts must identify the same build and configuration. Stop on the first failed layer and exercise `=0`, matching the staged rollback contract (`012-default-on-decision/plan.md:176-179`). |

Ranked actions:

1. Make a machine-readable `cutover-evidence.json` the sole P4 promotion authority; render completion prose from it.
2. Implement the permanent automated and 014 parity suites before changing defaults, including bridge/hook propagation and red-control tests.
3. Add a non-frozen manual-playbook executor or receipt collector; fix the `sk-doc` verdict taxonomy before generating router scenarios.
4. Harden LUNA into a strict routed-subject gate with exact configuration provenance, held-out gold, and no timeout forgiveness.
5. Make archive creation fail when a run label already exists, then reconcile the stale 011 completion claim against the recorded post-flip sweep.
