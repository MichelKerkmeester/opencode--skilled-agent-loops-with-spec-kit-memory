## 1. RESEARCH QUESTION

Where should the reranker integrate in CocoIndex? Compare three patterns: (1) post-CocoIndex rerank of top-K externally, (2) inline query-stage rerank inside CocoIndex, and (3) a new MCP tool wrapping search+rerank, with trade-offs in latency, complexity, and observability.

## 3. FINDINGS

1. The CocoIndex retrieval call site is `query_codebase()`, not the MCP tool wrapper. MCP `server.search()` parses parameters, optionally refreshes, then calls `client.search()`; `client.search()` serializes a `SearchRequest`; the daemon dispatches to `ProjectRegistry.search()`; `ProjectRegistry.search()` calls `query_codebase()` and converts `QueryResult` into IPC `SearchResult`. Evidence: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/server.py:108-230`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/client.py:125-179`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/daemon.py:456-516`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/daemon.py:733-800`.

2. The best technical insertion point is after candidate lookup and local heuristic scoring, before final result return; this matches the phase spec and the standard retrieve-then-rerank architecture. `query_codebase()` already embeds the query, fetches `fetch_k = unique_k * 4` candidates, performs sqlite-vec/full-scan lookup, then runs `_dedup_and_rank_rows()` and logs a stage currently named `"rerank"`. Evidence: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:298-355`; phase scope says “after embedder top-K retrieval, before result return” at `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-cocoindex-stack/007-reranker-integration/spec.md:57-62`; Sentence Transformers says retrieve a large candidate list first, then CrossEncoder scores candidates: https://github.com/huggingface/sentence-transformers/blob/main/examples/sentence_transformer/applications/retrieve_rerank/README.md.

3. Pattern 1, post-CocoIndex external rerank, is the lowest-complexity evaluation path but the weakest product integration. It can overfetch with existing `limit`, rerank outside the daemon, and avoid protocol/schema changes; however, callers must remember to request K > final N, CLI/MCP behavior diverges, and internal stage timing/correlation only covers the first-stage search. Evidence: current MCP API exposes only `query`, `limit`, `offset`, `refresh_index`, `languages`, and `paths` at `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/server.py:125-165`; request schema has no rerank controls at `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/protocol.py:21-29`; observability already uses request-scoped stage logs at `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/observability.py:63-79`.

4. Pattern 2, inline query-stage rerank inside CocoIndex, gives the cleanest steady-state architecture if implemented in `query_codebase()` as an optional second-stage scorer over a bounded candidate set. It preserves one public `search` behavior for MCP and CLI, lets one request ID capture embedding/index/rerank timing, and can reuse the existing score-audit pattern from mk-spec-memory: preserve vector/heuristic score separately, write reranker score, and return sorted results. Evidence: score-audit contract in `.opencode/skills/system-spec-kit/mcp_server/lib/search/pipeline/stage2-fusion.ts:42-47`; Stage 3 is “the only stage that may change scores after Stage 2” in `.opencode/skills/system-spec-kit/mcp_server/lib/search/pipeline/stage3-rerank.ts:11-24`; `applyCrossEncoderReranking()` preserves original rows, stores `stage2Score`, sets `rerankerScore`, syncs score aliases, and gracefully falls back on errors at `.opencode/skills/system-spec-kit/mcp_server/lib/search/pipeline/stage3-rerank.ts:347-502`.

5. Pattern 2 should be “inline query-stage op,” not an indexing-flow op, because reranking depends on the live query text and candidate set. The current CocoIndex query path is manual sqlite-vec lookup plus Python ranking, while the index flow only stores chunks/embeddings; forcing cross-encoder work into indexing would not have the query/document pair needed by CrossEncoder reranking. Evidence: `QueryResult` stores per-query score fields at `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/schema.py:24-36`; `query_codebase()` generates the query embedding at request time and reads candidates at `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:298-329`; CrossEncoder literature describes scoring query-document pairs after first-stage retrieval, not at indexing time: https://cs.uwaterloo.ca/~jimmylin/publications/Pradeep_etal_ECIR2022.pdf.

6. Pattern 3, a new MCP tool wrapping search+rerank, is useful for shadow rollout but likely creates API drift if promoted as the only implementation. A separate `search_reranked` tool could keep default latency stable and expose opt-in comparisons, but it duplicates the existing `search` request path and would not help CLI `ccc search` unless a second CLI path is added. Evidence: current tool contract is already a single semantic `search` entry point at `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/server.py:108-165`; CLI `search` calls `_search_with_wait_spinner()` then prints the same `SearchResponse` shape at `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/cli.py:375-409`; Cohere frames rerank as sorting results from an existing search solution, which supports a wrapper for experiments but not necessarily a separate long-term product surface: https://docs.cohere.com/docs/rerank.

7. Latency/observability constraints favor a feature-gated inline implementation with bounded K and graceful fallback. The phase target is rerank adds <500ms p95; CocoIndex MCP already has a request timeout budget defaulting to 10s; mk-spec-memory’s local reranker is opt-in, RAM-gated, model-file-gated, capped at 50 candidates, and returns unchanged ordering on unavailable preconditions or failures. Evidence: phase latency target at `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-cocoindex-stack/007-reranker-integration/spec.md:57-62`; request timeout defaults/clamps at `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/observability.py:15-55`; local reranker gates and graceful behavior at `.opencode/skills/system-spec-kit/mcp_server/lib/search/local-reranker.ts:4-6`, `.opencode/skills/system-spec-kit/mcp_server/lib/search/local-reranker.ts:206-240`.

## 4. RECOMMENDATIONS

1. Recommend Pattern 2 for production: add an optional inline query-stage reranker inside `query_codebase()` after first-stage candidate retrieval/dedup and before final response construction. Treat the existing `log_stage(... stage="rerank")` as the natural observability anchor, but rename/split it in implementation planning into `heuristic_rank` and `cross_encoder_rerank` to avoid hiding neural reranker cost.

2. Use Pattern 1 only for fixture evaluation and model/K sweeps. It is ideal for validating lift on the 18-pair fixture without touching daemon protocol, but it should not become the final integration because it makes overfetch/final-limit semantics caller-owned.

3. Use Pattern 3 only as a temporary shadow or opt-in MCP surface if operator safety requires it. If created, it should call the same internal reranker function planned for Pattern 2 so the wrapper does not fork ranking behavior.

4. Implementation-shape recommendation for a future plan: introduce `rerank_enabled`, `rerank_k`, and maybe `rerank_model` config/request plumbing; fetch/dedup at `rerank_k` (bounded, likely 10/20/50 sweep), run cross-encoder on `{query, content}` pairs, preserve `raw_score` and add a reranker signal/score audit field, sort by reranker score, then apply `offset/limit` for the final response.

5. GAP: no authoritative local benchmark was found in this iteration for Apple MPS latency at K=10/20/50 for the candidate reranker models. The next iteration should measure K-vs-latency and top-3 hit-rate lift on the 18-pair fixture before default-on recommendations.

6. GAP: current CocoIndex `SearchResult` protocol exposes `score`, `raw_score`, `path_class`, and `rankingSignals`, but not `stage2Score` or `rerankerScore`; implementation planning must decide whether to extend the IPC/MCP schema or encode reranker provenance only in `rankingSignals`. Evidence: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/protocol.py:96-115`.
