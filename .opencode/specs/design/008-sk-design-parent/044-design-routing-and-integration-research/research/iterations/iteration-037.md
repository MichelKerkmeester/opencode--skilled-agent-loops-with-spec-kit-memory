## F3 - The corpus chains through explicit follow-up recommendations, not silent auto-execution

The command files commonly end by naming a possible next command, but they do not silently execute it. `/design-screen` outputs a complete screen spec and suggests `/responsive-audit` as follow-up. Current `/design:*` wrappers have the negative half of this pattern: if a request spans more than the pinned mode, defer to the hub instead of forcing the mode. [SOURCE: .opencode/commands/design/interface.md:13] [SOURCE: .opencode/commands/design/interface.md:15] They lack the positive half: a parseable `NEXT_OPTIONS` or `HANDOFF` field after a successful mode/task run.

Buildable recommendation: encode chaining as explicit handoff metadata, not execution. Add `nextCommands` and `handoffRequires` to command metadata and generated wrappers. The returned status block should include `PRODUCES`, `NEXT_OPTIONS`, and `HANDOFF_REQUIRED=false|true`, while preserving `STATUS=OK|FAIL|ASK|DEFER`. Auto-chaining should remain forbidden unless the user explicitly requested a multi-stage workflow.

Enforceability: ENFORCEABLE for wrapper schema and command-fixture replay. Advisory for whether the user should actually take the suggested next step.

# Questions Answered

- Q1/D2: The corpus supports adding a task-command layer beyond the five pinned mode wrappers. Command grammar should be task-specific, concrete, and output-bound.
- Q2/D3: Utilization can be made provable by requiring command recipes to declare expected skill/mode bundles and by checking transcript proof fields against the `context_loading_contract.md` fields.
- Q5/all: The enforceable core is metadata schema, generated-wrapper drift checks, router/command fixture replay, context-manifest proof fields, and no-auto-chain status grammar. Aesthetic quality and open-ended user phrasing remain advisory.
