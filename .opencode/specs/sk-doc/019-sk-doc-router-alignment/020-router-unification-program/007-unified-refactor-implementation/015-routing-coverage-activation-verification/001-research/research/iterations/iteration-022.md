# Iteration 22 — gpt-5.6-sol ultra — focus: integration

The integration plan is cyclic as written. P2 must produce parity evidence before P3 can start (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/plan.md:168`, `:252`), but the Lane C alignment requires P3 eligibility before its parity harness can start (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/014-benchmark-alignment/spec.md:168`). Pull the pure, read-only eligibility and manifest-discovery API into P0; leave production-consumer cutover in P3.

Verified claims:

- P4 currently requires sibling alignment packets merely to be “available” (`012-default-on-decision/plan.md:253`), while 013 and 014 explicitly remain `Planned` and `Not started` (`013-create-skill-alignment/implementation-summary.md:22`, `:25`; `014-benchmark-alignment/implementation-summary.md:22`, `:25`). Packet existence is therefore not a sufficient P4 gate.
- Create-skill cannot be implemented early: it waits for the minter, freshness predicate, canonical manifest location, and data-driven discovery (`013-create-skill-alignment/implementation-summary.md:54`, `:92-94`). Its proper position is after P3 interfaces stabilize but before P4.
- The authority must be centralized before downstream work. Create-skill explicitly forbids a local allowlist or alternate freshness algorithm (`013-create-skill-alignment/spec.md:127`) and requires data-driven runtime discovery (`013-create-skill-alignment/spec.md:162-163`).
- sk-doc ownership requires separate phases. Feature catalogs have distinct root and per-feature scaffolds (`.opencode/skills/sk-doc/create-feature-catalog/SKILL.md:440-441`), while create-benchmark owns Lane C storage/index authoring but not renderer-owned reports (`.opencode/skills/sk-doc/create-benchmark/SKILL.md:454`, `:478-489`).
- Durable evidence is a promotion gate, not closeout paperwork. Every Lane C run must use a fresh sibling run-label folder, keep `baseline/` frozen, and synchronize the index (`.opencode/skills/sk-doc/create-benchmark/SKILL.md:485-494`); P2 currently says only to “record” canary evidence (`012-default-on-decision/plan.md:167-168`).
- New unnamed gap: the 015 research JSONL rows omit canonical `type`, `newInfoRatio`, `status`, and novelty-justification fields (`015-routing-coverage-activation-verification/001-research/research/deep-research-state.jsonl:18`). The loop contract requires those fields (`.opencode/skills/system-deep-loop/deep-research/SKILL.md:269-275`, `:342`), so reducer-driven convergence cannot safely treat this lineage as canonical.

Ranked actionable recommendations:

1. Make 015 a phased implementation parent with this dependency spine:

   | Child | Scope | Exit gate |
   |---|---|---|
   | `001-research` | Existing research lineage | Research synthesized; JSONL contract reconciled |
   | `002-runtime-authority-observability` | Stable resolver, default/kill-switch semantics, env propagation, hook/cache handling, serving snapshot, pure eligibility/status API | P0 observable and rollback-proven |
   | `003-sk-code-authority-drift` | `code-opencode` routing authority, shared discovery, duplicated-map removal, manifest re-minting, P1 drift CI | One authority; re-mint-required CI green |
   | `004-sk-doc-authoring-contracts` | Fix the four mode templates and validators | Template fixtures and validators green |
   | `005-catalog-playbook-coverage` | Seven-hub catalogs and serving-authority playbook matrix | Complete inventories and compiled-execution evidence fields |
   | `006-benchmark-parity-archive` | Implement 014, typed parity, immutable run archives and indexes | Deterministic parity matrix, scorer hashes unchanged, archived report |
   | `007-canary-luna-acceptance` | One-profile flag-on run with GPT-5.6 LUNA HIGH through real hook/advisor paths | All-LUNA pass, serving snapshot archived, `=0` rollback proven |
   | `008-eligibility-onboarding` | P3 consumer cutover plus 013 minter/discovery/create-skill implementation | Both consumers agree; generated ready hub verified |
   | `009-staged-default-on` | Sequential seven-hub P4 cutover | Stop-on-first-failure gates, per-hub rollback, final default-on and fleet kill-switch proof |

   Children 003 and 004 may run in parallel after 002. Everything else joins in the order shown.

2. Replace “sibling alignment phases available” with machine-checkable completion: implemented status, named tests green, immutable archive present, serving snapshot valid, scorer hashes unchanged, and rollback receipt present. The existing execution rules already say later stages require evidenced exit criteria (`012-default-on-decision/plan.md:392-396`).

3. Repair the P2/P3 cycle by exposing the pure predicate in 002. Lane C and the canary consume it read-only; 008 later replaces the production hardcoded consumers.

4. Normalize future research-state appends before 015 synthesis or dependency automation consumes them. Preserve existing rows and add a reducer/migration adapter rather than rewriting research history.
