## Analysis: opus-concurrency-race — IPC client cap 8→64

### Consistency of the cap (CLEAN)
The cap is consistent everywhere I checked:
- Both real module copies declare `DEFAULT_MAX_SECONDARY_CLIENTS = 64` and `diff` reports them **IDENTICAL** (`shared/ipc/socket-server.ts:21` vs `system-code-graph/mcp_server/lib/ipc/socket-server.ts:21`).
- `system-spec-kit/mcp_server/lib/ipc` and `system-skill-advisor/mcp_server/lib/ipc` are thin re-export shims of the shared module, so they cannot drift.
- All nine daemon env blocks pin `SPECKIT_MAX_SECONDARY_CLIENTS="64"` (3 daemons × 3 runtimes): `opencode.json:28,60,81`, `.claude/mcp.json:18,49,69`, `.codex/config.toml:68,97,113`. All read the same env name (`socket-server.ts:66`). No inconsistency.

The cap-enforcement is also race-free: the `size >= maxClients` check (`socket-server.ts:303`) and the `activeSockets.add` (`:310`) run in the same synchronous tick of the connection handler, before any `await`, so there is no TOCTOU window. Exactly 64 concurrent clients; the 65th is refused. Good.

### Back-pressure / "busy" refusal path (RESIDUAL DEFECT)
Tracing the refusal path past the socket server reveals a coupling that the cap raise mitigates by *threshold* but does not fix *structurally*:

1. **Refusal is silent close, not a protocol signal.** At cap, the server does `socket.end(); socket.destroy()` with **no JSON-RPC reply** (`socket-server.ts:303-307`). The module comment (`:18-20`) acknowledges this is "indistinguishable from a dead daemon."

2. **The liveness probe reads that close as death.** `probeDaemon` resolves `'dead'`/`closed-before-reply` when the socket closes before a JSON-RPC reply (`launcher-ipc-bridge.cjs:227-229`). The N-retry mitigation built for *busy* daemons (`:42-49`, `:348-357`) does **not** cover cap exhaustion — every retry reconnects into the *still-full* cap and is refused again, so all attempts return `'dead'`.

3. **`'dead'` triggers a reap of the live owner.** `maybeBridgeLeaseHolder` returns `action:'respawn'` (`:421-424`) → `respawnAfterDeadSocket` → `reapOwnerBeforeRespawn(ownerPid)`. That gate only consults `processLiveness` (pid exists?) — a still-running, **heartbeating** owner is *not* `'dead'`, so it falls through to **SIGTERM then SIGKILL** the live daemon (`mk-spec-memory-launcher.cjs:754-771`). The pre-reap guards check only that the lease *pid is unchanged* (`:807-818`), never the lease's `lastHeartbeatIso`/`ttlMs` freshness — even though `classifyOwnerLease:433-444` already computes exactly that staleness signal for the acquire path.

Net: when the 64-slot cap is exhausted, a healthy spec-memory daemon serving 64 clients can be SIGKILLed by an arriving launcher, aborting in-flight SQLite/FTS work for all of them. The cap raise pushes this out of the "realistic fleet" range but the kill remains *possible* because the reap decision trusts the socket probe over the robust heartbeat signal it already has.

4. **Probes share the cap, amplifying #1-#3 below 64 real sessions.** Every accepted connection — including liveness probes — is added to `activeSockets` and spins up a full `createServer()` (`socket-server.ts:310,328`), holding a slot from connect until the probe destroys the socket after the reply. A synchronized launcher burst (a large parallel fan-out all probing at once) can therefore transiently saturate the 64-slot cap with *probe* traffic alone — i.e., trigger the false-reap with fewer than 64 long-lived sessions. The documented mitigation ("cap must exceed realistic session fleet", `:18-20`) reasons only about durable bridge sessions, not probe bursts.

These are hardening findings on an already-documented, threshold-mitigated tradeoff (not a live-broken behavior), so I rate them P2 — but the structural fix is cheap: consult lease heartbeat freshness in `reapOwnerBeforeRespawn`, and/or exempt liveness probes from the client cap. That removes the kill-a-live-daemon possibility regardless of cap value.
