## New Features

#### Async file I/O with closure-scoped lazy-mkdir guard in `ccc-feedback.ts`

The user-facing feedback handler used to call synchronous `mkdirSync` plus `appendFileSync` on the hot path, which blocked the event loop on every feedback record. The handler now uses awaited `mkdir` plus `appendFile` from `fs/promises`, gated by a closure-scoped `dirReady = false` flag at line 11 that ensures the directory check runs exactly once per process. A safe-stringify fallback wraps the `JSON.stringify` call so a non-serializable feedback value never throws inside the persistence path. There is no `SKILL_ADVISOR_DEBUG`-style enable gate here because `ccc-feedback.ts` is user-facing feedback persistence that must always run â€” the user is explicitly asking the system to record something and the silence-by-default pattern would lose data.

&nbsp;

#### Writes remain AWAITED to preserve the MCP error-return contract

The naive port of H-3 would have made the file writes fire-and-forget â€” kick off the write, return success immediately. The council's section 10.1 review flagged this as a contract violation: lines 64 through 72 of `ccc-feedback.ts` return an MCP error response when the write fails, and that error path only exists if the handler awaits the write. The implementation that landed therefore keeps the awaited contract intact. The behavioral payoff of H-3 in this handler is the closure-scoped lazy-mkdir guard, not write asynchrony â€” `mkdir` runs once per process instead of once per record, and the directory existence check no longer touches the disk on the hot path.

&nbsp;

#### 4-tier configuration precedence in `mk-code-graph.js`

The plugin used to read configuration only from `rawOptions` passed at construction. The plugin now loads configuration through `loadConfig()` with the same 4-tier precedence introduced in `mk-skill-advisor.js` v0.3.0: `rawOptions` wins first, then a JSON config file at `~/.config/opencode/plugin/mk-code-graph.json`, then `MK_CODE_GRAPH_*` environment-variable fallbacks, then hardcoded defaults. The two plugins share the precedence pattern by design so operators have a single mental model when overriding either one.

&nbsp;

#### Awaited config promise at module init

Same shape as `mk-skill-advisor.js` v0.3.0: the 4-tier config is loaded as a `configPromise` at module init, and the plugin factory awaits that promise before option normalization. The council's section 10.2 review flagged the race-prone synchronous path on both plugins; both releases ship the awaited fix together.

## Backward Compatibility

This release is backward-compatible for every existing user. Operators who passed configuration via `rawOptions` only continue to work â€” `rawOptions` still wins at the top of the precedence chain. Operators who used environment variables only continue to work â€” the `MK_CODE_GRAPH_*` variables are additive. Operators who used neither continue to work â€” the hardcoded defaults are unchanged from v1.0.3.0. The feedback JSONL file format on disk is unchanged; existing files are appended through the new awaited code path without schema changes. The MCP error-return shape is unchanged; callers that handle the existing error envelope continue to work. Source commits: 65e6f0479, 45c5a7118.

## Upgrade

No migration required. Pick up the new release and the plugin will discover any pre-existing `~/.config/opencode/plugin/mk-code-graph.json` on first invocation. The feedback handler will start using the async write path on the next MCP feedback call.
