# Iteration 16 — gpt-5.6-sol high — focus: sk-code-align

The headline finding: the accepted direction is sound, but the proposed implementation boundaries are not. `HUB_CHILD` is both an allowlist and a runtime dispatch table, existing manifests lack enough data to replace that dispatch role, and copying only `resolve.cjs` cannot remove spec-tree coupling because its activation state, engine, loaders, and compiled artifacts remain spec-relative.

Verified claims:

- `COMPILED_ROUTING_HUBS` is a pure membership gate before advisor enrichment (`.opencode/skills/system-skill-advisor/mcp-server/handlers/advisor-recommend.ts:41-49`, `:326-334`). `HUB_CHILD`, however, maps hub IDs to rollout directories and is consumed to load hub-specific snapshot builders and router implementations (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/011-runtime-engine/lib/compiled-route.cjs:23-31`, `:35-58`). A manifest-derived eligibility predicate can replace the first structure, but not the second without a new uniform engine-location contract.

- Existing serving manifests contain only schema version, selected policy hash/generation, serving authority, and shadow state; they carry no `hubId`, engine entrypoint, artifact root, archetype, or source hashes (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/010-live-activation/activation/sk-code/manifest.json:1`). Consequently, “enumerate fresh manifests” cannot currently locate or instantiate the corresponding engine.

- Promoting only `resolve.cjs` into `.opencode/bin/lib/` would break both relative dependencies. It imports `./compiled-route.cjs` and derives activation state from `__dirname/../../010-live-activation/activation` (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/011-runtime-engine/lib/resolve.cjs:15-20`). The engine then derives its implementation root from its own spec-tree location and dynamically imports rollout-child harnesses and routers (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/011-runtime-engine/lib/compiled-route.cjs:20-22`, `:35-58`). This is the strongest previously unnamed gap: resolver promotion must cover the complete runtime dependency closure, not one file.

- Freshness is not manifest-local. The current snapshot builder rereads live `SKILL.md`, `hub-router.json`, and `mode-registry.json`, then recompiles the policy (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/006-parent-hub-rollout/001-sk-code/harness/build-artifacts.cjs:50-69`). The accepted decision requires a manifest digest versus live-config hash comparison (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/012-default-on-decision/decision-record.md:182-184`), so the shared predicate needs a standardized compiler/input-discovery interface rather than manifest existence alone.

- `verify_alignment_drift.py` currently exposes only repeatable roots and warning strictness (`.opencode/skills/sk-code/code-opencode/assets/scripts/verify_alignment_drift.py:110-123`), scans supported files individually (`:126-139`, `:457-497`), and is documented as lightweight and behavior-neutral (`.opencode/skills/sk-code/code-opencode/references/shared/alignment-verification-automation.md:32-51`). It cannot presently assert cross-file eligibility equality, runtime-copy freshness, forbidden spec-tree imports, or route parity.

- The compiled engine caches snapshots solely by hub ID with no selected-policy or source-hash invalidation (`.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/011-runtime-engine/lib/compiled-route.cjs:33-36`, `:58-61`). This is harmless while each CLI call creates a fresh process, but becomes load-bearing if promotion changes the advisor to an in-process/shared resolver.

Ranked actionable recommendations:

1. Promote a complete runtime bundle, not only the resolver: stable resolver, engine loader, activation manifests, and per-hub compiled artifacts. Use a fixed layout such as `.opencode/runtime/compiled-routing/<hub>/` with a uniform `engine.cjs` interface. The spec tree may remain provenance, but no production `require()` or activation read should point into it.

2. Split the current overloaded concept into `isCompiledEligible(hubId)` and `loadCompiledEngine(hubId)`. The first validates a fresh runtime manifest; the second uses a fixed bundle convention or a manifest-declared, root-guarded entrypoint. Delete `COMPILED_ROUTING_HUBS` after parity, but delete `HUB_CHILD` only after all seven engines satisfy the uniform loader contract.

3. Extend the runtime manifest schema with at least `hubId`, compiled bundle identity, input/source hashes, and selected-policy identity. Validate IDs against a strict hub grammar and guard resolved paths inside the runtime bundle root.

4. Keep `verify_alignment_drift.py` as the named top-level gate, but add an explicit compiled-routing contract subcheck backed by a dedicated helper. It should reject spec-tree imports from runtime code, compare runtime bundle hashes with generated source hashes, enumerate eligible manifests, and prove advisor/resolver eligibility parity. Route-gold behavioral parity should remain a separate executable test invoked by the same orchestration gate.

5. Key engine caching by `{hubId, selectedPolicy.effectivePolicyHash, sourceSignature}` or invalidate on manifest changes. Test re-minting within one long-lived process so default-on does not require a restart to serve the new generation.
