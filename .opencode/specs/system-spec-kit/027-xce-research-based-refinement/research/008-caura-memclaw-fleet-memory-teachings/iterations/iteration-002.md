## Mechanism
MemClaw’s `/search` request enters `core-api` as `SearchRequest`, which requires `tenant_id`, a non-empty `query`, optional `fleet_ids`, agent/type/status/time filters, and `top_k` capped at 1-20 (`core-api/src/core_api/schemas.py:303`, `core-api/src/core_api/schemas.py:314`). The REST route enforces readable tenant scope, applies a per-tenant search slot, derives an effective caller agent, and for low-trust agents forces fleet scoping before calling `search_memories` (`core-api/src/core_api/routes/memories.py:1408`, `core-api/src/core_api/routes/memories.py:1412`, `core-api/src/core_api/routes/memories.py:1425`, `core-api/src/core_api/routes/memories.py:1438`, `core-api/src/core_api/routes/memories.py:1456`).

The active path is a decomposed pipeline: resolve profile, extract temporal hint, classify query, embed plus entity boost, scored storage search, post-filter, serialize, optionally inject STM, then track recalls (`core-api/src/core_api/services/memory_service.py:2984`, `core-api/src/core_api/pipeline/compositions/search.py:17`, `core-api/src/core_api/pipeline/compositions/search.py:20`). Search profile resolution sets effective `top_k`, `min_similarity`, adaptive FTS weight, freshness, graph hops, and similarity blend (`core-api/src/core_api/pipeline/steps/search/resolve_search_profile.py:33`).

Candidate generation has two modes. First, `ClassifyQuery` tokenizes the query and searches the entity FTS index; if entity matches are specific enough, it graph-expands entities, loads linked memories, skips embedding and scored search, and ranks by graph-hop boost (`core-api/src/core_api/pipeline/steps/search/classify_query.py:1`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:75`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:99`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:124`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:125`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:378`). It explicitly rejects over-broad entity matches above a cap and falls through to hybrid search (`core-api/src/core_api/pipeline/steps/search/classify_query.py:82`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:90`, `core-api/src/core_api/constants.py:450`). Otherwise it routes to keyword or semantic search based on adaptive FTS weight (`core-api/src/core_api/pipeline/steps/search/classify_query.py:186`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:187`).

Hybrid candidate generation runs query embedding and entity/graph boost concurrently (`core-api/src/core_api/pipeline/steps/search/parallel_embed_entity_boost.py:1`, `core-api/src/core_api/pipeline/steps/search/parallel_embed_entity_boost.py:154`). Query embeddings use `get_query_embedding`, which supports instruction-aware asymmetric query/document embedding but falls back to symmetric `embed` for local/symmetric providers (`common/embedding/_service.py:257`, `common/embedding/_service.py:264`, `common/embedding/_service.py:290`, `common/embedding/_service.py:296`). The query embedding cache key includes model, vector dimension, instruction, tenant, and normalized query, with an in-process stampede guard for concurrent cold misses (`core-api/src/core_api/services/memory_service.py:2615`, `core-api/src/core_api/services/memory_service.py:2652`, `core-api/src/core_api/services/memory_service.py:2672`). Entity boost uses entity FTS, optional graph expansion, linked-memory collection, hop-based boosts, and a cap on boosted memories (`core-api/src/core_api/pipeline/steps/search/parallel_embed_entity_boost.py:60`, `core-api/src/core_api/pipeline/steps/search/parallel_embed_entity_boost.py:73`, `core-api/src/core_api/pipeline/steps/search/parallel_embed_entity_boost.py:92`, `core-api/src/core_api/pipeline/steps/search/parallel_embed_entity_boost.py:109`, `core-api/src/core_api/pipeline/steps/search/parallel_embed_entity_boost.py:113`). If entity boost times out after embedding succeeds, it degrades to vector-only search instead of failing the query (`core-api/src/core_api/pipeline/steps/search/parallel_embed_entity_boost.py:168`, `core-api/src/core_api/pipeline/steps/search/parallel_embed_entity_boost.py:195`).

Scoring is not RRF or cross-encoder reranking; it is a server-side CTE composite score. Storage computes pgvector cosine `vec_sim`, FTS `ts_rank_cd`, a blended similarity score, then blends similarity with memory weight and multiplies by freshness, entity boost, recall boost, temporal/date-range boosts, currency, and status penalty (`core-storage-api/src/core_storage_api/services/postgres_service.py:895`, `core-storage-api/src/core_storage_api/services/postgres_service.py:904`, `core-storage-api/src/core_storage_api/services/postgres_service.py:919`, `core-storage-api/src/core_storage_api/services/postgres_service.py:973`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1050`). Null-embedding rows remain discoverable through FTS-only fallback instead of being ranked as null/zero-vector noise (`core-storage-api/src/core_storage_api/services/postgres_service.py:908`, `core-storage-api/src/core_storage_api/services/postgres_service.py:919`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1071`). Scope filtering happens in the same scored query: tenant/readable-tenants, deleted rows, fleet visibility, caller visibility, agent/type/status, and valid-time filters (`core-storage-api/src/core_storage_api/services/postgres_service.py:1101`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1111`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1120`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1129`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1142`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1157`). Results are ordered by score and recency (`core-storage-api/src/core_storage_api/services/postgres_service.py:1188`).

Result shaping is intentionally bounded. MemClaw overfetches `top_k * SEARCH_OVERFETCH_FACTOR` from storage, then post-filters by minimum vector similarity and trims to the final requested `top_k` (`core-api/src/core_api/pipeline/steps/search/execute_scored_search.py:52`, `core-api/src/core_api/pipeline/steps/search/execute_scored_search.py:59`, `core-api/src/core_api/pipeline/steps/search/post_filter_results.py:20`, `core-api/src/core_api/pipeline/steps/search/post_filter_results.py:37`). FTS-only rows bypass the vector threshold via `has_embedding=False`, preserving recall during pending-embedding windows (`core-api/src/core_api/pipeline/steps/search/post_filter_results.py:21`, `core-api/src/core_api/pipeline/steps/search/post_filter_results.py:30`). Serialization returns `MemoryOut` items with entity links and rounded composite score as `similarity` (`core-api/src/core_api/pipeline/steps/search/load_and_serialize.py:85`). `/recall` then wraps search with a token-bounded LLM brief; its formatter exposes only type/title/status/content fields to the summarizer (`core-api/src/core_api/services/recall_service.py:44`, `core-api/src/core_api/services/recall_service.py:51`, `core-api/src/core_api/services/recall_service.py:65`).

## Teachings for Spec Kit Memory
- **Claim** · Add a bounded query-strategy classifier before normal hybrid retrieval. **Evidence** · MemClaw short-circuits specific entity queries to entity graph lookup and skips embedding/scored search, but declines over-broad entity matches above a calibrated cap (`core-api/src/core_api/pipeline/steps/search/classify_query.py:82`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:90`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:125`, `core-api/src/core_api/pipeline/steps/search/classify_query.py:128`, `core-api/src/core_api/constants.py:450`). **Maps-to** · 027 child 007 semantic-trigger-fallback plus new sub-packet for query-strategy classification; Spec Kit already has hybrid lexical+semantic retrieval. **Verdict** · ADAPT. **Risk** · Entity short-circuit can overfit to broad topical terms and skip the stronger fused scorer. **Confidence** · High. **Why it transfers (or not)** · It transfers if “entity” means local anchors such as spec folder, file path, symbol, memory title, trigger phrase, or causal node; it should not copy fleet graph traversal wholesale.

- **Claim** · Preserve lexical-only recall during vector gaps with overfetch-before-thresholding. **Evidence** · MemClaw admits null-embedding rows only when they match FTS, scores them by FTS rather than zero-vector blend, overfetches `top_k * 2`, then trims after min-sim filtering (`core-storage-api/src/core_storage_api/services/postgres_service.py:908`, `core-storage-api/src/core_storage_api/services/postgres_service.py:919`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1071`, `core-api/src/core_api/pipeline/steps/search/execute_scored_search.py:59`, `core-api/src/core_api/pipeline/steps/search/post_filter_results.py:21`, `core-api/src/core_api/constants.py:473`). **Maps-to** · 027 child 007 semantic-trigger-fallback; Spec Kit already has separate SQLite index plus separate vector store and pending vectors. **Verdict** · ADOPT. **Risk** · Lexical-only rows can crowd out semantic rows if FTS normalization is too permissive. **Confidence** · High. **Why it transfers (or not)** · This is directly relevant to a local self-maintaining index: SQLite/BM25 rows should remain searchable while the separate vector store is lagging, failed, or reconciling.

- **Claim** · Cache query embeddings with embedder-version keys and a cold-miss stampede guard. **Evidence** · MemClaw’s query cache key includes model, vector dimension, instruction, tenant, and normalized query; concurrent cold callers await the same in-flight future (`core-api/src/core_api/services/memory_service.py:2599`, `core-api/src/core_api/services/memory_service.py:2652`, `core-api/src/core_api/services/memory_service.py:2661`, `core-api/src/core_api/services/memory_service.py:2672`, `core-api/src/core_api/services/memory_service.py:2687`). **Maps-to** · new sub-packet; Spec Kit already has local ollama/nomic vectors and embedder pointer management. **Verdict** · ADAPT. **Risk** · Stale cache if the key omits active embedder manifest, dimension, prompt/instruction, or normalization version. **Confidence** · Medium-high. **Why it transfers (or not)** · The tenant dimension does not transfer, but model/dimension/instruction/query and active embedder manifest do; it improves local latency without adopting fleet cache infrastructure.

- **Claim** · Keep retrieval diagnostics as first-class output for tuning, not just logs. **Evidence** · Diagnostic mode widens candidate capture and records all candidates, search params, retrieval strategy, and original top_k (`core-api/src/core_api/pipeline/steps/search/execute_scored_search.py:52`, `core-api/src/core_api/services/memory_service.py:3084`, `core-api/src/core_api/services/memory_service.py:3087`, `core-api/src/core_api/services/recall_service.py:205`, `core-api/src/core_api/services/recall_service.py:210`). **Maps-to** · new sub-packet; Spec Kit already has shadow-first learning-feedback reducers and rich MCP search surfaces. **Verdict** · ADAPT. **Risk** · Diagnostic payloads can become token-expensive if exposed by default. **Confidence** · Medium. **Why it transfers (or not)** · Single-user local systems benefit from transparent “why this result” traces, but should keep them opt-in/debug-profile only.

## Negative knowledge
- Do not copy MemClaw’s tenant/fleet/trust visibility machinery into Spec Kit Memory. It exists to prevent cross-tenant/fleet disclosure and agent-scope leaks (`core-api/src/core_api/routes/memories.py:1408`, `core-api/src/core_api/routes/memories.py:1438`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1101`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1129`).

- Do not copy PostgreSQL/pgvector-specific CTE implementation details as-is. MemClaw stores vectors in `pgvector` and text search in PostgreSQL `TSVECTOR` (`common/models/memory.py:4`, `common/models/memory.py:24`, `common/models/memory.py:42`); Spec Kit uses SQLite plus a separate vector store, so only the scoring shape transfers.

- Do not introduce RRF/cross-encoder reranking from MemClaw; this pass found composite scoring, not RRF or a reranker. The active scoring is explicit blend/multiplier math in storage (`core-storage-api/src/core_storage_api/services/postgres_service.py:973`, `core-storage-api/src/core_storage_api/services/postgres_service.py:1050`).

- Do not copy per-agent retrieval profiles literally. MemClaw advertises per-agent retrieval tuning for fleets (`README.md:267`, `README.md:269`), while Spec Kit is single-user/local; any equivalent should be a local profile or shadow-learned configuration, not agent governance.

- Defer recall-count boost and decay lessons to iteration 009. MemClaw multiplies scores by recent recall count (`core-storage-api/src/core_storage_api/services/postgres_service.py:954`, `core-storage-api/src/core_storage_api/services/postgres_service.py:963`), but this overlaps ranking/decay and may conflict with Spec Kit’s FSRS decay/co-activation model.
