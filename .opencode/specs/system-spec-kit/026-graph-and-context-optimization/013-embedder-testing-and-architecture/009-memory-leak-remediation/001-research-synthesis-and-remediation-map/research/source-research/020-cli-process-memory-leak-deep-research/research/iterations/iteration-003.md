# Iteration 003: Deep-review, AI council, and multi-agent command process / lock / cleanup audit

## Focus
Audit the deep-review workflow, the AI council skill + agent + persistence library, and the multi-agent commands (`/improve:agent`, `/spec_kit:complete :with-research`) for process behavior, nested dispatch risk, long-running children, stale locks, missing cleanup, and unsafe retry loops that could interact with `system-spec-kit` memory workflows. Out of scope (iters 001-002 closed): the deep-research spawn-inventory itself, the deep-research state-machine, and the `runAuditedExecutorCommand` `kind:`/`type:` mismatch (re-checked only as a cross-flow propagation).

## Findings

### A. Deep-review has NO advisory lock and NO inter-flow serialization

1. The deep-review YAML defines no `lock_file` path, no `step_acquire_lock` step, and no `on_halt`/`on_cancel`/`on_workflow_exit` release hooks. A `Grep` for `step_acquire_lock|advisory_lock|lock_file|lock_released|\.deep-review\.lock` against the asset returned zero matches, whereas the same grep against `deep_start-research-loop_auto.yaml` returns 10 hits (lock_file path, step_acquire_lock body, three release hooks, two state-event emissions, etc.). [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:83-109`]; [SOURCE: `.opencode/commands/deep/assets/deep_start-research-loop_auto.yaml:105`]; [SOURCE: `.opencode/commands/deep/assets/deep_start-research-loop_auto.yaml:179-195`]
2. Consequence: two concurrent `/deep:start-review-loop` runs against the same packet (e.g. operator triggers a second one because the first looks stuck) share the same `{artifact_dir}/deep-review-state.jsonl`, `deep-review-findings-registry.json`, and `prompts/iteration-{N}.md` paths with NO file-locking gate at the workflow boundary. Both runs will: (a) re-render the same prompt file (last writer wins), (b) `appendFileSync` interleaved JSONL bytes (the malformed-line risk iter-002 #25 documented for deep-research applies word-for-word here, but now without an advisory lock even guarding the obvious case of two operator runs), (c) overwrite each other's `deep-review-dashboard.md` via the reducer. There is no PID file, no `process.kill(pid, 0)` probe, nothing.
3. The `step_classify_session.on_invalid_state` branch halts when state artifacts are "incomplete or contradictory" with `message: "Existing deep-review state in review/ is incomplete or contradictory. Repair or archive the invalid review packet before continuing."` — but this gate fires only on detected mismatch, not on detected concurrent ownership. A second run starting before the first wrote any state still passes `on_fresh`. [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:221-223`]

### B. Deep-review inherits every deep-loop defect iter-002 identified for deep-research

4. **Marker-scan self-trigger (iter-002 #5-7 cross-confirmation):** `head -5` of `.opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl` shows the literal first line is `DEEP-REVIEW`. The deep-review YAML's `step_marker_scan` aborts with exit 2 when `head -1` of the previous iteration's rendered prompt matches `^(DEEP-REVIEW|DEEP-RESEARCH|CODE-REVIEW)$`. The shipping template matches that regex exactly, so any multi-iteration deep-review run that uses `render_prompt_pack` HALTS at the start of iteration 2 unless the orchestrator overrides the template (as this packet does). The defect is cross-flow: both deep-research and deep-review have the same false-positive HALT against their own canonical templates. [SOURCE: `.opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:1`]; [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:620-634`]
5. **`record_executor_audit` `type:` field mismatch (iter-002 #1-4 cross-confirmation):** the deep-review `record_executor_audit` step writes `executor: { type: "{config.executor.type}", model, reasoningEffort, serviceTier }` to the state log. The executor-config schema defines the field as `kind:`, not `type:`. So deep-review's persisted executor records have the same silently-stripped `executor.kind` field as deep-research, and any reducer or dashboard filtering on `executor.kind === 'cli-codex'` will undercount deep-review dispatches identically. [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:904-912`]; [SOURCE: `.opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/executor-config.ts:21-28`]
6. **Reducer-shared `missing_iteration_file` divergence (iter-002 #12-14 cross-confirmation):** deep-review's `step_reduce_review_state` documents `failure_modes.missing_iteration_file: "no-op + error"` and shells `node .opencode/skills/deep-review/scripts/reduce-state.cjs {spec_folder}`. The deep-research finding that this contract is unimplemented (`process.exit(3)` on ENOENT) applies — the reducers are sibling scripts under `.opencode/skills/deep-{research,review}/scripts/` and share the same `parseIterationFile` / `readUtf8` shape. Hard-crashing the reducer mid-loop in deep-review breaks the workflow before `step_generate_dashboard`, leaving the dashboard stale and the next `step_check_convergence` reading old findings. [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:954-971`]

### C. Deep-review CLI dispatch branches lack inter-iteration cleanup

7. **cli-devin branch leaves three files per iteration:** the `if_cli_devin` block writes (a) `{state_paths.prompt_dir}/agent-config-iter-{current_iteration}.json` (sed-substituted from `agent-config-deep-review-iter.json`), (b) `{state_paths.prompt_dir}/devin-stdout-iter-{current_iteration}.log`, (c) `{state_paths.prompt_dir}/devin-stderr-iter-{current_iteration}.log`. No `rm`, no rotation policy, no synthesis-time sweep. Default `max_iterations: 7` ⇒ up to 21 stale files per packet for the cli-devin variant. After multiple operator-triggered re-runs (each renumbering iterations from 1), older runs' files remain. [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:822-848`]
8. **cli-claude-code branch uses `--permission-mode acceptEdits` against a documented READ-ONLY review target:** `claude -p ... --permission-mode acceptEdits` auto-approves edits without prompts; the YAML override is explicit ("claude-code default permission-mode is 'plan' (read-only); we override to 'acceptEdits' for iteration writes"). But §1 of `/deep:start-review-loop` says "treat_review_target_as_read_only" is an ALWAYS rule, and the `@deep-review` agent contract is "modify_files_under_review" is NEVER. Containment depends entirely on the dispatched agent's behavioral discipline — there is no runtime sandbox restricting writes to `{artifact_dir}`. A misbehaving model or prompt-injected payload can edit the very files under review. [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:778-796`]; [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:1290-1308`]
9. **cli-opencode and cli-devin branches inherit the manual SIGKILL discipline iter-001 #10 documented for deep-research:** the per-iter shell heredocs do not detach into their own process group, so when the YAML runtime's external `timeout_seconds` fires it kills only the immediate child shell — the cli-opencode/cli-devin sub-processes (MCP daemons, sandbox helpers, embedder warmups, ccc search workers, gtimeout wrappers) keep running until the operator manually `pkill -9`s them. This is identical to iter-001 finding 10 but now confirmed for the deep-review YAML branches independently. [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:797-848`]

### D. ai-council `writeStateJsonl` truncates the "append-only" state log

10. Despite §14 of `ai-council.md` explicitly stating "ai-council-state.jsonl is append-only. NEVER rewrite or truncate it", the named export `writeStateJsonl(packetSpecFolder, content, options)` calls `writeFileScoped(aiCouncilRoot, 'ai-council-state.jsonl', content, options)`, which under the hood calls `fs.writeFileSync(targetPath, content, 'utf8')` — a TRUNCATING write. Only `appendStateLine` (line 644-651) is genuinely append-safe via `fs.appendFileSync`. [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/persist-artifacts.js:458-475`]; [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/persist-artifacts.js:522-526`]; [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/persist-artifacts.js:644-651`]; [SOURCE: `.opencode/agents/ai-council.md:676`]
11. The contract violation matters because `writeArtifacts(packetSpecFolder, rendered)` — the CLI's batch-persistence path — invokes `writeStateJsonl` FIRST in its files array (before `writeConfig`, `writeStrategyMd`, `writeDeliberation`, `writeReport`, and per-seat writes). So every CLI-driven persistence run starts by wiping any prior state log and re-laying the freshly-rendered events with `DEFAULT_TIMESTAMP = '1970-01-01T00:00:00.000Z'`. A council that ran round 1 + 2 of `appendStateLine` events, then crashed before round-3 persistence, then was re-run via `node persist-artifacts.cjs <packet> --input-file council-report.md`, loses rounds 1 + 2's audit history. [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/persist-artifacts.js:602-635`]; [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/persist-artifacts.js:29`]
12. The §13 resume contract ("Resume after interruption: read the state log and resume from the next incomplete event") in `ai-council.md` assumes append-only state. With `writeArtifacts` truncating on every CLI invocation, the resume protocol cannot tell apart "council ran rounds 1-3 then was re-persisted" from "council started fresh". The forensic decision rule "(highest round_end event).round + 1" returns wrong answers after a truncation. [SOURCE: `.opencode/agents/ai-council.md:647`]

### E. ai-council state rotation silently breaks append-only audit

13. `rotateIfNeeded(stateJsonlPath, incomingBytes, maxBytes)` in `audit-trail.js` renames the JSONL file to `<path>.1` when `currentBytes + incomingBytes > DEFAULT_MAX_BYTES = 10 MiB`. `fs.rmSync(backupPath, { force: true })` runs first — so any existing `.1` from a prior rotation is deleted before the rename. There is no `.2`, `.3`, … ladder. Two rotations on a busy packet lose all events older than the most recent rotation cycle. [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/audit-trail.js:73-81`]; [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/audit-trail.js:24`]
14. Downstream readers that parse only the active `ai-council-state.jsonl` (e.g. `parseStateLog`) silently miss rotated events. The audit-trail module exports `appendArtifactWrittenEvent`, which calls `appendJsonlEvent` ⇒ `rotateIfNeeded` on EVERY artifact_written audit emission. So a long-running council that writes many seats can silently rotate during a single session without the council code ever seeing the cutover. [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/audit-trail.js:96-103`]; [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/audit-trail.js:115-127`]; [SOURCE: `.opencode/skills/sk-ai-council/scripts/lib/persist-artifacts.js:93-99`]

### F. ai-council agent permissions vs skill SKILL.md drift

15. The runtime agent `.opencode/agents/ai-council.md` frontmatter is `permission: { ... bash: deny, ... patch: deny, ... task: allow, ... }`. The skill `.opencode/skills/sk-ai-council/SKILL.md` frontmatter is `allowed-tools: [Read, Write, Edit, Bash, Glob, Grep]`. The two surfaces disagree on whether the council may shell out: the agent denies Bash; the skill description claims Bash is allowed. Operators reading SKILL.md may design Bash-using council workflows that the runtime rejects with a permission denial at execution time. [SOURCE: `.opencode/agents/ai-council.md:6-22`]; [SOURCE: `.opencode/skills/sk-ai-council/SKILL.md:4`]
16. `permission.task: allow` plus §0 "Adaptive dispatch" (Depth 0 ⇒ Task tool parallel dispatch) means the council CAN spawn sub-agent Task dispatches when it detects itself as Depth 0. Depth detection is "If task context includes `Depth: 1` or explicit LEAF/nesting constraints, operate at Depth 1. Otherwise, operate at Depth 0." There is no runtime field that reliably carries `Depth: 1` from an orchestrator wrapper; the council infers depth from absence rather than presence. A council mistakenly dispatched at apparent Depth 0 by an orchestrator that didn't explicitly tag the context could recursively spawn Task-tool dispatches. [SOURCE: `.opencode/agents/ai-council.md:36-49`]; [SOURCE: `.opencode/agents/ai-council.md:17`]

### G. improve:agent loop has no resume / no automated archive — hard halt on stale runtime

17. `improve_deep-agent-improvement_auto.yaml.step_session_boundary_gate` enforces `lineage_mode_must_equal_new`, `existing_runtime_must_be_archived`, and `dormant_resume_fields_disallowed`. On failure: "Halt with STATUS=FAIL and instruct the operator to archive the prior improvement packet before re-running." There is NO automated archive step — the operator must manually `mv {spec_folder}/improvement {spec_folder}/improvement_archive/...` between runs. Compared to deep-review's `step_classify_session` (auto-resume / auto-restart / completed-session branches), this is a strict single-shot policy. A partial/crashed/interrupted improve run blocks ALL future improve runs on the same packet until manual cleanup. [SOURCE: `.opencode/commands/improve/assets/improve_deep-agent-improvement_auto.yaml:119-125`]
18. **Per-iteration agent dispatch:** `phase_loop.step_generate_candidate.action: "Dispatch @deep-agent-improvement to write one bounded candidate"`. The dispatch repeats every iteration of `phase_loop` (default `max_iterations: 5`). The dispatched agent has `permission.bash: allow`, `permission.external_directory: allow`, `permission.task: deny`, `permission.memory: deny`. So the dispatched agent cannot recursively dispatch — that surface is closed — but it can run arbitrary Bash and reach outside the packet root. [SOURCE: `.opencode/commands/improve/assets/improve_deep-agent-improvement_auto.yaml:154-156`]; [SOURCE: `.opencode/agents/deep-agent-improvement.md:6-19`]
19. **Per-event Node spawn fan-out:** every state transition in the improve loop fires a `node .opencode/skills/deep-agent-improvement/scripts/improvement-journal.cjs --emit <event>` subprocess. For one 5-iteration session the events are: `session_start` (1) + per-iter `candidate_generated` + `candidate_scored` + `benchmark_completed` + `legal_stop_evaluated` + optional `blocked_stop` (≈5 events × 5 iter = 25) + `session_end` (1) ≈ 27 Node spawns. Each loads `node` fresh, requires resolution, and writes to the same journal file with `fs.appendFileSync`. No timeout is documented on these journal-emit steps; if `improvement-journal.cjs` hangs (e.g. NFS file lock), the loop hangs without a typed dispatch_failure event. [SOURCE: `.opencode/commands/improve/assets/improve_deep-agent-improvement_auto.yaml:142-202`]
20. **Per-iteration ad-hoc `node -e` inline scripts:** four steps (`step_record_candidate_lineage`, `step_record_mutation_coverage`, `step_measure_benchmark_stability`, `step_detect_trade_offs`) shell out via `node -e "..."` with multi-line inline scripts that `require()` packet-helper modules and `fs.writeFileSync` JSON outputs. Each is a fresh `node` boot, each parses JSON inline. No timeouts documented. If any of these hang (e.g. broken `require` after a refactor), the loop stalls without a dispatch_failure record. [SOURCE: `.opencode/commands/improve/assets/improve_deep-agent-improvement_auto.yaml:160-186`]

### H. `/spec_kit:complete :with-research` nests deep-research; nested lock-acquire path

21. `spec_kit_complete_auto.yaml.optional_workflows.research_integration.workflow_reference: "deep_start-research-loop_auto.yaml"` with `insert_point: "after_phase_2"`. When the `:with-research` flag is set OR `smart_detect` fires above `uncertainty_threshold: 60`, the complete workflow embeds the FULL deep-research auto workflow inline. That nested workflow runs the full `step_acquire_lock`/`pre_dispatch_audit`/iteration loop/`step_release_lock` machinery from inside the complete run. [SOURCE: `.opencode/commands/spec_kit/assets/spec_kit_complete_auto.yaml:41-60`]
22. If a prior crashed deep-research run left `.deep-research.lock` held (iter-002 #17-18: macOS advisory locks survive crash; no PID liveness probe; recovery is "confirm-only or explicit recovery-only"), the embedded research workflow halts at `step_acquire_lock`. The complete workflow has no documented fallback to skip the research integration on lock contention; the complete run is blocked behind a stale advisory lock owned by a process that may no longer exist. [SOURCE: `.opencode/commands/deep/assets/deep_start-research-loop_auto.yaml:179-185`]
23. The complete workflow also has no inter-run lock of its own. Two concurrent `/spec_kit:complete` runs against the same packet may race on `spec.md`/`plan.md`/`tasks.md` writes, and on the canonical save's `generate-context.js` invocation — which mutates `description.json` and `graph-metadata.json`. The spec-memory mutation-freshness rule ("`memory_save` and `memory_bulk_delete` must invalidate the entity-density cache after commit") means concurrent writes can leave the cache pointing at stale rows.

### I. Cross-flow memory-write race: deep-review save + deep-research save + improve journal

24. **Deep-review `phase_save`:** `step_generate_context` runs `node generate-context.js /tmp/save-context-data-<session-id>.json {spec_folder}`. This routes content into `decision-record.md` / `implementation-summary.md` / `handover.md` and updates `description.json` + `graph-metadata.json` (per `CLAUDE.md` Memory Save Rule). `step_index_memory` then calls MCP `memory_save({ filePath })`, which mutates `memory_index` rows and (per the CLAUDE.md "Memory mutation freshness" rule) must invalidate the entity-density cache. [SOURCE: `.opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:1238-1247`]
25. **Race conditions with no flow-level coordination:**
    a) Deep-review and deep-research save concurrently to the same packet: both write `implementation-summary.md` / `handover.md` and both call `memory_save`. Deep-research has its own packet-local lock for state but does not lock the canonical save target. Deep-review has no packet lock at all. Last writer wins on the canonical doc; first cache invalidation is harmless but the second can race against in-flight indexer reads.
    b) The improve loop writes `agent-improvement-state.jsonl`, `improvement-journal.jsonl`, `mutation-coverage.json`, `trade-off-report.json`, and per-iter `iteration-{iteration}-repeatability.json` to `{spec_folder}/improvement/`. None of these go through `generate-context.js`, but if a concurrent deep-review/deep-research save on the SAME `{spec_folder}` regenerates `graph-metadata.json`, the improve loop's packet-local files are NOT in the canonical spec-doc set the memory router writes back. They survive the save — but the regenerated `description.json` may now under-represent the actual packet shape (e.g. status, importance_tier reset per the iter-002 cross-context). Operators relying on memory search to find the improve runtime files will see stale derived metadata until the next manual `memory_index_scan`.
    c) `audit-trail.js.rotateIfNeeded` can rotate ai-council's state log inside a single seat write, while a concurrent council-graph upsert reads the same file via the council MCP tool. No advisory lock guards the rotate-rename window. A reader hitting the file mid-`fs.renameSync` sees ENOENT or `.1`-rotated content; the council MCP would surface this as a `degraded` graph state with no specific recovery hint.
