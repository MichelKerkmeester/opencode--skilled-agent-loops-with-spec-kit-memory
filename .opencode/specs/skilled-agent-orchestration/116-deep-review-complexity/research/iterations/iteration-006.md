# Iteration 006: Scope Selection And File Targeting

## Focus

Investigate deep-review scope selection and file/resource targeting. The goal was to determine whether shallow outcomes come from reviewing too broad a dimension, too few files, weak hotspot selection, or missing code graph / semantic search requirements.

## Actions Taken

1. Read the active packet state, dashboard, strategy, state log, and iterations 001-005 before inspecting new surfaces.
2. Inspected deep-review setup and iteration YAML for context loading, scope discovery, dimension ordering, graph convergence, resource-map prompting, and prompt-pack rendering.
3. Compared the deep-review prompt pack, strategy template, and agent instructions with deep-research focus selection.
4. Inspected resource-map extraction behavior and the resource-map coverage contract.
5. Inspected graph convergence signals and tests/fixtures around scope discovery, dimension rotation, graph-aware stop, and resource-map coverage.

## Findings

1. Deep-review scope discovery is broad and enumerative, not hotspot-ranked. For a spec-folder target it reads packet docs, discovers referenced implementation files and associated tests, then sets scope to all discovered files; skills similarly include SKILL.md, references, assets, scripts, agents, and commands; arbitrary files expand globs and immediate cross-references [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-review_auto.yaml:227] [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-review_auto.yaml:235] [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-review_auto.yaml:249]. The outputs are `review_scope_files` and `cross_reference_targets`, but there is no ranked target queue, changed-file risk score, dependency centrality, bug-class hotspot list, or mandatory semantic/code-graph expansion step [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-review_auto.yaml:254].

2. The review loop selects a dimension before it selects a deep file cluster. Initialization orders dimensions as correctness, security, traceability, and maintainability, with an optional resource-map traceability pass when a map exists [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-review_auto.yaml:258]. Each loop then extracts `next_dimension` as the next uncovered dimension from the strategy queue [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-review_auto.yaml:386]. That makes the primary search axis a taxonomy bucket. File focus is present only as broad `review_scope_files` and strategy table state, not as a reducer-owned "next highest-risk file/path/invariant" queue [SOURCE: .opencode/skills/deep-review/assets/deep_review_strategy.md:134].

3. The prompt pack hands the agent the whole scope plus one dimension, which can make "few files" and "too broad a dimension" both true depending on budget. The rendered state includes `Dimension`, `Review Target`, and `Review Scope Files` [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:11]. The agent is asked to perform only 3-5 analysis actions, review one dimension, and near the ceiling write verified findings rather than expanding discovery [SOURCE: .opencode/agents/deep-review.md:155] [SOURCE: .opencode/agents/deep-review.md:158] [SOURCE: .opencode/agents/deep-review.md:159]. So the workflow can review too few files relative to the broad discovered scope, but the root cause is not a low file count by itself; it is the absence of a required risk-ranked file/flow selection method.

4. Resource-map behavior is a traceability coverage gate, not a bug-hunting target selector. Deep-review detects `{spec_folder}/resource-map.md`, stores whether it is present, and otherwise records that the coverage gate is skipped [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-review_auto.yaml:145]. When present, the contract requires at least one iteration to read the map and applied reports, cross-check `target_files`, classify touched/untouched/absent paths, and emit `resource-map-coverage` findings [SOURCE: .opencode/skills/deep-review/references/loop_protocol.md:306]. That is useful for "did implementation paths match the declared inventory?", but it does not prioritize files for correctness/security review and disappears entirely when the map is absent.

5. Graph convergence is a stop gate over emitted graph data, not a context-loading or hotspot-discovery requirement. The deep-review YAML calls `deep_loop_graph_convergence` before the inline vote [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-review_auto.yaml:418], but the handler returns `CONTINUE` with no blockers when the graph is empty [SOURCE: .opencode/skills/system-spec-kit/mcp_server/handlers/coverage-graph/convergence.ts:168]. Review graph signals measure dimension coverage, finding stability, P0 resolution, evidence density, and hotspot saturation [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/coverage-graph/coverage-graph-signals.ts:418] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/coverage-graph/coverage-graph-signals.ts:480]. Hotspots are only FILE nodes carrying `metadata.hotspot_score`, and saturation means at least two dimensions cover that file [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/coverage-graph/coverage-graph-signals.ts:499]. If iterations do not emit graph nodes, graph convergence cannot guide target selection.

6. Code graph and semantic search are available, but not mandatory discovery gates. The command allows `mcp__cocoindex_code__search`, `code_graph_query`, and `code_graph_context` [SOURCE: .opencode/commands/spec_kit/deep-review.md:4], while the auto YAML agent tool list includes CocoIndex but not the code graph tools [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-review_auto.yaml:76]. The agent says to use CocoIndex only when exposed and exact symbols are unknown, and describes code graph tools as support rather than a substitute for file:line evidence [SOURCE: .opencode/agents/deep-review.md:157] [SOURCE: .opencode/agents/deep-review.md:239]. This makes semantic/code-graph search optional, not a required precondition for reviewing unfamiliar or high-risk targets.

7. Existing tests and playbooks verify the presence of scope rules, dimension rotation, resource-map visibility, and graph blockers, but not hotspot selection quality. The scope-discovery playbook passes if all target types have resolution rules and dimensions are ordered correctly [SOURCE: .opencode/skills/deep-review/manual_testing_playbook/02--initialization-and-state-setup/007-scope-discovery-and-dimension-ordering.md:26]. The dimension-rotation playbook passes if next-dimension extraction, dispatch focus injection, and convergence dimension coverage exist [SOURCE: .opencode/skills/deep-review/manual_testing_playbook/03--iteration-execution-and-state-discipline/010-strategy-next-focus-and-dimension-rotation.md:137]. The blocked-stop fixture proves missing dimensions and active P0 block stop, but the completed iterations contain only `filesReviewed`, dimensions, counts, and convergence summaries [SOURCE: .opencode/skills/deep-review/scripts/tests/fixtures/blocked-stop-session/review/deep-review-state.jsonl:2]. I did not find a test that fails when a review omits semantic/code-graph discovery for unfamiliar code, lacks a ranked hotspot queue, or covers a broad scope with too few justified file choices.

8. Deep-research has a stronger focus-selection loop even without a dedicated file targeter. Its prompt injects focus area, remaining key questions, and last-three summaries [SOURCE: .opencode/skills/deep-research/assets/prompt_pack_iteration.md.tmpl:11]. Its YAML chooses `least_explored` from unanswered questions not covered by the last focus [SOURCE: .opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:386], and the agent chooses the narrower evidence-answerable focus when tied [SOURCE: .opencode/agents/deep-research.md:122]. That explains why bug-finding research can keep drilling into one defect hypothesis while deep-review rotates over a dimension/file scope combination.

## Questions Answered

- Does shallow output come from reviewing too broad a dimension? Yes, partly. Dimension rotation is the primary loop axis, and each dimension can cover a broad, heterogeneous scope before a file/flow hotspot is selected.
- Does it come from too few files? Not directly. The workflow can discover many files, but the per-iteration action budget can only inspect a subset. The missing piece is a required explanation for which files, flows, or invariants were chosen and which were deferred.
- Does it come from weak hotspot selection? Yes. Hotspot saturation exists as a stop signal if graph FILE nodes with `hotspot_score` exist, but no setup or dispatch step requires creating a hotspot-ranked queue before review.
- Does it come from missing code graph or semantic search requirements? Yes. The tools are allowed and documented, but use is conditional/optional; there is no mandatory code graph or semantic-search pass for unfamiliar, broad, or concept-defined targets.

## Questions Remaining

- What is the smallest enforceable schema change: `targetSelection` on the iteration record, a `candidateMatrix`, per-delta `target_candidate` records, or all three?
- Should mandatory semantic/code-graph discovery trigger only for code/spec targets above a scope-size threshold, or for every review where exact symbols are unknown?
- How should graphless environments behave: require a text hotspot ledger as a substitute, or mark graph context unavailable and block STOP for high-risk targets?
- Should resource-map evidence feed the same target-selection ledger, or remain a traceability-only coverage gate?

## Next Focus

Design the minimum target-selection contract for deep-review: fields, thresholds, validator checks, and fallback behavior when code graph or CocoIndex is unavailable. The next pass should map this to concrete implementation surfaces without changing code in this research packet.
