# Iter 4 — query.py deep-read — expansion, candidate assembly, path_class application timing

## TL;DR (2-3 sentences)

`query.py` does not implement synonym expansion, code-aware query rewriting, camelCase decomposition, or identifier splitting before dense embedding; it passes the raw query string to the configured embedder, with only an optional model-specific query prompt name. Path class can affect ranking twice: `query.py` applies a pre-rerank heuristic boost/penalty when the query looks implementation-oriented, and the reranker adapters can multiply model scores by path-class factors when the Phase 2 env flag is enabled. The only query text normalization in this path is narrow and stage-specific: lowercasing for implementation-intent detection and regex token extraction/OR quoting in the FTS5 lane.

## Findings (numbered, with citations)

1. There is no general query expansion layer in `query.py`. The raw `query` string is carried into dense embedding and FTS; `query.py` does not maintain synonym maps, identifier splitting, camelCase decomposition, snake_case decomposition, or code-aware query rewriting. The only query-derived branch in `query.py` is `_has_implementation_intent()`, which lowercases the query and checks static substrings such as `implementation`, `function`, `handler`, `class `, `method`, `callers`, and `definition of`; that branch changes scoring, not recall text. FTS has its own regex tokenizer and OR-quoted token query, but that is lexical normalization rather than semantic expansion. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:55-74`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:579-585`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:88-103`; grep evidence: `rg -n "synonym|expand|camel|case|split|normalize|lower|strip|regex|token|query" ...`.

2. Dense query embedding uses the same env-provided embedder object as indexed document chunks, with possible query-side prompt metadata. `query_codebase()` retrieves `embedder = env.get_context(EMBEDDER)` and calls `await embedder.embed(query, query_prompt_name)`, while `shared.create_embedder()` builds and stores the module-level embedder used by `CodeChunk.embedding`. That means Phase 2's bench pin to `sbert/BAAI/bge-code-v1` controlled the dense query model and the already-built document vector model assumption, while the reranker/path-class lanes were separate tests. The caveat is prompt asymmetry: `query_prompt_name` may be non-null for registered models or env override, but document chunks are only annotated with the embedder object, not an explicit document prompt in the cited code. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:579-595`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/shared.py:35-72`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/shared.py:143-202`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/run-phase2-smoke.sh:48-51`.

3. The exact `query_codebase()` path to rerank is: validate DB path, get env contexts, embed query, convert embedding to bytes, compute `fetch_k`, run hybrid or vector-only candidate lookup, then rank/dedup and optionally rerank. In hybrid mode, vector rows come from `_hybrid_vector_rows()`; FTS rows come from `query_fts(conn, query, fetch_k, ...)`; both become `RankedRow` lists and pass through `rrf_fuse()` with env-configured weights. Missing FTS-only chunk records are fetched by ID, then `_dedup_and_rank_hybrid_rows()` produces the sorted unique `candidates`. Only then does `query_codebase()` call `reranker.rerank(query, candidates, top_k=config.rerank_top_k, model_name=config.rerank_model)`, and final output is sliced by `_window_results()`. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:573-596`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:598-640`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:666-695`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:533-553`.

4. `path_class` enters pre-rerank scoring inside `query.py`, but only for implementation-intent queries. In vector-only `_ranked_result()` and hybrid `_hybrid_ranked_result()`, a query that trips `_has_implementation_intent()` gets `+0.05` for implementation chunks and `-0.05` for `spec_research` or `docs`; canonical resource paths get `+0.10` regardless of that intent branch. `_dedup_and_rank_hybrid_rows()` sorts by this score before alias dedup, so the path-class heuristic can affect which candidates enter the rerank head. This supports Iter 3's conclusion that rerank sees the post-fusion/post-heuristic head, not the raw RRF list. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:350-396`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:399-444`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:481-515`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/cocoindex-internals-deep-dive/iterations/iteration-003.md:17-21`.

5. `path_class` can also enter post-model rerank ordering, but this happens in the adapter layer and is flag-gated. The cross-encoder adapter scores `(query, candidate.content)` pairs for the head, calls `_apply_path_class_boost(scores, head)`, logs the boosted scores if configured, replaces candidate scores, and sorts the reranked head. `_apply_path_class_boost()` is a no-op unless `COCOINDEX_RERANK_PATH_CLASS_BOOST` is truthy; with the flag on, it multiplies model scores by factors from env or defaults. Jina v3 uses the same helper after `model.rerank(query, docs)`. Therefore the answer is "both" for path-class timing, but the two applications are different: additive pre-rerank heuristic in `query.py` for implementation-intent queries, and multiplicative post-model adapter scoring only when the Phase 2 boost flag is enabled. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/reranker.py:20-47`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/reranker.py:139-183`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/rerankers_jina_v3.py:140-190`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/config.py:20-30`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/016-embedder-testing-and-architecture/004-code-index-stack/011-rerank-model-fit-investigation/research/phase2-bench/run-phase2-smoke.sh:73-91`.

6. Silent normalization risks are real but narrow. Dense search gets the raw query string except for any embedder-internal behavior and possible `query_prompt_name`; `query.py` itself does not strip punctuation before embedding. Implementation-intent detection lowercases the query and uses substring checks, so phrasing like "implementation", "function", or "definition of" can silently turn on path-class/canonical scoring. The FTS lane strips anything outside `[A-Za-z0-9_./:-]+`, keeps dots/slashes/colons/hyphens as token characters, ORs the remaining quoted tokens, and relies on SQLite's `unicode61 remove_diacritics 2`; this can lose punctuation meaningful to code, fail to decompose camelCase, and broaden lexical recall via OR. Dedup also normalizes chunk content by `strip().split()` before hashing, but that affects alias collapsing rather than query text. Citations: `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:72-82`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/query.py:247-272`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:11-12`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:25-35`, `.opencode/skills/mcp-coco-index/mcp_server/cocoindex_code/fts_index.py:88-132`.
