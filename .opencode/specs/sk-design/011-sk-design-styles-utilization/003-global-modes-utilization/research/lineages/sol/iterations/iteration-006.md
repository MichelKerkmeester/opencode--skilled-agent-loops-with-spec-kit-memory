## Findings

1. **`MOTION_CORPUS_CONTEXT v1` belongs after the restraint gate and before choreography; zero corpus use is a first-class successful outcome.** The request is `{surface, register, motionBudget, targetInteraction, frequencyTier, inputModes[], purpose, stateOrScenePath, existingAnimationSystem?, reducedMotionBar, performanceConstraints[], corpusGeneration}`. Gate order remains frequency -> keyboard input -> one named purpose -> register ceiling. `gateDecision:no-motion` terminates with `queryIssued:false`, `hydratedReferences:[]`, and an instant equivalent; `gateDecision:eligible` merely allows temporal discovery. A static interface anchor may constrain visual tone, but it supplies no temporal authority. This preserves phase 001's zero-or-one motion hydration and the current workflow's rule that a gate failure is correct, not degraded. [SOURCE: .opencode/skills/sk-design/design-motion/SKILL.md:262-275] [SOURCE: .opencode/skills/sk-design/design-motion/references/animation_decision_framework.md:92-101] [SOURCE: .opencode/specs/sk-design/011-sk-design-styles-utilization/001-research-utilization/research/lineages/sol/research.md:134-145] [INFERENCE: placing the corpus adapter between gate and choreography prevents retrieval from manufacturing a reason to animate]

2. **Eligibility must be polarity-aware and evidence-typed before ranking.** `MOTION_EVIDENCE_ELIGIBILITY v1` first requires matching generation/hash, provenance, and a positively asserted temporal slice; it hard-excludes static-only similarity, incidental temporal vocabulary, explicit prohibition of the requested behavior, and source text that names no usable state/scene or pattern. Eligible records then receive a reviewable 10-point rank: `+3` explicit non-negated motion section/philosophy, `+2` target pattern or purpose match, `+2` named state/trigger/scene, `+1` material or property, `+1` measured/declared cadence, and `+1` a reduced-motion statement. The first `+3` and at least one of the two `+2` fields are mandatory; a reduced-motion mention alone never makes a record eligible. Every card labels evidence `measured` (captured occurrence/property evidence), `declared` (source rationale), or `inferred`, with source slice, prohibitions, and limitations. The score orders already-eligible evidence; it never overrides the gate or selects motion automatically. [SOURCE: .opencode/specs/sk-design/011-sk-design-styles-utilization/001-research-utilization/research/lineages/sol/iterations/iteration-008.md:20-37] [SOURCE: .opencode/skills/sk-design/design-motion/SKILL.md:53-66] [INFERENCE: mandatory positive polarity plus purpose/state composition directly addresses the holdout's BM25 negation failures]

3. **The safe negative baseline distinguishes “no motion” from “no corpus motion evidence.”** `MOTION_NEGATIVE_BASELINE v1` records `{decision:no-motion|target-native-motion, reason:frequency|keyboard|purpose-fail|register|no-eligible-evidence|source-prohibition, affectedStates, instantEquivalent, preservedFeedback, queryIssued, rejectedEvidence[], reducedMotionPath}`. Frequency, keyboard, purpose, or register failure proves the target interaction stays instant. By contrast, a static bundle or an empty eligible set proves only `no corpus temporal authority`; if the gate passed, motion may still be derived from the mode's own strategy and target evidence with `temporalOwner:target-native`. Explicit source prohibition can be retained as a counterexample but cannot prohibit target motion unless the target independently shares that interaction and reason. Reduced motion preserves state through instant swaps, opacity/color/focus changes, or static progress cues rather than deleting feedback.
