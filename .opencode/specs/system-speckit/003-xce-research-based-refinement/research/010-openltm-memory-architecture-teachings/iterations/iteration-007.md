## Mechanism
OpenLTM implements provenance as a separate per-memory chain, not as frontmatter. `memory_provenance` is one-to-many, cascades with the memory, and stores `source_type`, `source_ref`, `actor`, `created_at`, and freeform `metadata`; pre-existing rows are backfilled as `legacy` (`migrations/008_add_memory_provenance.sql:1`, `migrations/008_add_memory_provenance.sql:6`, `migrations/008_add_memory_provenance.sql:21`). The actual source taxonomy is `learn`, `git-backfill`, `evaluate-session`, `import-bundle`, `user-promotion`, `janitor-rollup`, `legacy`, so the prompt’s “session” label is not a literal enum value; session is represented indirectly via `source_ref` defaulting to `sessionId` on learn (`migrations/008_add_memory_provenance.sql:9`, `packages/openltm-core/src/db.ts:377`).

Audit is a separate append-oriented mutation log. `memory_audit.memory_id` is intentionally not a foreign key so audit rows survive memory deletion, and each row records `op`, required `actor`, optional `session_id`, `before_json`, `after_json`, and `created_at` (`migrations/009_add_memory_audit.sql:1`, `migrations/009_add_memory_audit.sql:6`). The initial op enum is `insert/update/forget/deprecate/supersede/redact/restore`, later rebuilt to add `archive` (`migrations/009_add_memory_audit.sql:9`, `migrations/014_update_audit_op_check.sql:6`). I did not verify SQL-level immutability triggers; the listed migrations create ordinary tables and indexes.

The DAO captures slim before/after snapshots that intentionally exclude embeddings, selecting memory metadata, lifecycle timestamps, supersession fields, workspace/agent IDs, and `created_by` (`packages/openltm-core/src/dao/provenanceAudit.ts:15`, `packages/openltm-core/src/dao/provenanceAudit.ts:18`). `insertProvenance` and `insertAudit` are narrow helpers, while `queryAudit` filters by memory, op, session, and since, ordered newest-first with a default limit of 50 (`packages/openltm-core/src/dao/provenanceAudit.ts:27`, `packages/openltm-core/src/dao/provenanceAudit.ts:61`).

Write paths use the audit/provenance layer selectively: duplicate reinforcement snapshots before/after and writes an `update` audit (`packages/openltm-core/src/db.ts:321`, `packages/openltm-core/src/db.ts:334`); new learns write `created_by`, a provenance row, and an `insert` audit (`packages/openltm-core/src/db.ts:352`, `packages/openltm-core/src/db.ts:377`); forget snapshots before deletion and writes a `forget` audit with `after_json=null` (`packages/openltm-core/src/db.ts:606`, `packages/openltm-core/src/db.ts:611`). Janitor archive preserves full memory JSON in `memory_archive`, writes an `archive` audit, then deletes the memory (`packages/openltm-core/src/janitor/archive.ts:45`, `packages/openltm-core/src/janitor/archive.ts:53`). Audit writes are best-effort: `tryAudit` catches and logs failures without rolling back the primary mutation (`packages/openltm-core/src/db.ts:132`).

Recall can optionally carry provenance. When `includeProvenance` is set, OpenLTM batch-fetches provenance rows for all returned IDs to avoid N+1 queries and attaches them to each memory (`packages/openltm-core/src/db.ts:563`, `packages/openltm-core/src/dao/provenanceAudit.ts:89`). The architecture narrative frames this as fixing “what, but not where it came from” and “who/what wrote this?” gaps (`docs/03-architecture.md:710`, `docs/03-architecture.md:727`). It also sketches future replay/diff use cases from the audit log (`docs/03-architecture.md:908`).

## Negative knowledge
- Do not copy OpenLTM’s taxonomy verbatim: its schema says `learn`, not `session`, and mixes mechanism (`learn`) with origin (`git-backfill`) and lifecycle actor (`janitor-rollup`).
- Do not copy fail-open audit writes for canonical or governed mutations; audit loss should be explicit where compliance/debuggability matters.
- Do not treat comments saying “append-only” as enforcement; without immutability triggers or restricted write paths, this is convention plus DAO discipline.
- Do not duplicate our existing FSRS/causal-edge lifecycle model with OpenLTM’s lighter audit trail; use provenance/audit to explain mutations, not to replace ranking or causal lineage.

## Open questions
- Should system-spec-kit use a strict enum for `provenance_source_kind`, or a small controlled vocabulary plus freeform `provenance_source` for compatibility?
- Should deletion audit rows live in `memory_history`, mutation ledger, or a new tombstone-only table?
- Should `includeTrace` expose source-kind provenance by default for canonical continuity rows, or only under debug/research profiles?
