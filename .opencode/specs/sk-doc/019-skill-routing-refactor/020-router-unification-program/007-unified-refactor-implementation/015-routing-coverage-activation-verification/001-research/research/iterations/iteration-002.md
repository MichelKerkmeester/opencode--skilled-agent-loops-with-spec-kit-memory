## Iteration 2 Report — Benchmark compiled-lane

**Headline:** The minimal additive compiled-parity lane is feasible **without touching any of the 3 frozen files**, but a naive "set the flag and diff" implementation will produce **vacuously green** results because the compiled engine has a *second* gate (activation-manifest serving authority), emits a *disjoint shape* from the scorer's `observed` contract, and resolves its resolver from a mutable spec-tree path.

### Verified data flow (the seam)
- Orchestrator `run-skill-benchmark.cjs:30-37` requires the frozen trio plus two **non-frozen** siblings: `executor-dispatch.cjs` and `build-report.cjs`.
- For every playbook scenario, `runPlaybook` (run-skill-benchmark.cjs:178) → `dispatchScenario` (executor-dispatch.cjs:124) → `routeSkillResources` (router-replay.cjs:643, frozen) → normalized `observed` shape → `evaluateRouteGold({scenario, observed})` (score-skill-benchmark.cjs:885, frozen) → `row.routeGold` (run-skill-benchmark.cjs:219).
- `evaluateRouteGold` reads **only** `observed.observedIntents` + `observed.observedResources` + `observed.routedOut` from the observed (score-skill-benchmark.cjs:886, 903-904). Any caller can re-invoke it with a synthesized observed.
- `_args.cjs:24-37` is a generic `--key value` parser; a new `--compiled-parity` flag requires **zero edits** to `_args.cjs`.

### Minimal additive lane (touches ZERO frozen files)
1. New sibling `compiled-parity.cjs` (sits next to `executor-dispatch.cjs`, same dir).
2. Orchestrator hook at `run-skill-benchmark.cjs:219` — after `row.routeGold = evaluateRouteGold(...)` for the legacy lane, call `compiledParity({scenario, legacyObserved: observed, skillRoot, skillId})` → attaches `row.compiledParity`.
3. New verdict branch in `run-skill-benchmark.cjs:308-310` — `BLOCKED-BY-COMPILED-DRIFT` (exit 3), mirroring the existing `BLOCKED-BY-ROUTE-GOLD` pattern. Edit is to the orchestrator's verdict switch, not the frozen scorer.

### Four hazards the spec must address (3 are novel)
- **F-2-2 Vacuous-parity gate (P0, novel).** `resolve.cjs:41-42` requires *both* `SPECKIT_COMPILED_ROUTING=1` AND `servingAuthority==='compiled'` in the activation manifest. Flag-on with `servingAuthority:'legacy'` → `resolveRoute` returns null → `compiled-route.cjs:39` prints `{servingAuthority:'legacy', hubId}` → a naive lane reads "compiled matches legacy" and passes. The lane MUST assert the manifest's state for the target hub before treating the output as a real compiled decision.
- **F-2-3 Shape impedance (P0, novel).** The compiled engine emits `targetQualifiedIds` (e.g. `"sk-code/code-review/code-review/workflow/review-cache"`), `decisionAction` (route/clarify/defer/reject), and `observedIntents` — verified in `.../006-parent-hub-rollout/001-sk-code/compiled/route-gold.typed.json`. The frozen scorer reads `observedResources` as surface paths (score-skill-benchmark.cjs:904, 919-927). Direct parity comparison is vocabulary-incompatible. The lane MUST translate via the leaf-resource contract library (`router-replay.cjs:191 CONTRACT_LIB_PATH` → `leaf-resource-contract.cjs`, exists at `.opencode/skills/sk-doc/create-skill/scripts/lib/`, 19774 bytes), the same bridge `buildResourceContract` (router-replay.cjs:235) already uses.
- **F-2-4 Spec-tree resolver coupling (P1, novel).** `compiled-route.cjs:16-21` resolves the resolver from `.opencode/specs/sk-doc/019-sk-doc-router-alignment/020-router-unification-program/007-unified-refactor-implementation/011-runtime-engine/lib/resolve.cjs`. Specs are mutable planning artifacts; any rename of 011-runtime-engine (sk-doc packets restructure often) silently breaks the resolver, `compiled-route.cjs:36-39` catches and returns the legacy sentinel, and the lane goes vacuously green. The lane MUST pre-flight that path's existence and surface breakage as drift, not parity.
- **F-2-5 Hub-type vs compiled-eligibility mismatch (P2, novel).** `isHubTypeSkill` (run-skill-benchmark.cjs:68) keys off `hub-router.json` (7 hubs have one). Compiled-serving eligibility is keyed off the activation manifest *plus* the `COMPILED_ROUTING_HUBS` allowlist in `advisor-recommend.ts`. A hub can have route-gold auto-on while being compiled-N/A. The lane MUST distinguish `n/a (not compiled-eligible)` from `drift`, otherwise N/A hubs get scored as either false-pass or false-fail.

### Ranked recommendations
1. **Land `compiled-parity.cjs` + the two orchestrator edits first** — this is the entire "additive lane without touching frozen files" surface and unblocks everything.
2. **Build the lane around a `compiledParity` status enum**: `{match, drift, vacuous, n/a, resolver-missing}` — only `match` is a pass; `vacuous` and `resolver-missing` are hard failures (the trap states), `drift` is a real regression, `n/a` is informational.
3. **Reuse the frozen `evaluateRouteGold` as the comparison oracle**: call it twice (legacy observed, compiled observed synthesized via the leaf contract) and assert `(legacyGold.pass === compiledGold.pass) && setEq(legacyGold.observedIntents, compiledGold.observedIntents)`. Editing it is forbidden; calling it is the intended API.
4. **Assert the resolver path + manifest state in `compiled-parity.cjs`'s pre-flight**, before any scenario loop — fail loud on spec-tree drift, never fail safe.
