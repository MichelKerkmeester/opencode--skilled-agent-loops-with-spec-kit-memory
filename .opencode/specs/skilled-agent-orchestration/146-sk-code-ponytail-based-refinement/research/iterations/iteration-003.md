## Iteration 003 — Lens: "Intensity Sliders" (seat opus-2)

### How ponytail's slider actually works (and why it's cheap)

Ponytail exposes `lite | full | ultra` via three resolution layers — env `PONYTAIL_DEFAULT_MODE` > config-file `defaultMode` > hardcoded `'full'` (`ponytail-config.js:53-73`) — plus a runtime `/ponytail` switch. But the mechanically important detail is in `filterSkillBodyForMode` (`ponytail-instructions.js:11-37`): **only the intensity-table rows and the worked example are mode-specific.** The ladder, the rules, and the entire "When NOT to be lazy" safety floor are emitted *verbatim at every level*. So the slider tunes exactly one variable — *how aggressively to simplify* — over a constant floor. It is a single-axis dial slicing one table.

That is precisely why the pattern does **not** transfer cleanly to sk-code / sk-code-review, whose "intensity" is multi-axis (checklist breadth × verification depth × gate strictness) sitting on **hard, non-lowerable floors** (the Iron Law; baseline security minimums).

### Q1 — A "verification intensity" slider for sk-code

Mapping levels onto the existing §2 Verification Commands (`sk-code/SKILL.md:188-193`):

| Level | OPENCODE | WEBFLOW |
|---|---|---|
| **lite** | `verify_alignment_drift.py --root <scope>` only — no targeted tests | `minify-webflow.mjs` + `verify-minification.mjs` only — no runtime test, no browser evidence |
| **full** (today's Iron Law) | alignment verifier **+** targeted tests (vitest/pytest/shellcheck/JSON/spec-validate) | + `test-minified-runtime.mjs` + desktop/mobile console-clean **when runtime behavior changes** |
| **ultra** | full + spec `validate.sh --strict` on every touched folder regardless of change | browser/frame evidence **always** (not just "when behavior changes") + cross-viewport |

The problem: **the §2 table already encodes the lite↔full distinction**, but keyed on an *objective* signal ("when runtime behavior changes") rather than a user dial. And the Iron Law (`SKILL.md:140`, `phase_detection.md:86-90`) makes "full" the **floor** — "NO completion claim without fresh verification evidence." A user-set **lite** is therefore a documented license to claim done without the evidence the Iron Law mandates. The only *honest* slider keeps full as the immovable floor and offers ultra as an add-on — at which point "lite" has nowhere to go and the slider collapses to "the optional escalation that already exists as conditional logic." **Low value, high integration risk.**

### Q2 — A "review depth" slider for sk-code-review (skeptical)

sk-code-review **already is a 3-tier depth slider** — it just isn't named one:

- **ALWAYS** (`SKILL.md:88`) — security + correctness + quality + fix-completeness, every invocation. This is the floor.
- **CONDITIONAL** (`:89`) — solid / removal / test-quality, loaded when intent score crosses threshold. ≈ "full."
- **ON_DEMAND** (`:90`, triggered by `ON_DEMAND_KEYWORDS` at `:148` — "comprehensive", "deep review", "full review", "all checks") — full mapped set. ≈ "ultra."

So a depth slider duplicates tiers that exist. It is **better than the status quo in only two narrow ways**: (1) the ON_DEMAND tier is currently reachable only by hitting the right keyword incantation — a stable *name* would make it addressable; (2) ponytail-style env/config **persistence** would let "always deep on this repo" survive across invocations instead of being re-derived from prompt text each time. Both are small.

It is **worse** in the way that matters: the ALWAYS tier is a hard security floor the Precedence Matrix forbids relaxing (`SKILL.md:95`, `review_core.md:82-84`), and the severity contract is deliberately fixed — "Do not adopt `score>=4` or any numeric threshold as a blocker" (`review_core.md:39`). A "lite" depth that drops below ALWAYS or softens gates contradicts doctrine. So the only legal depth slider is "name + persist the existing ON_DEMAND tier" — a thin alias, not a new mechanism.

### Q3 — Where the level resolves (concrete)

- **Ponytail:** `PONYTAIL_DEFAULT_MODE` env > `~/.config/ponytail/config.json` `defaultMode` > `'full'` (`ponytail-config.js:53-73`).
- **sk-code-review already resolves depth by prompt keyword** — `ON_DEMAND_KEYWORDS` in the smart-router pseudocode (`SKILL.md:148, 245-248`). That layer exists today.
- **The idiomatic missing layer is an env var**, because the repo already gates this skill that way: `SK_CODE_REVIEW_MIN_CHANGED_LINES` (`SKILL.md:483`). So the concrete mechanism is **`SK_CODE_REVIEW_DEPTH=lite|full|ultra` resolved in §9**, biasing intent selection toward the ON_DEMAND set, layered over the existing keyword resolution. A flag-file/JSON config (ponytail's middle layer) is *non-idiomatic* here — the repo has no per-skill `config.json` convention, only env-var gates + the `.opencode/.sk-code-review-cache/` dir.
- **For sk-code there is no existing depth resolver** — verification level is fixed by the Iron Law — so a slider there is net-new mechanism, not an alias.

### Q4 — Negative knowledge

**Largely redundant, and actively harmful at the lite end.**

1. **sk-code:** the §2 "when runtime behavior changes" conditional already *is* the lite↔full boundary, driven by an objective signal that's more trustworthy than a user mood-dial. A slider adds a named "ultra" at best and risks legitimizing Iron-Law bypass at worst.
2. **sk-code-review:** LOADING_LEVELS (ALWAYS/CONDITIONAL/ON_DEMAND) already *is* the depth slider, and **M-2** (`:478-510`) already *is* the safe "lite" — but M-2's skip is auto-disarmed for sensitive paths (auth/config/persistence/deps/sandbox/public-facing). A user-set "lite" is **strictly worse than M-2**, because it could suppress depth on exactly the risky diff M-2 protects.
3. **Maintenance asymmetry:** ponytail's slider costs almost nothing because the variable lives in one isolated table (`filterSkillBodyForMode`). In these skills "intensity" is woven through Phase-3 commands, the Precedence Matrix, M-1/M-2, and the severity contract — threading a level through all of them is disproportionate cost for marginal gain.

**The one graft worth keeping:** borrow ponytail's *resolution pattern*, not its slider — a `SK_CODE_REVIEW_DEPTH` env var that names + persists the already-existing ON_DEMAND tier, additive, floor untouched, reusing the `SK_CODE_REVIEW_MIN_CHANGED_LINES` idiom. Everything beyond that is duplication.
