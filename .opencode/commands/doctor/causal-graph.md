# SpecKit Doctor — Causal Graph

Diagnose and rebuild causal-edge integrity inside `mcp_server/database/context-index.sqlite`. The command surfaces causal graph coverage, stale relationship signals, and add-only auto-link candidates for the `causal_edges` table.

```yaml
role: Expert Operator running causal-edges integrity diagnostics + add-only auto-link
purpose: Detect low coverage, orphaned records, drifted lineage, and stale edge observations
action: Discovery -> analysis -> (run: snapshot -> add-only auto-link -> verify -> rollback-on-regression)
```

## 1. PURPOSE

Restore the causal graph to useful coverage without destroying existing lineage evidence. The `causal_edges` table drifts when new spec docs are created without explicit links, when ad-hoc `memory_save` calls skip link suggestions or leave references unresolved, and when decision lineage evolves across packets faster than the graph is backfilled.

This command exposes that drift through the interactive workflow and reports the actions needed for creating only high-confidence new edges. It never deletes existing causal edges, never prunes low-weight edges, and treats coverage below 60% as the gold-battery failure floor inherited from the causal graph routing packet.

## SUBSYSTEM CONTRACT

| Aspect               | Value                                                                       |
| -------------------- | --------------------------------------------------------------------------- |
| Command              | `/doctor:causal-graph`                                                      |
| Modes                | (stats + drift), , (auto-link), |
| Target table         | `causal_edges` inside `mcp_server/database/context-index.sqlite`            |
| Snapshot             | YES — snapshots host context-index DB before any link mutation              |
| Mutation boundary    | ADD-ONLY edges; never deletes existing edges                                |

## STALENESS SIGNALS

The diagnostic phase detects:

1. **Low coverage** — `link_coverage_percent` below 60% of spec-doc records
2. **Orphan records** — high-importance memory records with no incoming or outgoing causal edges
3. **Divergent edge weights** — relation weights skewed enough to imply repeated upsert or extraction imbalance
4. **Stale edge observations** — old `last_observed_at` equivalents (`extracted_at` / `last_accessed`) relative to recent packet edits
5. **Relation imbalance** — one relation dominates the recent insertion window and risks masking real lineage variety
6. **Unresolved lineage** — `memory_drift_why` shows sparse or missing paths around active phase packets

| First-time auto-link on a packet                                       | `--confidence-threshold=0.9` |
| You only want candidate links                                          | `--dry-run`                          |

## MUTATION BOUNDARIES (Phase 3 enforced)

| Allowed Targets                                                          | Forbidden Targets                                                                             |
| ------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------- |
| `mcp_server/database/context-index.sqlite`                               | Any spec folder doc (`.opencode/specs/**/*.md`, `.opencode/specs/**/*.json`)                  |
| `mcp_server/database/context-index.sqlite.pre-doctor-causal-graph.*.bak` | Any skill, agent, or command file                                                             |
| New rows in `causal_edges` via `memory_causal_link` only                 | Any other database, including code graph, skill graph, deep-loop, eval, and vector DBs        |
|                                                                          | Any `DELETE FROM causal_edges`, `memory_causal_unlink`, or direct SQL mutation of causal rows |

Phase 3 first activity (`validate_targets_and_policy`) runs canonical-path validation and add-only policy validation BEFORE any write: resolved targets via `realpath`, exact-match against allowed_targets, no `..` escapes, no symlinks outside repo root, and no forbidden operation strings. Failure halts before mutation.

## WORKFLOW PHASES (delegated to YAML)

```
Phase 0 — Discovery
  Purpose: probe memory_causal_stats; capture coverage and relation health
  Tools: memory_causal_stats, Bash(stat for host DB mtime)
  Output: { total_edges, coverage_percent, health, orphaned_edges, relation_balance }

Phase 1 — Analysis
  Purpose: sample up to 10 candidate records and classify drift
  Tools: memory_drift_why, memory_search
  Output: { candidate_records[], drift_signals[], recommended_action }

Phase 2 — Snapshot
  Purpose: VACUUM INTO snapshot of context-index.sqlite before any edge insert
  Tools: Bash(sqlite3 ... 'VACUUM INTO ...'), disk-free preflight
  Output: { snapshot_path, snapshot_bytes }
  Halt-on: disk-free < 2x current DB size

Phase 3 — run
  First activity: validate_targets_and_policy
  Then: memory_causal_link for each candidate with confidence >= threshold and no existing same source/target/relation edge
  Output: { attempted: N, inserted: M, skipped: K, threshold }

Phase 4 — Post-verify
  Purpose: gold-battery coverage check; coverage must be >= 60% and must not regress from baseline
  Tools: memory_causal_stats, memory_drift_why on sampled linked records
  Output: { gold_battery_pass: bool, coverage_percent, regressions[] }

Phase 5 — Rollback (only on regression in interactive workflow or user request in confirm)
  Purpose: restore context-index.sqlite from snapshot
  Tools: Bash(cp -p)
  Output: { rolled_back: true, restored_at }

Phase 6 — Report
  Purpose: emit final status + state log
  Output: { status: OK|APPLIED|ROLLED_BACK|FAIL, state_log_path }
```

## 6. INSTRUCTIONS

The orchestrator (`assets/doctor_causal-graph.yaml`) executes the workflow phases above. As the operator running this command:

1. **Resolve mode + flags** via the Unified Setup Phase prompt above. Do not advance until ALL inputs are bound (`execution_mode`, `intent`, `confidence_threshold`, `no_snapshot`, `dry_run`).
2. **Load the matching YAML** from `assets/`. The surviving YAML file is `doctor_causal-graph.yaml`.
3. **Execute the YAML phases sequentially**. Phase 0 and Phase 1 are read-only. Phase 2-5 only run in modes.
4. **In modes, never skip the snapshot phase** unless the user passed `--no-snapshot` explicitly. The validator records the waiver and rollback becomes unavailable.
5. **Honor the confidence threshold**. The threshold default is 0.7 and MUST NOT be lowered. Any candidate below threshold is skipped.
6. **Honor add-only semantics**. If `memory_drift_why` shows the same source/target/relation already exists, skip the candidate rather than calling `memory_causal_link`.
7. **Always emit the state log** in Phase 6, regardless of success or failure. The log feeds `/doctor:update`'s cross-subsystem dashboard.

Read the YAML file end-to-end before starting Phase 0. Do not paraphrase; the YAML defines exact tool sequences, parameters, halt conditions, and rollback behavior.
