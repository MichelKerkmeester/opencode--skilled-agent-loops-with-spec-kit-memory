# v3.5.0.1, The Transition Gets Its Map

A patch release with two stories. The small one: a day of defect fixes on the runtime's integration surfaces — Codex sessions now run their own hooks instead of Claude's, and two documentation lies were corrected. The big one: the complete research program for moving all three MCP servers to dual-stack CLI fallbacks landed as a fully phased, gap-audited spec tree.

Seven autonomous research runs, three GO verdicts, sixteen validated packets, and every implementation phase scoped down to its acceptance criteria — ready to build, nothing built yet.

## Codex Runs Its Own Hooks

#### The rewiring

The live `.codex/hooks.json` had drifted: it registered Claude's hook scripts for every event, so Codex sessions were primed through the wrong adapters. `SessionStart` and `UserPromptSubmit` now point at the Codex-native hooks, which read the Codex stdin envelope and answer in kind — both smoke-verified. The `PreCompact` registration was removed outright rather than rewired: the hook contract lists Codex compaction as unsupported, so the entry was dead weight pointing at a Claude-envelope script.

#### Two paper cuts

The Codex MCP config note claimed the code-graph database default was the legacy shared path and the skill-local path was legacy — it is exactly the other way around, and the note now matches the launcher source. And the skill-advisor's Gemini hook catalog pointed at the spec-kit shim as the implementation; it now names the active implementation and the shim separately.

## The Bridge Fix That Was Correctly Un-Shipped

#### Working, racy, reverted

The OpenCode code-graph plugin bridge has been inert since a skill extraction moved three of its imports — it is why sessions report "Code Graph: unavailable." Re-pointing the imports made it run again and emit the right transport payload. A fresh-model review then flagged what the smoke test could not: the runnable bridge initializes the memory database directly in its own process, outside the daemon's single-writer lease — the same corruption class the daemon architecture exists to prevent.

The fix was reverted. Broken-and-inert is safer than working-and-racy; the bridge stays down until the transition program delivers the IPC-backed transport that fixes it properly.

#### The sweep that wasn't

A planned sweep of orphaned launcher processes ended the same honest way: parent-process classification showed all nine running launchers belonged to live sessions, so nothing was killed. The true orphan class — launchers that outlive an exited owner — is a scoped fix inside the transition program.

## The MCP→CLI Transition Program

#### Why it exists

The motivating incident hit twice during the program's own research day: the MCP transport dropped mid-session and took its tools with it, unrecoverable until restart. The 028 program is the evidence-based answer — a CLI over each existing daemon, dual-stack, with the MCP registration untouched.

#### Three verdicts from seven runs

Spec-memory went deepest: feasibility fan-out across three models, a buildable CLI design, convergence-driven risk resolution, and a closure run that ended with zero unknowns — 37 tools portable, eight named design deltas, measured overheads. Code-index and skill-advisor each got a forced-ten-iteration study against the settled spec-memory premise: code-index is GO at 8/8 tools with blocked-read rendering named as its top risk, and skill-advisor is GO at 9/9 with its existing Python fallback script reconciled as a legacy facade after a measured 10-of-10 scorer-parity check — and a measured 824.8ms one-shot native path that bans per-prompt bridge calls from hooks forever.

#### Audited before anyone builds

Three adversarial gap-audit lanes then reviewed the program itself: zero blockers, nineteen documentation gaps. Among them: a cross-workstream blind spot no per-system run could see (nothing tested three CLIs spawning three daemons simultaneously) and a dual-failure mode nobody had pinned (MCP down and the daemon dead inside a prompt hook). Every gap is patched into the specs with an owning task.

#### What is scoped now

Each workstream carries a research record plus three implementation phases with requirements, acceptance criteria, handoff chains and estimates — roughly 25 to 34 engineering days of scoped work. Hooks for Claude Code, Codex and Devin plus an OpenCode plugin are required pairing for every CLI.
