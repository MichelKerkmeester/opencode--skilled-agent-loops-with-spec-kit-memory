## Mechanism
OpenLTM’s janitor is an explicit sequential pass guarded against concurrent runs, with per-stage error capture and run-result counters (`packages/openltm-core/src/janitor/index.ts:51`, `packages/openltm-core/src/janitor/index.ts:52`, `packages/openltm-core/src/janitor/index.ts:74`, `packages/openltm-core/src/janitor/index.ts:116`). The order is embed missing memories, decay, archive, promote, then dedup-stage (`packages/openltm-core/src/janitor/index.ts:75`, `packages/openltm-core/src/janitor/index.ts:82`, `packages/openltm-core/src/janitor/index.ts:89`, `packages/openltm-core/src/janitor/index.ts:96`, `packages/openltm-core/src/janitor/index.ts:103`). It persists last-run counters in `settings`, seeded by migration 012, then runs WAL checkpoint and `ANALYZE` hygiene (`packages/openltm-core/src/janitor/index.ts:130`, `migrations/012_seed_janitor_settings.sql:5`, `packages/openltm-core/src/janitor/index.ts:139`).

Scheduling is optional: `ltm.janitor.intervalMinutes` defaults to `0`, so auto-run is disabled unless configured (`packages/openltm-core/src/janitor/providers/types.ts:140`, `packages/openltm-core/src/janitor/index.ts:183`, `packages/openltm-core/src/janitor/index.ts:189`). A separate Honker scheduler defines `@every 6h`, leader-elected enqueue, worker claim/retry, and inert fallback when Honker is unavailable; actual server startup wiring is UNKNOWN from the scoped files (`packages/openltm-core/src/scheduler/index.ts:18`, `packages/openltm-core/src/scheduler/index.ts:10`, `packages/openltm-core/src/scheduler/index.ts:61`, `packages/openltm-core/src/scheduler/index.ts:75`).

Embedding maintenance uses a `memory_embeddings` side table with model/dim metadata and optional vec0 dual-write (`migrations/010_memory_embeddings_split.sql:5`, `packages/openltm-core/src/dao/embeddings.ts:36`, `packages/openltm-core/src/dao/embeddings.ts:48`). The janitor asks for up to 10,000 missing IDs and embeds batches of 50 by default (`packages/openltm-core/src/janitor/embeddings.ts:86`, `packages/openltm-core/src/janitor/embeddings.ts:92`, `packages/openltm-core/src/janitor/embeddings.ts:107`). Caveat: the missing-ID DAO filters only `status='active'`, while the later fetch allows active/pending, so pending proposal embedding is effectively UNKNOWN/active-only (`packages/openltm-core/src/dao/embeddings.ts:69`, `packages/openltm-core/src/janitor/embeddings.ts:99`).

Decay is materialized: migration 011 adds indexed `decay_score`, and recall uses `ORDER BY decay_score DESC` for default relevance (`migrations/011_add_decay_score_and_archive.sql:4`, `migrations/011_add_decay_score_and_archive.sql:6`, `packages/openltm-core/src/db.ts:524`, `packages/openltm-core/src/db.ts:526`). The formula is `importance * confidence * 0.5^(age/half_life)`, except importance 5 and `confirm_count >= 10` do not age-decay; half-lives are 180/90/30/14 days for importance 4/3/2/else (`packages/openltm-core/src/janitor/decay.ts:14`, `packages/openltm-core/src/janitor/decay.ts:17`, `packages/openltm-core/src/janitor/decay.ts:20`, `packages/openltm-core/src/janitor/decay.ts:22`, `packages/openltm-core/src/janitor/decay.ts:24`, `packages/openltm-core/src/janitor/decay.ts:26`). Deprecation triggers at `importance < 5`, `confirm_count < 10`, and `decay_score < 0.25`; archive later evicts only deprecated, importance <=2, recall_count <=1, score <0.10, max 100/run, preserving JSON first (`packages/openltm-core/src/janitor/decay.ts:31`, `packages/openltm-core/src/janitor/archive.ts:4`, `packages/openltm-core/src/janitor/archive.ts:17`, `packages/openltm-core/src/janitor/archive.ts:49`).

Promotion scans active unlinked `context_items` of type decision/gotcha with content >20 chars, dedups by normalized key, then creates `status='pending'` memories with source `auto-promote`, confidence 0.8, architecture/gotcha category, gotcha importance 4, and decision importance default 3 (`packages/openltm-core/src/janitor/promote.ts:51`, `packages/openltm-core/src/janitor/promote.ts:64`, `packages/openltm-core/src/janitor/promote.ts:83`, `packages/openltm-core/src/janitor/promote.ts:89`, `packages/openltm-core/src/janitor/providers/types.ts:139`). Approval flips pending to active; rejection deletes the pending memory and resets the context item (`packages/openltm-core/src/janitor/promote.ts:112`, `packages/openltm-core/src/janitor/promote.ts:122`, `packages/openltm-core/src/janitor/promote.ts:134`, `packages/openltm-core/src/janitor/promote.ts:146`). Dedup intends pairwise active-memory cosine matching at 0.85 plus optional LLM JSON verdict, then saves pending `dedup:idA:idB` proposals; however it still selects `memories.embedding` even though migration 010 drops that column, so the current dedup pass appears schema-stale (`packages/openltm-core/src/janitor/dedup.ts:73`, `packages/openltm-core/src/janitor/dedup.ts:90`, `packages/openltm-core/src/janitor/dedup.ts:114`, `packages/openltm-core/src/janitor/dedup.ts:160`, `packages/openltm-core/src/janitor/dedup.ts:211`, `migrations/010_memory_embeddings_split.sql:24`).

## Negative knowledge
Do not copy fixed half-life decay as the primary relevance model; FSRS plus co-activation is more expressive. Do not copy O(n²) all-pairs dedup as-is; use vector top-K/proposal staging. Do not hard-archive spec continuity by simple score thresholds; governed retention and provenance should dominate. Do not adopt Honker-style scheduling unless we first prove current self-maintenance leaves stale work.

## Open questions
Should system-spec-kit expose a first-class `pending_memory_proposals` review surface? Should dedup proposals preserve causal edges as `supports`/`supersedes` candidates instead of merges? Should maintenance reporting be a dry-run janitor dashboard over existing tools rather than a new writer?
