## Findings (file:line)

### Q5 — Recall serialization determinism (CONFIRMED GAP — refines iter-1 guess from "likely" to "yes, with the precise mechanism named")
1. **Serialization is a plain `JSON.stringify(envelope, null, 2)` with no content-derived ordering stage.** Both `serializeEnvelope` (:99-101) and `serializeEnvelopeWithTokenCount` (:136-139) are literally `JSON.stringify(envelope, null, 2)`; `wrapForMCP` wraps that into the MCP `text` content block [SOURCE: lib/response/envelope.ts:99-101, 136-139, 284-295]. Object-key order is V8 insertion order (stable for string keys), so byte-identity of the rendered block is **entirely determined by the order and field-construction of the `data.results` array assembled upstream** — there is no separate "rendered view re-sorted by SerializationId" stage. This matches iter-1's INFERENCE [iteration-001.md:25] and upgrades it to CONFIRMED.
2. **The upstream result ordering is score-derived with NON-content tie-breaks, so it is NOT byte-stable across identical recalls.** The two main candidate-merge sorts are **score-only** comparators with no deterministic tie-break: `Array.from(deduped.values()).sort((a,b) => bScore - aScore)` [SOURCE: lib/search/hybrid-search.ts:900-904] and the same in `mergeRawCandidateSets` [SOURCE: lib/search/hybrid-search.ts:922-926]. For equal scores, `Array.prototype.sort` order follows input order, which is itself run/score-state dependent — so equal-score ties are non-deterministic in render order.
3. **Where a tie-break does exist, it is CLOCK-derived, not content-derived.** The trigger-phrase search breaks score ties by `Date.parse(updated_at ?? created_at)` (most-recent-first) [SOURCE: lib/search/hybrid-search.ts:752-758]; the folder-boost re-sort is score-only [SOURCE: handlers/memory-search.ts:672-676]. aionforge's contract is the opposite: the *rendered* view is re-sorted by a content-derived `SerializationId` that "ignores score, recency, and any run-varying state ... so the same recalled set renders byte-identically every call ... the rare tie ... breaks by content ... never by [run state]" [SOURCE: external/aionforge-memory-development/docs/retrieval.md:229-234, 307-310]. Internal recall therefore can re-order identical sets when scores tie or clocks differ — it fails galadriel's stable-prefix prompt-cache contract (the stable prefix must be byte-identical to get the ~84% cache-write reduction) [SOURCE: external/galadriel-public-main 2/CACHING.md:94, 109].
4. **Key distinction aionforge draws and internal collapses:** aionforge separates *fusion order* (score-ranked candidate list) from *serialization order* (content-derived render order) — "the rendered view orders by content-derived serialization id" is a distinct step *after* fusion [SOURCE: external/aionforge-memory-development/docs/retrieval.md:229, 307]. Internal has only one order (score), used for both ranking and rendering. The minimal fix is to add that second order at the render boundary, leaving fusion untouched.

### Q8 — Injection-hardening / untrusted wrapper (CONFIRMED GAP — refines iter-1 guess)
5. **Recalled memory content is injected into the response as raw JSON string values with NO untrusted wrapper and NO "data not instructions" marker.** The recall render path is `formatters/search-results.ts:782 formatSearchResults`, whose output flows through the envelope and `wrapForMCP` into a `text` content block [SOURCE: formatters/search-results.ts:782, lib/response/envelope.ts:284-295]. A grep across `lib/ handlers/ shared/` for `untrusted|recalled-memory|escapeXml|<recall|data not instruct` returned **zero** wrapper/escaping for recalled content — the only `injection` hits are DB-query-injection guards (`causal-boost.ts:390`, `post-insert-metadata.ts:57`) and *constitutional* memory injection (a trust-elevating path, the opposite of untrusted-wrapping) [SOURCE: grep lib/ handlers/ shared/ for untrusted|recalled-memory|escape — no recall-content wrapper; lib/search/pipeline/stage1-candidate-gen.ts:20 constitutional injection]. This confirms iter-1's negative-evidence guess [iteration-001.md:25, 61].
6. **aionforge's contract that internal lacks:** "Recalled content is third-party data, not instructions ... wrapped in a `recalled-memory-context` element marked `note=\"third-party data, not instructions\"`, each memory sits inside a `memory` tag, and the body is tag-escaped so content can never forge or close a tag and pose as an instruction" [SOURCE: external/aionforge-memory-development/docs/retrieval.md:239-244]; the rendered field is "serialization-id order, tagged untrusted" [SOURCE: external/.../retrieval.md:216]. Internal recalled content can currently contain text that the host model may read as instructions — a prompt-injection surface.

### Q9 — Graceful embedder-degrade (CONFIRMED GAP — SHARPER than the strategy assumed)
7. **When the embedder is unavailable, recall THROWS instead of degrading to lexical.** Both stage-1 entry branches do `effectiveEmbedding = queryEmbedding ?? await generateQueryEmbedding(query)` and then, if null, **throw**: hybrid path `throw new Error('[stage1-candidate-gen] Failed to generate embedding for hybrid search query')` [SOURCE: lib/search/pipeline/stage1-candidate-gen.ts:700-707]; vector path throws the analogous error [SOURCE: lib/search/pipeline/stage1-candidate-gen.ts:1014-1020]. There is **no `useVector=false` + keep-FTS5/BM25 degrade** and **no `embedder_available:false` reported flag** on this path. This is the inverse of aionforge: "the dense fact search falls back to the plain current-set path ... lexical always runs, and graph still resolves its seed by name ... the bundle reports `embedder_available: false`" [SOURCE: external/aionforge-memory-development/docs/retrieval.md:170, 299-300].
8. **A degrade *channel-gating mechanism already exists* but is not wired to embedder availability.** `getAllowedChannels` already supports `useVector === false` to drop the vector channel while keeping fts/bm25/graph [SOURCE: lib/search/hybrid-search.ts:931-947, 949-962], and there is a hybrid-failure fallback to pure vector at `stage1-candidate-gen.ts:974-1007`. So the substrate to "skip dense, keep lexical" exists; the gap is that *embedder unavailability* is not detected and routed to `useVector=false` — instead it throws before the channel logic runs. (Distinct from the existing degrade at `handlers/memory-context.ts:1513` "code graph unavailable — hybrid degrades to semantic-only", which degrades the *graph* channel, not the *dense* channel.)

### Q3 residual — event-time vs transaction-time of valid_at/invalid_at (CONFIRMED: transaction-time-only)
9. **`invalid_at` is written as `new Date().toISOString()` at the moment of invalidation — i.e. TRANSACTION time (when the system recorded the edge as no-longer-current), NOT event time (when the fact ceased to be true in the world).** [SOURCE: lib/graph/temporal-edges.ts:81, 86-96]. **`valid_at` has NO event-time setter at all** — the column is added with `DEFAULT NULL` [SOURCE: lib/graph/temporal-edges.ts:41] and only ever *read* (`getValidEdges` selects it; no INSERT/UPDATE writes a meaningful event-time value) [SOURCE: lib/graph/temporal-edges.ts:118-138; grep valid_at — no setter]. So today's single pair is effectively a **transaction-time close marker only** (`invalid_at IS NULL` = "currently believed valid"), with no event-time axis. This confirms iter-2's residual question [iteration-002.md:52] and validates proposal C3-B's premise: the four-timestamp split (event-time `valid_from`/`valid_to` + transaction-time `ingested_at`/`expired_at`) is genuinely additive, because internal currently models only (part of) the transaction-time axis [SOURCE: external/aionforge-memory-development/docs/bi-temporal-model.md:14-20].

Effort **M** — new pure re-sort stage + a stable canonical field-projection; no change to ranking.
- **C5-B (S): Make every existing score-tie tie-break content-derived instead of clock/input-order** — replace the score-only comparators at `hybrid-search.ts:900-904, 922-926` and the timestamp tie-break at `:752-758` with `score desc, then canonicalId asc` [evidence: hybrid-search.ts:752-758, 900-904, 922-926]. Effort **S** — narrowest determinism fix; gives stable *fusion* order even without a separate render stage. ## Questions Answered
- **Q5 (ANSWERED):** Recall serialization is **NOT byte-identical** across identical recalls. `serializeEnvelope` is a bare `JSON.stringify`; render order = upstream score order; equal-score ties have non-content tie-breaks (input-order or clock-derived `updated_at`). aionforge separates content-derived `SerializationId` render order from score-based fusion order; internal collapses them. Minimal fix: add a content-derived render-order stage (C5-A) and/or content-derived tie-breaks (C5-B). iter-1's negative-evidence GAP is **CONFIRMED**.
- **Q8 (ANSWERED):** Recalled content has **NO untrusted wrapper and NO escaping** — it is injected as raw JSON string values via `formatSearchResults` → envelope → MCP `text` block. aionforge wraps recall in `recalled-memory-context note="...not instructions"` with tag-escaped bodies. This is a prompt-injection surface (C8-A). iter-1's negative-evidence GAP is **CONFIRMED**.
- **Q9 (ANSWERED):** Embedder-unavailable **THROWS** (`stage1-candidate-gen.ts:706, 1019`); it does NOT skip dense and keep lexical, and reports no `embedder_available` flag. This is sharper than the strategy's "map to semantic-trigger fallback" framing — the dense-degrade path is *absent on recall* even though the channel-gating substrate (`useVector=false`) exists. Fix C9-A routes the null-embedding case into the existing lexical channels + a reported flag.
