### B. Q3 Conflicts And Ruled-Out List

2. **Research, strategy, testing programs, and operations are ruled out as first-class `sk-design` capabilities.** The corpus covers user research, UX strategy, validation, operations, and stakeholder utilities, but those exceed taste-led build/visual scope unless reduced to a concrete observable UI check. The adoption should be a crosswalk, not a second score model. ### D. New-Mode Judgment

No new `sk-design` mode is justified.

The strongest candidate for a new cluster is `visual-critique`, but it does not need separate routing, a separate backend, or a separate ownership model. Its useful content is a set of audit lenses, and `audit` already owns critique, accessibility, performance, anti-slop detection, scoring, production hardening, and findings. [SOURCE: .opencode/skills/sk-design/SKILL.md:28] [SOURCE: .opencode/specs/skilled-agent-orchestration/154-sk-design-parent/024-designer-skills-research/research/iterations/iteration-006.md:35]

The same is true for `interaction-design` and `design-systems`: their adoptable parts split cleanly into `motion`, `interface`, `audit`, and `foundations`. The current registry already gives those homes, and the hub requires new modes to be explicit registry extensions with new packets rather than hub logic. [SOURCE: .opencode/skills/sk-design/mode-registry.json:14] [SOURCE: .opencode/skills/sk-design/SKILL.md:84] [SOURCE: .opencode/skills/sk-design/SKILL.md:96]

A new mode would only become justified if a future cluster has all three properties: distinct user intent, distinct output contract, and distinct ownership that cannot be expressed as a packet edit to one of the five modes. None of the 9 plugins meets that threshold in the current corpus evidence.

## Questions Answered

- **Q1: What is in scope vs out of scope across all 9 plugins?** Answered in the ledger above. In scope: concrete build-facing visual, motion, interface-flow, foundations, accessibility, production-hardening, and audit-evidence checks. Out of scope: research, strategy, validation programs, design ops, governance, adoption, decks, case studies, negotiation, and broad lifecycle workflows.
- **Q2: Where do adoptable items live?** Existing modes cover the useful material: `audit` receives critique/release/evidence hardening; `interface` receives flow-floor, copy, illustration, and restraint checks; `foundations` receives layout/color/theme refinements; `motion` receives state-machine and motion-token refinements. `md-generator` receives no item from this corpus. [SOURCE: .opencode/skills/sk-design/SKILL.md:23]
- **Q3: What conflicts or duplicates should be ruled out?** Ruled out: command-suite import, parallel scoring, duplicate cognitive-law/color/responsive/token/icon imports, research/testing/ops/strategy workflows, evidence overclaims, governance/documentation systems, and new modes for critique, interaction, emotional design, design ops, or design systems.
- **Q4: What is the draft prioritized backlog?** Answered in the ranked skeleton. The adoption order should start with audit visual-critique crosswalk, interface flow floor, audit hardening bundle, motion state-machine fragment, and foundations layout refinements.
- **Does any cluster justify a new mode?** No. The visual-critique cluster is high leverage but belongs in `audit`; no corpus cluster crosses the distinct-intent/distinct-output/distinct-owner threshold for a sixth mode.
