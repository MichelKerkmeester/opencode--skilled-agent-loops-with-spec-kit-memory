## 2. PROBLEM & PURPOSE

### Problem Statement
The canon-conformance program needs a single, machine-validated audit input before any per-lane conformance work begins. Without a resolved `--lane-config` and a confirmed-live sk-doc adapter, the deep-alignment loop can silently emit blanket `could-not-validate` P1s (a known prior failure mode when `validate_document.py`'s `template_rules.json` path was broken), which would corrupt every downstream child checklist.

### Purpose
Establish the audit foundation: an immutable BASE census, a schema-valid deep-alignment `--lane-config` covering the two sk-doc lanes (command-docs, agent-docs), and a live confirmation that the sk-doc adapter returns REAL findings (P0/P1/P2 keyed off actual `validate_document.py` results) rather than blanket adapter errors. Downstream phases 001-003 consume these findings as their verifier contracts.

## 3. SCOPE

### In Scope
- The immutable BASE commit hash + census of the 29 command docs / 13 agents × 3 runtimes (recorded in `implementation-summary.md`).
- `lane-config.json`: a schema-valid deep-alignment lane-config (2 sk-doc/docs lanes; `scoping.cjs` exit 0).
- A live deep-alignment sk-doc audit run confirming real findings, executor `openai/gpt-5.6-luna-fast --variant xhigh`.

### Out of Scope
- Any command, agent, or codex file EDIT — this phase only reads and audits.
- The codex-parity surfaces (`.codex/prompts/`, `sync-prompts.cjs`) — verified by their own `--check`, not by the deep-alignment sk-doc lane.
- Fixing findings — that is phases 001-003.

<!-- ANCHOR:risks -->
## 6. RISKS & DEPENDENCIES

- **Adapter path drift**: the sk-doc adapter historically emitted blanket `could-not-validate` when `template_rules.json` resolution broke. Mitigation: REQ-002 gates on a real finding before downstream reliance.
- **Dependency**: the deep-alignment engine (`deep-alignment/`) and the sk-doc `validate_document.py` (`shared/` variant) must both be runnable from repo root.
