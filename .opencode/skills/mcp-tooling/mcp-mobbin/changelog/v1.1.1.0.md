## [**1.1.1.0**] - 2026-07-16

Live Code Mode discovery ran on 2026-07-16, pre-auth (direct stdio MCP probe of CodeMode-MCP: `initialize`, then `tools/call` on `list_tools`, `search_tools`, `tool_info`, with `UTCP_CONFIG_FILE=.utcp_config.json`). The dated fixture `references/discovery-fixture-2026-07-16.json` is the packet's new ground truth, and this release supersedes the one-tool research baseline with the live three-tool inventory.

#### Three tools supersede the one-tool baseline

- Need: The whole packet was built on the official `mobbin/skills` one-public-tool record (`search_screens` only), with `search_flows` explicitly listed as a tool that "does not exist and is never invented".
- Change: Live pre-auth `list_tools` returned **THREE** read-only search tools — registry names `mobbin.mobbin.{search_screens,search_flows,search_sections}`, TS callables `mobbin.mobbin_search_screens(...)` etc. — recorded as a dated supersession (2026-07-16) across SKILL.md, README, INSTALL_GUIDE, tool-surface.md, mcp-wiring.md, troubleshooting.md, the feature catalog (root + flows/screens/apps/elements leaves), the examples, the playbook (root + DISCOVER-001, FLOWS-001, SCREENS-001), the server package README, the asset checklist, and both scripts. All three are read-only search tools; the mutation-refusal check is recorded as passed. Flow research now prefers `search_flows` (ordered per-screen previews are retrieved fact; anything beyond them stays labeled inference); website-section research uses `search_sections`.
- Why it matters: The packet's central "never invent `search_flows`" rule would have made an agent refuse a real live tool; the supersession is recorded with the fixture cited so the boundary now tracks the observed inventory.

&nbsp;

#### The `deep` conflict is resolved

- Need: The research record preserved an input-vs-server-behavior conflict on `deep` search and forbade hardcoding it.
- Change: The fixture's `search_screens` schema shows `mode?: "deep" | "standard" | "fast"` — a client-settable input (`"deep"` = AI-powered relevance pipeline for nuanced queries; `"fast"` = deprecated alias for `"standard"`). The schema also exposes `exclude_screen_ids` and `image_format`, and the declared output is `{ query, screens[{id, image_url, mobbin_url, app_name, platform}] }` — the research-documented `index`/`failed[]` fields are absent from the declared schema, flagged for verification on the first authenticated call.
- Why it matters: Agents can now use `deep` mode deliberately instead of tiptoeing around a preserved conflict, and the response-shape docs no longer assert fields the live schema does not declare.

&nbsp;

#### Discovery is pre-auth; naming confirmed

- Change: `tools/list` and full schemas needed no OAuth — discovery is pre-auth; authenticated CALLS remain operator-gated (those items stay SKIP-valid with exact commands). The convention-derived naming prediction held exactly and is flipped from INFERRED to CONFIRMED everywhere, with the dotted registry form and the underscore TS form both recorded.
- What stays open: OAuth end-to-end (Inferred), authenticated response shapes, inline-image fidelity through `call_tool_chain`, Free-denial semantics, per-plan caps, Finance+ coverage.
