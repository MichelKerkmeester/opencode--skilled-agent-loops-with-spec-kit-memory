# Research: Deep-Loop Unification Merge Design — Validation Synthesis

## Methodology

20-replica multi-model fanout via `deep-loop-runtime/scripts/{fanout-run,fanout-merge}.cjs`: 10x `openai/gpt-5.5-fast` (xhigh), 5x `zai-coding-plan/glm-5.2` (max), 5x `claude-sonnet-5` (xhigh, via `cli-claude-code`). **15/20 completed successfully** (10 gpt55-fast + 5 glm52, 165 raw findings merged into `deep-research-findings-registry.json`). **5/20 (all `sonnet5-*`) failed terminally** — root-caused via direct log inspection to `Not logged in · Please run /login`: `cli-claude-code`'s headless subprocess dispatch cannot access this machine's macOS-Keychain-stored Claude Code credentials (confirmed reproducible: `env -i HOME PATH claude -p ...` fails identically; full inherited env succeeds; adding `SSH_AUTH_SOCK`/`__CFBundleIdentifier` to the stripped env did not fix it — a genuine, unresolved gap in `executor-audit.ts`'s env-isolation design, not a config typo).

**Substitution**: the 5 sonnet5 iterations were re-run as 5 parallel in-session Plan-agent dispatches (model: sonnet), each independently verifying a different subsystem of the merge design against live files (not trusting the existing plan docs). This satisfies the intent (5 real Sonnet-5-xhigh research passes) via a different mechanism than originally planned.

**Convergence floor**: satisfied trivially and more strongly than required — `antiConvergence.minIterations: 3` is baked into every individual replica's own config, and fan-out only merges after all lineages independently terminate (no cross-lineage early-stop path exists to violate the "≥3 per model" requirement).

---

## VERDICT

**The mechanical design (structural layout, Class A/B path-repair rule, `2.0.0.0` versioning, identity-only rename scope) is sound.** But **do not execute phase 002 as currently written** — one finding is severe enough to corrupt real content on the very first irreversible step, and several others will make the plan's own exit gates fail for reasons unrelated to the merge itself, creating false-regression confusion. All are fixable with targeted plan.md edits, not a redesign. Full revision list in §5.

## 1. Severe finding: Stage 2's `SKILL.md → README.md` demotion will destroy real content

`deep-loop-runtime/README.md` **already exists** (18,543 bytes, real distinct content — "the pre-existing operator-facing overview") alongside `SKILL.md` (22,320 bytes, the routable-skill identity file). Confirmed via direct `ls`/`head`. Phase 002's plan says, verbatim: *"demote `runtime/SKILL.md` → `runtime/README.md` (strip routable-skill frontmatter, keep content)"* — written as if `README.md` doesn't already exist. As literally written this either silently overwrites real content or fails outright on `git mv` to an existing target. Compounding: `deep-improvement/assets/non_dev_ai_system/profiles/deep-loop-runtime.json` (a live, consumed Lane-D self-benchmark config, reached via `/deep:ai-system-improvement --self-target deep-loop-runtime`) has `"frozen_surface": [{"live_relpath": "SKILL.md", "section_anchor": "## 3. HOW IT WORKS", ...}]` — already treating SKILL.md and README.md as two distinct, differently-roled artifacts. Neither phase 002 nor 003's plan names this file anywhere; grep-based verification (REQ-002's `.cjs`/`.ts`-only pattern) is structurally blind to it since the breakage is a *rename decision*, not a *skill-name string*.

**Required fix**: rewrite Stage 2 to explicitly reconcile the two files (merge content, or keep README.md as-is and give SKILL.md's content a different destination) before any `git mv` runs. Add the Lane-D profile as an explicit Stage 3b/003 fix target.

## 2. Both structural-move site inventories undercount

**Forward coupling (Class A)**: `compile-command-contracts.cjs` has 44 literals per direct count (confirmed both by my own agent and independently by the raw fanout registry, which counted ~50 — the discrepancy is likely inclusive/exclusive counting of 2 additional self-referencing literals at lines 22-23 pointing at `executor-audit.ts`/`post-dispatch-validate.ts`, rendered into contract markdown, not `sha256File`-checked, so non-crashing but leaves dead paths in every compiled `/deep:*` contract). `runtime/tests/unit/` has **10** files referencing the old name, not 7 — 3 extra (`meta-loop-lane-d-packaging.vitest.ts`, `check-contract-drift.vitest.ts`, `prompt-pack.vitest.ts` — the last also hardcodes a machine-specific absolute path, a pre-existing portability issue). `render-command-contract.cjs` needs a **second** edit beyond its one documented `require()` fix — its own `WORKSPACE_ROOT` constant.

**Reverse coupling (Class B)**: 16 code files reference `deep-loop-runtime`, not 12 — missing: `deep-ai-council/scripts/orchestrate-session.cjs` (3 requires), `deep-research/scripts/runtime-capabilities.cjs`, `deep-review/scripts/{reduce-state,runtime-capabilities}.cjs`. One file (`deep-ai-council/scripts/replay-graph-from-artifacts.cjs`) needs a **different fix shape entirely** — a repo-root-detection walk that needs a segment *inserted*, not hop-decremented; the plan's binary Class A/B framework has no bucket for it. None of `orchestrate-session.cjs`, `replay-graph-from-artifacts.cjs`, or 7 `deep-ai-council/scripts/tests/*.vitest.ts` files are covered by any wired `npm test` — if a fix here is subtly wrong, no automated gate catches it.

**`system-spec-kit` tooling-borrow**: real, but ~15+ sites, not 4 — most are hardcoded literals inside `test:council`'s own constituent test files (`multi-ai-council-{audit-trail,permission-scope,rollback}.vitest.ts`, `council-helpers-smoke.vitest.ts`, `council-playbook-anchor-integrity.vitest.ts` — the last does `readdirSync` on a path built from a hardcoded `deep-loop-workflows` literal as its *first statement*, meaning `test:council` will hard-crash immediately post-move unless this file is also fixed, even though it's `system-spec-kit`-owned and outside 002's stated scope). Two more production-code sites entirely missing from any stage: `lib/deep-loop/artifact-root.cjs` (needs the same one-more-`..` fix as `package.json`/`tsconfig.json`) and its test companion `tests/unit/artifact-root.vitest.ts`. **Worst of this class**: `tests/unit/dependency-seams.vitest.ts` doesn't fail post-move — it silently stops testing what it claims to test (a sibling-reach-in guard whose computed comparison path becomes structurally wrong, so the assertion trivially passes regardless of whether a real regression exists). Two more files (`optimizer-manifest-anti-convergence.vitest.ts`, `deep-research-convergence-floor.vitest.ts`) share a `resolve(testDir,'..','..','..','..')` idiom needing an *added* hop — mischaracterized by the plan as "same 3 ups" Class A sites.

**Required fix**: expand Stage 3a/3b's file tables to include all sites above; treat `dependency-seams.vitest.ts` specially (its assertion needs re-deriving `skillsRoot` correctly, not just a path-string edit, or it becomes a permanently-blind guard).

## 3. Test-suite baselines in the plan are stale, and two gates are already red pre-merge

Live `cd deep-loop-runtime && npm test`: **71 files / 659 tests**, not the plan's "47/379" — the skill has grown substantially since that figure was recorded. **2 deterministic (non-flaky, reproduced across 4 runs) failures exist today, unrelated to the merge**: `check-contract-drift.vitest.ts` (pre-existing compiled-contract drift from 2 recent unrelated commits) and `executor-provenance-mismatch.vitest.ts`. Neither is in `loop-lock.vitest.ts` as the plan's stale note claims.

`cd system-spec-kit/mcp_server && npm run test:council`: **currently RED** (2/9 files failing), for reasons unconnected to the merge (`multi-ai-council-runtime-parity.vitest.ts` expects a `.toml` agent file that has never existed; `persist-artifacts.cjs` test expects a missing fixture artifact).

**Required fix**: 002's Definition of Done ("`npm test` green", "`test:council` succeeds") must explicitly baseline against "2 known pre-existing `runtime` failures + 2 known pre-existing `test:council` failures, none merge-related" — as currently worded, both gates will be false immediately after a mechanically perfect merge, and whoever executes this will waste time chasing phantom regressions.

## 4. Sequencing gap: phase 002's own exit gate cannot pass using only phase 002's own scope

002's Phase-3 verification requires a live `/deep:research` or `/deep:review` short run. But all three `/deep:*` commands open with an identical hardcoded line invoking `deep-loop-runtime/scripts/render-command-contract.cjs` by path — a fix explicitly assigned to **003**, which is gated on 002 completing first. Right after 002's physical move and before 003 begins, this verification step hits a dead path. **Required fix**: either pull the 3-file command-router fix forward into 002's own Stage 3a, or rewrite the live-verification step to bypass the `/deep:*` command surface until 003 lands. Additionally: if this live-verification step uses a `cli-claude-code` executor (a natural choice), it will hit the exact same Keychain wall documented above — **specify `cli-opencode` or plain in-session dispatch for 002's live-verification step, not `cli-claude-code`, until that gap is separately fixed.**

## 5. Advisor-corpus (003 Stage I) — mostly correct, with real field-coverage gaps

Confirmed accurate: `--emit-routing-projection` regenerates 3 marked blocks (not 2) atomically in one pass; `MERGED_DEEP_SKILL_ID` genuinely hand-authored/duplicated/unguarded in both `aliases.ts` and `skill_advisor.py` with zero automated cross-check; the divergence ratchet (`local-native-divergence-ratchet.vitest.ts`) genuinely has no auto-pass path — every ledger change requires a matching, verified entry or CI fails loudly; "skill name never appears in prompt text" verified true (0 leaks across all label fixtures).

**Gaps**: `local-native-approved-divergences.json`'s `gold` field (19/75 entries) needs updating alongside `nativeTop`/`localTop`/`reason` — currently unlisted. `tests/scorer/fixtures/intent-prompt-corpus.ts` has 2 more hardcoded `expectedSkill: 'deep-loop-workflows'` labels in a fourth fixture file, also unlisted. **Two more independent, hard-pinned regression gates exist over the same corpus that Stage I doesn't mention**: `scorer-eval-baseline-ratchet.vitest.ts` (pins exact counts + SHA-256 hashes of the label files — editing `skill_top_1` values changes the hash and will fail this gate unless `capture-scorer-eval-baseline.mjs --write` is re-run) and `python-ts-parity.vitest.ts` (hardcodes `toBe(105)`/`toBe(101)` counts plus a literal 4-id array, 2 of which are ledger rows needing the same rename). None of this will slip past Stage J's full-vitest catch-all, but Stage I as written will look "done" while two more gates still fail. Also: the plan's literal `T001` baseline-capture command (`score-routing-corpus.py`) is **currently broken independent of the merge** — `gate3-corpus-runner.mjs` imports a nonexistent path (`shared/dist/gate-3-classifier.js` doesn't exist at that resolution; the real file is one directory over, under `system-spec-kit/shared/`). A real baseline number was captured anyway via direct module invocation: `{"total": 193, "correct": 106, "accuracy": 0.5492}` (Python-side; not the same figure as the already-existing, properly-pinned `scorer-eval-baseline.json`, which shows `0.7617` for the TS-side scorer — different measurement, don't conflate them).

Two more sibling-graph bugs found, distinct from Stage G's described "collapse duplicate edge" case: `cli-opencode/graph-metadata.json` and `system-spec-kit/graph-metadata.json` both have the *same string repeated twice* in `manual.related_to` (not "one edge per old skill" — a naive rename leaves the duplication intact, needs explicit dedup). Converse risk: `system-skill-advisor/graph-metadata.json` has 4 *legitimately distinct* edges all targeting `deep-loop-workflows` (each with different `context` — routes research/review/agent-improvement/ai-council-delegation separately) — collapsing these mechanically would destroy real routing information; Stage G's rule needs scoping to "identical target with no distinguishing context."

`skill-graph.json` (a separate compiled snapshot, not `graph-metadata.json`) has `deep-loop-runtime`/`deep-loop-workflows` as mutual `depends_on`/`enhances`/`siblings` edges — post-merge these become self-loops, which need deletion, not a text substitution, and risk tripping the script's own cycle-detection.

## 7. `fallback-router.ts` (phase 004) — recommend staying deferred, and the "one call site" framing undersells the real work

Confirmed zero real callers (only its own test files). Confirmed real design gaps beyond the plan's single-bullet scope: (a) a layering violation — `fanout-pool.cjs` is an explicitly generic primitive with zero model/quota-pool awareness by design, wiring belongs in `fanout-run.cjs` instead; (b) the pool's worker arrays are fixed-size, seeded once — "re-dispatch a new lineage" requires runtime array growth that doesn't exist and isn't designed; (c) the plan hand-authors a duplicate `glm-5.2 -> mimo-v2.5-pro` registry rather than sourcing the real one that already exists in a different skill (`sk-prompt-models/assets/model_profiles.json`); (d) `resolveFallback`'s own `approvedModelIds` safety-gate parameter is dropped by the plan's call-site sketch. **Decisive evidence it's not needed for this merge**: this exact research run is the natural live test of the motivating scenario (GLM-5.2 exhausting retries) — and it never happened. All 5 glm52 replicas completed normally; the only 5 failures in the whole 20-replica run were 100% attributable to the unrelated Keychain/`cli-claude-code` gap, which a model-fallback router would not have fixed. **Recommendation: keep 004 deferred; if ever picked up, rewrite its plan first per (a)-(d).**

## 8. New, cross-cutting infra finding (not part of the merge design itself, surfaced by running this research)

3 independent replicas (2 different model families: `gpt55-fast-2`, `glm52-4`, `glm52-5`) converged on the same finding: `deep_research_auto.yaml`'s `step_detect_spec_present`/`step_writeback_spec_findings` steps mutate the root `spec.md` during pre-init and post-synthesis, but this is **not structurally gated** on the fan-out boundary (`config.fanout_lineage_artifact_dir`) — it relies on the dispatched model reading and honoring a prose instruction ("do not touch any path outside your lineage directory") rather than a code-enforced skip. **In this specific run, no harm occurred** — verified via `git status` (spec.md is untracked/new, not modified) and every lineage's own state log, which shows each one correctly self-gated (`"writeback":"deferred_lineage_boundary"`). But this is a prose-trusted safety boundary, not a code-enforced one — the same risk class as several other "silent unless someone happens to comply" findings across this whole research effort. Worth a future hardening pass on `deep-loop-runtime`/`deep-loop-workflows` itself (outside this merge packet's scope), not a blocker for this merge.
