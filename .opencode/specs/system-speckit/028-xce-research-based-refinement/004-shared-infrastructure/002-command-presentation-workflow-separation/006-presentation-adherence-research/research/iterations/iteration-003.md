# Iteration 003 (mimo-v2.5-pro)

**Summary:** The doctor/ family's high render-contract adherence correlates with three phrasing patterns absent from the memory/ family: (1) an explicit 'Presentation Boundary' section enumerating what content lives only in the presentation asset, (2) contrast phrasing ('the presentation contract, not this router') that creates an either-or mental model, and (3) a numbered execution-order step that mandates reading the presentation asset before any rendering. The memory/ family uses softer descriptive language ('use it as the display source of truth') without enumeration or contrast, which mid-tier models interpret as advisory rather than binding. Standardizing memory/ commands to match the doctor/ pattern — Presentation Boundary section, contrast phrasing, and numbered read-first step — should measurably improve render-contract adherence for dispatched mid-tier models.

## [P1][DOC-DRIFT] memory/ commands lack Presentation Boundary enumeration section
- Evidence: doctor/speckit.md:65-73, doctor/mcp.md:53-61, doctor/update.md:50-58 all have '## Presentation Boundary' sections that explicitly list what lives only in the presentation asset. memory/search.md, memory/save.md, memory/manage.md, memory/learn.md have no such section.
- Detail: Every doctor/ command file has a dedicated 'Presentation Boundary' section that enumerates the concrete content classes owned by the presentation asset (e.g., 'Sub-action menu, accepted answers, and cancellation display', 'MCP assessment, install, repair, verification, and final-report display templates'). The memory/ family omits this entirely, leaving the model to infer what counts as 'display' vs. router-owned logic. Mid-tier models need the enumeration to anchor their rendering decisions.
- Fix sketch: Add a '## Presentation Boundary' section to each memory/ command file that enumerates the content classes owned by the corresponding presentation asset, mirroring the doctor/ pattern.

## [P1][REFINEMENT] memory/ uses weaker 'use it as the display source of truth' vs doctor/'s 'not this router' contrast phrasing
- Evidence: memory/search.md:18 says 'read the presentation asset and use it as the display source of truth'. doctor/mcp.md:42 says 'Use the presentation contract, not this router, for user prompts, dashboards, result summaries, and next-step display.' doctor/update.md:36 uses identical contrast phrasing.
- Detail: The doctor/ family explicitly contrasts the two sources ('the presentation contract, not this router') which creates a clear either-or mental model for the dispatched model. The memory/ family uses a softer positive-only phrasing ('use it as the display source of truth') that lacks the negative anchor telling the model what to NOT do. Mid-tier models benefit from the explicit contrast to avoid blending router prose with presentation templates.
- Fix sketch: Rewrite the memory/ command reference lines to use contrast phrasing: 'Use the presentation asset, not this command file, for result display, dashboards, and user prompts.'

## [P2][REFINEMENT] Asset table label inconsistency: 'Presentation' vs 'Presentation source of truth'
- Evidence: memory/search.md:16 labels the asset column 'Presentation'. doctor/mcp.md:22, doctor/speckit.md:23, doctor/update.md:22, speckit/plan.md:21, speckit/resume.md:21, create/agent.md:15 all label it 'Presentation source of truth'.
- Detail: The doctor/, speckit/, and create/ families consistently label the presentation asset as 'Presentation source of truth' in their asset tables. The memory/ family uses the shorter label 'Presentation'. While minor, the 'source of truth' phrasing reinforces the authority contract. Mid-tier models may treat a bare 'Presentation' label as optional reference material rather than authoritative specification.
- Fix sketch: Change the asset table label in all memory/ command files from 'Presentation' to 'Presentation source of truth'.

## [P2][REFINEMENT] memory/search.md:47 uses vague 'Render compact, parseable output using the presentation asset' without MUST or enumeration
- Evidence: memory/search.md:47 says 'Render compact, parseable output using the presentation asset.' Compare deep/start-research-loop_presentation.md:9: 'the consolidated setup prompt MUST be the first visible response'. The deep/ family uses MUST for critical render directives; memory/search.md uses a bare imperative with no modal verb.
- Detail: The deep/ and speckit/ families use RFC-2119-style 'MUST' for critical presentation directives (e.g., 'MUST be the first visible response'). memory/search.md:47 uses a plain imperative 'Render' without a modal verb, which mid-tier models may interpret as a suggestion rather than a binding constraint. The search_presentation.md section 2 defines a specific fenced template, but the router never uses MUST-level language to direct the model to it.
- Fix sketch: Rewrite memory/search.md:47 as 'Render retrieval results using the format defined in the presentation asset section 2; do not emit free prose for result display.'

## [P2][REFINEMENT] memory/ family has no explicit step to READ the presentation asset before rendering
- Evidence: doctor/mcp.md:30 'Read .opencode/commands/doctor/assets/doctor_mcp_presentation.md.' is step 1 of Execution Order. doctor/speckit.md:43 is identical. doctor/update.md:29 is identical. create/agent.md:21 is identical. memory/search.md has no numbered step that says 'Read the presentation asset' — it appears only as a prose line at line 18.
- Detail: Every doctor/, speckit/, and create/ command includes an explicit numbered step 1 in its Execution Order that says 'Read [presentation asset path]'. The memory/ family embeds the read directive as a prose sentence ('Before asking startup questions or displaying results, read the presentation asset...'). The numbered-step pattern is more salient for mid-tier models because it appears as a mandatory sequential action rather than a background instruction.
- Fix sketch: Add a numbered '## Execution Order' or '## Workflow Routing' section to memory/ commands with step 1 being 'Read the presentation asset at [path].'

The search file does include 'Contract' but the title prefix 'Memory Search' doesn't match the command name '/memory:search', unlike doctor/ which uses '/doctor:mcp'.

The search presentation uses a descriptive name '# Memory Search Presentation Contract' that doesn't include the slash-command name. While cosmetic, mid-tier models may use the title to anchor their understanding of the asset's scope. A consistent naming convention aids pattern matching across families.
