## UX principles for this integration

- Infer, do not ask: `source_kind`, idempotency keys, audit actor, and stale state should be server-derived.
- Surface on exception, not success: normal save/search output should stay compact unless something changes the user's next action.
- Save/search hot paths must remain one-step: safety features run behind the command, not as follow-up chores.
- Diagnostics are pull-based: dashboards, `/doctor`, `includeTrace`, and `/memory:manage` carry detail.
- Advisory means advisory: duplicate, stale, and feedback signals should not block or mutate canonical truth without explicit measured promotion.
- Use plain language: "retry recognized", "saved normally", "omitted stale candidates", "audit incomplete" beats internal provenance jargon.

## Friction anti-patterns to avoid

- Asking the developer to choose `source_kind`; zero-friction alternative: infer it from tool/caller and only expose it in diagnostics.
- Requiring user-provided idempotency tokens; zero-friction alternative: server receipt from content/request fingerprint and replay on identical retry.
- Turning near-duplicates into hard rejects or review queues; zero-friction alternative: save normally and show at most one advisory hint.
- Showing feedback widgets after every search; zero-friction alternative: passive shadow event capture and dashboard-only diagnostics.
- Printing audit rows on every mutation; zero-friction alternative: append silently, warn only when audit failed.
- Making delete ask "soft delete or hard delete?"; zero-friction alternative: normal delete tombstones, retention sweep purges later.
- Hiding stale/deprecated matches without traceability; zero-friction alternative: default active results plus debug/weak-result omission diagnostics.
- Consolidating MCP tools into a generic operation dispatcher; zero-friction alternative: keep discoverable tools, add an ownership map for maintainers and `/doctor` routing.
