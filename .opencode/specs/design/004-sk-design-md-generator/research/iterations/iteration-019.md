# Iteration 019 — Track B (deepseek)

## Focus
Remaining correctness fixes (findInteraction matching, classifyShadow dedup, touch-target filtering, extractA11yAsync wiring)

## Findings
1. **[P0] B2: extractA11yAsync never wired** — Confirmed. extractA11yAsync (a11y-extract.ts:361) exports tabOrder/langAttribute/skipLinkDetected/altTextCoverage but is never imported or called in extract.ts. Only the synchronous extractA11y is called (extract.ts:445). The browser is already closed (extract.ts:422) before a11y assembly, so async capture never runs. Four fields are silently missing from A11yTokens output; the writer must invent them.
   - Recommendation: Import extractA11yAsync in extract.ts. Inside the per-page loop (extract.ts:296-336), call extractA11yAsync(page) while the Page object is alive. Merge results across pages (take langAttribute/skipLinkDetected from first page, aggregate altTextCoverage). Wire merged async fields into a11yTokens before assigning at line 469. Remove the dead-code a11y computation inside clusterTokens (cluster.ts:1506-1568) which is overwritten at extract.ts:469.
   - Evidence: a11y-extract.ts:361-391 (exported but unused), extract.ts:19 (import misses extractA11yAsync), extract.ts:422 (browser.close before a11y assembly), extract.ts:445 (only sync extractA11y called)
2. **[P0] B13: findInteraction drops real interaction data** — Confirmed. interactionLookup keys use capture.componentType (e.g., 'link', 'button', 'tab') from classifyComponent(). But findInteraction constructs the primary key as `${el.tag}|${el.className}|${el.tag}` using el.tag for the componentType slot (cluster.ts:1294). For <a> elements (tag='a', componentType='link') and role-based components (tag='div', componentType='button'), the primary lookup never matches. The linear-scan fallback (cluster.ts:1297-1301) checks el.className.includes(cap.element.classes) — this fails when the representative element has fewer classes than the captured element (common since only one element per first-class-prefix is captured). When both fail, hoverDiff/focusVisibleDiff/focusDiff/activeDiff/disabledStyle are all set to null (cluster.ts:1347-1351) despite real data existing in InteractionCapture.
   - Recommendation: Replace the key construction in findInteraction with a classifyComponent equivalent: `${el.tag}|${el.className}|${classifyByTagRole(el.tag, el.role)}` where classifyByTagRole mirrors classifyComponent from interaction-capture.ts. Remove the fragile linear-scan fallback or invert its class-substring direction to check if cap.classes overlaps el.className rather than el.className.contains(cap.classes).
   - Evidence: cluster.ts:1288 (interactionLookup key uses componentType from classifyComponent), cluster.ts:1294 (findInteraction key uses el.tag instead of componentType), interaction-capture.ts:118-126 (classifyComponent returns 'link' for <a>, 'button' for role=button), cluster.ts:1347-1351 (null fallback drops all interaction state)
3. **[P1] B14: classifyShadow duplicated with drift risk** — Confirmed. classifyShadow is defined twice in cluster.ts: once as an exported function (cluster.ts:374-396) and once as an inline function inside clusterTokens (cluster.ts:1047-1072). Both implementations are currently identical. The inline copy (line 1047) shadows the exported copy and is the one actually used for shadow token classification (line 1090). If a maintainer fixes a classification bug in the exported version, the inline copy would drift, silently leaving shadow classification broken. Zero-blur border-shadows are correctly separated from elevation because spread>0 gates the border-shadow branch, and the blind fallthrough returns 'elevation' which is harmless for zero-blur shadows.
   - Recommendation: Delete the inline classifyShadow (cluster.ts:1047-1072) and call the exported classifyShadow (cluster.ts:374) from within clusterTokens. Both accept the same parameter and return the same type. This makes the exported function the single source of truth.
   - Evidence: cluster.ts:374-396 (exported classifyShadow), cluster.ts:1047-1072 (duplicate inline classifyShadow), cluster.ts:1090 (inline version is the one actually called for shadowTokens)
4. **[P1] B16: global-min touch/font skewed by hidden elements** — Confirmed with nuance. Two independent minTouchTarget/minFontSize computations exist: one in a11y-extract.ts (lines 306-338, the one that ships to output) and one in cluster.ts (lines 1532-1568, overwritten at extract.ts:469 so dead code). The a11y-extract.ts version already has reasonable guards: minTouchTarget filters to interactive tags (button/a/input/select/textarea) and excludes zero-area elements; minFontSize requires textContent.trim(). However, both miss visibility filtering — a 1px-wide interactive element hidden via CSS can produce a misleading minimum that flags false WCAG violations. The cluster.ts dead-code version is worse: minFontSize scans all elements with positive parsePxValue without checking textContent or visibility.
   - Recommendation: Consolidate to a single computation (eliminate the dead-code cluster.ts version). In a11y-extract.ts, add visibility guards: skip elements where computed display is 'none', visibility is 'hidden', opacity is '0', or the element is positioned off-screen. For minFontSize, additionally skip font-size < 8px unless the element has an explicit aria-label. For minTouchTarget, report the most common smallest interactive size rather than the global minimum.
   - Evidence: a11y-extract.ts:306-325 (minTouchTarget filters interactive tags but not visibility), a11y-extract.ts:327-338 (minFontSize requires textContent but not visibility), cluster.ts:1532-1568 (dead-code duplicate with weaker filtering), extract.ts:469 (cluster a11yTokens overwritten by extractA11y result)
