## Findings

1. **Rank 1: mandatory candidate coverage matrix / search ledger.** Leverage: very high. Risk: medium. Affected surface: deep-review prompt pack, state format, post-dispatch validation, reducer, dashboard, synthesis report, and tests. Verification cost: medium-high. Evidence: deep-review's current iteration narrative and JSONL contract require dimension, severity, findings, traceability, and optional graph events, but no candidate-search rows [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:76] [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:81] [SOURCE: .opencode/skills/deep-review/assets/prompt_pack_iteration.md.tmpl:86]. The state schema requires active finding details but makes `ruledOut`, coverage, novelty justification, and graph events optional [SOURCE: .opencode/skills/deep-review/references/state_format.md:205] [SOURCE: .opencode/skills/deep-review/references/state_format.md:207]. Prior evidence shows explicit bug-class coverage was absent from surveyed structured finding details [SOURCE: .opencode/specs/skilled-agent-orchestration/116-deep-review-complexity/research/iterations/iteration-003.md:28]. Acceptance criteria: each non-trivial code/spec review iteration must emit `candidateMatrix` or `searchLedger` rows with `dimension`, `bugClass`, `invariant`, `producer`, `consumer`, `negativeTestSearched`, `evidenceRefs`, `disposition`, and optional `linkedFindingId`; every row disposition is one of `finding`, `ruled_out`, `deferred`, `blocked`, or `not_applicable`; clean dimensions are illegal unless at least one cited row explains what was searched and why no finding resulted.

2. **Rank 2: stricter post-dispatch schema validation.** Leverage: very high because it turns the ledger from guidance into a gate. Risk: medium-low if introduced behind target/risk thresholds. Affected surface: `.opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts`, deep-review fixtures, prompt pack examples, and state-format docs. Verification cost: medium. Evidence: the validator knows the canonical review fields [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:112], rejects missing required fields [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:366], and type-checks `filesReviewed`, `dimensions`, `findingDetails`, and `newFindingsRatio` [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:376] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:390] [SOURCE: .opencode/skills/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:397]. It does not yet validate the internal shape of `findingsNew`, rich finding details, candidate rows, graph events, or null-search evidence. Acceptance criteria: validator fails records where `findingsNew` is not `{P0,P1,P2}`, where `findingDetails` entries lack required rich fields when findings exist, and where required candidate/search-ledger rows are missing or lack cited evidence; legacy packet tolerance is explicit and versioned rather than silent.

3. **Rank 3: null-search ledger persisted through reducer, dashboard, and synthesis.** Leverage: high. Risk: medium. Affected surface: reducer ingestion/rendering, dashboard template/rendered sections, report synthesis YAML, state registry, and deep-review report examples. Verification cost: medium. Evidence: synthesis derives verdict from active severity counts [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:1088], reports active counts and scope in the executive summary [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:1171], and requires a rich active finding registry [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:1183]. Ruled-out claims and sources reviewed are relegated to the Audit Appendix without a row schema [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:1208]. The reducer-rendered dashboard surfaces severity, progress, dimension coverage, graph convergence, trend, next focus, and active risks [SOURCE: .opencode/skills/deep-review/scripts/reduce-state.cjs:1119] [SOURCE: .opencode/skills/deep-review/scripts/reduce-state.cjs:1215], but not search debt. Acceptance criteria: reducer owns a `searchDebt` / `ruledOutCandidates` rollup; dashboard shows searched, ruled-out, deferred, blocked, and unsearched candidate classes; review reports include a first-class Search Ledger section before or near the Active Finding Registry, not only in the final appendix.

4. **Rank 4: focus narrowing and target-selection ledger.** Leverage: high for broad or unfamiliar targets. Risk: low-medium. Affected surface: deep-review YAML setup/iteration focus, prompt pack state injection, agent instructions, strategy format, and tests/playbooks. Verification cost: medium. Evidence: scope discovery enumerates all files for spec, skill, agent, track, or file targets [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:227] [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:254], then dimension ordering makes correctness, security, traceability, and maintainability the primary sequence [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:258]. 5. **Rank 5: producer/consumer/invariant graph events.** Leverage: high once the candidate ledger exists. Risk: medium-high because it extends graph vocabulary and convergence semantics. Affected surface: graph event schema, coverage graph upsert/convergence handlers, reducer graph rollups, prompt examples, and graph tests. Verification cost: high. Acceptance criteria: graph accepts node kinds such as `BUG_CLASS`, `INVARIANT`, `PRODUCER`, `CONSUMER`, and `TEST`; stop blockers can name missing producer/consumer/invariant coverage; graphless environments must emit a text ledger fallback instead of silently treating absent graph evidence as clean coverage.

Risk: medium because stricter stopping can increase loop cost or block valid low-risk reviews. Affected surface: convergence reference, YAML stop logic, reducer registry, blocked-stop events, and tests. Verification cost: medium-high. Evidence: the highest-weight convergence signal is dimension coverage [SOURCE: .opencode/skills/deep-review/references/convergence.md:211], and the YAML composite vote weights dimension coverage at 0.45 [SOURCE: .opencode/commands/deep/assets/deep_start-review-loop_auto.yaml:471]. 7. **Rank 7: synthesis/report changes.** Leverage: medium-high for operator trust and follow-on planning, but insufficient without earlier state enforcement. Risk: low-medium. Affected surface: synthesis YAML, `review-report.md` contract, planning packet fields, reducer report inputs, examples. Verification cost: medium. Acceptance criteria: defaults move only after candidate/search-ledger gates exist, or are scoped to high-risk targets; tests confirm increased iterations improve candidate coverage rather than only adding more empty clean passes.

## Questions Answered

- Why can focused deep-research surface more bugs? It externalizes the search process: focus, questions, observations, edges, invariants, and ruled-out directions are first-class iteration artifacts [SOURCE: .opencode/skills/deep-research/assets/prompt_pack_iteration.md.tmpl:41] [SOURCE: .opencode/skills/deep-research/assets/prompt_pack_iteration.md.tmpl:53]. Deep-review externalizes severity reporting, active findings, dimensions, and verdicts more strongly than candidate generation or null-search proof.
- Which improvement has the best leverage? A mandatory candidate/search ledger, enforced by stricter post-dispatch validation, is the highest-leverage change because it forces pre-validation candidate generation and gives convergence, graph, dashboard, and synthesis something real to consume.
- Should producer/consumer/invariant graph events be first? Not first. They are powerful but depend on a stable candidate ledger and fallback behavior. Otherwise graphless runs remain shallow or overly blocked.
- Should higher iteration defaults be first? No. More iterations help only after the loop knows what deeper search means. Without candidate rows, extra iterations can produce more dimension bookkeeping.
- How should deep-review become less surface-level? Move fix-completeness and research-style evidence discipline earlier: each review iteration should prove what bug classes, invariants, producer/consumer paths, and negative tests were searched before it claims a clean dimension or final PASS.

## Questions Remaining

- How should legacy packets be tolerated while new packets fail closed on missing search-ledger evidence?
- Should graph expansion ship in the same packet as schema validation, or as a follow-up once candidate rows are stable?
- What migration path should update existing docs, examples, fixtures, and reducer dashboards without making old review reports unreadable?
