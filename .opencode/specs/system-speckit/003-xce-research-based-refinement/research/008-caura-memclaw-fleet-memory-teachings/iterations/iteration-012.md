## Mechanism
MemClaw treats memory quality as end-task behavior, not just retrieval overlap. Its public table reports LLM-judge accuracy on LoCoMo and LongMemEval, token savings versus full-context prompting, and warm-cache search latency (`docs/performance.md:7`, `docs/performance.md:9`, `docs/performance.md:10`, `docs/performance.md:11`). The README repeats the same framing and explicitly points methodology details to `docs/performance.md` rather than embedding the harness in this repo (`README.md:279`, `README.md:281`, `README.md:283`, `README.md:285`, `README.md:286`, `README.md:287`).

The core philosophical move is that accuracy is judged over the retrieval-then-answer pipeline: “LLM-judge over benchmark-defined questions, not `recall@k` over a fixed gold set,” because the pipeline as a whole gets product credit (`docs/performance.md:27`, `docs/performance.md:29`). Token efficiency is normalized against a “no memory system” baseline: tokens sent to the answering LLM divided by prompt plus full prior context (`docs/performance.md:30`). Latency is p50/p95 for `POST /search` under warm-cache, single-tenant load, with cold-cache caveat (`docs/performance.md:31`).

MemClaw is unusually explicit about what the numbers do not prove. It says LoCoMo and LongMemEval measure “one agent, one user, one long conversation,” and cannot evaluate fleet learning, inheritance by new agents, scoped visibility, or cross-tenant leak behavior (`docs/performance.md:13`, `docs/performance.md:33`, `docs/performance.md:35`, `docs/performance.md:37`, `docs/performance.md:38`, `docs/performance.md:39`, `docs/performance.md:40`). It also says fleet-only properties “None of this moves a recall@k number. All of it moves whether you can deploy” (`docs/performance.md:42`).

The in-repo benchmark scaffolding is mostly operational and regression-oriented, not the public LoCoMo/LongMemEval scoring harness. The raw public benchmark harness is external in `keystone-benchmark` (`docs/performance.md:68`). In-repo categories are unit, integration, and benchmark markers (`pytest.ini:7`, `pytest.ini:8`, `pytest.ini:9`, `pytest.ini:10`). The benchmark test measures scoring-formula latency only, explicitly “not the SQL query execution time” (`tests/test_benchmark_scoring.py:1`, `tests/test_benchmark_scoring.py:4`, `tests/test_benchmark_scoring.py:5`, `tests/test_benchmark_scoring.py:116`). ## Negative knowledge
MemClaw's public LoCoMo/LongMemEval headline numbers should not be imported into Spec Kit claims. The repo itself says these benchmarks are single-chatbot shaped (`docs/performance.md:13`) while MemClaw is fleet-shaped (`README.md:36`), and the raw scoring harness lives in an external `keystone-benchmark` repo, not this vendored code (`docs/performance.md:68`).

Fleet leaderboard machinery has little payoff for Spec Kit's local, single-user store. MemClaw's differentiators include tenant/fleet scopes, cross-agent outcome propagation, trust tiers, and cross-tenant leak prevention (`docs/performance.md:42`, `docs/performance.md:52`, `docs/performance.md:54`, `docs/performance.md:55`, `docs/performance.md:56`). Spec Kit should borrow metric honesty and regression discipline, not fleet-wide benchmark positioning.

Provider/model benchmark scripts are secondary for memory-quality methodology. The in-repo latency harness is useful as a shape (`scripts/latency_test.py:2`, `scripts/latency_test.py:216`), but model-provider comparisons and warm-cache service latency are less important than local retrieval quality, token injection cost, and drift/readiness in Spec Kit.
