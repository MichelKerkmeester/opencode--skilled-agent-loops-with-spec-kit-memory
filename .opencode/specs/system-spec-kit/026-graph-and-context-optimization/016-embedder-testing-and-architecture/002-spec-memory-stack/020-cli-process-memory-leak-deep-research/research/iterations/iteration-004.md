# Iteration 004: cli-X self-invocation guards, nested routing rules, and cross-CLI loop risks

## Focus
Audit the five cli-* skills (cli-claude-code, cli-codex, cli-opencode, cli-devin, cli-gemini) and the deep-loop dispatch infrastructure for self-invocation guards, nested routing rules, and cross-CLI loop risks. The question driving the iteration: can Claude call Claude, Codex call Codex, OpenCode call OpenCode (without the documented parallel-session exception), Gemini call Gemini, or a chain `claude → codex → opencode → devin → gemini` ever circle back to the originating runtime — and are the existing guards enforceable, consistent, and uniformly scoped across the five skills? Out of scope (closed in iters 001-003): the spawn inventory itself, the state-machine off-by-ones, the ai-council append-only contract violations, and improve-loop archive policy.

## Findings

### A. Self-invocation guards are documentation, not runtime enforcement

1. **cli-devin's own playbook admits the guard is LLM-honor-system.** The DV-019 manual_testing_playbook entry's "v1.0.2.0 SKIP RATIONALE (reaffirmed)" states verbatim: "the self-invocation guard runs at the calling-AI orchestrator layer (cli-devin's SKILL.md §2 pseudocode is executed by the AI loading the skill, not by the `devin` binary which has no concept of the guard). Wave-2 confirmed there is no shell-only way to test this — only an actual calling-AI session that loads cli-devin under simulated `DEVIN_*` env can demonstrate the refusal." The same model holds for the four sibling skills: `refuse(...)` is an undefined pseudo-call, `has_parallel_session_keywords(prompt)` (cli-opencode) and `has_cloud_handoff_keywords(prompt)` (cli-devin) are signatures with no concrete implementation anywhere in the repo, and `detect_self_invocation()` is Python prose in a Markdown file. ### D. Cross-CLI loop risks the per-skill guards cannot catch

9. **Each guard only checks its OWN runtime signature; a multi-hop chain never trips any single guard.** cli-claude-code's `detect_self_invocation()` checks `CLAUDECODE`, `claude` in ancestry, `~/.claude/state/<id>/lock`. cli-codex checks `CODEX_*`, `codex` in ancestry, `~/.codex/state/<id>/lock`. cli-opencode checks `OPENCODE_*`, `opencode` in ancestry, `~/.opencode/state/<id>/lock`. cli-devin checks `DEVIN_*`, `devin` in ancestry, `~/.config/devin/sessions/<id>/lock`. cli-gemini checks `GEMINI_*`, `gemini` in ancestry, `~/.gemini/state/<id>/lock`. The intersection of the five guards is empty. A chain `claude → cli-codex` (allowed: no `CODEX_*` env in Claude Code) → child codex session dispatches `cli-claude-code` (allowed: the codex child does NOT inherit `CLAUDECODE` because it is a separate runtime) → second-level Claude Code is now running underneath a Claude Code grandparent with no guard trip. Every two-hop cross-CLI loop is undetectable by design. [INFERENCE: comparison of the env / ancestry / lockfile token sets in cli-claude-code SKILL.md:65-79, cli-codex SKILL.md:62-77, cli-opencode SKILL.md:69-85, cli-devin SKILL.md:69-86, cli-gemini SKILL.md:62-77 — each lists ONLY its own provider tokens; no skill consults a shared cross-runtime registry.]

10. **The "deep-flow session" exception explicitly allows N back-to-back dispatches without re-checking the guard.** Every cli-X ALWAYS rule on single-dispatch discipline (cli-claude-code SKILL.md ALWAYS rule 10 at line 350; cli-codex ALWAYS rule 12 at line 357; cli-opencode ALWAYS rule 14 at line 296; cli-devin ALWAYS rule 15 at line 372; cli-gemini ALWAYS rule 10 at line 309) ends with the identical exception: "Within a deep-flow session (deep-review / deep-research): the operator authorizes the whole multi-iteration session at start — iterations chain back-to-back with kill-between as the safety mechanism, NOT a per-iteration operator confirmation prompt." If a child iteration leaks an env var into the next iteration's parent (e.g., a `claude` child sets `CLAUDECODE=1` on the shell that then dispatches iteration N+1), the next iteration's guard would catch it — but the exception explicitly removes any per-iteration recheck. [SOURCE: `.opencode/skills/cli-claude-code/SKILL.md:350`]; [SOURCE: `.opencode/skills/cli-codex/SKILL.md:357`]; [SOURCE: `.opencode/skills/cli-opencode/SKILL.md:296`]; [SOURCE: `.opencode/skills/cli-devin/SKILL.md:372`]; [SOURCE: `.opencode/skills/cli-gemini/SKILL.md:309`]

11. **The five "legitimate exceptions" are asymmetric and semantically inconsistent.** cli-opencode (SKILL.md:13-15, 87-95) carves out an exception when the prompt contains "parallel detached", "ablation suite", "worker farm", or "share URL" — these spawn a DIFFERENT OpenCode session id and state directory and are documented as legitimate self-from-OpenCode dispatch. cli-devin (SKILL.md:13-15, 88-96) carves out an exception when the prompt contains "cloud handoff", "hand off to cloud", or "devin cloud" — a DIFFERENT semantic (the work migrates to a separate Devin cloud sandbox; the local devin doesn't keep running). cli-codex (SKILL.md:12-16), cli-claude-code (SKILL.md:12-16), and cli-gemini (SKILL.md:12-16) have NO exception at all — they are fully closed against self-invocation. An operator who learned "parallel detached" semantics from cli-opencode and tried the same on cli-codex / cli-claude-code / cli-gemini would receive different behavior across the three: cli-opencode allows the exception, the other three refuse, and cli-devin uses a wholly different keyword family. [SOURCE: `.opencode/skills/cli-opencode/SKILL.md:13-15,87-95`]; [SOURCE: `.opencode/skills/cli-devin/SKILL.md:13-15,88-96`]; [SOURCE: `.opencode/skills/cli-codex/SKILL.md:12-16`]; [SOURCE: `.opencode/skills/cli-claude-code/SKILL.md:12-16`]; [SOURCE: `.opencode/skills/cli-gemini/SKILL.md:12-16`]

### E. Path drift and other inconsistencies that weaken the guard chain

12. **cli-claude-code SKILL.md says agents live at `.claude/agents/`; the linked `references/agent_delegation.md` says `.opencode/agents/`.** The SKILL.md routing table (line 263-264) reads "Claude Code agents in `.claude/agents/*.md` shape HOW Claude Code processes the task." But `references/agent_delegation.md:20` reads "Documents the active Claude Code agents in `.opencode/agents/`" and `:44` reads "Loads agent definition from .opencode/agents/<name>.md". A `--agent <name>` dispatch in a `claude -p` invocation could pick the wrong path depending on which doc the orchestrator consulted, and silently degrade the agent's behavioral discipline (e.g., a missing scope-lock invariant). [SOURCE: `.opencode/skills/cli-claude-code/SKILL.md:263-264`]; [SOURCE: `.opencode/skills/cli-claude-code/references/agent_delegation.md:20,44`]

13. **The `if_native` branch of the deep-research YAML dispatches `agent: deep-research` — the same LEAF agent whose body strictly forbids sub-agent dispatch (HARD BLOCK).** `spec_kit_deep-research_auto.yaml:578-583` runs `dispatch: agent: deep-research, model: opus, context_source: rendered_prompt_pack`. The deep-research agent body at `.opencode/agents/deep-research.md:45-50` ("0. ILLEGAL NESTING AND SCOPE LOCK (HARD BLOCK) ... NEVER create sub-tasks or dispatch sub-agents. NEVER use the Task tool.") forbids Task-tool nesting. The YAML's `dispatch:` mechanism is separate from the agent-internal Task tool, so the LEAF rule is not actually violated here — but if a future change wires the YAML dispatch THROUGH the Task tool (a plausible runtime consolidation), the same workflow would recursively self-dispatch the LEAF agent and immediately trip the HARD BLOCK. The boundary between "workflow dispatch" and "agent dispatch" is enforced by convention, not by a runtime contract. [SOURCE: `.opencode/commands/spec_kit/assets/spec_kit_deep-research_auto.yaml:578-583`]; [SOURCE: `.opencode/agents/deep-research.md:45-50`]

14. **The ai-council agent's depth-detection is by absence, not by presence.** `.opencode/agents/ai-council.md:42-44` reads: "If task context includes `Depth: 1` or explicit LEAF/nesting constraints, operate at Depth 1. Otherwise, operate at Depth 0." There is no runtime field that reliably carries `Depth: 1` from an orchestrator; the council infers depth from the absence of the tag. A council dispatched at apparent Depth 0 by an orchestrator that forgot to tag the context will recursively spawn Task-tool dispatches (`permission.task: allow` at line 16) — and from any of those Task dispatches, a Bash-allowed sibling could shell out to `claude -p`, `codex exec`, or `opencode run` and escape the depth-1 constraint without ever crossing the runtime self-invocation guards. (Iter-003 #16 documented depth detection itself; this iteration shows the depth bypass intersects the cli-* guard gap.) [SOURCE: `.opencode/agents/ai-council.md:42-44`]; [SOURCE: `.opencode/agents/ai-council.md:16`]

15. **`deep-research.md` agent has `bash: allow` and no prohibition against shell-launching a sibling CLI.** The HARD BLOCK invariants (line 36-41) cover sub-agent dispatch, Task tool, packet scope, and citation. They do NOT prohibit the LEAF agent from running `bash -c "claude -p ..."` directly. Combined with finding #2 (the dispatch wrapper does not check runtime), a maliciously-crafted iteration prompt could direct the LEAF deep-research agent to shell out to `claude -p` from inside a Claude Code parent — and there is no skill/agent/wrapper guard between the agent body and the recursive `claude` invocation. [SOURCE: `.opencode/agents/deep-research.md:6-22`]; [SOURCE: `.opencode/agents/deep-research.md:36-58`]

## Edge Cases
- **Ambiguous input:** "OpenCode calling OpenCode" is partially-allowed by cli-opencode's "parallel detached" exception. I treated the exception as documented-but-asymmetric (Finding 11), not as a guard hole — the operator who knows the exception keyword unlocks the legitimate use case. The asymmetric finding is that no sibling skill has the equivalent.
- **Contradictory evidence:** the YAML's `if_cli_gemini` and `if_cli_claude_code` branches contradict `ExecutorNotWiredError`'s "not yet wired" comment (Finding 7). I treat the YAML as the authoritative wiring evidence (the cli-X branches run in production deep-research runs, including the current iteration packet) and `ExecutorNotWiredError` as stale dead code.
- **Missing dependencies:** runtime-detection.ts has no OpenCode branch (Finding 6). Whether this is "missing" or "intentional" depends on whether `system-spec-kit` is intended to run inside OpenCode. The MCP server it powers DOES run inside OpenCode (per `opencode.json` MCP registration), so the absence is a real gap, not a deliberate exclusion.
- **Partial success:** the cli-devin manual_testing_playbook §07 (`001-self-invocation-refused.md`, `002-cloud-handoff-exception.md`) treats the guard as an "operator-runnable manual test, NOT a shell-automatable binary test" — which is honest about the LLM-honor-system nature but explicitly accepts that automated regression coverage is impossible. The other four skills' playbooks have similar manual-only entries; I read only DV-019 and CO-008/CO-031 directly but the pattern is consistent.

## Reflection
- What worked and why: starting from a head-to-head diff of the five SKILL.md §2 sections immediately surfaced the cli-codex / cli-gemini Bash-guard gap (Finding 4) — a one-line asymmetry that's invisible from reading any single skill in isolation. Then walking `runAuditedExecutorCommand` end-to-end and grepping for `detectRuntime` / `self-invocation` callers confirmed Finding 2 in two steps. The combination of "diff the five skills" + "trace the wrapper" was the right shape for an audit-style iteration.
- What did not work and why: I attempted to find a shared cross-CLI guard module (`cli-runtime-registry.ts` / `cross-cli-guard.ts`) and it does not exist. Time spent looking was useful in that it confirmed the negative — the unified guard is genuinely absent, not just hidden. Confirming absence required two grep passes; a single targeted grep against `runAuditedExecutorCommand` callsites would have been faster.
- What I would do differently: for iteration 005, I would start with the runtime-detection.ts callgraph (`getRecoveryApproach`, `detectCodexHookPolicy`, hook policy consumers) to map exactly which subsystems make decisions on `runtime` and would be wrong when the parent is OpenCode (Finding 6 is upstream of an unknown number of downstream defects). The current iteration documented the missing OpenCode branch but did not trace its downstream consumers.
