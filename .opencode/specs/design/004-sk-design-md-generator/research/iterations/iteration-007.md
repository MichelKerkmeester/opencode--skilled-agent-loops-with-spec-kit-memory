# Iteration 007 — Track D (deepseek)

## Focus
Adopt coverage-based token election + DTCG $type/$value provenance from design-extract

## Findings
1. **[P0] Coverage Election vs Composite Scoring** — design-extract uses a binary coverage gate: tokens appearing across >= N canonical pages are 'elected' as system tokens; single-page tokens are demoted or dropped. Our `classifyColorStability()` (cluster.ts:400-435) uses a composite 5-signal score with weighted adds/subtracts, where pageCoverage is just one of five signals (30pts max). On single-page extractions, pageCoverage is always 1.0, making the coverage signal meaningless and allowing gradient colors used only in a hero section (frequency=6, gradientColor-only) to score high enough to escape L4 classification — this is the likely L4 leakage mechanism. The raw OKLCH deltaE < 3 clustering (cluster.ts:698-724) is orthogonal: it merges near-neighbor colors but doesn't prevent leakage on its own.
   - Recommendation: Add a pre-classification HARD GATE in `clusterTokens()` (cluster.ts:569) BEFORE the stability pass at line 1640: if `totalPages >= 2` and `pagesCoverage < 0.2` (or `c.pages.size === 1` with frequency <= 5), force content layer. This gates the classifier input. Also fix single-page coverage inflation: when `totalPages === 1`, compute coverage as `frequency / totalElements` instead of pagesCoverage=1.0. File: `tool/scripts/cluster.ts`, phase: EXTRACT Phase 1 output assembly.
   - Evidence: Cluster.ts classifyColorStability signal weights (lines 400-435) + design-extract README coverage-election semantics
2. **[P0] DTCG $type/$value as Provenance Anchor** — W3C DTCG 2025.10 mandates every token as `{"$type": "...", "$value": ...}` where $type is a hard schema slot (color, dimension, shadow, typography, gradient, cubicBezier, etc.). This creates a type-lattice: a DESIGN.md claim about shadows must cite a `$type:"shadow"` token, a claim about gradients must cite a `$type:"gradient"` token. The $type is enforced at token construction time — you cannot emit a token without declaring its type. Our current flat tokens.json shape (e.g., `{hex, frequency, usedAs}`) has no type field. Our validate.ts only checks hex existence (validate.ts:57-93) — hex is an insufficient anchor because: (a) gradients aren't hex, (b) shadows have hex components but the claim is about shadow semantics, not the hex color inside, (c) two semantically distinct tokens can share a hex after OKLCH clustering. DTCG's typed slots are strictly stronger than hex-only sentinel tracing — they enable validator rules like: 'every shadow claim in DESIGN.md §6 must match a $type:"shadow" token by value hash.'
   - Recommendation: Adopt DTCG format in `tokens.json` output. Phase 1: add `$type` and `$value` to ColorToken, ShadowToken, RadiusToken, TypographyLevel, GradientInfo in `types.ts` (lines 409-475). Phase 2: in `cluster.ts` assembly (lines 1576-1643), wrap each token array with DTCG envelope: `colorTokens.map(c => ({$type: 'color', $value: c.hex, ...c}))`. Phase 3: add `checkClaimAnchoring()` to `validate.ts` that verifies every DESIGN.md prose claim about a token category (gradient, shadow, spacing) maps to an existing DTCG-typed token. The DTCG spec also defines composite tokens (a shadow token referencing a color token by $id) — this is the bridge to composite/semantic tokens.
   - Evidence: W3C DTCG 2025.10 Editor's Draft + design-extract DTCG output format + our validate.ts hex-only check (lines 57-93)
3. **[P1] Semantic/Composite Token Vocabulary** — design-extract outputs three token layers: (1) primitive (raw values: hex, px, ms), (2) semantic (role-named: `color.surface.primary`, `shadow.card.elevated`), (3) composite (multi-value tokens referencing primitives: `shadow.card.elevated` = `{offsetX, offsetY, blur, spread, color: ref.color.surface.shadow}`). This creates a CONSTRAINED VOCABULARY for the AI writer: instead of free prose like 'the shadows create a floating effect with soft edges,' the writer references `shadow.card.elevated` which has a known, verifiable $value. The vocabulary space is bounded to only elected tokens, preventing invention. Our current DESIGN.md writer has no such vocabulary — every section is free prose with only hex codes as fact anchors. The writer can invent concepts like 'the design uses a layered floating card system' without any token supporting 'layered' or 'floating.'
   - Recommendation: After adopting DTCG format (P0), add a semantic/composite token pass in `cluster.ts`. The OKLCH scales already discovered (cluster.ts:769-796) can bootstrap semantic color scales. Shadow/radius tokens can be auto-named by intensity (elevation-1, elevation-2). The key constraint: `tool/resources/writing_style_guide.md` must be updated to require every DESIGN.md design claim to cite at least one semantic token ID. File: `tool/scripts/cluster.ts` (add semantic naming pass after stability classification, ~line 1640), `tool/resources/writing_style_guide.md` (add citation mandate).
   - Evidence: design-extract multi-layer token output (primitive + semantic + composite) + our cluster.ts OKLCH scale detection (lines 769-796) showing latent scale structure ready for semantic naming
4. **[P1] DTCG Token Shape as Single Highest-Value Adoption** — The DTCG `$type`/`$value` format is the single highest-value architecture to adopt because: (1) it is standards-track (W3C Community Group, 20+ reference implementations), making our output immediately consumable by other tools (Style Dictionary, Tokens Studio, design-extract's own emitters); (2) it directly fixes our provenance problem — $type creates a hard schema anchor that validate.ts can enforce; (3) it is a prerequisite for composite/semantic tokens (composites reference primitives by $id, which DTCG standardizes); (4) it is the format design-extract already emits, aligning us with the stronger competitor; (5) the change is localized to 3 files (types.ts, cluster.ts, validate.ts) + 1 resource doc (writing_style_guide.md), with no changes to the Playwright extraction pipeline.
   - Recommendation: Concrete phase change: `tool/scripts/types.ts` — add `DTCGToken`, `DTCGTokenBundle` interfaces. `tool/scripts/cluster.ts` — add `toDTCG()` function called before writing tokens.json (extract.ts:token write path). `tool/scripts/validate.ts` — add `checkDTCGClaimAnchoring()` that cross-references DESIGN.md token-type mentions against the DTCG-typed token index. `tool/resources/writing_style_guide.md` — mandate citation format: '`shadow.card.elevated` ($type: shadow, $id: shadow-3)'. This is a Phase 1 (EXTRACT) + Phase 3 (VALIDATE) change. No Phase 2 (WRITE) changes needed if the writer already works from tokens.json — it just gets richer data.
   - Evidence: W3C DTCG 2025.10 spec structure + our types.ts (lines 409-475) showing current flat token shapes ready for DTCG wrapping + validate.ts (lines 318-350) showing extension point for claim anchoring

## Questions Answered
- Coverage election (binary site-wide share threshold) IS more robust against single-page one-off leakage than our composite scoring because it's a HARD GATE: a token appearing on 1 page never reaches 'system' regardless of frequency. Our 5-signal composite can be gamed on single-page extractions where pageCoverage=1.0. Slot it as a PRE-FILTER before classifyColorStability() in cluster.ts:569-843.
- DTCG $type/$value IS a substantially stronger claims-to-token binding than hex-only sentinels. A hex check (validate.ts:57-93) treats all color usages identically regardless of semantic role. A typed token anchors every DESIGN.md claim to a specific $type slot — a 'gradient-depth' claim becomes unprovable if no $type:"gradient" token exists, enforced at validation time rather than relying on the writer's discipline.
- Composite/semantic tokens give the writer a bounded, machine-verifiable vocabulary. Instead of free prose inventing concepts like 'layered floating cards,' the writer must cite `shadow.card.elevated` (a composite referencing primitive colors). This eliminates the largest source of unverifiable claims.

## Summary
design-extract's coverage-based election is a harder anti-leakage gate than our composite scoring (especially on single-page extractions where our pageCoverage signal inflates). DTCG $type/$value format is the highest-value adoption: it creates a provenance lattice where every DESIGN.md claim can cite a typed token slot, making claims provable or unprovable at validation time — directly fixing our weak claim-provenance. Adopt DTCG token shape in types.ts/cluster.ts, add pre-classification coverage gate, and add semantic/composite naming as a follow-on layer.
