### Q1: Self-maintaining memory-index move-reconciliation under rapid concurrent spec-folder moves across sessions

- Findings:
  1. The handler entry point is `memory_index_scan`, gated by an atomic in-DB scan lease stored in the `config` table under keys `last_index_scan` and `scan_started_at`. Evidence: `.opencode/skills/system-spec-kit/mcp_server/handlers/memory-index.ts:351-357` calls `acquireIndexScanLease({ now, cooldownMs: INDEX_SCAN_COOLDOWN })` before any indexing. [HIGH]
  2. The scan lease is itself a single-DB-row reservation performed inside `db.transaction(...)` (`reserveLeaseTx`), so the "check + set" of `scan_started_at` is atomic at the SQLite level. Evidence: `.opencode/skills/system-spec-kit/mcp_server/core/db-state.ts:442-502`. [HIGH]
  3. SQLite lock/contention errors are detected by `isSqliteContentionError` (codes `SQLITE_BUSY`/`SQLITE_LOCKED` or message `database is locked` / `database table is locked`) and the lease acquisition **fails CLOSED** (returns `acquired:false, reason:'contention'`) so two callers cannot both believe they hold the scan lease. Evidence: `.opencode/skills/system-spec-kit/mcp_server/core/db-state.ts:387-401` and `:505-522`. [HIGH]
  4. The default scan lease TTL is `INDEX_SCAN_COOLDOWN * 2` = `30_000 * 2 = 60_000 ms` (env override `SPECKIT_INDEX_SCAN_LEASE_EXPIRY_MS`). Evidence: `.opencode/skills/system-spec-kit/mcp_server/core/db-state.ts:89` (`const DEFAULT_SCAN_LEASE_EXPIRY_MS = INDEX_SCAN_COOLDOWN * 2;`) and `.opencode/skills/system-spec-kit/mcp_server/core/config.ts:126` (`INDEX_SCAN_COOLDOWN = 30000`). [HIGH]
  5. The lease TTL is **refreshed by a heartbeat** set inside the same handler: `leaseHeartbeatMs = Math.max(10000, Math.floor(leaseExpiryMs / 3))` with `leaseHeartbeat.unref?.()` so it never holds the event loop open. Evidence: `.opencode/skills/system-spec-kit/mcp_server/handlers/memory-index.ts:400-408`. [HIGH]
  6. The `refreshScanLease` writer overwrites `scan_started_at` and is best-effort (try/catch swallows errors). Evidence: `.opencode/skills/system-spec-kit/mcp_server/core/db-state.ts:543-552`. [HIGH]
  7. Move reconciliation runs **only after the lease is held** and before the delete+reindex cycle, and is gated by an explicit precondition `if (filesToDelete.length > 0 && filesToIndex.length > 0)`. Evidence: `.opencode/skills/system-spec-kit/mcp_server/handlers/memory-index.ts:688-700`. [HIGH]
  8. The reconciliation algorithm pairs a vanished `oldPath` with a new `newPath` by matching **grandparent dir + basename** (sibling folder rename only). It then re-validates uniqueness against the live `memory_index` and cross-checks `document_type` to prevent a mismatched repoint. Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:561-591`. [HIGH]
  9. The matched row is updated in place via a single `UPDATE memory_index SET file_path=?, canonical_file_path=?, spec_folder=?, file_mtime_ms=?, updated_at=datetime('now') WHERE id=? AND embedding_status != 'failed'`, and only counted as reconciled if `updated.changes === 1`. Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:603-614`. [HIGH]
  10. **The SELECT (uniqueness guard) and UPDATE (repoint) inside `reconcileMoves` are NOT wrapped in a `db.transaction(...)`**; each prepared statement runs as its own auto-commit. Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/incremental-index.ts:581` (`db!.prepare(selectSql).all(...)`) and `:603` (`db!.prepare(...).run(...)`) — no surrounding `db.transaction(...)` and the for-loop body executes both. [HIGH]
  11. The mutation is **NOT issued through `runInTransaction`** (the centralized transaction wrapper in `transaction-manager.ts:146-167`), so a process-level failure between the two statements would leave a half-reconciled state. Evidence: search for `runInTransaction` inside `incremental-index.ts` returns no matches. [HIGH]
  12. The scan-level race is serialised by the scan lease (Q1 #1, #2), so **concurrent `memory_index_scan` calls** in the same DB are coalesced. However, **a parallel launcher process against the SAME database is NOT excluded by the scan lease** — the lease is per-DB-row, not per-process. A second launcher invoking `memory_index_scan` would still call `acquireIndexScanLease`, so the lease does block it; the gap is the in-place UPDATE outside a transaction (claim #10), which can be torn by an unrelated writer in between. [MEDIUM]
  13. The post-scan **alias-divergence reconcile** runs `runDivergenceReconcileHooks`, which itself is wrapped in `runInTransaction` for the per-path retry/escalation ledger writes (using `json_extract(decision_meta, '$.normalizedPath')` for O(1) COUNT and a `DEFAULT_DIVERGENCE_RECONCILE_MAX_RETRIES = 3` cap with a deduplicated escalation entry). Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/mutation-ledger.ts:257-323` and `:360-441`. [HIGH]
  14. The mutation-ledger uses append-only `prevent_ledger_update` / `prevent_ledger_delete` triggers to guarantee retry history is tamper-evident. Evidence: `.opencode/skills/system-spec-kit/mcp_server/lib/storage/mutation-ledger.ts:244-251`. [HIGH]
  15. **The divergence-reconcile path is a `runInTransaction`**, but the **reconcileMoves path is NOT** — i.e. the most safety-critical path (folder-rename repointing that updates `file_path`/`canonical_file_path`/`spec_folder`) is the one path missing the transaction wrapper. Evidence: same lines as #10/#13. [HIGH]
  16. Rapid concurrent spec-folder moves: under normal conditions the scan lease (`#1`) serialises them, but a crash in the loop between `SELECT` and `UPDATE` (`#10`) leaves the row pointing at the old path while the filesystem carries the new path; the next scan would see `filesToDelete` empty and `filesToIndex` populated, and the loop body would skip without re-issuing a reconciliation because the old path is no longer in `filesToDelete`. The row is then stuck on a dead path until manual `memory_crud_delete` + reindex. [LOW — only observed via code path; no stress test result found]

- Verdict: **Largely correct under the scan-lease invariant, but the SELECT/UPDATE pair inside `reconcileMoves` is un-transactioned** — the single most important race window in move reconciliation. Concurrent scans are serialised; concurrent *non-scan* writers (async-embedding flush, an un-leased hand-coded index update) can tear the SELECT/UPDATE and leave a half-reconciled row. The alias-divergence path is properly wrapped via `runInTransaction`, so the divergence case is fine; the rename case is not. [MEDIUM-HIGH overall confidence on the structural claim; the crash-window severity (#16) is [LOW] because no failing case is reproduced in the source.]

### Q2: Shared-daemon lease model — ungraceful owner death + secondary re-election latency

- Findings:
  1. The launcher writes an exclusive, fsync'd owner lease file `OWNER_LEASE_FILE_NAME = '.spec-memory-owner.json'` via `writeOwnerLeaseFileExclusive` (`fs.openSync(path, 'wx', 0o600)` + `fs.fsyncSync(fd)`) so two launchers cannot both create the lease atomically. Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:85` (constant), `:298-312` (`writeOwnerLeaseFileExclusive`). [HIGH]
  2. The owner lease includes `ownerPid`, `ppid`, `executablePath`, `startedAtIso`, `lastHeartbeatIso`, and `ttlMs: 60000`. Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:314-324` (`buildOwnerLease`). [HIGH]
  3. The default heartbeat interval is `Math.max(1000, Math.floor(ttlMs / 2))` = **30 s** for the default `ttlMs = 60000`. Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:427-443` (`startOwnerLeaseHeartbeat`). [HIGH]
  4. A lease is classified stale by **`classifyOwnerLease`** along three branches: `stale-pid` (process dead via `process.kill(pid,0) => ESRCH`), `ppid-1-orphan` (expected ppid mismatch + reparented to PID 1), and `stale-heartbeat-reclaim` (`Date.now() - heartbeat > ttlMs * 2` = **120 s without heartbeat**). Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:346-363`. [HIGH]
  5. `acquireOwnerLeaseFile` does a "reclaim and re-read" double-check after winning the exclusive open — `writeOwnerLeaseFile(lease)` then `readOwnerLeaseFile(...)` — so two launchers racing a stale lease cannot both believe they own it. Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:390-403`. [HIGH]
  6. The heartbeat timer is owned by the lease-holding launcher; on `refreshOwnerLeaseFile` failure it calls `shutdownLauncherForSignal('SIGTERM')` then `process.exit(128)`, and `process.on('exit', clearAllLeaseFiles)` is registered so a graceful shutdown always unlinks the owner lease. Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:432-441` and `:1336`. [HIGH]
  7. `process.on('exit')` does **NOT run on SIGKILL** or hard segfaults that bypass Node's exit handlers. A SIGKILL'd owner therefore **leaves the lease file with a dead `ownerPid` and no last-heartbeat update**; the file is only cleaned when the *next* launcher calls `acquireOwnerLeaseFile` and hits the `stale-pid` branch in `classifyOwnerLease` (line 348). [MEDIUM]
  8. The shared-daemon lease (separate from the owner lease) is a JSON file at `leasePath()` containing `{ pid, childPid, startedAt, socketPath, modelServerPid }`; `leaseHeldFromFile` classifies it as held if `process.kill(pid,0)` returns OK, and `staleReclaimable:true` on `ESRCH`. Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:478-507`. [HIGH]
  9. Secondary (non-lease-holder) launchers do **not poll the lease file**; they detect the lease-holder's death transitively through the daemon socket. The session proxy enters `handleBackendFailure` on socket close, then `reattach()` runs **at most 40 attempts** with backoff ladder `PROBE_BACKOFF_MS = [100, 250, 500, 1000, 1500]`, capped at the last value. Evidence: `.opencode/bin/lib/launcher-session-proxy.cjs:11-12` (constants) and `:693-721` (reattach loop), and a comment confirming the bounded total: *“a dead/booting daemon surfaces a retryable -32001 to the client in ~tens of seconds (~41s with the backoff ladder).”* (`.opencode/bin/lib/launcher-session-proxy.cjs:13-17`). [HIGH]
  10. The `reattach` loop gives up at `state = 'CLOSED'` after `failPendingAndEnd()` and the secondary cannot recover without a new launcher process. Evidence: `.opencode/bin/lib/launcher-session-proxy.cjs:714-721`. [HIGH]
  11. **Re-election latency for secondary launchers** = sum of `PROBE_BACKOFF_MS` over 40 attempts = `100+250+500+1000+1500+1500*35` ≈ **55 350 ms** in the pathological case, matching the ~41 s comment when early exits happen. The secondary does **not** independently reclaim the lease — it only waits for *some* new launcher to appear (a fresh `mk-spec-memory-launcher.cjs` invocation) and then bridge through the new owner via the proxy. Evidence: same lines as #9/#10. [HIGH]
  12. The reclaimer must first `acquireOwnerLeaseFile` (which does the heartbeat-2-TTL stale check from #4), then `writeLeaseFile` (the shared lease), then `reprobe` to detect lost-lease races (`writeLeaseHeldJsonRpcError(...,'lease-reprobe-lost')` and `process.exit(0)`). Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:1389-1396`. [HIGH]
  13. **Failure-mode: owner SIGKILL with daemon still alive.** The `process.on('exit')` handler does not run, the lease file stays at `ownerPid=<dead>`, and the daemon (a child) may or may not survive depending on the signal's effect on descendants. A new launcher run will reclaim via `stale-pid` (instant), but **existing secondary launchers only discover the death via the daemon socket death** and have to ride the 40-attempt reattach ladder before failing. If the daemon child survived the SIGKILL of the parent (e.g. SIGKILL on the launcher only, daemon reparented to PID 1), secondary launchers attached to the now-orphaned daemon may keep issuing writes to a daemon that is no longer represented by any lease holder — a **split-brain** window. Evidence: synthesis of #4, #7, #8, #9, #10. [MEDIUM]
  14. The owner lease has an env flag `MK_SPEC_MEMORY_STRICT_SINGLE_WRITER` (default true) that, when disabled, **skips the lease check entirely** — `log('MK_SPEC_MEMORY_STRICT_SINGLE_WRITER is disabled; skipping lease check')`. Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:1339-1372`. [HIGH]
  15. A second lease **outside the strict single-writer path** (the `legacyLeaseResult` check on line 1342) routes secondary launchers through `bridgeOrReportLeaseHeldAndExit` which can return `-32001` (retryable) JSON-RPC errors if the bridge is unavailable. Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:515-525` and `:1343-1346`. [HIGH]
  16. There is also a `respawnAfterDeadSocket` path that reaps the lease child on a confirmed-dead socket (respawn lock acquired via `acquireRespawnLockFile`); it gates on the lease containing a `childPid`. Evidence: `.opencode/bin/mk-spec-memory-launcher.cjs:671-740`. [HIGH]

- Verdict: **Two distinct layers of stale-lease detection work, but they are layered, not redundant.** Owner-lease staleness is detected on next-launcher-start (instant) and the heartbeat-2×TTL window (120 s) catches abandoned-but-live owners. Secondary launchers do *not* poll the owner lease; they only react to the daemon socket death, so a secondary's re-election latency is bounded by the **40-attempt session-proxy reattach ladder ≈ 41–55 s** in the worst case, with no automatic takeover. The principal ungraceful-death exposure is the **SIGKILL-on-owner-with-orphan-daemon** case, which can leave secondaries attached to a daemon whose lease is stale until the next launcher invocation. [HIGH confidence on the static behaviour; the split-brain claim is [MEDIUM] because it requires a specific signal/parent-death propagation to manifest.]
