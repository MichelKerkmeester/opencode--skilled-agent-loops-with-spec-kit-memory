## Review: vector shard attach/rename path — search resilience (inode integrity)

I traced the real attach/detach/rename flow across three rename sites and the single shared connection model.

**Connection model.** The daemon uses one shared `db` singleton keyed by resolved path (`vector-index-store.ts:1000`, `db_connections` map). All read paths (`getDb()`) and the reindex worker (`jobDb`) resolve to the same handle, and SQLite calls are synchronous/single-threaded — so no query interleaves with a `fs.renameSync`. The integrity of the "live attachment vs rename" window therefore rests entirely on **detach-before-rename discipline**, because the only attachment-identity check in the codebase is path-string based.

**`get_attached_vector_path` is path-string only (no inode compare).** `vector-index-store.ts:460-463` returns the `file` column from `PRAGMA database_list`, path-resolved. There is no inode/file-identity comparison anywhere in the module. After a `rename(2)` swaps a new file into the active shard path, an attachment that stays live keeps the *orphaned old inode* open while `PRAGMA database_list` still reports the *same path string* — so the path-string check (`attachedPath === path.resolve(shardPath)`) gives a false "already correctly attached" verdict (`attachActiveVectorShard` fast-path at `:1562-1605`).

**The two genuinely-live rename sites are mostly disciplined:**
- In-process checkpoint restore goes through `reopenActiveDatabase` (`:2296-2308`), which `detachActiveVectorShard` **and** `close_db({retainLocks:true})` *before* `swapFn()` runs the renames — connection is fully closed, no live inode. Safe.
- Reindex swap (`reindex.ts:649-680`) detaches before the rename, then re-attaches the fresh inode via the slow path at `:737`. Safe **on the happy path**.

**The gap (my finding):** the reindex detach is wrapped in a blanket catch (`reindex.ts:663-667`) whose comment asserts "Not attached on this connection; the rename is still safe" — treating *every* throw as the benign not-attached case. `detachActiveVectorShard` (`vector-index-store.ts:1664-1674`) execs `DETACH DATABASE active_vec`, which can also throw when the schema is busy/locked, leaving the attachment **live**. If that ever happens, the code renames over the path with the connection still bound to the old inode, and the re-attach at `:737` hits the path-string fast-path (`:1562`), runs the integrity probe against the *orphaned (but valid) old shard*, passes, and concludes "already attached" — silently serving stale pre-reindex vectors until the next restart, even though the job reports success and flips the active-embedder pointer. There is no inode backstop to catch this.

This is low-probability (in the single-writer staging model a `DETACH` at swap time normally succeeds with no open cursor on `active_vec`), so I rate it **P2 / defensive** rather than P1 — no demonstrated trigger, but the swallowed catch encodes a false safety invariant and the angle's expected inode-compare backstop is genuinely absent. Everything else on this angle is sound.

Verdict: **CONDITIONAL** — one real-but-low-severity hardening finding; core detach-before-rename design is correct.
