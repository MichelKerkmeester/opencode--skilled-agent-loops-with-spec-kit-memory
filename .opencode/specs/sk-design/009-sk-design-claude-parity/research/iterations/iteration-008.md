## Findings

1. **Use a two-layer benchmark: deterministic routing baseline plus live/manual feel gates.** The existing benchmark already separates offline router replay from live model dispatch: router mode replays `hub-router.json` and `mode-registry.json`, while live mode grades advisor behavior, observed activation, usefulness, and browser-class scenarios. A parity refactor should pass the frozen router baseline first, then add live/manual Claude-feel scenarios rather than replacing the benchmark harness. [SOURCE: .opencode/skills/sk-design/benchmark/README.md:18] [SOURCE: .opencode/skills/sk-design/benchmark/README.md:20] [SOURCE: .opencode/skills/sk-design/benchmark/README.md:21]
2. **Golden prompts should assert procedure selection, packet loading, and outcome quality.** The current playbook already requires each scenario to name the realistic request, expected mode, expected resource paths, advisor behavior, pass/fail criteria, and failure triage. For Claude parity, each new golden prompt should add expected private procedure cards and proof gates on top of the existing mode assertions. [SOURCE: .opencode/skills/sk-design/manual_testing_playbook/manual_testing_playbook.md:47] [SOURCE: .opencode/skills/sk-design/manual_testing_playbook/manual_testing_playbook.md:49] [SOURCE: .opencode/skills/sk-design/manual_testing_playbook/manual_testing_playbook.md:53]
3. **Evidence gates must remain context-bound and proof-field based.** The shared context contract requires a context manifest before design/build decisions and says no palette, layout, motion, copy, accessibility, score, release, or readiness claim passes until the files behind the claim are named as loaded. Acceptance tests should therefore fail any parity run that produces design recommendations without a loaded-context manifest and final proof fields. [SOURCE: .opencode/skills/sk-design/shared/context_loading_contract.md:44] [SOURCE: .opencode/skills/sk-design/shared/context_loading_contract.md:46] [SOURCE: .opencode/skills/sk-design/shared/context_loading_contract.md:65]
4. **Anti-slop and polish checks need explicit lane coverage, not a single subjective score.** Claude's polish pass requires accessibility, AI-slop, hierarchy/rhythm, and interaction-state reviews, then aggregation/dedup/prioritization; its AI-slop procedure lists concrete failure patterns such as gratuitous gradients, decorative emoji, default left-border cards, generic fonts, pure white/black, random colors, random spacing, and warm-editorial house style. A practical benchmark should require each lane to be present with severity/confidence and concrete evidence. [SOURCE: .opencode/specs/sk-design/009-sk-design-claude-parity/external/claude/skills/polish-pass.md:22] [SOURCE: .opencode/specs/sk-design/009-sk-design-claude-parity/external/claude/skills/polish-pass.md:71] [SOURCE: .opencode/specs/sk-design/009-sk-design-claude-parity/external/claude/skills/ai-slop-check.md:25] [SOURCE: .opencode/specs/sk-design/009-sk-design-claude-parity/external/claude/skills/ai-slop-check.md:110]
5. **`design-md-generator` preservation requires positive and negative controls.** Its current playbook treats the live extract-write-validate pipeline, real `tokens.json`, v3 Style Reference output, verbatim values, validation output, and no-fabrication rule as non-negotiable. Parity tests must prove the refactor still routes URL extraction to `md-generator`, writes only sandboxed outputs, validates against tokens, catches planted phantom hex/Quick-Start fabrications, and never downgrades to generic Claude token notes. [SOURCE: .opencode/skills/sk-design/design-md-generator/manual_testing_playbook/manual_testing_playbook.md:9] [SOURCE: .opencode/skills/sk-design/design-md-generator/manual_testing_playbook/manual_testing_playbook.md:11] [SOURCE: .opencode/skills/sk-design/design-md-generator/manual_testing_playbook/manual_testing_playbook.md:125] [SOURCE: .opencode/skills/sk-design/design-md-generator/manual_testing_playbook/manual_testing_playbook.md:216]
6. **Routing/advisor invariants are release blockers, not nice-to-have checks.** The parent hub promises one public advisor-routable `sk-design` identity, five modes, registry-driven routing, smallest-useful-mode defaults, and no per-mode logic in the hub. The registry also defines read-only tool surfaces for the four reference-base modes and mutating permissions only for `md-generator`; parity is invalid if private Claude-like procedures create new public identities, bypass the registry, or grant write/Bash access to guidance modes. ## Ruled Out

- **Pure vibe review as a benchmark:** ruled out because existing benchmark/playbook contracts require exact prompts, advisor outputs, mode resolution, resources loaded, tools used, and verdict evidence. [SOURCE: .opencode/skills/sk-design/manual_testing_playbook/manual_testing_playbook.md:70]
- **Replacing current benchmark assets:** ruled out because the frozen baseline is the comparison anchor, and new baselines should be sibling folders rather than overwriting `baseline/`. [SOURCE: .opencode/skills/sk-design/benchmark/README.md:49]
- **Treating Claude parallel agents as the acceptance criterion:** ruled out for OpenCode parity because the testable invariant is lane coverage and proof fields; the external material itself has a Codex variant that runs reviews sequentially when no verifier subagent exists.
