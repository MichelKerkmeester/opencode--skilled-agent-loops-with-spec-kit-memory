## Questions Answered

- Why can focused deep-research bug-finding surface more bugs than deep-review? Because it preserves the bug-search process itself as state: narrow focus, unanswered questions, observations, edges, invariants, and ruled-out directions. Deep-review currently preserves the review report and active finding lifecycle more strongly than it preserves candidate generation or null-search evidence.
- Should the first implementation change be schema validation, candidate ledger, graph events, or prompt wording? Candidate ledger plus prompt-visible contract first, post-dispatch validation immediately behind it. Graph and convergence need stable ledger semantics to consume.
- Should `candidateMatrix` and `searchLedger` both survive? Choose one canonical field now: `searchLedger`. Allow `candidateMatrix` as a temporary migration alias for one schema version if needed, but final docs and tests should converge on `searchLedger` because it covers findings, ruled-out rows, deferred rows, blocked rows, and not-applicable rows.
- How should enforcement be scoped? Enforce for new `reviewDepthSchemaVersion >= 1` records on non-trivial code/spec/skill/agent/track/file reviews. Low-risk documentation-only or metadata-only reviews can use a lighter ledger, but a clean PASS still needs at least cited null-search rows or an explicit `not_applicable` rationale.
- Should graph expansion ship in the same packet as schema enforcement? Not as the first blocking change. Add text/JSON ledger enforcement first; then extend graph node kinds and candidate-aware stop blockers once row semantics and graphless fallback are proven.
- What legacy policy is safest? Preserve old packets as readable and warn-only. Fail closed only for new schema-versioned records or new command runs that opt into `reviewDepthSchemaVersion >= 1`.

## Questions Remaining

- Exact thresholds still need implementation-time calibration: minimum ledger rows per dimension should derive from `targetSelection` risk categories, not a fixed universal count.
- The candidate taxonomy should start conservative and evolve from seeded-bug tests: contract/data-flow, boundary/negative-test, producer/consumer, invariant, security sink/source, resource-map traceability, and graph hotspot.
- The final synthesis should decide whether the first implementation packet includes reducer/report persistence or stops after prompt/schema/validator work. My recommendation is to include reducer/dashboard visibility in the same packet if scope allows, because validator-only enforcement creates state that operators still cannot inspect easily.
