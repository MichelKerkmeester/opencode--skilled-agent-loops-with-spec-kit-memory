## Mechanism
MemClaw makes writes safe with layered gates, not a single global “save” primitive. The `Memory` row records tenant/fleet/agent scope, `source_uri`, `run_id`, metadata, content hash, `client_request_id`, soft-delete state, status, visibility, and supersession fields (`common/models/memory.py:19`, `common/models/memory.py:26`, `common/models/memory.py:33`, `common/models/memory.py:39`, `common/models/memory.py:41`, `common/models/memory.py:57`). The README frames the pipeline as production fleet memory with governance, tenant isolation, write-time visibility stamping, and full audit logging (`README.md:253`, `README.md:254`, `README.md:256`).

Dedup is multi-tier. Exact dedup computes a content hash over tenant/fleet/content (`core-api/src/core_api/pipeline/steps/write/compute_content_hash.py:24`) and rejects exact duplicates scoped to tenant/fleet/agent (`core-api/src/core_api/pipeline/steps/write/check_exact_duplicate.py:1`, `core-api/src/core_api/pipeline/steps/write/check_exact_duplicate.py:23`). Storage confirms that agent-scoped dedup is deliberate so identical content from different agents can both persist (`core-storage-api/src/core_storage_api/services/postgres_service.py:604`). Semantic dedup uses similarity bands: high similarity auto-rejects, middle band calls an LLM judge, and low-confidence duplicate judgments are accepted but queued for human review (`core-api/src/core_api/pipeline/steps/write/check_semantic_duplicate.py:7`, `core-api/src/core_api/pipeline/steps/write/check_semantic_duplicate.py:19`, `core-api/src/core_api/pipeline/steps/write/check_semantic_duplicate.py:193`, `core-api/src/core_api/pipeline/steps/write/check_semantic_duplicate.py:212`). The review queue snapshots both new and candidate content even when rejected writes never persist (`common/models/dedup_review.py:7`, `core-storage-api/src/core_storage_api/services/postgres_service.py:723`). Fast write mode does not reject near-duplicates; it stores advisory metadata like `near_duplicate_of` and lets callers decide (`core-api/src/core_api/pipeline/compositions/write.py:51`, `core-api/src/core_api/pipeline/steps/write/detect_near_duplicate.py:25`, `core-api/src/core_api/pipeline/steps/write/detect_near_duplicate.py:106`).

Idempotency is split into response-level replay and row-level bulk retry safety. Single writes can use `Idempotency-Key` from a header or metadata fallback (`core-api/src/core_api/routes/memories.py:868`, `core-api/src/core_api/routes/memories.py:880`, `core-api/src/core_api/middleware/idempotency.py:54`). The idempotency row stores `(tenant_id, idempotency_key)`, request hash, response body, status code, expiry, and `is_pending` (`common/models/idempotency.py:30`, `common/models/idempotency.py:42`, `common/models/idempotency.py:53`). Reusing a key with a different body is a 422 conflict (`core-api/src/core_api/middleware/idempotency.py:139`, `core-api/src/core_api/middleware/idempotency.py:198`). Concurrent same-key requests are handled by claim-then-poll with a pending sentinel (`core-api/src/core_api/middleware/idempotency.py:158`, `core-storage-api/src/core_storage_api/services/postgres_service.py:4566`). Bulk writes require `X-Bulk-Attempt-Id`; per-item ids become `f"{attempt_id}:{index}"`, and retries resolve as `duplicate_attempt` instead of duplicating (`README.md:587`, `core-api/src/core_api/routes/memories.py:991`, `core-api/src/core_api/services/memory_service.py:998`, `core-api/src/core_api/services/memory_service.py:1123`). Storage enforces that with a partial unique index on `(tenant_id, COALESCE(fleet_id,''), client_request_id)` and `ON CONFLICT DO NOTHING`, then re-reads canonical ids for conflicted rows (`common/models/memory.py:96`, `core-storage-api/src/core_storage_api/services/postgres_service.py:282`, `core-storage-api/src/core_storage_api/services/postgres_service.py:344`, `core-storage-api/src/core_storage_api/services/postgres_service.py:376`).

Provenance is mostly scope and source metadata, not a rich `created_by` field on memory rows. Memory provenance includes `tenant_id`, `fleet_id`, `agent_id`, `source_uri`, `run_id`, and metadata (`common/models/memory.py:19`, `common/models/memory.py:26`, `core-api/src/core_api/pipeline/steps/write/write_memory_row.py:50`, `core-api/src/core_api/pipeline/steps/write/write_memory_row.py:58`). Writes emit audit rows with tenant, agent, action, resource type/id, detail, and timestamp (`common/models/audit.py:17`, `core-api/src/core_api/pipeline/steps/write/write_memory_row.py:105`, `core-api/src/core_api/services/memory_service.py:1396`). Bulk audit detail also records `source: "bulk"` (`core-api/src/core_api/services/memory_service.py:1420`). Keystone governance rules add an optional `author_user_id` and audit it (`core-api/src/core_api/mcp_server.py:2185`, `core-api/src/core_api/mcp_server.py:2388`, `core-storage-api/src/core_storage_api/routers/keystones.py:121`).

Protection of manually curated or caller-provided data exists, but it is selective. Enrichment fills gaps only; agent-provided `memory_type`, `weight`, status, and temporal fields win over LLM-derived values (`core-api/src/core_api/pipeline/steps/write/merge_enrichment_fields.py:28`, `core-api/src/core_api/pipeline/steps/write/merge_enrichment_fields.py:58`). Deferred enrichment snapshots agent-provided fields so a worker redelivery cannot downgrade explicit values (`core-api/src/core_api/services/memory_service.py:1572`, `core-api/src/core_api/services/memory_service.py:1612`, `core-api/src/core_api/services/memory_service.py:1656`). Metadata PATCH defaults to top-level merge instead of wholesale replace, with explicit `metadata_mode="replace"` required for full overwrite (`core-api/src/core_api/routes/memories.py:1356`, `core-api/src/core_api/services/memory_service.py:2359`, `core-storage-api/src/core_storage_api/services/postgres_service.py:508`). However, top-level memory columns are explicitly last-write-wins with no optimistic lock (`core-api/src/core_api/routes/memories.py:1356`). Keystone rules are stronger: system-managed collections reject public document writes (`core-storage-api/src/core_storage_api/services/postgres_service.py:3672`), and the MCP keystone setter checks existing and new scope so a low-trust agent cannot overwrite a broader governance rule by narrowing the incoming shape (`core-api/src/core_api/mcp_server.py:2298`, `core-api/src/core_api/mcp_server.py:2331`).

## Negative knowledge
Fleet tenant/fleet/agent scoping should not be copied as a local architecture primitive. MemClaw’s memory model and dedup indexes are built around `tenant_id`, nullable `fleet_id`, and `agent_id` (`common/models/memory.py:19`, `common/models/memory.py:86`, `common/models/memory.py:96`). Spec Kit Memory is single-user/single-tenant, so those dimensions should collapse to simpler provenance and source-kind fields.

Postgres-specific conflict machinery should not be copied literally. MemClaw relies on `pgvector`, PostgreSQL JSONB, partial unique indexes with `COALESCE(fleet_id,'')`, and `INSERT ... ON CONFLICT` (`common/models/memory.py:4`, `common/models/memory.py:96`, `core-storage-api/src/core_storage_api/services/postgres_service.py:344`). SQLite can implement equivalent guarantees with unique indexes, transactions, and `INSERT OR IGNORE`/`ON CONFLICT`, but not by copying the Postgres shape.

Tenant concurrency slots, rate limits, trust levels, install credentials, and cross-tenant audit are fleet controls, not local-memory controls. MemClaw’s routes enforce tenant, usage limits, trust, fleet writes, and per-tenant slots around writes (`core-api/src/core_api/routes/memories.py:870`, `core-api/src/core_api/routes/memories.py:902`, `core-api/src/core_api/routes/memories.py:916`). Spec Kit Memory should avoid importing those abstractions unless it becomes multi-user.

LLM judge dedup and human review queues are likely too heavy for the local default path. MemClaw’s semantic dedup can call an LLM judge and enqueue review rows (`core-api/src/core_api/pipeline/steps/write/check_semantic_duplicate.py:187`, `core-storage-api/src/core_storage_api/services/postgres_service.py:722`). Spec Kit Memory should prefer deterministic thresholds, traceable preview output, and optional manual resolution over always-on judge/review infrastructure.

Document upsert is not safe enough for constitutional/manual local memory without extra gates. MemClaw’s document upsert overwrites `data` wholesale on conflict (`core-storage-api/src/core_storage_api/services/postgres_service.py:3670`, `core-storage-api/src/core_storage_api/services/postgres_service.py:3690`), and safety for keystones is provided upstream by trust and TOCTOU checks (`core-api/src/core_api/mcp_server.py:2298`, `core-api/src/core_api/mcp_server.py:2331`). Spec Kit Memory should not use blind upsert for manually curated records.
