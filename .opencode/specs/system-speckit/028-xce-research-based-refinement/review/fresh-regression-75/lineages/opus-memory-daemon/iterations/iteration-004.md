## Review: opus-memory-daemon — History + access-tracker concurrency

I traced both slice files and their real call sites (`vector-index-schema.ts` init path, `reconsolidation.ts`, `vector-index-mutations.ts`, `memory-retention-sweep.ts`). Findings below are grounded in code I read.

### What's solid (regression-checked)
- **History write ordering is correct.** `CURRENT_TIMESTAMP` has 1-second granularity, but every read path tie-breaks on `rowid` (`getHistory` L348-349, `resolveHistorySpecFolderStatement` L169, `getHistoryEventsForLineage` L368). Same-second events stay correctly ordered.
- **No intra-process counter race.** Node JS + synchronous better-sqlite3 means each `trackAccess`/`flushAccessCounts` runs to completion atomically; `access_count = access_count + 1` is atomic at the SQL level, so even cross-process daemon overlap can't corrupt the counter.
- **Accumulator-overflow flush (L90-98) doesn't mutate the Map during iteration** — `flushAccessCounts` only touches the DB; `clear()` happens after. No iterator invalidation.
- **Float accumulation is safe** — 0.1×5 reaches exactly 0.5 in IEEE-754, so the threshold fires on the 5th access as intended.

### Findings

**P1 — Legacy history rebuild runs outside any transaction → audit-log data loss on mid-migration crash.** `initHistory(database)` is invoked at `vector-index-schema.ts:2024`, *after* `run_all_migrations()` has already committed and returned (transaction defined L1964, invoked+closed L1974-1978). The legacy rebuild inside `history.init` (`history.ts:103-133`) issues `ALTER TABLE RENAME` → `CREATE` → `INSERT…SELECT` → `DROP` as separate `database.exec` calls, each auto-committing. A crash/SIGKILL between the RENAME and the INSERT (very plausible in this daemon-re-election epic) leaves an empty new `memory_history` plus an orphaned `memory_history_old`. Because the rebuilt table no longer contains `CHECK(actor IN` or `FOREIGN KEY`, the re-run guard at `history.ts:101` is then **false**, so the copy never re-executes — the entire history log is stranded permanently. Fix: wrap L103-133 in `database.transaction(() => { … })()`.

**P2 — `specFolderCache` is never invalidated, so history rows can get stale spec_folder attribution.** `resolveSpecFolder` returns the cached value whenever present (`history.ts:241-243`); the cache is only cleared on `init()` (L155) and only refreshed when a caller passes an explicit override (L236-238). `reconsolidation.ts:393` records an `UPDATE` event with **no** spec_folder argument, so it relies on cache/index. If a memory's folder changed after first resolution, the history row is attributed to the old folder — a traceability defect feeding `getHistoryStats`. Fix: invalidate/refresh the cache entry when `memory_index.spec_folder` is updated, or have callers pass the known folder.

**P2 — `reset()` and overflow-flush discard accumulated counts on flush failure.** `trackAccess` correctly re-stores the value when `flushAccessCounts` returns false (`access-tracker.ts:108-110`), but `reset()` (L251-255) and the overflow path (L93-97) call `flushAccessCounts` best-effort and then `clear()` unconditionally, so a transient DB-busy failure silently drops those popularity counts. Low impact (popularity only), but inconsistent with the careful handling in `trackAccess`. Fix: only delete/clear entries whose flush succeeded.

I checked the "unbounded growth" angle: `memory_history` has no prune path anywhere (`memory-retention-sweep.ts` only *appends* history), but a permanent audit log is a defensible design choice, so I'm not reporting it as a defect.
